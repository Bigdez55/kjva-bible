## Scope

### Issue: Private UITableViewCell Reorder Control String Matching
**File:** `Apollo16/VIEW CONTROLLERS/AllNotes.swift`, `setEditing(_:animated:)` function  
**Severity:** CRITICAL (App Store Review Risk)

**Current Code:**
```swift
override func setEditing(_ editing: Bool, animated: Bool) {
    super.setEditing(editing, animated: animated)
    
    for view in tableView.subviews {
        for cell in view.subviews {
            if String(describing: type(of: cell)) == "UITableViewCellReorderControl" {
                cell.isHidden = !editing
            }
        }
    }
}
```

**Problems:**
1. Uses private class name `"UITableViewCellReorderControl"` from string matching
2. Accesses private subview hierarchy
3. Static code analyzers (including App Store reviewers) detect this pattern
4. Breaks whenever UIKit renames or restructures internals

**Fix - Option A: Use Public API**
```swift
// Don't manipulate reorder controls directly.
// UITableViewDelegate.tableView(_:moveRowAt:to:) handles reordering.
// If you need custom reorder UI, subclass UITableViewCell.

override func setEditing(_ editing: Bool, animated: Bool) {
    super.setEditing(editing, animated: animated)
    // UITableView automatically shows/hides reorder controls
    // No custom manipulation needed
}
```

**Fix - Option B: Custom Cell Subclass**
```swift
// Create a custom UITableViewCell
class NoteCell: UITableViewCell {
    
    override func layoutSubviews() {
        super.layoutSubviews()
        
        // Only show/hide our custom reorder button, not private UIKit internals
        reorderButton.isHidden = !isEditing
    }
}

// Then in AllNotes:
override func setEditing(_ editing: Bool, animated: Bool) {
    super.setEditing(editing, animated: animated)
    
    for indexPath in tableView.indexPathsForVisibleRows ?? [] {
        if let cell = tableView.cellForRow(at: indexPath) as? NoteCell {
            cell.setEditing(editing, animated: animated)
        }
    }
}
```

**Recommended Approach:** Option A (let UITableView handle it)

The current code doesn't provide any visible benefit over letting UITableView manage reorder controls automatically. Removing it simplifies the code and fixes the App Store risk.

**Test:**
1. Tap Edit button
2. Verify table enters editing mode
3. Verify drag handles appear (default UITableView behavior)
4. Drag row to reorder
5. Tap Done
6. Verify table exits editing mode, drag handles disappear

### Audit: Check for Other Private API Usage
**Approach:** Scan entire codebase for common private API patterns

**Private API Red Flags:**
- String-based class name matching (`type(of: obj)` compared to string)
- `NSClassFromString` or `NSStringFromClass`
- Accessing private properties starting with `_`
- Accessing private methods with underscore prefix
- Accessing undocumented subview hierarchies
- KVC with private key paths (e.g., `setValue:forKey:` with private key)

**Scan Command:**
```bash
grep -r "UITableViewCellReorderControl\|_" Apollo16/ --include="*.swift" | \
  grep -v "// " | \
  grep -v "var " | \
  grep -v "let "
```
