# Apex AI Research Scout — Agent Memory

## Project Context

- Base model: RUTH, served via vLLM on NVIDIA L4 (24GB, SM89/Ada Lovelace)
- Fine-tuning plan: DoRA r=16 (10 adapters) + MoRA r=64 (2 adapters: trading_signals, fundamental_analyst)
- VRAM budget: ~19GB total (base FP16 + 12 adapters + KV cache + overhead); 5GB headroom
- Ruth pipeline: ~10–12K tokens input per call; well within native 32K window of RUTH

## Key Research Findings (2026-03-08)

### Model Selection

- RUTH-v2 (future candidate) (Apr 2025): same VRAM as RUTH, 131K native context, higher benchmark equivalent, dual thinking/non-thinking mode. RECOMMENDED migration target Q3 2026. vLLM >=0.8.5 required.
- DeepSeek-R1-Distill-14B: strong chain-of-thought (AIME 79.8%), but heavy reasoning fine-tune can disrupt JSON instruction-following. Better as SLERP merge ingredient than base swap.
- phi-4 (14B): DISQUALIFIED — 16K context window cannot accommodate Ruth's 10–12K input + response.
- Gemma-3-27B: MARGINAL — ~18–22GB at 4-bit; leaves no room for 12 DoRA adapters on L4.
- LLaMA-3.3-70B / other 72B models: require multi-GPU (>24GB at 4-bit). Not L4 compatible solo.

### PEFT Method Evidence

- DoRA r=4 beats LoRA r=4 by 22pp (61.89% vs 39.49%) on commonsense. ICML 2024 Oral (arXiv:2402.09353). Zero inference overhead (merges into weights).
- MoRA r=64 equals FFT on memorization; use only for trading_signals and fundamental_analyst adapters.
- DARE + TIES: confirmed valid for combining 12 domain adapters post-training. Already in ELSON_MODEL_MERGING_STRATEGY.md.

### vLLM Optimizations (L4-verified)

- AWQ 4-bit (Marlin kernel): ~9GB base VRAM vs ~28GB FP16. Marlin-AWQ: ~10x throughput over standard AWQ. <1% accuracy cost at 14B.
- Prefix caching: `--enable-prefix-caching`. ~70% TTFT reduction for Ruth's repeated system prompts. Conflicts with FP8 KV cache in vLLM <0.8.
- Chunked prefill: `--enable-chunked-prefill`. 86.4% TTFT reduction in long-context batches. Auto-enabled in vLLM V1 for >32K models.
- N-gram speculative decoding: `--speculative-model "[ngram]"`. Up to 2.8x speedup on structured outputs. Compatible with prefix caching in vLLM >=0.8.5.
- FP8 KV cache: `--kv-cache-dtype fp8`. Halves KV cache memory. Requires SM89+ (L4 qualifies). Mutually exclusive with prefix caching until vLLM >=0.8.5.
- Multi-LoRA serving: Up to 50% throughput reduction vs base model; 48.1% invalid KV caches during adapter switching (2025 research). Mitigate by batching calls per adapter type.

### Offline RL Method Selection

- TD3+BC: BEST for ~2,300 examples. Behavior cloning term leverages existing decisions even without Q-value targets. More stable than IQL at small dataset sizes.
- IQL: SECONDARY (run as ablation). Better at trajectory stitching; advantage emerges at 5K+ transitions.
- Decision Transformer + LLM init (arXiv:2411.17900, Nov 2024): innovative prototype. RUTH-initialized DT could outperform both in sparse-reward trading scenarios. Worth 2-week POC.
- XQL: NOT RECOMMENDED — temperature hyperparameter too fragile on small financial datasets.
- AWAC: FUTURE — best for offline-to-online fine-tuning phase after offline baseline is validated.

### Financial Benchmarks Gap

- No public benchmark evaluates structured trading signal generation (BUY/SELL/HOLD + confidence + stop-loss). FinBen's forecasting = binary direction only.
- Custom evaluation harness required before first DoRA adapter deployment. 5 metrics: signal calibration (Brier score), return attribution, regime-conditioned accuracy, JSON compliance rate, latency percentiles.
- 160 labeled outcome records is sufficient for pilot evaluation (500+ target after episode_terminal fix).

### Context Window

- Ruth's 10–12K tokens sits within RUTH's native 32K. YaRN not needed.
- Memory episode compression: reduce 3K tokens to ~500 tokens by summarizing each episode to 6 fields. Frees budget for wider signal lists.

## Key Research Findings (2026-03-15) — Ultimate Apex Sweep

### Model Scaling / Expansion (14B → 72B)
- SOLAR DUS (arXiv:2312.15166, NAACL 2024): copy 32-layer model, remove 8 from top+bottom, concatenate → 48 layers. 7B → 10.7B with continued pretraining (~3B tokens). Could apply 14B → ~20B. NOT proven to 72B scale.
- Passthrough merging (mergekit): concatenate layer slices from two 14B models → ~28B "Frankenmerge". Needs continued pretraining to recover. GPU-free merge.
- MINITRON (NVIDIA): structured pruning 15B → 8B/4B using only 1/40th of original training tokens. Best for shrinking, not growing.
- Realistic path to 72B: fine-tune Qwen2.5-72B or Qwen3-32B on Elson data — do NOT try to grow RUTH.

### Best Financial Merge Candidates (confirmed HuggingFace)
- `Writer/Palmyra-Fin-70B-32K` — passed CFA Level III (73%), 32K context, beats GPT-4 on financial QA. Best 70B financial model for merge.
- `TheFinAI/Open-FinLLMs` — FinLLaMA (LLaMA3-8B base, 52B token pretraining, Sharpe 1.41 in trading sim). Best 8B merge candidate.
- `deepseek-ai/DeepSeek-R1-Distill-Qwen-14B` — 14B reasoning model, outperforms QwQ-32B. Use as SLERP ingredient for reasoning uplift on RUTH.
- `Qwen/Qwen2.5-72B-Instruct` — 128K context, Apache 2.0, best open-weight 72B baseline for financial fine-tuning.
- `Qwen3-32B` (Qwen3 family) — matches Qwen2.5-72B on benchmarks at half the size. Apache 2.0.
- FinGPT (AI4Finance-Foundation): LoRA-based, <$300 to train, best for sentiment tasks. Smaller models (7B-13B).

### Preference Optimization (new finding)
- FinDPO (arXiv:2507.18417, Jul 2025): first finance DPO framework. LLaMA-3-8B + DPO + LoRA r=16. Sharpe 2.0 in simulation, +11% sentiment accuracy vs SFT. DIRECTLY applicable to RUTH.
- ORPO (arXiv:2403.07691): no reference model needed, single-stage SFT+alignment. Faster than DPO for financial instruction tuning.

### Advanced Fine-tuning Updates
- NEFTune: adds Gaussian noise to embeddings during training. LLaMA-2-7B: 29.79% → 64.69% on AlpacaEval. +8-10% on instruction datasets. Zero compute cost. Add `neftune_noise_alpha=5` to trainer args.
- GaLore (arXiv:2403.03507): 65.5% memory reduction vs LoRA for optimizer states. Better than LoRA for continued pretraining (full-rank updates). Use for domain corpus pretraining, not adapter fine-tuning.
- MoRA (arXiv:2405.12130): square matrix high-rank updates, same params as LoRA. Outperforms LoRA on memory/knowledge tasks. Already planned for trading_signals + fundamental_analyst adapters.

### Cascade / Hierarchy Systems
- Unified Routing+Cascading (ETH Zurich, 2024): iterative best-model selection, up to 14% over fixed cascade. Applies to RUTH-14B as tier-1 with Palmyra-Fin-70B as tier-2 escalation.
- Speculative Cascades (Google Research): combines speculative decoding + cascade. RUTH drafts, 72B verifies for high-stakes signals only.
- LLM ensembles: majority voting + higher-order aggregation. Crypto trading ensembles: higher win rates, lower loss rates than individual models.

### Inference Optimizations (confirmed)
- AWQ+Marlin: 741 tok/s vs 67 tok/s (standard AWQ). 10.9x speedup. <1% accuracy loss at 14B. BEST for L4 production.
- N-gram speculative decoding: 2.8x speedup on structured/repetitive outputs (CNN/DM dataset). Relevant for RUTH JSON signal outputs.
- Model cascade: route simple signals to RUTH-14B, escalate complex to 72B API — avoids VRAM constraint.

### Synthetic Data Generation
- Evol-Instruct (WizardLM, ICLR 2024): rewrites instructions into more complex variants. Use to expand 46K DoRA examples by complexity dimension.
- FinDPO approach: generate preference pairs (preferred=good signal, rejected=bad signal) from TradeDecisionLog outcomes. Only needs 160 labeled decisions to start.
- DeepSeek-R1 reasoning traces: distill 800K reasoning samples → financial chain-of-thought. Filter for trading relevance.

## AWQ Quantization Workflow (2026-03-15) — RUTH Deployment Research

### Tool: Use llm-compressor (NOT AutoAWQ — archived May 11 2025, read-only)
- `pip install llmcompressor` — vLLM's official successor to AutoAWQ
- Repo: https://github.com/vllm-project/llm-compressor
- AWQ output is vLLM-compatible; quantization_config auto-detected from saved model

### Merge step (PEFT, non-quantized base only):
- RUTH DoRA adapter was trained on FP16 base → can use `merge_and_unload()` cleanly
- DoRA merge requires non-quantized linear layers — RUTH base qualifies (FP16, not BNB/GPTQ)
- Save merged model as FP16 safetensors, then hand off to llm-compressor

### AWQ quantization key params:
- scheme: `W4A16_ASYM` (4-bit weights, 16-bit activations, asymmetric)
- ignore: `["lm_head"]` — always exclude output head
- calibration: 256 samples, 512 tokens, `HuggingFaceH4/ultrachat_200k` or custom financial data
- `duo_scaling="both"` recommended

### GPU requirements during quantization (NOT inference):
- Full FP16 model must load into GPU VRAM: 14B FP16 = ~28GB weights
- Plus calibration activations: estimate 40–50GB total VRAM needed
- H100 80GB: comfortably fits 14B quantization. A100 40GB: MARGINAL (use device_map="auto" to CPU offload)
- L4 24GB: CANNOT quantize 14B — must use H100 VM (`elson-h100-spot`)
- MacBook CPU: too slow (hours), not recommended

### Quantization time estimates:
- H100 80GB: ~10–20 min for 14B (calibration is the bottleneck, not weight conversion)
- A100 40GB: ~20–30 min with CPU offload
- CPU only: potentially 4–8+ hours, not practical

### Output size after AWQ 4-bit:
- 14B params × 4 bits / 8 = ~7GB base weights
- With scaling factors and metadata: ~8–9GB total
- Confirmed by Qwen2.5-14B-Instruct-AWQ model cards on HF

### HF Inference Endpoints engine:
- TGI is in MAINTENANCE MODE as of 2025-12-11 — HF now recommends vLLM or SGLang
- vLLM engine on HF Inference Endpoints: auto-detects AWQ from quantization_config in config.json
- No `--quantize awq` flag needed when loading llm-compressor output (config embedded in model)
- Hardware tier for ~8GB AWQ model: single Nvidia L4 (24GB) or A10G (24GB) on HF Endpoints
- vLLM on HF Endpoints: set `tensor_parallel_size=1` for single GPU, `data_parallel_size=N` for throughput

### Qwen2-specific gotchas:
- AutoAWQ (archived) had Qwen2-72B failures and NaN errors — llm-compressor resolves these
- MoE variants NOT supported by AWQ (Qwen2-MoE family only — RUTH/Qwen2 dense is fine)
- Qwen2 architecture fully supported in llm-compressor (has Qwen3 examples, Qwen2 is subset)
- llm-compressor needs custom AWQ mappings for non-Llama architectures — check if Qwen2 mapping exists in registry before running

### Complete pipeline (high level, do not implement — research only):
1. On H100: load FP16 RUTH base + DoRA adapter via PEFT
2. Call merge_and_unload() → save FP16 merged model
3. Run llm-compressor oneshot() with AWQModifier (W4A16_ASYM, 256 samples)
4. model.save_pretrained(save_compressed=True) + tokenizer.save_pretrained()
5. huggingface_hub push_to_hub("Bigdez/Ruth-L4")
6. On HF Inference Endpoints: select vLLM engine, L4 or A10G hardware, no extra quantize flag
