Phase 1 GCP infrastructure tasks (G5-G7) have been successfully completed. The L4 GPU VM is running, vLLM is serving the RUTH model with int8 quantization, and Cloud Scheduler is configured for market-hours-only operation. However, a **CRITICAL security gap** was identified during this audit:

This violates the execution plan specification of "Port 8000 Cloud Run VPC only" and exposes the AI model to unauthorized access and potential abuse. This must be remediated before production use.

### 2. SECURITY: CRITICAL FAILURE

**Finding:** Firewall rule `allow-vllm` permits access from 0.0.0.0/0 (entire internet).

**Current Configuration:**
```yaml
name: allow-vllm
sourceRanges:
- 0.0.0.0/0
targetTags:
- http-server
allowed:
- IPProtocol: tcp
  ports:
  - '8000'
```

**Execution Plan Specification (G5):**
> "Port 8000 accessible via VPC only (firewall rule: deny external, allow internal)"

**Risk Assessment:**
- vLLM OpenAI-compatible API has NO built-in authentication
- Anyone with the IP (136.109.205.148) can:
  - Query the model for financial advice
  - Generate unlimited completions (cost/quota abuse)
  - Extract model behavior patterns
  - Launch denial-of-service attacks
- Estimated time to discovery: hours to days via automated internet scanning

**Required Remediation (BLOCKING for C2):**
1. Create VPC Serverless Connector for Cloud Run (elson-backend) in us-west1
2. Update Cloud Run service to use VPC connector
3. Update firewall rule `allow-vllm` to restrict source to VPC CIDR block only
4. Remove public External IP from L4 VM (optional but recommended)
5. Update backend env vars: VLLM_BASE_URL=http://10.138.0.4:8000 (internal IP)

**Estimated Cost:** VPC connector ~$7/mo (e2-micro, 2-10 instances)

**Responsible Agent:** GCP Agent (infrastructure) + Repo Agent (env var update)

**Decision Required:** Immediate remediation or accept risk and document?
