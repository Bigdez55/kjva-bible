# Single-User Architecture Lockdown (2026-02-21)

## Context
Alpaca individual trading account can only support single user (bigdez55@gmail.com). Multi-user registration violated Alpaca terms and created security risk. Implemented defense-in-depth security lockdown while preserving multi-tenant database structure to avoid massive refactor.

## Implementation Strategy
4-phase execution: (1) Registration/Login Lockdown, (2) Admin Endpoint Hardening, (3) Session Hardening & Rate Limiting, (4) Cache Metrics (no changes needed).

## Key Files Modified
- `backend/app/core/config.py` - ALLOWED_USER_EMAIL setting, JWT expiry (15min/7day), rate limit config (5 attempts/15min)
- `backend/app/api/api_v1/endpoints/auth.py` - Email whitelist checks (403 on register, 401 on login with generic message)
- `backend/app/api/api_v1/endpoints/admin_market_data.py` - Added Depends(get_current_admin_user) to 5 endpoints + new /status endpoint
- `frontend/src/pages/RegisterPage.tsx` - Single-user mode warning banner
- `cloudbuild.yaml` - Added ALLOWED_USER_EMAIL to --update-secrets
- `backend/tests/test_auth_single_user.py` - NEW (10 test cases, 2 core passing)

## Security Improvements
1. Email whitelist - ONLY bigdez55@gmail.com can register/login
2. Admin endpoint protection - All 6 admin market data endpoints require is_superuser=True
3. JWT hardening - 15min access (down from 120min), 7-day refresh (down from 30-day)
4. Rate limiting - 5 failed login attempts per 15 minutes (prevents brute force)
5. Generic error messages - Login returns "incorrect email or password" (no information leakage)

## Lessons Learned
1. **Multi-tenant preservation**: Kept User table + FKs intact; added lockdown layer on top. Easier than full schema migration.
2. **Generic error messages**: Login uses same "incorrect email or password" for unauthorized email AND wrong password (prevents email enumeration).
3. **Test DB schema drift**: Test fixtures use SQLite in-memory; missing columns from migrations. Core auth logic still testable via API.
4. **JWT expiry tuning**: 15min/7day balances security vs UX (was 120min/30day - too permissive for single-user app).
5. **Rate limit config externalization**: Moved hardcoded LOGIN_RATE_LIMIT/WINDOW to settings.LOGIN_RATE_LIMIT_* (now configurable via env vars).
