## Lessons Learned
- vLLM port conflict: startup-script-url runs AT BOOT but vllm.service also starts at boot; if vllm.service starts first and binds port 8000, the startup script gets OSError 98. The systemd service is the authoritative launcher -- the startup script is only for initial model download on a fresh disk.
- RUTH bfloat16 dense model = ~28GB; L4 has 22.5GB VRAM; ALWAYS use --quantization bitsandbytes (NF4) -- without it the model OOMs at load time before serving a single request.
- bitsandbytes is installed in user site-packages (/home/bigdez55/.local); systemd unit MUST set Environment=HOME=/home/bigdez55 and Environment=PYTHONUSERBASE=/home/bigdez55/.local or the package won't be found.
- vLLM 0.15.1 uses v1 engine (multiprocess architecture with EngineCore subprocess); --enforce-eager is NOT needed for single-GPU setup with bnb4bit (enough headroom for CUDA graph capture).
- Always verify _scan_portfolio() calls ALL check methods, not just some
- AdvisorLayer needs explicit endpoint wiring to be useful
- Documentation can drift from code -- always cross-check tracker vs execution plan
- Phase 2 Local Agent wrote async code for sync codebase -- always verify DB session pattern
- Double-prefix bug: router decorators had /sentinel + api.py prefix /sentinel = /sentinel/sentinel
- Numeric(18,8) required for financial columns, not Float (precision loss in fintech)
- Migration a1b2c3d4e5f6 depends on add_ssn_columns_2025_02 (Alembic chain)
- Notifications table must exist before migration runs (adds columns to it)
- `gcloud sql instances patch` does NOT take --region flag
- Cloud Build smoke test only accepts "healthy" (not "degraded")
- `gcloud ai custom-jobs create --config` takes only the jobSpec portion (workerPoolSpecs etc.), NOT the full CustomJob object
- Vertex AI GPU image requires accelerator in machineSpec; CPU images are very old (1.4 only) -- no viable CPU path via Vertex AI for new Python
- For on-demand CPU merge jobs: use n1-highmem-16 (16 vCPU, 104GB RAM) with pd-balanced disk in us-west1-b (not us-west1-a to avoid SSD quota conflict with L4 VM)
- **CRITICAL (2026-02-21):** Cloud SQL was UNTRACKED by Alembic (no alembic_version table) despite having all 43 tables. Always run `alembic stamp head` after `create_all()`. Use `alembic heads` before creating migrations to detect branching.
- Alembic merge resolution: If multiple heads exist, run `alembic merge -m "msg" <head1> <head2>`, stamp DB with both heads, then upgrade to merge revision.
- alembic/env.py uses `DATABASE_URL` env var (not `SQLALCHEMY_DATABASE_URL`) for Cloud SQL connections.
