## 7. CODE vs GCP DRIFT (CRITICAL SECTION)

### DRIFT-03 (HIGH): Uncommitted Dockerfile changes NOT deployed
The working tree has changes to 24 files that are NOT committed or deployed:

**Dockerfile changes (uncommitted):**
```
+COPY backend/alembic ./alembic
+COPY backend/alembic.ini ./alembic.ini
```
The deployed container does NOT contain Alembic migration files.
The uncommitted entrypoint.sh adds `alembic upgrade head` before gunicorn start.
**Impact:** New migration files (auto_trading_sessions, trade_decision_log) have never
run against production Cloud SQL. These tables may exist only if created by `create_all()`.

**requirements-docker.txt changes (uncommitted):**
```
+google-cloud-storage>=2.14.0
+onnxruntime>=1.18.0
+polygon-api-client>=1.13.0
```
**Impact:** Any code using `polygon` or `onnxruntime` imports will crash in production
if those packages are not already installed. The deployed image was built from the
committed requirements-docker.txt which lacks these 3 packages.

**market_data.py + polygon.py changes (uncommitted):**
158 lines changed in market_data.py, 30 new lines in polygon.py.
These are NOT deployed.

### DRIFT-04 (HIGH): Alembic migrations never run against production
- The deployed entrypoint.sh does NOT run `alembic upgrade head`
- The deployed Dockerfile does NOT copy alembic files into the container
- Production DB was initially set up with `create_all()` and `alembic stamp head`
- Migration chain: initial -> ... -> merge_heads_2026_02_23 -> add_bot_sessions_20260224
- The `auto_trading_sessions` table may not exist in production
- The `trade_decision_logs` table may not exist in production

### DRIFT-05 (MEDIUM): Python version mismatch documentation
- Dockerfile: `python:3.13-slim` (Python 3.13)
- MEMORY.md / project docs: "Python 3.11+"
- Local system: Python 3.13.7
- **Impact:** No functional issue, but documentation is misleading

### DRIFT-06 (MEDIUM): ALPACA_API_KEY vs ALPACA_PAPER_API_KEY confusion
- Cloud Run mounts BOTH `ALPACA_API_KEY` and `ALPACA_PAPER_API_KEY`
- `config.py` has separate fields for both
- `ALPACA_API_KEY` is the legacy generic field; `ALPACA_PAPER_API_KEY` is the new explicit field
- Code in `alpaca.py` may use either depending on paper/live mode
- **Risk:** If both point to different Alpaca accounts, mode switching could use wrong credentials

### DRIFT-07 (MEDIUM): Redis rate limiting still disabled in production
- `REDIS_URL` secret exists in Secret Manager (created 2026-02-22)
- But no Memorystore Redis instance exists (billing disabled blocks creation)
- `security.py` correctly handles this: `_create_redis_client()` returns None on failure
- Rate limiting degrades gracefully but is NOT active
- **Impact:** Login brute-force protection is not operational
