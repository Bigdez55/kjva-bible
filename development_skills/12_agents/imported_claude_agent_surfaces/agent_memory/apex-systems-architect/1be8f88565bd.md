## FIX 1: XKABI Rights System Coherence
### Files
- kernel/xenos/include/xkabi_rights.h
- aetherboot/src/capability_init.c

### Verdict: CORRECT with one noted gap

### Findings

**F1-A: Canonical include is correct.**
capability_init.c now does `#include "xkabi_rights.h"` (line 46) after typedef-ing
xkabi_rights_t as UINT32 (line 44). The old D-S1-05 local definitions used sequential
0x01-0x80 values that diverged from the kernel's bit-shifted values (e.g. NET_SEND=0x10
vs correct 1u<<17=0x20000). That divergence is eliminated. The include path resolves to
the kernel header directory — the compile command must carry -Ikernel/xenos/include.

**F1-B: Protocol classification grants are correct for AetherBoot.**
AetherBoot runs in UEFI pre-ExitBootServices — ring-0 equivalent with full firmware access.
The classify_protocol() function grants:
  - GOP handle   → XKABI_RIGHT_GPU_EXEC | XKABI_RIGHT_GPU_MAP  (bit 19, bit 1) — correct
  - Block IO     → XKABI_RIGHT_STORAGE_READ | STORAGE_WRITE (bits 15, 16) — correct
  - Network      → XKABI_RIGHT_NET_SEND | NET_RECV (bits 17, 18) — correct
  - Console out  → XKABI_RIGHT_CONSOLE_OUT (bit 4) — correct

**F1-C: XKABI_RIGHT_FIRMWARE_VAR is NOT granted by capability_init.c.**
No EFI_VARIABLE_WRITE_PROTOCOL GUID or EFI_RUNTIME_SERVICES_SUPPORTED GUID is
classified. This is architecturally intentional: AetherBoot does not grant firmware
variable write rights to kernel handles enumerated from EFI. However, if Sprint 4
introduces a UEFI secure boot enrolment path (MOK enrolment, kernel signing), a
FIRMWARE_VAR handle will need to be explicitly added with strict scope constraints.
This is a gap to track, not a defect.

**F1-D: XKABI_RIGHT_DERIVE not granted to any boot handle.**
Correct. AetherBoot handles should not be able to spawn sub-capabilities autonomously.
The DERIVE bit (bit 23) is reserved for post-boot kernel capability management only.

**F1-E: Unknown handles receive XKABI_RIGHT_NONE.**
classify_protocol() returns XKABI_RIGHT_NONE for any handle whose installed protocols
do not match the 4 recognized GUIDs. All handles are enumerated (AllHandles, not
ByProtocol) and stored — zero-rights entries are still recorded in the cap table.
This is correct: the kernel can audit what handles existed at boot even if they have
no rights.

**F1-F: Version coupling risk.**
If xkabi_rights.h adds or shifts bits in a future sprint (e.g. adding a
XKABI_RIGHT_DMA bit at position 11 or 12), capability_init.c's classify_protocol()
will silently grant stale combined values. Mitigation: the file has a clear comment
"ADDING A RIGHT: add here ONLY. Propagate to GSD parser string table." The propagation
note does not mention capability_init.c. Sprint 4 should add capability_init.c to that
propagation note. This is a documentation gap, not a correctness defect today.

**F1-G: Cap table storage model is architecturally sound.**
g_cap_table is static BSS in the bootloader image. UEFI maps bootloader image pages as
EfiBootServicesCode/Data. After ExitBootServices, those pages remain valid until the
kernel explicitly reclaims them. xenos_boot_info_t.mem_map_addr carries the physical
address. The kernel's cap_table_init() reads the magic (0x584B414249434150 = "XKABICAT")
and must copy before reclaiming EfiBootServicesData. The magic validation is present in
xkabi_capabilities.c. The copy-before-reclaim sequencing is a Sprint 4 kernel init
responsibility that must be explicitly tested.

## FIX 2: TLS 1.3 Audit Architecture
### Files
- sec/xsec/tls/tls13.c
- sec/xsec/audit/audit.c
- sec/xsec/include/xsec.h

### Verdict: CORRECT with one forensic gap (P1 for Sprint 4)

### Findings

**F2-A: XSEC_MODULE_TLS13 = 7 is the correct module identifier.**
Confirmed from xsec.h lines 508-516:
  XSEC_MODULE_SHA256=1, HMAC=2, AES_GCM=3, X25519=4, CHACHA20=5, ED25519=6,
  TLS13=7, X509=8, XPKG=9.
All 6 new audit call sites use XSEC_MODULE_TLS13. Correct.

**F2-B: Audit ring concurrency is correct for concurrent TLS sessions.**
The ring buffer write protocol (audit.c lines 145-163):
  1. pal_atomic_add(&write_idx, 1u) — atomic fetch-and-increment gives each concurrent
     caller a unique slot index.
  2. slot = idx % XSEC_AUDIT_RING_SIZE — each slot is owned by exactly one writer.
  3. Field writes are non-atomic but per-slot exclusive — no races between writers.
  4. pal_atomic_add(&total_events, 1u) — completion marker is atomic.
The spinlock is used ONLY for dump/query (xsec_audit_dump, xsec_audit_query_recent),
not for the hot write path. This is the correct lock-free producer, locked consumer pattern.
Concurrent TLS sessions each claim a unique slot. No ABA risk because slot indices are
monotonically increasing (write_idx never resets). CORRECT.

**F2-C: Overflow counter correctness.**
The overflow check compares total_before (loaded BEFORE incrementing write_idx) against
XSEC_AUDIT_RING_SIZE. There is a minor TOCTOU window: two threads can both observe
total_before < RING_SIZE, both claim slots N and N+1, and only the second one overwrites
a valid entry, but neither increments dropped_events. The dropped_events counter is
therefore a lower-bound approximation, not exact. For a security audit log this is
acceptable — it detects high-volume overflow without false-zero guarantees. The comment
in audit.c correctly states "detect data loss" not "exactly count data loss."

**F2-D: Audit call site placement — all 6 are BEFORE the return statement.**
Verified by grep output: every xsec_audit_log() in tls13.c precedes the return of the
error code. This is architecturally correct — an audit event that fires after the error
return would never execute. The order is:
  [tls_send_alert()] → xsec_audit_log() → return XSEC_ERR_*
This is the right sequence: alert is sent to peer first (notifying them of rejection),
then the event is recorded locally, then the error is returned to the caller.

**F2-E: Event type mapping is semantically correct.**
  - Alert received from peer              → XSEC_AUDIT_TLS_ALERT_RECV  (correct)
  - AEAD tag mismatch on decrypt          → XSEC_AUDIT_AUTH_TAG_MISMATCH  (correct)
  - Unexpected message type in handshake  → XSEC_AUDIT_HANDSHAKE_FAILED  (correct)
  - Unsupported cipher suite              → XSEC_AUDIT_HANDSHAKE_FAILED  (correct)
  - Certificate chain failures            → XSEC_AUDIT_CERT_REJECTED  (correct)
  - CertificateVerify decode failures     → XSEC_AUDIT_HANDSHAKE_FAILED  (correct)
  - ServerFinished HMAC mismatch          → XSEC_AUDIT_HANDSHAKE_FAILED  (correct)
  - Handshake complete                    → XSEC_AUDIT_HANDSHAKE_COMPLETE  (correct)
  - Session close                         → XSEC_AUDIT_SESSION_CLOSED  (correct)

**F2-F: FORENSIC GAP — no session ID or peer IP in audit detail strings.**
xsec_tls_ctx_t carries: client_random[32], server_random[32], hostname[256].
None of these are included in the 64-byte detail string of audit events. An operator
receiving XSEC_AUDIT_HANDSHAKE_FAILED cannot determine WHICH session failed or from
WHICH peer IP. The detail string contains only a human-readable description.

This is a P1 Sprint 4 prerequisite: add a session_id field (e.g. truncated SHA-256 of
client_random, 8 hex chars) to xsec_tls_ctx_t and prepend "sid=XXXXXXXX " to audit
detail strings in tls13.c. The 64-byte detail limit (XSEC_AUDIT_DETAIL_LEN) is tight
but sufficient for "sid=XXXXXXXX: <description>" patterns.

**F2-G: Non-Ed25519 chain advisory log is architecturally correct.**
Line 861-865: XSEC_ERR_NOT_SUPPORTED from xsec_x509_verify_chain() is logged as
XSEC_AUDIT_CERT_REJECTED but does NOT abort the handshake (the TODO-S4 comment is
explicit). This is a deliberate Sprint 3 posture: log-and-continue for RSA/ECDSA chains
until those verifiers are implemented. It is NOT a security regression vs. S2 (TLS was
not implemented at all in S2). Sprint 4 must harden this: non-Ed25519 chains must be
rejected after RSA-PSS/ECDSA stubs are removed and replaced with real verifiers.

## FIX 3: X.509 TBS Chain of Trust
### Files
- sec/xsec/cert/x509.c (lines 459-609)

### Verdict: CORRECT — implementation matches RFC 5280 precisely

### Findings

**F3-A: tbs_start capture point is canonical RFC 5280.**
RFC 5280 §4.1.2 defines:
  Certificate ::= SEQUENCE {
      tbsCertificate   TBSCertificate,   -- signed over this entire DER encoding
      signatureAlgorithm AlgorithmIdentifier,
      signature        BIT STRING
  }
The signature is computed over the complete DER encoding of TBSCertificate, which
includes the outer SEQUENCE tag (0x30), the length field (1-4 bytes, variable), and
the content. Capturing tbs_start = c.pos BEFORE der_enter() (which consumes tag + length)
means tbs_start points to the 0x30 tag byte. After der_enter(), c.pos points to the
first content byte, and tbs_seq_end = c.pos + tbs_len points one byte past the last
content byte. Therefore:
  out->tbs_data = der + tbs_start         (points to 0x30 tag)
  out->tbs_len  = tbs_seq_end - tbs_start (tag + length_bytes + content_bytes)
This is exactly the byte range that the CA signs. CORRECT.

**F3-B: tbs_len manipulation blast radius.**
tbs_len is derived from the DER length field inside the input buffer (controlled by the
cert issuer, already signature-verified by xsec_ed25519_verify before tbs_data is used).
The only path where tbs_len is used for cryptographic verification is:
  xsec_ed25519_verify(issuer->pub_key, cert->tbs_data, (size_t)cert->tbs_len, cert->signature)
in xsec_x509_verify_chain(). The bounds check tbs_seq_end > tbs_start (line 609) prevents
a zero or negative span. However, tbs_seq_end itself is bounded by der_enter()'s check:
  if (c.pos + len > c.end) { c->error = true; return 0u; }
which prevents tbs_len from extending beyond the input buffer. If an attacker provides a
malformed cert where tbs_len overlaps signatureAlgorithm or signature bytes, the
xsec_ed25519_verify call will simply fail (signature won't match). No out-of-bounds
read is possible because tbs_data + tbs_len <= der + der_len (enforced by der_enter).
BOUNDS CHECK IS SUFFICIENT.

**F3-C: Comment accuracy.**
The comment at lines 463-466 is accurate:
  "tbs_start points to the 0x30 tag byte. tbs_seq_end - tbs_start then covers
   tag + variable-length field + content = full DER encoding per RFC 5280 §4.1.2"
This is a correct description of what the code does and why. Sufficient for future
maintainers. The fix tag D-S3-P1-003 is referenced, linking to the original defect.

**F3-D: One subtle issue — tbs_len assignment uses tbs_seq_end, not c.pos at line 609.**
tbs_seq_end is computed as c.pos + tbs_len immediately after der_enter() (line 472),
BEFORE the TBS content is parsed. The parser then uses c.pos = tbs_seq_end (line 586)
to skip to the end of TBS. The final assignment:
  out->tbs_len = tbs_seq_end > tbs_start ? (tbs_seq_end - tbs_start) : 0u
uses tbs_seq_end correctly — it does NOT use the final c.pos value (which has been
advanced past TBS). This is correct. A naive maintainer might mistakenly replace
tbs_seq_end with c.pos here; the ternary guard prevents a silent zero but would not
prevent using the wrong end marker. The comment should note that c.pos must NOT be
substituted for tbs_seq_end here. This is a documentation-only recommendation.

## Cross-Layer Integration Analysis

### XKABI Fix → XPKG refusal_gate.c Gate 2 (CAPABILITY)
**Current state:** Gate 2 uses a path-prefix heuristic (/sys/, /dev/, /kernel/, /boot/, /init/)
and returns XPKG_GATE_RESULT_WARN for restricted paths. It does NOT read XKABI rights
bitmasks from xkabi_rights.h at all. The comment in refusal_gate.c explicitly defers
this: "Full capability manifest is a Sprint 4 item (XKABI integration)."

**Impact of XKABI rights fix:** Zero impact on Gate 2 today. The fix only corrects
the bitmask values that AetherBoot and GENSD use. Gate 2 will need to be upgraded in
Sprint 4 when xpkg_package_t gains a capabilities field (uint32_t) aligned to the
xkabi_rights.h bitmask. At that point Gate 2 must compare pkg->metadata.capabilities
against an allowed-capabilities manifest and use XKABI_RIGHT_* constants from the
canonical header. The rights fix ensures those constants will be correct when that
upgrade lands.

**Sprint 4 prerequisite:** Add `uint32_t capabilities` field to xpkg_manifest_t and
xpkg_package_t. Upgrade Gate 2 to compare against a per-install-path allowed-rights mask
derived from xkabi_rights.h. Wire XPKG refusal_gate.c to include xkabi_rights.h.

### TLS Audit Fix → GENSD supervisor.c Health Check
**Current state:** supervisor.c has no TLS awareness. supervisor_probe_http() is a
Sprint 1 stub that always returns HEALTH_OK. It does not call any XSEC function and
does not read from the XSEC audit ring.

**Impact of TLS audit fix:** Zero impact on supervisor.c today. The supervisor does not
route through TLS for its health probes. However, when Sprint 4 wires supervisor_probe_http()
to XNET TCP (which will use XSEC TLS for HTTPS endpoints), TLS handshake failures will
generate XSEC_AUDIT_HANDSHAKE_FAILED events. The supervisor currently has no mechanism
to inspect the audit ring and escalate health probe failures beyond its own journal_write().

**Sprint 4 prerequisite:** Add an optional audit-ring query to supervisor_tick():
after a HEALTH_FAIL event, call xsec_audit_query_recent() and scan for
XSEC_AUDIT_HANDSHAKE_FAILED events with matching session context. If found, log with
LOG_ERROR (not LOG_WARN) to indicate that the health probe failure was a TLS attack
vector rather than a transient network issue. Also periodically check
xsec_audit_get_dropped() > 0 and journal a WARNING.
