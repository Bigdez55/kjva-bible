The hard staged-name-check gate (`git diff --cached --name-only` returning exactly the intended file) prevents OUTBOUND contamination but does NOT prevent INBOUND contamination during the window between `add` and `commit`. The atomic single-Bash-call `add && commit` pattern is the only reliable mitigation. When even that fails (a parallel agent commits in the window), the work still lands — git's tree-of-blobs preserves it; only the commit attribution shifts.

## Recovery from accidental `git reset`

If you reset a commit another agent has already documented as durable (referenced in STAGE4_PROGRESS or another file): DON'T clean-restage as if it were stray work. Use `git reset --hard <reflog-sha>` to restore. The reflog persists for 90 days. The "destructive" label is misleading when the operation is the inverse of YOUR own reset mistake.

I made this mistake mid-slice and recovered correctly via `git reset --hard ff3ba95` after consulting advisor. Without that consult I'd have re-attempted "clean restaging" and obliterated peer documentation.
