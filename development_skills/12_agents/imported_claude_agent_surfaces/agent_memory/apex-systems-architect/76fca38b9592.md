## AUDIT FINDINGS

### 1. API CONTRACT CONSISTENCY

#### 1a. Does XSTORE use PAL for all I/O?

**FINDING: PARTIAL — by design, correct architecture.**

`xstore.h` (§10, lines 313–317) defines `xstore_t` with I/O callbacks:
```c
xstore_status_t (*read_page)(void *io_ctx, uint32_t page_no, uint8_t out[XSTORE_PAGE_SIZE]);
xstore_status_t (*write_page)(void *io_ctx, uint32_t page_no, const uint8_t in[XSTORE_PAGE_SIZE]);
xstore_status_t (*sync)(void *io_ctx);
void            *io_ctx;
```
XSTORE does NOT call `pal_pages_alloc` directly for page I/O — it uses the caller-injected I/O callbacks (correct IOC pattern). Internal in-memory state uses `pal_spinlock_t` (from pal.h). The page cache `xstore_cache_entry_t.last_access` uses `pal_time_now_ns()`. This is correct: XSTORE is I/O-agnostic via callbacks, not raw PAL calls. No raw arrays used for backing storage.

**Verdict: COMPLIANT.** The I/O abstraction is clean. The caller provides the PAL-backed block I/O driver as a callback.

#### 1b. Does XMIND use XBLOB for weight loading?

**FINDING: GUARDED — conditional compilation, incomplete default path.**

`weights_loader.c` (§9 comment, line 7):
> "When compiled with -DXMIND_HAS_XBLOB, xmind_weights_load() accepts a blob store and a 32-byte SHA-256 hash..."

`wl_alloc_pages()` (line 363) calls `pal_pages_alloc` + `pal_map_pages` directly — this is correct for the weight tensor memory itself (weights are mmapped pages, not XBLOB blobs). XBLOB is used for the model file retrieval path, which is `#ifdef XMIND_HAS_XBLOB` guarded.

Sprint 8 limitation explicitly acknowledged (line 433):
> "Weight pointers remain NULL; xmind_forward() will produce zero-valued outputs until Sprint 9 completes the loader."

**Verdict: ARCHITECTURAL GAP (P2).** The XBLOB-backed loader is conditionally compiled. In a production build without `-DXMIND_HAS_XBLOB`, XMIND falls back to `pal_file_open()` which is NOT declared in `pal.h` — it is declared locally in `weights_loader.c` as an extension stub (§1). This means the file I/O path has no implemented backend in the current codebase.

### 3. XJIT SAFETY BOUNDARY

#### 3a. NX Bit Management — Code Page Executability

**FINDING: CRITICAL GAP (P1).**

`xjit_codebuf_t` (xjit.h lines 223–227) is defined as:
```c
typedef struct {
    uint8_t  buf[XJIT_CODE_BUF_SIZE];
    uint32_t pos;
    uint32_t capacity;
} xjit_codebuf_t;
```

`buf[]` is a **static array embedded inside `xjit_func_t`**. It is stack or BSS memory, never dynamically allocated via `pal_pages_alloc`. As a consequence:

- There is NO call to `pal_map_protect()` with `PAL_MAP_EXEC` anywhere in `jit/` (confirmed by grep — zero matches for `pal_map_protect`, `PAL_MAP_EXEC`, `mprotect`, `NX` in the entire jit/ subtree).
- `xjit_get_fn()` (codegen.c line 861) casts `f->code.buf` directly to a function pointer and returns it.
- On a real XENOS target with W^X enforced at the VMM level, executing code from a data-mapped region (stack/BSS) will fault.
- The current design works only because the freestanding target has no W^X enforcement yet.

**Recommendation:** `xjit_codebuf_t.buf` must be replaced with a PAL-allocated executable mapping. After `xjit_compile()` completes emission, the implementation must call `pal_map_protect(map_handle, PAL_MAP_READ | PAL_MAP_EXEC)` to flip the page from RW to RX before returning the function pointer. The allocation itself must be `PAL_MAP_READ | PAL_MAP_WRITE` during emission, then transitioned.

---

#### 3b. Buffer Bounds Guard in `xj_emit8`

**FINDING: COMPLIANT with PANIC behavior.**

`xj_emit8()` (codegen.c lines 68–73):
```c
static void xj_emit8(xjit_codebuf_t *cb, uint8_t b) {
    if (cb->pos >= cb->capacity) {
        pal_panic("XJIT: code buffer overflow");
    }
    cb->buf[cb->pos++] = b;
}
```

Bounds check is present and fires `pal_panic()` on overflow — correct for a freestanding kernel context. The panic is hard-fail, not silent corruption.

**Verdict: COMPLIANT.** No silent buffer overflow possible.

---

#### 3c. `xj_patch_rel32` Bounds Check

**FINDING: MISSING BOUNDS CHECK (P1).**

`xj_patch_rel32()` (codegen.c lines 454–463):
```c
static void xj_patch_rel32(xjit_codebuf_t *cb,
                            uint32_t patch_offset,
                            uint32_t target_offset) {
    int32_t rel = (int32_t)(target_offset - (patch_offset + 4u));
    cb->buf[patch_offset]     = (uint8_t)(rel & 0xFF);
    cb->buf[patch_offset + 1] = (uint8_t)((rel >>  8) & 0xFF);
    cb->buf[patch_offset + 2] = (uint8_t)((rel >> 16) & 0xFF);
    cb->buf[patch_offset + 3] = (uint8_t)((rel >> 24) & 0xFF);
}
```

There is NO bounds check on `patch_offset` before the four writes. If `patch_offset >= cb->capacity - 3`, writes at `patch_offset+1`, `+2`, `+3` will overflow `cb->buf[]` into adjacent struct fields (`pos`, `capacity`, then `xjit_func_t` layout). The call site in the XJIT_LABEL handler (line 833) does pass values derived from `f->code.pos` at time of emission, which was bounds-checked by `xj_emit8()`. However, this is an implicit invariant, not an explicit check. A caller providing a malformed `patch_offset` has no guardrail.

**Recommendation:** Add `if (patch_offset + 3u >= cb->capacity) { pal_panic("XJIT: patch OOB"); return; }` at the top of `xj_patch_rel32`.

### 4. BOOT CHAIN INTEGRATION (Sprint 8)

**FINDING: SEQUENCE CORRECT, WITH ONE STAGE NUMBERING ANOMALY (P3).**

`boot_chain.c` `xchain_run()` (lines 533–612) executes stages in this order:

| xchain_run Order | Function | Stage Label | Critical? |
|-----------------|----------|-------------|-----------|
| 1st | `xchain_kernel_ready()` | Stage 0 — XACPI | YES — aborts on fail |
| 2nd | `xchain_storage()` | Stage 2 | NO — warn and continue |
| 3rd | `xchain_security()` | Stage 3 — SHA-256 KAT | YES — aborts on fail |
| 4th | `xchain_network()` | Stage 4 | NO — warn and continue |
| 5th | `xchain_packages()` | Stage 5 | NO — warn and continue |
| 6th | `xchain_display()` | Stage 6 | YES — aborts on fail |
| 7th | `xchain_compositor()` | Stage 7 | YES — aborts on fail |
| 8th | `xchain_shell()` | Stage 8 | YES — aborts on fail |
| 9th | `xchain_ai()` | Stage 9 | NO — warn, offline mode |

This matches the contract: `kernel_ready → storage → security → network → packages → display → compositor → shell → AI → done`.

**ANOMALY:** Stage 1 ("PAL init") is missing from `xchain_run()`. The header comment documents it as "Stage 1: xchain_pal_init — already in boot_seq". This means PAL init is split across `boot_sequence.c` and `boot_chain.c` — a sequencing dependency exists at the file boundary that is not validated by `xchain_run()` itself. If `boot_sequence.c` calls `xchain_run()` before `pal_init()`, all PAL calls inside `xchain_run()` will fail silently (TSC uninitialized, console silent, handle table unzeroed).

**XMIND Fallback on Failure:** `xchain_ai()` returns a non-PAL_OK code on memory probe failure. In `xchain_run()` line 600, AI failure is non-critical (`WARN: AI init failed (offline mode)`). The system boots without AI — correct graceful degradation. However, `xchain_ai_ready()` accessor (line 620) correctly reflects the false state. Downstream consumers of `xchain_ai_ready()` must gate XMIND operations on this accessor — this is the intended contract.

**Verdict:** Boot sequence is correct. One sequencing split (PAL init in boot_sequence.c vs xchain_run) should be documented as a hard pre-condition.

### 5. XMIND WEIGHT SECURITY

**Source:** `ai/xmind/src/weights_loader.c` (lines 1–120, 350–449)

#### 5a. GGUF Magic Verification

**FINDING: VERIFIED (lines 96–98, §2).**

```c
#define GGUF_MAGIC   0x46554747UL   /* "GGUF" little-endian */
#define GGUF_VER_MIN 1u
#define GGUF_VER_MAX 3u
```

The magic constant is defined and version range is bounded (v1–v3 accepted). The scanner function `gguf_scan_metadata()` (§5, implicit from structure) must check both before parsing. Sprint 8 loads only 64 KB of metadata (`#define ... 64KB scan buffer` per MEMORY.md). This is a safe bounded read.

#### 5b. Size Limit on Weight Load (OOM Prevention)

**FINDING: PARTIAL (P2).**

`wl_alloc_pages()` (line 363) calls `pal_pages_alloc(n_pages, ...)` where:
```c
uint64_t n_pages = (bytes + PAL_PAGE_SIZE_4K - 1u) / PAL_PAGE_SIZE_4K;
```

There is NO upper-bound check on `bytes` before the division. If the GGUF file reports a tensor requiring >32 GB of memory (malformed or adversarial file), `n_pages` will be a large value. `pal_pages_alloc()` will reject it if the kernel PMM is exhausted, but this is a reactive OOM, not a proactive size limit.

The Llama 3.2 3B weight tensor expected size is approximately:
- Q4_0 quantized: ~2.0 GB
- Full float32: ~6.4 GB

There is no `XMIND_MAX_WEIGHT_BYTES` constant or check in `wl_alloc_pages()` that would reject an adversarial size request before hitting the PMM.

**Recommendation:** Add `if (bytes > XMIND_MAX_WEIGHT_BYTES) return NULL;` at the top of `wl_alloc_pages()`, where `XMIND_MAX_WEIGHT_BYTES` = 8 GB (conservative ceiling for the 3B model class).

#### 5c. NULL Check After `pal_mem_alloc` / `wl_alloc_pages`

**FINDING: COMPLIANT.**

`wl_alloc_pages()` (lines 369–378):
```c
if (pal_pages_alloc(...) != PAL_OK) { return (void *)0; }
if (pal_map_pages(...) != PAL_OK) {
    pal_pages_free(ph); return (void *)0;
}
return (void *)va;
```

Returns NULL on failure. The caller must check the return before writing to `m->wq[l]`.

In `xmind_alloc_state()` (xmind.c lines 148–151), `XM_ALLOC` macro:
```c
#define XM_ALLOC(ptr, count) \
    do { \
        (ptr) = (float *)xm_heap_alloc((count) * f); \
        if ((ptr) == (void *)0) { goto oom; } \
    } while (0)
```

NULL is checked immediately after allocation. On NULL, execution jumps to `oom:` label which frees all previously allocated buffers. **Correct partial-allocation cleanup.**

The weight pointer assignments in `weights_loader.c` (the `m->wq[l]` path) follow the same pattern via `wl_alloc_pages()` return checks.

**Verdict: COMPLIANT.** NULL check present and cleanup is correct.

### 6. SUPPLEMENTARY FINDINGS

#### ASA-S6-P1-01: XJIT Code Pages — No NX/W^X Enforcement
**File:** `jit/xjit/include/xjit.h:223`, `jit/xjit/src/codegen.c:861`
**Issue:** `xjit_codebuf_t.buf` is a static array in struct. Emitted code is cast to a function pointer from data memory with no `PAL_MAP_EXEC` transition. On W^X-enforced targets this will fault.
**Recommendation:** Allocate code buffer via `pal_pages_alloc` + `pal_map_pages(RW)`, emit, then `pal_map_protect(RX)` before returning function pointer.
**Severity:** P1 — security boundary violation on hardened targets.

#### ASA-S6-P1-02: XJIT `xj_patch_rel32` — No Bounds Check on `patch_offset`
**File:** `jit/xjit/src/codegen.c:454–463`
**Issue:** Four direct `cb->buf[patch_offset + N]` writes without checking `patch_offset + 3 < cb->capacity`. Relies on implicit invariant from emission path.
**Recommendation:** Add explicit bounds check with `pal_panic()` at function entry.
**Severity:** P1 — potential struct corruption if invariant breaks under future refactoring.

#### ASA-S8-P2-01: XMIND Weight Loader — No OOM Size Ceiling
**File:** `ai/xmind/src/weights_loader.c:363–378`
**Issue:** `wl_alloc_pages(bytes)` passes `bytes` from GGUF tensor size directly to PMM with no upper-bound check. Adversarial GGUF could trigger a very large PMM allocation attempt.
**Recommendation:** Add `if (bytes > (8ULL * 1024 * 1024 * 1024)) { return NULL; }` guard.
**Severity:** P2 — OOM vector on malformed model files.

#### ASA-S8-P2-02: XMIND Weights Not Loaded (Sprint 8 Limitation)
**File:** `ai/xmind/src/weights_loader.c:430–435`
**Issue:** Sprint 8 explicitly leaves weight tensor pointers NULL. `xmind_forward()` will produce zero-valued outputs. This is a documented limitation, not a bug, but it means XMIND is not usable for inference until Sprint 9 tensor load is implemented.
**Recommendation:** Track as Sprint 9 P0 — wire actual tensor data reads from GGUF and populate `m->wq`, `m->wk`, `m->wv`, `m->token_emb` from the allocated pages.
**Severity:** P2 — functional gap, not a stability hazard.

#### ASA-S6-P2-03: `xm_heap_free` is a No-Op Stub
**File:** `ai/xmind/src/xmind.c:183`
**Issue:** Comment confirms "xm_heap_free() is a no-op stub in Sprint 6 (page leak accepted)." The `xmind_shutdown()` function calls `xm_heap_free()` for all 7 + (2×28) KV cache buffers, but none of those pages are returned to the PMM. On repeated `xmind_init()`/`xmind_shutdown()` cycles (e.g., model reload), physical pages leak permanently.
**Recommendation:** Implement `xm_heap_free()` to call `pal_unmap()` + `pal_pages_free()` using stored mapping handles. Requires storing the `pal_handle_t` alongside the pointer — suggest adding `xmind_alloc_record_t {void *ptr; pal_handle_t ph; pal_handle_t mh;}` array.
**Severity:** P2 — memory leak on model reload cycles.

#### ASA-S4-P2-04: XSTORE I/O Callbacks — No PAL-Backed Default Driver
**File:** `store/xstore/include/xstore.h:313–318`
**Issue:** XSTORE uses caller-injected I/O callbacks. No default PAL-backed driver (`pal_pages_alloc` + NVMe DMA via `pal_sg_build`) is provided in the repository. Consumers (XPKG, XMIND tokenizer) must wire their own storage callbacks. Boot chain `xchain_storage()` (boot_chain.c:89) validates the XBLOB physical region but does NOT call `xblob_open()` or `xstore_open()`.
**Recommendation:** Provide a reference `xstore_pal_io_t` adapter that wraps `pal_pages_alloc` / NVMe DMA in the XSTORE I/O callback pattern. Gate XPKG and XMIND-tokenizer on `xchain_storage_ok` accessor.
**Severity:** P2 — without a wired driver, XSTORE is structurally sound but operationally unused at runtime.

#### ASA-S5-P3-01: Boot Chain Stage 1 (PAL Init) Sequencing Split
**File:** `init/gensd/boot_chain.c:533`
**Issue:** `xchain_run()` has no Stage 1. The comment (line 28) says "Stage 1: xchain_pal_init — already in boot_seq." This creates a cross-file pre-condition: `boot_sequence.c` MUST call `pal_init()` before `xchain_run()`. If this order is violated (e.g., during refactoring), all stage functions in `xchain_run()` will silently operate on an uninitialized PAL (TSC=0, handle table unzeroed, serial console not initialized).
**Recommendation:** Add a compile-time or runtime assertion at the top of `xchain_run()`: `if (!pal_is_initialized()) { return PAL_ERR_STATE; }`. Requires adding `bool pal_is_initialized(void)` to `pal.h`.
**Severity:** P3 — refactoring hazard, not a current bug.

#### ASA-S9-P3-02: GENISO Not Wired to GENSD Boot Chain
**File:** `tools/iso/gen_iso.c` (Sprint 9)
**Issue:** GENISO produces the bootable ISO manifest but has no runtime hookup to GENSD's boot chain or XPKG installer flow. The `giso_verify_manifest()` function exists in GENISO but is never called by `xchain_run()` or any boot stage to verify the running image's manifest integrity at boot time.
**Recommendation:** Wire `giso_verify_manifest()` into `xchain_kernel_ready()` using the manifest embedded in the boot modules (handoff). This closes the verified boot loop from AetherBoot → XENOS → GENSD.
**Severity:** P3 — missing verified boot closure, not a stability issue.

#### ASA-S9-P3-03: XHPET/XLAPIC Not Integrated into PAL Timer Path
**File:** `kernel/xenos/include/xhpet.h`, `kernel/xenos/include/xlapic.h`
**Issue:** XHPET and XLAPIC drivers exist as standalone Sprint 9 modules. PAL-Aether's `pal_time_now_ns()` still uses RDTSC (lines 292–298 of pal_aether.c). `pal_timer_create/set/cancel()` still route through `XK_SYS_TIMER_*` syscalls. XHPET and XLAPIC are not yet wired as the hardware backend for PAL timer services inside XENOS.
**Recommendation:** Wire `xhpet_read_ns()` as the `pal_time_now_ns()` backend inside XENOS kernel mode (RDTSC remains as the userspace fallback). Wire `xlapic_timer_init()` as the periodic tick source for PAL timer deadlines.
**Severity:** P3 — RDTSC is a valid and performant fallback; this is a hardening/precision gap.

### Key Architectural Risks (Ranked)

1. **XJIT W^X (P1-01):** The static `buf[]` in `xjit_codebuf_t` must become a PAL-allocated executable mapping before XJIT can run on a kernel with proper memory protection. This is a 1-sprint fix but is a hard blocker for security-grade deployment.

2. **XMIND xm_heap_free no-op (P2-03):** Every call to `xmind_shutdown()` permanently leaks all KV cache pages. On a 16 GB HP EliteBook x360 running repeated inference sessions, this will eventually exhaust physical memory. Must be fixed before any sustained inference workload.

3. **XSTORE Not Called at Boot (P2-04):** The storage layer is architecturally sound but has no runtime instantiation in the boot chain. XPKG package resolution and XMIND tokenizer XSTORE integration path (`#ifdef XMIND_HAS_XSTORE`) cannot function until a caller wires `xstore_open()` with a PAL NVMe I/O adapter.

4. **Verified Boot Loop Open (P3-02):** GENISO produces and signs the manifest, but `giso_verify_manifest()` is never called at runtime. AetherBoot → XENOS verified boot is not closed until this is wired into `xchain_kernel_ready()`.
