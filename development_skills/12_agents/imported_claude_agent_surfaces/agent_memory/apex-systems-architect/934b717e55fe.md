## Detailed Findings

### F-01 [FAIL/HIGH] IP4 Checksum Verification Logic Inverted (ip4.c:452)

```c
uint16_t computed = xnet_ip4_checksum(hdr_copy, ihl_bytes);
if (computed != saved_cksum) {
    return XNET_ERR_CHECKSUM;
}
```

**Issue:** The checksum validation is incorrect. The standard way to verify an IP checksum is to compute the checksum over the *entire header including the checksum field*. If the result is 0x0000, the checksum is valid. The code instead zeros the checksum field, recomputes, and compares against the saved value. While this *can* work, it produces the *complement* of the original checksum when computed over the header-with-zero-checksum-field. The comparison `computed != saved_cksum` will fail because `xnet_ip4_checksum` returns `~sum` (the complement), which equals `saved_cksum` only if computed correctly.

**Wait -- re-analyzing:** Actually, this pattern IS correct. When you zero the checksum field and recompute, the result IS the same value the sender computed. This is standard practice. The code is mathematically sound. **Reclassified: PASS.**

**Correction: F-01 is PASS. Adjusting.**

### F-01 [FAIL/HIGH] IP Pseudo-Header Checksum Byte Order Mismatch (ip4.c:89-108)

```c
uint32_t xnet_ip4_pseudo_checksum(xnet_ipv4_t src_ip, xnet_ipv4_t dst_ip,
                                   uint8_t protocol, uint16_t segment_len) {
    uint32_t sum = 0;
    sum += (src_ip >> 16) & 0xFFFFU;
    sum += src_ip & 0xFFFFU;
```

**Issue:** `xnet_ipv4_t` is documented as "stored in network byte order" in the header (`xnet.h:99-100`), but `netif.c:198` comment says "All addresses are in host byte order internally; conversion to network byte order happens at the wire format layer." The pseudo-header checksum function receives addresses in host byte order (as evidenced by `udp_checksum` and `tcp_checksum` callers which pass `src_ip`/`dst_ip` after `xnet_ntohl()`). However, the pseudo-header RFC requires addresses in network byte order for checksum computation. The current code splits host-byte-order IPs into 16-bit words and sums them -- this produces a different result than summing network-byte-order words on little-endian machines.

**Analysis:** On little-endian (x86_64), host-order `0xC0A80001` (192.168.0.1) splits as `0xC0A8 + 0x0001 = 0xC0A9`. Network-order `0x0100A8C0` splits as `0x0100 + 0xA8C0 = 0xA9C0`. These are NOT the same partial sum. However, the checksum is a ones-complement sum -- the byte-swap at both ends means the final folded result will actually match because both sender and receiver use the same convention consistently. Since XNET is both sender and receiver of its own checksums, this works **only if XNET always talks to itself**. When talking to external hosts (Linux, Windows), checksums will be **incorrect on little-endian**.

**Severity:** HIGH -- TCP/UDP checksums will be rejected by any non-XNET host on little-endian hardware.

**Fix:** The pseudo-header checksum must operate on network-byte-order values:
```c
uint32_t src_n = xnet_htonl(src_ip);
uint32_t dst_n = xnet_htonl(dst_ip);
sum += (src_n >> 16) & 0xFFFFU;
sum += src_n & 0xFFFFU;
sum += (dst_n >> 16) & 0xFFFFU;
sum += dst_n & 0xFFFFU;
sum += xnet_htons(protocol);
sum += xnet_htons(segment_len);
```

### F-02 [FAIL/HIGH] TCP FIN in ESTABLISHED Does Not Check Sequence Validity (tcp.c:832-836)

```c
if (flags & TCP_FLAG_FIN) {
    t->rcv_nxt++;
    t->state = TCP_CLOSE_WAIT;
    tcp_send_segment(t, TCP_FLAG_ACK, t->snd_nxt, NULL, 0);
}
```

**Issue:** The FIN is accepted regardless of whether the segment's sequence number is valid. RFC 793 Section 3.9 step 5 requires checking that the segment sequence number falls within the receive window. A spoofed FIN with an out-of-window sequence number would improperly transition the connection to CLOSE_WAIT. Additionally, the `rcv_nxt++` here conflicts with `tcp_process_data` which also increments `rcv_nxt` for FIN. Double-increment if data+FIN in same segment.

**Fix:** Add sequence validation before FIN processing. Guard against double FIN increment if `tcp_process_data` already handled it.

### F-03 [FAIL/HIGH] Gossip Incarnation Number Truncation (gossip.c:234, 354)

```c
hdr->incarnation = xnet_htonl((uint32_t)g_self.incarnation);
// ...
uint64_t incarnation = (uint64_t)xnet_ntohl((uint32_t)hdr->incarnation);
```

**Issue:** `incarnation` is declared as `uint64_t` in `xnet_gossip_node_t` but is serialized/deserialized as `uint32_t` via `xnet_htonl/ntohl`. This truncates the upper 32 bits. The wire format struct declares `uint64_t incarnation` but only 4 bytes are converted. After 2^32 incarnation bumps (possible under sustained SUSPECT storms), the protocol will wrap incorrectly.

**Fix:** Implement `xnet_htonll/ntohll` for 64-bit network byte order conversion, or document the 32-bit limitation.

### F-04 [FAIL/MEDIUM] ARP Cache Poisoning: Unsolicited ARP Updates (arp.c:405-422)

```c
xnet_arp_entry_t *entry = arp_find_entry(sender_ip);
if (!entry) {
    entry = arp_find_free();
}
if (entry) {
    entry->state = ARP_STATE_RESOLVED;
    xnet_memcpy(entry->mac, arp->sender_mac, 6);
}
```

**Issue:** Any ARP packet (request or reply) unconditionally updates or creates a cache entry for the sender. This means an attacker can send unsolicited ARP replies to poison the cache. RFC 826 recommends only updating existing entries (merge flag pattern), not creating new ones from unsolicited traffic.

**Fix:** Only update existing entries from unsolicited ARP packets. Only create new entries when:
1. We sent an ARP request for that IP (entry is PENDING), or
2. The ARP request targets our IP (we need to reply anyway)

### F-05 [FAIL/MEDIUM] Fragment Reassembly Static Buffer Not Thread-Safe (ip4.c:407)

```c
static uint8_t g_frag_out_buf[FRAG_MAX_SIZE];
```

**Issue:** `g_frag_out_buf` is a single static 64KB buffer used by `xnet_ip4_input` to hold reassembled datagrams. If two interfaces receive the last fragment of different datagrams simultaneously, both calls to `ip4_reassemble` will copy into the same buffer. The `g_frag_lock` protects the fragment table but NOT the output buffer usage after `pal_spin_unlock` in `ip4_reassemble`. The buffer is written inside the lock, but the caller (`xnet_ip4_input`) reads it after the lock is released (lines 500-504), creating a TOCTOU race.

**Fix:** Either protect `g_frag_out_buf` usage through the full dispatch path, or allocate per-interface reassembly buffers.

### F-06 [FAIL/MEDIUM] mDNS Query Never Returns Results (mdns.c:396-404)

```c
while (pal_time_now_ns() < deadline) {
    pal_thread_yield();
    /* Response arrives via mdns_recv_callback -> xnet_mdns_input */
    /* In Sprint 3 we don't have async result storage; return timeout */
}
return XNET_ERR_TIMEOUT;
```

**Issue:** `xnet_mdns_query` always returns `XNET_ERR_TIMEOUT` regardless of whether a response is received. The mDNS input handler does not store query results, making service discovery non-functional.

**Fix:** Implement a pending query structure similar to `dns_do_resolve` pattern.

### F-07 [FAIL/MEDIUM] Gossip PING_REQ Not Fully Implemented (gossip.c:412-416)

```c
case GOSSIP_MSG_PING_REQ:
    /* Indirect ping request: ping target on behalf of requester */
    /* For Sprint 3: forward the PING_REQ as a direct PING to embedded target */
    /* The target is encoded in the first piggyback event */
    break;
```

**Issue:** PING_REQ handling is a no-op. This breaks the SWIM protocol's indirect ping mechanism, which is the core of its failure detection reliability. Without indirect pings, false positives (marking alive nodes as dead due to network asymmetry) will be significantly higher.

**Fix:** Extract target node from first piggyback event, send PING to target, and relay ACK back to requester.

### W-01 [WARN/HIGH] Duplicated Helper Functions Across All Files

Every .c file contains identical copies of `xnet_memset`, `xnet_memcpy`, `xnet_memcmp`, etc. This is ~50 lines duplicated across 12 files (600 lines total). While this works for freestanding compilation, it violates DRY and makes bug fixes require 12 edits.

**Recommendation:** Create `net/xnet/core/xnet_util.h` with `static inline` implementations. Include from all .c files.

### W-02 [WARN/MEDIUM] CRC32 Table Init Not Thread-Safe (ethernet.c:62-77)

```c
static bool g_crc32_table_ready = false;
static void crc32_init_table(void) {
    if (g_crc32_table_ready) return;
    // ... populate table ...
    g_crc32_table_ready = true;
}
```

**Issue:** Two concurrent calls to `xnet_eth_crc32` could both enter `crc32_init_table` before the flag is set, causing a benign but wasteful race. In practice the table is deterministic so double-init is harmless, but it violates the project's spinlock-everything policy.

### W-03 [WARN/MEDIUM] DNS Single-Pending-Query Limitation (dns.c:259)

```c
static xnet_dns_pending_t g_dns_pending;
```

Only one DNS query can be in-flight at a time. If two threads call `xnet_dns_resolve` concurrently, the second will overwrite the first's pending state, causing the first to timeout or receive wrong results.

**Recommendation:** Use a pending query array (e.g., 8 slots) keyed by QID.

### W-04 [WARN/MEDIUM] Socket Table Per-Socket 16KB UDP Buffer (socket.c:56-57)

```c
#define UDP_RECV_BUF_SIZE  (16U * 1024U)
typedef struct {
    uint8_t udp_recv_buf[UDP_RECV_BUF_SIZE];
```

With `SOCKET_MAX = 256`, the socket table alone consumes `256 * (16384 + overhead) = ~4.2MB`. Combined with TCP TCBs (128 * ~100KB each for ring buffers + rexmit queues), total static allocation is approximately **17MB** for the networking stack.

**Recommendation:** Document memory budget. Consider on-demand allocation for sockets that haven't received data.

### W-05 [WARN/MEDIUM] TCP Sequence Number in RST for Non-ACK Segments (tcp.c:722)

```c
tcp_send_rst(dst_ip, dst_port, src_ip, src_port,
             0, seq + data_len + 1, true);
```

**Issue:** When sending RST for a non-ACK segment, the ACK number should be `seq + seg_len` where `seg_len` includes SYN/FIN flag consumption. The current code adds `data_len + 1` which is incorrect if the segment has no SYN/FIN (should be just `data_len`), and incorrect if it has both SYN and data (should be `data_len + 1` for SYN but code adds +1 unconditionally).

### W-06 [WARN/LOW] XNET_IPV4 Macro Produces Host-Order Value

```c
#define XNET_IPV4(a, b, c, d) \
    ((xnet_ipv4_t)(((uint32_t)(a) << 24) | ...))
```

This produces a host-byte-order value (a in MSB), which is consistent with the internal convention but confusing since `xnet_ipv4_t` header comment says "stored in network byte order." The code internally uses host order everywhere. The header comment at line 99 is misleading.

### W-07 [WARN/LOW] TCP Data in FIN_WAIT States Not Processed (tcp.c:840-858)

In `TCP_FIN_WAIT_1` and `TCP_FIN_WAIT_2`, incoming data segments are not passed to `tcp_process_data`. RFC 793 allows data reception in these states until FIN is received from the peer.

### W-08 [WARN/LOW] UDP ICMP Port Unreachable Passes Wrong Header (udp.c:354-356)

```c
xnet_icmp_dest_unreachable(src_ip, ICMP_CODE_PORT_UNREACH, pkt, UDP_HDR_LEN);
```

RFC 792 requires the ICMP error to include the original **IP header** + 8 bytes, not the UDP header. The `pkt` parameter points to the UDP segment, not the enclosing IP header.

### W-09 [WARN/LOW] No `sendto`/`recvfrom` in Socket API

The BSD socket API is missing `sendto()`/`recvfrom()` which are essential for UDP communication. The current `xnet_send()`/`xnet_recv()` require a prior `connect()` for UDP, which is non-standard usage.
