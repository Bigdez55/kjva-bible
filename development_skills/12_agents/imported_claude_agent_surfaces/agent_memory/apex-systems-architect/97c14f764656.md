## SECTION 1: WAL FIX DESIGN ASSESSMENT

### The Fix (mvcc.c lines 196-226)

```c
if (we->op == XSTORE_WAL_OP_INSERT) {
    xstore_key_t k; k.data = we->key; k.len = we->key_len;
    xstore_val_out_t vout;
    vout.buf = entry.val; vout.buf_len = XSTORE_MAX_VAL_LEN; vout.out_len = 0;
    if (xstore_btree_lookup(store, txn->snapshot_lsn, &k, &vout) == XSTORE_OK) {
        entry.val_len = (uint16_t)vout.out_len;
    }
}
```

### Question A: Is txn->snapshot_lsn the right snapshot?

VERDICT: Currently SAFE, but for the wrong architectural reason. Latent debt.

Evidence from btree.c line 510:
```c
(void)snapshot_lsn; /* Full per-version chain in Sprint 5 */
```

The btree does NOT enforce snapshot visibility — it ignores snapshot_lsn entirely
and finds the key by binary search alone. This means:

- The lookup WILL find the value inserted by xstore_txn_put() because it is
  physically present in the leaf page.
- The lookup would also find any OTHER transaction's committed value for the
  same key, if one exists, because there is no per-version filtering.
- When Sprint 5 wires per-version chain filtering into btree_lookup, the current
  fix will BREAK: btree_lookup with snapshot_lsn (established at txn BEGIN)
  will not see the insert performed by xstore_txn_put() during THIS transaction,
  because the insert's LSN was assigned AFTER snapshot_lsn.

Correct snapshot for this read: `store->next_lsn` (current head), or a
transaction-private "write LSN" captured at xstore_txn_put() time.

### Question B: Lock ordering — deadlock risk?

VERDICT: NO DEADLOCK. Lock acquisition sequence is:

1. `btree_lookup` → `xstore_cache_get` → acquires `cache->lock`, releases it
   before returning.
2. `xstore_wal_append` → acquires `wal_lock` (after btree_lookup returns).

There is zero lock overlap between cache->lock and wal_lock in this path.
Both are pal_spinlock_t (non-reentrant), but they are never held simultaneously
in xstore_txn_commit.

### Question C: Should this be xstore_txn_get() instead of xstore_btree_lookup()?

VERDICT: xstore_btree_lookup is the correct choice here. xstore_txn_get
(mvcc.c line 314) calls xstore_btree_lookup with txn->snapshot_lsn — they
are functionally equivalent. Using btree_lookup directly avoids the txn active
check and is slightly more efficient. No correctness difference under current code.

### Question D: WAL-FIRST alternative design analysis

The WAL-FIRST approach (write val to WAL before B+ tree) would require storing
val bytes in xstore_write_entry_t. Analysis:

- XSTORE_TXN_MAX_WRITES = 512 entries
- Adding val[XSTORE_MAX_VAL_LEN] = 4096 bytes per entry
- Stack cost: 512 × 4096 = 2 MB per transaction — unacceptable for freestanding
- Pointer alternative: store `const uint8_t *val_ptr` + `uint32_t val_len` in
  write_entry, pointing to the caller's original buffer.
- BREAKS the published contract: "caller may free/reuse buffers after this call"
- Conclusion: WAL-FIRST with pointer is architecturally incompatible with the
  published API contract. B-tree-as-staging is the correct pattern for this system.

### WAL Replay Correctness

WAL replay (wal.c lines 128-167) correctly reconstructs inserts from WAL entries:
```c
val.data = e->val;
val.len  = e->val_len;
(void)xstore_btree_insert(store, &rec_txn, &key, &val);
```

If val_len=0 is recorded in the WAL (silent failure path in the fix), replay will
call btree_insert with an empty value — silently corrupting the recovered key's
value to zero-length. This is a data integrity hazard.

## ARCH FINDINGS

### ARCH-01: [P1] WAL Fix — Snapshot LSN Forward Incompatibility
**Category:** Design debt, latent correctness failure
**File:** store/xstore/txn/mvcc.c:220
**Description:** btree_lookup is called with `txn->snapshot_lsn` to recover the
value of an insert performed during the current transaction. This works today only
because btree.c ignores snapshot_lsn (line 510: `(void)snapshot_lsn`). When Sprint
5 wires per-version filtering, the lookup will return XSTORE_ERR_NOT_FOUND because
the insert's write LSN > snapshot_lsn. The WAL entry will record val_len=0, and
crash recovery will silently restore the key with an empty value.
**Recommendation:** Before Sprint 5 adds MVCC visibility to btree_lookup, the WAL
commit path must use `store->next_lsn` (or a post-insert capture) as the snapshot
for this recovery read. Alternatively, capture `val_ptr + val_len` at txn_put time
in a separate per-write staging buffer separate from xstore_write_entry_t.

### ARCH-02: [P1] WAL Fix — Silent Failure Path on btree_lookup Miss
**Category:** Data integrity hazard
**File:** store/xstore/txn/mvcc.c:223-225
**Description:** When btree_lookup fails during commit (any error), val_len=0 is
silently written to the WAL entry. The comment says "should not occur in normal
operation" but provides no assertion, no counter, and no distinct error code.
A WAL entry with val_len=0 for an INSERT operation will replay as a zero-length
value insertion, silently corrupting the recovered key.
**Recommendation:** Assert or panic on btree_lookup failure in this path, or return
XSTORE_ERR_STATE and abort the transaction. Never silently proceed with val_len=0.
Add a diagnostic counter: `store->wal_val_recover_fails++`.

### ARCH-03: [P2] OCC Conflict Check — write_set.page_no Always Root Page
**Category:** Correctness gap, OCC granularity
**File:** store/xstore/txn/mvcc.c:302-303, 331-332
**Description:** Both xstore_txn_put and xstore_txn_delete record
`we->page_no = store->sb.root_page` for every write. The OCC conflict check
(xstore_mvcc_conflict_check) queries mvcc_query_write(we->page_no) for each
write entry. Since all writes map to the same page_no (root_page = 1), ALL
transactions share one OCC slot. This causes:
1. False conflicts: two transactions writing different keys to different leaf
   pages will conflict if any third transaction has committed since snapshot.
2. False non-conflicts: two transactions writing to the same leaf page but
   tracked through different page_no values (if root ever changes) will miss
   each other.
**Recommendation:** btree_insert must return the actual leaf page_no it wrote to,
and xstore_txn_put must store that in write_entry.page_no. This requires a minor
API change to xstore_btree_insert (add `uint32_t *out_leaf_page_no` parameter).
Acceptable for Sprint 5 scope.

### ARCH-04: [P2] btree_lookup — snapshot_lsn Completely Unenforced
**Category:** MVCC contract gap
**File:** store/xstore/btree/btree.c:510
**Description:** `(void)snapshot_lsn;` — the entire MVCC visibility guarantee
of "reads the latest committed value whose commit LSN <= snapshot_lsn" (per
xstore.h §13) is not implemented. Any read sees the physically latest value
regardless of snapshot. This is documented as deferred to Sprint 5 but is a
critical gap: snapshot isolation is the core MVCC contract.
**Current Risk:** Low in single-writer workloads. High if XPKG or XORCHESTRA
issues concurrent read+write transactions.
**Recommendation:** Sprint 5 must wire per-slot LSN comparison using the lsn_hi
field already stored in xstore_leaf_slot_t.

### ARCH-05: [P2] Page Cache — Linear Scan O(n) for Every Cache Access
**Category:** Performance
**File:** store/xstore/io/page_cache.c:75-82
**Description:** `_cache_find` performs a full linear scan over all 1024 cache
slots on every access (hit or miss). At 1024 entries this is bounded, but the
file itself notes "For Sprint 5 we will add an open-addressing directory for
O(1) lookup." Under XMIND inference (which will issue repeated page reads for
the B+ tree provenance store), cache lookup will be called thousands of times
per inference pass, making O(n) scan a measurable bottleneck.
**Recommendation:** Sprint 5 priority: add a hash-indexed directory (power-of-two
open addressing, page_no as key). Target O(1) amortized.

### ARCH-06: [P2] mvcc_record_write — Page 0 Tracking Blind Spot
**Category:** Correctness
**File:** store/xstore/txn/mvcc.c:58, 82
**Description:** The MVCC shadow table uses `page_no == 0` as the empty-slot
sentinel. The comment notes "page 0 = superblock, tracked separately" but there
is no code that actually tracks page 0 separately. If a write ever touches page 0
(the superblock — which checkpoint does via write_page), it will silently fail to
insert into the shadow table (the `if (e->page_no == 0 || e->page_no == page_no)`
branch will find an empty slot on the first iteration and set it, but only if
page_no != 0). In the case where page_no IS 0, the first condition matches, the
second fails, and the slot is reused — the insert is a no-op.
**Recommendation:** Use UINT32_MAX as the empty-slot sentinel instead of 0.
Reserve slot 0 explicitly for superblock tracking if needed.

### ARCH-07: [P3] WAL Ring — LSN/slot Assignment Race with wal_head Counter
**Category:** Concurrency
**File:** store/xstore/wal/wal.c:72, 81
**Description:** LSN is assigned via `pal_atomic_add(&store->next_lsn, 1)` (atomic),
but `store->wal_head++` on line 81 is NOT atomic — it is protected only by
`store->wal_lock` (spinlock held). This is internally consistent (the spinlock
covers the entire append critical section), but the LSN and wal_head can diverge
if any code path reads next_lsn outside the lock (which xstore_txn_begin does
at line 142: `uint64_t snap = store->next_lsn;` — under txn_lock, not wal_lock).
The snapshot LSN can thus be a value between two WAL appends that have been
LSN-assigned but not yet wal_head-incremented. Under Sprint 4's single-writer
pattern this is harmless. Under multi-writer Sprint 5+, this needs formal analysis.
**Recommendation:** Document the two-lock protocol. Consider whether next_lsn
should be incremented only inside wal_lock to guarantee snapshot_lsn always
reflects a stable WAL boundary.

### ARCH-08: [P3] Cursor API — Sprint 4 Leaf Chaining Missing
**Category:** Feature gap, functional correctness
**File:** store/xstore/btree/btree.c:781-786
**Description:** xstore_btree_cursor_advance sets cursor->valid=false and returns
XSTORE_ERR_NOT_FOUND when a leaf page is exhausted, rather than advancing to the
next leaf. Sprint 4 documents this explicitly as a known limitation. This makes
range scans across page boundaries impossible until Sprint 5.
**Impact:** Any XPKG dependency resolution query or XBLOB index scan that spans
more than one leaf page (>~10 keys at max value size, or >~200 keys at minimum)
will silently truncate results.
**Recommendation:** Sprint 5 must add a next_leaf_page pointer to the leaf page
header (or re-descend from root — the current approach) before XPKG is used
in production.

## SECTION 2: API CONTRACT CORRECTNESS

### xstore_txn_put Contract Audit

Published contract (xstore.h §13):
> "Key and value bytes are copied — caller may free/reuse buffers after this call"

**Tracing the val copy path:**

1. `xstore_txn_put` → `xstore_btree_insert` (btree.c:525)
2. `xstore_btree_insert` → `_leaf_insert_at` (btree.c:313)
3. `_leaf_insert_at` → `xstore_memcpy(rec + key_len, val->data, val_len)` (line 338)
4. Data is now in the in-memory page buffer inside `xstore_cache_entry_t.data[]`

CONTRACT STATUS: UPHELD. The value is deep-copied into the page cache at call time.

**Page eviction analysis:**

If the dirty page containing this key is evicted between txn_put and txn_commit:
- Eviction path (page_cache.c:113-117): `write_page(io_ctx, e->page_no, e->data)`
- On btree_lookup miss: `read_page(io_ctx, page_no, e->data)` (page_cache.c:176)
- The value is therefore recoverable from the backing store.

CONTRACT STATUS: SAFE under eviction. The round-trip through write_page/read_page
preserves the value byte-for-byte (assuming write_page/read_page correctness, which
is a caller responsibility per §11).

**Edge case: val_len change on second put to same key:**

btree_insert line 556-561:
```c
if (val->len != s->val_len) {
    return XSTORE_ERR_CONFLICT;
}
```

An update that changes value size returns XSTORE_ERR_CONFLICT, not XSTORE_ERR_EXISTS.
This is a **contract mismatch**: callers expecting to update a key with a different-
sized value will receive XSTORE_ERR_CONFLICT and may incorrectly retry (OCC retry
logic in xstore_put). The correct error code should be XSTORE_ERR_EXISTS or a new
XSTORE_ERR_SIZE_CHANGE.

### ARCH-09: [P2] btree_insert Size-Change Returns Wrong Error Code
**Category:** API contract violation
**File:** store/xstore/btree/btree.c:556-561
**Description:** Attempting to upsert a key where val_len differs from the stored
val_len returns XSTORE_ERR_CONFLICT. The xstore_put retry loop treats XSTORE_ERR_CONFLICT
as "retry" — causing up to 3 wasted transaction attempts before surfacing the error.
Callers expecting "update" semantics will see confusing retry behavior.
**Recommendation:** Return XSTORE_ERR_INVAL or a dedicated XSTORE_ERR_SIZE_CHANGE code.
Update the API comment to document this restriction explicitly.

## SECTION 4: CROSS-LAYER API MISMATCH ANALYSIS

### 4A: XJIT (xjit.h) vs XEMU jit_backend.c

**xjit_get_fn signature (xjit.h §13):**
```c
typedef uint64_t (*xjit_fn_t)(uint64_t arg0, uint64_t arg1, uint64_t arg2, uint64_t arg3);
xjit_fn_t xjit_get_fn(xjit_func_t *f);
```

**XEMU call site (xjit_backend.c:402):**
```c
rf->rax = fn((uint64_t)rf->rax, 0ULL, 0ULL, 0ULL);
```

VERDICT: **COMPATIBLE**. The 4-argument signature matches. RAX is passed as arg0,
returned as the uint64_t result. The JIT IR emits `XJIT_RET VR_RAX`, which codegen
compiles to `mov rax, <result>; ret`. Return value lands in RAX per x86-64 ABI.

**XJIT register encoding note (potential future issue):**
xjit.h §6 defines `XJIT_RSI = 4u` and `XJIT_RDI = 5u` with the comment "NOTE: RSI=6
in hw, but we use our own index". The HW_ENC macro correctly maps these back:
`XJIT_RSI → 6u`, `XJIT_RDI → 7u`. This is internally consistent.

**Block cache memory concern:**
xjit_backend.c §3 comment: "each entry is ~176 KB. 256 entries = ~44 MB"
The actual size of xjit_func_t:
- ir[4096] × 24 bytes = 98,304 bytes
- labels[256] × sizeof(xjit_label_t) = 256 × (4+4+32+4) = 11,264 bytes
- ranges[256] × 12 = 3,072 bytes
- ra: map[256]×8 + preg_used[16] + ... ~2,100 bytes
- code: buf[65536] + pos(4) + cap(4) = 65,544 bytes
- Total: ~180 KB per entry × 256 = ~46 MB static global

This is a P2 concern: 46 MB of static BSS for the JIT cache will stress the
HP EliteBook x360's memory budget during early boot when XENOS has not yet
set up the full virtual memory manager. The cache size should be reduced or
made dynamically allocated once PMM is reliable.

### ARCH-10: [P2] XEMU JIT Cache — 46 MB Static BSS
**Category:** Resource constraint
**File:** vm/xemu/src/xjit_backend.c:169
**Description:** `static xemu_jit_block_t s_cache[256]` — 256 entries × ~180 KB
= ~46 MB of static BSS. This is allocated at compile time and present in the
kernel image from boot. On a 16 GB device this is marginal; on a device with
memory pressure during early boot (before full PMM), it risks BSS segment
placement failures.
**Recommendation:** Reduce XEMU_JIT_CACHE_SIZE to 32 or 64 entries (6-12 MB),
or allocate the cache from pal_pages_alloc() after PMM is initialized.

### 4B: XMIND API vs boot_chain.c

**boot_chain.c xchain_ai() (lines 453-479):**
The function does NOT call:
- `xmind_init()`
- `xmind_alloc_state()`
- `xmind_forward()`
- Any function from xmind.h

It only calls `pal_pages_alloc(100 pages)` as a memory availability probe, then
immediately frees the pages. `xchain_ai_ready()` returns 1 if this probe succeeds.

**XMIND API (xmind.h §11):**
The correct initialization sequence is:
1. `xmind_init(m, &cfg)` — validates config, zeroes model
2. Assign weight pointers (wq, wk, ..., token_emb, rope_*)
3. `xmind_alloc_state(m)` — allocates KV cache + scratch buffers
4. `xmind_forward(m, token, pos)` — inference

**Verdict:** The boot chain does not exercise the XMIND API at all. The
`xchain_ai_ready()` flag means "the kernel has 400 KB of free memory" — not
"XMIND is initialized and ready for inference." This is a **semantic contract
violation** between the boot chain status API and its intended meaning.

### ARCH-11: [P1] boot_chain xchain_ai_ready() Semantically Incorrect
**Category:** API contract violation, boot correctness
**File:** init/gensd/boot_chain.c:453-479, 621
**Description:** xchain_ai_ready() returns 1 when a 400 KB memory probe succeeds.
Callers (XSHELL, XORCHESTRA service registry) expecting this flag to mean "XMIND
is initialized and available for inference" will proceed to call xmind_forward()
on an uninitialized model, causing UB (null pointer dereference on m->state.logits,
m->wq[0], etc.).
**Recommendation:** xchain_ai must either:
a) Call xmind_init + xmind_alloc_state and report real initialization success, OR
b) Be renamed xchain_ai_memory_ok() with a clear comment that model initialization
   is deferred to the first inference request.
The status accessor xchain_ai_ready() must not be exported until (a) is implemented.

## SECTION 5: ADDITIONAL FINDINGS

### ARCH-12: [P3] xstore_exists Snapshot LSN Off-By-One
**Category:** Correctness
**File:** store/xstore/xstore.c:237
**Description:**
```c
xstore_status_t rc = xstore_btree_lookup(store, store->next_lsn, key, &out);
```
`store->next_lsn` is the NEXT LSN to be assigned — i.e., one beyond the highest
committed entry. Using `next_lsn` means "see everything including any in-progress
uncommitted write that happened to atomically-increment next_lsn before the
xstore_txn_commit COMMIT entry." This is technically sound today (btree ignores
snapshot_lsn) but will become a dirty read bug when MVCC visibility is enforced
in Sprint 5. Should use `store->next_lsn - 1` or equivalent "committed head LSN."

### ARCH-13: [P3] pal_channel_send — PAL msg.data Pointer Cast
**Category:** API contract gap
**File:** pal/src/pal_aether.c:664
**Description:**
```c
(uint64_t)(uintptr_t)msg->data,
```
The `msg->data` field is `uint8_t data[PAL_MSG_MAX_ENVELOPE]` — an array, not a
pointer. The cast passes the array's address (correct) but the kernel syscall
receives a virtual address that must be mapped in the kernel address space. If
the caller passes a stack-allocated pal_msg_t, the kernel must copy the data
before the call returns. Whether the XENOS XK_SYS_CHAN_SEND implementation
performs this synchronous copy is undocumented. This should be audited when
XENOS channel implementation is available.

### ARCH-14: [P3] XJIT Compilation — pal_panic Declared but Not Defined in PAL
**Category:** Linkage gap
**File:** jit/xjit/include/xjit.h:36-37
**Description:**
```c
__attribute__((noreturn)) void pal_panic(const char *msg);
```
`pal_panic` is declared in xjit.h but is not defined in pal.h or pal_aether.c.
It must be provided by some other translation unit at link time. If it is absent,
any XJ_ASSERT failure will link-error in production builds. The PAL contract
should either define pal_panic() or XJIT should use its own halt macro.

## SECTION 6: FINAL SCORES

### WAL Fix Verdict: NEEDS REVISION

The fix is architecturally sound in its general approach (B-tree-as-staging-buffer
for WAL value recovery) and does not introduce deadlocks. However two P1 issues
must be resolved before Sprint 5:

1. The snapshot_lsn used in btree_lookup must be updated to reference a
   post-insert LSN, not the pre-transaction snapshot. Otherwise the fix silently
   breaks when MVCC visibility is enforced in btree_lookup.
2. The silent val_len=0 path on btree_lookup failure must be hardened with an
   assert or explicit error return.
