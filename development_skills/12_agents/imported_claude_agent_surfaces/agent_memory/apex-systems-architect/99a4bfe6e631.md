### 1.2 VMM — PML4 Index Math
STATUS: PASS — Sprint 13 fix CONFIRMED

HHDM_BASE = 0xFFFF888000000000:
  bits[47:39] = (0xFFFF888000000000 >> 39) & 0x1FF
             = 0x1111 & 0x1FF = 0x111 = 273d
  vmm.c line 95: #define PML4_IDX_HHDM 273u — CORRECT.
  sprint13.yml explicitly tests grep PML4_IDX_HHDM.*273 — PASS.
  The bug (272 vs 273) from pre-Sprint-13 is definitively FIXED.

KERN_BASE = 0xFFFFFFFF80000000:
  bits[47:39] = (0xFFFFFFFF80000000 >> 39) & 0x1FF
             = 0x1FFFFFFFFFFF >> 39 & 0x1FF — must compute:
             0xFFFFFFFF80000000 >> 39 = 0x1FFFFFFF0 >> 0 = ...
             direct: (0xFFFFFFFF80000000 >> 39) = 0x1FFFFFFF >> 0 bits:
             bit39=bit47=1, so index = 0x1FF = 511.
  vmm.c line 96: #define PML4_IDX_KERN 511u — CORRECT.

PDPT index for KERN_BASE:
  (0xFFFFFFFF80000000 >> 30) & 0x1FF = ?
  0xFFFFFFFF80000000 >> 30 = 0x3FFFFFFFE → & 0x1FF = 0x1FE = 510.
  vmm.c line 316: const uint32_t kern_pdpt_idx = 510u — CORRECT.

HHDM 2MB page loop: for gb in [0, 128): pdpt[gb] → pd → pd[0..511] each 2MB.
  Each pd entry: phys_2m = gb*1GB + m*2MB. Physical coverage = [0, 128GB). CORRECT.
  PTE flags: PTE_PS | PTE_PRESENT | PTE_WRITE | PTE_GLOBAL. NOTE below (1.2a).

FINDING 1.2a (P1) — HHDM PAGES MISSING PTE_NX BIT:
  All 128GB HHDM 2MB pages are mapped with PTE_PS | PTE_PRESENT | PTE_WRITE | PTE_GLOBAL.
  PTE_NX is NOT set. This means all physical RAM is executable via HHDM.
  Since EFER.NXE is set at step 4.5 (before vmm_init step 5), the NX bit
  is active but not applied. An attacker who writes shellcode to any physical
  page could execute it via the HHDM alias.
  Sprint 1 audit (VMM-01) documented this exact issue. It has NOT been fixed in any sprint.
  FIX REQUIRED: Add PTE_NX to all HHDM PDE entries:
    pd[m] = phys_2m | PTE_PRESENT | PTE_WRITE | PTE_PS | PTE_GLOBAL | PTE_NX;
  BLOCKER for bare-metal security. P1.

FINDING 1.2b (P2) — KERN PDE ENTRIES MISSING PTE_NX ON DATA SECTIONS:
  KERN_MAP_SIZE = 512MB mapped at KERN_BASE with all 2MB PDEs flagged:
    PTE_PRESENT | PTE_WRITE | PTE_PS | PTE_GLOBAL — no PTE_NX.
  Comment says "text sections remapped RO+X in Sprint 2 by ELF loader".
  Sprint 2 through 13 = 12 sprints later — this has not happened.
  Data sections (BSS, rodata, heap) are executable at kernel virtual addresses.
  P2 risk; P1 if XEMU translation or XJIT code generation is active.

### 1.3 Heap — VMM Backing
STATUS: FUNCTIONAL — with CRITICAL correctness gap

heap.c does NOT use vmm_map(). It calls pmm_alloc_pages() directly and accesses
via HHDM (phys + HHDM_BASE). This is architecturally correct for freestanding
slab allocators — no requirement to VMM-map slab pages, since HHDM already covers
all physical RAM.

However: large allocations in kmalloc() call pmm_alloc_pages(order, PMM_ZONE_NORMAL)
while holding g_heap_lock. During that lock, _add_to_free_list() is called from
the split path in pmm_alloc_pages(), which acquires g_pmm_lock. This is a nested
lock sequence: g_heap_lock → g_pmm_lock. The established lock ordering in the
architecture is g_vmm_lock → g_pmm_lock → g_heap_lock. Acquiring g_pmm_lock
while holding g_heap_lock INVERTS the documented order.

FINDING 1.3a (P1) — LOCK INVERSION: g_heap_lock held while calling pmm_alloc_pages()
  kmalloc() acquires g_heap_lock at line 249, then calls pmm_alloc_pages() at
  line 262 (large path) or line 305 (_slab_create -> pmm_alloc_pages) (small path).
  pmm_alloc_pages acquires g_pmm_lock internally.
  If any code path acquires g_pmm_lock first and then tries g_heap_lock,
  deadlock is possible. On single-CPU this is not immediately dangerous (no
  true concurrency), but the fiber scheduler can context-switch and a fiber
  waiting on g_pmm_lock that another fiber is holding while also waiting on
  g_heap_lock produces a cooperative deadlock.
  DOCUMENTED ORDER: g_vmm_lock → g_pmm_lock → g_heap_lock.
  heap.c violates this by calling into PMM while holding its own lock.
  FIX: Release g_heap_lock before the PMM call, re-acquire after, re-validate
  cache state (another fiber may have modified it during the window).

FINDING 1.3b (P1) — kfree() LARGE-ALLOC PATH: DOUBLE PMM FREE RISK
  kfree() checks large_hdr_t magic (LARGE_ALLOC_MAGIC = 0xB1607B1Au).
  The check is: maybe_large = ptr - sizeof(large_hdr_t); if (magic == LARGE_ALLOC_MAGIC)
  Any slab allocation whose first 4 bytes of its slot happen to equal 0xB1607B1A
  will be misidentified as a large allocation and incorrectly freed via PMM.
  This is the HEAP-01 P2 finding from Sprint 1 that has NOT been fixed.
  No freelist range check is performed. A maliciously crafted or corrupt slot
  content can trigger pmm_free_pages() on an arbitrary physical address.
  FIX: Track large allocation addresses in a separate bitmap or hash, rather
  than relying on a forgeable magic pattern in the allocation itself.

### 2.2 XLAPIC — HHDM Mapping
STATUS: PASS

xlapic.h §9: XLAPIC_PHYS_TO_VIRT(p) = (volatile uint32_t *)((p) + 0xFFFF888000000000ULL)
xlapic_init: ctx->mmio = XLAPIC_PHYS_TO_VIRT(ctx->mmio_phys). CORRECT.

Default fallback 0xFEE00000 used if MSR reads zero. CORRECT for emulator compat.

FINDING 2.2a (P2) — LAPIC TIMER CALIBRATION INACCURACY (KNOWN):
  xlapic_timer_init uses 1M-iteration busy loop instead of xhpet_udelay().
  On i7-8565U at 2.1GHz idle, 1M loop iterations ≈ 476µs, not 10ms.
  bus_hz is underestimated by ~21x. Timer intervals will be ~21x shorter than
  requested. xenos_timer_calibrate() is called if g_xlapic.bus_hz > 0, but
  g_xlapic.bus_hz was set by the inaccurate calibration.
  The legacy g_lapic_bus_hz default = 100MHz is more accurate than the
  XLAPIC calibrated value on most x86 hardware.
  DOCUMENTED in memory as LAPIC Timer Calibration Risk. NOT FIXED.
  FIX: replace volatile loop with xhpet_udelay(&g_xhpet, 10000000ULL) in
  xlapic_timer_init. Requires XHPET initialized before XLAPIC timer (step 10d
  before step 10c in kmain — currently step 10c comes before 10d, WRONG ORDER).

FINDING 2.2b (P1) — DRIVER INIT ORDER: XLAPIC BEFORE XHPET
  In kmain():
    Step 10c: xlapic_init(&g_xlapic) — timer calibrated with busy-loop
    Step 10d: xhpet_init(&g_xhpet) — HPET initialized here
  xlapic_timer_init requires HPET for accurate calibration, but HPET is not
  yet available at step 10c. The single-CPU fiber scheduler boot proceeds with
  an inaccurate timer. On bare-metal, this affects all time-sensitive boot
  operations including TLS session timeouts, pal_thread_sleep_ns accuracy,
  and GENSD health check intervals.
  FIX: Reorder: XHPET init (10c) → XLAPIC timer calibration (10d).
  Note: XLAPIC_init (enable/ID/mask LVTs) can run early; only timer_init
  (which needs HPET) should be deferred until after XHPET is ready.

## DOMAIN 3: STACK LAYOUT AND KERNEL STACK SAFETY

### 3.1 Bootstrap Stack (entry.asm)
SIZE: 16KB (STACK_SIZE = 16384 bytes, _stack .. _stack_top in .bss)
  entry.asm line 413: STACK_SIZE equ 16384

FINDING 3.1a (P1) — BOOTSTRAP STACK TOO SMALL FOR TLS CONTEXT:
  xsec_tls_ctx_t = ~17.3KB (dominated by record_buf[16640]).
  If any TLS session is initiated from kmain() boot context (before GENSD
  launches dedicated fiber stacks), the 16KB bootstrap stack will OVERFLOW.
  Architecture memory note confirms: "minimum stack requirement = 32KB".
  The kernel stack for ring-0 interrupt/syscall entry is g_kern_stack = 32KB
  (main.c line 62: KERN_STACK_SIZE = 32768). This is correctly sized.
  But the _xenos_start bootstrap stack (entry.asm .bss) = 16KB.
  If kmain() itself triggers any code path that allocates a TLS context on
  the stack before switching to g_kern_stack, overflow occurs.
  Mitigation: kmain() switches RSP to g_kern_stack via gdt_reload() TSS.rsp0
  ONLY on interrupt/syscall entry, not for normal kmain() stack usage.
  The bootstrap 16KB stack is active for all of kmain()'s local variables and
  call frames. pal_init(), xacpi_init(), xpci_init() must NOT put TLS contexts
  on stack.
  RISK LEVEL: Medium — depends on whether TLS is used in boot paths.
  CURRENT STATUS: No TLS in boot path (XPKG/XNET not called from kmain()).
  ADVISORY: document that kmain() callee functions must not allocate >8KB of
  stack-local data. Consider switching to g_kern_stack early in kmain step 2.

### 3.2 Fiber Stacks
SIZE: 64KB per fiber (XENOS_FIBER_STACK_ORDER=4, 16 pages × 4KB)
  This is adequate for TLS (17.3KB required; 64KB available). PASS.

### 3.3 TLS recv buffer — Static Verification
Audit of tls13.c for recv buffer placement:
  Line ~87 (xsec_tls_ctx_t definition in xsec.h): record_buf[16640] is a field
  in xsec_tls_ctx_t. The ctx is passed by POINTER to all TLS functions.
  The caller allocates the ctx. If caller allocates on stack, overflow risk.
  In tls13.c, no stack-local xsec_tls_ctx_t instances were found in the
  critical handshake paths. All internal buffers (poly_key_block, etc.) are
  <= 64 bytes. The ~32KB "static scratch buffer" concern from Sprint 3 was
  resolved by the streaming Poly1305 fix (P2-POLY-STACK-01). CONFIRMED FIXED.
  chacha20.c line 387: poly_zero[16] declared static const — not on stack. PASS.
  poly1305_state_t (80 bytes) IS stack-allocated in chacha20_poly1305_core. CORRECT.

## DOMAIN 4: INTERRUPT AND EXCEPTION HANDLING

### 4.1 IDT Setup Ordering
STATUS: PASS with IST caveat

entry.asm step 8: lidt [_idt256_ptr] — IDT loaded with ZERO entries.
entry.asm step 9: call kmain
kmain step 2: idt_init() — fills 256 entries and re-issues lidt.

CRITICAL WINDOW: From _xenos_start through kmain step 2, the IDT is loaded
but all 256 entries are zero (null descriptor). Any exception during this window
(e.g., #GP on misaligned access in serial_early_puts) will triple-fault.
This window includes:
  - _serial_init_16550() port I/O
  - pal_console magic validation

ASSESSMENT: Acceptable risk. The window is ~50 instructions of known-safe code.
No user-mode exists, no interrupts enabled (IF=0 from entry.asm pushfq step 3).

### 4.2 BSS Clearing
STATUS: PASS

entry.asm step 2 sets RSP = [_stack_top], and _stack is in .bss.
NASM .bss section: resb (reserve bytes) — initialized to zero by ELF loader.
AetherBoot is responsible for zeroing .bss before jumping to _xenos_start.
sprint13.yml verifies AetherBoot build. No explicit BSS zero loop in entry.asm.
RISK: If AetherBoot fails to zero BSS (e.g., if it mmaps ELF without zeroing),
g_free_list (PMM) and g_cr3_phys (VMM) start non-zero — corrupting allocators.
FINDING 4.2a (P2): Add BSS zero loop before _serial_init_16550 in entry.asm
  as a defence-in-depth measure (some bootloaders do not zero BSS).

### 4.3 ISR Error Code Vectors
STATUS: PASS

Vectors with hardware error codes correctly use ISR_ERR: 8,10,11,12,13,14,17,21,29,30.
All others use ISR_NOERR (push dummy 0).
Cross-referenced with Intel SDM Vol.3A Table 6-1. No mismatches. PASS.

### 4.4 RSP Alignment in _common_isr_entry
STATUS: PASS

Stack state at _common_isr_entry:
  Hardware pushes 5 qwords (SS, RSP, RFLAGS, CS, RIP) = 40 bytes.
  ISR stub pushes 2 qwords (error_code, vec) = 16 bytes.
  Total before GPR save: 56 bytes below the pre-interrupt RSP.
  56 % 16 = 8 — RSP is 8-byte misaligned (as required for call: call aligns to 16).
  _common_isr_entry pushes 15 GPRs = 120 bytes. Total = 176. 176 % 16 = 0.
  sub rsp, 8 — RSP = 184 bytes below pre-interrupt RSP. 184 % 16 = 8.
  call xenos_isr_dispatch — pushes 8 bytes return addr, RSP = 192. 192 % 16 = 0.
  Callee sees RSP 16-byte aligned. CORRECT per System V AMD64 ABI. PASS.

### 4.5 Double Fault / IST
STATUS: GAP (P1, KNOWN)

idt_init() sets all IDT entries with ist=0 (legacy RSP0 stack from TSS).
Vector 8 (#DF) should use IST=1 to guarantee a known-good stack even if
the kernel stack itself is corrupted (e.g., stack overflow → #PF → #DF).
IST-01 from Sprint 1 audit: NOT FIXED across all 13 sprints.
Impact: double fault on stack overflow triggers another #DF on the same
corrupted RSP, causing a triple fault and silent reboot. Diagnosis impossible.
FIX: Allocate two 4KB IST stacks, configure _tss0.ist[0] = IST_stack_top,
set idt[8].ist = 1. This is a 1-day fix.

## DOMAIN 5: SMP READINESS

### 5.1 BSP/AP Distinction
STATUS: SINGLE-CPU ONLY — NO AP SUPPORT

xlapic_init() correctly reads BSP flag: ctx->is_bsp = (apic_base_msr & XLAPIC_MSR_BSP) ? 1 : 0.
IPI broadcast function xlapic_send_ipi_broadcast() implemented.
However:
  - No Application Processor (AP) startup sequence (INIT-SIPI-SIPI protocol).
  - No per-AP TSS, per-AP interrupt stack, per-AP GS_BASE.
  - g_percpu[MAX_CPUS=8] array exists but only [0] is populated.
  - g_fibers[] fiber table has no per-CPU partitioning.
  - All VMM operations use single g_vmm_lock — correct for single CPU.
  - All PMM operations use single g_pmm_lock — will become contention hot spot on SMP.

### 5.2 ChaCha20 Thread Safety
STATUS: NOT THREAD-SAFE — DOCUMENTED GAP

chacha20.c: chacha20_block() uses only stack-local state (state[16], working[16]).
chacha20_encrypt() uses stack-local keystream[64]. NO static/global state in
encryption path. THREAD-SAFE for concurrent encrypt calls with different keys/nonces.

poly1305_state_t is stack-allocated in chacha20_poly1305_core(). THREAD-SAFE.
poly1305_mac() wrapper uses stack-local poly1305_state_t. THREAD-SAFE.

EXCEPTION: chacha20_poly1305_core() line 387: static const uint8_t poly_zero[16]
is a read-only static constant. Zero-write path (read from this) is thread-safe.
PASS.

However: xsec_audit_log() (called on MAC failure) uses a lock-free ring with
pal_atomic_add. This is safe for concurrent TLS sessions on multi-core.

CONCLUSION: ChaCha20-Poly1305 is thread-safe by construction. The "thread safety
gap" flagged in earlier sprints has been resolved by the streaming Poly1305 refactor
(P2-POLY-STACK-01). No global mutable state in the encryption path.

### 5.3 VMM on SMP
STATUS: SINGLE-LOCK BOTTLENECK (P2)

vmm_map/vmm_unmap acquire g_vmm_lock for the entire mapping operation including
per-page invlpg. On SMP with concurrent map/unmap, this serializes all address
space modifications. Acceptable for Sprint 13 single-CPU boot. Will require
per-PCID or per-address-space fine-grained locks for multi-core.

Additionally: vmm_unmap does NOT issue TLB shootdown IPIs to other CPUs.
On SMP, remote CPUs may still have stale TLB entries after vmm_unmap. This is
safe on single-CPU (single TLB), fatal on SMP.

### 5.4 Spinlock Implementation
STATUS: CANNOT VERIFY WITHOUT pal_aether.c spinlock bodies

The PAL spinlock (pal_spinlock_t, pal_spin_lock, pal_spin_unlock) must use
atomic XCHG or LOCK CMPXCHG. If implemented as plain flag writes (as a stub),
all mutual exclusion is illusory on SMP. Review required at Sprint 14 SMP bringup.

### 6.2 Boot Handoff Structure
STATUS: TWO-STRUCTURE MISMATCH (P1)

GEN.OS uses TWO boot info structures:
  (A) xenos_boot_info_t (xenos.h) — 8-field lean structure used in pmm_init/kmain
  (B) xenos_boot_handoff_t (boot_handoff.h) — full 4096-byte packed structure
      with memory map, modules, ACPI, framebuffer, APIC addresses, CPU features,
      TPM event log, cmdline, HP hardware flags

kmain() receives xenos_boot_info_t *bi (structure A), NOT xenos_boot_handoff_t.
Structure A has: mem_map_addr, mem_map_count, mem_map_desc_size (raw EFI map fields).
Structure B has: mem_regions[128] typed xenos_mem_region_t (post-processed regions).

These are PARALLEL but INCONSISTENT representations of the same data:
  - boot_handoff.h version = 3, XENOS_HANDOFF_VERSION = 3
  - xenos_boot_info_t has magic XENOS_BOOT_MAGIC = "AETHER01"
  - xenos_boot_handoff_t has magic XENOS_HANDOFF_MAGIC = "GENOSHOF"
  - pmm_init() reads raw EFI descriptors via mem_map_addr/count/desc_size
    → requires struct A
  - boot_handoff_t has xenos_mem_type_t regions — different type system
    than efi_memory_type_t used by pmm_init

If AetherBoot actually passes struct B (GENOSHOF handoff), but kmain() interprets
it as struct A (AETHER01), the magic check at kmain step 1 fails and the kernel
halts. If AetherBoot passes struct A, struct B fields (lapic_phys, cpu_features,
etc.) are never consumed.

The sprint13.yml CI only checks for "AETHER01" magic and bi->acpi_rsdp_phys.
It does NOT verify which structure AetherBoot actually writes.
aetherboot/src/aetherboot.c is not readable (no init/boot/ path found in glob).

FINDING 6.2a (P1): Dual boot structure ambiguity. Either:
  (a) AetherBoot writes struct A (xenos_boot_info_t) — then struct B is dead code
  (b) AetherBoot writes struct B (xenos_boot_handoff_t) — then struct A is an alias
      and magic mismatch will HALT the kernel on every boot
  A definitive resolution requires reading aetherboot/src/aetherboot.c.
  The lab assessment given sprint13.yml verifications: likely AetherBoot writes A.
  The handoff_t appears to be a more complete aspirational spec not yet consumed.

### 6.3 Boot Order — Interrupts
STATUS: CONCERN (P2)

kmain step 14: __asm__ volatile("sti") — interrupts enabled.
kmain step 10: apic_init() — LAPIC enabled, all LVTs masked.
kmain step 10c: xlapic_init() — LAPIC re-initialized, all LVTs masked.
Interrupts are disabled from entry.asm (RFLAGS bit 1 only, IF=0) until step 14.
This is safe. PASS.
