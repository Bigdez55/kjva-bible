# Cloud + XMesh + Network Audit — 2026-04-24

## Summary
Mesh code compiles (freestanding C) but is NOT initialized in any runtime path.
`xnet_gossip_init`, `xnet_mdns_init`, `handoff_init`, `cross_device_init`, `node_agent_init`
have ZERO call-sites outside their own TUs. xnet_tick() DOES call gossip_tick/mdns_tick,
but both are no-ops when `g_gossip_initialized=false`.

## Key findings
- No `genos-meshd` daemon exists. The name appears only in `tests/mesh/test_meshd.c` and sprint41.yml.
- docker-compose has 5 containers, not 10. The "10 services" claim is false — they are
  14 daemons inside supervisord in `genos-citadel` container; separately, 5 HTTP apps
  run inside `genos-cloud` (identity/sync/update/gate/provenance mounted under FastAPI).
- `genos-cloud` does NOT include kernel/soulmgrd/eventjournald/gaterunnerd/ahkid/companion/
  forged/telemetryd. Those live in `genos-citadel` (Citadel/docker/supervisord.conf).
- RDMA: zero ibverbs calls in tree. `qp_relay.c` is a Linux-kernel-only module (out-of-tree).
  Production path uses `qp_relay_stubs.c` which is FAIL-CLOSED (-EPERM).
  SoftRoCETransport is UDP-based emulation (port 4791), not real RDMA.
- Two divergent copies of `vdram_rdma.py`: `devices/cloud/cloud/vdram_rdma.py` (497 LOC)
  vs `devices/cloud/cloud/tc_agent/vdram_rdma.py` (524 LOC).
- `node_agent.c` is freestanding SWIM. No entrypoint links it into GENSD.
- `xstore_adapter.py` / `xblob_cloud_adapter.py` fall back to in-memory dict "stub mode"
  when FFI unavailable — persistence silently lost.

## Required fixes to make cloud+mesh operational
1. Wire `xnet_gossip_init(ip, 7946)` + `xnet_mdns_init(hostname)` into GENSD xnet_daemon_main
2. Link `node_agent_init/run` as a GENSD service (new entry in boot_chain.c)
3. Remove duplicate vdram_rdma.py; keep tc_agent version
4. Enforce FFI-required mode (fail startup if xstore_ffi missing in production)
5. Either build/run `genos-meshd` or delete the name from test
6. Document that cloud has 5 services (not 10)
