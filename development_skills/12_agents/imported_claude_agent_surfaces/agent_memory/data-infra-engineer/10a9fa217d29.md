## Sprint 42 Orange Suite XSTORE Backend

**Why:** Sprint 41 identified that mail/calendar/drive called `xstore_open()` with a 6-arg form that no longer matched the 8-arg callback API in xstore.h. Notes used an even older 3-arg form with `XSTORE_OPEN_RDWR | XSTORE_OPEN_CREATE` flags that no longer exist. All four apps needed a shared RAM-disk-backed XSTORE instance.

**Files created/modified:**
- `shell/xshell/include/orange_store.h` — `orange_ramdisk_ctx_t`, 3 callback declarations, 3 lifecycle functions
- `shell/xshell/src/orange_store.c` — full implementation (PASS: -fsyntax-only -Werror)
- `apps/orange/notes.c` — added `orange_notes_set_store()` injection point; replaced old 3-arg `xstore_open()` with graceful-degradation fallback

## Spec Errors I Corrected (CRITICAL to remember for future work)

### PAL API — pal_pages_alloc
- Spec used: `pal_pages_alloc(32, PAL_MEM_LOCAL, 0, 0)` returning `uint8_t *`
- Actual: `pal_pages_alloc(num_pages, page_size, flags, numa_node, out_handle*)` returns `pal_status_t`
- `PAL_MEM_LOCAL` does NOT exist — correct flag is `PAL_MEM_NORMAL` (value 0)
- Returns a `pal_handle_t` via out_handle, NOT a pointer
- Page size argument: use `PAL_PAGE_SIZE_4K` (= 4096)

### PAL API — pal_map_pages
- Signature: `pal_map_pages(pages_handle, virt_addr, offset_pages, num_pages, flags, out_handle*, out_vaddr*)`
- Returns `pal_status_t` + mapped VA via `uintptr_t *out_vaddr`
- `PAL_MAP_RW` does NOT exist — use `PAL_MAP_READ | PAL_MAP_WRITE`
- `virt_addr = 0` means "PAL chooses the virtual address"

### PAL API — pal_memcpy / pal_memset
- These do NOT exist in pal.h
- Use `xstore_memcpy()` / `xstore_memset()` from `xstore.h` (already inlined, no extern needed)

### PAL API — VA leak ordering (from Sprint 41 post-mortem)
- ALWAYS call `pal_unmap(map_handle)` BEFORE `pal_pages_free(pages_handle)`
- Reversing the order leaks 336 MB of VA per restart

### xstore_open() API (current, as of Sprint 37+)
- 8 arguments: `(store*, read_page_cb, write_page_cb, sync_cb, io_ctx, cache_buf, wal_buf, create_bool)`
- Old 3-arg form (`store*, "name", flags`) and old 6-arg form both removed
- `XSTORE_OPEN_RDWR` and `XSTORE_OPEN_CREATE` flags do NOT exist

## Notes.c Injection Architecture

- `orange_notes_set_store(xstore_t *store)` is now a PUBLIC function in notes.c
- Sets `s_notes_store_ptr = store` and `s_store_open = 1`
- All persist operations use `s_notes_store_ptr` (pointer) not `&s_notes_store` (direct)
- Destroy path only calls `xstore_close(&s_notes_store)` if `s_notes_store_ptr == &s_notes_store` (ownership guard)
- Mail/Calendar/Drive already had `orange_{app}_set_store()` — no changes needed there

## PAL_FREESTANDING Guard Pattern
- All Orange .c files define `#define PAL_FREESTANDING` unconditionally at top
- When compiling with `-DPAL_FREESTANDING` on CLI this causes `-Werror,-Wmacro-redefined`
- Fix: use `#ifndef PAL_FREESTANDING / #define PAL_FREESTANDING / #endif` guard
- Applied this guard in `orange_store.c`; notes.c has this as a pre-existing issue (not our bug)

## RAM Disk Grow Strategy
- Initial: 32 pages (128 KB), doubling: 32→64→128→...→4096 (16 MB max)
- Grow: allocate new pages → map → memcpy old data → pal_unmap old → pal_pages_free old → swap pointers
- Lock scope: `disk->lock` held for entire grow+write in `write_page` callback
- Read callbacks do NOT take the lock (capacity only grows, pointer swap is atomic with write lock)

## Medallion Tier Assignment
- RAM disk + xstore_t instance = Bronze (volatile, in-process, not persisted across reboots)
- App data written via xstore_put() = Silver
- ETL decision: ELT not applicable; this is an in-process storage backend

**How to apply:** When any future work touches PAL memory APIs or XSTORE I/O callbacks, verify against these exact signatures before writing spec or implementation code.
