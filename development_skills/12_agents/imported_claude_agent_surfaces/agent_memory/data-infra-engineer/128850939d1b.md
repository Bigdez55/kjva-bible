## XSTORE FFI — Three Compile-Blocking Bugs (UNRESOLVED as of HEAD)

**Bug 1 — open() signature mismatch (compile failure):**
`py_xstore_ffi_open()` at `store/xstore/ffi/python_ffi.c:423` calls `xstore_open(path, &g_store)` with 2 args.
Real `xstore_open()` in `store/xstore/include/xstore.h:386` requires 8 parameters: `(xstore_t*, read_page_fn, write_page_fn, sync_fn, io_ctx, cache_buf, wal_buf, create)`.
This is a compile-time hard failure. The Sprint 36/37 "P0-FFI-1/2/3 fixed" claim in MEMORY.md is incorrect.

**Bug 2 — scan() wrong struct fields (compile failure):**
`python_ffi.c:535` initializes `xstore_val_out_t val_out = { .data = NULL, .len = 0 }`.
Correct fields per `xstore.h:173-177` are `.buf`, `.buf_len`, `.out_len`. Fields `.data` and `.len` do not exist on `xstore_val_out_t` (they exist on `xstore_val_t` — the immutable write descriptor). Lines 551-552 dereference `val_out.data` and `val_out.len` — both wrong.

**Bug 3 — scan() NULL txn dereference (runtime crash):**
`python_ffi.c:518` calls `xstore_cursor_open(g_store, NULL, &lower, ...)`.
`xstore_cursor_open()` unconditionally dereferences `txn->active` — NULL txn causes immediate crash.

**Why:** pal_linux.c was the expected PAL-Linux shim for the POSIX (Python FFI) environment. It was never created — MEMORY.md "Phase 0 = PAL-Linux" is stale. The FFI was written against an older 2-arg xstore_open() prototype and never updated when the signature expanded to 8 args.

**How to apply:** Before any work on XSTORE Python FFI, treat the module as entirely non-functional and requiring a full rewrite of `py_xstore_ffi_open()` (provide in-memory page buffer callbacks + static cache/wal buffers) and a full fix of `py_xstore_ffi_scan()` (use `.buf`/`.buf_len`/`.out_len` and pass a valid txn handle or the cursor open overload that accepts NULL).

## XBLOB hash index — non-persistent (known open issue)

`store/xblob/index.c` — 1M-slot open-addressing hash table is entirely in-memory.
On process restart, index is empty. Raw blob slot data on backing store survives but is unreachable without index reconstruction.
GC log at `gc.c` replays refcounts but does NOT rebuild the hash index. There is no index scan/rebuild path.
**Impact:** Any restart of a process using XBLOB = total index loss. Content-addressed lookup returns XBLOB_ERR_NOT_FOUND for all existing blobs until re-put.

## nvme_sync() is a no-op — durability gap

`store/xstore/io/nvme_backend.c:205` — `nvme_sync()` returns `XSTORE_OK` without issuing NVMe Flush (OPC=0x00).
On controllers with Volatile Write Cache (VWC), committed WAL entries are not durable against power loss.
`xstore_wal_checkpoint()` calls `store->sync()` and relies on it for durability — that guarantee is void on VWC hardware.

## btree internal node split — not implemented

`store/xstore/btree/btree.c` — internal node split returns `XSTORE_ERR_CAPACITY` when internal node is full.
At order 128 (127 keys per internal node), the tree caps at approximately 127^2 * 255 leaf entries ≈ 4.1M keys before failing. Leaf splits work; only internal splits are missing.

## PAL-Aether is the ONLY PAL backend — pal_linux.c does not exist

`pal/src/` contains: `pal_aether.c`, `pal_buddy.c`, `pal_builtins.c`, `pal_file_xvfs.c`, `pal_heap.c`, `pal_net_posix.c`.
`pal_linux.c` does NOT exist. MEMORY.md "Phase 0 = PAL-Linux backend, wraps Linux syscalls" is stale documentation.
PAL-Aether went directly from Sprint 1. Any build target that expects `pal_linux.c` will fail to link. `pal_net_posix.c` exists as a POSIX adapter for the `pal_net_*` functions.

## PAL AArch64 is fully functional (not stubs)

- `cntvct_el0` virtual counter read — LIVE
- `cntfrq_el0` frequency discovery — LIVE
- `svc #0` syscall convention (x8=nr, x0-x3=args) — LIVE
- PL011 UART via `g_uart_base` MMIO — LIVE (requires AetherBoot to set base)
- CSPRNG: `RNDR` instruction (ARMv8.5-RNG) with `cntvct_el0` LCG fallback — LIVE
  WARNING: fallback is NOT cryptographically secure (cycle counter + LCG)
- SHA-256: pure-C FIPS 180-4 implementation, no arch-specific code — LIVE for both x86/ARM64

## xkabi_native/ — fully functional PAL-wired implementations

All 15 .c files are real implementations against PAL syscalls, not stubs:
- `native_process.c`: XK_SYS_PROCESS_CREATE/START/EXIT/KILL all wired via `_xk_syscall4()`
- `native_vmar.c`: pal_pages_alloc + pal_map_pages, 512-slot mapping table, canonical VA check
- `native_vnode.c`: 9 xk_vnode_* functions wired to XVFS (extern xvfs_open/read/write/close/seek/stat/chmod/mkdir). Claims "All 9 xk_vnode_* functions are live — no more NOT_SUPPORTED stubs."
  Exception: `xk_vnode_setattr()` for size>0 truncation returns `XK_ERR_NOT_SUPPORTED`
  Exception: `xk_vnode_ioctl()` for unknown commands returns `XK_ERR_NOT_SUPPORTED`
- `native_iommu_stub.c`: wired to kernel `iommu_vtd_create_context/map/unmap` — LIVE
- System info topics 0/1/2: `xk_system_info()` populated — LIVE

## XISC Linux personality — ENOSYS surface (25 return points, confirmed by grep)

Unconditional ENOSYS (always fail): sendmsg(46), recvmsg(47), getsockname(51), getpeername(52), socketpair(53), setsockopt(54), getsockopt(55), fork(57), vfork(58), tgkill(234), getpgrp(111), unsupported futex ops, mmap file-backed, mmap shared.
Conditional ENOSYS (fail if XNET weak link absent): socket, connect, accept, sendto, recvfrom, shutdown, bind, listen, fstat/stat family.
Thread model is live: `linux_thread_ctx_t` has fd_table[256], vmar_table, TLS FS/GS, PID/PPID/UID/GID.

## XPKG signature verification — real (not stub)

`pkg/xpkg/verify/pkg_verify.c`: Ed25519 verify via `xsec_ed25519_verify()` wired to XSEC.
GENOS_SYSTEM_PUB_KEY hardcoded (32-byte dev key, matches XPKG_ROOT_FINGERPRINT in boot_chain.c).
Revocation list: SHA-256 of public key, O(n) scan at verify time.
Key snapshot pattern prevents races with concurrent add/remove.
S3-DI-H05 fixed: content_hash covers [pay_off, sig_off) (payload only), not [meta_off, sig_off).
Lock ordering documented: s_key_lock always acquired before s_rev_lock.

## XPKG install is degraded — in-memory VFS, volatile registry

Files extracted to `g_vfs_table[256]`, max 4096 bytes/file — lost on process exit.
Registry `g_installed_registry` is static in-memory array — lost on restart.
Repo fetch via XNET TCP (all 4 functions weak-linked). IPv4 dotted-decimal only, no DNS.
9-gate refusal machine in `refusal_gate.c` is fully real.

## XAS — functional x86-64 assembler, no AArch64 emission

`xas/src/xas.c`: 20 XISC mnemonics to x86-64, two-pass, correct ELF64 PT_LOAD at 0x400000.
`hlt` maps to `UD2` (safe in user mode). Only x86-64 target supported by design.
