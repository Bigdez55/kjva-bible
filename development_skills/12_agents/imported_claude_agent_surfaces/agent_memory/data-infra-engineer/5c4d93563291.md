## 1. PROVENANCE PIPELINE AUDIT

### 1.1 provenance_client.py — SERVICE_URL PATH MISMATCH (CRITICAL — FIXED)

**Finding:** SERVICE_URL default was `http://provenance:8000/api/v1/events`.
provenance/app.py registers `POST /log` (not `/api/v1/events`).
Every emit() call against a live service would have returned 404, silently
falling back to the JSONL file for ALL events regardless of service health.

**Fix applied:** SERVICE_URL default changed to `http://provenance:8000/log`.

**Provenance service route map (confirmed from app.py):**
- `GET /healthz` — liveness probe (SELECT 1 — correct)
- `POST /log` — write provenance record
- `GET /provenance` — list all (unbounded — pagination P2 gap remains)
- `GET /provenance/{action_id}` — get by ID

### 1.2 provenance_client.py — Thread Safety (CRITICAL — FIXED)

**Race 1:** `_entry_count += 1` was unguarded. Two concurrent callers see count N,
both increment to N+1. File has N+2 entries but counter says N+1. Rotation was
eventually suppressed.
**Fix:** `threading.Lock()` wraps `_rotate_if_needed_locked() + open(append) + _entry_count += 1`
as one atomic critical section.

**Race 2:** `flush_fallback_to_service()` read file with `read_text()`, then wrote
survivors back with `write_text()`. A concurrent `_write_fallback()` that appended
a record between the read and write had its entry permanently clobbered.
**Fix:** flush now holds the same lock during the entire read-stream-writeback-unlink
cycle. Write-back uses `tempfile.mkstemp() + fsync + os.replace()` (atomic rename).

**Race 3 (known, documented):** Multi-process use of same FALLBACK_DIR (two daemon
instances) is not protected by `fcntl.flock`. Single-process use is fully safe.
This is a LOW risk documented in the test suite.

### 1.3 provenance_client.py — Additional Fixes

| Issue | Severity | Status |
|-------|----------|--------|
| UnicodeDecodeError on binary/corrupt fallback file in __init__ | LOW | FIXED: `errors='replace'` |
| OSError (disk full) propagated out of emit() | LOW | FIXED: caught in emit(), returns False |

| flush read_text() loads up to 10MB into RAM | HIGH | FIXED: stream line-by-line via open() iterator |
| No fsync after fallback write | LOW | FIXED: fsync added after every append |

## 2. DATA FLOW TRACING

### 2.1 thermal_predictor.py → write_state() → /run/genos/thermal-prediction.json

**Previous state:** `STATE_PATH.write_text(json.dumps(...))` — NOT atomic.
Reader sees empty/partial file between truncate and close. SIGKILL leaves truncated file.

**Fixed:** tmpfile + fsync + os.replace() — atomic on same tmpfs filesystem.

**Schema validation:** `schema_version: 1` CONFIRMED present in write_state() output.

**monotonic_ns field ADDED:** Consumers can use this for unambiguous ordering
independent of NTP clock jumps.

**ThermalState TypeScript type (compositor/genos-shell/src/renderer/types.ts) — SCHEMA MATCH ANALYSIS:**

```typescript
export type ThermalState = {
  schema_version: number;
  [deviceId: string]: { gpu_c: number; cpu_c: number; phase: string; alert: string } | number;
};
```

Python `_write_state()` output per device:
```python
{"gpu_c": v.gpu_temp_c, "cpu_c": v.cpu_temp_c, "phase": v.phase, "alert": v.alert}
```

**MATCH: CONFIRMED.** All 4 device-level fields (gpu_c, cpu_c, phase, alert) align with the TypeScript type.

**DISCREPANCY NOTED:** The TypeScript type allows `| number` for the index signature (to accommodate `schema_version: number` and `timestamp: number` at the top level). This is a TypeScript design workaround — the type is technically unsound (it conflates device objects with number fields). No runtime impact, but it means TypeScript will not flag accessing a number as if it were a device object. Document as LOW.

**dbus_state_bridge.py — read_thermal() validation:**
```python
if data.get("schema_version") != 1:
    return None
```
CONFIRMED: schema_version guard is present. Returns None (excluded from snapshot) if version wrong. Correct fail-safe behavior.

**Consumer chain:** thermal_guardian._write_state() → /run/genos/thermal-state.json → dbus_state_bridge.read_thermal() → build_snapshot()["thermal"] → STATE_FILE write → Electron shell IPC.

**Latency budget:** guardian polls at 500ms; bridge runs every 2s asyncio.sleep; total consumer latency = up to 2.5s from thermal event to shell UI update. Acceptable for thermal monitoring use case.

### 2.3 fabric_energy_controller.py → write_report() → /run/genos/fabric-energy-state.json

**Previous state:** non-atomic write_text() in write_report().
**Fixed:** tmpfile+fsync+os.replace().

**FabricEnergyState TypeScript type vs write_report() output — SCHEMA MATCH ANALYSIS:**

TypeScript expects:
```typescript
{ schema_version, timestamp, metrics: { current_lanes, state, total_energy_joules,
  baseline_energy_joules, savings_pct, lane_transitions, idle_time_s, dense_time_s,
  current_power_w }, gate: { target_savings_pct, achieved } }
```

Python write_report() emits all these fields. **MATCH: CONFIRMED.**

**New field `monotonic_ns` added** to write_report() output. TypeScript FabricEnergyState
type does not include it — TypeScript will ignore the unknown field at runtime. No issue.

**Energy accumulation reset on restart** — still an open gap (Demo 9 gate risk).
Total energy resets to 0 each time the process starts. Fix: load prior totals from
existing report on startup. OPEN as HIGH.
