## SECTION 1: WAL CORRECTNESS — INSERT VALUE RECOVERY FIX ANALYSIS

### What the fix does (mvcc.c lines 212-226)

```c
if (we->op == XSTORE_WAL_OP_INSERT) {
    xstore_key_t k;
    k.data = we->key;
    k.len  = we->key_len;
    xstore_val_out_t vout;
    vout.buf     = entry.val;
    vout.buf_len = XSTORE_MAX_VAL_LEN;
    vout.out_len = 0;
    if (xstore_btree_lookup(store, txn->snapshot_lsn, &k, &vout) == XSTORE_OK) {
        entry.val_len = (uint16_t)vout.out_len;
    }
}
```

At commit time, for each INSERT write-set entry, the code reads the value back from the B+ tree
page cache using a lookup keyed on the write_set key, and copies it into the WAL entry's val[]
field. This addresses the prior P1-FUNC-01 finding where WAL INSERT entries had val_len=0.

### FINDING FE-WAL-01 (P0): FIX IS SEMANTICALLY INCORRECT — snapshot_lsn PREDATES THE INSERT

The btree lookup uses `txn->snapshot_lsn` — the LSN captured at `xstore_txn_begin()` time,
which is `store->next_lsn` at that instant.

The actual insert in `xstore_btree_insert()` (btree.c:532) uses:
```c
uint64_t lsn = pal_atomic_add(&store->next_lsn, 1);
```

This LSN is strictly greater than `txn->snapshot_lsn` (because `snapshot_lsn` was captured
before the insert's `pal_atomic_add`). The value is stored in the page with this newer LSN.

When Sprint 5 enables per-version MVCC chains in `xstore_btree_lookup()`, the lookup at
`txn->snapshot_lsn` will return `XSTORE_ERR_NOT_FOUND` because the value was committed at a
higher LSN than the snapshot.

Currently (Sprint 4), btree.c line 510 has:
```c
(void)snapshot_lsn; /* Full per-version chain in Sprint 5 */
```
The snapshot_lsn is completely ignored. The fix works NOW by accident — the lookup finds the
most recent physical value regardless of the LSN passed.

**Verdict:** The fix resolves the immediate val_len=0 symptom but is a time-bomb. It will
silently produce empty WAL entries when Sprint 5 activates version chains.

### FINDING FE-WAL-02 (P0): LOOKUP-AT-COMMIT DEPENDS ON PAGE STILL BEING IN CACHE

The fix assumes the value written to the btree during `xstore_txn_put()` is still present in
the page cache at commit time. This is NOT guaranteed:

1. `xstore_txn_put()` calls `xstore_btree_insert()` → marks page dirty in cache.
2. A subsequent `xstore_cache_get()` for a different page triggers clock-hand eviction.
3. Eviction (`_cache_evict()`) writes the dirty page to backing store and frees the cache slot.
4. At `xstore_txn_commit()`, the btree_lookup calls `xstore_cache_get()` which re-reads the
   page from backing store — this will succeed and return the correct value.

So the lookup will work even after cache eviction, since `xstore_cache_get()` reloads from
the backing store on miss. However, this introduces a silent read amplification: every commit
for every key now potentially incurs a backing-store read on top of the write.

The deeper problem remains (DIE-XSTORE-01 from prior audit): the btree page was dirtied and
could have been flushed to backing store before the WAL COMMIT record was written. If power is
lost after the dirty eviction but before the WAL COMMIT is written, the page on disk has the
data but the WAL has no COMMIT for it — this is a WAL-first violation.

### FINDING FE-WAL-03 (PASS): EMPTY TREE EDGE CASE

When the store is new (empty B+ tree, root is a leaf with num_keys=0), `xstore_btree_lookup()`
for a non-existent key returns `XSTORE_ERR_NOT_FOUND`. In the recovery path:
```c
if (xstore_btree_lookup(...) == XSTORE_OK) {
    entry.val_len = (uint16_t)vout.out_len;
}
/* On lookup failure: val_len=0, replay remains best-effort. */
```
On NOT_FOUND, `entry.val_len` remains 0 with a code comment acknowledging this case. The WAL
replay would insert a zero-length value. This is incorrect but the comment treats it as
"best-effort." For an empty tree this scenario is unlikely in practice.

### FINDING FE-WAL-04 (PASS): DUPLICATE KEY EDGE CASE

If `xstore_txn_put()` is called twice for the same key within one transaction (a re-put with
the same val_len), `xstore_btree_insert()` executes the in-place upsert path (btree.c:554-566)
which updates the value bytes and returns `XSTORE_OK`. The write_set records one entry per
`xstore_txn_put()` call, so there may be two write_set entries for the same key. At commit time,
both entries attempt btree_lookup and both succeed — the WAL gets two INSERT entries for the
same key. Replay applies them in order, so the second value wins. This is correct behavior.

### FINDING FE-WAL-05 (P1): CONCURRENT TRANSACTIONS — LOOKUP AT COMMIT RETURNS WRONG VERSION

Txn T1 inserts key K=V1 and proceeds to commit.
Concurrent Txn T2 also inserts key K=V2 (different value, same val_len) and commits first
(OCC check passes because both record root_page — see FE-MVCC-01).

When T1 reaches the btree_lookup at commit time, the tree already has V2 (from T2's commit).
The lookup returns V2, not V1. T1's WAL entry records V2 as the value for its INSERT of K.

On WAL replay after crash, T1's INSERT replays V2 — which is T2's value, not T1's. The
committed state (T1 and T2 both committed) should show V2 as the last committed write.
In this case the result is accidentally correct because T1 overwrote T2's value in the btree
(T1 committed after T2, so T1's write is the final page state). But the WAL replay would
reproduce the LAST physical page state, not the strict transaction order.

Under different orderings this produces incorrect replay. This is a consequence of reading
back from the shared mutable page state rather than from a stable write buffer.

### WAL CORRECTNESS VERDICT

| Criterion | Status | Finding |
|-----------|--------|---------|
| val recovery present in WAL | YES | FE-WAL-01, FE-WAL-02 |
| val recovery semantically correct | NO (by accident only) | FE-WAL-01 |
| Sprint 5 safe | NO | FE-WAL-01 |
| Empty tree safe | DEGRADED | FE-WAL-03 |
| Concurrent txn correctness | DEGRADED | FE-WAL-05 |
| WAL-first ordering | NO | DIE-XSTORE-01 (prior audit) |
| WAL CRC replay (P0-SEC-02) | STILL BROKEN | No fix applied |

**The WAL CRC replay bug (P0-SEC-02) from the prior audit is NOT fixed in the current code.**
`xstore_wal_replay()` (wal.c:117-118) still computes CRC with the stored checksum field included
rather than zeroed, meaning every CRC comparison fails and all WAL entries are skipped during
crash recovery. The WAL replay mechanism remains non-functional.

## SECTION 2: MVCC SNAPSHOT ISOLATION

### Snapshot Timestamp Monotonicity

`xstore_txn_begin()` (mvcc.c:140-142):
```c
pal_spin_lock(&store->txn_lock);
uint64_t txn_id = store->next_txn_id++;
uint64_t snap   = store->next_lsn;
pal_spin_unlock(&store->txn_lock);
```

`next_txn_id` is incremented atomically under `txn_lock`. Each call produces a strictly
increasing txn_id. `snap = store->next_lsn` captures the current LSN watermark under the
same lock. `next_lsn` is a `volatile uint64_t` incremented only via `pal_atomic_add`.

FINDING FE-MVCC-01 (PASS): snapshot_lsn is monotonically non-decreasing — a later begin()
call will see an equal-or-higher next_lsn. Strictly increasing only if at least one WAL append
has occurred between the two begin() calls.

### Read-Your-Writes Semantics

FINDING FE-MVCC-02 (DEGRADED): As noted in FE-WAL-01, the btree ignores snapshot_lsn entirely
(btree.c:510). This means:
- Read-your-writes: WORKS (by accident — the in-page value is always the latest version).
- Snapshot isolation from concurrent commits: ABSENT (concurrent commits are visible to the
  current txn's reads, violating SI semantics).

### Rollback Correctness

`xstore_txn_abort()` (mvcc.c:266-275):
```c
xstore_memset(txn->write_set, 0,
               txn->write_count * sizeof(xstore_write_entry_t));
txn->write_count = 0;
txn->active      = false;
return XSTORE_OK;
```

FINDING FE-MVCC-03 (P1): ROLLBACK DOES NOT UNDO BTREE MUTATIONS

`xstore_txn_put()` calls `xstore_btree_insert()` immediately, mutating the page cache.
`xstore_txn_abort()` only clears the write_set — it does NOT undo the btree mutation.

Consequence: aborted transactions leave orphaned key-value pairs in the B+ tree. If the
same key is then read by another transaction, it may see the aborted value. If a different
transaction inserts the same key after the abort, `xstore_btree_insert()` finds the key
already present (upsert path) and handles it — but only if val_len matches. If val_len differs,
it returns `XSTORE_ERR_CONFLICT` for a key that should be considered absent.

This is a second-order manifestation of the BTREE-FIRST problem (DIE-XSTORE-01). The correct
fix is to not call btree_insert until commit, at which point abort truly has nothing to undo.

### MVCC Shadow Table Correctness

FINDING FE-MVCC-04 (P1): Root-page conflict granularity (DIE-XSTORE-04, still present):
`xstore_txn_put()` and `xstore_txn_delete()` both set `we->page_no = store->sb.root_page`.
The OCC conflict check at commit time therefore reports conflict if ANY transaction committed
to ANY page since the snapshot — all writes collide at the root_page. No fix applied.

### MVCC Verdict

| Property | Status |
|----------|--------|
| snapshot_lsn monotonicity | PASS |
| Read-your-writes | ACCIDENTAL PASS (snapshot_lsn ignored) |
| SI: snapshot isolation from concurrent commits | FAIL (snapshot_lsn ignored) |
| Rollback undoes all changes | FAIL — btree not reverted |
| OCC conflict precision | FAIL — root-page collision, false positives |
| Write-skew prevention | NOT ATTEMPTED (documented accepted limitation) |

## SECTION 3: B+ TREE INTEGRITY

### Node Split Correctness

FINDING FE-BTREE-01 (PASS): Leaf split path (`_leaf_split()`, btree.c:360-423):
- Allocates right sibling via `_page_alloc()` — CORRECT
- Copies slots [mid, n) from left to right with correct rec_off calculation — CORRECT
- Exports separator key as first key of right sibling — CORRECT (B+ tree convention: all keys
  in right subtree >= separator)
- Truncates left to [0, mid) — CORRECT
- Marks both pages dirty — CORRECT

FINDING FE-BTREE-02 (PASS): Root split path (btree.c:605-619):
- Allocates new root internal node — CORRECT
- Sets left_child to original root, right_child to new sibling — CORRECT
- Updates `store->sb.root_page` — CORRECT
- Marks new root dirty — CORRECT

FINDING FE-BTREE-03 (P1): INTERNAL NODE SPLIT NOT IMPLEMENTED
btree.c:645-646:
```c
/* Parent internal node is full: Sprint 4 returns CAPACITY error. */
return XSTORE_ERR_CAPACITY;
```
Trees deeper than one level of internal nodes silently fail on leaf splits when the parent
is full. At order 128 this limits the tree to approximately 128 leaf pages (524,288 keys at
max density) before inserts into an already-split subtree start failing. This is acceptable
for Sprint 4 but is a hard capacity ceiling.

### Key Ordering Invariants

FINDING FE-BTREE-04 (PASS): Binary search in `_leaf_find_slot()` uses `_key_cmp()` which is
lexicographic with correct shorter-prefix-is-less semantics. The insertion path preserves sort
order: `_leaf_insert_at()` shifts the offset table right and inserts at the binary-search
position. Insert-after-split correctly routes the new key to the correct sibling by comparing
against the separator key. Order invariants are maintained.

FINDING FE-BTREE-05 (P1): MISSING LEAF CHAINING — RANGE SCANS TRUNCATED
`xstore_btree_cursor_advance()` (btree.c:780-786) sets `cursor->valid = false` at page
boundaries. Range scans over keys spanning multiple leaf pages silently terminate. No fix applied.

### Page Cache Dirty Bit Management

FINDING FE-BTREE-06 (PASS): `xstore_btree_insert()` calls `xstore_cache_mark_dirty()` after
every leaf mutation and after root creation — CORRECT.

FINDING FE-BTREE-07 (P2): `xstore_btree_delete()` calls `xstore_cache_mark_dirty()` but does
NOT compact the record data region. Deleted record bytes remain in the page data area (hole).
`_leaf_free_space()` only accounts for records at the maximum rec_off — holes before the last
record are uncounted. Insert candidates will be packed past the highest existing rec_end, so
the hole is permanent. Over repeated insert+delete cycles the page appears full while physically
holding fewer than the theoretical maximum keys. (This is DIE-XSTORE-05 from prior audit.)

FINDING FE-BTREE-08 (P2): DIRTY PAGE EVICTION DOES NOT UPDATE PAGE CHECKSUM
`_cache_evict()` (page_cache.c:113-118) writes dirty pages to backing store without updating
`hdr->checksum`. The checksum is only updated during `xstore_cache_flush()` at checkpoint time.
An evicted dirty page has a stale checksum on disk. If page-level integrity verification is
added (Sprint 5), these pages will fail the integrity check incorrectly.

## SECTION 4: XBLOB GC SAFETY — PIA-S4-P1-01 LOCK ORDERING FIX VERIFICATION

### Stated Fix in gc.c

The fix documented as PIA-S4-P1-01 is present in `gc.c` lines 138-153:

```c
/* Log the free event outside store_lock to avoid acquiring
 * gc_lock under store_lock (lock-order inversion / deadlock risk).
 * Copy hash to stack before releasing. */
uint8_t freed_hash[XBLOB_HASH_LEN];
uint32_t hi;
for (hi = 0u; hi < XBLOB_HASH_LEN; hi++) {
    freed_hash[hi] = e->hash[hi];
}
pal_spin_unlock(&blob->store_lock);
xblob_gc_log_append(blob, XBLOB_GC_OP_FREE, freed_hash, 0);
pal_spin_lock(&blob->store_lock);
```

The fix releases `store_lock` before calling `xblob_gc_log_append()` (which acquires `gc_lock`).
It copies the hash to the stack before releasing to avoid use-after-free from a concurrent
write modifying the index entry after the lock is dropped.

### Lock Ordering Analysis

FINDING FE-GC-01 (PASS): No deadlock path exists in the current codebase:
- `blob.c xblob_put()`: acquires store_lock, operates, releases store_lock, THEN acquires
  gc_lock (sequential, no nesting)
- `blob.c xblob_delete()`: same sequential pattern
- `gc.c xblob_gc_scan()`: acquires store_lock; calls gc_log_append (acquires gc_lock) only
  AFTER releasing store_lock (the fix)

No code path acquires gc_lock while holding store_lock. No code path acquires store_lock while
holding gc_lock. Lock ordering is SAFE.

### Residual Race in the Fix Window

FINDING FE-GC-02 (P1): TOCTOU BETWEEN TOMBSTONE AND LOG APPEND

The sequence in the GC scan loop (gc.c:131-153):
1. Entry is inspected: `ref_count == 0` and `state == OCCUPIED`.
2. `xblob_free_slot()` called — bitmap bit cleared.
3. `e->state = XBLOB_INDEX_SLOT_TOMBSTONE` — index entry tombstoned.
4. `pal_spin_unlock(&blob->store_lock)` — lock released.
5. `xblob_gc_log_append()` called — log entry written.
6. `pal_spin_lock(&blob->store_lock)` — lock reacquired, loop continues.

Between steps 4 and 6, a concurrent `xblob_put()` for the same hash will:
- Call `xblob_index_find()`, find the tombstone, and skip it (tombstones are not matches).
- Call `xblob_index_insert()`, allocating a new slot for a NEW index entry.
- The GC log at step 5 records `XBLOB_GC_OP_FREE` for this hash.

On crash recovery, `xblob_gc_log_replay()` replays the FREE record and sets
`blob->index[idx].ref_count = 0` for the new entry. This is incorrect — the new entry was
just put() with refcount=1.

Severity: P1 under concurrent put()+gc() workloads. The 32-byte hash copy to stack before
release protects against use-after-free but does not prevent the logical race.

### GC Compaction Insert Error Check

FINDING FE-GC-03 (P1): COMPACTION DOES NOT CHECK xblob_index_insert() RETURN VALUE
gc.c lines 193-195:
```c
xblob_index_insert(blob, chunk[j].hash,
                    chunk[j].blob_offset, chunk[j].blob_len);
/* Restore ref_count (xblob_index_insert sets it to 1) */
uint32_t ri = xblob_index_find(blob, chunk[j].hash);
```
The return value of `xblob_index_insert()` is not captured or checked. If the insert fails
(e.g., table full due to probe collision during partial rebuild), the entry is permanently
lost. The `xblob_index_find()` on the next line will return `(uint32_t)-1` and the
ref_count restore is skipped. The blob's data slot remains allocated (bitmap bit set) but
the index entry is gone — permanent storage leak.

### GC Stack Allocation

FINDING FE-GC-04 (P1): 14 KB STACK ALLOCATION IN KERNEL CONTEXT
gc.c:178: `xblob_index_entry_t chunk[GC_COMPACT_CHUNK];` = 256 × 56 = 14,336 bytes.
On a freestanding XENOS kernel context, kernel stacks are typically 8–16 KB. This single
declaration may overflow the kernel stack when called from a deeply nested call path.

## SECTION 5: WAL RING FOOTPRINT ANALYSIS

### Calculation

From xstore.h:
- `XSTORE_WAL_RING_SIZE = 65536` entries
- `xstore_wal_entry_t` = 8+8+4+1+1+2+4+4+256+4096 = **4384 bytes** (verified by _Static_assert)
- `XSTORE_WAL_BUF_SIZE = 65536 × 4384 = 287,309,824 bytes = ~274 MB`

### Is This a Real Problem?

YES — it is a real operational constraint, not a theoretical concern.

**Hardware context:** HP EliteBook x360 with 16 GB RAM. 274 MB = 1.7% of physical RAM.

**Allocation requirements:**
- The buffer is caller-provided (passed to `xstore_open()` as `wal_buf`).
- The buffer must be physically contiguous for the DMA/PAL subsystem.
- If allocated as a static global: will blow the BSS segment.
- If allocated on the kernel boot stack: immediate stack overflow.
- Correct allocation: via `pal_pages_alloc()` using DMA/MMIO-mapped physically contiguous pages.

**Runtime behavior:** The ring is IN USE — it is not sparse. Every WAL append writes a
full 4384-byte entry at `&store->wal[slot]`. The 274 MB is actively touched on every
transactional write.

**Is the entry size correct?** The `val[XSTORE_MAX_VAL_LEN]` field (4096 bytes) inflates
every WAL entry to 4384 bytes even for DELETE operations and COMMIT records where val is empty.
For a ring of 65536 entries, 65536 × 4096 = 256 MB is attributed to the value payload alone.

**Sprint 14 action:** This is a real problem that needs attention. The entry size must be
reduced — the most effective approach is to use a variable-length WAL record format rather
than a fixed-size entry with the maximum val size. For Sprint 4's scope (small key-value
stores for provenance and XPKG state), the practical WAL buffer can be much smaller. A
variable-length format with 4096-entry ring and sequence-numbered records would reduce the
buffer to ~4-8 MB at typical workloads.

**Deferred assessment:** The 274 MB WAL is a Sprint 14 target, not a Sprint 4/5 blocker,
provided the caller allocates the buffer correctly via PAL pages rather than on the stack.
The header comment at xstore.h §6 incorrectly describes entries as "fixed-size 128-byte
records" — the actual size is 4384 bytes. This documentation error (P2-DOC-01) may mislead
implementors into under-allocating the buffer.

## SECTION 6: XSTORE WIRING IN BOOT CHAIN

### xchain_storage() Analysis (boot_chain.c lines 89-124)

The function does NOT call `xstore_init()` or `xblob_open()`. It performs a memory map
validation only:

```c
static pal_status_t xchain_storage(const xenos_boot_handoff_t *handoff) {
    /* Validate: the XBLOB region must be marked as usable or reserved */
    if (handoff) {
        for (i = 0u; i < handoff->n_mem_regions; i++) {
            const xenos_mem_region_t *r = &handoff->mem_regions[i];
            if (r->phys_base <= XCHAIN_BLOB_REGION_PHYS &&
                end >= blob_end &&
                (r->type == XENOS_MEM_USABLE || r->type == XENOS_MEM_RESERVED)) {
                return PAL_OK;
            }
        }
    }
    return (pal_status_t)-1;  /* PAL_ERR_NOT_FOUND */
}
```

FINDING FE-BOOT-01 (DEFERRED — KNOWN): xchain_storage() DOES NOT CALL xstore_open()

The `§2` header comment in boot_chain.c explicitly states:
> "Full xblob_open() wiring is deferred to Sprint 9 (XSTORE driver)."

This is a documented architectural deferral. The boot chain validates that the XBLOB region
exists in the physical memory map, but does not initialize the storage engine. XSTORE and
XBLOB must be initialized by the caller after `xchain_run()` completes, once the NVMe driver
is available.

FINDING FE-BOOT-02 (PASS): NVMe-absent path is safe

In `xchain_run()` (boot_chain.c:549-554):
```c
rc = xchain_storage(handoff);
s_result.storage_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] WARN: storage init failed (%u)\n", rc);
}
```
Storage failure is non-critical — the boot chain continues. `s_result.storage_ok = 0` is
recorded and the `xchain_get_result()` accessor exposes it. The AI stage later checks memory
availability but does NOT gate on `storage_ok`. This is acceptable because XMIND can operate
from a RAM-loaded model without persistent storage.

FINDING FE-BOOT-03 (P2): XCHAIN_BLOB_REGION_PHYS IS A HARDCODED PLACEHOLDER

`XCHAIN_BLOB_REGION_PHYS = 0x8000000000ULL` (512 GiB). The comment says "This is a
placeholder — the real layout comes from XSTORE's partition table discovered during XSTORE
init." If the HP EliteBook x360 has less than 512 GiB of physical address space mapped to
the NVMe device at this offset, the validation will always fail. The memory map scan has no
way to succeed unless the platform's firmware actually maps storage at 0x8000000000.

This is a boot-time false negative: `xchain_storage()` will return failure on real hardware,
and `storage_ok = 0` on every boot. Since storage failure is non-critical, the system boots
but storage is never validated as ready.

FINDING FE-BOOT-04 (P2): NULL HANDOFF PATH SKIPS VALIDATION

If `handoff` is NULL (bare boot test environment), `xchain_storage()` returns failure
immediately. The WARN log is printed, but storage proceeds in degraded mode. For production
boots this is acceptable; for integration test environments it means storage is always
marked as failed regardless of actual NVMe availability.

## GO/NO-GO VERDICT

### Production Readiness: NO-GO

**Crash safety is non-functional.** Two independent P0 issues prevent the WAL from providing
any crash recovery:

1. P0-SEC-02: WAL replay CRC verification always fails → all entries skipped → replay is a no-op.
2. FE-WAL-02 (DIE-XSTORE-01): btree mutations reach the backing store before WAL COMMIT →
   WAL-first ordering violated.

In the event of a crash between checkpoint and the next checkpoint, committed data is lost.
The public API contract ("a transaction is durable after xstore_txn_commit() returns XSTORE_OK")
is false.

**XBLOB index persistence is absent.** All blob mappings are lost on every process restart.
This makes XBLOB unsuitable for any persistent use case until DIE-XBLOB-01 is resolved.

### What IS Correct

- B+ tree key ordering and split mechanics: functionally correct for the single-writer case.
- XBLOB SHA-256 content integrity: correct; every get() verifies the hash.
- XBLOB deduplication: correct.
- Boot chain storage probe: correctly scoped as non-critical and non-blocking.
- XBLOB GC lock ordering fix (PIA-S4-P1-01): present and correctly implemented for the
  no-concurrent-put() case.
- Snapshot LSN capture: monotonically non-decreasing, structurally correct for when
  per-version chains are eventually implemented.

### Minimum Viable Fix List for Sprint 14 Production Readiness

| Priority | Fix | Impact |
|----------|-----|--------|
| 1 | Fix WAL replay CRC: zero checksum field before calling xstore_wal_crc() in wal_replay(), then compare. | Restores all crash recovery. |
| 2 | WAL-first ordering: buffer values in write_set at put() time; call btree_insert inside commit() AFTER WAL+COMMIT are fsynced. | Eliminates BTREE-FIRST ordering violation. |
| 3 | XBLOB index persistence: write/load index buffer via blob I/O callbacks on close/open. | Makes XBLOB durable across restarts. |
| 4 | Fix snapshot_lsn in val recovery lookup: use store->next_lsn or retain value in write_set. | Prevents Sprint 5 silent corruption. |
| 5 | txn_abort() must undo btree mutations: restructured by fix #2 (abort before commit = nothing to undo). | Resolved as a side effect of fix #2. |
| 6 | GC compaction: check insert return value; abort and restore on failure. | Prevents permanent storage leaks. |
| 7 | GC stack allocation: convert 14 KB chunk array to static or caller-provided buffer. | Prevents kernel stack overflow. |
