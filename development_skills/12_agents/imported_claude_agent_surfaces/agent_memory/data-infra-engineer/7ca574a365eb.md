## Files

- `devices/cloud/platform/services/common/storage_backend.py` — feature flag + DualWriteBackend orchestrator
- `devices/cloud/platform/services/common/xstore_adapter.py` — XStoreKV FFI wrapper

## DualWriteBackend Contract (implemented 2026-03-24)

**Write ordering**: Postgres written FIRST (authoritative). XSTORE written after as best-effort. If Postgres raises, XSTORE is NOT written (prevents split-brain). If XSTORE raises after a successful Postgres write, the exception is caught, logged as WARNING with `exc_info=True`, and the method returns normally to the caller.

**Read path**: Postgres is the only value returned. XSTORE is read in parallel for shadow validation. Discrepancies are logged (pg_only / xs_only / field_mismatch) but never affect the return value or raise. XSTORE read failure is also non-blocking.

**Delete path**: Postgres deleted authoritatively. XSTORE deletion attempted after; failure is non-blocking.

**xstore_kv=None mode**: All XSTORE legs are silently skipped. DualWriteBackend degrades cleanly to pure-postgres mode.

## XStoreKV Additions (2026-03-24)

- `list_keys_by_prefix(prefix, limit)` — new method returns raw key strings (no value deserialization). Required for cutover bulk-delete sweeps.
- `_scan_ffi()` — internal helper extracted to avoid code duplication between list_by_prefix and list_keys_by_prefix.
- `__del__` now uses `hasattr(self, "_opened")` guard to survive interpreter-shutdown GC where instance attrs may already be collected.
- Removed module-level `STORAGE_BACKEND` variable (was duplicating `storage_backend.py` env read; eliminated split-read risk).

## ETL Decision

ETL (not ELT). JSON serialization happens in Python before either store receives bytes. The C FFI layer must never receive raw Python objects.

## Medallion Tier

DualWriteBackend operates on Silver-tier records. It is not a Bronze ingestion layer.

## Import Path

Tests and services must add `devices/cloud/platform/services` to sys.path before importing `common.storage_backend` or `common.xstore_adapter`. The `tests/test_sprint37*.py` files use the wrong path (`platform/services` — does not exist); those tests are pre-existing broken imports unrelated to these files.

## Why: Sprint 37 Phase 2 — 4-phase Postgres → XSTORE migration
- Phase 0: FFI hardening (Sprint 36)
- Phase 1: XStoreKV adapter (Sprint 37)
- **Phase 2: dual-write mode (this implementation)**
- Phase 3: Sync/MinIO migration
- Phase 4: Identity migration (highest risk)

## How to apply

When a service needs dual-write mode: instantiate `DualWriteBackend` with the service's existing `pg_write/pg_read/pg_delete` callables and an `XStoreKV` instance. The service's HTTP handlers do not change — all dual-write logic is inside the backend.
