# Sprint 0/1 Kernel Data Layer Audit — 2026-03-03

## Files Audited
- init/gensd/journal.c
- init/gensd/service.c
- kernel/linux-validation/vdram/{vdram.h, mr_registry.c, pf_handler.c, tier_migration.c, prefetch_engine.c}

## CRITICAL Defects (must fix before any production use)

### VDRAM-01 — tier_migration.c lines 126, 178 — SILENT DATA CORRUPTION
- `pos = (pm - ((struct vdram_mr *)pm->lru_node.prev)->pages) << PAGE_SHIFT;`
- `pm->lru_node.prev` is a `struct list_head*` pointing to another page_meta's lru_node, NOT a vdram_mr*
- The cast is completely wrong; `->pages` dereferences garbage; NVMe write/read lands at wrong offset
- Every T1↔T2 migration silently corrupts data
- Fix: add `struct vdram_mr *owner_mr` to `struct vdram_page_meta`; use `pm->owner_mr->pages`

### VDRAM-02 — prefetch_engine.c + vdram.h — COMPILE FAILURE / OOB WRITE
- `node->prefetch_state` referenced in 4 functions in prefetch_engine.c
- Field `prefetch_state` does NOT exist in `struct vdram_node` in vdram.h
- Will not compile; if patched, causes OOB write at sizeof(vdram_node) offset
- Fix: add `struct prefetch_state *prefetch_state;` to vdram_node in vdram.h

### JNL-01 — journal.c lines 262–321 — TORN HEADER READ UNDER CONCURRENT WRITE
- journal_flush_serial() snapshots wp/total under lock, then releases lock before scanning ring
- Concurrent journal_write() can overwrite ring bytes mid-scan
- Torn 16-byte header read can produce garbage mlen, misaligning all subsequent entry reads
- Fix: copy entire ring to staging buffer under spinlock, then scan staging buffer unlocked

## HIGH Defects

### SVC-01 — service.c lines 520–562 — KAHN DUPLICATE-DEP BUG
- in_degree[i] incremented once per requires[] entry (not unique deps)
- BFS decrements in_degree[v] only once per u dequeue (break on first match)
- Duplicate dep entry in GSD file: in_degree never reaches 0, service never queued
- Service placed in fallback block, false cycle log emitted, wrong startup order
- Fix: deduplicate requires[] in parse_requires() before in-degree computation

### VDRAM-03 — pf_handler.c line 309 — SPINLOCK + SLEEP VIOLATION
- pf_attestation_gate() called under ctx.page->lock (spinlock)
- T3 slow path calls rdma_transport_connect() + kzalloc(GFP_KERNEL) — both can sleep
- BUG_ON in DEBUG kernel; silent deadlock/corruption in production
- Acknowledged in inline comment; stub-only in validation vehicle, CRITICAL in production

### VDRAM-04 — tier_migration.c line 228 — lkey/rkey CONFUSION
- `sge.lkey = pm->rkey;` — assigns remote key to local key field
- RDMA NIC rejects work request with WC_LOC_PROT_ERR; all T3 migrations fail
- Fix: sge.lkey = ctx->mr->lkey (the local access key from the source MR registration)

### VDRAM-05 — pf_handler.c lines 427–436 — NODE DESTROY RACES WITH FAULTS
- vdram_node_destroy() calls vdram_mr_free() while page fault handler may hold mr->ref > 1
- vdram_mr_free WARN_ON leaks MR; subsequent access via dangling owner_node = UAF

## MEDIUM Defects

### JNL-04 — journal.c lines 277-320
- Forward scan starts at write_pos which is mid-entry after wrap
- Random binary data can pass weak header validation (lvl <= 4, mlen <= 255)
- Spurious entries printed; fix: add 4-byte magic sentinel to journal_entry_t

### VDRAM-06 — mr_registry.c lines 208–219
- rkey_rotate nulls old slot, synchronize_rcu() (~10-50ms), then installs new slot
- Window where all rkey lookups return NULL → RDMA requests fail with REM_ACCESS_ERR

### VDRAM-07 — mr_registry.c lines 238–250
- vdram_mr_lookup_rkey() increments ref but no vdram_mr_put() exists anywhere
- Callers that look up and later free MR will leak it (vdram_mr_free asserts ref==1)

### VDRAM-08 — pf_handler.c lines 36–52
- pf_find_mr holds mr_lock across O(n) list scan
- Scalability bottleneck at high MR count; production fix: RCU interval tree

### SVC-02 — service.c lines 565–576
- Cycle fallback uses declaration order (arbitrary); returns -1 which caller should honor

## LOW Defects
