### OPEN — REMAINING ISSUES (post-P0–P3)

#### HIGH — IDB Singleton Has No Version Migration Path
DB_VERSION = 1. onupgradeneeded only handles first-time creation (no version guards).
Schema change to add indexes or columns will corrupt existing user data.
Risk: HIGH. No DB_VERSION bump logic means adding indexes silently fails on existing DBs.

#### HIGH — IDB Has No Secondary Indexes
createObjectStore uses only keyPath "id". Filter at main.tsx:183 is a full in-memory scan.
tags has no index. updatedAt has no index. Search is O(n) over all notes in memory.
Scalability limit: ~5,000 notes before noticeable lag (browser benchmark baseline).

#### HIGH — Calendar is a Single JSON Blob (Scalability Cliff)
go-calendar/electron/preload.ts:45-54: every sync call serializes the ENTIRE events array
into OrangeDocument.body. The body field is a string; sync service stores it opaque.
Scalability limit: 1 MiB cap on payload (MAX_PAYLOAD_LEN guard). At ~2 KB/event average
that is ~500 events before hard rejection. At ~500 bytes/event (minimal) it is ~2,000 events.
No incremental sync. No per-event CRUD. Concurrent save from two sessions = last-write-wins
with total data loss for the losing writer.

#### HIGH — No Conflict Resolution Anywhere in IDB<->SyncClient Flow
IDB is write-authoritative locally. SyncClient.saveDoc() is last-write-wins remotely.
No vector clock, no CRDT, no optimistic lock (updatedAt is written at save time so two
clients writing at the same millisecond both succeed with different state).
Impact: silent data loss on concurrent multi-device editing.

#### MEDIUM — SyncClient.saveDoc / getDoc / list Have No Timeout
AI action calls in preload have 10s AbortController timeouts.
But sync.saveDoc(), sync.getDoc(), sync.list(), sync.upload() have NO AbortSignal.
A hung sync endpoint will hang the preload process indefinitely (no circuit breaker).

#### MEDIUM — go-drive download/remove Bypass SyncClient
download() and remove() use raw fetch() with manually assembled Authorization header.
Token rotation / refresh logic only needs to be patched in SyncClient but silently
misses these two operations. Tracked from P1 audit; partially addressed (upload routed
through SyncClient) but download/remove remain split.

#### MEDIUM — Yjs Is Installed But Not Used in Any Renderer
yjs@^13.6.0 is in go-notes/package.json dependencies.
doc-model.ts uses Y.Doc, createDocModel(), toOrangeDocument() — but go-notes/src/main.tsx
does NOT import doc-model.ts and does NOT use Y.Doc anywhere.
The Note type (main.tsx:17) is a plain object, not a Y.Doc.
Yjs CRDT infrastructure exists in common/ but no renderer actually instantiates it.
This means the CRDT conflict-resolution capability that Yjs provides is entirely unused.
It is dead weight in the bundle (~130 kB gzipped).

#### MEDIUM — 500ms Debounce Data Loss Window
If the Electron process is killed (OOM kill, SIGKILL, power loss) during the 500ms debounce
window, the in-flight edit in pendingSaveRef is lost.
The unmount flush (main.tsx:113-118) covers React unmount, not process kill.
Loss window: up to 500ms of keystrokes. On a 120 WPM typist that is ~10 characters.
Acceptable for a local-first app but must be documented as a known trade-off.

#### MEDIUM — upload() Content-Type Mismatch in go-drive
go-drive/electron/preload.ts:66: Blob created with type "application/octet-stream".
SyncClient.upload() appends this Blob to FormData without overriding the MIME type.
The browser will use the Blob's type as the part Content-Type in the multipart form.
For text files (the only content the UI supports: textarea input) the correct type is
"text/plain; charset=utf-8". Using octet-stream means the sync server will treat all
text uploads as binary. Downstream text search/indexing will fail.
Fix: derive MIME from extension using the same IMAGE_MIME allowlist pattern in go-drive/src/main.tsx.

#### LOW — TOKEN Empty-String Guard Missing in Preload
GENOS_SESSION_TOKEN defaults to "". An empty Bearer token is sent on every request.
Sync service must reject empty tokens — preload should guard before attempting sync.

#### LOW — No Data Retention / Export-All Policy
IndexedDB persists until storage pressure eviction or manual clear. No TTL, no archive.
No user-facing export-all function exists.

#### LOW — go-calendar Has No Local Persistence (IDB)
go-notes uses IDB as a local Bronze layer with SyncClient as remote Silver.
go-calendar has NO IDB layer — it reads entirely from SyncClient on mount.
If the sync service is unreachable at startup, the calendar is blank.
Fix: add an IDB Bronze layer for calendar events, same pattern as go-notes.

## Architecture Decisions Recorded
- ELT decision: local IDB is ETL (renderer transforms before put); sync service is ELT
- go-notes IDB is Bronze tier (raw, append/upsert by id)
- go-calendar sync doc is Silver tier (but stored as opaque blob — not truly Silver)
- SyncClient /docs/ is Silver tier (validated OrangeDocument schema)
- Gold tier: not yet implemented — needed for search indexes, tag aggregations, calendar views
- Yjs Y.Doc infrastructure in doc-model.ts is built but completely unused in renderers
- DB_VERSION = 1, no migration path — schema evolution risk remains HIGH
