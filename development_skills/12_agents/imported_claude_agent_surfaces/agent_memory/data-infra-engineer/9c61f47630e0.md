### HIGH-001 — provenance/app.py: no connection pool configured
File: platform/services/provenance/app.py, line 94
```

# engine_kwargs only has pool_pre_ping=True for PostgreSQL
```

### HIGH-002 — provenance/app.py: _get_last_audit_token ORDER BY record_time with no covering index on (record_time, audit_token)
File: platform/services/provenance/app.py, lines 266-271
```python
row = db.execute(
    select(ProvenanceRow.audit_token)
    .order_by(desc(ProvenanceRow.record_time))
    .limit(1)
).scalar_one_or_none()
```
The ORM declares index=True on record_time (line 137), but that index covers
only record_time. The query SELECT audit_token ORDER BY record_time DESC LIMIT 1
must do a heap fetch for the audit_token column on every write. On PostgreSQL
this is a two-page lookup rather than an index-only scan. A covering index
(record_time DESC, audit_token) would make this index-only.
Also: this query is inside _chain_lock and executed on every POST /log —
it is the gating latency of the write path.
Fix: add to provenance schema.sql:
  CREATE INDEX IF NOT EXISTS idx_provenance_chain_tip
      ON provenance (record_time DESC)
      INCLUDE (audit_token);

### HIGH-003 — provenance/app.py: GET /provenance fetches all rows unbounded
File: platform/services/provenance/app.py, lines 371-374
```python
rows = db.execute(
    select(ProvenanceRow).order_by(ProvenanceRow.record_time)
).scalars().all()
return {"items": [_row_to_dict(r) for r in rows]}
```
No LIMIT, no cursor pagination. As the audit log grows this query will:
- Return all rows to Python (memory explosion)
- Serialize every row to dict in a single response body
- Block the DB connection for the full scan duration
For a security audit log this table can grow to millions of rows.
Fix: add limit/offset or cursor (after_id) query params; default page_size=500.

### HIGH-005 — provenance_client.py: fsync on every fallback write
File: tools/provenance_client.py, lines 202-204
```python
f.write(line)
f.flush()
os.fsync(f.fileno())
```
Each fsync() forces a full journal commit to NVMe. On HP EliteBook NVMe,
fsync latency is ~200-500 µs. At 100 events/sec = 20-50 ms/sec of pure
fsync stall, saturating one CPU core's wait time. On battery this is
especially expensive (prevents C-state deep sleep between writes).
Batching strategy: collect writes in a deque; fsync only when:
  a) batch size >= N (e.g. 50 entries), OR
  b) time since last fsync >= T seconds (e.g. 2s), OR
  c) _rotate_if_needed_locked() triggers.
Risk: up to N-1 events lost on crash. Acceptable for a fallback path that
already tolerates service unavailability. The primary path (POST /log to
provenance service) is synchronous HTTP, so durability is the service's job.

### HIGH-006 — bridge daemons: unconditional fsync + atomic write on every poll tick
Files:
  tools/bridges/entropy_bridge.py, lines 178-179 (poll every 2s)
  tools/bridges/vdram_bridge.py, lines 200-201 (poll every 2s)
  tools/bridges/mesh_bridge.py, lines 179-180 (poll every 5s)

All three daemons call os.fsync() on the tmpfile before os.replace() on
every poll cycle, even when the data has not changed since the last write.
On a 2-second poll cycle: entropy_bridge = 30 fsyncs/min; vdram_bridge =
30 fsyncs/min; mesh_bridge = 12 fsyncs/min. Total: 72 fsync syscalls/minute
writing to /run/genos/ which is typically tmpfs (RAM-backed). On tmpfs,
fsync() is a no-op at the VFS layer — it is NOT expensive. However if
/run/genos/ is bind-mounted to a real filesystem (or the OS changes), the
cost becomes real. Additionally, the unconditional write wastes CPU cycles
parsing and serialising data that has not changed.
Fix: add a content-hash comparison before writing. Store previous_json_hash
in the daemon's loop state; skip _atomic_write if hash unchanged.
Impact: eliminates ~70% of writes in steady state (most telemetry is stable).

### MEDIUM-007 — identity/app.py: _login_attempts dict is unbounded (no TTL eviction)
File: platform/services/identity/app.py, lines 56-101
The _login_attempts dict accumulates one entry per unique username attempted.
There is no periodic eviction of stale entries. A brute-force attempt against
10,000 different usernames creates 10,000 dict entries that are never cleaned.
On a long-running pod this is a slow memory leak.
_record_failed_attempt() prunes timestamps within the observation window but
does NOT remove the dict key when attempts list becomes empty after window
expiry.
Fix: in _record_failed_attempt(), after pruning:
  if not attempts and float(record.get("locked_until", 0.0)) <= now:
      del _login_attempts[username]

### MEDIUM-008 — identity/app.py: expired sessions not purged (table grows unbounded)
File: platform/services/identity/schema.sql, line 117-121
The schema comment says "Maintenance: expire old sessions. Scheduled via
pg_cron or migrate.py cleanup task." No such cleanup task exists anywhere
in migrate.py or the codebase. The identity_sessions table will grow by one
row per login, forever. At 100 logins/day = 36,500 rows/year. The
token_hash lookup (index on token_hash) stays fast, but the table wastes
storage and backup size.
Fix: add a DELETE FROM identity_sessions WHERE expires_at < NOW() - INTERVAL '30 days'
step to a periodic maintenance job, or implement a cleanup endpoint in migrate.py --cleanup.

### MEDIUM-009 — identity/app.py: create_user makes two sequential SELECT queries where one would do
File: platform/services/identity/app.py, lines 409-413
```python
existing_username = db.execute(select(IdentityUser).where(IdentityUser.username == req.username)).scalar_one_or_none()
if existing_username: raise ...
existing_email = db.execute(select(IdentityUser).where(IdentityUser.email == req.email)).scalar_one_or_none()
if existing_email: raise ...
```
Two separate round-trips to the DB. A single query with OR could replace them,
but the real fix is to just attempt the INSERT and catch IntegrityError
(which is already done on line 421). The pre-flight checks are redundant and
add 2x DB round-trips on the happy path.
Fix: remove the two pre-flight SELECTs; rely solely on the IntegrityError
catch at line 421 to distinguish username vs. email conflict (inspect
exc.orig.pgcode or the constraint name in exc.orig.diag.constraint_name).

### MEDIUM-010 — identity/app.py: bcrypt work factor not configurable
File: platform/services/identity/app.py, line 416
```python
hashed = bcrypt.hashpw(req.password.encode("utf-8"), bcrypt.gensalt()).decode("utf-8")
```
bcrypt.gensalt() defaults to rounds=12. This is correct but not tunable.
On HP EliteBook (4-core i7) bcrypt at rounds=12 takes ~250ms. This blocks
the FastAPI worker thread for 250ms on every registration and login.
FastAPI runs sync route handlers in a thread pool (anyio to_thread), so
this does not block the event loop, but it does saturate the thread pool
under concurrent login load.
Fix: expose IDENTITY_BCRYPT_ROUNDS env var (default 12); at high concurrency
consider offloading to a dedicated process pool.

### MEDIUM-011 — sync/app.py: upload_file assembles full file in RAM before MinIO write
File: platform/services/sync/app.py, lines 149-164
The comment (ADV-001 from wave 4 audit) notes this is acceptable for 100 MB.
However, the chunks list grows to MAX_UPLOAD_BYTES / READ_CHUNK_SIZE = 1,562
chunks before b"".join(). Peak memory per upload = 2x file size (chunks list
+ joined bytes). Under concurrent uploads (k8s pod with 512 MB limit) this
is a real risk.
Reassessed severity: MEDIUM (not just advisory) because the 100 MB cap +
k8s memory limit means a 5-concurrent-upload scenario hits 1 GB RAM.
Fix: use MinIO put_object with a streaming source. MinIO client supports
length-aware streaming. Read the file size from Content-Length or just use
a chunked writer.

### MEDIUM-012 — sync/app.py: MinIO connection object is module-level singleton with no reconnect logic
File: platform/services/sync/app.py, lines 74-79
```python
client = Minio(MINIO_ENDPOINT, ...)
```
The Minio client is a module-level singleton. If MinIO restarts, the client
object holds stale TCP connections. The minio-py library does handle
reconnection internally per-request (each operation opens a new HTTP
connection via urllib3), but there is no health-check retry on the lifespan
ensure_bucket() call. If MinIO is slow at startup, ensure_bucket() fails and
the pod crashes.
Fix: wrap ensure_bucket() in a retry loop identical to the DB retry pattern
used in identity and provenance (with exponential backoff up to max_retries).

### MEDIUM-013 — provenance/app.py: GET /provenance and temporal query load entire result set into Python list before serializing
File: platform/services/provenance/app.py, lines 371-374 and 406-412
Both `.scalars().all()` calls materialize the full result set into a Python
list before the list comprehension runs. For large audit windows this is
O(N) RAM. SQLAlchemy supports yield_per() for server-side cursors.
Fix: use .scalars().yield_per(500) and stream the JSON response using
FastAPI StreamingResponse with a generator.

### LOW-014 — bridge daemons: mkdir() called on every _atomic_write()
Files:
  tools/bridges/entropy_bridge.py line 173
  tools/bridges/mesh_bridge.py line 174
  tools/bridges/vdram_bridge.py line 195
```python
path.parent.mkdir(parents=True, exist_ok=True)
```
This syscall is made on every poll cycle (every 2-5 seconds). On Linux,
mkdir with exist_ok=True calls stat() then mkdir() — two syscalls per write
even when the directory already exists. This is the same bug fixed in
thermal_predictor.py and thermal_guardian.py (wave 1B audit).
The directory is /run/genos/ which is created at daemon startup and never
removed. Calling mkdir() 30x/min is pure waste.
Fix: move path.parent.mkdir() to a one-time call at module import time
(or in run_loop() before the while loop), identical to the thermal daemon fix.

### LOW-016 — identity/app.py: healthz opens a new DB connection per request via Depends(get_db)
File: platform/services/identity/app.py, lines 399-404
The /healthz endpoint acquires a session from the pool, executes SELECT 1,
and closes it. Under a high-frequency k8s liveness probe (default every 10s)
this is fine. But if the probe interval is reduced to 1s (aggressive), it
generates 60 pool checkouts/minute for zero-value health data. Provenance
/healthz (lines 303-311) has the same pattern but uses engine.connect()
directly (bypasses the pool — actually worse, it creates a raw connection
each time).
Fix for provenance/healthz: use SessionLocal() instead of engine.connect()
to stay within the pool.
