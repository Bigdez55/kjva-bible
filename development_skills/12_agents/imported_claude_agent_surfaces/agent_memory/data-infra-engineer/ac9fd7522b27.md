### 1.2 Sync Service

**Files compared:**
- Data models: `platform/services/sync/app.py` (Pydantic, no ORM — MinIO-only)
- DDL: `platform/services/sync/schema.sql`

The sync service has NO SQLAlchemy ORM. It uses Pydantic models (`SaveDocRequest`) for request validation only. All persistence goes to MinIO (object storage). The schema.sql represents the **target future state** — a PostgreSQL-backed metadata store that has not yet been implemented in the application layer.

**Current architecture gap:**
- `sync_entries` and `sync_conflicts` tables will be created by migrate.py.
- The sync service app.py will never write to them — all data goes to MinIO.
- `sync_entries.payload` is JSONB in the schema, but the current MinIO path stores the full Pydantic JSON directly in the object body.
- There is NO synchronization between the MinIO state and the future PostgreSQL state. A migration sprint is required to backfill from MinIO to PostgreSQL before the app layer switches.

**Route shadow bug (CRITICAL):**

```
# sync/app.py line 175
@app.get("/docs/{doc_id}")          # registered FIRST in source order

# sync/app.py line 185
@app.get("/docs/list")              # registered SECOND — DEAD ROUTE
```

FastAPI (Starlette router) matches routes in registration order. `/docs/list` is a string that satisfies the pattern `/docs/{doc_id}` where doc_id = "list". Therefore `GET /docs/list` will ALWAYS be routed to `get_doc()` with `doc_id="list"`, and will return 404 because no document with id "list" exists in MinIO. The `list_docs()` function is unreachable.

**Confirmed impact:** Any client calling `GET /docs/list` receives a 404. The go-notes Electron preload calls `SyncClient.getDoc("list")` or equivalent — this call path is silently broken.

### 1.3 Provenance Service

**Files compared:**
- ORM: `platform/services/provenance/app.py` (SQLAlchemy ProvenanceRow)
- Inline SQL: `platform/init/genos_platform_init.py` run_migrations()

The provenance service has NO separate schema.sql file. The ORM is the DDL authority for the provenance table.

| Column | ORM (app.py) | genos_platform_init.py | Verdict |
|--------|-------------|------------------------|---------|
| action_id | String(36) PK | UUID PK | TYPE MISMATCH |
| user_id | Text NOT NULL | TEXT NOT NULL | ALIGNED |
| action_time | DateTime(tz=True) NOT NULL | TIMESTAMPTZ NOT NULL | ALIGNED |
| record_time | DateTime(tz=True) NOT NULL | TIMESTAMPTZ NOT NULL | ALIGNED |
| action_type | Text NOT NULL | TEXT NOT NULL | ALIGNED |
| target | Text NOT NULL | TEXT NOT NULL | ALIGNED |
| details | Text NOT NULL | TEXT NOT NULL | ALIGNED |
| reversible | Boolean NOT NULL | BOOLEAN NOT NULL | ALIGNED |
| audit_token | Text NOT NULL | TEXT NOT NULL | ALIGNED |

**UUID vs String(36) type mismatch:** genos_platform_init.py creates `action_id UUID PRIMARY KEY` — a native PostgreSQL UUID column. The ORM maps `action_id` as `String(36)` and inserts Python `str(uuid.uuid4())`. PostgreSQL will silently cast the string to UUID on INSERT. On SELECT, psycopg3 will return a Python uuid.UUID object, which SQLAlchemy String mapping will call `str()` on. The round-trip works but is technically incorrect typing — any tool that inspects the column type will see UUID, not VARCHAR.

**Ordering issue:** The inline SQL creates the provenance table in the same transaction as identity and session tables. If the ORM's `create_all` runs (on service startup) BEFORE genos_platform_init.py runs the inline SQL, the provenance table will be created with String(36) type for action_id. If genos_platform_init.py runs first, the column is UUID. The behavior depends on which process wins the startup race.

### 1.4 Migration Runner Analysis

**File:** `platform/services/migrate.py`

| Safety Property | Status | Detail |
|----------------|--------|--------|
| Advisory lock | CONFIRMED | pg_advisory_lock(0x4745_4E4F) — session-level, auto-released on close |
| Idempotency | CONFIRMED | IF NOT EXISTS in all DDL; checksum-based skip; ON CONFLICT DO NOTHING in tracking INSERT |
| Rollback capability | CONFIRMED (dev only) | --rollback + GENOS_ALLOW_ROLLBACK=true; CASCADE drop in reverse dependency order |
| Schema files referenced | identity/schema.sql, sync/schema.sql, gate/schema.sql | provenance has NO schema.sql — not in MIGRATION_SEQUENCE |
| Transaction safety | CONFIRMED | Each file in one transaction; rollback on error |
| Retry on connect | CONFIRMED | 20 retries, 3s initial delay, 1.5x backoff, cap 30s |
| Tracking table | CONFIRMED | schema_migrations (Bronze tier, append-only) |
| Dry-run mode | CONFIRMED | --dry-run prints checksums without DB touch |
| Provenance table | NOT COVERED | migrate.py does not include provenance — only genos_platform_init inline SQL creates it |

**Migration source conflict:**

Two systems modify the PostgreSQL schema:
1. `genos_platform_init.py::run_migrations()` — runs via `kubectl exec statefulset/postgres -- psql` at platform init time (line 201). Creates identity_users, identity_permissions, identity_sessions, provenance in ONE inline SQL block.
2. `migrate.py` — runs as a k8s Job (00_migrations.yaml) applying schema.sql files for identity, sync, and gate.

**Execution order in main():** genos_platform_init.py runs `apply_manifest("01_postgres.yaml")` then `wait_for_rollout("statefulset/postgres")` then `run_migrations()` — all BEFORE any service Deployments are applied. The migration Job (00_migrations.yaml) is also applied before service Deployments. The race between psql inline execution and the k8s Job depends on Job scheduling latency.

**Consequence:** If inline SQL wins: identity tables exist with 6 columns (no display_name, is_active, created_at, updated_at). When migrate.py runs, `IF NOT EXISTS` skips table creation but does NOT add missing columns. The 4 schema.sql columns are never created. If migrate.py wins: tables have 10 columns; inline SQL's `CREATE TABLE IF NOT EXISTS` is a no-op. The first scenario is the more dangerous path.

## SECTION 2 — TELEMETRY PIPELINE AUDIT

### 2.1 signal_baseline.py — Mathematical Correctness

**Algorithm:** Tolerance band = `warning_tolerance * (expected_max - expected_min)`

```
NORMAL:   expected_min <= value <= expected_max
WARNING:  deviation from nearest bound <= tolerance
CRITICAL: deviation > tolerance
```

**Edge case analysis:**
- `expected_min == expected_max` (zero span): tolerance = 0.0. Any value != expected_min is immediately CRITICAL. This is correct behavior for a single-point baseline. No crash.
- Negative values: handled correctly (wifi dBm uses negative range [-70, -30]).
- `warning_tolerance = 0.0`: tolerance = 0.0, meaning any out-of-band value is immediately CRITICAL. Correct behavior.
- `warning_tolerance > 1.0`: mathematically valid — the tolerance band could be larger than the original range. No guard exists against this; user-defined baselines could produce unexpected WARNING ranges.

**Verdict: PASS** — no mathematical errors. One recommendation: add validation in `SignalBaseline.__post_init__` that `0 <= warning_tolerance <= 1.0`.

**Data boundary concern:** `check()` takes a raw `float`. There is no upper or lower cap on input values. A sensor feeding `float('inf')` or `float('nan')` would produce a nonsensical `deviation` value. Python's `math` operations on infinity produce infinity (which is > tolerance, yielding CRITICAL — safe). NaN comparisons (`NaN < x`) always return False — a NaN input would match `expected_min <= NaN <= expected_max` as False, then the direction check `NaN < expected_min` as False, so it would execute the "above" branch with `deviation = NaN - expected_max = NaN`. `NaN <= tolerance` is False, so it would return CRITICAL. NaN inputs produce CRITICAL status — arguably correct but silently wrong.

### 2.2 signal_scorer.py — Mathematical Correctness

**Weights:** FREQUENCY(0.20) + MAGNITUDE(0.35) + DURATION(0.20) + PROPAGATION(0.25) = **1.00** — confirmed.

**Normalization functions:**
- `_normalize_frequency(f)` = `1 - 1/(1 + 0.3f)` — sigmoid-like, range (0, 1). At f=1: 0.231. At f=10: 0.750. At f=∞: 1.0. Saturates gracefully. Correct.
- `_normalize_magnitude(m)` = clamp(m, 0, 1) — correct.
- `_normalize_duration(d)` — returns 0.05 floor for d<=0 (instantaneous events), min(1, d/300) otherwise. Correct. Floor prevents zero-duration events from contributing nothing to score.
- `_normalize_propagation(layers)` = `(clamped - 1) / 4` for clamped in [1,5]. At 1 layer: 0.0. At 5 layers: 1.0. Correct.

**Decay formula:** `e^(-0.1 * age_hours)`. Half-life = ln(2)/0.1 = 6.93 hours. Confirmed correct.

**Edge case:** `compute_decayed_score(score, query_time=now - 3600)` — past query time. Code uses `max(0.0, age_hours)` which clamps to 0.0, yielding `e^0 = 1.0`. Correct — no negative decay.

**Verdict: PASS** — no mathematical errors.

### 2.3 tei_calculator.py — Mathematical Correctness

**Weights:** 0.25+0.20+0.15+0.25+0.15 = **1.00** — confirmed.

**Weighted harmonic mean formula:**
```
H = sum(w_i) / sum(w_i / s_i)
```
With all weights summing to 1.0: `H = 1.0 / sum(w_i / s_i)`. Scores floored to 0.01. Division-by-zero protected. Correct.

**Why harmonic mean is correct here:** The harmonic mean is pulled toward the minimum — a single metric at 0.01 with weight 0.25 contributes `0.25/0.01 = 25` to the denominator, dominating the sum. This correctly makes a critical bottleneck collapse the TEI score.

**_normalize_metric edge case:** `span <= 0` returns 2.5 (midpoint). Prevents division by zero for degenerate ranges. Correct.

**_normalize_inverse_metric:** Lower is better. At `value <= ideal_min`: returns 5.0 (perfect). At `value = ideal_max`: returns 2.5. Above max: linear decay to 0.0 at `ideal_max + 2*span`. Correct.

**Grade thresholds:** CRITICAL < 1.0, DEGRADED [1.0, 2.0), NOMINAL [2.0, 3.5), OPTIMAL >= 3.5. Consistent with docstring. Correct.

**Verdict: PASS** — no mathematical errors.

### 2.4 causal_triples.py — Pattern Correctness

**Pattern matching:** `re.IGNORECASE` on all patterns. First-match-wins per line. Empty lines skipped. Correct.

**Confidence validation:** `__post_init__` raises `ValueError` for confidence outside [0.0, 1.0]. Correct.

**Journal entry handling:** `entry.get("MESSAGE", "")` — safe for missing key. `entry.get("_SYSTEMD_UNIT", "")` — safe. Empty message skipped. Correct.

**Pattern ordering concern:** The OOM pattern `r"out\s+of\s+memory|oom.killer|oom_kill"` uses `oom.` where `.` is a regex wildcard (matches any character). This will match `oomXkiller` etc. — false positive potential but low risk in real kernel logs.

**Verdict: PASS** — no correctness errors. One cosmetic fix: `oom.killer` should be `oom\.killer` for strict matching.

### 2.5 variance_decomp.py — Mathematical Correctness

**SS_regression formula:**
```
SS_reg = (cov(factor, measurements))^2 / var(factor) * (n-1)
```

This is the Ordinary Least Squares formula for the explained sum of squares from a simple linear regression. The derivation: for y = a*x + b + e, the OLS R-squared is `cov(x,y)^2 / (var(x) * var(y))`. Multiplying by `SS_total = var(y) * (n-1)` gives `cov(x,y)^2 / var(x) * (n-1)`. **Confirmed correct.**

**Multicollinearity handling:** If sum of contributions > 1.0 (possible when factors are correlated), normalize proportionally and set residual = 0.0. This is a valid first-order correction — not a full partial R-squared decomposition but appropriate for operational use. Warning is logged.

**Zero-variance edge case:** When `ss_total < 1e-15` (all measurements identical), returns `systematic_r2=0.0` and `system_state="ANALYZABLE"`. Debatable: if the system produces perfectly constant measurements, it could be argued as UNPREDICTABLE (no signal). However, returning ANALYZABLE for zero-variance is mathematically consistent — R-squared is undefined (0/0) and setting it to 0 is a conservative choice.

**CI approximation:** `se = 2.0 * (1 - r2) * sqrt(k / dof) / sqrt(n)`. This is a heuristic, not the standard Fisher z-transform CI for R-squared. The standard formula uses `se(R^2) ≈ 2*R*(1-R^2)/sqrt(n-1)`. The implementation's formula grows with k (number of factors) and shrinks with n — it is operationally reasonable for the intended use case but should not be cited as a formal statistical CI.

**Verdict: PASS with 1 note** — heuristic CI is acceptable for operational dashboards but must not be presented as a formal 95% CI to statistical consumers.

## SECTION 3 — DATA FLOW TRACING

### 3.1 Authentication Flow (User → Identity → PostgreSQL)

```
Client
  |
  |-- POST /auth/login {username, password}
  |
Identity Service (identity/app.py)
  |-- select(IdentityUser).where(username=req.username)  [O(log n) via unique index]
  |-- bcrypt.checkpw(password, hash)                     [compute-bound, ~100ms]
  |-- encode_token(uid, "access", 30min)                 [Ed25519 sign]
  |-- encode_token(uid, "refresh", 7d)                   [Ed25519 sign]
  |-- INSERT IdentitySession(token_hash, token_jti, uid, username, expires_at)
  |-- db.commit()
  |
  |<-- TokenResponse {access_token, refresh_token, expires_in}

Client stores access_token (30min TTL) and refresh_token (7d TTL)
```

**Data integrity concerns:**
- Login does NOT check `valid_to` on the IdentityUser — a future-dated or soft-deleted user (via the SCD Type 2 pattern in schema.sql) would still authenticate. The ORM's `is_active` column doesn't exist, so there is no active-user filter.
- No rate limiting on /auth/login — brute force protection absent at the service layer.

### 3.2 Document Persistence Flow (Sync → MinIO)

```
Client (go-notes Electron)
  |
  |-- POST /docs/save {id?, title, body, tags}  [Bearer: access_token]
  |
Sync Service (sync/app.py)
  |-- decode_bearer()                            [JWT validate]
  |-- uid = claims["sub"]
  |-- doc_id = req.id or uuid4()
  |-- payload = req.model_dump_json().encode()   [Pydantic serialize]
  |-- put_object(DOC_BUCKET, "docs/{uid}/{doc_id}.json", payload)
  |
  |<-- SaveDocResponse {id, status: "saved"}

GET /docs/{doc_id}:
  |-- get_object(DOC_BUCKET, "docs/{uid}/{doc_id}.json")
  |-- model_validate_json(obj.read())

GET /docs/list:  [DEAD ROUTE — shadowed by /docs/{doc_id}]
```

**Broken flow:** GET /docs/list is unreachable. See Section 1.2.

**Missing flow:** No version control on documents. If two clients save the same doc_id simultaneously, last-write-wins with no conflict record. The sync_conflicts table in schema.sql is never populated.

### 3.3 Audit Flow (Tool call → Provenance → PostgreSQL)

```
Any service with bearer token
  |
  |-- POST /log {user_id, action_type, target, details, reversible, action_time?}
  |
Provenance Service (provenance/app.py)
  |-- decode_bearer()
  |-- with _chain_lock:                          [threading.Lock — single-process serialization]
  |     |-- _get_last_audit_token(db)            [SELECT audit_token ORDER BY record_time DESC LIMIT 1]
  |     |-- build_audit_token(previous, item, ...)  [SHA-256 chain]
  |     |-- INSERT ProvenanceRow(...)
  |     |-- db.commit()
  |
  |<-- {status: "logged", action_id, audit_token}
```

**Concurrency concern:** `_chain_lock` is a `threading.Lock()` — process-local. If the provenance service ever scales to multiple replicas, the chain lock provides no cross-replica serialization. Two pods could compute audit tokens against the same "previous" record, creating a fork in the chain. The current k8s deployment (single replica Deployment) prevents this, but there is no guardrail preventing a replica count of > 1.

**Unbounded list:** `GET /provenance` returns ALL records with no pagination. At high audit volume this becomes a full table scan + unbounded JSON response. Recommend adding LIMIT/OFFSET or cursor-based pagination.

### 3.4 System Event → Telemetry → Signal Scoring

```
Kernel / systemd
  |-- /sys/class/thermal/thermal_zone0/temp  (procfs/sysfs)
  |-- /proc/meminfo                           (procfs)
  |-- dmesg output                            (kernel ring buffer)
  |-- journalctl -o json                      (systemd journal)
  |
Telemetry Layer (platform/services/telemetry/)
  |-- BaselineRegistry.check_signal(metric, value)   [signal_baseline.py]
  |-- score_event(event_type, ...)                   [signal_scorer.py]
  |-- ThroughputEfficiencyIndex.calculate(metrics)   [tei_calculator.py]
  |-- extract_from_dmesg(lines)                      [causal_triples.py]
  |-- decompose_boot_time(measurements, factors)     [variance_decomp.py]
  |
  |-- [IN MEMORY ONLY — no persistence layer]
```

**Critical gap:** The entire telemetry pipeline computes in memory. There is NO sink — no SQLite store, no PostgreSQL table, no time-series database. Telemetry results are computed and discarded. The signal scoring, TEI values, causal triples, and variance decompositions are not persisted anywhere.

**What is needed:** A Bronze-tier append-only telemetry store. Recommended: SQLite on local disk (on-device, no network dependency) with a schema like:

```sql
CREATE TABLE IF NOT EXISTS telemetry_events (
    id          TEXT PRIMARY KEY,          -- UUID
    collected_at REAL NOT NULL,            -- Unix epoch (time.time())
    metric_name  TEXT NOT NULL,
    raw_value    REAL,
    signal_status TEXT,                    -- NORMAL/WARNING/CRITICAL
    tei_score    REAL,
    event_type   TEXT,
    causal_cause TEXT,
    causal_effect TEXT,
    r2_systematic REAL,
    system_state TEXT                      -- ANALYZABLE/UNPREDICTABLE_CHAOS
);
CREATE INDEX IF NOT EXISTS idx_telemetry_collected_at ON telemetry_events(collected_at DESC);
```
