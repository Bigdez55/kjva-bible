# Sprint 33 Data Layer E2E Audit
**HEAD:** 279b28f | **Date:** 2026-03-18 | **Auditor:** data-infra-engineer

## XSTORE B+Tree — Verified Parameters
- Page 4096B, leaf order 256, internal order 128
- Superblock page 0, root initialized on page 1
- Variable-length record layout: records packed from byte 4 (XSTORE_LEAF_CHAIN_HDR), offset table at page tail
- 8-byte `xstore_leaf_slot_t` _Static_assert verified
- Leaf sibling chain wired at split (btree.c:401-408) — enables O(1) cursor advance
- Internal node cascading split DEFERRED (returns XSTORE_ERR_CAPACITY at internal node full) — Sprint 5 item
- Merge/underflow: lazy deletion only, no merge — documented in btree.c:40
- UPSERT gap (F-BT-01 P2): same-key value-size change returns ERR_CONFLICT (-7) instead of delete-then-reinsert; `xstore_put()` convenience wrapper does NOT perform this fallback

## XSTORE WAL — Verified
- Ring 65536 × 4384B entries (~287 MB buffer) — caller-provided
- Entry CRC32C covers bytes [16, sizeof) — skips lsn+txn_id so LSN can be assigned without recompute
- Durability: `store->sync()` called INSIDE `xstore_txn_commit()` after COMMIT record, before returning XSTORE_OK (mvcc.c:262-268) — correct
- Checkpoint order: flush dirty pages → append CHECKPOINT → write superblock → sync() → advance tail
- WAL_REPLAY_MAX_TXNS = 256; overflow returns XSTORE_ERR_REPLAY_OVERFLOW (-15) with console message (P2-S20-WAL-01 fixed)
- checkpoint LSN race: documented in wal.c:199-229, correctly resolved by pal_atomic_add pre-add semantics — NOT a defect

## XSTORE MVCC — Verified
- Isolation: Snapshot Isolation (write-skew acknowledged + accepted for single-device)
- Snapshot: `store->next_lsn` captured at `xstore_txn_begin()` (mvcc.c:142)
- OCC conflict check: shadow table g_mvcc_table[8192] (open-addressing, Knuth multiplicative hash)
- Static global MVCC table — must become per-store for multi-store future
- WRITE-SET PAGE_NO BUG (F-MVCC-01 P2): write_set records root_page for ALL writes instead of actual leaf page touched — causes false OCC conflicts under concurrent writers to different leaf pages (mvcc.c:317, mvcc.c:346)
- XSTORE_TXN_MAX_WRITES = 512; XSTORE_MAX_TXN_INFLIGHT = 64

## XSTORE Python FFI — Verified
- Module: `xstore_ffi`, 512 LOC, PEP 489 single-phase init
- Exposed: set, get, delete, begin_txn, commit_txn, abort_txn
- All 15 error codes + MAX_KEY_LEN + MAX_VAL_LEN + MAX_TXN exported as named constants
- Value buffer: 4096B stack (XSTORE_MAX_VAL_LEN) — no heap crossing FFI
- Thread safety: GIL-protected; g_txn_table + g_txn_slot_used protected by GIL
- FFI txn slot full: returns -100 (NOT exported as named constant — F-FFI-XSTORE-01 P3)

## XBLOB Storage — Verified
- SHA-256 via pal_sha256() on put (blob.c:93-94); re-hashed on every get for at-rest integrity (blob.c:231-236)
- Fixed-slot: XBLOB_SLOT_STRIDE = (64+16MB+4095)&~4095 = 16777280B per slot
- 16384 slots (256×64-bit bitmap); max store = ~256 GB
- Index: 1M open-addressing slots, 75% load factor (XBLOB_INDEX_LOAD_MAX), linear probing, tombstone deletion
- GC: ref-counting, GC log ring 16384 entries × 56B = ~896 KB
- GC log overflow (F-BLOB-01 P2): under >16384 put/delete cycles without xblob_gc_scan(), ring overwrites; older entries skipped by seq-mismatch check; ref-counts revert to on-disk index values after next open()
- GC compaction: triggered at tombstone_count > total_used/4; chunked 256-entry rebuild
- Dedup: transparent; dedup_saves counter tracked

## XBLOB Python FFI — Verified
- Module: `xblob_ffi`, 396 LOC, PEP 489 single-phase init
- Exposed: put (returns 32-byte hash), get (None on miss), delete, exists
- All 11 error codes + HASH_LEN + MAX_BLOB_SIZE exported as named constants
- g_get_buf: 16 MB static module-level buffer (python_ffi.c:76); GIL-protected
- Sub-interpreter data race (F-BLOB-02 P3): g_get_buf shared across sub-interpreters under Python 3.12+ no-GIL scenarios
- Hash verification propagated: XBLOB_ERR_CORRUPT → RuntimeError in Python (python_ffi.c:222-224)

## Alert Thresholds to Implement
- wal_entries > 60000 (92% of 65536) → WARN, trigger checkpoint
- cache_evictions rate > 1000/sec → WARN
- txn_aborts / txn_commits > 0.1 → WARN (F-MVCC-01 symptom)
- index_tombstones / index_used_slots > 0.25 → trigger xblob_gc_scan()

- index_used_slots > 786432 (75% of 1M) → WARN

**Why:** F-INT-01 blocks complete PostgreSQL/MinIO decommission. F-BT-01 + F-MVCC-01 are latent correctness risks under concurrent Python service writes. GC log overflow (F-BLOB-01) is a durability risk under XPKG burst-install workloads.
**How to apply:** Prioritize F-INT-01 migration in next sprint. F-BT-01 fix is a 20-line change to xstore_put() convenience wrapper. F-MVCC-01 fix requires btree.c to return the actual dirty leaf page number.
