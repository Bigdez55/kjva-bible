# XSEC Audit Subsystem — Data Infrastructure Assessment
# Date: 2026-03-06

## 1. Ring Buffer Structure (audit.c / xsec.h)

### Capacity
- `XSEC_AUDIT_RING_SIZE = 1024` entries. Confirmed.
- `XSEC_AUDIT_DETAIL_LEN = 64` bytes per detail string.
- Static memory footprint: 1024 x sizeof(xsec_audit_entry_t).
  - Each entry: 8 (timestamp_ns) + 4 (event_type enum) + 4 (module_id enum) + 64 (detail) = 80 bytes.
  - Total ring: ~80 KB in BSS. Plus ctx fields (~24 bytes). Total ~80 KB.

### Entry Structure (xsec_audit_entry_t)
Fields:
  - `timestamp_ns`  : uint64_t — pal_time_now_ns() at event time.
  - `event_type`    : xsec_audit_event_t enum (int32_t).
  - `source_module` : xsec_module_id_t enum (int32_t).
  - `detail`        : char[64] — bounded, null-terminated, audit_copy_detail() enforces the cap.

### Lock-Free vs Spinlock — HYBRID (not fully lock-free)
- xsec_audit_log(): atomics only. `pal_atomic_add` on write_idx, `pal_atomic_load` on total_events,
  `pal_atomic_add` on dropped_events, `pal_atomic_add` on total_events. NO spinlock on the hot path.
  Correct classification: lock-free for the write path.
  HOWEVER: the slot write (entry->timestamp_ns, event_type, source_module, detail) is a
  non-atomic multi-word store. A reader calling xsec_audit_query_recent() concurrently
  under the spinlock may observe a partially written entry if xsec_audit_dump/query_recent
  reads the slot while xsec_audit_log() is mid-write on another CPU.
  This is a DATA RACE on the entry fields — the fix applied in Sprint 3 addressed the
  init-race (S3-DI-L01) but the slot-torn-read race on the entry payload is still open.
- xsec_audit_query_recent(): acquires pal_spinlock. Reads from ring under lock.
- xsec_audit_dump(): acquires pal_spinlock. Reads from ring under lock.
- Classification: log-path is lock-free; read-path is spinlock-serialized.

### Overflow Handling
- Strategy: DROP OLDEST (ring wrap). write_idx monotonically increments; slot = idx % 1024
  overwrites the oldest slot when full.
- Visible events: min(total_events, 1024).
- dropped_events counter: atomically incremented when total_events >= 1024 before a write.
  Exposed via xsec_audit_get_dropped().
- S3-DI-H01 fix confirmed applied: overflow is now detectable by callers.

### Persistence
- RAM ONLY. The ring lives entirely in BSS of the XSEC module. Zero persistence.
- No drain function. No GENSD journal integration. No XSTORE write path.

## 3. Data Flow Analysis — CRITICAL DURABILITY GAP

### Confirmed Full Data Flow:
```
Crypto primitives (aes_gcm, chacha20, x25519, ed25519)
TLS13 handshake state machine (tls13.c — 18 xsec_audit_log calls)
X.509 chain verification (x509.c — 12 xsec_audit_log calls)
XPKG refusal_gate (refusal_gate.c — 2 xsec_audit_log calls, events 10+11)
   |
   v
xsec_audit_log() [lock-free atomic write to in-memory ring]
   |
   v
xsec_g_audit.entries[slot] [BSS ring, 80 KB, RAM-only]
   |
   [MISSING LINK — no drain, no bridge, no IPC to GENSD]
   |
   v
GENSD journal.c [4 MB ring in BSS — NOT connected to xsec]
   |
   v
journal_flush_serial() [prints to pal_console_puts — serial only, no NVMe write]
   |
   [NO XSTORE WRITE PATH EXISTS IN journal.c]
```

### Gap 1: CRITICAL — No drain function from audit ring to GENSD journal
- Zero code exists that reads xsec_audit_query_recent() or iterates xsec_g_audit.entries
  and writes results to journal_write().
- The GENSD journal itself is also RAM-only (BSS ring), so even if a drain existed,
  events would still be in volatile memory.

### Gap 2: CRITICAL — xsec_audit_init() is never called in the boot chain
- boot_chain.c xchain_security() performs a SHA-256 KAT but does NOT call xsec_audit_init().
- gensd_init() in main.c calls journal_init() and supervisor_init() but NOT xsec_audit_init().
- xsec_audit_log() returns XSEC_ERR_STATE until xsec_audit_init() is called.
- Result: ALL audit events emitted during early boot (key generation, handshake, cert verify)
  are SILENTLY DISCARDED until someone explicitly calls xsec_audit_init().

### Gap 3: CRITICAL — No NVMe/XSTORE write path for audit events
- journal.c has no write_page or pal_nvme callback. It is a pure in-memory log.
- XSTORE is listed in GENSD essential_services as "xstore-init" (oneshot), but
  journal.c has no code that calls xstore_txn_put() or xstore_put() for any event type.
- On power loss or kernel panic, ALL security audit events are lost.

### Gap 4: MEDIUM — GENSD journal reader is a stub
- journal_read_recent() at journal.c line 233 explicitly returns nothing:
  "Sprint 1 stub: return nothing (full reader implemented in Sprint 3)"
  This is Sprint 3 GENSD code but the function was never implemented. Still stub.

## 4. Ten Test Type Assessment

### 1. SMOKE — PASS
xsec_audit_init() + xsec_audit_log() call sequence works. Smoke test in test_sprint3.c
lines 179-181 passes. But: the boot chain never calls xsec_audit_init(), so in production
boot the first event will return XSEC_ERR_STATE. Smoke test passes in isolation; production
smoke would fail on first log call.

### 2. FUNCTIONAL — PARTIAL PASS
- 9 of 11 event types are tested (test_sprint3.c line 1543 only covers events 1-9).
- Events XSEC_AUDIT_PKG_GATE_FAIL (10) and XSEC_AUDIT_PKG_GATE_FORCED (11) are NOT in
  the functional test matrix.
- Module ID uniqueness tested for 8 of 9 modules (XSEC_MODULE_XPKG missing from array
  at test line 1527).

### 3. INTEGRATION — FAIL
- No consumer of the audit ring exists outside of xsec_audit_dump() (serial console only).
- GENSD does not read audit events. XSHELL has no CLI to query them.
- refusal_gate.c calls xsec_audit_log() correctly but the events are silently lost
  when xsec_audit_init() has not been called (which is always the case in production boot).

### 4. REGRESSION — PASS
- S3-DI-L01 (init race) fix is in place.
- S3-DI-H01 (silent overflow) fix is in place: dropped_events counter is correct.
- No existing audit events were removed — events 1-9 are still defined and all callers
  in tls13.c / x509.c / crypto modules still use the same constants.

### 5. LOAD — RING SATURATES IN 1 SECOND AT 1000 EVENTS/S
- At 1000 events/s, ring saturates after 1024ms (just over 1 second).
- After saturation, dropped_events increments for every new event.
- The pal_atomic_add is cheap (x86 lock xadd), so the ring can handle well above 1000/s.
- Load test in test_sprint3.c fills RING_SIZE + 100 = 1124 events and verifies the
  dropped_events counter shows > 0. Correct.
- RISK: At high load (1000 events/s sustained), important early security events
  (KEY_GENERATED) are evicted within ~17 minutes. Since no drain exists, those events
  are permanently lost.

### 6. STRESS — OLDEST ENTRY OVERWRITTEN (ring wrap, no block)
- Ring is non-blocking. New event always succeeds (XSEC_OK returned).
- The slot occupied by the oldest entry is overwritten silently.
- dropped_events counter increments. No alerting mechanism notifies GENSD.
- The dump function (xsec_audit_dump) correctly reports the "WARNING: events dropped"
  line when dropped_events > 0.
- RISK: Under burst load (handshake storm), all 1024 slots can wrap multiple times.
  A forensic audit trail becomes impossible.

### 7. SECURITY — AUDIT RING IS KERNEL-SPACE ONLY, BUT HAS TORN-READ RISK
- xsec_g_audit is a BSS global in the kernel address space. User processes have no
  direct access (XENOS process isolation). Passes for kernel-mode protection.
- However: the slot write in xsec_audit_log() is not atomic (4 separate field assignments
  without a write barrier between them). On a multi-core XENOS future: a reader on CPU-B
  calling xsec_audit_dump() under the spinlock can observe a partially written entry if
  the spinlock is acquired before CPU-A finishes writing all 4 fields.
  Status: OPEN race on the write path. Low severity today (single-core initial boot),
  HIGH severity when SMP cores are enabled.

### 8. UI — NO CLI EXISTS
- xsec_audit_dump() outputs to pal_console_puts() (serial console only).
- No XSHELL command exists to query the audit ring.
- No structured output format for dashboard consumption.
- This is a MEDIUM gap for operational visibility.

### 9. FUZZ — DETAIL FIELD IS SAFE, BUT NULL DETAIL POINTER IS HANDLED
- audit_copy_detail() handles NULL src by writing '\0' immediately. Safe.
- Max detail length enforced at XSEC_AUDIT_DETAIL_LEN - 1 = 63 chars. Correct.
- The xsec_audit_log() function accepts any xsec_audit_event_t / xsec_module_id_t integer.
  The switch in audit_event_name() has a `default: return "UNKNOWN"` fallback.
  Fuzz input of out-of-range enum values will be stored in the ring without crashing.
  The enum field is stored as-is (int32_t under the hood), so raw integer injection
  is possible if a caller passes an unsanitized value. Low risk in kernel space.

### 10. RELIABILITY — FAIL (no crash recovery)
- On power loss, all audit events are lost. The ring is in BSS (DRAM).
- journal_read_recent() is a stub — returns zero entries. No replay capability exists.
- GENSD idle loop calls journal_flush_serial() every 30 seconds to print to serial,
  but serial output is also volatile (no NVMe sink).
- There is NO point-in-time recovery for security audit events.

## 5. XSTORE Integration Opportunity

### Feasibility: YES — Strong Fit
XSTORE's key-value API at xstore.h is a direct fit:
- Key: "sec:audit:{lsn}" (up to 256 bytes, fits easily)
- Value: serialized xsec_audit_entry_t (80 bytes, well within XSTORE_MAX_VAL_LEN = 4096)
- XSTORE provides WAL crash-safety: a committed xstore_txn_commit() is durable on next
  xstore_checkpoint().

### Recommended XSTORE Key Scheme
```
sec:audit:{16-digit-zero-padded-lsn}
   e.g. "sec:audit:0000000000001024"
```
- LSN = xsec_g_audit.total_events at write time (monotonic, never wraps practically).
- This key scheme enables:
  - Forward range scan: xstore_cursor_open(lower="sec:audit:", upper="sec:audit:~")
  - Last-N query: seek to high LSN and walk backward (cursor backward reserved Sprint 5)
  - Idempotent replay: LSN is the natural unique business key

### Value Serialization (80 bytes, little-endian)
```
[8B: timestamp_ns][4B: event_type][4B: source_module][64B: detail]
```
No padding needed. Fits on a single XSTORE leaf node slot.

### Integration Architecture (to implement in Sprint 4)
1. XSEC provides an opaque drain callback registration:
   xsec_audit_set_drain_fn(xsec_audit_drain_fn_t fn, void *ctx)
   Called by GENSD after XSTORE is initialized. Non-blocking.

2. GENSD "xsec-audit-drain" background thread (1-second poll):
   - Calls xsec_audit_query_recent() for new events since last drain LSN.
   - For each new event: xstore_put(key="sec:audit:{lsn}", val=serialized_entry).
   - Calls xstore_checkpoint() every 100 drained events.

3. xsec_audit_init() must be added to GENSD boot chain at Stage 3 (security),
   immediately after SHA-256 KAT passes and before any crypto operations begin.
