## 1. provenance_client.py — JSONL Fallback Pipeline

### CONFIRMED CORRECT
- Bounded storage: MAX_FILE_BYTES (10MB) + MAX_ENTRIES (10000) guard dual-axis. Worst-case 20MB guaranteed.
- POSIX rename() for rotation is atomic on Linux — no partial-read window.
- _entry_count resets to 0 on rotation — counter and file stay in sync.
- flush_fallback_to_service() processes rotated file first (higher priority than active), then active — correct ordering.
- partial flush: remaining lines written back via path.write_text() before deletion — no data loss on partial reconnect.
- SERVICE_URL overridable via env var — correct for local dev / k8s DNS.
- CONNECT_TIMEOUT_S = 2 — prevents hung emit() calls blocking callers.

### RACE CONDITIONS — CRITICAL
1. **_entry_count is not thread-safe.** `_write_fallback()` does `self._entry_count += 1` with no lock. Two concurrent callers both see count N, both increment to N+1. Actual entries written = N+2, counter says N+1. Eventually the file overflows MAX_ENTRIES without rotating.
   - Fix: `threading.Lock()` around `_rotate_if_needed()` + `open(append)` + `self._entry_count += 1` as a single critical section.

2. **TOCTOU between stat() and open(append).** `_rotate_if_needed()` calls `stat()`, then returns. Another thread calls `open(fallback_file, "a")` between the stat and the rename. The file being renamed may already be open for append by the second thread. On Linux the open fd remains valid after rename (inode stays alive), so the write lands in the rotated file. This is not data loss but breaks the expectation that fallback_file.1 is immutable after rotation.
   - Fix: Hold the lock across the entire `_rotate_if_needed()` + `open()` sequence.

3. **flush_fallback_to_service() vs _write_fallback() concurrent execution.** `flush_fallback_to_service()` reads the entire file with `path.read_text()`, then writes survivors back with `path.write_text()`. If `_write_fallback()` appends a new record BETWEEN `read_text()` and `write_text()`, that new record is clobbered by the write_text() overwrite. This is permanent data loss.
   - Fix: Lock must encompass both flush reading AND writing. Or use a separate tmp file + atomic rename for the write-back.

4. **_count_existing_entries() on __init__ races with concurrent writers on an existing file.** Single-process use is safe; multi-process (e.g., two daemon instances against the same FALLBACK_DIR) has no advisory lock. Low likelihood in production but not documented.

### HIGH Issues
- `flush_fallback_to_service()` uses `path.read_text().strip().split("\n")` — loads the entire file into RAM. For a full 10MB file of large JSON records this is ~10MB RAM spike on reconnect. For an edge daemon this may be acceptable but is not streaming.
  - Better: iterate with `path.open()` line-by-line, write survivors to a tempfile, then atomic-rename.
- `_try_service()` returns False on HTTP non-200 but does not distinguish transient (503) from permanent (400 malformed JSON). A malformed record will loop forever in flush, consuming reconnect budget.
  - Fix: Skip (log and discard) records that return 4xx. Only retry on 5xx/network error.
- No file-level advisory lock (`fcntl.flock`) — safe for single-process but fragile if multiple processes share the directory.

### LOW Issues
- `emit()` returns bool but callers are not required to check it. Silent fallback is the design intent, but a metrics counter (fallback_total) would support alerting on sustained outage.
- No `flush=True` or `f.flush()` + `os.fsync()` after the append write. On power loss, the last record(s) may not reach disk. For an audit fallback, fsync is strongly recommended.
- `monotonic_ns` is per-process-relative; it is not meaningful for cross-process or post-restart ordering. Document this limitation.

## 2. thermal_predictor.py — write_state() Atomic Write

### CONFIRMED CORRECT
- EWMA initialization guard (`if pred.ewma_gpu_c == 0.0`) — correctly seeds on first sample.
- Linear regression denominator guard (`abs(denom) < 1e-10`) — division by zero cannot occur.
- Confidence (R²) clamped to [0.0, 1.0] — no out-of-range float escapes.
- `math.isinf()` check before `round()` — avoids `OverflowError` on infinite TTT values.
- `deque(maxlen=HISTORY_SIZE)` — bounded history, no unbounded memory growth.
- Callback exception isolation (`try/except Exception: log.exception`) — one bad callback cannot kill the prediction loop.

### ATOMIC WRITE — CRITICAL
`write_state()` line 225:
```python
STATE_PATH.write_text(json.dumps(state, indent=2))
```
`Path.write_text()` is NOT atomic. Internally it does `open(path, 'w') → write() → close()`. A consumer reading `/run/genos/thermal-prediction.json` between the open-truncate and the close will read a partial/empty file. If the daemon is SIGKILLed mid-write, the file is left truncated. Any downstream reader that does `json.loads(path.read_text())` will raise `JSONDecodeError` and must handle it defensively.

Fix — use write-to-tmpfile + atomic rename:
```python
def write_state(self) -> None:
    import tempfile, os
    state = { ... }  # same as current
    STATE_PATH.parent.mkdir(parents=True, exist_ok=True)
    fd, tmp = tempfile.mkstemp(dir=STATE_PATH.parent, prefix=".thermal-pred-")
    try:
        with os.fdopen(fd, "w") as f:
            json.dump(state, f, indent=2)
            f.flush()
            os.fsync(f.fileno())
        os.replace(tmp, STATE_PATH)  # atomic on same filesystem (POSIX)
    except Exception:
        os.unlink(tmp)
        raise
```
`os.replace()` is guaranteed atomic on POSIX when src and dst are on the same filesystem. `/run/genos/` is a tmpfs mount — same filesystem as the tempfile written to the same directory. This pattern is safe.

### HIGH Issues
- `STATE_PATH.parent.mkdir(parents=True, exist_ok=True)` is called on every `write_state()` invocation (every 500ms per the main loop). `mkdir()` is a syscall even when the directory already exists. Move this to `__init__` or call it once.
- No schema_version increment path. When the JSON schema changes in a future version, readers have no signal to adapt. The `"schema_version": 1` field is correct to have; a migration plan should be documented.
- `time.time()` for the top-level `timestamp` field is NTP-wall-clock sensitive. If NTP steps the clock during a rapid thermal event, timestamps in the state file will jump. Add `monotonic_ns` alongside `timestamp` for unambiguous ordering.
- `deque` is not JSON-serializable. The history field is intentionally excluded from `write_state()` output (correct), but the dataclass `history` field would cause errors if someone naively calls `json.dumps(pred)`. Document or add `__json__` guard.

### LOW Issues
- EWMA seed via `if pred.ewma_gpu_c == 0.0`: if a real GPU temperature is exactly 0.0°C (impossible in practice but valid float), the EWMA is re-initialized every sample. Consider using a sentinel `Optional[float] = None` instead.
- `_compute_trend()` uses `list(pred.history)[-min(n, 30):]` — slices up to 30 samples. But if `n >= MIN_SAMPLES_FOR_PREDICTION (6)` and `n < 30`, this is fine. If `n > 30`, only the last 30 are used despite HISTORY_SIZE=120. This is intentional (recent trend), but not documented.
- test_write_state_creates_file monkeypatches `mod.STATE_PATH` at module level — this is not thread-safe in parallel pytest. Consider using `tmp_path` fixture injection into ThermalPredictor via constructor parameter instead of module-level patching.

## 3. entropy_classifier.c — Shannon LUT Lookup Path

### LUT CORRECTNESS ANALYSIS

**LUT formula per comment:** `lut[f] = round(-f/256 * log2(f/256) * 256)` for f in [0,255].

Manual verification of key boundary values:
- `f=0`: formula → 0 (by convention). LUT[0] = 0. CORRECT.
- `f=1`: `-(1/256)*log2(1/256)*256 = -(1/256)*(-8)*256 = 8`. LUT[1] = 0. **DISCREPANCY** — expected ~8, table shows 0.
- `f=2`: `-(2/256)*log2(2/256)*256 = -(2/256)*(-7)*256 = 14`. LUT[2] = 488. **DISCREPANCY** — expected ~14 (in units of 1/256 entropy), 488 appears to be in units of 1/256 * something larger.

**Root cause: the LUT formula comment is misleading.** The actual scale factor appears to be 256*256 (i.e., `round(-f/256 * log2(f/256) * 65536)`), not 256. This is consistent with LUT[2] = 488: `-(2/256)*log2(2/256)*65536 = -(2/256)*(-7)*65536 ≈ 3584`. Still doesn't match 488.

Re-deriving from the actual computation path in `entropy_classifier_score()`:
```c
u64 p = ((u64)freq[i] << 16) / len;   // p in Q16 fixed-point (0..65536)
u32 lut_idx = min_t(u32, freq[i], 255);
entropy_x256 += (p * log2_lut[lut_idx]) >> 16;
```
`p` represents `freq[i]/len` scaled by 2^16. The contribution to entropy is `p * lut[freq[i]] >> 16`. For this to yield `-p * log2(p)` in units of 1, `lut[f]` must encode `-log2(f/len)` scaled by some factor. But `lut_idx = freq[i]` (raw count), NOT `freq[i]/len` (probability). This is the **fundamental indexing error**.

### CRITICAL: LUT INDEX IS WRONG
The LUT maps raw byte frequency counts (0..255), not probability percentages. The entropy formula `H = -Σ p_i log2(p_i)` requires `p_i = freq[i] / len`. The correct index into a frequency-based LUT should be `freq[i] * 256 / len` (normalized to 0..255), not `freq[i]` itself.

As coded, `lut_idx = min_t(u32, freq[i], 255)` uses the raw count. For a 1000-byte packet where one byte appears 50 times, `lut_idx = 50` but the actual probability index should be `50*256/1000 = 12`. The LUT lookup retrieves the entropy contribution for a byte appearing 50/256 ≈ 19.5% of the time, when the actual probability is 50/1000 = 5%. The error grows with packet size.

**Impact:** For packets longer than 256 bytes (virtually all real packets), the entropy score is systematically wrong. Higher-frequency bytes get an inflated LUT value (they look more common than the LUT represents), pushing scores toward values characteristic of shorter packets. Classification thresholds (>200 for IB, <80 for UET) will misfire.

**Correct LUT design (two options):**
1. Keep the LUT indexed by raw count but make it packet-length-dependent (impractical for a fixed table).
2. Normalize the index: `u32 lut_idx = min_t(u32, (freq[i] * 255) / len, 255);` — this maps actual probability into the LUT correctly.

**Fix for line 71:**
```c
// Replace:
u32 lut_idx = min_t(u32, freq[i], 255);
// With:
u32 lut_idx = min_t(u32, (u32)(((u64)freq[i] * 255) / len), 255);
```

### LUT TRUNCATION — HIGH
The LUT has zeros from index 181 onward (entries 181..255 = 0). The formula `-f/256 * log2(f/256)` does NOT evaluate to 0 for f > 180. At f=181: `-(181/256)*log2(181/256) ≈ -(0.707)*(-0.528) ≈ 0.373` — clearly non-zero. The trailing zeros appear to be a generation error (possibly the Python one-liner was run with integer division truncation). This means any byte appearing more than 181 times in a 256-byte window gets an entropy contribution of 0, artificially lowering the score for high-frequency bytes (low-entropy data). This partially counteracts the indexing error but in an unpredictable way.

**LUT generation command (corrected, using true floating-point):**
```python
import math
lut = [round(-f/256 * math.log2(f/256) * 256) if f > 0 else 0 for f in range(256)]
# Verify: lut[1]=8, lut[128]=256, lut[255]=3
```

### CONFIRMED CORRECT
- NULL and zero-length guards in `entropy_classifier_score()` — CORRECT.
- Single-pass byte frequency count — cache-friendly, O(n). CORRECT.
- `min_t(u32, freq[i], 255)` prevents out-of-bounds LUT access — CORRECT (bounds safe even with wrong semantics).
- `min_t(u64, normalized, 255)` prevents overflow of u8 return — CORRECT.
- `entropy_classify_route()` payload_offset: `min_t(u32, 40, pkt_len/4)` — protects against short packets. CORRECT.
- `EXPORT_SYMBOL_GPL` on both exported symbols — correct for GPL kernel module.

### HIGH Issues
- The `fabric_protocol` enum is not defined in `entropy_classifier.h`. It declares `typedef enum fabric_protocol fabric_proto_t` as a forward declaration with no definition visible. If `entropy_classifier.h` is the only header included by callers, they cannot allocate or switch on `fabric_proto_t` values. The enum body (FABRIC_IB, FABRIC_ROCE, FABRIC_UET) must be defined in the header or in a shared header that both this file and callers include.
- Fixed payload_offset = 40 bytes assumes IPv6 + no extension headers. For IPv4 (20 bytes header) + UDP (8 bytes) = 28 bytes; for RDMA over UDP (RoCEv2) the BTH header adds 12 bytes. A static 40-byte skip will include transport header bytes in the entropy calculation for IPv4 packets, slightly polluting the measurement.
- No SIMD annotation or intrinsic hint. The comment claims "<50ns per packet classification (SIMD-accelerated on DPU ARM cores)" but the C code has no `__attribute__((target("neon")))` or explicit SIMD. Compiler auto-vectorization may handle the frequency loop, but the LUT-lookup loop has a gather pattern that defeats most auto-vectorizers.

### LOW Issues
- `u32 freq[256] = {}` — zero-initializer is C99/C11 valid but some strict kernel lint tools prefer explicit `= {0}`. Minor.
- The comment says "Returns 0-255 normalized entropy score" but the normalization denominator is `8ULL * 256 = 2048`. Max theoretical `entropy_x256` for a truly uniform 256-symbol source over 256 bytes is: each symbol appears once, `p=1/256`, contribution = `(1/256) * log2_lut[1] >> 16` per symbol. With lut[1]=0 (per current LUT), this yields 0 — the normalization produces 0 for maximum entropy due to the LUT error at index 1. This is another symptom of the LUT generation bug.
