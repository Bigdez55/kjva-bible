## XSTORE Findings

### DIE-XSTORE-01: P0 — Atomicity — BTREE-FIRST, WAL-SECOND ordering inverts the WAL contract

**ACID Property:** Atomicity + Durability
**Affected files:** mvcc.c (xstore_txn_put), btree.c (xstore_btree_insert), wal.c (xstore_wal_append)

**Root Cause:**

`xstore_txn_put()` (mvcc.c line 293) calls `xstore_btree_insert()` immediately — this mutates the
in-memory page cache and marks the page dirty. Only when `xstore_txn_commit()` is later called does
the WAL entry get appended. The correct WAL-first protocol is:

```
WAL append → fsync → btree mutate → commit_lsn assigned
```

The current implementation does:

```
btree mutate (page cache dirty) → WAL append → (no fsync) → return OK
```

**Crash scenario — data in page, not in WAL:**

1. `xstore_txn_put()` mutates page cache. Page is dirty.
2. Page cache eviction fires (clock hand sweeps dirty slot): `write_page()` is called immediately
   from `_cache_evict()` (page_cache.c line 115). The uncommitted data is now on disk.
3. Process crashes before `xstore_txn_commit()` writes the WAL COMMIT.
4. On `xstore_open()`: WAL replay finds no COMMIT for this txn_id. Operation is discarded.
5. But the page on disk already has the data — the btree physical state is ahead of the WAL.
6. Result: page file and WAL are inconsistent. The store is corrupt.

**P0 justification:** This is not a hypothetical race. The clock-based page cache eviction runs on
every `xstore_cache_get()` call when all slots are referenced — which is routine during multi-key
inserts. An uncommitted write can reach disk before its WAL entry exists.

**Fix:** `xstore_txn_put()` must buffer the write in the write_set only and NOT call
`xstore_btree_insert()`. The btree insert must happen inside `xstore_txn_commit()`, after all WAL
entries (including COMMIT) have been written and fsynced. The current fix in mvcc.c (reading value
back from btree at commit time) addresses the wrong problem — the btree should not be written until
commit is durable.

### DIE-XSTORE-02: P1 — Atomicity — snapshot_lsn used for WAL value recovery is semantically incorrect and fragile

**ACID Property:** Atomicity (WAL replay correctness)
**Affected files:** mvcc.c (xstore_txn_commit lines 220), btree.c (xstore_btree_lookup line 510)

**Root Cause:**

The recent fix in `xstore_txn_commit()` recovers the value to write into the WAL entry by calling:

```c
xstore_btree_lookup(store, txn->snapshot_lsn, &k, &vout)
```

`txn->snapshot_lsn` is assigned at `xstore_txn_begin()` as `store->next_lsn` at that instant.
The value was written into the btree by `xstore_txn_put()` using an LSN obtained from
`pal_atomic_add(&store->next_lsn, 1)` — which is strictly greater than `snapshot_lsn`.

The lookup call passes `snapshot_lsn` to `xstore_btree_lookup()`, which in the current code
executes:

```c
// btree.c line 510
(void)snapshot_lsn; /* Full per-version chain in Sprint 5 */
```

The snapshot_lsn parameter is completely ignored — the lookup finds the most recent value in the
page regardless of LSN. This is why the fix appears to work: the lookup succeeds by accident.

**The Sprint 5 breakage scenario:**

When Sprint 5 enables per-version MVCC chains, `xstore_btree_lookup()` will respect
`snapshot_lsn`. At that point:

- The value was inserted with LSN `snapshot_lsn + N` (where N >= 1).
- The lookup passes `snapshot_lsn` (before the insert).
- The lookup returns `XSTORE_ERR_NOT_FOUND`.
- `entry.val_len` stays 0.
- The WAL entry records a zero-length value.
- WAL replay inserts an empty value → silent data corruption.

**Fix:** The lookup should use `store->next_lsn` (latest watermark) rather than `txn->snapshot_lsn`,
or — better — the value should be retained in the write_set at put() time (making the btree lookup
at commit unnecessary entirely, which also resolves DIE-XSTORE-01).

### DIE-XSTORE-03: P1 — Durability — xstore_txn_commit() returns XSTORE_OK before any fsync

**ACID Property:** Durability
**Affected files:** mvcc.c (xstore_txn_commit), wal.c (xstore_wal_checkpoint)

**Root Cause:**

`xstore_txn_commit()` appends WAL entries and the COMMIT record via `xstore_wal_append()`.
`xstore_wal_append()` writes to the in-memory WAL ring buffer only — it does not call
`store->sync()`. The function then returns `XSTORE_OK`.

The WAL ring buffer is in caller-provided memory. If the process crashes or the machine loses power
between `xstore_txn_commit()` returning and the next `xstore_checkpoint()` call, the WAL entries
(including COMMIT) are gone. The data in the btree page cache is also gone (in-memory).
The transaction is effectively lost despite `commit()` having returned success.

**Contradiction with public API contract:**

`xstore.h` line 34 states: "A transaction is durable only after `xstore_txn_commit()` returns
`XSTORE_OK`." The internal wal.c comment says "durable after `store->sync()` has been called."
These two statements cannot both be true given the current implementation.

**Fix option A (correct):** Call `store->sync()` at the end of `xstore_txn_commit()` after the
COMMIT WAL entry. This makes every commit durable at the cost of one fsync per transaction.

**Fix option B (documented relaxation):** Update the public contract to state "WAL entries survive
until the next checkpoint; checkpoint is required for durability." All callers must then manage
checkpoint frequency explicitly.

### DIE-XSTORE-04: P1 — Isolation — All writes tracked against root_page, creating false conflict storms

**ACID Property:** Isolation (Optimistic Concurrency Control)
**Affected files:** mvcc.c (xstore_txn_put lines 301-303, xstore_txn_delete lines 330-332)

**Root Cause:**

In both `xstore_txn_put()` and `xstore_txn_delete()`, the write_set entry's `page_no` is set to:

```c
we->page_no = store->sb.root_page;
```

This is documented with a comment: "we store the superblock root_page as a conservative conflict
target." The OCC conflict check (`xstore_mvcc_conflict_check()`) checks whether any concurrent
committer wrote to `we->page_no` after the snapshot LSN.

Because ALL writes from ALL transactions map to the same root_page, the MVCC shadow table records
every commit against root_page. Any two concurrent write transactions will always conflict at the
OCC check, regardless of whether they touch overlapping keys.

**Impact:** Under any concurrent write workload (XPKG installs, AI context updates, provenance
logging), every second writer will receive `XSTORE_ERR_CONFLICT` and must retry. At 3 auto-retries
(`XSTORE_RETRY_MAX=3`), all writes on any key become effectively serialized at the store level,
negating the benefits of OCC.

**Fix:** `xstore_btree_insert()` and `xstore_btree_delete()` must return the actual leaf page_no
that was modified, and that value must be recorded in `we->page_no`. The btree traverse loop already
has `cur` pointing to the final leaf — this value just needs to be surfaced back to the MVCC layer.

### DIE-XSTORE-05: P2 — Consistency — Leaf delete leaves record bytes unreclaimable without page compaction

**ACID Property:** Consistency (B+ tree space invariant)
**Affected files:** btree.c (xstore_btree_delete lines 689-695)

**Root Cause:**

`xstore_btree_delete()` removes a slot from the leaf's offset table by shifting the table entries
left (`*dst = *src` loop). The offset table entry at slot `n-1` (the old last entry) is logically
gone, and `hdr->num_keys--` is correct.

However, the actual key+value record bytes at `page[DATA_OFF + slot->rec_off]` through
`page[DATA_OFF + slot->rec_off + key_len + val_len]` remain in the page data region, unreferenced
but occupying space.

`_leaf_free_space()` computes free space by scanning all `num_keys` existing slots' `rec_off +
key_len + val_len` to find `rec_end`. Since the deleted slot is no longer in the table, its data
bytes are not counted — but they are also not reclaimed. The `rec_end` value from `_leaf_free_space`
will not reach those bytes; new records are packed beyond the highest existing `rec_end`. The
deleted bytes become a permanent "hole" in the data region that is never reused.

**Consequence:** Over N insert+delete cycles on the same page, the page fills up despite holding
fewer than the theoretical max keys, causing premature splits and tree bloat.

**Fix:** A leaf page compaction pass (sliding remaining records to close holes) must be triggered
after delete, or a tombstone-in-data approach must track the hole for reuse.

### DIE-XSTORE-06: P2 — Atomicity — WAL ring saturation during commit creates split-brain state

**ACID Property:** Atomicity
**Affected files:** mvcc.c (xstore_txn_commit lines 229-240), btree.c, wal.c

**Root Cause:**

Inside `xstore_txn_commit()`, if `xstore_wal_append()` returns `XSTORE_ERR_WAL_FULL`, the code
auto-checkpoints and retries once. If the auto-checkpoint fails (e.g., I/O error), the commit
returns failure with `txn->active = false`.

But `xstore_txn_put()` already applied the write to the btree page cache. The btree now has the
data. The WAL has no COMMIT record. The txn is marked inactive, so no retry or abort is possible.

This is a split-brain: btree has the data, WAL doesn't. The page cache is dirty. If a checkpoint
runs later (triggered by another transaction), it will flush the dirty page to disk without a
corresponding WAL entry — making the orphaned data durable without a WAL record. On next open,
WAL replay cannot redo it (no WAL record exists) and cannot undo it (redo-only WAL). The page
on disk has data that has no authoritative WAL record.

**This is a second manifestation of the BTREE-FIRST issue (DIE-XSTORE-01).**

### DIE-XSTORE-07: P2 — Consistency — Dual next_lsn advancement creates LSN aliasing

**ACID Property:** Consistency (LSN monotonicity and WAL/page correlation)
**Affected files:** btree.c (xstore_btree_insert line 532, xstore_btree_delete line 673), wal.c (xstore_wal_append line 72)

**Root Cause:**

`xstore_btree_insert()` advances `next_lsn` via `pal_atomic_add(&store->next_lsn, 1)` to obtain
an LSN for the page header (`hdr->lsn`). Later, `xstore_wal_append()` also advances `next_lsn`
via `pal_atomic_add(&store->next_lsn, 1)` to assign the WAL entry's `lsn` field.

These are two independent increments. The LSN written into the btree page header and the LSN
written into the corresponding WAL entry are different values. Crash recovery code in wal.c
uses `e->lsn != lsn` to validate WAL slot ownership, but the page header's LSN has no direct
relationship to the WAL entry LSN. Any future tooling that tries to correlate "which WAL entry
created this page version" cannot do so reliably.

Additionally, `xstore_wal_checkpoint()` computes `checkpoint_lsn = store->next_lsn - 1` after
appending the CHECKPOINT entry. With multiple concurrent btree operations also advancing next_lsn,
the checkpoint_lsn may be higher than expected, causing the WAL tail to advance past entries that
have not yet been flushed to pages.

### DIE-XBLOB-01: P1 — Atomicity/Durability — Index is purely in-memory; total loss on process crash

**ACID Property:** Atomicity + Durability
**Affected files:** xblob.c (xblob_open), blob.c (xblob_put), gc.c (xblob_gc_log_replay)

**Root Cause:**

XBLOB's hash index (the `xblob_index_entry_t *index` pointer) is a caller-provided in-memory
buffer. The index maps SHA-256 hash → (blob_offset, blob_len, ref_count). On `xblob_open(create=
false)`, the index is NOT read from disk — it is rebuilt from the GC log via `xblob_gc_log_replay`.

`xblob_gc_log_replay()` only sets `ref_count` on existing index entries:

```c
blob->index[idx].ref_count = e->new_refcount;  // gc.c line 98
```

But `xblob_index_find()` at line 94 of gc.c can only find the entry if it already has `state ==
XBLOB_INDEX_SLOT_OCCUPIED` — meaning the entry must already be present in the in-memory index.
After a crash, all entries are zeroed (state == FREE). The GC log replay finds no entries and
does nothing.

The `xblob_open(create=false)` path at xblob.c line 87 then rebuilds the index from the index
buffer's `state` field — but if the buffer is freshly zeroed (as it would be on any new process
invocation), all states are FREE and no entries are found.

**Result:** After any process restart, the XBLOB index is empty even if the backing blob storage
contains valid blobs. Every `xblob_get()` returns `XBLOB_ERR_NOT_FOUND`. All previously stored
blobs are effectively inaccessible (the data bytes exist on disk at their offsets, but the index
mapping is gone).

**Fix:** The index buffer must be persisted to durable storage (via the blob I/O callbacks) on
every `xblob_put()` and loaded back into the caller-provided buffer on `xblob_open()`. The GC log
replay must then patch ref_counts on top of the loaded index, not on an empty one. Alternatively,
a separate index persistence region must be maintained in the blob store backing device.

### DIE-XBLOB-02: P1 — Durability — GC log writes are not fsynced; refcount state is volatile

**ACID Property:** Durability
**Affected files:** gc.c (xblob_gc_log_append), xblob.c (xblob_close)

**Root Cause:**

`xblob_gc_log_append()` writes entries to the in-memory `gc_log` ring buffer. It does not call
`blob->io.sync()`. The only sync call is in `xblob_close()`.

If the process crashes between a `xblob_put()` (which calls `xblob_gc_log_append()` with
`XBLOB_GC_OP_INC`) and an explicit close or flush, the GC log entry is lost. On next open,
`xblob_gc_log_replay()` will not see the INC record. The ref_count for that blob will be lower
than the true value.

**Concrete impact:** If `xblob_put()` is followed by `xblob_delete()` on the same hash — both
succeeded — and then the process crashes before the GC log is synced, on recovery the GC log
shows: no INC (lost), but the DEC may or may not have made it to disk depending on ordering.
If neither made it: ref_count = 0 (from index rebuild) → `xblob_gc_scan()` will free the blob
even though a consumer still holds a reference.

**Fix:** `xblob_gc_log_append()` must call `blob->io.sync()` after writing the log entry, or at
minimum after every `xblob_put()` / `xblob_delete()` (group commit). This matches the WAL
durability requirement: the log entry must be on durable storage before the operation returns OK.

### DIE-XBLOB-03: P1 — Atomicity — TOCTOU in xblob_put: slot allocated before index insert; crash leaves orphaned bitmap bit

**ACID Property:** Atomicity
**Affected files:** blob.c (xblob_put lines 120-175)

**Root Cause:**

`xblob_put()` for a new blob executes this sequence under `store_lock`:

1. `xblob_alloc_slot()` → marks bitmap bit, returns offset (line 120)
2. `blob->io.write(hdr)` (line 154)
3. `blob->io.write(data)` (line 162)
4. `xblob_index_insert()` (line 170)

If the process crashes between steps 1 and 4, the bitmap bit is set but no index entry exists.

On `xblob_open(create=false)`, the bitmap is reconstructed from the index (xblob.c lines 93-98):

```c
uint32_t slot_no = xblob_offset_to_slot(index_buf[i].blob_offset);
blob->alloc_bitmap[word] |= (1ULL << bit);
```

Since the index entry was never written (crash before step 4), this slot_no will not be found in
any index entry. The bitmap bit for this slot will remain CLEAR after rebuild. The orphaned slot
can be reallocated, and the partial header/data written in steps 2-3 will be silently overwritten.

This is acceptable behavior (no corruption), but it means the data written in steps 2-3 is
permanently silently discarded rather than explicitly detected. If the I/O callbacks are expensive
(e.g., NVMe writes), these orphaned writes waste I/O for no durable result.

**Additionally:** If step 3 (data write) completes but `xblob_index_insert()` fails with
`XBLOB_ERR_INDEX_FULL`, `xblob_free_slot()` is called to reclaim the bitmap bit. However, the
written bytes remain on the backing store — no cleanup write is issued. This is a storage leak
that is only recovered if the slot is later allocated and the header is overwritten.

### DIE-XBLOB-04: P2 — Consistency — Index entry CRC does not cover state or ref_count fields

**ACID Property:** Consistency (on-disk integrity)
**Affected files:** index.c (xblob_index_insert, xblob_index_inc_ref, xblob_index_dec_ref)

**Root Cause:**

`xblob_index_entry_t` layout (56 bytes):
- Bytes 0: `state` (uint8_t)
- Bytes 1-3: `_pad[3]`
- Bytes 4-7: `ref_count` (uint32_t)
- Bytes 8-39: `hash[32]`
- Bytes 40-47: `blob_offset` (uint64_t)
- Bytes 48-51: `blob_len` (uint32_t)
- Bytes 52-55: `crc` (uint32_t)

The CRC is computed over bytes [8, 52): hash + blob_offset + blob_len. The `state` byte (0) and
`ref_count` (bytes 4-7) are NOT included in the CRC. Bit flips in `state` (e.g., OCCUPIED→FREE
= 0x01→0x00) or `ref_count` are undetected by any integrity check.

A corrupted `state` byte turning OCCUPIED to TOMBSTONE would cause the entry to be skipped during
lookups, making the blob invisble — a silent data loss from the consumer's perspective. A corrupted
`ref_count` decrement to 0 would cause the blob to be freed by the next GC scan.

**Fix:** The CRC must cover bytes [0, 52) — the full entry excluding only the CRC field itself.
All three CRC computation sites (insert, inc_ref, dec_ref) must be updated consistently.

### DIE-XBLOB-05: P1 — Atomicity — GC compaction has no rollback; index entries permanently lost on insert failure

**ACID Property:** Atomicity
**Affected files:** gc.c (xblob_gc_scan lines 163-196)

**Root Cause:**

The compaction path in `xblob_gc_scan()` processes the index in chunks of 256. For each chunk:

1. Marks each OCCUPIED entry in `blob->index[i].state = XBLOB_INDEX_SLOT_FREE` (clearing it from
   the live table) while saving the entry in the local `chunk[]` array on the stack.
2. After clearing, calls `xblob_index_insert()` for each saved entry to re-insert it at its
   optimal slot.

The critical flaw: step 1 (clear) and step 2 (re-insert) are interleaved across chunks. When
processing chunk K, entries from chunks 0..(K-1) have already been re-inserted. Entries from
chunks (K+1)..N are still present in the table (not yet cleared). This means that during
compaction, `xblob_index_insert()` must probe past still-occupied entries from later chunks.

If `xblob_index_insert()` returns `XBLOB_ERR_INDEX_FULL` for any entry in chunk K:

```c
xblob_index_insert(blob, chunk[j].hash, chunk[j].blob_offset, chunk[j].blob_len);
/* No error check — return value is ignored */
```

The return value is NOT checked. The insert failure is silently ignored. The entry's blob_offset
and blob_len are permanently lost. The blob data remains on disk but is unreachable — a storage
leak. Its bitmap slot remains allocated (slot was in the original index entry's blob_offset), so
the slot is also never reused. This is permanent storage corruption.

**Fix:** The return value of `xblob_index_insert()` inside the compaction loop MUST be checked.
On failure, the compaction must abort and the entire table must be restored from a snapshot taken
before compaction began. Alternatively, compaction must be done into a separate scratch buffer,
validated in full, then swapped in atomically.

### DIE-XBLOB-06: P3 — Efficiency — xblob_gc_log_append acquires gc_lock while called from xblob_gc_scan which holds store_lock

**ACID Property:** Not an ACID violation — lock contention observation
**Affected files:** gc.c (xblob_gc_scan line 138), gc.c (xblob_gc_log_append line 43)

**Observation:**

`xblob_gc_scan()` holds `store_lock` for its entire duration. Inside the scan loop, it calls
`xblob_gc_log_append()`, which acquires `gc_lock`. The lock order throughout the XBLOB codebase is:

- `blob.c xblob_put()`: acquire store_lock → release store_lock → acquire gc_lock (sequential)
- `blob.c xblob_delete()`: acquire store_lock → release store_lock → acquire gc_lock (sequential)
- `gc.c xblob_gc_scan()`: acquire store_lock → (while holding store_lock) acquire gc_lock

Only `xblob_gc_scan()` ever holds both locks simultaneously, with order store_lock → gc_lock. No
other code path holds gc_lock while acquiring store_lock. No deadlock risk in the current codebase.

However, if a future operation acquires gc_lock first then store_lock, a deadlock will occur.
The lock order must be documented as a contract: `store_lock` always precedes `gc_lock`.

## WAL Correctness Analysis — mvcc.c Fix Verification

**Question 1: Is snapshot_lsn correct for read-your-own-writes in the btree lookup?**

No. `snapshot_lsn` was assigned at `txn_begin` time and is less than the LSN used when the value
was written into the btree by `xstore_txn_put()`. The lookup appears to work only because
`xstore_btree_lookup()` currently ignores `snapshot_lsn` (line 510: `(void)snapshot_lsn`).
When Sprint 5 enables version chains, this will silently return NOT_FOUND. See DIE-XSTORE-02.

**Question 2: Does xstore_btree_lookup acquire wal_lock, creating a deadlock?**

No deadlock exists. The call sequence in `xstore_txn_commit()` is:
`btree_lookup(acquires+releases cache_lock)` → `wal_append(acquires+releases wal_lock)`.
These are sequential, not nested. The auto-checkpoint path (WAL_FULL inside commit) calls
`xstore_wal_checkpoint()`, which acquires `wal_lock` only at step 5 (tail advancement), after
`xstore_cache_flush()` (which acquires `cache_lock`) has completed. No circular lock dependency.

**Question 3: If btree_lookup returns NOT_FOUND at commit time, what happens?**

`val_len` remains 0 in the WAL entry. On crash recovery, `xstore_wal_replay()` calls
`xstore_btree_insert()` with `val.len = 0`. The btree accepts this (0 is a valid val_len per the
val_t descriptor; `XSTORE_MAX_VAL_LEN = 4096` and val_len=0 passes the `<= MAX_VAL_LEN` check).
The key is re-inserted with an empty value — this is silent data corruption. The real value from
before the crash is replaced with an empty value.

**Question 4: ATOMICITY — Is this WAL-first?**

No. This is the core finding of DIE-XSTORE-01. The fix in mvcc.c addresses WAL entry content
completeness, but it does not fix the ordering: btree is still mutated before the WAL COMMIT is
durable. The lookup-at-commit is a workaround that assumes the page cache still holds the value
at commit time (it usually will, but this is not guaranteed if eviction occurred between put() and
commit()). If the page was evicted between put() and commit(), the lookup returns NOT_FOUND and
the WAL entry has val_len=0 → corruption on replay.

## Recommended Fix Order

1. **DIE-XSTORE-01 (P0):** Restructure to WAL-first. Buffer writes in write_set with value bytes.
   Apply btree mutation inside commit AFTER WAL+COMMIT are fsynced.

2. **DIE-XBLOB-01 (P1):** Design a persistence region for the XBLOB index on the backing store.
   Load it on open before GC log replay.

3. **DIE-XBLOB-05 (P1):** Add error checking to compaction's insert loop. Abort compaction and
   restore if any insert fails.

4. **DIE-XSTORE-03 (P1):** Add `store->sync()` call at end of `xstore_txn_commit()`.
