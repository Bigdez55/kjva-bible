# Sprint 3 Schema Coherence Audit (2026-03-06)

## Scope
XPKG binary format, XNET wire structs, XSEC x509/audit schemas, cross-module coherence.

## Module-by-Module Findings

### 1. XPKG Binary Format (pkg_format.c, xpkg.h)
- Magic: 0x58504B47 ("XPKG"), format_version=1, flags bitfield
- Header: DECLARED as 20 bytes but xpkg_archive_header_t has 7 fields summing to 24 bytes
- P1-XPKG-HDR-1: total_size field (4B) declared in struct but NEVER SERIALIZED on wire
  - parse_header sets total_size = buf_len (inferred from context)
  - serialize writes only 6 fields at offsets [0..19], total_size is MISSING from wire
  - Struct declares 24 bytes, wire format is 20 bytes -- MISMATCH
- Forward-compat: format_version field exists (GOOD), XPKG_FLAG_COMPRESSED/DELTA reserved (GOOD)
- content_hash: covers payload only [payload_offset, signature_offset) -- self-ref loop FIXED (S3-DI-H05)
- Metadata: TOML-like key=value, LF-terminated -- NOT a standard format, fragile for Sprint 5 app migration
- P2-XPKG-META-1: No content-type field, no architecture field, no min-OS-version field
- P2-XPKG-META-2: No install timestamp, no installed-by field (provenance gap)
- xpkg_package_t: ~79KB per package (256 files * 294B + 16 deps * ~140B + metadata)

### 2. XNET Wire Structs (xnet.h)
- ALL wire-format structs have __attribute__((packed)) -- CORRECT
  - xnet_eth_hdr_t (line 166), xnet_arp_pkt_t (184), xnet_ip4_hdr_t (214)
  - xnet_icmp_hdr_t (255), xnet_udp_hdr_t (269), xnet_tcp_hdr_t (292)
  - gossip.c: xnet_gossip_msg_hdr_t (92), xnet_gossip_event_t (105)
- ALL multi-byte fields use xnet_htons/htonl on write, xnet_ntohs/ntohl on read -- CORRECT
  - Verified in: arp.c, ethernet.c, icmp.c, udp.c, ip4.c, socket.c, mdns.c
- Non-wire structs (cache, routing, stats) correctly OMIT packed -- CORRECT
  - xnet_sockaddr_in_t, xnet_if_stats_t, xnet_route_entry_t, dns cache entry, etc.
- P1-XNET-DNS-1: DNS header is built manually (byte-by-byte) in dns.c, no dns_hdr_t struct
  - Acceptable for Sprint 3 but should be formalized for Sprint 4
- GRADE: A (wire format correctness is solid)

### 3. X.509 Certificate (x509.c, xsec.h)
- xsec_x509_cert_t: COMPLETE for TLS usage
  - version, serial[20], subject/issuer[128], not_before/not_after (epoch secs)
  - pub_key[64] + pub_key_len, signature[512] + sig_len
  - tbs_data/tbs_len for signature verification -- CORRECT
  - is_ca from BasicConstraints -- CORRECT
  - der_data/der_len for zero-copy access -- CORRECT
- Ed25519 signature: 64 bytes (XSEC_ED25519_SIG_LEN=64) -- CORRECT
- Signature buffer: XSEC_X509_SIG_MAX=512 bytes (handles RSA-4096 too) -- CORRECT
- Chain verification: Ed25519 only, RSA/ECDSA rejected -- ACCEPTABLE for Sprint 3
- Hostname validation: CN-based with wildcard support, null-byte injection protection -- CORRECT
- P1-X509-SAN-1: No Subject Alternative Name (SAN) support -- needed for multi-domain certs
- GRADE: A- (solid for Ed25519-only, SAN gap for production)

### 4. Audit Event Schema (audit.c, xsec.h)
- xsec_audit_entry_t: FIXED SIZE (80 bytes on x86-64)
  - timestamp_ns (8B) + event_type (4B) + source_module (4B) + detail[64] (64B)
- Timestamp: YES, pal_time_now_ns() at event time -- CORRECT
- Sequence number: NO explicit sequence field
  - write_idx is monotonic but NOT stored per-entry
  - P1-AUDIT-SEQ-1: No sequence number in xsec_audit_entry_t (ordering lost on ring read)
- Ring buffer: 1024 entries, atomic write_idx, spinlock for reads
- Overflow: dropped_events counter added (S3-DI-H01 fix) -- GOOD
- PROVENANCE VIOLATION: ring overwrites destroy old entries (KNOWN, flagged in Sprint 3 capacity audit)
- GRADE: B (functional, missing sequence number, ring overwrites violate provenance)

### 5. Cross-Schema Coherence

#### 5a. XPKG <-> XSEC Ed25519 Key Format: COHERENT
- XPKG: pub_key[32] in xpkg_trusted_key_t (xpkg.h:234)
- XSEC: XSEC_ED25519_PUB_LEN=32, pub[32] in xsec_ed25519_verify (xsec.h:268)
- XPKG calls xsec_ed25519_verify(key_store[k].pub_key, msg, msg_len, sig) -- CORRECT
- Signature: XPKG metadata.signature[64] matches XSEC_ED25519_SIG_LEN=64 -- CORRECT
- XPKG signs content_hash[32] (SHA-256), XSEC verifies arbitrary message -- COMPATIBLE

#### 5b. XPKG <-> XSEC X.509 Certificate: NOT CONNECTED
- XPKG trusted key store uses raw Ed25519 pub_key[32]
- XSEC x509 uses xsec_x509_cert_t with pub_key[XSEC_X509_PUBKEY_MAX=64]
- P2-CROSS-1: No mechanism to extract Ed25519 pub_key from x509 cert for XPKG trust store
- P2-CROSS-2: TLS certificate chain validation and XPKG signature verification are independent paths
  - In Sprint 5 (app store), package repo fetch over TLS should validate server cert
    AND package signature should be verifiable via x509 publisher cert chain
  - Currently these are two separate trust models with no bridge

#### 5c. XSEC Audit <-> XPKG Events: NOT CONNECTED
- XSEC audit logs crypto events (cert verify, handshake, etc.)
- XPKG has NO audit logging (install, remove, upgrade, gate check results not logged)
- P1-CROSS-3: xpkg_install/xpkg_remove should emit xsec_audit_log events

#### 5d. Hash Algorithm Coherence: COHERENT
- XPKG content_hash: SHA-256 via pal_sha256() -- CORRECT
- XPKG per-file hash: SHA-256 via pal_sha256() -- CORRECT
- XPKG revocation list: SHA-256 of pub_key via pal_sha256() -- CORRECT
- XSEC x509: SHA-256/SHA-512 for Ed25519 -- CORRECT
- All SHA-256 uses are 32 bytes, all Ed25519 sigs are 64 bytes -- CONSISTENT

## Forward-Compatibility for Sprint 4 XSTORE

### XPKG Package Format
- format_version field allows future version bumps -- GOOD
- flags field has COMPRESSED and DELTA reserved -- GOOD
- Metadata is text-based key=value, extensible by adding new keys -- ADEQUATE
- MISSING for Sprint 5: architecture, content-type, min-os-version, capabilities-required

### Recommendations for XSTORE Schema
1. When XPKG registry moves from BSS to XSTORE, add:
   - installed_at_ns (action_time), recorded_at_ns (record_time) per provenance model
   - installed_by (user/system/auto-update)
   - _end timestamp columns for bitemporal close-out on remove/upgrade
2. XSEC audit ring should flush to XSTORE append-only log (provenance-safe)
3. XPKG trusted key store should support x509 certificate chains in Sprint 5
4. DNS cache should be backed by XSTORE for persistence across reboots

### Overall Grade: B+
- Wire format correctness: A (XNET packed structs, byte-order, checksums all correct)
- Crypto coherence: A (Ed25519 key sizes, SHA-256 usage fully consistent across modules)
- Provenance readiness: C (no timestamps on packages, audit ring overwrites, no install log)
- Forward-compat: B (version fields present, but missing Sprint 5 metadata fields)
