## Foundation Design Decisions (Sprint 0)

### XSTORE + XBLOB (Sprint 4 IMPLEMENTED, 4011 LOC)

- XSTORE: B+ tree order 256, 4096B pages, CRC32C, WAL 65K ring, MVCC snapshot isolation
- XBLOB: content-addressed, SHA-256 keys, 16MB max blob, 1M-slot open-addressing index
- XBLOB dedup: put() increments refcount on duplicate hash, no new storage written
- Both use pal_sha256() (same as XSEC/XPKG) -- single authoritative SHA-256
- WAL: redo-only, fixed-size entries (4384B each), 287MB total ring buffer
- P0: WAL key_len uint8_t truncates 256->0; MVCC conflicts on root_page (serial)
- P1: No XPKG-to-XSTORE wiring, no XPKG-to-XBLOB wiring, no key namespace schema
- FROZEN tier NVMe spill: STILL A STUB (data lost on COLD->FROZEN demotion)
- See sprint4-data-architecture-audit.md for Sprint 4 findings (2 P0, 10 P1, 10 P2)

## Throughput Audit (2026-03-02) -- see throughput-audit.md
- Current max: ~100-200 events/sec (new TCP per event, sync emit, chain lock serialization)
- provenance_client uses urllib.request (no connection reuse, no keepalive)
- No circuit breaker: 2s timeout per emit during service outage
- No batch endpoint (POST /log/batch) on service
- No write-ahead buffer in client (caller blocks on every emit)
- flush_fallback_to_service holds _lock for entire flush (can block emitters ~20s)
- SQLAlchemy is synchronous (not async), chain_lock serializes all writes
- No deduplication (uuid4 per request, no idempotency key)
- GET /provenance has no pagination (unbounded result set)
- Priority fix order: connection reuse > circuit breaker > lock-free flush > batch endpoint > async SA
