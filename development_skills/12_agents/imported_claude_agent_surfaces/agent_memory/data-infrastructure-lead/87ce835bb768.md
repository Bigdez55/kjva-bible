# Provenance Pipeline Throughput Audit -- 2026-03-02

- tools/provenance_client.py (324 lines)

## HIGH Findings (6)
1. No HTTP connection reuse (urllib.request opens new TCP per event)
2. No batching / write-ahead buffer (sync one-at-a-time emit)
3. Chain lock serializes ALL writes (held during SELECT + INSERT + COMMIT)
4. Synchronous SQLAlchemy (not async, threadpool-bound)
5. Fallback lock held during fsync (blocks caller for disk I/O)
6. flush_fallback_to_service holds lock for entire duration (blocks all emitters)

## MEDIUM Findings (6)
1. No circuit breaker (2s timeout per emit during outage)
2. No batch endpoint (POST /log/batch does not exist)
3. No deduplication (uuid4 per request, no idempotency key)
4. No pagination on GET /provenance
5. JSONL format overhead (minor, keep for debuggability)
6. No auto-flush mechanism for fallback files

## LOW Findings (3)
1. No explicit connection pool config on SQLAlchemy engine
2. _count_existing_entries reads entire file on startup (acceptable)
3. DRY violation: _try_service and _try_service_with_status are near-duplicates

## Throughput Estimates
- Current: ~100-200 events/sec sustained, ~100 burst
- After connection reuse + batch + circuit breaker: ~5000-10000 events/sec
- p99 target (50ms): currently MISS, achievable after HIGH-1+2+MEDIUM-1

## Priority Implementation Order
1. Connection reuse (urllib3 pool or http.client keepalive) + DRY refactor
2. Circuit breaker (3 consecutive failures -> skip HTTP for 30s)
3. Lock-free flush (atomic swap file, flush without lock, write back survivors)
4. Write-ahead buffer + batch endpoint (POST /log/batch)
5. Batch within chain lock (single commit for N records)
6. Async SQLAlchemy (create_async_engine + asyncpg)
7. Batched fsync (included in buffer design)
8. Deduplication (client-generated idempotency key)
9. Pagination on GET /provenance
10. Auto-flush background thread
