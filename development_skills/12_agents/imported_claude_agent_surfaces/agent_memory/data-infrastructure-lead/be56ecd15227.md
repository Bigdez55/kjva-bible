# XSEC Audit Data Governance Findings (2026-03-06)

Full report: docs/compliance/xsec-audit-data-governance-report.md

## xsec_audit_entry_t Schema (80 bytes)
- timestamp_ns (8B) + event_type (4B) + source_module (4B) + detail[64] (64B)
- 11 event types defined (1-11), 9 modules (1-9)
- 39 production call sites across tls13.c, x509.c, aes_gcm.c, chacha20.c, x25519.c, ed25519.c, refusal_gate.c
- Ring: 1024 entries, 80KB total, BSS segment (in-memory only)

## P0 Findings (5)
- DATA-GOV-01: Single timestamp violates two-time-dimension provenance model
- DATA-GOV-03: No session_id -- cannot correlate events from one TLS handshake
- DATA-GOV-04: Ephemeral ring complies with no retention policy (102s max at 10 evt/s)
- DATA-GOV-12: Zero production consumers -- ring is write-only (no drain, no forward)
- DATA-GOV-13: Oldest events destroyed without flush -- provenance violation

## P1 Findings (6)
- DATA-GOV-02: write_idx/total_events race (two separate atomics, reader sees inconsistent state)
- DATA-GOV-05: No tamper-evidence (no hash chain, no checksums on entries)
- DATA-GOV-07: audit_event_name/audit_module_name miss PKG events 10,11 + module 9 (show UNKNOWN)
- DATA-GOV-09: GENSD journal has no security event entry type
- DATA-GOV-10: GENSD journal ring also overwrites (same provenance violation)
- DATA-GOV-11: journal_read_recent() still a stub -- blocks automated forwarding

## P2 Findings (2)
- DATA-GOV-06: extern xsec_g_audit allows any kernel module to read/write ring
- DATA-GOV-08: XSTORE WAL 46x write amplification for 96B audit events (batch mitigates to ~1.1x)

## PII Assessment: LOW
- No IP addresses logged (transport is opaque void *ctx)
- No private keys logged (constant strings only)
- Hostname not included in audit detail strings
- Package names may appear in gate check detail field

## XSTORE Integration Design
- Key: "sec:audit:batch:{batch_lsn}" -- batched 42 entries per value
- Value: [count:4B][entry0:96B]...[entry41:96B] (4,036B max, fits XSTORE_MAX_VAL_LEN)
- 96B entry = 80B original + session_id (8B) + record_time_ns (8B)
- HMAC-SHA256 chain per batch for tamper evidence
- Flush: every 1000ms OR at 768 entries (75% ring capacity)
- Write amp with batching: ~1.1x per event (acceptable)

## Ring Saturation Times
- 10 evt/s: 102s | 100 evt/s: 10.2s | 1000 evt/s (DDoS): 1.02s

## GENSD Journal Integration
- journal_write(service, level, msg) -- no structured security fields
- journal_read_recent() is STILL a stub (Sprint 1, not implemented in Sprint 3)
- Both XSEC ring and GENSD journal need XSTORE-backed persistence in Sprint 4
- Proposed: JOURNAL_ENTRY_SECURITY type=1 with structured payload

## Sprint 4 Implementation Phases
1. Schema extension (session_id, record_time_ns, fix audit_event_name)
2. XSTORE audit sink (audit_store.c, batch flush, HMAC chain)
3. Flush daemon (GENSD-supervised, 1000ms timer, 75% threshold)
4. GENSD journal bridge (security entry type, implement journal_read_recent)
