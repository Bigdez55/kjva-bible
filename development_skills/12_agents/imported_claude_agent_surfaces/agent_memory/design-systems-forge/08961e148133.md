## CONFIRMED FINDINGS (Wave 1 claims vs. actual evidence)

### DSF-W2-MOT-01 (P2) — Easing curve diverges from Material Design spec
CONFIRMED. tokens.ts:57-59 all three motion values use cubic-bezier(0.2, 0, 0.2, 1).
styles.css:55-57 echoes this faithfully. Material Design standard-easing is (0.4, 0, 0.2, 1).
The x1 param (0.2 vs 0.4) shortens the initial deceleration curve — motions feel snappier
but less physically grounded. This is a deliberate design decision or an oversight; there is no
comment explaining the deviation. Token is the single source so one edit in tokens.ts fixes all.

### DSF-W2-SPIN-01 (P1) — Button spinner does NOT halt under prefers-reduced-motion
CONFIRMED, and the bug is more subtle than Wave 1 reported.
Button.tsx:97-102 injects a <style> block with two @keyframes blocks:
  @keyframes btn-spin { to { transform: rotate(360deg); } }
  @media (prefers-reduced-motion: reduce) {
    @keyframes btn-spin { to { transform: rotate(360deg); } }
  }
The reduced-motion block redefines btn-spin with the SAME rotation keyframe — it does not halt,
substitute, or clip the animation. The animation prop on the SVG (line 93) is unconditional:
  style={{ animation: "btn-spin 0.7s linear infinite", ... }}
styles.css:425-433 has a blanket @media (prefers-reduced-motion: reduce) that sets
animation-duration: 0.01ms !important globally. This catches the spinner only because styles.css
is loaded at the document level. The component's inline <style> injected keyframe override will
win specificity races in some rendering contexts. The correct fix is in Button.tsx: in the
reduced-motion media query, redefine btn-spin to { opacity: 0.7; } or remove animation
entirely via animation: none on the SVG style.

### DSF-W2-OSK-01 (P2) — Dual OSK architecture: squeekboard + TypeScript VirtualKeyboard
CONFIRMED. Two separate on-screen keyboard systems exist:
1. compositor/genos-shell/system/tablet_mode.py:31-32 — launches squeekboard (Wayland GTK OSK)
   when tablet mode is detected via /sys/bus/platform/devices.
2. compositor/genos-shell/src/renderer/components/tablet-mode.tsx:230-396 — full QWERTY
   VirtualKeyboard React component rendered inside the Electron renderer process.
Both systems are active. tablet_mode.py runs as a daemon (detect_tablet_mode() polling loop,
2s interval). TabletModeManager triggers VirtualKeyboard on HTMLInput focus in tablet mode.
In a Wayland/Electron environment both will attempt to render on screen simultaneously.
No coordinator exists between the Python daemon and the JS layer. No IPC message from
tablet_mode.py reaches the Electron renderer to suppress VirtualKeyboard when squeekboard is
already running.

### DSF-W2-EYE-01 (P2) — Eye-toggle touch target is below 44px minimum
CONFIRMED. LockScreen.tsx:327-342 and Greeter.tsx:335-354 both render the eye-toggle button with:
  minHeight: "unset"
  padding: 4
The button renders at approximately 26x26px — the 18px SVG icon plus 8px total padding (4px each
side). This fails WCAG 2.5.5 (44x44px minimum) and the GEN.OS 48px tablet-mode target.
The parent password input is 48px tall (LockScreen) or 44px (Greeter); the button is absolutely
positioned inside it (right:12, top:50%, transform:translateY(-50%)) and explicitly overrides
the tablet-mode.css min-height rule via minHeight:"unset". Wave 1 reported the eye-toggle was
"removed" — it is NOT removed; it exists in both components with a deliberate minHeight:unset
override that suppresses the tablet-mode enforcement. The pattern appears twice.

## NEW FINDINGS

### DSF-W2-TOK-01 (P2) — Button.tsx has 4 hardcoded hover/active hex values
Button.tsx:22-23 (secondary hover/active): "#1592d0" / "#1280b8"
Button.tsx:33-34 (danger hover/active): "#d63a5f" / "#bf3354"
These do not appear in tokens.ts. Hover/active states for secondary and danger variants are
uncontrolled by the token system. A token change to colors.secondary or colors.danger will
not affect interactive states — the relationship between base and hover shade will silently
drift. Required: add secondaryMuted, dangerMuted tokens analogous to existing primaryMuted.

### DSF-W2-TOK-02 (P3) — Greeter.tsx has one hardcoded midpoint color
Greeter.tsx:243: background gradient uses "#0d1820" as the 50% stop. This is between
colors.neutral0 (#0d1115) and colors.neutral100 (#151b22) but is neither token value.
It should be extracted as a named token or composed from existing tokens.

### DSF-W2-TOK-03 (P3) — FirstRunWizard.tsx hardcodes 5 wallpaper colors
FirstRunWizard.tsx:77-81: wallpaper palette is "#ff7518", "#1aa7ec", "#17b978", "#a855f7",
"#f59e0b". The first three duplicate tokens.primary/secondary/success but are hardcoded
strings. "#a855f7" (purple) and "#f59e0b" (amber) have no corresponding token at all —
two colors are used in the shipped UI that are outside the token vocabulary.

### DSF-W2-CSS-01 (P2) — styles.css tablet-mode touch target is 56px, not 48px
styles.css:414-418 (.tablet-mode .touch-target): min-width: 56px, min-height: 56px.
tablet-mode.css:12 (--touch-target-min: 48px) is the canonical tablet target and is applied
to body.tablet-mode button et al. The .touch-target class override in styles.css sets a
larger (56px) value exclusively for elements bearing the .touch-target class. This creates
two different touch target floors in tablet mode (48px for most elements, 56px for
.touch-target-classed elements). The inconsistency is not documented and could cause
visual rhythm breaks in the dock.

### DSF-W2-A11Y-01 (P2) — Dialog.tsx focus trap may miss dynamically injected elements
Dialog.tsx:30-34: focusable elements are queried once on open (querySelectorAll at mount).
If children render asynchronous content (e.g., a list that loads after the dialog opens),
those elements will not be included in the Tab cycle. The query is not re-run on content
change. Standard pattern: use a MutationObserver or re-query on every Tab keydown.

### DSF-W2-A11Y-02 (P3) — Toast dismiss button is 24x24px, below 44px minimum
Toast.tsx:99-103: close button has minWidth:24, minHeight:24. This is below the WCAG 2.5.5
44px target. The button carries only 4px of effective hit area margin (the icon is ~14px).
In tablet mode, tablet-mode.css would raise it to 48px via the body.tablet-mode button rule,
but the inline style minHeight:24 wins over the stylesheet rule, neutralizing the override
(same mechanism as the eye-toggle defect above — inline styles defeat tablet-mode.css).

### DSF-W2-A11Y-03 (P3) — Sidebar.tsx carries no ARIA landmark role
Sidebar.tsx renders <aside> which is a valid landmark, but the element has no aria-label.
Multiple asides on a page are indistinguishable to screen readers without a label.
Required: aria-label prop (e.g., aria-label="Navigation" or passed in by consumer).

### DSF-W2-A11Y-04 (P3) — Dropdown.tsx missing motion transition
Dropdown.tsx uses a native <select> — no custom motion, which is correct for a native widget.
However, the component imports `radius` and `spacing` tokens but NOT `motion`. If this component
is later migrated to a custom listbox (required for full styling control), motion must be added.
Not a defect today; flag for future migration.

### DSF-W2-CSS-02 (P3) — tab role in tablet-mode.css uses 44px, not --touch-target-min
tablet-mode.css:107: body.tablet-mode [role="tab"] sets min-height: 44px (literal).
All other tablet-mode rules use var(--touch-target-min, 48px). The tab target floor is 4px
lower than every other interactive element. Should be var(--touch-target-min, 48px).

## VERIFIED CORRECT (Wave 1 finding disputed or nuanced)

### Token pipeline is clean and complete
tokens.ts exports: colors (12 values), typography (families/scale/weight), spacing (6 steps),
radius (4 steps), shadows (3 steps), motion (3 steps). All 8 components import from "../tokens".
styles.css mirrors all values exactly (verified line by line — all 12 color hex values match).
Token pipeline: tokens.ts -> component inline styles + CSS custom props in styles.css.
There is no generated CSS-from-tokens step; parity is maintained manually. Risk: drift over time.

### 48px touch targets ARE implemented in tablet-mode.css
tablet-mode.css:12 sets --touch-target-min: 48px. Line 25 applies it to button, [role="button"],
a, checkbox, radio, select. Wave 1 claim that "eye-toggle 44px touch target was removed" is
partially correct — the eye-toggle exists but uses minHeight:"unset" to opt out of enforcement.

### Dialog focus trap is functional for static content
Dialog.tsx:30-55: focus trap implementation is correct for static children (first/last element
wrap, Escape closes, initial focus on first element). The bug (DSF-W2-A11Y-01) applies only
to async content scenarios.

### OSK VirtualKeyboard token consumption is complete
tablet-mode.tsx imports { colors, radius, spacing, typography } from "@ds/tokens" and uses
all four throughout Key and VirtualKeyboard components. All 44px key heights are inline
(line 169, 211) consistent with Button.tsx pattern.
