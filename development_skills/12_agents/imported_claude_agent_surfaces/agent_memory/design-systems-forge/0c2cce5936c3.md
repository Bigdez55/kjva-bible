## SECTION 3 — TOP 5 ELECTRON FEATURES MISSING FROM XFRAME C

**Feature 1: Lock button in TopBar**
The Electron TopBar renders a "Lock" button with onClick=onLock and aria-label="Lock". The XFRAME topbar.c has no lock action element. The lockscreen exists in XSHELL but is only accessible via the shell's keyboard shortcut path, not from a visible topbar control. Users expecting a visible lock button cannot find one.
Fix: Add a lockscreen_btn node to xshell_topbar_t, wire xshell_lockscreen_lock() as its action.

**Feature 2: Animated entry/exit for Dock and Launcher**
The Electron layer would use CSS `transition` for dock slide-in/out and launcher opacity fade. The XFRAME dock uses binary `xframe_node_set_visible()` — instant show/hide. The launcher similarly. The XCOMP animation engine (`xanim_start`, `xanim_spring`) exists and is production-ready but is never called by any shell component. XTOKEN_DURATION_NORMAL=220ms defines the intended transition speed; it goes unused in all Sprint 5 shell code.
Fix: Call xanim_start(dock_surface, XANIM_PROP_Y, ..., XTOKEN_DURATION_NORMAL, XANIM_EASE_OUT_CUBIC) on show/hide.

**Feature 3: WiFi SSID display**
The Electron TopBar renders: `WiFi: {state.wifi.ssid}` — showing the actual network name. The XFRAME topbar shows only "WiFi" or "No WiFi". The XSHELL struct already tracks wifi_rssi and wifi_connected; it does not store the SSID string. No SSID field exists in xshell_topbar_t.
Fix: Add char wifi_ssid[64] to xshell_topbar_t; update tb_format_wifi() to use it.

**Feature 4: Volume level display**
The Electron TopBar shows `Volume: {state.volume.level}%`. The XFRAME topbar renders the static string "VOL" with no numeric level. No volume_level field exists in xshell_topbar_t; no xshell_topbar_set_volume() function is declared.
Fix: Add uint32_t volume_level to xshell_topbar_t; add set_volume() API; update text formatting.

**Feature 5: Event dispatch from Dock and Launcher tiles**
The Electron Dock calls `onLaunch(app.id)` when a dock button is clicked. The Electron Launcher calls `onLaunch(app.id)` when a tile is clicked. In XFRAME C, xshell_dock_pin_app() and the launcher tile builder do not register any click handlers — no xframe event listener or callback is attached to dock icon cells or launcher tiles. Users can see the UI but cannot launch apps through it.
Fix: Add xshell_launch_fn callback to xshell_dock_t and xshell_launcher_t structs; wire xframe node event registration when XFRAME adds pointer event support.

## SECTION 4 — DARK MODE / THEME SYSTEM

**Status: Dark-only. No theme-switching architecture exists.**

- design_tokens.h defines a single fixed dark palette. There is no `XTOKEN_COLOR_*_DARK` / `XTOKEN_COLOR_*_LIGHT` split.
- No theme context variable. No runtime token override mechanism.
- Sprint 3's design_tokens_net.h added `XTOKEN_COLOR_NET_*_LIGHT` variants as a workaround for the greeter/wizard surface (which appears on a lighter background). This is an ad-hoc pattern, not a systematic solution.
- Shell components (topbar, dock, launcher, Orange suite) are all hardwired to the dark palette.
- The Electron JS layer uses CSS class-based theming (implied by the design brief) but the compiled JS components show no theme-switching logic in the audited files.

**Risk:** If a light mode or high-contrast accessibility mode is required, there is no infrastructure to implement it without touching every component individually. This is a significant architectural gap.

## SECTION 5 — MOTION DESIGN

### 5.1 Animation Engine Capabilities

animation.c implements:
- 8 easing functions: linear, in/out quad, in/out cubic, decelerate (OUT_CUBIC), accelerate (IN_CUBIC)
- Spring physics: semi-implicit Euler with settle detection
- Per-surface property animation: X, Y, Z, opacity, scale_X, scale_Y
- Frame-rate independent: nanosecond timestamps
- Pool: 256 animation slots

### 5.2 Duration Token Cross-Validation

animation.h documents:
```
XANIM_DURATION_FAST   = 120ms
XANIM_DURATION_NORMAL = 220ms
XANIM_DURATION_SLOW   = 340ms
```

design_tokens.h defines:
```
XTOKEN_DURATION_FAST   = 120ms
XTOKEN_DURATION_NORMAL = 220ms
XTOKEN_DURATION_SLOW   = 340ms
```

Values MATCH. But animation.h defines its own constants rather than `#include "design_tokens.h"` and using `XTOKEN_DURATION_*` directly. This is a minor coupling gap — if the design token values ever change, animation.h must be updated separately.

### 5.3 Shell Animation Usage

No shell component calls xanim_start() or xanim_spring():
- topbar.c: no animations
- dock.c: binary visibility toggle only (no slide animation despite being described as "slides out")
- launcher.c: binary visibility toggle (no fade or scale entry)
- notes.c: no animations
- calendar.c: no animations

### 5.4 Reduced-Motion Support

No shell component checks a `prefers_reduced_motion` flag. The XFRAME widget layer has a `prefers_reduced_motion` field in widget structs (documented in sprint3 audit). Shell-level components do not inherit or propagate this. The dock's auto-hide comment in the source reads:
```
/* model slide with opacity/visibility for Sprint 5 (no animation engine yet) */
```
This indicates the animation gap was anticipated but not resolved.

## SECTION 6 — VISUAL HIERARCHY & CONSISTENCY

### 6.1 Elevation Model

GEN.OS uses a three-layer color depth model:
```
BG_PRIMARY   (0xFF0A0A1A) — deepest / desktop layer
BG_SECONDARY (0xFF1A1A2E) — panels, topbar, dock, sidebar
BG_TERTIARY  (0xFF2A2A3EU) — cards, inputs, icon backgrounds, dropdown
```

This model is consistently applied:
- Topbar: BG_SECONDARY — correct (sits above desktop)
- Dock: BG_SECONDARY (container) + BG_TERTIARY (icons) — correct (two levels)
- Launcher: SCRIM overlay + BG_TERTIARY (search/tiles) — correct (highest visual layer)
- Orange Notes: BG_PRIMARY (root) + BG_SECONDARY (toolbar/sidebar) + BG_TERTIARY (selected item) — correct
- Orange Calendar: BG_PRIMARY (root) + BG_SECONDARY (nav/header) — correct

**No shadow tokens exist.** Floating elements (dock, launcher, context menu) rely solely on background color contrast for visual separation. The dock sits on BG_SECONDARY but the desktop behind it is BG_PRIMARY — only a 10/10/20 RGB difference. No drop shadow separates them visually.

### 6.2 Border Radius Consistency

| Component | Radius Used | Correct? |
|-----------|-------------|----------|
| button.c (standalone) | XTOKEN_RADIUS_BTN = MD = 8px | Correct |
| dialog.c (card) | XTOKEN_RADIUS_CARD = LG = 12px | Correct |
| Dock icon background | XTOKEN_RADIUS_MD = 8px | Correct |
| Dock root container | XTOKEN_RADIUS_LG = 12px | Correct |
| Launcher search bar | XTOKEN_RADIUS_FULL = 9999px | Correct |
| Launcher tile | XTOKEN_RADIUS_LG = 12px | Correct |
| Launcher icon box | XTOKEN_RADIUS_MD = 8px | Correct |
| Notes toolbar buttons | XTOKEN_RADIUS_SM = 4px | INCONSISTENT — should be XTOKEN_RADIUS_BTN (8px) |
| Calendar nav buttons | XTOKEN_RADIUS_SM = 4px | INCONSISTENT — should be XTOKEN_RADIUS_BTN (8px) |

All Orange Suite buttons use XTOKEN_RADIUS_SM (4px) instead of XTOKEN_RADIUS_BTN (8px). This creates a visual discrepancy: buttons inside Orange apps look sharper than the system button widget.

## FINDINGS — DSF FINDING REGISTER

### P1 Findings (Must fix before Sprint 6 shell stabilisation)

**DSF-SHELL-01: P1 — No event dispatch from Dock or Launcher tiles**
Components: dock.c, launcher.c
Gap: Dock icons and launcher tiles are inert XFRAME_NODE_BOX nodes. No pointer event handlers are registered. Users cannot launch applications through the dock or launcher. The Electron JS layer has onLaunch(app.id) callbacks.
Fix: When XFRAME adds pointer event registration, wire a launch callback. Interim: add xshell_dock_launch_fn and xshell_launcher_launch_fn function pointer fields to the respective structs, and document the wiring point.

**DSF-SHELL-02: P1 — No interactive states on dock icons or launcher tiles**
Components: dock.c, launcher.c
Gap: No hover, press, or focus states. Users get zero visual feedback when hovering over or pressing dock/launcher elements. The reference button.c implements all five states correctly.
Fix: Register pointer-enter/leave/down/up events on each icon cell and tile node; apply XTOKEN_COLOR_ACCENT_HOVER/ACTIVE on state change.

**DSF-SHELL-03: P1 — Dock auto-hide uses binary toggle, not animation**
Component: dock.c
Gap: Despite the Sprint 5 comment noting animation was pending, the dock still uses instant visibility toggle. XTOKEN_DURATION_NORMAL=220ms and the full animation engine exist. A 220ms ease-out-cubic Y-slide would match the Electron CSS transition expectation.
Fix: Replace `xframe_node_set_visible(dock->root, 0/1)` with `xanim_start(dock_surface, XANIM_PROP_Y, show_y, hide_y, XTOKEN_DURATION_NORMAL, XANIM_EASE_OUT_CUBIC)`.

### P2 Findings (Should fix within Sprint 6)

**DSF-TOPBAR-01: P2 — Volume indicator shows static "VOL" not numeric level**
Component: topbar.c
Gap: XFRAME shows "VOL"; Electron shows "Volume: 75%". No volume_level field in xshell_topbar_t. No set_volume API.
Fix: Add uint32_t volume_level to xshell_topbar_t; add xshell_topbar_set_volume() API; update volume_node text formatting.

**DSF-TOPBAR-02: P2 — WiFi shows "WiFi"/"No WiFi" not SSID**
Component: topbar.c
Gap: xshell_topbar_t has no SSID field. tb_format_wifi() ignores the SSID. Electron shows the actual network name.
Fix: Add char wifi_ssid[64] to xshell_topbar_t; pass SSID through set_wifi() API; render as "SSID" or "No WiFi".

**DSF-TOPBAR-03: P2 — No Lock button in topbar**
Component: topbar.c
Gap: The Electron TopBar has a visible Lock button. XFRAME has no lock affordance in the topbar.
Fix: Add xframe_node_t *lock_btn node to xshell_topbar_t; wire to xshell_lockscreen_lock() via callback.

**DSF-ORANGE-01: P2 — Orange Suite buttons use XTOKEN_RADIUS_SM (4px) not XTOKEN_RADIUS_BTN (8px)**
Components: notes.c, calendar.c
Gap: In-app toolbar and nav buttons are visually sharper than system buttons. Breaks visual consistency across the design system.
Fix: Replace `XTOKEN_RADIUS_SM` with `XTOKEN_RADIUS_BTN` on all in-app button nodes.

**DSF-ORANGE-02: P2 — Calendar event highlight is a raw ARGB literal**
Component: calendar.c
Gap: `0x20FFFFFF` (8% white) used for event cell highlight. Not in design_tokens.h. Changing the event indicator color requires editing calendar.c directly rather than the token file.
Fix: Add `XTOKEN_COLOR_CAL_EVENT_HIGHLIGHT 0x20FFFFFFU` to design_tokens.h (or design_tokens_cal.h following Sprint 3 extension pattern).

**DSF-TOK-01: P2 — animation.h defines its own XANIM_DURATION_* instead of consuming XTOKEN_DURATION_***
Component: display/xcomp/animation.h
Gap: animation.h documents 120/220/340ms constants that MATCH design_tokens.h values but do not reference them. If design token durations are ever adjusted, animation.h would silently diverge.
Fix: `#include "design_tokens.h"` in animation.h; `#define XANIM_DURATION_FAST XTOKEN_DURATION_FAST` etc.

**DSF-TOK-02: P2 — Spacing tokens unused in all Sprint 5 shell components**
Components: topbar.c, dock.c, launcher.c, notes.c, calendar.c
Gap: XTOKEN_SPACE_SM/MD/LG exist and match the raw float values used (8/16/24). Shell components pass raw floats to xframe_node_set_padding/gap instead of casting the tokens.
Fix: Replace all raw float spacing literals with `(float)XTOKEN_SPACE_*` casts throughout Sprint 5 shell and Orange code.

**DSF-TOK-03: P2 — No XTOKEN_TOPBAR_HEIGHT or XTOKEN_NAVBAR_HEIGHT token**
Components: topbar.c, dock.c (via XSHELL_DOCK_HEIGHT), notes.c toolbar, calendar.c nav bar
Gap: 48px is the topbar height used across topbar.c, notes.c, and calendar.c, but there is no XTOKEN_TOPBAR_HEIGHT. If the height changes, every component must be edited individually. XSHELL_DOCK_HEIGHT=80 exists in xshell.h (shell layer) but not in design_tokens.h (design system layer).
Fix: Add `XTOKEN_TOPBAR_HEIGHT 48` and `XTOKEN_DOCK_HEIGHT 80` to design_tokens.h.

### P3 Findings (Should fix in Sprint 7 or when related feature is implemented)

**DSF-ANIM-01: P3 — prefers_reduced_motion not checked in any shell component**
Components: dock.c, launcher.c
Gap: The XFRAME widget layer supports a prefers_reduced_motion flag. Shell components do not propagate or check it. Users with vestibular disorders get no accommodation.
Fix: Add a global xshell_reduced_motion flag; check it before starting any animation; skip to final state immediately if set.

**DSF-THEME-01: P3 — Design system has no light-mode theme architecture**
All components
Gap: design_tokens.h is dark-only. No runtime theme-switching mechanism. Sprint 3 added ad-hoc _LIGHT variants for net tokens only.
Fix: Architect a token split: XTOKEN_COLOR_*_DARK vs XTOKEN_COLOR_*_LIGHT per category; add a global xtoken_theme_t selector; generate both sets at compile time.

**DSF-VISUAL-01: P3 — No shadow/elevation tokens**
All floating components
Gap: design_tokens.h has no XTOKEN_SHADOW_* tokens. Floating elements (dock, launcher, context menu) rely solely on background color contrast. On displays with narrow gamut or in accessibility high-contrast modes, the elevation distinction may be insufficient.
Fix: Add XTOKEN_SHADOW_SM/MD/LG as shadow descriptor tokens; implement shadow rendering in XCOMP blend pass.

**DSF-CLOCK-01: P3 — Topbar clock uses epoch-relative time not real-time**
Component: topbar.c
Gap: tb_format_clock() computes HH:MM from ms-since-boot. The displayed clock starts at "00:00" on boot. The Electron TopBar uses `new Date().toLocaleTimeString()` for actual system time. The XFRAME clock is not a real clock.
Fix: Use pal_rtc_read() (or equivalent real-time clock PAL API) rather than the monotonic tick counter.

**DSF-ORANGE-03: P3 — Calendar 0.5px border sub-pixel (below XTOKEN_BORDER_WIDTH_SM=1)**
Component: calendar.c
Gap: `xframe_node_set_border_width(cell, 0.5f)` is thinner than the minimum defined token value. On low-DPI displays this will alias to 0 or 1 unpredictably.
Fix: Use `(float)XTOKEN_BORDER_WIDTH_SM` (1px); optionally add `XTOKEN_BORDER_WIDTH_XS 1` if sub-pixel intent is meaningful.

**DSF-A11Y-01: P3 — No ARIA equivalents in XFRAME C shell components**
Components: topbar.c, dock.c, launcher.c
Gap: The Electron JS components have aria-label on every interactive element and semantic container elements (nav, header, section). The XFRAME C shell has no equivalent — xframe_node_t has an aria_label[] field (from Sprint 3 widget pattern) but it is never set in shell components.
Fix: Set aria_label on all xshell shell nodes using xframe_node_set_aria_label(); ensure screen reader event pipeline is wired.
