## 4. ANIMATION SYSTEM

### Engine capabilities (animation.c/h)

Two animation types:
1. `xanim_start()` — easing animation with 9 curves (linear, in/out quad, in/out cubic,
   in-out-cubic, decelerate=OUT_CUBIC, accelerate=IN_CUBIC)
2. `xanim_spring()` — semi-implicit Euler spring physics (stiffness, damping)

Animatable surface properties (6):
- XANIM_PROP_X, _Y, _Z (position/layer)
- XANIM_PROP_OPACITY
- XANIM_PROP_SCALE_X, _Y

Pool: 256 slots, spinlock-protected, framerate-independent via pal_time_now_ns().
Loop support: flag on easing animations.
Settle detection: velocity < 0.5 AND displacement < 0.5.
Max timestep: 50ms (spring explosion guard on wake-from-sleep).

Duration constants in animation.h:
```c
XANIM_DURATION_FAST   = 120u
XANIM_DURATION_NORMAL = 220u
XANIM_DURATION_SLOW   = 340u
```
These MATCH XTOKEN_DURATION_* values but are DEFINED INDEPENDENTLY.
Finding DSF-TOK-01 (open from Sprint 5 audit): still unresolved.

### Animation engine usage gap

`xanim_start()` / `xanim_spring()` are defined only in animation.c (self-references).
Zero calls exist in any widget, shell component, or Orange app.
The animation engine is entirely unused by all UI code.

Finding: DSF-ANIM-01 (CRITICAL) — 100% of animation engine capability is dead code
from the UI layer perspective. All dock show/hide, launcher open/close, modal entry/exit,
and toast appearance/dismissal are instantaneous (no animation).

### prefers_reduced_motion

No prefers_reduced_motion check in any C source file (confirmed via grep).
Finding DSF-ANIM-02 (P3): no accessibility motion preference support.

## 5. FONT RENDERING SYSTEM (fonts.c/h)

### Capabilities

- Parser: TrueType/OpenType sfnt container (offset table + table directory)
- Required tables: head, hhea, maxp, cmap, loca, glyf, hmtx
- Cmap: Format 4 BMP (Unicode BMP plane, codepoints 0x0000–0xFFFF)
- Outline: Quadratic bezier splines (TrueType format), simple glyphs only
- Rasterizer: 4x vertical supersampling scanline conversion, non-zero winding rule
- Cache: 256-slot LRU, keyed on (font_id, codepoint, size_px), 4096B max per entry
- Compositing: Coverage × source alpha → SRC_OVER to ARGB8888 destination
- API: xfont_load_mem (in-memory), xfont_set_size, xfont_rasterize_glyph,
       xfont_render_string (UTF-8), xfont_measure_string, xfont_free, xfont_shutdown

### Gaps

| Gap ID       | Description                                              | Severity |
|--------------|----------------------------------------------------------|----------|
| DSF-FONT-01  | xfont_load() is a stub — file I/O requires VFS (not yet wired). Only xfont_load_mem() works. | P2 |
| DSF-FONT-02  | Compound glyph support missing (n_cont < 0 → return -1). Composite characters (accented, ligatures) fall through to fallback glyph. | P2 |
| DSF-FONT-03  | cmap Format 4 only — no Format 12 (full Unicode, Supplementary Multilingual Plane). Emoji (U+1F***) will all produce fallback rectangle. | P2 |
| DSF-FONT-04  | No subpixel (horizontal) antialiasing. Only 4x vertical supersampling. No LCD RGB subpixel rendering. | P3 |
| DSF-FONT-05  | No kerning (kern/GPOS tables not parsed). Inter-character spacing is advance-width only. | P3 |
| DSF-FONT-06  | No OpenType layout (GSUB/GPOS) — no ligatures, no contextual substitution, no Arabic shaping. | P3 |
| DSF-FONT-07  | No bold/italic: font weight and style are not tracked. XTOKEN_FONT_WEIGHT_* tokens missing. No synthesized bold (stroke expansion) or italic (skew transform). | P2 |
| DSF-FONT-08  | Arena is a bump allocator — freed fonts do not reclaim arena space. 4MB total arena for all font data. | P3 |
| DSF-FONT-09  | No right-to-left (BiDi) support. Text always rendered left-to-right. | P3 |
| DSF-FONT-10  | No CJK-specific rendering path. CJK codepoints work if the loaded font covers them via cmap4, but there is no fallback font chain for CJK coverage. | P3 |

### Unicode coverage assessment

Coverage depends entirely on the TTF file loaded at runtime via xfont_load_mem().
The engine supports:
- ASCII (U+0020–U+007F): Full coverage if font provides it
- Latin Extended (U+0080–U+024F): Coverage depends on font
- BMP (U+0000–U+FFFF): Covered IF font has cmap Format 4 entries AND simple glyph data
- SMP (U+10000+): NOT covered — cmap4 lookup returns 0 for > 0xFFFF codepoints
- Emoji: NOT covered (all SMP)
- Compound glyphs: NOT covered (fallback rectangle rendered)

## 6. VECTOR GRAPHICS SYSTEM (vector.c/h)

### Capabilities

Path construction:
- move_to, line_to, quad_to (quadratic bezier), cubic_to (cubic bezier), close
- Pre-built shapes: rect, rounded_rect, circle, ellipse

Rendering:
- xvec_fill: non-zero winding rule, 4x vertical supersampling AA
- xvec_stroke: outline expansion with configurable cap (BUTT/ROUND/SQUARE) and join (MITER/ROUND/BEVEL)
- Miter limit clamping on MITER joins
- SRC_OVER compositing to ARGB8888 destination

Limits:
- Path pool: 16 paths (XVEC_PATH_MAX)
- Commands per path: 4096 (XVEC_CMD_MAX)
- Coordinates per path: 8192 (XVEC_COORD_MAX)
- Flatten tolerance: 0.25 pixels (XVEC_FLATTEN_TOLERANCE)

Algorithms:
- Flattening: De Casteljau recursive subdivision (O(n × log(1/tol)) per curve)
- Fill: edge-list scanline, 4x vertical supersampling
- Stroke: half-width perpendicular offset expansion

### Gaps

| Gap ID       | Description                                               | Severity |
|--------------|-----------------------------------------------------------|----------|
| DSF-VEC-01   | No gradient fill (linear, radial). All fills are flat color. | P2    |
| DSF-VEC-02   | No clip path / stencil operation. Overflow clipping uses XFRAME's overflow_hidden flag (bounding-box only), not vector clip shapes. | P2 |
| DSF-VEC-03   | No SVG parser — no way to load .svg files. Icons must be hardcoded as path commands or rasterized PNG. | P3 |
| DSF-VEC-04   | No dash pattern for strokes (dashed/dotted lines for focus rings, etc.). | P3 |
| DSF-VEC-05   | No transform matrix (translate, scale, rotate) on paths. All coordinates must be pre-transformed. | P3 |
| DSF-VEC-06   | Path pool of 16 is very small. Complex screens using xvec_path_* for multiple decorative elements could exhaust the pool simultaneously. | P2 |
| DSF-VEC-07   | No even-odd winding rule (only non-zero). Donuts/compound shapes with holes require careful point ordering. | P3 |

## 7. DESIGN GAP INVENTORY (G-SERIES)

### G-01: Shadow / Elevation System — MISSING

No shadow tokens (XTOKEN_SHADOW_*) in design_tokens.h.
No shadow drawing capability in vector.c (would require blur/spread).
XCOMP has no Gaussian blur or box-shadow implementation.
Cards, dialogs, toasts, and dropdowns all appear flat — no depth perception.
This makes the UI look 2D/flat rather than layered.

Impact: All elevation implied by background color difference only (BG_PRIMARY / BG_SECONDARY / BG_TERTIARY). Acceptable for minimal dark theme but insufficient for generative UI or dynamic surfaces.

### G-02: Font Weight / Style Hierarchy — MISSING

No bold, medium, or italic variants in the rendering pipeline.
XTOKEN_FONT_WEIGHT_* tokens do not exist.
The font rasterizer loads a single TTF and renders all text at that weight.
Title text, body text, captions, and labels all render at identical weight.

Impact: Typography has no visual hierarchy. Headings are not visually distinct from body text except through size difference.

### G-03: Light Mode Token Architecture — STILL MISSING (P3, from Sprint 5)

design_tokens_net.h documents `_LIGHT` variants for greeter/wizard surfaces.
The base design_tokens.h has no light mode architecture.
No theme-switch mechanism exists in xframe_app_t or the compositor.
GEN.OS is dark-only.

### G-04: Interactive State Tokens — PARTIAL

XTOKEN_COLOR_ACCENT_HOVER and XTOKEN_COLOR_ACCENT_ACTIVE exist.
XTOKEN_COLOR_ERROR does not have hover/active variants.
No XTOKEN_OPACITY_HOVER, XTOKEN_OPACITY_PRESSED, XTOKEN_OPACITY_DISABLED tokens.
Button.c constructs hover tints manually (0x20000000 | ACCENT).

### G-05: Animation Integration — CRITICAL GAP

The XCOMP animation engine (xanim_start, xanim_spring) is feature-complete but
zero-connected to the XFRAME UI layer.
No widget, shell component, or app calls xanim_start() or xanim_spring().
All transitions are instantaneous: dock show/hide, launcher open, modals, toasts.
The motion design language promised by animation.h is entirely unrealized.

### G-06: Customization / Theme Override System — MISSING

No runtime theming: colors, spacing, and radii are compile-time constants.
No user accent color selection.
No font swap (different TTF at runtime).
No density mode (compact / default / spacious).

### G-07: Generative UI / Adaptive Layouts — MISSING

No dynamic layout adaptation beyond the XTOKEN_LAYOUT_* breakpoints in
design_tokens_net.h.
No grid layout (only flex row/column).
No responsive container query analog.
No AI-driven layout suggestions from XMIND → XFRAME path.
XMIND and XFRAME are entirely decoupled — no API surface for generative UI.

## 8. ACCESSIBILITY AUDIT

### WCAG 2.1 AA status

Color contrast:
- design_tokens_net.h verifies all 11 network/package tokens against BG_SECONDARY
  (luminance 0.012). All pass at ≥ 4.5:1.
- Base design_tokens.h has no inline contrast verification for core tokens.
- XTOKEN_COLOR_TEXT_SECONDARY (0xB0B0C0) on XTOKEN_COLOR_BG_PRIMARY (0x0A0A1A):
  approximate contrast ~10:1 — PASS.
- XTOKEN_COLOR_TEXT_DISABLED (0x606070) on BG_PRIMARY: approximate contrast ~2.5:1 — FAIL (4.5:1 required).
  Finding DSF-A11Y-02 (P1): disabled text fails WCAG contrast.

Keyboard navigation:
- dialog.c has keyboard focus trap while open.
- button.c has on_focus_in/on_focus_out/on_key handlers (Enter/Space).
- dropdown.c has keyboard navigation with highlighted item tracking.
- scroll.c has NO keyboard scroll handler (confirmed open from Sprint 5).
- No Tab-order management across widgets (no tabindex analog).

ARIA/labels:
- xframe_node_t has no aria_label, role, or state fields.
- No shell or app component sets an ARIA label.
- Screen reader support: zero.
- Finding DSF-A11Y-01: open since Sprint 5.

Focus ring:
- XTOKEN_COLOR_FOCUS_RING and XTOKEN_BORDER_WIDTH_FOCUS exist.
- No visual focus ring drawing in any widget's paint code (unverified — paint.c
  not fully read, but no widget explicitly paints a focus ring).
