## SECTION 1 — XSEC ERROR CODE CATALOG

### Full xsec_status_t table (xsec.h §1)

| Code | Value | Description |
|------|-------|-------------|
| XSEC_OK | 0 | Success |
| XSEC_ERR_INVAL | -1 | Invalid argument |
| XSEC_ERR_NOMEM | -2 | Buffer too small |
| XSEC_ERR_AUTH_FAIL | -3 | Authentication tag mismatch |
| XSEC_ERR_VERIFY_FAIL | -4 | Signature verification failed |
| XSEC_ERR_CERT_EXPIRED | -5 | Certificate validity window exceeded |
| XSEC_ERR_CERT_INVALID | -6 | Certificate parse or chain error |
| XSEC_ERR_TLS_HANDSHAKE | -7 | TLS handshake failed |
| XSEC_ERR_TLS_RECORD | -8 | TLS record layer error |
| XSEC_ERR_TLS_ALERT | -9 | TLS fatal alert received |
| XSEC_ERR_NOT_SUPPORTED | -10 | Algorithm or feature not supported |
| XSEC_ERR_IO | -11 | Transport send/recv failure |
| XSEC_ERR_STATE | -12 | Invalid state machine transition |
| XSEC_ERR_OVERFLOW | -13 | Buffer overflow prevented |

### Error code consistency findings

STRENGTH: The set covers all major failure categories for a crypto+TLS library at this scope.
No integer gaps exist (0 to -13 contiguous). No duplicated values.

FINDING DSF-TOK-02 (P3): Naming inconsistency between semantic families.
- Generic errors use bare noun form: XSEC_ERR_INVAL, XSEC_ERR_NOMEM, XSEC_ERR_IO, XSEC_ERR_STATE, XSEC_ERR_OVERFLOW, XSEC_ERR_NOT_SUPPORTED.
- Crypto-domain errors use _FAIL suffix: XSEC_ERR_AUTH_FAIL, XSEC_ERR_VERIFY_FAIL.
- Cert errors use noun + state compound: XSEC_ERR_CERT_EXPIRED, XSEC_ERR_CERT_INVALID.
- TLS errors use TLS_ subsystem + noun: XSEC_ERR_TLS_HANDSHAKE, XSEC_ERR_TLS_RECORD, XSEC_ERR_TLS_ALERT.

The _FAIL suffix is inconsistent: XSEC_ERR_AUTH_FAIL and XSEC_ERR_VERIFY_FAIL are the only
two codes that use this pattern. All other failures are named without the _FAIL suffix. This
is a minor naming drift, not a functional problem.

FINDING DSF-TOK-03 (P2): Missing XSEC_ERR_TLS_DECODE_ERROR.

tls13.c uses XSEC_ERR_TLS_HANDSHAKE for at least two structurally distinct failure
categories that the RFC 8446 protocol itself distinguishes:

  Category A — Decode failures (malformed message bytes):
    Line 881:  pt_len < 4+2+2 (too short for header+alg+sig_len fields)
    Line 893:  cv_pos + sig_len_cv > pt_len (sig extends past message end)
    Line 927:  sig_len_cv != 64 (Ed25519 signature has wrong byte count)
    These all trigger TLS_ALERT_DECODE_ERROR alerts to the peer.

  Category B — Cryptographic / state failures:
    Line 688:  Received wrong message type (unexpected message)
    Line 711:  Unsupported cipher suite
    Line 939:  xsec_ed25519_verify() returned non-OK
    Line 983:  ServerFinished HMAC mismatch
    These represent actual protocol or crypto failures.

Currently both categories collapse into XSEC_ERR_TLS_HANDSHAKE (-7). This means:
  - A caller cannot distinguish "message was malformed/truncated" from "crypto verification
    failed" without inspecting the audit log.
  - When XNET wraps XSEC-TLS, it cannot map decode errors to appropriate XNET error codes.
  - The RFC 8446 design already provides this distinction via separate alert types
    (decode_error vs. bad_certificate vs. handshake_failure).

RECOMMENDED ADDITION:
  #define XSEC_ERR_TLS_DECODE  -14  /* TLS message malformed or truncated */

With this change:
  - All paths that send TLS_ALERT_DECODE_ERROR return XSEC_ERR_TLS_DECODE.
  - XSEC_ERR_TLS_HANDSHAKE is reserved for state-machine and crypto verification failures.
  - XSEC_ERR_TLS_RECORD (-8) continues to cover record-layer framing errors (wrong content
    type byte, length out of range at the wire layer before handshake message parsing).

Score: 7/10 (strong coverage, contiguous values, but two naming inconsistencies and one
missing decode-vs-handshake distinction identified).

## SECTION 2 — FUNCTION NAMING CONVENTION AUDIT

### Pattern inventory across xsec.h

Observed naming convention:
  xsec_{algorithm}_{verb}[_{qualifier}]()

All public functions were catalogued against this template:

SHA-256 family:
  xsec_sha256_init / xsec_sha256_update / xsec_sha256_final / xsec_sha256
  xsec_sha256_selftest
  Pattern: CONSISTENT. Streaming trio (init/update/final) + one-shot + selftest.

SHA-512 family:
  xsec_sha512_init / xsec_sha512_update / xsec_sha512_final / xsec_sha512
  Pattern: CONSISTENT with SHA-256. No selftest for SHA-512 — acceptable (SHA-256 drives
  HMAC/TLS tests; SHA-512 is internal to Ed25519).

HMAC-SHA-256 family:
  xsec_hmac_sha256_init / xsec_hmac_sha256_update / xsec_hmac_sha256_final
  xsec_hmac_sha256
  xsec_hkdf_extract / xsec_hkdf_expand / xsec_hkdf_expand_label
  Pattern: CONSISTENT for streaming trio. HKDF functions use xsec_hkdf_ prefix
  (algorithm name, not subsystem), which is correct given RFC 5869 defines HKDF as
  a distinct construction.

AES-GCM family:
  xsec_aes256_key_expand
  xsec_aes_gcm_encrypt / xsec_aes_gcm_decrypt
  Pattern: NOTE — the key expansion function uses xsec_aes256_ prefix (includes bit width)
  while encrypt/decrypt use xsec_aes_gcm_ (drops the 256). This is a minor inconsistency:
  either both should carry the 256 qualifier or neither should.
  FINDING DSF-NAME-01 (P3): xsec_aes256_key_expand vs. xsec_aes_gcm_encrypt prefix mismatch.

ChaCha20-Poly1305 family:
  xsec_chacha20_poly1305_encrypt / xsec_chacha20_poly1305_decrypt
  Pattern: CONSISTENT. Full algorithm name in prefix.

CONSISTENCY BETWEEN AES-GCM and ChaCha20-Poly1305:
  xsec_aes_gcm_encrypt(key, nonce, aad, aad_len, pt, pt_len, ct, tag)
  xsec_chacha20_poly1305_encrypt(key, nonce, aad, aad_len, pt, pt_len, ct, tag)
  Argument order: IDENTICAL. Buffer ownership: IDENTICAL. Return type: IDENTICAL.
  Assessment: FULLY CONSISTENT. The AEAD API pair is one of the strongest consistency
  achievements in the Sprint 3 design.

X25519 family:
  xsec_x25519_keypair / xsec_x25519_shared
  Pattern: CONSISTENT. Keypair generation + DH step.

Ed25519 family:
  xsec_ed25519_keypair / xsec_ed25519_sign / xsec_ed25519_verify
  Pattern: CONSISTENT. Three functions, three distinct operations.

Ed25519 vs. X25519 symmetry:
  Both use xsec_{algorithm}_keypair() with out_priv / out_pub output params.
  Both follow the established out_* naming for output parameters.
  Assessment: CONSISTENT.

TLS 1.3 family:
  xsec_tls_init / xsec_tls_handshake / xsec_tls_send / xsec_tls_recv / xsec_tls_close
  Pattern: CONSISTENT. Clean lifecycle: init → handshake → {send/recv}* → close.

X.509 family:
  xsec_x509_parse / xsec_x509_verify_chain / xsec_x509_verify_hostname
  Pattern: CONSISTENT within the X.509 group.

FINDING DSF-NAME-02 (P3): xsec_x509_verify_hostname is semantically a validation, not a
verification. In the codebase, verify_ prefix is consistently used for cryptographic
operations (signature verification, chain verification). A hostname check is a string
comparison, not a crypto operation. A more precise name would be
xsec_x509_match_hostname() or xsec_x509_check_hostname(). This is a minor semantic drift
rather than a consistency violation, but worth flagging for Sprint 4 when the X.509 API
may be extended.

Audit log family:
  xsec_audit_init / xsec_audit_log / xsec_audit_query_recent / xsec_audit_dump
  xsec_audit_get_dropped
  Pattern: CONSISTENT. Note: xsec_audit_get_dropped returns uint64_t (not xsec_status_t)
  which is correct — it's a query, not an operation.

## SECTION 3 — XNET-TLS INTEGRATION API DESIGN

### Current state

xnet.h defines no XSEC-facing or TLS-facing API surface. There is no:
  - xnet_tls_connect() / xnet_tls_accept()
  - xnet_tls_context_t wrapping xsec_tls_ctx_t + a TCP socket descriptor
  - Any adapter between xnet_tcp_send/recv and xsec_tls_transport_t

The integration is left entirely to the application layer (or future shell integration).

### How integration currently must work

An application wanting TLS-over-TCP today must:
  1. Call xnet_tcp_connect() to get a socket descriptor (int sd).
  2. Manually define a xsec_tls_transport_t with send/recv callbacks that call
     xnet_tcp_send(sd, ...) and xnet_tcp_recv(sd, ...).
  3. Allocate and initialize xsec_tls_ctx_t on the stack or statically.
  4. Call xsec_tls_handshake() then xsec_tls_send/recv().

This works but the transport glue is boilerplate that every caller must repeat. The risk
is that callers will implement this incorrectly (wrong error mapping, forgotten hostname
verification, etc.).

### Design assessment

The xsec_tls_transport_t callback struct (send/recv + opaque ctx) is exactly the right
abstraction boundary. XSEC does not know about sockets. XNET does not know about TLS.
The layering is clean. This matches the Schannel / BoringSSL model where the TLS layer is
transport-agnostic.

### Recommendation — XNET-TLS glue layer (Sprint 5 / Shell integration)

A thin adapter module (not in xnet.h or xsec.h, but in a new file, e.g.,
net/xnet/tls_adapter.c + include/xnet_tls.h) should provide:

  typedef struct {
      int               tcp_sd;       /* XNET socket descriptor          */
      xsec_tls_ctx_t    tls;          /* XSEC TLS context (inline)        */
  } xnet_tls_conn_t;

  xnet_status_t xnet_tls_connect(xnet_ipv4_t ip, uint16_t port,
                                  const char *hostname,
                                  xnet_tls_conn_t *out_conn);

  xnet_status_t xnet_tls_send(xnet_tls_conn_t *conn,
                               const void *data, size_t len);

  xnet_status_t xnet_tls_recv(xnet_tls_conn_t *conn,
                               void *buf, size_t len, size_t *out_len);

  xnet_status_t xnet_tls_close(xnet_tls_conn_t *conn);

This adapter:
  - Lives in XNET (not XSEC) because it owns the socket lifecycle.
  - Calls xsec_x509_verify_hostname() automatically in xnet_tls_connect().
  - Maps xsec_status_t to xnet_status_t via a small translation table.
  - Provides the boilerplate xsec_tls_transport_t binding once, not per-caller.
  - Should NOT be in xnet.h core — put it in include/xnet_tls.h (opt-in include).

This is a Sprint 5 / Shell integration item. Do not add to xnet.h directly.

## SECTION 4 — PAL TIMER vs. EVENT API UNIFICATION ANALYSIS

### Current PAL event API (pal.h §7)

  pal_status_t pal_event_create(pal_handle_t *out_handle);
  pal_status_t pal_event_signal(pal_handle_t event_handle);
  pal_status_t pal_event_wait(pal_handle_t event_handle, uint64_t timeout_ns);
  pal_status_t pal_event_reset(pal_handle_t event_handle);

Operations: create / signal / wait / reset.
All operations take the handle directly. No intermediate config struct.

### Current PAL timer API (pal.h §8)

  pal_status_t pal_timer_create(pal_handle_t *out_handle);
  pal_status_t pal_timer_set(pal_handle_t timer_handle, uint64_t deadline_ns);
  pal_status_t pal_timer_cancel(pal_handle_t timer_handle);

Operations: create / set / cancel.
No wait operation — the timer presumably fires into an event or callback (not specified
in pal.h). The callback/event wiring is absent from the current API.

### Asymmetry analysis

The event API is self-contained: create, signal, wait, reset are all you need.
The timer API is not: create + set + cancel, but where does the timer deliver its signal?
A timer that fires but delivers nothing is only useful for polling via pal_time_now_ns(),
which would be a busy-wait anti-pattern.

FINDING DSF-PAL-01 (P2): Timer delivery mechanism is unspecified.
pal.h §8 states "fires callback or signals event after deadline" in the comment, but the
API exposes no way to attach an event or callback. The create() function takes only
out_handle. There is no pal_timer_create_with_event() or pal_timer_create_with_fn().

### Unified "waitable object" abstraction — assessment

Windows NT uses a single WaitForSingleObject() that accepts handles of type: event,
mutex, semaphore, or timer. The POSIX timerfd model unifies timers and file descriptors.
Mach ports unify everything.

For GEN.OS PAL at this stage, a fully unified waitable object is premature — the handle
system already uses typed handles (PAL_OBJ_TIMER vs PAL_OBJ_EVENT) so the foundation
is present. The minimum viable fix is not a full abstraction merger but a targeted
timer delivery fix.

### Recommended fix — minimal, Sprint 4 scope

Option A (preferred): Timer-to-event binding.
  pal_status_t pal_timer_create(pal_handle_t event_handle,
                                 pal_handle_t *out_timer_handle);

  When the timer fires, it calls pal_event_signal(event_handle) automatically.
  Caller then waits on the event with pal_event_wait(). This is a single-line change
  to the create() signature and keeps both APIs orthogonal. The event handle may be
  PAL_HANDLE_INVALID to indicate fire-and-forget (audit log only).

Option B: Unified wait — add to pal.h:
  pal_status_t pal_wait_any(const pal_handle_t *handles, uint32_t count,
                             uint64_t timeout_ns, uint32_t *out_signaled_idx);

  This is more powerful but is a larger API surface addition. Defer to Sprint 5.

Option A is the recommended Sprint 4 path. It requires changing only the
pal_timer_create signature (which is a breaking change — assess blast radius: currently
used in pal_aether.c and any Sprint 3 test code that calls pal_timer_create).

## SECTION 5 — XPKG REFUSAL GATE API CONSISTENCY ASSESSMENT

### Gate public API (xpkg.h §24)

  xpkg_status_t xpkg_gate_check(const xpkg_package_t *pkg,
                                  const uint8_t *archive_buf,
                                  uint32_t archive_len,
                                  const xpkg_gate_config_t *config,
                                  xpkg_gate_report_t *out_report);

  bool xpkg_gate_all_pass(const xpkg_gate_report_t *report);

### Pattern compliance with xpkg conventions

1. Return type: xpkg_gate_check() returns xpkg_status_t. COMPLIANT with xpkg pattern
   (all XPKG operations return xpkg_status_t).

2. Error code: Returns XPKG_ERR_GATE (-6) on hard FAIL, XPKG_OK on WARN-only or
   full-pass. This is correct — WARN is advisory and does not change the return code.
   The distinction between PASS and WARN is communicated via out_report->all_pass, not
   the return value. This is a deliberate, well-documented design.

3. Output parameter naming: out_report follows the out_* convention. COMPLIANT.

4. NULL handling: `if (!pkg || !out_report) { return XPKG_ERR_INVAL; }` — correct.
   config == NULL is handled by substituting a default config (not treated as error).
   COMPLIANT — nullable config is a good ergonomic choice.

5. Internal gate result type (xpkg_gate_result_t): Defined cleanly with three values
   (PASS=0, WARN=1, FAIL=2). Used only inside the gate machinery. Not leaked through
   the xpkg_status_t return. COMPLIANT.

6. Struct-based report output: xpkg_gate_report_t containing an array of xpkg_gate_check_t
   structs. Each check contains gate id, result, and a 128-byte detail string. This is a
   richer result than a simple status code — appropriate for a multi-gate pipeline where
   the caller needs to display per-gate results to the user.

### Findings

FINDING DSF-PKG-01 (P3): all_pass field in xpkg_gate_report_t is redundant and
potentially misleading.
  - all_pass is set to false on any WARN, but the function still returns XPKG_OK.
  - The caller must check both return value (for hard FAIL) and all_pass (for WARNs).
  - Recommendation: replace all_pass with a uint32_t warn_count + uint32_t fail_count
    summary, which is unambiguous and allows callers to distinguish "0 warns, 0 fails"
    from "3 warns, 0 fails" without iterating the checks array. This is a low-priority
    ergonomic improvement.

FINDING DSF-PKG-02 (P3): xpkg_gate_config_t §16 comment says "Gate 7 SIZE limit" for
disk_quota_bytes but Gate 6 is SIZE (XPKG_GATE_SIZE = 6, zero-indexed). Comment is
off by one. Cosmetic but could mislead future editors.

FINDING DSF-PKG-03 (P3): §17 comment says "Policy Rules (Gate 9)" but gate 8 (XPKG_GATE_POLICY = 8,
zero-indexed) is the policy gate. 9-gate pipeline is 0-indexed. The section heading
should read "Gate 8" to match the enum value. Matches the pattern noted in previous
audits about gate numbering.

### xpkg_gate_check() internal structure assessment

The short-circuit on FAIL (fills remaining checks with "Not evaluated" and returns
XPKG_ERR_GATE immediately) is correct and matches the stated failure semantics in the
file header. The force[] flag demoting FAIL to WARN is correctly guarded by an audit
log message. The enabled[] flag skipping a gate entirely is correctly modeled as a
synthetic PASS.

Assessment: Gate API is well-structured. The result struct pattern is correct for this
use case. The inconsistencies are cosmetic (P3) only.

### §F ADR-API-001: Sprint 4 Binding API Naming Standard

These rules are MANDATORY for Sprint 4+ headers. They resolve the ambiguities documented
across both audit sessions (2026-03-05 and 2026-03-06).

#### F.1 Error code ordering (canonical)

```c
#define X_SS_OK              0
#define X_SS_ERR_NOMEM      -1   /* Buffer / pool exhausted          */
#define X_SS_ERR_INVAL      -2   /* Invalid argument                 */
#define X_SS_ERR_PERM       -3   /* Permission / capability denied   */
#define X_SS_ERR_BUSY       -4   /* Resource temporarily unavailable */
#define X_SS_ERR_TIMEOUT    -5   /* Operation timed out              */
#define X_SS_ERR_NOT_FOUND  -6   /* Object not found                 */
#define X_SS_ERR_EXISTS     -7   /* Object already exists            */
#define X_SS_ERR_IO         -8   /* I/O or transport error           */
#define X_SS_ERR_NOT_SUPPORTED -9 /* Feature not available           */
#define X_SS_ERR_OVERFLOW   -10  /* Buffer or value overflow         */
/* Domain-specific codes begin at -11 */
```

NOMEM always -1. INVAL always -2. Domain codes follow, never precede, the common block.

#### F.2 Enum member prefix rule

All enum members carry full subsystem prefix: `XNET_TCP_STATE_CLOSED` not `TCP_CLOSED`.
Exception: wire-format protocol constants (`ETHERTYPE_*`, `IPPROTO_*`, `ARP_*`, `ICMP_*`) follow
BSD convention and are exempt — but a block comment at the section header MUST document
the intentional exemption.

#### F.3 \_FAIL suffix ban

New error codes must not use `_FAIL` suffix. Use `ERR_AUTH` (not `ERR_AUTH_FAIL`).
Existing `XSEC_ERR_AUTH_FAIL`, `XSEC_ERR_VERIFY_FAIL` are grandfathered.

#### F.4 Lifecycle pair rule

Every subsystem MUST expose `x{ss}_init()` and `x{ss}_shutdown()` as public API.
`xsec_init()` / `xsec_shutdown()` must be added in Sprint 4.

#### F.5 \_input function return type rule

All `_input` functions MUST return `x{ss}_status_t`. void return on `_input` is banned.
Fix: `xnet_mdns_input` must return `xnet_status_t` before Sprint 5.

#### F.6 Output parameter naming

- Never bare `out` — always `out_{noun}`
- Bytes received: `out_recvd`
- Bytes sent: `out_sent`
- Output struct: `out_{type_noun}` (out\_pkg, out\_report, out\_node, out\_key)

#### F.7 Buffer naming within a module

Choose one name per buffer concept, apply uniformly within a module.
`archive_buf`/`archive_len` preferred over `buf`/`buf_len` when buffer has specific semantic.
