## FINDING CATALOGUE

### F-01 (P1) — XNET §24 socket API: bare `int` return type
Location: xnet.h lines 633–644

All 10 BSD-socket-style functions (xnet_socket, xnet_bind, xnet_listen,
xnet_accept, xnet_connect, xnet_send, xnet_recv, xnet_close,
xnet_setsockopt, xnet_getsockopt) return plain `int` instead of
`xnet_status_t`. Every other function in xnet.h returns `xnet_status_t`.

Impact: callers switching between the raw TCP/UDP API (xnet_tcp_connect,
xnet_udp_bind — both return xnet_status_t) and the socket API must
mentally context-switch between two error conventions. There is no
generic error path that handles both without a cast or wrapper. A
developer assigning the result of xnet_socket() to an xnet_status_t
variable gets a valid compile without warning but loses negative
distinction between sd=-1 (XNET_SD_INVALID, not an error code) and
sd=XNET_ERR_NOMEM=-1 (they are the same value on failure).

Recommended fix for Sprint 4: do NOT change the return type — POSIX
socket semantics require the fd/sd to be the return value for
xnet_socket/accept/connect. Instead, document the dual convention
explicitly in the header CONVENTIONS block and add a XNET_IS_ERROR(r)
macro that handles both (negative int = error, non-negative = descriptor).

### F-02 (P2) — `xpkg_index_entry_t.dep_names` is a CSV blob
Location: xpkg.h line 211

```c
char dep_names[256];  /* Comma-separated dependency names */
```

The rest of XPKG uses typed structs. xpkg_dep_t is a proper struct with
name[64] + min_version + max_version + optional. But the index entry
(the lightweight per-package record in a repo index) collapses all
dependency names into a single CSV string. This creates three friction
points:
1. The resolver (dep_resolve.c) must parse the CSV to get dep names —
   bespoke string parsing in a security-sensitive code path.
2. Version constraints are silently lost in the index entry; a developer
   querying an index entry cannot know version bounds without fetching
   the full .xpkg archive.
3. The 256-byte cap is invisible — a package with 16 deps × 16-char names
   = 256 chars exactly, which overflows on name separator chars.

Alternative for Sprint 4: add dep_names as char dep_names[XPKG_MAX_DEPS_PER_PKG][64]
array and dep_count uint32_t. Yes, this costs ~1KB vs ~256 bytes per entry,
but the index fits XPKG_MAX_INSTALLED=512 entries — total index size grows
by ~500KB which is acceptable for an on-disk repo index cache.

### F-03 (P2) — No local Makefile for XSEC or XNET
Finding confirmed: `find sec/xsec/ -name "Makefile*"` and
`find net/xnet/ -name "Makefile*"` return empty.

XPKG has Makefile.xpkg with `make -f Makefile.xpkg check`.
XSEC and XNET have no equivalent. A developer working on XSEC must
reconstruct the 9-flag clang invocation from the CI YAML by hand, or
run the full CI job to get feedback. This costs 3–5 minutes per iteration
cycle vs <10 seconds with a local `make check`.

### F-04 (P2) — selftest() coverage asymmetry in XSEC
xsec_sha256_selftest() is declared (line 68).
xsec_sha512 — no selftest declared.
HMAC, AES-GCM, ChaCha20-Poly1305, Ed25519, X25519 — no selftests declared.

SHA-256 is the only primitive with a NIST vector selftest in the public
API. A developer who wants to verify their cross-compiled build of, say,
AES-GCM or Ed25519 against NIST test vectors has no entry point. This
is a particular problem for freestanding targets where you cannot run
the unit test binary natively.

### F-05 (P2) — CI sprint3.yml: PAL-Aether version string inconsistency
Lines 11, 340, 350: reference "PAL-Aether v1.1 (PMM/VMM/IPC wired)"
Line 421 (gate banner): says "PAL-Aether: 1/1 (v1.2 — PMM/VMM/IPC/Timers wired)"

The gate *summary* (which developers read on pass) says v1.2, but the
step *names* and *echo* in the compile step say v1.1. A new contributor
reading CI output will see contradictory version signals in the same run.

### F-06 (P3) — XSEC utility functions duplicated in XPKG
xsec.h §12 provides: xsec_memcpy, xsec_memset, xsec_strlen, xsec_memcmp,
xsec_ror32, xsec_ror64, xsec_be32, xsec_be64, xsec_put_be32, etc.

xpkg.h §26 provides: xpkg_memcpy, xpkg_memset, xpkg_strlen, xpkg_memcmp,
xpkg_strcmp, xpkg_strncmp, xpkg_strlcpy — plus byte-order helpers.

These are structurally identical inline implementations with different
prefixes. XPKG already lists xsec.h as a dependency in its Makefile
INCLUDES. A developer writing XPKG code has to decide which prefix to use
and may unknowingly mix them. The only real difference is that xsec_ct_eq
(constant-time equality) exists only in XSEC — there is no xpkg_ct_eq —
which means a developer writing security-sensitive XPKG code that compares
hashes may inadvertently reach for xpkg_memcmp (which is NOT constant-time).

### F-07 (P3) — Missing CONVENTIONS block in xsec.h
xnet.h has an explicit CONVENTIONS block (lines 14–20):
  - packed structs for wire layout
  - byte order helpers
  - XNET_OK / XNET_ERR_* conventions
  - xnet_status_t on all APIs

xsec.h has a PURPOSE + CONSTRAINTS block but no CONVENTIONS block
equivalent. The dual-convention reality (all functions return
xsec_status_t EXCEPT the four inline CT helpers which return bool or void)
is not stated anywhere. Specifically, xsec_ct_eq returns bool and
xsec_secure_zero returns void — both correct, but undocumented as a
deliberate design choice. The section §12 header comment does not
explain WHY those are inline vs. regular functions.

### F-08 (P3) — xnet_tick() has no documentation on required call frequency
xnet.h line 701:
```c
void xnet_tick(void);    /* Periodic maintenance — call at ~1 Hz */
```
ARP TTL is 30 minutes. DNS cache TTL is presumably seconds. TCP keepalive
intervals are configurable. "~1 Hz" is documented in the comment, but
there is no note about what breaks at <1 Hz (TCP timers drift, ARP entries
go stale without expiry, gossip suspects time out incorrectly). A developer
integrating XNET into GENSD without reading the .c files has no way to
know whether calling tick() at 10 Hz (harmless) vs 0.1 Hz (breaks TCP
timers) matters, or which subsystems are sensitive to the call rate.

### F-09 (P3) — XPKG `force` flag in xpkg_gate_config_t has no guard comment
Location: xpkg.h line 330

```c
bool force[XPKG_GATE_COUNT];    /* Override FAIL → WARN (dangerous) */
```

The word "dangerous" is in the comment, but there is no guidance on which
gates can safely be forced vs. which represent security invariants that
must never be overridden (e.g., XPKG_GATE_SIGNATURE, XPKG_GATE_REVOKED).
A developer writing a testing harness may set force[XPKG_GATE_SIGNATURE]=true
to bypass signature checks during development and forget to revert it.
There is no compile-time protection and no runtime assertion.

## BUILD SYSTEM ASSESSMENT

### Discoverability (new contributor path)
1. Developer clones repo, finds pkg/xpkg/Makefile.xpkg — can run
   `make -f Makefile.xpkg check` immediately. Good.
2. Developer wants to check XSEC changes. Finds no Makefile. Must read
   sprint3.yml to reconstruct the clang invocation. Friction: HIGH.
3. Developer wants to check XNET changes. Same problem. Friction: HIGH.
4. Developer wants a single command to validate all of Sprint 3.
   No root-level target exists. Must run CI remotely. Friction: CRITICAL
   for tight iteration cycles.

### Include path documentation
Makefile.xpkg documents all three include paths in the INCLUDES variable
with source comments. This is the best-documented build file in Sprint 3.
The CI YAML shows the correct include paths but they are embedded in shell
script strings inside YAML steps — not extractable without reading the
full file.

### Missing: `make sprint3-check` at repo root
There is no root Makefile or script that runs XSEC + XNET + XPKG checks
in sequence. Makefile.xpkg covers XPKG only.

## ERROR CODE QUALITY

### XSEC
13 error codes. Quality: GOOD.
Each code has a one-line comment. The only ambiguity is XSEC_ERR_TLS_HANDSHAKE
covers 9 distinct TLS failure modes. Already noted in sprint3-p2-hardening-audit.md.
Mitigation: xsec_audit_dump() / xsec_audit_query_recent() are the correct
diagnostic path — but this is not stated anywhere near the error code definition.
Add a comment on line 42 pointing developers to xsec_audit_query_recent().

### XNET
20 error codes. Quality: EXCELLENT.
The richest error vocabulary of the three subsystems. XNET_ERR_AGAIN (-15),
XNET_ERR_TOOBIG (-16), XNET_ERR_CHECKSUM (-17), XNET_ERR_TRUNCATED (-18),
XNET_ERR_NOBUFS (-19), XNET_ERR_SHUTDOWN (-20) cover every observable
network failure mode at the packet level. No ambiguity found.

### XPKG
16 error codes. Quality: GOOD.
XPKG_ERR_GATE (-6) is correctly generic — the gate *report* (xpkg_gate_report_t)
carries per-gate detail strings. This is the right design. The detail[128]
field in xpkg_gate_check_t gives developers the diagnostic text they need.
XPKG_ERR_CYCLE (-13) is especially good — cycle detection is silent in many
package managers. Explicit surfacing here is developer-friendly.
