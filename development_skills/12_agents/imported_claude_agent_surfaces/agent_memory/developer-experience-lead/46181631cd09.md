## 4. QEMU BOOT SCRIPT ANALYSIS

**RESULT: MOSTLY CORRECT — ONE CI BUG (P2)**

### `scripts/qemu-boot.sh` functionality:

CORRECT behaviors:
- FAT32 ESP structure: creates `/EFI/BOOT/BOOTX64.EFI` (UEFI discovery path) AND
  `/EFI/GENOS/BOOTX64.EFI` + `/EFI/GENOS/xenos.elf` (GEN.OS custom paths)
- OVMF search: checks 4 known paths across Ubuntu/Fedora/Homebrew layouts
- Missing artifact handling: creates stub `.EFI` with `dd` if BOOTX64.EFI not built;
  warns (not errors) if xenos.elf missing
- `--dry-run` flag: prints QEMU command without executing
- `--headless` flag: adds `-display none -nographic`
- `set -euo pipefail`: correct error handling
- QEMU flags: `-machine q35 -bios ${OVMF} -drive virtio -serial stdio -no-reboot`

**BUG (P2): `mkfs.fat` + `mcopy` usage is silent-fallback only**
Lines 84-97: if `mkfs.fat` or `mcopy` is not installed, the script creates a 64MB
`/tmp/genos-boot.img` that is NOT formatted as FAT32 — it's all-zeros. OVMF will
refuse to boot from it. The EFI directory structure exists in `/tmp/genos-efi` but
never gets copied into the disk image without `mtools`. The `-bios` OVMF invocation
doesn't use pflash (no OVMF_VARS.fd) — won't persist UEFI boot entries but will boot.

**BUG (P2): Sprint 13 CI `qemu-dry-run` job installs `qemu-utils` only**
`.github/workflows/sprint13.yml` line 194:
```
sudo apt-get install -y --no-install-recommends ovmf qemu-utils
```
`qemu-utils` provides `qemu-img` but NOT `qemu-system-x86_64`. The script calls
`qemu-system-x86_64` in `QEMU_CMD`. The dry-run exits before executing (line 131),
so the dry-run CI job PASSES even without `qemu-system-x86`. But if a developer
runs the script without `--dry-run` after following the CI job's install instructions,
they'd hit "command not found: qemu-system-x86_64".

Correct CI package for full QEMU: `qemu-system-x86` (provides `qemu-system-x86_64`).
The `qemu-boot-test.md` skill correctly documents `qemu-system-x86`.

**CORRECT: `--dry-run` outputs a valid QEMU command**
The command constructed includes machine type, CPU, memory, OVMF bios, virtio drive,
serial output, and display. It is a valid QEMU invocation (confirmed by reading lines
105-122).

## 5. CROSS-PLATFORM (macOS) DEV TOOLCHAIN

**RESULT: DOCKER PATH EXISTS — NATIVE macOS PATH INCOMPLETE**

### Docker path (correct and documented):
- `build/Dockerfile.buildenv` — Ubuntu 24.04 with full Linux ISO build toolchain
- `build/buildenv.sh` — smart launcher: auto-detects Docker socket, detects stale images
  via SHA256 of Dockerfile, supports `verify`, interactive, and command modes
- Platform: `linux/amd64` — required for UEFI PE32+ tools (grub-efi-amd64-bin, etc.)
- Works on Apple Silicon via Rosetta 2 / amd64 emulation

**Gap (P1): `build/Dockerfile.buildenv` is missing clang-18, lld-18, nasm**
The Docker image installs: debootstrap, qemu, python3, node, build-essential, mtools...
But is missing:
- `clang-18` (required for all C freestanding compilation)
- `lld-18` (required for ELF link and PE32+ link)
- `nasm` (required for ASM files: entry.asm, isr.asm, syscall.asm)
- `llvm-18` (required for lld-link for BOOTX64.EFI)
- `cppcheck` (required for static analysis)

A developer who follows the documented path (`./build/buildenv.sh`) and then tries
to run `make -f kernel/xenos/Makefile` inside the container will hit:
```
make: clang-18: No such file or directory
```
This is a P1 friction point — the container promises a complete build environment
but cannot actually build any GEN.OS C code.

**Gap (P2): No native macOS setup guide**
For freestanding C development on macOS:
- Apple clang 15.x ≠ clang-18 (different target triple support)
- `clang --target=x86_64-unknown-none-elf` works on macOS but flag support varies
- `lld-link` not available from Xcode command line tools
- `nasm` available via `brew install nasm`
- `clang-18` available via `brew install llvm@18`
- No `scripts/setup-macos.sh` or documented `brew install` sequence exists

**Existing partial path (not documented for new contributors):**
The `build/buildenv.sh` script is the intended macOS path but the Docker image
missing clang-18 makes it broken for C kernel development.
