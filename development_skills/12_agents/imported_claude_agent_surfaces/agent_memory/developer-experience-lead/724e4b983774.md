### Verdict: 4 of 6 are under-specified

The four "unexpected message type" messages (lines 694, 788, 912, 1030) all omit
the ACTUAL received value. The developer reading an audit log cannot distinguish:
- Did the peer send the wrong handshake type (e.g., got Certificate where ServerHello expected)?
- Did the peer send the wrong content type (e.g., got APPLICATION_DATA instead of HANDSHAKE)?
- Did the peer send an alert?

The code at each site has the actual values available in local variables:
- `ct_type` (content type, line 692 context)
- `inner` (inner content type, lines 786, 803, 910, 1028 context)
- `msg_buf[0]` (handshake message type)

The `xsec_audit_log` signature is: `xsec_audit_log(event, module, message_string)`.
The message is a bare `const char *`. There is no structured field for actual/expected
values, which is the root constraint. The audit subsystem cannot format printf-style.

### Additional audit message issues found

Line 1046: "ServerFinished verify fail" — terse, inconsistent capitalization with the
others ("ServerFinished" vs "Server Finished"), no RFC reference.

Line 366-368: "tls alert received from peer" — lowercase "tls" inconsistent with
all other messages which use uppercase "TLS". Minor but breaks log grep patterns.

## 2. x509.c TBS SEQUENCE Comment (D-S3-P1-003)

### Comment at lines 463-466

```c
/* Record start of TBSCertificate SEQUENCE *before* der_enter() so that
 * tbs_start points to the 0x30 tag byte. tbs_seq_end - tbs_start then
 * covers tag + variable-length field + content = full DER encoding per
 * RFC 5280 §4.1.2 (D-S3-P1-003: fixed; no hardcoded header offset). */
```

### Assessment

WHAT (what happened): Clear — "record start before der_enter()".
WHY (why it must be before): Partially clear — "so tbs_start points to the 0x30 tag byte".
    Missing: WHY does this matter? A new contributor does not know that der_enter() advances
    the cursor past the tag+length bytes, so calling it first would make tbs_start point
    INTO the TBS content, not at the SEQUENCE header.
WHAT is covered: Stated as "tag + variable-length field + content". Correct but compressed.
    Missing: explicit statement that the variable-length DER length field can be 2-5 bytes
    (multi-byte form), which is precisely WHY a hardcoded "+2" offset was wrong.
RFC reference: Present (RFC 5280 §4.1.2). Good.

### Confusing element

The phrase "no hardcoded header offset" is meaningful only to someone who knows the bug
that was fixed. A first-time reader has no context for what offset was hardcoded or why.
The task ID "(D-S3-P1-003: fixed;" embeds the fix reference inline in the comment, which
is useful for blame tracking but breaks the comment's readability as a standalone explanation.

### Follow-on comment at lines 606-607

```c
/* out->tbs_data/tbs_len span the full DER-encoded TBSCertificate (tag + length + value)
 * — the exact byte range the signature is computed over per RFC 5280 §4.1.2. */
```

This is better — it states the security purpose (signature computation) which answers
WHY this range matters. The two comments are complementary but separated by 140 lines,
so the purpose is only explained at the use-site, not at the capture-site.

## 3. capability_init.c Include Path Discoverability

### The include at line 46

```c
#include "xkabi_rights.h"
```

### Problems found

**Problem A: Two xkabi_rights.h files exist**

- `kernel/xenos/include/xkabi_rights.h` — CORRECT file (81 lines, hardware rights
  at bits 12-22, software at bits 0-11, XKABI_RIGHT_* names). This is the one referenced
  in the file-header comment at line 21.
- `xisc/include/xkabi_rights.h` — DIFFERENT file (39 lines, XK_RIGHT_READ/WRITE/EXECUTE
  generic object rights, NO hardware bit assignments). Completely different semantics.

If a developer adds `-Ixisc/include` before `-Ikernel/xenos/include` in their local
build invocation, they will silently compile against the wrong header. The XKABI_RIGHT_*
macros used by capability_init.c (e.g., XKABI_RIGHT_GPU_EXEC, XKABI_RIGHT_NET_SEND)
are undefined in xisc/include/xkabi_rights.h and would cause compile errors — so at
least the failure is loud, not silent. But the error message "undeclared identifier
'XKABI_RIGHT_GPU_EXEC'" does not tell the developer which of the two files is wrong.

**Problem B: No comment explaining why quoted (not angle-bracket) include**

The `#include "xkabi_rights.h"` form searches relative to the compiler's include path.
Since there is no Makefile for aetherboot (confirmed: no Makefile.aetherboot exists),
the correct include path (-Ikernel/xenos/include) is documented only in xenos.yml
line 160. A developer running a local compile without reading the CI YAML will not
know to add it.

**Problem C: No Makefile.aetherboot**

xpkg has `pkg/xpkg/Makefile.xpkg`. aetherboot has no equivalent local make target.
The CI YAML at xenos.yml lines 155-170 is the only place the full compile command
for capability_init.c is documented. A developer cannot run `make check` locally.

**Problem D: typedef placement before include**

Line 44: `typedef UINT32 xkabi_rights_t;` appears before line 46: `#include "xkabi_rights.h"`.
The kernel's xkabi_rights.h does NOT define xkabi_rights_t — that type is only in
xisc/include/xkabi_rights.h. So the typedef is locally defined to avoid pulling in
the xisc header. This is correct but opaque. A new contributor seeing a typedef before
an include will find this ordering unusual and may not understand why the type is not
defined in the header being included.

## 4. Ten Test Types Assessment

### 1. Smoke — Can a new developer understand capability_init.c from header comment?
PASS (7/10). The PURPOSE and DESIGN blocks at lines 7-23 are clear. The "zero ambient
authority" goal is stated. FAIL point: "The table is stored at the end of the boot-time
memory map buffer" (line 24) is confusing — is it overlapping the memory map? A new
developer needs a diagram or a size-safety assertion comment to understand there is no
overflow risk.

### 2. Functional — Do tls13.c error messages guide to root cause?
PARTIAL (5/10). Messages correctly identify WHICH handshake step failed. They do NOT
include the actual received value (see section 1). A developer debugging a test harness
must enable raw packet capture alongside the audit log to identify the actual peer behavior.

### 3. Integration — Is the include path documented in README or CONTRIBUTING?
FAIL (2/10). CONTRIBUTING.md (20 lines total) contains no build instructions, no include
paths, no mention of capability_init.c or xkabi_rights.h. The only authoritative source
is xenos.yml. No README.md was found at the root. A new contributor cannot discover the
correct compile command without reading the CI YAML.

### 4. Regression — Did any comments become stale after the fixes?
YES, one regression found:
- capability_init.c line 39: "Formerly duplicated with incorrect sequential values (0x01-0x80)."
  The original incorrect local defines in capability_init.c used values 0x01-0x80. This
  statement is historically accurate but will confuse future contributors who will not know
  what the old values looked like or why they were wrong. The comment explains WHAT was fixed
  but not WHY the old values were semantically wrong (they did not match the bit positions
  in the kernel's hardware enforcement logic).

### 5. Load — N/A
Audit message throughput is not a bottleneck concern for handshake operations.
Document: xsec_audit_log is a write to a fixed circular ring buffer per Sprint 3 audit.c.
No load concern identified.

### 6. Stress — N/A
Freestanding C with no heap. No stress failure mode from the DX perspective.
Document: tls_enc_scratch and tls_enc_ct_buf are module-static; single-threaded per
ctx assumed. Comment at lines 193-196 documents this rationale.

### 7. Security — Do audit messages avoid leaking sensitive data?
PASS (9/10). None of the 6 new audit messages include key material, memory addresses,
or PII. The "ServerHello unsupported cipher suite" message notably omits the offered
suite value — which is actually CORRECT from a security perspective (cipher suite
values are not secret, but including them would give an attacker confirmation of what
was detected). The omission is acceptable but the rationale is not documented.

One borderline case: "CertificateVerify: peer public key too short for Ed25519" (line
1000) does not leak the actual key length, which is correct. No issue.

### 8. UI — Is there a tool to display audit events during development?
FAIL (1/10). No developer-facing audit viewer tool exists. The xsec_audit_dump() and
xsec_audit_query_recent() functions exist in the XSEC header, but there is no CLI
wrapper, no formatted output tool, and no VS Code task that surfaces audit events
during a test run. A developer must call xsec_audit_dump() manually in test code and
parse the raw ring buffer output.

### 9. Fuzz — N/A
Fuzz testing of TLS record parsing would be a test-forge deliverable, not a DX gap.
Document: tls_recv_record bounds-checks all length fields before use. No unbounded
buffer reads identified in the 6 new code paths.

### 10. Reliability — Are there build-time guards if xkabi_rights.h is missing?
PARTIAL (6/10). If xkabi_rights.h is not found, the compiler will fail with "file not
found" — a loud, fast failure. The correct file (kernel/xenos/include/xkabi_rights.h)
has a standard include guard (#ifndef XKABI_RIGHTS_H). However, there is no static_assert
that verifies the expected rights values are in range. A developer who accidentally
includes the wrong file (xisc/include/xkabi_rights.h) would get a compile error on
XKABI_RIGHT_GPU_EXEC (undefined), not a silent wrong-values bug. Outcome: fails loud.

## 7. Proposed Developer Guide Section: Capability Rights

(To be added to CONTRIBUTING.md or a new docs/capability-rights-guide.md)

Key points to document for new contributors:
1. Single source of truth: kernel/xenos/include/xkabi_rights.h only. Never redefine.
2. Bit layout: bits 0-11 = software rights (GENSD-visible), bits 12-22 = hardware rights
   (kernel-enforced per syscall), bit 23 = DERIVE, bits 24-31 = reserved/zero.
3. Two different headers with same filename exist: kernel/xenos/include/ is the boot +
   kernel header; xisc/include/ is a generic object-rights header. They are not interchangeable.
4. aetherboot compile: requires -Ikernel/xenos/include (see xenos.yml line 160).
5. The principle: every EFI handle gets exactly the rights its protocol implies, and no more.
   Unknown handles get XKABI_RIGHT_NONE — default-deny enforced at EFI handle enumeration time.
