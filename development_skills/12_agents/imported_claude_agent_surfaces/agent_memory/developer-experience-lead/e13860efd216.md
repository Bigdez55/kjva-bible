## Critical Bugs (P1)

### F-01 — CI header comment file counts are wrong (sprint3.yml L8, L10)
  Line 8:  says "7 source files" for XSEC — actual is 9 (crypto*6 + x509 + audit + tls13)
  Line 10: says "12 source files" for XNET — actual is 11
Neither the job loop nor the gate summary banner is wrong — only the comment block.
Fix: update lines 8 and 10 in .github/workflows/sprint3.yml.

## High Friction (P2)

### F-02 — No Makefile.xsec or Makefile.xnet
XPKG has Makefile.xpkg (make -f Makefile.xpkg check).
XSEC and XNET have no equivalent. Developers must read the full CI YAML to run
a local compile check for those subsystems.

### F-03 — test_sprint3.c is a single 30K-token monolith (10 test types, ~1200+ lines)
There is no way to run a single test type or subsystem in isolation.
No test binary is produced — compile-only gate only.
Recommendation: split into test_xsec.c, test_xnet.c, test_xpkg.c.

### F-04 — xnet.h §24 socket API returns bare `int` — contradicts CONVENTIONS block
CONVENTIONS block says "All public APIs return xnet_status_t".
§24 (socket.c) declares 10 functions returning bare int (POSIX style).
The comment block's claim is false for this section — a developer reading the
CONVENTIONS block will be surprised when they reach §24.
Fix: add a note to the CONVENTIONS block: "Exception: §24 socket API returns int
(POSIX-compatible descriptor-or-negative convention)".

### F-05 — ASSERT_MEMEQ calls xsec_memcmp instead of t_memcmp
The test harness defines t_memcmp (local inline) but ASSERT_MEMEQ uses xsec_memcmp.
If XSEC subsystem has a bug, XNET-only tests fail with a confusing linker/API error
rather than a clean test failure. Creates invisible cross-subsystem coupling.

## Medium Friction (P3)

### F-06 — xpkg_index_entry_t.dep_names is a CSV blob, not a typed array
  char dep_names[256];  /* Comma-separated dependency names */
All other dep structures use xpkg_dep_t arrays. No iteration helper is provided.
This is the only field in all three headers that encodes structure in a string.

### F-07 — xpkg_main cmd_repo_refresh uses a nested extern declaration (non-portable)
  void xpkg_repo_get_list(const xpkg_repo_t **out, uint32_t *out_count);
This is declared inside cmd_repo_refresh() function body — a GCC extension,
not ISO C11. It is accepted by clang -fsyntax-only but is non-portable and
surprising. Should be in the §3 external declarations block.

### F-08 — No root-level make sprint3-check target
Developer must know subsystem-specific make invocations; nothing at the repo root.

### F-09 — ASSERT_OK / ASSERT_EQ failure messages omit actual values
ASSERT_OK failure: "[FAIL] name -- status not OK"  (what was the code?)
ASSERT_EQ failure: "[FAIL] name -- values not equal"  (what were a and b?)
Better: include the actual values in the failure string.

### F-10 — test_func_ethernet_crc32 has a tautological assertion (always passes)
  ASSERT_TRUE("CRC-32(empty) is deterministic", crc_empty == crc_empty);
crc_empty == crc_empty is always true regardless of value. Tests nothing.

### F-11 — test_load_arp_cache has a vacuous size assertion
  ASSERT_EQ("ARP hw_len...", (uint32_t)sizeof(uint8_t), 1u);
sizeof(uint8_t)==1 is universally true. Not an ARP test.

### F-12 — No XNET or XPKG "summary" CI step (XSEC has one, others do not)
Minor consistency gap. xsec-compile-check has "XSEC summary: 9/9 files verified".
xnet-compile-check and xpkg-compile-check have no equivalent step.

## What Is Excellent (preserve)

- xsec.h §1 error codes: 13 distinct codes, each with trailing comment. AUTH_FAIL vs
  VERIFY_FAIL vs CERT_INVALID are separate — no ambiguity. Best in class.
- xpkg.h §1 error codes: 16 codes. XPKG_ERR_CYCLE (separate from ERR_DEPENDENCY)
  and XPKG_ERR_REVOKED are precisely named. Better than most production pkg managers.
- xpkg.c print_error(): complete status->string table, format "[xpkg] ERROR: cmd failed:
  reason (code N)" — satisfies what/why/next. Template for all subsystems.
- xpkg.c cmd_help() gate visualization: numbered gates in a grid. Developers can
  cross-reference gate IDs in logs directly to help text.
- xnet.h CONVENTIONS block: sets byte-order, naming, error-code rules at the top.
  Should be adopted in xsec.h and xpkg.h.
- xsec.h §12 xsec_memcmp comment: "NOT constant-time, use xsec_ct_eq for secrets".
  This is precisely the pit-of-success guardrail that prevents side-channel bugs.
- CI sprint3-gate job: structured pass/fail matrix + Sprint 4 prerequisites listed.
  This is the right template for all sprints — developers always know what is next.
- SHA-pinned actions with date comments on every checkout step.
- xpkg_gate_id_t is an enum, not magic numbers — gate IDs are navigable in IDEs.
- xpkg_gate_report_t.checks[].detail[128] carries human-readable explanation per gate.
- test harness xorshift32 PRNG for fuzz tests: elegant freestanding solution.
- SECTION() macro in test harness provides clear visual grouping in console output.

## Patterns to Preserve Across All Sprints

- Section dividers: /* === * SN SECTION NAME * === */ — scannable in any editor
- One-shot + streaming variants for all crypto primitives (XSEC pattern)
- Caller-provided buffers everywhere — no internal allocation
- _selftest() functions on crypto primitives for NIST vector validation
- Inline constant-time helpers (xsec_ct_eq, xsec_secure_zero) in the header itself
- CONVENTIONS block at the top of headers (XNET pattern — adopt for XSEC and XPKG)
- Gate report detail strings as human-readable text (XPKG pattern)
- SHA-pinned CI actions with date comments
- CI gate summary job that lists Sprint N+1 prerequisites
