## Corrected baseline (prior memory was wrong)

- **Reality**: most workflows are grep/AST/structural gates, not compile gates.

## Five direct answers

### Q1 — Can `docker compose up` produce a working GEN.OS cloud?
**Partial / build-time risky.**
- All 5 services have real Dockerfiles (tools/launch/Dockerfile is a 3-stage build; devices/cloud/platform/docker/Dockerfile.cloud is valid uvicorn bundle; Citadel/docker/Dockerfile.citadel runs supervisord with 16 real programs).
- Cloud service (`main.py`, 70 LOC) correctly mounts 5 FastAPI sub-apps (identity/sync/update/gate/provenance) — all `app.py` files exist.
- Desktop (`tools/launch/entrypoint.sh`) executes `qemu-system-x86_64` with OVMF + disk image built from `$BUILD/xenos.elf` and `$BUILD/aetherboot/BOOTX64.EFI` — but those artifacts are produced by `boot-test.sh` inside the Docker builder stage, which depends on the `ghcr.io/bigdez55/genos-toolchain:latest` image.
- **Gap**: the toolchain image is a pre-built GHCR reference — no workflow in .github/workflows/ builds or pushes it. If it's stale or private, `docker compose up` of `genos-desktop` fails at build time.
- **Gap**: no `docker compose up` smoke test exists in any CI workflow. Zero workflows grep for `docker-compose` or `docker compose`.

### Q2 — Is there a working ISO pipeline?
**No. The primary pipeline is orphaned from CI.**
- `build/genos-1.0-amd64.iso` (3.7 GB, Feb 26) exists on disk but its own validation report (`build/genos-1.0-amd64.iso.validation.json`) reports `"status": "FAIL"` with `iso-assemble.py` exit code 64. The 7 required checks include: `iso_exists: FAIL`, `iso_sha256_sidecar_exists: FAIL`, `iso_md5_sidecar_exists: FAIL`, `iso_assemble_validation_present: FAIL`. The SHA256/MD5 sidecars on disk are from a DIFFERENT run than the validation report.
- `build/genos-1.0-amd64.iso.gate.json` has `verdict: PASS` but that's a refusal-manifest gate (unrelated to ISO structure — it's validating `platform/refusal/examples/S_S_good.json`).
- **Zero CI workflows reference `iso-assemble.py`, `build-and-validate.sh`, `release-gate.sh`, `chroot-setup.sh`, or `genos-build.py`** (verified via grep on all 74 workflows).
- `iso-build.yml` exists and runs — but it builds **stub test payloads**: creates a 4 KB fake MZ header as `test_aetherboot.efi` and an 8 KB fake ELF header as `test_xenos.elf`, then runs `genos-iso-build` (the C host tool) to assemble a minimal ISO. This verifies the ISO assembly tool's structural correctness (PVD, BRD, VDST, boot catalog, MZ at sector 20). It does NOT produce a bootable ISO with real AetherBoot + XENOS.

### Q3 — Is there a release pipeline?
**No.**
- Zero workflows contain `docker push`, `ghcr.io/...push`, or `gh release create`.
- Zero workflows use `on: push: tags:` triggers.
- The 9 `build/images/genos-*-1.0.0.tar` files (≈246 MB each, total 2.2 GB) are Docker save-format images but are not referenced by `docker-compose.yml` (which uses `build:` contexts instead of `image:`) and no workflow loads, builds, or publishes them. They are artifacts from a prior manual run — a museum, not a pipeline.

### Q4 — Staging CI gate?
**No.**
- `cloud.yml` is the only "cloud CI" workflow. It runs `ruff check devices/cloud/` and `pytest tests/cloud/` — which contains exactly 1 test file (`test_tier1_services.py`, 376 LOC). No deployment step. No staging target.
- No environment is declared via `environment:` in any workflow.

### Q5 — Sprint 50 100-test boot matrix?
**No.**
- `sprint50-namespace-isolation.yml` is NOT a boot matrix. It is 10 grep-based structural checks for namespace-isolation code (`XKABI_RIGHT_CROSS_NAMESPACE`, `owner_ns_id`, `_chan_ns_check`, etc.) plus Python AST/ruff/pylint on `Citadel/council/gates/magen_gate.py`.
- The actual hardware boot matrix exists at `tests/hardware/boot-matrix.sh` (344 LOC, 20 QEMU profiles) and `tests/hardware/boot-matrix-md.sh` (525 LOC). **Zero CI workflows reference them**.
- `qemu-boot-test.yml` runs a SINGLE QEMU boot via `tools/boot/boot-test.sh` and checks for `[XENOS]` sentinel. One profile, not a matrix.

## P0 findings

1. **iso-assemble.py pipeline is orphaned and broken** — the 3.7 GB ISO is stale and the validation JSON says FAIL. No CI ever runs this pipeline. If the user expected "main's last green build = bootable ISO," that's not the case.
2. **No release pipeline exists** — no tag-triggered workflow, no Docker registry push, no GitHub Release artifact upload. "Deploying" means manually running scripts.
3. **Toolchain image dependency is implicit** — `tools/launch/Dockerfile` expects `ghcr.io/bigdez55/genos-toolchain:latest` which no workflow publishes. If stale: all `docker compose up` on desktop/console/arm64-boot fails cold.
4. **100-test boot matrix never runs in CI** — `tests/hardware/boot-matrix.sh` with 20 QEMU CPU/RAM/machine profiles exists but is invoked by nothing. BOOT-VERIFIED claim rests on single-profile `qemu-boot-test.yml` only.
5. **No `docker-compose` validation workflow** — no CI gate ensures the fleet actually starts cleanly. Fleet health is unverified on every commit.

## P1 findings

6. `iso-build.yml` produces stub ISO (fake MZ + fake ELF) — tool validation only, not artifact.
7. `cloud.yml` has 1 test file for 5 mounted services — thin coverage.
8. `gate.json` in the ISO artifact bundle validates a refusal manifest, not the ISO itself — naming is misleading.
9. 9 × 246 MB tar images in `build/images/` are unreferenced by docker-compose.yml (uses `build:` contexts) and unbuilt by any workflow.
10. MEMORY.md claim "53 of 61 workflows are syntax-only" was false — actual is 1 of 74. Prior audit generalized incorrectly.
