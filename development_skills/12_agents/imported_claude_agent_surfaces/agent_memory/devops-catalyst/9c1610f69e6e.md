### P0 -- CRITICAL (must fix before push to origin/main)

### P1 -- HIGH (fix in current sprint)

**P1-01: ci.yml uses ubuntu-22.04 for 4 jobs**
- Files: ci.yml lines 160, 183, 197, 215
- Jobs: poc-test-suites-ubuntu22, poc-static-analysis, poc-security-barriers, poc-demo-gates
- Impact: These jobs run on ubuntu-22.04, creating a split runner environment.
  The poc-test-suites-matrix job uses matrix [ubuntu-24.04, macos-14] but the
  poc-test-suites-ubuntu22 specifically targets 22.04. While intentional for
  backward-compat testing, it means ubuntu-22.04 is the gating dependency for
  poc-security-barriers and poc-demo-gates via `needs: poc-test-suites-ubuntu22`.
- Recommendation: Rename to `poc-test-suites-ubuntu24` and update to ubuntu-24.04,
  or add ubuntu-24.04 to the matrix and remove the standalone 22.04 job.

**P1-02: ci.yml uses mutable tags for setup-python and setup-node**
- 4x `actions/setup-python@v5` (ci.yml lines 20, 56, 101, 233)
- 2x `actions/setup-node@v4` (ci.yml lines 258, 275)
- Impact: Mutable tags can change without warning, potentially introducing
  supply-chain attacks or breaking changes. All other workflows are clean --
  only ci.yml has this problem.
- Recommendation: Pin to SHA:
  - actions/setup-python v5.3.0: @65d7f2d534ac1bc67fcd62888c5f4f3d2cb2b236
  - actions/setup-node v4.1.0: @39370e3970a6d050c480ffad4ff0ed4d3fdee5af

**P1-03: ci.yml has zero freestanding compile checks for Sprint 2 (XDISP/XCOMP/XFRAME)**
- Sprint 2 produced 50 files / 30,499 lines of freestanding C code.
- ci.yml contains header compile checks (xisc-header-check) and hosted tests (pytest)
  but ZERO freestanding cross-compile checks with clang -target x86_64-unknown-none-elf.
- The 29 compile checks from Sprint 2 are verified locally but not enforced in CI.
- Impact: Regression in any Sprint 2 C file would go undetected by CI.
- Recommendation: Add a sprint2-compile-check job to ci.yml (or create sprint2.yml)
  with the canonical clang freestanding flags for all 29 Sprint 2 .c files.

**P1-04: xenos.yml (Sprint 1) uses gcc instead of clang for freestanding compile**
- Lines 89, 102-117, 125-137, 158-168
- All Sprint 3-9 workflows use `clang -target x86_64-unknown-none-elf` for freestanding
  compile checks, establishing a canonical standard.
- xenos.yml uses `gcc -ffreestanding -fno-stack-protector ...` without `-target`.
- gcc on ubuntu-24.04 defaults to x86_64-linux-gnu, not x86_64-unknown-none-elf.
  This means Sprint 1 compile checks are NOT truly freestanding cross-compile checks.
- Impact: Code that accidentally depends on Linux ABI could pass Sprint 1 CI but
  fail when linked into the actual GEN.OS kernel image.
- Recommendation: Migrate xenos.yml to clang with canonical flags.

**P1-05: xenos.yml conditional skip pattern creates silent pass risk**
- Lines 135, 149, 166: `[ -f "$f" ] || { echo "WARN: $f not yet written -- skip"; continue; }`
- If a file is accidentally deleted or renamed, CI will silently skip it and still pass.
- All Sprint 3-9 workflows do NOT have this pattern -- they assume files exist (correct).
- Impact: A missing source file would not cause CI failure in Sprint 1.
- Recommendation: Remove the `[ -f ]` guards. All Sprint 1 files exist and are committed.

### P2 -- MEDIUM (fix in next sprint)

**P2-01: Sprint 3-9 have no artifact upload**
- Only foundation.yml and xenos.yml upload build artifacts.
- Sprint 3-9 produce no artifacts, meaning there is no way to inspect
  intermediate build state after CI runs.
- Impact: Debugging CI failures requires re-running the entire pipeline.
- Recommendation: At minimum, upload the sprint gate summary as an artifact.

**P2-02: Sprint 7/8/9 missing `permissions: contents: read`**
- foundation.yml, xenos.yml, ci.yml, sprint3.yml, sprint4.yml, sprint5.yml, sprint6.yml
  all declare `permissions: contents: read` (principle of least privilege).
- sprint7.yml, sprint8.yml, sprint9.yml do NOT declare permissions.
- Impact: Without explicit permissions, jobs inherit the default (which may be
  write on some repository configurations). This is a security hardening gap.
- Recommendation: Add `permissions: contents: read` to sprint7/8/9.yml.

**P2-03: Sprint 7/8/9 missing timeout-minutes on most jobs**
- sprint7.yml: No timeout-minutes on any job (6 jobs)
- sprint8.yml: No timeout-minutes on any job (8 jobs)
- sprint9.yml: No timeout-minutes on any job (8 jobs)
- Sprint 3-6 consistently set 15-minute timeouts on compile jobs, 5 minutes on gate jobs.
- Impact: A hung job (e.g., clang spinning on malformed input) would consume
  runner minutes until the GitHub Actions 6-hour default kicks in.
- Recommendation: Add timeout-minutes: 15 to compile/analysis jobs, 5 to gate jobs.

**P2-04: PAL-Aether version inconsistency across workflows**
- sprint3.yml line 11, 340, 350: references "PAL-Aether v1.3"
- sprint3.yml line 433: says "PAL-Aether v1.3: all 5 sections wired (Sprint 4)"
- sprint4.yml line 14, 226, 236: references "PAL-Aether v1.2"
- MEMORY.md says PAL-Aether v1.2 is current with all 5 sections wired.
- Contradiction: Sprint 3 calls it v1.3, Sprint 4 calls it v1.2.
- Impact: Version confusion across sprints; no single source of truth.
- Recommendation: Establish canonical version in pal.h and reference it in all workflows.

**P2-05: Sprint 3 CFLAGS_COMMON env defined but never referenced**
- sprint3.yml lines 51-65: Defines `CFLAGS_COMMON` as a YAML env var
- All compile steps in sprint3.yml hardcode the flags inline instead of using
  `${{ env.CFLAGS_COMMON }}` or `$CFLAGS_COMMON`.
- Same issue in sprint4.yml, sprint5.yml, sprint6.yml.
- sprint7.yml uses `${{ env.CFLAGS_FREESTANDING }}` correctly (partially).
- sprint8.yml and sprint9.yml use `${{ env.CFLAGS }}` correctly.
- Impact: Maintenance burden -- changing a flag requires editing every step.
- Recommendation: Refactor sprint3-6 to use the env var.

**P2-06: Branch filter inconsistency across workflows**
- foundation.yml, xenos.yml, ci.yml, sprint3-6: push to ["main", "master"]
- sprint7.yml: push to any branch (no `branches:` filter at all)
- sprint8.yml, sprint9.yml: push to [main] only (no "master")
- Impact: sprint7.yml triggers on ALL branches for its paths, which is more
  liberal than intended. sprint8/9 would not trigger on a "master" branch push.
- Recommendation: Standardize to `branches: ["main"]` across all workflows
  (since the repo uses "main" as primary branch).

### P3 -- LOW (track for future improvement)

**P3-01: xenos.yml ABFLAGS missing -std=c11**
- Line 158-161: ABFLAGS for tpm.c and capability_init.c does not include -std=c11.
- KFLAGS and GFLAGS both include -std=c11.
- Impact: gcc defaults to -std=gnu17 on modern ubuntu, which is more permissive.
  capability_init.c line 52 has a C23 digit separator (`0x584B4142'49434150ULL`)
  that would be rejected by -std=c11 but accepted by -std=gnu17.
  The macro is never expanded (line 228 uses a literal), so currently benign.

**P3-02: foundation.yml uses hosted gcc build (not freestanding)**
- Sprint 0 is a hosted POSIX validation layer (pal_linux.c, sys/eventfd.h).
- This is intentionally different from Sprint 1+ freestanding builds.
- No action needed -- documented as expected behavior.

**P3-03: Sprint 8/9 use clang-18 explicitly; Sprint 3-7 use `clang` (distro default)**
- sprint8.yml line 29: `CLANG: clang-18`
- sprint9.yml line 33: `CLANG: clang-18`
- sprint3-6.yml: `sudo apt-get install -y clang` (gets distro default)
- sprint7.yml: `CLANG: clang` env var + `sudo apt-get install -y clang`
- Impact: On ubuntu-24.04, `clang` is clang-18 by default, so this is currently
  a no-op difference. But if the distro bumps to clang-19, sprint3-7 would
  silently change compilers.
- Recommendation: Standardize to clang-18 across all sprint workflows.

**P3-04: Sprint 5 file count check uses WARN instead of FAIL**
- sprint5.yml line 329: `echo "WARN: expected 14 .c, got $C_COUNT (may include archived files)"`
- Sprint 3 and 4 use hard `exit 1` on file count mismatch.
- Sprint 5 downgrades this to a warning, which means a missing file would not fail CI.
- Same issue in sprint6.yml line 240.

**P3-05: Sprint 8/9 LOC counting uses `if [ -f "$f" ]` guard**
- sprint8.yml line 331, sprint9.yml line 343
- These guards silently skip missing files during LOC counting.
- Impact: Low -- LOC counting is informational, not a gate check. But it masks
  potential issues where expected files are missing.

**P3-06: No clang-tidy job in any workflow**
- cppcheck is consistently run across all 10 workflows.
- clang-tidy (which catches different classes of bugs) is never used.
- Impact: Lower static analysis coverage than possible.

**P3-07: ci.yml npm audit runs at repo root**
- ci.yml line 91: `working-directory: .` -- npm audit runs at the root, but
  package.json may not exist there. If it does not exist, npm audit exits 0
  (no packages to audit = no vulnerabilities = pass).
- Impact: Potential false-pass if package.json is only in subdirectories.

## 7. GREY BOX: FLAG PROGRESSION ANALYSIS

The compile flag chain shows clear evolution:

- **Sprint 0**: `gcc` hosted build (libc available, POSIX API)
- **Sprint 1**: `gcc -ffreestanding -fno-stack-protector -fno-pie -mno-red-zone -Wall -Wextra -Werror -O2 -std=c11` (partial freestanding)
- **Sprint 3-6**: `clang -target x86_64-unknown-none-elf -ffreestanding -fno-stack-protector -fno-pie -mno-red-zone -mno-mmx -mno-sse -mno-sse2 -Werror -Wall -Wextra -Wno-unused-parameter -O2 -std=c11 -fsyntax-only` (canonical)
- **Sprint 7**: Same canonical flags via CFLAGS_FREESTANDING env var
- **Sprint 8-9**: Same canonical flags via CFLAGS env var, explicitly using clang-18

The canonical flag set is consistent from Sprint 3 onward. The key improvement was
adding `-target x86_64-unknown-none-elf` and `-fsyntax-only` starting in Sprint 3.

Sprint 1 (gcc) is the outlier: no `-target`, no `-fsyntax-only`, no `-mno-mmx/-sse/-sse2`.
