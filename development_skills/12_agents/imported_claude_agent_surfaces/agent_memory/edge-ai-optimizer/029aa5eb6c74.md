## 1. AES-256-GCM (aes_gcm.c, 498 lines)

### 1.1 Current Implementation Analysis

**Architecture:**
- Constant-time S-box via full 256-entry linear scan (`aes_sbox_ct()`, line 58)
- Schoolbook GF(2^8) multiplication for MixColumns (`gf_mul()`, line 84)
- Byte-by-byte GF(2^128) multiplication for GHASH (`ghash_mul()`, line 254)
- CTR mode constructed manually via `gcm_inc32()` + `aes256_encrypt_block()`
- 14-round AES-256 with textbook SubBytes/ShiftRows/MixColumns/AddRoundKey

**Operation Counts per AES-256 Block (16 bytes):**
- `aes_sub_bytes()`: 16 S-box lookups x 256 iterations = 4,096 comparisons per round
- 14 rounds (13 full + 1 final without MixColumns) = 57,344 S-box comparisons
- `aes_mix_columns()`: 13 rounds x 4 columns x 4 `gf_mul()` calls = 208 GF(2^8) muls
- Each `gf_mul()` = 8-iteration loop with 5 ops each = ~40 ops
- MixColumns total: 208 x 40 = 8,320 operations per block

**Operation Counts per GHASH block (16 bytes):**
- `ghash_mul()` iterates 128 bits (16 bytes x 8 bits)
- Each bit: 16-byte XOR (conditional) + 16-byte right-shift + conditional XOR
- Total: 128 x (16 + 16 + 16) = ~6,144 byte operations per GHASH block

### 1.2 Throughput Estimates on i5-8265U

**Software CT S-box Path (current):**

```
Per-block AES-256 encrypt:
  S-box:       57,344 comparisons x ~5 cycles = 286,720 cycles
  MixColumns:  8,320 ops x ~3 cycles           =  24,960 cycles
  ShiftRows:   14 rounds x ~10 cycles           =     140 cycles
  AddRoundKey: 15 rounds x ~20 cycles           =     300 cycles
  Total per block:                              ~312,120 cycles
  At 3.9 GHz boost: ~80 us per block

Per 16 KB TLS record (1024 blocks):
  AES-CTR:     1024 x 80 us                    = ~82 ms
  GHASH:       1024 x ~30 us (6144 byte-ops)   = ~31 ms
  Total:                                        = ~113 ms per 16 KB record
  Throughput:  16384 / 0.113                    = ~145 KB/s = ~0.14 MB/s
```

**With AES-NI (Sprint 4 target):**

```
Per-block AES-256 encrypt (AESENC pipeline):
  _mm_aesenc_si128 x 13 rounds                 =  13 cycles (pipelined)
  _mm_aesenclast_si128 x 1                      =   1 cycle
  _mm_xor_si128 (AddRoundKey) x 15              =  15 cycles
  Key schedule overhead (amortized):             =   2 cycles
  Total per block:                              ~  31 cycles
  At 3.9 GHz boost: ~8 ns per block

Per 16 KB TLS record (1024 blocks):
  AES-CTR:     1024 x 8 ns                     = ~8.2 us
  GHASH (PCLMULQDQ):  1024 x ~10 ns            = ~10.2 us
  Total:                                        = ~18.4 us per 16 KB record
  Throughput:  16384 / 0.0000184                = ~890 MB/s
```

**Speedup from AES-NI: ~6,100x (0.14 MB/s --> 890 MB/s)**

### 1.3 Correctness Notes

- The CT S-box linear scan is correct for constant-time security
- `ghash_mul()` line 265: uses `if (xi & 0x80u)` which is a branch on secret data
  - This is a timing side-channel in GHASH (not the S-box, which was already fixed)
  - P2 severity: GHASH input is ciphertext (not secret), but the hash key H is secret
  - Should be converted to constant-time mask: `mask = -(xi >> 7); result ^= V & mask;`
- `ghash_mul()` line 280: `if (lsb)` is another branch on V state derived from H
  - Same fix: use constant-time mask

### 1.4 Memory Footprint

- `xsec_aes256_key_t`: 60 x uint32_t round keys = 240 bytes
- GHASH: 3 x 16-byte temporaries = 48 bytes
- GCM core stack: ~400 bytes total (key + H + J0 + S + len_block + icb + E_J0)
- S-box table: 256 bytes (read-only, .rodata)
- Rcon table: 44 bytes
- **Total per-call stack: ~700 bytes** -- excellent for constrained environments

## 2. SHA-256 / SHA-512 (sha256.c, 455 lines)

### 2.1 Current Implementation Analysis

**Architecture:**
- Standard FIPS 180-4 compression function
- SHA-256: 64 rounds per 64-byte block, 32-bit working variables
- SHA-512: 80 rounds per 128-byte block, 64-bit working variables
- Message schedule W[64] (SHA-256) / W[80] (SHA-512) fully expanded in memory
- Uses `xsec_ror32()` / `xsec_ror64()` for rotations (likely inline via compiler)
- No SIMD, no message schedule optimization, no loop unrolling

**Key observation:** i5-8265U does NOT have SHA-NI extensions. SHA-NI is only
available on Goldmont+ (Atom), Ice Lake, and later architectures. Whiskey Lake
(Coffee Lake refresh) does not include it.

### 2.2 Throughput Estimates on i5-8265U

**SHA-256 (current software):**

```
Per-block compression (64 bytes input):
  Message schedule: 48 rounds x ~6 ops = 288 ops
  Compression rounds: 64 x ~20 ops     = 1,280 ops
  Total per block:                      ~1,568 ops
  At ~2 IPC, 3.9 GHz:                  ~201 ns per block
  Throughput: 64 / 201e-9               = ~318 MB/s
```

**SHA-512 (current software):**

```
Per-block compression (128 bytes input):
  Message schedule: 64 rounds x ~8 ops = 512 ops
  Compression rounds: 80 x ~24 ops     = 1,920 ops
  Total per block:                      ~2,432 ops
  At ~2 IPC, 3.9 GHz:                  ~312 ns per block
  Throughput: 128 / 312e-9              = ~410 MB/s
```

SHA-512 is actually FASTER per byte than SHA-256 on 64-bit CPUs because it
processes 128-byte blocks with 64-bit arithmetic that maps naturally to x86-64
registers. This is relevant for Ed25519 which uses SHA-512 exclusively.

### 2.3 SIMD Optimization Opportunities

Since SHA-NI is unavailable, the main optimization avenue is AVX2:

**AVX2 SHA-256 Parallelism Strategy:**
- Cannot parallelize rounds within a single hash (data dependency on a..h)
- CAN parallelize 4 or 8 independent hash computations (multi-buffer)
- 8-way parallel SHA-256 via AVX2: ~2,500 MB/s aggregate throughput
- Useful for: batch signature verification, merkle tree hashing, package verification

**Message Schedule Optimization (single-hash speedup):**
- Current: W[64] array fully expanded (256 bytes on stack)
- Optimization: rolling W[16] window, compute on-the-fly
- Saves: 192 bytes stack, marginal speed improvement from better cache locality
- Estimated gain: ~5-10% single-hash throughput

**SHA-512 on x86-64 is already well-optimized** by nature:
- 64-bit rotations map to single `rorq` instructions
- 64-bit additions map to single `addq` instructions
- No byte-swapping overhead on modern CPUs (bswap is 1 cycle)

### 2.4 Memory Footprint

- SHA-256 context: 8 x uint32 state + 64-byte buffer + counters = ~112 bytes
- SHA-256 compress stack: W[64] = 256 bytes + 8 working vars = 288 bytes
- SHA-512 context: 8 x uint64 state + 128-byte buffer + counters = ~216 bytes
- SHA-512 compress stack: W[80] = 640 bytes + 8 working vars = 704 bytes
- **SHA-512 W[80] is the largest single stack allocation in all 5 primitives**
- Not a problem on desktop (default stack is 1-8 MB), but notable for kernel threads

## 3. ChaCha20-Poly1305 (chacha20.c, 478 lines)

### 3.1 Current Implementation Analysis

**ChaCha20 Architecture:**
- Standard 20-round (10 double-round) construction per RFC 8439
- Scalar quarter-round macro `QR()` with `ROTL32()` on 32-bit words
- State: 16 x uint32_t = 64 bytes (working copy + initial state)
- Block function produces 64 bytes of keystream per invocation
- Encryption XORs keystream with plaintext byte-by-byte

**Quarter-Round Operation Count:**
- 4 additions + 4 XORs + 4 rotations = 12 operations per quarter-round
- 8 quarter-rounds per double-round (4 column + 4 diagonal)
- 10 double-rounds: 8 x 12 x 10 = 960 operations per block
- Plus: 16 additions for state+working merge, 16 stores = 992 total ops

**Poly1305 Architecture:**
- 3-limb accumulator: {acc_lo, acc_hi, acc_top} using uint64_t
- Uses `__uint128_t` for intermediate products (compiler generates mulq/imulq)
- Schoolbook 3x2 multiplication for 130-bit x 128-bit product
- Partial reduction via 2^130 = 5 (mod p)
- Stack-local scratch buffer: POLY_SCRATCH_MAX = 32,816 bytes

### 3.2 Throughput Estimates on i5-8265U

**ChaCha20 (current scalar):**

```
Per-block (64 bytes output):
  Quarter-rounds: 960 ops
  State merge + serialize: 32 ops
  At ~3 IPC, 3.9 GHz:  ~254 ns per 64-byte block
  Throughput: 64 / 254e-9 = ~252 MB/s

Per 16 KB TLS record:
  256 blocks x 254 ns = ~65 us
  With XOR overhead:   ~80 us total
```

**Poly1305 (current __uint128_t):**

```
Per 16-byte block:
  Load + accumulate: ~10 ops
  3x2 schoolbook mul (__uint128_t -> mulq pairs): ~12 multiplies
  Each mulq: ~3-4 cycles on Whiskey Lake
  Reduction (3 shifts + adds): ~15 ops
  Total: ~80 cycles per 16-byte block
  At 3.9 GHz: ~20.5 ns per block

Per 16 KB TLS record:
  1024 blocks x 20.5 ns = ~21 us
```

**ChaCha20-Poly1305 Combined (16 KB record):**

```
  ChaCha20 encryption:  ~80 us
  Poly1305 MAC:          ~21 us
  Key generation:        ~0.3 us (1 ChaCha20 block for poly_key)
  Padding/length:        ~0.1 us
  Total:                 ~101 us per 16 KB record
  Throughput: 16384 / 101e-6 = ~162 MB/s
```

**With AVX2 SIMD (Sprint 4+ target):**

```
ChaCha20 AVX2 (4-block parallel, 256-bit vectors):
  4 blocks processed simultaneously
  Quarter-round: 4 x vpaddq + 4 x vpxorq + 4 x vprold = 12 AVX2 ops
  10 double-rounds x 8 QR: 960 AVX2 ops -> 240 per-block equivalent
  Throughput: ~800+ MB/s (4x theoretical, ~3.2x realistic with overhead)

Poly1305 with __uint128_t is already near-optimal on x86-64:
  mulq instruction pair is the fastest 64x64->128 path
  AVX2 does not help Poly1305 (no 64-bit multiply in AVX2)
  VPCLMULQDQ (future, not on Whiskey Lake) would help but not available

ChaCha20-Poly1305 Combined with AVX2:
  ChaCha20: ~25 us (3.2x speedup)
  Poly1305: ~21 us (unchanged)
  Total:    ~46 us per 16 KB record
  Throughput: ~356 MB/s
```

**Speedup from AVX2 ChaCha20: ~2.2x combined (162 --> 356 MB/s)**

### 3.3 Stack Pressure Warning

```
POLY_SCRATCH_MAX = 32,816 bytes (line 371)
+ poly_key_block[64] + poly_key[32] + keystream[64]
+ pad[16] + tag[16] + computed_tag[16]
= ~33,024 bytes peak stack usage in chacha20_poly1305_core()
```

This is the largest stack frame in the entire crypto subsystem. On the XENOS
kernel, thread stack sizes may be limited. If the default kernel thread stack is
8 KB (common for microkernel designs), this will cause a stack overflow.

**Recommendation:** Make POLY_SCRATCH_MAX caller-configurable or implement
streaming Poly1305 to avoid buffering the entire record.

### 3.4 Memory Footprint

- ChaCha20 block: 128 bytes (state[16] + working[16])
- Poly1305 core: ~80 bytes (r_bytes + s_bytes + accumulator limbs)
- Scratch buffer: 32,816 bytes (dominates)
- **Total per-call: ~33 KB** -- this is the P1 stack issue from previous audit

## 4. Ed25519 (ed25519.c, 1,023 lines)

### 4.1 Current Implementation Analysis

**Architecture:**
- GF(2^255-19) field: 10-limb representation, alternating 25/26-bit limbs
- Limb type: `int32_t` (fe10) -- differs from x25519.c which uses `uint64_t` (fe)
- Schoolbook multiplication: 100 products per `fe10_mul()`, 64-bit intermediates
- Point representation: Extended twisted Edwards (X, Y, Z, T)
- Point addition: Hisil unified formula (8 field multiplies + 6 add/sub)
- Point doubling: 4 squarings + 4 multiplies + 3 add/sub
- Scalar multiplication: bit-by-bit double-and-add with `fe10_cmov()` (constant-time)
- 255 iterations: 255 doublings + up to 255 additions (always computed, conditionally selected)

**Critical Performance Path -- Scalar Multiplication:**

```
ge_scalarmult_base() / ge_scalarmult():
  255 iterations of:
    ge_p3_dbl():  4 fe10_sq + 4 fe10_mul + 3 fe10_add/sub
    ge_p3_add():  4 fe10_mul + 6 fe10_add/sub (always computed)
    4 x fe10_cmov(): 4 x 10 conditional moves

  Per iteration:
    8 fe10_mul + 4 fe10_sq = 12 field multiplications
    9 fe10_add/sub = trivial (10 additions each)
    4 fe10_cmov = 40 conditional word operations

  Total field multiplications: 255 x 12 = 3,060 fe10_mul calls
  (fe10_sq is implemented as fe10_mul(h, f, f) -- not optimized)
```

### 4.2 Throughput Estimates on i5-8265U

**fe10_mul() Cost:**

```
100 int64_t multiplications (schoolbook 10x10)
  - 50 of these pre-multiplied by 19 (wrap-around)
  - 10 pre-multiplied by 2 (even-indexed doublings)
Total multiply ops: 100 mulq-equivalent
Plus: 10 precompute (2*f) + 9 precompute (19*g) = 19 multiplies
Total: ~119 multiply operations
Carry propagation: ~24 ops
At 3 cycles per imulq on Whiskey Lake: ~420 cycles per fe10_mul()
```

**Point Operations:**

```
ge_p3_dbl():
  4 fe10_sq + 4 fe10_mul + 3 fe10_add/sub + extra add
  = 8 x 420 + 3 x 30 = 3,450 cycles

ge_p3_add():
  8 fe10_mul + 6 fe10_add/sub
  = 8 x 420 + 6 x 30 = 3,540 cycles

Per scalar mult iteration:
  dbl + add + 4 cmov = 3,450 + 3,540 + 120 = 7,110 cycles

Total scalar multiplication:
  255 x 7,110 = 1,813,050 cycles
  At 3.9 GHz: ~465 us per scalarmult
```

**Ed25519 Sign (complete):**

```
1 x SHA-512(seed, 32):           ~0.2 us (tiny input)
1 x ge_scalarmult_base (R=r*B):  ~465 us
1 x SHA-512(R||pub||msg):        ~0.3 us (small msg) + O(msg_len)
1 x sc_reduce (k mod l):         ~1 us
1 x schoolbook 12x12 multiply:   ~2 us
1 x sc_reduce (S mod l):         ~1 us
Total (small message):            ~470 us
Throughput: ~2,128 signs/sec
```

**Ed25519 Verify (complete):**

```
1 x ge_frombytes (decode pub):        ~20 us (1 inversion + sqrt)
1 x SHA-512(R||pub||msg):             ~0.3 us
1 x sc_reduce:                        ~1 us
1 x ge_scalarmult_base ([S]B):        ~465 us
1 x ge_scalarmult ([k](-A)):          ~465 us
1 x ge_p3_add:                        ~1 us
1 x ge_p3_tobytes (encode, 1 inv):    ~15 us
1 x xsec_ct_eq (32 bytes):            ~0.01 us
Total (small message):                 ~967 us
Throughput: ~1,034 verifies/sec
```

### 4.3 Optimization Opportunities

**HIGH IMPACT -- fe10_sq() specialization:**
- Currently: `fe10_sq(h, f) { fe10_mul(h, f, f); }` (line 128)
- Squaring has only 55 unique products (not 100) due to symmetry
- a[i]*a[j] == a[j]*a[i], so half the products can be doubled instead of recomputed
- Expected speedup: ~1.5-1.7x on field squaring
- Since squaring is ~40% of scalar mult operations, overall ~1.2x speedup
- Estimated verify with fe10_sq optimization: ~800 us (~1,250 verifies/sec)

**MEDIUM IMPACT -- Precomputed base point table:**
- Current: bit-by-bit scan with full ge_p3_add every iteration
- Optimization: precompute [1]B, [2]B, [4]B, [8]B for 4-bit window
- Reduces 255 full additions to ~64 table lookups + additions
- Estimated speedup for ge_scalarmult_base: ~2.5x
- Sign time: ~200 us; Verify time: ~670 us

**LOW IMPACT -- 5-limb uint64_t representation (like x25519.c):**
- Ed25519 uses fe10 (10 x int32_t), X25519 uses fe (5 x uint64_t)
- 5-limb requires `__uint128_t` intermediates (25 products vs 100)
- Estimated speedup: ~2x on field multiplication
- Significant rewrite required; both representations are standard

### 4.4 Memory Footprint

- fe10: 10 x int32_t = 40 bytes per field element
- ge_p3: 4 x fe10 = 160 bytes per point
- ge_scalarmult stack: Q + T + r = 3 x 160 = 480 bytes
- ge_p3_add stack: 8 x fe10 + 2 tmp = 400 bytes
- fe10_mul stack: 20 x int64_t + 10 x int64_t + 10 x int64_t = 320 bytes
- sc_reduce: 24 x int64_t + carry[12] = 288 bytes
- Sign total stack: ~2.5 KB (multiple nested calls)
- Verify total stack: ~3.2 KB (two scalar multiplications active)
- **Stack usage is reasonable for desktop; tight for 8 KB kernel stacks**

## 5. X25519 (x25519.c, 475 lines)

### 5.1 Current Implementation Analysis

**Architecture:**
- GF(2^255-19) field: 5-limb, 51-bit per limb, `uint64_t`
- Uses `__uint128_t` (typedef u128) for 64x64->128 multiplication
- Schoolbook multiplication: 25 primary products + 10 wrap-around (x19)
- Montgomery ladder: 255 iterations, constant-time via `fe_cswap()`
- Per iteration: 4 `fe_mul()` + 6 `fe_sq()` + 4 `fe_add()` + 4 `fe_sub()`
  (Actually: 4 mul + 2 sq + 4 add + 4 sub from reading the ladder body)

**Critical observation:** `fe_sq()` is implemented as `fe_mul(a, a)` (line 167)
-- same inefficiency as ed25519.c. Squaring symmetry is not exploited.

### 5.2 Throughput Estimates on i5-8265U

**fe_mul() Cost (5-limb, __uint128_t):**

```
25 primary u128 multiplications (cast to __uint128_t, compiled to mulq)
10 wrap-around multiplications (a[i]*19 precomputed, then x b[j])
  Actually: 10 products use (a.v[i] * P19) pre-multiplied as uint64_t
  Then cast to u128 for the product -- this is 10 mulq pairs
Total multiplies: 35 mulq instructions (each ~3-4 cycles on Whiskey Lake)
Carry propagation: 6 shifts + 6 masks + 6 adds + 1 wrap-around
Total: ~35 x 3.5 + 30 = ~152 cycles per fe_mul()
```

**Montgomery Ladder (255 iterations):**

```
Per iteration (reading x25519_ladder, lines 376-396):
  fe_add(x_2, z_2) -> A:          1 fe_add + 1 fe_reduce
  fe_sq(A) -> AA:                  1 fe_mul = 152 cycles
  fe_sub(x_2, z_2) -> B:          1 fe_sub + 1 fe_reduce
  fe_sq(B) -> BB:                  1 fe_mul = 152 cycles
  fe_sub(AA, BB) -> E:             1 fe_sub
  fe_add(x_3, z_3) -> C:          1 fe_add
  fe_sub(x_3, z_3) -> D:          1 fe_sub
  fe_mul(D, A) -> DA:              1 fe_mul = 152 cycles
  fe_mul(C, B) -> CB:              1 fe_mul = 152 cycles
  fe_sq(fe_add(DA, CB)) -> x_3:   1 fe_add + 1 fe_mul = 152 cycles
  fe_mul(x_1, fe_sq(sub)) -> z_3: 1 fe_sub + 2 fe_mul = 304 cycles
  fe_mul(AA, BB) -> x_2:          1 fe_mul = 152 cycles
  fe_mul(a24_fe, E) + AA:         1 fe_mul + 1 fe_add = 152 cycles
  fe_mul(E, aa_a24e) -> z_2:      1 fe_mul = 152 cycles

  Total multiplies per iteration: ~10 fe_mul (including sq-as-mul)
  Total per iteration: 10 x 152 + 6 x 30 (add/sub) = 1,700 cycles

Total ladder: 255 x 1,700 = 433,500 cycles

fe_invert (255 squarings + 11 multiplies):
  255 x 152 + 11 x 152 = 40,432 cycles

Total X25519 scalar multiplication:
  Ladder + inversion: 433,500 + 40,432 = 473,932 cycles
  At 3.9 GHz: ~121 us

Full DH exchange (xsec_x25519_shared):
  clamp + ladder + inversion + zero-check
  ~122 us per key exchange
  Throughput: ~8,197 DH exchanges/sec
```

### 5.3 Optimization Opportunities

**HIGH IMPACT -- fe_sq() specialization:**
- Currently fe_sq calls fe_mul(a, a) -- 35 multiplies
- Squaring symmetry: a[0]*a[1] == a[1]*a[0], so 10 of 25 primary products are duplicates
- Optimized: 15 unique products doubled + 5 diagonal squares = 20 mulq
- Plus 6 wrap-around (vs 10): total ~26 mulq
- Estimated fe_sq: ~100 cycles (vs 152 for fe_mul)
- 6 of 10 per-iteration ops are squarings: saves ~312 cycles/iteration
- Total speedup: ~15% on X25519 (121 us -> ~103 us)

**MEDIUM IMPACT -- Assembly mulq/adcq for fe_mul:**
- Compiler-generated code from `__uint128_t` casts is generally good
- Hand-written x86-64 assembly can use mulq + adcq chains more efficiently
- Saves register spills in the schoolbook accumulation
- Estimated gain: ~10-15% (depends on compiler quality)
- Risk: Maintenance burden, must be conditionally compiled for x86-64

**LOW IMPACT -- a24 as immediate rather than fe:**
- Line 392-394: constructs `a24_fe` as a full field element every iteration
- `fe_mul(a24_fe, E)` does a full 5x5 multiplication for a single-limb x 5-limb
- Specialized `fe_mul_121665()` would replace 35 mulq with 5 mulq + reduction
- Saves: ~120 cycles per iteration = 30,600 cycles total = ~8 us
- Easy win with low risk

### 5.4 Memory Footprint

- fe: 5 x uint64_t = 40 bytes per field element
- fe_mul stack: 5 x u128 = 80 bytes accumulators
- Ladder: 7 named fe (x_1..z_3, swap) = 280 bytes
- Per-iteration temporaries: 9 fe (A, AA, B, BB, E, C, D, DA, CB, a24_fe, aa_a24e) = 440 bytes
- fe_invert: 9 fe + loop = 360 bytes
- **Total X25519 stack: ~1.2 KB** -- very efficient

### 6.3 AI Pipeline Impact Assessment

For the XMIND AI inference path (Ollama on localhost):

```
Current path: HTTP plaintext on 127.0.0.1:11434 -- NO crypto in hot path
Future path:  If XNET+TLS replaces system networking for Ollama:
  - Typical AI query:    ~500 bytes prompt, ~2000 bytes response
  - ChaCha20-Poly1305:   500B encrypt + 2000B decrypt = ~0.4 us + ~1.6 us = 2 us
  - AES-256-GCM (SW):    500B encrypt + 2000B decrypt = ~2.5 ms + ~10 ms = 12.5 ms
```

## 8. AES-NI Integration Guide for aes_gcm.c

### 8.1 Prerequisites

**CPU Feature Detection:**
```c
/* Detect AES-NI at runtime via CPUID */
static bool aes_ni_available(void) {
    uint32_t eax, ebx, ecx, edx;
    /* CPUID leaf 1: ECX bit 25 = AES-NI */
    __asm__ volatile("cpuid"
        : "=a"(eax), "=b"(ebx), "=c"(ecx), "=d"(edx)
        : "a"(1), "c"(0));
    return (ecx >> 25u) & 1u;
}
```

**Compiler Flags:**
```
clang -target x86_64-unknown-none-elf -ffreestanding -maes -mpclmul ...
```

**Header (freestanding-safe):**
```c
#include <immintrin.h>   /* _mm_aesenc_si128, _mm_aeskeygenassist_si128 */
/* OR for freestanding, use __builtin_ia32_aesenc128 directly */
```

### 8.2 AES-NI Key Expansion

```c
/*
 * AES-256 key expansion using AESKEYGENASSIST.
 * Produces 15 round keys (14 rounds + initial) in __m128i format.
 * Must be called once per key; round keys cached in xsec_aes256_key_t.
 */
static void aes256_key_expand_ni(const uint8_t key[32],
                                  __m128i rk[15]) {
    __m128i key_lo = _mm_loadu_si128((const __m128i *)key);
    __m128i key_hi = _mm_loadu_si128((const __m128i *)(key + 16));
    rk[0] = key_lo;
    rk[1] = key_hi;

    /* Round 2-14: alternating AESKEYGENASSIST on hi/lo halves */
    /* Each round uses a different RCON value */
    /* Helper macro for the key expansion assist: */
    #define AES256_EXPAND_EVEN(rk_out, rk_prev_lo, rk_prev_hi, rcon) do { \
        __m128i assist = _mm_aeskeygenassist_si128(rk_prev_hi, rcon); \
        assist = _mm_shuffle_epi32(assist, 0xFF); /* broadcast word 3 */ \
        __m128i tmp = _mm_slli_si128(rk_prev_lo, 4); \
        tmp = _mm_xor_si128(rk_prev_lo, tmp); \
        tmp = _mm_xor_si128(tmp, _mm_slli_si128(tmp, 4)); \
        tmp = _mm_xor_si128(tmp, _mm_slli_si128(tmp, 4)); \
        rk_out = _mm_xor_si128(tmp, assist); \
    } while(0)

    #define AES256_EXPAND_ODD(rk_out, rk_prev_lo, rk_prev_hi) do { \
        __m128i assist = _mm_aeskeygenassist_si128(rk_prev_hi, 0); \
        assist = _mm_shuffle_epi32(assist, 0xAA); /* broadcast word 2 */ \
        __m128i tmp = _mm_slli_si128(rk_prev_lo, 4); \
        tmp = _mm_xor_si128(rk_prev_lo, tmp); \
        tmp = _mm_xor_si128(tmp, _mm_slli_si128(tmp, 4)); \
        tmp = _mm_xor_si128(tmp, _mm_slli_si128(tmp, 4)); \
        rk_out = _mm_xor_si128(tmp, assist); \
    } while(0)

    AES256_EXPAND_EVEN(rk[2],  rk[0], rk[1],  0x01);
    AES256_EXPAND_ODD (rk[3],  rk[1], rk[2]);
    AES256_EXPAND_EVEN(rk[4],  rk[2], rk[3],  0x02);
    AES256_EXPAND_ODD (rk[5],  rk[3], rk[4]);
    AES256_EXPAND_EVEN(rk[6],  rk[4], rk[5],  0x04);
    AES256_EXPAND_ODD (rk[7],  rk[5], rk[6]);
    AES256_EXPAND_EVEN(rk[8],  rk[6], rk[7],  0x08);
    AES256_EXPAND_ODD (rk[9],  rk[7], rk[8]);
    AES256_EXPAND_EVEN(rk[10], rk[8], rk[9],  0x10);
    AES256_EXPAND_ODD (rk[11], rk[9], rk[10]);
    AES256_EXPAND_EVEN(rk[12], rk[10],rk[11], 0x20);
    AES256_EXPAND_ODD (rk[13], rk[11],rk[12]);
    AES256_EXPAND_EVEN(rk[14], rk[12],rk[13], 0x40);

    #undef AES256_EXPAND_EVEN
    #undef AES256_EXPAND_ODD
}
```

### 8.3 AES-NI Block Encryption

```c
/*
 * AES-256 encrypt one 128-bit block using AES-NI.
 * 13 rounds of AESENC + 1 round of AESENCLAST.
 * Inherently constant-time (hardware pipeline).
 */
static inline __m128i aes256_encrypt_block_ni(const __m128i rk[15],
                                               __m128i plaintext) {
    __m128i state = _mm_xor_si128(plaintext, rk[0]);   /* Initial round key */
    state = _mm_aesenc_si128(state, rk[1]);
    state = _mm_aesenc_si128(state, rk[2]);
    state = _mm_aesenc_si128(state, rk[3]);
    state = _mm_aesenc_si128(state, rk[4]);
    state = _mm_aesenc_si128(state, rk[5]);
    state = _mm_aesenc_si128(state, rk[6]);
    state = _mm_aesenc_si128(state, rk[7]);
    state = _mm_aesenc_si128(state, rk[8]);
    state = _mm_aesenc_si128(state, rk[9]);
    state = _mm_aesenc_si128(state, rk[10]);
    state = _mm_aesenc_si128(state, rk[11]);
    state = _mm_aesenc_si128(state, rk[12]);
    state = _mm_aesenc_si128(state, rk[13]);
    return _mm_aesenclast_si128(state, rk[14]);
}
```

### 8.4 PCLMULQDQ-Accelerated GHASH

```c
/*
 * GF(2^128) multiplication using PCLMULQDQ (carryless multiply).
 * Implements GHASH multiplication for AES-GCM.
 * Uses Karatsuba decomposition for efficiency.
 */
static inline __m128i ghash_mul_hw(__m128i H, __m128i X) {
    /* Karatsuba: H*X = (H_hi*X_hi)*x^128 + ((H_hi+H_lo)*(X_hi+X_lo)
     *              - H_hi*X_hi - H_lo*X_lo)*x^64 + H_lo*X_lo */
    __m128i tmp0 = _mm_clmulepi64_si128(H, X, 0x00);  /* H_lo * X_lo */
    __m128i tmp1 = _mm_clmulepi64_si128(H, X, 0x11);  /* H_hi * X_hi */
    __m128i tmp2 = _mm_clmulepi64_si128(H, X, 0x01);  /* H_lo * X_hi */
    __m128i tmp3 = _mm_clmulepi64_si128(H, X, 0x10);  /* H_hi * X_lo */
    __m128i mid  = _mm_xor_si128(tmp2, tmp3);

    /* Combine: result = {tmp1_hi, tmp1_lo ^ mid_hi, tmp0_hi ^ mid_lo, tmp0_lo} */
    __m128i mid_hi = _mm_srli_si128(mid, 8);
    __m128i mid_lo = _mm_slli_si128(mid, 8);
    __m128i result_hi = _mm_xor_si128(tmp1, mid_hi);
    __m128i result_lo = _mm_xor_si128(tmp0, mid_lo);

    /* Reduce 256-bit result mod x^128 + x^7 + x^2 + x + 1 */
    /* Using the standard Barrett/shift reduction */
    __m128i t0 = _mm_clmulepi64_si128(result_lo,
                    _mm_set_epi64x(0, 0x87), 0x00);  /* poly = x^7+x^2+x+1 */
    __m128i t1 = _mm_shuffle_epi32(t0, 78);           /* swap hi/lo 64-bit */
    result_lo = _mm_xor_si128(result_lo, t1);
    t0 = _mm_clmulepi64_si128(result_lo,
            _mm_set_epi64x(0, 0x87), 0x01);
    return _mm_xor_si128(_mm_xor_si128(result_lo, t0), result_hi);
}
```

### 8.5 Integration Architecture

The recommended integration pattern is **runtime dispatch**:

```c
/* In aes_gcm.c: */

/* Forward declarations for both paths */
static void aes256_encrypt_block_sw(const xsec_aes256_key_t *key,
                                     const uint8_t in[16], uint8_t out[16]);
static __m128i aes256_encrypt_block_ni(const __m128i rk[15], __m128i pt);

/* Runtime dispatch (checked once, cached in key structure) */
typedef struct {
    uint32_t round_keys[60];  /* Software path round keys (existing) */
    __m128i  rk_ni[15];       /* AES-NI round keys (new) */
    uint32_t num_rounds;
    bool     use_aesni;       /* Set once during key expand */
} xsec_aes256_key_t;

xsec_status_t xsec_aes256_key_expand(const uint8_t key[32],
                                      xsec_aes256_key_t *out) {
    /* Always expand software keys (for fallback) */
    expand_software(key, out);

    /* Try AES-NI path */
    if (aes_ni_available()) {
        aes256_key_expand_ni(key, out->rk_ni);
        out->use_aesni = true;
    } else {
        out->use_aesni = false;
    }
    return XSEC_OK;
}
```

## 9. AVX2 ChaCha20 SIMD Opportunities

### 9.1 4-Block Parallel Strategy

ChaCha20 is ideal for SIMD because:
- All operations are 32-bit (add, XOR, rotate) -- map to vpaddq, vpxord, vprold
- 4 independent states can be processed in 256-bit YMM registers
- No data dependencies between blocks (counter increments independently)

**Approach:** Process 4 consecutive keystream blocks in parallel:

```
Block N:   state[0..15] with counter = N
Block N+1: state[0..15] with counter = N+1
Block N+2: state[0..15] with counter = N+2
Block N+3: state[0..15] with counter = N+3

Pack word[i] from all 4 blocks into one YMM register:
  ymm_state_0 = { block_0[0], block_1[0], block_2[0], block_3[0] }
  ymm_state_1 = { block_0[1], block_1[1], block_2[1], block_3[1] }
  ...
  ymm_state_15 = { block_0[15], block_1[15], block_2[15], block_3[15] }

Then QR operates on 4 blocks simultaneously:
  ymm_a = _mm256_add_epi32(ymm_a, ymm_b);  // 4 blocks at once
  ymm_d = _mm256_xor_si256(ymm_d, ymm_a);
  ymm_d = _mm256_or_si256(
      _mm256_slli_epi32(ymm_d, 16),
      _mm256_srli_epi32(ymm_d, 16));  // rotate left 16
  ...
```

### 9.2 Expected AVX2 Throughput

```
Per 4-block group (256 bytes output):
  Same number of SIMD ops as 1 scalar block: ~240 instructions
  AVX2 32-bit ops: ~1 cycle each (IPC ~2 for integer SIMD on Whiskey Lake)
  Total: ~120 cycles for 4 blocks = 30 cycles per block

  Overhead: state setup, XOR with plaintext, tail handling = ~50 cycles
  Effective: ~80 cycles per block = ~20 ns at 3.9 GHz

  Throughput: 64 bytes / 20 ns = 3.2 GB/s (theoretical peak)
  Realistic (with memory + XOR): ~800 MB/s
```

### 9.3 Tail Block Handling

When input length is not a multiple of 256 bytes (4 blocks):
- Process complete 4-block groups with AVX2
- Fall back to scalar `chacha20_block()` for remaining 1-3 blocks
- No performance penalty since tail is processed once per record
