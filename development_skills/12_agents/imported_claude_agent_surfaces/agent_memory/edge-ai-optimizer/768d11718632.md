## 1. MODEL DISTRIBUTION VIA XPKG

### Assessment: PARTIALLY READY — requires architectural changes

**Problem:** XPKG is designed for traditional software packages, not 3GB+ AI model weights.

**Capacity Constraints Identified:**

| Constant | Value | Impact on AI Models |
|----------|-------|---------------------|
| `XPKG_MAX_FILES_PER_PKG` | 256 | Sufficient — model is 1-3 files |
| `xpkg_file_entry_t.size` | `uint32_t` | BLOCKER: max 4GB per file, but `xpkg_archive_header_t.total_size` is also `uint32_t` — 4GB limit per archive |
| `xpkg_format_parse()` | Takes `const uint8_t *buf, uint32_t buf_len` | BLOCKER: Requires entire archive in memory. A 3GB model cannot be loaded into a single buffer on a 16GB system alongside Ollama |
| `XPKG_FLAG_COMPRESSED` | Reserved (bit 0) | NOT IMPLEMENTED: Compression would reduce 3GB download to ~2GB but the flag exists as a stub only |
| `XPKG_FLAG_DELTA` | Reserved (bit 1) | NOT IMPLEMENTED: Delta updates would allow incremental model updates (critical for quantization swaps) |

**Recommendations (ranked by impact):**

1. **Streaming parse API** (HIGH IMPACT, MEDIUM COMPLEXITY): Add `xpkg_format_parse_stream()` that processes the archive in chunks rather than requiring the full buffer in memory. For a 3GB model, the current API would require 3GB of contiguous buffer space — impossible alongside a running Ollama instance.

2. **Large file support** (HIGH IMPACT, LOW COMPLEXITY): Change `total_size` in `xpkg_archive_header_t` and `size` in `xpkg_file_entry_t` to `uint64_t`. This is a format version bump (v1 -> v2) but essential for AI model packages.

3. **Implement compression** (MEDIUM IMPACT, HIGH COMPLEXITY): The `XPKG_FLAG_COMPRESSED` flag exists but is not implemented. GGUF model files compress ~30-40% with LZ4. This saves ~1GB on download and storage.

4. **Implement delta packages** (MEDIUM IMPACT, HIGH COMPLEXITY): When swapping quantizations (Q4 -> Q6), delta packages could reduce the update size from 3GB to ~500MB.

**Tradeoff acknowledged:** Streaming parse adds complexity to the format parser and breaks the zero-allocation constraint (needs state tracking). Recommend a separate `xpkg_model_t` package type with streaming-aware APIs.

## 2. MODEL INTEGRITY VIA XSEC Ed25519

### Assessment: READY — excellent fit

**Ed25519 API is well-suited for model signing:**

- `xsec_ed25519_sign()` / `xsec_ed25519_verify()` — 64-byte signatures over arbitrary-length messages
- `xsec_sha256()` — content hash of model weights (used as the message for Ed25519)
- `xpkg_metadata_t.content_hash[32]` — SHA-256 slot already exists
- `xpkg_metadata_t.signature[64]` — Ed25519 signature slot already exists

**Verification workflow for AI models:**

```
1. SHA-256(model_weights.gguf) -> content_hash[32]
2. Ed25519_verify(publisher_pub_key, content_hash, signature) -> XSEC_OK / XSEC_ERR_VERIFY_FAIL
3. XPKG Gate 1 (SIGNATURE) + Gate 2 (INTEGRITY) enforce this automatically
```

**Memory impact of verification:** Negligible. `xsec_sha256_ctx_t` is 108 bytes. SHA-256 can be computed incrementally in streaming fashion over a 3GB file using `xsec_sha256_update()` with small chunk reads.

**Key revocation:** `xpkg_key_is_revoked()` + `XPKG_MAX_REVOKED_KEYS=64` provides defense against compromised model publishers. The 9-gate refusal machine (Gate 4: PROVENANCE) checks revocation status.

**Recommendation:** No changes needed. The existing XSEC Ed25519 + XPKG gate system is production-ready for model integrity verification.

## 3. SECURE MODEL DOWNLOAD VIA XNET + TLS 1.3

### Assessment: READY WITH CAVEATS

**TLS 1.3 Implementation:**

- `xsec_tls_ctx_t` provides full TLS 1.3 client with X25519 key exchange + AES-256-GCM / ChaCha20-Poly1305
- Pluggable transport via `xsec_tls_transport_t` callbacks — can be wired directly to XNET TCP sockets
- X.509 certificate chain verification (`xsec_x509_verify_chain()`) ensures server identity

**Download pipeline:**

```
DNS resolve (xnet_dns_resolve) -> TCP connect (xnet_tcp_connect) -> TLS handshake (xsec_tls_handshake) -> TLS recv loop (xsec_tls_recv) -> SHA-256 verify -> Ed25519 verify -> XPKG install
```

**Caveat — HTTP not implemented:** XNET provides raw TCP + TLS, but there is no HTTP/1.1 or HTTP/2 client. Model download from a repository URL requires an HTTP client layer on top of TLS. This is a Sprint 4+ dependency.

**Recommendation:** Implement a minimal HTTP/1.1 GET client (~300 lines) on top of `xsec_tls_send/recv` for model fetching. Chunked transfer-encoding support is critical for streaming large downloads.

## 4. BANDWIDTH: 16KB TCP RING BUFFERS FOR 3GB DOWNLOADS

### Assessment: FUNCTIONAL BUT SUBOPTIMAL

**Current TCP buffer configuration:**

| Parameter | Value | Impact |
|-----------|-------|--------|
| `TCP_RCVBUF_SIZE` | 16KB | Limits TCP receive window |
| `TCP_SNDBUF_SIZE` | 16KB | Irrelevant for downloads |
| `TCP_MSS_DEFAULT` | 1460 bytes | Standard for Ethernet |
| `TCP_TCB_MAX` | 128 | Far more than needed |

**Bandwidth calculation (pessimistic):**

- Window size: 16KB = 16,384 bytes
- RTT (typical broadband): 30ms
- Theoretical throughput: 16,384 / 0.030 = ~546 KB/s = ~4.4 Mbps
- Time to download 3GB model: 3,072 MB / 0.546 MB/s = **94 minutes**

**Bandwidth calculation (with window scaling):**

- TCP window scaling (RFC 7323) is NOT implemented in XNET
- Without window scaling, maximum window = 65,535 bytes (16-bit field)
- Even at 64KB window: 64KB / 30ms = ~2.1 MB/s, download = ~24 minutes

**This is unacceptable for AI model delivery.**

**Recommendations (ranked by impact):**

1. **Increase TCP_RCVBUF_SIZE to 256KB for model downloads** (HIGH IMPACT, LOW COMPLEXITY): At 256KB window / 30ms RTT = ~8.5 MB/s = 6 minutes for 3GB. Memory cost: 256KB per connection (acceptable for a single download connection).

2. **Implement TCP window scaling (RFC 7323)** (HIGH IMPACT, MEDIUM COMPLEXITY): Allows windows up to 1GB. Essential for high-bandwidth downloads.

3. **Background download with resume** (MEDIUM IMPACT, MEDIUM COMPLEXITY): Download model in background, save progress to disk, resume on failure. Prevents re-downloading 3GB on connection drop.

4. **Consider multi-connection download** (LOW IMPACT, HIGH COMPLEXITY): HTTP range requests across 4 connections. Overkill for initial implementation.

**Tradeoff acknowledged:** Increasing buffer to 256KB adds 256KB RAM per TCP connection. With `TCP_TCB_MAX=128`, worst case is 32MB — but model download should use a dedicated TCB pool of 1-2 connections. Recommend a `TCP_RCVBUF_MODEL_SIZE` constant for download-specific connections.

## 5. MEMORY PRESSURE: XNET + XSEC + XPKG ALONGSIDE OLLAMA

### Assessment: MANAGEABLE — requires careful lifecycle coordination

**Memory footprint analysis (static allocations):**

| Component | Structure | Size | Count | Total |
|-----------|-----------|------|-------|-------|
| **XNET** | | | | |
| TCP TCBs | ~100KB each (snd_buf + rcv_buf + rexmit + ooo) | ~100KB | 128 max | **12.5MB max** |
| ARP cache | `ARP_CACHE_SIZE=256` entries | ~24B each | 256 | **6KB** |
| Route table | `XNET_ROUTE_MAX=32` entries | 20B each | 32 | **640B** |
| DNS cache | `DNS_CACHE_SIZE=128` entries | ~280B each | 128 | **35KB** |
| Gossip nodes | `GOSSIP_MAX_NODES=256` | ~50B each | 256 | **12.5KB** |
| | | | | **~12.6MB max** |
| **XSEC** | | | | |
| TLS context | `xsec_tls_ctx_t` | ~17.2KB | per session | **17.2KB** |
| Audit ring | `XSEC_AUDIT_RING_SIZE=1024` entries | ~80B each | 1024 | **80KB** |
| | | | | **~97KB** |
| **XPKG** | | | | |
| Registry | `XPKG_MAX_INSTALLED=512` packages | ~88KB each | 512 | **44MB max** |
| Dep graph | `XPKG_MAX_GRAPH_NODES=512` | ~160B each | 512 | **80KB** |
| Repo index | `XPKG_MAX_INSTALLED=512` entries | ~380B each | 512 | **190KB** |
| Install plan | `XPKG_INSTALL_PLAN_MAX=512` | ~76B each | 512 | **38KB** |
| | | | | **~44.3MB max** |

**Total Sprint 3 static memory (worst case): ~57MB**

**Critical finding: XPKG registry at 512 x ~88KB = 44MB is the dominant consumer.** Each `xpkg_package_t` contains a 256-entry file table (`xpkg_file_entry_t` at ~300B each = ~77KB per package). With 512 packages, this is enormous.

**Memory map during model download:**

```
Component                   RAM Usage
-------------------------------------------
OS + Desktop + Services     ~6GB
Ollama (model loaded)       ~3GB (Q4_K_M weights + KV cache)
XPKG registry (in use)      ~44MB (worst case, 512 packages)
XNET TCP (download conn)    ~100KB (1 TCB)
XSEC TLS (active session)   ~17KB
Download buffer (streaming) ~1MB (application-level read buffer)
-------------------------------------------
Total during model download: ~9.2GB
Available headroom:          ~6.8GB
```

**Verdict:** Memory pressure is manageable IF:
1. Ollama is not running inference during model download (model loaded but idle = OK)
2. XPKG does not load the full registry into RAM during download (lazy load)
3. Download uses streaming parse, not full-buffer load

**Recommendation:** Implement a model download coordinator that:
- Pauses Ollama inference during download (or unloads the model)
- Uses streaming SHA-256 verification (already supported by `xsec_sha256_update`)
- Limits TCP connections to 1-2 during model fetch
- Resumes on connection failure

## 6. FUTURE XMIND INTEGRATION (Sprint 6)

### APIs Sprint 3 provides that XMIND will need:

**From XNET:**
- `xnet_tcp_connect()` / `xnet_tcp_send()` / `xnet_tcp_recv()` — for Ollama HTTP API (localhost:11434). XMIND's inference pipeline communicates with Ollama over HTTP on loopback.
- `xnet_dns_resolve()` — for RAG document fetching (future web search grounding)
- mDNS (`xnet_mdns_query()`) — for discovering other GEN.OS devices on LAN (future federated inference)
- Gossip (`xnet_gossip_*`) — for mesh AI inference coordination across devices

**From XSEC:**
- `xsec_sha256()` — for RAG document deduplication (content-addressable indexing)
- `xsec_aes_gcm_encrypt/decrypt()` — for encrypting conversation history at rest
- `xsec_ed25519_verify()` — for verifying model weights after XPKG install
- Audit log — for tracking all AI inference security events

**From XPKG:**
- `xpkg_install()` — for model package installation
- `xpkg_gate_check()` — 9-gate verification before loading any model into Ollama
- `xpkg_get_info()` — for querying installed model metadata (version, quantization)
- `xpkg_upgrade()` — for hot-swapping model quantizations (Q4 <-> Q6)

**Missing APIs XMIND will need (Sprint 3 gaps):**

1. **No HTTP client** — XMIND needs HTTP POST to Ollama API. Requires HTTP/1.1 on top of XNET TCP.
2. **No streaming response parser** — Ollama returns NDJSON streaming responses. Need a line-delimited JSON parser.
3. **No local IPC** — XNET is network-oriented. XMIND may prefer Unix domain sockets or shared memory for Ollama communication (faster than TCP loopback).
4. **No model-specific XPKG package type** — No `.xmodel` or model-aware metadata (quantization level, context length, parameter count).

## 7. TLS CRYPTO + LLM INFERENCE COEXISTENCE

### Assessment: FEASIBLE — CPU contention is the risk, not memory

**Memory coexistence:**

| Workload | RAM |
|----------|-----|
| TLS session (1 active) | 17.2KB |
| Ollama loaded model | ~2.0GB (Q4_K_M weights) |
| Ollama KV cache (2048 ctx) | ~500MB |
| Total | ~2.5GB |

**Memory verdict:** TLS adds 17KB to a 2.5GB workload. Negligible. No memory conflict.

**CPU contention (the real risk):**

| Operation | CPU Profile |
|-----------|-------------|
| TLS handshake (X25519 + AES-GCM) | Burst: 100% single core for ~50ms |
| TLS record decrypt (AES-256-GCM) | Per-record: ~0.1ms for 16KB record |
| LLM token generation | Sustained: 100% all cores during inference |

**Scenario analysis:**

- **Downloading model while NOT running inference:** No conflict. TLS crypto is light.
- **Background TLS traffic during inference:** Minimal impact. AES-GCM decryption of a TLS record takes ~0.1ms. Token generation takes ~60ms. TLS processing is <0.2% of CPU time per token.
- **TLS handshake during inference:** Brief spike. X25519 scalar multiplication takes ~2ms on i5-8350U. This could add 1 token of latency. Acceptable.

**Thermal analysis:**

- Sustained LLM inference: CPU at ~75-80C
- Adding TLS background traffic: +1-2C (negligible)
- Downloading model (high-bandwidth TLS): +3-5C due to interrupt handling and memory copies. If CPU is already at 78C during inference, this could push to 83C — still below throttle point (90C).

**Recommendation:** Do not download models during active inference. Implement a workload coordinator:
1. If user is chatting with AI -> defer model updates
2. If AI is idle (no inference for 5+ minutes) -> allow background model download
3. If user requests model switch -> unload current model -> download/verify -> load new model
