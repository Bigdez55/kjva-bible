## 1. AES-256-GCM (aes_gcm.c) — PERFORMANCE ANALYSIS

### 1.1 Constant-Time S-Box Scan

**File:** `sec/xsec/crypto/aes_gcm.c`, lines 58-69

The `aes_sbox_ct()` function performs a full 256-iteration linear scan for every S-box
lookup. This is correct for constant-time security.

**Performance impact per AES-256 block encryption (14 rounds):**
- `aes_sub_bytes()` calls `aes_sbox_ct()` 16 times per round
- 14 rounds x 16 lookups x 256 iterations = **57,344 comparisons per block**
- Key expansion (`sub_word`) adds another 4 lookups x 15 words = 60 lookups
- Key expansion total: 60 x 256 = **15,360 additional comparisons**

**Estimated cost on i5-8250U (Kaby Lake R):**
- Each iteration: ~3 instructions (XOR, subtract, shift, mask, OR)
- At ~3.4 GHz boost: ~57,344 * 3 / 3.4e9 = ~50 us per block (software path)
- For a 16 KB TLS record (1024 blocks): ~51 ms per record encryption
- This is **well within tolerance** for TLS record processing at edge

**VERDICT: ACCEPTABLE for HP EliteBook x360.**

The 256-iteration scan costs ~50 us per block in software. For typical AI inference
payloads (JSON responses, <4KB), this is <15 ms — invisible to the user.

### 1.2 AES-NI Hardware Path — NOT IMPLEMENTED

**FINDING [P2-PERF-001]: AES-NI mentioned in comments (line 24) but no implementation exists.**

The comment at line 24 claims "AES-NI hardware path (x86_64 with AES-NI) --
single-instruction, inherently constant-time" but no AES-NI intrinsics
(`_mm_aesenc_si128`, etc.) are present anywhere in the codebase.

The Intel i5-8250U in the HP EliteBook x360 **does support AES-NI**. Using it would:
- Reduce AES-256 block encryption from ~50 us to ~0.3 us (>100x speedup)
- Eliminate the constant-time concern entirely (hardware is inherently CT)
- Reduce TLS record encryption for 16 KB from ~51 ms to ~0.3 ms

**Recommendation:** Implement AES-NI detection via CPUID and use hardware AES
when available. This is the single highest-impact optimization for TLS throughput
on the target hardware. Priority: P2 (Sprint 4/5 timeframe, before XMIND).

### 1.3 GF(2^8) Multiplication — Non-Constant-Time

**FINDING [P3-SEC-001]: `gf_mul()` (line 84) uses branching (`if (b & 1u)`, `if (hi_bit)`).**

This is used in `aes_mix_columns()` which operates on non-secret data (state bytes
that are already mixed), so the timing leak is limited. However, for defense-in-depth,
a branchless version using conditional XOR masks would be preferable.

Impact on edge AI: Negligible. MixColumns is called 13 times per block with 4
columns each = 52 calls to `gf_mul`. The branch misprediction cost is ~2-5 ns
per call, adding ~260 ns total — unmeasurable.

### 1.4 GHASH — GF(2^128) Multiplication

The `ghash_mul()` function (line 251) performs 128 iterations of bit-level
multiplication. This is standard schoolbook GF(2^128). On the i5-8250U with
PCLMULQDQ support, this could be accelerated ~50x, but the current software
implementation is adequate for Sprint 3 payloads.

### 1.5 Memory Footprint

- Stack: ~96 bytes state + 16 bytes keystream + 240 round_keys = ~356 bytes transient
- `xsec_aes256_key_t`: 244 bytes (60 words + num_rounds)
- S-box table: 256 bytes (`.rodata`, shared)
- Rcon table: 44 bytes

**Total static: ~300 bytes. Total transient per call: ~600 bytes. Negligible.**

## 2. ChaCha20-Poly1305 (chacha20.c) — PERFORMANCE ANALYSIS

### 2.1 ChaCha20 Block Function

20 rounds (10 double-rounds), 8 quarter-rounds each = 80 quarter-rounds.
Each QR: 4 additions, 4 XORs, 4 rotations = 12 operations.
Total: 80 * 12 = 960 operations per 64-byte keystream block.

**Estimated cost:** ~0.3 us per block on i5-8250U.
For 16 KB payload: 256 blocks x 0.3 us = ~77 us. **Excellent performance.**

ChaCha20 is inherently constant-time (no table lookups, no data-dependent branches).
This makes it the **preferred cipher suite for edge deployment** over AES-256-GCM
(until AES-NI is implemented).

### 2.2 Poly1305 MAC

Uses `__uint128_t` for 130-bit arithmetic — Clang on x86_64 compiles this to
efficient `mul`/`imul` instructions. The 3x2 schoolbook multiply with partial
reduction is ~20 multiplications per 16-byte block.

**Estimated cost:** ~0.5 us per 16-byte block. For 16 KB: ~512 us. Acceptable.

### 2.3 CRITICAL FINDING: Static Poly1305 Scratch Buffer

**FINDING [P1-SEC-002]: `poly_scratch` at line 367 is declared `static`.**

```c
static uint8_t poly_scratch[POLY_SCRATCH_MAX];  // POLY_SCRATCH_MAX = 32,848 bytes
```

This has two implications:

1. **Thread safety:** If two concurrent TLS sessions call ChaCha20-Poly1305
   simultaneously, they will corrupt each other's MAC computation. With
   `OLLAMA_NUM_PARALLEL=1` this is currently safe, but it is a latent bug
   for any future multi-session TLS use.

2. **Memory:** 32,848 bytes permanently allocated in `.bss` even when ChaCha20
   is not in use. This is wasteful but not critical (32 KB of 16 GB).

3. **Security:** The `xsec_secure_zero(poly_scratch, scratch_pos)` at line 427
   correctly zeroes after use. However, the static lifetime means the buffer
   is a persistent target for memory disclosure attacks.

**Recommendation:** Convert to stack allocation with a maximum bound matching
the TLS record limit (16,384 + overhead), or pass a caller-provided scratch
buffer. Priority: P1 for thread safety, P2 for security.

### 2.4 Memory Footprint

- `chacha20_block()`: 128 bytes stack (state + working)
- `chacha20_encrypt()`: 64 bytes stack (keystream)
- `poly1305_mac()`: ~80 bytes stack (r_bytes, s_bytes, block, u128 accumulators)
- `poly_scratch` static: **32,848 bytes** (always allocated)

**Total static: ~33 KB. Total transient per call: ~272 bytes.**

## 3. X25519 Key Exchange (x25519.c) — PERFORMANCE ANALYSIS

### 3.1 Montgomery Ladder

255 iterations, each performing:
- 5 field additions/subtractions
- 4 field multiplications
- 1 field squaring
- 1 conditional swap (constant-time)

Each `fe_mul` does 25 schoolbook products with `__uint128_t` = 25 `mul` instructions.
Plus carry propagation.

**Estimated cost per scalar multiplication:**
- 255 * (4 mul + 1 sq) = 255 * 5 = 1,275 field multiplications
- Each fe_mul: ~25 MUL64 + carries = ~50 instructions = ~15 ns at 3.4 GHz
- Total: 1,275 * 15 ns = ~19 us
- Plus inversion (fe_invert): ~255 squarings + 11 multiplications = ~4 us
- **Total X25519: ~23 us**

This is fast enough to be imperceptible during TLS handshake.

### 3.2 Constant-Time Properties

- `fe_cswap()` uses branchless mask-and-XOR swap. Correct.
- Montgomery ladder processes all 255 bits regardless of scalar value. Correct.
- All-zero output check uses `xsec_ct_eq()`. Correct.

### 3.3 Memory Footprint

- `fe` struct: 40 bytes (5 x uint64_t)
- Ladder locals: ~12 field elements = ~480 bytes
- `fe_invert`: ~4 field elements = ~160 bytes
- Key material: 3 x 32 bytes = 96 bytes

**Total transient: ~736 bytes. Negligible.**

## 4. Ed25519 Signatures (ed25519.c) — PERFORMANCE ANALYSIS

### 4.1 Signing Cost

Ed25519 sign requires:
1. SHA-512 of seed (one-time during key derivation)
2. SHA-512 of (nonce_prefix || message) — O(msg_len)
3. Scalar multiplication of basepoint (fixed-base) — ~255 doublings + additions
4. SHA-512 of (R || A || message) — O(msg_len)
5. Modular reduction mod L (group order)

The `fe10` representation uses 10 limbs of `int32_t` with schoolbook multiplication
producing `int64_t` intermediates. Each `fe10_mul` does 100 products.

**Estimated cost:**
- Basepoint scalar mul: ~255 * (1 doubling + ~0.5 additions) = ~383 group operations
- Each group operation: ~8 field multiplications
- Total field muls: ~3,064
- Each fe10_mul: 100 products + reduction = ~200 instructions = ~60 ns
- **Total sign: ~184 us** (plus SHA-512 of message)

For XPKG model signature verification (the primary AI use case), signing is done
on the build server, not on-device. Only verification matters for edge.

### 4.2 Verification Cost

Verification requires one double-scalar multiplication (Shamir's trick or
separate multiplications). The current implementation appears to use separate
scalar muls, which means:
- 2 scalar multiplications = ~368 us
- Plus point decompression (~1 inversion) = ~60 us
- Plus 2x SHA-512 hashes of the message

**Total verify: ~430 us for a small message.** For model verification (2 GB file),
the SHA-512 hash dominates at ~1-2 seconds on i5-8250U.

**VERDICT: Acceptable. Model verification is a one-time cost at download/install.**

### 4.3 Memory Footprint

- `fe10`: 40 bytes per field element
- Extended point (`ge_p3`): 4 field elements = 160 bytes
- Precomputed basepoint table (if present): could be large
- Stack during sign/verify: ~2-3 KB (multiple temporaries)

**Total transient: ~3 KB. Acceptable.**

## 5. TLS 1.3 (tls13.c) — HANDSHAKE LATENCY BUDGET

### 5.1 Handshake Computational Cost

A full TLS 1.3 client handshake performs:
1. X25519 keypair generation: ~23 us
2. X25519 shared secret: ~23 us
3. HKDF-Extract + HKDF-Expand-Label (multiple): ~5 us each x ~8 = ~40 us
4. AES-GCM or ChaCha20 encryption of Finished: ~5 us
5. Ed25519 CertificateVerify (if required): ~430 us
6. Certificate parsing + chain validation: ~100 us

**Total CPU cost: ~621 us (~0.6 ms)**

Network RTT dominates: a typical TLS 1.3 1-RTT handshake takes 50-200 ms on LAN,
100-500 ms on WAN. The cryptographic overhead is <1% of total handshake time.

**VERDICT: TLS handshake will NOT block AI inference pipeline.** Even worst-case
(500 ms WAN + 0.6 ms crypto), the handshake completes well within the 1.5s TTFT
target for the first Ollama API call.

### 5.2 TLS Context Memory

From `xsec_tls_ctx_t` in `xsec.h` (lines 323-368):

| Field | Size |
|-------|------|
| State/transport/cipher/bool | ~36 bytes |
| ECDHE keys (4 x 32) | 128 bytes |
| Key schedule secrets (7 x 32) | 224 bytes |
| Session keys (2x32 + 2x12) | 88 bytes |
| Transcript hash (SHA-256 ctx) | 108 bytes |
| Sequence numbers | 16 bytes |
| Random values (2 x 32) | 64 bytes |
| Peer cert key + hostname | 324 bytes |
| **record_buf[16640]** | **16,640 bytes** |
| record_buf_len | 4 bytes |
| **TOTAL** | **~17.2 KB** |

The `record_buf` at 16,640 bytes dominates. This is the TLS record buffer
(16,384 max plaintext + 256 overhead). This is per-session.

**For Ollama API (single localhost connection): 1 x 17.2 KB = negligible.**
For model download from remote: still only 1 concurrent TLS session = 17.2 KB.

### 5.3 Stack Usage During Encrypted Record Send

**FINDING [P2-PERF-002]: Double stack buffers in `tls_send_record_encrypted()`.**

Lines 200 and 242 of tls13.c allocate two large stack buffers:
```c
uint8_t scratch[TLS13_MAX_RECORD + TLS13_RECORD_OVERHEAD];  // 16,640 bytes
uint8_t ct_buf[TLS13_MAX_RECORD + TLS13_RECORD_OVERHEAD];   // 16,640 bytes
```

Total stack usage: **~33.3 KB per TLS send call.** On a kernel with 8 KB or 16 KB
stack, this will cause stack overflow. The XENOS kernel stack size needs to
accommodate this.

**Recommendation:** Use the `ctx->record_buf` for one of these buffers, or perform
in-place encryption. Priority: P2 (before production TLS use).

### 5.4 Double Send Bug

**FINDING [P1-BUG-001]: `tls_send_record_encrypted()` sends the header twice.**

Lines 257-258 call `tls_send_record_plain()` which sends a 5-byte header with
empty payload, then lines 260-271 send another 5-byte header followed by the
actual ciphertext. This results in a malformed TLS record on the wire.

The first call sends: `[CT=23][0303][0000]` (empty record)
The second call sends: `[CT=23][0303][ct_len]` + ciphertext + tag

This will cause TLS peers to reject the connection.

**Priority: P0 — must be fixed before any TLS connection can work.**

## 6. TCP Stack (tcp.c) — MEMORY FOOTPRINT ANALYSIS

### 6.1 TCB Memory Model

Each TCB (Transmission Control Block) contains:

| Component | Size |
|-----------|------|
| State + addressing | ~48 bytes |
| Sequence variables (7 x uint32) | 28 bytes |
| MSS (2 x uint16) | 4 bytes |
| **snd_buf ring (16 KB)** | **16,396 bytes** |
| **rcv_buf ring (16 KB)** | **16,396 bytes** |
| **rexmit[32] (32 x ~1488)** | **~47,616 bytes** |
| **ooo[8] (8 x ~1472)** | **~11,776 bytes** |
| RTO + congestion + timers | ~68 bytes |
| Options + listen state | ~24 bytes |
| **TOTAL per TCB** | **~92,356 bytes (~90 KB)** |

**128 TCBs = 128 x 90 KB = ~11.25 MB**

This is allocated as a static global `g_tcbs[128]` (line 230), meaning **11.25 MB
is permanently consumed in `.bss` whether connections exist or not**.

### 6.2 Impact on AI Inference

**FINDING [P1-MEM-001]: TCP TCB table consumes 11.25 MB of the 16 GB RAM budget.**

With the current static allocation:
- OS + services: ~6 GB
- TCP table: ~11.25 MB
- Ollama + model: ~2.5 GB
- Available: ~7.5 GB

This is technically fine for the 16 GB target, but the 11.25 MB is wasteful for
an edge device that will typically have 1-5 active TCP connections.

**Recommendation:**
1. Reduce `TCP_TCB_MAX` from 128 to 32 for edge deployment (saves ~8.4 MB)
2. Reduce `TCP_REXMIT_QUEUE_MAX` from 32 to 8 (saves ~67 KB per TCB)
3. Consider lazy allocation of rexmit/ooo buffers (only when connection established)

### 6.3 Ring Buffer Size

16 KB send + 16 KB receive per connection. For AI model download, a larger
receive buffer would improve throughput:

**TCP window = min(rcv_wnd, cwnd)**

With 16 KB receive window and typical WAN RTT of 50 ms:
- Bandwidth = 16,384 / 0.050 = **327 KB/s**

For a 2 GB model download: 2,048,000 / 327 = **6,263 seconds = 104 minutes.**

This confirms the Sprint 3 assessment finding. TCP window scaling or enlarged
buffers are critical for model download performance.

**Recommendation:** For model download connections, use 256 KB buffers.
This requires the configurable buffer approach noted in the Sprint 3 assessment.

### 6.4 Download Throughput vs AI Inference Thermal

During a 2 GB model download over TCP+TLS:
- TCP processing: checksum computation, ACK generation — light CPU load
- TLS decryption: ChaCha20-Poly1305 at ~77 us per 16 KB record
  = ~77 us / 0.016 MB = 4.8 ms per MB = ~0.5% CPU for 100 KB/s download

**VERDICT: TLS decryption during download will NOT cause thermal issues.**
Even at 10 MB/s (hypothetical fast connection), TLS would use ~4.8% CPU.
This leaves ample thermal headroom for concurrent AI inference.

## 7. DNS Resolver (dns.c) — MEMORY AND LATENCY

### 7.1 DNS Cache Memory

```
DNS_CACHE_SIZE = 128 entries
Entry size: ~292 bytes (bool + char[256] + uint16 + uint32 + uint64 + bool + ipv4 + ipv6)
Total: 128 x 292 = ~37,376 bytes = ~36.5 KB
```

Plus `g_dns_pending` state: ~48 bytes.
Plus `g_dns_servers[3]`: 12 bytes.

**Total DNS memory: ~37 KB. Negligible for edge AI.**

### 7.2 DNS Resolution Latency

- Cache hit: O(128) linear scan = ~1 us. Instant.
- Cache miss: Up to 3 servers x 3 retries x (5s / 9) timeout = 5 seconds worst case
- Typical: 1 query, 20-100 ms RTT = 20-100 ms

**For AI inference (Ollama on localhost):** DNS is not involved. Ollama binds to
127.0.0.1:11434 — no DNS resolution needed.

**For model download:** DNS resolves the model repository hostname once. The result
is cached with server-provided TTL. Subsequent downloads skip DNS.

**VERDICT: DNS has zero impact on AI inference latency.**

### 7.3 Spin-Wait Concern

**FINDING [P2-PERF-003]: DNS resolution uses spin-wait with `pal_thread_yield()`.**

Lines 443-450:
```c
while (pal_time_now_ns() < deadline) {
    pal_spin_lock(&g_dns_pending_lock);
    bool got_it = g_dns_pending.received || g_dns_pending.nxdomain;
    pal_spin_unlock(&g_dns_pending_lock);
    if (got_it) goto done;
    pal_thread_yield();
}
```

During DNS resolution (up to 5 seconds worst case), this burns CPU cycles
spinning. If AI inference is running concurrently, this reduces available
CPU for token generation.

**Impact on AI:** During the brief DNS resolution window (20-100 ms typical),
one CPU core is consumed by the spin-wait. On a 4-core i5, this reduces
Ollama's available compute by 25% for those milliseconds. Negligible in practice
since DNS resolution happens once per model download, not during inference.

## 9. EDGE AI COEXISTENCE VALIDATION

### Question 1: Is constant-time AES S-box scan acceptable?

**YES.** At ~50 us per block (software path), a 4 KB AI response encrypts in
<15 ms. This is within the "interactive" latency perception band (<1s). With
AES-NI (P2-PERF-001), this drops to <0.1 ms.

### Question 2: Are crypto algorithms suitable for edge deployment?

**YES, with caveats.** ChaCha20-Poly1305 is the recommended cipher suite for
Sprint 3 edge deployment because it is inherently constant-time and fast without
hardware acceleration. AES-256-GCM should be preferred once AES-NI is implemented.

### Question 3: Are network stack memory footprints reasonable?

**PARTIALLY.** DNS (37 KB) and TLS (17.2 KB per session) are fine. TCP's 11.25 MB
static allocation is excessive for edge — reducing `TCP_TCB_MAX` to 32 saves
8.4 MB and is strongly recommended.

### Question 4: Will TLS handshake latency block AI inference?

**NO.** TLS handshake crypto costs ~0.6 ms. Network RTT dominates. The handshake
occurs once at connection establishment, not per inference request. For localhost
Ollama, TLS is not used (HTTP over loopback).

### Question 5: Is DNS cache sizing appropriate?

**YES.** 128 entries at ~37 KB is minimal. DNS is not involved in localhost AI
inference. For model download, one DNS lookup is cached for the session duration.
