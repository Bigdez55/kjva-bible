## 1. TLS USAGE IN AI PIPELINE

### Finding: NO — Ollama communication is plaintext HTTP over localhost

**Evidence from source code:**

- `ai/genesys-ai/src/model_manager.py:39` — `self.base_url = os.getenv("OLLAMA_URL", "http://127.0.0.1:11434")`
- All Ollama API calls use `http://` prefix, never `https://`
- `ai/companion/src/agent-bridge.ts:100` — `this._baseUrl = config?.agentBaseUrl ?? "http://localhost:8800"`
- Internal service URLs in `agent.py:163-166` — `http://sync:8000`, `http://provenance:8000`, `http://browser-bridge:8787`

**Analysis:** The entire GENESYS AI inference pipeline operates over plaintext HTTP on localhost/internal network. The Poly1305 fix in `chacha20.c` (3-limb arithmetic rewrite) does NOT affect the runtime AI inference path. Ollama binds to `127.0.0.1:11434` — traffic never leaves the loopback interface.

**However:** The Poly1305 fix IS critical for model download. `model_manager.py:77-79` calls `POST /api/pull` to download models from Ollama's registry. Ollama itself uses HTTPS (TLS) to fetch model blobs from `registry.ollama.ai`. If GEN.OS replaces Ollama's download path with XNET+TLS in Sprint 6 (XMIND), the corrected Poly1305 implementation becomes a hard dependency for model integrity during download.

| Question | Answer |
|----------|--------|
| TLS used for Ollama inference API? | **NO** — plaintext HTTP on localhost |
| Affected by Poly1305 fix at runtime? | **NO** — no TLS in inference hot path |
| Affected by Poly1305 fix for model download? | **YES (future)** — when XNET+TLS replaces Ollama's built-in downloader |
| ChaCha20 `poly_scratch` thread-safety fix relevant? | **YES (future)** — stack-local buffer (line 374) now prevents data races if concurrent TLS sessions download models |

## 2. MODEL PACKAGE VERIFICATION CHAIN (Ed25519)

### Finding: YES — Ed25519 is the sole trust anchor for XPKG model packages

**Evidence from source code:**

- `pkg/xpkg/verify/pkg_verify.c:307` — `xsec_ed25519_verify(key_store[k].pub_key, msg, msg_len, sig)`
- Every XPKG package (including future model packages) MUST pass Ed25519 signature verification before installation
- Trust hierarchy: SYSTEM > USER > COMMUNITY (lines 291-293)
- Revocation check precedes every verify attempt (line 301)

**Analysis:** The Ed25519 carry propagation fix in `ed25519.c:105` (int64_t intermediates before truncation to int32_t) and the scalar reduction fix (lines 564-656) are **critical** for model package verification. Without correct carry propagation:

1. Signature verification could produce false negatives — rejecting valid model packages signed by the GEN.OS SYSTEM key
2. Worse: false positives could accept tampered model weights, enabling model poisoning attacks

The `xpkg_verify_signature()` function is the ONLY gate between a downloaded model file and installation on the device. A corrupted Ed25519 verify would break the entire model supply chain.

| Question | Answer |
|----------|--------|
| Ed25519 used for model verification? | **YES** — via `xpkg_verify_signature()` |
| Affected by Ed25519 carry fix? | **YES, CRITICAL** — incorrect carry = wrong verify results |
| Impact of false negative? | Model update refused — user stuck on old model |
| Impact of false positive? | Poisoned model weights installed — catastrophic |

## 3. DNS PROTECTION FOR MODEL UPDATES

### Finding: YES — DNS random QID protects model update resolution

**Evidence from source code:**

- `net/xnet/app/dns.c:381-386` — Comment explicitly states: "Sequential IDs allow an attacker to predict the next QID and forge responses"
- Fix: `pal_random_bytes(qid_bytes, 2u)` replaces sequential `g_dns_qid` counter
- `g_dns_qid` is retained but no longer drives QID generation (line 387)

**Analysis:** When GENESYS AI resolves model update servers (e.g., `registry.ollama.ai` or a future GEN.OS model repository), DNS is the first network operation. With predictable sequential QIDs, an attacker on the same network segment could:

1. Predict the next QID (trivial: QID = previous + 1)
2. Race a forged DNS response pointing `registry.ollama.ai` to an attacker-controlled server
3. Serve a poisoned model file (Llama 3.2 3B with backdoor weights)

The random QID fix raises the attack from trivial (sequential prediction) to requiring ~32K guesses per attempt (16-bit random QID space). Combined with Ed25519 signature verification in XPKG, this creates defense in depth:

- **Layer 1 (DNS):** Random QID prevents trivial DNS poisoning
- **Layer 2 (TLS):** Certificate verification prevents MITM on download (when using XNET+TLS)
- **Layer 3 (XPKG):** Ed25519 signature verification prevents tampered model installation

| Question | Answer |
|----------|--------|
| DNS used for model updates? | **YES** — resolves model registry hostnames |
| Protected by random QID fix? | **YES** — prevents trivial DNS poisoning |
| Sufficient alone? | **NO** — 16-bit QID space still brute-forceable; TLS + Ed25519 required |

## 5. TCP INITCWND AND MODEL DOWNLOAD SPEED

### Current Implementation: initcwnd = 10 MSS (RFC 6928)

**Evidence:** `net/xnet/transport/tcp.c:464` — `t->cwnd = 10u * t->snd_mss;`

This is ALREADY the optimal value per RFC 6928. The question asks about improvement from initcwnd=1 to initcwnd=10. Let me calculate the concrete benefit for a 2GB model download.

### Slow Start Analysis

**Parameters:**
- Connection: 50 Mbps, 50ms RTT
- MSS: 1460 bytes (standard Ethernet)
- Model size: 2 GB = 2,147,483,648 bytes
- BDP (bandwidth-delay product): 50 Mbps * 50ms = 312,500 bytes

**With initcwnd = 1 MSS (1,460 bytes):**
```
RTT 0:  cwnd =    1 MSS =     1,460 B  (cumulative:       1,460 B)
RTT 1:  cwnd =    2 MSS =     2,920 B  (cumulative:       4,380 B)
RTT 2:  cwnd =    4 MSS =     5,840 B  (cumulative:      10,220 B)
RTT 3:  cwnd =    8 MSS =    11,680 B  (cumulative:      21,900 B)
RTT 4:  cwnd =   16 MSS =    23,360 B  (cumulative:      45,260 B)
RTT 5:  cwnd =   32 MSS =    46,720 B  (cumulative:      91,980 B)
RTT 6:  cwnd =   64 MSS =    93,440 B  (cumulative:     185,420 B)
RTT 7:  cwnd =  128 MSS =   186,880 B  (cumulative:     372,300 B)
RTT 8:  cwnd =  214 MSS =   312,440 B  (BDP reached — exit slow start)

RTTs to reach full throughput: ~8 RTTs = 400ms
Data transferred during slow start: ~372 KB
```

**With initcwnd = 10 MSS (14,600 bytes):**
```
RTT 0:  cwnd =   10 MSS =    14,600 B  (cumulative:      14,600 B)
RTT 1:  cwnd =   20 MSS =    29,200 B  (cumulative:      43,800 B)
RTT 2:  cwnd =   40 MSS =    58,400 B  (cumulative:     102,200 B)
RTT 3:  cwnd =   80 MSS =   116,800 B  (cumulative:     219,000 B)
RTT 4:  cwnd =  160 MSS =   233,600 B  (cumulative:     452,600 B)
RTT 5:  cwnd =  214 MSS =   312,440 B  (BDP reached — exit slow start)

RTTs to reach full throughput: ~5 RTTs = 250ms
Data transferred during slow start: ~452 KB
```

### Download Time Calculation

```
Bulk transfer time at 50 Mbps:
  2 GB / 50 Mbps = 2,147,483,648 * 8 / 50,000,000 = 343.6 seconds

Slow start penalty (initcwnd=1):
  8 RTTs * 50ms = 400ms, but sub-BDP throughput costs additional time
  Effective penalty: ~0.4s wall-clock vs. already-at-BDP

Slow start penalty (initcwnd=10):
  5 RTTs * 50ms = 250ms
  Effective penalty: ~0.25s wall-clock

Time saved: 400ms - 250ms = ~150ms
```

**However**, the real benefit is in short-lived connections (HTTP/1.1 with reconnects). If the model download uses multiple TCP connections (e.g., Ollama's chunked download), each connection pays the slow start cost independently:

```
If download uses N connections:
  Savings = N * 150ms

For a 2GB download split into 64MB chunks (32 connections):
  Savings = 32 * 150ms = 4.8 seconds

For a single persistent connection:
  Savings = 150ms (negligible vs. 343s total)
```

| Question | Answer |
|----------|--------|
| Current initcwnd value? | **10 MSS** (already RFC 6928 optimal) |
| Time saved vs. initcwnd=1 (single conn)? | **~150ms** — negligible for 343s download |
| Time saved vs. initcwnd=1 (32 connections)? | **~4.8 seconds** — noticeable UX improvement |
| Bottleneck for model download? | **TCP window scaling** (16KB buffers) — not initcwnd |

## CRITICAL PATH IMPLICATIONS

1. **Ed25519 carry fix is the highest-priority P2 fix for the AI pipeline.** Without correct signature verification, the entire model supply chain is compromised. This fix must be validated with known-answer tests using actual GEN.OS SYSTEM signing keys before any model package is distributed through XPKG.

2. **The Poly1305 rewrite has no immediate runtime impact** on AI inference (localhost HTTP), but becomes critical in Sprint 6 when XMIND replaces Ollama with native XNET+TLS model download.

3. **DNS random QID is defense-in-depth** — it raises the bar for model poisoning but is not sufficient alone. The Ed25519 signature check in XPKG is the definitive guard.

4. **TCP initcwnd is already optimal.** The real download bottleneck identified in the Sprint 3 E2E audit is the 16KB TCP receive window (capped at ~327 KB/s). Window scaling or 256KB buffers remain the P1 priority for model download performance.

5. **Memory impact is zero for practical purposes.** The 1.5 KB code size increase is lost in the noise of a 3GB model footprint.
