### 1.2 Inner Loop Analysis -- xmind_matmul_q4

The critical hot path is `tensor.c` lines 163-184:

```c
for (r = 0u; r < rows; r++) {
    float acc = 0.0f;
    for (b = 0u; b < n_blocks_per_row; b++) {
        float blk_scale = blk->scale;
        for (i = 0u; i < 16u; i++) {
            uint8_t byte   = blk->nibbles[i];
            int32_t lo_raw = (int32_t)(byte & 0x0Fu) - 8;
            int32_t hi_raw = (int32_t)(byte >> 4u)  - 8;
            float   w_lo   = (float)lo_raw * blk_scale;
            float   w_hi   = (float)hi_raw * blk_scale;
            acc += w_lo * x[base_col + i * 2u];
            acc += w_hi * x[base_col + i * 2u + 1u];
        }
    }
    out[r] = acc;
}
```

**Per inner iteration (16 nibble-byte iterations per block):**
- 2 nibble extractions (integer bit ops)
- 2 int-to-float conversions
- 2 float multiplies (dequant: raw * scale)
- 2 float multiplies (dot: w * x)
- 2 float adds (accumulate)
- Total: 4 float muls + 2 float adds + 2 int-to-float = **8 FP ops per byte, 256 FP ops per Q4 block**

### 1.3 Scalar x87 Performance Estimate

On i5-8250U at ~3 GHz with `-mno-sse -mno-sse2` (x87 FPU only):
- x87 FMUL throughput: ~1 per 5 cycles (not pipelined for scalar legacy mode)
- x87 FADD throughput: ~1 per 3 cycles
- INT-to-x87 (FILD): ~5 cycles
- Effective throughput for Q4 inner loop: ~20-30 cycles per weight pair

**Conservative estimate for matmul-dominant workload:**
- Total matmul FLOPs per token: ~2.77 GFLOP
- At ~3 GHz, 1 scalar FP op per ~4 cycles average: ~750 MFLOP/s effective
- **Time per token (matmul only): 2.77G / 750M = ~3.7 seconds**
- With non-matmul overhead (~5%): **~3.9 seconds per token**
- Plus attention dot products growing with pos: ~4.6s at pos=0 to ~18s at pos=2048

**TTFT = ~4.6 seconds (single token, pos=0). TPS = ~2.0 tok/s (early sequence).**

This is firmly in the BROKEN perception tier (> 5 seconds for TTFT).

### 1.4 AVX2 Performance Projection

With AVX2 (256-bit, 8 floats parallel) + 4-thread parallelism:
- AVX2 VFMADD throughput: 2 per cycle on Kaby Lake = 16 FLOP/cycle
- 4 threads at 3.0 GHz (AVX2 downclocks from 3.4 to ~3.0): 48 GFLOP/s theoretical peak
- At ~40% utilization (memory-bound Q4 decode): ~19.2 GFLOP/s effective
- **Time per token: 2.87G / 19.2G = ~0.15 seconds**
- With overhead: **~0.2 seconds per token = ~5 TPS single-thread, ~14 TPS 4-thread**

Target of 15 TPS is achievable with AVX2 + 4-thread matmul.

### 2.4 Stack Usage (CRITICAL)

The FFN scratch in `transformer.c` line 253:
```c
#define XM_FFN_SCRATCH (XMIND_MAX_HEADS * XM_MAX_HEAD_DIM)  // 32 * 256 = 8192
float gate[XM_FFN_SCRATCH];  // 32 KB on stack
float up[XM_FFN_SCRATCH];    // 32 KB on stack
```

**P1 BUG: 64 KB of stack-allocated scratch per xm_ffn() call.** With the attention scratch
(`float scores[XMIND_MAX_SEQ]` = 8 KB) and head_out (1 KB), peak stack usage during a
single forward pass is ~65 KB. This is dangerously high for a freestanding kernel-adjacent
context. If XENOS provides 4-8 KB stacks, this will overflow.

The actual FFN dim is 8192 (not 8192*4 as XM_FFN_SCRATCH allows), so the 32 * 256 = 8192
cap happens to match. But the 64 KB stack allocation is still excessive.

### 2.5 Total Memory Budget

| Component | Size | Notes |
|-----------|------|-------|
| Weights | 2.05 GB | Memory-mapped, read-only sequential access |
| KV cache (full 2048 ctx) | 336 MB | Random access per attention head |
| State buffers | 433 KB | Hot, fits in L2 |
| Stack scratch | ~65 KB | P1: excessive for freestanding |
| **Total XMIND** | **~2.42 GB** | Fits 3 GB budget |
| OS + services (estimated) | ~2 GB | XENOS + GENSD + XDISP + XCOMP |
| **System total** | **~4.4 GB** | Fits 16 GB with 11.6 GB headroom |

**VERDICT: Memory is NOT a constraint.** 2.42 GB fits well within the 3 GB model budget.
16 GB system RAM provides ample headroom for OS, display stack, and inference concurrently.

### 2.6 Memory Access Pattern

**Weights (matmul): Strictly sequential within each row.** The Q4 matmul iterates blocks
left-to-right across columns. Each block is 20 bytes, read once per output element.
Hardware prefetcher will be effective. This is a streaming workload.

**KV cache (attention): Semi-random.** For each query head, the attention function iterates
over all cached positions (t=0..pos) reading k_t and v_t. Within each position, access is
sequential across head_dim=96 floats (384 bytes). But jumping between positions has a stride
of kv_stride * 4 = 768 * 4 = 3072 bytes. For 28 layers at 2048 positions, this scans the
entire 336 MB KV cache. At long contexts, this becomes the memory bandwidth bottleneck.

**Memory bandwidth on i5-8250U:** DDR4-2400 dual channel = ~38.4 GB/s theoretical, ~25 GB/s
practical. Per-token weight read: 2.05 GB (must read all weights once per token).
At 25 GB/s: 2.05 / 25 = **82 ms** -- this is the absolute memory bandwidth floor.
At 15 TPS target: 67 ms per token budget. **Memory bandwidth is the bottleneck after AVX2.**

## III. XJIT SSE/AVX OPPORTUNITY (Task 3)

### 3.1 Current XJIT Capability

Source: `jit/xjit/src/codegen.c` (925 LOC)

**XJIT generates integer-only x86-64 code.** The codegen emits:
- 22 integer opcodes: MOV_RI, MOV_RR, ADD, SUB, IMUL, AND, OR, XOR, SHL, SHR, CMP, JMP, JEQ, JNE, JLT, JGT, CALL, RET, LOAD, STORE, LABEL, PHI, NOP
- 14 GP registers: RAX, RCX, RDX, RBX, RSI, RDI, R8-R15
- REX prefix encoding for 64-bit operands
- Spill/reload via RBP-relative addressing

**No SSE2/AVX2 code generation exists.** There are:
- Zero VEX prefix encodings (no 0xC4/0xC5 bytes)
- Zero XMM/YMM register references
- Zero floating-point instructions (no MOVSS, ADDPS, VFMADD, etc.)
- No SIMD IR opcodes in the IR definition

### 3.2 The SSE Prohibition -- Kernel vs. Userspace

**P0 BLOCKER CONFIRMED:** Every CI workflow (sprint2.yml through sprint13.yml) compiles
XMIND with `-mno-sse -mno-sse2`. This is inherited from XENOS kernel flags.

**Critical architectural question:** XMIND is a userspace inference runtime, NOT kernel code.
The `-mno-sse` flag exists in the kernel to prevent SSE register clobbering in interrupt
handlers (since XENOS does not yet implement XSAVE/XRSTOR for XMM/YMM context switching).

**XMIND running in userspace SHOULD be allowed to use SSE/AVX** because:
1. Userspace context switches save/restore the full FPU/SSE/AVX state via XSAVE
2. The i5-8250U supports SSE4.2, AVX2, and FMA3
3. The ABI default for x86-64 uses SSE2 for all floating-point

**However:** XENOS kernel does not yet implement XSAVE/XRSTOR (per prior audit finding).
Until the kernel saves XMM/YMM state on context switch and interrupt, ANY use of SSE/AVX
in userspace will corrupt register state when the scheduler preempts the inference thread.

**This is the single most important optimization gate for XMIND.**

### 3.3 XJIT SIMD Roadmap Assessment

Even if SSE/AVX were enabled at compile time, XJIT cannot generate SIMD code because:
1. No float/vector IR opcodes exist
2. No XMM/YMM register allocator
3. No VEX prefix encoding in codegen.c
4. No memory alignment handling for SIMD loads

Estimated effort to add AVX2 matmul via XJIT: ~410 LOC additions (per prior audit estimate).
This is a Sprint 12+ target and is NOT viable for near-term inference acceleration.

**Recommendation:** Use compiler intrinsics (`_mm256_*`) for SIMD matmul, NOT XJIT. This
delivers immediate AVX2 performance without modifying the JIT infrastructure.

## IV. iGPU COMPUTE PATH (Task 4)

### 4.1 Intel UHD 620 -- Hardware Capability

The Intel UHD 620 (Kaby Lake Refresh, GT2) contains:
- 24 Execution Units (EUs)
- ~460 GFLOPS fp32 theoretical peak
- Shared system memory (no dedicated VRAM)
- OpenCL 2.0 support in hardware

**This is significant compute capability** -- 460 GFLOPS dwarfs the CPU's ~48 GFLOPS (AVX2).

### 4.2 Current iGPU Driver (intel_igpu.c)

Source: `kernel/xenos/drv/intel_igpu.c` (535 LOC)

The driver is a **GOP framebuffer driver only**:
- Maps physical framebuffer via HHDM
- Provides put_pixel(), clear(), draw_rect(), draw_text_8x16()
- Write-combining mapping for sequential pixel writes
- No command streamer, no batch buffer submission, no EU dispatch

**Zero GPGPU/compute capability is exposed.** The driver does not:
- Initialize the GPU command streamer
- Submit batch buffers
- Program EU shader dispatch
- Expose shared memory for GPU-accessible tensor storage
- Implement any Intel Gen9 render/compute pipeline

### 4.3 DRM/KMS Layer (drm.c)

Source: `display/xdisp/drm.c` (1927+ LOC)

The DRM layer handles mode setting (CRTCs, connectors, encoders, framebuffers) via raw
ioctls on /dev/dri/card0. This is display-plane only. There is no:
- DRM render node (/dev/dri/renderD128) access
- GEM buffer object management for compute
- i915 batch buffer submission (I915_GEM_EXECBUFFER2)
- GPU fence/sync primitive support

### 4.4 GPU Compute Feasibility Assessment

**Could XMIND use the UHD 620 for tensor operations?** In theory, yes -- the 24 EUs
provide 460 GFLOPS that would make inference memory-bandwidth-limited rather than
compute-limited. However:

**Blockers (in order of severity):**
1. **No GPU command streamer driver** -- requires ~2,000-5,000 LOC of Gen9 hardware programming
2. **No GPU memory management** -- GEM objects, GTT/GGTT page table setup required
3. **No shader compiler** -- GPGPU compute needs compiled EU kernels
4. **Memory bandwidth shared** -- iGPU uses same DDR4 as CPU, so no bandwidth gain
5. **Context switch overhead** -- GPU dispatch latency (~50-100us) per kernel launch

**The shared memory architecture means GPU compute does NOT increase memory bandwidth.**
Since inference is memory-bandwidth-limited after AVX2, GPU compute would not improve TPS
beyond what CPU AVX2 achieves. The GPU would need to process data already in system RAM.

**Verdict: GPU compute is NOT worth pursuing for XMIND inference.** The engineering cost
(~5,000-10,000 LOC for a Gen9 compute driver) far exceeds the benefit. AVX2 on CPU is
the correct optimization path. GPU compute should be reserved for display composition
(XCOMP) where it can offload blend/composite operations.

## V. INFERENCE REQUEST QUEUING (Task 5)

### 5.1 Singleton Architecture

Source: `xmind.c` lines 105-108:
```c
static xmind_model_t s_xmind_model;

xmind_model_t *xmind_get_global(void) {
    return &s_xmind_model;
}
```

XMIND uses a **process-wide global singleton** (`s_xmind_model`). There is exactly one
model instance, one KV cache, and one set of state buffers.

### 5.2 Synchronous Inference -- No Queue

`xmind_forward()` is a **fully synchronous, blocking function call**:
- Takes a single token and position
- Runs the entire 28-layer forward pass
- Writes logits directly into `m->state.logits`
- Returns only after the full computation completes

There is:
- **No request queue** -- no ring buffer, no FIFO, no priority system
- **No mutex/lock** -- no concurrency protection on the singleton
- **No thread safety** -- `s_xmind_model` has no atomic fields, no lock guards
- **No batch inference** -- processes exactly one token at a time

### 5.3 Concurrent Access Hazard (P1 BUG)

If two processes or threads call `xmind_forward()` simultaneously on the global singleton:

1. **State corruption:** Both would write to `m->state.x`, `m->state.xb`, `m->state.q`,
   `m->state.k`, `m->state.v`, `m->state.attn`, and `m->state.logits` simultaneously.
   The interleaved writes would produce garbage results.

2. **KV cache corruption:** Both would write to `m->kv.k[layer]` and `m->kv.v[layer]`
   at their respective positions, potentially overwriting each other's cached values.

3. **Position counter race:** `m->kv.n_cached = pos + 1u` on line 215 would race.

**There is no protection against this.** In a single-threaded freestanding context (which
is the current deployment model), this is not triggered. But any future multi-service
architecture (e.g., XSHELL AI assistant + background document analysis) would hit this.

### 5.4 Missing Inference API Surface

The current API (`xmind_forward` + `xmind_sample`) requires the caller to:
1. Manually manage the token position counter
2. Call forward() in a loop for autoregressive generation
3. Know to call xmind_sample() after each forward()
4. Handle tokenization/detokenization externally

There is no:
- `xmind_generate(prompt, max_tokens, callback)` -- high-level generation API
- Streaming callback for token-by-token output
- Cancellation mechanism for long generations
- Timeout or thermal-aware throttling integration

### P1 -- HIGH (Blocks Production Deployment)

| ID | Finding | Impact | Effort |
