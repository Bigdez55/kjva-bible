## NEW Findings

### P1

#### GS-S4-P1-001: x25519.c fe_pack lacks constant-time canonical reduction
- **File**: `sec/xsec/crypto/x25519.c:201-257`
- **Carry-forward from**: GS-S3-P1-007
- **Issue**: fe_pack performs two reduction passes but no final constant-time conditional subtraction of p. Values exactly equal to p (or p+1..p+18 after two passes) will serialize as non-canonical field elements. RFC 7748 requires canonical output.
- **Impact**: Non-canonical X25519 shared secrets may cause interop failures. In theory could leak 1 bit of information about the shared secret (whether it was >= p).
- **Fix**: After two reduction passes, compute t-p in a separate variable, then constant-time select between t and (t-p) based on the carry-out using `mask = -(carry) & ~0ULL`.

#### GS-S4-P1-002: tls_recv_record_encrypted has 33KB stack VLAs (receive path)
- **File**: `sec/xsec/tls/tls13.c:338-339`
- **Issue**: The SEND path was fixed with module-static buffers (S3-DI-C02), but the RECEIVE path still declares `ct_buf[16640]` and `pt_buf[16385]` as local arrays on the stack, totaling ~33KB. On a freestanding kernel with 8-16KB stacks, this will overflow into adjacent kernel memory.
- **Impact**: Stack overflow on every TLS record receive. Silent memory corruption of adjacent kernel data structures.
- **Fix**: Promote ct_buf and pt_buf to module-static storage (same pattern as send path), or use the context's record_buf field.

#### GS-S4-P1-003: Audit ring write_idx/total_events temporal inconsistency
- **File**: `sec/xsec/audit/audit.c:145-163`
- **Issue**: write_idx is incremented BEFORE the entry is written (line 145), then total_events is incremented AFTER (line 162). Between these two atomics, a concurrent reader calling xsec_audit_query_recent() will compute ring_count from total_events (which hasn't been incremented yet) but write_idx already advanced. The reader may read a stale/partially-written entry from the slot that was just claimed. Additionally, the dropped_events check on line 150-153 reads total_events BEFORE it is incremented for the current event, which means the first event to wrap the ring will NOT be counted as dropped.
- **Impact**: Under concurrent audit log access, readers may observe corrupted entries. The dropped_events counter is off-by-one.
- **Fix**: Use a single monotonic counter (remove total_events, derive from write_idx). Take the spinlock for writes as well, or use a proper sequence lock pattern.

### P2

#### GS-S4-P2-001: Ed25519 ge_frombytes point decompression not constant-time
- **Carry-forward from**: GS-S3-P2-002
- **File**: `sec/xsec/crypto/ed25519.c:471-486`
- **Issue**: Point validation uses `if(check_bytes[k]!=0) ok=false` loop and subsequent `if(!ok)` branch. This leaks timing information about whether the first square root succeeded.
- **Impact**: Side-channel on public key decompression. Lower severity since public keys are not secret, but in batch verification contexts could leak information.
- **Fix**: Use `xsec_ct_eq(check_bytes, zero_buf, 32)` and constant-time conditional multiply.

#### GS-S4-P2-002: Ed25519 verify does not check S < l (signature malleability)
- **Carry-forward from**: GS-S3-P2-003
- **File**: `sec/xsec/crypto/ed25519.c` (verify function)
- **Issue**: RFC 8032 section 5.1.7 requires that S (bytes 32-63 of signature) be less than the group order l. Without this check, for any valid signature (R,S), the signature (R, S+l) is also valid, enabling signature malleability.
- **Impact**: An attacker can produce alternative valid signatures for the same message. This breaks systems that use signature uniqueness as a security property (e.g., transaction deduplication).
- **Fix**: Before verifying, check that `sig[32..63]` interpreted as LE integer is < l. Reject if not.

#### GS-S4-P2-003: X.509 DER parser does not reject indefinite-length encoding
- **File**: `sec/xsec/cert/x509.c:96-111`
- **Issue**: der_read_length() handles short form (1 byte) and long definite form (2-5 bytes) but does not explicitly reject indefinite-length encoding (first byte = 0x80). When first=0x80, num_bytes=0, the function returns 0 length without error. This allows crafted certificates with indefinite-length SEQUENCE tags to pass parsing with zero-length bodies, potentially skipping critical fields.
- **Impact**: Crafted DER input could cause the parser to skip fields or misalign the cursor, potentially accepting malformed certificates.
- **Fix**: Add explicit check: `if (first == 0x80u) { c->error = true; return 0u; }` after the short-form check.

#### GS-S4-P2-004: TLS transcript hash uses SHA-256 for all cipher suites
- **File**: `sec/xsec/tls/tls13.c:429-431, xsec.h:349`
- **Issue**: The transcript hash context is `xsec_sha256_ctx_t`. TLS_AES_256_GCM_SHA384 (0x1302) requires SHA-384 for transcript hashing per RFC 8446 section 4.4.1. Currently AES-GCM is rejected in ServerHello (correctly), but the code comments in ClientHello (line 613) acknowledge this limitation. If AES-GCM is ever enabled without adding SHA-384 transcript support, Finished MACs and key derivation will produce wrong values.
- **Impact**: No current impact (AES-GCM is rejected). Sprint 4 dependency -- must be resolved before enabling AES-GCM.
- **Fix**: Sprint 4 must add xsec_sha384_ctx_t and select transcript hash based on negotiated cipher suite.

1. Full software AES-256-GCM with CT S-box scan (aes_gcm.c) -- functional but slow (~200 KB/s)
2. AES-GCM correctly rejected in TLS ServerHello (tls13.c:710)

4. GCM mode (GHASH, CTR, tag computation) is complete and correct
5. Clear TODO-S4 comment documenting the plan (aes_gcm.c:29-32)

### What Sprint 4 needs to enable AES-256-GCM:
1. **CPUID detection**: Check CPUID.1:ECX bit 25 (AES-NI) + bit 1 (PCLMULQDQ)
2. **AES-NI intrinsics**: Replace aes_sbox_ct + key schedule with `__builtin_ia32_aesenc` / `__builtin_ia32_aesenclast` / `__builtin_ia32_aeskeygenassist`
3. **PCLMULQDQ GHASH**: Replace software ghash_mul with `__builtin_ia32_pclmulqdq128` for ~100x GHASH speedup
4. **SHA-384 transcript**: Add xsec_sha384_ctx_t for TLS_AES_256_GCM_SHA384 transcript hashing
5. **Runtime dispatch**: If AES-NI not present (non-x86 or old CPU), fall back to ChaCha20-Poly1305 only
6. **HP EliteBook x360 1030 G4**: Whiskey Lake i7-8665U supports AES-NI + PCLMULQDQ. Expected ~1 GB/s AES-GCM.

### Risk: LOW
The HP EliteBook target has AES-NI. Sprint 4 implementation is straightforward -- the GCM mode framework is already correct, only the AES block cipher and GHASH multiply need hardware acceleration.

## TOP 3 Security Risks for HP EliteBook Bare-Metal Deployment

### Risk 1: SYSRETQ Privilege Escalation (P0 -- GS-S01-001)
- **Threat**: Non-canonical RIP in RCX register during SYSRETQ causes General Protection Fault in ring 0 instead of ring 3. CVE-2012-0217 class vulnerability. Any userspace process can escalate to kernel.
- **Mitigation**: MUST validate canonical address before SYSRETQ. Use IRETQ for suspect addresses.
- **Sprint**: 1 (carry-forward, unresolved)

### Risk 2: TLS Receive Path Stack Overflow (P1 -- GS-S4-P1-002)
- **Threat**: Every TLS record receive allocates ~33KB on the kernel stack. On bare metal with 8-16KB kernel stacks, this silently corrupts adjacent memory. An attacker sending any TLS application data record triggers the overflow.
- **Mitigation**: Promote VLAs to module-static or context-attached buffers (same pattern as send path fix).
- **Sprint**: 3 (new finding)

### Risk 3: X25519 Non-Canonical Output (P1 -- GS-S4-P1-001)
- **Threat**: Non-canonical field elements from X25519 shared secret computation could cause TLS handshake failures with strict peers, or in worst case leak 1 bit of the shared secret per handshake.
- **Mitigation**: Add constant-time conditional subtraction of p in fe_pack().
- **Sprint**: 3 (carry-forward from GS-S3-P1-007)
