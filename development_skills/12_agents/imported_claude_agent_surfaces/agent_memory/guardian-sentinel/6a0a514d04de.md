# Security Efficiency Audit -- 2026-03-02

## Audit Scope
5 platform services + validator_loader + 3 k8s manifests at HEAD (main branch, post-daf390d)

## Key Findings

### P1 (Critical)
- **No per-IP rate limiting** on identity /auth/login endpoint. Per-username lockout exists (5 attempts/15min) but no IP-level throttling. slowapi recommended.
- **No request body size limit** on identity, provenance, gate, admission endpoints. Sync has MAX_UPLOAD_BYTES. Other services rely only on k8s 512Mi memory limit as implicit guard.

### P2 (Quality -- Performance)
- **JWT decoded from scratch every request** in identity, sync, provenance. No LRU cache. ~0.3-0.5ms waste per request.
- **Sync PEM re-normalized every request** (normalize_pem() called inside decode_bearer instead of at module init). Identity and provenance handle this correctly.
- **bcrypt rounds implicit** -- gensalt() without rounds= parameter defaults to 12. Should be explicit + env-tunable.
- **Failed auth not logged with source IP** -- only lockout triggers logged, individual failures silent.

### P3 (Polish)
- No CORS middleware on any service -- confirm Electron architecture before adding.
- Provenance `import json` inside build_audit_token() function body instead of top-level.
- /healthz used for k8s liveness probe causes cascading restarts during DB outage -- should use /health for liveness.
- Gate/admission reload schema+registries from disk on every /validate call -- could cache at startup.

## Verified Correct
- hmac.compare_digest on ALL token comparisons (gate, admission, registry, validator_loader)
- JWT keys loaded at startup in identity (module-level _load_signing_keys)
- Empty token guard in admission webhook (prevents compare_digest('','') bypass)
- Health endpoints exempt from auth on all services
- Permission check done before expensive DB fetch in update_user
- Pydantic extra="forbid" on all request models
- K8s security contexts complete (runAsNonRoot, readOnlyRootFilesystem, no priv escalation)
