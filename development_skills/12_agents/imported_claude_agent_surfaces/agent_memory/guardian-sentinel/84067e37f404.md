## Key Patterns Discovered

### Dependency Split Problem
- pyproject.toml defines project deps (includes httpx)
- platform/services/requirements.txt is what Dockerfiles actually install
- These two files are NOT synchronized -- httpx is in pyproject but not requirements.txt
- CI installs via pyproject (`pip install -e ".[dev]"`), Docker via requirements.txt
- Result: tests pass in CI but containers crash at runtime

### Docker Build Coverage Gap
- DOCKER_SERVICES in genos-build.py lists 9 services
- But k8s manifests reference 11 images (adds fabric-energy + merkle-collective)
- Dockerfile.merkle-collective EXISTS but is never invoked by build pipeline
- Dockerfile.fabric-energy does NOT exist at all
- IMAGE_TARBALLS in chroot-setup.sh also only lists 9 tarballs

### CI Blind Spots
- TypeScript: 12 workspaces, zero CI coverage (no tsc, no ESLint, no build)
- Docker: 10 Dockerfiles, zero CI builds
- k8s: 16 manifests, zero validation (no kubeval/kubeconform)
- Integration: no service-to-service tests

### GENOS_SERVICE_TOKEN Pattern
- 4 services read os.environ["GENOS_SERVICE_TOKEN"] at MODULE LOAD (line 14)
- registry_app.py, gate_service_app.py, artifact_store_app.py, update/app.py
- Their k8s manifests have NO env section at all
- Pod crashes immediately with KeyError before any endpoint can respond
- Only admission-webhook (07) and genesys-ai (09) correctly provide this env var

### Dockerfile Uniformity Issues
- All use single-stage builds (no multi-stage)
- All use tag-only base images (no digest pinning)
- merkle-collective uses python:3.11-slim; all others use 3.12-slim
- genesys-ai uses the platform services requirements.txt but needs httpx (not in it)
