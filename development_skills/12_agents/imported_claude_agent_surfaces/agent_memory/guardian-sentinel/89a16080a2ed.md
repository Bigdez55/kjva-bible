## FIX 2: P1 CertificateVerify Verification (tls13.c:830-921)

**Status: VERIFIED CORRECT with 1 P1 finding**

- Parses signature scheme (2 bytes) at cv_pos=4 -- CORRECT per RFC 8446 section 4.4.3
- Parses signature length (2 bytes) -- CORRECT
- Bounds check `cv_pos + sig_len_cv > pt_len` -- CORRECT
- Builds signed content: 64x0x20 + "TLS 1.3, server CertificateVerify" (33 bytes) + 0x00 + transcript_hash -- CORRECT (130 bytes total matches buffer)
- `transcript_get()` called BEFORE `transcript_update()` for CertificateVerify -- CORRECT (line 887 vs 920)
- Calls `xsec_ed25519_verify(pub, content, len, sig)` -- signature matches xsec.h:268-270
- Rejects non-Ed25519 schemes (0x0807) -- CORRECT
- Zeroes cv_content and cv_transcript after use -- CORRECT (line 918-919)

**P1-SIG-LEN-01** (tls13.c:892-899): Ed25519 signature MUST be exactly 64 bytes (RFC 8032).
  Code checks `peer_cert_pub_key_len >= 32` but NEVER validates `sig_len_cv == 64`.
  If server sends sig_len_cv=65 or sig_len_cv=32, the verify function reads 64 bytes
  from `msg_buf + cv_pos` which may read past the actual signature data (within the
  bounds-checked `pt_len` buffer, but semantically wrong). Could accept truncated or
  padded signatures depending on ed25519_verify implementation.
  **Fix**: Add `if (sig_len_cv != 64u) { reject }` before the verify call.

## FIX 3: P1 Hostname Validation (x509.c:703-783 + tls13.c:806-818)

**Status: VERIFIED CORRECT with 2 P2 findings**

- TLS integration: hostname checked after cert parse, before transcript update -- CORRECT
- Null/empty checks on cert and hostname -- CORRECT
- Wildcard: `*.example.com` matches `foo.example.com` but NOT `foo.bar.example.com` -- CORRECT per RFC 6125
- Case-insensitive comparison via `ct_tolower()` -- CORRECT
- CN extraction scans for "CN=" then reads until comma or NUL -- CORRECT
- Buffer capped at 255 chars + NUL -- CORRECT

**P2-NO-SAN-01** (x509.c:749-780): Only checks CN (Common Name). RFC 6125 section 6.4.4
  states that if SAN (Subject Alternative Name) dNSName is present, CN MUST be ignored.
  Modern certificates use SAN exclusively. This implementation will fail hostname validation
  on any certificate that puts the hostname only in SAN (which is most production certs).
  Acceptable for Sprint 3 freestanding scope but must be addressed before production TLS.

**P2-CT-TOLOWER-01** (x509.c:713-715): `ct_tolower()` uses branching (`c >= 'A' && c <= 'Z'`).
  Not constant-time for hostname comparison. Low risk (hostname is not secret), but
  inconsistent with the project's constant-time philosophy.

## FIX 4: PAL-Aether v1.2 Wiring (pal_aether.c)

**Status: VERIFIED with 1 P1 finding**

- `pal_thread_create()` (line 516-536): Validates cfg/out/entry, issues XK_SYS_THREAD_CREATE,

- `pal_thread_join()` (line 544-549): Gets PAL_OBJ_THREAD handle, issues XK_SYS_THREAD_JOIN -- CORRECT
- `pal_sg_build()` (line 829-853): Validates inputs, gets PAL_OBJ_PAGES handle, issues
  XK_SYS_SG_BUILD with phys_base/out/max/total_size -- CORRECT

**P1-TIMER-OBJ-TYPE-01** (pal_aether.c:863,871,879): Timer functions use `PAL_OBJ_EVENT`
  instead of `PAL_OBJ_TIMER`. `PAL_OBJ_TIMER` exists in pal.h:109 but is never used.
  - `pal_timer_create()` line 863: `_alloc_handle(PAL_OBJ_EVENT)` should be `PAL_OBJ_TIMER`
  - `pal_timer_set()` line 871: `_get(h, PAL_OBJ_EVENT)` should be `_get(h, PAL_OBJ_TIMER)`
  - `pal_timer_cancel()` line 879: `_get(h, PAL_OBJ_EVENT)` should be `_get(h, PAL_OBJ_TIMER)`
  This causes type confusion: a timer handle could be used as an event handle and vice versa.
  An `pal_event_wait()` on a timer handle would succeed (wrong behavior). A `pal_timer_set()`
  on an event handle would also succeed (dangerous).
  **Fix**: Replace all three `PAL_OBJ_EVENT` references in timer functions with `PAL_OBJ_TIMER`.
