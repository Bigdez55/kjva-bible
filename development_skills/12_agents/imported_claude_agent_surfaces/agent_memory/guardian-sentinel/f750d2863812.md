## P0 Findings

### GS-S3-P0-001: AES-GCM table-lookup S-Box timing side-channel
- **File**: `sec/xsec/crypto/aes_gcm.c:24-41`
- **Issue**: AES_SBOX[256] lookup table indexed by secret data. On x86 this leaks key material via cache-timing (Bernstein 2005, Osvik-Shamir-Tromer 2006).
- **Impact**: Full AES key recovery via cache-timing attack on shared hardware.
- **Fix**: Implement bitsliced AES or use AES-NI intrinsics with runtime feature detection.

## P1 Findings

### GS-S3-P1-001: TLS 1.3 CertificateVerify NOT verified
- **File**: `sec/xsec/tls/tls13.c:773-784`
- **Issue**: Step 6 receives CertificateVerify but does NOT verify the signature. It only checks inner content type == handshake. The server's identity is never authenticated.
- **Impact**: Man-in-the-middle attack -- any attacker can impersonate any server.
- **Fix**: Parse CertificateVerify, extract signature algorithm + signature, verify against server certificate public key over transcript hash.

### GS-S3-P1-002: TLS 1.3 Certificate NOT parsed or validated
- **File**: `sec/xsec/tls/tls13.c:757-769`
- **Issue**: Step 5 comment says "skip validation for now". Certificate is received but not parsed via xsec_x509_parse() or chain-verified. No certificate pinning, no trust anchor check.
- **Impact**: TLS provides encryption but zero authentication -- equivalent to opportunistic encryption.
- **Fix**: Parse Certificate message, extract cert chain, call xsec_x509_verify_chain(), verify server identity.

### GS-S3-P1-003: X.509 tbs_data pointer calculation unreliable
- **File**: `sec/xsec/cert/x509.c:442-603`
- **Issue**: Multiple conflicting assignments to `out->tbs_data` and `out->tbs_len` (lines 442-444, 594-603). The TBS SEQUENCE TLV boundary is not precisely tracked because `tbs_start` is captured AFTER entering the outer SEQUENCE, but the code tries to subtract 4 to include the tag+length bytes, which fails for variable-length DER length encodings.
- **Impact**: Ed25519 signature verification over wrong byte range => valid certs may be rejected or forged certs accepted.
- **Fix**: Record cursor position BEFORE `der_enter()` for TBS SEQUENCE, and compute tbs_len = (post-TBS position) - (pre-TBS position).

### GS-S3-P1-004: ChaCha20-Poly1305 static scratch buffer (thread-unsafe)
- **File**: `sec/xsec/crypto/chacha20.c:367`
- **Issue**: `static uint8_t poly_scratch[POLY_SCRATCH_MAX]` (32KB+ static buffer) is shared across all callers. Concurrent TLS connections would corrupt each other's MAC computation.
- **Impact**: Authentication tag corruption under concurrent use, leading to either denial of service or potential auth bypass.
- **Fix**: Move scratch buffer to caller-provided context or use stack allocation with bounded maximum size.

### GS-S3-P1-005: X.509 chain verification skips non-Ed25519 signatures
- **File**: `sec/xsec/cert/x509.c:679`
- **Issue**: RSA-PSS and ECDSA signatures are explicitly not verified ("parse-only stub"). Certificates with these algorithms pass chain verification without any signature check.
- **Impact**: Forged certificates with RSA/ECDSA signatures will be accepted as valid.
- **Fix**: Return XSEC_ERR_NOT_SUPPORTED for unverifiable signature algorithms rather than silently passing.

### GS-S3-P1-006: TLS 1.3 double record send in tls_send_record_encrypted
- **File**: `sec/xsec/tls/tls13.c:257-258`
- **Issue**: Line 257 calls `tls_send_record_plain()` with empty payload, then lines 260-271 construct and send a SECOND header + ciphertext. This sends a malformed empty record followed by the real record, which violates RFC 8446 and will cause interop failures with conformant TLS implementations.
- **Impact**: Protocol violation -- non-compliant TLS records.
- **Fix**: Remove the spurious `tls_send_record_plain()` call at line 257.

### GS-S3-P1-007: fe_pack in x25519.c lacks full canonical reduction
- **File**: `sec/xsec/crypto/x25519.c:201-257`
- **Issue**: `fe_pack()` performs two reduction passes but does NOT do a final conditional subtraction of p. After two passes the value may still be >= p (e.g., exactly p itself would reduce to p, not 0). The code comments mention "constant-time friendly" but no conditional subtraction is implemented.
- **Impact**: Non-canonical output from X25519 shared secret computation. May cause interoperability issues and potentially leaks information about the shared secret.
- **Fix**: Add a constant-time conditional subtraction: compute `t - p`, select the result if `t >= p`.

## P2 Findings

### GS-S3-P2-001: AES gf_mul not constant-time
- **File**: `sec/xsec/crypto/aes_gcm.c:56-68`
- **Issue**: `gf_mul()` has a branch `if (b & 1u)` on operand data. This is used in MixColumns which processes state derived from plaintext, not key material, so the risk is lower than the S-Box issue.
- **Fix**: Use branchless conditional XOR: `result ^= a & -(b & 1u)`.

### GS-S3-P2-002: Ed25519 ge_frombytes point validation not constant-time
- **File**: `sec/xsec/crypto/ed25519.c:459-472`
- **Issue**: The check loop `for(k=0;k<32;k++) { if(check_bytes[k]!=0) ok=false; }` and subsequent branching (`if (!ok)`) leaks information about whether the first sqrt succeeded. Should use `xsec_ct_eq()`.
- **Fix**: Replace with `xsec_ct_eq(check_bytes, zero_buf, 32)`.

### GS-S3-P2-003: Ed25519 verify does not check S < l
- **File**: `sec/xsec/crypto/ed25519.c:953-1009`
- **Issue**: RFC 8032 section 5.1.7 requires checking that S (bytes 32-63 of signature) is less than l (the group order). Without this check, malleability attacks are possible -- multiple valid signatures exist for the same message.
- **Fix**: Add a check that `sig[32..63]` as a little-endian integer is < l before proceeding.

### GS-S3-P2-004: SHA-256 finalization corrupts bit_count during padding
- **File**: `sec/xsec/crypto/sha256.c:168-173`
- **Issue**: `xsec_sha256_final()` calls `xsec_sha256_update()` to process padding bytes, which increments `ctx->bit_count`. Then it appends the (now-modified) `ctx->bit_count` as the length field. The padding update adds `pad_len * 8` extra bits to the count, making the final length field incorrect.
- **Impact**: SHA-256 produces wrong hashes for all inputs. This would be caught by the selftest.
- **Status**: The selftest vectors would fail if this is truly broken. The code may work because the bit_count before padding represents the original message length + padding length, and the final 8-byte append also adds to bit_count, but the value serialized is the ALREADY-INCREMENTED one. Need to verify: save bit_count BEFORE padding update.
- **Fix**: Save `uint64_t total_bits = ctx->bit_count` before padding update, serialize that saved value.

### GS-S3-P2-005: Poly1305 final reduction not fully constant-time
- **File**: `sec/xsec/crypto/chacha20.c:289-298`
- **Issue**: The final reduction uses `if (carry)` branch on line 290, which leaks whether the accumulator was >= p. This is a timing side-channel on the MAC computation.
- **Fix**: Use constant-time conditional select: compute both paths and select based on carry using bitwise masking.

### GS-S3-P2-006: Audit ring buffer write_idx/total_events double-increment race
- **File**: `sec/xsec/audit/audit.c:129-139`
- **Issue**: `write_idx` and `total_events` are incremented separately via two atomic operations. Between them, a reader calling `xsec_audit_query_recent()` could see `total_events < write_idx`, leading to incorrect ring calculation. These should be a single atomic counter or protected by the same lock.
- **Fix**: Remove `total_events` and derive it from `write_idx` (they track the same thing).

## P3 Findings

### GS-S3-P3-001: Large stack frames in TLS
- **File**: `sec/xsec/tls/tls13.c:200,242,320-321`
- **Issue**: Multiple ~16KB stack buffers (`scratch[16640]`, `ct_buf[16640]`, `pt_buf[16385]`). On a kernel stack (typically 8-16KB), this will overflow.
- **Fix**: Use the context's `record_buf` field or add caller-provided scratch buffers.

### GS-S3-P3-002: Missing xsec_le64 helper in xsec.h
- **File**: `sec/xsec/include/xsec.h`
- **Issue**: `xsec_put_le64` exists but no `xsec_le64` (load little-endian uint64). ChaCha20 works around this with `xsec_le32` pairs.
- **Fix**: Add `xsec_le64()` for consistency.

### GS-S3-P3-003: TLS server handshake not implemented
- **File**: `sec/xsec/tls/tls13.c:902-905`
- **Issue**: Server-side TLS handshake returns XSEC_ERR_NOT_SUPPORTED. Documented as Sprint 4 work.
- **Status**: Acceptable for Sprint 3 scope, but must be tracked.
