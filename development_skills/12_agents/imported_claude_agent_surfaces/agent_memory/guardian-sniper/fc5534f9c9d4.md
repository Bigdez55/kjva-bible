## Regulatory Knowledge

### PDT (FINRA Rule 4210(f)(8)(B)(ii))
- Alpaca's `pattern_day_trader` flag is REACTIVE not preventive -- violation already occurred by the time it's set
- Platform must count day trades proactively: 3 round-trips in 5 business days, HARD_STOP on 4th if equity < $25K
- Cash accounts avoid PDT but require T+1 settlement (SEC Rule 15c6-1(a) amended May 2024)
- `TradingService.validate_trade()` in `trading.py` has ZERO PDT logic currently

### Wash Sales (IRC Section 1091)
- 61-day window is correct: 30 before + sale day + 30 after
- **Crypto included as of 2025 tax year** (IIJA 2021 amendments, effective Jan 1, 2025)
- Training data in `expand_training_data.py` line 94 does NOT mention crypto wash sales -- model may give wrong advice
- "Substantially identical" definition needed for options on same underlying and for wrapped crypto tokens

### Alpaca Crypto API Constraints
- Supported order types: market, limit, stop_limit, trailing_stop
- NO plain `stop` orders for crypto -- only `stop_limit`
- Trailing stop may NOT be supported for crypto -- verify at implementation time
- Time in force must be `gtc` not `day` (crypto trades 24/7)
- `_prepare_order_data()` in alpaca.py line 233 hardcodes `"time_in_force": "day"` -- MUST override for crypto
- NO staking support via Alpaca API

### Alpaca Options API
- Sides: buy_to_open, sell_to_open, buy_to_close, sell_to_close
- Order types: market, limit, stop, stop_limit
- Multi-leg via `legs[]` parameter
- Tier 1: covered calls + cash-secured puts only; Tier 2: spreads; No Tier 3 (no naked options)
- `get_account_features()` already reads `options_approved_level` but NO execution gate exists

### Audit Log Compliance (SEC Rule 17a-4)
- DB triggers alone are insufficient -- superusers can disable triggers
- Minimum controls: INSERT-only GRANT, hash chaining, bitemporal timestamps, WORM offsite replication
- Retention: 6 years minimum (first 2 in easily accessible location)
