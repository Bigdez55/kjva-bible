# Sprint 11-13 Boot Path Hardware Audit (2026-03-07)

## Scope
HP EliteBook x360 1030 G4 boot compatibility audit across:
- AetherBoot page tables (aetherboot.c S12)
- HHDM/KERN_BASE consistency (6 headers)
- ABI transition (UEFI ms_abi -> System V)
- kmain step ordering (HHDM before MMIO drivers)
- elitebook_x360.c integration
- XHPET/XLAPIC HHDM access patterns

## Findings Summary

### AREA 1: Page Tables (aetherboot.c S12 build_page_tables) -- PASS with NOTE
- Identity map: 4x 1GB huge pages (PDPE1G) at PML4[0]
- HHDM: 4x 1GB huge pages at PML4[273]
- Kernel: 2MB pages at PML4[511], PDPT[510]
- PDPE1G support: Whiskey Lake (i5-8265U) DOES support 1GB pages
  (CPUID 0x80000001 EDX bit 26 = 1 on all Intel since Westmere)
- PTE_G (global) bit set on identity+HHDM pages: requires CR4.PGE
  UEFI firmware sets CR4.PGE on all modern Intel platforms. PASS.
- PTE_NX NOT set on identity/HHDM: acceptable since EFER.NXE is
  not yet enabled at AetherBoot stage. NXE is enabled at Step 4.5.
- NOTE: identity+HHDM only cover 4GB. MMIO devices above 4GB
  (unlikely on EliteBook G4) would be inaccessible until vmm_init.

### AREA 2: HHDM Base Consistency -- PASS
All six definitions are identical: 0xFFFF888000000000
- xenos.h:64         HHDM_BASE = 0xFFFF888000000000ULL
- xhpet.h:152        XHPET_HHDM_BASE = 0xFFFF888000000000ULL
- xlapic.h:180       XLAPIC_HHDM_BASE = 0xFFFF888000000000ULL
- xacpi.h:292        XACPI_HHDM_BASE = 0xFFFF888000000000ULL
- vmm.c:95           PML4_IDX_HHDM = 273 (= 0xFFFF888000000000 >> 39 & 0x1FF)
- aetherboot.c:454   HHDM_BASE_VIRT = 0xFFFF888000000000ULL, pml4[273]

### AREA 3: KERN_BASE Consistency -- PASS
- xenos.ld:24        KERNEL_BASE = 0xFFFFFFFF80000000
- xenos.h:71         KERN_BASE = 0xFFFFFFFF80000000ULL
- aetherboot.c:453   KERN_BASE_VIRT = 0xFFFFFFFF80000000ULL
- vmm.c:96           PML4_IDX_KERN = 511
- Verification: (0xFFFFFFFF80000000 >> 39) & 0x1FF = 511. CORRECT.
- PDPT index: (0xFFFFFFFF80000000 >> 30) & 0x1FF = 510. Matches
  pdpt_kern[510] in aetherboot.c:518 and vmm.c:316.

### AREA 4: ABI Transition -- PASS
- uefi_types.h:38    EFIAPI = __attribute__((ms_abi))
- efi_main() declared with EFIAPI -- uses ms_abi (RCX,RDX,R8,R9)
- do_kernel_jump() at line 702-716:
  - Loads CR3 with new page tables
  - Puts handoff pointer in RDI via inline asm
  - Jumps to kernel virtual entry point
  - entry.asm:88: "mov r15, rdi" -- receives boot_info in RDI (System V)
- The ABI transition is CORRECT: efi_main runs ms_abi, do_kernel_jump
  manually loads RDI (System V first arg) via asm, then jumps to
  _xenos_start which expects RDI per System V ABI.

### AREA 5: kmain Step Ordering -- PASS
Boot sequence verified:
  Step 1: serial_init (COM1)
  Step 2: IDT init
  Step 3: GDT + TSS
  Step 4: PMM init (buddy allocator from EFI mmap)
  Step 4.5: EFER.NXE enabled (before VMM)
  Step 5: vmm_init() -- HHDM ESTABLISHED HERE (PML4[273], 128GB, 2MB pages)
  Step 6: heap_init
  Step 7: PAL init
  Step 8: vDRAM
  Step 9: SYSCALL MSRs
  Step 10: apic_init (uses HHDM -- safe, Step 5 done)
  Step 10a: XACPI (uses HHDM -- safe)
  Step 10b: XPCI init
  Step 10c: XLAPIC init (uses HHDM via XLAPIC_PHYS_TO_VIRT -- safe)
  Step 10d: XHPET init (uses HHDM via XHPET_PHYS_TO_VIRT -- safe)

HHDM is established at Step 5. All MMIO-accessing drivers run at
Step 10+. Order is CORRECT.

### AREA 6: elitebook_x360.c Integration -- FLAG (3 issues)

**P1-01: Platform mismatch (STILL OPEN from Sprint 9)**
elitebook_x360.c:7-8 says "Intel Tiger Lake / Alder Lake platform"
but the HP EliteBook x360 1030 G4 is Whiskey Lake (8th gen).
Tiger Lake is 11th gen (EliteBook x360 1040 G8).
EC register addresses may differ between Whiskey Lake and Tiger Lake.
Battery registers (0x2C-0x36), fan registers (0x93-0x95), camera
register (0x40), Sure Start (0x50), hinge (0x61) are all platform-
specific and have NOT been verified against Whiskey Lake EC firmware.

**P1-02: Thunderbolt MMIO without HHDM (STILL OPEN from Sprint 9)**
elitebook_x360.c:306-307:
  volatile uint32_t *pm_reg =
      (volatile uint32_t *)(uintptr_t)(HP_TBT_MMIO_BASE + HP_TBT_PCIE_PWR);
HP_TBT_MMIO_BASE = 0xFED80000 -- this is a PHYSICAL address.
The code accesses it as a raw virtual address WITHOUT adding HHDM_BASE.
After vmm_init(), identity mapping from AetherBoot is replaced by VMM's
page tables. The low 4GB identity map is NOT preserved by vmm_init()
(it only creates HHDM and KERN mappings). This will PAGE FAULT.
Fix: access should be (HHDM_BASE + HP_TBT_MMIO_BASE + HP_TBT_PCIE_PWR).

Additionally, Whiskey Lake does NOT have integrated Thunderbolt 4.
The 1030 G4 has Thunderbolt 3 via an Alpine Ridge or Titan Ridge
controller, with a DIFFERENT MMIO base than Tiger Lake's built-in TBT4.
This entire section should be conditionally compiled or runtime-detected.

**P1-03: Driver not called from kmain()**
hp_elitebook_x360_init() is never called from main.c.
The Makefile (line 53) compiles elitebook_x360.c and links it into
xenos.elf, but grep shows NO reference to hp_elitebook_x360_init
in main.c. The driver is dead code at runtime.

### AREA 7: WiFi PCI ID -- FLAG (STILL OPEN from Sprint 9)
xpci.h:144 defines XPCI_DEV_INTEL_WIFI_AX201 = 0x02F0
The HP EliteBook x360 1030 G4 ships with Intel Wireless-AC 9560 (CNVi),
PCI device ID 0x2526 (iwlwifi). AX201 (0x02F0) is the Tiger Lake/
Ice Lake WiFi 6 part and is NOT present in the 1030 G4.
This does not cause a boot failure (xpci_find_device simply won't
match), but it means WiFi hardware detection will silently fail.

### AREA 8: LAPIC Timer Calibration -- FLAG (known from Sprint 9)
lapic.c:209-210:
  volatile uint32_t delay = 1000000u;
  while (delay > 0u) { delay--; }
This is a busy-loop calibration stub. The comment says "replace with
xhpet_udelay". On Whiskey Lake at 3.9GHz boost, 1M loop iterations
takes approximately 250-500 microseconds, NOT 10ms. This means:
  bus_hz = elapsed * 1600
With only ~250us of actual time, elapsed will be ~40x too small,
making bus_hz ~40x lower than reality. The LAPIC timer will fire
at ~40x the intended frequency.
The comment at line 207 acknowledges this: "Full calibration wiring
to XHPET is done via elitebook bring-up."

### AREA 9: Handoff Structure Mismatch -- FLAG
AetherBoot uses xenos_handoff_t (aetherboot.c:400-416) with magic
XENOS_BOOT_MAGIC = 0x4145544845523031 ("AETHER01").
The kernel main.c uses xenos_boot_info_t (xenos.h:89-108) with the
SAME magic. These two structs have IDENTICAL layouts (field-by-field
match verified). PASS.

However, boot_handoff.h defines a DIFFERENT struct xenos_boot_handoff_t
with magic XENOS_HANDOFF_MAGIC = 0x47454E4F53484F46 ("GENOSHOF") and
a completely different layout (4096-byte packed struct with HP-specific
fields). elitebook_x360.c:374 takes a const xenos_boot_handoff_t*
parameter. kmain() never creates or populates this struct. This is
why hp_elitebook_x360_init cannot be wired -- the handoff types differ.

### AREA 10: AetherBoot HHDM Coverage vs VMM HHDM -- PASS with NOTE
- AetherBoot: 4GB HHDM (4x 1GB huge pages)
- VMM: 128GB HHDM (2MB pages, HHDM_MAP_LIMIT = 128GB)
- EliteBook has 16GB RAM -- VMM's 128GB is sufficient
- LAPIC at 0xFEE00000 -- within 4GB, accessible immediately
- HPET at 0xFED00000 -- within 4GB, accessible immediately
- Framebuffer typically below 4GB on EliteBook -- accessible
- The VMM replaces AetherBoot's page tables at Step 5 with full
  128GB HHDM. All MMIO below 128GB is then accessible. PASS.
