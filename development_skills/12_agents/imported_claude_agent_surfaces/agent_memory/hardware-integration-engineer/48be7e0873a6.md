# Sprint 15 NIC Driver Correctness Audit (2026-03-07)

## Files Audited
- `net/xnet/drv/virtio_net.c` (746 LOC)
- `net/xnet/drv/e1000.c` (787 LOC)
- `net/xnet/drv/include/virtio_net.h` (366 LOC)
- `kernel/xenos/drv/pci.c` (265 LOC)
- Total: 2,164 LOC

## Score: 8/10

Both drivers are well-structured, follow freestanding C conventions, implement correct
error cleanup paths, and register properly with the XNET stack. The code quality is
high for Sprint 15 scope. However, there are several hardware protocol issues that
must be fixed before first-packet-on-wire in Sprint 16.

## Issues Found

### P1-01 MEDIUM: virtio-net queue size capping is illegal in legacy mode
- File: virtio_net.c:296-298
- The driver caps qsz to VIRTIO_NET_QUEUE_SIZE (128) if the device reports more.
- In legacy virtio (Virtio 1.0 Appendix A), the queue size register is READ-ONLY.
  The device reports the queue size; the driver MUST use it or not use the queue.
  The driver cannot negotiate a smaller queue. If QEMU reports 256, allocating
  only 128 descriptors means the device will DMA into memory beyond the allocation.
- FIX: Either use the device-reported qsz directly, OR write the desired smaller
  qsz back via VIRTIO_IO_QUEUE_SIZE (only valid for virtio 1.0 modern, NOT legacy).
  For legacy: remove the capping, use device qsz as-is.
- IMPACT: Data corruption / page fault -- device writes to unallocated memory.

### P1-02 MEDIUM: virtio-net skips FEATURES_OK status bit (modern device path)
- File: virtio_net.c:484-487
- The handshake jumps from ACKNOWLEDGE|DRIVER directly to ACKNOWLEDGE|DRIVER|DRIVER_OK.
- For LEGACY mode, this is CORRECT -- legacy virtio does NOT use FEATURES_OK.
- But the driver also tries to probe modern device ID 0x1041 (virtio_net.c:699-701).
  Modern virtio REQUIRES FEATURES_OK between DRIVER and DRIVER_OK.
- FIX: If the matched device_id is 0x1041 (modern), the handshake must include
  FEATURES_OK. Or simply remove the modern device probe since the driver only
  implements the legacy I/O-space register interface (modern uses PCI capabilities).
- IMPACT: If a modern virtio-net device is present, the driver will misconfigure it.

### P1-03 MEDIUM: E1000 missing TIPG (Transmit Inter-Packet Gap) register setup
- File: e1000.c, e1k_init_tx()
- The E1000 TIPG register (offset 0x00410) is NOT programmed.
- The Intel 82540EM manual (section 14.5) specifies TIPG must be set for
  proper frame spacing: IPGT=10, IPGR1=8, IPGR2=6 -> 0x00602006.
- QEMU's E1000 emulation will work without it (TIPG defaults to a sane value),
  but real hardware or stricter emulators may fail.
- FIX: Add `e1k_write(dev, 0x00410, 0x0060200A);` in e1k_init_tx().
- IMPACT: LOW for QEMU, HIGH for any real E1000 hardware.

### P1-04 LOW: E1000 RX tail pointer semantics inverted
- File: e1000.c:618, 338, 665-668
- Init sets RDT = E1000_NUM_RX_DESC - 1 (127) and rx_tail = 0.
- In the poll loop, the driver reads descriptors starting at rx_tail (0).
- After consuming a descriptor at `old_tail`, it writes `old_tail` to E1000_RDT.
- This is semantically correct for the E1000: RDT points to the last descriptor
  the driver has given back. Writing old_tail (the one just consumed) gives it back.
- PASS -- no bug, but the naming `rx_tail` as the software scan pointer is
  confusing since the E1000 spec calls the hardware register "tail" too.
  Consider renaming to `rx_next` or `rx_scan` for clarity.

### P2-01 LOW: boot_chain.c treats xnet_drivers_probe return as count, not status
- File: init/gensd/boot_chain.c:335-337
- `int32_t drv_count = xnet_drivers_probe();` -- variable named "drv_count"
- `xnet_drivers_probe()` returns XNET_OK (0) on success or XNET_ERR_NOT_FOUND (-6).
- The printf says "%d NIC driver(s) registered" which would print "0 NIC driver(s)"
  on success, or "-6 NIC driver(s)" on failure.
- The return value is not checked for error -- xchain_network() always returns PAL_OK
  even if no NIC was found. This means boot continues without networking silently.
- FIX: Check rc != XNET_OK and either warn or set a global "network unavailable" flag.
- IMPACT: Silent boot with no network; confusing diagnostic output.

### P2-02 LOW: Duplicate e1k_alloc_dma / vn_alloc_dma / memset / memcpy
- Files: virtio_net.c and e1000.c both define identical DMA alloc + memset + memcpy.
- These should be factored into a shared xnet_drv_common.c or into virtio_net.h.
- IMPACT: Code duplication only; no functional bug.

### P2-03 INFO: E1000 uses E1000_RX_BUF_SIZE for TX buffer slot size
- File: e1000.c:373, 570, 574
- TX buffer pool uses `E1000_RX_BUF_SIZE` (2048) per slot. This works but the
  constant name is misleading. Should define E1000_TX_BUF_SIZE = 2048 separately.

### P2-04 INFO: No poll integration path
- Neither virtio_net_poll() nor e1000_poll() is called from any timer/tick loop.
- The boot chain calls xnet_drivers_probe() but never sets up periodic polling.
- Sprint 16 must wire a timer callback or IRQ handler to call *_poll() functions.
