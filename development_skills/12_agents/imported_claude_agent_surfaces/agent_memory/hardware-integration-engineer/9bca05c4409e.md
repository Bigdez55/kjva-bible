## 1. NETIF ABSTRACTION -- INTEL NIC COMPATIBILITY

### 1.1 Assessment: SUFFICIENT for both I219-LM and AX201

The `xnet_netif` abstraction is hardware-agnostic and well-suited:
- 6-byte MAC address: CORRECT for both Ethernet (I219-LM) and Wi-Fi (AX201)
- MTU range validated: `ETHER_HDR_LEN` (14) to `ETHER_MAX_JUMBO` (9000)
- tx_fn callback pattern: Clean driver hook -- each NIC provides its own tx_fn
- 8-slot interface table: Sufficient (eth0, wlan0, lo = 3 of 8)
- Interface flags: UP, BROADCAST, LOOPBACK, PROMISC, RUNNING -- standard set

### 1.2 Integration Points Needed for Intel NICs

**I219-LM Ethernet (e1000e-class):**
- PCI BAR0 MMIO register base for TX/RX descriptor rings
- TX descriptor ring: 256 entries, each 16 bytes (TDESC legacy format)
- RX descriptor ring: 256 entries, each 16 bytes
- Interrupt: MSI-X vector for TX/RX completion
- MAC address: Read from NVM (EEPROM) at power-on, 3-word read at offset 0x00-0x02
- Link status: PHY register via MDIO (register 0x01, bit 2)
- tx_fn implementation: write TX descriptor, bump TDT (Transmit Descriptor Tail)

**AX201 Wi-Fi 6 (iwlwifi-class):**
- PCIe BAR0 for CSR registers
- Firmware upload required (ucode sections loaded to SRAM)
- TX queue: 512-entry TFD (Transmit Frame Descriptor) ring
- 802.11 encapsulation: requires additional header (24-30 bytes) beyond Ethernet
- tx_fn must handle: Ethernet-to-802.11 frame conversion in firmware
- MAC address: Read from OTP/NVM section of firmware image

### 1.3 Missing Netif Features for Wi-Fi

| Feature | Status | Impact |
|---------|--------|--------|
| SSID/BSS association | MISSING | Cannot connect to Wi-Fi network |
| WPA3/SAE authentication | MISSING | No security handshake |
| 802.11 frame encapsulation | MISSING | Wi-Fi frames differ from Ethernet |
| Power save mode | MISSING | Battery drain on Wi-Fi |
| Scan/probe request | MISSING | Cannot discover networks |
| Channel management | MISSING | Cannot tune radio |

**Verdict:** Netif is ready for wired Ethernet (I219-LM). Wi-Fi (AX201) requires a
substantial 802.11 MAC layer above netif -- Sprint 5+ scope.

## 2. ETHERNET LAYER -- HARDWARE COMPATIBILITY

### 2.1 CRC32: CORRECT

```c
// Polynomial: 0xEDB88320 (reversed IEEE 802.3) -- CORRECT
// Algorithm: Table-driven, 256-entry lookup -- CORRECT
// Init: 0xFFFFFFFF, final XOR: 0xFFFFFFFF -- CORRECT per IEEE 802.3
```

The CRC32 implementation matches the IEEE 802.3 standard exactly. Both the I219-LM
and AX201 compute FCS in hardware, so software CRC is a fallback/validation path only.

### 2.2 CRC32 Init Race: HW-S3-02 (from prior audit, STILL OPEN)

```c
static bool g_crc32_table_ready = false;  // NOT ATOMIC
static void crc32_init_table(void) {
    if (g_crc32_table_ready) return;       // TOCTOU race on SMP
    ...
    g_crc32_table_ready = true;
}
```

**Issue:** Two cores calling `xnet_eth_crc32()` simultaneously on first use can race
on table initialization. The `g_crc32_table_ready` flag is a plain `bool`, not atomic.
On x86_64 this is unlikely to cause corruption (store is naturally aligned and the
table computation is idempotent), but it violates kernel coding standards.

**Fix:** Use `pal_atomic_load/store` or call `crc32_init_table()` once from `xnet_init()`.

### 2.3 Minimum Frame Size: CORRECT

```c
#define ETHER_MIN_PAYLOAD  46U    // 64 - 14 (hdr) - 4 (FCS) = 46
```

Payload is zero-padded to 46 bytes when undersized. Frame minimum is 14 + 46 = 60
bytes (without FCS). Hardware appends 4-byte FCS, yielding 64 bytes on the wire.
This matches IEEE 802.3 section 3.1.1. CORRECT.

### 2.4 VLAN Tagging: EXPLICITLY EXCLUDED

```c
case ETHERTYPE_VLAN:
    /* VLAN tagging -- not supported; drop */
    return XNET_OK;
```

VLAN frames (EtherType 0x8100) are silently dropped. This is acceptable for Sprint 3.
The I219-LM supports hardware VLAN tag insertion/stripping -- when VLAN support is
added, the driver can strip the tag before delivering to netif, making it transparent
to the stack.

### 2.5 FCS Policy: CORRECT

The comment at ethernet.c:232-237 documents that drivers must NOT deliver FCS bytes
to the stack. Both I219-LM and AX201 strip FCS in hardware before DMA to host memory.
This matches the design.

## 3. DMA CONSIDERATIONS

### 3.1 Current State: SOFTWARE COPY (No DMA)

The current packet I/O path is entirely software-copy:

```
TX path: xnet_ip4_output() -> stack-allocated buffers -> xnet_eth_build()
         -> memcpy into eth_frame[] -> tx_fn() -> ??? (no driver yet)

RX path: driver calls xnet_netif_input(frame, len) -> xnet_eth_input()
         -> protocol dispatch (all pointer-based, zero-copy within stack)
```

Key observation: `xnet_ip4_output()` at line 666-667 builds the IP packet in a
stack-allocated buffer:
```c
uint8_t ip_pkt[IP4_HDR_LEN + ETHER_MAX_PAYLOAD];  // 1520 bytes on stack
uint8_t eth_frame[ETHER_HDR_LEN + IP4_HDR_LEN + ETHER_MAX_PAYLOAD]; // 1534 bytes
```

**Total stack usage per xnet_ip4_output call: ~3054 bytes.** This is acceptable for
a kernel with 8KB+ stacks but should be documented as a constraint.

### 3.2 PAL DMA API Status

```c
// pal_aether.c:829 -- pal_sg_build() is WIRED to XENOS syscall XK_SYS_SG_BUILD
// However, the XENOS kernel handler returns ENOSYS (not implemented)
// pal_pages_alloc() with PAL_MEM_DMA flag: allocates below 4GB (DMA32 zone)
```

`pal_sg_build()` issues the syscall but the kernel does not handle it yet.

### 3.3 Sprint 4 DMA Wiring Requirements

To enable zero-copy DMA networking for the I219-LM:

**Step 1: DMA buffer allocation**
- Allocate TX/RX descriptor rings via `pal_pages_alloc(PAL_MEM_DMA)`
- Allocate packet buffers in DMA-accessible memory (below 4GB for I219-LM)
- Each RX buffer: 2KB aligned (I219-LM requirement)
- Each TX buffer: no alignment requirement, but 16-byte alignment recommended

**Step 2: Scatter-gather list**
- Implement `XK_SYS_SG_BUILD` handler in XENOS kernel
- Build SG list from physical pages for multi-buffer packets
- I219-LM supports up to 8 SG entries per TX descriptor (TDESC.POPTS.IXSM)

**Step 3: Cache coherency**
- x86_64 is cache-coherent for DMA (no explicit flush needed for I219-LM)
- However, MFENCE before doorbell write is required (already documented in MEMORY.md)
- For non-coherent devices (future): add `pal_dma_sync_for_device/cpu()` API

**Step 4: TX path rewrite**
- Replace stack-allocated buffers with pre-allocated DMA ring buffers
- tx_fn receives a DMA buffer descriptor instead of a pointer-to-copy
- TX completion interrupt frees the DMA buffer back to the ring

**Step 5: RX path rewrite**
- Pre-post RX DMA buffers to I219-LM RX descriptor ring
- RX interrupt: harvest completed descriptors, call `xnet_netif_input()`
- Refill RX ring with fresh DMA buffers

## 4. TCP/IP CHECKSUM AND HARDWARE OFFLOAD

### 4.1 IP Header Checksum: SOFTWARE ONLY, CORRECT

```c
uint16_t xnet_ip4_checksum(const void *hdr, size_t hdr_len) {
    // ones-complement 16-bit sum -- RFC 791 compliant
    // Odd-byte handling: shifts trailing byte left by 8 -- see finding below
}
```

The I219-LM supports TX IP header checksum offload (TXCMD.IP = 1 in TX descriptor).
When enabled, the NIC computes the IP header checksum in hardware, and the driver
should set `ip_hdr.checksum = 0` before posting to the TX ring.

**Current impact:** None. Software checksum is always correct. Offload is a
performance optimization for Sprint 5+.

### 4.2 TCP/UDP Pseudo-Header Checksum: CORRECT

```c
uint32_t xnet_ip4_pseudo_checksum(src_ip, dst_ip, protocol, segment_len) {
    // Converts IPs to NBO before splitting into 16-bit words -- CORRECT
    // Returns partial sum (not folded) -- CORRECT for TCP/UDP to extend
}
```

The pseudo-header checksum correctly converts to network byte order before
computation. TCP and UDP callers add their own header+data words and fold.

### 4.3 Checksum Offload Architecture for Sprint 5+

Both I219-LM and AX201 support TX checksum offload:

| Offload | I219-LM | AX201 | XNET Status |
|---------|---------|-------|-------------|
| TX IP checksum | Yes (TXCMD.IP) | Yes (fw) | Software only |
| TX TCP checksum | Yes (TXCMD.TCP) | Yes (fw) | Software only |
| TX UDP checksum | Yes (TXCMD.UDP) | Yes (fw) | Software only |
| RX IP checksum | Yes (RXDESC.IPE) | Yes (fw) | Software verify |
| RX TCP checksum | Yes (RXDESC.TCPE) | Yes (fw) | Software verify |
| RX UDP checksum | Yes (RXDESC.UDPE) | Yes (fw) | Software verify |

**Recommendation:** Add `XNET_IF_FLAG_TX_CSUM_IP`, `XNET_IF_FLAG_TX_CSUM_TCP`,
`XNET_IF_FLAG_TX_CSUM_UDP` flags to `xnet_netif`. When set, the IP/TCP/UDP output
paths skip software checksum and set the field to 0, letting hardware compute it.

### 4.4 FINDING: IP Checksum Odd-Byte Handling (HW-S3-03, MEDIUM)

```c
// ip4.c:68
if (hdr_len & 1U) {
    sum += (uint32_t)(*(const uint8_t *)words) << 8;
}
```

The odd trailing byte is shifted left by 8 bits, placing it in the HIGH byte of a
16-bit word. This is the correct behavior for network byte order (big-endian) data
on a big-endian machine. However, on x86_64 (little-endian), the byte should be
placed in the LOW byte position (no shift), because the 16-bit words were already
read in little-endian order by the `*words++` loop.

**Analysis:** For IPv4 headers specifically, `hdr_len` is always a multiple of 4
(IHL is in 32-bit words), so this code path is NEVER reached for IP header checksums.
For the pseudo-header checksum, the function is called with even-length data only.
The odd-byte path is dead code for all current callers. However, `xnet_ip4_checksum`
is declared as a general-purpose checksum in xnet.h and could be called with odd
lengths in the future. The `<< 8` is technically wrong on little-endian for a generic
ones-complement sum function.

**Severity:** MEDIUM (dead code today, latent bug for future callers)
**Impact on Intel NICs:** NONE (IP headers are always 4-byte aligned)

## 5. ENDIANNESS AUDIT

### 5.1 xnet_htons: CORRECT

```c
static inline uint16_t xnet_htons(uint16_t x) {
    return (uint16_t)((x >> 8) | (x << 8));  // Swap bytes -- correct for LE->BE
}
```

On x86_64 (little-endian), this correctly swaps bytes to produce network byte order.
The `__BYTE_ORDER__` compile-time check provides a no-op path for big-endian targets.

### 5.2 xnet_htonl: CORRECT

```c
static inline uint32_t xnet_htonl(uint32_t x) {
    return ((x >> 24) & 0x000000FFU) |
           ((x >>  8) & 0x0000FF00U) |
           ((x <<  8) & 0x00FF0000U) |
           ((x << 24) & 0xFF000000U);
}
```

Standard 4-byte reversal. CORRECT for x86_64 little-endian.

### 5.3 xnet_htonll: FINDING (HW-S3-04, LOW)

```c
// gossip.c:69 -- LOCAL definition, NOT in xnet.h
static inline uint64_t xnet_htonll(uint64_t h) {
    uint32_t hi = xnet_htonl((uint32_t)(h >> 32));
    uint32_t lo = xnet_htonl((uint32_t)(h & 0xFFFFFFFFU));
    return ((uint64_t)hi) | (((uint64_t)lo) << 32);  // REVERSED
}
```

**Issue:** The `hi` and `lo` words are assembled in the wrong order. After `htonl`,
the original high 32 bits are in `hi` and the original low 32 bits are in `lo`.
For correct 64-bit byte reversal, the result should be `((uint64_t)lo << 32) | hi`,
which places the original low bytes (now reversed) in the high position.

Wait -- let me re-analyze. Given input `0xAABBCCDDEEFF0011`:
- `h >> 32` = `0xAABBCCDD`, `htonl` -> `0xDDCCBBAA` = `hi`
- `h & 0xFFFFFFFF` = `0xEEFF0011`, `htonl` -> `0x1100FFEE` = `lo`
- Result: `hi | (lo << 32)` = `0x1100FFEEDDCCBBAA`

Expected htonll: full byte reversal = `0x1100FFEEDDCCBBAA`. **CORRECT.**

The assembly `hi | (lo << 32)` puts the byte-reversed low word in the high position
and the byte-reversed high word in the low position, which is the full 8-byte reversal.
**No bug here.** The naming is confusing but the math is right.

### 5.4 xnet_htonll Missing from xnet.h: FINDING (HW-S3-05, LOW)

`xnet_htonll` / `xnet_ntohll` are defined as `static inline` in `gossip.c` only.
They are not available to other translation units. If any future module needs 64-bit
byte order conversion, it would have to duplicate the implementation.

**Recommendation:** Move `xnet_htonll` / `xnet_ntohll` to `xnet.h` alongside htons/htonl.

## 6. ADDITIONAL HARDWARE FINDINGS

### 6.1 FINDING: tx_fn Callback Receives Copy, Not Live Struct (HW-S3-06, INFO)

```c
// netif.c:313 -- Fix S3-DI-M04
struct xnet_netif iface_snap = *iface;   /* 128-ish bytes, all POD */
pal_spin_unlock(&g_netif_lock);
xnet_status_t rc = tx(&iface_snap, frame, frame_len);
```

The tx_fn callback receives a stack copy of the interface struct. This is safe but
means the driver cannot use the `xnet_netif_t*` to access mutable state (e.g., driver
private data). Real NIC drivers (I219-LM, AX201) will need a way to map the interface
index back to driver-private state (ring buffers, MMIO base, etc.).

**Recommendation:** Either:
(a) Add a `void *driver_priv` field to `struct xnet_netif` that the snapshot copies, or
(b) Have tx_fn use `if_idx` (already available from the interface name or a new param)
    to look up driver state in its own table.

### 6.2 FINDING: Stack-Allocated Packet Buffers (HW-S3-07, INFO)

```c
// ip4.c:666-674
uint8_t ip_pkt[IP4_HDR_LEN + ETHER_MAX_PAYLOAD];     // 1520 bytes
uint8_t eth_frame[ETHER_HDR_LEN + IP4_HDR_LEN + ETHER_MAX_PAYLOAD]; // 1534 bytes
```

~3054 bytes of stack per `xnet_ip4_output()` call. With XENOS kernel stacks (likely 8KB
or 16KB), this is acceptable but tight if `ip4_output` is called from deep call chains.
DMA ring buffers in Sprint 4 will eliminate this pattern.

### 6.3 FINDING: No Promiscuous Mode Toggle API (HW-S3-08, LOW)

`XNET_IF_FLAG_PROMISC` is defined but there is no API to set/clear it. The I219-LM
supports promiscuous mode via the RCTL.UPE (Unicast Promiscuous Enable) register bit.
Packet capture and debugging require this.

### 6.4 FINDING: No Link State Change Notification (HW-S3-09, LOW)

The netif abstraction has UP/RUNNING flags but no mechanism for the NIC driver to
report link state changes (cable unplug, Wi-Fi disconnect). The I219-LM generates
a Link Status Change interrupt (ICR.LSC) that should toggle RUNNING.

**Recommendation:** Add `xnet_netif_set_link_state(uint32_t if_idx, bool link_up)`
for drivers to call from their interrupt handler.
