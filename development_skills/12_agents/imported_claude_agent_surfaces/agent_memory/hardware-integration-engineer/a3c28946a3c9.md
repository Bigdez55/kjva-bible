# Sprint 3 Hardware Integration Audit — 2026-03-05

## Scope
READ-ONLY audit of Sprint 3 files for hardware-level correctness on HP EliteBook x360 1030 G4.

## Files Audited
1. pal/src/pal_aether.c — PAL-Aether syscall wiring (PMM/VMM/IPC/Thread/DMA)
2. sec/xsec/crypto/aes_gcm.c — AES-256-GCM software path
3. net/xnet/core/netif.c — Network interface abstraction
4. net/xnet/link/ethernet.c — Ethernet CRC32 + frame handling
5. kernel/xenos/drv/nvme.c — NVMe PCIe MMIO driver

## Findings Summary

### PASS: Syscall ABI (Validation #1)
- pal_aether.c `_xk_syscall4`: RAX=nr, RDI=a1, RSI=a2, RDX=a3, R10=a4
- syscall.asm `_syscall_entry`: RAX=nr, RDI=a1, RSI=a2, RDX=a3, R10=a4
- Shuffle chain in asm is dependency-safe (reads before writes)
- RCX/R11 correctly saved/restored (SYSCALL clobbers them)
- Clobber list in inline asm: "rcx", "r11", "memory" -- correct

### PASS: Syscall Numbers Match (Validation #1 cont.)
- pal_aether.c defines match kernel/xenos/core/main.c exactly (1-28)
- All currently return XENOS_ENOSYS except THREAD_YIELD(0) and THREAD_SELF(1)

### ADVISORY: DMA Scatter-Gather (Validation #2)
- pal_sg_build() passes phys_base to kernel via syscall -- correct pattern
- Missing: no explicit cache flush before DMA (CLFLUSH/WBINVD)
- Intel IOMMU (VT-d) on EliteBook requires DMAR table parsing -- not yet implemented
- Status: Acceptable for Sprint 3 (syscall stub returns ENOSYS); needs Sprint 4 work

### PASS: Thread Stack Alignment (Validation #3)
- syscall.asm Step 7: kernel_rsp0 is 16-byte aligned by gdt_reload()
- 2 pushes (RCX, R11) = 16 bytes, RSP stays 16-byte aligned
- call pushes 8 bytes -> 8-byte misaligned at callee entry (correct per SysV ABI)
- Thread creation passes stack_size to kernel but kernel stub returns ENOSYS

### PASS: Network Spinlock SMP Safety (Validation #4)
- g_netif_lock uses pal_spinlock_t with __atomic_exchange_n ACQUIRE
- pal_spin_unlock uses __atomic_store_n RELEASE
- All netif functions lock before access, unlock on all exit paths
- Stats use pal_atomic_add (__atomic_fetch_add SEQ_CST) -- correct for SMP
- ISSUE-LOW: xnet_netif_send reads g_interfaces[if_idx] AFTER unlock (line 304)
  - tx_fn pointer is copied under lock (line 301), which is fine
  - But the callback receives &g_interfaces[if_idx] which could be concurrently
    unregistered. Race window is small but real. Recommend: copy netif data under lock.
- ISSUE-LOW: xnet_netif_set_ipv4 printf at line 214 reads g_interfaces after unlock.
  Same pattern. Not crash-critical (worst case: stale name in debug log).

### PASS: Ethernet CRC32 (Validation #5)
- Polynomial 0xEDB88320 is correct (bit-reversed IEEE 802.3 0x04C11DB7)
- Init: 0xFFFFFFFF, XOR out: 0xFFFFFFFF -- standard
- Table generation algorithm is canonical: for each byte, 8 iterations of
  shift-right with conditional XOR of polynomial
- Thread safety issue: g_crc32_table_ready is a plain bool, not atomic.
  Two CPUs calling xnet_eth_crc32() simultaneously on first use could
  race on table init. Non-critical (idempotent writes) but not clean.

### PASS: Timer/HPET/LAPIC Compatibility (Validation #6)
- RDTSC calibration via CPUID leaf 0x15 -- correct for Whiskey Lake
- Fallback to 3.4 GHz if leaf 0x15 fails -- reasonable for i7-8565U
- pal_time_now_ns() overflow math is correct (divides to MHz first)
- Timer syscalls (CREATE/SET/CANCEL) wired to kernel -- stubs return ENOSYS
- pal_thread_sleep_ns() uses busy-wait via RDTSC -- acceptable for early boot
- HPET/LAPIC timer not yet wired (kernel timer infrastructure needed)

### AES-GCM Software Path Assessment
- S-box is constant-time: full 256-entry linear scan with branchless select
- Uses volatile-style OR accumulator (compiler cannot optimize away scan)
- GHASH uses schoolbook GF(2^128) multiply -- correct but slow
- gf_mul() for MixColumns uses branch on bit test -- NOT constant-time
  - This is a MINOR side-channel: MixColumns operates on state bytes
    derived from S-box output + round key XOR. Exploitability is low
    because the attacker would need to correlate timing with internal
    state after S-box application. Not a P0 issue.
- Key material zeroed via xsec_secure_zero (volatile writes) -- correct
- Tag comparison uses xsec_ct_eq (constant-time) -- correct
- Decryption zeros plaintext on auth failure -- correct

### NVMe Driver Assessment
- BAR0 mapping via HHDM (phys + 0xFFFF888000000000) -- correct pattern
- 64-bit register access: low word first for writes (NVMe 1.4 sec 3.1.1) -- correct
- Memory fence before doorbell: __atomic_thread_fence SEQ_CST -- correct
- Phase bit tracking for CQ -- correct per NVMe 1.4 sec 4.6.2
- Queue allocation uses PAL_MEM_DMA | PAL_MEM_ZEROED -- correct
- PRP-based I/O limited to single 4KB pages -- documented limitation
- CID allocator wraps at 0xFFFF, skips 0 -- correct
