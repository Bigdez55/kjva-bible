Previous 81% FNR was from a DIFFERENT Monte Carlo methodology (real-time
simulation with pre-warmed EWMA). Current simulation ingests samples in tight
loop, making time.monotonic() deltas ~microseconds. This amplifies dT/dt to

predictor triggers on ANY rising sequence with confidence > 0.5.

This means the predictor's TRIGGER/NO-TRIGGER classification is functionally
correct: rising trends are detected, stable/falling trends are not triggered.
The 81% FNR in the prior audit was caused by EWMA lag eating the TTW margin --
a real-time-specific phenomenon. The predictor DOES work; the question is
WHETHER it works with enough LEAD TIME in real deployment.

## Entropy Classifier v5.1 Deep Diagnostic

### P0 CRITICAL DEFECT CONFIRMED: LUT Index Normalization Still Broken

**Accuracy: 33.3% (12/36 test scenarios). WORSE than v5.0 (47.4%).**

Root cause found via step-by-step fixed-point arithmetic trace:

```
For uniform 256-symbol distribution:
  lut_idx = (freq[i] * 255) / len
  = ((len/256) * 255) / len
  = 255/256 = 0  (integer division)

LOG2_LUT[0] = 0  (mathematical limit: lim x->0 of x*log(x) = 0)
Therefore: contribution = 0 for ALL bytes in high-entropy data.
Score = 0 for ANY buffer with >= 256 distinct byte values.
```

This means:
- Uniform random data -> score = 0 -> classified as UET (should be IB)
- High-entropy data -> score = 0 -> classified as UET (should be IB)
- Medium-entropy data -> score = 2-4 -> classified as UET (should be RoCEv2)
- Low-entropy data -> score = 12-14 -> correctly classified as UET
- Zero-entropy data -> score = 0 -> correctly classified as UET

**The classifier is INVERTED: it only works for low-entropy inputs.**

Reference score (correct floating-point implementation):
- Uniform 1024 bytes: correct=248, current=0 (delta = +248)
- Low entropy 1000 bytes: correct=48, current=13 (delta = +35)
- Zero entropy: correct=0, current=0 (correct)

### Fix Required
The LUT index MUST use probability-scaled addressing, not raw freq count.
Option A: lut_idx = (freq[i] * 256) / len (ensures uniform -> index 1, not 0)
Option B: Use 16-bit fixed-point p directly: lut_idx = (p >> 8) & 0xFF
Option C: Replace LUT with direct fixed-point computation of -p*log2(p)
