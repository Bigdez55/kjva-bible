## Findings

### P0 CRITICAL

#### D-XMIND-P0-01: token_emb stored as fp32, not Q4 -- 375 MB waste
- File: xmind.h line 223: `float *token_emb;`
- transformer.c line 351: logit projection uses fp32 dot product over token_emb
- 32000 * 3072 * 4 = 375 MB fp32 vs 58.6 MB at Q4
- 316 MB wasted. On 16GB device, this is 2% of total RAM.
- Risk: R2=0.92 SIGNAL. All other Llama implementations (llama.cpp, etc.) quantize the embedding.
- Fix: Store token_emb as xmind_q4_block_t*, use xmind_matmul_q4 for logit projection.

#### D-XMIND-P0-02: xm_heap_free is a no-op -- memory leak on every shutdown
- File: xmind.h lines 109-114: `static inline void xm_heap_free(void *ptr) { (void)ptr; }`
- PAL handle (ph, mh) not tracked per allocation (lines 100-106)
- Every xmind_alloc_state() + xmind_shutdown() cycle leaks KV cache + scratch
- KV cache alone: 336 MB leaked per init/shutdown cycle
- Sprint 8 load test (20 cycles, test_sprint8.c line 492): leaks 6,720 MB
- Risk: R2=0.95 SIGNAL. This will OOM the device after 4-5 init cycles.

### P1 HIGH

#### D-XMIND-P1-01: PRNG seed is deterministic constant
- File: sampler.c line 186: `0xDEADBEEFCAFEBABEull`
- xmind_sampler_init with seed=0 falls back to this constant
- xorshift64 is non-cryptographic (stated in comments, line 30)
- But deterministic seed = identical output sequences across sessions
- For any security-adjacent use (code generation, etc.), this is exploitable
- Risk: R2=0.78 SIGNAL. The seed should derive from PAL entropy source.

#### D-XMIND-P1-02: No bounds check on KV cache overflow at ctx=2048
- File: transformer.c line 215: `m->kv.n_cached = pos + 1u;`
- xmind_forward line 299: `XM_ASSERT(pos < m->cfg.ctx_len)` -- halts the system
- No graceful degradation. When pos hits 2048, the CPU halts forever (HLT loop).
- No ring-buffer, no eviction, no sliding window.
- Risk: R2=0.88 SIGNAL. Generation beyond 2048 tokens = system hang.

#### D-XMIND-P1-03: FFN stack allocation = 64 KB per call
- File: transformer.c lines 261-262: `float gate[8192]; float up[8192];`
- XM_FFN_SCRATCH = 32 * 256 = 8192 floats = 32 KB each, 64 KB total
- This is on the stack, not heap. x86-64 default stack = 8 MB
- For Llama 3.2 3B (ffn_dim=8192): exactly at the limit.
- Not a bug (8192 <= 8192), but fragile. Any future model with larger FFN = ASSERT crash.
- Stack depth: xmind_forward -> xm_ffn -> 64KB. No recursive calls, so safe for now.
- Risk: R2=0.71 SIGNAL. BORDERLINE for this exact model. Fails for larger models.

#### D-XMIND-P1-04: Duplicate xmind_tokenize definitions in tokenizer.c
- File: tokenizer.c has TWO external definitions of xmind_tokenize:
  - Section 6 (line 138): first definition
  - Section 14 (line 528): xmind_tokenize_bpe, but dispatch via xmind_tokenize_dispatch is `unused`
- The BPE dispatch wrapper (line 570) is marked `__attribute__((unused))` and NEVER called
- The actual public xmind_tokenize() always uses byte-level mode
- BPE path is dead code in all builds (even with XMIND_HAS_XSTORE)
- Risk: R2=0.82 SIGNAL. BPE tokenization is non-functional despite Sprint 7 implementation.

#### D-XMIND-P1-05: weights_loader.c allocates incorrect weight sizes
- File: weights_loader.c lines 500-502:
  ```
  w_size = hidden_dim * (hidden_dim / XMIND_Q4_BLOCK) * sizeof(xmind_q4_block_t)
  ```
  This computes 3072 * 96 * 20 = 5,898,240 bytes for ALL 7 weight matrices.
  But wk, wv have rows=768 (not 3072), and w1, w2, w3 involve ffn_dim=8192.
  All 7 pointers (wq through w3) get identically-sized buffers.
- wk/wv: allocated 5.6 MB but need 1.4 MB (overallocation: 4.2 MB waste per matrix per layer)
- w1/w3: allocated 5.6 MB but need 15.7 MB (underallocation: will corrupt on write)
- w2: allocated 5.6 MB but need 15.7 MB (underallocation)
- Sprint 8 note says "zero weights" so this doesn't crash yet, but Sprint 9 weight loading WILL.
- Risk: R2=0.91 SIGNAL. Sprint 9 will hit buffer overflow writing w1/w2/w3 weight data.

### P2 MEDIUM

#### D-XMIND-P2-01: No SIMD -- all math is scalar x87
- File: tensor.c line 7: "No SIMD (x87 FPU only)"
- xmind_matmul_q4 inner loop (lines 173-181): 3 branches, 4 integer ops, 2 FP multiplies, 2 FP adds per nibble pair
- No SSE, no AVX. Compile with -msse4.2 or -mavx2 would require intrinsics.
- Scalar throughput: 0.25-0.41 tok/s (unusable for interactive)
- Risk: R2=0.88 SIGNAL. Without SIMD, XMIND is 10-30x slower than llama.cpp on same hardware.

#### D-XMIND-P2-02: Attention score inline in stack buffer
- File: transformer.c line 119: `float scores[XM_ATTN_SCRATCH]` = 2048 * 4 = 8 KB per head
- Called in a loop over 32 heads, but each iteration overwrites (not cumulative). OK.
- However: the outer head loop (line 220) calls xm_attention_head serially.
- No multi-head parallelism. Each head is computed sequentially.
- Risk: R2=0.65 SIGNAL. Parallel head computation would improve throughput ~2-3x.

#### D-XMIND-P2-03: Q4_0 dequant in matmul inner loop (no block caching)
- File: tensor.c lines 173-181: dequant is fused with dot product
- The same block may be dequantized multiple times if the same row is accessed
  by multiple heads (not the case for matmul, but could be for future batching)
- The nibble extraction (mask + shift + subtract 8) is 3 ops per weight,
  repeated for every matmul. Caching dequantized blocks would trade memory for compute.
- Risk: R2=0.58 SIGNAL.

#### D-XMIND-P2-04: Softmax uses custom xm_expf with 1.5e-6 relative error
- File: tensor.c lines 62-86: degree-5 minimax polynomial
- Max relative error stated as 1.5e-6 -- adequate for softmax normalization.
- But: clamping to [-87, 87] means very negative logits become exactly 0.0.
  This is actually correct behavior (exp(-87) < 1e-38 ~ denormal).
- Risk: R2=0.82 SIGNAL. Functionally correct, but precision slightly lower than libm.

### P3 LOW

#### D-XMIND-P3-01: BPE rule search is O(n) linear scan
- File: tokenizer.c line 326: `for (i = 0u; i < s_n_bpe_rules; i++)`
- With 8192 rules, merge pass is O(tokens * rules) = O(n * 8192)
- Iterative merge: up to 8192 passes, each O(n * 8192) = O(n * 67M) worst case
- Fix: sort rules by (left, right) pair and binary search, or hash table.

#### D-XMIND-P3-02: Newton-Raphson sqrtf only 3 iterations
- File: tensor.c lines 95-109
- Quake fast-inverse-sqrt seed + 3 NR iterations = ~6 ULP accuracy
- libm sqrtf is 0.5 ULP. Difference is ~0.0001% in practice.
- For RMSNorm denominator, 6 ULP is sufficient.

#### D-XMIND-P3-03: GGUF array metadata skipped
- File: weights_loader.c line 311: `pos += 12u; /* minimal skip */`
- Comment acknowledges this may desync the scanner.
- If GGUF file has array metadata before key config fields, values will be wrong.
- Fallback to preset mitigates, but GGUF conformance is incomplete.

## XJIT Acceleration Analysis

### XJIT Opcode Set (23 opcodes)
NOP, MOV_RI, MOV_RR, ADD, SUB, IMUL, AND, OR, XOR, SHL, SHR,
CMP, JMP, JEQ, JNE, JLT, JGT, CALL, RET, LOAD, STORE, LABEL, PHI

### XMIND Operation Requirements
- matmul_q4: FP32 multiply-accumulate, nibble extract, float conversion
- rmsnorm: FP32 square, sum, reciprocal sqrt, scale
- softmax: FP32 exp, max, sum, divide
- silu: FP32 exp, divide, multiply
- rope: FP32 cos/sin lookup, multiply, add/subtract
- dot: FP32 multiply-accumulate

### XJIT Cannot Accelerate XMIND: R2=0.98 SIGNAL
- XJIT has ZERO floating-point opcodes. All 23 opcodes operate on uint64_t GP registers.
- No FADD, FMUL, FDIV, FSQRT. No SSE/AVX XMM register support.
- No FP32 load/store with conversion. LOAD/STORE are uint64_t only.
- IMUL is integer multiply (x86 IMUL r/m64), not floating-point.
- XJIT could theoretically JIT the integer nibble extraction loop (4-bit unpack),
  but this is ~5% of matmul compute time. The FP multiply-accumulate is 95%.
- XJIT leverage ratio: 0% of XMIND FLOPs can be accelerated by XJIT.
- Verdict: XJIT and XMIND are architecturally disjoint. No causal acceleration edge exists.

## Counterfactual: SSE4.2/AVX2 Speedup

### P(Y | do(X = SSE4.2 enabled))
- matmul_q4 inner loop is 4-wide vectorizable (4 floats per SSE register)
- Nibble extract NOT vectorizable (bit manipulation, branching)
- Vectorizable fraction of matmul: ~60-65% (FMA portion)
- Overall speedup: 2.5-3.5x
- Expected tok/s: 0.6-1.4 (still sub-interactive)

### P(Y | do(X = AVX2 enabled))
- 8-wide fp32 FMA (vfmadd231ps)
- vpshufb can vectorize nibble extraction (4-bit unpack via shuffle)
- Vectorizable fraction: ~80-85%
- Overall speedup: 4-6x
- Expected tok/s: 1.0-2.5 (marginally interactive for short outputs)

### P(Y | do(X = AVX-512, Ice Lake))
- 16-wide fp32 FMA
- Full nibble vectorization with vpternlogd + vpshufb
- Vectorizable fraction: ~90%
- Overall speedup: 6-10x
- Expected tok/s: 1.5-4.1
- 4-core: 4.8-13.1 tok/s (interactive for decode)

## XSTORE Integration
- tokenizer.c line 360: `#ifdef XMIND_HAS_XSTORE` guards BPE loading
- But: the BPE dispatch is dead code (P1-04 above)
- Even when XMIND_HAS_XSTORE is defined, xmind_tokenize() uses byte-level mode
- The XSTORE BPE dictionary is loaded but never consumed by the public tokenize API
- Verdict: XSTORE integration is structurally wired but functionally broken.

## Summary

### Performance Model (Llama 3.2 3B Q4_0 on HP EliteBook x360 G4)
- **Tokens/sec (current, scalar):** 0.25-0.41 tok/s
- **Tokens/sec (with AVX2 + 4-core):** 3.2-8.0 tok/s
- **Tokens/sec (with AVX-512 + 4-core, Ice Lake):** 4.8-13.1 tok/s

- **Causal bottleneck:** COMPUTE (scalar throughput << memory bandwidth ceiling)
- **XJIT leverage:** 0% (integer-only ISA, zero FP opcodes)

- P0: 2 (fp32 embedding waste, heap_free no-op leak)
