## NEW P0 FINDING: SHA-256/SHA-512 Length Field Corruption (D-S3-P0-SHA)

sha256_final (line 169) calls xsec_sha256_update(ctx, pad, pad_len) which
modifies ctx->bit_count (line 118: += pad_len*8). Then line 172 reads the
CORRUPTED bit_count to write the length field.

For "abc" (3 bytes): bit_count should be 24 (0x18). After padding update,
bit_count = 448 (0x1C0). Wrong length field -> wrong hash for EVERY input.

Same pattern in sha512_final (line 431 corrupts bit_count, lines 434-435 read it).

Selftest has NEVER EXECUTED (compile-only, no runtime). Bug latent.

### Contagion
sha256 -> HMAC -> HKDF -> TLS 1.3 key schedule -> ALL TLS sessions broken
sha512 -> Ed25519 -> TLS CertVerify, XPKG verify -> ALL signatures broken

### Fix
Save bit_count BEFORE the padding update:
```
uint64_t saved_bits = ctx->bit_count;  // ADD THIS LINE
xsec_sha256_update(ctx, pad, pad_len);
xsec_put_be64(pad, saved_bits);        // USE SAVED VALUE
```

## Sprint 4 Risk Model

Top 3 categories for XSTORE (B+ tree, WAL, MVCC):

1. **Integer overflow in page arithmetic** (R2=0.85 from ed25519 pattern)
   B+ tree page splits involve page_count * page_size arithmetic.
   WAL segment offsets can overflow uint32_t at >4GB WAL.

2. **State scope errors in transaction rollback** (R2=0.78 from chacha20 pattern)
   MVCC version counters declared in tight scopes but needed post-loop.
   WAL replay loops with checkpoint state that must persist across iterations.

3. **Missing validation on WAL record lengths** (R2=0.88 from tls13 pattern)
   WAL records contain length fields read from disk. If not bounds-checked
   before memcpy, buffer overflows on corrupted WAL.
