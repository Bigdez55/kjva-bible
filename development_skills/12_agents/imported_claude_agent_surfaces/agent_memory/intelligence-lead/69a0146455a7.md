## AMENDMENT 2026-03-06 — Deep Mathematical Verification

### A. ed25519.c P1-A (scalar loop) — PREVIOUS FINDING WAS FALSE POSITIVE
Previously flagged: `for (i = 255u; i <= 255u; --i)` with uint32_t i never iterates.
CORRECTION: With uint32_t i, unsigned wraparound after i=0 gives 0xFFFFFFFF > 255,
so the condition i <= 255u becomes false. The loop iterates i = 255, 254, ..., 1, 0.
That is exactly 256 iterations, processing all 256 bits of the scalar. CORRECT.
This is a valid unconventional countdown. No fix required.

### B. ed25519.c P1-C (fe10_mul int64->int32 truncation) — CONFIRMED CORRECT
The carry sequence at lines 109-120 of fe10_mul fully reduces all 10 int64_t
intermediates before the final int32_t casts at lines 122-125. After carry propagation,
each h[i] is within the 25- or 26-bit capacity of its slot, fitting in int32_t with margin.
The carry sequence (12 steps) matches NaCl/SUPERCOP fe25519_mul exactly. CORRECT.

### C. x25519.c P1-E (TWICE_P[0] bias) — PREVIOUS FINDING WAS FALSE POSITIVE
Previously flagged as "OFF BY 60". CORRECTION:
  MASK51 = 2^51 - 1.
  TWICE_P[0] = (MASK51 - 18u) * 2u = (2^51 - 19) * 2 = 2^52 - 38. This IS 2p[0].
  fe_sub adds TWICE_P * 2 = 4p before subtraction.
  4p[0] = (2^52 - 38) * 2 = 2^53 - 76 = 4*(2^51 - 19). EXACTLY 4p[0]. CORRECT.
  The previous audit miscomputed by comparing against an incorrect reference value.
  Reclassified from P1 to CLOSED (not a bug).

### D. chacha20.c P2-B (poly1305 final reduction) — CONFIRMED CORRECT (2026-03-06)
Mathematical proof:
  Final reduction computes acc + 5 across all 3 limbs.
  t5  = acc_lo + 5  (carry into bit 64 if needed)
  t5h = acc_hi + carry_from_t5  (carry into bit 128 if needed)
  sum_top = acc_top + carry_from_t5h  (bits 128-130 of acc+5)
  do_reduce = sum_top >> 2: non-zero iff bit 130 of acc+5 is set, i.e., acc+5 >= 2^130,
  i.e., acc >= p = 2^130-5. Correct constant-time reduction test.
  When do_reduce is true: lower 128 bits of (acc+5) used. CORRECT.
  When do_reduce is false: acc < p, lower 128 bits of acc used. acc_top drops to 0 mod 2^128.
  Reclassified from P2-BUG to CONFIRMED CORRECT.

### E. chacha20.c P2-C (poly_scratch static buffer) — FIXED (2026-03-06)
poly_scratch changed to stack-local in the production version. Thread-safety issue resolved.

## P2 Findings (correctness risk or performance concern, pre-shipping)

### P2-A: x25519.c — fe_pack does not perform final conditional subtraction
After two carry-propagation passes, canonical reduction of the output to [0, p) is
not guaranteed. RFC 7748 does not mandate canonical form for X25519 output, but
downstream signature and comparison code may depend on it.

### P2-D: gossip.c — incarnation number serialisation truncates uint64 to uint32
File: net/xnet/mesh/gossip.c, line 234.
hdr->incarnation = xnet_htonl((uint32_t)g_self.incarnation);
After ~4 billion incarnation bumps, high 32 bits lost. Node refuted many times could
be incorrectly declared dead after wraparound. Fix: use xnet_htonll/ntohll.

### P2-E: gossip.c — SUSPECT-to-DEAD timeout
Uses same constant for both ping-failure window and suspicion period.
SWIM recommends suspicion period proportional to log(N). Review parameterisation.

## Deep Verification Results — 2026-03-06

### 1. Ed25519 fe10 Field Arithmetic — MATHEMATICALLY CORRECT

Representation verified:
- 10 limbs, int32_t. Even limbs (0,2,4,6,8): 26-bit. Odd limbs (1,3,5,7,9): 25-bit.
- Bit positions: 0, 26, 51, 77, 102, 128, 153, 179, 204, 230. Total 255 bits. CORRECT.

fe10_mul product decomposition:
- Cross-limb terms with i+j >= 10 apply factor 19 (from 2^255 ≡ 19 mod p). CORRECT.
- Doubling factors (f1_2, f3_2, etc.) handle symmetric cross-terms in schoolbook mul.
- All 100 products f[i]*g[j] accounted for across h[0]..h[9]. CORRECT.
- Matches NaCl/SUPERCOP fe25519_mul reference exactly.

Carry sequence (12 steps, lines 109-120):
- Alternates between even-limb (26-bit, >>26) and odd-limb (25-bit, >>25) reductions.
- Two-sided pass: 0-4 first, then 0-4 second time after limb-9 wrap.
- Wrap: c9 = carry from limb 9; h[0] += c9 * 19 (reduction constant). CORRECT.
- Final step 12 propagates wrap carry into limb 1. CORRECT.

Curve constants ED25519_D, ED25519_D2, ED25519_SQRTM1 match NaCl/SUPERCOP reference values.

### 2. X25519 TWICE_P — MATHEMATICALLY CORRECT

MASK51 = 2^51 - 1 = 0x0007ffffffffffff.

p in 5-limb 51-bit form:
  p[0] = 2^51 - 19 = MASK51 - 18
  p[1..4] = 2^51 - 1 = MASK51

TWICE_P values:
  TWICE_P[0] = (MASK51 - 18u) * 2u = 2*(2^51-19) = 2^52 - 38. EXACTLY 2p[0]. CORRECT.
  TWICE_P[1..4] = MASK51 * 2u = 2*(2^51-1) = 2^52 - 2. EXACTLY 2p[1..4]. CORRECT.

fe_sub adds 4p = 2*TWICE_P before subtracting:
  4p[0] = 2*(2^52-38) = 2^53-76 = 4*(2^51-19). CORRECT. No underflow possible.

### 3. Poly1305 Final Reduction — MATHEMATICALLY CORRECT

Accumulator: acc_lo (bits 0-63), acc_hi (bits 64-127), acc_top (bits 128-129).
After each block's partial carry: acc_top <= 3 (2 bits). VERIFIED.

Final reduction:
  t5 = acc_lo + 5. t5h = acc_hi + carry(t5). sum_top = acc_top + carry(t5h).
  do_reduce = sum_top >> 2 != 0  iff  bit-130 of (acc+5) is set  iff  acc >= p.
  Constant-time: no branch on secret accumulator value (branch on do_reduce only
  after it is fully computed, which is a derived comparison result, not raw acc).
  When true: tag = lower 128 bits of (acc+5). CORRECT.
  When false: tag = lower 128 bits of acc (acc_top lost, but acc_top*2^128 mod 2^128=0).

Block accumulation: block[block_len]=0x01 sets the "hibit" per RFC 8439 §2.5.1. CORRECT.
Multiplication (acc*r) via 3x2 schoolbook with 2^130 ≡ 5 partial reduction. CORRECT.

### 4. AES-256-GCM — MATHEMATICALLY CORRECT

Key schedule (FIPS 197 §5.2):
  Nk=8, Nr=14, 60 words. i%8==0 path: SubWord(RotWord(W[i-1])) XOR Rcon[i/8].
  i%8==4 path: SubWord(W[i-1]) (AES-256 extra SubWord). CORRECT.
  RCON index range: i=8..59, i%8==0 at i=8,16,24,32,40,48,56 -> Rcon[1..7]. In-bounds. CORRECT.

MixColumns: uses GF(2^8) irreducible polynomial 0x1b (x^8+x^4+x^3+x+1). CORRECT.
MixColumns matrix matches FIPS 197 §5.1.3 exactly.

ShiftRows: row 0 unchanged, row 1 left-1, row 2 left-2, row 3 left-3 (=right-1). CORRECT.

GHASH reduction polynomial:
  R = 0xe1 || 0...0. 0xe1 = x^7+x^6+x^5+x^0 in GCM bit-reflected representation.
  Implements x^128 + x^7 + x^2 + x + 1 per NIST SP 800-38D. CORRECT.

GCM construction (NIST SP 800-38D §7.1 case len(IV)=96):
  H = AES-K(0^128). J0 = nonce||0x00000001. E(K,J0) computed before increment.
  ICB = gcm_inc32(J0) = nonce||0x00000002. GCTR starts at ICB. CORRECT.
  GHASH input: PAD(AAD) || PAD(CT) || LE64(|AAD|) || LE64(|CT|).
  Wait — code uses xsec_put_be64 for lengths. NIST SP 800-38D §7.1 specifies BE64. CORRECT.
  tag = GHASH(H, ...) XOR E(K,J0). CORRECT.
