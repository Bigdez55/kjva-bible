## CORRECTNESS VERDICTS (per component)

### 1. Transformer (transformer.c)

CORRECT:
- Self-attention: Q·Kᵀ / sqrt(head_dim) — scale computed via Quake fast-inverse-sqrt
  (one Newton-Raphson step). Sufficient accuracy for head_dim=96.
- GQA: kv_head = h * n_kv_heads / n_heads (integer div). With kv_heads=8, heads=32 → factor 4. CONFIRMED.
- Softmax: numerically stable (max-subtraction). Inline exp in attention loop matches xm_expf.
- RoPE: applied to Q (all n_heads) and K (n_kv_heads only). Pair-wise rotation on pre-computed tables. CONFIRMED.
- RMSNorm: pre-norm applied to both attention and FFN sub-layers. CONFIRMED.
- SwiGLU: gate=w1*xb, up=w3*xb, hidden=SiLU(gate)*up, out=w2*hidden. CONFIRMED.
- Residual connections: x += xb after both attention and FFN. CONFIRMED.
- Weight tying: logits computed as token_emb^T × x. CONFIRMED.

ISSUES:
- P0: Weight pointers (wq,wk,wv,wo,w1,w2,w3) are ZERO-initialized stub buffers in Sprint 8.
  xmind_forward() will produce all-zero logits. Acknowledged in code comments.
- P1: FFN stack arrays gate[XM_FFN_SCRATCH] and up[XM_FFN_SCRATCH] where
  XM_FFN_SCRATCH = XMIND_MAX_HEADS * XM_MAX_HEAD_DIM = 32 * 256 = 8192 floats = 32KB each.
  Llama 3.2 3B ffn_dim = 8192 exactly hits this cap — ZERO MARGIN.
  If ffn_dim ever exceeds 8192, XM_ASSERT fires (correct guard) but also means two
  32KB arrays on the stack simultaneously = 64KB stack usage per xm_ffn() call.
  Combined with attention stack (XM_ATTN_SCRATCH = 2048 floats = 8KB) = ~72KB total
  per-frame stack during a forward pass. Must verify XENOS thread stack size >= 128KB.
- P1: head_dim=96 for Llama 3.2 3B. XM_MAX_HEAD_DIM=256. head_out[XM_MAX_HEAD_DIM]
  allocates 256 floats (1KB) but only uses 96. Wasteful but not incorrect.
- P2: Inline exp in xm_attention_head (§3) duplicates xm_expf body — divergence risk
  if xm_expf is ever updated.

### 2. Tensor / Math Primitives (tensor.c)

CORRECT:
- xm_expf: 2^(x*log2e) bit trick + degree-5 polynomial on [0,1). Max error <1.5e-6. CONFIRMED.
- xm_sqrtf: Quake seed + 3 Newton-Raphson iterations. ~6 ULP accuracy. Sufficient.
- xmind_rmsnorm: rms = sqrt(mean(x^2)+1e-5), out[i] = weight[i]*(x[i]/rms). Matches LLaMA reference.
- xmind_matmul_q4: correct Q4_0 dequant in inner loop: w=(nibble-8)*scale. Per-block scales. CONFIRMED.
- xmind_softmax: max-subtraction stable variant. CONFIRMED.
- xmind_silu: out[i] = gate[i]*sigmoid(gate[i])*up[i]. Correct SwiGLU. CONFIRMED.
- xmind_rope: pair-wise rotation, GQA-aware (n_kv_heads for K). CONFIRMED.

ISSUES:
- P2: xmind_matmul_q4 is pure scalar (no SIMD). For Llama 3.2 3B, the Q×K matmul
  (32 heads × 96 dim) and FFN projection (hidden=3072, ffn=8192) will be the
  performance bottleneck. Estimated 8-12 tok/s on i7 with scalar float.
- P2: No AVX2/SSE4 path. On EliteBook x360 (Intel i7-8th gen), AVX2 would yield
  ~4-6x speedup on matmul. Currently unimplemented.

### 3. Quantization (quantize.c)

CORRECT:
- Q4_0 format: block=32 weights, scale=max_abs/7.0f, nibble=q+8.
- Dequant formula: w = (nibble-8)*scale. CONFIRMED correct.
- Per-block scales: each xmind_q4_block_t has its own float scale. This is
  per-row when blocks are laid out row-major (cols/32 blocks per row). CONFIRMED.
- Round-trip self-test included (xmind_verify_q4_roundtrip). Correct bound check.

ISSUES:
- P2: Q4_0 is a symmetric scheme (no zero_point). GGUF Q4_K_M (used by
  llama3.2:3b-instruct-q4_K_M in production Ollama) uses asymmetric quantization
  with separate min values. XMIND implements Q4_0 only. If loading Q4_K_M files,
  weight dequantization will be WRONG — blocks parsed as Q4_0 will misread
  the k-quant block layout (which has different header bytes).
  This is a critical compatibility gap for the target Ollama model.

### 4. Tokenizer (tokenizer.c)

ISSUES — CRITICAL:
- P0: Llama 3.2 uses SentencePiece BPE with vocab_size=128,256. XMIND byte-level
  fallback is vocab_size=259 (3 special + 256 byte tokens). The model expects
  token IDs in [0,127999] range. Running inference with byte-level tokenizer
  produces COMPLETELY WRONG token distributions. Outputs will be incoherent.
- P0: XMIND_VOCAB_SIZE constant = 32,000 (header line 35). Llama 3.2 3B actual
  vocab = 128,256. This constant caps xmind_init() validation and will REJECT
  the correct vocab_size at initialization time (128256 > 32000 → XMIND_ERR_INVAL).
- P1: xmind_tokenize_dispatch() is declared `static __attribute__((unused))`.
  It is NEVER called externally. The public symbol xmind_tokenize (§6) is always
  the externally-linked function regardless of s_bpe_loaded flag.
  BPE merging is therefore dead code even when XMIND_HAS_XSTORE is defined.
- P2: bpe_find_rule() does O(n_rules) linear scan per adjacent pair per pass.
  For n_rules=8192 and a 512-token sequence, worst-case = 8192*512*8192 ops.
  Must use a hash map for production BPE.
- P2: XMIND_BPE_MAX_RULES=8192. Llama 3.2 uses ~100,000 merge rules. Even if
  BPE were wired correctly, the table is 12x too small.

### 5. Sampler (sampler.c)

CORRECT:
- Temperature greedy (temp=0) → argmax. CONFIRMED.
- Temperature scaling before softmax. CONFIRMED.
- xorshift64 PRNG: period 2^64-1. Non-crypto. Sampling only. CONFIRMED safe.
- xm_rand_float: 23-bit mantissa trick for uniform [0,1). Correct.
- Seed=0 guard: defaults to 0xDEADBEEFCAFEBABEull. CONFIRMED.

ISSUES:
- P2: top-p approximation uses threshold=(1-p)*max_prob instead of true
  cumulative CDF sort. This is a conservative nucleus that includes MORE
  tokens than true top-p (never fewer). At p=0.9, may include tokens
  outside the true nucleus if probability mass is concentrated at one token.
  For p=1.0: threshold=(1-1.0)*max_prob=0 → all tokens with prob>0 qualify → correct.
- P2: No top-k implementation. Common complement to top-p in production samplers.

### 6. Weights Loader (weights_loader.c)

ISSUES — CRITICAL:
- P0: Sprint 8 explicitly does NOT parse tensor data offsets or read actual
  weight values. All weight pointers are stub zero-page allocations.
  xmind_forward() produces zero logits. Acknowledged in comments.
  Sprint 9 dependency unresolved.
- P0: xm_heap_alloc() / wl_alloc_pages(): ph and mh PAL handles are suppressed
  with `(void)` and discarded. No per-pointer tracking. xmind_weights_unload()
  NULLs the pointers but cannot call pal_pages_free(ph) or pal_unmap_pages(mh)
  because the handles are gone. UNRECOVERABLE MEMORY LEAK on every model load/unload.
- P1: GGUF ARRAY type skip (+12 bytes) in gguf_scan_metadata §5 is incorrect.
  GGUF ARRAY = uint32_t elem_type (4B) + uint64_t count (8B) + count*elem_size data.
  The code skips only 12 bytes (type+count header) but leaves the array data
  unread, desyncing the parse position for all subsequent KV pairs.
  Any GGUF file with an ARRAY-typed metadata field before the numeric fields
  (e.g. tokenizer.ggml.tokens is ARRAY type) will produce wrong or zero config.
- P1: No SHA-256 verification of weights despite being the standard GGUF security check.
- P2: s_scan_buf[64KB] is a static global — correct (not stack). No overflow risk. CONFIRMED.
- P2: GGUF header n_kv read at offset 16 (bytes 16-23). Correct: magic(4)+ver(4)+n_tensors(8)=16. CONFIRMED.
- P2: GGUF magic mismatch → XMIND_ERR_CORRUPT immediately. CONFIRMED.
- P2: GGUF version >3 → XMIND_ERR_CORRUPT immediately. CONFIRMED.

### 7. Main API (xmind.c)

CORRECT:
- xmind_init(): validates all dimensions, checks n_heads*head_dim==hidden_dim. CONFIRMED.
- xmind_alloc_state(): allocates all scratch + KV cache buffers with OOM rollback. CONFIRMED.
- xmind_shutdown(): correctly frees state buffers (not weight pointers). CONFIRMED.
- Global singleton pattern. CONFIRMED.

ISSUES:
- P0: xm_heap_free() is a no-op stub (xmind.h §3). All state buffers allocated by
  xmind_alloc_state() leak on xmind_shutdown(). The comment acknowledges this as
  "Sprint 6 minor leak" but it is actually a full inference-state leak (~200MB for
  Llama 3.2 3B KV cache at 2048 ctx) on every shutdown/reinit cycle.
- P2: attn buffer allocated as n_heads × ctx_len floats (line 160).
  For Llama 3.2 3B: 32*2048*4 = 256KB. But attn is used as n_heads × head_dim
  output assembly buffer (line 225 in transformer.c). These are different sizes.
  Buffer is over-allocated (ctx_len >> head_dim) which wastes memory but is safe.

## INTEGRATION GAP ANALYSIS

### XMIND ↔ XBLOB
- Conditional: -DXMIND_HAS_XBLOB enables xmind_weights_load_blob().
- Sprint 8 path: retrieves blob, validates GGUF magic, then falls back to
  preset config WITHOUT parsing tensor offsets. Weight data is NOT loaded.
- Status: Partial integration (config path works; weight loading does not).

### XMIND ↔ PAL
- xm_heap_alloc/free are backed by pal_pages_alloc + pal_map_pages. CONFIRMED.
- Handle leak is the critical gap (see P0 above).
- No SIMD abstraction layer used — intentional (PAL does not expose SIMD in Sprint 3).

## CAPABILITY ASSESSMENT

### Can XMIND run Llama 3.2 3B inference?
Answer: NO (Sprint 8 state). Partial — architecture is correct, weights are zeros.

Blocking issues in priority order:
1. XMIND_VOCAB_SIZE=32000 rejects Llama 3.2's 128256 vocab at init time (hard block)
2. Weight loader does not parse tensor data — all weights are zero (hard block)
3. Q4_0 vs Q4_K_M format mismatch — wrong dequant for the target model file
4. Tokenizer vocab mismatch — byte-level produces wrong token IDs
5. Memory handle leak — prevents reliable model reload cycles

### Token/second estimate (Intel i7-8th gen, INT4, scalar):
- Attention per token: 32 heads × 96 dim × 2048 ctx = ~6.3M float ops
- FFN per token: 3072×8192 + 8192×3072 = ~50M float ops
- Logit projection: 32000×3072 = ~98M float ops (bottleneck)
- Per-layer total: ~57M ops × 28 layers = ~1.6B ops/token
- Scalar float throughput on i7-8th gen: ~8-12 GFLOP/s (no SIMD)
- Estimated: ~5-7 tok/s scalar | ~20-28 tok/s with AVX2 (unimplemented)
- Current (zero weights, no actual compute): not measurable

## FINDING REGISTRY

IL-01: P0 — Weight pointers are NULL/zero after weights_loader Sprint 8.
  xmind_forward() produces zero logits. Sprint 9 must implement tensor offset parsing.

IL-02: P0 — xm_heap_free() is a no-op; PAL handles ph/mh discarded on alloc.
  Unrecoverable memory leak on every xmind_shutdown() call (~200MB for 3B model).

IL-03: P0 — XMIND_VOCAB_SIZE=32000 < Llama 3.2's 128256.
  xmind_init() will REJECT the correct config with XMIND_ERR_INVAL.
  Must increase to 131072 or higher.

IL-04: P0 — Tokenizer is byte-level (vocab=259) in runtime default.
  Llama 3.2 requires SentencePiece BPE with 128256 tokens.
  Byte-level token IDs produce completely incoherent model output.

IL-05: P1 — Q4_0 dequant incompatible with Q4_K_M (target Ollama model format).
  Q4_K_M uses asymmetric k-quant blocks with different byte layout.
  Loading a Q4_K_M GGUF with XMIND's Q4_0 parser = wrong weights.

IL-06: P1 — GGUF ARRAY metadata skip (+12 bytes) desyncs parse position.
  tokenizer.ggml.tokens is ARRAY type and appears before numeric fields.
  Config scan will produce wrong/zero values on real Llama GGUF files.

IL-07: P1 — FFN stack: gate[8192] + up[8192] = 64KB on stack simultaneously.
  At Llama 3.2 3B ffn_dim=8192 this is exactly at cap (XM_FFN_SCRATCH=8192).
  Zero margin. Thread stack must be >= 128KB.

IL-08: P1 — BPE dispatch dead code: xmind_tokenize_dispatch() is static+unused.
  Public xmind_tokenize is always the byte-level §6 function regardless of
  s_bpe_loaded. BPE merging never activates even with XMIND_HAS_XSTORE.

IL-09: P2 — No SIMD (AVX2/SSE4) in matmul. ~4-6x performance gap vs potential.
  Estimated: 5-7 tok/s scalar vs 20-28 tok/s with AVX2.

IL-10: P2 — top-p uses threshold=(1-p)*max_prob approximation (not true CDF sort).
  Conservative nucleus — includes more tokens than true top-p.

IL-11: P2 — XMIND_BPE_MAX_RULES=8192. Llama 3.2 has ~100,000 merge rules.
  Table is 12x too small for production BPE even if wiring were correct.

IL-12: P2 — No SHA-256 verification of loaded weight files.
  GGUF integrity check absent despite xsec/xsec having sha256 implementation.

IL-13: P2 — Logit projection (weight tying) uses fp32 token_emb.
  For vocab=32000: 32000×3072 = ~98M multiplies per token in fp32.
  This is the single largest per-token compute step. Should be quantized.

IL-14: P2 — attn scratch buffer over-allocated as n_heads×ctx_len instead of
  n_heads×head_dim. For 3B model: 32*2048*4=256KB vs needed 32*96*4=12KB.
  Wasteful but not incorrect.
