## FINDING REGISTER

### IL-01: P1 — state.attn Buffer Semantically Misnamed and Oversized
File: xmind.c line 160 + xmind.h (xmind_state_t comment)
Allocation: n_heads × ctx_len floats (32 × 2048 = 65,536 floats = 262,144 bytes)
Usage: n_heads × head_dim floats (32 × 96 = 3,072 floats = 12,288 bytes)
Attention scores (per-head) are stack-allocated in xm_attention_head(), not in state.attn.
state.attn is actually the per-head output assembly buffer.
Impact: 250 KB wasted per model. No correctness defect (oversized buffer is safe).
Fix: Rename to attn_out; reallocate as n_heads × head_dim.

### IL-02: P0 — RoPE Tables Zero-Initialized, Never Populated
Files: weights_loader.c lines 525-527; tensor.c §9
rope_cos and rope_sin are allocated via wl_alloc_pages() with PAL_MEM_ZEROED.
No xmind_rope_init_tables() function exists.
Effect: All cos values = 0.0, all sin values = 0.0.
RoPE rotation: q'[2i] = q[2i]*0 - q[2i+1]*0 = 0 for every token position.
Every Q/K vector is zeroed by RoPE. No positional information injected.
Fix: Implement xmind_rope_init_tables() with Llama 3.2 rope_theta = 500,000:
  theta_i = 1 / (500000.0 ^ (2i / head_dim))
  cos_table[pos * half + i] = cosf(pos * theta_i)
  sin_table[pos * half + i] = sinf(pos * theta_i)

### IL-03: P1 — FFN Stack Arrays at Exact XMIND_MAX Limit
File: transformer.c xm_ffn() lines 261-262
float gate[XM_FFN_SCRATCH] and float up[XM_FFN_SCRATCH]
XM_FFN_SCRATCH = XMIND_MAX_HEADS × XM_MAX_HEAD_DIM = 32 × 256 = 8192 floats
Llama 3.2 3B ffn_dim = 8192. Zero margin.
Two arrays × 8192 × 4 bytes = 65,536 bytes stack per call.
Risk: Stack overflow in freestanding kernel (typical stack 8-16KB).
Fix: Move gate[] and up[] to static globals or a per-model scratch allocation.

### IL-04: P3 — scale=1.0f for Zero Blocks (Benign)
File: quantize.c line 90
scale = (max_abs > 0.0f) ? (max_abs / 7.0f) : 1.0f;
Zero blocks store scale=1.0 and nibbles=8 (midpoint) → dequantizes to 0. Correct.
Impact: None. Documentation clarity only.

### IL-05: P0 — Default Tokenizer is Byte-Level, Not BPE
File: tokenizer.c §6
Default xmind_tokenize() encodes each byte as token (byte + 3).
Vocabulary: 3 special + 256 byte = 259 entries.
Llama 3.2 3B requires SentencePiece BPE tokenizer, vocab = 128,256.
Byte-level IDs 3-258 are completely different from Llama 3.2 BPE token IDs.
Feeding byte tokens into embedding table indexed for BPE produces random embeddings.
BPE loading path exists (#ifdef XMIND_HAS_XSTORE) but is disabled in CI.
BPE dispatch function is dead code (see IL-07).

### IL-06: P0 — XMIND_VOCAB_SIZE = 32,000 (Should be 128,256)
File: xmind.h line 35
#define XMIND_VOCAB_SIZE 32000u
Llama 3.2 3B actual: 128,256 tokens.
Consequences:
  (a) xmind_init() returns XMIND_ERR_INVAL for vocab_size > 32,000.
  (b) Sprint 8 preset hardcodes vocab_size=32,000 — wrong for Llama 3.2.
  (c) Logit projection loop only computes 32,000 logits — tokens 32,001-128,255
      are permanently inaccessible to the sampler.
  (d) Embedding table allocated as 32,000 × 3,072 × 4 = 393 MB instead of
      correct 128,256 × 3,072 × 4 = 1,576 MB.
Fix: Change XMIND_VOCAB_SIZE to 128256u; update preset to vocab_size=128256.

### IL-07: P1 — BPE Dispatch Dead Code (External Symbol Always Byte-Level)
File: tokenizer.c lines 569-578
xmind_tokenize_dispatch() is declared __attribute__((unused)) static.
It is never called externally.
The externally-linked symbol xmind_tokenize is always the §6 byte-level function.
Even with s_bpe_loaded == 1 (XSTORE-backed BPE loaded), external callers
get byte-level tokenization. BPE merge path is unreachable from public API.
Fix: Make xmind_tokenize_dispatch the primary externally-linked xmind_tokenize;
mark the byte-level version as xmind_tokenize_bytes.

### IL-08: P3 — bpe_find_rule() is O(n_rules) Linear Scan
File: tokenizer.c lines 322-336
With s_n_bpe_rules up to 8192, and O(n_tokens × n_passes) calls needed,
worst case is O(8192 × n_tokens × n_passes) per tokenization call.
For a 512-token sequence: 8192 × 512 × 512 = ~2.1 billion comparisons.
Fix: Hash (left_id << 32 | right_id) into a uint64_t → rule_index table.

### IL-09: P2 — top-p Uses Conservative Threshold Approximation
File: sampler.c lines 136-138
threshold = (1.0f - p) * max_prob  (approximation, not true CDF)
True nucleus sampling: sort by probability, cumulate until sum >= p.
This approximation includes more tokens than specified p when distribution is
non-uniform. For p=0.9, effective nucleus is larger than intended.
Edge case p=1.0: threshold = 0 → all tokens qualify → correct behavior.
Impact: Slightly higher entropy than specified. Acceptable for sprint.

### IL-10: P3 — Default RNG Seed is Fixed
File: sampler.c line 186
Seed=0 fallback: 0xDEADBEEFCAFEBABEull (hard constant).
All unseeded samplers produce identical token sequences.
Fix for production: seed from HPET counter or TPM RDRAND.

### IL-11: P1 — No Pre-Load Memory Budget Check
File: weights_loader.c §8
No PAL page count query before allocation.
Llama 3.2 3B memory requirements:
  Weights: ~1.2 GB (Q4_0 compressed)
  KV cache (ctx=2048): 28×2×2048×8×96×4 = ~843 MB
  Embeddings (fp32): 128256×3072×4 = ~1,576 MB (if vocab fixed)
  Total: ~3.6 GB — exceeds typical HP EliteBook DRAM headroom
Fix: Query pal_mem_available() before allocation; return XMIND_ERR_NOMEM early.

### IL-12: P1 — GGUF ARRAY Type Skip Causes Metadata Desync
File: weights_loader.c line 310-311
case GGUF_VAL_ARRAY:
    pos += 12u;  /* elem_type(4) + count(8) — does NOT skip element data */
Real GGUF arrays: element_data_size = count × sizeof(elem_type).
Without skipping element data, pos desynchronizes after the first ARRAY entry.
All real Llama 3.2 3B GGUF files contain ARRAY metadata (tokenizer vocab, merges).
Practical effect: all n_layers/n_heads/etc. reads after first ARRAY will be wrong;
code falls back to hardcoded preset. Metadata scan is effectively non-functional.
Fix: Implement per-element-type width dispatch and skip count × width bytes.

### IL-13: P2 — Weight Matrix Allocation Sizes Wrong for Non-Square Matrices
File: weights_loader.c lines 500-511
w_size = hidden_dim × (hidden_dim / Q4_BLOCK) × sizeof(q4_block_t)
        = 3072 × 96 × 20 = ~5.9 MB (for Llama 3.2 3B)
Correct sizes:
  wq[l]: (n_heads × head_dim) × (hidden_dim / Q4_BLOCK) → 3072 × 96 × 20 ✓ (correct by coincidence)
  wk[l]: (n_kv_heads × head_dim) × (hidden_dim / Q4_BLOCK) → 768 × 96 × 20 = 1.5 MB (over-allocated 4x)
  wv[l]: same as wk — over-allocated 4x
  w1[l]: ffn_dim × (hidden_dim / Q4_BLOCK) → 8192 × 96 × 20 = 15.7 MB (under-allocated 2.7x)
  w3[l]: same as w1 — under-allocated 2.7x
  w2[l]: hidden_dim × (ffn_dim / Q4_BLOCK) → 3072 × 256 × 20 = 15.7 MB (under-allocated 2.7x)
Fix: Compute per-matrix dimensions from config before allocation.

### IL-14: P2 — xm_heap_free() is a No-Op; Permanent Memory Leaks
File: xmind.h lines 109-114
xm_heap_free(ptr) { (void)ptr; }
Every xmind_alloc_state() + xmind_shutdown() cycle leaks all state buffers.
Every wl_alloc_pages() call leaks PAL page + map handles.
Fix: Store (ph, mh) in a per-allocation slab; recover on free.

## KEY ARCHITECTURAL FACTS CONFIRMED

- head_dim = 96 for Llama 3.2 3B (= 3072 / 32). NOT 128.
- GQA: kv_head = h * n_kv_heads / n_heads. n_kv=8, n_heads=32 → factor 4.
- SwiGLU: gate=w1×xb, up=w3×xb, hidden=SiLU(gate)×up, out=w2×hidden. CORRECT.
- RoPE: pair-wise rotation on q[2i], q[2i+1]. GQA-aware (only n_kv_heads for K). CORRECT.
- Q4_0: 32 elements, float32 scale, 16 nibble bytes = 20B total. CORRECT.
- GGUF magic: 0x46554747. Versions 1-3 accepted. Immediate XMIND_ERR_CORRUPT on mismatch.
- xm_heap_free() = no-op (documented Sprint 6 decision, must fix for production).
- rope_theta for Llama 3.2: 500,000 (NOT 10,000 from Llama 2).
