## 2. Source File Inline Documentation Assessment

### 2.1 pal/include/pal.h (14-Section API Contract)

**Score: 5/5 — Exemplary**

- Full 14-section structure with section header banners using Unicode box-drawing (═══)
- Each section has a block comment explaining: what it replaces, behavioral contracts, usage conventions
- Every function has `@param` annotations and return value semantics
- Sky OS anti-pattern lessons embedded (e.g., §3: "The Sky OS mistake: using raw pointers across abstraction boundaries")
- Explicit "every behavior not specified here, backends MUST NOT rely on, callers MUST NOT assume" meta-contract
- Format: `__attribute__((format(printf, 1, 2)))` applied where correct
- Freestanding compatibility is documented and guarded

**What is documented:**
- §1 Primitive types with freestanding fallback and why it exists
- §2 Status codes with inline error name semantics
- §3 Handle system with typed/opaque/idempotent contracts
- §4 Physical memory with NUMA advisory policy, page sizes
- §5 Virtual memory with flag semantics
- §6 Threading with priority classes and affinity semantics
- §7 Synchronization with spinlock implementation (inline + documented)
- §8 Timekeeping with monotonic guarantee
- §9 IPC channels with AetherLink lineage noted
- §10 Crypto with SHA-256 one-shot convenience
- §11 DMA/scatter-gather
- §12 Console with format subset documented
- §13 Platform query with thermal reading
- §14 Backend initialization with backend swap contract

**Minor gaps:** The PAL_FREESTANDING assumption that `size_t = uint64_t` on x86-64 is flagged in sprint0-code-integrity-audit.md but not yet added as a `static_assert` comment to the header itself.

### 2.2 kernel/xenos/core/main.c (XENOS kmain, 14-Step Init)

**Score: 4/5 — Very Good**

- File header documents purpose, lists all 7 exported functions, provides compile command
- 14-step init sequence is documented inline at each step with the step number, name, and brief rationale
- Each subsection (§0–§10) uses the same section banner format as pal.h — consistent house style
- External symbol declarations have inline ABI comments explaining sizes and offsets
- MSR addresses are documented with full SDM volume references (vol.4 §2.1)
- TSS descriptor bit-field encoding is comprehensively commented (24 lines of field layout)
- STAR MSR encoding comments describe exact ring-0/ring-3 CS/SS selector arithmetic
- GDT layout comments explain the 11-entry, 16-byte TSS structure

**Gaps:**
- Step 4.5 (EFER.NXE pre-VMM enablement) has a good comment but the security rationale for enabling NXE _before_ VMM is only noted via a ticket reference (`Security Sentinel fix P2-10`) without explaining what attack the ordering prevents in more depth
- The ISR dispatch `_fatal_exc[]` array uses C99 designated initializer syntax that may confuse junior contributors unfamiliar with sparse initialization — brief comment would help
- Syscall dispatch table (§8) has `TODO-S1` and `TODO-S2` stubs but no target ticket links or acceptance criteria that would let Sprint 2 engineers know what "wiring" means concretely

### 2.3 kernel/xenos/cap/xkabi_capabilities.c (XKABI Implementation)

**Score: 4/5 — Very Good**

- File header documents the invention number (INVENTION #02), core security invariants
- Handle encoding (bits [63:40]/[39:16]/[15:0]) is documented at file header AND at §5 with the actual bit arithmetic repeated for clarity
- Security limitation of Sprint 1 (no HMAC, handles forgeable) is explicitly warned with mitigation path (Sprint 2 HMAC)
- Every public function has a full Doxygen-style block: purpose, `@param`, return semantics, thread-safety claim
- Generation epoch design (TOCTOU fix) is cross-referenced to the Linux validation audit date
- The `XKABI_LINEAGE_MAX_DEPTH` constant documents why 16 entries are chosen
- `xkabi_cap_verify` five-check sequence is numbered and each check explains what attack it prevents

**Gaps:**
- §4 "Module-Private Globals" documents the BSS placement rationale but does not document the maximum theoretical memory consumption (6MB BSS) in the source — this appears only in the §1 constant comment
- The `xkabi_audit()` function uses `__atomic_fetch_add` but the comment note ("held inside spinlock callers already hold, OR called after release") describes a dual-mode usage that could be confusing — the locking discipline is not immediately obvious to a new contributor
- No documentation of what `object_id=0` auto-assignment means for external callers — is it recommended? Required?

### 2.4 init/gensd/supervisor.c (GENSD Supervisor)

**Score: 4/5 — Very Good**

- File header explains purpose, restart policy, health probe types, freestanding constraint, thread safety model
- Exponential backoff algorithm is described in the header comment with exact formula (`delay = min(delay * 2, 30000)`)
- Each public API function has a full block comment
- The `supervisor_watch()` function documents the opaque struct layout via a local typedef with explicit byte offsets — a temporary measure awaiting `gensd_types.h` (Sprint 3)
- Sprint 1 stub behavior of `supervisor_probe_http()` is documented with the full Sprint 2 implementation outline (7 numbered steps)
- The known blocking behavior of `pal_thread_sleep_ns()` in `supervisor_restart()` is explicitly flagged with a `TODO(Sprint 2)` and cross-referenced to the event-horizon analysis document

**Gaps:**
- `sv_watch_entry_t` duplicates fields from `gsd_service_t` (documented as a temporary workaround) but there is no cross-reference back to `init/gensd/service.c` to confirm the field layout matches at compile time — a future struct mismatch would be silent
- The 64-service limit (`SV_MAX_WATCHED`) is not explained — why 64? This is an arbitrary but important constraint for system sizing
- No documentation of whether `supervisor_tick()` is safe to call from a signal handler context (it is probably not, but this is unspecified)

## 4. Top 5 Documentation Gaps That Will Slow Sprint 2 Development

### GAP-1: No QEMU Boot Debugging Runbook (CRITICAL — Sprint 2 Blocker)

**Impact:** Every Sprint 2 engineer working on XDISP (DRM/KMS compositor) or XFRAME (native UI) will need to boot the kernel in QEMU. Without a runbook, each engineer will independently reverse-engineer the QEMU invocation, the expected serial output at each of the 14 boot steps, and how to recover from triple-faults. This is a 2–4 hour cost per engineer per boot debugging session.

**What is missing:**
- QEMU command line for launching the XENOS kernel binary (flags: `-kernel`, `-serial stdio`, `-m`, `-cpu`, `-machine`, `-enable-kvm`, etc.)
- Expected serial output: the 14 `[XENOS] [N/14]` messages and their success indicators
- What a triple-fault looks like (QEMU reboots silently) and how to detect it (`-d cpu_reset`)
- GDB remote stub setup (`-s -S` flags, how to load XENOS ELF symbols)
- Common failure modes: bad magic value, PAL init failure, LAPIC MMIO fault
- How to interpret the `isr_frame_t` register dump (which field is which)
- NASM/ELF linker command to produce a flat binary vs ELF64 kernel image

**Severity:** This gap will cause significant time loss in Sprint 2 because XDISP requires framebuffer access that is debugged almost exclusively through QEMU.

### GAP-2: No XENOS Kernel Contributor Guide (HIGH — Sprint 2 Onboarding)

**Impact:** Any engineer joining Sprint 2 to implement XDISP, XFRAME, or complete the contra-rotation scheduler will not know: which files to modify, how to add a new subsystem to the 14-step boot sequence, how to write a new PAL-Aether backend function, or what the XKABI syscall dispatch contract requires.

**What is missing:**
- "Adding a new kernel subsystem" checklist: create .c/.h files, add to Makefile, add to xenos.yml, add weak stub in main.c, add to boot sequence
- "Adding a PAL-Aether function" protocol: implement in pal_aether.c, add declaration to pal.h §N, add test to test_pal.c
- "Adding a XKABI syscall" protocol: assign number in xkabi_syscalls.h, add case to xenos_syscall_dispatch(), add XKABI right if needed
- "Registering a GENSD service" protocol: add to service table in service.c, define start/stop/health, call supervisor_watch()
- The PAL discipline (zero Linux headers) and how CI enforces it
- The capability model: why every new kernel object needs a capability, how to derive caps for child objects

**Severity:** Without this guide, Sprint 2 will incur 1–3 days of onboarding per engineer as they read source code to infer these conventions.

### GAP-3: No ADRs for XENOS/PAL/GENSD Core Design Decisions (HIGH — Architecture Continuity)

**Impact:** The 4 existing ADRs cover XISC/Transformer Cloud. The most consequential design decisions in Sprint 0 and Sprint 1 — PAL abstraction strategy, XENOS as a microkernel vs monolithic kernel, GENSD vs systemd, AetherBoot vs EDK2, Contra-Rotation scheduler vs CFS — are undocumented as decisions. Sprint 2 engineers will not know why these choices were made, making it difficult to judge when a proposed change violates the design intent.

**Missing ADRs (priority order):**

1. **ADR-PAL-001: PAL Abstraction Strategy** — Why a 14-section flat API instead of VFS-style or object-oriented? Why freestanding fallback types instead of using stdint.h? Why handles instead of direct pointers? What is the Sky OS anti-pattern being avoided?
2. **ADR-XENOS-001: XENOS Microkernel Architecture** — Why microkernel vs monolithic? What goes in the kernel vs user space? Why 14-step boot sequence (vs lazy init)? Why static capability table (64K slots, 6MB BSS) vs dynamic?
3. **ADR-GENSD-001: GENSD vs systemd** — Why write a custom init system? What systemd features are deliberately excluded? How does GENSD interact with the XKABI process model?
4. **ADR-XKABI-001: Handle Encoding Format** — Why 64-bit bit-packing for Sprint 1? Why 64K cap table limit? Why is HMAC deferred to Sprint 2? What are the security implications of the current design?
5. **ADR-AETHER-001: AetherBoot vs EDK2** — Why avoid EDK2? What is the maintenance burden of a zero-EDK2 UEFI loader? What UEFI features are not supported?
6. **ADR-SCHED-001: Contra-Rotation Scheduler** — What is contra-rotation? Why dual-ring scheduling? What problem does it solve that CFS does not?

**Severity:** Without these ADRs, design erosion will occur as Sprint 2 engineers make locally reasonable decisions that violate the global architecture intent.

### GAP-4: GENSD Service Dependency Model Not Documented (MEDIUM — Sprint 2 Service Integration)

**Impact:** Sprint 2 will add new services (XDISP, potentially xnet). Without a documented dependency DAG and startup ordering protocol, new services will be wired incorrectly, causing boot hangs or race conditions.

**What is missing:**
- A startup dependency graph (which service requires which to be running first)
- The GENSD `requires[]` field semantics — what does it mean for service A to require service B? Does GENSD wait for B's health probe to pass before starting A, or does it just sequence them?
- The health probe contract — what does `HEALTH_OK` mean? Is HTTP /healthz a strong guarantee or best-effort?
- The restart policy interaction with the dependency model — if a required service restarts, does the dependent service restart too?
- Maximum service count justification — why 64 services? What happens at 65?
- The one-shot vs daemon distinction for system initialization use cases (e.g., TPM attestation is a one-shot, display server is a daemon)

**Severity:** Medium impact in Sprint 2, but becomes HIGH in Sprint 3 when the full service mesh must integrate with XKABI process handles.

### GAP-5: No Testing Protocol Documentation for Kernel Work (MEDIUM — Quality Assurance)

**Impact:** The mandatory E2E testing protocol (10 test types, 3 methodologies, 4 static analysis tools) applies to all agents but there is no document explaining how this protocol applies to freestanding kernel code. Standard tools (pytest, ESLint) don't apply. Custom procedures are needed.

**What is missing:**
- How to run smoke tests against a QEMU-booted XENOS kernel (serial output parsing)
- What functional tests exist for kernel subsystems (currently: none beyond compile checks)
- How to fuzz the syscall dispatch table
- How to run memory safety checks on freestanding C (ASAN doesn't apply; alternatives?)
- How to run regression tests: which `.o` objects are compared between builds?
- What constitutes a "security test" for the capability system (XKABI)
- How to run cppcheck/clang-tidy against kernel sources locally (flags are in xenos.yml but not documented for local use)
- Where test artifacts are stored and for how long (currently: 7-day retention in CI)

**Severity:** Without this, Sprint 2 engineers will skip testing kernel changes beyond "does it compile and boot to idle," creating a quality regression risk as XDISP and XFRAME are added.
