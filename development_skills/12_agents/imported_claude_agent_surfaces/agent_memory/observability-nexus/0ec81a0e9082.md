## 1. Audit Ring Capacity Model

### Ring Buffer Specification
- **Size:** 1024 entries (XSEC_AUDIT_RING_SIZE)
- **Entry size:** ~88 bytes (8B timestamp + 4B event + 4B module + 64B detail + padding)
- **Total BSS footprint:** ~90 KB
- **Write protocol:** Lock-free atomic fetch-and-increment on write_idx
- **Overflow behavior:** Oldest entry silently overwritten; dropped_events counter incremented atomically (S3-DI-H01 fix)

### Overflow Rate Analysis

| Scenario | Events/operation | Ops/sec | Events/sec | Time to overflow |
|----------|-----------------|---------|------------|-----------------|
| TLS handshake (success) | 1 (HANDSHAKE_COMPLETE) | - | - | - |
| TLS handshake (fail, all paths) | 1-2 (HANDSHAKE_FAILED + possible CERT_REJECTED) | - | - | - |
| TLS record decrypt fail | 1 (AUTH_TAG_MISMATCH) | - | - | - |
| TLS session close | 1 (SESSION_CLOSED) | - | - | - |
| X.509 chain verify (3-cert chain, success) | ~4-7 (per-cert checks + final CERT_VERIFIED) | - | - | - |
| X.509 chain verify (failure) | 1-3 (CERT_REJECTED at first failing cert) | - | - | - |
| X.509 hostname verify (fail) | 1 (CERT_REJECTED) | - | - | - |
| Normal TLS session lifecycle | ~5-8 total events | - | - | - |
| **Sustained TLS connections** | 5-8 | 10 conn/s | 50-80 | **12-20 seconds** |
| **TLS under attack (bad certs)** | 2-3 | 100 conn/s | 200-300 | **3-5 seconds** |
| **Idle system (no TLS)** | 0 | 0 | 0 | Never |

### Critical Finding: Ring Overflow Under Attack

At 100 failed TLS connections/sec (a modest brute-force or scanning attack), the ring
overflows in ~3-5 seconds. All forensic evidence of earlier attack phases is lost.
The dropped_events counter (S3-DI-H01 fix) alerts to data loss but cannot recover
the overwritten entries.

**XPKG gate events are NOT in the ring at all** (see Section 2), so package operations
do not contribute to ring pressure. If XPKG were wired in (9 gates per install), a
burst of 100 package installs would generate 900+ events and overflow in ~1 second.

### Overflow Mitigation Assessment
- [GOOD] dropped_events counter exists (S3-DI-H01 fix)
- [GOOD] xsec_audit_dump() reports visible vs total counts
- [GOOD] xsec_audit_query_recent() returns most recent N entries
- [GAP] No overflow callback/hook to flush ring to persistent storage
- [GAP] No priority-based retention (all events treated equally; critical CERT_REJECTED
  events overwritten by routine SESSION_CLOSED events)
- [GAP] No ring size tunability at runtime

### Dead Audit Event Types: 3 of 9 (33%)

1. **XSEC_AUDIT_KEY_GENERATED (=1):** Defined for X25519/Ed25519 keypair generation.
   Neither `xsec_x25519_keypair()` nor `xsec_ed25519_keypair()` emit this event.
   Only called in test harness.

2. **XSEC_AUDIT_DECRYPT_FAIL (=6):** Defined for generic decryption failures.
   AES-GCM and ChaCha20 return XSEC_ERR_AUTH_FAIL but do NOT call xsec_audit_log().
   Only the TLS record layer wraps their failure with AUTH_TAG_MISMATCH (event 7).
   Direct callers of xsec_aes_gcm_decrypt() outside TLS get zero audit trail.

3. **XSEC_AUDIT_TLS_ALERT_RECV (=8):** Defined for incoming TLS alerts.
   tls13.c sends alerts via tls_send_alert() but NEVER logs received alerts.
   An attacker sending fatal alerts to tear down sessions is invisible in the audit log.

## 5. Recommended New Audit Event Types for Sprint 4

| New Event | Value | Module | Rationale |
|-----------|-------|--------|-----------|
| XSEC_AUDIT_PKG_GATE_FAIL | 10 | new XSEC_MODULE_XPKG (=9) | Package gate failure with gate ID in detail |
| XSEC_AUDIT_PKG_GATE_FORCED | 11 | XSEC_MODULE_XPKG | FAIL->WARN force demotion (admin override trail) |
| XSEC_AUDIT_TCP_RST_RECV | 12 | new XSEC_MODULE_XNET (=10) | TCP reset received (connection teardown) |
| XSEC_AUDIT_TCP_STATE_CHANGE | 13 | XSEC_MODULE_XNET | State machine transitions (SYN_SENT, ESTABLISHED, etc.) |
| XSEC_AUDIT_DNS_QUERY | 14 | XSEC_MODULE_XNET | DNS query sent (name + QID, no PII) |

Additionally, the 3 dead events should be wired into production code:
- KEY_GENERATED: emit from xsec_x25519_keypair() and xsec_ed25519_keypair()
- DECRYPT_FAIL: emit from xsec_aes_gcm_decrypt() and xsec_chacha20_poly1305_decrypt() on tag mismatch
- TLS_ALERT_RECV: emit from tls13.c alert reception path
