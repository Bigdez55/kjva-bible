making the dump output misleading for package security events.

## 2. GENSD JOURNAL STRUCTURE

**File:** `init/gensd/journal.c` (326 LOC)

### 2.1 Format: Binary packed, NOT structured text, NOT JSON

Wire format per entry:
```
[timestamp_ns: 8B][service_hash: 4B][level: 2B][msg_len: 2B][msg: N bytes]
Fixed header: 16 bytes. Variable tail: msg[msg_len].
```

- Service name is hashed via fnv1a32 (compact, but non-reversible without lookup table)
- Messages are human-readable strings (max 255 bytes)
- No correlation_id, no trace_id, no span_id
- No structured fields within the message body

### 2.2 Log Level Concept: YES (5 levels)

| Value | Name   |
|-------|--------|
| 0     | DEBUG  |
| 1     | INFO   |
| 2     | WARN   |
| 3     | ERROR  |
| 4     | FATAL  |

### 2.3 Persistence: RAM-ONLY (4MB BSS ring buffer)

- Ring buffer in `static uint8_t g_ring[4MB]` -- BSS segment
- No persistence to XSTORE, no persistence to disk
- On power loss or panic: all journal data is LOST
- Ring overwrites oldest entries silently (no dropped counter)
- `journal_read_recent()` is a STUB -- returns nothing (comment says "Sprint 3")

### 2.4 Journal Users

| File              | journal_write calls |
|-------------------|---------------------|
| gensd/main.c      | 16                  |
| gensd/supervisor.c | 8                   |
| gensd/journal.c   | 5 (self-test/init)  |
| **Total**         | **29**              |

No other subsystem calls journal_write(). The journal is GENSD-internal only.

### 2.5 DEFECT: journal_read_recent() is still a Sprint 1 stub

Line 236-240: Returns NULL/0. Four sprints later, the reader is still unimplemented.
This means no programmatic access to journal entries -- only serial flush.

**Severity: P1** -- the primary structured log store has no read API.

## 4. XPKG REFUSAL GATE LOGGING

**File:** `pkg/xpkg/gate/refusal_gate.c` (599 LOC)

### 4.1 Audit Integration: PRESENT (2 call sites)

| Line | Event              | Condition                    |
|------|--------------------|------------------------------|
| 537  | PKG_GATE_FORCED    | FAIL force-demoted to WARN   |
| 545  | PKG_GATE_FAIL      | All non-PASS outcomes        |

### 4.2 Coverage Assessment

- FAIL on any gate -> xsec_audit_log(PKG_GATE_FAIL) at line 545
- FAIL force-demoted -> xsec_audit_log(PKG_GATE_FORCED) at line 537 FIRST,
  then also PKG_GATE_FAIL at line 545 (double-logged for forced demotions)
- WARN outcomes -> xsec_audit_log(PKG_GATE_FAIL) at line 545
- PASS outcomes -> NOT logged (correct; PASS is the happy path)

### 4.3 DEFECT: Forced demotion double-logs

When `config->force[g]` is true and result is FAIL:
1. Line 537: logs PKG_GATE_FORCED
2. Line 538: result demoted to WARN
3. Line 544: `result != XPKG_GATE_RESULT_PASS` is true (WARN != PASS)
4. Line 545: logs PKG_GATE_FAIL

So a single forced-demotion event produces TWO audit entries: one FORCED, one FAIL.
The FAIL entry is misleading because the gate actually WARNed (was demoted).

**Severity: P2** -- audit noise, not data loss, but confusing for forensics.

### 4.4 Console logging: PRESENT (pal_console_printf)

Every gate result is printed to console via pal_console_printf (lines 548-560).
Gate skip, gate force, and gate results all have console output.

**Verdict: XPKG gate is the most observability-complete subsystem outside XSEC.**

## 9. GO/NO-GO VERDICT

**VERDICT: CONDITIONAL GO -- with mandatory P0 remediation**

The observability platform has significant gaps but the core security subsystem
(XSEC) is well-instrumented at 83% error-path coverage. The primary blockers are:

1. **TCP is completely dark** -- this is a security and reliability risk. Any TCP-level
   attack, connection failure, or performance degradation is invisible.

2. **XSTORE/XBLOB are completely dark** -- data corruption, write failures, and
   B-tree splits are invisible. For a data-path subsystem, this is unacceptable.

3. **The audit dump function misreports XPKG events as "UNKNOWN"** -- a regression
   from the Sprint 3 hardening that added PKG_GATE_FAIL/FORCED events without
   updating the dump name lookup.

The platform can ship for demonstration purposes but CANNOT be considered
production-ready without TCP and data-path observability.
