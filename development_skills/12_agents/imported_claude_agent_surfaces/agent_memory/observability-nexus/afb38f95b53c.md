## 1. AUDIT RING BUFFER CORRECTNESS

### 1.1 Lock-Free Write Protocol Analysis

**File:** `sec/xsec/audit/audit.c` (283 LOC)

The ring buffer uses a two-phase write protocol:
1. `pal_atomic_add(&write_idx, 1)` -- atomic fetch-and-increment gives each writer a unique slot
2. Non-atomic write to `entries[slot]` -- safe because each writer owns its slot
3. `pal_atomic_add(&total_events, 1)` -- atomic increment of event counter

**VERDICT: CONDITIONALLY CORRECT**

The slot allocation via atomic FAI is sound for concurrent writers: no two writers can
receive the same slot index. However, there is a subtle race window:

- **TOCTOU in dropped_events counting (lines 150-153):**
  Writer A atomically claims slot 1025 (write_idx returns 1025).
  Writer A reads total_events (which is, say, 1024) and increments dropped_events.
  Writer B claims slot 1026 and reads total_events BEFORE Writer A's total_events
  increment at line 162. Writer B sees total_events=1024, also increments dropped_events.
  This means dropped_events can OVERCOUNT by a small amount under extreme concurrency.

  **Impact:** Minor -- dropped_events may slightly overcount. This is an acceptable
  conservative error (better to overcount drops than undercount).

- **Entry visibility race:** Between slot claim (line 145) and total_events increment
  (line 162), a concurrent reader calling `xsec_audit_query_recent()` could read a
  partially-written entry (timestamp written, detail not yet copied). The query function
  takes the spinlock, which is not held during writes, so it cannot prevent this.

  **Impact:** Unlikely in practice (reader would need to query during the ~microsecond
  write window) but not impossible under high concurrency.

### 1.2 Atomic Head/Tail Management

- `write_idx` (volatile uint64_t): Monotonically increasing via `pal_atomic_add`. **CORRECT.**
- `total_events` (volatile uint64_t): Monotonically increasing via `pal_atomic_add`. **CORRECT.**
- `dropped_events` (volatile uint64_t): Monotonically increasing via `pal_atomic_add`. **CORRECT.**
- Slot mapping: `slot = idx % XSEC_AUDIT_RING_SIZE` -- no modular arithmetic error. **CORRECT.**

### 1.3 Ring Capacity Assessment

- **1024 entries, ~88 bytes each, ~90KB BSS**
- TLS handshake lifecycle: 5-8 events per connection
- Normal operation (10 conn/s): overflow in 12-20 seconds
- Under attack (100 conn/s): overflow in 3-5 seconds
- **No flush-to-persistent hook exists**

### 1.4 Dropped Event Counter

- **Implemented** (S3-DI-H01 fix)
- `xsec_audit_get_dropped()` returns the count via `pal_atomic_load`
- `xsec_audit_dump()` prints a WARNING line when dropped > 0
- Slight overcount possible under extreme concurrency (see 1.1 above)
- **VERDICT: FUNCTIONAL and CORRECT for operational purposes**

### 1.5 Init Guard

- `xsec_audit_init()` must be called once before any `xsec_audit_log()` call
- The old auto-init race (S3-DI-L01) was fixed: `xsec_audit_log()` returns
  `XSEC_ERR_STATE` if `!initialized`
- **VERDICT: CORRECT**

## 3. AUDIT TRAIL INTEGRITY

### 3.1 Post-Write Tamper Resistance

**VERDICT: NO TAMPER RESISTANCE**

- Entries are written to a writable BSS array (`xsec_g_audit.entries`)
- No write-once semantics: any code with access to the global symbol can modify entries
- No cryptographic chaining (no hash chain linking entry N to entry N-1)
- No signature or MAC on entries
- In a freestanding kernel, this is acceptable for Sprint 3 (no untrusted code runs
  in the same address space). For Sprint 4+, if userspace processes exist, the audit
  ring must be protected by capability checks or moved to a privileged region.

### 3.2 Per-Entry Metadata

Each `xsec_audit_entry_t` contains:
- `timestamp_ns` (uint64_t): from `pal_time_now_ns()` -- **monotonic, PRESENT**
- `event_type` (enum): typed event classification -- **PRESENT**
- `source_module` (enum): typed module identification -- **PRESENT**
- `detail` (char[64]): human-readable context string -- **PRESENT**

**Missing:**
- **Sequence number:** No per-entry monotonic sequence counter. The write_idx gives
  ordering but is not stored in the entry itself. After ring wrap, there is no way
  to determine the absolute ordering of two entries without external correlation.
- **Thread/CPU ID:** No identification of which execution context emitted the event.
- **Severity level:** All events have the same weight; critical CERT_REJECTED events
  can be overwritten by routine SESSION_CLOSED events.

### 3.3 Dropped Event Counter Accuracy

As analyzed in section 1.1, the dropped_events counter may slightly overcount under
extreme concurrency but will never undercount. **Operationally correct.**

must prioritize wiring existing event types into production code paths and adding TCP
observability from scratch.
