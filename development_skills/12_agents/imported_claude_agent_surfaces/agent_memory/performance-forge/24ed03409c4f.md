## 3. CRITICAL PATH: FRAME PIPELINE

```
vsync signal
  |
  v  [0.000 ms] DRM VBlank IRQ -> compositor wakeup
  |
  v  [0.042 ms] Z-sort 128 surfaces (insertion sort, worst case O(n^2))
  |
  v  [0.813 ms] SRC_OVER blend pass (5 surfaces @ 40% coverage avg)
  |             1920*1200*0.4*5 = 4,608,000 pixels
  |             Each pixel: read src (4B) + read dst (4B) + write dst (4B) = 12B
  |             Total memory: 55.3 MB per frame
  |             At 34 GB/s LPDDR4x: 1.63 ms theoretical
  |             With cache misses + ALU: ~0.81 ms (50% efficiency)
  |
  v  [0.100 ms] DRM page flip (kernel, non-blocking)
  |
  v  [0.010 ms] Damage region reset
  |
  =====================================
  Total compositor: ~0.97 ms per frame
  Remaining budget: 15.70 ms (for shell rendering + XFRAME layout)
  =====================================
  |
  v  [Shell: Electron React reconciliation -- NOT in XCOMP path]
  |  Electron renders to a GEM surface buffer; XCOMP just composites it.
  |  Shell rendering is off the critical path of the compositor.
  |
  v  [XFRAME layout (native path):
  |   2048 nodes * 2 passes * 50 cycles/node = 0.053 ms
  |   Paint: 200 dirty nodes * ~1000 cycles/node = 0.051 ms
  |   Total XFRAME: ~0.10 ms]
```

**Frame budget verdict**: WITHIN BUDGET at 7.59 ms total (worst case) of 16.67 ms.
The compositor uses only ~6% of the frame budget. Even with pessimistic assumptions
(10 overlapping surfaces, 60% coverage), the blend pass stays under 2.5 ms.

**Bottleneck classification**: MEMORY-BOUND. The SRC_OVER blend loop is limited
by memory bandwidth, not CPU arithmetic. At 1920x1200 ARGB8888, each full-screen
composite pass moves 27.6 MB through the memory bus. The integer multiply in
BLEND_MUL255 is a single-cycle operation; the stalls are cache line fetches.

**Optimization opportunity**: Damage-driven partial compositing is already
implemented (damage rects). The main improvement would be SSE2/AVX2 vectorized
blending (process 4-16 pixels per instruction). Expected improvement: 2-4x for
the blend pass, but since blend is already <1 ms, the ROI is low.

**Throughput estimates** (i7-1065G7 @ 3.9 GHz boost):

| Implementation          | FLOP/cycle/core | 4-core FLOP/s | Tokens/sec |
|--------------------------|-----------------|---------------|------------|
| Scalar fp32 (current)    | ~1 useful       | 15.6 GFLOP/s  | **2.6**    |

**NOTE**: The "useful" FLOP/cycle accounts for Q4 dequantization overhead.
With Q4_0, the inner loop must unpack nibbles and multiply by scale before
the dot product. This roughly halves the theoretical peak compared to raw fp32.

Current scalar implementation: `xmind_matmul_q4()` in `tensor.c` processes
one element at a time. The inner loop dequantizes a nibble (2 ops: mask + shift,
subtract 8, multiply by scale) and accumulates a dot product (1 FMA). This
gives approximately 1 useful FLOP per 3-4 cycles, yielding ~1 GFLOP/s per core.

**Verdict**: At 2.6 tokens/sec scalar, first-token latency for a 100-token
prompt would be ~38 seconds. This is OVER BUDGET (target: <5s batch). AVX2+FMA
SIMD is mandatory for Sprint 5/6 to reach the 10+ tokens/sec minimum threshold.

With AVX2+FMA at 4 cores: 40.9 tokens/sec, first-token latency for 100-token
prompt: ~2.4 seconds. This MEETS the <5s budget.

**Memory bandwidth check**: At 40 tok/s with AVX2, the matmul reads 1.65 GB of
Q4 weights per token. At 40 tok/s = 66 GB/s. LPDDR4x peak is 34 GB/s, so at
>20 tok/s the system becomes MEMORY-BOUND, not compute-bound. The practical
AVX2 ceiling is approximately 20 tokens/sec due to memory bandwidth.

### 4B. XCOMP Compositor: MEMORY-BOUND

As analyzed in the frame pipeline section. The blend loop reads/writes ~55 MB

### 4C. XSTORE B+ Tree: I/O-BOUND (when flushing), COMPUTE-BOUND (for CRC)

- B+ tree lookups: O(log256 N) = 2-3 page reads for millions of keys. Binary
  search within a page: O(log 256) = 8 comparisons. This is fast.
- Page cache linear scan (`_cache_find`): O(1024). At ~5 cycles/comparison,
  this is 5,120 cycles = 1.3 us. For a hot path called on every page access,
  this should be O(1) via hash directory. P2 finding.
- CRC32C computation: Byte-by-byte Sarwate algorithm. Per 4096-byte page:
  4096 * 8 bit-iterations * ~3 cycles = 98,304 cycles = 25 us per page.
  With CRC32C instruction (SSE4.2, available on i7-1065G7): ~0.25 cycles/byte
  = 1,024 cycles = 0.26 us. That is a 96x improvement. P2 finding.
- WAL append: Each append writes a 4,384-byte entry. At NVMe write latency
  of ~10 us, this is fine for throughput but the entry size is wasteful.

### 4D. XBLOB: I/O-BOUND

- SHA-256 on put: ~12 cycles/byte software. For 16 MB blob: 12 * 16M = 192M
  cycles = 49 ms. With SHA-NI extensions (available on Ice Lake): ~2 cycles/byte
  = 8 ms. P2 finding.
- SHA-256 on every read (integrity verification): Same cost. For small blobs
  (<1 KB) this is negligible; for large blobs it dominates read latency.
- Index lookup: O(1) average with linear probing. Good.

### 4E. XNET TCP: COMPUTE-BOUND (byte-by-byte ring copy)

- `ring_write`/`ring_read`: Byte-by-byte copy with modular arithmetic per byte.
  For a 1460-byte segment: 1460 iterations * ~5 cycles = 7,300 cycles = 1.9 us.
  With memcpy: ~1460/64 * 5 = 114 cycles = 29 ns. That is 65x slower. P1 finding.
- TCB lookup: Linear scan of 128 entries. At ~10 cycles/entry: 1,280 cycles =
  0.33 us. Acceptable for the connection rate of a single-node OS.

### 4F. XSEC Crypto: COMPUTE-BOUND

| Cipher                   | Cycles/byte | Throughput @ 3.9 GHz | Verdict           |
|--------------------------|-------------|----------------------|-------------------|
| ChaCha20 (scalar)        | 60.0        | 65 MB/s              | ACCEPTABLE        |

| ChaCha20-Poly1305 AEAD   | 61.25       | 63.7 MB/s            | ACCEPTABLE        |

ChaCha20-Poly1305 at 63.7 MB/s is sufficient for TLS 1.3 record encryption.

ChaCha20-Poly1305 and 1,806x slower than AES-NI. A 16 KB TLS record would
take 44.4 ms to encrypt -- this consumes 44% of a 100 ms API budget and
is flatly unusable for data-heavy operations. AES-NI is mandatory.

## 5. ALL FINDINGS (P0 / P1 / P2 / P3)

### P0 -- CRITICAL (Must fix before Sprint 4 merge)

**P0-WAL-MEM-01: XSTORE WAL buffer requires 273 MB**
- Location: `store/xstore/include/xstore.h` line 209
- `XSTORE_WAL_BUF_SIZE = 65536 * 4384 = 286,654,464 bytes`
- Root cause: `xstore_wal_entry_t` embeds `key[256] + val[4096]` inline
- Fix: Separate WAL entry header (32 bytes) from payload. Store key/val
  in the page itself; WAL entry only needs lsn + txn_id + page_no + op + lengths
  + checksum = 32 bytes. New WAL buffer: 65536 * 32 = 2 MB.
- Alternative: Reduce WAL ring to 4096 entries + compact entry format = 128 KB.
- Impact: Saves 271-273 MB of RAM.
- Risk: Low. WAL replay logic needs update but the format change is mechanical.

**P0-AES-CT-01: AES-256-GCM constant-time S-box is 1,806x slower than AES-NI**
- Location: `sec/xsec/crypto/aes_gcm.c`, function `aes_sbox_ct()`
- Each byte: 256-iteration linear scan * 14 rounds * 16 bytes/round = 57,344
  iterations per 16-byte block.
- Fix: Implement AES-NI path using `_mm_aesenc_si128` / `_mm_aesenclast_si128`
  intrinsics. i7-1065G7 (Ice Lake) supports AES-NI.
- Impact: 0.36 MB/s -> 650 MB/s (1,806x improvement).
- Risk: Medium. Requires CPU feature detection (CPUID check) and fallback path.
- Carried forward from Sprint 3 audit.

**P0-GHASH-01: GHASH bit-by-bit multiplication is 32x slower than table-based**
- Location: `sec/xsec/crypto/aes_gcm.c`, function `ghash_mul()`
- 128 iterations per multiply with 16-byte inner XOR loop = 2,048 XOR ops.
- Fix: 4-bit lookup table (256 entries * 16 bytes = 4 KB) reduces to 32
  iterations. PCLMULQDQ instruction reduces to ~6 cycles total.
- Impact: 64 cycles/byte -> 2 cycles/byte (32x improvement with table,
  160x with PCLMULQDQ).
- Risk: Low (4-bit table) to Medium (PCLMULQDQ).
- Carried forward from Sprint 3 audit.

**P0-XMIND-SCALAR-01: XMIND matmul is scalar fp32 -- 2.6 tokens/sec**
- Location: `ai/xmind/src/tensor.c`, function `xmind_matmul_q4()`
- Inner loop processes one element at a time. No SIMD.
- Fix: AVX2 + FMA path processing 8 elements per cycle. Q4 dequantization
  can use `_mm256_and_si256` for nibble extraction.
- Impact: 2.6 tok/s -> ~20 tok/s (memory-bandwidth-limited ceiling).
- Risk: Medium. Requires careful alignment handling and CPU feature detection.
- Timeline: Sprint 5/6 (XMIND SIMD track).

### P1 -- HIGH (Fix in Sprint 4-5)

**P1-TCP-RING-01: Byte-by-byte ring buffer copy is 65x slower than memcpy**
- Location: `net/xnet/transport/tcp.c`, functions `ring_write()`, `ring_read()`
- Fix: Two-phase copy (head-to-end, wrap-to-remainder) using xnet_memcpy.
- Impact: 1.9 us -> 29 ns per 1460-byte segment.
