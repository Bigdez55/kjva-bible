## Critical Findings

### P0 — GHASH GF(2^128) multiplication is O(128) bit-by-bit (aes_gcm.c:223-254)
- Inner loop: 128 iterations, each with 16-byte XOR + 16-byte right-shift
- Per-block cost: ~2048 XOR + ~2048 shift operations
- TLS record (16KB) = 1024 GHASH blocks = ~2M XOR ops per record
- RECOMMENDATION: Switch to 4-bit table-based GHASH (16-entry * 16-byte = 256B table)
  - Expected speedup: 4x (128 iterations -> 32 iterations with 4-bit nibbles)

### P0 — ChaCha20-Poly1305 static 32KB scratch buffer (chacha20.c:367)
- POLY_SCRATCH_MAX = 32,816 bytes allocated as `static` BSS
- NOT thread-safe: concurrent AEAD calls corrupt shared buffer
- Wastes 32KB BSS even when not in use
- RECOMMENDATION: Streaming Poly1305 MAC (feed blocks incrementally, no scratch)

### P1 — AES gf_mul() bit-by-bit in MixColumns (aes_gcm.c:56-68)
- Called 32 times per round (4 columns * 4 mul * 2 constants)
- 13 rounds * 32 = 416 calls per block, each with 8-iteration loop
- RECOMMENDATION: Use xtime() inline for constants 0x02/0x03 only
  - gf_mul(0x02, x) = (x<<1) ^ (x & 0x80 ? 0x1b : 0)
  - gf_mul(0x03, x) = gf_mul(0x02, x) ^ x
  - Eliminates all 8-iteration loops in MixColumns

### P1 — TCP ring buffer byte-by-byte copy (tcp.c:106-126)
- ring_write/ring_read copy one byte at a time with modular arithmetic
- 16KB buffer at 1Gbps = ~80K packets/sec, each copying up to 1460 bytes
- RECOMMENDATION: Two-phase memcpy (head-to-end, wrap-to-tail)

### P1 — IP fragment reassembly: received[] is bool array, not bitmap (ip4.c:247)
- `bool received[FRAG_MAX_SIZE / 8]` = 8192 bools = 8KB per entry
- 16 entries * 8KB = 128KB wasted (bools are 1 byte each, not 1 bit)
- RECOMMENDATION: Use uint8_t bitmap[FRAG_MAX_SIZE / 64] = 1KB per entry
  - Saves 112KB of static memory across 16 entries

### P1 — dep_resolve conflict detection is O(V^2) (dep_resolve.c:194-218)
- Nested loop: every pair of nodes compared by name string
- 512 max nodes: worst case 131,072 string comparisons
- RECOMMENDATION: Sort nodes by name first (O(V log V)), then linear scan for duplicates

### P2 — SHA-256 W[] array is 256 bytes on stack (sha256.c:66)
- W[64] * 4 bytes = 256 bytes per compress call
- SHA-512 W[80] * 8 bytes = 640 bytes per compress call
- Acceptable for kernel stack, but could use 16-word rotating window

### P2 — x25519 fe_sq() calls fe_mul(a, a) — no squaring optimization (x25519.c:167-169)
- Dedicated squaring saves ~40% multiplications (15 vs 25 products)
- Montgomery ladder calls fe_sq 4x per bit = 1020 squarings per scalar mul
- RECOMMENDATION: Implement dedicated fe_sq with halved cross-products

### P2 — ed25519 fe10_mul truncates to int32_t before reduce (ed25519.c:105-109)
- Products are int64_t but stored as int32_t before fe10_reduce
- Correctness risk: large intermediate values may overflow 32-bit
- This follows the ref10 pattern but needs verification that limb bounds hold

### P2 — CRC32 table init is not thread-safe (ethernet.c:60-77)
- `g_crc32_table_ready` bool checked without lock
- Race condition: two threads may both init simultaneously
- Low risk (idempotent result) but technically UB
- RECOMMENDATION: Init table at boot via explicit init function

### TCP TCB Size Breakdown (per connection)
- Ring buffers: 2 * 16KB = 32KB
- Retransmit queue: 32 * ~1.5KB = ~48KB
- Out-of-order buffer: 8 * ~1.5KB = ~12KB
- Metadata: ~200B
- **Total per TCB: ~92KB**
- **128 TCBs: ~11.5MB**

### CRITICAL: Total XNET + XSEC Static Memory
- TCP TCBs: ~11.5MB
- IP fragment table: ~1.1MB
- Other tables: ~130KB
- **Total: ~12.7MB**

This is **excessive** for the HP EliteBook x360 resource budget. The XNET subsystem alone would consume 12.7MB of BSS, competing with the 16GB total RAM budget where ~5GB is already committed.

### Recommended Reductions
1. Reduce TCP_TCB_MAX from 128 to 32: saves ~8.8MB (32 connections is sufficient for edge device)
2. Reduce TCP_REXMIT_QUEUE_MAX from 32 to 8: saves ~70KB per TCB
3. Reduce TCP ring buffers from 16KB to 8KB: saves 16KB per TCB
4. With all optimizations: 32 TCBs * ~30KB = ~960KB (down from 11.5MB)
5. Use bitmap for IP fragment received[]: saves 112KB
6. Reduce FRAG_TABLE_MAX from 16 to 8: saves ~550KB

**Optimized total: ~1.7MB** (down from 12.7MB, 87% reduction)

## Latency Budget: Crypto Operations per TLS Packet

### TLS 1.3 Record Processing (1400-byte payload)
| Operation | Est. Cycles (i5-8265U) | Est. Time @ 3.9GHz |
|-----------|----------------------|---------------------|
| AES-256-GCM encrypt (1400B) | ~25,000 | 6.4us |
| GHASH (1400B, bit-by-bit) | ~180,000 | 46us |
| GHASH (4-bit table) | ~45,000 | 11.5us |
| ChaCha20 (1400B) | ~8,000 | 2.1us |
| Poly1305 (1400B) | ~6,000 | 1.5us |
| SHA-256 (32B HMAC) | ~3,000 | 0.8us |
| **AES-GCM total (current)** | **~208,000** | **53us** |
| **AES-GCM total (optimized)** | **~73,000** | **19us** |
| **ChaCha20-Poly1305 total** | **~17,000** | **4.4us** |

### Observations
- GHASH is the dominant bottleneck in AES-GCM (87% of encrypt time)
- ChaCha20-Poly1305 is ~12x faster than current AES-GCM implementation
- At 1Gbps line rate (~80K packets/sec), AES-GCM saturates ~4 cores
- ChaCha20-Poly1305 can sustain 1Gbps on ~0.35 cores
- RECOMMENDATION: Default to ChaCha20-Poly1305 for TLS on this hardware

### X25519 Key Exchange Latency
- Montgomery ladder: 255 iterations * (4 fe_mul + 4 fe_sq + 6 fe_add/sub)
- ~1200 field multiplications per scalar mul
- Est: ~800,000 cycles = ~205us per key exchange
- Acceptable for TLS handshake (happens once per connection)

### Ed25519 Sign/Verify Latency
- Double scalar multiplication for verify: ~3000 field multiplications
- Est: ~2,000,000 cycles = ~512us per verify
- Acceptable for package signature verification

## Spinlock Contention Analysis

| Lock | Held During | Contention Risk |
|------|------------|-----------------|
| g_tcp_lock | Entire TCB table scan, segment processing | HIGH — single global lock for 128 TCBs |
| g_route_lock | Route lookup (O(32)) | LOW — fast scan, infrequent updates |
| g_frag_lock | Fragment reassembly | MEDIUM — held during memcpy of fragment data |
| g_gossip_lock | Member scan, event queue | MEDIUM — O(256) scans under lock |

### TCP Lock Contention (P1)
- Single `g_tcp_lock` protects all 128 TCBs
- Every incoming segment acquires this lock for the full TCB scan
- At high packet rates, this becomes a serialization bottleneck
- RECOMMENDATION (Sprint 4+): Per-TCB locks or hash-based striping
