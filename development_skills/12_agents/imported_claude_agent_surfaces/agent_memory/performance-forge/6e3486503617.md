# GEN.OS Kernel Engineering Skills Extraction (2026-03-31)

Source: 20 kernel files across core/, mm/, drv/, sched/, fs/

## Skills Catalog

### DEBUGGING (Skills 1-12)

**Skill 1: RDTSC Boot Telemetry**
- Category: debugging, performance
- Technique: Wrap each init step with `_rdtsc()` before/after, store delta in `g_boot_step_cycles[]`
- Pattern: `_step_tsc = _rdtsc(); /* init step */; g_boot_step_cycles[N] = _rdtsc() - _step_tsc;`
- Why: Cycle-accurate timing of every boot phase without any timer dependency (RDTSC is always available ring-0)
- Measurement: cycles / g_tsc_mhz = microseconds per step
- File: kernel/xenos/core/main.c:3481-3612 (14 steps instrumented)

**Skill 2: Progressive Serial Console (COM1 115200 8N1)**
- Category: debugging
- Technique: Initialize 16550 UART at 115200 baud before any subsystem. Three signal bytes: 'X' (entry.asm), 'S' (stack OK), 'K' (kmain reached)
- Pattern: `_outb(0x3F8, 'K')` proves handoff chain; `serial_early_puts()` for pre-PAL text
- Why: When GPU, VMM, heap are all uninitialized, serial is the ONLY output path
- Measurement: If 'K' appears on serial, the entire AetherBoot->entry.asm->kmain chain works
- File: kernel/xenos/core/main.c:3501-3507, 669-699

**Skill 3: Compile-Time Diagnostic Toggle (XENOS_SERIAL_DIAG)**
- Category: debugging, performance
- Technique: `#ifdef XENOS_SERIAL_DIAG` guards on verbose serial output; `#define XENOS_RELEASE` suppresses
- Why: Saves ~3.5ms at 115200 baud. Critical messages (FATAL, step timing) always print regardless
- Measurement: Serial output at 115200 = ~86us/char; 40-char message = 3.4ms
- File: kernel/xenos/core/main.c:35-54

**Skill 4: Panic Handler -- Zero-Dependency Crash Dump**
- Category: debugging
- Technique: Duplicated serial I/O functions in panic.c (no external calls). Static buffers only. No heap, no locks, no spinlocks
- Pattern: CLI -> capture all GPRs via inline asm -> RBP chain walk -> NMI broadcast -> serial dump -> HLT
- Why: If heap or lock subsystem caused the panic, calling them would deadlock or crash again
- Measurement: Stack trace depth (max 32 frames), all 16 GPRs + CR0/CR2/CR3/CR4/RFLAGS captured
- File: kernel/xenos/core/panic.c:1-200

**Skill 5: Reentrant Panic Guard**
- Category: debugging
- Technique: `volatile uint32_t s_panic_reentrant` checked with atomic test-and-set. If panic fires during panic, immediately HLT
- Why: Prevents infinite fault loop (panic -> page fault -> panic -> ...)
- Measurement: Single atomic check, zero overhead in non-panic path
- File: kernel/xenos/core/panic.c:55, main.c (OOM: `__sync_lock_test_and_set`)

**Skill 6: Watchdog Timer -- TSC-Based Hang Detection**
- Category: debugging
- Technique: Store `s_last_pet_tsc` via RDTSC. Timer ISR checks `elapsed = now - last`. Warning at 80%, panic at 100% of timeout
- Pattern: `watchdog_pet()` from scheduler tick; `watchdog_check()` from LAPIC ISR
- Why: Detects kernel hangs (infinite loops, deadlocks) within configurable timeout
- Measurement: timeout_ticks = (timeout_ms * tsc_hz) / 1000
- File: kernel/xenos/core/watchdog.c:1-197

**Skill 7: _Static_assert for Structure Size Validation**
- Category: debugging
- Technique: `_Static_assert(sizeof(page_frame_t) <= 64, "...")` at compile time
- Why: Catches struct layout drift that would corrupt memory at runtime. Zero runtime cost
- Examples: page_frame_t<=64, slab_t<=64, large_hdr_t<=32, xenos_fiber_t==40, nvme_sqe_t==64
- File: pmm.c:94, heap.c:97,120, main.c:323,353, nvme.c:169

**Skill 8: Weak Symbol Override for Progressive Integration**
- Category: debugging
- Technique: `extern uint32_t scheduler_fiber_rss(uint32_t) __attribute__((weak))` then `if (scheduler_fiber_rss) { ... }`
- Why: Allows subsystems to compile independently. OOM killer works with or without scheduler. Exception handler works with or without process model
- Measurement: Single pointer check per call (branch predictor handles it)
- File: oom.c:170-173, exception.c:104-107, main.c:557 (xenos_sched_yield)

**Skill 9: Stack Canary from RDRAND**
- Category: debugging, security
- Technique: `rdrand %0; setc %b1` at early boot. Replace compile-time `__stack_chk_guard` with hardware RNG value
- Why: Defeats stack buffer overflow exploits that rely on known canary. MUST execute before any -fstack-protector function
- Measurement: Single RDRAND instruction (~150 cycles on Whiskey Lake)
- File: kernel/xenos/core/main.c:3544-3557

**Skill 10: Exception State Capture with CR2**
- Category: debugging
- Technique: On vector 14 (#PF), read CR2 via `mov %%cr2, %0` to get the faulting linear address
- Pattern: exception_capture() saves ISR frame + CR2 + timestamp into static record pool
- Why: CR2 is only valid immediately after #PF; must be captured before any other page fault occurs
- File: kernel/xenos/core/exception.c:54-58, 94-98

**Skill 11: Hex Formatting Without sprintf**
- Category: debugging
- Technique: Manual hex conversion via shift-and-mask: `_hex_chars[(val >> (i*4)) & 0xF]`
- Why: In a freestanding kernel, sprintf/snprintf are unavailable. Panic handler needs hex output
- File: kernel/xenos/core/panic.c:106-119

**Skill 12: Compiler Barrier for MMIO Ordering**
- Category: debugging, hardware
- Technique: `__asm__ volatile("" ::: "memory")` as a compiler memory barrier without generating instructions
- Why: Ensures stores to ring buffer entries are visible before advancing the head pointer
- File: ps2.c:111, vdram_pf.c:66

### PERFORMANCE (Skills 13-24)

**Skill 13: O(1) Buddy Order via __builtin_ctzll / __builtin_clzll**
- Category: performance, memory
- Technique: `ctz(addr) - PAGE_SHIFT` gives alignment order in one instruction vs iterative probe loop
- Why: PMM init went from 25ms to 0.1ms (250x speedup) on 512MB RAM
- Measurement: ~180 iterations total for 512MB, single lock acquire per region
- File: kernel/xenos/mm/pmm.c:278-324

**Skill 14: Batched Lock Acquisition in PMM Init**
- Category: performance
- Technique: Acquire `g_pmm_lock` ONCE for entire contiguous range, not per-block insertion
- Why: Previous code did ~3600 lock round-trips for 20 regions. Now: 1 lock per region
- Measurement: Lock contention reduced from O(blocks) to O(regions)
- File: kernel/xenos/mm/pmm.c:353-402

**Skill 15: 2MB Large Pages for HHDM (Reduced TLB Pressure)**
- Category: performance
- Technique: Set PTE_PS bit at PD level to map 2MB pages instead of 4KB. For 128GB HHDM = 65,536 TLB entries vs 33,554,432
- Why: Each 4KB TLB miss costs ~10-30 cycles. 2MB pages reduce miss rate by 512x for linear access
- Measurement: TLB coverage: 512 entries * 2MB = 1GB with 2MB pages vs 512 * 4KB = 2MB with 4KB
- File: kernel/xenos/mm/vmm.c:273-310

**Skill 16: PTE_GLOBAL for Kernel Mappings (Survives CR3 Switch)**
- Category: performance
- Technique: Set PTE_GLOBAL on HHDM and kernel ELF pages. These survive TLB flush on CR3 write
- Why: Context switch writes CR3 (~200 cycles + TLB flush). Global pages avoid re-walking kernel PTEs
- Measurement: On process switch, user TLB flushed but kernel entries retained
- File: kernel/xenos/mm/vmm.c:307, 352

**Skill 17: CPUID-Based LAPIC Calibration (Zero-Wait)**
- Category: performance
- Technique: CPUID leaf 0x15 gives TSC/crystal ratio + crystal Hz directly. No calibration delay needed
- Why: Saves 10ms (was HPET 10ms loop, then 1ms, then 0ms with CPUID). On Whiskey Lake: 25MHz crystal
- Measurement: CPUID 0x15: ECX=25000000 (crystal Hz), EAX/EBX = TSC ratio
- File: kernel/xenos/drv/lapic.c:226-289

**Skill 18: HPET 1ms Calibration Fallback (Was 10ms)**
- Category: performance
- Technique: HPET hardware counter for precise 1ms delay. 1ms at 25MHz/div16 = 1562 ticks = 0.06% error
- Why: Saves 9ms vs 10ms loop. Hardware counter eliminates volatile loop inaccuracy
- File: kernel/xenos/drv/lapic.c:307-349

**Skill 19: Slab Allocator O(1) Alloc/Free**
- Category: performance, memory
- Technique: Freelist embedded in free slots (first 8 bytes = next pointer). Pop = O(1), Push = O(1)
- Pattern: `ptr = slab->freelist; slab->freelist = *(void**)ptr;`
- Why: No heap fragmentation for common sizes. 10 size classes: 8/16/32/64/128/256/512/1K/2K/4K
- Measurement: Single pointer dereference + store per alloc/free
- File: kernel/xenos/mm/heap.c:156-329

**Skill 20: Partial Slab Promotion (Move-to-Front)**
- Category: performance
- Technique: When allocating, if a partial slab is found deeper in the list, move it to front
- Pattern: `prev->next = slab->next; slab->next = cache->slabs; cache->slabs = slab;`
- Why: Next allocation for same size class hits the partial slab immediately (cache-friendly)
- File: kernel/xenos/mm/heap.c:315-319

**Skill 21: NVMe CQ Phase Bit Toggle**
- Category: performance, hardware
- Technique: CQ completion detection via phase bit (P bit toggles each time head wraps). No doorbell read needed
- Why: Avoids MMIO read for completion polling -- phase bit is in the CQE already in cache
- Measurement: Save one 32-bit MMIO read (~100ns) per completion
- File: kernel/xenos/drv/nvme.c:183-194

**Skill 22: Lazy Preemption (Timer Flag + Safe Point Yield)**
- Category: performance, scheduler
- Technique: LAPIC timer ISR sets `need_resched` flag. Actual context switch only fires at `preempt_enable()` boundaries
- Why: Avoids context switch in middle of critical sections. Timer sets intent; safe point executes
- Measurement: ~2 cycles for flag check on preempt_enable() fast path
- File: kernel/xenos/sched/contra_rotation.c:166-186

**Skill 23: Naked Function for Zero-Overhead Context Switch**
- Category: performance, scheduler
- Technique: `__attribute__((naked))` on xenos_fiber_switch() -- pure inline asm, no prologue/epilogue
- Pattern: Push 6 callee-saved regs, swap RSP, conditionally reload CR3, pop regs, ret
- Why: No compiler-generated stack frame overhead. Exact control over register save/restore
- Measurement: Kernel-to-kernel: ~50 cycles. Process switch: ~250 cycles (CR3 write + TLB flush)
- File: kernel/xenos/core/main.c:412-479

**Skill 24: Contra-Rotating Thermal Scheduler**
- Category: performance, scheduler
- Technique: Three-phase cycle: COMPUTE (60-80C) -> COOLDOWN -> LOAD. Thermal ceiling IS the scheduling primitive
- Pattern: Device A computes while B cools and C loads -- no idle time
- Why: On 15W TDP laptop, thermal throttling destroys throughput. Overlap compute/cool/load across devices
- Measurement: PHASE_MIN_COMPUTE_NS=2s, COOLDOWN=500ms, LOAD=1s. Poll thermal every 500ms
- File: kernel/xenos/sched/contra_rotation.c:1-148

### HARDWARE DRIVER PATTERNS (Skills 25-37)

**Skill 25: Volatile MMIO Pointer Pattern**
- Category: hardware
- Technique: Cast HHDM-mapped physical address to `volatile uint32_t*`. All register reads/writes through volatile
- Pattern: `ctx->mmio[off >> 2u]` for 32-bit register at 4-byte-aligned offsets
- Why: Prevents compiler from caching, reordering, or eliminating hardware register accesses
- File: lapic.c:78-84, hpet.c:43-49, iommu_vtd.c:55-71, tpm2_tis.c:86-102

**Skill 26: HPET 64-bit Counter Double-Read (Roll-Over Protection)**
- Category: hardware
- Technique: Read lo, hi, lo again. If lo2 < lo1, carry into hi occurred -- re-read hi
- Pattern: `lo1=rd32(off); hi=rd32(off+4); lo2=rd32(off); if(lo2<lo1) hi=rd32(off+4); return (hi<<32)|lo2`
- Why: 64-bit MMIO counter read as two 32-bit reads can tear across carry boundary
- Measurement: Extra 32-bit read only on carry (once per ~4.29 billion ticks)
- File: kernel/xenos/drv/hpet.c:56-64

**Skill 27: PCI Config Space via I/O Ports (Mechanism #1)**
- Category: hardware
- Technique: Write 32-bit address to 0xCF8, read/write data at 0xCFC. Sub-dword access via shift-and-mask
- Pattern: 16-bit read: `dword = read32(reg & 0xFC); return (dword >> ((reg & 0x02) << 3)) & 0xFFFF`
- Why: Universally supported PCI access method. No MMIO config space needed for enumeration
- File: kernel/xenos/drv/pci.c:40-100

**Skill 28: PCI BAR Enumeration + Multi-Function Detection**
- Category: hardware
- Technique: Read vendor ID at fn 0. If 0xFFFF, skip device. Check header type bit 7 for multi-function
- Pattern: Scan bus 0..MAX, dev 0..31, fn 0..7 (only if multi-function bit set)
- Why: Correctly handles both single-function and multi-function PCI devices
- File: kernel/xenos/drv/pci.c:155-195

**Skill 29: TPM 2.0 TIS FIFO State Machine**
- Category: hardware
- Technique: COMMAND_READY -> write bytes to DATA_FIFO -> poll EXPECT -> GO -> poll DATA_AVAIL -> read response
- Pattern: Bounded polling loops (5M iterations for STS, 2B for self-test)
- Why: TPM command/response protocol requires strict state machine adherence. Bounded loops prevent boot hang
- File: kernel/xenos/drv/tpm2_tis.c:147-200

**Skill 30: Big-Endian Wire Serialization (TPM Commands)**
- Category: hardware
- Technique: Manual byte serialization with shift/mask: `buf[0] = (val >> 24) & 0xFF` etc.
- Why: TPM 2.0 wire format is big-endian; x86 is little-endian. Type-punning would give wrong byte order
- File: kernel/xenos/drv/tpm2_tis.c:28-30 (design note), command builder helpers

**Skill 31: EC Register I/O with IBF/OBF Polling**
- Category: hardware
- Technique: Wait for IBF=0 (input buffer empty), then send command/register byte. Wait for OBF=1, then read data
- Pattern: `ec_wait_ibf() -> outb(CMD, READ) -> ec_wait_ibf() -> outb(DATA, reg) -> ec_wait_obf() -> inb(DATA)`
- Why: HP EliteBook x360 EC is on LPC bus at ports 0x62/0x66. Without polling, data corruption occurs
- File: kernel/xenos/drv/elitebook_x360.c:62-100

**Skill 32: Thermal Fan Curve with Emergency P-State Throttle**
- Category: hardware, performance
- Technique: Linear fan ramp 50-70C. Above 85C: write MSR_IA32_PERF_CTL to force 800MHz P-state
- Pattern: `wrmsr(0x199, (rdmsr() & 0xFFFF0000) | 0x00000800)` -- ratio=8 = 800MHz
- Why: On slim chassis laptop, thermal runaway can damage hardware. P-state cap is last resort
- Measurement: < 50C=off, 50-70C=30-80% linear, 70C+=100%, 85C+=emergency throttle
- File: kernel/xenos/drv/elitebook_x360.c:149-200

**Skill 33: IOMMU VT-d DMAR Table Parsing**
- Category: hardware
- Technique: Walk XSDT to find "DMAR" signature. Parse DRHD entries for IOMMU MMIO base addresses
- Pattern: Variable-length remapping structures with type/length prefix
- Why: DMA isolation requires programming IOMMU root/context tables at the physical addresses found in DMAR
- File: kernel/xenos/drv/iommu_vtd.c:100-200

**Skill 34: NVMe Controller Reset + Admin Queue Setup**
- Category: hardware
- Technique: Disable controller (CC.EN=0), wait CSTS.RDY=0, set AQA/ASQ/ACQ, configure CC, enable, wait CSTS.RDY=1
- Why: NVMe spec requires full controller reset before any command. Race condition if skipped
- Measurement: NVME_RESET_TIMEOUT_MS = 5000 (5s max for controller ready)
- File: kernel/xenos/drv/nvme.c:56-117

**Skill 35: PS/2 SPSC Ring Buffer (256-entry, Power-of-2)**
- Category: hardware, performance
- Technique: 256-entry ring with head/tail. Mask with `& 0xFF`. ISR produces, main loop consumes. No locks
- Why: Power-of-2 size makes modular arithmetic a single AND instruction. Single-producer/single-consumer is lock-free
- Measurement: Zero contention. Overflow drops newest event (head not advanced)
- File: kernel/xenos/drv/ps2.c:96-113

**Skill 36: I/O Port Delay via Unused Port Read**
- Category: hardware
- Technique: `inb(0x80)` -- read POST diagnostic port as a tiny I/O delay
- Why: Some legacy hardware (8042 PS/2 controller) requires a brief delay between I/O operations
- Measurement: ~1us delay per inb(0x80) on modern x86
- File: kernel/xenos/drv/ps2.c:56-59

**Skill 37: ACPI Table Checksum Validation**
- Category: hardware
- Technique: Sum all bytes of table; valid if result == 0 (mod 256). Applied to RSDP (first 20 bytes) and all XSDT entries
- Why: Firmware tables in memory can be corrupted. Checksum catches single-bit errors
- File: kernel/xenos/drv/xacpi.c:73-79, 140

### MEMORY MANAGEMENT (Skills 38-47)

**Skill 38: HHDM (Higher-Half Direct Map)**
- Category: memory
- Technique: Map all physical RAM at virtual offset `HHDM_BASE = 0xFFFF888000000000`. Access any phys addr as `(void*)(phys + HHDM_BASE)`
- Why: Eliminates temporary mappings for physical memory access. PMM page descriptors, page tables, DMA buffers all accessible via simple pointer arithmetic
- File: vmm.c:11, pmm.c:28, heap.c:25 (all use HHDM for physical page access)

**Skill 39: In-Page Metadata (page_frame_t in First 64 Bytes)**
- Category: memory
- Technique: Store page_frame_t descriptor in the first 64 bytes of each free 4KB page itself
- Why: No separate metadata array needed. Saves RAM proportional to physical memory size. Fits in first cache line
- Measurement: _Static_assert(sizeof(page_frame_t) <= 64) -- one cache line
- File: kernel/xenos/mm/pmm.c:87-95

**Skill 40: Buddy Coalescing on Free**
- Category: memory
- Technique: On free, XOR address with buddy address `(phys ^ (PAGE_SIZE << order))`. Search free list for buddy. If found, merge and try order+1
- Why: Prevents fragmentation. Two adjacent order-0 blocks merge to order-1, etc., up to order-10 (4MB)
- Measurement: Worst case: O(MAX_ORDER) = 11 iterations per free
- File: kernel/xenos/mm/pmm.c:495-541

**Skill 41: Zone-Aware Allocation (DMA32 + NORMAL)**
- Category: memory
- Technique: Two zones: DMA32 (0-4GB) and NORMAL (>4GB). Fallback: if NORMAL empty, try DMA32
- Why: Legacy devices require DMA addresses below 4GB. Preserves DMA32 pages for those devices
- Pattern: Split buddy blocks at 4GB boundary to prevent cross-zone blocks
- File: kernel/xenos/mm/pmm.c:64-68, 353-368

**Skill 42: Static Pool Allocation (No Malloc at Init)**
- Category: memory
- Technique: All global state as static arrays: `g_fibers[16]`, `g_processes[16]`, `g_jobs[16]`, `g_caches[10]`, `s_fd_table[256]`
- Why: During early boot, heap is not available. Static allocation guarantees deterministic memory layout
- Examples: XENOS_FIBERS_MAX=16, NUM_SLAB_CLASSES=10, XVFS_MAX_FD=256, XK_SYS_MAX=64
- File: main.c:232,332,349, heap.c:127, xvfs.c:75, ps2.c:99

**Skill 43: Slab Header Magic for Use-After-Free Detection**
- Category: memory, debugging
- Technique: `slab->magic = SLAB_MAGIC` on create, `slab->magic = 0` on free. Check in kfree
- Pattern: `if (slab->magic != SLAB_MAGIC) { log + ignore }` -- catches corruption and double-free
- Why: Without heap metadata guards, use-after-free is silent data corruption
- File: kernel/xenos/mm/heap.c:169, 197, 366-371

**Skill 44: OOM Handler -- Tiered Recovery**
- Category: memory
- Technique: Step 1: reclaim empty slabs (LIFO order). Step 2: retry allocation. Step 3: kill lowest-priority fiber. Step 4: panic
- Pattern: OOM score = (RSS * 10) / priority. PID 0/1 never killed. Reentrant-safe via atomic guard
- Why: Graceful degradation before panic. Kill the biggest, least important process first
- Measurement: Target = 2x requested pages for headroom
- File: kernel/xenos/mm/oom.c:239-349

**Skill 45: vDRAM Page Fault Fast Path (Zero Page + Async Fetch)**
- Category: memory, performance
- Technique: On #PF in vDRAM window (PML4[272]), immediately map zero page. Enqueue async fetch to SPSC ring
- Why: CPU would loop on #PF without immediate resolution. Zero page unblocks execution; real data arrives later
- Measurement: Fast path: single ring enqueue (~20 cycles). Fetch thread replaces zero page with real data
- File: kernel/xenos/mm/vdram/vdram_pf.c:47-74

**Skill 46: 5-Tier Memory Migration (Advisory Scoring)**
- Category: memory
- Technique: Score-based migration: promote if score<0.3, demote if >0.7, evict if >0.9. Tiers: Hot DRAM -> Warm -> NVMe cache -> Remote DRAM -> Object store
- Why: On 16GB laptop, AI model (2.5GB) + OS must coexist. Tiered memory extends effective capacity
- Pattern: Pinned pages skipped. Stale advisory entries skipped
- File: kernel/xenos/mm/vdram/tier_migration.c:46-150

**Skill 47: Demand-Zero via HHDM Identity Bootstrap**
- Category: memory
- Technique: Before HHDM, AetherBoot provides identity map (phys==virt). PMM uses `g_hhdm_live` flag to switch addressing mode
- Pattern: `_phys_to_pf()` returns `g_hhdm_live ? (phys + HHDM_BASE) : phys`
- Why: Chicken-and-egg: PMM needs page access to build free lists, but HHDM needs PMM for page table pages
- File: kernel/xenos/mm/pmm.c:111-129, 559-561

### SECURITY HARDENING (Skills 48-53)

**Skill 48: W^X Enforcement in VMM**
- Category: security, memory
- Technique: If both VM_WRITE and VM_EXEC are set, silently drop VM_EXEC. Writable pages are never executable
- Why: Prevents code injection via writable+executable pages. Hardware NX enforced via EFER.NXE
- File: kernel/xenos/mm/vmm.c:171-175

**Skill 49: CR4 Hardening (SMEP/SMAP/UMIP/OSXSAVE)**
- Category: security
- Technique: CPUID-gated CR4 bit enablement. SMEP: prevent kernel executing user pages. SMAP: prevent kernel accessing user data. UMIP: block SGDT/SIDT from user mode
- Pattern: Check CPUID leaf 07H bits, then OR into CR4
- Why: Each mitigation closes a different kernel exploitation primitive
- File: kernel/xenos/core/main.c:3653-3677

**Skill 50: Speculative Execution Mitigations (IBRS/STIBP/SSBD)**
- Category: security, performance
- Technique: Write IA32_SPEC_CTRL MSR with IBRS|STIBP|SSBD bits. IBRS restricts indirect branch prediction. STIBP prevents sibling hyperthread speculation. SSBD blocks speculative store bypass
- Why: Mitigates Spectre v2, SpectreNG, SSBD on Whiskey Lake silicon
- Measurement: ~10-30% overhead on indirect branches (acceptable for security)
- File: kernel/xenos/core/main.c (kmain step 4.6 hardening block)

**Skill 51: IST1 Stack for Double Fault (#DF)**
- Category: security, debugging
- Technique: Dedicated 4KB IST1 stack for #DF handler, separate from main kernel stack
- Why: #DF fires when stack overflows during fault handling. Same stack = triple fault = reboot. IST provides clean stack
- Measurement: 4KB static allocation, configured in TSS
- File: kernel/xenos/core/main.c:147-154

**Skill 52: Exception Write-State CS/RIP Validation**
- Category: security
- Technique: Reject writes to exception state where CS RPL==0 (kernel) or RIP > 0x7FFFFFFFFFFF (kernel space)
- Why: Prevents privilege escalation via debugger exception state modification
- File: kernel/xenos/core/exception.c:186-200

**Skill 53: EFER.NXE Before First CR3 Load**
- Category: security
- Technique: Enable NXE in EFER MSR at step 4.5 (before VMM step 5), not at syscall_init (step 9)
- Why: Closes 4-step window where NX bits in PTEs would be ignored. Re-enable in step 9 is idempotent
- File: kernel/xenos/core/main.c:3635-3651

### BUILD & TOOLING (Skills 54-57)

**Skill 54: Freestanding C Compilation Flags**
- Category: build
- Technique: `-ffreestanding -mno-red-zone -fno-stack-protector -std=c11 -Wall -Wextra`
- Why: -ffreestanding removes libc dependency. -mno-red-zone prevents ISR stack corruption. -fno-stack-protector (initially; later enabled with custom __stack_chk_guard)
- File: All kernel .c files (compile header comments)

**Skill 55: PAL_FREESTANDING Guard Pattern**
- Category: build
- Technique: `#ifndef PAL_FREESTANDING / #define PAL_FREESTANDING / #endif` at top of every kernel .c file
- Why: Ensures PAL (Platform Abstraction Layer) headers produce freestanding-safe definitions. Prevents accidental libc pulls
- File: Every kernel source file

**Skill 56: Freestanding memset/memcpy Reimplementation**
- Category: build
- Technique: Local `xa_memset()`, `vtd_memset()`, `tis_memset()`, `exc_memcpy()` etc. in each translation unit
- Why: No libc. Each driver is self-contained. Byte-by-byte loop is correct everywhere; optimized versions (rep stosq) added selectively
- File: xacpi.c:61-65, iommu_vtd.c:41-45, tpm2_tis.c:72-76, exception.c:40-51

**Skill 57: Compile-Time Size Override via #ifndef**
- Category: build
- Technique: `#ifndef XENOS_FIBERS_MAX / #define XENOS_FIBERS_MAX 16u / #endif`
- Why: Allows test builds to override constants without editing source. E.g., reduce fiber count for constrained test
- File: contra_rotation.c:80-82, various headers
