## 1. TCP Performance (net/xnet/transport/tcp.c)

### 1.1 Initial Congestion Window (initcwnd)

**Line 464**: `t->cwnd = 10u * t->snd_mss;`

CORRECT per RFC 6928. With MSS=1460, cwnd=14,600 bytes (10 segments).
This allows bursting 10 segments in the first RTT, which is the modern
standard. Google measured this as optimal for typical web page loads.

**Verdict: WITHIN BUDGET. Correct RFC 6928 implementation.**

### 1.2 Jacobson/Karels RTO Algorithm (RFC 6298)

**Lines 433-456**: `tcp_update_rtt()`

Analysis of the implementation:

```
First measurement:
  SRTT   = R'           (line 436) -- CORRECT per RFC 6298 s2.2
  RTTVAR = R' / 2       (line 437) -- CORRECT per RFC 6298 s2.2

Subsequent measurements:
  RTTVAR = (3 * RTTVAR + |SRTT - R'|) / 4   -- CORRECT (beta = 1/4)
  SRTT   = (7 * SRTT + R') / 8              -- CORRECT (alpha = 1/8)
  RTO    = SRTT + 4 * RTTVAR                -- CORRECT per RFC 6298 s2.3
```

RTO clamped to [200ms, 60s] at lines 453-454. RFC 6298 recommends minimum
1 second for the initial RTO, which is set at line 76: `TCP_RTO_INIT_NS = 1s`.
The 200ms minimum is aggressive but acceptable for a LAN-oriented edge device.

**Minor concern**: Integer arithmetic with uint64_t nanoseconds. The expression
`3 * t->rttvar_ns` could theoretically overflow if rttvar_ns > 6.1e18 ns
(~194 years). Not a practical concern.

**Verdict: WITHIN BUDGET. Textbook-correct Jacobson/Karels.**

### 1.3 Slow Start and Congestion Avoidance

**Lines 475-488**: `tcp_congestion_on_ack()`

- Slow start (cwnd < ssthresh): `cwnd += acked_bytes` -- CORRECT, doubles per RTT
- Congestion avoidance (cwnd >= ssthresh): `cwnd += MSS^2 / cwnd` -- CORRECT Reno AIMD
- Cap at line 487: `cwnd <= TCP_SNDBUF_SIZE (16KB)` -- This is a problem (see below)

**FINDING (P1): cwnd cap at 16KB limits throughput**

Line 487 caps cwnd at TCP_SNDBUF_SIZE (16KB). This means the congestion window
can never grow beyond ~11 segments. For a connection with 100ms RTT:

  Max throughput = cwnd / RTT = 16384 / 0.1 = 163 KB/s

This is far below the ~125 MB/s that a GbE link could sustain. For local k3s
service-to-service communication on the EliteBook (RTT < 1ms), this gives
~16 MB/s, which is adequate. But for any external connection, this is a
severe throughput limiter.

Recommendation: Cap cwnd at min(65535, peer_window) instead of send buffer size.
The send buffer limits how much data the application can enqueue, not how much
the congestion window should allow in flight.

### 1.4 Fast Retransmit

**Lines 805-821**: Triple duplicate ACK triggers fast retransmit.

- Detects 3 dup ACKs correctly (line 808)
- Calls `tcp_congestion_on_loss()` which sets cwnd = 1*MSS (line 498)
- Retransmits the segment at snd_una

**FINDING (P2): No fast recovery (RFC 5681 s3.2)**

After fast retransmit, cwnd is set to 1*MSS (timeout-style behavior). RFC 5681
specifies fast recovery: cwnd = ssthresh + 3*MSS, then inflate cwnd by MSS for
each additional dup ACK. The current implementation effectively falls back to
slow start after triple dup ACK, which is correct but suboptimal. This halves
throughput during loss events compared to proper Reno fast recovery.

### 1.5 Ring Buffer Byte-by-Byte Copy

**Lines 106-126**: `ring_write()` and `ring_read()` copy one byte at a time
with modular arithmetic per byte.

For a 1460-byte MSS segment: 1460 iterations, each with a modulo operation.
The modulo on a power-of-2 buffer could be optimized to a bitwise AND, but
the fundamental problem is byte-by-byte copying.

**Performance estimate**: ~3 cycles/byte (load + store + mod) = ~4,380 cycles
per MSS segment. A two-phase memcpy would be ~0.25 cycles/byte with 64-byte
cache line streaming = ~365 cycles. That is a 12x improvement on the copy path.

At 80K packets/sec (GbE), this is 350M vs 29M cycles/sec. On a 3.9GHz
i5-8265U, that is 9% vs 0.7% of a single core.

**Verdict: P1. Two-phase memcpy is a simple, high-impact fix.**

### 1.6 Global TCP Lock Contention

**Line 231**: Single `g_tcp_lock` protects the entire TCB table.

Every TCP operation (input, send, recv, timer tick) acquires this lock.
`xnet_tcp_input()` holds the lock from line 653 through line 905 --
the ENTIRE state machine processing, including checksum verification
(line 641-648... actually the checksum is computed before the lock at
line 641, but the memcpy into seg_copy IS under the lock at line 644).

Wait -- re-reading: the checksum block at lines 641-648 is INSIDE the
lock (acquired at line 653). Actually no: the lock is acquired at 653,
and the checksum block at 641-648 is BEFORE the lock. Let me re-read...

Lines 640-653:
```
    /* Verify checksum */
    {
        uint8_t seg_copy[...];
        ...
    }
    ...
    pal_spin_lock(&g_tcp_lock);
```

CORRECT: checksum is computed OUTSIDE the lock. Good design -- the expensive
O(n) checksum doesn't hold the lock.

However, the state machine processing (lines 653-905) holds the lock for
the entire duration, including `tcp_send_segment()` calls which invoke
`xnet_ip4_output()`. The comment at line 30 says "No lock held during
actual IP send" but the code DOES hold g_tcp_lock during tcp_send_segment
calls within the state machine (e.g., line 769, 828, 835).

**FINDING (P1): Lock held during IP output contradicts design intent**

The lock is held during all tcp_send_segment calls inside xnet_tcp_input.
This serializes all TCP processing with IP output. For Sprint 3 with
128 TCBs this is manageable, but it will be a scaling bottleneck.

### 1.7 TCB Memory Budget

Per-TCB breakdown:
- 2x ring buffers: 2 * (16384 + 12) = 32,792 bytes
- Retransmit queue: 32 * ~1488 bytes = ~47,616 bytes
- OOO buffer: 8 * ~1476 bytes = ~11,808 bytes
- Metadata (seq vars, timers, options): ~120 bytes
- **Total per TCB: ~92,336 bytes (~90 KB)**
- **128 TCBs: ~11.5 MB**

This is the single largest static allocation in XNET. See Memory Budget
section below for full analysis.

## 2. Crypto Throughput (sec/xsec/crypto/)

### 2.1 AES-256-GCM: CT S-Box (aes_gcm.c)

The constant-time S-box at line 58-69 scans all 256 entries per lookup.
This was a P0 security fix (GS-S3-P0-001) and is the correct approach
for software-only AES when cache-timing attacks are a threat.

**Throughput analysis (reiterated from prior audit)**:
- 276 sbox calls/block * 256 iterations = 70,656 iterations/block
- ~22,080 cycles/byte = ~177 KB/s at 3.9 GHz
- For a 16KB TLS record: ~5.7 ms per record encrypt

**Sprint 3 verdict: ACCEPTABLE as transitional. AES-NI is P0 for Sprint 4.**

The i5-8265U (Whiskey Lake) has AES-NI (CPUID.01H:ECX bit 25). With AES-NI:
- AESENC/AESENCLAST: ~4 cycles/byte = ~975 MB/s
- This is a 5,520x speedup. Non-negotiable for Sprint 4.

### 2.2 ChaCha20 Quarter Round (chacha20.c)

**Lines 29-35**: Standard macro-based quarter round.

```c
#define ROTL32(x, n)  (((x) << (n)) | ((x) >> (32u - (n))))
#define QR(a, b, c, d) ...
```

Analysis:
- 8 quarter rounds per double round, 10 double rounds = 80 QR invocations
- Each QR: 4 adds, 4 XORs, 4 rotates = 12 ALU operations
- Total: 960 ALU operations per 64-byte block = 15 ops/byte
- At ~1 cycle/op: ~15 cycles/byte = ~260 MB/s at 3.9 GHz

**Vectorization analysis**: The column rounds (lines 83-86) operate on
independent state words (0,4,8,12), (1,5,9,13), (2,6,10,14), (3,7,11,15).
These are 4-way parallel and map naturally to SSE2/AVX2 SIMD:
- `_mm_add_epi32`, `_mm_xor_si128`, `_mm_shuffle_epi32` for rotates
- Expected speedup with SSE2: ~3-4x (limited by diagonal round shuffles)
- With SSE2: ~4-5 cycles/byte = ~780 MB/s

The diagonal rounds (lines 88-91) require lane permutations which cost
1-2 shuffle instructions each. Still SIMD-friendly.

**Sprint 3 verdict: WITHIN BUDGET for software scalar path. SIMD is P2 for S4.**

The scalar implementation is clean and correct. The `ROTL32` macro compiles
to a single `rol` instruction on x86_64 with any optimization level.

### 2.3 Poly1305 MAC (chacha20.c:162-316)

The 3-limb accumulator with `__uint128_t` multiplication is the right approach
for x86_64. Clang compiles u128 multiplication to `mul`/`imul` pairs efficiently.

**FINDING (P2 -- resolved): poly_scratch is now stack-allocated**

Line 374: `uint8_t poly_scratch[POLY_SCRATCH_MAX];` is a LOCAL variable.
The prior audit flagged this as static (thread-unsafe). The current code
allocates ~32KB on the stack per AEAD call. This is:
- Thread-safe (each call gets its own stack frame)
- Stack-heavy: 32KB per call. XENOS kernel stack is typically 8-16KB.

**FINDING (P1): 32KB stack allocation may overflow kernel stack**

If POLY_SCRATCH_MAX = 32,816 bytes is allocated on a kernel stack of 8KB
or even 16KB, this WILL overflow. The TLS layer calls this function for
every record, and if the call chain is:

  tls13_process_record -> chacha20_poly1305_core -> poly_scratch[32KB]

...the total stack depth easily exceeds 16KB with the caller's frames.

**Recommendation**: Streaming Poly1305 (process blocks incrementally without
buffering the entire MAC input). This eliminates the 32KB scratch entirely.
Alternative: Use a caller-provided buffer or a per-connection scratch area
within the TCB/TLS session structure.

### 2.4 SHA-256 (sha256.c)

Standard FIPS 180-4 implementation. W[64] = 256 bytes on stack.

- 64 rounds, each: ~20 ALU operations
- ~1280 ops per 64-byte block = 20 ops/byte
- ~20 cycles/byte = ~195 MB/s at 3.9 GHz

SHA-NI (CPUID.07H:EBX bit 29) is available on Whiskey Lake:
- SHA256RNDS2, SHA256MSG1, SHA256MSG2: ~4 cycles/byte
- 5x speedup. P2 for Sprint 4.

**Sprint 3 verdict: WITHIN BUDGET. Standard implementation, no bugs.**

### 2.5 Ed25519 (referenced from prior audit)

- Double scalar multiplication: ~512 us per verify
- Montgomery ladder is constant-time (critical for key operations)
- fe10_mul uses int64_t intermediates, which is correct for 10-limb representation

**Sprint 3 verdict: WITHIN BUDGET. One-time cost per TLS connection.**

## 3. DNS Resolver (net/xnet/app/dns.c)

### 3.1 Cache Lookup Complexity

**Lines 218-226**: `dns_cache_find()` is a LINEAR SCAN over all 128 entries.

```c
for (uint32_t i = 0; i < DNS_CACHE_SIZE; i++) {
    if (!e->in_use) continue;
    if (e->type == type && xnet_strcmp(e->name, name) == 0) return e;
}
```

This is O(n) where n = DNS_CACHE_SIZE = 128. Each iteration performs a
string comparison (O(m) where m = hostname length, max 256).

Worst case: 128 * 256 = 32,768 byte comparisons per cache lookup.
At ~1 cycle/byte: ~33K cycles = ~8.5 us.

**Verdict: ACCEPTABLE for Sprint 3.** 128 entries is small enough that
linear scan is practical. The cache is consulted once per DNS resolution,
and the result is dominated by network RTT (milliseconds).

For Sprint 5+ (if cache grows to 1024+), switch to a hash table:
- FNV-1a hash of (name, type) -> bucket chain
- O(1) average case, O(n) worst case with collision chains
- 256-bucket table with 4-entry chains covers 1024 entries

### 3.2 Cache Allocation (Eviction)

**Lines 228-241**: `dns_cache_alloc()` finds a free slot, or evicts the
oldest entry (by cached_at_ns).

This is also O(n). It scans all 128 entries to find the oldest. Combined
with the find operation, a cache miss + insert requires 2 full scans = 256
iterations. Still under 20 us.

**FINDING (P3): No LRU eviction -- oldest-first may evict frequently-used entries**

The eviction policy evicts the entry with the oldest `cached_at_ns`, not
the least recently USED. A frequently-queried hostname with a long TTL
could be evicted in favor of a recently-cached but rarely-used entry.

For Sprint 3 with 128 entries, this is unlikely to cause problems. For
larger caches, consider LRU by tracking last-access time.

### 3.3 TTL Management

**Lines 552-566** (xnet_dns_resolve) and **609-623** (xnet_dns_cache_expire):

- On cache hit: checks `age_ns < ttl_ns` (correct)
- On expiry: sets `in_use = false` (correct)
- Periodic expiry via `xnet_dns_cache_expire()` called from `xnet_tick()`
- Negative caching: NXDOMAIN entries cached with 60s TTL (correct)
- Zero TTL: not cached (line 494: `if (ttl > 0)`) (correct)

**Verdict: CORRECT. TTL management is sound.**

### 3.4 DNS Cache Entry Size

Approximate per-entry size:
- `bool in_use`: 1 byte (+ 3 padding)
- `char name[256]`: 256 bytes
- `uint16_t type`: 2 bytes (+ 2 padding)
- `uint32_t ttl`: 4 bytes
- `uint64_t cached_at_ns`: 8 bytes
- `bool nxdomain`: 1 byte (+ 3 padding)
- `xnet_ipv4_t ipv4`: 4 bytes
- `xnet_ipv6_t ipv6`: 16 bytes
- **Total: ~300 bytes per entry** (with alignment padding)
- **128 entries: ~38.4 KB**

This is modest and well within budget.

### 3.5 Spin-Wait Polling

**Lines 450-457**: The DNS resolver busy-waits for responses:

```c
while (pal_time_now_ns() < deadline) {
    ...
    if (got_it) goto done;
    pal_thread_yield();
}
```

This burns CPU while waiting for DNS responses (up to 5 seconds total).
`pal_thread_yield()` helps but still consumes scheduling cycles.

**FINDING (P2): Busy-wait DNS resolution wastes CPU**

For Sprint 3 this is acceptable (blocking stub resolver). Sprint 4+
should implement an event-driven resolver using the PAL channel/IPC
primitives to sleep-wait on a response event.

## 4. XPKG Dependency Resolution (pkg/xpkg/solver/dep_resolve.c)

### 4.1 Topological Sort Complexity

**Lines 224-281**: `topo_sort()` implements iterative DFS with on_stack
markers for cycle detection.

- Outer loop: O(V) to visit all unvisited nodes
- Inner loop: each edge visited exactly once across all DFS traversals
- **Total: O(V + E)** -- optimal

With XPKG_MAX_GRAPH_NODES = 512 and XPKG_MAX_DEPS_PER_PKG = 16:
- Max V = 512, max E = 512 * 16 = 8,192
- Worst case: 8,704 iterations

**Verdict: WITHIN BUDGET. Optimal time complexity.**

### 4.2 Quadratic Behavior Risk

#### 4.2.1 Conflict Detection: O(V^2)

**Lines 194-218**: `detect_conflicts()` compares every pair of nodes:

```c
for (uint32_t i = 0; i < g->node_count; i++) {
    for (uint32_t j = i + 1; j < g->node_count; j++) {
        if (xpkg_strcmp(g->nodes[i].name, g->nodes[j].name) == 0) { ... }
    }
}
```

At V = 512: 512 * 511 / 2 = 130,816 string comparisons.
Each comparison: O(64) for 64-byte name field.
Total: ~8.4M byte comparisons = ~8.4M cycles = ~2.2 ms at 3.9 GHz.

**Verdict: ACCEPTABLE for Sprint 3.** 2.2 ms is well within the package
installation latency budget (seconds). But this is an O(V^2) algorithm
hiding in a linear pipeline.

Fix for Sprint 4: Sort nodes by name (O(V log V)), then linear scan for
adjacent duplicates (O(V)). Total: O(V log V) instead of O(V^2).

#### 4.2.2 graph_find_node: O(V) per call

**Lines 84-92**: Linear scan of nodes by name. Called from `build_graph()`
for each dependency edge. With V nodes and E edges:
- Total calls: O(E)
- Cost per call: O(V)
- Total: O(V * E)

At max capacity: 512 * 8192 = 4.2M iterations. Each with a strcmp.
~268M byte comparisons = ~69 ms.

**FINDING (P2): graph_find_node creates O(V*E) behavior in build_graph**

For typical package installations (V < 50, E < 200), this is under 1 ms.
Only pathological cases with 500+ packages would notice. But a name-to-index
hash map would make this O(E) total.

#### 4.2.3 repo_find_best: O(R * P) per call

**Lines 135-160**: Scans all packages in all repos. Called once per
dependency resolution. Cost depends on total repo size.

### 4.3 Stack Usage

`xpkg_resolve()` allocates `xpkg_dep_graph_t` on stack (line 553).
Estimated sizes:
- xpkg_dep_graph_t: 512 nodes * ~145 bytes + 512 * 4 + 8 = ~76 KB
- dfs_stack_t: 512 * 8 + 4 = ~4 KB
- work_queue_t: 512 * 4 + 8 = ~2 KB
- **Total stack frame: ~82 KB**

**FINDING (P1): 82KB stack allocation is excessive for freestanding kernel**

This is the same class of problem as the Poly1305 scratch buffer. If the
XENOS kernel stack is 16KB, this WILL overflow. Even with a 64KB stack,
this leaves only ~18KB for the entire call chain above xpkg_resolve.

**Recommendation**: Move xpkg_dep_graph_t to a module-static variable
(guarded by a spinlock) or use a caller-provided buffer. Package resolution
is inherently single-threaded (only one package install at a time), so a
static allocation is safe.

### 5.3 Budget vs. Doctrine

Per the Resource Budget Manifesto, XNET+XSEC should fit within the
"Platform services (20%)" CPU allocation and share the ~5GB committed
memory envelope. The 12.7 MB of static BSS is small relative to 16GB
total RAM, but it is wasteful when most connections will never be used
simultaneously.

**Recommended reductions (reiterated)**:
1. TCP_TCB_MAX: 128 -> 32 (saves 8.8 MB)
2. TCP ring buffers: 16KB -> 8KB (saves 16KB/TCB = 512KB for 32 TCBs)
3. TCP_REXMIT_QUEUE_MAX: 32 -> 8 (saves ~36KB/TCB)
4. IP frag entries: 16 -> 8, use bitmap (saves ~600KB)

**Optimized total: ~1.7 MB** (87% reduction)

## 7. Sprint 4 Recommendations (Priority-Ordered)

### P0: AES-NI + PCLMULQDQ Implementation
- Files: sec/xsec/crypto/aes_gcm.c
- Impact: 5,520x AES throughput improvement (177 KB/s -> 975 MB/s)
- CPUID detection: leaf 1, ECX bit 25 (AES-NI), bit 1 (PCLMULQDQ)
- Runtime dispatch: check CPUID at init, set function pointers
- Effort: Medium (intrinsics for AESENC, AESENCLAST, PCLMULQDQ)

### P0: Stack Overflow Prevention
- Files: chacha20.c (32KB scratch), dep_resolve.c (82KB graph)
- Impact: Prevents kernel stack overflow under normal operation
- Fix: Streaming Poly1305 (no scratch), static dep_graph
- Effort: Medium

### P1: TCP Memory Reduction
- File: tcp.c
- Impact: 11.5 MB -> ~1.0 MB (91% reduction)
- Changes: TCP_TCB_MAX=32, ring buffers=8KB, rexmit queue=8
- Effort: Low (constant changes only)

### P1: TCP Ring Buffer Optimization
- File: tcp.c lines 106-126
- Impact: 12x copy throughput improvement
- Fix: Two-phase memcpy (head-to-end, wrap-to-start)
- Effort: Low

### P1: TCP cwnd Cap Fix
- File: tcp.c line 487
- Impact: Removes artificial 16KB throughput ceiling
- Fix: Cap cwnd at 65535 (or peer window) instead of SNDBUF_SIZE
- Effort: Trivial

### P2: Fast Recovery (RFC 5681)
- File: tcp.c lines 805-821
- Impact: 2x throughput during loss recovery
- Fix: Implement NewReno fast recovery state machine
- Effort: Medium

### P2: ChaCha20 SIMD (SSE2)
- File: chacha20.c
- Impact: 3-4x throughput (260 MB/s -> ~900 MB/s)
- Effort: Medium (SSE2 intrinsics, runtime CPUID)

### P2: Event-Driven DNS
- File: dns.c
- Impact: Eliminates CPU spin-wait during resolution
- Fix: Use PAL channel/IPC for async wait
- Effort: Medium

### P3: XPKG Hash Map for graph_find_node
- File: dep_resolve.c
- Impact: O(V*E) -> O(E) graph build
- Effort: Low-Medium
