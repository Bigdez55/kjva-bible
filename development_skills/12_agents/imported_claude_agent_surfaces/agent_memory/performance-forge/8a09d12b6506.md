## SPRINT 4 -- XSTORE / XBLOB

### PF-XSTORE-01: [P1] WAL Ring Memory Footprint -- OVER BUDGET

WAL entry struct (xstore.h line 191-206):
- sizeof(xstore_wal_entry_t) = 8+8+4+1+1+2+4+4+256+4096 = 4384 bytes
- XSTORE_WAL_RING_SIZE = 65536
- Total WAL buffer: 65536 * 4384 = **~274 MB**

This is CRITICALLY OVER BUDGET for a 16GB system. The WAL ring alone consumes
1.7% of total RAM. This is the single largest static allocation in GEN.OS.

Recommendation: Reduce WAL entry to compact form. Key (256B) and val (4096B)
should NOT be inlined in the WAL entry. Store key/val data in separate page
buffers referenced by page_no + offset. A compact WAL entry would be ~32 bytes,
reducing total from 274MB to ~2MB.

### PF-XSTORE-02: [P2] B+ Tree Height for 1M Entries -- WITHIN BUDGET

- Order 256 leaf, order 128 internal
- Height = ceil(log_128(1M/256)) = ceil(log_128(3906)) = ceil(1.70) = 2 internal levels + 1 leaf = **3 levels total**
- Lookup cost: 3 page reads (12KB I/O) -- acceptable
- Verdict: WITHIN BUDGET. Tree height <= 3 confirmed for 1M keys.

### PF-XSTORE-03: [P3] Page Cache Sizing -- WITHIN BUDGET

- 1024 pages * sizeof(xstore_cache_entry_t)
- sizeof(xstore_cache_entry_t) = 4+1+1+1+8+4096 = ~4112 bytes (with padding ~4120)
- Total: 1024 * 4120 = **~4.1 MB**
- For a 16GB system this is 0.025% of RAM -- well within budget.
- Cache hit ratio at 1024 pages with B+ tree order 256: working set of ~4000 pages
  for 1M keys means ~25% cache hit rate for random access. Sequential scans
  will achieve near-100% due to clock algorithm.
- Verdict: WITHIN BUDGET. Could increase to 4096 pages (16MB) for better hit ratio.

### PF-XSTORE-04: [P2] Page Cache Lookup is O(n) Linear Scan -- OVER BUDGET

page_cache.c line 75-82: `_cache_find()` performs a linear scan over 1024 entries
for every cache hit check. This is called on EVERY btree node access, including
inside `xstore_cache_mark_dirty()`.

- Worst case: 1024 comparisons per lookup
- On the insert hot path: cache_get (lookup) + cache_mark_dirty (lookup) = 2048 comparisons
- At ~1 cycle per comparison: ~500 ns per lookup pair
- Impact: p95 insert latency inflated by ~1 us from cache overhead alone

Recommendation: Replace linear scan with open-addressing hash directory.
O(1) average lookup. Implementation effort: ~100 LOC. Sprint 5 notes this is planned.

### PF-XSTORE-05: [P2] CRC32C is Bit-by-Bit Software -- OVER BUDGET

xstore.h line 616-631: `xstore_crc32c_byte()` processes 8 iterations per byte.
Same pattern in xblob.h. No lookup table, no CRC32C instruction (SSE4.2).

- WAL entry CRC: 4384 bytes * 8 iterations = 35,072 iterations per WAL append
- Page checksum: 4088 bytes * 8 = 32,704 iterations per dirty page flush
- i5-8265U supports CRC32C via SSE4.2 (`crc32` instruction): ~0.17 cycles/byte
- Software bit-by-bit: ~8 cycles/byte = **47x slower**

Recommendation: Use `_mm_crc32_u64` intrinsic (SSE4.2). Requires `-msse4.2` flag
(conflicts with `-mno-sse` freestanding). Alternative: 4-bit lookup table reduces
to 2 iterations/byte (4x improvement with no ISA dependency).

### PF-XBLOB-01: [P1] Index Memory Footprint -- OVER BUDGET

- XBLOB_INDEX_SLOTS = 1M (1,048,576)
- sizeof(xblob_index_entry_t) = 56 bytes
- Total: 1,048,576 * 56 = **56 MB**
- xblob.h line 148 declares this as `XBLOB_INDEX_BUF_SIZE`
- This is caller-provided (not BSS), but still 56MB of committed memory

Combined with WAL (274MB), XSTORE+XBLOB alone consume **~334 MB**.
This exceeds the entire XSTORE+XBLOB resource budget allocation.

Recommendation: Reduce index to 256K slots (14MB) for typical workloads.
At 75% load factor, 256K slots supports 192K blobs -- sufficient for
early GEN.OS. Scale to 1M only when storage usage warrants it.

### PF-XBLOB-02: [P2] GC Compaction is Stop-the-World

gc.c line 157-197: When tombstone density > 25%, the entire 1M-entry table
is rebuilt while holding `store_lock`. At 56 bytes * 1M entries:
- Full scan: ~56MB of sequential reads
- Re-insert 750K live entries (75% load): 750K * hash + probe
- Estimated time: 50-200ms depending on cache pressure
- During this time ALL blob operations are blocked

Recommendation: Incremental compaction -- process 4096 entries per tick,
spread over ~256 ticks. Or use Robin Hood hashing to eliminate tombstones entirely.

### PF-XBLOB-03: [P3] GC Log Replay is O(max_seq * SLOTS)

gc.c line 84: Replay iterates from seq=0 to max_seq, each iteration doing
`xblob_index_find()` which is O(1) amortized. But if max_seq is large
(e.g., 100K operations), replay scans 100K entries.

At 16384 ring slots, effective replay is bounded by ring size. Acceptable.

## SPRINT 6 -- XMIND (AI Inference)

### PF-XMIND-01: [P0] No SIMD -- Scalar-Only Math -- CRITICAL

tensor.c line 7: "No SIMD (x87 FPU only)."

ALL math primitives (matmul_q4, rmsnorm, softmax, rope, silu, dot) are
purely scalar fp32 loops. On i7-8565U with AVX2:
- Scalar: 1 FLOP/cycle
- SSE2: 4 FLOPs/cycle (4x)
- AVX2: 8 FLOPs/cycle (8x)
- AVX2+FMA: 16 FLOPs/cycle (16x)

The logit computation alone (transformer.c line 351-358):
  vocab_size(32000) * hidden_dim(3072) = 98.3M multiply-adds
  Scalar at 3.9 GHz: ~25ms per forward pass for logits ALONE
  AVX2+FMA: ~1.5ms

Total forward pass (28 layers of attention + FFN + final logits):
  Estimated scalar: **800ms-1200ms per token** (first token latency)
  With AVX2+FMA: **50-80ms per token**

The `-mno-sse2` freestanding constraint is the ROOT CAUSE. The kernel
compiles with `-mno-sse` to avoid saving SSE state in interrupt handlers.
XMIND must be compiled as a SEPARATE translation unit with `-mavx2 -mfma`
and linked separately, with SSE/AVX state save/restore around inference calls.

Recommendation: P0 -- mandatory for Sprint 6 ship. Without SIMD, XMIND
cannot meet the 2000ms first-token latency budget. With AVX2+FMA, it can
achieve ~60ms/token = 16 tokens/second, meeting the budget comfortably.

### PF-XMIND-02: [P1] Attention is O(n^2) -- No Flash Attention

transformer.c line 118-175: `xm_attention_head()` computes:
- scores[t] for t in [0..pos]: O(pos * head_dim)
- softmax over [0..pos]: O(pos)
- weighted V sum: O(pos * head_dim)
- Total per head: O(pos * head_dim)
- Total per layer: O(n_heads * pos * head_dim)
- All layers: O(n_layers * n_heads * pos * head_dim) = O(n * d)

Stack allocation: `float scores[XM_ATTN_SCRATCH]` where XM_ATTN_SCRATCH = 2048
= 8KB on stack per head. This is fine for kernel stacks >= 16KB.

At ctx_len=2048 (full context), per-layer attention:
  32 heads * 2048 positions * 96 dims = 6.3M FLOPs per layer
  28 layers = 176M FLOPs for attention alone

This is standard O(n*d) causal attention (NOT O(n^2) in the quadratic sense
since we process one token at a time with KV cache). The implementation is
correct for autoregressive decoding. Flash attention is not needed for
single-token decode -- it matters for prefill (prompt processing).

Verdict: ACCEPTABLE for decode. Will need flash attention for prompt prefill
in Sprint 7+ when batch processing is added.

### PF-XMIND-03: [P1] Weight Memory Footprint for Llama 3.2 3B

Q4_0 format: 20 bytes per 32 weights = 0.625 bytes/weight

Llama 3.2 3B parameter count breakdown (from boot_chain.c config):
- n_layers=28, hidden_dim=3072, n_heads=32, n_kv_heads=8, head_dim=96, ffn_dim=8192

Per layer Q4 weights:
  wq: 3072*3072 = 9.44M params -> 9.44M * 0.625B = 5.9 MB
  wk: 3072*768  = 2.36M params -> 1.47 MB
  wv: 3072*768  = 2.36M params -> 1.47 MB
  wo: 3072*3072 = 9.44M params -> 5.9 MB
  w1: 8192*3072 = 25.2M params -> 15.7 MB
  w2: 3072*8192 = 25.2M params -> 15.7 MB
  w3: 8192*3072 = 25.2M params -> 15.7 MB
  Per layer total: ~61.9 MB

28 layers: 28 * 61.9 = **~1,733 MB (1.69 GB)** for Q4 weights

Non-quantized weights (fp32):
  token_emb: 32000 * 3072 * 4 = 393 MB
  rms_att: 28 * 3072 * 4 = 0.33 MB
  rms_ffn: 28 * 3072 * 4 = 0.33 MB
  rms_final: 3072 * 4 = 12 KB
  rope tables: 2 * 2048 * 48 * 4 = 0.75 MB
  Total non-quantized: ~395 MB

**PROBLEM**: token_emb is 393MB in fp32. This should also be Q4 quantized.
With Q4: 32000 * 3072 * 0.625 = **61 MB** (6.4x savings).

Total model memory:
  With fp32 token_emb: 1,733 + 395 = **~2,128 MB (2.08 GB)**
  With Q4 token_emb:   1,733 + 61 + 1.4 = **~1,795 MB (1.75 GB)**

### PF-XMIND-04: [P1] KV Cache Memory

From boot_chain.c line 447:
  2 * n_layers * ctx_len * n_kv_heads * head_dim * 4 bytes
  = 2 * 28 * 2048 * 8 * 96 * 4 = **~352 MB**

(boot_chain.c says ~439 MB but the correct calculation is 352 MB)

### PF-XMIND-05: [P1] Inference State Scratch Buffers

State buffers (all fp32):
  logits: 32000 * 4 = 125 KB
  x: 3072 * 4 = 12 KB
  xb: 3072 * 4 = 12 KB
  q: 32 * 96 * 4 = 12 KB
  k: 8 * 96 * 4 = 3 KB
  v: 8 * 96 * 4 = 3 KB
  attn: 32 * 2048 * 4 = 256 KB  (NOTE: header says n_heads*ctx_len but code uses head_dim)
  Total state: ~423 KB

Stack scratch in FFN:
  gate[XM_FFN_SCRATCH]: XMIND_MAX_HEADS(32) * XM_MAX_HEAD_DIM(256) * 4 = 32 KB
  up[XM_FFN_SCRATCH]: 32 KB
  Total FFN stack: ~64 KB

### PF-XMIND-06: [P1] Total XMIND Memory Assessment

Component              | Memory
-----------------------|--------
Q4 weights (28 layers) | 1,733 MB
token_emb (fp32)       | 393 MB
RMS/rope weights       | 1.4 MB
KV cache               | 352 MB
State scratch          | 0.4 MB
FFN stack scratch      | 0.06 MB
**TOTAL**              | **~2,480 MB (2.42 GB)**

Remaining for OS after XMIND: 16,384 - 2,480 = 13,904 MB
OS overhead (kernel+k3s+compositor+shell+services): ~5,000 MB (from budget)
Available for user workloads: ~8,900 MB

Verdict: XMIND CAN run in 16GB with the OS. However, token_emb should be
Q4-quantized to save ~332 MB, bringing XMIND total to ~2,148 MB.

CAN INFERENCE RUN IN 4GB? NO. Minimum is ~2.1 GB for weights + 352 MB KV cache
+ overhead = ~2.5 GB. 4GB is insufficient with OS overhead.

## SPRINT 6 -- XJIT

### PF-XJIT-01: [P2] sizeof(xjit_func_t) -- LARGE STRUCT

From xjit.h:
  ir[4096]: 4096 * 24 = 98,304 bytes
  labels[256]: 256 * 44 = 11,264 bytes  (44 = 4+4+8*4+4)
  ranges[256]: 256 * 12 = 3,072 bytes
  ra (regalloc): 256*8 + 16 + 4 + 4 = 2,072 bytes
  code: 65536 + 4 + 4 = 65,544 bytes
  frame_size + compiled + pad: 8 bytes
  **TOTAL: ~180 KB per xjit_func_t**

This is a stack-hostile structure. Must be heap-allocated or static.
180KB is fine for a singleton JIT context but NOT for recursive compilation.

### PF-XJIT-02: [P3] IR is Not SSA -- Acceptable for One-Pass

IR uses virtual registers (up to 256 vregs) with live range tracking.
Not true SSA (no phi nodes effectively -- XJIT_PHI is ignored).
This is correct for a one-pass linear-scan JIT.

### PF-XJIT-03: [P3] Register Allocation is Linear Scan -- Appropriate

regalloc.c implements linear scan over live ranges with 14 allocatable
physical registers and 64 spill slots. Linear scan is O(n * k) where
n = IR instructions and k = registers. At 4096 max IR and 14 registers,
this is ~57K operations -- sub-millisecond.

### PF-XJIT-04: [P3] Code Buffer is Fixed 64KB

XJIT_CODE_BUF_SIZE = 65536. At ~5 bytes average per x86-64 instruction,
this supports ~13,000 native instructions. For JIT-compiled QuickJS
bytecode, this is sufficient for most functions.

Panic on overflow (codegen.c line 69-72) is the correct behavior --
functions exceeding 64KB should fall back to the interpreter.

## SPRINT 5 -- XSHELL

### PF-XSHELL-01: [P3] XSHELL Uses XFRAME Event Loop -- CORRECT

xshell.c line 243: `xframe_run(g_shell.xframe_app)` drives the 60fps loop.
XSHELL does NOT spin-poll. The tick callback (line 98) is called once per
frame by XFRAME. This is the correct design.

### PF-XSHELL-02: [P3] Scene Graph Redraw Frequency -- ACCEPTABLE

The tick callback updates:
- Notification auto-dismiss (O(16) check per frame)
- topbar_tick (clock update, typically once per second)
- dock_tick (hover timeout check)

No scene graph rebuild occurs in tick -- only state updates.
Scene graph is only modified on app launch/close events.
Verdict: WITHIN BUDGET.

## TOP 3 PERFORMANCE RISKS

### RISK 1: [P0] XMIND Scalar-Only Inference -- SHOWSTOPPER

Without SIMD (AVX2+FMA), inference is 8-16x slower than necessary.
Estimated 800-1200ms per token vs 50-80ms with AVX2+FMA.
FIRST TOKEN LATENCY BUDGET: 2000ms -- barely met without SIMD,
and will FAIL at longer context lengths.

Mitigation: Compile XMIND with `-mavx2 -mfma` as a separate compilation
unit. Add SSE/AVX state save/restore in the kernel scheduler around
XMIND execution contexts.

### RISK 2: [P1] WAL Entry Size Explosion (274 MB)

The WAL ring inlines full key (256B) and value (4096B) in every entry.
65536 entries * 4384 bytes = 274 MB. This is architectural waste.

Mitigation: Compact WAL entries to ~32 bytes (lsn, txn_id, page_no, op,
key_offset, val_offset). Store key/val data in the page cache. Reduces
WAL from 274 MB to 2 MB.

### RISK 3: [P1] CRC32C Software Path -- Cross-Cutting Tax

Every page write, WAL append, GC log entry, and index CRC uses the
bit-by-bit software CRC32C. At 8 cycles/byte vs 0.17 cycles/byte
with SSE4.2, this is a 47x penalty on every I/O path.

Mitigation: 4-bit lookup table (4x improvement, no ISA dependency) as
immediate fix. CRC32C instruction via SSE4.2 as P0 for Sprint 5.

## INFERENCE FEASIBILITY ASSESSMENT

**Can XMIND run Llama 3.2 3B on 16GB with OS overhead?**

YES, with caveats:

| Component              | Memory    |
|------------------------|-----------|
| XMIND total            | 2,480 MB  |
| OS (kernel+services)   | ~1,000 MB |
| XSTORE+XBLOB (current) | 335 MB   |
| XSHELL+XCOMP+XDISP    | ~150 MB   |
| System overhead        | ~500 MB   |
| **Total committed**    | **4,465 MB** |
| **Remaining for user** | **11,919 MB** |

The system fits. However:
1. WAL must be compacted (274 MB -> 2 MB) to avoid waste
2. token_emb should be Q4 quantized (393 MB -> 61 MB) to free 332 MB
3. SIMD is MANDATORY for acceptable inference speed
4. KV cache (352 MB) is the second largest consumer and scales linearly
   with ctx_len. At ctx_len=4096, KV cache doubles to 704 MB.

With optimizations applied:
  XMIND: 2,148 MB (Q4 token_emb)
  XSTORE+XBLOB: 63 MB (compact WAL + 256K index)
  Total saved: ~604 MB
  **Optimized total: ~3,861 MB committed, 12,523 MB for user workloads**
