## Fix 1: gf_mul() Constant-Time Branchless (aes_gcm.c:84-98)

### Before (branching)
```
if (b & 1) result ^= a;     // branch on secret-dependent bit
if (a & 0x80) a ^= 0x1b;    // branch on key-dependent bit
```
- 8 iterations, 2 branches each = 16 branches/call
- Well-predicted branches: ~1-2 cycles each (97%+ prediction on repetitive patterns)
- Total: ~20-30 cycles/call (assuming good prediction)

### After (branchless CT)
```
b_mask = (uint8_t)(0u - (b & 1u));    // 2 ops: AND + NEG
result ^= a & b_mask;                 // 2 ops: AND + XOR
hi_mask = (uint8_t)(0u - ((a >> 7u) & 1u));  // 3 ops: SHR + AND + NEG
a = (uint8_t)(a << 1u);               // 1 op: SHL
a ^= 0x1bu & hi_mask;                 // 2 ops: AND + XOR
b >>= 1u;                             // 1 op: SHR
```
- 8 iterations * 11 ops = 88 ALU ops
- On Whiskey Lake (4 ALU ports): ~88/4 = 22 cycles + 8 loop overhead = ~30 cycles
- No branch misprediction risk

### Per-AES-block impact
- MixColumns: 4 columns * 8 gf_mul calls = 32 calls/round * 13 rounds = 416 calls/block
- Delta per call: ~0-5 cycles (branchless is similar or slightly better due to no mispredict)
- Delta per block: ~0 to +2,080 cycles (negligible in context of CT S-box domination)

### Verdict: NEGLIGIBLE
The gf_mul CT fix costs ~0-5 cycles per call. The branchless version may actually be
FASTER on Whiskey Lake due to eliminating branch misprediction on random key material.
The 416 calls/block are completely dominated by the 70,656-iteration CT S-box scan.

## Fix 2: fe10_mul() Carry Propagation (ed25519.c:105-121)

### What was added
16 carry propagation operations (lines 109-120):
```
c0=(h0+(1LL<<25))>>26; h1+=c0; h0-=c0<<26;   // 3 ops: add+shift+sub
c4=(h4+(1LL<<25))>>26; h5+=c4; h4-=c4<<26;   // ... repeated 10 times
c9=(h9+(1LL<<24))>>25; h0+=c9*19; h9-=c9<<25; // final wrap: multiply by 19
c0=(h0+(1LL<<25))>>26; h1+=c0; h0-=c0<<26;   // second pass on c0
```

### Cycle cost per fe10_mul call
- 12 carry steps: each = ADD + SHIFT + SUB = 3 ops = ~1 cycle (ILP)
- Total carry chain: 12 * 3 = 36 ALU ops
- With ILP on Whiskey Lake (4 ports, pairs interleave): ~12-15 cycles
- Plus the schoolbook multiply itself: 100 int64 MULs + 19 int64 MULs (g*19)
  = ~119 MULs at 3 cycles each (IMUL r64) = ~357 cycles
- Carry propagation adds ~12-15 cycles to a ~360-cycle function = ~3.5% overhead

### Ed25519 sign/verify multiplication count
- fe10_mul: ~500 calls per verify (double-scalar-mul + inversion)
- fe10_sq (calls fe10_mul): ~250 calls per verify
- Total: ~750 fe10_mul calls per Ed25519 verify

### Total extra cost per Ed25519 verify
- 750 calls * 15 extra cycles = 11,250 cycles
- At 3.4 GHz: ~3.3 us additional
- Baseline Ed25519 verify: ~175 us
- Overhead: ~1.9%

### Verdict: ACCEPTABLE
The carry propagation is a CORRECTNESS FIX (prevents ~15-30% of multiplications
from producing wrong results due to int64->int32 truncation). The 1.9% overhead
is the cost of getting correct signatures. Non-negotiable.

## Fix 3: Poly1305 acc_top Carry Propagation (chacha20.c:271-281)

### What was added
Per 16-byte block (lines 274-281):
```
uint64_t cx = acc_top >> 2u;            // 1 op: SHR
acc_top &= 0x3u;                        // 1 op: AND
u128 tc = (u128)acc_lo + (u128)cx * 5u; // 2 ops: MUL + ADD (128-bit)
acc_lo = (uint64_t)tc;                  // 1 op: MOV
u128 t_hi = (u128)acc_hi + (uint64_t)(tc >> 64u);  // 2 ops: SHR128 + ADD
acc_hi = (uint64_t)t_hi;               // 1 op: MOV
acc_top += (uint64_t)(t_hi >> 64u);     // 2 ops: SHR128 + ADD
```

Plus in final reduction (lines 293-296):
```
u128 t5 = (u128)acc_lo + 5u;
u128 t5h = (u128)acc_hi + (uint64_t)(t5 >> 64u);
uint64_t sum_top = acc_top + (uint64_t)(t5h >> 64u);
uint64_t do_reduce = sum_top >> 2u;
```

### Cycle cost analysis
- Per-block carry: ~10 ALU ops, 2 of which are 128-bit (u128 add + u128 shift)
  - u128 ops compile to 2x64-bit on x86_64: ~4 extra ops
  - Total: ~14 ops per block = ~4-5 cycles (ILP)
- Final reduction: ~8 ops (one-time) = ~3 cycles

### For a 16KB TLS record
- 16,384 / 16 = 1,024 blocks
- Extra per-block cost: 1,024 * 5 cycles = 5,120 cycles
- Final reduction: 3 cycles (one-time)
- Total extra: ~5,123 cycles

### Baseline Poly1305 cost for 16KB
- Per-block: 3x2 schoolbook multiply (6 u128 MULs) + reduction + accumulate
  = ~60-80 cycles/block (u128 MUL = ~8 cycles on Whiskey Lake via MULQ+ADC chains)
- Baseline: 1,024 * 70 cycles = ~71,680 cycles
- Extra: 5,123 / 71,680 = ~7.1% overhead on Poly1305 phase

### Versus full ChaCha20-Poly1305 for 16KB
- ChaCha20 encryption: 1,024 * 64 / 64 = 256 chacha20_block calls
  - Each: 20 rounds * 16 QR ops + setup = ~320 ops = ~100 cycles
  - Total: 256 * 100 = ~25,600 cycles
- Total ChaCha20-Poly1305: 25,600 + 71,680 = ~97,280 cycles
- Extra 5,123 / 97,280 = ~5.3% of total AEAD cost

### At 3.4 GHz
- Extra latency per 16KB record: 5,123 / 3.4e9 = ~1.5 us
- Baseline: 97,280 / 3.4e9 = ~28.6 us
- New total: ~30.1 us

### Verdict: ACCEPTABLE
5.3% overhead on ChaCha20-Poly1305. This is a correctness fix -- without it,
acc_top overflow corrupts the accumulator for messages > ~8KB, producing
incorrect authentication tags. The 1.5 us extra per 16KB record is invisible
at TLS record processing timescales.

## Fix 4: TCP initcwnd = 10 MSS (tcp.c:464)

### Change
```
t->cwnd = 10u * t->snd_mss;   // RFC 6928: initial cwnd = 10 MSS
```
Previous (implied): cwnd = 1 MSS (RFC 2581 default)

### Slow-start analysis for 1MB file transfer

With MSS = 1460 bytes (TCP_MSS_DEFAULT):

**initcwnd = 1 MSS:**
| RTT | cwnd (segments) | Cumulative bytes |
|-----|-----------------|-----------------|
| 0   | 1               | 1,460           |
| 1   | 2               | 4,380           |
| 2   | 4               | 10,220          |
| 3   | 8               | 21,900          |
| 4   | 16              | 45,260          |
| 5   | 32              | 91,980          |
| 6   | 64              | 185,420         |
| 7   | 128             | 372,300         |
| 8   | 256             | 745,660         |
| 9   | 512             | ~1,493,180 (done) |
RTTs to transfer 1MB: ~9-10 RTTs in slow start

**initcwnd = 10 MSS (RFC 6928):**
| RTT | cwnd (segments) | Cumulative bytes |
|-----|-----------------|-----------------|
| 0   | 10              | 14,600          |
| 1   | 20              | 43,800          |
| 2   | 40              | 102,200         |
| 3   | 80              | 218,800         |
| 4   | 160             | 452,200         |
| 5   | 320             | 919,400         |
| 6   | 640             | ~1,852,600 (done) |
RTTs to transfer 1MB: ~6-7 RTTs in slow start

### Latency improvement
- RTTs saved: 10 - 7 = 3 RTTs (approximately log2(10) = 3.32 RTTs)
- For 10ms RTT: 3.32 * 10ms = **33.2ms saved**
- For 50ms RTT: 3.32 * 50ms = **166ms saved**
- For 100ms RTT: 3.32 * 100ms = **332ms saved**

### Impact on small objects (web pages, API responses)
- Object < 14.6KB (10 MSS): transfers in 1 RTT instead of ceil(log2(N))+1 RTTs
- Typical API JSON response (2-8KB): 1 RTT vs 2-4 RTTs
- For 10ms RTT: saves 10-30ms per API call

### Verdict: SIGNIFICANT POSITIVE IMPACT
This is a net WIN. RFC 6928 (Google's initcwnd=10 proposal from 2013) is standard
practice. Saves 33ms on 10ms-RTT connections for bulk transfers, and eliminates
1-3 RTTs for typical API/web responses. Zero downside on modern networks.
Thermal impact: zero (no extra computation).

## Fix 5: DNS pal_random_bytes() for Query ID (dns.c:383-385)

### Change
```
uint8_t qid_bytes[2];
pal_random_bytes(qid_bytes, 2u);    // CSPRNG instead of sequential counter
```

### Cost analysis
- pal_random_bytes(2) on XENOS:
  - Calls RDRAND instruction (available on Whiskey Lake)
  - RDRAND: ~460 cycles on Whiskey Lake (Intel spec: 200-800 cycles)
  - For 2 bytes: 1 RDRAND call (generates 64 bits) + mask = ~460 cycles
  - At 3.4 GHz: ~135 ns

- Alternative if RDRAND unavailable (software fallback):
  - ChaCha20 block + extract: ~100 cycles (if key already expanded)
  - Worst case (cold key): ~500 cycles = ~147 ns

### Context
- DNS queries are network-bound (5-50ms RTT per query)
- Extra 135ns per query = 0.00027% of a 50ms query
- DNS cache hit rate in practice: 80-95%, so queries are rare

### Verdict: NEGLIGIBLE
135ns on a network operation measured in milliseconds. This is a critical
security fix (prevents Kaminsky-style cache poisoning) with zero measurable
performance impact.

## Composite Thermal Impact on HP EliteBook x360

### Steady-state TLS workload (16KB records, ChaCha20-Poly1305)
- Baseline: ~28.6 us/record = ~35,000 records/sec max throughput
- After P2 fixes: ~30.1 us/record = ~33,200 records/sec
- Throughput reduction: ~5.1%
- Power impact: negligible (ALU ops, not cache-miss-heavy)
- CPU temperature increase: < 0.5C (within noise floor of thermal measurement)

### During Ed25519 signature verification
- Baseline: ~175 us per verify
- After fix: ~178.3 us per verify
- Burst verify (certificate chains, 3 signatures): 534.9 us (was 525 us)
- Thermal impact: ZERO (one-shot operation, < 1ms)

### TCP initcwnd=10 thermal impact
- No additional computation
- May cause brief burst of higher NIC activity (10 segments vs 1)
- NIC power draw increase: ~0.01W for ~1ms = negligible

### Overall thermal verdict
All P2 fixes combined add < 0.5C to CPU package temperature under sustained
crypto workload. The HP EliteBook x360's thermal envelope (sustained 85C target)
is not threatened.

## Net Performance Verdict

**All P2 hardening fixes are WITHIN BUDGET.**

- Crypto fixes: correctness-critical with minimal overhead (1.9%-5.3%)
- TCP initcwnd: substantial positive performance improvement (saves 33-332ms)
- DNS CSPRNG: invisible overhead on a network-bound operation

**AES-NI remains P0 URGENT** -- the CT S-box (P0 fix from earlier) dominates all
crypto performance. The P2 fixes analyzed here are rounding errors compared to
the 320x throughput regression from CT S-box. Until AES-NI is implemented,
AES-GCM software path is ~200 KB/s (OVER BUDGET). ChaCha20-Poly1305 should be
the default cipher suite until AES-NI lands.

### Recommended cipher suite priority for TLS 1.3:
1. TLS_CHACHA20_POLY1305_SHA256 (default until AES-NI)
2. TLS_AES_256_GCM_SHA384 (enable after AES-NI implementation)
