A 4 MB kernel at byte granularity = 4M iterations. With 8-byte `__builtin_memcpy`
or `uint64_t *` aliased copy this becomes 500K iterations = ~8x improvement. Currently
acceptable (2-5 ms) but needlessly slow.

### 2D. P0 Memory Findings (MUST FIX)

**P0-MEM-01: XSTORE WAL ring = 274 MB -- OVER BUDGET**

Source: `store/xstore/include/xstore.h:84,204,209`
```
XSTORE_WAL_RING_SIZE = 65536 entries
sizeof(xstore_wal_entry_t) = 8+8+4+1+1+2+4+4+256+4096 = 4384 bytes
Total = 65536 x 4384 = 287,309,824 bytes = ~274 MB
```

This is 274 MB of static allocation for a WAL ring that should need at most 4-8 MB.
The problem is the WAL entry embeds a full 4096-byte value inline.

**Fix**: Reduce `XSTORE_WAL_RING_SIZE` to 4096, OR use out-of-line values (WAL entry
stores only key + page_no, not the full value). Expected savings: 270 MB (98.5%).

**P0-MEM-02: XBLOB hash index = 56 MB -- OVER BUDGET**

The 1M-slot content-addressed hash index at ~56 bytes/slot = 56 MB.
For an embedded OS with at most thousands of blobs, this is 100x overprovisioned.

**Fix**: Reduce to 16K slots (~896 KB) or use on-demand growth.

**P0-MEM-03 (carried from Sprint 3): TCP TCB table = 11.5 MB**

128 TCBs with 16KB+16KB send/receive buffers each. Most embedded systems need < 32.

**Fix**: Reduce to 32 TCBs + 4KB buffers = ~416 KB (96% reduction).

## 3. HPET AND LAPIC TIMER ACCURACY

### 3A. HPET Resolution Analysis

Source: `kernel/xenos/drv/hpet.c`, `kernel/xenos/include/xhpet.h`

**HPET Clock Period**: Read from GCAP_ID[63:32] = `period_fs` (femtoseconds per tick).

Typical Intel HPET: period_fs ~= 69,841 fs (14.318 MHz)
- 1 tick = 69.841 ns
- **Minimum resolution: ~70 ns per tick**
- Maximum resolution (HPET spec floor): 100 ns per tick (XHPET_CLK_PERIOD_MAX = 0x05F5E100)

**The __int128 overflow fix** (hpet.c:150,157) is correct -- prevents uint64 overflow
at large tick counts (>1 day of uptime at 14.318 MHz).

**Conversion accuracy**:
- `xhpet_ticks_to_ns()`: uses `(ticks * period_fs) / 1000000` with __int128
- `xhpet_ns_to_ticks()`: uses `(ns * 1000000) / period_fs` with __int128
- Error: zero for any practical value (full 128-bit precision)

**Minimum achievable timer period (xhpet_set_periodic)**:
- `period_ticks = freq_hz / hz`
- At 14.318 MHz, 1 kHz timer = 14318 ticks (exact)
- At 14.318 MHz, 10 kHz timer = 1431 ticks (~0.05% error)
- At 14.318 MHz, 100 kHz timer = 143 ticks (~0.1% error)
- At 14.318 MHz, 1 MHz timer = 14 ticks (~2% error) = ~1 us minimum period


### 3B. LAPIC Calibration Assessment

Source: `kernel/xenos/drv/lapic.c:190-236`

**CRITICAL FINDING P1-TIMER-01: LAPIC calibration uses a volatile decrement loop, NOT HPET.**

```c
volatile uint32_t delay = 1000000u;
while (delay > 0u) { delay--; }
```
(lapic.c:209-210)

This is a busy-loop of 1M decrements, which the comment claims targets ~10ms.
On i5-8250U at 3.9 GHz, a volatile decrement takes ~2-3 cycles per iteration.
At 3 cycles: 1M * 3 / 3.9e9 = 0.77 ms (NOT 10 ms).

**Consequence**: The calibrated `bus_hz` will be ~13x too HIGH, causing:
- LAPIC timer running ~13x faster than intended
- Scheduler ticks at ~780 Hz instead of 60 Hz
- Excessive interrupt overhead

**Fix**: Wire `xhpet_udelay(&g_xhpet, 10000000)` (10ms) as the comment suggests.
The HPET is initialized before LAPIC timer calibration (step 10d before scheduler init).

### 3C. TSC Calibration Assessment

Source: `aetherboot.c:431-436`

```c
UINT64 t0 = rdtsc();
BS->Stall(100000ULL);   /* 100 ms */
UINT64 t1 = rdtsc();
return (t1 - t0) * 10ULL;
```

This is CORRECT for pre-ExitBootServices calibration. UEFI Stall is hardware-timed
(ACPI PM timer typically). Accuracy is ~1-2% which is sufficient for TSC-based
timekeeping when refined later by HPET cross-calibration.

**P1-TSC-01**: i5-8250U supports CPUID leaf 0x15 (TSC frequency).
If available, this gives EXACT TSC frequency with zero stall time (100 ms savings).

## 4. PAGE TABLE OVERHEAD

### 4A. Current Page Table Layout

Source: `aetherboot.c:462-527`

5 pages allocated (20 KB total):

```
Page 0: PML4
  [0]   -> PDPT_id    (identity map)
  [273] -> PDPT_hhdm  (HHDM at 0xFFFF888000000000)
  [511] -> PDPT_kern  (kernel at 0xFFFFFFFF80000000)

Page 1: PDPT_id
  [0..3] -> 4 x 1GB huge pages (identity map 0-4GB)

Page 2: PDPT_hhdm
  [0..3] -> 4 x 1GB huge pages (HHDM 0-4GB)

Page 3: PDPT_kern
  [510] -> PD_kern

Page 4: PD_kern
  [0..N] -> 2MB pages for kernel image at kern_phys
```

### 4B. Coverage Assessment

| Mapping           | Range                          | Size Mapped | Pages Used |
|-------------------|-------------------------------|-------------|------------|
| Identity          | 0x0 - 0x100000000             | 4 GB        | 2 (PML4 entry + PDPT) |
| HHDM              | 0xFFFF888000000000 + 0-4GB    | 4 GB        | 2 (PML4 entry + PDPT) |
| Kernel text/data  | 0xFFFFFFFF80000000 + 0-Nx2MB  | N x 2MB     | 2 (PDPT + PD) |

**Is 4 GB HHDM sufficient?**

For 8 GB RAM: NO. Only the first 4 GB of physical memory is accessible via HHDM.
For 16 GB RAM: Definitely NO.

**P0-PGTABLE-01: HHDM maps only 4 GB of physical RAM.**

The HP EliteBook x360 ships with 8 GB or 16 GB LPDDR3. Memory above 4 GB is
inaccessible to the kernel via HHDM, meaning:
- PMM cannot manage memory above 4 GB
- XMIND model weights cannot be loaded into upper RAM
- Effective usable RAM capped at ~4 GB even on 16 GB hardware

**Fix**: Map more PDPT entries. Each 1 GB huge page costs zero page-table pages
(it uses a PDPT entry directly). For 16 GB coverage:
- Need 16 PDPT entries (currently 4)
- No additional page allocations needed (same PDPT page has 512 slots)
- Simply change the loop bound from 4 to 16 in `build_page_tables()`

```c
// Current (aetherboot.c:513):
for (UINT64 i = 0; i < 4; i++)
    pdpt_hhdm[i] = (i * 0x40000000ULL) | PTE_P | PTE_RW | PTE_PS | PTE_G;

// Fix: use actual_gb = total_memory_bytes / (1 << 30), clamped to 512
```

**Overhead for full 16 GB**: Still 20 KB (5 pages). The existing PDPT pages have
room for 512 entries = 512 GB. No additional pages needed.

For a proper dynamic scheme (post-boot VMM), the cost per 1 GB of mapped physical
memory is: 0 bytes (using 1 GB huge pages in PDPT). For 2 MB granularity mapping
(needed for mixed permissions), cost is: 1 PD page (4 KB) per 1 GB = 64 KB for 16 GB.

## 5. COMPOSITOR FRAME RATE

### 5A. Compositing Pipeline Analysis

Source: `display/xcomp/compositor.c`

Pipeline per frame (xcomp_frame()):
```
Phase A: pal_spin_lock(&g_comp_lock)                        ~0.05 us
Phase B: xdisp_buf_map(g_back_buf)                          ~1-5 us (MMIO map)
Phase C: Full-screen background fill                        ** BOTTLENECK
Phase D: Walk scene graph, blend each surface               ** BOTTLENECK
Phase E: xdisp_buf_unmap(g_back_buf)                        ~1-5 us
Phase F: xdisp_buf_submit(back, crtc, wait_vsync=1)         ~0-16.67 ms (vsync)
Phase G: Swap front <-> back handles                        ~0.001 us
         pal_spin_unlock(&g_comp_lock)                      ~0.05 us
```

### 5B. Background Fill Cost (Phase C)

```c
for (fill_row = 0; fill_row < g_output_h; fill_row++) {
    uint32_t *row = back_pixels + fill_row * back_stride_px;
    for (fill_col = 0; fill_col < g_output_w; fill_col++) {
        row[fill_col] = XCOMP_BG_COLOUR;
    }
}
```

At 1920x1200 ARGB8888:
- Pixels: 1920 x 1200 = 2,304,000
- Bytes: 2,304,000 x 4 = 9,216,000 (9.2 MB)
- Scalar store throughput on Whiskey Lake: ~1 store/cycle (with write-combining)
- At 3.9 GHz: 2,304,000 / 3.9e9 = 0.59 ms minimum (best case)
- Without write-combining (uncached framebuffer): ~3-5 ms
- **Estimate: 1-5 ms per frame for background fill alone**

### 5C. Alpha Blend Cost (Phase D)

The `blend_pixel()` function per pixel:
- 4 extractions (shift + mask)
- 1 multiply + 1 divide for effective alpha
- 4 multiply + 4 divide + 4 add for RGBA channels
- 4 clamp branches
- Total: ~20-30 ALU ops per pixel

For full-screen damage (worst case, 1 opaque surface covering screen):
- 2,304,000 pixels x 25 ops / 3.9 GHz = 14.77 ms
- **This EXCEEDS the 16.67 ms frame budget with only 1 surface!**

For typical case (5-10 surfaces with 10% screen damage each):
- ~230,400 pixels x 10 surfaces x 25 ops / 3.9 GHz = 1.48 ms per surface x 10 = 14.8 ms
- Still dangerously close to budget.

### 5F. Compositor Verdict

```
PERFORMANCE FORGE REPORT
========================
Component:     XCOMP Compositor Frame Rate
Metric:        Full-screen composition time
Baseline:      14.8-19.8 ms for 1 full-screen opaque surface
Target:        p99 < 16.67 ms (60 FPS guarantee)
Status:        OVER BUDGET for full-screen damage
```

**P1-COMP-01: Full-screen scalar blend exceeds 16.67 ms frame budget.**

Root cause: The `blend_pixel()` inner loop uses integer division by 255 (expensive
on x86 without SIMD). Each pixel costs ~25 ALU ops.

Fix options (ranked by impact):
1. **Skip blend for fully opaque surfaces** (opacity=255, alpha=0xFF). Just memcpy
   the source rows. Cost: 9.2 MB memcpy = ~2-3 ms. Saves ~12 ms for common case.
2. **Replace div-by-255 with multiply-shift approximation**: `(x * 0x8081) >> 23`
   Saves 2-3 cycles per channel, ~50% improvement in blend.
3. **SSE2/AVX2 SIMD blend**: Process 4-8 pixels per instruction. i5-8250U has AVX2.
   Expected improvement: 4-8x throughput.
4. **Dirty-region-only background fill**: Only clear regions not covered by opaque
   surfaces. Saves 1-5 ms for typical desktop with opaque topbar + content area.

**P2-COMP-02: Background fill is always full-screen even with no damage.**

Every frame fills all 9.2 MB regardless of damage regions. The compositor could
skip the fill for regions covered by opaque committed surfaces.

### VERDICT

```
==============================================================
GO/NO-GO:  CONDITIONAL GO
==============================================================

The system boots in ~3.2 seconds (well within 12-second target),
the compositor achieves 60 FPS for typical desktop workloads
(small damage regions), HPET provides ~70 ns timer resolution,
and the full AI inference stack fits in 8 GB.

However, THREE P0 findings prevent unconditional GO:

1. WAL ring (274 MB) and XBLOB index (56 MB) consume 330 MB
   of RAM for data structures that need ~5 MB. On 8 GB hardware
   this is 4% of total RAM wasted on empty ring slots.

2. HHDM maps only 4 GB of 8-16 GB physical RAM, making upper
   memory inaccessible to the kernel. This is a hard blocker
   for the XMIND 1.8 GB model on 8 GB hardware.

3. The LAPIC timer calibration is off by ~13x due to a
   calibration loop that runs 13x faster than assumed.

FIX P0-MEM-01, P0-MEM-02, P0-PGTABLE-01, and P1-TIMER-01
before the next production milestone.
==============================================================
```
