## Findings — Severity P1 (HIGH)

### PIA-S4-P1-01: store/xblob/gc.c:118-141 — Lock Ordering Inconsistency

What: `xblob_gc_scan` acquires `store_lock`, then calls `xblob_gc_log_append`
which acquires `gc_lock`. This creates a nested acquisition order:
store_lock -> gc_lock.

All other callers of `xblob_gc_log_append` (xblob_put:113, xblob_delete:278,
xblob_ref:302) release `store_lock` BEFORE calling `xblob_gc_log_append`,
meaning they acquire gc_lock without holding store_lock.

Where: store/xblob/gc.c:138 (call to xblob_gc_log_append inside gc_scan
while store_lock is held from line 118)

Why: If any future code path acquires gc_lock first and then attempts to acquire
store_lock (e.g., a GC callback inspecting blob metadata), the result is
deadlock. The inconsistency is currently latent because no code acquires
gc_lock then store_lock. However, lock ordering must be globally consistent
to be safe under refactoring.

How: Release store_lock before calling xblob_gc_log_append in gc_scan:

  /* In xblob_gc_scan, before appending GC log entry: */
  pal_spin_unlock(&store->store_lock);
  xblob_gc_log_append(store, XBLOB_GC_SCAN_COMPLETE, 0, NULL);
  /* Re-acquire only if subsequent operations require it */

Established ordering: ALL callers must follow gc_log_append AFTER releasing
store_lock. Add a comment in gc.c documenting this invariant.

### PIA-S8-P1-02: ai/xmind/src/weights_loader.c:377 — PAL Page Handle Leak

What: `wl_alloc_pages` discards both `pal_handle_t ph` (page handle) and
`pal_handle_t mh` (map handle) via explicit `(void)ph; (void)mh;` casts.
All per-layer weight allocations (~7 matrices per layer × 28 layers = 196+
allocations, plus embeddings/RMSNorm/RoPE tables) are permanently leaked.
`xmind_weights_unload` frees the pointer arithmetic view of memory but cannot
release the underlying PAL page allocations.

Where: ai/xmind/src/weights_loader.c:377 (wl_alloc_pages function)

Why: On a long-running GEN.OS session with repeated model load/unload cycles
(e.g., XORCHESTRA restarting XMIND after a configuration change), the platform
heap will be exhausted by accumulated leaked weight pages. The Llama 3.2 3B
Q4_0 model requires ~2GB of weight data — each reload cycle leaks ~2GB worth
of PAL page handles from the kernel page allocator.

How: The fix requires maintaining a handle array in `xmind_weights_t`:

  /* In xmind_weights_t struct (xmind.h): */
  pal_handle_t alloc_handles[XMIND_MAX_ALLOC_HANDLES];
  uint32_t     n_alloc_handles;

  /* In wl_alloc_pages: */
  if (w->n_alloc_handles < XMIND_MAX_ALLOC_HANDLES) {
      w->alloc_handles[w->n_alloc_handles++] = ph;
  }

  /* In xmind_weights_unload: */
  for (uint32_t i = 0; i < w->n_alloc_handles; i++) {
      pal_page_free(w->alloc_handles[i]);
  }

Sprint 8 source acknowledges this as accepted debt. Flagging as P1 because
the leak is unbounded on production hardware with a model runtime.

### PIA-S4-P2-01: store/xstore/btree/btree.c:786 — Cursor Advance Always Returns NOT_FOUND

What: `xstore_btree_cursor_advance` returns `XSTORE_ERR_NOT_FOUND` unconditionally
after exhausting the current leaf page. Leaf chaining (next-leaf pointer
traversal) is documented as deferred to Sprint 5 but Sprint 5 has shipped.

Where: store/xstore/btree/btree.c:786 (cursor advance function body)

Why: Any consumer of XSTORE cursors — including XPKG's package enumeration
(`xpkg_list_installed`), XBLOB index iteration, and XSTORE stats reporting —
will see only entries in the first B+ tree leaf page, silently missing all
entries on subsequent pages. For a store with more than ~170 entries (4096B
page / ~24B per leaf entry), data is invisible to callers using the cursor API.

How: Implement leaf chaining. Leaf page header must store `next_page_no`.
On cursor advance past end of current leaf, load `cursor->page_no = leaf.next_page_no`
and set `cursor->slot = 0`. Return XSTORE_ERR_NOT_FOUND only when next_page_no == 0.

Status: Sprint 5 deferred marker not resolved. AT RISK.

### PIA-S9-P2-02: kernel/xenos/drv/hpet.c:148 — ticks_to_ns Overflow

What: `xhpet_ticks_to_ns` computes `(ticks * ctx->period_fs) / 1000000ULL`.
With `period_fs` up to `XHPET_CLK_PERIOD_MAX` (1e8 fs) and `ticks` at a
typical 64-bit counter value after hours of uptime (~1e15 at 14.3 MHz),
the intermediate product `ticks * period_fs` = 1e23, which overflows uint64_t
(max ~1.8e19). The code comment acknowledges the 2^32 ns cap but does not
enforce it in code.

Where: kernel/xenos/drv/hpet.c:148

Why: After ~1300 seconds of uptime at 14.3 MHz, `xhpet_ticks_to_ns` returns
a wrapped (incorrect) nanosecond value. `xhpet_dump` calls `xhpet_ticks_to_ns`
for uptime display — the displayed uptime would be wrong. More critically,
`xhpet_udelay` calls `xhpet_ns_to_ticks` (which is safe) but if any caller
uses `xhpet_ticks_to_ns` for deadline arithmetic, incorrect overflow values
propagate to timer decisions.

How: Split the division to avoid overflow:

  uint64_t xhpet_ticks_to_ns(const xhpet_ctx_t *ctx, uint64_t ticks) {
      if (!ctx || ctx->period_fs == 0ULL) { return 0ULL; }
      /* Divide ticks first to reduce magnitude before multiplying */
      /* ns = (ticks / 1000000) * period_fs + (ticks % 1000000) * period_fs / 1000000 */
      uint64_t q = ticks / 1000000ULL;
      uint64_t r = ticks % 1000000ULL;
      return q * ctx->period_fs + (r * ctx->period_fs) / 1000000ULL;
  }

### PIA-S7-P2-03: vm/xemu/src/xjit_backend.c:308 — Conservative remaining=32 Risks OOB Read

What: `lift_block` passes `remaining = 32u` as a fixed constant to the
x86-64 decoder regardless of actual bytes remaining in the ELF segment being
lifted. For instructions at the end of a segment with fewer than 32 bytes
remaining, the decoder may read past the mapped ELF segment boundary.

Where: vm/xemu/src/xjit_backend.c:308

Why: In a freestanding environment without virtual memory protection, reading
past an ELF segment end reads adjacent mapped memory (next segment or unmapped
region). If unmapped, this causes a page fault during JIT lifting. If mapped,
it silently decodes garbage bytes as valid instructions, producing incorrect
JIT output.

How: Thread the actual remaining byte count from the caller through to
`lift_block`. The ELF loader in `elf_loader.c` exposes segment bounds via
`xemu_segment_t` — use `seg_end - guest_rip` as the true remaining count,
capped at 32 for the decoder's working buffer size.

### PIA-S8-P2-04: ai/xmind/src/transformer.c — Stack Allocation in FFN May Overflow

What: `xm_ffn` allocates `float gate[XM_FFN_SCRATCH]` and `float up[XM_FFN_SCRATCH]`
on the stack. If `XM_FFN_SCRATCH = XMIND_MAX_HEADS * XM_MAX_HEAD_DIM` with
XMIND_MAX_HEADS=64 and XM_MAX_HEAD_DIM=256, each array is 64*256*4 = 65536 bytes.
Two such arrays = 128KB per `xm_ffn` call frame.

Where: ai/xmind/src/transformer.c (xm_ffn function, gate/up arrays)

Why: In XENOS kernel context, the kernel stack is typically 16KB-64KB. A single
`xm_ffn` call overflows the kernel stack. Even in userspace context, 128KB of
stack per FFN call (called once per layer, 28 layers = 3.5MB total stack depth
for a single forward pass) will exhaust stack space on constrained systems.

How: Promote `gate` and `up` to module-static scratch buffers:

  static float s_ffn_gate[XM_FFN_SCRATCH];
  static float s_ffn_up[XM_FFN_SCRATCH];

This is safe because XMIND is single-threaded (documented in MEMORY.md).
The same pattern is already used for `s_scan_buf` in weights_loader.c.

Note: `float scores[XMIND_MAX_SEQ]` in `xm_attention_head` = 2048*4 = 8KB
is called `n_heads` times per layer but returns before the next call — stack
frame reuse keeps peak at 8KB. Acceptable. Only FFN is critical.

### PIA-S8-P2-05: ai/xmind/src/weights_loader.c:309-312 — GGUF Array Metadata Skip Bug

What: In `gguf_scan_metadata`, the `GGUF_VAL_ARRAY` case skips only 12 bytes
(4 for elem_type, 8 for count) without iterating over or skipping the actual
array elements. For any non-empty metadata array, the scanner loses sync with
the byte stream and misparses all subsequent metadata entries.

Where: ai/xmind/src/weights_loader.c:309-312

Why: GGUF metadata commonly includes string arrays (e.g., `tokenizer.ggml.tokens`,
`llama.attention.head_count_kv`). A misparse of any entry causes all subsequent
metadata reads to return wrong types, wrong sizes, and potentially corrupt
stack/heap if buffer bounds are not strictly enforced in the value readers.

How: The scanner must compute and skip the element payload:

  case GGUF_VAL_ARRAY: {
      uint32_t elem_type;
      uint64_t elem_count;
      /* read elem_type (4B) and elem_count (8B) */
      ...
      /* skip elem_count * sizeof(elem_type) bytes */
      uint64_t elem_sz = gguf_val_size(elem_type);  /* helper needed */
      off += (uint32_t)(elem_count * elem_sz);
      break;
  }

This is required for correct GGUF model loading in production.

### PIA-S8-P2-06: kernel/xenos/drv/xacpi.c:109 — XSDT Table Count Misalignment

What: `xa_enumerate_tables` computes `n_tables = body_len / 8u`. If `body_len`
is not a multiple of 8 (possible if firmware writes a non-aligned XSDT), the
division truncates and the last table pointer is silently skipped.

Where: kernel/xenos/drv/xacpi.c:109

Why: Missing a table pointer means a device table (e.g., HPET ACPI table,
SRAT, or SSDT) is not registered in the XACPI namespace. Subsequent calls to
`xacpi_find_table` for that device return NULL, causing device initialization
failure (e.g., HPET not found -> no high-resolution timer -> kernel falls back
to degraded timer or panics).

How: Assert alignment and log a warning if misaligned:

  if (body_len % 8u != 0u) {
      /* ACPI firmware bug: XSDT body_len not 8-byte aligned */
      /* Round up to include the partial entry */
      n_tables = (body_len + 7u) / 8u;
  }

### PIA-S9-P3-04: tools/iso/gen_iso.c — combined_hash is XOR-Only

What: `giso_add_component` XORs each component's SHA-256 into `combined_hash`
(gen_iso.c:175-177). XOR of SHA-256 hashes is not collision-resistant: any
two components with the same XOR-paired hash produce an identical `combined_hash`.

Where: tools/iso/gen_iso.c:175-177

Why: The `combined_hash` is printed in `giso_dump` and conceptually serves as
a manifest integrity check. With XOR, an attacker who can swap component pairs
with matching XOR differentials produces the same `combined_hash` with different
content. The Ed25519 sig slot (`giso_manifest_t`) mitigates this for verified
boot, but the `combined_hash` itself is not a trustworthy integrity primitive.

How: Replace XOR accumulation with an HMAC-SHA-256 or SHA-256(concat(all_hashes))
running hash using XSEC primitives. For Sprint 9, annotate the field as
"non-cryptographic manifest fingerprint" and document that the Ed25519 sig
(when populated) is the authoritative integrity proof.

### PIA-S7-P3-05: kernel/xenos/drv/xacpi.c — WMI Settings Table Not Locked

What: `s_hp_settings[XACPI_HP_WMI_MAX_SETTINGS]` and `s_hp_n_settings` are
static globals modified by `xa_hp_wmi_init_defaults` (called from `xacpi_init`)
and read/written by `xacpi_hp_wmi_get`/`xacpi_hp_wmi_set`. No spinlock
protects these globals.

Where: kernel/xenos/drv/xacpi.c (~line 602)

Why: If `xacpi_hp_wmi_set` is called concurrently from an interrupt handler
or a second CPU core (in future SMP scenarios) while `xacpi_hp_wmi_get`
is reading, a torn read/write is possible. For Sprint 9 (single-core boot
context), this is benign. For production SMP kernel, it is a data race.

How: Add a `pal_spinlock_t g_wmi_lock = PAL_SPINLOCK_INIT` and acquire/release
around all accesses to `s_hp_settings` and `s_hp_n_settings`.

### MEDIUM: g_boot_info_ptr dead store (main.c:81, 1177)
- `static xenos_boot_info_t *g_boot_info_ptr` is assigned at line 1177 but
  never read in main.c or any other file. The comment says "read by gensd_start()"
  but gensd_start() receives `bi` directly as a parameter — it does not use this
  global. Dead store. Compiler may warn under -Wunused-variable (suppressed by static).
- Risk: misleads future developers into believing gensd_start() reads the global.
- Fix: remove the global and the assignment, or add a static_assert with a
  comment explaining the intended future use (e.g., SMP AP boot path).

### MEDIUM: THREAD_JOIN spin condition incomplete (main.c:847)
- `while (g_fibers[slot].state == XF_READY)` — only spins on XF_READY.
- Does not handle the case where a fiber was created but then its slot was freed
  (XF_FREE) by another path before join was called. In that scenario, join returns
  0 (success) having never seen XF_DONE, giving a false success result.
- In the current cooperative single-CPU scheduler this cannot happen in practice
  (no concurrent modification), but the semantics are incomplete and unsafe for
  SMP extension.
- Fix: spin on `state != XF_DONE && state != XF_FREE`, or assert state == XF_DONE
  after the spin exits.

### LOW: Stack frame comment ambiguity (main.c:819)
- Comment "Layout (growing down): r15,r14,r13,r12,rbx,rbp,ret_addr" describes
  the memory layout from low-to-high address, which is correct.
- The push sequence in xenos_fiber_switch is push rbp, rbx, r12, r13, r14, r15.
- The pre-built frame in THREAD_CREATE writes *--sptr in the order r15 first
  (lowest address), which correctly mirrors the pop sequence.
- No functional bug. Comment is terse and could mislead — annotate with
  "layout from low to high address (first popped = r15 = lowest)".

### LOW: gensd_build_handoff hardcodes 1 GB memory region (gensd/main.c:95-97)
- `g_chain_handoff.mem_regions[0].length = 0x40000000ULL` (1 GB flat).
- Does not parse bi->mem_map_addr/bi->mem_map_count to enumerate real EFI
  memory descriptors. xchain_run stages that consume memory map will see
  inaccurate RAM layout.
- Acceptable for Sprint 11 (PMM is separate). Must be fixed before Sprint 12
  process isolation which requires accurate memory regions for process mapping.

### INFO: boot_handoff.h _tail_pad formula risk (boot_handoff.h:198-218)
