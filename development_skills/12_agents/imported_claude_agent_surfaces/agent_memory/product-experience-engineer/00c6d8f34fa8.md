## Critical UX Gaps Found in Sprint 3 APIs

### XPKG — No real-time progress event mechanism
- xpkg_state_t has 6 states (AVAILABLE, DOWNLOADING, INSTALLING, INSTALLED, REMOVING, BROKEN)
  but there is NO callback/event hook for state transitions.
- The shell has no way to know when a package moves from DOWNLOADING to INSTALLING — it would
  have to poll xpkg_get_info() on a timer. This is architecturally hostile to reactive UX.
- REQUIRED for Sprint 5: xpkg_progress_cb_t — a callback registered at xpkg_init() that fires
  on every state transition with (pkg_name, old_state, new_state, percent_complete).

### XPKG — Gate report not surfaced to user
- xpkg_gate_report_t contains 9 gate results with 128-byte detail strings — rich human-readable
  error context. But xpkg.c's print_error() only prints the top-level status code.
- Gate 0 SIGNATURE failure = "signature verification failed" — same string as a corrupted key.
- Gate 2 CAPABILITY failure = "refusal gate rejected package" — user has no idea what capability.
- REQUIRED: Gate report must be exposed over D-Bus so the shell can render a structured failure
  dialog (which gate, what detail, what the user can do about it).

### XPKG — install/upgrade are VFS stubs in Sprint 3
- xpkg.c cmd_install() explicitly prints: "VFS not yet integrated (Sprint 4). Use API directly."
- This means the GUI package manager in Sprint 5 CANNOT call the CLI. It must call the C API
  directly via a privileged service bridge. Design the bridge now, not in Sprint 5.

### XNET — No link state event / carrier detect hook
- xnet_netif_t is forward-declared (full definition in netif.c). No callback for IF_FLAG_RUNNING
  transitions. The TopBar WiFi widget has no kernel-sourced event to react to — it currently
  polls the D-Bus bridge state.
- REQUIRED: xnet_if_state_cb_t — fires when XNET_IF_FLAG_UP or XNET_IF_FLAG_RUNNING changes.
  Bridges to D-Bus dbus_state_bridge.py for shell consumption.

### XNET — xnet_if_stats_t is rich but unconnected to shell
- rx_bytes, tx_bytes, rx_errors, tx_errors, rx_dropped are all tracked.
- No shell widget consumes this data. No D-Bus schema field for network throughput.
- Sprint 5 opportunity: real-time bandwidth widget in TopBar system tray area.

### XNET — Gossip/mDNS present but UX story is undefined
- GOSSIP_NODE_ALIVE/SUSPECT/DEAD states are meaningful — a suspect node may be a peer device
  the user was sharing documents with via CRDT sync (Sprint 2).
- When a gossip peer goes DEAD, the CRDT offline_first.c oplog diverges silently. The user
  needs to know their document is now in offline/pending-sync state.
- REQUIRED: Link gossip node state changes to the CRDT sync status indicator in Orange Suite.

### XSEC — TLS 1.3 handshake state machine has 10 states but no user-visible signal
- xsec_tls_state_t progresses IDLE → CLIENT_HELLO → ... → APPLICATION_DATA → CLOSED/ERROR
- XSEC_TLS_STATE_ERROR is the only failure state — it does not distinguish alert type (see
  XSEC_ERR_TLS_ALERT, XSEC_ERR_TLS_HANDSHAKE, XSEC_ERR_CERT_EXPIRED, XSEC_ERR_CERT_INVALID).
- The browser padlock must map these to three distinct states: SECURE / INSECURE / CERT_ERROR.

### XSEC — xsec_audit_ctx_t ring buffer is kernel-only
- 1024-entry audit ring with HANDSHAKE_COMPLETE, CERT_REJECTED, AUTH_TAG_MISMATCH events.
- xsec_audit_query_recent() exists but nothing reads it and pipes it to the UI.
- Sprint 5 opportunity: Security Event Log in the System Settings app (future).

## Sprint 5 Shell Integration Recommendations (Prioritized)

### R1 — XPKG Progress Bridge (P0 — blocks package manager UX entirely)
Design a privileged GENSD service (xpkg-daemon) that:
1. Calls xpkg_init() on boot with system-level trust keys.
2. Exposes a D-Bus interface: genos.PackageManager with signals:
   - PackageStateChanged(name: string, state: xpkg_state_t, percent: u8)
   - GateReportAvailable(name: string, gate_results: array of {gate_id, result, detail})
   - RepoRefreshComplete(repo_name: string, count: u32, success: bool)
3. The shell TopBar (or a dedicated Software widget) subscribes to these signals.
4. The XFRAME PackageInstaller widget (Sprint 5) renders a multi-step progress UI driven by
   PackageStateChanged events.

### R2 — XNET Link-State D-Bus Signal (P0 — WiFi widget accuracy)
Extend dbus_state_bridge.py to subscribe to an XNET link-state event (when implemented):
- Add wifi.tx_bytes and wifi.rx_bytes to BridgeState schema
- Add wifi.link_quality (0-100, derived from rx_errors/rx_packets ratio)
- TopBar WifiWidget gets animated strength bars driven by link_quality, not binary connected flag

### R3 — Browser Security Indicator Model (P1)
Map xsec_tls_state_t + xsec_status_t to three browser chrome states:

  XSEC_TLS_STATE_APPLICATION_DATA + XSEC_OK          → SECURE (green padlock)
  XSEC_TLS_STATE_APPLICATION_DATA + XSEC_ERR_CERT_*  → CERT_WARN (yellow shield)
  XSEC_TLS_STATE_ERROR + any alert                   → INSECURE (red broken padlock)
  XSEC_TLS_STATE_IDLE or connecting states            → LOADING (animated ring, no padlock)

The GENESYS Browser preload bridge needs a new channel: shell:tls-state { url, state, cipher }
so the renderer can update the address bar lock icon without re-inspecting cert data.

### R4 — First-Run WiFi Setup (P1 — FirstRunWizard.tsx integration)
FirstRunWizard currently has no network setup step. With XNET available:
- Add Step 3 "Connect to Network" between account creation and companion setup.
- Call xnet_netif_up() via the GENSD bridge, then xnet_dns_resolve() as a connectivity test.
- Show: interface list (xnet_netif_get_ipv4) + SSID field + passphrase field.
- On success: green check + "Connected to [SSID]". On failure: "Could not connect — check
  passphrase or try ethernet" with a Skip button (offline mode is valid).
- mDNS announce (xnet_mdns_announce) after successful connection so the device appears to peers.

### R5 — Package Discovery UI (P2 — Software Center)
xpkg_repo_search() returns name, version, size_bytes, dep_names[256].
This is sufficient to render a basic Software Center card grid:
- Card: package name (large), version badge, size in human-readable (KB/MB/GB), install button.
- Search input debounced 300ms → xpkg_repo_search() → re-render results.
- Empty state: "No packages match your search" with a clear-search action.
- Installed state: green "Installed" chip replacing install button (derived from xpkg_list_installed).
- Broken state: orange "Repair" button (xpkg_upgrade with same version = reinstall).

### R6 — Offline Mode Clarity (P2 — CRDT + XPKG interaction)
When XNET link is DOWN (xnet_netif_down event):
- Orange Suite apps show a subtle "Offline — changes will sync when reconnected" banner.
  This maps to CRDT offline_first.c oplog state already in Sprint 2.
- XPKG disables repo-refresh and remote install commands — shows "Network required" inline.
  Package install from local .xpkg file (VFS, Sprint 4) remains available.
- The Software Center Install button changes to "Download when online" if the user taps while
  offline — queues the install intent, executes when xnet_netif_up fires.

### R7 — Error State Lexicon (P2 — all surfaces)
Map every XPKG and XNET error code to a user-facing string + recovery action:

XPKG_ERR_SIGNATURE  → "This package cannot be verified. It may have been tampered with."
                       [Cancel] [Install anyway — requires admin auth]
XPKG_ERR_REVOKED    → "The publisher's key has been revoked. This package is unsafe to install."
                       [Cancel only — no bypass]
XPKG_ERR_CONFLICT   → "Installing [X] would break [Y] (v1.2.0). Update [Y] first, then retry."
                       [Cancel] [Show affected packages]
XPKG_ERR_INTEGRITY  → "The download appears corrupted. Try downloading again."
                       [Retry] [Cancel]
XPKG_ERR_CAPACITY   → "Not enough storage space ([N] MB required, [M] MB available)."
                       [Open Storage Settings] [Cancel]
XNET_ERR_TIMEOUT    → "Network timed out. Check your connection and try again."
                       [Retry] [Use offline mode]
XNET_ERR_HOSTUNREACH→ "Cannot reach the package server. Check DNS or try a different network."
                       [Retry] [Add manual repo]
XSEC_ERR_CERT_EXPIRED → "This site's security certificate expired on [date]. Proceed with caution."
                       [Go back] [Proceed (advanced)]

## XPKG CLI UX Assessment — Current State (Sprint 3)

### What works well
- 16 human-readable error strings in print_error() — every status code maps to natural language.
- print_pkg_info() outputs structured data: name, version, description, author, license, size,
  install path, state, dep count, file count.
- search output uses %-32s fixed-width columns — readable in a terminal.
- repo-refresh shows per-repo status + totals: "2 OK, 1 failed" is clear.
- cmd_list empty state prints "(none)" — edge case handled correctly.
- xpkg_gate_all_pass() boolean simplifies the install decision path.

### What is missing for GUI integration
1. No machine-readable output mode (--json flag). The GUI cannot parse pal_console_printf output.
   Recommendation: xpkg_main_json() variant that writes JSON to a caller buffer instead of console.
2. No progress percentage. XPKG_STATE_INSTALLING gives no indication of how many of 256 files
   have been written. Add: xpkg_install_progress_t { uint32_t files_done; uint32_t files_total; }
3. No download progress. XPKG_STATE_DOWNLOADING has no bytes_received / bytes_total pair.
   When XNET is integrated in Sprint 4, the fetch layer must report progress back to XPKG.
4. Disk size shown in bytes — needs human-readable formatting for GUI labels (KB / MB / GB).

## 9-Gate Refusal Machine — UX Design Implications

The 9 gates run in sequence on every install. From a UX perspective:

Gates 0-1 (SIGNATURE, INTEGRITY) — Hard block. No user bypass. Error is non-negotiable.
  UI: Red shield icon. "This package is unsafe" message. Only action: Cancel.

Gate 2 (CAPABILITY) — Hard block by default. Admin can force[2]=true.
  UI: Orange warning. Show capability list. "This package requests [X]. Allow?"

Gate 3 (PROVENANCE) — Warn for COMMUNITY trust, fail for unknown key.
  UI: Yellow badge on package card. "Published by community contributor. Install with caution."

Gates 4-5 (VERSION, CONFLICT) — Contextual. Show resolution plan.
  UI: Dependency tree view. "Installing X will also update Y and Z."

Gate 6 (SIZE) — Show storage impact clearly before confirmation.
  UI: "This package requires 450 MB. You have 2.1 GB available." Progress ring on disk icon.

Gates 7-8 (DEPENDENCY, POLICY) — Surface as a checklist in the install confirmation dialog.
  UI: Expandable "What will be installed" section showing xpkg_install_plan_t entries.

The install confirmation dialog should be a single-screen summary that collapses the full
gate report into three sections: SECURITY (gates 0-3), COMPATIBILITY (4-5), RESOURCES (6-8).

## Trust Level → Visual Treatment

XPKG_TRUST_SYSTEM    → Blue "GEN.OS Verified" badge (shield + checkmark)
XPKG_TRUST_USER      → Green "Trusted by You" badge (person + checkmark)
XPKG_TRUST_COMMUNITY → Amber "Community" badge (group icon, no checkmark)
Revoked key          → Red "Publisher Revoked" badge — install blocked

These badges appear on:
- Package cards in the Software Center
- The install confirmation dialog header
- xpkg info output (as a visual label, not just text)
