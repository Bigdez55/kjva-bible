## 2. Error Message Quality Assessment: 6/10

### What Works
- print_error() in xpkg.c maps all 16 status codes to human-readable strings
- The format "[xpkg] ERROR: <command> failed: <reason> (code <N>)" is machine-parseable
  AND human-readable — good dual-mode design
- Gate names are printed inline during the check loop (e.g., "[gate 3/PROVENANCE]")

### Deficiencies

**SEVERITY HIGH — Gate failure gives no gate name to the user**
xpkg_install() calls print_error("install", XPKG_ERR_GATE) after a gate failure.
The user sees:
  [xpkg] ERROR: install failed: refusal gate rejected package (code -6)
They do NOT see which gate failed or what the gate's detail message said.
The gate_report is fully populated at this point — it is simply not surfaced via the CLI.

**SEVERITY HIGH — XPKG_ERR_GATE is a bucket error code**
All 9 gates share a single error code. A policy block (gate 8) looks identical to a
signature failure (gate 0) at the CLI output level. The user cannot distinguish them
without reading raw gate-report logs that are interleaved with install progress logs.

**SEVERITY MEDIUM — VFS stub messages expose internals**
"NOTE: VFS not yet integrated (Sprint 4). Use xpkg_install() API directly with
archive buffer." — this is an internal engineering note, not a user message.
A user will read this and have no idea what to do.

**SEVERITY MEDIUM — remove failure does not name the dependent**
"[remove] FAIL: 'foo' is required by other packages" does not name WHICH packages
depend on foo. The user cannot take corrective action without that information.

**SEVERITY LOW — repo-refresh partial failure returns XPKG_ERR_STORAGE**
If 1 of 3 repos fails to refresh, the error code is XPKG_ERR_STORAGE, which implies
a local disk problem. The actual issue is a network fetch failure. Code-to-message
mapping is misleading here.

**SEVERITY LOW — "Already initialized" on re-init is not an error but returns XPKG_OK
with a console message. Caller cannot distinguish silent re-init from first init.**

## 3. Refusal Gate UX Analysis

### Gate Detail Message Quality (per gate)

| Gate | Name        | FAIL Detail                                      | Quality |
|------|-------------|--------------------------------------------------|---------|
| 0    | SIGNATURE   | "No trusted key validated this package signature"| GOOD    |
| 0    | SIGNATURE   | "Publisher key is revoked"                       | GOOD    |
| 1    | INTEGRITY   | "SHA-256 mismatch: archive has been tampered with"| EXCELLENT — names the attack |
| 2    | CAPABILITY  | "Package targets restricted path — requires SYSTEM trust review" | GOOD (WARN) |
| 3    | PROVENANCE  | "No repositories configured — provenance cannot be verified" | GOOD |
| 4    | VERSION     | "New version is older than installed version (downgrade blocked)" | EXCELLENT |
| 5    | CONFLICT    | "File path collision with installed package"     | POOR — no path, no package name |
| 6    | SIZE        | "Package size exceeds disk quota"                | POOR — no numbers shown |
| 7    | DEPENDENCY  | "Required dependency not currently installed — will attempt resolution" | WARN, not FAIL — OK |
| 8    | POLICY      | "Package blocked by administrator policy rule"   | GOOD |

### Best Gate Message
Gate 1 (INTEGRITY) and Gate 4 (VERSION) produce the most actionable messages.
Gate 1 explicitly says "tampered with" which is the correct user-facing framing for
a hash mismatch — it avoids cryptographic jargon while being precise.
Gate 4 says "downgrade blocked" which names the policy instead of just saying "failed".

### Worst Gate Messages
Gate 5 (CONFLICT): "File path collision with installed package" — does not name the
conflicting package or the conflicting file path. The user cannot resolve this without
running a manual diff.
Gate 6 (SIZE): "Package size exceeds disk quota" — does not show the package size or
the quota limit. The user cannot determine how much space to free.

### Gate Report CLI Accessibility
CRITICAL GAP: The xpkg_gate_report_t struct is fully populated and available inside
xpkg_install() after the gate check. The detailed per-gate results ARE computed. But
they are only emitted via the internal logging loop (pal_console_printf "[gate-report]
Gate[N] PASS/WARN/FAIL: <detail>"). The CLI cmd_install() never receives or displays
this report — it only sees XPKG_ERR_GATE.

There is no `xpkg gate-report <name>` command, no persistent gate report storage,
and no way to retrieve the last failure report after the install returns.

## 4. Installation Feedback Assessment

### Progress Signals Present
1. "[install] Installing '<name>' v<M>.<m>.<p>" — start banner: GOOD
2. "[gate N/NAME] PASS/WARN/FAIL: <detail>" — per-gate live logging: GOOD
3. "[install] Dependency plan: N package(s)" — dep count shown: GOOD
4. "[install] [INSTALL/UPGRADE/KEEP] '<dep>' v<M>.<m>.<p>" — per-dep action: GOOD
5. "[install] extract: <path> (N bytes, mode=XXXX)" — per-file extraction: VERBOSE
6. "[install] Extracted N files" — extraction summary: GOOD
7. "[install] SUCCESS: '<name>' installed at slot N" — completion: ADEQUATE

### Feedback Deficiencies
- "installed at slot N" exposes the internal registry array index to the user.
  The slot number has no user-facing meaning.
- No timing information (how long the install took)
- No total byte count / disk space consumed in the success message
- Extraction logging is file-by-file (up to 256 files) with no progress percentage.
  On a 256-file package this produces 256 console lines before the summary.
- No visual distinction between informational progress and warnings/errors.
  All output uses the same [install] prefix regardless of severity.

### Rollback Communication
- Rollback IS automatic on upgrade failure (xpkg_upgrade re-registers old_pkg_copy
  with XPKG_STATE_BROKEN)
- The message "[upgrade] ROLLBACK: restoring old version '<name>'" IS emitted
- However the restored package is marked BROKEN, not INSTALLED — a user running
  `xpkg list` after a failed upgrade will see their package listed as [broken],
  which is alarming and requires follow-up documentation/help text

## 5. Package Discovery Assessment

### Search Capability
- `xpkg search <query>` exists and works
- Case-insensitive substring match across name AND dep_names fields
- Results capped at 32 entries (hardcoded in cmd_search)
- Priority-ordered across multiple repos

### Discovery Deficiencies
- CRITICAL (Sprint 3): search always returns 0 results because repo_refresh() is a
  stub that populates 0 packages. The index is always empty until XNET is live.
  The user will search, get "No packages found", and have no indication this is a
  stub behavior vs. a genuine empty index.
- No `xpkg search` with empty query does NOT list all packages (it does, technically,
  but the stub issue means 0 results always)
- No category/tag filtering
- No "installed" vs "available" filter flag (e.g., xpkg list --broken)
- search results show size_bytes but not description — user cannot evaluate packages
  from search results alone; must follow up with `xpkg info`
- No `xpkg repo-list` to see configured repos — after adding repos the user has no
  read-back command to confirm what is configured

## 6. UX Friction Inventory (by severity)

### P0 — Blocks meaningful use
1. install/upgrade/verify commands are VFS stubs — the only functional install path
   is the C API, not the CLI. Users cannot install any package via the CLI in Sprint 3.
   The stub message tells them to "use the API directly" which is not a user action.

2. Gate failure does not surface the gate name or detail message via the CLI. The user
   sees "refusal gate rejected package (code -6)" with no path to diagnosis.

3. repo-refresh is an XNET stub — all repos return 0 packages. search, info (repo
   fallback), and any dependency resolution from repos are non-functional.

### P1 — Significant friction, workaround exists
4. Gate 5 (CONFLICT) failure gives no conflicting package name or file path.
5. Gate 6 (SIZE) failure gives no numbers (package size, quota limit).
6. remove failure for reverse-dependency block names no dependents.
7. "installed at slot N" success message leaks internal implementation detail.
8. Rollback leaves package in BROKEN state with no recovery instruction.
9. No `xpkg repo-list` command — cannot verify repo configuration.

### P2 — Polish / progressive disclosure
10. Per-file extraction logging generates up to 256 lines with no progress %.
11. VFS stub messages contain internal engineering commentary.
12. repo-refresh partial failure returns misleading XPKG_ERR_STORAGE code.
13. search results omit description field — requires a second `info` lookup.
14. No `xpkg gate-report <name>` for post-install gate history.
15. No `xpkg list --broken` or `xpkg list --available` filters.

## 8. Top 3 UX Improvements for Sprint 5

### Improvement 1 (P0): Gate Failure Report Surfacing
The gate_report is already fully populated at the point of failure. Sprint 5 should:
- Print the failing gate name and detail string as part of the CLI error output
- Persist the last gate report per package name in a static xpkg_last_gate_report
  module variable
- Add `xpkg gate-report <name>` command to retrieve it

Target output when gate 5 fails:
  [xpkg] ERROR: install failed at Gate [5/CONFLICT]
  [xpkg]   detail: File path collision — /opt/genos/bin/xterm owned by 'xterm-base'
  [xpkg]   Run 'xpkg gate-report xterm-ng' for the full report.

### Improvement 2 (P0/P1): Actionable Stub Messaging
install/upgrade/verify commands should output a single clear stub notice that
communicates user-visible impact, not engineering status:
  [xpkg] 'install' requires VFS (available in GEN.OS Sprint 4).
  [xpkg] To install a package now, use the programmatic API: xpkg_install().
  [xpkg] Tracking: S4-VFS-01

This replaces the current engineering note with a time-bounded, reference-linked
message that a developer user can act on.

### Improvement 3 (P1): Quantitative Context in Gate Messages
Gates 5 and 6 need data in their failure strings. The detail buffer is 128 bytes —
sufficient for a path or a size comparison. Specifically:
- Gate 5 CONFLICT: include the first conflicting file path and the owning package name
- Gate 6 SIZE: include the package size in human-readable form (e.g., "1.2 GB")
  and the quota limit
- Gate 4 VERSION: already excellent — no change needed

These changes require no new infrastructure — only the detail string composition
in the two gate functions needs updating.

## 9. Positive Findings

- Command dispatch table (s_commands[]) is clean and extensible — adding a new
  command is a one-line addition
- Dual-mode flags (--help/-h, --version/-v) follow POSIX conventions
- help text includes a 9-gate summary — users can see what the security pipeline
  does without reading source
- Gate WARN semantics are correct: advisory, non-blocking, logged — this is
  the right UX model for uncertain cases (provenance, reinstall, dependency pending)
- Rollback on upgrade failure IS implemented and communicated (P0 concern addressed
  for upgrade path specifically)
- list_cb correctly shows [broken] state alongside [installed] — state is visible
- Dependency plan preview before extraction (Step 3 output in xpkg_install) is
  excellent UX — user sees what will happen before it happens
- Case-insensitive search is correct default behavior for a package manager

## 10. Key Constants / Limits the UX Team Should Know
- Max 512 installed packages (XPKG_MAX_INSTALLED)
- Max 16 repos (XPKG_MAX_REPOS)
- Max 32 search results hardcoded in cmd_search (not a constant, inline literal)
- Gate detail buffer: 128 bytes (xpkg_gate_check_t.detail)
- Max 64 policy rules (XPKG_POLICY_RULES_MAX)
- Default disk quota: 4 GB (xpkg_gate_config_default)
- Default install prefix: /opt/genos
