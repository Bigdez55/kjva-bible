**Auditor:** Product Experience Engineer
**Scope:** Sprint 5 — XSHELL native shell + Orange Suite C apps + FirstRunWizard.tsx
**Prior audit:** xshell-orange-audit-2026-03-06.md (reference)
**Method:** White-box source code inspection of all 7 XSHELL components, 4 Orange apps,

## 1. Shell Component Audit

### 1.1 XSHELL Core (xshell.c)

**Score: 2 — Partial**

**Elements rendered:**
- Scene root (full-screen column, 1920x1200, token color)
- Topbar + Dock wired into scene root children
- App registration table (32 slots), notification ring buffer (16 slots, 5s auto-dismiss)
- App launch/close/focus lifecycle with window_root BOX nodes

**XFRAME API usage:** Correct and thorough. `xframe_init`, `xframe_app_create`,
`xframe_node_create/set_direction/set_size/set_bg_color/set_flex_grow/add_child`, `xframe_run`.

**Keyboard navigation:** None implemented. Input dispatch is caller's responsibility (no key
event loop in xshell.c itself).

**Touch/tablet input:** Not present. No touch event routing, no swipe-from-edge gesture.

**Design tokens:** 100% token-referenced. `XTOKEN_COLOR_BG_PRIMARY`, `XTOKEN_COLOR_BG_SECONDARY`
used throughout. Zero hardcoded ARGB values.

**Findings:**

PEE-XSHELL-01: P1 — No inactivity-driven lockscreen trigger. `xshell_tick` auto-dismisses
notifications but never calls `xshell_lockscreen_lock()` on idle timeout. A user who walks
away from the machine without manually locking is unprotected. User impact: session hijacking
on unattended device. Recommendation: Add idle_since_ms field to xshell_t; call
xshell_lockscreen_lock at configurable timeout (default 5 min) from tick.

PEE-XSHELL-02: P2 — No window management chrome. `xshell_app_launch` creates a minimal BOX
window_root but no title bar, close button, resize handles, or minimize control. Users have
no discoverable mechanism to quit or manage apps from the window itself.
User impact: apps feel like kiosks, not a desktop OS.
Recommendation: Sprint 6 — add xshell_window_t with title bar chrome (48px) containing app
name + close/minimize buttons wired to xshell_app_close.

PEE-XSHELL-03: P2 — No empty-state for notification bell. `notif_node` text is always "BELL"
regardless of whether there are 0 or 16 notifications. No badge count visible.
User impact: users cannot know at a glance that they have unread notifications.
Recommendation: On each tick, format notif_count into bell node text: "0" vs "3".
Color: neutral when 0, XTOKEN_COLOR_WARNING when >0.

PEE-XSHELL-04: P3 — `xshell_tick` is declared with `__attribute__((unused))` — it is NEVER
registered with xframe_run as the per-frame callback. Notification auto-dismiss and sub-component
ticks (topbar clock, dock auto-hide) will never fire at runtime.
User impact: clock never updates, dock never auto-hides, notifications never expire.
Recommendation: Wire tick as xframe_app callback before calling xframe_run().

### 1.2 Topbar (topbar.c)

**Score: 2 — Partial**

**Elements rendered:**
- Left: active app title (flex-grow, XTOKEN_COLOR_TEXT_PRIMARY)
- Center: clock (HH:MM, token color, 24px horizontal margin)
- Right group (flex-end, gap 12): Wifi text, Battery %, Volume "VOL", Notification "BELL"

**XFRAME API usage:** Full. All nodes use `xframe_node_set_*` property setters.
Clock tick fires every 1000ms with ms-based time derivation (no real-time clock; derived from boot epoch).

**Keyboard navigation:** N/A — topbar is display-only.

**Touch/tablet input:** Not present. Right-side indicators are not tappable.

**Design tokens:** 100% compliant. Battery color uses green/yellow/red thresholds via token
constants. No hardcoded values.

**Findings:**

PEE-TOPBAR-01: P1 — Clock derives time from ms-since-boot, not wall clock. At every boot,
clock starts at 00:00 and advances from there. A device booted at 14:30 shows 00:00 → 00:01...
User impact: incorrect system time display — fundamental utility failure.
Recommendation: Add real-time clock read via PAL (pal_rtc_get or XACPI/HPET time base).
Sprint 9 XHPET provides the counter. An RTC read from CMOS (I/O port 0x70/0x71) needs a PAL
wrapper.

PEE-TOPBAR-02: P2 — Volume indicator is static text "VOL" with no numeric value or dynamic
update path. `xshell_topbar_set_*` API exists for wifi and battery but not volume.
User impact: no volume feedback, no mute/unmute control surface.
Recommendation: Add xshell_topbar_set_volume(tb, uint32_t pct, int muted) API.

PEE-TOPBAR-03: P2 — Notification bell is static text "BELL" with no count badge and no
interactive action. Clicking it should open a notification center overlay or drop-down.
User impact: notifications accumulate invisibly with no review path.
Recommendation: Wire bell node click to show a notifications panel (new component or launcher-
style overlay listing active xshell_notif_t entries).

PEE-TOPBAR-04: P3 — WiFi RSSI value is read into tb->wifi_rssi but not displayed. tb_format_wifi
ignores the rssi parameter entirely (line 85: `(void)rssi`). Users see only "WiFi" or "No WiFi"
with no signal strength indication.
Recommendation: Map RSSI to signal bars (e.g., ≥ -50 = "WiFi 4", ≥ -70 = "WiFi 3", etc.)
or use a Unicode signal-strength glyph if font supports it.

### 1.3 Dock (dock.c)

**Score: 2 — Partial**

**Elements rendered:**
- Root: centered row, XTOKEN_COLOR_BG_SECONDARY, radius=XTOKEN_RADIUS_LG, padding 8/16
- Per entry: icon cell (64x80) → icon_bg (48x48, XTOKEN_RADIUS_MD) → 2-letter label
  → app name label (12px height) → running dot (6x6 circle, visible when running)
- Auto-hide: after XSHELL_DOCK_HIDE_DELAY (2000ms) of no hover, `xframe_node_set_visible(root, 0)`

**XFRAME API usage:** Full. `dk_rebuild_nodes` reconstructs the entire node tree on each
pin/unpin or running-state change.

**Keyboard navigation:** None. No Tab/arrow key traversal of dock items.

**Touch/tablet input:** `xshell_dock_on_hover` exists for hover events, but no touch
tap API. On a touchscreen, pointer-hover events won't fire reliably; the dock will
auto-hide without recovery.
User impact: on EliteBook x360 in tablet mode, dock becomes inaccessible after hide.
Recommendation: Add xshell_dock_on_touch(dock, int x, int y) — any touch within ±64px of
bottom edge shows dock.

**Design tokens:** 100% compliant. Running dot uses XTOKEN_COLOR_ACCENT.

**Findings:**

PEE-DOCK-01: P1 — Dock icon is a 2-character text abbreviation (dk_make_icon_label). First 2
chars of app name are used (e.g., "No" for "Notes", "Ca" for "Calendar"). This is not
visually distinguishable for apps sharing the same first two letters. Icons lack visual identity.
User impact: brand confusion and low recognition for first-time users.
Recommendation: Sprint 6 — implement icon asset loader (PNG/SVG fallback from XBLOB store).
As an interim measure, use first letter only with a per-app accent color derived from a color
table keyed to app name hash.

PEE-DOCK-02: P1 — Touch/tablet auto-hide recovery absent. On tablet mode, once dock
auto-hides after 2 seconds with no hover, there is no swipe gesture or bottom-edge touch
target to bring it back. Keyboard shortcut (Ctrl+N for launcher) is documented but not
wired. Dock recovery requires hover, which tablets don't emit.
User impact: P1 on HP EliteBook x360 tablet mode — dock permanently disappears.

PEE-DOCK-03: P2 — Click/tap actions on dock icons are not wired. Dock icon BOX nodes have
no XFRAME event callback. Clicking a pinned app icon does not launch the app.
User impact: dock is display-only, not interactive.
Recommendation: Wire xframe_node event callback when XFRAME event API is available; add
launch-on-click handler calling xshell_app_launch(dock->entries[i].name, NULL).

PEE-DOCK-04: P3 — Running dot uses transparent background (0x00000000) when not running,
which relies on parent's background being a distinct color. If parent bg changes (e.g., theme
change), invisible dots become visually absent rather than empty-circle. Should use a bordered
empty-circle instead for clearer empty-state.

### 1.4 Launcher (launcher.c)

**Score: 2 — Partial**

**Elements rendered:**
- Full-screen overlay (1920x1200, XTOKEN_COLOR_SCRIM, centered, gap 24, hidden by default)
- Search bar container (600x48, XTOKEN_RADIUS_FULL, padding 24, with "Search" label + query text)
- App grid (600px wide, row wrap, gap 16)
- Tile per app (120x120, XTOKEN_COLOR_BG_TERTIARY, XTOKEN_RADIUS_LG):
  icon box (64x64, XTOKEN_COLOR_ACCENT) + first-letter label + app name label

**Catalog:** Hard-coded 8 apps: Notes, Calendar, Mail, Drive, Terminal, Browser, Settings, Files.

**Keyboard navigation:** Partial.
- `xshell_launcher_on_keychar(l, c)` — appends printable ASCII to search query, re-filters.
- `xshell_launcher_on_backspace(l)` — removes last char.
- No arrow-key navigation between tiles. Tab focus between tiles is absent.
- No Enter key to launch the selected tile.

**Touch/tablet input:** Not present. Tile tap events not wired.

**Design tokens:** 100% compliant. XTOKEN_COLOR_SCRIM for overlay, XTOKEN_COLOR_ACCENT for icon.

**Findings:**

PEE-LAUNCHER-01: P1 — No empty state when search returns zero matches. If user types a query
matching no app, `la_rebuild_tiles` adds no nodes to grid_node. Grid displays as blank area
with no "No results" message. User has no feedback that their search failed vs. the UI is broken.
User impact: user confusion, perceived system freeze.
Recommendation: After la_rebuild_tiles, if idx==0 and search_len>0, add a single TEXT node
with "No results" (XTOKEN_COLOR_TEXT_SECONDARY, centered).

PEE-LAUNCHER-02: P1 — Tile click/tap not wired. Tiles are 120x120 BOX nodes with no event
callback. Launcher cannot actually launch apps.
User impact: launcher is a search display with no action.
Recommendation: Same as dock — wire click callback to xshell_app_launch(app_name, NULL) when
XFRAME event API is available.

PEE-LAUNCHER-03: P2 — No arrow-key navigation or focus ring. A keyboard user who types a
search query and sees results has no way to select and launch a result with keyboard alone
(no arrow + Enter flow).
Recommendation: Track selected_idx state; arrow keys change it; Enter triggers launch.
Highlight selected tile with XTOKEN_COLOR_ACCENT border.

PEE-LAUNCHER-04: P2 — App catalog is a 8-entry static const array baked at compile time.
There is no mechanism for dynamically registered or user-installed apps to appear in the
launcher. As the OS grows (Sprint 6+), all new apps will be invisible from the launcher
without a source code change.
Recommendation: Expose xshell_launcher_register_app(l, name, icon_path) API allowing
xshell_init and future app install hooks to populate the catalog dynamically.

PEE-LAUNCHER-05: P3 — Search prefix match is left-anchored only. Searching "rive" won't
match "Drive". For a 8-app catalog this is low impact now; will become significant as catalog
grows. Recommendation: Change to substring match (strstr equivalent).

### 1.5 Lockscreen (lockscreen.c)

**Score: 2 — Partial (security-adjusted)**

**Elements rendered:**
- Full-screen overlay (1920x1200, 0xFF0A0A1A — hardcoded deep dark, NOT a token — see finding)
- Large clock (HH:MM, XTOKEN_COLOR_TEXT_PRIMARY)
- 6 PIN dots (20x20 circles, bordered, fill-on-entry via ls_update_dots)
- "Incorrect PIN" label (XTOKEN_COLOR_ERROR, hidden by default)
- Numeric keypad: 10 keys (72x72 circles, row layout, 1-9 then 0)

**Inactivity timeout:** NOT present in lockscreen.c itself. Must be triggered externally
via xshell_lockscreen_lock(ls). See PEE-XSHELL-01 — xshell_tick never calls it.

**PIN entry:** xshell_lockscreen_on_digit() appends digits, auto-verifies at 6 digits via
xshell_lockscreen_unlock().

**Failed attempts counter:** YES. `attempt_count` incremented per failure; input rejected
at XSHELL_PIN_MAX_ATTEMPTS (5). Fail message shown.

**Lock state atomicity:** ls->locked set to 1 synchronously within xshell_lockscreen_lock().
No spinlock — single-threaded XFRAME; safe.

**Findings:**

PEE-LOCK-01: P0 — PIN hash is XOR-accumulation stub (ls_compute_hash). Fixed 6-byte key
{0x5A, 0x3C, 0x1F, 0x72, 0x4B, 0x8E}. An attacker who reads stored_hash can trivially
reverse each digit: digit[i] = stored_hash[i] ^ KEYS[i]. This is not a hash — it is
symmetric encryption with a hardcoded key stored alongside the ciphertext. All 6 stored
bytes individually reveal each digit. Security collapses to zero.
Source comment acknowledges: "NOT cryptographically secure — replaced by XSEC SHA-256 in Sprint 6."
User impact: P0 — full credential exposure if storage is accessible.
Recommendation: Sprint 6 — replace with xsec_sha256(salt || digits) where salt is
random 32 bytes stored separately. XSEC SHA-256 is implemented (Sprint 3). This is a
contractual Sprint 6 item, not an oversight.

PEE-LOCK-02: P1 — Default PIN "000000" is silently accepted when no PIN has been set
(ls->hash_set == 0). There is no requirement to set a PIN before the lockscreen activates.
A device that has never had a PIN configured trivially unlocks with 000000. No UI warning
is presented that the default PIN is in use.
User impact: first-time users are unprotected without knowing it.
Recommendation: On first lock attempt when hash_set==0, show a "Set up PIN" flow instead
of locking with default. Or require PIN setup during OOBE/first-run.

PEE-LOCK-03: P1 — After XSHELL_PIN_MAX_ATTEMPTS (5) failures, keypad input is silently
blocked but the UI gives no updated feedback beyond the "Incorrect PIN" label shown after
the last failure. There is no "device locked — contact administrator" message, no lockout
timer display, no escalation path.
User impact: lockout appears as a broken UI rather than a security state.
Recommendation: On attempt_count >= max: replace fail_node text with "Too many attempts.
Device locked." and show a countdown timer via tick (e.g., 30-second cooldown). After
cooldown, allow 5 more attempts (or escalate to remote wipe path).

PEE-LOCK-04: P2 — Lockscreen background uses hardcoded ARGB 0xFF0A0A1A, not a design token.
This breaks theme-ability and is a code standard violation (confirmed from codebase policy:
never hardcode hex when a token exists). XTOKEN_COLOR_BG_LOCKSCREEN or XTOKEN_COLOR_NEUTRAL_900
should be defined and used.
User impact: visual inconsistency if theme changes; also discovered via security UX review
(lock screen appearance should be consistent and trust-affirming).

PEE-LOCK-05: P2 — Keypad is rendered as a flat ROW (not 3x4 grid). All 10 keys (1-9, 0) are
in a single horizontal row (xframe_node_set_direction(keypad, XFRAME_DIR_ROW)). This does not
match the conventional phone/ATM keypad layout (3 columns, 4 rows). The resulting UI is
10 circles in a row — unusable for muscle-memory PIN entry.
User impact: unfamiliar layout increases error rate and frustration for locked-out users.
Recommendation: Create 4 rows: [1,2,3], [4,5,6], [7,8,9], [backspace,0,enter].
Requires switching keypad to XFRAME_DIR_COLUMN with 4 row children each set to XFRAME_DIR_ROW.

PEE-LOCK-06: P2 — No backspace key on the keypad UI. `xshell_lockscreen_on_digit` accepts
digits but there is no on_backspace API for the lockscreen. If user mis-enters digit 3 of 6,
they must wait for auto-verify to fail, then re-enter all 6 digits. The keypad row has no
visual or functional backspace button.
Recommendation: Add xshell_lockscreen_on_backspace() API; add a Backspace key to the keypad
grid (at position 10 in the 3x4 layout).

PEE-LOCK-07: P3 — Touch targets: keypad keys are 72x72 pixels — above the 44px minimum.
This is correct. PIN dots at 20x20 are below the 44px minimum but they are display-only, not
interactive, so this is acceptable.

### 1.6 Greeter (greeter.c)

**Score: 2 — Partial**

**Elements rendered:**
- Full-screen overlay (1920x1200, XTOKEN_COLOR_BG_PRIMARY, column centered)
- Logo placeholder (120x120 BOX, XTOKEN_COLOR_ACCENT, XTOKEN_RADIUS_LG, "GEN.OS" label)
- Card (400x auto, XTOKEN_COLOR_BG_SECONDARY, XTOKEN_RADIUS_LG, XTOKEN_SPACE_LG padding, gap 16)
  - "Username" label + user_node BOX (input display, XTOKEN_INPUT_HEIGHT, token border)
  - "Password" label + pass_node BOX (masked display)
  - Session selector: static TEXT "Session: XSHELL"
  - Login button (XTOKEN_BTN_HEIGHT, XTOKEN_COLOR_ACCENT, "Log In")

**XFRAME API usage:** Full. Card uses XTOKEN_SPACE_LG, XTOKEN_INPUT_HEIGHT, XTOKEN_BTN_HEIGHT,
XTOKEN_RADIUS_INPUT, XTOKEN_BORDER_WIDTH_SM — excellent token coverage.

**Keyboard navigation:** No Tab key between username/password, no Enter key to submit.
Login callback (gr_on_login_click) is defined but never wired to login_btn event.

**Error state:** No error node. Authentication failure produces no user-visible feedback
(only pal_console_printf). The greeter has no error display region.

**Findings:**

PEE-GREETER-01: P1 — Login callback `gr_on_login_click` is defined but never wired to
login_btn as a click handler. XFRAME BOX nodes have no event API in Sprint 5 (same constraint
as dock/launcher). The login button is visually present but non-functional. The entire greeter
cannot authenticate a user.
User impact: device is unbootable to desktop (greeter blocks but cannot proceed).
Recommendation: Wire when XFRAME event API lands. Interim: handle login via keyboard Enter
key routed through xshell's input dispatch.

PEE-GREETER-02: P1 — No authentication error display node. If login_cb returns failure
(wrong password), there is no mechanism to show the user "Incorrect password." The caller's
callback would need to call greeter internals to update state, but no such API exists.
User impact: silent failure — user clicks Log In, nothing happens, no feedback.
Recommendation: Add fail_node (similar to lockscreen) to greeter struct; expose
xshell_greeter_set_error(g, msg) API.

PEE-GREETER-03: P1 — Username and password fields are BOX nodes with TEXT children, not
XFRAME text_input widgets (xwidget_input_t). They display empty string by default and have
no xwidget_input_on_char routing. There is no active field selection, cursor, or character
echo for either field. The greeter is visually complete but input is impossible.
User impact: No user can type their credentials.
Recommendation: Replace user_node and pass_node BOX+TEXT with xwidget_input_create() from
XFRAME widgets (text_input.c, fully implemented in Sprint 2). Wire xwidget_input_on_char
from greeter's keyboard event handler. Use xwidget_input_is_password=true for pass field.

PEE-GREETER-04: P2 — Session selector is a static TEXT node "Session: XSHELL". No toggle
or interaction to switch between XSHELL_SESSION_XSHELL and XSHELL_SESSION_FALLBACK. The
struct field g->session_type defaults to XSHELL_SESSION_XSHELL and is never changed.
User impact: users who need the fallback TTY session cannot access it.
Recommendation: Add clickable session toggle (Tab-style toggle button or dropdown) using
xshell_context_menu or a simple two-state button.

PEE-GREETER-05: P2 — gr_make_mask and gr_strcpy are defined but used only in the
_gr_unused suppression stub. Password masking (showing N asterisks as user types) is
planned but not connected to the pass_node display. The pass field shows nothing as user
types (if typing were even possible given PEE-GREETER-03).

PEE-GREETER-06: P3 — Logo node is 120x120 BOX with "GEN.OS" text. This is a placeholder.
The actual GEN.OS branding (orange G mark) should be rendered here via XFRAME paint or
an SVG asset. Low priority but visible on every boot.

### 1.7 Context Menu (context_menu.c)

**Score: 1 — Stub**

**Elements rendered:**
- Floating overlay (200px wide, auto height, XTOKEN_COLOR_BG_TERTIARY, XTOKEN_RADIUS_MD, border)
- Per item: row (36px height, padding 16px sides) with label TEXT (color indicates enabled/disabled)
- Separator: 1px horizontal line with 4px vertical margin
- Position: placed via `xframe_node_set_margin(overlay, y, 0, 0, x)` — correct offset approach

**XFRAME API usage:** Full for layout. Event wiring explicitly deferred.

**Click handling:** `cm_item_click` callback is defined and `s_click_ctxs` is pre-populated,
but the comment at line 129 states: "no XFRAME event API on BOX nodes in current sprint."
The `(void)cm_item_click;` suppression makes this explicit — click callbacks are connected to
nothing. The menu is display-only.

**Keyboard navigation:** Completely absent. No Up/Down arrow key navigation, no Enter, no Esc.

**Findings:**

PEE-CTX-01: P1 — All menu items are non-interactive. The entire component is display-only.
Context menu conceptually demands interaction — showing a menu that does nothing is a P1 UX
failure. User expectation: right-click → menu appears → click item → action fires. Current
state breaks step 3.
User impact: user loss of trust in OS responsiveness; may assume system is frozen.
Recommendation: This is blocked on XFRAME BOX event API landing. Priority task for Sprint 6:
implement xframe_node_set_on_click() as the first event API surface (minimal: click rect
hit-test, invoke callback). Unblock dock + launcher + context menu simultaneously.

PEE-CTX-02: P1 — No Escape key to dismiss menu. Menu stays visible until
xshell_context_menu_hide() is called externally. There is no keyboard or click-outside
dismissal logic. A shown context menu cannot be dismissed by the user at all.
User impact: context menu blocks interaction with rest of screen until caller dismisses it,
which currently has no trigger.
Recommendation: Wire Esc key in xshell's keyboard event dispatcher to call
xshell_context_menu_hide if menu is visible.

PEE-CTX-03: P2 — Position overflow not handled. `xshell_context_menu_show(m, x, y)` places
menu at exact (x, y) via margin. If x is 1800 with a 200px menu, it extends past 1920px.
If y is 1100 with 8 items x 36px = 288px, menu clips at 1200px.
User impact: context menu items cut off at screen edge — particularly likely on dock and
near desktop edges.
Recommendation: Clamp: effective_x = min(x, 1920 - 200); effective_y = min(y, 1200 - item_count * 36 - 8).

PEE-CTX-04: P3 — Item row height is 36px, which is below the 44px touch target minimum.
In tablet mode, menu items will be difficult to tap precisely.
Recommendation: Increase to 44px minimum row height.

## 2. Orange Suite App Audit

### 2.1 Orange Notes (notes.c) — Score: 2 — Partial

**XFRAME or XCOMP?** Pure XFRAME. All nodes via `xframe_node_create/set_*`. No raw XCOMP calls.

**XSTORE integration?** None. Notes are stored in a static 64-slot in-memory array (`s_notes[64]`).
No persistence across app destroy/create cycles.

**CRDT sync?** None. No xsync_crdt_* calls.

**Implemented:**
- 3-panel layout (Toolbar 48px, Sidebar 240px, Editor flex)
- CRUD: notes_on_new, notes_on_delete, notes_on_pin, notes_on_select
- Sidebar sorted: pinned notes first (orange dot), then by ID descending
- Editor: title + divider + body (plain TEXT nodes, not xwidget_editor_t)
- Default "New Note" created on launch — non-empty initial state (good FTUE)
- Title truncation with ellipsis at 32 chars

**Findings:**

PEE-NOTES-01: P1 — Editor uses XFRAME_NODE_TEXT not xwidget_editor_t. Notes body and title
are plain TEXT nodes with no cursor, selection, or character input routing. The editor
is display-only: `s_editor_body = xframe_node_create(XFRAME_NODE_TEXT)`. The XFRAME editor
widget (editor.c, Sprint 2) exists and supports cursor navigation, line ops, caret blink.
User impact: Notes cannot be typed into. Core feature absent.
Recommendation: Replace s_editor_title and s_editor_body with xwidget_editor_create(); route
keyboard events to xwidget_editor_on_keydown/char.

PEE-NOTES-02: P1 — No XSTORE persistence. All notes lost on app close. `orange_notes_destroy`
zeroes s_notes array. No xstore_put/get calls.
User impact: notes app is functionally a scratchpad that resets every session.
Recommendation: Sprint 6 — on notes_on_new/delete/pin: call xstore_put(XSTORE_NOTES_DB, id, serialized_note).
On orange_notes_create: query xstore_scan to restore notes from persistent B+ tree.

PEE-NOTES-03: P2 — No sidebar scroll. Sidebar list (s_sidebar_list) is a column BOX with
no xwidget_scroll_t wrapper. With 64 max notes, the list will overflow the visible area with
no scroll mechanism. Scroll widget (scroll.c, G-03 RESOLVED in Sprint 4) is available.
Recommendation: Wrap s_sidebar_list in xwidget_scroll_create with XSCROLL_FLAG_VERTICAL.

PEE-NOTES-04: P2 — Sidebar note items (48px height) have no click/tap action. Tapping a note
does not call notes_on_select(). Same XFRAME event gap.

PEE-NOTES-05: P3 — Toolbar buttons "New", "Delete", "Pin" are 72x32px — below 44px touch
target height minimum. In tablet mode, tapping these is imprecise.
Recommendation: Increase to 44px minimum height (same as input height token).

### 2.2 Orange Calendar (calendar.c) — Score: 2 — Partial

**XFRAME or XCOMP?** Pure XFRAME. Correct use of xframe_node_create throughout.

**XSTORE integration?** None. Events in static s_events[256] array, lost on destroy.

**CRDT sync?** None.

**Implemented:**
- Month-view: 6-row x 7-col grid (42 cells), weekday header row
- Tomohiko Sakamoto weekday algorithm (correct, tested math)
- Leap year handling (correct)
- Event management: orange_calendar_add_event with date clamping
- Event dot indicator in cell (semi-transparent white highlight when event_count > 0)
- Nav bar: Prev / Month-Year label / Today button / Next
- Initial view: March 2026 hardcoded

**Findings:**

PEE-CAL-01: P1 — Calendar view is hardcoded to March 2026 (`s_view_year=2026, s_view_month=3`).
There is no mechanism to initialize to the actual current date. The "Today" button
(today_btn) is visually present but has no event callback — it is non-interactive.
User impact: calendar always opens to March 2026 regardless of actual date.
Recommendation: Initialize s_view_year/month from PAL RTC read (same fix as PEE-TOPBAR-01).
Wire today_btn click to reset view to current month/year.

PEE-CAL-02: P1 — Prev/Next navigation buttons are BOX nodes with no event callbacks. Month
cannot be changed by user interaction. xshell_context_menu pattern (display-only) repeated here.
User impact: single-month calendar with no navigation.

PEE-CAL-03: P1 — No event creation UI. `orange_calendar_add_event` API exists and works, but
there is no dialog, form, or UI to create events from within the app. Events can only be
injected programmatically. No day-cell click handler to open a "New Event" dialog.
User impact: calendar is read-only for end users.

## 3. First-Run Experience Audit

### 3.1 FirstRunWizard.tsx (compositor/genos-shell)

**Score: 2 — Partial**

**Steps present:**
- Language (free-text input, pre-populated "en-US") — should be a dropdown/select
- Timezone (free-text input, pre-populated "UTC") — should be a searchable dropdown
- Keyboard Layout (free-text input, pre-populated "us") — should be a dropdown
- Username (pre-populated "genos") — problematic default, see finding
- Password (type="password", autoComplete="new-password")
- Terms acceptance (checkbox, aria-label on the input itself — note: input is inside a
  label element, so aria-label on input is redundant but not harmful)

**Steps NOT present:**
- Network configuration (WiFi setup) — absent
- AI/XMIND setup (model selection, privacy consent) — absent
- Time zone picker UI (only a text field, no zone map or search)
- Progress indicator across steps (single-page form, not a multi-step wizard)

**Error recovery:**
- createUser failure: setError(result.error) shown in role="alert" region — correct.
- Terms not accepted: setError("Please accept the terms...") — correct.
- No step-level validation (e.g., empty username proceeds until createUser fails with a
  network error rather than an inline "username required" message).

**Accessibility:**
- All labels use `<label style={labelStyle}><input /></label>` wrapping — correct.
- Error region: role="alert", aria-live="assertive", aria-atomic="true" — all three
  required attributes present — correct.
- Continue button: aria-label="Complete setup and continue", aria-busy={loading},
  disabled when loading or terms not accepted — correct.
- Min-height 44px on all inputs and button — touch target compliant.
- Token-sourced styling — no hardcoded hex.

**Motion:** `transition: border-color motion.fast` on inputs. Button transitions background
and opacity. No entrance animation on form mount (no Framer Motion — acceptable in this
shell Electron layer).

**Findings:**

PEE-FRW-01: P1 — Single-page flat form, not a multi-step wizard. All fields visible
simultaneously creates cognitive overload (Hick's Law: reduce choices per screen). OOBE
best practice: step 1 = locale (Language/Timezone/Keyboard), step 2 = account
(Username/Password/Terms), step 3 = network (WiFi), step 4 = AI consent.
Current design also has no progress indicator (step 1 of N). User has no sense of
completion proximity.
User impact: overwhelm on first boot, especially for non-technical users.
Recommendation: Refactor to step-based wizard with `step` state variable (1..N),
render only current step's fields, add a progress bar or step dots at top.

PEE-FRW-02: P1 — No network/WiFi setup step. A first-time user must configure wifi before
they can use any networked feature (Mail, Browser, Drive sync). This is a critical OOBE gap.
User impact: user completes setup and finds no internet access with no guidance on how to
connect.
Recommendation: Add WiFi setup step (step 3) with SSID scan results from XNET/D-Bus
(or window.genosShell.scanWifi() IPC), password input, and connect action.

PEE-FRW-03: P1 — No XMIND/AI setup step. Sprint 6 delivers XMIND. The OOBE should present
AI privacy consent ("GEN.OS uses an on-device AI model. No data leaves your device. Enable?")
and optionally allow model selection. Currently absent.

PEE-FRW-04: P1 — Default username is pre-populated as "genos". This is a security issue:
predictable username aids credential-stuffing attacks on any network-accessible service.
WCAG/security best practice: leave username field empty and require user input.
User impact: low-security default for users who press Continue without changing it.

PEE-FRW-05: P2 — No password strength indicator. User can set password="" or "a" and
proceed to createUser (no client-side minLength enforcement visible in the component —
only blocked at API level with a 422 error which shows as generic "User creation failed").
Prior audit (Wave 2.2 W2-PF-004) noted: minLength(10) not enforced in FirstRunWizard.tsx.
CONFIRMED STILL OPEN. The TSX component has no minLength validation.
Recommendation: Add inline validation: if (state.password.length < 10) setError("Password
must be at least 10 characters.") before the API call.

PEE-FRW-06: P2 — Language, Timezone, and Keyboard Layout are free-text inputs with no
dropdown, autocomplete list, or validation. A user who types "en-US " (trailing space) or
"utc" (lowercase) may encounter silent failures downstream.
Recommendation: Replace with `<select>` elements or searchable comboboxes with curated
option lists.

PEE-FRW-07: P2 — No step-back navigation. User cannot return to a prior step if using a
multi-step wizard (once implemented). Not applicable to current flat form, but must be
designed in when PEE-FRW-01 is addressed.

PEE-FRW-08: P3 — The `<p>` subtitle "Configure your system preferences below." has
`opacity: 0.7` applied inline. Opacity on text reduces effective contrast ratio. At
neutral900 (#f7fbff) on neutral100 (#151b22) background, opacity 0.7 reduces the contrast
significantly — may fail WCAG AA 4.5:1 for normal text at sm scale.
Recommendation: Remove opacity; use XTOKEN_COLOR_TEXT_SECONDARY directly.
