## Settings Strong-Implementation Wiring

### New File: `devices/desktop/shell/xshell/src/settings_shim.c`

Provides 16 strong symbols that either:
- **Signature-adapt** (dispatch.c contract → real backend ABI)
- **Delegate** missing names → correct subsystem function
- **Fallback** to XSTORE when backend not linked (weak extern pattern)

### Makefile: `kernel/xenos/Makefile` XSHELL_SRCS additions

Three files added (before xshell.c):
```
settings_dispatch.c
settings_init.c
settings_shim.c
```

### Key Patterns for This Codebase

- **Weak extern function pointer test**: `extern int fn(args) __attribute__((weak)); if (fn) { fn(...); }` — safe NULL check
- **Weak volatile global test**: `if (&g_xdisp_fb_w) { w = g_xdisp_fb_w; }` — tests symbol presence not value
- **XSTORE_VAL macro**: `XSTORE_VAL(buf, len)` — creates xstore_val_t from buffer
- **XSTORE_KEY_STR macro**: `XSTORE_KEY_STR("namespace:key")` — string key literal
- **g_settings_store extern**: all settings backends use `extern xstore_t *g_settings_store` — provided by settings_init.c

### Known Boot-Stubs Signature Mismatches (Do NOT Try to Reconcile)

boot_stubs.c §14 has `void settings_display_set_brightness(uint32_t pct)` but settings_display.c has `int settings_display_set_brightness(uint32_t pct)`. The weak stub loses to the strong definition — ABI-compatible (caller ignores void return when calling int-return function). Do NOT add conflicting strong void-return wrappers in the shim — this causes duplicate symbol errors.
