# Boot UX + Developer Experience Audit — Sprint 0 + Sprint 1
**Date:** 2026-03-03
**Auditor:** Product Experience Engineer
**Scope:** init/gensd/{splash.c, journal.c, supervisor.c, main.c} + kernel/xenos/core/main.c

---

## Executive Summary

The Sprint 0/1 boot subsystem is architecturally sound with professional code organization, but the **developer-facing output layer suffers from three compounding problems**: inconsistent prefix taxonomy, timestamp opacity in the journal flush, and a supervisor restart log that is informative but not actionable enough for rapid triage. The splash screen is well-designed for the framebuffer layer but lacks brand specificity. The kernel init sequence has verbose-but-useful step markers.

**Overall Boot Experience Grade: B-**
**Developer Ergonomics Grade: C+**
**Cognitive Load Score: 7.2 / 10** (higher = more load; target is 4.0 for production)

## 1. Splash Screen Audit (init/gensd/splash.c)

### What It Renders
- Background: `#0A0A1A` (deep space dark) — full framebuffer fill
- Title: "GEN.OS" at 8x scale (64px tall, 384px wide) — white, centered horizontally, vertical 1/3
- Subtitle: "Sprint 1 -- AetherKernel v0.1" at 2x scale, gold (`#B5830C`)
- Progress bar: full-width minus 32px margins, 16px tall, at bottom - 48px
- Progress label: "nnn%" centered inside bar in white

### UX Assessment

**POSITIVE:**
- Framebuffer-first design. Splash appears before GENSD init banner — correct ordering.
- Percentage label inside the bar is a strong UX choice. Removes ambiguity.
- Spinlock protection on `splash_update_progress()` prevents render corruption from concurrent calls.
- Headless fallback (logs `[SPLASH] Framebuffer not available — headless mode`) is correct behavior.
- `splash_hide()` is idempotent — double-hide is a no-op.

**NEGATIVE (UX Issues):**

1. **Brand String is Generic.** "GEN.OS" as a bitmap string in the system's own 8x16 font is not GENESYS-branded. There is no logomark, wordmark, or stylized treatment — just capital letters. At Sprint 2, this should be replaced by a pre-rasterized bitmap logo stored in the driver or a dedicated logo.h file.

2. **Subtitle exposes internal sprint nomenclature.** "Sprint 1 -- AetherKernel v0.1" is developer-facing metadata surfaced to end users. The double-dash `--` is also an ASCII approximation of an em-dash, which looks unprofessional at 2x scale. For user-facing builds, this line should read something like "Starting system..." or be removed entirely. Sprint metadata belongs in a developer boot mode only.

3. **Hard-cut hide transition.** `splash_hide()` immediately fills with `COLOR_BACKGROUND` and presents. There is a Sprint 2 TODO comment for a fade, which is correct. The abrupt visual cut is jarring on real hardware — flagged for Sprint 2 priority.

4. **No status label during service start sequence.** The progress bar fills from 0% to 33% to 66% to 100% as services start, but there is no text label indicating *which* service is starting. "Starting network..." is more informative than watching a bar fill silently.

5. **COLOR_GOLD BGRA encoding comment error.** The spec comment says `COLOR_GOLD = #B5830C (BGRA: 0xFF0C83B5)`. BGRA byte order for `#B5830C` should be `B=0x0C, G=0x83, R=0xB5, A=0xFF` → `0xFF0C83B5`. This is consistent with how the macro is defined but the parenthetical "BGRA: 0xFF0C83B5" in the file header spec section at line 20 correctly matches `COLOR_GOLD 0xFF0C83B5U` at line 96 — no bug, but confusing documentation mixing RGB and BGRA representations in the same table.

### Accessibility Note (Text Output)
Serial log `[SPLASH] Boot splash displayed (1920x1080)` is clean and parseable. No issues.

## 2. Journal Flush Output Audit (init/gensd/journal.c)

### What the Serial Output Looks Like
```
[JOURNAL] Flush: 47 entries total, ring_pos=8032
[JOURNAL] [INFO ] ts=12s hash=0x3a4b8f12 GENSD initialized — Sprint 1
[JOURNAL] [WARN ] ts=14s hash=0x7c2d1099 service restarting (backoff)
[JOURNAL] [ERROR] ts=15s hash=0x7c2d1099 start failed
[JOURNAL] Flush complete (47 entries displayed)
```

### UX Assessment

**POSITIVE:**
- Level names are padded to 5 chars (`"INFO "`, `"WARN "`), producing a clean column. Excellent.
- Timestamp in seconds (`ts=Ns`) is readable. Better than raw nanoseconds.
- Level range 0–4 maps to DEBUG/INFO/WARN/ERROR/FATAL — industry-standard semantics.
- 64-entry cap on flush output prevents terminal flooding.

**NEGATIVE (Developer Ergonomics Issues):**

1. **`hash=0x3a4b8f12` is opaque.** The service name is stored only as a FNV-1a hash in the journal entry. When flushing to serial, a developer sees `hash=0x7c2d1099` and cannot tell which service this is without a lookup table or running the hash function themselves. This is the single largest debuggability regression in the journal system.

   **Root Cause:** `journal_entry_t` stores only `service_hash` (4 bytes), not the full name. The name is available at write-time but not at read-time.

   **Impact:** When a service fails and the developer reads the serial dump, they must cross-reference hash values manually. During a production incident this adds 2–5 minutes of cognitive overhead per failure event.

2. **Timestamp resolution truncated to seconds.** `ts_s = timestamp_ns / 1e9` discards millisecond precision. For diagnosing rapid restart loops (service restarts 3× in 800ms), second-level granularity is insufficient. A format like `ts=12.847s` or `ts=12847ms` would be far more useful.

3. **No forward-scan reliability indicator.** `journal_flush_serial()` scans forward from `write_pos` and skips invalid entries by advancing 1 byte. There is no count of skipped bytes or corrupted entries. A developer cannot tell whether the output is complete or partially corrupted.

4. **Log level usage is inconsistent between modules.**
   - `supervisor.c` uses literal `LOG_WARN` for "service restarting" — correct.
   - `supervisor.c` uses `LOG_INFO` for "service restarted (exec stub)" — arguably should be `LOG_WARN` since stub restarts are not real recovery.
   - `main.c` uses `LOG_INFO` for "registering essential services" — correct.
   - `main.c` uses `LOG_FATAL` for "unexpected exit from run_loop" — correct.
   - `journal_write()` calls are consistent overall, but "service restarted (exec stub)" at INFO feels misleading — it implies recovery when it is actually a no-op.

### Cognitive Load Analysis
Each journal line requires the developer to:
1. Parse the severity level (easy — labeled)
2. Convert seconds to a mental timeline (medium — no sub-second precision)
3. Resolve the service hash to a service name (hard — requires external lookup)

**Cognitive Load Score for Journal Flush: 7.8 / 10** — high.

## 3. Supervisor Restart Log Audit (init/gensd/supervisor.c)

### What the Serial Output Looks Like
```
[SV]   xapi-health: liveness probe FAIL (health=1)
[SV]   xapi-health: restart #1 in 1000 ms
[SV]   xapi-health: RESTART_NEVER — deactivating
[SV]   xapi-health: restarted (exec stub)
```

### UX Assessment

**POSITIVE:**
- Service name appears in every log line — no hash opacity. This is the correct pattern that journal.c should emulate.
- Exponential backoff is logged with the current delay value in milliseconds — actionable.
- Restart count (`restart #N`) gives a direct fatigue signal. After restart #5 at 30,000ms backoff, a developer knows the service is in a death spiral.
- `RESTART_NEVER — deactivating` is exceptionally clear policy transparency.

**NEGATIVE:**

1. **`health=1` is an enum value, not a string.** The line `liveness probe FAIL (health=1)` requires the developer to remember that 1 = `HEALTH_FAIL` and 2 = `HEALTH_TIMEOUT`. Should print `HEALTH_FAIL` or `HEALTH_TIMEOUT` as a string, not the raw integer.

2. **`(exec stub)` in production log output.** When `supervisor_restart()` marks `running=true` and logs "restarted (exec stub)", this stub annotation leaks implementation detail into what should be a behavioral log. In Sprint 2 when real exec is implemented, this string must be updated. Using a compile-time conditional (`#ifdef SPRINT1_STUB`) would make the sprint-specific nature explicit.

3. **Blocking sleep in supervisor thread.** The `TODO(Sprint 2)` comment at line 208 is well-written and references a tracking document. This is a developer-experience positive — the issue is documented inline where the developer would look. However, the UX consequence is that during a 30-second backoff, the entire supervisor thread is blocked. This is a silent behavioral hole with no log output during the wait window. A developer would see:
   ```
   [SV]   xapi-health: restart #5 in 30000 ms
   [SV]   xapi-health: restarted (exec stub)
   ```
   With a 30-second silence between them. No heartbeat. No indication the system is alive.

4. **No explicit "max restarts reached" state.** The backoff caps at 30,000ms but restarts continue indefinitely. A service that fails 1,000 times will log 1,000 restart lines with no "giving up" escalation. The supervisor should emit a LOG_ERROR or LOG_FATAL after N consecutive failures (e.g., 10).

### Cognitive Load Analysis
Supervisor logs score significantly better than journal flush:
1. Service name is always visible (easy)
2. Restart number + delay are both present (easy)
3. Health result requires enum lookup (medium)

**Cognitive Load Score for Supervisor: 5.5 / 10** — acceptable, fixable with string health names.

## 4. GENSD Main Boot Sequence Audit (init/gensd/main.c)

### What the Serial Output Looks Like
```
[GENSD] ============================================
[GENSD]  GEN.OS Service Descriptor Daemon (GENSD)
[GENSD]  Sprint 1 — AetherKernel v0.1
[GENSD] ============================================
[GENSD] Starting: xnet-loopback (rights=0x00000003 exec=/opt/genos/services/xnet-loopback)
[GENSD] Starting: xstore-init (rights=0x0000000c exec=/opt/genos/services/xstore-init)
[GENSD] Starting: xapi-health (rights=0x00000003 exec=/opt/genos/services/xapi-health)
[GENSD] === Service Status ===
[GENSD]  xnet-loopback             RUNNING  restarts=0
[GENSD]  xstore-init               STOPPED  restarts=0
[GENSD]  xapi-health               RUNNING  restarts=0
[GENSD] === End Status ===
[GENSD] Boot sequence complete. Entering supervision.
```

### UX Assessment

**POSITIVE:**
- Banner box around GENSD header is excellent. Visually anchors the GENSD section in a multi-section boot log.
- `gensd_print_status()` with padded service name column (`%-24s`) produces a clean table. Highly readable.
- `RUNNING` / `STOPPED` state vocabulary is clear and binary.
- `restarts=N` column is a direct signal of service health.
- `rights=0x00000003` in start log is transparent about capability grants — good for security auditing.

**NEGATIVE:**

1. **`rights=0x00000003` is opaque without a bitmask legend.** A developer who doesn't have `xkabi_rights.h` open will not know that `0x00000003` = `NET_SEND | NET_RECV`. The log should name the rights, or at minimum the hex should include a short annotation: `rights=NET_SEND|NET_RECV(0x3)`.

2. **`WARN: failed to register %s` has no reason code.** Line 375-377: `pal_console_printf("[GENSD] WARN: failed to register %s\n", ...)`. `gensd_register_service()` returns -1 for both "table full" and "duplicate name" — two different failure modes. The console log doesn't indicate which. A developer seeing this during a boot regression has no path to diagnosis without adding prints.

3. **Sprint metadata in user-facing banner.** Same issue as splash subtitle: "Sprint 1 — AetherKernel v0.1" in the GENSD banner is developer/internal content. In shipped builds this should be replaced with a version string from a build-time constant.

4. **Journal flush at boot-complete is useful but position is surprising.** `journal_flush_serial()` is called immediately after "Boot sequence complete." — this dumps the entire in-memory ring to serial. The output interleaves with the status table, making it hard to visually separate the structured status from the historical log dump. Adding a clear separator before the flush output would help:
   ```
   [GENSD] === Journal Dump (boot) ===
   ```

5. **`STOPPED` for xstore-init is potentially misleading.** `xstore-init` is a ONESHOT with `RESTART_NEVER`. It exits cleanly on success. Displaying it as `STOPPED` alongside a `RUNNING` daemon creates a false equivalence — a developer might think it failed. The status should distinguish `COMPLETE` (oneshot finished OK) from `STOPPED` (daemon exited unexpectedly) or `FAILED`.

### Cognitive Load Analysis
The main.c boot output is the clearest of the four files:
1. Banner is visually anchored (instant orientation)
2. Service table is columnar and readable (low load)
3. Rights hex requires lookup (medium load)
4. STOPPED vs COMPLETE ambiguity (medium load)

**Cognitive Load Score for main.c: 5.2 / 10** — good, best of the four files.

## 5. Kernel Boot Output Audit (kernel/xenos/core/main.c)

### What the Serial Output Looks Like (Step Sequence)
```
[XENOS] Sprint 1 boot — AetherBoot handoff
[XENOS] GEN.OS XENOS Microkernel v0.1.0
[XENOS] Boot info: AETHER01 magic OK
[XENOS] [2/14] IDT init
[XENOS] IDT initialized (256 interrupt gates)
[XENOS] [3/14] GDT reload + TSS
[XENOS] GDT reloaded, TSS installed (ltr 0x48)
[XENOS] [4/14] PMM init
[XENOS] [4.5] EFER.NXE enabled (NX active from first PML4 load)
[XENOS] [5/14] VMM init
[XENOS] [6/14] Heap init
[XENOS] [7/14] PAL init
[XENOS] PAL backend: pal-aether-v1
[XENOS] CPUs=1  TSC=2400000000 Hz  RAM @ EFI map
[XENOS] FB: 1920 x 1080 @ phys=0x00000000fd000000 fmt=1
[XENOS] [8/14] vDRAM init
[XENOS] vdram_init_kernel: stub (Sprint 2)
[XENOS] [9/14] SYSCALL init
[XENOS] SYSCALL MSRs configured
[XENOS] [10/14] APIC init
[XENOS] LAPIC enabled, ID=0x0, SVR=0x1ff
[XENOS] [11/14] Scheduler init
[XENOS] cr_sched_init: stub (Sprint 2)
[XENOS] [12/14] Cap table init
[XENOS] cap_table_init: stub (Sprint 2)
[XENOS] [13/14] GENSD start
[XENOS] gensd_start: stub (Sprint 2)
[XENOS] [14/14] Idle loop — boot sequence complete
```

### UX Assessment

**POSITIVE:**
- `[N/14]` step counter is outstanding. Gives the developer an exact position in the init sequence at all times. Knowing there are 14 total steps means a hang at `[7/14]` immediately narrows the failure to PAL init.
- Step labels are concise and descriptive: "IDT init", "GDT reload + TSS", "SYSCALL init".
- `[XENOS] PAL backend: pal-aether-v1` — naming the active backend is helpful for multi-backend testing.
- `CPUs=N TSC=N Hz` — hardware fingerprint on boot is excellent for regression tracking.
- `FB: WxH @ phys=0xADDR fmt=N` — framebuffer params on serial let a developer verify GOP handoff without attaching a display debugger.
- Exception handler output in `xenos_isr_dispatch()` is comprehensive: vector number, name, RIP, CS, RFLAGS, RSP, error code, all GPRs, and CR2 for page faults. This is a well-designed panic dump.
- `[XENOS] Boot info: AETHER01 magic OK` — explicit magic validation confirmation is good operational hygiene.
- Step 4.5 (`EFER.NXE`) has an in-code rationale comment explaining WHY it's between 4 and 5, not at step 9. The serial log reflects this with `[4.5]` notation — excellent transparency.

**NEGATIVE:**

1. **`[N/14]` markers are PRE-step, completions are POST-step — inconsistent pattern.**
   The output pattern is: announce step → call function → function prints its own completion.
   Example:
   ```
   [XENOS] [2/14] IDT init          ← pre-announcement (from kmain)
   [XENOS] IDT initialized (256 interrupt gates)  ← completion (from idt_init())
   ```
   But step 8 is:
   ```
   [XENOS] [8/14] vDRAM init        ← pre-announcement
   [XENOS] vdram_init_kernel: stub (Sprint 2)  ← from the weak stub
   ```
   And steps 11–13 are stubs that print their own messages. This is inconsistent: some steps confirm completion with context, some just say "stub". A developer cannot tell whether a step completed successfully or was a no-op without inspecting each function.

2. **"stub (Sprint 2)" appears 3 times in the boot log.** At Step 8, 11, 12, and 13. These stub messages are correct to have during development, but they increase cognitive noise for someone debugging a real problem. They pattern-match to "error" visually because they represent missing functionality. Flagging: at scale, stub count should shrink each sprint. Consider a single consolidated stub summary rather than inline messages.

3. **`LAPIC enabled, ID=0x0, SVR=0x1ff` requires hex literacy.** `SVR=0x1ff` means `APIC_SVR_ENABLE (bit 8) | APIC_SPURIOUS_VEC (0xFF)`. Only a developer familiar with the LAPIC register encoding can parse this. Consider adding a decoded note: `SVR=0x1ff (enabled, spurious_vec=0xFF)`.

4. **Magic validation failure has no magic value in the log.** Line 647-650:
   ```c
   serial_early_puts("[XENOS] FATAL: boot info magic invalid\n");
   ```
   There is no output of `bi->magic` or `XENOS_BOOT_MAGIC` values. A developer investigating a mismatch cannot determine whether `bi` was null (null pointer) or the magic was a different value (ABI mismatch). Output like `FATAL: magic=0x00000000, expected=0x4145544801` is far more actionable.

5. **No duration markers.** The entire kmain sequence has no timing information. A developer cannot tell if `PMM init` took 2ms or 200ms. Adding TSC-derived timestamps (even just `dt=NNms` after each step completion) would allow performance regression detection.

### Cognitive Load Analysis
The kernel boot output is the best-designed of all five files:
1. Step counter gives immediate orientation (very low load)
2. Hardware fingerprint confirms correct hardware init (low load)
3. LAPIC SVR hex requires register knowledge (medium load)
4. Stub markers create visual noise (medium load)
5. No timing data prevents performance understanding (medium load)

**Cognitive Load Score for kmain: 5.8 / 10** — good. Elevated by stub noise and hex LAPIC output.

## 6. Accessibility Assessment (Text Output)

This audit covers serial output and console logging — not visual UI. Accessibility considerations for text output:

1. **Color usage:** None of the C serial output uses ANSI color codes. This is correct for a bare-metal freestanding environment where terminal color support cannot be assumed. No accessibility concern.

2. **Line length:** All log lines are under 120 characters. Readable in standard 80- and 132-column terminals.

3. **Symbol usage:** `===`, `---`, and Unicode box-drawing characters (`═`) are used only in source code comments/headers, not in runtime output. Runtime output is clean ASCII. No concern.

4. **Prefix taxonomy consistency issue:** Different prefixes are used for different severity levels:
   - `[GENSD] WARN:` (space-separated keyword)
   - `[GENSD] ERROR:` (same)
   - `[SV]   service: RESTART_NEVER` (service-name-first, CAPS policy name)
   - `[XENOS] FATAL:` (same)
   - `[JOURNAL] [WARN ]` (bracketed level inside brackets)

   A developer scanning a mixed-subsystem boot log must learn four different pattern formats to identify severity. This increases cognitive load for accessibility-adjacent reasons — particularly for developers with attention or reading-processing differences.

5. **`[SV]   %s:` three-space padding.** The three spaces after `[SV]` produce visual alignment with `[GENSD]` and `[XENOS]` prefixes of similar length. This is intentional and good.

## 8. Top 3 UX Improvements for Sprint 2

### Priority 1 (P0): Service Name Resolution in journal_flush_serial()

**Problem:** `hash=0x7c2d1099` is opaque. Developers cannot identify the failing service without a hash lookup.

**Fix:** Store a capped service name (16 bytes) alongside the hash in `journal_entry_t`, OR maintain a static hash→name lookup table populated by `journal_write()` calls. The second approach avoids enlarging every journal entry.

**Impact:** Reduces cognitive load during incident triage from ~8 minutes to ~30 seconds per failure event. Eliminates the single largest debuggability gap in the journal subsystem.

**Implementation hint (read-only observation):** A 256-entry hash-to-name map (256 × 16 bytes = 4KB) in BSS, populated on first write of each unique service hash, would allow `journal_flush_serial()` to print `service=xapi-health` instead of `hash=0x7c2d1099`.

---

### Priority 2 (P1): Health Result String in supervisor_tick()

**Problem:** `liveness probe FAIL (health=1)` requires enum→string mental translation.

**Fix:** Introduce a `static const char *health_names[] = {"OK", "FAIL", "TIMEOUT"}` array and print `health=FAIL` instead of `health=1`.

**Impact:** Eliminates a lookup step during triage. Low implementation cost (5 lines). High daily-use payoff for every developer watching a boot.

**Also:** The `health=` prefix in supervisor_tick() and the `(exec stub)` annotation in supervisor_restart() should both be addressed together in the same Sprint 2 patch.

---

### Priority 3 (P1): Boot Step Timing in kmain()

**Problem:** No timing data. A developer cannot identify slow boot steps or detect performance regressions across sprints.

**Fix:** Capture `bi->tsc_hz` (available at boot time — already printed at Step 7). Between each step, compute `(rdtsc() - step_start_tsc) * 1000 / tsc_hz` to get milliseconds. Print as `dt=NNms`.

Example output:
```
[XENOS] [4/14] PMM init
[XENOS] PMM: 3824 pages free (15MB), dt=12ms
[XENOS] [5/14] VMM init
[XENOS] VMM: HHDM mapped, dt=4ms
```

**Impact:** Sprint 2 introduces the scheduler and process isolation — both have non-trivial init cost. Without timing data, a developer cannot distinguish a regression (5→50ms) from expected behavior. This is essential instrumentation for a production OS.

**Note:** `serial_early_puts()` is available before PAL, but TSC timing requires calibration (available after Step 7 `pal_init()`). Only Steps 8–14 can be timed precisely. Steps 2–7 can use TSC cycles with a fixed 2.4GHz estimate from the boot info `tsc_hz` field.

## 9. Additional Findings (Advisory)

### A-1: Splash Subtitle Should be Conditional on Build Mode

"Sprint 1 -- AetherKernel v0.1" should be emitted only in debug/dev builds. In release builds, replace with "Starting..." or a GEN.OS version string from a `GENOS_VERSION` build constant. Apply the same to the GENSD banner in `gensd_init()`.

### A-2: ONESHOT vs DAEMON Status Distinction in gensd_print_status()

`STOPPED` for xstore-init (a ONESHOT) looks identical to a failed daemon. Add a `COMPLETE` state for ONESHOTs that exited cleanly (exit code 0). This disambiguates normal completion from failure.

### A-3: Supervisor Heartbeat During Backoff Sleep

The 30-second blocking sleep in `supervisor_restart()` produces complete radio silence. Add a periodic tick (every 5s) during the sleep using a loop with shorter `pal_thread_sleep_ns()` intervals, printing `[SV] xapi-health: waiting N/30s before restart`. This makes the system appear alive during long backoffs.

This is already tracked in the TODO comment at supervisor.c:208 referencing `event-horizon-analysis.md §Contagion-Map-2 P1-SUPERVISOR-SLEEP`.

### A-4: Boot Magic Diagnostic Value in FATAL Path

kmain() line 647: print both the received magic and the expected magic on magic mismatch. Format: `[XENOS] FATAL: boot magic mismatch (got=0x%llx, expected=0x%llx)`.

### A-5: Journal Timestamp Sub-second Precision

Change `ts_s = ts_ns / 1e9` to print `ts=NNN.NNNs` (3 decimal places of seconds) or `ts=NNNNNms`. The raw nanosecond value is available — truncating to integer seconds loses meaningful resolution for rapid-restart-loop diagnosis.
