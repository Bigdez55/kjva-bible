## Section 1: Verified Pipeline Stages (What Works End-to-End)

### Stage A — Boot Chain calls xshell_start() [VERIFIED WORKING]
boot_chain.c §8 xchain_shell() calls xshell_start() at line 532.
The weak stub is overridden by xshell.c which defines pal_status_t xshell_start(void)
at line 257 — this calls xshell_init() → xshell_greeter_show() → xshell_run(). CORRECT.

### Stage B — xshell_start() calls greeter_show [VERIFIED WORKING]
xshell.c line 265-267: if (g_shell.greeter) xshell_greeter_show(g_shell.greeter).
greeter.c xshell_greeter_show() sets g->visible=1, calls xframe_node_set_visible(overlay, 1),
and calls xframe_node_mark_dirty(overlay). CORRECT.

### Stage C — xframe_init() and app creation [VERIFIED WORKING]
xshell.c xshell_build_scene() calls xframe_init() then xframe_app_create(1920, 1200).
main.c xframe_app_create() calls xcomp_surface_create(width, height), stores the returned
pointer in app->surface and sets app->surface_id = surf->pool_idx + 1. CORRECT.

### Stage D — XCOMP surface_create [VERIFIED WORKING]
compositor.c xcomp_surface_create() claims a pool slot, zeroes and initializes the struct,
inserts into z-sorted scene graph, returns the pointer. Surface starts invisible/uncommitted
with no buffer. CORRECT.

### Stage E — xframe_run() paint loop [VERIFIED — PARTIALLY]
main.c xframe_run() loop:
  - calls xframe_layout() when needs_layout=1
  - calls xframe_paint(app, g_sw_fb, paint_w, paint_w, paint_h) — real buf pointer passed
  - calls xcomp_surface_commit((xcomp_surface_t *)app->surface)

paint.c xframe_paint():
  - if buf != NULL, clears buffer to 0x00000000 (transparent black)
  - calls paint_subtree(app->root, buf, stride, w, h)
  - paint_fill_rect() has a REAL software rasterizer — writes argb pixels directly to buf[]
  - paint_rounded_rect() falls back to paint_fill_rect() (sharp corners, real pixels)
  - paint_border() calls paint_fill_rect() four times — real pixels
  - paint_text() IS A STUB — logs to console, writes ZERO pixels

So: BOX nodes with non-transparent bg_color DO paint real pixels into g_sw_fb. TEXT nodes
write nothing visible.

### Stage F — xcomp_surface_commit() [VERIFIED WORKING — MARKS STATE ONLY]
compositor.c line 949-976: sets committed=1, visible=1, and if no explicit damage was set,
damages the entire surface (w=s->width, h=s->height). CORRECT signaling.

### Stage G — xcomp_frame() composites surface pixels into back buffer [VERIFIED WORKING]
compositor.c xcomp_frame():
  - Maps g_back_buf via xdisp_buf_map() — real mmap from DRM dumb buffer
  - Fills back buffer with XCOMP_BG_COLOUR (0xFF000000, opaque black)
  - Walks scene graph: for each surface, checks committed/visible/buf_handle/damage
  - Maps surface's buf_handle via xdisp_buf_map()
  - Calls blend_surface_region() — real Porter-Duff SRC_OVER pixel blending
  - Unmaps surface buffer, clears damage+committed flags
  - Calls xdisp_buf_submit(g_back_buf, crtc_id, 1) — real DRM SETCRTC + vsync wait
  - Swaps back/front buffer handles

### Stage H — xdisp DRM/KMS pipeline [VERIFIED WORKING]
drm.c xdisp_drm_init(): opens /dev/dri/card0, enumerates connectors (eDP priority),
selects mode (1920x1200 preferred), allocates init dumb framebuffer, calls SETCRTC.
gpu_buf.c xdisp_buf_alloc(): CREATE_DUMB + ADDFB → returns GEM handle.
gpu_buf.c xdisp_buf_map(): MAP_DUMB + mmap → returns CPU pointer.
gpu_buf.c xdisp_buf_submit(): calls xdisp_drm_set_framebuffer(fb_id) + wait_vsync.

## Section 2: Remaining Gaps (Critical Blockers)

### GAP-01 [P0 — PIPELINE BREAK]: g_sw_fb is never attached to the XCOMP surface.

This is the central, fatal gap that breaks the entire pixel path in Sprint 15.

The sequence in xframe_run() is:
  1. xframe_paint(app, g_sw_fb, ...)  — pixels written into g_sw_fb (correct)
  2. xcomp_surface_commit(surf)        — marks surf committed + damaged (correct)

BUT xcomp_frame() (called from a separate XCOMP thread or loop) does this per-surface:
  snap_buf = cur->buf_handle  — this is the GEM handle from xdisp_buf_alloc()

The XCOMP surface's buf_handle was NEVER SET. xcomp_surface_create() initializes
buf_handle = XCOMP_SLOT_FREE (0). Nothing in xframe_app_create() or anywhere else
calls xcomp_surface_attach_buf(surf, gem_handle).

xcomp_frame() at line 1324 explicitly skips surfaces where:
  snap_buf == XCOMP_SLOT_FREE || snap_buf == XCOMP_SLOT_SENTINEL

So xcomp_frame() reads: snap_buf=0 → skips this surface entirely → no pixels composited.

Furthermore: even if buf_handle were set, xcomp_frame() calls xdisp_buf_map(snap_buf)
to get the source pixels to blend. That would return a pointer to the GEM dumb buffer
mmap — NOT to g_sw_fb. The g_sw_fb array and the GEM buffer are two completely separate
memory regions. XCOMP would blend pixels from an uninitialized GEM buffer, not from
the XFRAME-painted software framebuffer.

XFRAME paints into g_sw_fb (a static uint32_t array in BSS).
XCOMP composites from xdisp GEM dumb buffers (mmap'd kernel memory).
These two memory regions are NEVER CONNECTED. g_sw_fb pixels never reach XCOMP.

### GAP-02 [P0 — ARCHITECTURE]: XCOMP and XFRAME run in separate loops with no shared clock.

xframe_run() (called from xshell_run) is the XFRAME loop that paints into g_sw_fb
and calls xcomp_surface_commit(). This runs at 60fps via pal_thread_sleep_ns.

xcomp_frame() / xcomp_run() is the XCOMP compositor loop. BUT: xcomp_run() is NEVER
called anywhere in the boot chain or in xshell.c. Neither xchain_compositor() nor
xframe_run() calls xcomp_run(). The XCOMP compositor loop never starts.

xcomp_surface_commit() signals the surface as committed. But if xcomp_frame() never
runs, nothing ever composites surfaces → DRM scanout buffer is never updated past the
initial blank black frame submitted in xcomp_init().

### GAP-03 [P0 — MISSING FUNCTION]: xdisp_init() has no real implementation.

boot_chain.c declares xdisp_init() as __attribute__((weak)) and calls it at line 455
with (fb_phys, width, height, pitch, fmt). The comment says
"xdisp_init() — display/xdisp/drm.c" but drm.c only exports xdisp_drm_init() (no args).
There is NO real xdisp_init() implementation anywhere in the codebase.

Result: xchain_display() calls the weak stub which returns PAL_OK and does nothing.
The GOP framebuffer physical address from AetherBoot is passed in but immediately
discarded. This is acceptable IF xcomp_init() → xdisp_drm_init() runs independently
(which it does — it opens /dev/dri/card0 directly). But the fb_phys from the handoff
is never stored or used. In QEMU with virtio-gpu this is fine because DRM device
enumeration works. On bare hardware with only a GOP framebuffer (no DRM), this would
be fatal.

### GAP-04 [P1]: paint_text() writes zero pixels — all text labels invisible.

paint.c paint_text() (lines 360-384) discards buf, stride, buf_w, buf_h via (void) casts
and only calls pal_console_printf. All TEXT nodes in the greeter (username label,
password label, "Log In" button text, "GEN.OS" logo text, session label) are invisible.
Only the colored BOX shapes render.

### GAP-05 [P1 — GREETER NODE TREE NOT IN XCOMP FRAME PATH]:

The greeter overlay node is attached to shell->scene_root (greeter.c line 210-212).
scene_root is the root of the XFRAME node tree. This is correct for XFRAME paint.
But there is only ONE xcomp_surface_t for the entire shell app (created in xshell_build_scene).
All shell UI paints into one surface. This is architecturally fine for a single-surface model.
The issue is GAP-01: that surface's buf_handle is never set to a GEM handle.

### GAP-06 [P2 — THREAD ARCHITECTURE]: xshell_run() enters xframe_run() which blocks.
xcomp_run() must run on a separate thread. Neither is launched concurrently.
xchain_shell() calls xshell_start() which blocks in xframe_run(). xcomp_run() is never
called. These must run on separate PAL threads.

## Section 3: The g_sw_fb → GEM Buffer Disconnect — Root Cause Analysis

The fundamental Sprint 15 design assumption appears to have been:
  "xframe paints into g_sw_fb, then calls xcomp_surface_commit() which flushes g_sw_fb
   to the physical display."

But xcomp_surface_commit() does NOT copy pixels. It only sets committed=1 and damage rect.
The actual pixel transfer happens in xcomp_frame() via xdisp_buf_map(buf_handle) which
maps a GEM dumb buffer — a completely separate memory region from g_sw_fb.

The missing link is: XFRAME must allocate a GEM buffer for each surface, paint into it
(instead of or in addition to g_sw_fb), and attach that GEM handle to the XCOMP surface.

Two architecturally valid solutions exist:

OPTION A (Shared GEM buffer — correct architecture):
  xframe_app_create() allocates a GEM buffer via xdisp_buf_alloc(width, height, ARGB8888),
  calls xcomp_surface_attach_buf(surf, gem_handle), then in xframe_run() paints directly
  into xdisp_buf_map(gem_handle) instead of g_sw_fb. After paint, calls xdisp_buf_unmap()
  then xcomp_surface_commit().

OPTION B (Copy path — simpler but wasteful):
  Keep painting into g_sw_fb, then in xframe_run() after paint: copy g_sw_fb into the
  GEM buffer (xdisp_buf_map → memcpy → xdisp_buf_unmap), then commit. 9MB copy per frame
  at 60fps = ~540MB/s, within UHD620 bandwidth but unnecessary.

OPTION A is the correct path.

## Section 4: Minimum Changes for Sprint 16 to Produce Visible Output in QEMU

Listed in dependency order:

FIX-01 [CRITICAL — 1 function, ~10 lines]:
  xframe_app_create() in main.c must allocate a GEM buffer and attach it to the surface.
  After line 273 (xcomp_surface_set_visible), add:
    uint32_t gem = xdisp_buf_alloc(width, height, XDISP_FORMAT_ARGB8888);
    if (gem != 0) {
        xcomp_surface_attach_buf(surf, gem);
        app->gem_handle = gem;   // new field in xframe_app_t
    }
  And paint directly into xdisp_buf_map(app->gem_handle) in xframe_run() instead of g_sw_fb.

FIX-02 [CRITICAL — xframe.h, ~1 line]:
  Add uint32_t gem_handle field to xframe_app_t struct.

FIX-03 [CRITICAL — xframe_run(), ~10 lines]:
  In xframe_run() paint block, replace:
    xframe_paint(app, g_sw_fb, ...)
  with:
    uint32_t stride_bytes;
    uint32_t *paint_buf = (uint32_t*)xdisp_buf_map(app->gem_handle, &stride_bytes);
    if (paint_buf) {
        xframe_paint(app, paint_buf, stride_bytes/4, paint_w, paint_h);
        xdisp_buf_unmap(app->gem_handle);
    }

FIX-04 [CRITICAL — threading, ~20 lines in xshell.c or boot_chain.c]:
  xcomp_run() must run on a dedicated PAL thread before xframe_run() blocks.
  In xshell_start() before calling xshell_run():
    pal_thread_create(&comp_thread, xcomp_run_wrapper, NULL, PAL_THREAD_REALTIME);
  Or in xchain_compositor() after xcomp_init():
    pal_thread_create(&comp_thread, (void*)xcomp_run, NULL, PAL_THREAD_REALTIME);

FIX-05 [REQUIRED — xframe_app_destroy(), ~3 lines]:
  Free the GEM buffer on destroy: xdisp_buf_free(app->gem_handle).

FIX-06 [OPTIONAL but recommended for Sprint 16 visual proof]:
  Add a real xdisp_init() wrapper in drm.c or a new xdisp/init.c that stores fb_phys
  and calls xdisp_drm_init() internally. Minimum: just rename or alias.

FIX-07 [P1 — paint_text(), ~80 lines]:
  Implement 8x16 bitmap font in paint.c (see Sprint 15 audit G-05). Without this,
  all button labels, username/password labels, and logo text are invisible. The greeter
  will show as colored rectangles only — functional as a boot proof but not readable.

## Section 5: Regression Check — Sprint 15 Changes

Sprint 15 introduced these specific changes. Checking each for regressions:

REGRESSION-CHECK-01: xframe_app_create() now calls xcomp_surface_create().
  STATUS: No regression. The call is correctly gated on initialization and the returned
  pointer stored in app->surface. xcomp_surface_destroy() called correctly in app_destroy().

REGRESSION-CHECK-02: g_sw_fb static buffer at 1920x1200x4 = 9,216,000 bytes in BSS.
  STATUS: No regression in terms of correctness. BSS is zero-initialized by the loader.
  Space cost: 8.8 MB of BSS. On XENOS with adequate RAM (HP EliteBook: 16GB) this is fine.
  NOTE: Once FIX-01/FIX-03 are applied and paint goes directly into the GEM buffer,
  g_sw_fb becomes dead code and should be removed to recover the BSS space.

REGRESSION-CHECK-03: xshell_start() now defined in xshell.c, overriding weak stub.
  STATUS: No regression. The weak stub mechanism works correctly. xshell_start() calls
  xshell_init() → greeter_show() → xshell_run() in the correct order.

REGRESSION-CHECK-04: xcomp_surface_set_visible(surf, 1) called immediately on create.
  STATUS: Minor concern. The surface is made visible before any buffer is attached or
  any content is committed. xcomp_frame() will skip it (snap_buf==0 check at line 1324),
  so no visual corruption. But semantically, visible=1 without a buffer is misleading.
  Not a regression — just an ordering issue that becomes moot once FIX-01 attaches a buffer.

REGRESSION-CHECK-05: xframe_run() passes g_sw_fb (non-NULL) to xframe_paint().
  STATUS: Correct — paint.c no longer takes the NULL path that was the Sprint 14 bug.
  paint_fill_rect() correctly writes pixels when buf != NULL. This is a genuine fix.
