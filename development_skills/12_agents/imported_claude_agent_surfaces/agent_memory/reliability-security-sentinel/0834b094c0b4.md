## DETAILED FINDINGS

### F-001 (P2) — provenance_client.py: _entry_count not persisted across restarts
- File: tools/provenance_client.py line 44
- `self._entry_count = 0` on `__init__`. If the process restarts after writing 9,999 entries
  to disk, `_entry_count` resets to 0. The size-based check (MAX_FILE_BYTES) will still
  trigger rotation correctly, but `_entry_count >= MAX_ENTRIES` will never fire for a
  restarted process until it writes 10,000 more entries in the new session.
- Impact: at most 2x MAX_ENTRIES (20,000 entries) before size check catches it. Not data loss,
  but the entry-count guard is effectively bypassed on restart.
- Fix: initialize `_entry_count` from `len(self.fallback_file.read_text().splitlines())`
  if the file exists, OR rely solely on the size check and remove `_entry_count`.

### F-002 (P2) — flush_fallback_to_service() reads entire file into memory
- File: tools/provenance_client.py line 99
- `lines = path.read_text().strip().split("\n")` — at 10 MB worst case (20 MB with rotated
  file), this materializes the full file in memory before streaming to the service.
- If the service is slow (network reconnect), the process holds 10-20 MB of audit log in RAM.
- For a daemon on the EliteBook x360 (16 GB RAM), this is LOW impact, but should be noted.
- Fix: iterate line-by-line with a file handle using `for line in f` to keep memory O(1).

### F-003 (P2) — provenance_client.py: no fsync / non-atomic rotation
- File: tools/provenance_client.py lines 78-91
- `open(self.fallback_file, "a")` and `.write(line)` — no `f.flush()` or `os.fsync()`.
  On OS crash between write and sync, the JSONL entry is lost.
- `self.fallback_file.rename(self.fallback_rotated)` — on ext4/POSIX, rename is atomic.
  However `self.fallback_rotated.unlink()` before rename is a two-step operation with a
  crash window between unlink and rename where the rotated file is gone but the active file
  is also gone.
- Fix: rename active to rotated first (atomically replaces old rotated), then reset.
  i.e., `self.fallback_file.rename(self.fallback_rotated)` is already atomic on POSIX.
  The `unlink` before rename creates the window. Swap order: rename first, unlink is
  implicit (rename replaces the destination). The code does `unlink` then `rename` —
  this is WRONG. If the process crashes between unlink and rename, the rotated data is lost.
  FIX: Remove the `self.fallback_rotated.unlink()` — `Path.rename()` on POSIX atomically
  replaces the destination file if it exists.

### F-004 (P2) — thermal_guardian.py: device_id accepted via CLI but not validated
- File: tools/thermal_guardian/thermal_guardian.py lines 141-143
- `--devices` accepts arbitrary strings from `argparse`. Strings become dict keys and are
  passed to `_read_temp()`. The `nvidia-smi` invocation does NOT use `device_id` in the
  subprocess arguments — `device_id` is only used as a dict key — so NO injection risk
  in the subprocess call itself.
- However, `device_id` is written verbatim to JSON state at `/run/genos/thermal-state.json`
  as a top-level key. A device_id of `"../../etc/passwd"` or a JSON-special character
  would be safely serialized by `json.dumps()`, so no injection into the JSON file.
- VERDICT: subprocess call is safe (list form). JSON serialization is safe. No injection.
  The P2 classification is for the state file using unconstrained device_id as a key —
  downstream consumers of the JSON should treat device_id keys as untrusted.
- Recommendation: validate device_id format at argparse level (alphanumeric + hyphens).

### F-005 (P2) — thermal_guardian.py: blocking subprocess.run() inside async def
- File: tools/thermal_guardian/thermal_guardian.py lines 56-64
- `_read_temp()` is declared `async def` but calls `subprocess.run(..., timeout=2)`.
  `subprocess.run()` is a blocking synchronous call. The asyncio event loop is blocked
  for up to 2 seconds per device per poll.
- With 7 devices polled concurrently via `asyncio.gather()`, the tasks are created
  concurrently but each task blocks the event loop during its subprocess call. The
  effective wall time is 2 seconds (serial blocking) not 2/7 seconds.
- For the demo/POC scenario (single machine, simulation mode), the impact is LOW because
  the simulation fallback in the `except Exception` clause never calls subprocess.
- In production with real nvidia-smi: blocking a 0.5s asyncio loop for 2s causes the
  entire event loop to stall, dropping scheduled iterations.
- Fix: use `asyncio.create_subprocess_exec()` or `loop.run_in_executor()` for the
  subprocess call to keep the event loop non-blocking.

### F-006 (P2) — fabric_energy_controller.py: device_path path traversal risk
- File: tools/fabric_energy/fabric_energy_controller.py lines 168, 174, 180, 187
- `device_path` is accepted from CLI `--device` argument. In production mode:
  `open(f"{self.device_path}/lanes", "w")` — if device_path is supplied as
  `"/dev/../etc/cron.d"`, the open() call would target `/etc/cron.d/lanes`.
- The service file uses default `/dev/bluefield0` and `FABRIC_SIMULATION=1`, so in
  the current deployment this path is never reached. But the production code path is
  reachable via `--device` argument.
- Fix: validate `device_path` with `Path(device_path).resolve().is_relative_to(Path("/dev"))`
  before storing it. Reject any path that does not resolve under `/dev/`.
- Additionally: `int(f.read().strip())` with no bounds — a corrupted sysfs register
  returning an out-of-range integer (e.g., 9999 lanes) would pass `scale_lanes()`'s
  validation only at the call site. The `_read_lane_count()` return is used only for
  comparison, not for another call to `scale_lanes()`, so blast radius is low.

### F-007 (P2) — fabric_energy_controller.py: sysfs integer parsing without bounds
- File: tools/fabric_energy/fabric_energy_controller.py lines 175, 181, 188
- `int(f.read().strip())` on sysfs paths — no maximum value check.
- `credits_pending` could theoretically overflow if a corrupted DPU returns a huge value.
  Python's arbitrary-precision integers prevent integer overflow, but the semantic contract
  (credits == 0 means drained) could be violated by a stuck non-zero value.
- `power_mw` divided by 1000.0 — if DPU returns negative value, `read_power_watts()`
  returns negative watts, which propagates into energy accounting silently.
- Fix: add range assertions: `assert 0 <= credits_pending < 2**32`.

### F-008 (P3) — test_pf_to_mesh_pipeline.c: Merkle buffer size implicit
- File: tests/integration/test_pf_to_mesh_pipeline.c lines 301-302
- `uint8_t commitment[64]` is stack-allocated and passed to `tier_ops.merkle_commit()`.
  The mock writes exactly 64 bytes via `memset(out, 0xAB, 64)`. The interface contract
  in `vdram_tier_ops.h` declares `uint8_t *commitment_out` with no size annotation.
- If a future production implementation writes more than 64 bytes, this is a stack buffer
  overflow. There is no `_Static_assert` or size parameter to prevent this.
- Recommendation: define `VDRAM_MERKLE_COMMITMENT_SIZE 64` in `vdram_tier_ops.h` and
  use it everywhere: `uint8_t commitment[VDRAM_MERKLE_COMMITMENT_SIZE]`.
  This creates a single authoritative size and enables compile-time checking.

### F-009 (P3) — test_pf_to_mesh_pipeline.c: dummy_cert zero-initialized
- File: tests/integration/test_pf_to_mesh_pipeline.c lines 242, 273
- `uint8_t dummy_cert[32] = {0}` — fine for test mocks. The production attestation_ops
  implementation must validate cert length and content. Note for production implementation:
  do NOT accept zero-filled certificates as valid.

### F-010 (P3) — xisc.h: `imm` field is unsigned 8-bit only
- File: xisc/include/xisc.h lines 90-97
- `uint32_t imm : 8` — the immediate field is unsigned. XISC instructions that need
  negative immediates (e.g., LOAD with negative offset) cannot express them.
- The comment says `/* immediate value */` without specifying signedness.
- If the JIT compiler (xisc_llvm_jit.c) assumes signed immediates, it will
  sign-extend this 8-bit field. If it assumes unsigned, negative offsets are impossible.
- Recommendation: explicitly document signedness in the header comment, or change to
  `int32_t imm : 8` with a `_Static_assert` confirming the struct still packs to 32 bits.
  NOTE: changing to signed bitfield is valid in C11 (implementation-defined sign extension
  is the only caveat — gcc/clang both handle this predictably).
