# Frontend Security Audit — 2026-03-14

**Why:** Scheduled security audit of frontend/src before next production deployment.
**How to apply:** Treat all P1 items as blockers for the next deploy. P2 items must be sprint-scheduled.

## P1: JWT + refreshToken in localStorage — CONFIRMED, 20+ files

Both `token` (JWT) and `refreshToken` are stored in localStorage. Every service file
reads directly from localStorage (adminApi, educationApi, tradingApi, analyticsApi, etc.).
This means any XSS vulnerability anywhere in the app can steal both tokens.

Files with direct localStorage.getItem('token'):
- services/api.ts (interceptor), services/auth.ts, services/adminApi.ts, services/tradingApi.ts,
  services/analyticsApi.ts, services/aiTradingApi.ts, services/optionsApi.ts, services/sentinelApi.ts,
  services/riskManagementApi.ts, services/beneficiaryApi.ts, services/familyApi.ts,
  services/wealthAdvisoryApi.ts, services/marketDataApi.ts, services/deviceManagementApi.ts,
  services/subscriptionApi.ts, services/tradingWorkspaceApi.ts, services/autoTradingApi.ts,
  services/websocketService.ts, components/analytics/TradeJournal.tsx, components/alerts/AlertManager.tsx,
  hooks/useStreamingAdvisory.ts, hooks/useMarketStatus.ts, hooks/useAuth.ts (2fa_email in sessionStorage),
  store/slices/authSlice.ts, store/slices/userSlice.ts

Two parallel auth slices exist (authSlice.ts + userSlice.ts) — both write token to localStorage.
Logout() in authSlice.ts does NOT clear refreshToken (only userSlice/auth.ts does).

## P1: No client-side brute-force rate limiting on login form

LoginPage.tsx dispatches auth/login with no attempt counter, no lockout, no CAPTCHA.
Backend is responsible for lockout (login_attempt_limit: 5 seen in deviceManagementApi).
If backend rate limit fails or is misconfigured, login is fully open.

## P1: SSN transmitted in plain registration payload

RegisterPage.tsx submits `ssn: formData.ssn` in the register dispatch payload to
`authSlice.register()` which calls `POST /auth/register` via axios. The SSN is included
in the request body — correctly transmitted over HTTPS, but the authSlice.ts register
thunk type definition includes ssn as a plain string field. The backend encrypts it
(AES-256-CBC confirmed). SessionStorage draft CORRECTLY excludes ssn (destructured out).
Status: ACCEPTABLE if HTTPS is enforced end-to-end. No plaintext rest-storage of SSN.

## P2: MSW mockServiceWorker.js present in public/ directory

File: frontend/public/mockServiceWorker.js
This file is deployed to production in the static bundle. It is the Mock Service Worker
that intercepts fetch/XHR for test mocking. In production it registers as a service worker
if startWorker() is called. Confirmed that no production code path calls it (only src/mocks/).
Risk: If activated somehow, it intercepts ALL API traffic in the browser. Remove from public/.

## P2: CRA build output contains .map files (build/static/js/)

The `build/` directory (CRA output, not vite) contains .js.map files for every chunk.
Vite build (dist/) correctly has sourcemap: false — no .map files.
The build/ directory appears to be a legacy CRA build artifact committed to the repo or
generated locally. If CRA build is used for production instead of vite, source maps are exposed.
Must confirm production uses `npm run build` (vite) not `react-scripts build` (CRA).
If CRA path is used anywhere in CI/CD, add GENERATE_SOURCEMAP=false to the build step.

## P2: Dual auth slice state inconsistency (authSlice + userSlice)

authSlice.logout() clears `token` but NOT `refreshToken` from localStorage.
Only auth.ts/authService.logout() + userSlice also remove refreshToken.
useAuth.logout() calls both dispatch(logoutAction()) + dispatch(authLogoutAction()) + authService.logout() — appears correct.
But direct dispatches of authSlice.logout (without useAuth) leave refreshToken dangling.
Remediation: Add localStorage.removeItem('refreshToken') inside authSlice logout reducer.

## P2: export URL link injection in AccountPage.tsx:1730

```typescript
const exportUrl = `${api.defaults.baseURL}/documents/export?format=csv`;
link.href = exportUrl;
```
api.defaults.baseURL comes from resolveApiBaseUrl() which is safe (origin-checked).
Risk is LOW — not user-controlled. But the link.click() pattern without auth header means
the export endpoint must validate the JWT from the httpOnly cookie or the URL will 401.

## P2: StockProfilePage external link injection (profile.website)

Line 774: `href={profile.website}` — profile.website comes from backend API response.
It is NOT sanitized. A malicious `javascript:` URI in the website field would execute code.
The `rel="noopener noreferrer"` is present but does NOT prevent javascript: protocol execution
when the user clicks the link. Remediation: validate URL scheme before rendering.

## P3: No CSP nonce — inline styles used extensively

CSP in nginx.conf: `style-src 'self' 'unsafe-inline'`
The entire MUI + Tailwind stack uses extensive inline styles (sx prop, styled components).
Removing unsafe-inline requires nonce-based CSP or hash-based approach.
Current CSP is better than no CSP but allows XSS via style injection.

## P3: No CSRF token on mutations

withCredentials: true is set on axios — cookies are sent. But no X-CSRF-Token header
is added to mutation requests. Backend uses JWT Bearer which is CSRF-safe by nature
(attacker cannot read localStorage). However, if migration to httpOnly cookies completes,
CSRF protection becomes mandatory for all state-mutating endpoints.

## P3: logger.ts uses console.log in production for WARN/ERROR

Logger class sets level to WARN in production. formatMessage() always calls console.log
(not console.warn/error). In production, error logs appear as console.log — reduces
signal-to-noise in browser DevTools and any log aggregation tools.

## CONFIRMED SAFE items

- dangerouslySetInnerHTML: ZERO usages found across all .ts/.tsx files
- eval() / Function(): ZERO usages found
- No API keys, secrets, or tokens hardcoded in source files
- No .env files committed to repo (none found in frontend/)
- process.env.REACT_APP_SENTRY_DSN and REACT_APP_GA_TRACKING_ID are empty strings at build time
  (only set via VITE_* env vars, which default to "" in vite.config.ts)
- Vite build: sourcemap: false — dist/ has NO .map files
- Dockerfile: multi-stage, non-root user (appuser), nginx:alpine base
- nginx.conf: X-Frame-Options SAMEORIGIN, X-Content-Type-Options nosniff, frame-ancestors none
- CSP connect-src is restrictive (specific Cloud Run + elsontrade.com domains only)
- window.open() usage is ONLY for mailto: links (safe)
- target="_blank" links all have rel="noopener noreferrer"
- filterSensitiveData() in monitoring.ts removes token, password, ssn, account_number from error reports
- SSN excluded from sessionStorage registration draft (correctly destructured out)
- Vite define block does NOT expose any secret env vars — only public config

## Priority Remediation Order

1. (P1) Remove JWT from localStorage — migrate to reading from httpOnly cookie exclusively
   All 20+ service files need the localStorage.getItem('token') call removed.
   services/api.ts interceptor already has the pattern: if no localStorage token, cookies used.
   Target: make localStorage.getItem('token') return null (stop writing it).

2. (P1) Fix authSlice.logout to also remove refreshToken:
   Add `localStorage.removeItem('refreshToken');` to the logout reducer.

3. (P2) Remove mockServiceWorker.js from public/ (or add to .dockerignore / nginx deny rule)

4. (P2) Confirm production build path uses vite (npm run build), not CRA (react-scripts build)
   Add GENERATE_SOURCEMAP=false as env var to cloudbuild.yaml frontend build step as defense.

5. (P2) Sanitize profile.website before rendering as href — reject non-http(s) schemes.

6. (P3) Add X-CSRF-Token header once httpOnly cookie migration completes.
