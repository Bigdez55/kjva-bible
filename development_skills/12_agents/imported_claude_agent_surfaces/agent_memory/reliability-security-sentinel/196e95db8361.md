**The 286 MB is NOT accounted for by these kernel statics alone (~300 KB max above).**
**Most likely cause: XMIND model weight data (Llama 3.2 3B at Q4_0 ≈ 2 GB on disk) is being
pre-allocated as BSS or mapped into the kernel's .bss section by the build system. The
`xmind_model_t.state` fields (x[], xb[], q[], k[], v[], attn[], logits[]) are heap-allocated
at runtime, but weight pointers and the fallback pool may cause large BSS reservations
if the ISO build includes a pre-packed model binary linked as a symbol.**

**Action required**: Run `size -A kernel.elf | sort -k2 -rn | head -30` on the built kernel
binary to identify the exact symbols contributing to BSS. Likely candidates beyond the above:
- A baked-in model weight array (check ai/xmind/ for any `static const` or `static` weight
  arrays, or any `.incbin` / `KEEP` directives in the linker script)
- A large page-table pool in vmm.c (static PT page pool for early boot)
- XNET packet buffer arrays

## 2. BOOT SECURITY — AetherBoot §16

### ECDSA P-256 Kernel Signature Verification
- **Status: IMPLEMENTED** — `verify_kernel_signature()` present and wired at §16 in aetherboot.c
- Production path: reads `xenos.sig` from ESP, computes SHA-256(kernel), calls `xsec_ecdsa_p256_verify()`
- Correct fail behavior: hard-halts + security recovery console on EFI_SECURITY_VIOLATION
- **OPEN P1**: `GENOS_KEYED_BUILD` flag controls whether real public key or zero-filled placeholder is compiled in
- The `#warning` at §16 fires on builds with GENOS_SECURE_BOOT but without GENOS_KEYED_BUILD
- Zero-key detection is present at runtime — verify_kernel_signature() rejects zero-key builds
- **Gate**: ISO build pipeline MUST pass -DGENOS_KEYED_BUILD=1 and run `embed-pubkey.sh` (all 7 gates)

- **Kernel GENSD stage PCRs (Sprint 47)**: PCR[1-5] via tpm2_tis.c — COMPLETE

- **OPEN P2**: XSTORE_ROOT_HASH in boot_chain.c is all-zeros dev sentinel — must be provisioned before production ISO (logged Sprint 47)

### KASLR

## PRODUCTION DEPLOYMENT BLOCKERS

### P0 Blockers
1. BSS 286 MB — root cause must be identified and fixed before production ISO. Investigate linker script for large .bss symbol injection. A 286 MB kernel image will fail to boot on any real system with <512 MB RAM.
2. `xdev_perm_check` weak stub in boot_stubs.c returns ALLOW (0) — production link MUST include device_perm.c with strong definition to prevent camera/mic bypass.
3. GENOS_KEYED_BUILD=1 with a real ECDSA P-256 key MUST be set for every production ISO build. Zero-key ISO = EFI_SECURITY_VIOLATION = non-bootable. Key ceremony required.
4. XSTORE_ROOT_HASH in boot_chain.c is all-zeros dev sentinel — must be provisioned by ISO build pipeline before production ISO (Sprint 47 OPEN).

### P1 Blockers
5. `scheduler_fiber_rss` and `scheduler_fiber_kill` are unimplemented — OOM kill step is dead code. Production kernel with no process kill capability will panic on first unrecoverable OOM.
6. XNET stubs — network stack not linked into boot kernel (Sprint 47). No network = no remote attestation, no XPKG updates.
7. XKABI grant/revoke is a console print stub — capability delegation to user processes is non-functional.
8. `xcomp_init` and `xdisp_init` are strong stubs — no actual compositor surface pool, no scene graph, no blend pipeline until Sprint 48.

### P2 Items
9. KASLR only 5-bit entropy (32 positions). Extend to 7-bit (128 positions) before production.
10. Audit ring DETAIL_LEN=64 bytes — truncates long TLS/cert detail strings.
11. `s_xm_ffn_gate` and `s_xm_ffn_up` (64 KB combined BSS) in XMIND transformer — acceptable but should be sized to actual model's ffn_dim rather than XMIND_MAX_HEADS×XM_MAX_HEAD_DIM.
12. Watchdog TSC-based — frequency calibration depends on `bi->tsc_hz` from AetherBoot. If AetherBoot passes 0, watchdog is disabled (correctly logged, non-fatal).
13. P1-S34-NVME-01 still open: nvme_setup_prps() stores virtual address in cmd->prp2. All multi-page NVMe I/O (>8KB) will fault or corrupt. BLOCKS storage-intensive operations.
