## NEW FINDINGS (2026-03-25)

### NEW-1 (P0) — openai + sse-starlette missing from requirements-docker.txt
- Root Dockerfile (used by Cloud Build) installs requirements-docker.txt
- openai absent → every get_openai_client() call raises ImportError at runtime → silent LLM fallback
- sse-starlette absent → ruth streaming endpoint fails on import at container startup
- Fix: add to requirements-docker.txt:
    openai>=1.58.0
    sse-starlette>=1.8.2

### NEW-2 (P1) — Dockerfile uses python:3.13-slim, project targets 3.11+
- Dockerfile:2 and :35: FROM python:3.13-slim
- __pycache__ files in repo compiled for 3.12 and 3.14
- onnxruntime>=1.18.0 has no PyPI 3.13 wheels — will source-build and fail without libonnx
- psycopg2-binary: 3.13 wheels available but late; risk of source-build if pip version is old
- Fix: change to FROM python:3.11-slim

### NEW-3 (P1) — ALPACA_API_KEY (generic) never set → health always degraded
- monitoring.py:130 checks settings.ALPACA_API_KEY
- config.py:138 reads os.getenv("ALPACA_API_KEY") — default None
- cloudbuild.yaml only wires ALPACA_PAPER_API_KEY and ALPACA_LIVE_API_KEY (not generic)
- settings.ALPACA_API_KEY is always None in production
- Fix: change monitoring.py health check to: settings.ALPACA_PAPER_API_KEY or settings.ALPACA_LIVE_API_KEY

### NEW-4 (P1) — cryptography>=46.0.5 missing from requirements-docker.txt
- backend/requirements.txt:23 has cryptography>=46.0.5 (CVE-2026-26007 fix)
- requirements-docker.txt does not pin cryptography
- Production container resolves to transitive version — may be below 46.0.5
- websockets>=12.0 and sendgrid also missing (less critical)

### NEW-5 (P1) — gunicorn missing from backend/requirements.txt
- entrypoint.sh:11 calls python -m gunicorn
- requirements-docker.txt has gunicorn>=21.0.0 (root Dockerfile — OK)
- backend/requirements.txt does NOT have gunicorn
- If backend/Dockerfile is used for any deployment, container crashes at entrypoint

### NEW-6 (P1) — _DB_USER default is postgres superuser
- cloudbuild.yaml:365: _DB_USER: postgres
- Production user is elson_user
- If Cloud Build triggered without _DB_USER override, migration job runs as postgres
- Schema objects created by postgres; elson_user app connections may get permission denied
- Fix: change default to _DB_USER: elson_user

### NEW-7 (P2) — KILL_SWITCH_OVERRIDE not pre-wired
- main.py:190 reads KILL_SWITCH_OVERRIDE env var
- Not in Secret Manager, not in cloudbuild.yaml
- Incident response requires manual Cloud Run env var update — unacceptable delay under SEV-1
- Fix: pre-create Secret Manager entry with value "false"; wire in cloudbuild.yaml

### NEW-8 (P2) — LOCAL_LLM_BASE_URL dual-purpose variable
- config.py:400: LOCAL_LLM_BASE_URL default = https://router.huggingface.co
- tiered_inference_router.py:292: uses LOCAL_LLM_BASE_URL as hf_router_url for T1-T5
- If set to EliteBook IP for local inference, ALL free tiers (T1-T5) route to EliteBook
- Should be split: HF_ROUTER_URL (HF serverless) vs LOCAL_LLM_BASE_URL (local llama.cpp)

### NEW-9 (P2) — Dockerfile HEALTHCHECK start-period=5s too short
- Dockerfile:75: --start-period=5s
- Cloud SQL proxy needs 2-5s startup; entrypoint.sh also runs alembic migrations
- Cloud Run liveness probe correctly uses initialDelaySeconds=30
- Fix: --start-period=60s

### NEW-10 (P3) — Dead DB_URL variable in migrate-db step
- cloudbuild.yaml:177: DB_URL variable constructed but never used
- Actual migration uses DATABASE_URL constructed separately at line 208

### NEW-11 (P3) — naive datetime.now() in monitoring endpoints
- monitoring.py:226: datetime.now().isoformat() (no timezone)
- Other endpoints use datetime.now(UTC) — inconsistent
- Fix: replace with datetime.now(UTC).isoformat()

### NEW-12 (INFO) — HF_API_TOKEN correctly wired
- cloudbuild.yaml:125: HF_API_TOKEN=HF_TOKEN:latest
- config.py:378: os.getenv("HF_API_TOKEN", "")
- Chain is correct — not an issue
