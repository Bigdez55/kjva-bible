## P1 FINDINGS — BLOCK PRODUCTION USE

### P1-BOOT-02: e_phoff not bounds-checked vs elf_size before dereference
- File: aetherboot/src/aetherboot.c, line 634
- Code: `Elf64_Phdr *ph = (Elf64_Phdr *)(UINTN)(g_kern_elf_pa + eh->e_phoff);`
- e_phoff is attacker-controlled (from xenos.elf on disk).
- No check that `e_phoff < elf_size` or `e_phoff + e_phnum * sizeof(Elf64_Phdr) <= elf_size`.
- A malicious or corrupted ELF sets e_phoff = elf_size+1, causing ph to point outside the
  allocated buffer. The subsequent loop at lines 635-640 reads OOB from EfiLoaderData memory.
- Fix: assert `eh->e_phoff + (UINTN)eh->e_phnum * sizeof(Elf64_Phdr) <= elf_size` after ELF
  magic validation (line 630) before computing ph pointer.

### P1-BOOT-03: p_paddr + p_memsz UINT64 overflow in load_max computation
- File: aetherboot/src/aetherboot.c, line 638
- Code: `UINT64 end = ph[i].p_paddr + ph[i].p_memsz;`
- Both p_paddr and p_memsz are attacker-controlled UINT64 values.
- If p_paddr = 0xFFFFFFFFFFFFFF00 and p_memsz = 0x200, end wraps to 0xFF (small value).
- load_max stays underestimated. kern_span (line 650) allocates too few pages.
- The segment copy loop at line 675 then writes past the end of g_kern_load_pa — physical
  memory corruption at pre-kernel stage before any fault handler is active.
- Fix: before line 638, check `ph[i].p_paddr <= UINT64_MAX - ph[i].p_memsz`. Also reject any
  segment where p_paddr + p_memsz > 4GB (bootloader only maps 4 GB physical at load time).

### P1-BOOT-04: p_filesz not bounded against kern_span — OOB write in segment copy
- File: aetherboot/src/aetherboot.c, line 675
- Code: `for (UINT64 b = 0; b < ph[i].p_filesz; b++) seg_dst[b] = seg_src[b];`
- seg_off = ph[i].p_paddr - load_min. seg_dst = dst_base + seg_off.
- No check that `seg_off + ph[i].p_filesz <= kern_span`.
- Crafted ELF: p_filesz > (kern_span - seg_off) causes write past g_kern_load_pa allocation.
- Also: no check that `ph[i].p_offset + ph[i].p_filesz <= elf_size` — seg_src can point
  outside the ELF read buffer (OOB read feeding the OOB write).
- Fix: assert `seg_off + ph[i].p_filesz <= kern_span` before the copy. Assert
  `ph[i].p_offset + ph[i].p_filesz <= elf_size` to bound seg_src.

### P1-BOOT-01: NX ambiguity on kernel code pages in AetherBoot page tables
- File: aetherboot/src/aetherboot.c, line 524
- Code: `pd_kern[i] = (kern_phys + i*0x200000) | PTE_P | PTE_RW | PTE_PS;`
- PTE_NX is not defined in aetherboot.c; no NX bit (bit 63) is set in kernel PD entries.
- This is CORRECT — code pages must be executable. PTE_NX is absent = executable. PASS.
- HOWEVER: if the UEFI firmware set EFER.NXE=1 (standard on modern UEFI), the CPU will
  fault if NX were ever OR'd in. The code is safe AS-IS but only because PTE_NX is undefined.
- Risk: a future developer adding `#define PTE_NX (1ULL<<63)` for HHDM hardening (P2-VMM-02
  below) would need to explicitly exclude it from kernel code page entries or face a boot fault.
- Advisory: add a comment at line 524 explicitly stating NX is intentionally absent for
  executable kernel pages. Define PTE_NX with a note. Medium risk without the comment.

## P2 FINDINGS — FIX BEFORE RELEASE

### P2-ACPI-01: rsdp_phys not upper-bounded before HHDM dereference
- File: kernel/xenos/drv/xacpi.c, xacpi_init lines 722-735 + main.c line 1116
- xacpi_init checks rsdp_phys != 0 and validates signature+20-byte checksum after dereference.
- No check that rsdp_phys < HHDM_MAP_LIMIT (128 GB) before dereference.
- A corrupted handoff with rsdp_phys = 0xFFFFFFFF00000000 causes XACPI_PHYS_TO_VIRT to compute
  an unmapped virtual address -> kernel triple-fault in early boot before IDT is active.
- P2 not P1 because UEFI ConfigurationTable pointer is firmware-provided and trusted; requires
  physical access or firmware compromise.
- Fix: add `if (rsdp_phys >= HHDM_MAP_LIMIT) return PAL_ERR_INVAL;` in xacpi_init before line 729.

### P2-VMM-02: HHDM pages mapped without PTE_NX — all 128 GB physical RAM is executable
- File: kernel/xenos/mm/vmm.c, line 288
- Code: `pd[m] = phys_2m | PTE_PRESENT | PTE_WRITE | PTE_PS | PTE_GLOBAL;`
- All 128 GB HHDM is mapped RWX. Any physical page reachable via HHDM can be executed.
- If attacker can place shellcode at a known physical address (e.g., via a write primitive),
  it is trivially executable.
- Fix: add PTE_NX to HHDM data page entries. Kernel code does not execute via HHDM;
  it executes via KERN_BASE mapping which is separate (and also lacks NX — see pass below).

### P2-ELF-01 (DEFERRED): p_flags ignored — W^X not enforced at boot
- File: aetherboot/src/aetherboot.c, line 524
- PF_X is defined (line 360) but never used in build_page_tables().
- All kernel segments mapped RW+exec (RW, NX absent). .data pages are executable.
- vmm.c line 328 comment acknowledges: "text sections remapped RO+X in Sprint 2 by ELF loader"
- This is a known deferred item, NOT a new finding. Status: OPEN per Sprint 2 roadmap.

## PASS FINDINGS (VERIFIED SECURE)

### PASS-VMM-01: PML4_IDX_HHDM = 273 is CORRECT and CONSISTENT
- vmm.c line 95: `#define PML4_IDX_HHDM 273u`
- Mathematical verification: 0xFFFF888000000000 >> 39 = 0x1FFFF11 & 0x1FF = 0x111 = 273. CORRECT.
- aetherboot.c line 505: `pml4[273] = pdpt_hhdm_pa`. CONSISTENT.
- Zero remaining references to the old wrong value 272 found in either file. PASS.

### PASS-VMM-02: PML4_IDX_KERN = 511 is CORRECT and CONSISTENT
- vmm.c line 96: `#define PML4_IDX_KERN 511u`
- aetherboot.c line 506: `pml4[511] = pdpt_kern_pa`. CONSISTENT. PASS.

### PASS-BOOT-05: do_kernel_jump CR3+RDI+jmpq sequence is correct
- aetherboot.c line 704: CR3 loaded first (address space switch).
- Line 709: RDI = (UINT64)(UINTN)hi — handoff pointer loaded.
- Line 710: jmpq *%0 — indirect jump to entry_vaddr.
- Stack (RSP) is in first 4 GB -> covered by identity map at pml4[0]. PASS.
- g_xenos_handoff is a PE32+ static -> loaded below 4 GB by UEFI -> covered by identity map. PASS.
- g_xenos_handoff populated before ExitBootServices (lines 879-893), not modified after. PASS.
- No code path modifies g_xenos_handoff between EBS and do_kernel_jump. PASS.

### PASS-BOOT-06: ExitBootServices retry does not corrupt handoff
- Lines 832-834: retry only re-calls GetMemoryMap to get fresh map_key, then EBS again.
- g_xenos_handoff not repopulated in retry path. mem_map_count uses map_size2/desc_size2 from
  the first GetMemoryMap call; retry updates map_key but not mem_map_count. This is a minor
  inconsistency (count from first map, key from second) but not a security issue. PASS.

### PASS-BOOT-01: ELF header validation is adequate
- aetherboot.c lines 622-630: magic (4 bytes via __builtin_memcpy), ELFCLASS64 (e_ident[4]),
  ELFDATA2LSB (e_ident[5]), EM_X86_64 (e_machine), ET_EXEC|ET_DYN (e_type). PASS.

### PASS-BOOT-02: File size bounded by KERN_ELF_BUF_PAGES before allocation
- aetherboot.c line 594: rejects elf_size == 0 and elf_size > 8 MB. PASS.

### PASS-BOOT-03: Empty PT_LOAD set rejected
- aetherboot.c line 641: load_min == (UINT64)-1 || load_max == 0 -> return 0. PASS.

### PASS-BOOT-04: kern_pages clamped to [1..512]
- aetherboot.c lines 521-522: `if (kern_pages == 0) kern_pages = 1; if (kern_pages > 512)
  kern_pages = 512;` PASS. 512 * 2 MB = 1 GB max kernel mapping. Reasonable.

### PASS-ACPI-01: RSDP validation in xacpi_init is adequate
- xacpi.c xa_rsdp_valid(): 8-byte signature comparison + 20-byte byte-sum checksum = 0. PASS.
- xacpi_init: null ctx guard, zero rsdp_phys guard, validation before any field access. PASS.
- XSDT and all SDTs also checksum-validated before use. PASS.
- main.c Step 10a: `if (rsdp_phys)` guard before xacpi_init call. PASS.
- Fallback chain: bi->acpi_rsdp_phys (Sprint 13) -> EFI mmap scan -> BIOS ROM scan. PASS.

### PASS-UEFI-01: EFI_LOADED_IMAGE_PROTOCOL struct layout matches UEFI 2.10 spec §8.1

### PASS-UEFI-02: EFI_SIMPLE_FILE_SYSTEM_PROTOCOL matches UEFI 2.10 spec §13.5

### PASS-UEFI-03: EFI_FILE_PROTOCOL vtable order matches UEFI 2.10 spec §13.5
