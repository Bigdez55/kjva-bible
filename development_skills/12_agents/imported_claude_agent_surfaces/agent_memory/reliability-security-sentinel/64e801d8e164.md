## 1. Process Isolation Gap — XK_SYS_EXEC Analysis

### What the code does (main.c lines 960-988)

XK_SYS_EXEC (syscall 29) takes a1=entry_function_pointer, a2=argument, and creates a
cooperative kernel fiber sharing the SINGLE kernel address space. It is functionally
identical to XK_SYS_THREAD_CREATE (syscall 7) with no architectural distinction — both
call pmm_alloc_pages(), build the same stack frame, and set XF_READY. The only difference
is the log message and the parameter name "exec_slot" vs "slot".

There is NO:
- Separate page table (PML4) per "process"
- Ring-3 privilege boundary (all fibers run at CPL0)
- Capability gate on what entry function address can be passed
- Memory range check: a1 can be any uint64_t cast to a function pointer

### Risk Classification by Environment

**QEMU dev environment (Sprint 15 boot test):**
- Blast Radius: ZERO external users. Single-developer boot test.
- Threat actors: None. Single-image QEMU VM with no network exposure.
- Risk classification: LOW for this specific use case.
- Verdict: ACCEPTABLE for Sprint 15 QEMU boot test under documented constraint.
  The shared address space is a known architectural placeholder. All code running
  in this environment is trusted kernel code. There are no untrusted user programs.

**Production (any physical hardware deployment):**
- Blast Radius: TOTAL. A malicious service calling XK_SYS_EXEC with an arbitrary
  entry address can jump to any kernel function, read any kernel memory, overwrite
  interrupt handlers, etc.
- Risk classification: P0 CRITICAL for production.
- Verdict: MUST NOT ship XK_SYS_EXEC in its current form in any production ISO.

### Minimum Viable Isolation for Sprint 16

Sprint 16 must add at minimum:
1. A separate PML4 per "process" — vmm.c already has vmm_map(); a per-process
   cr3 register value must be stored in xenos_fiber_t and loaded on context switch
   in xenos_fiber_switch (currently does NOT swap CR3).
2. Entry address validation: the a1 function pointer must reside within a known
   code segment range (checked against the ELF text segment boundaries loaded by
   XEMU/XBUILD). Arbitrary kernel addresses must be rejected — XENOS_EINVAL.
3. Ring-3 transition: a user fiber should enter at CPL3. This requires IRETQ to
   user-mode rather than the current direct trampoline call at CPL0.
4. The XK_SYS_EXEC implementation is identical to XK_SYS_THREAD_CREATE. In Sprint 16
   they must diverge: THREAD_CREATE = kernel fiber (trusted, CPL0), EXEC = user
   process (untrusted, CPL3, separate PML4). Current code conflates them.

### Specific Code Change Required (Sprint 16)

File: `kernel/xenos/core/main.c`
- xenos_fiber_t: add `uint64_t cr3` field (currently missing)
- xenos_fiber_switch: add `mov cr3, [saved_cr3]` after RSP swap (requires naked asm change)
- XK_SYS_EXEC case: validate a1 within [text_start, text_end] or reject. Allocate
  new PML4 via vmm. Switch to CPL3 via IRETQ.
- Add XENOS_EXEC_VALIDATION compile-time gate flag; assert it is set before any
  release ISO build.

## 2. Boot Chain Security Audit — boot_chain.c

### SHA-256 KAT abort behavior (VERIFIED SECURE)

`xchain_security()` at line 266 computes SHA-256("") and compares byte-by-byte against
the hardcoded expected digest. On ANY mismatch it returns PAL_ERR_INVAL (line 283).

`xchain_run()` at lines 659-665 checks this return code:
```
s_result.security_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] CRITICAL: security KAT FAILED\n");
    return rc;   // <-- ABORTS THE ENTIRE BOOT CHAIN
}
```
The early return propagates to the caller (boot_sequence.c / kmain). If boot_sequence.c
treats a non-PAL_OK return from xchain_run() as fatal (which it must — audit this call
site in Sprint 15), the KAT failure correctly halts boot.

FINDING: The KAT abort is CORRECT and SECURE in boot_chain.c itself. The security
assumption is that the xchain_run() caller in boot_sequence.c/kmain also treats a
non-PAL_OK return as fatal. This must be verified in Sprint 15.

### Weak function stubs — safety assessment

All five weak stubs (xnet_init, xdisp_init, xcomp_init, xshell_start, xmind_init) return
PAL_OK silently. This is SAFE for the following reasons:

1. The stubs are __attribute__((weak)) — they are replaced at link time by real
   implementations when those subsystems are compiled in. The weak stub is only
   active when the subsystem is genuinely absent from the image.

2. Boot chain failure modes for weak-stub subsystems are non-critical (network,
   AI) or produce a WARN (shell). Display and compositor failures ARE critical and
   abort the chain (lines 684-695) — but their real implementations will be linked.

3. In a QEMU test where XDISP is not linked (weak stub active), xchain_display()
   will FAIL at the fb_phys == 0 check (line 399) since QEMU passes a real
   framebuffer in the handoff. This means xchain_display will call the real
   xdisp_init() if XDISP is linked, or fail at framebuffer validation if not.
   This is correct behavior — it does NOT silently succeed with no framebuffer.

FINDING: Weak stubs are SAFE. They cannot produce a false-positive "security passed"
state because security (SHA-256 KAT) does not use a weak stub path.

### Privilege escalation path via service spawn

The only service spawn call is xshell_start() (line 519). xshell_start() is either:
- The weak stub (returns PAL_OK, no process created)
- The real xshell.c implementation which calls XK_SYS_EXEC

Since XK_SYS_EXEC creates a kernel fiber at CPL0 with no capability restriction, any
service spawned by xshell_start() inherits full kernel privileges. This is NOT an
escalation — it is a flat privilege model. There is no lower privilege to escalate FROM.

FINDING: No escalation path in the boot chain itself. The risk is the flat CPL0 model
(covered under S15-P1-01 above). Services cannot escalate because they start at the
maximum privilege level. This is a design debt for Sprint 16, not a Sprint 15 boot blocker.

### XPKG root key — placeholder status (S15-P1-02)

`XPKG_ROOT_FINGERPRINT[32]` in boot_chain.c lines 338-343 contains ASCII bytes for
"GENOSROOT TKEY0001" followed by zeros and 0x01. This is NOT a valid Ed25519 public key.
The validation at lines 373-379 only checks that the fingerprint is non-zero — it
accepts the placeholder.

ADR-S14-08 documents this gap explicitly: trust anchor lookup is NOT implemented in
xpkg_verify_signature(). XPKG_ENFORCE_TRUST_ANCHOR defaults to 0.

Classification: P1 before any release build. ACCEPTABLE for QEMU boot test — no
untrusted packages will be installed during a first-boot QEMU test, and
XPKG_ENFORCE_TRUST_ANCHOR=0 means the placeholder does not block operation.

Action: Sprint 15 must implement the trust anchor lookup in xpkg_verify_signature()
per ADR-S14-08 before any package installation workflow is tested.

## 3. Virtio-net DMA Safety Requirements

Sprint 15 will write a new virtio-net driver. No virtio-net files exist yet (Glob
returned no matches). The following requirements must be enforced during design:

### DMA Safety in QEMU (no physical IOMMU)

QEMU's virtio presents a virtio-mmio or virtio-pci device to the guest. The guest (XENOS)
sees a PCI BAR or MMIO region for the virtio queue descriptors. DMA occurs when XENOS
writes a guest-physical address into the virtqueue descriptor ring and the host (QEMU)
reads from or writes to that guest-physical address directly.

In QEMU: IOMMU is optional (enabled with `-device intel-iommu` or `-machine q35,iommu=on`).
For Sprint 15 QEMU boot test without IOMMU, DMA safety requirements are:

1. **Descriptor ring addresses must be within XENOS-owned physical memory.**
   Before writing a buffer address into a virtqueue descriptor, verify the physical
   address lies within the range provided by the XENOS memory map (not in MMIO regions,
   not at physical 0, not past the top of RAM). Failure: QEMU can write to arbitrary
   guest-physical memory including kernel code/data.

2. **Receive buffer pre-allocation from a dedicated DMA pool.**
   Receive buffers must be allocated from a dedicated pool, NOT from the heap or stack.
   The physical address of these buffers is exposed to the device. If a stack buffer's
   physical address is given to virtio, QEMU writes received packet data onto the kernel
   stack — a reliable stack overflow primitive.
   Recommended: Allocate a fixed receive ring of N×2048-byte buffers from PMM at driver
   init. These buffers remain pinned (never freed while the NIC is active).

3. **Maximum safe receive buffer size: 2048 bytes per descriptor.**
   Ethernet MTU is 1514 bytes (1500 payload + 14 header). With virtio-net header
   (12 bytes for virtio_net_hdr), safe upper bound is 1526 bytes. Allocate 2048-byte
   buffers (one page-aligned PMM allocation per descriptor) for alignment simplicity.
   Using 4096-byte buffers is also safe but wastes memory; using <1526 risks truncation.
   Using heap memory (heap_alloc() addresses) is UNSAFE — the heap is contiguous with
   kernel data, a device write overflow corrupts kernel state.

4. **Input validation on received packets before passing to XNET:**
   a. Bounds check: packet_len must be [14, 1514]. Reject len < 14 (no Ethernet header)
      and len > 1514 (jumbo frames not supported by XNET). Do NOT pass oversized frames.
   b. EtherType whitelist: XNET handles IPv4 (0x0800), ARP (0x0806), IPv6 (0x86DD).
      Drop all other EtherTypes before passing to XNET to reduce attack surface.
   c. virtio_net_hdr validation: flags field must only contain known bits
      (VIRTIO_NET_HDR_F_NEEDS_CSUM=1, VIRTIO_NET_HDR_F_DATA_VALID=2). Unknown flag bits
      must be masked or the packet dropped. Do not trust gso_type from an untrusted device.
   d. Source from ring: read descriptor length from used_len in the used ring, not from
      a field inside the packet. used_len comes from the host but is bounded by the
      buffer size you pre-allocated — it cannot exceed 2048.

5. **IOMMU note for production (HP EliteBook x360):**
   Intel VT-d is present on Whiskey Lake. For production, XENOS must enable IOMMU
   translation before the NIC driver starts. This prevents a compromised device from
   issuing DMA to arbitrary physical addresses. Sprint 15 QEMU boot test can proceed
   without IOMMU; production Sprint 16 requires IOMMU setup in vmm.c.

## 4. AES-NI Implementation Plan

### P0-S3-AES-01 STATUS: RESOLVED

Direct file read of `sec/xsec/crypto/aes_gcm.c` line 35 confirms:
```c
static volatile const uint8_t AES_SBOX[256] = { ... };
```
The `volatile` qualifier is present. This forces the compiler to emit actual memory read
instructions for each AES_SBOX[i] access inside aes_sbox_ct(), preventing the optimizer
from collapsing the 256-iteration scan to a direct index. P0-S3-AES-01 is RESOLVED.

The aes_sbox_ct() function at line 59 uses a branchless equality comparison:
```c
uint32_t eq = (uint32_t)((uint8_t)i ^ idx);
eq = (eq - 1u) >> 8u;
result |= (uint8_t)(AES_SBOX[i] & (uint8_t)eq);
```
This is a correct constant-time select. The volatile on AES_SBOX ensures all 256 reads
occur. PASS.

### AES-NI Sprint 15 P2 Implementation Guidance

ADR-S14-07 documents this as Sprint 14 P2 (carried to Sprint 15). The implementation
plan uses `__builtin_ia32_aesenc` intrinsics (approved in ADR-S14-01).

#### CPUID Feature Detection (correct sequence)

```c
// Detect AES-NI: CPUID EAX=1, check ECX bit 25
static inline int cpu_has_aesni(void) {
    uint32_t ecx = 0u, edx_unused = 0u;
    __asm__ volatile(
        "cpuid"
        : "=c"(ecx), "=d"(edx_unused)
        : "a"(1u), "c"(0u)
        : "ebx"
    );
    return (ecx >> 25u) & 1u;  // bit 25 = AESNI
}
```
Note: EBX is clobbered by CPUID and must be listed. On PIC code, EBX may be in use —
for freestanding kernel code this is not a concern (no GOT/PLT), but the clobber is
required for correctness.

#### Alignment Requirements

AES-NI instructions (AESENC, AESENCLAST, AESDEC, AESDECLAST, AESKEYGENASSIST) operate
on XMM registers. There are NO alignment requirements on the memory operands for these
instructions — they accept unaligned addresses, unlike SSE2 MOVDQA which requires
16-byte alignment. However:
- The AES round key array (uint8_t rk[11*16]) should be 16-byte aligned for performance.
  Use `__attribute__((aligned(16)))`.
- The state buffer (16 bytes for one AES block) should also be 16-byte aligned.
- PCLMULQDQ (for GHASH) similarly has no alignment requirement on memory operands.

#### Key Schedule (AES-128, AESKEYGENASSIST)

The AES-128 key schedule expands 16 bytes → 11 round keys (176 bytes total).
The correct sequence using AESKEYGENASSIST is:

```
Round key 0 = original key (load directly)
For i in 1..10:
    temp = AESKEYGENASSIST(rk[i-1], RCON[i])
    temp = PSHUFD(temp, 0xFF)          // broadcast word 3 to all four words
    rk[i-0] = rk[i-1]
    rk[i-0] = XOR(rk[i-0], PSLLDQ(rk[i-0], 4))   // shift left 4, XOR
    rk[i-0] = XOR(rk[i-0], PSLLDQ(rk[i-0], 4))
    rk[i-0] = XOR(rk[i-0], PSLLDQ(rk[i-0], 4))
    rk[i-0] = XOR(rk[i-0], temp)
```

RCON values for AES-128 (10 rounds):
```
0x01, 0x02, 0x04, 0x08, 0x10, 0x20, 0x40, 0x80, 0x1B, 0x36
```
These are passed as the imm8 to AESKEYGENASSIST.

The key expansion uses PSLLDQ (byte-shift left on XMM) to build the left-shift-XOR
chain. This is the Intel-documented expansion algorithm. Using `__builtin_ia32_aeskeygenassist128`,
`__builtin_ia32_pslldqi128`, and `__builtin_ia32_pxor128` (or their `_mm_*` equivalents
from `<wmmintrin.h>`, which is permissible under ADR-S14-01) implements this correctly.

For AES-256 (which XSEC currently implements in software), the key schedule is more
complex: it requires two AESKEYGENASSIST calls per round (one with imm8=RCON, one with
imm8=0x00). Since XSEC uses ChaCha20-Poly1305 as the TLS cipher and AES-GCM is currently
rejected (ADR-S14-07), AES-128 key schedule is sufficient for the initial AES-NI sprint
(disk encryption use case). AES-256 expansion can follow.

#### Integration Point

The new file should be `sec/xsec/crypto/aes_gcm_ni.c`, compiled only when
`-maes -mpclmul` flags are set (clang/gcc both support this for freestanding targets).
A runtime dispatch in `aes_gcm.c` selects the NI path via the `cpu_has_aesni()` check
at first call, cached in a module-level flag.

## 5. WAL Checkpoint Security Audit

### xstore_checkpoint() — EXISTS and is implemented

`xstore_checkpoint()` is NOT a stub. It exists in `store/xstore/xstore.c` at line 155:
```c
xstore_status_t xstore_checkpoint(xstore_t *store) {
    if (!store || !store->open) return XSTORE_ERR_NOT_OPEN;
    if (store->read_only) return XSTORE_ERR_READONLY;
    return xstore_wal_checkpoint(store);
}
```
It delegates to `xstore_wal_checkpoint()` in `wal/wal.c` (line 180), which is a full
implementation: flush dirty pages → append CHECKPOINT WAL entry → write superblock → sync().

It is called in two places:
- `xstore_open()` line 128: after btree init on a fresh store — CORRECT
- `xstore_close()` line 148: final checkpoint on close — CORRECT

### Crash-Safety Invariants (verified from wal.c)

The checkpoint implementation maintains the following invariants:

1. **Dirty pages before superblock.** Step 1 (cache_flush) precedes step 3 (superblock
   write). If power fails after cache_flush but before superblock write, the pages are
   durable but checkpoint_lsn is stale — the WAL replay will re-apply already-flushed
   data. This is idempotent (INSERT to already-present key is a no-op by convention)
   and safe.

2. **Superblock checksum.** Line 211: `store->sb.hdr.checksum = xstore_crc32c(sb_bytes + 8, ...)`.
   The checksum covers the superblock excluding the first 8 bytes (magic + checksum field
   itself). On recovery, xstore_open() must validate this checksum before trusting
   checkpoint_lsn. VERIFY this is done in xstore_open() — if not, a torn write to the
   superblock could corrupt checkpoint_lsn, causing replay to skip committed transactions
   (data loss) or replay already-checkpointed data (harmless duplicate, but signals gap).

3. **WAL tail advance is LAST.** Step 5 (wal_tail = checkpoint_lsn + 1) runs AFTER
   sync() returns. This is correct — the tail must not advance until the checkpoint is
   durable. If sync() returns an error, wal_tail is NOT advanced. PASS.

4. **checkpoint_lsn derivation — minor race (S15-P2-02).** At line 200:
   ```c
   uint64_t checkpoint_lsn = store->next_lsn - 1;
   ```
   `next_lsn` is a pal_atomic_add counter. This reads next_lsn AFTER appending the
   CHECKPOINT entry (line 197). The value `next_lsn - 1` is the LSN assigned to the
   just-appended CHECKPOINT entry. This is correct in a single-threaded checkpoint path.
   In a concurrent scenario (multiple writers + concurrent checkpoint), next_lsn - 1 could
   be a different entry's LSN if another append raced between xstore_wal_append() returning
   and the next_lsn read. The checkpoint is protected by no explicit mutex — only wal_lock
   is held during append. This is a low-risk race on a single-CPU cooperative kernel
   (XENOS fibers are non-preemptive) but becomes a real issue if checkpoint is ever called
   concurrently with WAL appends. Document as S15-P2-02.

5. **WAL FULL during checkpoint.** If `xstore_wal_append(store, &cp_entry)` returns
   XSTORE_ERR_WAL_FULL (ring full), xstore_wal_checkpoint() returns that error at line 198
   without advancing the tail. The ring cannot drain itself because tail advance requires
   a successful checkpoint. This is a potential deadlock: ring full, checkpoint fails because
   ring full, ring cannot drain. The caller (xstore_txn_commit) is told to invoke
   checkpoint when the ring is full — but if checkpoint needs to append to the ring,
   this is circular. MITIGATION: Reserve one ring slot for the checkpoint entry
   (ring_used >= XSTORE_WAL_RING_SIZE - 1 triggers a pre-emptive checkpoint before the
   ring is completely full). This is a Sprint 15 design task.

**S15-P1-03: XSEC P1 findings (CN bypass, KABI derive race) — still open.**
- P1-S3-CERT-01: x509.c CN null-byte bypass — not fixed as of this audit
- P1-S3-CERT-04: CN extraction from formatted string — not fixed
- P1-S3-KABI-01: xkabi_cap_derive revoked-parent race — not fixed
- These do not block QEMU boot (TLS is not exercised in first boot; XKABI race is

### P2 — Sprint 16 Target

**S15-P2-01: XK_SYS_EXEC is a copy-paste of XK_SYS_THREAD_CREATE.**
- main.c lines 810-839 (THREAD_CREATE) vs lines 960-988 (EXEC) are structurally
  identical. Sprint 16 must make them diverge: THREAD_CREATE stays as kernel fiber,
  EXEC becomes user-process entry point with separate page table.

**S15-P2-02: WAL checkpoint_lsn derived from next_lsn - 1 without mutex.**
- wal.c line 200. Safe on single-CPU cooperative XENOS. Becomes a race under SMP
  or if checkpoint is called from an interrupt-driven path. Document and add mutex
  guard when SMP support is added.

**S15-P2-03: Boot chain storage failure is non-critical but AI requires storage.**
- boot_chain.c lines 652-657: storage failure logs WARN and continues.
- If XBLOB region is not in the memory map, xchain_ai() will still attempt to
  initialize XMIND, which in Sprint 14+ tries to load weights from XBLOB.
  This will fail at weight load time, not at init time, producing a confusing
  late-stage failure.
- Fix: if s_result.storage_ok == 0, skip xchain_ai() or downgrade it to
  probe-only mode (no weight load attempt).

**S15-P2-04: virtio-net DMA safety design must be reviewed before driver implementation.**
- See Section 3 above for full requirements.
- Key gate: receive buffer physical addresses must be validated against XENOS memory map
  before being placed in the virtqueue descriptor ring.

**S15-P2-05: WAL ring full during checkpoint — circular dependency.**
- See Section 5, invariant 5. Reserve ring slot for checkpoint entry.
- Low risk in current sprint (single boot test, minimal WAL traffic), but must be
  addressed before XSTORE is used for any significant workload.
