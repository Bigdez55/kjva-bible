# ajv Version Conflict — Root Cause Analysis (2026-02-24)

## Status: DIAGNOSED — NOT YET FIXED

## The Core Conflict

`package.json` has a blanket override `"schema-utils": { "ajv": "^6.14.0" }` that forces ALL
schema-utils instances to receive ajv@6, including schema-utils@4.3.3 which requires `"ajv": "^8.9.0"`.

This produces `npm ls ajv` exit 1 with ELSPROBLEMS (two invalid entries).

## The Four Correctly-Scoped Overrides (DO NOT REMOVE)

```json
"@eslint/eslintrc": { "ajv": "^6.14.0" },   // eslintrc@2 needs ajv@6
"eslint": { "ajv": "^6.14.0" },             // eslint@8 needs ajv@6
"file-loader": { "ajv": "^6.14.0" },        // file-loader uses schema-utils@3 needs ajv@6
"fork-ts-checker-webpack-plugin": { "ajv": "^6.14.0" }  // uses schema-utils@2 needs ajv@6
```

## The Offending Override (REMOVE THIS)

```json
"schema-utils": { "ajv": "^6.14.0" }   // blanket — overrides ALL schema-utils including v4
```

## Recommended Fix

Remove only the `"schema-utils": { "ajv": "^6.14.0" }` line from `package.json` overrides.

Then run (note: requires --legacy-peer-deps due to typescript@5 vs react-scripts peerOptional):
```bash
cd frontend && PATH="/opt/homebrew/opt/node@22/bin:$PATH" npm install --legacy-peer-deps
```

Then verify:
```bash
PATH="/opt/homebrew/opt/node@22/bin:$PATH" npm ls ajv 2>&1
# ACCEPT: exit 0, no "invalid:" lines
PATH="/opt/homebrew/opt/node@22/bin:$PATH" npm run build 2>&1
# ACCEPT: "Compiled successfully"
PATH="/opt/homebrew/opt/node@22/bin:$PATH" npm test -- --watchAll=false 2>&1
# ACCEPT: 332+ passed, 0 regressions
```

## Why This Is NOT a Production Risk

- The `node_modules/css-minimizer-webpack-plugin/node_modules/ajv@6.14.0` entries in the lockfile
  are NOT physically installed on disk — those packages have no nested node_modules/ajv directory.
  They correctly resolve to the hoisted ajv@8.18.0 at runtime.
- The conflict is a lockfile/npm-ls reporting issue compounded by the blanket override.
- No production bundle content is affected. ajv is a build-tool dep only.

## Is package-lock.json the Source?

No. The lockfile (lockfileVersion: 3) faithfully records the overrides. It is NOT independently
corrupt. Removing the bad override from package.json and running npm install regenerates the
lockfile correctly — no manual lockfile editing required.
