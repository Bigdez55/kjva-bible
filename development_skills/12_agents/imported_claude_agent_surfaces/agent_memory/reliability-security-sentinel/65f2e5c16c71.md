## 1. XKABI Capability Handle v2 (xkabi_capabilities.c)

### HMAC MAC truncation — 8 bytes (64 bits)

**Assessment: ADEQUATE for current threat model, with one structural caveat.**

HMAC-SHA256 truncated to 8 bytes provides 2^64 brute-force resistance against a
random-guess attacker. The HMAC key is per-process-handle-table (32 bytes from
get_random_bytes). This is sound. Birthday attack against MAC values requires
2^32 attempts before a 50% collision probability — that is a quadrillion operations.
Not feasible for a local process.

**STRUCTURAL ISSUE (P2):** The HMAC is computed over only the first 8 bytes of the
handle ([0:7] = object_id_lo48 + rights32[:2]). Bytes [8:15] contain the 8-byte MAC.
But bytes [6:9] are rights32 (all 32 bits), and bytes [10:13] are generation. The
comment says "HMAC over [0:7]" but rights32 spans bytes [6:9], so generation bytes
[10:13] are NOT covered by the MAC. An attacker who can observe a valid handle could
flip the generation field in bytes [10:13] without invalidating the MAC. The generation
field governs revocation epoch checks. If generation is not MAC-bound, revoked handles
can potentially be replayed by flipping the generation counter to an older epoch.

**FIX:** Expand HMAC coverage to all 16 bytes of the unprotected prefix, OR include
generation in the 8-byte MAC input. The MAC should commit to at minimum:
  [0:5] obj_id_lo48 + [6:9] rights32 + [10:13] generation.

### Constant-time comparison

**Assessment: CORRECT.** `crypto_memneq()` is the Linux kernel's constant-time
comparison. Correct function, correct length (8). No issue.

### Handle forgery resistance

**Assessment: STRUCTURALLY SOUND, one alias risk.**

The idr_find lookup uses `(int)(object_id & 0xFFFFFF)` — 24-bit truncation. The
object_id is 48 bits. Two different object_ids that differ only in bits [24:47]
will hash to the same IDR slot. An adversary who can allocate handles could craft
object_id collisions at the IDR layer, causing the wrong cap to be looked up
post-MAC verification. MAC only covers the 48-bit object_id; IDR lookup truncates
to 24 bits. This is a latent aliasing bug.

**FIX:** IDR lookup must use the full object_id. Either store the full 64-bit
object_id in the cap and verify it post-lookup, OR extend the IDR key space.

Also: xkabi_audit_log() casts the 32-bit rights to u16 in both XKABI_AUDIT_DENY
and XKABI_AUDIT_ALLOW events (line 202, 215). This silently truncates hw rights
bits [16:23] from the audit record. Hardware capability rights are not audited.

### Rights escalation prevention (u16→u32 transition)

**Assessment: CORRECTLY IMPLEMENTED for derive path. One gap in validate path.**

xkabi_cap_derive(): the check `(derived_rights & ~parent_rights) != 0` correctly
blocks widening. The parent must also have XKABI_RIGHT_DERIVE (bit 23). Both checks
are present. Rights narrowing is enforced.

**GAP (P1):** xkabi_cap_validate() extracts rights from the handle via
xkabi_verify_handle() and uses those extracted rights for the authorization check.
It does NOT re-read rights from the in-kernel cap object. An attacker with a valid
handle who can manipulate the handle bytes (between MAC coverage and rights field)
could potentially present escalated rights. However, since rights ARE covered by the
MAC (bytes [6:9]) this is only exploitable if the MAC is broken — which at 2^64 is
not feasible. This is acceptable given the MAC coverage. NOTE: if the HMAC coverage
bug above (generation not covered) is exploited, then generation-replay combined
with rights mutation becomes a concern.

## 2. RCU Correctness (mr_registry.c)

### Memory ordering guarantees

**Assessment: CORRECT.**

- Writers: hold rkey_table_lock (spinlock) and use rcu_assign_pointer() which
  includes the appropriate memory barrier (smp_wmb or equivalent) before the
  pointer store. Correct.
- Readers: rcu_read_lock() + rcu_dereference() in vdram_mr_lookup_rkey(). Correct.
- Deletions: RCU_INIT_POINTER() is used to NULL-out the table slot in both free
  and rotate paths. RCU_INIT_POINTER sets the pointer without a barrier, which
  is permitted because the lock is held. Correct.

No memory ordering violations found.

### TOCTOU race in rkey lookup/rotation

**Assessment: ONE LATENT RACE (P2).**

vdram_mr_lookup_rkey() increments mr->ref under rcu_read_lock. But rcu_read_lock
does NOT prevent the writer (vdram_mr_free or vdram_rkey_rotate) from executing
on another CPU after the RCU grace period. The sequence is:

  Reader:         rcu_read_lock() → rcu_dereference() → NULL check → atomic_inc(mr->ref) → rcu_read_unlock()
  Concurrent:                                            vdram_mr_free() → kfree(mr)

The problem: there is no synchronize_rcu() before kfree() in vdram_mr_free().
If the reader is in the window between rcu_dereference() and atomic_inc(), the
writer can proceed past spin_unlock() and call kfree(). The ref increment then
writes into freed memory.

The canonical fix is call_rcu() or synchronize_rcu() before kfree, OR use
atomic_inc_not_zero() with a separate "alive" flag that the free path sets to 0
before the grace period.

**FIX:** vdram_mr_free() must call synchronize_rcu() after nulling the rkey_table
slot and before kfree(mr). Or restructure with refcount_t and call_rcu.

### rkey collision detection loop

mr_generate_rkey() busy-loops calling get_random_bytes until it finds a free slot
(excluding zero). This loop is called while holding rkey_table_lock. In a nearly
full table (approaching MR_MAX_PER_NODE = 65536 entries) this loop could spin for
a very long time under the spinlock, causing soft-lockup. The probability of
needing >N retries is (used/65536)^N. At 99% full: P(>10 retries) ≈ 0.9^10 ≈ 35%.
This is a DoS vector if a privileged process can exhaust the rkey table.

**FIX:** Add an iteration limit (e.g., 100 retries) and return -ENOSPC if exceeded.

## 3. Provenance Client (provenance_client.py)

### URL injection via PROVENANCE_SERVICE_URL

**Assessment: P2 — OPEN REDIRECT / SSRF via env var.**

SERVICE_URL = os.getenv("PROVENANCE_SERVICE_URL", "http://provenance:8000/api/v1/events")

This env var is consumed verbatim with no validation. If an attacker can control
environment variables (compromised container, malicious deployment), they can redirect
audit records to an attacker-controlled server, or use SSRF to reach internal
cluster services. In the k3s deployment this env var should be locked down.

**FIX:** Validate SERVICE_URL on startup: must start with "http://provenance:" or
"http://localhost:" (for dev). Reject any other scheme or host. Use a whitelist
of allowed hosts, not a denylist.

### JSONL injection in log entries

**Assessment: LOW (P3) — mitigated by json.dumps but worth flagging.**

`line = json.dumps(record) + "\n"` — json.dumps will escape all characters including
newlines within string values, so injecting a newline into event_type or payload values
will NOT break the JSONL format. The newline delimiter is only added by this code.
No injection vector in normal use.

**Caveat:** The payload dict is caller-provided. If a caller passes a very large dict,
json.dumps will produce a very large line. With MAX_FILE_BYTES = 10MB and MAX_ENTRIES
= 10,000, a single 10MB payload would immediately trigger rotation and consume the
entire allowed space in one entry. No per-entry size limit exists.

**FIX (P3):** Add per-entry size cap (e.g., 64KB). Truncate or reject oversized payloads.

### File permissions on fallback directory

**Assessment: P2 — directory permissions not enforced.**

`self.fallback_dir.mkdir(parents=True, exist_ok=True)` — no mode argument. The
directory is created with the process umask applied. If the process umask is 0022,
the directory is 0755 — world-readable. Audit fallback files contain timestamped
event records which may leak operational metadata to other local users.

**FIX:** Create directory with mode 0o700:
  self.fallback_dir.mkdir(parents=True, exist_ok=True, mode=0o700)
Also call self.fallback_dir.chmod(0o700) after mkdir in case the directory
already existed with wrong permissions.

### flush_fallback_to_service() uses read_text() without size limit

**Assessment: P2 — OOM vector.**

`lines = path.read_text().strip().split("\n")` — reads the entire file into
memory at once. With MAX_FILE_BYTES = 10MB per file × 2 files = 20MB, this is
bounded but still allocates a 20MB string. In a low-memory environment (HP
EliteBook x360 under thermal pressure) this may OOM. Also, if an OS bug causes
a fallback file to exceed MAX_FILE_BYTES (e.g., due to a TOCTOU race in rotate),
this reads an unbounded file.

**FIX:** Use line-by-line iteration rather than read_text().

## 4. Thermal Guardian (thermal_guardian.py)

### Command injection via nvidia-smi subprocess

**Assessment: NO INJECTION RISK — but unsafe exception handling.**

`asyncio.create_subprocess_exec("nvidia-smi", "--query-gpu=temperature.gpu", ...)` —
create_subprocess_exec does NOT invoke a shell. Arguments are passed as a list.
No command injection is possible regardless of the content of device_id because
device_id is never passed to the subprocess. CLEAN.

**BUT:** The entire except clause falls through to a random simulation:
```python
except Exception:
    import random
    return 65.0 + random.uniform(-5, 15), 55.0 + random.uniform(-5, 10)
```
This catches ALL exceptions including asyncio.CancelledError (which should never
be swallowed), MemoryError, KeyboardInterrupt (Python 3: not Exception, but
BaseException, so this is OK), and legitimate nvidia-smi failures. More importantly,
it silently substitutes FAKE TEMPERATURE DATA. If nvidia-smi crashes due to a
driver compromise or hardware error, the thermal guardian will happily report
"65°C +/- 5" and the system will continue COMPUTE phase when it should COOLDOWN.
This is a reliability and safety failure mode.

**FIX:** Catch only specific exceptions (FileNotFoundError, asyncio.TimeoutError,
ValueError). Propagate asyncio.CancelledError. On error return 0.0 and set an
explicit "sensor_error" flag in DeviceThermal. The scheduler should treat sensor
errors as CRITICAL temperature.

### /sys/class/thermal/ path traversal

**Assessment: LOW — no traversal possible, but symlink risk.**

`Path("/sys/class/thermal").glob("thermal_zone*/temp")` — glob on sysfs. The
"thermal_zone*" pattern can match symlinks in /sys/class/thermal. Sysfs symlinks
are kernel-managed and cannot be attacker-controlled in a standard deployment.
No traversal risk in practice.

**Observation:** `cpu_paths[0].read_text().strip()` reads the first thermal zone
alphabetically. thermal_zone0 may not be the CPU — it could be a battery sensor,
chassis sensor, etc. The resulting "CPU temperature" may be meaningless. Not a
security issue but a reliability issue.

### STATE_PATH world-writability

**Assessment: P2.**

`STATE_PATH = Path("/run/genos/thermal-state.json")` — written via
`STATE_PATH.write_text(json.dumps(state_json))`. No file permissions set. If /run/genos
is world-writable, another process can overwrite thermal-state.json with fake data
(e.g., all temps = 0.0), causing the scheduler to never enter COOLDOWN. This is a
local privilege escalation of the scheduling policy.

**FIX:** Create /run/genos with mode 0o750 (root:genos-daemon). Write state file
with explicit mode: use os.open() with O_CREAT|O_WRONLY|O_TRUNC and mode 0o640,
or atomically write via a tempfile + rename.

## 5. XISC Opcode Space — Privilege Escalation Analysis

### Opcode inventory (from xisc/include/xisc.h)

0x00 NOP
0x01-0x05 ADD/SUB/MUL/DIV/MOD — arithmetic, no kernel crossing
0x10 LOAD, 0x11 STORE, 0x12 MOVE, 0x13 PREFETCH — memory with tier hints
0x20-0x26 JMP/JEQ/JNE/JLT/JGE/CALL/RET — control flow
0x30 CAP_DERIVE, 0x31 CAP_INVOKE, 0x32 CAP_REVOKE — capability ops
0x40 COMMIT, 0x41 VERIFY — Merkle
0x50 ANNOTATE — metadata
0x60 MATMUL, 0x61 MOELOAD, 0x62 ALLREDUCE, 0x63 ALLGATHER — AI/HPC
0xFF HALT

### Privilege escalation risk by opcode

**XISC_CAP_DERIVE (0x30) — MEDIUM RISK (P2)**
The JIT (xisc_llvm_jit.c) does NOT implement CAP_DERIVE, CAP_INVOKE, or CAP_REVOKE
in its opcode switch. They fall through to `default: pc++`. This means capability
opcodes in XISC bytecode are SILENTLY IGNORED by the JIT, not forwarded to xkabi_cap_derive().
This has a dual implication:
- An attacker who injects XISC bytecode cannot escalate via CAP_DERIVE/INVOKE
  (the opcodes do nothing).
- A legitimate program that emits CAP_DERIVE opcodes will silently fail (ops ignored).
There is NO path from XISC bytecode to xkabi_cap_derive() in the current JIT.
If future work wires these up, the JIT must validate:
  1. That the bytecode program has been verified against its SHA-256 checksum (xisc_program_t.checksum).
  2. That the capability operation is authorized by the process's handle table.

**XISC_CAP_INVOKE (0x31) — FUTURE HIGH RISK**
If CAP_INVOKE is ever implemented in the JIT to call arbitrary kernel functions via
a capability handle, this becomes the primary escalation vector. It must require:
  - XKABI handle validation (xkabi_cap_validate) before dispatch.
  - A strict allowlist of invokable kernel functions.
  - Never allow indirect dispatch through a function pointer derived from JIT-generated code.

**XISC_ALLREDUCE (0x62) — LOW/MEDIUM RISK**
The policy annotation for ALLREDUCE sets `attestation_required = 1` and
`merkle_commit_point = 1`. These hints flow to vDRAM via vdram_jit_hint_update().
The attestation_required field is advisory — whether the vDRAM fault handler
actually enforces it depends on the pf_handler.c implementation (not audited here).
If the fault handler treats attestation_required as informational rather than
mandatory, an ALLREDUCE over T3 (remote RDMA) could proceed without SPDM attestation.
This is a data integrity risk, not a local privilege escalation.

**XISC bytecode checksum — NOT VERIFIED by JIT**
xisc_program_t has a SHA-256 checksum field, but xisc_llvm_jit.c's jit_compile_block()
receives raw bytecode (const u8*, u32 len) — the xisc_program_t structure is not
passed. There is no checksum verification in the JIT path. An adversary who can
supply a modified bytecode buffer to xisc_jit_get_fn() will have it compiled
and executed without integrity verification.

**FIX:** The public entry point xisc_jit_get_fn() should accept a xisc_program_t*,
verify the magic (0x58495343), verify the checksum (SHA-256 of instrs), and then
compile. Unsigned/untampered-with bytecode must be rejected.

**GAP count: 0x64-0xFE are undefined.**
Undefined opcodes are consumed by `default: pc++` in the JIT. They are silently
skipped. This is safe for the JIT but means a crafted bytecode could use opcodes
in the reserved range to pad to specific addresses or alignment for code-reuse
attacks if the JIT execution model ever changes.

## P1 Findings Requiring Immediate Action

### K-04: UAF in vdram_mr_free()

The RCU reader increments mr->ref inside rcu_read_lock. The writer calls kfree(mr)
after spin_unlock without waiting for a grace period. Fix:

```c
// In vdram_mr_free():
spin_lock(&rkey_table_lock);
RCU_INIT_POINTER(rkey_table[mr->rkey % MR_MAX_PER_NODE], NULL);
spin_unlock(&rkey_table_lock);
synchronize_rcu();   // ADD THIS — wait for all RCU readers to exit
// Now safe to free
vfree(mr->pages);
kfree(mr);
```
