### 4b. CRITICAL UNRESOLVED FINDING — VPC Firewall

This means port 8000 on the vLLM VM is accessible from any IP address.
The VPC connector provides egress-only protection for Cloud Run.
The VM itself has no ingress restriction.

Risk: Model theft, inference abuse, prompt injection from external actors.
Required fix (before M2 deploy):
  gcloud compute firewall-rules update allow-vllm \
    --source-ranges=10.8.0.0/28,10.138.0.0/20 \
    --project=elson-33a95

10.8.0.0/28 = VPC connector range (Cloud Run egress)
10.138.0.0/20 = us-west1 subnet (VM-to-VM within region)

Once firewall is restricted to 10.8.0.0/28, no external party can intercept this traffic.

## 5. CIRCUIT BREAKER SECURITY BASELINE

### 5a. Persistence Mechanism (CRITICAL EPHEMERAL GAP)

File: `backend/app/trading_engine/engine/circuit_breaker.py`
Line 160: `self.status_file = "circuit_breaker_status.json"`
Line 177: `if os.path.exists(self.status_file):`
Line 187: `json.dump(self.circuit_breakers, f, indent=2)`

The status file is written to the CONTAINER FILESYSTEM.
Cloud Run containers are EPHEMERAL — every restart, scale-up, or new revision
creates a fresh container. The circuit_breaker_status.json file is LOST.

Consequence: A circuit breaker tripped due to extreme losses or API failure
will be silently RESET on the next container restart. The bot will resume
trading as if no breach occurred.

For the autonomous trading milestones, this MUST be remediated before M1 live deploy:
- Option A: Store circuit breaker state in Cloud SQL (dedicated table)
- Option B: Store in Redis (faster, but Redis is not yet active)
- Option C: Store in GCS with strong consistency guarantees

Option A (Cloud SQL) is recommended — already available, persistent, auditable.

### 5b. Kill Switch Status

No dedicated kill switch endpoint exists in the current codebase.
The only halt mechanism is POST /auto-trading/stop, which:
- Requires a valid JWT
- Requires the user to be logged in
- Requires the API to be reachable

If the API is under load or the user cannot log in, there is NO out-of-band halt.
KILL_SWITCH_OVERRIDE secret is identified as required but NOT YET CREATED.

Required before M1 live deploy:
1. Create KILL_SWITCH_OVERRIDE secret in Secret Manager
2. Build POST /admin/kill-switch endpoint authenticated by KILL_SWITCH_OVERRIDE header
3. Wire secret in --update-secrets
4. Document and test the halt-and-liquidate path

## 6. DATABASE ACCESS PATTERNS

### 6a. Connection Establishment (base.py)

Retry logic: 5 attempts, exponential backoff (2s, 4s, 8s, 16s = 30s total wait).
Production behavior: CRASH after all retries (no silent SQLite fallback when ENVIRONMENT=production).
Non-production behavior: Falls back to in-memory SQLite (acceptable for dev/test).
pool_pre_ping=True: Verifies connections before use. Prevents stale connection errors.
pool_recycle=3600: Recycles connections after 1 hour. Prevents long-lived idle connection issues.

### 6b. Alembic in Docker

STATUS: CONFIRMED PRESENT.
Dockerfile line 61-62: COPY backend/alembic ./alembic / COPY backend/alembic.ini ./alembic.ini
entrypoint.sh line 7-8: `python -m alembic upgrade head || echo "[alembic] WARNING: ..."`

CRITICAL CONCERN: The entrypoint.sh uses `|| echo` which means Alembic failures are
SWALLOWED and gunicorn starts anyway. If Alembic fails, the app starts with a stale schema.
This is intentional (prevents startup blocking on migration failures) but creates a silent
schema drift risk.

Recommendation: Change to a 2-phase deploy pattern:
1. Run Alembic migration as a separate Cloud Build step BEFORE deploying the new image.
2. Keep the entrypoint fallback as a safety net only.

Current Alembic head (last known): f18101facea7 (2026-02-21)

### 6c. Schema Drift Gate

CURRENT GATE STATUS: Models directory clean as of 2026-02-25.
Command to run before every deploy:
  git diff HEAD~5 -- backend/app/models/ | grep "Column("

Every new model change in the 24-week plan MUST be accompanied by an Alembic migration
and the migration MUST be run on Cloud SQL BEFORE the new image is deployed.
