## NEW FINDINGS (This Audit)

### P1 — OPEN: rkey rotation missing synchronize_rcu() on old slot
- File: kernel/linux-validation/vdram/mr_registry.c, vdram_rkey_rotate()
- Lines 153-155: RCU_INIT_POINTER clears old_rkey slot, immediately assigns new slot.
  There is NO synchronize_rcu() between the two assignments.
  A concurrent reader that entered rcu_read_lock() BEFORE the RCU_INIT_POINTER can
  dereference the old slot AFTER it is NULL-assigned if the reader is preempted at
  exactly the wrong moment. This is a classic RCU UAF on rotation.
  In vdram_mr_free() synchronize_rcu() IS called correctly.
  In vdram_rkey_rotate() it is missing.
- FIX: Add synchronize_rcu() after RCU_INIT_POINTER(rkey_table[old_rkey % MR_MAX_PER_NODE], NULL)
  BEFORE rcu_assign_pointer for new slot.
  Or restructure using call_rcu() callback to defer kfree-equivalent work.

### P1 — OPEN: gate_service / registry / update / artifact-store soft-fail on missing GENOS_SERVICE_TOKEN
- Files: platform/services/gate_service_app.py:13, registry_app.py:13,
  update/app.py:14, artifact_store_app.py:14
- Pattern: `os.environ.get("GENOS_SERVICE_TOKEN", "")` — soft fail, service STARTS with empty token.
  _verify_token() does check `not _SERVICE_TOKEN` before compare, so requests get 401.
  But the service starts silently — inconsistent with Phase 7 hardening pattern.
  If a pod starts before the Secret is mounted, it accepts no tokens but also logs no startup error.
  An operator may not notice the silent degradation.
- FIX: Replace with `os.environ["GENOS_SERVICE_TOKEN"]` (hard-fail at import) in all four services.
  Matches the pattern in sync/app.py, identity/app.py, provenance/app.py, api.py.

### P1 — OPEN: Thermal sensor failure masking — cr_read_temp_mc() returns 0°C on error
- File: kernel/linux-validation/scheduler/contra_rotation.c, cr_read_temp_mc()
- Line 87: thermal_zone_get_zone_by_name(NULL) — passing NULL gets UNSPECIFIED zone.
  Line 90: if (!IS_ERR(tz)) ... else temp_mc stays 0.
  0 milli-Celsius = 0°C, which is BELOW all cooling thresholds (THERMAL_CEILING_LOW_C=60,
  HIGH_C=80). A sensor failure causes every device to stay in CR_PHASE_COMPUTE indefinitely,
  defeating the thermal safety backstop. If actual temp exceeds safe ceiling while sensor
  reports 0°C, thermal throttle is never triggered. Risk: SoC damage on HP EliteBook x360.
- FIX: Return a SENTINEL value (e.g., -1 or INT32_MIN) on sensor error.
  In cr_schedule_consumer(): if temp_mc < 0 (sensor error), treat as THERMAL_CEILING_HIGH_C
  (worst-case safe assumption) and force transition to COOLDOWN.

### P1 — OPEN: genesys-ai pod still uses hostNetwork: true
- File: platform/k8s/09_genesys-ai.yaml:20
- hostNetwork: true is present with TODO comment acknowledging it.
  This gives the genesys-ai container access to the full host network stack, bypassing
  the NetworkPolicy isolation (10_network-policy.yaml only applies to pod-level traffic).
  A compromised genesys-ai pod has direct loopback access to ALL host services.
  PREVIOUS AUDIT (2026-02-25) marked this as RESOLVED — it has REGRESSED.
  OLLAMA_URL is still "http://127.0.0.1:11434" requiring hostNetwork.
  The TODO has not been actioned.
- FIX: Create a separate Ollama Deployment + Service in genos-system namespace.
  Change OLLAMA_URL to "http://ollama:11434". Remove hostNetwork: true.
  Blast radius of current state: full host network exposure from AI inference pod.

### P2 — OPEN: fabric-energy liveness probe reads state file from writable hostPath
- File: platform/k8s/11_fabric-energy.yaml:50-51
- Liveness probe: `python3 -c "import json; json.load(open('/run/genos/fabric-energy-state.json'))"`
- /run/genos is a hostPath (type: DirectoryOrCreate). Any process with host access
  can write a valid JSON file there, causing the liveness probe to PASS even if the
  fabric-energy controller has crashed or been replaced.
  State spoofing: attacker writes a crafted JSON file to /run/genos/, probe passes,
  Kubernetes never restarts the pod despite controller death.
- FIX: Add a signed/HMAC'd field to the state JSON that the pod generates with a secret key.
  Probe should verify the signature, not just JSON parsability.
  OR: Replace file-based probe with an httpGet probe to a /healthz endpoint.

### P2 — OPEN: RDMA session key zeroization gap — spdm_challenge_auth does not zeroize digest
- File: kernel/linux-validation/security/rdma_attestation.c, spdm_challenge_auth()
- Line 133: `u8 digest[32]` computed from cert chain.
  memzero_explicit(session_key) IS called in the out_disconnect path (line 260).
  BUT digest (line 133) is a stack variable in spdm_challenge_auth() and is NOT explicitly
  zeroed before the function returns. digest contains the SHA-256 of the full cert chain
  and is used as KDF input material — it is cryptographic key material.
  On stack frames in kernel context, this data can be read by subsequent callers of the
  same stack frame if it is not explicitly zeroed (compiler optimization warning).
- FIX: Add memzero_explicit(digest, sizeof(digest)) before return in spdm_challenge_auth().

### P2 — OPEN: XISC linux_loader.c has no ELF bytecode checksum/signature verification
- File: xisc/runtime/personality_linux/linux_loader.c
- validate_elf_header() checks magic + type only. There is no SHA-256 or signature
  verification of the ELF image before loading. A tampered ELF binary can be loaded
  without detection.
- Note: Most load_segment() and linux_load_elf() internals are TODO stubs, so this
  is a pre-production gap rather than an immediate live vulnerability.
  The opcode space privilege audit is also incomplete (xk_object_signal / xk_object_get_info
  both return XK_ERR_NOT_SUPPORTED without performing the operation).
- FIX: Before load_segment(), compute SHA-256 of the full ELF file and verify against a
  trusted manifest/signature before any VMO creation.

### P2 — OPEN: Identity service has no brute-force login lockout
- File: platform/services/identity/app.py, login() endpoint
- No failed attempt counter, no lockout after N failures, no CAPTCHA trigger.
  The login endpoint returns 401 with a constant 2ms response time (bcrypt cost).
  A determined attacker can enumerate usernames (409 on /users vs 401 on /auth/login)
  and then brute-force passwords via slow bcrypt timing.
  Rate limiting exists on the AI endpoints but NOT on /auth/login.
- FIX: Add per-IP and per-username failed attempt counter (Redis or in-memory with TTL).
  Lock out after 5 failures for 15 minutes. Log all failed attempts with client IP.

### P3 — OPEN: CI/CD pipeline missing SAST (Bandit/Semgrep) and CVE scan (Trivy/pip-audit)
- File: .github/workflows/ci.yml
- Full pipeline runs: ruff lint, pytest, cppcheck, TypeScript build, XKABI conformance.
  Missing: bandit -r platform/ (SAST), semgrep --config=p/owasp-top-ten, trivy fs,
  pip-audit, npm audit --audit-level=high.
  The Iron Gate requires all these gates. None are present.
- FIX: Add ci-security job after policy-and-tests with all four scanners.
  Block merge on CRITICAL CVE or HIGH SAST finding.

### P3 — OPEN: ISO build produces sha256 but has no GPG signing step
- Files: build/genos-1.0-amd64.iso.sha256 exists, but CI has no gpg --detach-sign step.
  The Sentinel standard requires: sha256sum + gpg --detach-sign --armor SHA256SUMS.
  COMPLIANCE_LOG.md shows "SHA256: pending build" — checksum automation unclear.
- FIX: Add ISO signing job to CI after build completes. Artifact: SHA256SUMS + SHA256SUMS.asc.
  Verify with gpg --verify before any distribution step.

### P3 — OPEN: xkabi_cap_validate() reads cap->generation after releasing ht->lock
- File: kernel/linux-validation/security/xkabi_capabilities.c, xkabi_cap_validate()
- Lines 217-221: ht->lock acquired, cap pointer retrieved, lock released.
- Line 228: cap->generation compared to current_epoch WITHOUT holding cap->lock or ht->lock.
  cap->lock is a separate per-cap spinlock. After ht->lock is released at line 221,
  another thread could revoke cap (cap->revoked = true under cap->lock) or even free it
  (idr_remove + kfree) before line 228 reads cap->generation. This is a TOCTOU / UAF.
- Note: This is subtle — the object_id match check at line 220 provides PARTIAL mitigation
  but does not prevent the generation read at line 228 from racing with kfree.
- FIX: Acquire cap->lock before releasing ht->lock, read generation under cap->lock.

## CHAOS ENGINEERING ASSESSMENT

### Provenance service down
- genesys-ai /provenance endpoint: returns stub {"status": "use /tools/execute with explain.recent"}
  with verify_service_token guard — PASSES auth, returns static response. No crash.
- Platform init health check loop polls http://provenance:8000/healthz with timeout.
  If provenance is down, init retries (max 10, 2s delay) then raises RuntimeError. GOOD.
- Orange Suite apps that call /provenance fallback to local IPC — no UI block. ACCEPTABLE.

### Thermal sensor failure
- CRITICAL FAILURE MODE (see N-03 above). Sensor failure returns 0°C.
  Scheduler never transitions to COOLDOWN. SoC can overheat without thermal protection.

### RCU grace period delayed
- vdram_mr_free(): synchronize_rcu() blocks until all readers exit. If a reader holds
  rcu_read_lock() for an extended time (e.g., page fault handler under memory pressure),
  vdram_mr_free() stalls. This is expected RCU semantics — not a bug but an availability risk.
  High latency migration (T3 RDMA path with 5s poll timeout) while holding rcu_read_lock
  could extend this. Mitigation: use call_rcu() for deferred free instead of synchronize_rcu().
  vdram_rkey_rotate() has NO synchronize_rcu() — see N-01.
