## 3. Root Cause Analysis — Ranked by Impact

### BOTTLENECK 1 (P0): /market-enhanced/quotes — 18s AUTH TIMEOUT ON 401 (Most Critical)
**Blast radius:** ALL users — this endpoint saturates the single CPU vCPU, blocking real user requests.

**Root cause chain:**
- The endpoint fires every ~30 seconds from an automated caller (frontend RTK polling at very short
  intervals OR an external scheduler/browser tab open without a valid session token).
- The request has NO valid Authorization header — it returns HTTP 401.
- BUT: the server still takes **16-32 seconds** to return the 401. This is the anomaly.
- On a 401, FastAPI's `get_current_active_user` should fail immediately (token validation is in-memory).
  A 16s delay to return a 401 means something BEFORE the auth check is timing out.
- Most likely: the Cloud SQL Auth Proxy connection attempt or the `get_db` Depends() call is
  blocking, waiting for a DB connection to be available from the pool when the pool is exhausted
  by concurrent requests that are all also waiting for vLLM or provider timeouts.
- Alternatively: the vLLM connection attempt fires even on auth-failed paths if middleware order
  is wrong.
- **Second possibility:** The market_streaming.py `_fetch_and_broadcast_quotes()` was previously
  sending internal HTTP requests to `/api/v1/market-enhanced/quotes`. The 2026-02-27 memory note
  says this was "fixed" but the fix is a code comment, not a verified deployment. If the running
  container predates the fix OR the fix was never deployed, the streaming loop IS the source of these
  401s. The request pair pattern (two entries ~0.1s apart every ~55-60s) strongly suggests a
  background polling loop firing twice concurrently.

**Evidence from logs:**
- Pairs of identical requests ~0.1-0.4s apart: `15:42:52` and `15:42:47` (gap=5s, different pair)
- Consistent ~55-second inter-pair gap aligns with a background loop sleeping 5s + 50s = 55s
- Only 5 symbols match exactly the `market_streaming.py` default test set: `AAPL,DIS,ECL,GOOGL,MSFT`

**Fix path:**
1. Confirm which revision is live: `gcloud run services describe elson-backend --format='value(status.traffic)'`
2. If revision predates the market_streaming fix: deploy immediately with the fix.
3. SEPARATELY: the 16-32s latency on a 401 is unacceptable regardless of caller. Auth middleware
   must return 401 immediately without waiting for DB pool or vLLM. Investigate if `get_db` is
   inside the auth dependency for this route.

**Short-term mitigation:** Remove the `/api/v1/market-enhanced/quotes` endpoint from being polled
by any background task. Frontend should call it only when a user is authenticated and viewing the
market page (user-triggered, not on a timer).

### BOTTLENECK 2 (P0): EFT Sequential For-Loop — get_trading_signals
**File:** `backend/app/api/api_v1/endpoints/ai_trading.py` lines 122-130
**Blast radius:** Any user clicking "Trading Signals" — single endpoint, but hangs the entire page.

**Root cause:**
```python
# BEFORE (BROKEN — sequential, each awaits separately):
for signal in signals:
    signal = await eft_enhance_response("trading_signals", signal, ...)
    enhanced_signals.append(signal)
```

With 10 symbols and vLLM stopped (weekend):
- Each call hits the 15s primary timeout → falls through to cloud_llm_fallback (port 8001) → another 30s timeout → returns None
- Total: 10 × 45s = **450 seconds** (7.5 minutes) — far exceeds the 300s Cloud Run timeout
- Result: HTTP 504 Gateway Timeout

With vLLM running:
- Each call takes ~3-5s for inference
- Total: 10 × 4s = **40 seconds** — still very slow UX

**Fix applied in this session:**
```python
# AFTER (FIXED — concurrent via asyncio.gather):
user_ctx = build_user_context(current_user)
enhanced_signals = list(
    await asyncio.gather(
        *[
            eft_enhance_response("trading_signals", signal,
                portfolio_context={"symbol": signal.get("symbol", "")},
                user_context=user_ctx)
            for signal in signals
        ]
    )
)
```
With gather + semaphore(4): 10 calls complete in ceil(10/4) × 4s = **~8 seconds** instead of 40s.

### BOTTLENECK 3 (P0): No EFT Circuit Breaker — vLLM VM Stopped on Weekends
**File:** `backend/app/services/eft_enhance.py`
**Blast radius:** ALL AI-enhanced endpoints during off-hours / weekends.

**Root cause:**
- vLLM VM (`elson-dvora-training-l4-2`) is stopped on weekends to save cost.
- When stopped, EVERY `eft_enhance_response()` call:
  1. Waits 15s for primary vLLM timeout
  2. Then calls `cloud_llm_fallback` (port 8001 on same stopped VM) → waits 30s
  3. Returns None
  - **Total wasted time per call: 45 seconds**
- With no circuit breaker, every user page load on the weekend makes N × 45s of wasted network calls.

**Fix applied in this session:** Module-level circuit breaker in `eft_enhance.py`:
- Opens after 2 consecutive failures
- Fast-fails for 5 minutes (returns None immediately)
- Resets on first success after cool-off
- This reduces weekend page load from "minutes of hanging" to <100ms EFT overhead.

### BOTTLENECK 4 (P1): get_multiple_quotes — batch_concurrency=1
**File:** `backend/app/services/enhanced_market_data.py` — `EnhancedMarketDataService.__init__`

**Root cause:** `batch_concurrency=1` means a semaphore(1) is used inside `get_multiple_quotes`.
Even though `asyncio.gather` is used for the batch tasks, the semaphore ensures only ONE quote
fetches at a time. For 5 symbols:
```
Sequential: Symbol1 → Symbol2 → Symbol3 → Symbol4 → Symbol5
Time: 5 × (provider_latency + 0.15s delay) = 5 × 1.0s = ~5s on cache miss
```

**Fix applied in this session:** `batch_concurrency=3`, `inter_quote_delay=0.05s`:
```
Concurrent: [Symbol1, Symbol2, Symbol3] → [Symbol4, Symbol5]
Time: 2 × (1.0s + 0.05s) = ~2.1s on cache miss
```
Saves ~3s per batch data load. Alpaca (primary provider) allows 200 calls/min — 3-concurrent is safe.

### BOTTLENECK 5 (P1): MarketDataCache Ignores MARKET_DATA_CACHE_TTL Env Var
**File:** `backend/app/services/enhanced_market_data.py` — `MarketDataCache.__init__`

**Root cause:**
- Cloud Run env var `MARKET_DATA_CACHE_TTL=15` is set (verified in service config).
- `MarketDataCache._ttl_map["quote"]` was hardcoded to `60`.
- The `EnhancedMarketDataService` never passed the env var to the cache.
- Result: cache is effectively 60s (4× longer than intended), causing MORE provider calls on refresh
  cycles while simultaneously being too long for real-time trading accuracy.

**Fix applied in this session:** Cache reads `settings.MARKET_DATA_CACHE_TTL` at `__init__` time.

### BOTTLENECK 6 (P2): /api/v1/sentinel/scan — 16-20s Synchronous Scan
**File:** `backend/app/api/api_v1/endpoints/sentinel.py`

**Root cause:**
- `run_sentinel_scan` is a synchronous `def` (not `async def`).
- `SentinelService.run_scan()` iterates all portfolios in a Python for-loop with no concurrency.
- Each portfolio scan runs synchronous SQLAlchemy queries.
- With Cloud Scheduler calling this every 15 minutes via the `clnzl6xzga-uw.a.run.app` URL,
  each scan takes 16-20s, tying up a container worker for that duration.
- The Cloud Scheduler URL bypasses the auth secret properly (HTTP 200 returned).

**Recommended fix (not applied — scope too large for this session):**
- Move `SentinelService.run_scan()` to a Cloud Run Job (not an HTTP endpoint) for isolation.
- OR: add `asyncio.gather` for parallel portfolio scanning.
- Short-term: add database index on `Portfolio.owner_id` if missing.

### BOTTLENECK 7 (P2): Provider rate_limit_delay=12s on AlphaVantage and Polygon
**File:** `backend/app/services/enhanced_market_data.py`

**Root cause:**
- `AlphaVantageProviderEnhanced.rate_limit_delay = 12.0` (5 calls/min free tier)
- `PolygonProviderEnhanced.rate_limit_delay = 12.0` (5 calls/min free tier)
- These are fallback providers (4th and 5th in priority list).
- But with `batch_concurrency=3` (post-fix), if Alpaca AND YFinance AND Finnhub all fail for a
  symbol, the 3rd concurrent slot will hit AlphaVantage and wait in the 12s rate_limit_delay lock.
- This can stall a batch slot for 12s on a bad Alpaca/YFinance day.

**Recommended fix:** Add a per-provider timeout cap of 5s in `get_quote()` — if the rate_limit_delay
itself would exceed 5s, skip the provider and return None immediately.

### BOTTLENECK 8 (P3): Cloud Run CPU Throttling (No always-on CPU annotation)
**Root cause:**
- No `run.googleapis.com/cpu-throttling: "false"` annotation on the service.
- Cloud Run's default is to throttle CPU to near-zero between HTTP requests, even with minScale=1.
- When a new request arrives, the CPU must "wake up", adding 50-200ms latency on the first await
  of each idle period.
- For a low-traffic platform like this, most requests hit an idle (throttled) state.

**Recommended fix:**
```yaml
run.googleapis.com/cpu-throttling: "false"
```
Or via gcloud:
```bash
gcloud run services update elson-backend \
  --no-cpu-throttling \
  --region=us-west1 \
  --project=elson-33a95
```
Note: This increases cost slightly (CPU billed even between requests at minScale=1).

### BOTTLENECK 9 (P3): aiohttp.ClientSession Created Per-Request
**File:** Multiple provider classes in `enhanced_market_data.py`

**Root cause:**
Every provider creates a new `aiohttp.ClientSession()` for every quote call:
```python
async with aiohttp.ClientSession() as session:
    async with session.get(url, ...) as response:
```
Each `ClientSession()` creation involves:
- A new TCP connection (no connection reuse)
- TLS handshake for HTTPS endpoints (~50-100ms per call)
- DNS resolution (unless cached by OS)

With 5 symbols × 1 provider call each = 5 new TLS handshakes per batch load.

**Recommended fix:** Use a module-level persistent `aiohttp.ClientSession` per provider, created once
at `__init__` and reused. Requires lifecycle management (close on app shutdown).

## 4. MARKET_DATA_CACHE_TTL — Cache Hit Rate Analysis

**Current TTL config (post-fix):**
- quote: `settings.MARKET_DATA_CACHE_TTL` = 15 seconds (market hours)
- historical: 3600 seconds (1 hour)
- news: 1800 seconds (30 minutes)

**Streaming poll interval:** `PersonalMarketStreaming.update_interval = 5.0` seconds.

**Cache efficiency calculation:**
With TTL=15s and poll_interval=5s:
- Polls 0, 5, 10 → all within 15s window → 2 cache hits after first miss
- Poll 15s → cache miss, new provider call
- Effective cache hit rate: 66% (2 hits per 3 polls)

**With old TTL=60s and poll_interval=5s:**
- 11 cache hits per 12 polls → 92% cache hit rate
- BUT: 60s stale data is too long for a trading platform during market hours

**Recommendation:** Leave MARKET_DATA_CACHE_TTL=15s for market hours. The 66% hit rate is correct
and intentional. The `MARKET_DATA_CACHE_TTL_AFTERHOURS=60` and `MARKET_DATA_CACHE_TTL_CLOSED=300`
env vars exist but are NOT currently read by the cache — they could be wired in for further
optimization when market is closed.
