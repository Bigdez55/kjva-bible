## AUDIT 4: MEMORY SAFETY — FINDINGS

### btree.c — P2 ADVISORY
- _leaf_insert_at (line 336): `xstore_memcpy(rec, key, key_len)` — rec is `page + XSTORE_DATA_OFF + rec_end`. rec_end is the sum of all (key_len + val_len) for existing slots, computed without overflow check. If adversarial key/val combo causes rec_end + key_len + val_len > XSTORE_PAGE_DATA_SIZE, write overflows the page buffer.
  - MITIGATING FACTOR: _leaf_free_space() is called at line 571 BEFORE insert_at, and checks `total_used < XSTORE_PAGE_DATA_SIZE`. If check passes, insert_at is safe. The guard is present. ADVISORY ONLY.
- _leaf_split line 400: `rs->rec_off = (uint16_t)right_rec_off` — right_rec_off can exceed uint16_t max (65535) if large records accumulate. Since max record = 256+4096 = 4352 bytes, and XSTORE_PAGE_DATA_SIZE = 4064, the maximum right_rec_off per page is bounded by 4064. No overflow. PASS.
- Binary search in _leaf_find_slot: bounds (lo, hi = hdr->num_keys) — if num_keys is corrupted (hdr corrupt), infinite loop risk. Mitigated by page CRC check on read. ACCEPTABLE.

### mvcc.c — PASS
- mvcc_record_write: linear probe bounded by MVCC_PAGE_TABLE_SIZE (8192) — bounded loop. PASS
- mvcc_query_write: same. PASS
- xstore_txn_abort: memset bounded to txn->write_count * sizeof(entry) — write_count guarded by XSTORE_TXN_MAX_WRITES at insert time. PASS

### wal.c — P1 ADVISORY
- xstore_wal_replay Phase 2 (line 147): constructs rec_txn and calls xstore_btree_insert(). The val field used is `e->val` with `e->val_len`. In the WAL entry definition, val bytes are stored in entry but the WAL commit path (mvcc.c line 204) sets `entry.val_len = 0` and does NOT write val bytes. Therefore during replay, val.data = e->val (possibly zero) and val.len = 0. xstore_btree_insert with val->len=0 will attempt to overwrite a key with a zero-length value. This is a replay semantic gap: INSERTs replayed from WAL will have empty values, corrupting the B+ tree.
  - This is a documented Sprint 4 limitation: "val is already applied to B+ tree pages". Replay is "best-effort". HOWEVER: if the WAL replay path is ever exercised in production (crash recovery), this will silently corrupt values. Document as P1 reliability gap for Sprint 5.
- WAL ring slot assignment (line 73): `lsn % XSTORE_WAL_RING_SIZE` — XSTORE_WAL_RING_SIZE = 65536 (power of 2). The check at line 66 `(wal_head - wal_tail) >= XSTORE_WAL_RING_SIZE` prevents overflow before assignment. PASS.

### transformer.c — P2 ADVISORY
- scores[XM_ATTN_SCRATCH] stack array (line 119): XM_ATTN_SCRATCH = XMIND_MAX_SEQ. If XMIND_MAX_SEQ is large (e.g., 2048), scores[] = 8KB on stack per head call. Called once per head per layer: 28 layers * 32 heads = 896 calls but not nested. Stack frame per call is bounded. Acceptable if XMIND_MAX_SEQ <= 2048. ADVISORY.
- xm_kv_store: `k_dst[i] = k_src[i]` loop — kv_stride = n_kv_heads * head_dim. Both src (m->state.k) and dst (m->kv.k[layer]) must be allocated to kv_stride floats minimum. xmind_alloc_state() is responsible. No bounds check here — relies on caller. ACCEPTABLE per design.
- xmind_forward: XM_ASSERT(token < m->cfg.vocab_size) — guarded. PASS.
- head_out[XM_MAX_HEAD_DIM] = 256 floats (1KB). m->state.attn write at `h * head_dim + d` — if head_dim > XM_MAX_HEAD_DIM (256), write exceeds head_out bounds. If Llama 3.2 3B: head_dim = 96 < 256. SAFE for target model. ADVISORY for future models.

### weights_loader.c — P2 ADVISORY
- gguf_scan_metadata: s_scan_buf is STATIC (line 232). Not stack-allocated. Zero stack risk. But it is a global — not re-entrant. If called concurrently (not expected in freestanding single-thread context). ADVISORY.
- wl_alloc_pages: NULL check on return: `if pal_pages_alloc fails return (void*)0`. Callers at lines 504-527 use the returned pointer without NULL check before writing `m->wq[l] =`. If alloc fails, NULL pointer stored but not dereferenced in this function. xmind_forward would dereference. P2 reliability gap.
- Documented known gap (line 611): "memory allocated by wl_alloc_pages() is leaked in Sprint 8". Sprint 9 fix committed. ACCEPTABLE.
- gguf_header_t: read from `hdr_buf + 16u` for n_kv (line 213). sizeof(gguf_header_t) = 4+4+8+8 = 24 bytes. hdr_buf is 24 bytes. Access at [16..24) is in-bounds. PASS.

## AUDIT 5: RELIABILITY GAPS

### WAL replay partial writes — ADVISORY
- wal.c: `if (e->lsn != lsn) continue` correctly skips slots from different epochs. CRC check `e->checksum != expected_crc` handles corrupted/torn entries. Partial write (entry partially written before crash) is handled: either lsn mismatch or CRC mismatch causes skip, entry is not replayed. PASS for partial write handling.
- Known gap: replayed INSERT entries will have val_len=0 (see AUDIT 4 above). P1 reliability gap documented.

### XBLOB GC compaction interruption — P2
- gc.c xblob_gc_scan: compaction at line 173 runs under store_lock. If interrupted mid-compaction (power cut), the index table is partially zeroed (line 176 marks each entry FREE before re-inserting). Recovery path: on next xblob_open(), xblob_gc_log_replay() replays GC log to restore ref_counts. However, the compaction zeroed index entries BEFORE logging them (the GC log append at line 138 is for freed blobs, not for compacted live blobs). If interrupted mid-compaction, live blob entries that were zeroed but not yet re-inserted are LOST. This is a P2 data integrity gap: compaction is not crash-safe.
- Mitigation: compaction only runs when tombstone_count > total_used/4. Conservative threshold. In Sprint 4 this is a background operation, not on the critical write path.

### XMIND corrupt/truncated weight files — P2
- weights_loader.c: gguf_verify_magic returns XMIND_ERR_CORRUPT if magic wrong or version out of range. PASS.
- gguf_scan_metadata: if file is truncated mid-metadata, the `if (pos + N > n_read) goto done` guards throughout handle graceful exit. PASS.
- If n_kv in header is adversarially large (e.g., UINT64_MAX), the loop `for (kv=0; kv < n_kv && pos+12 < n_read; kv++)` terminates when pos+12 >= n_read. Safe. PASS.
- Weight pointer NULL after alloc failure: xmind_forward would dereference NULL weight pointer. Not guarded in forward pass. P2 gap.

### XJIT emit buffer overflow — P2 ADVISORY
- codegen.c xj_emit8 (line 68-72): `if (cb->pos >= cb->capacity) { pal_panic(...) }`. Buffer overflow is caught and panics, not silently corrupted. PASS for safety. However, pal_panic in a freestanding kernel is a hard crash — no recovery path for a JIT compilation that runs out of buffer. P2 for production resilience.
- Buffer overflow check is BEFORE the write. Correct ordering. PASS.
- xj_patch_rel32 (line 456-462): accesses `cb->buf[patch_offset]` through `cb->buf[patch_offset+3]`. No bounds check on patch_offset. If patch_offset >= cb->capacity, OOB write. Patch offsets are recorded from forward-jump emission (line 686: patch_pos = f->code.pos + 1). Since f->code.pos passed emit8 checks, patch_pos is within emitted range. Safe under correct use. ADVISORY.
