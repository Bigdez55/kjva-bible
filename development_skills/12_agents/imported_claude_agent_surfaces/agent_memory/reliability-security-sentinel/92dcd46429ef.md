### SKILL 01: FIPS 180-4 SHA-256 / SHA-512 / SHA-384 — Streaming Hash
**File:** `sec/xsec/crypto/sha256.c:65-264`

**Threat Model:** Pre-image attacks, collision attacks, hash length extension.

**Implementation Pattern:**
- SHA-256 uses FIPS 180-4 §6.2.2 compress function: CH, MAJ, BSIG0, BSIG1, SSIG0, SSIG1 macros via `xsec_ror32()`.
- 64 round constants SHA256_K[] are cube-root fractions of the first 64 primes (FIPS 180-4 §4.2.2).
- Streaming API: `xsec_sha256_init` / `xsec_sha256_update` / `xsec_sha256_final`. Partial block buffer in ctx (`buf[64]`, `buf_len`). Full 64-byte blocks processed directly without intermediate copy.
- SHA-512 uses 128-bit bit count (`bit_count_hi`/`bit_count_lo`) for 2^128 byte input support.
- SHA-384 = SHA-512 with different IV constants (FIPS 180-4 §5.3.4), truncated to 48 bytes. Reuses `xsec_sha512_ctx_t`.

**Constant-Time:** SHA-256 compression has no secret-dependent branches. All bit operations are pure arithmetic. The `xsec_ct_eq()` function is used for comparing digests (not `memcmp`).

**Freestanding:** No libc. Uses `xsec_memcpy`, `xsec_memset`, `xsec_be32`, `xsec_put_be32`, `xsec_put_be64` from `pal.h`. All buffers caller-provided.

**Critical Detail — bit_count save before padding:** `saved_bits = ctx->bit_count` is captured before padding updates are fed through `xsec_sha256_update()` which increments `bit_count`. Failure to save would corrupt the final length field.

**Verification:** `xsec_sha256_selftest()` at line 207 — three NIST FIPS 180-4 KAT vectors:
- V1: SHA256("abc") = ba7816bf...15ad (Appendix B.1)
- V2: SHA256("") = e3b0c442...b855
- V3: SHA256(55×'a') = 9f4390f8...bd34

**Secure Zeroization:** `xsec_secure_zero(ctx, sizeof(*ctx))` in `xsec_sha256_final()` at line 187 — wipes all working state after output is extracted.

### SKILL 02: AES-256 Key Schedule — Constant-Time S-Box
**File:** `sec/xsec/crypto/aes_gcm.c:33-142`

**Threat Model:** Cache-timing side-channels (AES table attacks on standard `sbox[i]` lookups). An attacker sharing the CPU cache can observe which cache lines are accessed during key schedule and encryption rounds, recovering the key.

**Implementation Pattern — `volatile const` S-Box:**
```c
static volatile const uint8_t AES_SBOX[256] = { ... };
```
The `volatile` qualifier prevents the compiler from eliminating the full-table scan or reordering accesses.

**Implementation Pattern — Branchless CT S-Box Lookup (`aes_sbox_ct`):**
```c
uint32_t eq = (uint32_t)((uint8_t)i ^ idx);
eq = (eq - 1u) >> 8u;   // 0 → 0xFFFFFF; nonzero → 0
result |= (uint8_t)(AES_SBOX[i] & (uint8_t)eq);
```
Scans ALL 256 entries regardless of input. Uses arithmetic underflow to compute a mask: `eq-1` underflows to `0xFFFF...` only when `eq==0` (match), then the `>>8` shift clears the high bytes leaving `0xFF`. No branch on `i==idx`.

**GF(2^8) Multiplication (`gf_mul`):** Branchless via:
```c
b_mask = (uint8_t)(0u - (b & 1u));   // 0xFF if bit set, 0x00 otherwise
hi_mask = (uint8_t)(0u - ((a >> 7u) & 1u));
a ^= 0x1bu & hi_mask;
```
Reduction polynomial XOR is applied via mask, not branch.

**Key Schedule:** AES-256 Nk=8, Nr=14, produces 60 round key words. Every `sub_word()` call uses `aes_sbox_ct()`.

**Freestanding:** Zero libc. AES-NI hardware path selected by `#ifdef __AES__` at compile time.

**Verification:** KAT via NIST SP 800-38D test vectors in sprint45/test_crypto_kat.c.

### SKILL 03: AES-256-GCM AEAD — Buffer Overlap Guard + CT Tag Comparison
**File:** `sec/xsec/crypto/aes_gcm.c:416-558`

**Threat Model:**
1. Authentication bypass via timing oracle on tag comparison (allows adaptive chosen-ciphertext attacks).
2. Partial plaintext exposure before authentication: if decryption and tag check are separated, an attacker with oracle access can recover plaintext from partial decrypts.
3. Buffer overlap bug (S3-DI-H02): if `ct == pt`, GHASH runs on already-decrypted plaintext, producing wrong tag → spurious AUTH_FAIL.

**Implementation Pattern — Overlap Rejection:**
```c
const uint8_t *ct_end = ct + ct_len;
const uint8_t *pt_end = pt + ct_len;
if (!((pt_end <= ct) || (ct_end <= pt))) { return XSEC_ERR_INVAL; }
```

**Implementation Pattern — CT Tag Comparison:**
```c
if (!xsec_ct_eq(computed_tag, tag, 16u)) {
    xsec_secure_zero(pt, ct_len);   // zero partial plaintext before returning
    xsec_audit_log(XSEC_AUDIT_DECRYPT_FAIL, ...);
    return XSEC_ERR_AUTH_FAIL;
}
```
Tag comparison via `xsec_ct_eq()` (timing-safe). Plaintext buffer zeroed on auth failure before return.

**GHASH Implementation:** `ghash_mul()` at line 310 uses branchless GF(2^128) multiplication:
```c
uint8_t msb_mask = (uint8_t)(0u - ((xi >> 7u) & 1u));  // 0xFF or 0x00
for (k = 0; k < 16u; k++) { result[k] ^= V[k] & msb_mask; }
uint8_t lsb_mask = (uint8_t)(0u - (V[15] & 1u));
V[0] ^= 0xe1u & lsb_mask;   // reduction polynomial, branchless
```

**GCM Nonce Construction:** 96-bit nonce || 0x00000001 per NIST SP 800-38D §7.1. Counter increments rightmost 32 bits only (`gcm_inc32`).

**Zeroization After Use:** `xsec_secure_zero(&aes_key, ...)`, `xsec_secure_zero(H, ...)`, `xsec_secure_zero(S, ...)`, `xsec_secure_zero(E_J0, ...)` all called in `aes_gcm_core` before return (line 487-490).

### SKILL 04: AES-NI Hardware Acceleration — Runtime CPUID Feature Detection
**File:** `sec/xsec/tls/aes_ni.c:61-141`

**Threat Model:** Performance degradation forcing software fallback; key schedule timing exposure.

**Implementation Pattern — Runtime CPUID Gate:**
```c
bool aesni_available(void) {
    xsec_cpuid(1u, &eax, &ebx, &ecx, &edx);
    return ((ecx >> 25u) & 1u) && ((ecx >> 1u) & 1u);
}
```
Checks CPUID leaf 1, ECX bit 25 (AES-NI) AND bit 1 (PCLMULQDQ). Both required: AES-NI for cipher, PCLMULQDQ for GHASH acceleration.

**Key Schedule via `_mm_aeskeygenassist_si128`:** 10 round keys derived with hardware assist. `_aes128_keygen_assist()` uses `_mm_shuffle_epi32(kg, 0xFFu)` to splat the SubWord(RotWord) result, then cascade XOR through all four 32-bit words using `_mm_slli_si128`.

**KAT:** NIST FIPS 197 Appendix B vector hardcoded in `aesni_selftest()`.

**Freestanding:** Uses compiler-provided `<wmmintrin.h>` and `<immintrin.h>` — these are part of the compiler toolchain, not libc.

### SKILL 06: X25519 Curve25519 — Montgomery Ladder + Low-Order Point Rejection
**File:** `sec/xsec/crypto/x25519.c`

**Threat Model:**
1. Timing side-channel via data-dependent branching in scalar multiplication.
2. Small-subgroup attacks via low-order point inputs (contributes zero-output shared secret).
3. Non-canonical output (value in [p, 2p)) leaks one bit of shared secret.

**Implementation Pattern — 5-Limb 51-bit Field Representation:**
```c
typedef struct { uint64_t v[5]; } fe;
#define MASK51  ((uint64_t)0x0007ffffffffffffULL)
```
Each limb holds 51 bits. Reduction: carry propagation through 5 limbs with `v[0] += c * P19` wrap (since 2^255 ≡ 19 mod p).

**Implementation Pattern — Constant-Time Conditional Swap:**
```c
static void fe_cswap(fe *a, fe *b, uint64_t swap_bit) {
    uint64_t mask = (uint64_t)(-(int64_t)swap_bit);  // 0 or 0xffff...ffff
    for (i = 0; i < 5; i++) {
        uint64_t tmp = mask & (a->v[i] ^ b->v[i]);
        a->v[i] ^= tmp;
        b->v[i] ^= tmp;
    }
}
```
Called in Montgomery ladder at every bit of scalar without branching on scalar bits.

**Montgomery Ladder (x25519_ladder):** 255 iterations from bit 254 down to 0. `swap ^= k_t`, then `fe_cswap` before and after the differential addition step. No branch on `k_t`.

**Fermat Inversion (`fe_invert`):** 255 squarings + 11 multiplications using Bernstein's addition chain for 2^255-21.

**Low-Order Point Rejection (C3 fix):** 5-entry constant-time table of known low-order points checked via `x25519_is_low_order()`. All-zero output also rejected.

**Non-Canonical Reduction (`fe_pack`):** Constant-time conditional subtraction of p from the result if value >= p:
```c
uint64_t mask = (uint64_t)(-(int64_t)borrow);  // 0 if canonical, ~0 if not
for (ki = 0; ki < 5; ki++) {
    t.v[ki] = (tmp[ki] & ~mask) | (t.v[ki] & mask);
}
```

**Field Multiplication (`fe_mul`):** Schoolbook 5×5 = 25 products using `unsigned __int128` intermediates (`u128`). Cross-limb terms with index >= 5 get factor 19 applied.

### SKILL 07: ECDSA P-256 — Constant-Time Field Arithmetic + Signature Verification
**File:** `sec/xsec/crypto/ecdsa_p256.c`

**Threat Model:** Private key recovery via timing analysis of conditional branches in field operations during signature verification. Forged signature acceptance.

**Implementation Pattern — fe256[4] Representation:**
```c
typedef uint64_t fe256[4];  // little-endian limb order, all ops mod P256_P
```

**CT Conditional Select (`fe_cmov`):**
```c
uint64_t mask = (uint64_t)0 - flag;  // 0 or 0xFFFF...FFFF
for (i = 0; i < 4; i++) { r[i] = (r[i] & ~mask) | (a[i] & mask); }
```

**CT Conditional Swap (`fe_cswap`):**
```c
uint64_t t = mask & (a[i] ^ b[i]);
a[i] ^= t; b[i] ^= t;
```

**CT Zero Check (`fe_is_zero`):**
```c
uint64_t z = a[0] | a[1] | a[2] | a[3];
z |= z >> 32; z |= z >> 16; z |= z >> 8; ...
return (int)(1 - (z & 1));
```

**Verification Path:** `xsec_ecdsa_p256_verify()` implements FIPS 186-4 ECDSA verification. Called from `verify_kernel_signature()` in AetherBoot §16.

**Freestanding:** P-256 constants (P, N, b, Gx, Gy) embedded as static `fe256` literals. No heap.

### SKILL 08: ChaCha20-Poly1305 AEAD — Stack-Safe Streaming MAC
**File:** `sec/xsec/crypto/chacha20.c`

**Threat Model:**
1. `P2-POLY-STACK-01`: ~32KB scratch buffer from Poly1305 input construction — kernel stack overflow.
2. `P2-POLY-CT-01`: Final reduction via data-dependent branch on secret accumulator value.

**Implementation Pattern — ChaCha20 Quarter Round (branchless):**
```c
#define QR(a,b,c,d) \
    (a)+=(b); (d)^=(a); (d)=ROTL32((d),16u); \
    (c)+=(d); (b)^=(c); (b)=ROTL32((b),12u); ...
```
20 rounds = 10 double rounds (column + diagonal). Pure ARX, no branches.

**Poly1305 One-Time Key:** `chacha20_block(key, 0, nonce, ...)` with counter=0. First 32 bytes used as Poly1305 key. `xsec_secure_zero(poly_key_block, 64)` immediately after extraction.

**Streaming Poly1305 (P2-POLY-STACK-01 fix):** Instead of building the full MAC input in a scratch buffer, feed each piece directly:
```c
poly1305_update(&poly_st, aad, aad_len);
poly1305_update(&poly_st, poly_zero, pad_needed);
poly1305_update(&poly_st, ct, in_len);
poly1305_update(&poly_st, poly_zero, pad_needed);
poly1305_update(&poly_st, len_block, 16u);
```

**CT Final Reduction (P2-POLY-CT-01 fix):**
```c
uint64_t do_reduce = sum_top >> 2u;
uint64_t ct_mask   = (uint64_t)(0u - do_reduce);  // 0xFFF or 0
u128 acc_final = (acc_t5 & mask128) | (acc_acc & ~mask128);
```
No branch on `do_reduce`. Polynomial multiplication uses `unsigned __int128` (`u128`) for 130-bit intermediates.

**Zeroization:** `xsec_secure_zero(keystream, 64)` after each ChaCha20 block. `xsec_secure_zero(st, sizeof(*st))` in `poly1305_final`. `xsec_secure_zero(computed_tag, ...)` after comparison.

### SKILL 09: RSA-2048 PSS Verification — RFC 8017 §9.1.2 Step-by-Step
**File:** `sec/xsec/crypto/rsa.c:193-408`

**Threat Model:** Signature forgery via malformed PSS structure. Bleichenbacher-style padding oracle if padding check is not constant-time.

**Implementation Pattern — RSA Public Key Operation:**
`sig^65537 mod n` via square-and-multiply. 65537 = 2^16 + 1 = 16 squarings + 1 multiply. Not constant-time (acceptable for public-key ops; exponent and modulus are public).

**PSS Verification Steps (RFC 8017 §9.1.2):**
1. Trailer byte check: `em_buf[255] != 0xBC` → VERIFY_FAIL
2. Top-bit check: `em_buf[0] & 0x80` → VERIFY_FAIL
3. MGF1-SHA256 mask derivation via `pss_mgf1_sha256(h_prime, 32, db_mask, 223)`
4. DB = maskedDB XOR dbMask; clear top bit of DB[0]
5. **CT zero-check for padding string PS:**
   ```c
   volatile uint8_t ps_nonzero = 0u;
   for (i = 0; i < 190; i++) { ps_nonzero |= db[i]; }
   if (ps_nonzero != 0u) { ... }
   ```
   `volatile` prevents compiler from short-circuiting the loop.
6. `db[190] == 0x01` separator check
7. Reconstruct M' = 8×0x00 || mHash || salt; H'' = SHA256(M')
8. **CT comparison:** `xsec_ct_eq(h_prime, h_double_prime, 32)`

**Full Zeroization:** All intermediates (`em_buf`, `db_mask`, `db`, `m_prime`, `h_double_prime`) are zeroed before return on every code path.

**PKCS#1 v1.5 also implemented:** `xsec_rsa2048_pkcs1_verify()` with DigestInfo DER header validation.

### SKILL 10: HMAC-SHA256 + HKDF — TLS 1.3 Key Schedule
**File:** `sec/xsec/crypto/hmac.c`

**Threat Model:** Key derivation weakness, key confusion attacks in TLS 1.3 transcript binding.

**Implementation Pattern — HMAC (RFC 2104):**
```
H((K' XOR opad) || H((K' XOR ipad) || m))
```
- K' = SHA256(K) if `len(K) > 64`, else K zero-padded to 64 bytes.
- ipad = 0x36 repeated, opad = 0x5C repeated.
- Streaming: inner and outer SHA-256 contexts maintained in `xsec_hmac_sha256_ctx_t`.
- `xsec_secure_zero(k_prime, 64)` after ipad/opad derivation. ipad_key and opad_key zeroed in `_final()`.

**HKDF-Extract (RFC 5869):** `PRK = HMAC-SHA256(salt, IKM)`. If salt is NULL, uses 32 zero bytes.

**HKDF-Expand:** `T(i) = HMAC-SHA256(PRK, T(i-1) || info || i)`. Max output: 255*32=8160 bytes. Counter byte is appended as a single byte (not big-endian uint32).

**HKDF-Expand-Label (RFC 8446 §7.1):**
```
HkdfLabel = uint16(length) || uint8(label_len) || "tls13 " || label || uint8(context_len) || context
```
Prefix "tls13 " (with space) is required by TLS 1.3. `xsec_secure_zero(hkdf_label, 520)` after expand.

### SKILL 11: Security Audit Ring Buffer — Lock-Free Write with Temporal Safety
**File:** `sec/xsec/audit/audit.c`

**Threat Model:**
1. S3-DI-L01: Auto-init race: two concurrent first callers both entering `if (!initialized)`, second memset zeroes entries written by first.
2. GS-S4-P1-003: Temporal race — `write_idx` advanced before entry fields populated; concurrent reader sees uninitialized entry.
3. P2-S20-AUDIT-01: `dropped_events` counter falsely incremented on every write once ring is full (ring overwrite is normal, not an error).

**Implementation Pattern:**
```c
pal_spin_lock(&xsec_g_audit.lock);
// Detect overwrite, record in ring_overwrites (not dropped_events)
if (write_idx >= RING_SIZE) { pal_atomic_add(&ring_overwrites, 1u); }
// Write all fields BEFORE advancing write_idx
entry->timestamp_ns = pal_time_now_ns();
entry->event_type   = event;
...
// THEN advance write_idx
xsec_g_audit.write_idx++;
pal_spin_unlock(&xsec_g_audit.lock);
```

**Ring Read (most recent first):** `(write_idx - 1 - i + RING_SIZE) % RING_SIZE`

**Initialization Guard:** `if (!xsec_g_audit.initialized) return XSEC_ERR_STATE;` — hard error if called before `xsec_audit_init()`. No auto-init.

**1024-entry ring, zero heap.** All 13 audit event types documented with names via `audit_event_name()`.

### SKILL 12: ECDSA P-256 Kernel Signature Gate at Boot
**File:** `devices/desktop/aetherboot/src/aetherboot.c:1093-1400`

**Threat Model:** Kernel tampering or replacement between build and boot (supply chain attack). An attacker who controls the ESP filesystem could replace `xenos.elf` with malicious code.

**Implementation Pattern (§16, §16a):**
1. Dev path (`!GENOS_KEYED_BUILD`): zero-filled `g_db_pubkey_x/y` arrays. `verify_kernel_signature()` detects all-zero key and performs hash-only check (corruption detection). Falls back to `EFI_SUCCESS` only in dev mode.
2. Production path (`GENOS_SECURE_BOOT + GENOS_KEYED_BUILD`): real P-256 key from `genos_kernel_pubkey.h` (generated by `tools/secure-boot/generate-ecdsa-kernel-key.sh`).
3. Read `xenos.sig` (64 bytes: r[32] || s[32] big-endian) from ESP.
4. Compute SHA-256 of kernel image.
5. `xsec_ecdsa_p256_verify(hash, 32, sig_r, sig_s, g_db_pubkey_x, g_db_pubkey_y)`.
6. Failure → permanent security recovery console. No retry.

**Compile-Time Warning:** `#warning "Secure Boot enabled without GENOS_KEYED_BUILD"` fires when `GENOS_SECURE_BOOT=1` but no real key is present — ensures developer sees it at compile time, not boot time.

**Belt-and-Suspenders:** Hash comparison against `XPKG_ROOT_FINGERPRINT` runs BEFORE the ECDSA check. Both must pass for production boot.

**NEVER distribute an unsigned ISO.** GPG signature on SHA256SUMS required.

### SKILL 15: Pre-Boot Full Disk Encryption — PBKDF2 + AES-256-GCM
**File:** `devices/desktop/aetherboot/src/disk_encrypt.c`

**Threat Model:** Theft of storage device gives adversary unencrypted data. Brute-force boot-time PIN attack.

**Implementation Pattern:**
1. 256-byte random salt at DISK_ENCRYPT_SALT_OFFSET (0x000) on XSTORE partition.
2. 12-byte AES-GCM IV at 0x100.
3. 16-byte GCM tag at 0x10C.
4. PBKDF2-SHA256: 100,000 iterations, 32-byte output key.
5. AES-256-GCM decrypt of XSTORE superblock body (3812 bytes at 0x11C).
6. GCM authentication tag verified before plaintext trusted.
7. Max 3 attempts (`DISK_ENCRYPT_MAX_ATTEMPTS`), then `HLT` — prevents brute force.

**Password Zeroization (volatile loop):**
```c
volatile UINT8 *pwbuf = (volatile UINT8 *)password_buf;
for (UINT32 i = 0; i < DISK_ENCRYPT_MAX_PASS_LEN + 1u; i++) { pwbuf[i] = 0u; }
```
`volatile` prevents compiler from optimizing away the zero.

**Echo Suppression:** UEFI `SimpleTextInput` ConIn protocol used directly. No echo of typed characters.

**Encryption Flag:** `DISK_ENCRYPT_FLAG_MAGIC = 0xDEAD1BEF` at byte 0xFF8 — unencrypted signal to call `disk_encrypt_unlock()`.

### SKILL 16: Dual-Magic Stack Canary with RDRAND Seeding
**File:** `kernel/xenos/core/stack_guard.c`

**Threat Model:** Stack buffer overflows overwriting return addresses. Static canaries are predictable (attacker can embed expected value in payload).

**Implementation Pattern:**
```c
uint64_t g_stack_canary_lo = STACK_CANARY_MAGIC_LO_DEFAULT;
uint64_t g_stack_canary_hi = STACK_CANARY_MAGIC_HI_DEFAULT;
```

**RDRAND Seeding (`stack_guard_init_canary`):**
```asm
"rdrand %0\n\t"
"setc %1\n"
```
3 retry attempts per value. TSC XOR fallback if RDRAND unavailable. Zero-value guard: if value comes out 0, force to `0xA5A5A5A5A5A5A5A5`.

**Canary Layout (at `stack_base + STACK_GUARD_PAGE_SIZE`):**
```
[+0] STACK_CANARY_MAGIC_LO  (0xDEADC0DE5AFEC0DE runtime-randomized)
[+8] STACK_CANARY_MAGIC_HI  (0xBADCAFE0FEEDFACE runtime-randomized)
```

**Guard Page:** First 4KB of each stack (`stack_base` address) left unmapped. #PF handler calls `stack_guard_check_fault(fault_addr)`, which matches fault address against guard page range and panics if match found.

**Periodic Canary Check:** `stack_guard_check_canaries()` called from scheduler tick — detects overwrites that jumped over the guard page (large `alloca`, struct overflow).

**Stack Depth Estimation:** `stack_guard_get_depth(fiber_id, rsp)` = `stack_top - rsp`.

### SKILL 17: SMEP/SMAP/UMIP CPU Feature Gating (CPUID before write)
**File:** `kernel/xenos/core/main.c:Step 4.6` (Wave 0 security wiring)

**Threat Model:**
- Without SMEP: kernel can be tricked into executing user-mode code.
- Without SMAP: kernel can be tricked into reading user-mode data (TOCTOU, null-deref tricks).
- Without UMIP: user-mode code can execute SGDT, SIDT, SLDT, SMSW, STR — leaks kernel address layout.

**Implementation Pattern — CPUID-gated CR4 bits:**
```c
// CPUID.07H.EBX[7] = SMEP, [20] = SMAP, EDX[2] = UMIP
// All checked via "=a"/"=b" CPUID asm before setting bits in CR4
cr4 |= (1UL << 20);  // SMEP
cr4 |= (1UL << 21);  // SMAP
cr4 |= (1UL << 11);  // UMIP  -- NOTE: prior audit found bit 11 wrong; (1UL<<22) was wrong, bit 11 IS UMIP on CR4
```

**Note from Sprint 47 audit (P1-CR4-UMIP-WRONG-BIT):** `main.c:3787` had `(1UL<<11)` vs `(1UL<<22)`. Bit 11 is correct for CR4.UMIP (Intel SDM Vol 3A §2.5). Fixed in Sprint 47.

**Also enabled:** OSXSAVE (bit 18), OSFXSR (bit 9), OSXMMEXCPT (bit 10) for SSE/AVX state save.
CR0: MP set, EM cleared (FPU present), TS set (for lazy save/restore).

### SKILL 18: Spectre/Meltdown Mitigations — IA32_SPEC_CTRL + TSX Disable
**File:** `kernel/xenos/core/main.c:Step 9b`

**Threat Model:** Spectre v2 (BTI) — indirect branch poisoning across privilege levels. Spectre v4 (SSBD) — speculative store bypass. TSX (CVE-2019-11135 / TAA) — microarchitectural data sampling via transactional memory abort.

// Check CPUID.07H.EDX[29] for IBRS/IBPB support

### SKILL 19: IOMMU VT-d DMA Isolation
**File:** `kernel/xenos/drv/iommu_vtd.c`

**Threat Model:** DMA attacks: a malicious PCIe device (or compromised firmware) issuing DMA transfers to arbitrary physical addresses — bypassing OS memory isolation. Thunderstrike-style attacks via Thunderbolt.

**Implementation Pattern:**
1. Parse DMAR ACPI table (signature "DMAR" in XSDT) via `vtd_find_dmar()`.
2. ACPI checksum verification: `vtd_checksum(rsdp, 20) != 0` → reject.
3. Extract DRHD entries (type 0): IOMMU unit MMIO base addresses.
4. Set up root table (256-entry, one per PCI bus) and context tables (256 entries, one per PCI device/function).
5. Program 4-level IOVA page tables per device, restricting each device's addressable range.
6. IOTLB + context-cache invalidation after table updates.
7. Wired in `kmain()` Step 10a2 after XACPI, before XPCI.

**RSDP Address Validation:** `rsdp_phys >= 0x2000000000ULL` → reject (prevents HHDM mapping of non-physical addresses).

**DRHD Checksum Guard:** `vtd_checksum(hdr, hdr->length) == 0` required before any DRHD is accepted.

**Fail-Safe:** If no DMAR found, IOMMU not enabled (logged). Systems without VT-d hardware continue without DMA isolation.

### SKILL 20: XKABI Capability System — HMAC-Authenticated Handles
**File:** `kernel/xenos/cap/xkabi_capabilities.c`

**Threat Model:** Handle forgery from user-space. Without authentication, user-space can construct arbitrary handle values and gain elevated rights. Generation mismatch attacks after capability revocation.

**Implementation Pattern — 64-bit Handle Encoding:**
```
bits [63:48] = slot    (16 bits, index into caps[])
bits [47:32] = rights  (16 bits, XKABI_RIGHT_ALL masked)
bits [31:16] = gen     (16 bits, revocation epoch)
bits [15:0]  = hmac    (16 bits, truncated SHA-256 tag)
```

**HMAC Computation:**
```c
// msg[44] = secret_key[32] || slot_le[4] || rights_le[4] || gen_le[4]
pal_sha256(msg, 44, hash);
return (uint16_t)(hash[0] | (hash[1] << 8));  // lower 16 bits
```

**HMAC Secret Key:** Initialized from 4 × `pal_random_u64()` calls during `cap_table_init()` — not hardcoded, not in Git.

**Rights Narrowing Invariant:** Derived capabilities can only narrow rights, never widen. `cap_derive()` validates `(derived_rights & ~parent_rights) == 0`.

**Revocation:** Two-phase: collect-then-invalidate (UAF bug BUG-3.1 fix). Revocation increments `generation` field. All handles with stale generation are rejected.

**Free List:** O(1) slot reclamation via intrusive singly-linked list through `next_free` field. O(N) initialization at boot only.

**Audit Trail:** Every `cap_create/allow/deny/revoke/derive` logged via `xkabi_audit()` with monotonic serial number.

**No Ambient Authority:** XKABI_RIGHT_ALL = explicit set. No implicit `root` concept.

### SKILL 22: KASLR — RDRAND-Based Physical Load Address Randomization
**File:** `devices/desktop/aetherboot/src/aetherboot.c:580-617`

**Threat Model:** Return-oriented programming (ROP) chains require knowing kernel virtual/physical addresses. Static load address makes the kernel trivially exploitable once any memory disclosure is achieved.

**Implementation Pattern:**
```c
// P1-S20-KASLR-01 fix: apply slide BEFORE load_kernel_elf()
// (was AFTER — slide was 0 at load time, correction applied too late)
for (i = 0; i < 3 && !ok; i++) {
    __asm__ volatile("rdrand %0\n\t" "setc %1\n" : ...);
}
UINT64 slide = rnd % KASLR_MAX_SLIDE;         // [0, 64MB)
slide = slide & ~(KASLR_ALIGN - 1ULL);         // align down to 2MB boundary
```

**5-bit entropy** (2MB aligned, [0,64MB) = 32 possible positions). Protects against spray attacks requiring exact address prediction.

**Fallback:** If RDRAND unavailable → `slide = 0` (KASLR disabled, logged). Boot continues without slide.

**Scan loop:** If UEFI `AllocateAddress` fails at `kaslr_base`, scan in 2MB steps up to +128MB.

### SKILL 24: XSEC Error Code Taxonomy — No Ambiguous Error Codes
**File:** `sec/xsec/include/xsec.h:35-58`

**Threat Model:** Ambiguous error codes cause callers to fall through to success path. Mixing auth failure with I/O failure causes insecure handling.

**Pattern:**
```c
#define XSEC_OK                0
#define XSEC_ERR_AUTH_FAIL    -3   // authentication tag mismatch
#define XSEC_ERR_VERIFY_FAIL  -4   // signature verification failed
#define XSEC_TLS_WANT_READ   -15   // non-blocking: retry later
```
`AUTH_FAIL` (decryption tag) and `VERIFY_FAIL` (signature) are distinct codes — callers handle them differently. `WANT_READ` (-15) is an informational code, not an error — well-behaved non-blocking transport.

### SKILL 26: Secure Zeroization — `xsec_secure_zero` Pattern
**File:** All crypto files

**Threat Model:** Sensitive key material remaining in memory after use — exposed via memory dumps, hibernation images, or cold-boot attacks.

**Pattern:** `xsec_secure_zero()` (implemented via `volatile` write or compiler barrier) is called:
- After `sha256_final`: `xsec_secure_zero(ctx, sizeof(*ctx))`
- After GCM core: key schedule, H, S, E_J0 all zeroed
- After ChaCha20 block: `xsec_secure_zero(keystream, 64)`
- After Poly1305 final: `xsec_secure_zero(st, sizeof(*st))`
- After PSS verify: all intermediates zeroed on every code path
- After HMAC final: `xsec_secure_zero(inner_digest, 32)`, ipad_key, opad_key
- After HKDF expand: `xsec_secure_zero(t, 32)`, `xsec_secure_zero(hkdf_label, 520)`

**Critical:** Must be called on EVERY code path including error returns — not just the success path.

### SKILL 27: `volatile` Keyword for Security-Critical Operations
**File:** Multiple files

**Threat Model:** Compiler optimizes away "dead" writes to memory that won't be read again after a security check.

### SKILL 29: SHA-384 as Truncated SHA-512 — Context Reuse Pattern
**File:** `sec/xsec/crypto/sha256.c:471-538`

**Threat Model:** Code duplication of SHA-512 logic leading to divergent implementations with different bugs.

**Pattern:**
```c
typedef xsec_sha512_ctx_t xsec_sha384_ctx_t;  // same struct, different IV

```

### SKILL 35: Symmetric Cipher Selection — AES-NI vs Software Fallback
**File:** `sec/xsec/crypto/aes_gcm.c:229-298`

### SKILL 36: Address Space Layout — PML4 / HHDM / Kernel Mapping
**File:** `devices/desktop/aetherboot/src/aetherboot.c:619-711`

**Threat Model:** Kernel/user address overlap allows user-mode pointers to directly address kernel memory.

**Pattern (4-level page table):**
```
PML4[  0] → identity map (16 × 1GB huge pages for devices/RAM up to 64GB)
PML4[273] → HHDM = physical+0 → 0xFFFF888000000000 (direct phys map)
PML4[511] → kernel at KERN_BASE = 0xFFFFFFFF80000000 (top 2GB)
```

**2MB huge pages for kernel:** PD_kern uses `PTE_PS` for each 2MB block. `kern_phys + i*0x200000` for `i` in `[0, kern_pages)`.

**Guard:** `kern_pages > 512` → clamped. `p_memsz > kern_span - seg_off` → fatal, boot aborted.

**BSS zero init:** `rep stosq` (microcode-optimized 8B/iter) pre-zeros destination region before copying PT_LOAD segments. Defense-in-depth for BSS gap (p_memsz > p_filesz).
