## B. AETHERBOOT

### B1. KASLR Slide Generation (aetherboot.c lines 462–490)

PASS. kaslr_generate_slide():
- RDRAND with CF check, 10 retries. Line 466-475.
- CF=0 fallback: slide=0 (KASLR disabled), logged to serial. Lines 478-481.
- Clamp to [0, 64MB). Line 484.
- Align down to 2MB boundary via bitmask. Line 487.
- Result stored in g_kaslr_slide, passed in xenos_handoff_t.kaslr_slide. PASS.

NOTE: KASLR slide is generated and passed in the handoff struct but the kernel's
own VMM does not apply the slide to remapping the kernel ELF. The slide informs
randomization intent but the physical load address chosen by AetherBoot's
build_page_tables() is not currently shifted by the slide value.
This is a P1 architectural gap if physical KASLR is the security goal —
the slide is logged and stored but not applied to the physical load address.

OPEN ISSUE P1-S20-KASLR-01: g_kaslr_slide is generated but not applied to
kern_phys in the load sequence. load_kernel_elf() uses AllocatePages starting
at fixed physical addresses (0x200000 scan). The handoff struct carries the slide
value but build_page_tables() does not offset kern_phys by g_kaslr_slide.
Effect: KASLR slide is informational only; the physical load base is
determined by EFI availability, not randomized. Real physical KASLR requires
kern_phys = g_kaslr_slide + base_address as the AllocateAddress argument.
This is a known limitation documented in ADR-S19-01 but must be tracked as P1
before shipping KASLR as a security claim.

### B2. ELF Loader Bounds Checks (lines 686–779)

P1-BOOT-02 RESOLVED: e_phentsize + e_phnum + e_phoff bounds check at lines 688-694.
P1-BOOT-03 RESOLVED: p_memsz > 4GB or overflow check at lines 702-710; kern_span > 64MB check at lines 723-726.
P1-BOOT-04 RESOLVED: seg_off + p_filesz within kern_span AND p_offset + p_filesz within elf_size at lines 755-763.

All three previously-flagged ELF loader P1 findings are confirmed FIXED. PASS.

### B3. TPM Measurement

GS-P1-TPM-UNWIRED RESOLVED: tpm2_init(), tpm2_measure_aetherboot(), tpm2_print_pcr_log()
all called in efi_main. Lines 850, 952, 979. cap_init_from_efi() also wired. Line 879.
PCR[7] measurement before ExitBootServices confirmed. PASS.

### B4. ExitBootServices Sequence

PASS. Sequence:
1. GetMemoryMap (first call, pre-load).
2. AllocatePages for ELF buffer and kernel load region.
3. TSC calibration via Stall (before EBS).
4. KASLR slide generation (before EBS).
5. build_page_tables (before EBS).
6. GetMemoryMap re-call (immediately before ExitBootServices — map_key refresh).
7. tpm2_measure_aetherboot(NULL, 0) before EBS.
8. ExitBootServices with retry on failure.
9. Serial only after EBS.

One advisory: Retry on EBS failure gets a new map_key but does not re-call
build_page_tables with the potentially-changed memory map. The page tables
were built before the retry and include EfiBootServicesCode/Data which the
second GetMemoryMap may report differently. This is extremely unlikely to
cause a real problem (page tables just map more memory than needed) but is
technically non-conforming with UEFI spec.

**Status: CONDITIONAL PASS**
**P1-S20-KASLR-01: Slide generated but not applied to physical load address.**

## D. STORAGE (XSTORE WAL + XBLOB)

### D1. WAL Checkpoint Correctness

PASS. xstore_wal_checkpoint() (wal.c lines 180–247):
- Flushes dirty pages first (xstore_cache_flush). Line 185.
- Checkpoint LSN snapshot taken under wal_lock BEFORE append. Lines 216-218.
  This correctly captures the pre-add value since pal_atomic_add returns pre-add.
  The checkpoint entry will receive exactly checkpoint_lsn as its LSN.
- Superblock updated with checkpoint_lsn, wal_head, next_txn_id. Lines 224-226.
- CRC32c over superblock bytes [8, PAGE_SIZE). Line 232.
- sync() called before wal_tail advance. Line 238.
- wal_tail = checkpoint_lsn + 1 under lock. Lines 242-244.

OPEN ISSUE P2-S20-WAL-01: WAL_REPLAY_MAX_TXNS = 256 hard cap.
xstore_wal_replay() (lines 98-170) silently truncates committed_txns[] at 256.
If more than 256 transactions commit between checkpoints, recovery will silently miss
transactions beyond the 256th. The btree_insert/delete calls are "best-effort" anyway,
but silent data loss on recovery is a reliability risk. Should either increase the cap
or use a two-pass O(n) linear scan approach.

### D2. XBLOB Index Load Factor

PASS. xblob_alloc_slot() (blob.c lines 46-64):
- 256 words × 64 bits = 16384 slots max.
- Bitmap scan O(n) worst case but O(1) amortized with early-exit on first non-full word.
- Returns UINT64_MAX on full. Callers check this.

No load factor throttle exists — the store fills to 100% capacity before refusing writes.
P3 advisory: production deployments should enforce a high-water mark (e.g., 80% = 13107 slots)
to prevent write amplification at full capacity.

### D3. GC Tombstone Cleanup

Memory (PIA-S4-P1-01): lock ordering fix for xblob/gc.c confirmed in Sprint 10.
Not re-read in this audit (not in scope list) — status remains PASS from prior audit.

**Status: CONDITIONAL PASS**
**P2-S20-WAL-01: WAL replay txn cap at 256 — silent data loss risk on heavy workloads.**

### F2. Audit Ring Race Condition Fix (audit.c)

Sprint 18 fix (GS-S4-P1-003) CONFIRMED:
- xsec_audit_log() holds spinlock for the entire write+index-advance sequence.
- All entry fields written BEFORE write_idx is incremented. Lines 158-166.
- No partial-entry visibility. PASS.

OPEN ISSUE P2-S20-AUDIT-01: Overflow detection has a subtle logic gap.
Line 151: `if (xsec_g_audit.write_idx >= XSEC_AUDIT_RING_SIZE) { dropped++ }`
This ONLY counts a drop if write_idx >= 1024, then proceeds to write anyway (line 155
computes slot = write_idx % 1024). So the ring DOES wrap and overwrite (circular buffer),
but the drop counter fires for EVERY write after the ring fills (write_idx >= 1024),
not just when the oldest entry is being overwritten. This means after 1024 events,
every subsequent write increments dropped_events even though data IS being written
to the ring (overwriting oldest). The counter semantics are inaccurate: it counts
"events written while ring has wrapped" not "events lost." Not a data corruption bug
but the monitoring signal (xsec_audit_get_dropped()) is misleading. P2.

### F3. Open P1 XSEC Findings (from Sprint 3 deep audit)

- P1-S3-CERT-01: x509.c CN null-byte bypass via embedded 0x00 in DER string.
- P1-S3-CERT-04: xsec_x509_verify_hostname() scans formatted subject string — O= value can spoof CN.
- P1-S3-KABI-01: xkabi_cap_derive() revoked-parent race between two lock acquisitions.

These are NOT in tls13.c or audit.c (which are the Sprint 18 targets) but affect the
broader XSEC layer. They block production TLS authentication. For QEMU LOOPBACK tests
(no real certificate validation), these are non-blocking. For any test using real
certificate chains, they are P1 blockers.
