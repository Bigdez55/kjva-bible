# Pass 3 Supply Chain + Legal Compliance Audit — 2026-04-24

## OPEN FINDINGS (requires action before production release)

### CRITICAL / P0

**CRIT-01: XSTORE_ROOT_HASH all-zeros in boot_chain.c (P1-SECBOOT-KEY-01 still open)**
- `init/gensd/boot_chain.c` line 446: `static const uint8_t XSTORE_ROOT_HASH[32]` = all 0x00
- Sprint 47 CI gate exists (iso-root-hash-gate job) but ONLY fires on main/tags
- Feature branches can build with zero hash — check passes (sentinel skip) in runtime
- Sprint 47 CI gate job name = `iso-root-hash-gate` in `.github/workflows/sprint47-boot-hardening.yml`
- **Action**: ISO build pipeline MUST inject real XSTORE_ROOT_HASH before any production ISO

**CRIT-02: Dev JWT public key hardcoded in docker-compose.yml**
- Lines 68-69: `SYNC_JWT_PUBLIC_KEY` + `PROVENANCE_JWT_PUBLIC_KEY` both set to same Ed25519 public key `MCowBQYDK2VwAyEA9NX1RO0SxQy86JJ7hsV9o9cpFwZtBfTEobOKio3XeVw=`
- This is committed to git — if the corresponding private key was ever used/committed anywhere, all JWTs signed by it are permanently compromised
- No corresponding private key found in repo — public key only exposure
- Severity: If this key pair is used in production, every dev-environment token verifies against production = auth bypass risk

### MEDIUM / P2

**P2-SUPPLY-01: 9 Electron apps missing package-lock.json**
- Only `devices/desktop/design-system/` and `devices/mobile/` have package-lock.json
- Remaining ~9 Orange suite apps: no lockfile = non-reproducible npm installs
- Supply chain attack vector: transitive dep upgrades can inject malicious code between builds
- **Action**: Run `npm install --package-lock-only` in each app dir and commit

**P2-DOCKER-01: All 18+ production Dockerfiles use floating tags (not SHA-pinned)**
- `FROM python:3.12-slim` (13 Dockerfiles), `FROM node:22-slim` (1), `FROM ubuntu:24.04` (2)
- Zero SHA digest pins found in any Dockerfile
- If DockerHub is compromised or tag is reused, build silently ingests malicious base image
- **Action**: Pin each to `FROM python:3.12-slim@sha256:<digest>`

**P2-CRYPTO-KEY-01: XPKG_ROOT_FINGERPRINT is a static dev key**
- `init/gensd/boot_chain.c` line 832: `XPKG_ROOT_FINGERPRINT` contains a non-zero 32-byte value `{0x4a, 0x8f, 0x2c, ...}`
- This is committed to git = anyone who has this repo knows the package signing fingerprint
- Key ceremony documentation: not found in docs/
- **Action**: Document key ceremony process; XPKG_ROOT_FINGERPRINT for production ISO must come from a key not committed to git

**P2-CRYPTO-KEY-02: TPM enrollment path undocumented**
- TPM 2.0 TIS driver exists (`kernel/xenos/drv/tpm2_tis.c`) — Sprint 47 complete
- No runbook or ADR covers: how PCR values are provisioned, how TPM is enrolled on first boot, key rotation procedure
- **Action**: Write RUNBOOK-TPM-ENROLLMENT.md

**P2-PRIVACY-01: Medical ID stores sensitive health data without encryption-at-rest disclosure**
- `devices/desktop/shell/xshell/src/medical_id.c`: stores name, blood_type, allergies, medications, conditions, organ_donor in XSTORE via `g_settings_store`
- No evidence XSTORE B-tree is encrypted at rest (XSTORE is a plain B-tree, not AES-GCM like SoulManager)
- This is potentially HIPAA-adjacent data (US) and special category health data (GDPR Art. 9)
- **Action**: Either (a) apply AES-256-GCM to medical_id XSTORE keys, or (b) document that medical ID data is local-only on-device, user-controlled, and exempt from cloud sync

**P2-PRIVACY-02: Right to erasure — partial implementation**
- `_pg_delete_user()` exists in identity/app.py (line 362) — user deletion from PostgreSQL covered
- No evidence of cascading soul data erasure: SoulManager data keyed by agent name (not user UID) — user deletion does not purge soul memory
- GDPR Art. 17 requires erasure of ALL data, not just auth records
- **Action**: Implement `/identity/users/{uid}/erase` endpoint that purges identity + all soul data scoped to user

**P2-PRIVACY-03: No GDPR privacy policy or consent framework in codebase**
- Telemetry service exists (`devices/cloud/platform/services/telemetry/`)
- `MASTER_INTEGRATION_PLAN_v1.md` references `privacy_manager.py` as planned but not found in production services
- No opt-in/opt-out telemetry consent flow found
- **Action**: Implement consent gate before any telemetry data leaves device; provide privacy policy document

**P2-ACCESSIBILITY-01: TTS xaudio_pcm_write is weak extern — ADA/EU a11y risk**
- `ai/tts/tts_engine.c` line 195-196: `extern __attribute__((weak)) void xaudio_pcm_write(...)` 
- Pattern is intentional (headless CI) but if audio driver is not linked at runtime, TTS is silently no-op
- WCAG 2.1 Success Criterion 1.1.1 requires non-text content alternatives; EU Accessibility Act 2025 requires TTS for public software
- The contract check `if (xaudio_pcm_write != NULL)` means no error is raised when audio is absent
- **Action**: Add startup check that logs WARN when xaudio_pcm_write is NULL on non-headless builds

**P2-ACCESSIBILITY-02: WCAG AAA claimed but only contrast ratio verified**
- `high_contrast_themes.c` verifies 7:1 contrast ratio (AAA criterion 1.4.6) 
- WCAG AAA is a comprehensive standard (87 success criteria) — contrast alone is insufficient to claim AAA
- Focus indicators, keyboard navigation, timing, cognitive load criteria are NOT verified by existing tests
- **Action**: Audit against full WCAG 2.1 AAA criterion list; downgrade claim to WCAG AA until full AAA audit is completed
