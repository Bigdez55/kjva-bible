# XPKG Security + Reliability Audit — 2026-03-05

## Verdict: CONDITIONAL — 4 P1 findings, 8 P2 findings. Gate 0+1 ordering is CORRECT. 9-gate machine is STRUCTURALLY SOUND. Do not ship without P1 resolution.

## P1 FINDINGS (BLOCKING)

### P1-XPKG-01: Signature verification does NOT lock s_key_lock during read pass
**File:** pkg_verify.c §7 xpkg_verify_signature (lines 245-299)
**Finding:** `xpkg_verify_signature()` reads from `s_trusted_keys[]` and `s_key_count` without holding `s_key_lock`. The comment says "reads do not lock (keys are immutable once written in practice)" but this is not enforced — `xpkg_add_trusted_key()` and `xpkg_remove_trusted_key()` mutate the array under the lock. A concurrent add/remove while verification is in progress produces a data race. Under TSan this is a reportable race; on x86 with spinlock release it is a store-visibility issue.
**Impact:** A concurrent key removal during signature verification could cause verify to use a partially-zeroed key slot — producing either a false-pass (if zeroed key happens to match) or false-fail. False-pass is a P0 security escalation path.
**Fix:** Take s_key_lock for read during the verify loop, OR copy the key store under the lock into a local snapshot array before releasing.

### P1-XPKG-02: glob_match() in refusal_gate.c is recursive — stack overflow risk in freestanding env
**File:** refusal_gate.c §1 glob_match (lines 103-121)
**Finding:** `glob_match()` calls itself recursively on line 110: `if (glob_match(pattern, str)) { return true; }`. This is within a freestanding kernel-layer environment with no stack guard. A crafted package name containing many `*` characters (e.g., `"*a*b*c*d*e*f*g*h*"`) against a long package name can induce O(2^n) recursion depth. At XPKG_POLICY_RULES_MAX=64 rules, the call stack per gate 8 evaluation can blow the stack.
**Impact:** DoS via specially crafted package name matching a BLOCK policy — but also a BLOCK bypass if stack overflow causes gate_policy() to silently return a garbage value that is treated as PASS.
**Fix:** Convert glob_match to iterative using a small explicit match stack, or limit pattern complexity (max 2 wildcards per pattern with documented rationale). A simple NFA-based iterative match for '*' only is ~30 lines.

### P1-XPKG-03: install.c gate check condition allows gate WARN to proceed but gate_check returns XPKG_ERR_GATE on any WARN
**File:** install.c §7 xpkg_install (lines 257-260) and §9 xpkg_upgrade (lines 417-419)
**Finding:** Both `xpkg_install` and `xpkg_upgrade` use the condition:
  `if (st != XPKG_OK && !gate_report.all_pass)`
`gate_check()` returns `XPKG_ERR_GATE` when `all_pass == false`, which includes the case where all gates PASS except one emits WARN (e.g., reinstall of same version → Gate 4 WARN, or missing dep in registry → Gate 7 WARN). In that state: `st == XPKG_ERR_GATE` AND `gate_report.all_pass == false`. The condition `!gate_report.all_pass` evaluates to `true`, so the install IS blocked.
However — `xpkg_gate_all_pass()` in refusal_gate.c §5 returns false for any WARN, yet the stated design says WARNs are non-blocking. The condition should be: block only if any gate has `XPKG_GATE_RESULT_FAIL`. Currently, a WARN on Gate 7 (dep not yet installed but resolvable from repos) blocks the install even though the design intends it to proceed.
**Impact:** Functional correctness: most first-time installs will have unsatisfied deps in the registry (they haven't been installed yet), causing Gate 7 to WARN and blocking every fresh install. This is a logic error that makes XPKG non-functional for the common case.
**Fix:** Change the install check to: `if (st != XPKG_OK)` — i.e., only block when gate_check returns XPKG_ERR_GATE due to a FAIL. OR change gate_check to return XPKG_OK when only WARNs are present and XPKG_ERR_GATE only when at least one FAIL is present (not demoted). The latter is cleaner.

### P1-XPKG-04: xpkg_remove has TOCTOU between reverse-dep check and registry compact
**File:** install.c §8 xpkg_remove (lines 332-380)
**Finding:** The flow is:
  1. Lock → find slot → check reverse deps → save copy → mark REMOVING → UNLOCK
  2. (no lock) Simulate file removal
  3. Lock → registry_find() again → compact → UNLOCK
Between steps 1 and 3, another thread can call `xpkg_install()` which calls `graph_add_edge` / `build_graph` referencing the REMOVING-state entry. Also: in step 3, `registry_find(pkg_name)` is called again because the slot index may have shifted (it won't, since nothing changes between step 1 and 3 for that specific entry, but the logic re-finds by name, which is safe here). The real risk is in `xpkg_upgrade`: it marks the old package as REMOVING without holding the lock when it removes files, then re-acquires the lock and calls `registry_find()` again. If a concurrent remove completes between these, `registry_find()` returns -1 and `g_installed_registry.count` is decremented from an already-decremented value, potentially wrapping to UINT32_MAX.
**Impact:** Registry corruption under concurrent upgrade + remove of same package. Blast radius: all subsequent install/remove operations operate on a corrupted count.
**Fix:** Hold the registry lock across the full remove operation in `xpkg_upgrade`, or use a per-slot state machine with a "pending-remove" flag that blocks concurrent operations on the same slot.

## P2 FINDINGS (HIGH PRIORITY, NON-BLOCKING FOR THIS SPRINT)

### P2-XPKG-01: parse_header total_size field is silently dropped
**File:** pkg_format.c §parse_header (lines 81-100)
**Finding:** The on-disk header has 7 fields by xpkg.h §9, but parse_header reads only 6 (magic, format_version, flags, metadata_offset, payload_offset, signature_offset). The `total_size` field at bytes [20..23] is never read — instead `out->total_size = buf_len` is assigned. If the archive has been truncated and the caller passes the truncated length as buf_len, the HEADER_SIZE check (buf_len < 20) passes, but signature_offset + 64 > buf_len catches the truncation. This is safe but: (a) total_size in the header is never validated against the actual buffer length, meaning a malicious archive can embed `total_size = 0` without detection, and (b) the on-disk struct layout in the file comment is inconsistent (claims 20 bytes for 6 fields, 4+2+2+4+4+4 = 20, no room for total_size).
**Fix:** Clarify in the format spec: either total_size is stored at bytes [20..23] (making the header 24 bytes, fix HEADER_SIZE), or it is derived from buf_len (document this clearly and remove the contradictory comment). Current HEADER_SIZE=20 is consistent with 6 fields only. Verify the on-wire format is consistent across format and verify implementations — both use HEADER_SIZE=20 and read sig_off at byte 16.

### P2-XPKG-02: hex_to_hash decodes only first 32 bytes but has redundant split loop
**File:** pkg_format.c §hex_to_hash (lines 206-216)
**Finding:** The function decodes a 64-char hex string into 32 bytes. It is split into two loops: i=0..15 and i=16..31. Both loops have the same logic. The split serves no purpose and is confusing. More importantly, the second loop guard `i * 2 + 1 < hex_len` for i=31 requires hex_len >= 64, but hex_len is the `vlen` from `meta_find_value`, which may be exactly 64 with no null terminator. The guard `i * 2 + 1 < hex_len` for i=31 evaluates to 63 < 64 = true, which is correct. However for i=16 the first loop guard `i * 4 < hex_len` evaluates to 64 < 64 = false, so the first loop stops at i=15 and the second loop handles i=16..31. This is correct but fragile and confusing.
**Fix:** Merge into a single loop `for (i = 0; i < 32; i++)` with guard `i*2+1 < hex_len`. This is the standard pattern.

### P2-XPKG-03: Payload iteration in extract_by_index has no overflow guard on pos advancement
**File:** pkg_format.c §xpkg_format_extract_by_index (lines 632-643) and extract_by_path (lines 668-681)
**Finding:** `pos += FILE_HDR_TOTAL + file_size` — if a crafted archive has `file_size = UINT32_MAX - FILE_HDR_TOTAL`, this addition wraps to near-zero and the while condition `pos + FILE_HDR_TOTAL <= hdr.signature_offset` becomes true again, looping forever or until `current` wraps to `file_index`. The xpkg_format_iter_files function at line 719-720 has an overflow guard: `if (next <= pos) { break; }`. The extract functions do NOT have this guard.
**Fix:** Add `if (next_pos <= pos) { return XPKG_ERR_FORMAT; }` before advancing pos in both extract functions, mirroring the pattern in iter_files.

### P2-XPKG-04: dep_resolve work_queue has capacity XPKG_MAX_GRAPH_NODES = 512 but is circular — can silently drop nodes
**File:** dep_resolve.c §6 work_queue (lines 287-307)
**Finding:** `wq_push` uses a circular buffer of size XPKG_MAX_GRAPH_NODES (512). The push returns false when `next == q->head` (full). At line 427: `wq_push(&wq, new_node_idx)` — the return value is NOT checked. If the queue is full, the new dependency node is silently dropped. The graph will be incomplete, cycle detection will miss edges, and the install plan will omit packages.
**Fix:** Check the return value of `wq_push` at line 427 and return `XPKG_ERR_CAPACITY` on false. This is already done correctly at line 492 (`if (!wq_push(&wq, new_node_idx)) { return XPKG_ERR_CAPACITY; }`), so apply the same pattern to the unchecked call.

### P2-XPKG-05: gate_conflict path concatenation has off-by-one when install_path is exactly 255 chars
**File:** refusal_gate.c §gate_conflict (lines 297-307)
**Finding:** `prefix_len = xpkg_strlen(full_path)` — if install_path is 255 chars, `prefix_len = 255`. The guard `if (prefix_len < sizeof(full_path) - 2u)` evaluates to `255 < 510` which is true. Then `full_path[prefix_len++] = '/'` writes to `full_path[255]` and `full_path[256] = '\0'` writes one byte past the end of `char full_path[512]`... wait, 512 is the array size, so full_path[256] is in bounds. The issue is that `xpkg_strlcpy(full_path + prefix_len, incoming_path, sizeof(full_path) - prefix_len)` with prefix_len=256 leaves only 256 bytes for the file path. With incoming_path up to 255 chars, this is correct. However: if install_path[0] == '\0' (zero-length), prefix_len=0, and `full_path[prefix_len - 1]` at line 301 accesses `full_path[-1]` — a one-byte OOB read.
**Fix:** Guard `if (prefix_len > 0 && full_path[prefix_len - 1] != '/')` to prevent the -1 access on empty install_path.

### P2-XPKG-06: xpkg_list_installed reads count under lock then iterates without lock — TOCTOU
**File:** install.c §10 xpkg_list_installed (lines 487-501)
**Finding:** Count is read under lock, then the loop iterates 0..count-1 accessing package entries without any lock. If a concurrent remove() calls compact() and decrements count during the iteration, the callback `cb(p, user_data)` may observe a REMOVING or zeroed entry.
**Impact:** Cosmetic/minor at current single-threaded CLI usage. Becomes exploitable when XPKG is integrated into a daemon context with concurrent requests.
**Fix:** Copy the snapshot of packages[] under the lock into a local stack array (bounded by XPKG_MAX_INSTALLED=512 × sizeof(xpkg_package_t)). Note: sizeof(xpkg_package_t) is large (~87KB for 256 files × 294 bytes + metadata). Do NOT stack-allocate the full registry copy. Instead, take the lock for the full iteration (acceptable if the callback is fast and does not itself call xpkg operations).

### P2-XPKG-07: repo_index parse_index_buffer name copy reads entry_size bytes but entry_size may equal sizeof(xpkg_index_entry_t) which is >> 64
**File:** repo_index.c §parse_index_buffer (lines 215-216)
**Finding:** `uint32_t name_copy = entry_size < 64u ? entry_size : 64u;` then `xpkg_memcpy(e->name, buf + pos, name_copy - 1u)`. `entry_size = sizeof(xpkg_index_entry_t)` which is much larger than 64. So `name_copy = 64`, `xpkg_memcpy(e->name, buf + pos, 63u)`. This is correct but the variable name `name_copy` and the `entry_size < 64u` comparison are misleading — entry_size will never be < 64. More critically: the function advances `pos += 64u` after the name copy, regardless of what was copied. This is correct IF the binary format lays out name[64] first. However: the index entry struct in memory has padding from the compiler. The binary parse assumes a specific field layout (name[64] + version(12) + size_bytes(8) + hash[32] + dep_names[256] = 412 bytes) but reads them field-by-field. This works but must be validated — if the struct layout changes the binary format breaks silently.
**Fix:** Add a compile-time assertion: `_Static_assert(sizeof(xpkg_index_entry_t) == 412, "xpkg_index_entry_t layout must be 412 bytes for binary format compatibility");`. Document the binary format layout explicitly.

### P2-XPKG-08: Placeholder SYSTEM key in pkg_verify.c is ASCII text "GENOS_SYSTEM_PUBKEY_PLACEHOLDER\0"
**File:** pkg_verify.c §1 (lines 59-64)
**Finding:** The 32-byte GENOS_SYSTEM_PUB_KEY is the ASCII encoding of "GENOS_SYSTEM_PUBKEY_PLACEHOLDER" + 0x00. This is not a valid Ed25519 public key. Any package signed with any real key will fail Gate 0. The comment says "In production this is replaced with the actual GEN.OS signing public key" but there is no CI gate, build script check, or compile-time assertion that prevents this placeholder from being shipped in a production ISO.
**Impact:** If shipped, ALL packages fail Gate 0. Or if Gate 0 is force-enabled in production config, ALL packages bypass signature verification.
**Fix:** Add a build-time flag `GENOS_PRODUCTION_KEY_REQUIRED` that causes a compile error if the placeholder bytes are present. The production key injection must be part of the ISO signing pipeline.

## PASSING CHECKS

### CHECK 1: Ed25519 BEFORE content processing — PASS
Gate 0 (SIGNATURE) runs before Gate 1 (INTEGRITY) and both run before any file extraction in install.c. xpkg_install() calls gate_check() at Step 2, before extract_files() at Step 4. Order is correct.

### CHECK 2: Magic number validation — PASS
parse_header() checks `out->magic != XPKG_MAGIC` at line 102 before any further parsing. Both pkg_format.c and pkg_verify.c validate magic independently.

### CHECK 3: Buffer overflow resistance in format parsing — LARGELY PASS
- All string copies use xpkg_strlcpy with explicit dst_len.
- dep_count and file_count are clamped to XPKG_MAX_DEPS_PER_PKG and XPKG_MAX_FILES_PER_PKG.
- meta_get_str copies min(vlen, dst_len-1) bytes.
- payload walking uses `pos + FILE_HDR_TOTAL <= sig_off` guard.
- EXCEPTION: pos overflow in extract_by_index/path (see P2-XPKG-03).

### CHECK 4: Cycle detection — PASS
topo_sort() uses DFS with `on_stack` marker. Back edges (child.on_stack == true) are detected and return XPKG_ERR_CYCLE. This is correct Tarjan-style cycle detection.

### CHECK 5: Topological sort correctness — PASS
Iterative post-order DFS. Nodes are appended to topo_order[] when all adjacencies are processed (post-order). The comment at line 279 correctly notes "dependencies first, target last." build_install_plan() iterates topo_order in forward order (dependencies installed before dependents). CORRECT.

### CHECK 6: 9-gate completeness — PASS (all 9 gates implemented)
- Gate 0 SIGNATURE: Ed25519 via xsec_ed25519_verify. IMPLEMENTED.
- Gate 1 INTEGRITY: SHA-256 content hash + per-file hashes. IMPLEMENTED.
- Gate 2 CAPABILITY: Install path prefix heuristic (XKABI stub). IMPLEMENTED (Sprint 3 scope).
- Gate 3 PROVENANCE: Repo list check. IMPLEMENTED (weak — see advisory below).
- Gate 4 VERSION: Downgrade protection. IMPLEMENTED.
- Gate 5 CONFLICT: File path collision check. IMPLEMENTED.
- Gate 6 SIZE: Disk quota check with 80% warn threshold. IMPLEMENTED.
- Gate 7 DEPENDENCY: Installed-registry dep check (WARN if uninstalled). IMPLEMENTED.
- Gate 8 POLICY: Glob-rule allow/block. IMPLEMENTED (but see P1-XPKG-02).

### CHECK 7: Install/remove atomicity — PARTIAL PASS
Install commits to registry only after successful extract_files(). On extract failure, registry is NOT updated (correct). On extract success + registry full, package is unregistered (correct). Remove marks REMOVING before file deletion, then compacts. UPGRADE has a rollback path. Gaps are documented in P1-XPKG-04.

### CHECK 8: Repository index integrity — PARTIAL PASS
parse_index_buffer() validates magic and format_version. Package count is clamped. Per-field bounds checks are present. No signature/HMAC on the index buffer itself (acknowledged as future XNET milestone). A malicious index file can inject arbitrary package entries — this is the expected gap for Sprint 3.

### CHECK 9: No libc — PASS
All files include only "xpkg.h" and "xsec.h". All string/memory operations use xpkg_* inline functions. No printf, malloc, memcpy, strcpy from libc. The xpkg_memcmp in xpkg.h is not constant-time (P3 advisory — not relevant for non-secret data comparisons, only relevant if content_hash comparison were on secret data, which it isn't).

### CHECK 10: Thread safety — PARTIAL PASS
- pkg_verify.c: s_key_lock protects writes. Reads are unlocked (P1-XPKG-01).
- repo_index.c: s_repo_lock protects all mutations and reads in search/get.
- refusal_gate.c: s_policy_lock protects rules.
- install.c: g_installed_registry.lock protects registry mutations. list_installed reads without lock after count snapshot (P2-XPKG-06).
- dep_resolve.c: Stateless (graph is caller-stack-allocated). SAFE.

## ADVISORY (P3)

### P3-XPKG-01: Gate 3 PROVENANCE is a proxy check, not a true provenance check
Gate 3 checks if at least one repo is enabled, not if the package actually originates from a repo. A locally-crafted package provided via the API passes Gate 3 as long as any repo is configured. True provenance requires matching the package name+hash against a repo index entry. Full fix belongs in XNET integration (Sprint 4).

### P3-XPKG-02: xpkg_gate_check short-circuits remaining gates on first FAIL, filling them with "Not evaluated"
This is correct behavior but the all_pass flag and the XPKG_ERR_GATE return both trigger on the first FAIL, so the install.c condition (P1-XPKG-03) means a single gate FAIL short-circuits before WARNs on later gates are visible. The gate report is still populated for logging, which is the right design.

### P3-XPKG-03: s_config in xpkg.c wipe on shutdown uses volatile write loop
This is the correct pattern for key material zeroization. However the config struct contains URLs (not secret) rather than keys. The actual sensitive material (signing keys) is wiped in xpkg_remove_trusted_key() via the volatile loop in pkg_verify.c. CONSISTENT.

### P3-XPKG-04: xpkg_memcmp is not constant-time
The inline xpkg_memcmp in xpkg.h short-circuits on first mismatch. If it is used to compare hash values derived from attacker-controlled input where timing is observable, this enables a timing oracle. In the current codebase, all xpkg_memcmp calls compare SHA-256 output against stored hashes — the attacker controls the input data, not a secret key. No timing oracle exists for the current use. Advisory only.
