These ARE present in the .ts sources but NOT in the compiled .js files.

### P2-5 [NEW FINDING]: go-notes export HTML — filename not sanitized
File: orange/go-notes/src/main.tsx, lines 323-329
`link.download = \`${active.title || "note"}.html\``
Note title is renderer-controlled. Certain browsers/OS restrict filenames but in Electron
the Blob download is renderer-controlled. A note title like `../../etc/passwd.html` could
write outside expected paths if the browser's download dialog uses it literally.
Remediation: Sanitize active.title: strip path separators and limit length before use in download.
Severity: P2. Not an RCE risk in sandbox, but a reliability and data-hygiene issue.

### P2-6 [NEW FINDING]: go-calendar event data from load() is cast without type-checking
File: orange/go-calendar/src/main.tsx, lines 253-256
```
if (data && Array.isArray((data as Record<string, unknown>).events)) {
  setEvents((data as Record<string, unknown>).events as EventItem[]);
}
```
The events array is cast to EventItem[] without field-level validation. A corrupt or
server-injected event with unexpected types (e.g., ev.reminderMinutes = "DROP TABLE") will flow
into the UI state and sync back to the server. No range checks on reminderMinutes, no length
cap on title/location, no validation on recurrence field.
Remediation: Add a runtime coercion/validation function before setEvents(). Clamp reminderMinutes
to 0-10080 (1 week). Cap title/location/recurrence to reasonable lengths (e.g., 256 chars).

### P2-7 [NEW FINDING]: go-drive folder creation is local-only (no server call) — no sanitization
File: orange/go-drive/src/main.tsx, lines 175-178
```
setItems([{ id: crypto.randomUUID(), name, kind: "folder", path: \`/\${name}\`, size: 0 }, ...items]);
```
The folder name is taken from newFolderName without sanitization. It is only added to local state,
never sent to the server via window.goDrive.upload() — so no immediate server-side injection.
However, the resulting item.path = `/${name}` is later passed to window.goDrive.deletePath(item.name)
and window.goDrive.download(item.name) which DO go through sanitizePath() in preload.ts.
If the sanitizePath() fix is compiled (.ts -> .js), this path is safe. Without the compile, it is not.
Remediation: Cap newFolderName to 128 chars, restrict to [a-zA-Z0-9 _-] in renderer before setItems.
This provides defense-in-depth even before the preload validation layer.

### P3-7 [NEW FINDING]: SyncClient list() path not encoded in URL
File: orange/common/sync-client.ts, line 27
`fetch(\`${this.baseUrl}/files/list/${path}\`)`
The path segment is not encoded. If the preload's sanitizePath() passes "/" it is safe,
but any path containing characters like spaces or unicode would form a malformed URL.
Remediation: `fetch(\`${this.baseUrl}/files/list/${encodeURIComponent(path)}\`)`
Note: for "/" specifically, encodeURIComponent("/") = "%2F" which may break routing on the server.
Use path === "/" ? path : encodeURIComponent(path) as a conditional.

## security.test.ts Known Regex Behaviors (2026-02-26)

### Phone regex \b word-boundary leaves leading '(' unconsumed

Regex: `/\b(?:\+?1[-.\s]?)?\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}\b/g`

Input: `"Call (555) 123-4567 for help"`
Output: `"Call ([REDACTED-PHONE] for help"` — the `(` is NOT consumed.

Reason: `\b` requires a word-to-non-word transition. The space before `(` is non-word; `(` is also
non-word; so `\b` cannot fire there. The match starts at `5` (word char, after `(` non-word char).
The regex matches `555) 123-4567` and leaves the `(` in place.

This behavior is CORRECT for the security purpose (the digits are redacted). Tests using
`not.toContain("(555) 123-4567")` pass. Tests using `toBe("Call [REDACTED-PHONE] for help")` FAIL
and must use `toBe("Call ([REDACTED-PHONE] for help")` instead.

Same behavior applies in: go-mail body integration test at line 1280 — but that test uses
`not.toContain` so it passes correctly without adjustment.

### sanitizeExportFilenameForTest must strip '..' after replacing path separators

The test helper `sanitizeExportFilenameForTest` in `orange/common/security.test.ts` only replaces
`[/\\:*?"<>|]` with `_`. For input `"../../etc/passwd"`, the `/` chars become `_` yielding
`".._.._etc_passwd"` — which still contains `..`. The test assertion `not.toContain("..")` fails.

Fix: Add `.replace(/\.\./g, "")` as a second step after the filesystem char replace.
After fix: `"../../etc/passwd"` → `".._.._etc_passwd"` → `"__etc_passwd"` — no `..`, no `/`.

### 1 MiB sanitizePii timeout

Running `sanitizePiiForTest` on a 1 MiB string triggers 4 regex passes, each over 1 million chars.
This reliably exceeds vitest's default 5-second test timeout. The test is functionally correct;
it simply needs a longer timeout. Fix: add `, 30_000` as the third argument to `it()`.
