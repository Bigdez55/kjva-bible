## FINDING 1 — P0: AES S-Box Constant-Time (aes_sbox_ct)

### ID: P0-S3-AES-01
### Status: OPEN — BLOCKING

**Code (aes_gcm.c lines 58-69):**
```c
static const uint8_t AES_SBOX[256] = { ... };  // line 34

static inline uint8_t aes_sbox_ct(uint8_t idx) {
    uint8_t result = 0u;
    uint32_t i;
    for (i = 0u; i < 256u; ++i) {
        uint32_t eq = (uint32_t)((uint8_t)i ^ idx);
        eq = (eq - 1u) >> 8u;
        result |= (uint8_t)(AES_SBOX[i] & (uint8_t)eq);
    }
    return result;
}
```

**Issue: Missing volatile on AES_SBOX.**
The source code comment at lines 22-32 states that volatile is used to prevent
dead-store elimination of the full-table scan. This claim is FALSE. The AES_SBOX
declaration at line 34 is `static const uint8_t AES_SBOX[256]` — no volatile.
The loop body accesses `AES_SBOX[i]` through a non-volatile const pointer.

**Optimizer risk:** clang -O2 with LTO can:
1. Observe that only one value of `eq` is nonzero (when i == idx).
2. Collapse the 256-iteration loop to a single `AES_SBOX[idx]` load.
3. This reinstates the cache-timing side-channel the fix was designed to eliminate.

**Mask arithmetic — CORRECT (no issue here):**
- `eq = (i ^ idx)` → 0 when match, 1..255 when mismatch.
- `(0u - 1u)` in uint32_t is 0xFFFFFFFF (defined unsigned underflow).
- `0xFFFFFFFF >> 8u` = 0x00FFFFFF. Cast to uint8_t = 0xFF. Correct mask.
- `(nonzero_eq - 1u) >> 8u` = high bits of a value <= 254, which after >>8 gives 0. Correct.
- The `|=` operation is branchless. No branch prediction leak from this logic.

**Dead-store of `result` — NOT an issue:**
- `result` is returned from the function. Return values cannot be dead-store-eliminated.

**Fix required:**
Option A: `static volatile const uint8_t AES_SBOX[256]` — forces all 256 reads.
Option B: Add `__asm__ volatile("" :: "r"(AES_SBOX[i]) : "memory")` inside the loop as a
  compiler barrier after the read.
Option C: Implement the described AES-NI hardware path (AESENC/AESENCLAST use
  the CPU's own S-box, inherently constant-time, far faster). The comment describes
  this as "mode 1" but zero AES-NI intrinsic code exists in the file.

### Status: OPEN — NON-BLOCKING

**gf_mul() (aes_gcm.c lines 84-96) contains `if` branches:**
```c
if (b & 1u) { result ^= a; }
```

These are data-dependent branches on the polynomial multiplication operands.

plaintext XOR'd with round keys). Not a direct key-byte branch, but correlated
with round-key material — P2 for now. A branch-free version using the xtime
conditional-move pattern should replace this before production crypto use.

## FINDING 3 — Hostname Validation

### ID: P1-S3-CERT-01 — Null byte in CN bypass
### Status: OPEN — BLOCKING

**x509.c dn_to_string() line 224:**
```c
out[pos_written++] = (char)sv[k];
```
No filtering of null bytes (0x00) in the DER string value. If sv contains an embedded
null byte, it is copied into the output buffer. The null byte then terminates the
C-string used in hostname matching.

**Attack scenario:**
- Attacker obtains cert where CN = "good.com\x00.evil.com" (length 18 in DER).
- DER sv_len = 18. dn_to_string copies all 18 bytes including the null.
- cn_buf contains "good.com\0.evil.com" as bytes, but as a C-string it reads "good.com".
- hostname_match("good.com", "good.com") → true.
- The cert's actual CN is NOT "good.com" — it's "good.com\x00.evil.com".
  A CA that issues such a cert can create a false match.

**Fix:** In dn_to_string(), skip or reject any byte where `sv[k] == 0x00`:
```c
if (sv[k] == 0x00u) { continue; }  /* Reject embedded nulls in CN */
```
Or: reject the entire cert if any null byte appears in CN.

### ID: P1-S3-CERT-04 — CN extraction from pre-formatted string
### Status: OPEN — BLOCKING

**x509.c xsec_x509_verify_hostname() lines 755-758:**
```c
for (i = 0u; subj[i] != '\0'; ++i) {
    if (subj[i] == 'C' && subj[i + 1u] == 'N' && subj[i + 2u] == '=') {
        cn_start = subj + i + 3u;
```

The function does NOT parse the raw DER to extract CN. It scans the pre-formatted
cert->subject string (produced by dn_to_string in the format "O=foo, CN=bar").

**Attack scenario:**
- Cert has RDNs in order: SET{O="trust me CN=victim.com"}, SET{CN="evil.com"}
- dn_to_string() writes O= first (if O appears first in DER), producing:
  "O=trust me CN=victim.com, CN=evil.com"
- The scanner finds first "CN=" at position 15 (inside the O= value).
- cn_start points to "victim.com, CN=evil.com".
- cn_buf extracted until comma: "victim.com".
- hostname_match("victim.com", "victim.com") → true!
- The attacker's cert with CN=evil.com is accepted for victim.com.

**Fix:** Scan for "CN=" only when preceded by start-of-string or ", " separator.
A robust fix is to store CN in a dedicated `char cn[XSEC_X509_NAME_MAX]` field in
xsec_x509_cert_t, populated directly during DER parsing, bypassing the formatted string.

### ID: PASS — Wildcard in non-leftmost position
### Status: PASS

hostname_match() only checks pattern[0] == '*'. Non-leftmost wildcards (e.g., "foo.*.com")
do not trigger the wildcard branch and fall through to byte-by-byte exact match.
"foo.*.com" never equals "foo.bar.com" in exact match. PASS.

Multi-wildcard "*.*.com": pattern[0]=='*', pattern[1]=='.', enters wildcard branch.
wild_suffix = "*.com". Comparison of '*' against first byte of remaining hostname suffix
will fail unless hostname also starts with '*'. SAFE — inner wildcards treated as literals.

RFC 6125 §6.4.3 compliance: single-label wildcard only, must be leftmost label. PASS.

## FINDING 4 — PAL-Aether / XKABI TOCTOU

### ID: P1-S3-KABI-01 — xkabi_cap_derive revoke window
### Status: OPEN — BLOCKING

**xkabi_capabilities.c xkabi_cap_derive() lines 466-568:**

The function has a double-lock pattern:
1. Calls `xkabi_cap_verify(parent_handle, XKABI_RIGHT_DERIVE)` at line 481.
   verify() acquires lock, checks all conditions, releases lock.
2. Re-acquires lock at line 489 to read parent_rights, parent_oid, parent_ld.
3. Releases lock at line 493.
4. Calls xkabi_cap_create() (which acquires lock again internally) at line 509.

**Race window:** Between steps 1 and 2, another thread can call xkabi_cap_revoke()
on the parent handle. revoke() acquires lock, sets cap->revoked = true, increments
cap->generation. By the time derive's step 2 reads parent_rights, the parent is
revoked — but step 2 does NOT check cap->revoked. It reads rights unconditionally
from a revoked cap.

**Result:** A child capability is created from a revoked parent. The child is valid
(not revoked). This violates the monotonicity invariant: revoking a cap should
prevent all future derivations from it.

**Fix:** Inside the re-acquired lock at step 2, check `cap->revoked` before reading
rights. Return 0 (failure) if revoked:
```c
pal_spin_lock(&g_handle_table.lock);
if (g_handle_table.caps[p_slot].revoked) {
    pal_spin_unlock(&g_handle_table.lock);
    return 0u;  /* Parent revoked in race window */
}
uint32_t parent_rights = g_handle_table.caps[p_slot].rights;
...
pal_spin_unlock(&g_handle_table.lock);
```
