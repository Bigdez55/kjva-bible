## AUDIT 5: WAL Fix Correctness Analysis

### Question
Is `txn->snapshot_lsn` the right snapshot to use in the btree_lookup at commit time?
Will the lookup return NOT_FOUND for a value that was just inserted by xstore_txn_put()?

### Finding: SAFE — but with an important caveat

**Sequence of events:**

1. `xstore_txn_begin()` sets `txn->snapshot_lsn = store->next_lsn` (the current LSN watermark).
2. `xstore_txn_put()` calls `xstore_btree_insert(store, txn, key, val)` which:
   - Atomically increments `store->next_lsn` (via pal_atomic_add) to get a new LSN.
   - Physically writes the key+value into the leaf page in the in-memory page cache.
   - Marks the page dirty. The value is immediately visible in cache.
3. At `xstore_txn_commit()`, the WAL fix calls:
   `xstore_btree_lookup(store, txn->snapshot_lsn, &k, &vout)`

**Critical observation — btree.c line 510:**
```c
(void)snapshot_lsn; /* Full per-version chain in Sprint 5 */
```
The `snapshot_lsn` parameter is intentionally discarded in Sprint 4. The lookup performs
a pure key search in the B+ tree; if the key is physically present in the leaf page
(which it is, since xstore_btree_insert wrote it to the in-memory cache), the lookup
returns the value unconditionally. The visibility gate is a deferred Sprint 5 feature.

**Conclusion:** The lookup WILL find the just-inserted value in Sprint 4. The fix is
functionally correct under the Sprint 4 assumption that btree_lookup ignores LSN visibility.

**Forward risk (P1 — Sprint 5 forward blocker):**
When Sprint 5 activates per-version chains and makes `snapshot_lsn` meaningful, the
WAL commit lookup using `txn->snapshot_lsn` (set at transaction BEGIN, before the insert)
may fail to see the insert if the insert's LSN > snapshot_lsn. In that scenario, val_len
stays 0 in the WAL entry, and crash recovery reverts to best-effort (no data value in
WAL for replay). This is the same scenario RSS-REL-01 was designed to fix — the fix
will regress when Sprint 5 enables LSN visibility. The fix must be re-evaluated at
Sprint 5 sprint boundary. The correct Sprint-5-compatible approach would use a sentinel
snapshot of UINT64_MAX for the WAL lookup, or store the value in the write set at
put() time rather than at commit() time.
