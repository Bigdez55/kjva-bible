## II. DETAILED FAILURE MODE ANALYSIS

### A. XSTORE (Sprint 4) -- Score: 7/10

#### Circuit Breakers

**RA-XSTORE-01 [P2]: WAL write_page failure has no retry.**
- Location: `wal.c:213` -- `store->write_page()` called once; failure returns immediately.
- Scenario: Transient NVMe I/O error during checkpoint superblock write.
- Impact: Checkpoint fails, WAL ring remains full, subsequent commits fail with WAL_FULL.
- Mitigation: Add single retry with 1ms delay on write_page failure before propagating error.

**RA-XSTORE-02 [P3]: WAL replay ignores btree insert/delete errors (best-effort).**
- Location: `wal.c:162-165` -- `(void)xstore_btree_insert()` and `(void)xstore_btree_delete()`.
- Scenario: Replay of a committed txn fails because a page is corrupt. The error is silently swallowed.
- Impact: Committed data may be lost without any indication. Replay returns XSTORE_OK regardless.
- Mitigation: Count failed replay operations; if count > 0, return XSTORE_ERR_CORRUPT or log warning and set a "degraded replay" flag.

**RA-XSTORE-03 [P2]: WAL replay txn overflow is silent.**
- Location: `wal.c:121-123` -- `if (committed_count < WAL_REPLAY_MAX_TXNS)` silently drops txn IDs beyond 256.
- Scenario: System crashed with > 256 in-flight committed transactions in the WAL window.
- Impact: Transactions beyond the 256th committed txn are not replayed. Data loss for those transactions.
- Mitigation: Either increase WAL_REPLAY_MAX_TXNS or return an error when the limit is exceeded.

#### Retry Logic

**RA-XSTORE-04 [GOOD]: OCC auto-retry on XSTORE_ERR_CONFLICT.**
- Location: `xstore.c:169,177-192` -- `XSTORE_RETRY_MAX = 3`, convenience API retries on conflict.
- Assessment: Well-designed. Three retries with fresh snapshot on each attempt. Surfaces error after exhaustion.

**RA-XSTORE-05 [GOOD]: WAL_FULL auto-checkpoint and retry.**
- Location: `mvcc.c:209-216` -- catches WAL_FULL, checkpoints, retries append once.
- Assessment: Correct pattern. Single retry is reasonable since checkpoint should drain the ring.

**RA-XSTORE-06 [P2]: COMMIT record WAL append has no WAL_FULL handling.**
- Location: `mvcc.c:229-233` -- The COMMIT WAL entry append does NOT attempt auto-checkpoint on WAL_FULL.
- Scenario: Write entries succeeded (with auto-checkpoint), but the COMMIT entry itself hits WAL_FULL.
- Impact: Transaction is aborted (writes are in WAL but no COMMIT record). On replay, these entries are correctly discarded (uncommitted), but the caller loses work.
- Mitigation: Apply the same auto-checkpoint-and-retry pattern to the COMMIT append.

#### State Recovery

**RA-XSTORE-07 [GOOD]: WAL replay on xstore_open.**
- Location: `xstore.c:93-99` -- WAL replay is called during open. If replay fails, store is NOT marked open.
- Assessment: Correct. The store refuses to open with potentially inconsistent state.

**RA-XSTORE-08 [GOOD]: Superblock CRC validation on open.**
- Location: `xstore.c:82-85` -- CRC32C check of superblock bytes [8, PAGE_SIZE).
- Assessment: Corrupt superblock correctly returns XSTORE_ERR_CORRUPT.

**RA-XSTORE-09 [P3]: xstore_close checkpoint failure returns error but store is already closed.**
- Location: `xstore.c:146-153` -- If final checkpoint fails, store->open is set to false but error is returned.
- Impact: Caller knows the close failed, but the store is now in a "closed but not checkpointed" state. On next open, WAL replay will recover, so this is safe but not clean.

#### Partial Failure Handling

**RA-XSTORE-10 [GOOD]: WAL replay failure prevents store open.**
- Scenario: XSTORE starts, superblock loads, but WAL replay fails.
- Location: `xstore.c:96-99` -- `store->open = false; return rc;`
- Assessment: Correct. System reports error and refuses to open with inconsistent state.

### B. XBLOB (Sprint 4) -- Score: 7/10

#### Circuit Breakers

**RA-XBLOB-01 [P3]: No circuit breaker between XBLOB and XSTORE consumers.**
- Scenario: XBLOB's backing I/O fails. Consumers (XPKG, XMIND) have no fallback.
- Impact: Consumer operation fails with XBLOB_ERR_IO. No cached response or degraded mode.
- Mitigation: XPKG should cache the last-known package index locally. XMIND should report "AI offline" via notification.

#### Retry Logic

**RA-XBLOB-02 [P1]: No retry on blob write failure.**
- Location: `blob.c` (via xblob_io_t.write callback) -- write failures are not retried.
- Scenario: Transient NVMe error during xblob_put(). The blob is half-written.
- Impact: Index entry may reference a partially written blob. On subsequent xblob_get(), SHA-256 verification will detect the corruption (XBLOB_ERR_CORRUPT), but the index entry remains.
- Mitigation: Write blob body FIRST, verify by reading back and checking SHA-256, THEN insert index entry. Retry write once on failure.

#### State Recovery

**RA-XBLOB-03 [GOOD]: GC log replay on xblob_open.**
- Location: `gc.c:65-103`, `xblob.c:75-79` -- Two-pass CRC-validated replay rebuilds refcounts.
- Assessment: Solid crash recovery. CRC mismatch entries are skipped. Refcounts are restored from authoritative GC log.

**RA-XBLOB-04 [GOOD]: Index rebuild from occupied slots on open.**
- Location: `xblob.c:87-101` -- Statistics and bitmap are rebuilt from the index scan.
- Assessment: Self-healing pattern. Even if stats were wrong before crash, they are recalculated.

**RA-XBLOB-05 [P2]: GC compaction crash = probe sequence corruption risk.**
- Location: `gc.c:173-195` -- During compaction, all entries are set to FREE, then re-inserted in chunks.
- Scenario: Power loss between clearing entries (line 176-179) and re-inserting (line 184-189).
- Impact: All entries in the current chunk are lost (state=FREE, not yet re-inserted). GC log does not track compaction operations -- only FREE events.
- Mitigation: Log compaction start/end to GC log. On replay, if compaction was in-flight, rebuild index from blob headers by scanning the storage region.

**RA-XBLOB-06 [GOOD]: SHA-256 integrity verification on read.**
- Assessment: xblob_get() verifies the retrieved data's SHA-256 against the requested hash. Returns XBLOB_ERR_CORRUPT on mismatch. This prevents serving corrupt data.

### C. XMIND (Sprints 6-8) -- Score: 4/10

#### Circuit Breakers

**RA-XMIND-01 [P1]: No circuit breaker between XMIND and XSHELL.**
- Scenario: xmind_init() or xmind_alloc_state() fails. XSHELL has no degraded AI mode.
- Location: `xshell.c` -- No reference to XMIND anywhere. Shell init does not depend on XMIND, but there is no notification mechanism to tell the user AI is offline.
- Impact: If the boot_chain stage 9 (AI) fails, it is logged as a warning, but XSHELL shows no user-facing indicator.
- Mitigation: XSHELL should query xchain_ai_ready() and display "AI: offline" in the topbar when false.

**RA-XMIND-02 [P1]: xm_heap_free() is a no-op -- CONFIRMED STILL OPEN.**
- Location: `xmind.h:109-113` -- `static inline void xm_heap_free(void *ptr) { (void)ptr; }`
- Impact: Every xmind_shutdown() + xmind_alloc_state() cycle leaks ~336 MB of PAL pages.
- Scenario: Service crash -> supervisor restart -> xmind re-init -> leak accumulates.
- After 3-4 crash/restart cycles, system runs out of physical memory.
- Blast radius: System-wide. PAL page pool exhaustion affects ALL subsystems.
- Mitigation: Implement PAL slab allocator with handle-per-allocation tracking. Sprint priority: P0.

#### Retry Logic

**RA-XMIND-03 [P2]: Weight loader has no retry on PAL page allocation failure.**
- Location: `weights_loader.c:503-511` -- Per-layer weight allocation loop. If wl_alloc_pages() returns NULL for any layer, the remaining layers get NULL pointers but the function continues.
- Impact: Forward pass with NULL weight pointers = undefined behavior (likely crash on dereference).
- Mitigation: Check each wl_alloc_pages() return; on failure, free all prior allocations and return XMIND_ERR_NOMEM.

**RA-XMIND-04 [P2]: XBLOB weight loader (weights_loader.c:550-599) has no retry on xblob_get failure.**
- Location: `weights_loader.c:575-579` -- If xblob_get fails, returns XMIND_ERR_IO with no retry.
- Impact: Transient I/O error = AI permanently offline until manual restart.
- Mitigation: Retry xblob_get once with a brief delay.

#### State Recovery

**RA-XMIND-05 [GOOD]: OOM partial-cleanup in xmind_alloc_state.**
- Location: `xmind.c:180-201` -- goto oom label frees all previously allocated buffers and nulls pointers.
- Assessment: Correct pattern. Unfortunately undermined by RA-XMIND-02 (free is a no-op).

**RA-XMIND-06 [P3]: XM_ASSERT triggers infinite HLT loop.**
- Location: `xmind.h:65-71` -- `for(;;) { __asm__ volatile("hlt"); }`
- Impact: Assertion failure in XMIND = process hangs forever. GENSD supervisor cannot detect this (process is still "alive" in a HLT loop, but not responding).
- Mitigation: XM_ASSERT should call a process-exit syscall so GENSD detects the failure and restarts.

#### Partial Failure Handling

**RA-XMIND-07 [P2]: Tokenizer init failure does not prevent forward pass.**
- Scenario: xmind_tokenizer_load() fails. The tokenizer falls back to byte-level mode.
- Location: `tokenizer.c:425-432` -- Returns XMIND_ERR_INVAL, sets no global error flag.
- Assessment: The fallback to byte-level mode is CORRECT graceful degradation. However, there is no notification to the user that tokenization quality is degraded.
- Mitigation: Set a global flag and expose via xmind_get_global()->tokenizer_mode for UI display.

### D. XSHELL (Sprint 5) -- Score: 5/10

#### Circuit Breakers

**RA-XSHELL-01 [P1]: XSHELL_SESSION_FALLBACK is dead code -- CONFIRMED STILL OPEN.**
- Location: `xshell.h:277` defines XSHELL_SESSION_FALLBACK=1. `greeter.c:50` names it "Fallback TTY".
- But: No code path in boot_chain, GENSD, or XSHELL ever sets session_type to XSHELL_SESSION_FALLBACK.
- Impact: If XSHELL fails to initialize (xframe_init, xframe_app_create, any sub-component), the system halts permanently. There is NO text-mode fallback.
- Blast radius: System-wide. User has no recovery path except power cycle + ISO reinstall.

**RA-XSHELL-02 [P2]: xshell_init sub-component failure does not clean up prior components.**
- Location: `xshell.c:161-228` -- If dock_create fails (line 183), topbar was already created but is not destroyed. If launcher_create fails, both topbar and dock leak.
- Impact: Memory leak on init failure. On retry (if supervisor restarts), the leaked memory accumulates.
- Mitigation: Add cleanup labels (goto pattern) to destroy all previously created sub-components on failure.

**RA-XSHELL-03 [P3]: No mechanism for XSHELL to report degraded state.**
- Assessment: XSHELL has no health endpoint, no status accessor, no way for the supervisor to query "is the shell functional?" beyond process liveness.
- Mitigation: Expose xshell_is_healthy() that checks xframe_app validity and scene_root existence.

#### State Recovery

**RA-XSHELL-04 [P2]: xshell_run exits silently when xframe_run returns.**
- Location: `xshell.c:243` -- xframe_run() returns, g_shell.running = 0, function returns XSHELL_OK.
- Impact: If xframe_run crashes or exits unexpectedly, XSHELL reports success. GENSD will not know to restart.
- Mitigation: xshell_run should detect unexpected xframe_run exit and return XSHELL_ERR_INVAL so the caller (or supervisor) can react.

### E. GENSD Supervisor (Sprint 1, audited current state) -- Score: 5/10

#### Health Probes

**RA-GENSD-01 [P1]: HTTP health probe is STILL A STUB -- 5+ sprints overdue.**
- Location: `supervisor.c:158-171` -- `supervisor_probe_http()` returns HEALTH_OK always.
- Impact: Services that appear alive (process running) but are functionally dead (hung, deadlocked) are never detected.
- Comment says "Sprint 2" but we are now at Sprint 8+.

**RA-GENSD-02 [P1]: Blocking restart suspends ALL supervision.**
- Location: `supervisor.c:213-214` -- `pal_thread_sleep_ns(delay_ns)` blocks the supervisor thread.
- Impact: During a 30-second backoff sleep for one service, ALL other service health checks are suspended. A second service failure during this window goes undetected.
- Mitigation: Replace blocking sleep with non-blocking timer (pal_timer_create). Track per-service "next_restart_at" timestamp and skip restart in supervisor_tick() until that time arrives.

#### Crash Loop Detection

**RA-GENSD-03 [P2]: No crash-loop detection or maximum restart limit.**
- Location: `supervisor.c:189-225` -- restart_count is incremented but never compared to a maximum.
- Impact: A service that immediately crashes on start will be restarted infinitely with exponentially growing delays (up to 30s), consuming resources and polluting the journal.
- Mitigation: After N restarts (e.g., 10), mark service as "crash-looping" and deactivate. Emit a user notification via XSHELL.

**RA-GENSD-04 [P3]: Watch table full silently drops services.**
- Location: `supervisor.c:313-317` -- When count >= SV_MAX_WATCHED (64), new registrations are logged but otherwise silently dropped.
- Impact: Service registered late may never be supervised.

### F. boot_chain (Sprint 8) -- Score: 6/10

#### Critical vs. Non-Critical Classification

**RA-BOOT-01 [GOOD]: Clear critical/non-critical stage classification.**
- Critical stages (kernel_ready, security KAT, display, compositor, shell): failure aborts boot.
- Non-critical stages (storage, network, packages, AI): failure logs warning and continues.
- Assessment: Correct design. AI and network failures do not prevent the system from booting.

**RA-BOOT-02 [P1]: xchain_storage() NEVER FAILS -- CONFIRMED STILL OPEN.**
- Location: `boot_chain.c:116-119` -- Even when the XBLOB region is not found in the memory map, the function returns PAL_OK with a "proceeding (unverified)" log.
- Impact: Storage subsystem may be operating on invalid memory. Subsequent XSTORE/XBLOB operations could corrupt arbitrary memory.
- Mitigation: Return PAL_ERR_INVAL when the region is not found. Mark storage as degraded.

**RA-BOOT-03 [P1]: Shell IPC failure = permanent system halt -- CONFIRMED STILL OPEN.**
- Location: `boot_chain.c:591-596` -- xchain_shell() failure returns to xchain_run(), which returns non-PAL_OK to gensd_start(), which enters infinite halt loop (main.c:534-536).
- Impact: NO TTY fallback. NO serial console. NO recovery path. User must power cycle.
- Mitigation: On shell failure, attempt serial console output. Set session_type = XSHELL_SESSION_FALLBACK. Start a minimal text-mode shell for diagnostics.

**RA-BOOT-04 [P2]: No PAL init stage in xchain_run.**
- Location: `boot_chain.c:533-613` -- Stage 1 (PAL init) is assumed to be done by boot_sequence.c. If PAL init failed silently, all subsequent stages will fail cryptically.
- Mitigation: Add an explicit PAL readiness check at the start of xchain_run().

#### Graceful Degradation

**RA-BOOT-05 [P2]: No intermediate degradation between "full system" and "halt."**
- The boot chain has exactly two outcomes: full boot success or fatal halt.
- Missing: "core-only mode" (compositor + shell, no AI, no network, no packages).
- Missing: "emergency recovery mode" (serial console + diagnostics).
- The progressive degradation ladder from L3 (network offline) to L4 (storage degraded) to L5 (core only) to L6 (emergency TTY) has NO implementation.

## III. CHAOS SCENARIO EVALUATIONS

### Chaos 1: Power failure during WAL write

**Hypothesis:** "When power fails during WAL write, XSTORE database remains consistent on next open."

**Analysis:**
- WAL entry write (wal.c:76) copies data to ring slot, assigns LSN, computes CRC.
- If power fails BEFORE sync() (wal.c:217), the WAL entry may have partial data.
- On recovery, xstore_wal_replay() validates each entry with CRC (wal.c:117-118). Partially written entries fail CRC and are skipped.
- If power fails AFTER the entry is written but BEFORE the COMMIT record, the transaction is correctly discarded (no matching COMMIT).
- If power fails AFTER COMMIT but BEFORE checkpoint, the committed data is replayed.

**Verdict: SAFE.** The WAL design correctly handles power failure at every point in the write sequence. The only risk is RA-XSTORE-03 (silent txn overflow during replay), which is a capacity limit, not a correctness issue.

### Chaos 2: OOM during XMIND weight load

**Hypothesis:** "When OOM occurs during weight loading, the system recovers gracefully."

**Analysis:**
- xmind_alloc_state() (xmind.c:140-201): OOM triggers goto oom, which calls xm_heap_free() on all buffers.
- BUT xm_heap_free() is a no-op (xmind.h:109-113). Memory is leaked.
- weights_loader.c:503-511: Per-layer allocation does NOT check return values. NULL weight pointers proceed.
- On subsequent xmind_forward() with NULL weight pointers: UNDEFINED BEHAVIOR (dereference NULL -> crash or page fault).

**Verdict: UNSAFE.** OOM during weight load produces NULL weight pointers that will cause a crash during the first forward pass. The OOM cleanup path is structurally correct but functionally broken due to xm_heap_free no-op. After the crash, GENSD supervisor will restart, but the leaked memory from the first attempt makes a second OOM MORE likely, creating a death spiral.

### Chaos 3: XSHELL segfault

**Hypothesis:** "When XSHELL segfaults, GENSD restarts it within 5 seconds."

**Analysis:**
- GENSD supervisor checks process liveness via running flag (supervisor.c:132-143).
- A segfault in XSHELL (page fault) would be caught by the XENOS kernel fault handler, which would terminate the process and set running=false.
- supervisor_tick() runs every 1 second (main.c:253), detects running=false, calls supervisor_restart().
- supervisor_restart() sleeps for restart_delay_ms (default: 1000ms) then sets running=true (stub).
- Total restart time: 1s (detection) + 1s (backoff) = ~2 seconds on first crash.

**Verdict: PARTIALLY SAFE.** The detection and restart path works for the first crash. However:
1. HTTP health probe is stubbed (RA-GENSD-01), so a hung XSHELL is never detected.
2. Blocking restart suspends all supervision (RA-GENSD-02).
3. No crash-loop limit (RA-GENSD-03) -- infinite restart loop possible.
4. Restart is a stub (sets running=true, no actual process spawn) -- Sprint 2+ integration needed.

### Chaos 4: XBLOB GC crash

**Hypothesis:** "When XBLOB GC crashes mid-compaction, blobs are recoverable."

**Analysis:**
- GC scan (gc.c:120-145): Frees refcount=0 blobs by tombstoning index entries. Each free is logged to GC log. This phase is safe -- GC log replay will restore the state.
- Compaction (gc.c:173-195): Sets ALL entries to FREE, then re-inserts in chunks of 256.
- If crash occurs DURING compaction (between clear and re-insert), entries in the current chunk are lost.
- GC log only records FREE events, NOT compaction re-insert events.

**Verdict: UNSAFE for compaction phase.** Blobs whose index entries were cleared but not re-inserted are orphaned. The blob DATA still exists in storage, but the index entry is gone. Recovery requires a full storage scan (not implemented) to rebuild index from blob headers.

### Chaos 5: XNET stack corruption

**Hypothesis:** "When XNET stack corruption occurs, XSEC audit log captures it."

**Analysis:**
- XSEC audit log exists (sec/xsec/audit/) and logs security events.
- XNET has no explicit corruption detection mechanism. TCP checksums validate transport integrity, but there is no internal consistency check for the XNET state machine.
- If XNET's internal state (TCP TCB, ARP table, DNS cache) is corrupted, the corruption would manifest as protocol errors, but XSEC audit does not monitor XNET internal state.

**Verdict: NO DETECTION.** XNET corruption would not be captured by XSEC audit. A deep-health check for XNET (validating internal data structure invariants) is missing.

## V. RESILIENCE PATTERNS -- WHAT WORKS

1. **XSTORE WAL crash recovery**: Two-phase replay (collect committed txns, then re-apply) with CRC validation on every entry. Partial transactions correctly discarded.
2. **XSTORE OCC retry**: Convenience API retries conflicts 3 times with fresh snapshot.
3. **XSTORE WAL_FULL auto-checkpoint**: Single-retry pattern in mvcc.c catches ring saturation.
4. **XBLOB GC log replay**: CRC-validated, sequence-ordered replay rebuilds refcounts after crash.
5. **XBLOB SHA-256 on read**: Content addressing provides built-in corruption detection.
6. **XBLOB bitmap rebuild on open**: Stats reconstructed from index scan, self-healing.
7. **XMIND OOM pattern**: Goto-based cleanup frees all prior allocations (correct structure, broken by no-op free).
8. **XMIND config validation**: 10+ boundary checks in xmind_init prevent invalid models.
9. **GENSD backoff**: Exponential with 30s cap and overflow protection.
10. **boot_chain critical/non-critical**: AI and network failures don't prevent boot.
11. **gen_iso SHA-256 manifest**: Per-component integrity with required flag.

---

## VI. RESILIENCE PATTERNS -- WHAT FAILS

1. **No progressive degradation ladder**: System goes from "full" to "halt" with nothing in between.
2. **Supervisor blocking sleep**: One service restart blocks ALL monitoring.
3. **HTTP health probe stub**: 5+ sprints overdue. Hung services are invisible.
4. **xm_heap_free no-op**: Creates memory leak death spiral on service restart.
5. **SESSION_FALLBACK dead code**: TTY fallback defined but never wired.
6. **xchain_storage never fails**: Proceeding with unverified storage region.
7. **Weight loader unchecked allocs**: NULL dereference on OOM during weight load.
8. **XBLOB compaction not crash-safe**: Index entries can be lost during compaction.

### Experiment 1: XSTORE Power-Loss Durability

```
CHAOS EXPERIMENT: XSTORE-POWER-LOSS
==========================
Hypothesis: "When power fails at any point during xstore_put(), the database
            is consistent and recoverable on next xstore_open()."
Steady State: XSTORE open, 1000 keys inserted, checkpoint completed.
Fault Injection: Kill process at random point during xstore_put() (insert 1001st key).
                 Vary kill timing across: wal_append, btree_insert, txn_commit, sync.
Observation: On re-open, verify:
             - xstore_open() returns XSTORE_OK
             - Key count is either 1000 (txn rolled back) or 1001 (txn committed)
             - All 1000 original keys are readable with correct values
             - No XSTORE_ERR_CORRUPT on any get()
Duration: 100 iterations with randomized kill timing.
Abort Conditions: Any iteration produces XSTORE_ERR_CORRUPT on a previously committed key.
Recovery Verification: Full key scan after each iteration confirms data integrity.
```
