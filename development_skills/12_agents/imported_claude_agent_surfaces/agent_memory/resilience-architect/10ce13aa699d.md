## 1. XNET RESILIENCE ANALYSIS

### 1.1 TCP Connection Table Full (tcp.c:308-313)

**Question**: What happens when the TCP connection table (128 slots) is full?

**Finding**: `tcb_alloc()` returns NULL when all 128 TCBs are in use.

- **xnet_tcp_connect** (line 934-937): Returns `XNET_ERR_NOMEM`. GOOD.
- **xnet_tcp_listen** (line 999-1002): Returns `XNET_ERR_NOMEM`. GOOD.
- **xnet_tcp_input SYN handling** (line 670-673): Returns `XNET_ERR_NOMEM` when a SYN arrives for a listener but no TCB slot is available. GOOD -- but no RST is sent to the client.

**Bug F-26 (P2): No RST sent when TCB table is full on incoming SYN**
At line 670-673, when `tcb_alloc()` fails for an incoming SYN, the function returns `XNET_ERR_NOMEM` without sending RST. The remote peer will retransmit SYNs until it gives up. The correct behavior per RFC 793 is to send RST so the client fails fast.

**Bug F-27 (P3): No backpressure metric for TCB pool utilization**
There is no way for upper layers to query how many TCBs are in use vs. capacity. This makes it impossible to implement load shedding before the table is full.

### 1.2 TCP Listen Backlog Full (tcp.c:663-666)

When the backlog is full, the SYN is silently dropped (line 663-666). This is correct per RFC behavior -- the client will retransmit and may succeed when a slot opens. PASS.

### 1.3 TCP Retransmit Exhaustion (tcp.c:1291-1297)

After 12 retransmits, the connection is aborted (state set to CLOSED, TCB zeroed). This is correct and well-implemented. The exponential backoff doubles RTO on each retransmit. PASS -- gold standard for exhaustion handling.

### 1.4 DNS Cache Full (dns.c:228-241)

**Finding**: `dns_cache_alloc()` implements oldest-entry eviction when the cache is full. It scans all 128 entries, finds either a free slot or the entry with the smallest `cached_at_ns`, and returns it.

**Verdict**: PASS. This is correct LRU-like eviction (actually oldest-inserted, not oldest-accessed, but acceptable for a DNS cache). The evicted entry is overwritten in-place.

### 1.5 DNS Single-Query Bottleneck (dns.c:259, 413-423)

**Known Bug (from prior FMEA)**: `g_dns_pending` is a single global struct. Only one DNS query can be in-flight at a time. A second caller to `dns_do_resolve()` will overwrite the pending state, corrupting the first query.

**Severity**: P0 (was already flagged). No change.

### 1.6 ARP Failure After 3 Retries (arp.c:461-498)

**Finding**: ARP does not retry at all from the resolution path itself. The flow is:
1. `xnet_arp_resolve()` creates a PENDING entry and sends one ARP request.
2. Returns `XNET_ERR_AGAIN` to the caller.
3. `xnet_arp_expire()` (called every ~1s by `xnet_tick()`) evicts PENDING entries older than 10 seconds.
4. When evicted, all queued packets are dropped with a console log.

There is no automatic ARP retry mechanism. If the first ARP request is lost, the resolution will fail after 10 seconds.

**Bug F-28 (P2): No ARP retry for PENDING entries**
The `xnet_arp_expire()` function only evicts PENDING entries after timeout. It does not re-send ARP requests for entries that are still PENDING. In lossy networks, a single dropped ARP request means a 10-second timeout and packet loss. The fix is to add retry logic in `xnet_arp_expire()`: for PENDING entries that have been waiting > 2 seconds, re-send ARP request (up to 3 times).

### 1.7 SWIM Gossip DEAD Node Handling (gossip.c:720-742)

**Finding**: When a node is declared DEAD:
- `m->info.state` is set to `GOSSIP_NODE_DEAD` (line 726)
- `m->active` is set to `false` (line 727)
- `g_member_count` is decremented (line 728)
- A DEAD event is queued for infection-style dissemination (lines 731-737)

The DEAD node's entry in the `g_members[]` array is deactivated (`active = false`) but NOT zeroed. This means the slot is available for reuse by `gossip_alloc_member()`, but the stale data remains. This is correct -- the slot will be fully overwritten on next allocation.

**However**, `gossip_select_random()` at line 216 correctly skips DEAD nodes:
```c
if (g_members[i].info.state == GOSSIP_NODE_DEAD) continue;
```

**Verdict**: PASS. DEAD nodes are properly removed from active membership and the dissemination is correct.

### 1.8 XNET_INIT Failure Recovery (netif.c:456-467)

**Finding**: `xnet_init()` only zeros the interface table and sets `g_initialized`. If it fails (only possible failure: already initialized, returns `XNET_ERR_EXISTS`), no state is corrupted.

However, there is no cascading initialization of sub-modules (TCP, DNS, ARP, gossip). Each sub-module uses static initialization (`PAL_SPINLOCK_INIT`, zero-initialized global arrays). This means:
- If `xnet_init()` fails, upper layers should not call any XNET functions.
- There is no partial initialization state to recover from.
- `xnet_tick()` (line 487-492) calls all sub-module ticks unconditionally.

**Bug F-29 (P3): xnet_tick() does not check g_initialized**
`xnet_tick()` calls `xnet_arp_expire()`, `xnet_tcp_timer_tick()`, `xnet_dns_cache_expire()`, and `xnet_gossip_tick()` without checking if the stack has been initialized. While each sub-module's tick is safe on zeroed state, this is a defense-in-depth gap.

## 2. XPKG RESILIENCE ANALYSIS

### 2.1 Mid-Installation Failure Rollback (install.c:216-326)

**Finding**: The install workflow has 6 steps:
1. Parse archive -> failure returns error, no state change. CLEAN.
2. Gate check -> failure returns error, no state change. CLEAN.
3. Dependency resolution -> failure returns error, no state change. CLEAN.
4. File extraction -> failure returns error at line 292-295. **PROBLEM**: If extraction fails partway, already-extracted files are NOT cleaned up. However, since Sprint 3 uses simulated extraction (no VFS), this is acceptable now but will be CRITICAL in Sprint 4.
5. Registry registration -> failure returns `XPKG_ERR_CAPACITY`. No files to clean up (simulated). CLEAN.
6. Post-install hook -> always succeeds (stub). CLEAN.

**Verdict**: In Sprint 3 (simulated VFS), the rollback gap is cosmetic. In Sprint 4 (real VFS), this becomes P1.

**Bug F-30 (P1-DEFERRED): No file cleanup on partial extraction failure**
When `extract_files()` fails at line 292, already-extracted files remain on disk. Sprint 4 MUST implement a rollback mechanism: either (a) extract to a staging directory then atomic rename, or (b) maintain a manifest of extracted files for cleanup on failure.

### 2.2 Upgrade Rollback (install.c:386-477)

**Finding**: The upgrade workflow is well-designed:
1. Parse and verify new archive BEFORE touching the old version. GOOD.
2. Remove old version under lock (lines 426-442). GOOD.
3. Install new version (line 454). If this fails:
   - Rollback: re-register old version at line 458-466.
   - **BUT**: Old version is registered with state `XPKG_STATE_BROKEN` (line 463).

**Bug F-31 (P1): Upgrade rollback marks old version BROKEN instead of INSTALLED**
This was flagged in the prior FMEA and the code at line 463 still reads:
```c
old_pkg_copy.state = XPKG_STATE_BROKEN;
```
The old version was working before the upgrade attempt. Marking it BROKEN prevents it from being used or upgraded again. It should be `XPKG_STATE_INSTALLED`.

### 2.3 Corrupted Repository Index

**Finding**: `repo_index.c` parses repository indices from buffer. If the index is corrupted:
- `xpkg_repo_parse()` validates magic and version fields.
- On parse failure, returns `XPKG_ERR_FORMAT`.
- The repo is not added to the repo list.

**Verdict**: PASS. Corrupted indices are rejected cleanly.

### 2.4 All Trusted Keys Revoked (pkg_verify.c)

**Finding**: If `g_trusted_key_count == 0`, then `xpkg_verify_signature()` will iterate zero keys, find no match, and return `XPKG_ERR_SIGNATURE`. All packages will fail the signature gate.

**Verdict**: PASS. This is correct behavior -- if no keys are trusted, no packages should install. However, there is no mechanism to bootstrap from this state (no emergency key provisioning).

### 2.5 Unresolvable Dependency Conflict (dep_resolve.c)

**Finding**: The solver detects and reports:
- **Cycles**: Returns `XPKG_ERR_CYCLE` at line 259.
- **Conflicts**: Returns `XPKG_ERR_CONFLICT` at line 212.
- **Capacity**: Returns `XPKG_ERR_CAPACITY` when graph exceeds 512 nodes.

All error paths are clean returns with no state mutation. PASS.

## 3. XSEC RESILIENCE ANALYSIS

### 3.1 TLS 1.3 Handshake Failure -- Resource Cleanup (tls13.c)

**CRITICAL FINDING**: The `tls_client_handshake()` function has 20+ error return paths (lines 688, 693, 697, 701, 705, 711, 717, 778, 793, 844, 866, 881, 893, 927, 939, 943, 950, 968, 983). NONE of these error returns call `xsec_secure_zero()` on the key material that has been computed up to that point.

Key material at risk:
- `ctx->ecdhe_priv` (32 bytes) -- populated at init before handshake
- `ctx->shared_secret` (32 bytes) -- populated at Step 3 (line 745)
- `ctx->handshake_secret`, `ctx->master_secret` -- populated at Step 3
- `ctx->client_hs_traffic`, `ctx->server_hs_traffic` -- populated at Step 3
- `ctx->keys` -- populated at `install_handshake_keys()` (line 764)

The ONLY function that zeros these is `xsec_tls_close()` (lines 1122-1131). If the handshake fails and the caller does not call `xsec_tls_close()`, all key material remains in memory indefinitely.

**Bug (CONFIRMED from prior FMEA -- still present)**: TLS handshake error paths do not zero key material.

**Partial fix present**: Lines 953-954 zero `cv_content` and `cv_transcript` locally within Step 6. But this only covers the CertificateVerify local buffers, NOT the context-level secrets.

**Mitigation analysis**: The caller MUST call `xsec_tls_close()` even after handshake failure. This is documented in the module design but NOT enforced by the API.

### 3.2 X.509 Chain Verification Failure (tls13.c:834-846)

**Finding**: When hostname verification fails at line 838:
- Alert is sent: `TLS_ALERT_BAD_CERTIFICATE`. GOOD.
- Audit log entry created. GOOD.
- Returns `XSEC_ERR_TLS_HANDSHAKE`. GOOD.
- Key material NOT zeroed (same issue as 3.1). BAD.

**Verdict**: The connection is properly rejected, but key material leaks.

### 3.3 Poly1305 MAC Mismatch (chacha20.c -- confirmed in prior audit)

**Finding from prior FMEA**: The AEAD decrypt function in chacha20.c:
- Zeros plaintext on MAC failure. GOOD.
- Zeros tag. GOOD.
- Uses constant-time comparison. GOOD.
- Returns error without exposing plaintext. GOOD.

**Verdict**: PASS. This is the gold standard implementation in the codebase.

### 3.4 TLS Record Layer Error Handling (tls13.c:186-303)

**Finding**: `tls_send_record_encrypted()` uses module-static scratch buffers (fix S3-DI-C02 at lines 198-199). All error paths zero both buffers:
- Line 275-276: AEAD error path -- zeros scratch + nonce.
- Line 283-284: Overflow path -- zeros scratch + nonce.
- Line 289-290: Send header failure -- zeros scratch + nonce.
- Line 295-296: Send ciphertext failure -- zeros scratch + nonce.
- Line 301-303: Success path -- zeros scratch, ct_buf, nonce.

**Verdict**: PASS. Record layer encryption cleanup is thorough.

## 4. PAL-AETHER FAILURE PATHS

### 4.1 pal_pages_alloc OOM (pal_aether.c:428-448)

**Finding**: The flow is:
1. Validate arguments (lines 432-434). GOOD.
2. Issue `XK_SYS_PHYS_ALLOC` syscall (line 437-439). If kernel returns negative, it is converted to `pal_status_t` and returned. GOOD.
3. Allocate a handle via `_alloc_handle(PAL_OBJ_PAGES)` (line 441). If the handle table is full, returns `PAL_ERR_NOMEM`. **PROBLEM**: The kernel has already allocated the physical pages, but we cannot track them.

**Bug (CONFIRMED from prior FMEA -- still present)**: Kernel resource leak when handle allocation fails after syscall succeeds.

This affects:
- `pal_pages_alloc` (line 441-442): kernel pages allocated, handle fails = leaked pages.
- `pal_map_pages` (line 490-491): kernel mapping created, handle fails = leaked mapping.
- `pal_thread_create` (line 531-532): kernel thread created, handle fails = orphaned thread.
- `pal_timer_create` (line 863-864): kernel timer created, handle fails = leaked timer.

**Severity**: P1. Under sustained load, repeated handle-alloc failures will leak kernel resources until the system is starved.

### 4.2 pal_channel_create Failure (pal_aether.c:614-637)

**Finding**: This function has GOOD cleanup:
1. Syscall succeeds, returns packed pair (line 622-623).
2. Allocate handle A (line 625). If fails, error.
3. Allocate handle B (line 626). If fails, close handle A.
4. Lines 627-630: Both failure cases are handled -- if either handle fails, both are closed.

**However**: The KERNEL channel endpoints (kern_a, kern_b) are NOT freed if handle allocation fails. The `pal_handle_close()` only marks the PAL handle as invalid; it does not issue a kernel close syscall.

**Bug (extension of prior finding)**: `pal_handle_close()` at line 366-376 only sets `valid = false`. It does NOT issue `XK_SYS_HANDLE_CLOSE` to the kernel. So even when PAL-level cleanup succeeds, the kernel-side resource leaks.

### 4.3 Handle Table Exhaustion (pal_aether.c:329-348)

**Finding**: `_alloc_handle()` scans all 65535 valid slots (index 1 to 65535). If all are taken, returns `PAL_HANDLE_INVALID`. Every caller checks for this and returns `PAL_ERR_NOMEM`.

**Bug F-32 (P2): No handle utilization metric**
There is no API to query current handle count vs. capacity. Upper layers cannot implement backpressure or load shedding before handle exhaustion.

## 5. CROSS-CUTTING FAILURE MODES

### 5.1 pal_random_bytes Failure Propagation

**Finding**: `pal_random_bytes()` returns `PAL_ERR_NOT_SUPPORTED` if RDRAND fails after 10 retries. Callers:

| Call site | File:Line | Checks return? | Impact if unchecked |
|-----------|-----------|-----------------|---------------------|
| dns_do_resolve QID | dns.c:384 | NO (was flagged prior) | Predictable QID = cache poisoning |
| tcp_generate_isn | tcp.c:246 | NO | Predictable ISN = TCP hijacking |
| gossip_rand seed | gossip.c:166 | NO | Predictable node selection (low impact) |
| xsec_tls_init client_random | tls13.c:1057 | YES (prc check) | N/A -- properly handled |

**Severity**: P2 for tcp.c and dns.c (confirmed from prior audit).

### 5.2 Spinlock Contention Under Load

All four subsystems use single global spinlocks:
- `g_tcp_lock` -- protects all 128 TCBs
- `g_arp_lock` -- protects all 256 ARP entries
- `g_dns_lock` + `g_dns_pending_lock` -- protects DNS cache and pending query
- `g_gossip_lock` -- protects all 256 gossip members

Under high load (many concurrent connections), `g_tcp_lock` becomes the bottleneck since every send/recv/input/timer-tick acquires it. This is acceptable for Sprint 3 (single-core focus) but will not scale.

## 8. SPRINT 4 RECOMMENDATIONS

### P0 (Must Fix)
1. **DNS concurrent queries**: Replace single `g_dns_pending` with a pending-query table (at least 8 slots).

### P1 (Should Fix)
2. **TLS handshake key zeroing**: Add `xsec_secure_zero()` calls on all handshake error returns, OR document and enforce that callers MUST call `xsec_tls_close()` after any handshake failure.
3. **PAL handle-to-kernel leak**: When `_alloc_handle()` fails after a successful syscall, issue the corresponding kernel free/close syscall before returning error.
4. **XPKG upgrade rollback state**: Change line 463 from `XPKG_STATE_BROKEN` to `XPKG_STATE_INSTALLED`.
5. **XPKG VFS rollback**: Implement staging-directory extraction with atomic rename for Sprint 4 VFS integration.

### P2 (Should Address)
6. **TCP RST on full TCB table**: Send RST when `tcb_alloc()` fails for incoming SYN.
7. **ARP retry**: Add ARP re-request logic in `xnet_arp_expire()` for PENDING entries.
8. **pal_random_bytes check**: Add return-value checks in `tcp_generate_isn()` and `dns_do_resolve()`.
9. **Pool utilization APIs**: Add `xnet_tcp_pool_stats()`, `pal_handle_stats()` for backpressure.

### P3 (Nice to Have)
10. **xnet_tick initialized check**: Guard with `if (!g_initialized) return;`.
11. **Health check endpoints**: Expose per-module health via a standard API for GENSD integration.
12. **SYN cookies**: Implement SYN cookies to handle SYN floods without consuming TCB slots.
