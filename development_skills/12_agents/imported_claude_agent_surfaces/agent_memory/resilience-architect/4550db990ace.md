## II. CRITICAL ANALYSIS: THE FIVE REQUESTED RESILIENCE DOMAINS

### 1. WAL CRASH RECOVERY (CRITICAL -- verify mvcc.c fix)

**Question 1: Does wal.c xstore_wal_replay now correctly replay INSERTs with val data?**

YES -- verified. The fix in `mvcc.c:212-226` populates `entry.val` and `entry.val_len` by calling `xstore_btree_lookup()` against the page cache at commit time. This means WAL INSERT entries now contain the actual value data.

On replay, `wal.c:157-162` correctly reads `e->val` and `e->val_len` from the WAL entry:
```c
xstore_val_t val;
val.data = e->val;       // now populated with real data
val.len  = e->val_len;   // now > 0 for INSERT entries
(void)xstore_btree_insert(store, &rec_txn, &key, &val);
```

**Verdict: The fix is correct. WAL INSERT entries now carry self-contained val data for crash recovery.**

However, three residual issues remain:

**RA-WAL-01 [P2]: btree_lookup at commit time uses snapshot_lsn, not current state.**
- Location: `mvcc.c:220` -- `xstore_btree_lookup(store, txn->snapshot_lsn, &k, &vout)`
- The lookup uses `txn->snapshot_lsn` (the snapshot at txn_begin time). If another concurrent txn modified the same key and committed between our begin and commit, the lookup may return the old value or NOT_FOUND.
- In practice, this is guarded by the OCC conflict check at line 190 -- if another txn wrote to the same page_no, our txn would have been aborted with XSTORE_ERR_CONFLICT before reaching the WAL append code.
- **Assessment: SAFE in normal operation.** The OCC check ensures that if we reach the WAL append, no conflicting write occurred. The snapshot_lsn lookup will return our own dirty page data.

**Question 2: What happens if btree_lookup returns NOT_FOUND at commit time?**

Per `mvcc.c:220-225`:
```c
if (xstore_btree_lookup(...) == XSTORE_OK) {
    entry.val_len = (uint16_t)vout.out_len;
}
/* On lookup failure: val_len=0, replay remains best-effort. */
```

If the lookup fails (NOT_FOUND or other error), `val_len` remains 0 (from the memset). The WAL entry is still appended with key but no value. On replay, `wal.c:159` would set `val.len = 0`, and the btree_insert would insert a key with an empty value.

**RA-WAL-02 [P2]: WAL INSERT with val_len=0 replays as empty-value insert.**
- Scenario: btree_lookup fails during commit (page evicted from cache between put() and commit(), or concurrent page modification). The WAL entry records the key but not the value.
- On crash recovery, the key is re-inserted with val_len=0, which is semantically wrong (the value was non-empty at put time).
- Probability: Low. The comment says "This should not occur in normal operation" because the page was dirtied by xstore_txn_put() and is in the cache.
- Impact: If it does occur, committed data loses its value (key preserved, value becomes empty).
- Mitigation: Treat btree_lookup failure during commit as a commit failure -- abort the transaction.

**Question 3: Is "best-effort" still accurate?**

The comment at `mvcc.c:223` says "val_len=0, replay remains best-effort." This is now misleading. The fix made replay CORRECT for the normal case (val_len > 0). The "best-effort" fallback only applies to the edge case where btree_lookup fails, which should be impossible under normal operation.

**RA-WAL-03 [P3]: Stale "best-effort" comment is misleading.**
- The comment should be updated to reflect that val recovery is the normal path, and the fallback is a defensive guard for an edge case that should never occur.

**XSTORE WAL Power-Failure Analysis (Chaos Scenario 1):**

| Failure Point | Data Consistent? | Mechanism |
|---------------|------------------|-----------|
| During wal_append (before CRC write) | YES | Entry fails CRC check on replay, skipped |
| After wal_append, before COMMIT entry | YES | No COMMIT record -> txn discarded |
| After COMMIT entry, before sync() | **RISK** | Entry may be in write-back cache but not durable |
| After sync(), before checkpoint | YES | Full replay restores committed data |
| During checkpoint (superblock write) | YES | Old superblock CRC fails -> old checkpoint_lsn used |

**Critical finding: RA-WAL-04 [P1]: sync() after COMMIT is NOT called in xstore_txn_commit.**

Looking at `mvcc.c:179-263`, the `xstore_txn_commit()` function appends all WAL entries including the COMMIT record, but does NOT call `store->sync()`. The sync is deferred to `xstore_wal_checkpoint()`. This means:

**A committed transaction's WAL entries are NOT durable until the next checkpoint.**

If power fails after xstore_txn_commit() returns XSTORE_OK but before the next checkpoint, the committed transaction's WAL entries may be lost (they exist only in the WAL ring in memory, not synced to durable storage).

The WAL design comment at `wal.c:27-31` explicitly states:
```
A transaction is durable when:
  1. All WAL entries for the txn are appended (including COMMIT entry)
  2. store->io.sync() has been called
```

But `xstore_txn_commit()` only does step 1. Step 2 is deferred. This is a documented trade-off (WAL entries are in memory, pages are not yet synced), but the CALLER has no way to force a sync after commit without calling `xstore_checkpoint()`.

**RA-WAL-04 [P1]: xstore_txn_commit does not call sync() -- committed transactions are not durable until checkpoint.**
- Impact: Data loss window between commit and next checkpoint.
- This is a well-known WAL design choice (group commit / lazy sync). However, the convenience API (`xstore_put`) does not checkpoint after each put, so the default usage pattern has a durability gap.
- Mitigation: Either (a) add a `sync_on_commit` flag to xstore_txn_commit, or (b) document that durability requires explicit xstore_checkpoint() calls, or (c) add periodic auto-checkpoint in GENSD supervisor tick.

### 2. BOOT CHAIN RESILIENCE (Sprint 8)

**Question: If XSTORE init fails (stage 2), does boot abort or continue?**

Per `boot_chain.c:546-550`:
```c
rc = xchain_storage(handoff);
s_result.storage_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] WARN: storage init failed (%u)\n", rc);
}
// continues to next stage
```

Storage is NON-CRITICAL. Boot continues. However, **xchain_storage() NEVER actually fails** -- it always returns PAL_OK (lines 111 and 119), even when the region is not found in the memory map. This is confirmed as still open from the prior audit (RA-BOOT-02).

**RA-BOOT-01 [P1 -- STILL OPEN]: xchain_storage always returns PAL_OK.**
- Even if storage is genuinely broken, the boot chain will proceed with `storage_ok=1`.
- Downstream subsystems (XBLOB, XSTORE, XMIND weights via XBLOB) will encounter failures at runtime, not at boot.

**Question: If XMIND load fails (stage 9), does XSHELL start without AI?**

YES. Per `boot_chain.c:598-603`:
```c
rc = xchain_ai();
s_result.ai_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] WARN: AI init failed (offline mode)\n");
}
```

AI is NON-CRITICAL. The shell starts regardless. The status accessor `xchain_ai_ready()` returns `s_result.ai_ok`, which XSHELL could query. However, **XSHELL does not actually query this** (no reference to xchain_ai_ready in xshell.c). The user has no indication that AI is offline.

**RA-BOOT-02 [P1 -- STILL OPEN]: XSHELL does not display AI offline status.**
- The mechanism exists (xchain_ai_ready) but is not wired.

**Question: If XDISP init fails, does the system drop to text console?**

NO. Per `boot_chain.c:574-579`:
```c
rc = xchain_display(handoff);
s_result.display_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] CRITICAL: display init FAILED\n");
    return rc;
}
```

Display is CRITICAL. Boot aborts. The return propagates to `gensd_start()` which enters the infinite halt loop (`main.c:534-536`). There is NO serial console fallback, NO text-mode VGA, NO recovery path.

**RA-BOOT-03 [P1 -- STILL OPEN]: Display failure = permanent halt with no fallback.**
- The XSHELL_SESSION_FALLBACK enum exists but is never triggered.
- The boot chain has no intermediate degradation between "display works" and "system halts."

**Question: Is there crash loop detection in boot_chain?**

NO. `xchain_run()` executes exactly once. If GENSD's `gensd_start()` calls `xchain_run()` and it fails, the system halts forever. There is no retry, no backoff, no "try again with reduced configuration."

**RA-BOOT-04 [P2 -- STILL OPEN]: No boot retry or progressive fallback.**
- If a transient issue caused a critical stage to fail (e.g., memory pressure during compositor surface pool alloc), the system halts permanently instead of retrying.

### 3. GENSD SUPERVISOR

**Question: Does supervisor restart XSHELL on crash?**

YES, conditionally. The supervisor watches services registered via `supervisor_watch()`. Per `main.c:284-287`, the essential services include `xnet-loopback`, `xstore-init`, and `xapi-health`. XSHELL is NOT in this list -- it would need to be registered separately by boot_chain or a later stage.

**RA-GENSD-05 [P2]: XSHELL is not in the essential_services list.**
- XSHELL's lifecycle is managed by boot_chain stage 8 (shell IPC setup) and presumably spawned by the GENSD run_loop. But the essential_services table does not include it.
- If XSHELL is registered and watched, the supervisor WILL restart it on crash.
- The restart mechanism: `supervisor_probe_process()` checks the `running` flag. If `running=false` and type=DAEMON, result is HEALTH_FAIL, which triggers `supervisor_restart()`.

**Question: Is there a max restart count?**

NO. `restart_count` is incremented but never compared to a maximum (`supervisor.c:198`). The backoff caps at 30 seconds, but restarts continue indefinitely.

**RA-GENSD-03 [P2 -- STILL OPEN]: No maximum restart count. Infinite crash-loop possible.**

**Question: What is the restart backoff?**

Per `supervisor.c:108-113`:
- Initial: `restart_delay_ms` (default 1000ms from `service.c:368`)
- Growth: `delay * 2`
- Cap: `SV_MAX_BACKOFF_MS = 30000` (30 seconds)
- Overflow protection: `next < current_ms` check

Backoff sequence: 1s, 2s, 4s, 8s, 16s, 30s, 30s, 30s, ...

**RA-GENSD-01 [P1 -- STILL OPEN]: supervisor_restart blocks the entire supervisor thread.**
- `supervisor.c:213-214`: `pal_thread_sleep_ns(delay_ns)` -- this is a blocking sleep.
- During a 30-second backoff for one service, ALL other services go unmonitored.
- The TODO comment at lines 207-212 explicitly documents this as a P1 issue tracked since Sprint 1.

### 4. XBLOB GC COMPACTION FAILURE

**Question: Is compaction atomic?**

NO. Compaction in `gc.c:173-195` is a multi-phase operation:
1. Iterate all slots, copying OCCUPIED entries to a 256-entry chunk buffer
2. Set each slot to XBLOB_INDEX_SLOT_FREE as it is copied
3. Re-insert all entries from the chunk

If a crash occurs between step 2 (slot set to FREE) and step 3 (re-insert), the entries in the current chunk are LOST. The index has them as FREE, but the chunk buffer (stack memory) is gone.

**RA-XBLOB-01 [P2 -- STILL OPEN]: GC compaction is not crash-safe.**

**Question: Are live blob entries recoverable after compaction crash?**

The blob DATA still exists in storage (only the index entry was cleared). However:
- There is no mechanism to scan blob storage and rebuild the index from blob headers.
- The GC log only records FREE events, not compaction moves.
- On `xblob_open()`, `xblob_gc_log_replay()` will replay GC log entries but cannot recover index entries that were cleared during compaction.

**RA-XBLOB-02 [P2]: No blob storage scan/rebuild capability.**
- Mitigation: Implement a full-storage-scan recovery mode that reads each blob header, verifies SHA-256, and rebuilds the index.

**Question: Does xblob_gc_log_replay restore correct state?**

YES, for the normal (non-compaction) case. Per `gc.c:65-103`:
- Two-pass: find max_seq, then replay in sequence order
- CRC validation on each entry
- Refcount from GC log is authoritative over on-disk index
- Skips entries that fail CRC or whose slot belongs to a different epoch

**Assessment: GC log replay is RESILIENT for refcount recovery. It is NOT resilient for compaction crashes.**

### 5. XMIND OOM DURING LOAD

**Question: Is the partial load cleaned up (no leak)?**

STRUCTURALLY YES, FUNCTIONALLY NO.

`xmind_alloc_state()` at lines 180-201:
```c
oom:
    xm_heap_free(m->state.logits);  // no-op
    xm_heap_free(m->state.x);       // no-op
    // ... all 7 scratch buffers + all KV cache layers
    return XMIND_ERR_NOMEM;
```

The cleanup pattern is correct: it iterates all pointers, calls free, and nulls them. However, `xm_heap_free()` is a no-op (`xmind.h:109-113`):
```c
static inline void xm_heap_free(void *ptr) { (void)ptr; }
```

Every allocation that succeeded before the OOM failure is leaked.

**RA-XMIND-01 [P1 -- STILL OPEN]: xm_heap_free no-op causes progressive memory exhaustion.**
- Llama 3.2 3B KV cache: 2 * 28 * 2048 * 8 * 96 * 4 = ~336 MB
- On OOM, up to 336 MB is leaked.
- Supervisor restart -> re-attempt -> leak again -> PAL page pool exhaustion.
- Death spiral: 2-3 OOM cycles exhaust all system memory.

**Question: Does XMIND return an error the boot chain can handle?**

YES, partially. `weights_loader.c:491-496`:
```c
st = xmind_alloc_state(m);
if (st != XMIND_OK) {
    pal_console_printf("[XMIND-WL] xmind_alloc_state failed: %d\n", st);
    xmind_shutdown(m);
    return st;
}
```

The error propagates through `xmind_weights_load_file()` -> `xmind_weights_load()` -> caller. If the caller is `xchain_ai()`, the error causes `s_result.ai_ok = 0`, which is the non-critical "AI offline" path.

HOWEVER, the weight loader has a separate problem:

**RA-XMIND-02 [P2 -- STILL OPEN]: Per-layer weight allocation in weights_loader.c does NOT check return values.**

`weights_loader.c:503-511`:
```c
for (l = 0u; l < cfg.n_layers && l < XMIND_MAX_LAYERS; l++) {
    m->wq[l] = (xmind_q4_block_t *)wl_alloc_pages(w_size);
    m->wk[l] = (xmind_q4_block_t *)wl_alloc_pages(w_size);
    // ... 7 allocations per layer, NO NULL checks
}
```

If `wl_alloc_pages()` returns NULL (OOM), the weight pointer is set to NULL. The function continues and returns XMIND_OK. On the first forward pass, dereferencing a NULL weight pointer = crash.

Additionally, the embedding and RoPE allocations (lines 514-527) also have no NULL checks.

**RA-XMIND-03 [P1]: Weight loader proceeds with NULL pointers on OOM -- crash on first forward pass.**
- Impact: NULL dereference in transformer forward pass = page fault = XENOS terminates process.
- This is WORSE than an explicit OOM error because the boot chain reports AI as "OK" (`xchain_ai` returns PAL_OK), and XSHELL would not show "AI offline."
- Mitigation: Check every wl_alloc_pages return. On NULL, free all prior allocations and return XMIND_ERR_NOMEM.

## III. CHAOS SCENARIO EVALUATIONS

### Chaos 1: Power failure during WAL write -> database consistent?

**Verdict: MOSTLY SAFE, with one gap.**

| Failure Point | Consistent? | Notes |
|---------------|-------------|-------|
| Mid wal_append (partial entry) | YES | CRC mismatch -> entry skipped on replay |
| After wal_append, before COMMIT | YES | No COMMIT -> txn discarded on replay |
| After COMMIT, before sync | **GAP** | WAL entries in memory only; power loss = lost |
| After sync, before checkpoint | YES | Replay recovers committed data |
| Mid checkpoint (superblock) | YES | Old superblock CRC valid; old checkpoint_lsn used |

The gap between COMMIT and sync is a durability concern (RA-WAL-04). The database is CONSISTENT (no partial state) but may lose the most recent committed transaction.

### Chaos 2: OOM during XMIND weight load -> graceful recovery?

**Verdict: UNSAFE -- three compounding failures.**

1. `xmind_alloc_state()` OOM: cleanup is correct but xm_heap_free is no-op -> memory leaked (RA-XMIND-01).
2. `weights_loader.c` per-layer alloc: no NULL checks -> NULL pointers stored -> crash on forward pass (RA-XMIND-03).
3. After crash, supervisor restarts XMIND, but leaked memory from attempt 1 makes attempt 2 MORE likely to OOM -> death spiral.

**Recovery path: BROKEN.** The system cannot recover from XMIND OOM without a reboot.

### Chaos 3: XSHELL segfault -> GENSD restart within 5 seconds?

**Verdict: PARTIALLY SAFE, with caveats.**

- Detection: supervisor_tick runs every 1s, checks `running` flag. Segfault -> XENOS terminates process -> running=false. Detection time: ~1s.
- Restart: supervisor_restart sleeps for 1000ms (initial backoff). Total: ~2 seconds on first crash.
- Caveats:
  - XSHELL may not be in essential_services (RA-GENSD-05). If not watched, no restart.
  - Restart is a STUB (sets running=true, no actual exec). Sprint 2+ integration needed.
  - HTTP probe is stubbed -- a hung XSHELL (not crashed) would never be detected.
  - Blocking sleep during restart suspends all supervision.

**5-second target: ACHIEVABLE for first crash if XSHELL is watched. NOT achievable if supervisor is sleeping on a backoff for another service.**

### Chaos 4: XBLOB GC crash -> blobs recoverable?

**Verdict: UNSAFE during compaction, SAFE otherwise.**

- Normal GC (freeing refcount=0): SAFE. Each free is logged to GC log before tombstoning. On crash, GC log replay restores refcounts.
- Compaction: UNSAFE. Entries cleared but not yet re-inserted are lost. No recovery mechanism exists to rebuild index from blob storage headers.
- Blob DATA is not deleted during compaction (only index entries are shuffled). A full-storage-scan recovery could theoretically rebuild the index, but this is not implemented.

## V. THE mvcc.c WAL FIX -- DETAILED VERDICT

### What changed
Before the fix, WAL INSERT entries had `val_len=0` -- the key was recorded but not the value. On crash recovery, replay would insert keys with empty values.

After the fix, `xstore_txn_commit()` (mvcc.c:212-226) looks up each key in the B+ tree page cache and copies the value into the WAL entry before appending it.

### Assessment
| Criterion | Before Fix | After Fix |
|-----------|-----------|-----------|
| WAL INSERT carries value data | NO | YES |
| Crash replay restores key+value | NO (value lost) | YES |
| Self-contained WAL entry | NO (requires page cache) | YES |
| Replay correctness guarantee | Best-effort (always empty val) | Correct for normal path |
| Edge case: btree_lookup fails | N/A (no lookup) | val_len=0, replay inserts empty val |

### Remaining risks
1. **RA-WAL-02**: If btree_lookup fails during commit (unlikely but possible), the WAL entry records val_len=0. On replay, the key is inserted with an empty value. This is data corruption (silent value loss).
2. **RA-WAL-04**: Even with the fix, WAL entries are not synced to durable storage until checkpoint. A power failure after commit but before checkpoint loses the transaction entirely.

### Overall
The fix is a significant improvement. The "best-effort" characterization is no longer accurate -- WAL replay is now CORRECT for the normal path. The remaining risks are edge cases (btree_lookup failure) and a design-level durability gap (no sync on commit).

**XSTORE resilience score upgraded from 7/10 to 7.5/10.**

## VII. RESILIENCE PATTERNS -- SUMMARY

### What Works (Strong patterns confirmed in code)

1. **XSTORE WAL crash recovery with val data** -- Fix verified. Two-phase CRC-validated replay with self-contained entries.
2. **XSTORE OCC auto-retry** -- 3 retries with fresh snapshot. Clean conflict resolution.
3. **XSTORE WAL_FULL auto-checkpoint** -- Single retry after draining ring. Correct pattern.
4. **XBLOB GC log replay** -- CRC-validated, sequence-ordered refcount recovery.
5. **XBLOB SHA-256 on read** -- Content-addressed integrity verification.
6. **XBLOB bitmap rebuild on open** -- Self-healing statistics reconstruction.
7. **XMIND OOM cleanup pattern** -- Correct structure (undermined by no-op free).
8. **XMIND config validation** -- 10+ boundary checks prevent invalid models.
9. **GENSD exponential backoff** -- 30s cap with overflow protection.
10. **boot_chain critical/non-critical** -- AI and network failures don't prevent boot.

### What Fails (Patterns that need fixing)

1. **No sync on commit** -- Durability gap between commit and checkpoint.
2. **xm_heap_free no-op** -- Memory leak death spiral on service restart.
3. **Weight loader unchecked allocs** -- NULL weight pointers -> crash.

## IX. CHAOS EXPERIMENT DESIGNS (Updated)

### Experiment 1: XSTORE Durability Gap

```
CHAOS EXPERIMENT: XSTORE-COMMIT-SYNC-GAP
==========================
Hypothesis: "When power fails after xstore_put() returns XSTORE_OK but before
            xstore_checkpoint(), the committed key-value pair is lost."
Steady State: XSTORE open, 1000 keys committed and checkpointed.
Fault Injection: Call xstore_put() for key 1001, verify return XSTORE_OK.
                 Kill process BEFORE calling xstore_checkpoint().
Observation: Re-open store, verify:
             - key 1001 is NOT present (WAL entries were in memory only)
             - keys 1-1000 are intact
Duration: 10 iterations.
Abort Conditions: Any key 1-1000 is missing or corrupt.
Recovery Verification: All prior checkpointed data survives.
Expected Result: Key 1001 IS lost. This confirms the durability gap.
```
