### Required fix

The tls_client_handshake function needs a cleanup-on-error wrapper. Either:

Option A: Add error cleanup at the end of tls_client_handshake:

```c
// At the end, before return XSEC_OK:
ctx->state = XSEC_TLS_STATE_APPLICATION_DATA;
return XSEC_OK;

// For errors, add a goto cleanup pattern or have xsec_tls_handshake() call:
error:
    if (ctx->state != XSEC_TLS_STATE_CLOSED) {
        tls_send_alert(ctx, TLS_ALERT_LEVEL_FATAL, TLS_ALERT_INTERNAL_ERROR);
    }
    xsec_secure_zero(ctx->ecdhe_priv, sizeof(ctx->ecdhe_priv));
    xsec_secure_zero(ctx->shared_secret, sizeof(ctx->shared_secret));
    // ... zero all key material
    return rc;
```

Option B: Make xsec_tls_handshake() always call xsec_tls_close() on failure:

```c
xsec_status_t xsec_tls_handshake(xsec_tls_ctx_t *ctx) {
    xsec_status_t rc = tls_client_handshake(ctx);
    if (rc != XSEC_OK) {
        xsec_tls_close(ctx);  // zeros all keys, sends close_notify if possible
    }
    return rc;
}
```

Option B is simpler and ensures all cleanup happens regardless of failure point.

## 2. XNET Stack Failure Cascading

### ARP -> IP4 -> TCP cascade path

The chain is: tcp_send_segment -> xnet_ip4_output -> xnet_arp_resolve.

When ARP returns XNET_ERR_AGAIN (resolution pending), ip4_output propagates it. TCP behavior depends on call site:

- tcp_send_segment returns the error to its caller
- In xnet_tcp_connect (line 965): checked, SYN enqueued only on success -- CORRECT
- In xnet_tcp_input, all tcp_send_segment calls for ACKs (lines 769, 828, 846, 855, etc.) IGNORE the return value

This is functionally acceptable because TCP retransmit will resend. The peer will retransmit its segment, triggering another ACK attempt. The ARP cache will resolve within 1-2 seconds, and the next attempt succeeds. The cost is one extra round trip.

### TCP retransmit exhaustion cascade

When retransmit_count > 12 (line 1293):

1. `t->state = TCP_CLOSED` then `xnet_memset(t, 0, sizeof(*t))`
2. The memset zeros `in_use`, so the TCB slot is immediately reusable
3. Any blocked xnet_tcp_send/recv sees `!t->in_use` and returns XNET_ERR_NOT_FOUND
4. The error code should be XNET_ERR_CONNRESET (not NOT_FOUND) but the connection IS cleaned up

There is no resource leak. There is no notification callback -- the socket layer discovers the failure on its next send/recv attempt. This is the standard BSD socket behavior (blocking call returns error).

### Missing: no connection-abort callback

Higher layers (TLS, HTTP) have no way to register for async connection-abort notification. They must poll via send/recv. For TLS this means a TLS session over a retransmit-exhausted TCP connection will hang until the next tls_send/recv call discovers the error. This is acceptable for Sprint 3 but should be addressed in Sprint 4 with an event/callback mechanism.

## 3. XPKG Refusal Gate Bypass Analysis

### Gate-by-gate bypass assessment

Gate 0 (SIGNATURE): Calls xpkg_verify_signature with NULL key list. Security depends entirely on the verify function's internal key store. If key store is empty at boot, behavior is undefined. P1 risk.

Gate 1 (INTEGRITY): Calls xpkg_verify_integrity. If archive_buf is valid but content has been modified after signature was computed, the integrity check catches it. SOUND.

Gate 2 (CAPABILITY): Heuristic-only. Checks install_path prefix. Returns WARN for restricted paths, never FAIL. Attacker can install to /sys/ with only a warning. P2 by design (Sprint 4 will add XKABI integration).

Gate 3 (PROVENANCE): Returns PASS if ANY repo is enabled, regardless of whether the package exists in that repo. False sense of security. P2.

Gate 4 (VERSION): Sound. Checks against installed registry. Downgrades blocked with FAIL. Reinstalls allowed with WARN.

Gate 5 (CONFLICT): Sound. Has integer overflow protection (line 338: `if (next <= pos) break`). Bounds checks on archive offsets. File path collision detection is thorough.

Gate 6 (SIZE): Sound. Checks against configurable quota. Warns at 80%.

Gate 7 (DEPENDENCY): Returns WARN if deps missing (not FAIL). By design -- full resolver runs later.

Gate 8 (POLICY): Uses glob_match which is recursive. Stack overflow risk on patterns like `*****...` with long strings. P2.

### Config authentication gap

The xpkg_gate_config_t has two dangerous arrays:

- `enabled[XPKG_GATE_COUNT]`: if attacker sets enabled[0]=false, signature check is skipped
- `force[XPKG_GATE_COUNT]`: if attacker sets force[0]=true, signature FAIL becomes WARN

The gate_check function (line 464) validates pkg and out_report are non-NULL but does NOT validate config integrity. If config is loaded from disk, a local privilege escalation can bypass all gates.

Mitigation: Config should be signed or derived from a tamper-evident source. At minimum, gates 0 (SIGNATURE) and 1 (INTEGRITY) should be hardcoded as always-enabled, never-forceable.

### glob_match stack risk

The glob_match function (line 103-121) is recursive. Each `*` in the pattern can cause O(n) recursive calls where n is the remaining string length. A pattern like `*a*b*c*d*e*f*g*h` with a long non-matching string causes exponential blowup.

In freestanding C with a fixed-size kernel stack (typically 8-16KB), this is a real stack overflow risk if policy patterns come from an attacker-controlled source. Fix: add a depth counter or convert to iterative.

## 4. PAL-Aether Syscall Error Propagation

### Error mapping pattern

All PAL-Aether syscall wrappers use the same pattern:

```c
int64_t ret = _xk_syscall4(XK_SYS_..., ...);
if (ret < 0) return (pal_status_t)(-ret);
```

This correctly maps negative kernel error codes to positive PAL error codes. The pattern is consistent across all 10 syscall wrappers. No negative-to-unsigned misinterpretation issues.

### Kernel resource leak on handle table exhaustion

Three functions have the pattern:

```c
int64_t ret = _xk_syscall4(...);      // kernel allocates resource
if (ret < 0) return error;             // kernel error -- no leak
pal_handle_t h = _alloc_handle(...);   // PAL handle table
if (h == PAL_HANDLE_INVALID) return PAL_ERR_NOMEM;  // LEAK: kernel resource allocated, no handle to free it
```

Affected functions:

- pal_pages_alloc (line 441): leaks kernel physical pages
- pal_map_pages (line 490): leaks kernel virtual mapping
- pal_thread_create (line 531): leaks kernel thread (runs forever with no handle to join/kill)

Fix: on handle alloc failure, issue the corresponding free syscall:

```c
pal_handle_t h = _alloc_handle(PAL_OBJ_PAGES);
if (h == PAL_HANDLE_INVALID) {
    _xk_syscall4(XK_SYS_PHYS_FREE, (uint64_t)ret, 0, 0, 0);
    return PAL_ERR_NOMEM;
}
```

pal_channel_create (line 625-631) handles this correctly -- it closes both handles if either allocation fails. This should be the model for the other functions.

### pal_channel_recv truncation

Line 664: `out->data_len = (uint32_t)ret` truncates if kernel returns > 4GB. Practically impossible given PAL_MSG_MAX_ENVELOPE constraints, but the cast should have a bounds check for defense in depth.

### pal_timer_create type mismatch

Line 863: `_alloc_handle(PAL_OBJ_EVENT)` allocates a handle typed as EVENT, not TIMER. This means pal_timer_set and pal_timer_cancel use `_get(h, PAL_OBJ_EVENT)` which works, but if a separate PAL_OBJ_TIMER type is added later, all timer handles will fail validation. Minor design inconsistency, not a bug today.
