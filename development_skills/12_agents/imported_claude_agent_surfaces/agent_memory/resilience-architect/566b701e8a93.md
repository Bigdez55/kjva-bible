## I. MANDATORY TEST TYPE RESULTS

### 1. SMOKE: Error Codes Defined in All Subsystem Headers

| Subsystem | Header | Error Codes Defined | Verdict |
|-----------|--------|--------------------:|---------|
| XSTORE | xstore.h | 15 (XSTORE_OK through XSTORE_ERR_STATE) | PASS |
| XBLOB | xblob.h | 11 (XBLOB_OK through XBLOB_ERR_READONLY) | PASS |
| XMIND | xmind.h | 5 (XMIND_OK through XMIND_ERR_IO) | PASS |
| XNET/TCP | xnet.h (implied) | Via return codes in tcp.c | PASS |
| XSEC/TLS | xsec.h | Alert codes + XSEC_ERR_* | PASS |
| PAL | pal.h | PAL_OK, PAL_ERR_INVAL, etc. | PASS |
| GENSD | service.c | int return (-1/0) | WEAK -- no typed error enum |
| GENISO | gen_iso.h | PAL_ERR_INVAL only | WEAK -- no domain errors |

**Finding S4-SM-01 (P3):** GENSD service.c and main.c use bare int returns (-1/0) rather than typed error enums. This makes error propagation from service failures to the supervisor opaque. The caller of gensd_start_service() cannot distinguish "service not found" from "service table full" from "shutdown in progress."

### 2. FUNCTIONAL: Error Propagation XSTORE_ERR_IO -> boot_chain -> GENSD

**Trace path analyzed:**
1. `xchain_storage()` in boot_chain.c validates the XBLOB region in the memory map (lines 89-120)
2. If validation fails, it returns `PAL_OK` anyway (line 119: "proceeding (unverified)")
3. `xchain_run()` stores `s_result.storage_ok = (rc == PAL_OK)` (line 547)
4. Storage failure is NON-CRITICAL: boot continues with a WARN log (line 549)
5. The actual XSTORE engine (xstore_open) is NOT called during boot_chain -- only a memory region probe

**Finding S4-FN-01 (P1): xchain_storage() NEVER returns an error.** Even when the XBLOB region is NOT found in the handoff memory map, it returns PAL_OK (line 119). This means `s_result.storage_ok` is ALWAYS 1. The "non-critical" classification is moot because the probe cannot fail.

**Finding S4-FN-02 (P2): No XSTORE_ERR_IO -> boot_chain path exists.** The boot chain does NOT call xstore_open(). If the NVMe driver fails during boot and XSTORE cannot open its backing store, no boot stage detects this. XSTORE errors will only manifest when services attempt reads/writes AFTER boot completes.

### 3. INTEGRATION: GENSD Supervisor Handles Service Crashes

**Code path: supervisor.c supervisor_tick() (lines 345-379)**

The supervisor correctly:
- Rate-limits probes to once per second (line 357-358)
- Calls supervisor_probe_process() which checks the `running` flag (line 363)
- For DAEMON type: running=false -> HEALTH_FAIL (line 143)
- For ONESHOT type: always returns HEALTH_OK (line 139) -- correct
- Applies restart policy on failure (lines 367-377)
- Uses exponential backoff capped at 30s (lines 108-113)

**Finding S4-IN-01 (P1 -- CONFIRMED PRIOR G-11): supervisor_probe_http() ALWAYS returns HEALTH_OK.** (Line 170). This is documented as a "Sprint 1 stub" but we are now at Sprint 4. HTTP health probing remains non-functional. Every service with an HTTP health endpoint is assumed healthy regardless of actual state. The supervisor can only detect process death (running flag), not hang/slow/corrupt failures.

**Finding S4-IN-02 (P1): Supervisor restart blocks ALL supervision.** supervisor_restart() calls pal_thread_sleep_ns() with up to 30s backoff (line 214). While sleeping, NO other services are probed. If service A fails with 30s backoff, services B through N are unsupervised for 30 seconds. The code documents this as "P1-SUPERVISOR-SLEEP" (line 212) but it remains unfixed.

**Finding S4-IN-03 (P2): Supervisor watch table has no dynamic growth.** SV_MAX_WATCHED = 64 (line 80). If more than 64 services are registered, supervisor_watch() silently drops the registration (line 313-316). The journal logs "watch table full" but the service runs unsupervised.

### 4. REGRESSION: boot_chain.c Critical vs Non-Critical Stage Handling

**xchain_run() stage criticality matrix (lines 533-613):**

| Stage | Function | Classification | On Failure | Correct? |
|-------|----------|---------------|------------|----------|
| 0 | xchain_kernel_ready | CRITICAL | return rc (abort) | YES |
| 2 | xchain_storage | NON-CRITICAL | WARN, continue | NO -- see S4-FN-01 |
| 3 | xchain_security | CRITICAL | return rc (abort) | YES |
| 4 | xchain_network | NON-CRITICAL | WARN, continue | YES |
| 5 | xchain_packages | NON-CRITICAL | WARN, continue | YES |
| 6 | xchain_display | CRITICAL | return rc (abort) | YES |
| 7 | xchain_compositor | CRITICAL | return rc (abort) | YES |
| 8 | xchain_shell | CRITICAL | return rc (abort) | PARTIAL* |
| 9 | xchain_ai | NON-CRITICAL | WARN, continue | YES |

**Finding S4-RG-01 (P1): Stage 8 (Shell) failure has NO fallback path.** When xchain_shell() fails, xchain_run() returns a non-PAL_OK status, which propagates to GENSD main. GENSD's gensd_start() enters an infinite halt loop (lines 534-536). There is NO TTY fallback. The XSHELL_SESSION_FALLBACK enum exists in xshell.h (line 277) and greeter.c names it "Fallback TTY" (line 50), but NO code path in boot_chain or GENSD ever triggers it. A shell IPC failure = permanent system hang.

**Finding S4-RG-02 (P2): Stage 1 (PAL init) is MISSING from xchain_run().** The comment says "Stage 1: xchain_pal_init ... already in boot_seq" (line 24-25) but xchain_run() jumps from Stage 0 directly to Stage 2. If PAL init fails in boot_sequence.c but xchain_run() is the entry point, PAL failures are invisible.

### 5. LOAD: XSTORE WAL Ring Saturation (XSTORE_ERR_WAL_FULL)

**Behavior when WAL is full:**
- wal.c line 66-68: xstore_wal_append() checks `(wal_head - wal_tail) >= XSTORE_WAL_RING_SIZE` and returns `XSTORE_ERR_WAL_FULL`
- mvcc.c lines 209-217: xstore_txn_commit() catches WAL_FULL and auto-checkpoints:
  ```c
  if (rc == XSTORE_ERR_WAL_FULL) {
      xstore_status_t cp_rc = xstore_wal_checkpoint(store);
      if (cp_rc != XSTORE_OK) { txn->active = false; return cp_rc; }
      rc = xstore_wal_append(store, &entry);
  }
  ```

**Verdict: GOOD.** Auto-checkpoint on WAL saturation is correctly implemented. The retry is single-attempt (not infinite loop), and checkpoint failure propagates cleanly.

**Finding S4-LD-01 (P2): Auto-checkpoint is attempted only ONCE per WAL_FULL.** If the checkpoint succeeds but the re-append still returns WAL_FULL (possible if many concurrent txns filled the ring during checkpoint), the commit fails permanently. No second retry.

### 6. STRESS: All 64 XSTORE Txn Slots Occupied

**xstore_txn_begin() in mvcc.c (lines 131-152):**
- The function does NOT use _txn_alloc() to check slot availability
- It assigns txn_id and snapshot directly from the global counters
- The caller provides their own `xstore_txn_t *out_txn` (stack-allocated)
- The store->txns[64] array tracks in-flight slots but _txn_alloc is `__attribute__((unused))`

**Finding S4-ST-01 (P1): xstore_txn_begin() has NO concurrency limit.** The function always succeeds regardless of how many transactions are in-flight. The 64-slot txns[] array exists but _txn_alloc() is unused. This means:
- There is no backpressure when 64+ concurrent transactions are active
- The OCC conflict check at commit time will catch true conflicts, but the system has no way to shed load before commit
- Memory is bounded only by caller stack size (each xstore_txn_t contains a 512-entry write_set)

### 7. SECURITY: Failure Paths Don't Leak State

**TLS handshake cleanup -- xsec_tls_close() in tls13.c (lines 1175-1195):**
The close function correctly zeroes:
- ecdhe_priv, shared_secret, early_secret, handshake_secret, master_secret
- client/server handshake traffic secrets
- client/server application traffic secrets
- key material struct

**Finding S4-SC-01 (P1 -- CONFIRMED PRIOR #5): TLS handshake error paths STILL leak key material.** xsec_tls_close() correctly zeroes all secrets, but it is only called on the CLOSE path. If the handshake fails at any of 20+ error returns in the handshake functions, the caller must explicitly call xsec_tls_close(). The architecture note says "xsec_tls_close() MUST be called even on handshake failure" but this is a documentation contract, not an enforcement mechanism. A caller who returns early on handshake error without calling close leaks key material.

### 8. UI: XSHELL Stage Fails -- Does GENSD Fall Back?

**Finding S4-UI-01 (P1 -- see S4-RG-01): NO fallback to minimal shell.**

The complete failure chain:
1. xchain_shell() fails (IPC channel creation failure) -> boot_chain.c line 594: `return rc`
2. xchain_run() returns non-PAL_OK to caller
3. In GENSD: gensd_start() calls gensd_run_loop() which does NOT call xchain_run() directly
4. If boot_chain is called from boot_sequence.c (the kernel path), the kernel's kmain would receive the error
5. The kernel has no TTY fallback handler

XSHELL_SESSION_FALLBACK (value 1) is defined and named in the greeter, but:
- No boot stage ever sets session_type = XSHELL_SESSION_FALLBACK
- No code path creates a fallback TTY session
- The greeter simply displays the name "Fallback TTY" as a dropdown option string -- it does not implement one

**Result: A shell failure = black screen with no recovery path.**

### 9. FUZZ: XSTORE_ERR_CORRUPT During WAL Replay

**WAL replay in wal.c (starting after line 87):**

Examining the replay logic: if a WAL entry has a checksum mismatch during replay, the behavior depends on the implementation. The WAL CRC covers bytes [16, sizeof) per xstore_wal_crc(). During replay, entries with bad CRCs should be skipped (treat as incomplete write).

**Finding S4-FZ-01 (P2): WAL replay behavior on corruption is NOT clearly defined.** The xstore_wal_replay() function is declared in xstore.h but I could not verify whether it skips corrupt entries or halts. The xblob_gc_log_replay() in gc.c (lines 73-77) correctly SKIPS entries with bad CRCs: `if (e->checksum != expected_crc) continue;`. If XSTORE WAL replay does the same, it is correct. If it halts on corruption, data committed before the corruption point is lost.

### 10. RELIABILITY: NVMe Driver Failure -> XSTORE Fallback to RAM

**Finding S4-RL-01 (P1): XSTORE has NO fallback to RAM disk.**

XSTORE's I/O is abstracted through function pointer callbacks (xstore.h lines 313-318: read_page, write_page, sync). The callbacks are provided by the caller at xstore_open() time. If the NVMe driver fails:
- The read_page/write_page callbacks will return XSTORE_ERR_IO
- XSTORE propagates the error to the caller
- There is NO internal fallback to a RAM-backed store
- There is NO mechanism to hot-swap the I/O callbacks after open

The system has no degradation path between "full NVMe-backed XSTORE" and "no storage at all."

### Black Box Analysis: What Does a GEN.OS Failure Look Like?

From a system administrator's perspective:

| Failure Scenario | Observable Behavior | Recovery Action | Verdict |
|-----------------|---------------------|-----------------|---------|
| NVMe dies during operation | Services that use XSTORE return errors; data loss for uncommitted txns | Reboot; XSTORE WAL replays committed data | ACCEPTABLE |
| RAM exhaustion | XMIND alloc_state returns XMIND_ERR_NOMEM; boot_chain AI stage fails | AI enters offline mode (non-critical) | GOOD |
| Network cable unplugged | XNET IPC probe fails during boot | Network marked offline; boot continues | GOOD |
| SHA-256 KAT fails | Security stage aborts boot chain | System halts -- no user data at risk | CORRECT |
| Display cable disconnected | xchain_display returns PAL_ERR_INVAL | Boot halts -- no framebuffer = no UI | PROBLEMATIC* |
| AI model file missing | xmind_weights_load returns XMIND_ERR_INVAL | AI offline, system operational | GOOD |
| XBLOB index full | xblob_put returns XBLOB_ERR_INDEX_FULL | Write rejected; reads still work | ACCEPTABLE |
| Service crashes during operation | Supervisor detects running=false | Auto-restart with backoff (1s->30s cap) | GOOD |
| All services crash simultaneously | Supervisor restarts one at a time (blocking) | System recovers in 30s worst case per service | PROBLEMATIC** |

*Display halt on disconnected cable is correct for a graphical OS but lacks serial console fallback.
**Blocking supervisor restart means cascading restart takes O(n * backoff) time.

### Grey Box Analysis: boot_chain Stage Failure Handling Code Path

Tracing a hypothetical xchain_compositor() failure:

1. `pal_pages_alloc(128, ...)` returns non-PAL_OK (e.g., PAL_ERR_NOMEM)
2. xchain_compositor() returns the PAL error code (line 402)
3. xchain_run() stores `s_result.compositor_ok = 0` (line 584)
4. xchain_run() prints `[XCHAIN] CRITICAL: compositor init FAILED` (line 586)
5. xchain_run() returns the error code (line 587)
6. The caller (boot_sequence.c or kmain) receives the error
7. If kmain: system halts
8. If GENSD: gensd_start() enters infinite sleep loop (line 534)

**Gap: No intermediate degradation.** The compositor is classified CRITICAL, meaning its failure halts the system. But a compositor failure should degrade to a text-mode console, not a halt. The boot chain has a binary critical/non-critical model with no intermediate levels.

## III. SPECIFIC CHECK RESULTS

### Check 1: boot_chain.c critical_failure vs non_critical_continue
**Result:** Correctly implemented for the binary model. See regression test (section 4) for the full matrix. The gap is the LACK of intermediate degradation levels.

### Check 2: supervisor.c health check always returns HEALTH_OK
**Result: CONFIRMED P1.** supervisor_probe_http() is a stub returning HEALTH_OK unconditionally (line 170). supervisor_probe_process() works correctly for process liveness only.

### Check 3: xstore_open() returns specific error codes
**Result: GOOD.** xstore_open() is declared to return xstore_status_t with 15 distinct error codes. The header documents precise semantics for each (XSTORE_ERR_CORRUPT for bad page 0, XSTORE_ERR_IO for backing store failure, etc.).

### Check 4: XBLOB_ERR_INDEX_FULL in xblob_put()
**Result: GOOD.** xblob_put() calls xblob_index_insert() (blob.c line 170) which returns XBLOB_ERR_INDEX_FULL when load factor exceeds 75% (index.c line 78). The error propagates cleanly to the caller. The put() also correctly frees the allocated storage slot on index insertion failure (line 173).

### Check 5: XNET_ERR_NOT_FOUND propagation
**Result: PARTIAL.** The boot_chain probes network by creating an IPC channel pair (boot_chain.c line 252). This tests IPC, not XNET. Real XNET errors are not tested during boot. XNET_ERR_NOT_FOUND would only be relevant if xnet_init() were called during boot, which it is not.

### Check 6: xmind_init() returns error on alloc failure
**Result: GOOD.** xmind_init() itself does no allocation -- it validates config and stores it (xmind.c lines 77-118). xmind_alloc_state() has a correct partial-failure cleanup path (lines 180-201): on any allocation failure, all previously allocated buffers are freed and XMIND_ERR_NOMEM is returned. The `goto oom` pattern is correctly implemented.

**Finding S4-XM-01 (P2): xm_heap_free() is a no-op.** (xmind.h line 113: `(void)ptr`). The OOM cleanup path correctly CALLS free on all buffers, but free does nothing. Memory allocated before the failure point is permanently leaked. This is documented as "Sprint 6 leak accepted" but it means repeated init/shutdown cycles leak PAL pages.

### Check 7: TCP retransmission max count
**Result: GOOD.** tcp.c line 1299: `if (oldest->retransmit_count > 12)` -- after 12 retransmissions, the connection is aborted (line 1301: `t->state = TCP_CLOSED`, line 1302: TCB zeroed). This matches the Sprint 3 audit finding: "TCP retransmit exhaustion: 12 retries then abort. Gold standard."

### Check 8: TLS handshake failure sends proper alert
**Result: PARTIAL.** xsec_tls_close() sends close_notify alert only when state is APPLICATION_DATA (line 1179). If the handshake fails before reaching APPLICATION_DATA state, NO alert is sent. This is technically correct per RFC 8446 (alerts during handshake are optional for the closer), but the remote peer receives a TCP RST instead of a TLS alert, making debugging harder.

### Check 9: giso_verify_manifest() failure -> boot abort
**Result: GOOD.** giso_verify_manifest() (gen_iso.c lines 265-303) validates magic, version, component count, paths, file sizes, sector counts, and SHA-256 hashes for required components. Returns PAL_ERR_INVAL on any failure. The boot chain does not directly call this during xchain_run(), but the test suite (test_sprint9.c) extensively tests it with 15+ test cases including NULL, bad magic, zero file size, and fuzz.

### Check 10: AetherBoot TPM PCR measurement failure
**Result: GOOD (graceful degradation).** tpm.c implements correct degradation:
- If EFI_TCG2_PROTOCOL not found: g_tpm_ok = FALSE, all subsequent calls are no-ops returning TRUE (line 325)
- If SubmitCommand fails: logs warning, returns FALSE (lines 371-373)
- If PCR response code non-zero: logs warning, returns FALSE (lines 378-380)
- Boot ALWAYS continues regardless of TPM failures
- PCR measurement failure is logged but never halts boot
