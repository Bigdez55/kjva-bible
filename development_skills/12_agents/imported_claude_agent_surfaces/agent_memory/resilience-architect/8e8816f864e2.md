### 1. XMIND HEAP DEATH SPIRAL — RA-XMIND-02 [FIXED]

**File:** `ai/xmind/src/xmind.c` lines 32-99
**Prior Status:** P1 OPEN — xm_heap_free was a no-op, ~336 MB leaked per inference cycle
**Current Status:** FIXED in Sprint 10

**Evidence of Fix (xmind.c lines 43-99):**

The Sprint 10 fix introduced a heap slab allocator:
- 128-slot static slab (`s_xm_slab[128]`) tracks `(va, ph, mh)` tuples
- `xm_heap_alloc()` (line 54-81): allocates via `pal_pages_alloc + pal_map_pages`, registers handles in slab
- `xm_heap_free()` (line 83-99): linear scan of slab, calls `pal_pages_free(ph)`, compacts via swap-with-last
- `xmind_shutdown()` (line 281-305): iterates all state buffers calling `xm_heap_free()` then zeroes pointers

**Memory Growth Analysis:**

| Scenario | Sprint 6 (no-op free) | Sprint 10+ (slab free) |
|----------|----------------------|------------------------|
| Single inference cycle | ~336 MB leaked | 0 MB leaked |
| 10 cycles | ~3.36 GB leaked → OOM | 0 MB leaked |
| Slab overflow (>128 allocs) | N/A | Handle untracked, WARN logged |

Llama 3.2 3B allocation count: 7 scratch + 2 x 28 layers = 63 allocations. Well under 128-slot capacity.

**Residual Issues:**

**RA-XMIND-05 [P2] — xm_heap_free does not call pal_unmap_pages()**
- Location: `xmind.c:89` — only calls `pal_pages_free(s_xm_slab[i].ph)`
- The mapping handle `mh` is stored in the slab but never freed via `pal_unmap_pages(mh)`
- On PAL-Aether: the physical pages are freed, but the virtual address mapping persists in the page table
- Impact: Virtual address space leak. Each alloc/free cycle leaks one VA mapping entry
- For 63 allocs per inference cycle, this is ~63 VA mappings leaked per cycle
- Not a death spiral (virtual address space is 48-bit = 256 TB), but a correctness bug

**RA-XMIND-06 [P3] — Slab overflow silently succeeds**
- Location: `xmind.c:76-79` — if slab is full, allocation succeeds but handle is untracked
- The allocation is returned to the caller but can never be freed
- Warning is logged but no error returned
- Fix: return NULL when slab is full, or dynamically grow the slab

**Verdict: RA-XMIND-02 CLOSED. Death spiral eliminated. Score: 4/10 → 7/10**

### 2. HEALTH PROBE STATUS — RA-GENSD-01 [PARTIALLY FIXED]

**File:** `orchestration/xorchestra/src/health.c` lines 69-95
**Prior Status:** P1 OPEN — health probe always returned OK
**Current Status:** PARTIALLY FIXED — process handle check added, HTTP probe still stub

**Sprint 10 Fix in health.c (lines 72-89):**

```c
// State RUNNING + handle INVALID + uptime > 5000ms → unhealthy
if (svc->proc_handle == PAL_HANDLE_INVALID) {
    uint64_t uptime_ms = (svc->start_time_ms > 0u)
        ? (now_ms - svc->start_time_ms) : 0u;
    if (uptime_ms > 5000u) {
        return 0; /* unhealthy */
    }
}
```

This is a real improvement: services that claim RUNNING but have PAL_HANDLE_INVALID for > 5 seconds are now flagged CRITICAL. The health_tick function (lines 144-159) then transitions such services to FAILED state and triggers restart.

**Remaining Gap — RA-GENSD-01b [P2]: HTTP health probe still returns HEALTH_OK**

**File:** `init/gensd/supervisor.c` lines 190-215

The supervisor's `supervisor_probe_http()` at line 214 still returns `HEALTH_OK` unconditionally:
```c
/* TODO(Sprint 11): wire xkabi_tcp_connect(host, port, timeout_ms). */
return HEALTH_OK;
```

The URL parsing (sv_parse_url) was added in Sprint 10 and validates the URL format, but the actual TCP connect is not wired. Any service with an HTTP health endpoint will always pass its probe, even if the service process died and the socket is closed.

**What a Real Health Check Requires:**
1. TCP connect to `host:port` via `xkabi_tcp_connect()` with configurable timeout (e.g. 3000ms)
2. Send HTTP GET `/healthz` request (raw bytes, no HTTP library needed)
3. Read response: check for "200" status line within timeout
4. On ECONNREFUSED: immediate HEALTH_FAIL
5. On timeout: HEALTH_TIMEOUT (treat as FAIL)
6. On non-200 response: HEALTH_FAIL

**Specific Changes Needed:**
- `supervisor.c:207-214`: Replace TODO block with `xkabi_tcp_connect(host, port, timeout_ms)` call
- `xkabi_capabilities.c`: Grant supervisor process `XKABI_RIGHT_NET_SEND` capability
- Add `sv_http_get()` function: 64-byte static request buffer, 256-byte response buffer
- Add timeout tracking via `pal_time_now_ns()` differential

**Verdict: RA-GENSD-01 PARTIALLY CLOSED. Process-level probe works. HTTP probe remains stub. Score: 5/10 → 6.5/10**

### 3. SUPERVISOR RESTART FIX — RA-GENSD-02 [FIXED]

**File:** `init/gensd/supervisor.c` lines 233-274, 393-437
**Prior Status:** P1 OPEN — blocking pal_thread_sleep_ns() in restart path froze monitoring
**Current Status:** FIXED — deferred non-blocking restart confirmed

**Exact Code Implementing Non-Blocking Restart:**

**supervisor_restart() (lines 250-259):**
```c
/* Non-blocking deferred restart (RA-GENSD-02 fix):
 * Record the future restart timestamp and return immediately so the
 * supervisor loop continues monitoring all other services during the
 * backoff window. */
uint64_t delay_ns = (uint64_t)delay_ms * 1000ULL * 1000ULL;
entry->restart_due_ns = pal_time_now_ns() + delay_ns;
```

**supervisor_tick() (lines 404-412):**
```c
/* Check deferred restart: if restart_due_ns is set and has elapsed,
 * perform the actual spawn (RA-GENSD-02 fix — non-blocking backoff). */
if (e->restart_due_ns != 0u && now >= e->restart_due_ns) {
    supervisor_do_restart(e);
    continue; /* skip liveness probe this tick */
}
/* Skip liveness probe while a restart is pending */
if (e->restart_due_ns != 0u) continue;
```

**Analysis:**

The fix is architecturally correct:
1. `supervisor_restart()` sets `restart_due_ns` to a future timestamp and returns immediately (no sleep)
2. `supervisor_tick()` checks all entries in a non-blocking loop each second
3. When `now >= restart_due_ns`, `supervisor_do_restart()` is called
4. Backoff is correctly updated via `sv_next_backoff()` (doubles, capped at 30s)

**Residual Issues:**

**RA-GENSD-07 [P3] — supervisor_do_restart is still a stub**
- Location: `supervisor.c:265-274` — sets `*entry->running = true` but does not actually spawn a process
- Comment says "TODO(Sprint 11): call xkabi_exec(entry->exec, rights)"
- The service is marked running without any process existing → next health check will fail after 5s grace
- This creates a restart loop: restart → mark running → 5s → health fail → restart again
- Not a blocking issue (the loop is non-blocking), but it wastes supervisor cycles

**RA-GENSD-08 [P2] — No maximum restart count in supervisor**
- The supervisor backoff caps at 30s but never stops trying
- The XORCHESTRA health.c has `XORCH_MAX_RESTARTS = 5` before marking FAILED
- The supervisor has no equivalent — a permanently broken service causes infinite retry at 30s intervals
- Fix: add `SV_MAX_RESTART_COUNT` (e.g. 10), after which set `active = false`

**Verdict: RA-GENSD-02 CLOSED. Non-blocking restart verified. Score: 5/10 → 7/10**

### 4. BOOT SHELL FALLBACK — RA-BOOT-03 [PARTIALLY FIXED]

**File:** `init/gensd/boot_chain.c` lines 594-609
**Prior Status:** P1 OPEN — shell failure = permanent halt
**Current Status:** PARTIALLY FIXED — demoted to WARNING, rescue console referenced but not implemented

**Evidence of Fix (boot_chain.c lines 594-609):**

```c
/* Stage 8: Shell IPC (non-critical: fall back to rescue serial console).
 * Demoted from CRITICAL to WARNING so a compositor/xshell spawn failure
 * does not permanently halt the system (RA-BOOT-03 fix). */
rc = xchain_shell();
s_result.shell_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] WARN: shell IPC failed — "
                       "rescue serial console active\n");
    pal_console_printf("[XCHAIN] RESCUE> type 'reboot' or 'halt'\n");
    /* NOTE: A real rescue console (serial TTY loop) would be launched
     * here via pal_thread_create() on the serial UART.  The thread
     * accepts 'reboot', 'halt', and basic diagnostic commands.
     * Sprint 10 wires the thread creation; the loop body is in
     * init/gensd/rescue_console.c (Sprint 10 deliverable). */
}
```

**Assessment:**

The shell stage was correctly demoted from CRITICAL to WARNING. If xchain_shell() fails, the boot chain now continues to Stage 9 (AI) and completes. This eliminates the permanent halt cliff.

However:

**RA-BOOT-03b [P1] — rescue_console.c does not exist**
- The comment at line 606 references `init/gensd/rescue_console.c` as a "Sprint 10 deliverable"
- Glob search for `**/rescue_console*` returns zero files
- This means the rescue console was never implemented
- When shell fails, the user sees "RESCUE> type 'reboot' or 'halt'" on serial output but has no way to actually type those commands — there is no UART input polling loop
- The system continues booting and reaches the event loop, but with no graphical shell and no interactive recovery path
- The user must power-cycle the device

**Impact: The halt cliff is softened (boot continues) but the "rescue serial console" is vaporware. The user has no interactive recovery when xshell fails.**

**Verdict: RA-BOOT-03 PARTIALLY CLOSED. Halt eliminated. Rescue console missing. Score: 6/10 → 6.5/10**

### 5. STORAGE BOOT ASSUMPTION — RA-BOOT-02 [FIXED]

**File:** `init/gensd/boot_chain.c` lines 89-124
**Prior Status:** P1 OPEN — xchain_storage always returned PAL_OK
**Current Status:** FIXED — storage now validates memory map and can fail

**Evidence of Fix (boot_chain.c lines 96-123):**

```c
if (handoff) {
    for (i = 0u; i < handoff->n_mem_regions; i++) {
        const xenos_mem_region_t *r = &handoff->mem_regions[i];
        /* ... check if XBLOB region is within a USABLE or RESERVED region ... */
        if (r->phys_base <= XCHAIN_BLOB_REGION_PHYS &&
            end >= blob_end &&
            (r->type == XENOS_MEM_USABLE || r->type == XENOS_MEM_RESERVED)) {
            return PAL_OK;
        }
    }
}
/* Region not found — fail (RA-BOOT-02 fix) */
return (pal_status_t)-1; /* PAL_ERR_NOT_FOUND */
```

And in xchain_run (lines 550-554):
```c
rc = xchain_storage(handoff);
s_result.storage_ok = (rc == PAL_OK) ? 1u : 0u;
if (rc != PAL_OK) {
    pal_console_printf("[XCHAIN] WARN: storage init failed (%u)\n", rc);
}
// Boot continues — storage is non-critical
```

**Assessment:**

Storage validation is now real. If the XBLOB region is not found in the memory map, the stage fails with a WARNING and boot continues. This is correct behavior — storage is marked non-critical, so AI and other services that depend on XBLOB will fail gracefully when they try to access it.

**Residual Issue:**

**RA-BOOT-09 [P3] — No RAM-only fallback mode**
- When storage fails, there is no explicit "RAM-only mode" activated
- Services that depend on XBLOB/XSTORE will fail individually when they attempt access
- A more robust design would set a global `g_storage_available` flag and have dependent services check it at startup rather than discovering the failure at runtime
- Not critical — the current design degrades per-service, which is acceptable

**Verdict: RA-BOOT-02 CLOSED. Storage validation real. Score: 6/10 → 7.5/10**

### 6. KERNEL PANIC HANDLER [NEW FINDING]

**File:** `kernel/xenos/core/main.c` lines 619-675
**Status:** No dedicated panic function exists. Fatal exceptions trigger halt.

**Current Behavior (main.c lines 656-661):**

```c
if (vec < 32u && _fatal_exc[vec]) {
    serial_early_puts("[XENOS] FATAL EXCEPTION — HALTING\n");
    __asm__ volatile("cli" ::: "memory");
    while (1) {
        __asm__ volatile("hlt" ::: "memory");
    }
}
```

Fatal exceptions (DE=0, NMI=2, UD=6, DF=8, GP=13, PF=14, MC=18) trigger:
1. Full register dump to console (RIP, CS, RFLAGS, RSP, ERR, RAX-RDI, RBP)
2. CR2 dump for page faults
3. Serial output: "[XENOS] FATAL EXCEPTION — HALTING"
4. CLI + infinite HLT loop

**Assessment:**

This is a minimal panic handler. It provides diagnostic output but has significant gaps:

**RA-KERNEL-01 [P1] — No reboot-on-panic capability**
- The kernel only halts. There is no automatic reboot after fatal exception
- No watchdog timer is set up (XHPET exists but no watchdog mode)
- The user must physically power-cycle the device
- Standard OS practice: configurable panic_timeout (e.g., reboot after 30s)
- Fix: After dumping registers, set XHPET one-shot timer for 30s, then trigger ACPI reset register write or triple fault

**RA-KERNEL-02 [P1] — No register dump to persistent storage**
- Register state is only written to serial (COM1) and console
- If no serial cable is attached, the diagnostic data is lost on power-cycle
- No attempt to write to NVMe or a reserved memory region
- Fix: Write panic frame to a reserved 4KB "panic log" page at a known physical address; read it on next boot

**RA-KERNEL-03 [P2] — No NMI-based watchdog for hang detection**
- Non-fatal hangs (infinite loop in kernel code, deadlock) produce no exception
- No periodic NMI watchdog that checks if the kernel made progress
- Fix: Configure XLAPIC timer to send NMI every 10s; ISR checks a progress counter

**RA-KERNEL-04 [P2] — No IPI panic broadcast to other CPUs**
- `XLAPIC_VEC_PANIC` (0x32) is defined in xlapic.h but never sent during fatal exception
- On SMP, a panic on one CPU should halt all others
- Fix: In the fatal exception path, call `xlapic_send_ipi_broadcast(XLAPIC_VEC_PANIC)` before halting

**RA-KERNEL-05 [P3] — Non-fatal exceptions silently continue**
- Exceptions not in `_fatal_exc[]` (e.g., #DE=0 divide error IS fatal, but #OF=4 overflow, #BR=5 bound range) are logged but execution continues
- Returning from a #GP or #PF without fixing the cause will immediately re-trigger the exception in an infinite loop
- The `_fatal_exc[]` array correctly marks the dangerous ones as fatal, but the non-fatal path does not fix the faulting condition

**Verdict: Kernel panic handler is MINIMAL. Diagnostic output exists. No reboot, no persistent dump. Score: 4/10**

## VI. GO/NO-GO VERDICT

### CONDITIONAL GO

**Rationale:**

The four prior P1 blockers targeted in this audit (RA-XMIND-02, RA-GENSD-02, RA-BOOT-02, RA-BOOT-03) are all confirmed fixed or substantially improved. The system's resilience score improved from 5.75/10 to 7.0/10, crossing the 6.0 threshold for "acceptable."

**GO conditions (must be tracked):**

1. **RA-BOOT-03b (rescue_console.c):** Implement the serial rescue console or accept that xshell failure requires physical power cycle. This is the most user-facing resilience gap.

2. **RA-KERNEL-01 (reboot-on-panic):** Implement configurable panic timeout with ACPI reset. Without this, any kernel fault requires physical intervention.

3. **RA-GENSD-01b (HTTP health probe):** Wire xkabi_tcp_connect or accept that HTTP health endpoints are not validated. Process-level health check provides partial coverage.

**NO-GO would be triggered by:**
- Any of the 4 fixed issues regressing (RA-XMIND-02, RA-GENSD-02, RA-BOOT-02, RA-BOOT-03)
- Discovery of a new death spiral or cascade failure path
- Kernel panic frequency exceeding acceptable threshold in integration testing
