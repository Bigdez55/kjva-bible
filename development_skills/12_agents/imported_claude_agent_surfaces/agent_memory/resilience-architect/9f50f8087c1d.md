## FMEA Table

### FM-01: TCP SYN Flood Exhausts Connection Table

| Attribute | Value |
|-----------|-------|
| **Component** | tcp.c |
| **Failure Mode** | SYN flood fills all 128 TCBs |
| **Probability** | HIGH (trivial to trigger externally) |
| **Impact** | CRITICAL -- no new TCP connections possible, all services denied |
| **Current Handling** | Backlog full: SYN dropped silently (line ~663). No SYN cookie. No rate limiting. No half-open tracking. |
| **Blast Radius** | System-wide -- TLS, DNS-over-TCP, HTTP, XPKG downloads all fail |
| **Detection** | None -- no counter for dropped SYNs, no log at SYN drop |

**Recommended Mitigations:**
1. **SYN cookies** -- stateless half-open connection handling. Encode ISN with hash(src_ip, src_port, dst_port, secret) to validate ACK without consuming a TCB.
2. **SYN flood rate limiter** -- per-source-IP rate limit (max 5 SYN/s per source). Track in a small bloom filter or mini hash table.
3. **Half-open timeout** -- SYN_RECEIVED entries older than 10s should be reaped (currently only checked in timer_tick but no explicit SYN_RECEIVED timeout exists).
4. **Monitoring** -- add `g_tcp_stats.syn_drops` counter, log periodic summary.
5. **Circuit breaker** -- if > 90% of TCBs are in SYN_RECEIVED, switch to SYN-cookie-only mode.

**Degradation Strategy:**
- Level 1: Normal operation (< 80% TCB utilization)
- Level 2: SYN rate limiting active (80-90% utilization)
- Level 3: SYN cookies only (> 90% utilization)
- Level 4: Accept only connections from known/cached IPs (emergency)

### FM-02: DNS Resolution Total Failure

| Attribute | Value |
|-----------|-------|
| **Component** | dns.c |
| **Failure Mode** | All 3 retries x 3 servers = 9 attempts fail |
| **Probability** | MEDIUM (network outage, all DNS servers down, or DNS amplification attack) |
| **Impact** | HIGH -- hostname resolution fails, returns XNET_ERR_TIMEOUT |
| **Current Handling** | Returns XNET_ERR_TIMEOUT after 5s total. Negative cache (60s TTL) for NXDOMAIN. No fallback to cached stale entries. |
| **Blast Radius** | Service-level -- browser navigation fails, XPKG repo lookups fail, TLS cert fetches fail |
| **Detection** | Caller receives XNET_ERR_TIMEOUT. No global DNS health metric. |

**Recommended Mitigations:**
1. **Stale cache serving** -- On DNS failure, serve expired cache entries with a "stale" flag (RFC 8767 DNS serve-stale). Current code sets `ce->in_use = false` on expiry -- should keep entry and mark `stale` instead.
2. **Local hosts file** -- Static name-to-IP mappings for critical infrastructure (k3s API server, Ollama, GENSD services).
3. **DNS health circuit breaker** -- Track consecutive failures per server. After 5 failures, mark server as "down" for 30s before probing again.
4. **Exponential backoff** -- Current retry timing is fixed (timeout_per_attempt = DNS_TIMEOUT_NS / DNS_MAX_RETRIES). Use exponential backoff: 500ms, 1s, 2s.
5. **Concurrent query** -- Query all 3 servers simultaneously, use first response (reduces latency and improves reliability).

**Degradation Strategy:**
- Level 1: Normal DNS resolution
- Level 2: Stale cache serving (expired entries returned with warning)
- Level 3: Local hosts file only (critical services still reachable)
- Level 4: IP-only mode (user must provide raw IPs)

**Critical Bug Found:** The `g_dns_pending` global state is a single struct. Only ONE DNS query can be in-flight at a time. Concurrent callers will corrupt each other's state. This is a serialization bottleneck and a correctness bug.

### FM-03: Gossip Protocol -- Majority Node Failure

| Attribute | Value |
|-----------|-------|
| **Component** | gossip.c |
| **Failure Mode** | > 50% of nodes fail simultaneously (power event, network partition) |
| **Probability** | LOW (requires coordinated failure) |
| **Impact** | HIGH -- surviving nodes cannot form quorum, all departed nodes marked DEAD eventually |
| **Current Handling** | SWIM protocol with SUSPECT timeout (5s) then DEAD. Dead nodes are removed from membership. No quorum concept. |
| **Blast Radius** | Component-level -- gossip state becomes inaccurate, service discovery breaks |
| **Detection** | `xnet_gossip_member_count()` will show declining count. Console logs for each DEAD declaration. |

**Recommended Mitigations:**
1. **Suspicion amplifier** -- Increase suspect timeout proportional to cluster size. Current flat 5s is too aggressive for large clusters. Use `SUSPECT_TIMEOUT * max(1, log2(N))`.
2. **Partition detection** -- If member_count drops below a threshold (e.g., half of previously known count) within one tick interval, enter "partition-suspected" mode. Delay DEAD declarations.
3. **Rejoin protocol** -- When a DEAD node comes back, allow it to rejoin by bumping incarnation. Current code handles this via GOSSIP_MSG_JOIN.
4. **Netsplit healing** -- When partitions merge, conflicting DEAD declarations must be resolved. The incarnation number mechanism handles this, but there is no explicit merge protocol.
5. **Persistent seed list** -- Store known seeds to disk. On restart, attempt to rejoin all seeds rather than just one.

**Degradation Strategy:**
- Level 1: Full mesh operational
- Level 2: Partition suspected -- delay DEAD declarations, increase ping frequency
- Level 3: Minority partition -- operate in read-only mode for distributed state
- Level 4: Solo mode -- operate independently, rejoin when connectivity returns

### FM-04: TLS Certificate Chain Validation Failure Mid-Handshake

| Attribute | Value |
|-----------|-------|
| **Component** | tls13.c + x509.c |
| **Failure Mode** | Certificate chain validation fails (expired, unknown CA, signature mismatch) |
| **Probability** | MEDIUM (expired certs are common; untrusted CAs in novel ecosystem) |
| **Impact** | HIGH -- TLS handshake aborts, connection fails |
| **Current Handling** | tls13.c Step 5 comment says "skip validation for now" -- the Certificate message is accepted WITHOUT verification. x509.c has verify_chain() but it is NOT called during handshake. RSA/ECDSA signatures are parse-only stubs (not verified). |
| **Blast Radius** | CRITICAL -- this is a security vulnerability. Any certificate is accepted. MITM attacks are trivial. |
| **Detection** | Audit log on handshake completion. No certificate rejection logs during handshake. |

**CRITICAL SECURITY FINDING:**
The TLS 1.3 client handshake (tls13.c line ~757) accepts ANY certificate without calling `xsec_x509_verify_chain()`. This means:
- No certificate expiry check during TLS
- No CA trust chain validation during TLS
- No signature verification on the Certificate message
- CertificateVerify is received but not validated against the certificate's public key

**Recommended Mitigations:**
1. **Integrate x509 verification** -- After receiving Certificate (Step 5), parse the certificate chain via `xsec_x509_parse()`, then call `xsec_x509_verify_chain()`.
2. **CertificateVerify validation** -- In Step 6, verify the CertificateVerify signature using the public key extracted from the peer's certificate.
3. **Trust anchor store** -- Embed a static array of trusted root CA certificates (Ed25519 roots at minimum).
4. **Certificate pinning** -- For known GEN.OS services (k3s, Ollama, XPKG repos), pin expected certificate public keys.
5. **Graceful TLS degradation** -- On cert failure, send TLS_ALERT_BAD_CERTIFICATE. For self-signed certs on local services, allow a "trust-on-first-use" (TOFU) mode with user confirmation.

**Degradation Strategy:**
- Level 1: Full certificate chain verification
- Level 2: Trust-on-first-use for local services (192.168.x.x, 10.x.x.x)
- Level 3: Certificate pinning only (no chain validation, pinned keys only)
- Level 4: TLS disabled, fall back to plaintext with warning (emergency only)

### FM-05: Power Failure During Package Installation

| Attribute | Value |
|-----------|-------|
| **Component** | install.c |
| **Failure Mode** | Power loss between file extraction (Step 4) and registry commit (Step 5) |
| **Probability** | LOW (laptop has battery, but still possible) |
| **Impact** | CRITICAL -- partially extracted files on disk, registry does not reflect install, package in liminal state |
| **Current Handling** | No transaction journal. No write-ahead log. Files are extracted sequentially (simulated in Sprint 3). Registry update is a single in-memory assignment. No crash recovery. |
| **Blast Radius** | Component-level to system-level -- broken package can prevent boot if it is a system package |
| **Detection** | None -- no consistency check at boot. No checksum verification of installed state. |

**Recommended Mitigations:**
1. **Write-ahead log (WAL)** -- Before starting install, write an intent record to a journal file: `{op: INSTALL, pkg: name, version: v, timestamp: t}`. On successful completion, mark the WAL entry as COMMITTED. On boot, scan WAL for incomplete entries and roll back.
2. **Atomic rename** -- Extract files to a staging directory first, then atomically rename to the final location. If power fails during extraction, the staging directory is cleaned up at boot.
3. **Registry snapshot** -- Before mutating the registry, snapshot the current state to a shadow file. On recovery, restore from snapshot if the WAL shows an incomplete operation.
4. **fsync barriers** -- After each file write and after registry update, issue fsync to ensure durability.
5. **Package state machine** -- Add XPKG_STATE_EXTRACTING between INSTALLING and INSTALLED. On boot, any package in EXTRACTING state is automatically rolled back.

**Degradation Strategy:**
- Level 1: Full WAL + atomic rename (no data loss)
- Level 2: WAL only (files may be partially written, but package is not registered)
- Level 3: Boot-time consistency scan (detect and repair orphaned files)
- Level 4: Manual recovery (user runs `xpkg repair`)

### FM-06: Circular Dependency in Package Graph

| Attribute | Value |
|-----------|-------|
| **Component** | dep_resolve.c |
| **Failure Mode** | Package A depends on B, B depends on A (directly or transitively) |
| **Probability** | LOW (but increases with ecosystem growth) |
| **Impact** | MEDIUM -- install fails with XPKG_ERR_CYCLE, user cannot install the package |
| **Current Handling** | Explicit cycle detection via Tarjan-style on_stack marker in iterative DFS. Returns XPKG_ERR_CYCLE with console log showing the cycle edge. |
| **Blast Radius** | Component-level -- only the specific install operation fails |
| **Detection** | Console log: "[dep-resolve] CYCLE: 'A' -> 'B'". Status code XPKG_ERR_CYCLE returned to caller. |

**Assessment:** This failure mode is already well-handled. The iterative DFS with explicit stack avoids recursion (important for freestanding). Cycle detection is O(V+E).

**Recommended Improvements:**
1. **Full cycle path reporting** -- Currently only reports the cycle edge. Walk the DFS stack to report the full cycle path (A -> C -> B -> A).
2. **Cycle-breaking policy** -- Allow packages to declare a dependency as "build-only" or "weak", which can be broken during cycle resolution.
3. **Pre-upload validation** -- XPKG repo index builder should detect cycles before packages are published.

### FM-07: Static Pool Exhaustion (Memory)

| Attribute | Value |
|-----------|-------|
| **Component** | All Sprint 3 modules |
| **Failure Mode** | Any static pool (TCBs, ARP entries, DNS cache, gossip members) reaches capacity |
| **Probability** | MEDIUM (128 TCBs on a single-user OS is tight under heavy browsing) |
| **Impact** | HIGH -- specific subsystem stops accepting new work |
| **Current Handling** | Varies by module: |
| | TCP: returns XNET_ERR_NOMEM, connection attempt fails |
| | ARP: evicts oldest STALE then RESOLVED (good LRU-ish policy) |
| | DNS: evicts oldest entry (good) |
| | Gossip: returns NULL from alloc, new member rejected |
| | XPKG graph: returns XPKG_ERR_CAPACITY |
| **Blast Radius** | Component to service level depending on which pool |

**Recommended Mitigations:**
1. **Pool utilization metrics** -- Add `pool_used / pool_max` counters for each static pool. Report via health check endpoint.
2. **Proactive eviction** -- At 80% capacity, begin aggressive eviction of stale/idle entries (TIME_WAIT connections, expired ARP, aged DNS).
3. **Pool sizing review** -- 128 TCBs may be insufficient. Consider increasing to 256 or implementing connection pooling for frequently-accessed services.
4. **Backpressure signals** -- When pools approach capacity, propagate backpressure to callers (e.g., TCP: increase SYN drop rate, DNS: refuse non-cached queries).

**Per-Pool Exhaustion Effects:**

| Pool | At Exhaustion | Degradation |
|------|---------------|-------------|
| TCP TCBs (128) | New connections fail | Close TIME_WAIT early, RST idle connections |
| ARP Cache (256) | Evict oldest (already done) | Acceptable -- LRU eviction works |
| DNS Cache (128) | Evict oldest (already done) | Acceptable -- eviction + re-query |
| Gossip (256) | New members rejected | Acceptable for HP EliteBook (max ~10 mesh nodes) |
| XPKG Graph (512) | Install fails | Acceptable -- 512 transitive deps is extreme |

### FM-08: ARP Cache Stale Entry Accumulation

| Attribute | Value |
|-----------|-------|
| **Component** | arp.c |
| **Failure Mode** | ARP cache fills with stale entries from hosts that have left the network |
| **Probability** | LOW (256 entries is generous; 30-min TTL with expiry sweep) |
| **Impact** | LOW -- stale entries are evicted by arp_find_free() policy |
| **Current Handling** | Well-designed eviction: FREE > STALE > oldest RESOLVED > oldest PENDING. TTL expiry in xnet_arp_expire() transitions RESOLVED -> STALE (at 30min) and STALE -> FREE (at 60min). PENDING entries timeout after 10s. |
| **Blast Radius** | Contained -- only affects ARP resolution latency |
| **Detection** | Console log on PENDING timeout. No stale entry counter. |

**Assessment:** This is the best-handled failure mode in Sprint 3. The ARP cache has:
- Proper state machine (FREE -> PENDING -> RESOLVED -> STALE -> FREE)
- Bounded pending queue (3 packets/entry)
- Explicit timeout for PENDING (10s)
- TTL-based expiry with periodic sweep
- Smart eviction policy

**Minor Improvements:**
1. **ARP cache stats** -- Track hit/miss/eviction counters for observability.
2. **Gratuitous ARP on link-up** -- Already implemented in netif_up(). Good.
3. **Duplicate IP detection** -- Already logs warning. Could escalate to interface down if persistent.

## Additional Failure Modes Identified During Analysis

### FM-09: DNS Single-Query Bottleneck (BUG)

| Attribute | Value |
|-----------|-------|
| **Component** | dns.c |
| **Failure Mode** | Concurrent DNS queries corrupt shared g_dns_pending state |
| **Probability** | HIGH (multiple services resolve names simultaneously) |
| **Impact** | HIGH -- wrong IP returned, query results mixed, data corruption |
| **Current Handling** | Global `g_dns_pending` struct with spinlock. Second query overwrites first query's state. |
| **Blast Radius** | System-wide -- any service doing DNS gets potentially wrong results |

**Root Cause:** `g_dns_pending` is a single global struct. `dns_do_resolve()` writes to it, then spins waiting for callback. If two threads call `xnet_dns_resolve()` simultaneously, the second call overwrites `g_dns_pending.qid`, and the first query's response will be discarded (QID mismatch) or the wrong response will be delivered.

**Fix Required:** Convert `g_dns_pending` to an array (e.g., 8 pending slots) keyed by QID. Each resolve allocates a slot, and the callback matches by QID to deliver to the correct waiter.

### FM-10: TLS Double Record Send (BUG)

| Attribute | Value |
|-----------|-------|
| **Component** | tls13.c |
| **Failure Mode** | tls_send_record_encrypted sends a spurious empty record before the real one |
| **Probability** | HIGH (every encrypted send triggers this) |
| **Impact** | MEDIUM -- extra empty TLS record sent on every encrypted write (line 257-258 calls tls_send_record_plain with NULL/0, then immediately re-sends the full header) |

**Root Cause:** Line 257: `rc = tls_send_record_plain(ctx, XSEC_TLS_CT_APPLICATION_DATA, NULL, 0u);` sends a 5-byte TLS record header with 0-length payload. Then lines 260-271 send the actual record. The receiver will see a spurious empty record.

**Fix Required:** Remove the call on line 257-258. The code on lines 260-271 already sends the complete header + ciphertext + tag.

### FM-11: Upgrade Rollback Marks Package as BROKEN

| Attribute | Value |
|-----------|-------|
| **Component** | install.c |
| **Failure Mode** | Upgrade fails at Phase 3 (new version install), rollback re-registers old version as BROKEN |
| **Probability** | LOW-MEDIUM (depends on gate check quality) |
| **Impact** | HIGH -- package exists in registry but marked BROKEN, not INSTALLED. May not function. |
| **Current Handling** | Line 467: `old_pkg_copy.state = XPKG_STATE_BROKEN;` |

**Fix Required:** On rollback, restore the old package to XPKG_STATE_INSTALLED, not BROKEN. The old version was functional before the upgrade attempt. Marking it BROKEN is incorrect.

## Chaos Experiment Designs

### CHAOS-01: TCP SYN Flood

```
Hypothesis: "When 200 SYN packets arrive in 1 second, the system should
             continue serving existing connections and reject new SYNs gracefully"
Steady State: 10 established TCP connections, all responsive
Fault Injection: Send 200 SYN packets from random source IPs to port 80
Observation: Measure existing connection latency, count dropped SYNs, check TCB utilization
Duration: 30 seconds
Abort Conditions: Existing connections become unresponsive
Recovery Verification: After flood stops, new connections succeed within 5 seconds
```

### CHAOS-02: DNS Blackhole

```
Hypothesis: "When all DNS servers are unreachable, cached entries continue to serve"
Steady State: DNS resolution succeeds for 5 known hostnames
Fault Injection: Set all g_dns_servers to unreachable IPs (0.0.0.1, 0.0.0.2, 0.0.0.3)
Observation: Measure cache hit rate, time to XNET_ERR_TIMEOUT, stale entry behavior
Duration: 10 minutes
Abort Conditions: System panic or infinite spin
Recovery Verification: After restoring valid servers, DNS resolves within 5 seconds
```

### CHAOS-03: XPKG Install Interrupt

```
Hypothesis: "When power fails during package install, the system boots cleanly
             and the package is not in a half-installed state"
Steady State: Clean system with 3 packages installed
Fault Injection: Kill the install process at each of the 6 steps (parse, gate, dep, extract, register, hook)
Observation: Registry consistency, orphaned files, boot success
Duration: Per step
Abort Conditions: System fails to boot
Recovery Verification: After reboot, xpkg_list_installed shows consistent state
```
