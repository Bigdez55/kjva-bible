## 2. P0 / P1 / P2 Findings

### P0 — BOOT BLOCKERS (Must fix before ship)

**P0-CI-01: Sprint 3-14 test files compiled with -fsyntax-only ONLY**
- Files: tests/test_sprint3.c through tests/test_sprint13.c (14 files)
- Impact: 1,810 assertions across 14 files are DECORATIVE (never executed)
- Evidence: All sprint3-14 .github/workflows/*.yml use `-fsyntax-only` for test compilation
- Sprint 15 workflow (sprint15.yml) now uses `-c` flag for kernel -- this is the fix pattern
- Recommendation: Migrate all test compilations to `-c` and link into QEMU test binary

**P0-TEST-01: Zero tests for PMM, VMM, Heap (kernel/xenos/mm/)**
- Files: pmm.c (~1000 LOC), vmm.c (~400 LOC), heap.c (~400 LOC)
- Impact: The physical memory manager, virtual memory manager, and kernel heap have ZERO behavioral tests
- Risk: Memory corruption bugs will be undetectable until runtime crash
- These are the most critical kernel subsystems -- a PMM bug = unbootable system

**P0-TEST-02: Zero tests for display + UI stack (Sprint 2)**
- Files: display/xdisp/ (5 files, ~7000 LOC), display/xcomp/ (6 files, ~5000 LOC), ui/xframe/ (16 files, ~8000 LOC)
- Impact: 20,000 LOC with zero test coverage
- Risk: Layout regressions, compositor bugs, widget interaction failures all undetected

**P0-TEST-03: Zero fuzz harnesses (AFL/libfuzzer) for external input parsers**
- Affected parsers: GGUF weights_loader.c, DNS dns.c, TLS tls13.c, X.509 x509.c, AML xacpi.c
- Impact: Adversarial input can trigger memory corruption in any of these parsers
- Evidence: The GGUF parser (weights_loader.c) reads untrusted file format data; no fuzzing exists

### P1 — HIGH RISK (Fix within 1 sprint)

**P1-SEC-01: xsec_tls_ctx_t is ~17.2 KB -- stack allocation danger**
- File: sec/xsec/include/xsec.h:383-429
- The struct contains record_buf[16640] (16.25 KB alone)
- Kernel stack is only 32 KB (main.c:61 KERN_STACK_SIZE=32768)
- Test files allocate this on the stack (test_sprint3.c:662, test_sprint3_fixes.c:487, test_sprint3_p2_gaps.c:486)
- In a real kernel context, a local xsec_tls_ctx_t consumes >50% of kernel stack
- Recommendation: TLS context must be heap-allocated or placed in static/global storage

**P1-SEC-02: No bounds check in xacpi AML PkgLength decoder for malicious DSDT**
- File: kernel/xenos/drv/xacpi.c section 6 (AML PkgLength decoder)
- The AML parser walks untrusted DSDT data from ACPI firmware
- Missing: Maximum recursion depth check, pointer bounds validation
- Risk: Crafted DSDT could cause read-beyond-buffer or stack overflow

**P1-SEC-03: LAPIC timer calibration uses busy-loop instead of HPET reference**
- File: kernel/xenos/drv/lapic.c:209 (volatile uint32_t delay = 1000000u)
- The calibration accuracy depends on CPU speed -- wildly inaccurate on different hardware
- Documented in comments as "safe stub for bare-metal bring-up" but never replaced
- Impact: Timer frequency may be off by 10x on real hardware vs. QEMU

**P1-NET-01: No test coverage for virtio-net or e1000 drivers**
- Files: net/xnet/drv/virtio_net.c (~700 LOC), net/xnet/drv/e1000.c (~700 LOC)
- These are new Sprint 15 deliverables with zero tests
- Risk: DMA buffer handling bugs could corrupt arbitrary memory

**P1-NET-02: TCP infinite loops lack timeout guards**
- File: net/xnet/transport/tcp.c:1042, 1169 -- `for (;;)` with no maximum iteration/timeout
- Risk: Malicious peer could cause CPU starvation

**P1-STORE-01: xstore_wal_entry_t is 4,288 bytes -- stack concern**
- File: store/xstore/include/xstore.h:191-202
- Contains key[256] + val[4096] = 4,352 bytes per entry
- WAL_BUF_SIZE = 65536 * 4288 = ~268 MB -- caller must provide this buffer
- Not a bug per se, but the entry size makes stack-local operations dangerous

**P1-BUILD-01: Test harness macros duplicated in every test file**
- Files: All 17 test_sprint*.c files duplicate ASSERT/ASSERT_EQ/ASSERT_NE etc.
- Should be extracted to tests/test_harness.h
- Risk: Inconsistent assertion behavior across files
