## V. CI TEST EXECUTION AUDIT -- THE -fsyntax-only PROBLEM

### Current State

**Every single CI workflow uses `-fsyntax-only`.** This means:
- Tests are COMPILED (syntax and type checking) but NEVER EXECUTED
- All 1,746 assertions across 329 test functions have NEVER RUN in CI
- A test containing `ASSERT_EQ(0, 1)` would PASS the current CI pipeline
- Test pass/fail is determined solely by compile correctness

### Sprint 13 Progress

Sprint 13 introduces `sprint13-qemu-dry-run` job:
- Installs OVMF + qemu-utils
- Runs `scripts/qemu-boot.sh --dry-run`
- Verifies the script generates a valid `qemu-system-x86_64` command
- This is a DRY RUN ONLY -- no kernel is actually booted
- BUT: this is the FIRST step toward real runtime test execution

### What Would Be Needed for Runtime Execution

**Option A: QEMU Full Boot (HIGH effort, HIGHEST value)**
- Link test files into xenos.elf as a test mode (`kmain_test` entry point)
- QEMU boots xenos.elf in test mode, runs test_sprint{N}_main()
- Capture serial output, parse PASS/FAIL counts
- Requires: linker script, ELF build, QEMU automation, serial console parsing
- scripts/qemu-boot.sh is the foundation for this path

**Option B: Host-Native Test Runner (MEDIUM effort, HIGH value)**
- Compile tests with `-DTEST_HOST` using standard libc on ubuntu-24.04
- Stub PAL functions (pal_console_printf -> printf, pal_page_alloc -> mmap)
- Link and execute directly on CI runner
- Requires: host-mode PAL stubs, `#ifdef PAL_FREESTANDING` guards (already present)
- Tests that call xhpet_ticks_to_ns, xmind_init, etc. would need the actual implementation linked

**Option C: QEMU User Mode (LOW effort, MEDIUM value)**
- Compile test + module .o files for Linux x86_64
- Run via qemu-user-static
- Simpler than full boot but requires Linux ABI for syscalls
- Not suitable for bare-metal code that uses MMIO

**Recommendation**: Option B first (fastest path to runtime), Option A as goal.
