## Critical Path Analysis — Zero External Runtime Dependencies

### AetherBoot replaces GRUB
**Status: SOURCE COMPLETE, ISO NOT INTEGRATED**
`devices/desktop/aetherboot/src/aetherboot.c` — PE32+ UEFI app, zero EDK2, zero gnu-efi, raw UEFI 2.10. Capability table handoff, KASLR, disk encryption gate all implemented. The ISO build pipeline in `build/chroot-setup.sh` still runs `debootstrap + grub-mkimage`. The transition gate is: AetherBoot must be compiled to `BOOTX64.EFI`, placed in the FAT32 ESP, and `build/chroot-setup.sh` must stop calling `configure_grub()`. This is an integration sprint, not a design sprint.

### XENOS replaces Linux kernel
**Status: SOURCE COMPLETE FOR CORE — DRIVER COVERAGE PARTIAL**
XENOS kernel is real: GDT/IDT/TSS, SYSCALL MSR, PMM buddy allocator, VMM 4-level page tables, HHDM, XKABI 60 syscalls, fiber scheduler, vDRAM PF handler, SMP IPI. Driver coverage for HP EliteBook x360 is present: XACPI, XHPET, XLAPIC, XPCI, NVMe, audio (ALC285), WiFi, fingerprint, NFC, smart card. The gap is: WiFi driver requires `iwlwifi` firmware blobs regardless of which kernel runs — this is a hardware constraint, not a software originality issue. The ISO still boots `linux-image-amd64` via GRUB. XENOS is the boot target via AetherBoot on bare metal.

### GENSD replaces systemd
**Status: SOURCE COMPLETE, NOT PID 1 IN ISO**
`init/gensd/` — supervisor, journal, splash, boot_chain, service orchestration, daemon mains all present. The ISO installs 9 systemd service units (`genos-shell.service`, `genos-platform-init.service`, etc.). GENSD becomes PID 1 when the ISO is replaced by a native XENOS boot image. Transitional state is documented and correct. The systemd units are wrappers for GEN.OS services, not for systemd functionality — the design is forward-compatible.

### XDISP replaces Linux DRM
**Status: ARCHITECTURALLY INDEPENDENT — USES DRM IOCTL PROTOCOL NOT libdrm**
This is the most nuanced finding. `devices/desktop/display/xdisp/drm.c` explicitly states: "Zero dependency on libdrm. All DRM ioctl numbers, command structs, and protocol constants are defined inline." The code opens `/dev/dri/card0` via raw syscall and issues ioctls directly. On the transitional ISO this talks to the Linux i915 driver. Under XENOS native, the same ioctl protocol applies — XENOS would implement the DRM KMS interface, or the display stack would consume the GOP framebuffer already mapped by AetherBoot (already in xenos_boot_info_t). This is a BRIDGE, not a violation. The protocol constants (DRM_IOCTL_BASE, mode struct layouts) are open specs.

### XCOMP replaces labwc
**Status: SOURCE COMPLETE — labwc IS NOT IN THE PRODUCTION PATH**
ADR-S26-01 is explicit: labwc was transitional (S1-S5), is still in the QEMU/CI boot test pipeline only. The production shell is `genos-shell` (Electron process container). XCOMP (`devices/desktop/display/xcomp/`) has compositor.c, blend.c, surface.c, fonts.c, animation.c, vector.c — original C. labwc remains in `build/config/packages.list` for CI validation only. Not a runtime dependency on production hardware.

### XFRAME replaces Electron
**Status: STRUCTURAL DEPENDENCY — MULTI-SPRINT MIGRATION, DOCUMENTED**
This is the largest remaining structural dependency. 9 Electron-based application packages ship: genos-shell, genesys-browser, go-notes, go-docs, go-drive, go-calendar, go-mail, go-sheets, go-slides. ADR-S26-01 formally classifies Electron as "process container only" — no Electron UI primitives in renderers. XFRAME (freestanding C, flexbox-subset layout, QuickJS TypeScript bridge) is the canonical replacement. The migration path is clear but multi-sprint. This dependency is STRUCTURAL but not a violation — it is documented, bounded, and architecturally contained.

### XSTORE replaces PostgreSQL/SQLite
**Status: XSTORE SOURCE COMPLETE — MIGRATION IN PROGRESS (PHASE 1 OF 4)**
`store/xstore/` — B+ tree, WAL, MVCC, txn, IO all present. `store/xstore/ffi/python_ffi.c` (511 LOC) and `xstore_adapter.py` (XStoreKV) provide the Python bridge. The docker-compose currently uses `IDENTITY_DATABASE_URL=sqlite:////tmp/identity.db` (SQLite, not PostgreSQL). The k8s manifests still reference `postgres:16-alpine` StatefulSet. Identity service uses SQLAlchemy on SQLite. Migration plan (4 phases) is documented in xstore_adapter.py header. Phase 1 (Update service pilot via DualWriteBackend) is done. Phases 2-4 remain.

### XBLOB replaces MinIO
**Status: XBLOB SOURCE COMPLETE — SYNC SERVICE NOT YET MIGRATED**
`store/xblob/` — blob.c, gc.c, index.c, name_index.c, kernel_bridge.c all present. `store/xblob/ffi/python_ffi.c` (395 LOC) provides Python bridge. The Sync service (`devices/cloud/platform/services/sync/app.py`) still imports `from minio import Minio` and requires `MINIO_ACCESS_KEY` and `MINIO_SECRET_KEY` env vars. The k8s manifests deploy `minio/minio:RELEASE.2025-01-20` as a StatefulSet. Phase 3 migration (Sync/MinIO → XBLOB) is planned but not executed. **The MinIO server binary is AGPL-3.0 — this is the single highest-priority license risk in the system.**

### XNET HTTP replaces FastAPI
**Status: NOT PLANNED — FASTAPI IS INTENTIONAL**
FastAPI is the correct tool for the platform services layer (Python microservices running in Docker/k8s). The platform services layer (identity, sync, gate, registry, provenance) is not part of the "100% original OS" claim — it is the cloud control plane running above the OS. XNET (`net/xnet/`) implements the native network stack for XENOS. There is no architectural reason to replace FastAPI with an XNET HTTP server for Python services. ADR-S46-01 does not list FastAPI as a migration target. This dependency is INTENTIONAL.

### Original VPN replaces WireGuard
**Status: ORIGINAL IMPLEMENTATION OF WIREGUARD PROTOCOL — NOT A DEPENDENCY**
`net/xnet/vpn/vpn.c` implements the WireGuard Noise_IKpsk2 protocol using XSEC crypto primitives (xsec_x25519_shared, xsec_chacha20_poly1305_encrypt, xsec_hkdf_extract via weak extern references). The vpn.h comment explicitly states: "implemented as an original freestanding C module — no WireGuard kernel module." The data model mirrors the WireGuard configuration format but calls no external WireGuard code. WireGuard Noise_IKpsk2 is an open protocol specification (Jason A. Donenfeld, public). Using a protocol spec in an original implementation is identical in legal character to implementing TCP/IP. No GPL-2.0 WireGuard kernel module is linked or imported anywhere in the native XENOS path.

## License Risk Assessment

### CRITICAL — MinIO AGPL-3.0
MinIO's server binary is AGPL-3.0. Under AGPL, if you run MinIO as a networked service and distribute modified versions, you must provide source. More critically for GEN.OS: if GEN.OS is distributed as a product and includes the MinIO binary in its ISO or Docker images, the AGPL-3.0 copyleft may require open-sourcing the combined work or obtaining a commercial license from MinIO. The current k8s manifests and docker-compose reference `minio/minio:RELEASE.2025-01-20T14-49-07Z`. This is the single license exposure that could constrain GEN.OS distribution. **Resolution: Complete Phase 3 migration (XBLOB) before any commercial distribution. Until then, document that MinIO is a pre-production transitional dependency not bundled in any distributed artifact.**

### HIGH — firmware-iwlwifi / firmware-misc-nonfree
Intel proprietary firmware blobs. Cannot be replaced with original code — they are binary microcode for hardware. The GEN.OS "100% original software" claim correctly scopes to the operating system layer, not hardware firmware. However redistribution terms must be verified blob-by-blob. Standard approach: document firmware provenance, redistribute under terms that allow it (typical Intel firmware allows redistribution in OS distributions).

### MEDIUM — psycopg[binary] LGPL-3.0
LGPL-3.0 is satisfied by dynamic linkage. The `psycopg[binary]` package ships a pre-compiled `.so` dynamically loaded by Python. This does not trigger LGPL static-linking concerns. Once XSTORE migration eliminates PostgreSQL entirely, this risk disappears. Low priority.

### MEDIUM — GRUB GPL-3.0+ in EFI partition
GPL-3.0 requires source availability notice for GRUB if distributed. Standard Debian-derivative practice: include the GPL text and note that GRUB source is available at `https://www.gnu.org/software/grub/`. This is a standard compliance notice, not a structural risk. Transitional only — AetherBoot eliminates it.

### LOW — MIT/Apache-2.0/BSD (React, Electron, FastAPI, tiptap, etc.)
All require license text inclusion in distribution. Already noted in LICENSE-THIRD-PARTY.md. No copyleft risk.

### CLEARED — WireGuard GPL-2.0
The WireGuard kernel module (GPL-2.0) is NOT used. XVPN implements the Noise_IKpsk2 protocol specification in original C using XSEC primitives. Protocol specifications are not copyrightable. No GPL-2.0 exposure.

### CLEARED — Ed25519 ref10 carry schedule
Public domain (SUPERCOP/NaCl). Formally governed by ADR-S46-02. No license obligation.

## The "100% Original" Claim — Assessment

### What the claim means precisely

The claim is accurate for a specific and important scope: **the GEN.OS native operating system stack** — every component that runs below the application layer when booting on bare metal:

- AetherBoot: 100% original PE32+ UEFI loader
- XENOS: 100% original freestanding C microkernel
- GENSD: 100% original init system
- XDISP/XCOMP: 100% original display/compositor (uses DRM ioctl protocol, not libdrm)
- XNET: 100% original network stack (implements WireGuard Noise spec, not the module)
- XSEC: 100% original crypto library (public domain Ed25519 carry schedule — not a violation)
- XSTORE/XBLOB: 100% original storage engines
- XMIND/XJIT: 100% original inference engine and JIT compiler
- XPKG/XFRAME/XISC/PAL: all 100% original
- XSHELL (C layer): 100% original

### Where the claim requires qualification

The claim requires qualification in three areas:

**1. The ISO build pipeline is not the OS.** The current ISO boots a Debian-derived rootfs with Linux kernel, GRUB, and systemd as transitional scaffolding. These are explicitly documented in ADR-S46-01 as transitional. The "100% original" claim applies to GEN.OS Mode 3 Native — not to the transitional Mode 2 ISO. This distinction is clear in GENOS_DOCTRINE_v1.md and all relevant ADRs. It is not a contradiction.

**2. The desktop application layer uses Electron.** 9 applications use Electron as a process container. ADR-S26-01 establishes the formal boundary: Electron is the process host, XFRAME + React (original TypeScript) is the UI framework. No Electron UI primitives appear in renderers. The rendered output is 100% original XFRAME design. However, Electron/Chromium is the runtime environment. This is the most visible qualification to the claim and the longest migration timeline.

**3. The cloud platform services layer uses FastAPI, SQLAlchemy, and MinIO.** This layer is above the OS, not part of it. The claim was never "our cloud services use no frameworks" — it was "our OS kernel and runtime are 100% original." FastAPI is intentional and correct. SQLAlchemy and MinIO are transitional, with documented migration paths.

### Precise formulation that is accurate

"The GEN.OS native operating system — kernel, bootloader, init system, display stack, network stack, storage engines, cryptography library, AI inference engine, JIT compiler, package manager, UI toolkit, and shell — is 100% original C code with zero third-party libraries. Application delivery currently uses Electron as a transitional process container, with migration to the native XFRAME runtime planned. Cloud platform services use industry-standard Python frameworks above the OS layer. The ISO build pipeline uses transitional Debian scaffolding during the pre-native-hardware-boot phase."

## Prioritized Elimination Roadmap

### Priority 1 — CRITICAL RISK: MinIO AGPL-3.0 (Must complete before distribution)
- Complete XBLOB Phase 3 migration in sync/app.py
- Remove `minio/minio` from k8s manifests and docker-compose
- Replace `from minio import Minio` with XBLOB Python FFI calls
- Gate: No production distribution until this is done

### Priority 2 — HIGH VALUE: XSTORE migration (Identity service — Phase 4)
- Identity service (SQLAlchemy on SQLite) is the highest-risk migration (auth system)
- DualWriteBackend pattern from xstore_adapter.py is the correct approach
- Complete Phases 2-3 (Provenance, Sync) first to validate the pattern at lower risk
- Gate: Audit pass on each phase before advancing

### Priority 3 — HIGH VISIBILITY: AetherBoot ISO integration
- AetherBoot source is complete — this is purely an integration task
- Modify `build/chroot-setup.sh` to compile aetherboot.c → BOOTX64.EFI and place in ESP
- Remove `configure_grub()` call; remove `grub-efi-amd64` from packages.list
- Gate: QEMU boot test passes with AetherBoot + XENOS

### Priority 4 — HIGH VALUE: XENOS native boot ISO
- Replace `linux-image-amd64` with XENOS kernel binary
- Replace systemd PID 1 with GENSD — requires GENSD exec launch path fully wired
- Remove systemd from packages.list; remove all `.service` units from chroot setup
- Gate: Full boot test with XENOS → GENSD → genos-shell on target hardware

### Priority 5 — MEDIUM: Electron → XFRAME migration (long-horizon)
- Per ADR-S26-01, XFRAME is the canonical target
- Migration order: companion (simplest) → Orange Suite → genos-shell (last, highest risk)
- Each app requires XFRAME widget parity before migration
- Gate: Each app must pass full E2E test suite after migration

### Priority 6 — MEDIUM: Remove labwc from packages.list
- labwc is already not in the production runtime path (ADR-S26-01)
- Remove once XCOMP is the validated compositor in CI boot tests
- Low urgency since it is already correctly non-production

### Priority 7 — LOW: kernel/linux-validation/ removal
- Per ADR-S44-01, remove after mesh driver native port is complete
- All other three subsystems (vDRAM, scheduler, capabilities) are validated
- Single remaining gate: XENOS mesh driver in kernel/xenos/drv/
