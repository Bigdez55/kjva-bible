# XISC Architecture — Deep Dive Reference (2026-04-06)

## What XISC Is

XISC (Cross Instruction Set Computing) is the **computational model** at the heart of GEN.OS.
It is NOT an ISA, NOT a translator, NOT a compatibility layer. It answers: what does
computation MEAN independent of physical hardware? The contract is XKABI.

"XKABI is the constitution. Translation is a replaceable service behind it.
If anything bypasses XKABI, it is technical debt by definition."

## The 30-Opcode XISC Bytecode ISA

Defined in `xisc/include/xisc.h`. 32-bit instruction encoding:
  [opcode:8][rd:4][rs1:4][rs2:4][tier:4][imm:8]

Arithmetic group (0x00-0x05):
  0x00 NOP | 0x01 ADD | 0x02 SUB | 0x03 MUL | 0x04 DIV | 0x05 MOD

Memory with tier policy (0x10-0x13):
  0x10 LOAD  (rd, [rs+imm], tier)
  0x11 STORE ([rd+imm], rs, tier)
  0x12 MOVE  (between tiers)
  0x13 PREFETCH (hint to tier)

Control flow (0x20-0x26):
  0x20 JMP | 0x21 JEQ | 0x22 JNE | 0x23 JLT | 0x24 JGE | 0x25 CALL | 0x26 RET

Capability / XKABI (0x30-0x32):
  0x30 CAP_DERIVE (narrow handle)
  0x31 CAP_INVOKE (invoke via cap)
  0x32 CAP_REVOKE (revoke cap)

Merkle / verifiable (0x40-0x41):
  0x40 COMMIT (emit Merkle commitment)
  0x41 VERIFY (verify proof against root)

Metadata (0x50):
  0x50 ANNOTATE (embed policy annotation, NOP in codegen)

AI / HPC compute (0x60-0x63):
  0x60 MATMUL | 0x61 MOELOAD | 0x62 ALLREDUCE | 0x63 ALLGATHER

Halt:
  0xFF HALT

Tier hints: 0x00=AUTO 0x01=GPU(T0/HBM) 0x02=DRAM(T1) 0x03=NVMe/CXL(T2) 0x04=Remote RDMA(T3)

PLUS 32 cognitive opcodes in xcog.h (0x70-0x8F):
  Routing 0x70-0x76: DIRECT EXEC MEMORY SALIENCE LATERAL HIER BROADCAST
  Content 0x80-0x8D: INTENT ENTITY PREDICATE FILTER TEMPORAL CONTEXT SALIENCE EMIT
                     NEGATE QUANTIFY MODALITY REFERENCE SEQUENCE COMPOUND

xcog_instr_t is 64-bit (vs 32-bit xisc_instr_t): [opcode:8][subtype:8][ref:16][payload:32]

## Capability System — Handle Table Implementation

Source: `xisc/runtime/xkabi_native/native_internal.h` + `native_handles.c`

Table size: 4096 handles (NATIVE_MAX_HANDLES), 4096 objects (NATIVE_MAX_OBJECTS)
Handle type: xk_handle_t = uint32_t (opaque, 0 = invalid)
Handle entry fields:
  obj_id (u64), type (xk_obj_type_t), rights (u32), lineage_root (u64),
  generation (u64), grant_timestamp (u64), depth (u32), valid (u32),
  obj_index (u32 -- OPT-01: O(1) direct index into objects[]),
  next (u32 -- OPT-02: freelist for O(1) allocation)

12 capability rights (bits 0-11 of xk_rights_t):
  XK_RIGHT_READ(0) WRITE(1) EXECUTE(2) MAP(3) DUP(4) TRANSFER(5)
  SIGNAL(6) WAIT(7) INSPECT(8) MANAGE(9) GRANT(10) REVOKE(11)

11 hardware rights (bits 12-22, RFC-002 v5.1):
  MEM_REGISTER(12) RDMA_SEND(13) RDMA_RECV(14) STORAGE_READ(15) STORAGE_WRITE(16)
  NET_SEND(17) NET_RECV(18) GPU_EXEC(19) MESH_JOIN(20) ATTEST(21) DERIVE_HW(22)

7 cognitive rights (bits 23-29, xkabi_cog.h):
  COG_OBSERVE(23) COG_REASON(24) COG_COMMAND(25) COG_CREATE(26)
  COG_CONFIGURE(27) COG_ESCALATE(28) COG_EVOLVE(29)

XK_RIGHTS_ALL = 0x3FFFFFFF (bits 0-29)
XK_RIGHTS_VMO_DEFAULT = 0x00000FFF (bits 0-11, no hardware/cognitive rights)

Revocation (BUG-3.1 two-phase fix):
  Phase 1 (under lock): scan table, collect all entries sharing lineage_root into stack array
  Phase 2 (per victim, NOT under handle lock): release object refcount, then re-acquire lock
  to invalidate slot and push to freelist. Prevents UAF from holding lock during release.

ABA guard: native_object_lookup_fast() verifies obj->id == handle_entry->obj_id before
returning pointer. Generation counter on handle entry detects use-after-close.

## VA Security (xisc_va_security.h)

Canonical user addresses: bits 63:48 = 0, VA <= 0x00007FFFFFFFFFFF (128 TB)
Kernel addresses: bits 63:48 = 0xFFFF (all ones)
Macros:
  XISC_VA_IS_CANONICAL_USER(va)      -- check before dereferencing user pointers
  XISC_VA_IS_CANONICAL_KERNEL(va)    -- detect kernel addr passed as user buffer (REJECT)
  XISC_VA_VALIDATE_USER_BUFFER(ptr, size) -- check start canonical + no overflow + no wrap
  XISC_VA_VALIDATE_USER_STRING(ptr, max_len)

Both xk_vmo_read/write and xk_channel_write/read call XISC_VA_VALIDATE_USER_BUFFER
before any operation. This is the primary defense against kernel address leaks.

## Memory Intent Model (xkabi_memory.h)

xk_mem_intent_t enum:
  XK_MEM_LOCAL(0)      -- thread-local, no ordering, zero overhead
  XK_MEM_COHERENT(1)   -- shared-read, acquire on load
  XK_MEM_EXCLUSIVE(2)  -- shared-write, release/acquire
  XK_MEM_SEQUENTIAL(3) -- total order (expensive)
  XK_MEM_DEVICE(4)     -- uncacheable, MMIO
  XK_MEM_GUEST_TSO(128)-- x86 personality compat
  XK_MEM_GUEST_WEAK(129)-- ARM personality compat

Intent is immutable after VMO creation. Changing intent requires creating a new VMO.
Child VMO modes: XK_VMO_CHILD_COPY(0) XK_VMO_CHILD_COW(1) XK_VMO_CHILD_SLICE(2)
COW and SLICE are mutually exclusive.

## Channel IPC Implementation

Source: `xisc/runtime/xkabi_native/native_channel.c`

One 4KB page per channel pair, split into two 2048-byte ring halves:
  Half 0 (bytes 0-2047): A→B direction
  Half 1 (bytes 2048-4095): B→A direction

Ring layout: [head:4][tail:4][data:2040]
Wire format: [data_len:4][data:N][handle_count:4][handles:M]
Max data: XK_CHANNEL_MAX_DATA = 65536 (64 KiB) per spec, but ring is 2040 bytes practical
Max handles: XK_CHANNEL_MAX_HANDLES = 64

Write path: length bytes encoded individually (not as uint32_t*) to handle ring wrap safely.
Two-phase memcpy (OPT-04) for payload: ~100x speedup over byte-by-byte.
MPSC safety: per-channel spinlock on ring_lock field of native_object_t.

End determination: end=0 writes to ring[0], reads from ring[1]. end=1 inverts.

## XISC-to-XJIT Bridge (xisc_xjit_bridge.c)

Entry point: xisc_xjit_compile(code, n_instrs, func)
Returns via: xisc_jit_execute(func, a0, a1, a2, a3)

XISC → XJIT opcode mapping (1:1 where possible):
  NOP→NOP ADD→ADD SUB→SUB MUL→IMUL LOAD→LOAD STORE→STORE MOVE→MOV_RR
  JMP→JMP JEQ→JEQ JNE→JNE JLT→JLT JGE→JGT CALL→CALL RET→RET
  MATMUL→VEC_FMA_231 ANNOTATE→NOP HALT→RET
  DIV/MOD: lowered to CALL (no XJIT DIV opcode)
  CAP_DERIVE/INVOKE/REVOKE: lowered to MOV_RI(syscall_nr) + 3x MOV_RR + CALL

Tier-aware memory: if tier != AUTO, emit NOP with tier in imm before LOAD/STORE.
Cap syscall numbers: CAP_DERIVE=20, CAP_INVOKE=21, CAP_REVOKE=22.

JIT calling convention: xjit_fn_t = uint64_t(*)(u64,u64,u64,u64) — 4 args max.

## Architectural Policies (ENFORCED, 6 hard rules)

P1: Personalities NEVER call host syscalls directly. Only XKABI.
P2: No Linux headers above XKABI boundary.
P3: Translator executes instructions only; no process/memory/IPC/security ops.
P4: Conformance tests define correctness. If test fails, implementation is wrong.
P5: Rights can only narrow. No mechanism widens rights. Ever.
P6: Audit every handle create/dup/transfer/revoke/close. Feeds GEN.OS Provenance Service.

## P1 Defects — Deep Investigation 2026-04-11

### D-01 native_channel.c — Read-lock targets wrong object (LIVE DATA RACE)

File: xisc/runtime/xkabi_native/native_channel.c, line 294 (xk_channel_read)

RACE: xk_channel_read on end0 acquires obj0->ring_lock before reading ring[1].
xk_channel_write on end1 acquires obj1->ring_lock before writing ring[1].
These are DIFFERENT spinlocks protecting the SAME memory. Zero mutual exclusion.

CORRECT INVARIANT: ring[D] is owned by the endpoint whose write_direction==D.
  ring[0] owned by obj0 (end0 writes it) → protected by obj0->ring_lock
  ring[1] owned by obj1 (end1 writes it) → protected by obj1->ring_lock

FIX: In xk_channel_read, resolve peer via native_object_lookup(obj->backend.channel.peer_obj_id)
and acquire peer_obj->backend.channel.ring_lock instead of obj->ring_lock.
If peer_obj is NULL (peer closed), return XK_ERR_NOT_READY.

There are NO other lock ordering violations in the write path. The write path
correctly acquires obj->ring_lock before writing to ring[write_direction].

### D-02 native_vmo.c / native_objects.c — SLICE/COW child double-unmap/leak

File: xisc/runtime/xkabi_native/native_objects.c lines 64-72 (native_object_release VMO case)
File: xisc/runtime/xkabi_native/native_vmo.c lines 144-176 (SLICE), 180-222 (COW)

DEFECT: SLICE and COW children store parent's pages_h and map_h directly into
obj->backend.vmo.pages_h and obj->backend.vmo.map_h. When native_object_release
fires on the child (refcount → 0), the cleanup code calls:
  pal_unmap(obj->backend.vmo.map_h)       -- unmaps PARENT's mapping
  pal_handle_close(obj->backend.vmo.pages_h) -- closes PARENT's page allocation
This is a double-unmap if the parent is still alive, or a leak if it was already freed.

REF-COUNT MODEL: parent->refcount is incremented at SLICE/COW child creation (line 162/207),
so the parent object itself won't be freed prematurely. But the RESOURCE handles
(map_h, pages_h) are not reference-counted at the PAL level.

FIX: For SLICE/COW children (child_type != XK_VMO_CHILD_COPY), native_object_release
must NOT call pal_unmap or pal_handle_close. Instead it must decrement parent's refcount
and null out the child's map_h/pages_h fields before the switch falls through.
Add a child_type guard at the top of the VMO release case:

  if (obj->backend.vmo.child_type != 0) {  // not an independent copy
      obj->backend.vmo.map_h = PAL_HANDLE_INVALID;   // don't close parent's handle
      obj->backend.vmo.pages_h = PAL_HANDLE_INVALID;
      // decrement parent refcount
      native_object_t *par = native_object_lookup(obj->backend.vmo.parent_obj_id);
      if (par) native_object_release(par);
      // then fall through to freelist push
  }

The normal-path (COPY children and root VMOs) is correct — they own their own handles.

### D-03 native_wait.c — Spurious always-signaled (not an ABA, not an edge trigger)

File: xisc/runtime/xkabi_native/native_wait.c lines 76-90 (xk_wait_many)

DEFECT: Both XK_OBJ_EVENT (line 79) and XK_OBJ_CHANNEL (line 88) hardcode
is_signaled = 1 with comment "Simplified: assume signaled". This is not a
missing edge-trigger check or a condition variable ABA — it is incomplete
implementation. Every xk_wait_many() call returns XK_OK immediately with
all items marked pending, regardless of actual object state.

FIX for XK_OBJ_CHANNEL: Read head and tail from the peer's ring under the
peer's lock (same fix domain as D-01). Signal when head != tail:
  channel_ring_t *r = channel_get_ring(obj->backend.channel.ring_vaddr,
                                       1 - obj->backend.channel.end);
  uint32_t h = __atomic_load_n(&r->head, __ATOMIC_ACQUIRE);
  uint32_t t = __atomic_load_n(&r->tail, __ATOMIC_ACQUIRE);
  is_signaled = (h != t) ? 1 : 0;
Note: the read here does NOT need the peer lock because we are only
sampling head/tail for a non-binding readiness check (poll semantics).
The actual consumption is protected by the lock in xk_channel_read.

FIX for XK_OBJ_EVENT: Read obj->backend.event.signals and check whether
any bits in items[i].wait_for are set:
  xk_signals_t sigs = __atomic_load_n(&obj->backend.event.signals, __ATOMIC_ACQUIRE);
  is_signaled = (sigs & items[i].wait_for) ? 1 : 0;

FIX for XK_OBJ_TIMER: The existing check compares now_ns >= deadline_ns where
deadline_ns is the WAIT deadline, not the timer's deadline. The timer object
needs its own deadline field checked:
  is_signaled = (now_ns >= (uint64_t)obj->backend.timer.slack_ns) ? 1 : 0;
(slack_ns is currently the only time field on the timer object — a dedicated
deadline field should be added when xk_timer_set is implemented).

### D-04 native_vmar.c — VA truncation when stored as handle

File: xisc/runtime/xkabi_native/native_vmar.c line 136 (xk_vmar_allocate)

DEFECT: Line 136: *out = (xk_handle_t)vaddr;
xk_handle_t is uint32_t (confirmed by _Static_assert in native_internal.h: sizeof == 4).
uintptr_t on x86-64 is 64 bits. PAL assigns virtual addresses that are valid
x86-64 user-space VAs in the range [0x400000, 0x7FFF_FFFF_FFFF]. Any VA above
0xFFFF_FFFF (4 GB) will have its high 32 bits silently truncated. On a system
with ASLR or any VA above the 4 GB mark, the returned handle is completely wrong.
All subsequent xk_vmar_map/unmap/protect calls receive this corrupt value.

The same is NOT present in xk_vmar_map (line 196) which correctly returns
via *addr (uintptr_t*) rather than the handle slot.

FIX: xk_vmar_allocate must NOT put the VA into the xk_handle_t output slot.
It must allocate a proper handle table entry (via native_handle_alloc) for a
new XK_OBJ_VMAR object that stores the vaddr internally. The handle index (u32)
is then returned in *out. The mapping_entry_t table already tracks the relationship
between vaddr and map_h, so xk_vmar_map/unmap/protect can use the handle to look
up the vaddr from the entry.

Minimal fix without full VMAR object model:
  Allocate a native_object_t of type XK_OBJ_VMAR, store vaddr in a repurposed
  field (e.g., backend.vmo.mapped_vaddr), call native_handle_alloc, store result in *out.
  xk_vmar_map/unmap/protect then resolve (xk_handle_t vmar) → object → vaddr.
