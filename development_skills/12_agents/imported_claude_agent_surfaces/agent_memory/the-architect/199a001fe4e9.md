## Integration Point 2: XNET ↔ XSEC (TLS 1.3)

### Assessment
TLS 1.3 uses a **pluggable transport callback** pattern (`xsec_tls_transport_t`):
```c
typedef struct {
    int32_t (*send)(void *ctx, const uint8_t *buf, uint32_t len);
    int32_t (*recv)(void *ctx, uint8_t *buf, uint32_t buf_len);
    void *ctx;
} xsec_tls_transport_t;
```
This is the correct adapter pattern — XSEC does NOT import xnet.h,
preserving strict layering (XSEC has no knowledge of XNET internals).

### Gap G-01: Missing xnet_tls_adapter glue (CRITICAL for Sprint 4)
XNET provides `xnet_tcp_send/recv` and `xnet_send/recv` (socket API).
XSEC expects send/recv callbacks with signature `int32_t (*)(void*, uint8_t*, uint32_t)`.

There is NO existing adapter that wraps an XNET socket descriptor into
an `xsec_tls_transport_t`. This glue must be written in Sprint 4.

Required glue (does not exist yet):
```c
// File: net/xnet/tls/xnet_tls.c (new)
static int32_t xnet_tls_send_fn(void *ctx, const uint8_t *buf, uint32_t len);
static int32_t xnet_tls_recv_fn(void *ctx, uint8_t *buf, uint32_t buf_len);
xnet_status_t xnet_tls_wrap(int sd, xsec_tls_ctx_t *out_tls_ctx, bool is_client);
```

### Gap G-02: Server-side TLS handshake stub (MEDIUM)
`xsec_tls_handshake()` returns `XSEC_ERR_NOT_SUPPORTED` for server role
(tls13.c line 905). Client-only TLS is not sufficient for:
- Package repository server connections where GEN.OS acts as server
- Future device-to-device protocols
Server handshake planned Sprint 4. Must be tracked.

### Gap G-03: Certificate validation not wired in TLS handshake (HIGH)
tls13.c Step 5 receives the Certificate message but performs NO validation:
```c
/* Accept Certificate or CertificateRequest */
transcript_update(ctx, msg_buf, pt_len);
```
`xsec_x509_parse` and `xsec_x509_verify_chain` exist in xsec.h but are
NOT called during handshake. This means TLS connections are currently
unauthenticated — the server identity is not verified. This is acceptable
for Sprint 3 (scaffolding) but MUST be closed in Sprint 4 before any
real network usage.

## Integration Point 3: XPKG ↔ XSEC (Ed25519 Package Verification)

### Assessment
pkg_verify.c correctly calls `xsec_ed25519_verify()`:
```c
xsec_status_t xst = xsec_ed25519_verify(
    key_store[k].pub_key, msg, msg_len, sig);
```
Where `msg = pkg->metadata.content_hash` (32 bytes) and
`sig = pkg->metadata.signature` (64 bytes).

Type alignment: `pub_key[32]` matches `XSEC_ED25519_PUB_LEN = 32`. PASS.
Signature `sig[64]` matches `XSEC_ED25519_SIG_LEN = 64`. PASS.

### Gap G-04: xpkg_verify_init not in xpkg.h (MEDIUM)
`xpkg_verify_init()` is declared as a package-internal function in
pkg_verify.c (not in xpkg.h). It must be called by `xpkg_init()` to
seed the SYSTEM key. Verify `xpkg.c::xpkg_init()` calls it.
If not, SYSTEM key is never loaded → all signatures fail silently.
ACTION: confirm call chain in xpkg.c.

### Gap G-05: GENOS_SYSTEM_PUB_KEY is placeholder (CRITICAL)
```c
static const uint8_t GENOS_SYSTEM_PUB_KEY[32] = {
    0x47, 0x45, 0x4E, 0x4F, 0x53, 0x5F, 0x53, 0x59, ...
};
```
This is ASCII "GENOS_SYSTEM_PUBKEY_PLACEHOLDER" — NOT a real Ed25519 key.
The `xsec_ed25519_keypair()` + `xsec_ed25519_sign()` flow must be used to
generate the actual GEN.OS signing keypair during ISO build, and the
public key must be baked into this constant before any real package signing.
This is a PRODUCTION BLOCKER.

### Gap G-06: No xsec_audit_log integration in XPKG gate (LOW)
XPKG has its own print-based logging. XSEC has a formal audit ring buffer
(`xsec_g_audit`). Signature verification events in pkg_verify.c should
emit `xsec_audit_log(XSEC_AUDIT_CERT_VERIFIED/CERT_REJECTED, ...)` entries
to the XSEC audit ring so they appear in unified security event queries.
Currently pkg_verify.c only calls `pal_console_printf`.

## Integration Point 4: XNET ↔ Kernel (XENOS)

### Assessment
XNET does NOT include `xenos.h` directly — this is CORRECT architecture.
Kernel interfaces are mediated through PAL. XNET's driver boundary is
the `xnet_if_tx_fn` callback and `xnet_netif_input()` function.

The contract:
- Driver registers via `xnet_netif_register(name, mac, mtu, tx_fn, &idx)`
- Driver calls `xnet_netif_input(if_idx, frame, frame_len)` on RX
- XNET calls `tx_fn(netif, frame, frame_len)` for TX

### Gap G-07: No i219-LM NIC driver (CRITICAL for Sprint 4)
The HP EliteBook x360 1030 G4 has an Intel I219-LM NIC. There is no
driver in the tree that calls `xnet_netif_register()`. The network stack
is complete but unconnected to hardware. Sprint 4 must implement:
- `drivers/net/i219/i219.c` — Intel I219-LM driver
- Calls `xnet_netif_register("eth0", mac, 1500, i219_tx, &idx)`
- Routes DMA RX descriptors to `xnet_netif_input()`
- Uses `pal_pages_alloc(PAL_MEM_DMA)` + `pal_sg_build()` for DMA rings

## Integration Point 5: Socket API Alignment with PAL-Aether Syscalls

### Assessment
`xnet_socket/bind/listen/accept/connect/send/recv/close` are C-level
functions. When PAL-Aether replaces PAL-Linux (post Phase 2), the XENOS
syscall table must route network syscalls to these functions.

Current XENOS syscall dispatch (`xenos_syscall_dispatch` in syscall.asm)
has no network syscall numbers defined. The `int` return types on socket.c
functions (returning XNET error codes as negative ints) align naturally
with the Linux syscall convention — this is intentional and correct.

### Gap G-08: No syscall number assignments for network operations (MEDIUM)
XENOS syscall numbering is not defined for XNET. Sprint 4 must define
in a shared header (suggest `kernel/xenos/include/syscall_table.h`):
```c
#define XENOS_SYS_SOCKET    41
#define XENOS_SYS_BIND      49
#define XENOS_SYS_LISTEN    50
#define XENOS_SYS_ACCEPT    43
#define XENOS_SYS_CONNECT   42
#define XENOS_SYS_SEND      44
#define XENOS_SYS_RECV      45
#define XENOS_SYS_CLOSE      3
```
And `xenos_syscall_dispatch()` must route these to the XNET functions.

## Integration Point 6: Data Flow — Packet RX to Application

The full path reads:
```
NIC DMA interrupt
  → i219_rx_interrupt() [Gap G-07: not yet written]
  → xnet_netif_input(if_idx, frame, frame_len)    [netif.c]
  → xnet_eth_input(if_idx, frame, frame_len)       [ethernet.c]
  → EtherType dispatch:
      0x0806 → xnet_arp_input()                   [arp.c]
      0x0800 → xnet_ip4_input()                   [ip4.c]
        → protocol dispatch:
            IPPROTO_ICMP → xnet_icmp_input()      [icmp.c]
            IPPROTO_TCP  → xnet_tcp_input()        [tcp.c]
            IPPROTO_UDP  → xnet_udp_input()        [udp.c]
              → registered callback or socket buf  [socket.c]
```

VERDICT: The chain is fully defined and internally consistent. All
`xnet_*_input` function signatures match their callers. The only
missing link is the hardware entry point (Gap G-07).

## Integration Point 8: Error Code Consistency

| Module | Base type  | OK value | Notes                              |
|--------|------------|----------|------------------------------------|
| PAL    | int32_t    | 0        | PAL_ERR_* negative, -1 to -11      |
| XSEC   | int32_t    | 0        | XSEC_ERR_* negative, -1 to -13     |
| XNET   | int32_t    | 0        | XNET_ERR_* negative, -1 to -20     |
| XPKG   | int32_t    | 0        | XPKG_ERR_* negative, -1 to -16     |
| XFRAME | int (PAL)  | 0        | Returns PAL_ERR_* codes             |
| XCOMP  | int (PAL)  | 0        | Returns PAL_ERR_* codes             |

All modules agree: `int32_t`, `0 = success`, `negative = error`. PASS.

ERROR CODE OVERLAP RISK: Several numeric values collide across domains
(e.g., XSEC_ERR_INVAL = -1, XNET_ERR_NOMEM = -1, XPKG_ERR_FORMAT = -1).
This is acceptable because error codes are ONLY meaningful within their
own domain (a caller checks the specific typed status). However, the
Socket API's `int` return type (not `xnet_status_t`) is an inconsistency:
socket functions return raw XNET error codes as plain `int`, which could
be confused with POSIX errno values at a future syscall boundary.

### Gap G-09 (LOW): Socket API return type inconsistency
`xnet_socket/bind/listen/accept` return `int` not `xnet_status_t`. This
is intentional POSIX compatibility but creates ambiguity at the syscall
boundary. Document explicitly that negative `int` returns from socket API
are XNET_ERR_* codes, not errno values.

## Integration Point 9: Memory Model Compatibility

### XSEC
- All buffers caller-provided. No internal heap allocation. PASS.
- xsec_tls_ctx_t is 17KB+ on stack (record_buf[16640]). Callers must
  ensure sufficient stack depth. This is a hidden constraint — not documented
  in xsec.h. Recommend adding a comment: "sizeof(xsec_tls_ctx_t) ~17KB — allocate on heap or static storage, NOT stack."

### XNET
- Socket table: `g_socket_table[256]` × sizeof(xnet_socket_t). Each socket
  has `udp_recv_buf[16384]`. Total: 256 × ~16500 ≈ 4MB BSS. On the
  HP EliteBook x360 with 8GB RAM this is acceptable but should be noted
  in platform memory budgeting.
- ARP cache: `ARP_CACHE_SIZE = 256`. Route table: `XNET_ROUTE_MAX = 32`.
  All static arrays. No heap. PASS.

### XPKG
- Installed package registry: 512 × sizeof(xpkg_package_t) — each package
  has `files[256]` × sizeof(xpkg_file_entry_t) = 256 × 294 ≈ 75KB.
  Total registry: 512 × ~78KB ≈ 39MB static BSS. This is LARGE.
  Recommend reducing XPKG_MAX_FILES_PER_PKG to 64 (15MB) unless 256 is
  genuinely required.
- xpkg_dep_graph_t: 512 nodes × ~130 bytes = 67KB. Acceptable.

### Zero-copy review
XNET uses `xnet_eth_build` to copy frames into caller buffers. No
scatter-gather in the user-facing API. DMA path (when i219 driver exists)
must use `pal_sg_build` to avoid copying at DMA level.

XSEC tls13.c uses stack-local scratch buffers (up to 16640 bytes). These
are zeroed via `xsec_secure_zero` before returning. Correct. PASS.
