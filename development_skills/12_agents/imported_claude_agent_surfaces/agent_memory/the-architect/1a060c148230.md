## ISSUE 1 — genos_auth.py: Silent empty-string key at import time

### Severity: P1

### Location
platform/services/genos_auth.py lines 72-74:
    raw_key: str = os.getenv(public_key_env, "")

Called at module level in:
- platform/services/sync/app.py line 41: decode_bearer = make_decode_bearer("SYNC_JWT_PUBLIC_KEY")
- platform/services/provenance/app.py line 186: decode_bearer = make_decode_bearer("PROVENANCE_JWT_PUBLIC_KEY")

### What happens when the env var is not set at import time

`os.getenv(public_key_env, "")` returns `""` silently. The factory closes over
`raw_key = ""`. The process starts and passes health checks. Every subsequent
call to `decode_bearer` reaches:

    claims = jwt.decode(token, _normalize_pem(raw_key), algorithms=["EdDSA"], ...)

PyJWT is then asked to verify an EdDSA token against an empty-string PEM.
`_normalize_pem("")` returns `"\n"` (the trailing-newline append). PyJWT raises
`jwt.exceptions.InvalidKeyError` or `jwt.exceptions.DecodeError` immediately.
The inner `except jwt.PyJWTError` catches it and returns HTTP 401 to the caller.

Effect: EVERY authenticated request to sync and provenance returns 401 forever,
silently, with no startup-time error. The pod appears healthy to k8s. The
application is completely non-functional in production.

### Contrast with identity/app.py

identity/app.py lines 216-253: `_load_signing_keys()` is called at module level
BEFORE `app = FastAPI(...)`. If `IDENTITY_JWT_PRIVATE_KEY_PEM` is absent AND
`IDENTITY_ALLOW_DEV_KEYS != true`, it raises `RuntimeError` immediately —
crashing the process at startup with a descriptive message. k8s sees the
CrashLoopBackOff and the operator knows immediately.

Similarly: lines 168-173: `_raw_dsn = os.getenv(...)` followed by
`if not _raw_dsn: raise RuntimeError(...)`. Fast, loud, descriptive failure.

genos_auth.py does NOT have this guard. The factory accepts `""` silently.

### Required fix

Add a startup-time guard inside `make_decode_bearer` that raises `RuntimeError`
if `raw_key` is empty after the `os.getenv` call:

    raw_key: str = os.getenv(public_key_env, "")
    if not raw_key.strip():
        raise RuntimeError(
            f"Required environment variable {public_key_env!r} is not set. "
            "Populate it via k8s Secret or systemd EnvironmentFile."
        )

This aligns with the identity service pattern and satisfies the architectural
rule: "crash loudly if a critical service is unreachable after retries."
A missing signing key is a configuration failure that must abort startup,
not silently degrade to "every token is invalid."

### Why this is P1, not P0

The _consequence_ is total auth failure (all 401) rather than a security bypass.
An attacker cannot gain access — all authenticated calls are rejected. However,
the production service is completely broken and the operator receives no signal.
This satisfies the P1 definition: "breaks production functionality, must fix
before next deploy."

## ISSUE 2 — SSE /chat/stream: Synchronous generator blocks the async event loop

### Severity: P1

### Location
ai/genesys-ai/src/api.py lines 428-484:
    def chat_stream(req: ChatRequest, _request: Request) -> StreamingResponse:
        ...
        def _generate() -> Iterator[str]:
            for chunk in agent.run_stream(req.user, req.message):
                ...
        return StreamingResponse(_generate(), media_type="text/event-stream")

model_manager.py lines 179-202:
    with _client.stream("POST", ...) as response:
        for line in response.iter_lines():
            ...

### The problem

`chat_stream` is declared `def`, not `async def`. In FastAPI, a plain `def`
endpoint runs in a thread from the default `ThreadPoolExecutor` (typically 40
workers). The `StreamingResponse` that is returned wraps `_generate()`, which
is a synchronous `Iterator[str]`.

When Starlette streams the response body, it calls `next()` on the iterator
repeatedly. If the iterator is a synchronous generator, Starlette's ASGI
machinery calls it in a `run_in_threadpool` wrapper — but only for the FIRST
`__next__` call. Subsequent iteration within the `StreamingResponse._iterate`
method re-enters the synchronous path from within the async context.

The deeper problem is `_client.stream(...)` + `response.iter_lines()`.
`_client` is a synchronous `httpx.Client` (model_manager.py line 17). Each
call to `iter_lines()` does a blocking socket read. During an Ollama inference
that produces 512 tokens with 8-35s total latency, the generator is blocked
on I/O for the full stream duration. This occupies one FastAPI threadpool slot
for the entire stream lifetime.

At 40 default threadpool workers and Ollama latency of 8-35s per token stream:
- 40 concurrent streams saturate the pool in under 5 seconds
- /health, /healthz, and /chat all run on the same threadpool
- k8s liveness probe (GET /health) will be queued behind 40 blocked threads
- After 3 missed probes the pod is restarted — destroying in-flight streams

This is the same Thundering Herd pattern already documented in the genesys-ai
codebase (api.py lines 315-320): the non-blocking semaphore acquire was added
specifically to prevent threadpool saturation from blocking /healthz. The SSE
endpoint holds the semaphore for the full stream duration AND occupies a thread
for the full stream duration. Only ONE stream can be active at a time (semaphore
limit=1), so the thread saturation risk is bounded, but the architectural
pattern is incorrect and fragile.

### Correct architecture

The entire genesys-ai service should migrate its Ollama calls to `httpx.AsyncClient`
and the streaming endpoint should use `AsyncGenerator[str, None]`. However, per
the project language policy and the existing architectural decision to keep all
FastAPI endpoints as sync `def` (to match the sync httpx client), the minimum
safe fix is:

Option A (current pattern, minimum change): Accept that the stream holds one
threadpool thread for its duration. The semaphore(1) cap means maximum one
thread is ever blocked on Ollama at a time. Add a comment documenting this
intentional design. Raise ThreadPoolExecutor size to guarantee /health
never queues.

Option B (correct pattern): Convert `_client` in model_manager to
`httpx.AsyncClient`, convert `generate_stream` to `AsyncGenerator`, convert
`chat_stream` to `async def`, use `asyncio.to_thread` for the sync `run_stream`
call if needed. This is the architecturally correct path.

### Current state assessment

The semaphore(1) + non-blocking acquire means at most one stream is ever
active. This limits the blast radius significantly. But the architectural
classification of "sync generator inside async FastAPI" is a P1 defect that
will become a P0 if the semaphore limit is ever raised or removed.

## ISSUE 3 — Graceful shutdown ordering: lifespan teardown vs in-flight requests

### Severity: Advisory (with nuance)

### Location
platform/services/identity/app.py lines 355-363
platform/services/sync/app.py lines 78-87
platform/services/provenance/app.py lines 172-179

    @asynccontextmanager
    async def lifespan(_app: FastAPI):
        init_db()
        yield
        _app.state.ready = False

### How Uvicorn/Starlette handles shutdown with lifespan

The sequence when Uvicorn receives SIGTERM:

1. Uvicorn calls `lifespan.__aexit__` — i.e., resumes the lifespan generator
   after the `yield`. This sets `_app.state.ready = False`.
2. Uvicorn then stops accepting NEW connections.
3. Uvicorn waits for in-flight requests to complete (the drain window, controlled
   by `--timeout-graceful-shutdown`, default 0 in older Uvicorn, 30s in recent).
4. After the drain window, Uvicorn exits.

Step 1 (lifespan teardown) and Step 3 (in-flight drain) are SEQUENTIAL, not
concurrent. Lifespan teardown fires FIRST, then draining happens. This means
`_app.state.ready = False` is set BEFORE any in-flight requests are processed
to completion.

### Why this is (mostly) correct

The stated intent is: set `ready = False` so that the NEXT k8s readiness probe
(every 5s) sees 503 and removes the pod from the LB, after which the pod drains
in-flight requests from already-connected clients.

This is a valid pattern. The concern is timing: between the moment `ready = False`
is set (in lifespan teardown) and the moment k8s removes the pod from the LB
(up to 5s later), new requests can still arrive from connections that were
established before the LB update propagated. Those requests are served normally
because `app.state.ready` only gates the /healthz endpoint, not the actual
business endpoints.

The pattern is architecturally sound for its stated purpose. The only gap is:

- `terminationGracePeriodSeconds` must be greater than the k8s probe interval
  (5s) plus the Uvicorn drain window. If `terminationGracePeriodSeconds` is too
  short, the pod is SIGKILL'd before in-flight requests drain. This is a
  k8s manifest configuration concern, not a code defect.

- Uvicorn's default `--timeout-graceful-shutdown` is 0 (no drain) in some
  versions. For this pattern to work correctly, the Uvicorn command in the
  Docker image must include `--timeout-graceful-shutdown 30` or similar.
  This should be verified at the Docker/k8s layer.

### Verdict

The ordering is correct. Lifespan teardown (setting ready=False) fires before
draining, which is the intended behavior. The architectural concern is that the
full pattern only works if Uvicorn is started with a non-zero graceful shutdown
timeout. This is an operational/deployment concern, not a code defect.
Severity: Advisory — document the Uvicorn startup flag requirement explicitly.

## ISSUE 4 — dbus_state_bridge.py: Daemon threads and SIGTERM data corruption

### Severity: Advisory

### Location
compositor/genos-shell/system/dbus_state_bridge.py lines 26-62:
    def _call_with_timeout(fn, timeout_s=5.0):
        t = threading.Thread(target=_target, daemon=True)
        t.start()
        t.join(timeout=timeout_s)
        ...

### The concern

When the Python interpreter receives SIGTERM (or exits normally), daemon threads
are killed immediately — they do NOT run their `finally` blocks. The question
is whether a daemon thread mid-write to D-Bus state can corrupt persistent data.

### Analysis

The daemon threads in `_call_with_timeout` do the following:
- `read_battery()._fetch()`: Reads D-Bus properties, returns a dict. No writes.
- `read_wifi()._fetch()`: Same — reads only.
- `read_volume()._fetch()`: Same — reads only.

None of these functions write to disk. They read from D-Bus and return a dict
to the caller via the `result` list. The ONLY disk write in the entire bridge
is `_atomic_write(STATE_FILE, ...)` in `emit_state_loop()`, which runs in the
main asyncio coroutine, NOT in a daemon thread.

The `_atomic_write` function uses `tempfile.mkstemp` + `os.fsync` + `os.replace`.
If SIGTERM fires while `os.replace` is in progress, the kernel guarantees
atomicity at the rename(2) syscall level — readers always see either the old or
the new complete file.

If SIGTERM fires while `os.fsync` is in progress in the main coroutine (not in
a daemon thread), the file descriptor is open and the temp file exists but
`os.replace` has not yet been called. The temp file will be left in `/run/genos/`
with a `.tmp` suffix. The next run will create a new temp file. The orphaned
`.tmp` file will persist until the next clean write succeeds. This is not data
corruption — the state file itself is untouched because `os.replace` was never
called.

### Verdict

There is NO risk of data corruption from daemon thread termination on SIGTERM.
The daemon threads are read-only D-Bus callers. The atomic write is in the main
coroutine and is protected by `rename(2)` semantics. The only residual is a
potential orphaned `.tmp` file in `/run/genos/` on ungraceful shutdown, which
is benign. The `_atomic_write` exception path calls `os.unlink(tmp_path)` but
this runs only if the write itself raises an exception — a SIGKILL mid-fsync
would bypass this cleanup. Acceptable on a single-device OS.
Severity: Advisory — current implementation is architecturally correct.

## ISSUE 5 — Per-IP rate limiting: _ip_attempts grows unbounded

### Severity: P2

### Location
platform/services/identity/app.py lines 74-112

    _ip_attempts: dict[str, dict[str, object]] = {}
    _ip_attempts_lock = threading.Lock()

### The problem

`_ip_attempts` is a plain dict. New entries are created in `_record_ip_failed_attempt`
(line 97-98) when an IP address is seen for the first time. Entries are NEVER
removed. The per-username `_login_attempts` dict has the same pattern (line 57),
but the username space is bounded by the user table. The IP space is unbounded.

Each `_ip_attempts[ip]` entry holds:
- A list of up to `_IP_LOCKOUT_WINDOW_SEC` (900) float timestamps — typically
  10 entries (the lockout threshold)
- A `locked_until` float

Memory per entry: approximately 200-400 bytes. At 1 million unique IPs (a
sustained credential-stuffing attack with IP rotation): 200-400 MB of RAM.
The HP EliteBook x360 target has limited RAM.

The per-entry `attempts` list IS pruned inside `_record_ip_failed_attempt` —
old timestamps outside the 15-minute window are removed (lines 100-102). But
the dict KEY (the IP address) is never removed. An attacker sending one failed
request per IP from 1M IPs leaves 1M dict entries, each with an empty `attempts`
list and `locked_until = 0.0`. Memory growth is unbounded.

### The correct fix

Use `cachetools.TTLCache` (maxsize=50_000, ttl=1800) as a drop-in replacement.
Entries are automatically evicted after 30 minutes of inactivity and the dict
cannot exceed 50,000 entries. This bounds memory to approximately 20-40 MB
worst case. Alternatively, implement a background sweeper that removes entries
where `locked_until < time.time() - _IP_LOCKOUT_WINDOW_SEC` and `len(attempts) == 0`.

### Current risk on GEN.OS

GEN.OS is a single-device OS. The identity service is not externally exposed in
the standard deployment (k3s single-node, no external LB). The attack surface
requires the attacker to be on the local network or have access to the k3s
cluster. The risk is real but not immediately exploitable at scale. P2 is
correct.
