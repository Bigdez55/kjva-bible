# Dependency Overrides — ajv/schema-utils Conflict (2026-02-24)

## Problem Statement
Build fails with `TypeError: Ajv is not a constructor` at `schema-utils@4.3.3/dist/validate.js:22`.

`schema-utils@4.3.3` calls `require("ajv").default` — this works only with ajv@8+ (which exports a default).
ajv@6 does NOT have a `.default` export. So if schema-utils@4.3.3 resolves to ajv@6, it crashes.

## Forensic Findings

### On-disk ajv locations (after broken state)
```
node_modules/ajv                              => v8.18.0  (ROOT — correct)
node_modules/react-scripts/node_modules/ajv  => v6.14.0  (PROBLEM — shadows root for react-scripts subtree)
node_modules/webpack/node_modules/ajv         => v6.14.0  (PROBLEM — shadows root for webpack subtree)
node_modules/schema-utils/node_modules/ajv   => v6.14.0  (OK — serves schema-utils@2.7.1 which needs v6)
node_modules/ajv-formats/node_modules/ajv    => v6.14.0  (OK — ajv-formats@2 supports both)
```

### schema-utils@4.3.3 locations and their resolved ajv
```
css-minimizer-webpack-plugin/node_modules/schema-utils  => resolves ROOT ajv@8.18.0  OK
eslint-webpack-plugin/node_modules/schema-utils         => resolves ROOT ajv@8.18.0  OK
mini-css-extract-plugin/node_modules/schema-utils       => resolves ROOT ajv@8.18.0  OK
terser-webpack-plugin/node_modules/schema-utils         => resolves ROOT ajv@8.18.0  OK
react-scripts/node_modules/schema-utils                 => resolves react-scripts/node_modules/ajv@6.14.0  BROKEN
webpack/node_modules/schema-utils                       => resolves webpack/node_modules/ajv@6.14.0  BROKEN
```

### Root cause
`package.json` overrides contained:
```json
"schema-utils": { "ajv": "^6.14.0" }
```
This was added during CVE #405 remediation (2026-02-24, commit e5a9373) to upgrade ajv@6.12.6 → 6.14.0.
But it applies to ALL schema-utils subtrees including schema-utils@4.3.3 — forcing it onto ajv@6 = crash.

### Legitimate ajv@6 consumers (must keep ajv@6)
- `@eslint/eslintrc` — declares `ajv: ^6.12.4`
- `eslint` — declares `ajv: ^6.12.4`
- `file-loader` → `schema-utils@3.3.0` — declares `ajv: ^6.12.5`
- `fork-ts-checker-webpack-plugin` → `schema-utils@2.7.0` — declares `ajv: ^6.12.2`
- `schema-utils@2.7.1` (root) — declares `ajv: ^6.12.4`

## The Fix

### Remove from overrides
```json
"schema-utils": { "ajv": "^6.14.0" }
```

### Add to overrides
```json
"webpack": { "ajv": "^8.18.0" },
"react-scripts": { "ajv": "^8.18.0" }
```

### Keep in overrides (CVE #405 still covered)
```json
"@eslint/eslintrc": { "ajv": "^6.14.0" },
"eslint": { "ajv": "^6.14.0" },
"file-loader": { "ajv": "^6.14.0" },
"fork-ts-checker-webpack-plugin": { "ajv": "^6.14.0" }
```

## Why Not a Clean Reinstall?
A clean reinstall (`rm -rf node_modules package-lock.json && npm install`) would:
1. Take 5+ minutes in CI
2. Potentially resolve new package versions (introduces version drift)
3. Not fix the underlying override misconfiguration — it would return on next `npm install`

The override fix is O(1) to apply and is durable across all future installs.

## Why Not Eject?
Ejecting CRA is irreversible and creates 1000+ lines of webpack config to maintain.
The npm overrides mechanism is the correct, non-destructive tool for this class of problem.
