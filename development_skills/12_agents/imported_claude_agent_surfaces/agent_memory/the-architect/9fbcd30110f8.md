## What Was Done

`Citadel/council/runtime/memory/soul_manager.py` was already correctly hardened
with the fail-closed AES-GCM model before this session. The one genuine gap was
requirement #7: `GENOS_STRICT_CRYPTO=1` enforcement on `_MASTER_SECRET`.

Added at lines 241-281 (replacing the single `_MASTER_SECRET` assignment):

### Production Secret Injection
- New env var: `GENOS_SOUL_MASTER_SECRET_HEX` — 64 hex chars (32 bytes)
- Decoded and validated at module load; stored in `_MASTER_SECRET`
- Bad hex or wrong length → raises `SoulManagerCryptoError` at import time
- `_DEV_SEED` preserved as a named constant for test introspection

### `GENOS_STRICT_CRYPTO=1` Guard
- If `GENOS_SOUL_MASTER_SECRET_HEX` is NOT set AND `GENOS_STRICT_CRYPTO=1` →
  raises `SoulManagerCryptoError` at module load
- If `GENOS_SOUL_MASTER_SECRET_HEX` is NOT set AND strict mode is OFF →
  uses `_DEV_SEED` with a WARNING log (dev/CI-only path)

## Architecture of the Full Security Model

```
Module load time:
  GENOS_SOUL_MASTER_SECRET_HEX set?
    Yes → decode hex → validate 32 bytes → _MASTER_SECRET = injected key
    No  → _MASTER_SECRET = _DEV_SEED
          GENOS_STRICT_CRYPTO=1? → raise SoulManagerCryptoError (production gate)
          else → WARNING log (dev only)

Encrypt call:
  aes_gcm_bridge available?
    Yes → AES-256-GCM (Tier 1: libxsec.so, Tier 2: cryptography pkg)
    No  → GENOS_SOUL_ALLOW_PLAINTEXT=1?
            Yes → SHA-256 XOR fallback + ERROR log (dev only)
            No  → raise SoulManagerCryptoError (fail-closed)

Decrypt call:
  Same tier logic. On bridge unavailability without env escape → return None.
```
