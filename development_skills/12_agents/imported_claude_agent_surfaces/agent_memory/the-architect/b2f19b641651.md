## SKILL 01 — Freestanding C as the Sole Kernel Language

**Pattern Type:** language-policy, systems-boundary

**The Decision:** All code at kernel privilege (ring 0), bootloader, init system, network stack, storage engine, crypto library, and UI toolkit is C11 compiled with `-ffreestanding -mno-red-zone -fno-stack-protector`. No libc, no stdlib, no POSIX headers. `pal.h` is the only permitted interface boundary for system services. See ADR-S14-01.

**Alternatives Considered:** Rust (no_std) — excluded by language policy before technical evaluation. C++ freestanding — vtables, constructors, RTTI, name mangling complicate NASM linkage. Pure assembly — unmaintainable at scale.

**Trade-offs Gained:** Universal toolchain (clang/gcc), full ABI control via `__attribute__((packed))`, NASM linkage via C extern declarations, inline assembly for LAPIC/RDMSR/CPUID, constant-time crypto auditable at C level.

**Trade-offs Sacrificed:** No memory safety guarantees, manual resource management, undefined behavior risk, no package ecosystem — every algorithm (SHA-256, B+ tree, TLS 1.3, SWIM gossip) written from scratch.

**Applicability:** Any project requiring bare-metal execution with no prior OS. Applies when the host OS is itself being built. Use freestanding constraint to enforce zero-dependency discipline from day one.

## SKILL 02 — Platform Abstraction Layer (PAL) as the Single Hardware Interface

**Pattern Type:** architecture, abstraction-boundary, adapter

**The Decision:** A single header `pal/include/pal.h` defines EVERY service that kernel-level code may use. Two backends: PAL-Linux (wraps Linux syscalls for development/CI) and PAL-Aether (backed by XENOS kernel for production). All GEN.OS libraries compile against PAL types only — zero Linux headers permitted in production code. See `pal/include/pal.h`.

**Key PAL Invariants (from the header itself):**
- All functions return `pal_status_t` (0 = success, negative = error)
- All handles are opaque 64-bit tokens — no raw pointers across boundaries
- No heap allocation inside PAL — callers provide buffers
- If a behavior is not specified in `pal.h`, backends MUST NOT rely on it

**Alternatives Considered:** Direct Linux syscalls in kernel code (Sky OS mistake — built on undocumented behavioral dependencies). Per-module HAL layers (creates multiple abstraction boundaries, no single audit point).

**Trade-offs Gained:** Backend swap is a single link-time decision. CI validates logic against PAL-Linux without hardware. PAL-Aether replaces PAL-Linux and Linux disappears from the runtime. Every access is auditable.

**Trade-offs Sacrificed:** One more indirection layer. PAL must be extended to expose every new hardware primitive before callers can use it.

**Applicability:** Any multi-target codebase (dev vs production, x86 vs ARM, emulated vs bare-metal). Define the abstraction boundary first, implement both backends, enforce via CI banned-include scan.

## SKILL 03 — Opaque Handle System (Capability Tokens)

**Pattern Type:** security, API-design, capability-based-security

**The Decision:** Every PAL resource is referenced by a 64-bit opaque handle (`pal_handle_t`). Handles are process-local, non-forgeable, typed, and validated by the kernel before use. The XKABI capability system extends this: EFI handles are enumerated by AetherBoot and converted to typed capability entries with bitmask rights (bits 0-11 software rights, bits 12-22 hardware rights, bit 23 DERIVE right). See `pal.h §3` and ADR-S3-10.

**Sky OS Mistake Documented Explicitly:** "The Sky OS mistake: using raw pointers across abstraction boundaries. We use handles so backends can validate every access."

**Alternatives Considered:** Raw pointers (Sky OS approach — causes corruption on invalid dereference, no validation, no type safety). POSIX file descriptors (small integers in per-process table — linear search, not typed).

**Trade-offs Gained:** Backends validate handles before every operation. Double-close returns `PAL_OK` (idempotent, no UAF). A process cannot forge a handle for a resource it was not given. Handle type queryable at runtime.

**Trade-offs Sacrificed:** Cannot pass handles across serialization boundaries without a capability transfer protocol. Each handle query has validation overhead.

**Applicability:** Any API that manages resources across trust boundaries. Replace raw pointer returns with handle returns in any C API surface that will be accessed from untrusted code.

## SKILL 04 — Two-PCH Architecture (NOSSE Kernel vs SSE Userspace)

**Pattern Type:** build-system, compile-flags, performance

**The Decision:** Kernel code compiles with `-mno-sse -mno-mmx -mno-avx` (NOSSE). Userspace and application code compiles with SSE/AVX enabled. The boundary is enforced by two separate pre-compiled headers. This prevents the kernel from accidentally using floating-point registers that the kernel's interrupt handlers do not save/restore. The kernel has no floating-point register save in interrupt frames (isr_frame_t only saves GPRs).

**Alternatives Considered:** Allow SSE in kernel (requires saving/restoring YMM registers in every interrupt — 512 bytes of XSAVE area per interrupt — extreme overhead). Require SSE for kernel matmul paths (XMIND cannot run in kernel context anyway).

**Trade-offs Gained:** Interrupt handler stack overhead remains bounded. Kernel code is simpler, faster interrupt paths.

**Trade-offs Sacrificed:** Kernel matmul and AI inference cannot run in ring-0. Must run in userspace (correct design anyway).

**Applicability:** Any OS kernel with interrupt handling. The SSE boundary maps directly to "can this code be interrupted and resumed without flushing FP state." Use compiler flags to enforce the boundary at compile time.

## SKILL 05 — Boot Info Handoff Struct (AetherBoot → XENOS)

**Pattern Type:** system-boundary, protocol, API-design

**The Decision:** AetherBoot places a pointer to `xenos_boot_info_t` in RDI before jumping to `_xenos_start`. The struct is a versioned, append-only record containing: memory map pointer, framebuffer base, CPU count, TSC frequency, ACPI RSDP, KASLR slide, capability table physical address, WiFi firmware blob pointer, deferred serial buffer. Magic `0x4145544845523031` ("AETHER01") verified on receipt. See `kernel/xenos/include/xenos.h §2`.

**Key Design Principles:**
- Struct is append-only — new fields added at the end with explicit `_reserved` and `_pad` fields
- Zero value in any optional field means "not provided, use fallback"
- All pointers are physical addresses (kernel virtual memory is not set up at entry)
- Magic number allows version detection without a version field

**Alternatives Considered:** Register passing (limited to 6 registers, insufficient for the data volume). Stack pointer to struct (requires caller to manage stack layout). Flat memory address convention (fragile, no type checking).

**Trade-offs Gained:** ABI is formally documented and CI-verified via `_Static_assert` on struct size. New boot parameters are additive with zero risk of breaking existing code.

**Trade-offs Sacrificed:** Struct must never reorder fields or change existing field types. Binary compatibility must be maintained across AetherBoot and XENOS upgrades.

**Applicability:** Any bootloader-to-kernel or stage-to-stage boot handoff. Define the handoff struct with a magic number, make it append-only, use `_Static_assert` to verify sizes at compile time, and zero-value every optional field.

## SKILL 06 — HHDM (Higher-Half Direct Map) for Physical Memory Access

**Pattern Type:** memory-management, address-space-layout

**The Decision:** All physical RAM is mapped into the kernel's virtual address space at `HHDM_BASE = 0xFFFF888000000000` (PML4 index 273). `PHYS_TO_HHDM(pa)` is a single add: `pa + HHDM_BASE`. Kernel image at `-2 GB` (`KERN_BASE = 0xFFFFFFFF80000000`, PML4 index 511). See ADR-S14-02.

**Why PML4[273] specifically:** PML4[272] (`0xFFFF880000000000`) conflicts with AMD SME encryption bit at bit 47. Linux switched from 272 to 273 for this reason. GEN.OS follows the same reasoning for future AMD hardware portability.

**Alternatives Considered:** PML4[256] (0xFFFF800000000000) — AMD SME c_bit conflict. No HHDM / dynamic MMIO mapping only — chicken-and-egg: PMM needs physical access before VMM is ready.

**Trade-offs Gained:** `PHYS_TO_HHDM` is a zero-overhead add. PMM can access any physical page immediately after `vmm_init()`. MMIO access at known physical addresses is immediate.

**Trade-offs Sacrificed:** `HHDM_BASE` appears in 5+ headers — consistency must be verified by CI. Identity map at PML4[0] must be explicitly removed to prevent userspace physical memory access.

**Applicability:** Required for any bare-metal kernel that needs to access arbitrary physical addresses during initialization. Pre-map all physical memory once; never need dynamic mapping for kernel data access.

## SKILL 07 — Sprint-Gated CI (One Workflow Per Sprint)

**Pattern Type:** CI/CD, quality-gate, progressive-integration

**The Decision:** Each sprint gets its own `.github/workflows/sprintN.yml` workflow with jobs that validate exactly what that sprint introduced. Sprint workflows are never deleted — they accumulate. The final gate job in each workflow prints a named sentinel (e.g., `GEN.OS BOOT-READY`). The gate job depends on all other jobs completing successfully.

**Accumulation Effect:** By Sprint 42, the CI tree contains 42+ workflow files. Each one remains as a regression guard for its sprint's deliverables. A regression that breaks Sprint 19's KASLR slide will fail the `sprint19.yml` workflow.

**Alternatives Considered:** Single monolithic CI workflow (runs everything — slow, hard to identify which sprint introduced a regression). Per-subsystem workflows (cross-sprint regressions harder to detect).

**Trade-offs Gained:** Regression isolation — the failing workflow identifies the sprint that introduced the breakage. CI history tells the complete development story. Each sprint's contract is documented in YAML form.

**Trade-offs Sacrificed:** CI maintenance overhead grows linearly with sprint count. Old workflows may reference deprecated toolchain versions.

**Applicability:** Long-running platform projects where sprint-to-sprint contracts must be preserved. Use named sentinel strings in gate job output for observability dashboards.

## SKILL 08 — Architecture Decision Record (ADR) Discipline

**Pattern Type:** documentation, governance, institutional-knowledge

**The Decision:** Every significant architectural decision gets an ADR with: date, status, context (forces in play), decision, consequences (positive/negative/neutral), alternatives considered, and related decisions. ADRs are numbered by sprint (`ADR-S14-01`, `ADR-S3-05`). 52+ ADRs filed. ADRs are never deleted — only superseded. See `docs/adr/`.

**ADR Structure Standard:**
- Context: The forces that created the decision pressure
- Decision: The exact choice made, with parameters
- Consequences: Positive, Negative, Neutral — honest accounting
- Alternatives: What was rejected and why (often more valuable than the decision itself)
- Errata: Open issues and known defects at the time of filing

**Alternatives Considered:** Inline code comments (lost when code is deleted). Wiki pages (go stale, no version tie). No documentation (institutional knowledge lost when team members leave).

**Trade-offs Gained:** Future engineers understand why decisions were made, not just what they are. Architectural drift is detectable by comparing current code to ADR decisions. "Negative" and "errata" sections track known technical debt explicitly.

**Applicability:** Any project with more than 2 engineers or a lifespan beyond 6 months. File ADRs before writing code — the act of writing the ADR often surfaces better alternatives.

## SKILL 09 — Single Source of Truth for Constants (Anti-Drift Pattern)

**Pattern Type:** architecture, DRY, constants-management

**The Decision:** `kernel/xenos/include/xkabi_rights.h` is the ONLY place where XKABI right bitmasks are defined. A comment in the header explicitly states: "ADDING A RIGHT: add here ONLY. Propagate to GSD parser string table. Do NOT redefine in caller files." AetherBoot must include this header via `-Ikernel/xenos/include` build flag. See ADR-S3-10.

**The Bug This Prevented:** AetherBoot `capability_init.c` had 8 local `#define` macros with sequential values (0x01, 0x02, 0x04...) that were wrong. Canonical `XKABI_RIGHT_GPU_EXEC = 1u << 19 = 0x80000` was defined locally as `0x01`. Every capability table entry produced by AetherBoot was silently incorrect. Silent boot-time security misconfiguration, no compile error.

**Alternatives Considered:** Copy canonical values into bootloader header (dual maintenance, same failure mode). Generate header from JSON/YAML (build complexity disproportionate to problem size for ~15 constants).

**Trade-offs Gained:** Any future constant addition in `xkabi_rights.h` is automatically available to AetherBoot. The drift bug class is eliminated. A removed include flag produces a hard compile error.

**Applicability:** Any multi-component system where the same semantic constant (bitmask, enum, magic number) must be used in two+ separately compiled units. Single header with enforced include via build flags. Never copy-paste constants.

## SKILL 10 — Content-Addressable Storage (XBLOB SHA-256 Indexing)

**Pattern Type:** storage, data-integrity, deduplication

**The Decision:** XBLOB keys every blob by `SHA-256(content)`. `xblob_put()` computes the hash, stores data at that key, returns the key — idempotent for identical content. `xblob_get()` can re-verify integrity by recomputing hash. 1M-slot open-addressing hash table, load factor < 0.75. Two-phase mark-sweep GC. See ADR-S14-09.

**Uses:** Package artifacts (XPKG), AI model weights (XMIND), offline sync snapshots (XFRAME). The deduplication benefit is significant for shared library binaries across 100 packages.

**Key Properties:**
- Immutable blobs: SHA-256 key always refers to same bytes (no update-in-place races)
- Deduplication automatic: second `put` of same content is a no-op
- Integrity verification at no extra cost: re-hash on get catches bit rot
- GC correctness proof: a blob is live iff some reference to its key exists

**Alternatives Considered:** UUID-keyed (no deduplication, no integrity guarantee). Path-keyed (two paths with same content = two storage allocations). IPFS CID (adds 2-4 bytes overhead, self-describing but unnecessary complexity).

**Trade-offs Sacrificed:** SHA-256 of 1.5 GB XMIND weights takes ~1 second (one-time cost at put). Storage file compaction deferred (tombstoned entries remain until GC compaction sprint).

**Applicability:** Any storage system where the same content may be stored from multiple sources. Ideal for package managers, model registries, document sync, backup systems. The key derives from content — deduplication is a free structural property.

## SKILL 11 — FORCE + NO-STEAL WAL Policy (ARIES-Simplified)

**Pattern Type:** storage, crash-safety, database-internals

**The Decision:** XSTORE uses B+ tree (4096-byte pages, matches NVMe atomic write unit) with WAL FORCE + NO-STEAL policy. FORCE: all dirty pages must be written before `txn_commit()` returns. NO-STEAL: dirty pages from uncommitted transactions never written to disk. Result: REDO-only recovery. MVCC snapshot isolation: writers don't block readers. See ADR-S14-04.

**Why FORCE + NO-STEAL:** Recovery is a forward-only WAL replay. No UNDO log needed (NO-STEAL guarantees uncommitted data never reached disk). Recovery time proportional to WAL size since last checkpoint, not database size.

**The Documented Bug:** P1 errata in the ADR: `txn_commit()` wrote the COMMIT WAL record but did NOT call `pal_fsync()` before returning `XSTORE_OK`. The documented contract was violated by the implementation. The ADR explicitly records both the contract and the known gap.

**Alternatives Considered:** LSM-Tree (write-optimized but read amplification hurts XPKG range queries). WAL NO-FORCE + STEAL / ARIES full (requires UNDO log, 3x implementation surface, not justified for XSTORE workload). COW B-Tree (no WAL but free page list consistency harder on crash).

**Applicability:** Embedded databases, configuration stores, package databases. If the workload is read-heavy with infrequent writes, FORCE + NO-STEAL is correct, simple, and fast to recover.

## SKILL 12 — Minimal Syscall Surface (XKABI 60-Syscall Non-POSIX Interface)

**Pattern Type:** security, API-design, kernel-interface

**The Decision:** XKABI syscall table has exactly 60 declared syscalls in 7 groups (Physical Memory, Virtual Memory, Process/Thread, IPC/Capabilities, Timer/Clock, I/O, Security/Misc). Capability-based: every syscall that operates on a resource takes a PAL handle as its first argument, kernel verifies capability bits before proceeding. Non-POSIX by design. Linux compatibility via XEMU translation layer. See ADR-S14-14.

**Why Non-POSIX:** POSIX assumes file descriptor semantics (small integers), not capability handles. POSIX has 100+ syscalls irrelevant to GEN.OS (TTY control, SysV semaphores, POSIX shared memory). Each syscall is a kernel entry point = attack surface. XEMU handles Linux compat at lower kernel complexity cost.

**Comparison:** seL4 = 12 syscalls (too minimal, no I/O), Linux = ~300 syscalls, Windows NT = ~470 syscalls, XKABI = 60 syscalls.

**Alternatives Considered:** POSIX-compatible ~300 syscalls (300 kernel entry points = 300 potential vulnerability classes). seL4-style 12 syscalls (too minimal for current development stage — ~15-20 minimum for boot readiness).

**Applicability:** Any new kernel or hypervisor. Define the minimal set of operations needed for your use case. Every additional syscall is additional attack surface. Use a compatibility translation layer rather than POSIX syscalls in the kernel itself.

## SKILL 13 — KASLR via Hardware RNG with Graceful Fallback

**Pattern Type:** security, boot-sequence, entropy

**The Decision:** AetherBoot generates kernel ASLR slide via RDRAND instruction: attempt up to 10 retries (check CF flag per Intel SDM), fallback to slide=0 if all fail, clamp to [0, 64MB), align down to nearest 2MB boundary. Pass slide in `xenos_boot_info_t.kaslr_slide`. Kernel applies slide BEFORE loading ELF (Sprint 21 bug fix — original code applied after load, making slide always 0). See ADR-S19-01.

**5-bit entropy:** 32 possible slide values (0, 2MB, 4MB, ..., 62MB). Sufficient to defeat static-offset attacks. Expansion to 18-bit entropy (4KB alignment, [0, 1GB)) planned.

**The Sprint 21 Bug:** Slide was applied AFTER `load_kernel_elf()`, making it always 0. Fix: apply slide BEFORE loading. Documents how easy it is to invert the order and silently break security.

**Alternatives Considered:** RDTSC-based (monotonic, not random — predictable across boots). EFI GetTime (seconds resolution — few bits of entropy). Fixed compile-time slide (defeats randomization entirely).

**Applicability:** Any bootloader implementing KASLR. Always apply the slide before loading the binary, not after. RDRAND retry loop with CF-flag check is the correct hardware RNG interface. Always implement graceful fallback to slide=0 for hardware without RNG support.

## SKILL 14 — ET_EXEC Only, PIE Deferred (Attack Surface Reduction)

**Pattern Type:** security, progressive-complexity, scoped-implementation

**The Decision:** `XK_SYS_EXEC` (syscall 29) accepts only ET_EXEC (e_type == 2) ELF binaries. ET_DYN (PIE) binaries rejected with `XENOS_EINVAL`. Physical KASLR (from AetherBoot slide) provides physical-layer address randomization without requiring PIE. No dynamic linker exists — XPKG delivers statically linked binaries. See ADR-S23-01.

**Security Analysis:** Rejecting ET_DYN eliminates the entire class of relocation-based ELF exploits. A malformed ET_DYN binary with crafted relocation entries could exploit bugs in a relocation engine. No relocation engine = no relocation exploits.

**Alternatives Considered:** Support ET_DYN immediately (requires ASLR base selection, GOT/PLT relocation processing, auxv setup, dynamic linker — ~500-800 LOC of additional trusted kernel code).

**Applicability:** When building a new kernel or runtime, start with static binaries. The relocation engine is complex trusted code that must be fuzzed thoroughly. Physical/loader KASLR can provide address randomization without PIE complexity.

## SKILL 15 — Demand-Zero BSS (Virtual Reservation, Physical Allocation on Access)

**Pattern Type:** memory-management, performance

**The Decision:** `vmm_bss_demand_zero_init()` splits 2MB kernel pages covering BSS into 4KB pages, clears PRESENT bit on pages beyond 64KB. On first access, page fault fires, `vmm_bss_pf_handle(cr2)` allocates physical page, zeros it, sets PRESENT, returns to retry faulting instruction. Must be called AFTER IDT is functional (so fault handler can run). See `kernel/xenos/include/xenos.h §function declarations`.

**Why:** Large BSS sections (XMIND vocabulary tables, XSTORE B+ tree node pools) waste physical RAM if pre-allocated at boot. Demand-zero means only touched pages consume physical RAM.

**Applicability:** Any kernel or runtime with large uninitialised data regions. BSS demand-zero is a standard technique — implement after the IDT and VMM are both functional, never before.

## SKILL 16 — Weak Symbol Progressive Integration

**Pattern Type:** build-system, progressive-integration, stub-override

**The Decision:** Settings backends in `xshell` declare driver APIs with `__attribute__((weak))`. If the driver is absent at link time, the symbol resolves to NULL; the setter persists the value and returns 0 (graceful degradation). When the driver ships, no settings code changes — the symbol resolves automatically. See ADR-SETTINGS-ARCH-01.

**Pattern Used Throughout:** `init/gensd/boot_stubs.c` provides weak stubs for all subsystem init functions. Real implementations override weak stubs as subsystems are built. The system links and boots at every sprint even when subsystems are incomplete.

**Alternatives Considered:** Conditional compilation with `#ifdef` (causes compilation errors when symbol is absent, requires explicit guard management). Null pointer checks with explicit `dlopen` (runtime complexity, no compile-time safety).

**Applicability:** Any platform with a large number of optional or incrementally developed drivers/subsystems. Define the interface, provide a weak stub that degrades gracefully, override with real implementation when ready. The build never breaks.

## SKILL 17 — 9-Gate Refusal Machine (Named Sequential Pipeline)

**Pattern Type:** security, pipeline, error-reporting

**The Decision:** XPKG installation runs 9 named sequential gates (Signature, Integrity, Capability, Provenance, Version, Conflict, Size, Dependency, Policy) via `xpkg_gate_check()`. All 9 gates run regardless of earlier failures — caller gets a complete failure report, not just the first error. Per-gate `enabled` flag and `force` flag (demote FAIL to WARN). Gates 0 (Signature) and 1 (Integrity) cannot be forced in production. See ADR-S3-05.

**Key Design:** Separation of concerns — each gate tests exactly one property. Monolithic validators are opaque. Caller-side composition creates bypass paths. Centralized pipeline with no short-circuit provides defense in depth.

**Alternatives Considered:** Monolithic validation function (opaque failure, first-error-only). Caller-side check composition (every call site that forgets a check creates a bypass). Two gates only (signature + integrity) — history shows the other 7 each catch real production bugs.

**Trade-offs Gained:** Structured `xpkg_gate_report_t` with 9 named entries for audit. Full failure picture at once. Each gate unit-testable in isolation. Policy escape hatch (Gate 8) without modifying core logic.

**Applicability:** Any security-critical installation, deployment, or authorization pathway. Name the checks, run all of them, collect structured results. Never short-circuit on first failure in a security pipeline — incomplete information is operationally hostile.

## SKILL 18 — Cipher Suite Hard Rejection (Not Just Omission)

**Pattern Type:** security, API-design, policy-enforcement

**The Decision:** AES-256-GCM is not merely unsupported — it is actively rejected by XSEC TLS 1.3. The TLS handshake will not negotiate AES-GCM cipher suites. Connections from TLS 1.3 peers that only offer AES-GCM get a `handshake_failure` alert. This rejection is conditional — it applies until AES-NI intrinsics are implemented. The distinction between "omission" and "rejection" is documented explicitly in an ADR. See ADR-S14-07.

**Root Cause:** Software AES-GCM without AES-NI is 200 KB/s (constant-time bit-sliced) or exposes a timing oracle (table lookup). ChaCha20-Poly1305 is constant-time by construction (add/rotate/XOR only) at 800 MB/s.

**Pattern:** Document the reason for rejection, not just the fact. Distinguish policy enforcement from missing implementation. Set a clear condition under which the rejection is lifted (AES-NI implementation). This prevents future engineers from re-enabling without understanding the timing vulnerability.

**Applicability:** Any security API that must exclude unsafe options while preserving the ability to add them safely later. An explicit rejection with documented rationale is safer than a silent omission.

## SKILL 19 — Council of Advisors (Named Daemon Architecture)

**Pattern Type:** architecture, AI-governance, microservices

**The Decision:** 14 AI advisor daemons communicate over TCP loopback with a uniform framing protocol: `[4-byte big-endian uint32 payload length][UTF-8 JSON payload]`. Fixed port assignments (18600-18700). 7-gate sequential software creation pipeline: required gates (Ezri/Legal, Magen/Security, Sarah/Architecture, Ahki/Strategic) halt the chain on REJECT; advisory gates (Abigail, Cherev, Esther) append WARN without halting. See ADR-S29-02.

**Why Fixed Ports over Service Registry:** Any daemon can restart independently without re-registration. Port assignments memorizable by any engineer. Dynamic registry adds a bootstrap dependency (who starts the registry?).

**Why asyncio over threads:** Each daemon handles multiple concurrent connections. Single event loop per process, no data races from shared mutable state. CPU-bound subagents use `asyncio.to_thread()`.

**Why JSON over gRPC/protobuf:** Council is Python-only for foreseeable future. JSON debuggability outweighs wire efficiency at intra-loopback throughput.

**Applicability:** AI governance layers, multi-agent decision systems. Name each agent, assign fixed ports, define required vs advisory gates explicitly in the pipeline definition. Separation of concerns: Ruth (resource economics) excluded from software creation chain — different concern.

## SKILL 20 — Algorithm Validation Vehicle (Linux Scaffold Pattern)

**Pattern Type:** progressive-implementation, validation, governance

**The Decision:** `kernel/linux-validation/` contains prototype implementations of advanced algorithms (vDRAM tier migration, scheduler, capabilities, mesh networking) validated against a live Linux kernel. Never compiled into GEN.OS binaries. Exempt from linux-header-scan CI gate. Removal gated on native port audit completion. Time-bounded: ADR sets hard removal timeline at Phase 45+. See ADR-S44-01.

**Why Not Delete Immediately:** Historical reference documents critical design decisions (ring buffer layouts, scheduler rotation period, IOMMU mapping strategies). Premature deletion risks losing institutional knowledge.

**Why This Is Safe:** GEN.OS build system has zero knowledge of `linux-validation/` — cannot be accidentally included. Kbuild system requires Linux headers to compile (automatic unbuildability if Linux headers absent). `kernel/README.md` explicitly marks as validation prototypes.

**Applicability:** Any platform project where advanced algorithms must be proven correct before porting to a novel runtime. Build the algorithm with familiar tools first, validate against known-good behavior, then port. Formally govern the scaffold as temporary with a documented removal timeline.

## SKILL 21 — Transitional Build Architecture (Scaffolding vs Production)

**Pattern Type:** build-system, progressive-migration, governance

**The Decision:** GEN.OS ISO build uses Debian `debootstrap` as rootfs scaffold (TRANSITIONAL), Linux kernel in the bootable image (TRANSITIONAL — will be replaced by XENOS), systemd as PID 1 (TRANSITIONAL — will be replaced by GENSD), Ollama for AI (TRANSITIONAL — replaced by XMIND). Each transitional component has: a named canonical replacement, a migration status, and a guard preventing production inclusion. See ADR-S46-01.

**Guard Mechanisms:**
- `GENOS_PRODUCTION` compile-time guard: `#ifdef GENOS_PRODUCTION #error` in validation vehicles
- `STORAGE_BACKEND` env var defaults to `xstore` (changed Sprint 46 from postgres)
- `OLLAMA_URL` references replaced with `XMIND_URL`
- `docs/LICENSE-THIRD-PARTY.md` tracks all transitional component licenses

**Alternatives Considered:** Wait for all original components before shipping anything (never ships). Ship transitional components without documentation (permanent technical debt, license compliance risk).

**Applicability:** Any greenfield OS or platform project. The transitional layer keeps you shipping while the original stack is built. Document every transitional component with its canonical replacement from day one. Use environment variables to switch backends without code changes.

## SKILL 22 — Electron as Process Container (Not UI Framework)

**Pattern Type:** architecture, boundary-enforcement, security

**The Decision:** GEN Shell uses Electron solely for its process model (lifecycle, sandboxed renderer isolation, IPC bridge, `BrowserWindow` as frameless wrapper). All visible UI is original, authored in the renderer using XFRAME. Electron UI primitives (native dialogs, native menus, tray icons) are forbidden in `src/renderer/`. `contextIsolation: true`, `sandbox: true` are non-negotiable. See ADR-S26-01.

**The Invariant:** Renderer must not `import from 'electron'` or `import from 'node:*'`. All platform access flows through `window.genOS` surface exposed by `contextBridge`. The renderer is architecturally equivalent to a sandboxed web app.

**ESLint Enforcement Rule:** `no-restricted-imports` scoped to `src/renderer/**` bans electron, node:*, fs, path, child_process. Makes the architectural boundary machine-enforceable.

**Why Electron over Native C/GTK Shell:** TypeScript React hot-reload allows rapid UI iteration. XFRAME layout is TypeScript/React — cannot be consumed by C/GTK. Jest/Vitest + Playwright test coverage achievable. Renderer crash recovery automatic via `render-process-gone` handler.

**Applicability:** Any Electron application with strong security requirements. Define the process-layer/renderer boundary explicitly, enforce it with ESLint, document it in an ADR. The renderer should be as thin as possible.

## SKILL 23 — Four-Layer Settings Backend Delegation Model

**Pattern Type:** architecture, delegation, progressive-wiring

**The Decision:** Every settings backend in `xshell/src/settings_<domain>.c` routes to one of four delegation targets: Layer A (PAL-backed hardware), Layer B (driver-backed via weak extern), Layer C (shell/compositor-backed via weak extern), Layer D (platform service-backed via XSTORE key write + `gensd_notify_setting_changed()` weak extern). No settings backend may allocate heap memory — all XSTORE keys are compile-time string literals. See ADR-SETTINGS-ARCH-01.

**Graceful Degradation Mandate:**
1. If `g_settings_store` is NULL: return sensible default, never fail
2. If weak extern resolves to NULL (driver absent): persist value, return 0
3. If hardware read fails: fall back to last persisted XSTORE value
4. Return -1 ONLY for: null output pointer, invalid input range, XSTORE write failure as primary purpose

**IPC types header is append-only** — existing struct definitions never modified once shipped (binary compatibility with TypeScript IPC layer).

**Applicability:** Any platform with settings/configuration that must work before all hardware drivers are present. Weak externs for drivers allow the settings layer to compile and ship before any driver exists. The four-layer taxonomy forces explicit classification of every setting's dependency.

## SKILL 24 — SWIM Gossip for Cluster Membership (Not Raft)

**Pattern Type:** distributed-systems, protocol-selection

**The Decision:** XNET uses SWIM (Scalable Weakly-consistent Infection-style Membership) for cluster membership, not Raft. Parameters: 1s probe interval, 200ms direct timeout, 5s suspect timeout, k=3 indirect probers, 8 piggybacked updates per message, max 64 members, UDP port 9000. Incarnation numbers prevent stale SUSPECT from overriding ALIVE. See ADR-S14-12.

**Why SWIM over Raft:** Raft provides strong consistency for data replication — this is better than needed for membership management (weakly consistent is sufficient). SWIM is decentralized (no single point of failure), O(log N) convergence, UDP-efficient (no TCP connection overhead per probe).

**Documented Bug:** `sender_id`, `member_id`, and `lamport` 64-bit fields were NOT passed through `htonll()` before serialization. Works fine on all-little-endian x86-64 cluster but would break on big-endian node. Documented explicitly in the ADR with "does not affect Sprint 14 functionality but tracked for cross-architecture work."

**Applicability:** Multi-device or multi-node systems requiring membership tracking. SWIM at small scale (< 64 nodes) is simpler and sufficient. Use Raft only when you need strong consistency for data replication, not just membership.

## SKILL 25 — Flexbox-Subset Layout Engine (Sufficient > Complete)

**Pattern Type:** UI-toolkit, algorithm-selection, progressive-spec

**The Decision:** XFRAME implements a Flexbox-subset layout engine: single-axis (row/column), justify, align_items, flex_grow/shrink/basis, min/max constraints, padding, margin, gap. Deliberately not implemented: flex-wrap, CSS Grid, absolute positioning, percentage widths. Single-pass O(n) algorithm, no constraint solver, no backtracking. ~1,200 LOC. See ADR-S14-11.

**Why Flexbox Subset over Full CSS Box Model:** Full CSS box model ~10,000-20,000 LOC. Flexbox-subset covers 100% of real XFRAME use cases (topbar = horizontal row, sidebar = vertical column, dialogs = vertical with scroll). The subset is sufficient. Start with "sufficient" — expand to "complete" when use cases require it.

**Why Not Cassowary Constraint Solver:** More expressive but adds ~3,000 LOC of mathematical solver. Constraint conflicts produce solver errors, not obvious layout bugs. Debuggability worse.

**Applicability:** Any embedded UI toolkit or custom layout engine. Start with Flexbox-subset. It handles ~90% of real application layouts. The remaining 10% (grid, wrap, absolute) can be added when required. Build the layout algorithm first as a clean single-pass function, then add language bindings.

## SKILL 26 — CRDT Selection by Data Type (Not One-Size-Fits-All)

**Pattern Type:** distributed-systems, collaborative-editing, data-structures

**The Decision:** XFRAME implements five separate CRDTs matched to their use cases: RGA for character sequences (collaborative text, proven convergence Roh et al. 2011), LWW Register for scalar state (app settings, cursor position — last write wins is correct semantics), G-Counter for aggregation (analytics), OR-Set for membership (tag sets, contact lists, honors both add and remove), Map CRDT for nested structures (document metadata). See ADR-S14-10.

**The Anti-Pattern:** Using a single CRDT for all data shapes (e.g., using LWW for text → concurrent edits overwrite each other; using RGA for settings → tombstone accumulation for simple scalar).

**Applicability:** Any multi-device sync or collaborative editing system. Match the CRDT to the data semantics: sequences → RGA or YATA, registers → LWW, counters → G-Counter, sets → OR-Set, maps → Map CRDT. The wire protocol and storage schema must be designed around the CRDT primitives from the beginning — they cannot be swapped later without breaking sync history.

## SKILL 27 — vDRAM Tiered Memory (PDT + Migration Lock)

**Pattern Type:** memory-management, performance, progressive-wiring

**The Decision:** vDRAM uses an in-memory Page Descriptor Table (PDT) rather than direct MMU page-table walks for tier migration. Each `vdram_page_desc_t` tracks virtual address, physical address, current tier (HOT/WARM/COLD), and migration lock bit. Migration: (1) set `migrating` bit under spinlock, abort if already set; (2) `pal_pages_alloc()` + copy + `INVLPG`; (3) update PDT, release lock. See ADR-S29-01.

**Why PDT over Real MMU walks:** Real page-table migration requires CR3 reload + IPI TLB shootdown to all CPUs. SMP IPI shootdown not yet validated under concurrent load — wiring real page tables creates a latent UAF risk. PDT approach is safe at UP, testable without QEMU, replaces with real MMU walks when SMP is validated.

**Pattern:** Build the observable behavior (tier tracking, migration events) before building the real implementation (MMU page walks). Use an intermediate data structure that mirrors the real state without the risky operations.

**Applicability:** Any incremental memory management feature where the full implementation depends on a lower-level subsystem not yet validated (SMP, IOMMU, DMA). Implement the safe intermediate, commit to the upgrade path.

## SKILL 28 — Async Daemon Microservice per AI Advisor

**Pattern Type:** AI-governance, service-design, concurrency

**The Decision:** Each Council advisor is an asyncio TCP daemon. CPU-bound subagents (Ruth's 11 specialized analyzers) launch via `asyncio.to_thread()` to avoid blocking the event loop. asyncio provides cooperative concurrency without OS thread context switch overhead and without data races from shared mutable state. Single event loop per process. See ADR-S29-02.

**Why asyncio over threads:** Multiple concurrent client connections per daemon (gate runner queries multiple advisors). Single event loop = no lock contention on shared state. Ruth's subagents are CPU-bound (not I/O-bound) so they correctly use `to_thread()`.

**Why not message bus (ZeroMQ-style):** Requires a broker process (single point of failure). External library dependency violates language policy (Python, TypeScript, C ONLY). Point-to-point TCP is sufficient for 14-daemon topology.

**Applicability:** Python microservice architectures with mixed CPU-bound and I/O-bound work. Use asyncio for I/O, `asyncio.to_thread()` for CPU. Fixed port assignments eliminate service discovery complexity at small scale.

## SKILL 29 — Per-Context State over Global Statics (Thread Safety)

**Pattern Type:** concurrency, safety, state-management

**The Decision:** XISC native layer moved TID allocation (`next_tid`), thread tracking, file descriptor table (`fd_table`), and virtual memory tracking (`vmar_table`) from global static arrays into `linux_thread_ctx_t` (per-thread context). Previous implementation had `s_thread_table[16]` and `s_next_tid` as global statics — unsafe in multi-thread personality layer. See ADR-S39-01.

**Fixed Capacity for vmar_table:** 64 entries per context (fixed array, not linked list or hash map). Rationale: no test program uses > 12 concurrent mappings, avoids secondary allocator dependency in dispatch layer, 64 * 24 bytes = 1,536 bytes negligible per-thread overhead.

**Applicability:** Any compatibility layer, runtime, or shim that handles concurrent callers. Audit every global static. Global mutable state in a multi-threaded context requires either locks (overhead) or per-context state (preferred). Per-context state scales to arbitrary concurrency without contention.

## SKILL 31 — GQA + Q4_0 Quantization for Edge AI

**Pattern Type:** AI-engineering, hardware-constraints, quantization

**The Decision:** XMIND uses Q4_0 quantization (4 bits/weight, 32-element blocks, one scale factor per block). Llama 3.2 3B at Q4_0 = ~1.5 GB weights + ~175 MB GV cache (with GQA 4:1 head ratio) = ~1.7 GB total. Leaves >28 GB for OS and applications on 32 GB target hardware. GQA reduces KV cache from 700 MB to 175 MB by sharing 1 KV head across 4 query heads. See ADR-S14-06.

**Why Q4_0 over Q8_0:** Q8_0 = 3 GB model, ~15 tokens/sec (DRAM bandwidth limited). Q4_0 = 1.5 GB model, ~30 tokens/sec. Doubles token rate at the cost of ~1-2% perplexity increase.

**Why GGUF format:** Most widely used Q4_0 quantization format. Models from llama.cpp `--quantize q4_0` load directly — GGUF compatibility gives access to the entire public quantized model ecosystem.

**Applicability:** Any on-device AI inference on hardware without discrete GPU. Q4_0 is the practical sweet spot for transformer inference on CPU-only hardware at 3B parameter scale. GQA is essential for managing KV cache memory at longer context lengths.

## SKILL 32 — One-Pass JIT with Linear Scan Register Allocation

**Pattern Type:** compiler-design, performance, minimalism

**The Decision:** XJIT uses a 22-opcode one-pass IR (not SSA), linear scan register allocation (one forward pass over live-range intervals), and direct x86-64 byte emission into a 176 KB static buffer. No intermediate assembly text, no assembler invocation, no ELF file. 256-entry direct-mapped block cache keyed by guest RIP for XEMU. <100 µs compile latency for a 50-instruction basic block. See ADR-S14-06.

**Why One-Pass over SSA:** SSA construction requires dominance tree (Lengauer-Tarjan, ~500 LOC), phi nodes, SSA destruction. One-pass is fast enough for short basic blocks (5-20 instructions). Optimization benefit of SSA form is marginal at this scale.

**Why Linear Scan over Graph Coloring:** Graph coloring is O(n²) or worse. Linear scan is O(n) with a single forward pass over live intervals. For a 14-register ISA, graph coloring provides minimal benefit over linear scan.

**Applicability:** JIT compilers for interpreted VMs, binary translators, dynamic recompilers. If the primary use case is short basic blocks (< 50 instructions), one-pass + linear scan is the correct tradeoff. SSA and graph coloring are for production optimizing compilers with long compilation pipelines.

## SKILL 33 — GENSD Sequential Priority Boot with Critical/Non-Critical Classification

**Pattern Type:** init-system, boot-sequence, resilience

**The Decision:** GENSD starts 10 services in strict priority order (XSTORE→XSEC→XPKG→XNET→XDISP→XCOMP→XORCHESTRA→XSHELL→XMIND→XPKG-UPDATES). Critical services (XSTORE, XSEC, XDISP, XCOMP, XSHELL) → rescue mode if 3 restarts fail. Non-critical (XPKG, XNET, XMIND) → GEN.OS boots to usable state without them. See ADR-S14-13.

**Why Sequential over Parallel Dependency-Resolved Boot:** Sequential priority boot simpler to reason about, sufficient for current sprint. Parallel boot requires explicit dependency declaration per service and dependency-resolution algorithm — deferred to later sprint.

**Restart Policies:** XMIND uses `RESTART_NEVER` (AI inference is non-essential). All critical services use `RESTART_ON_FAIL` with cooldown. Cooldown using deferred timer (not blocking sleep) to keep supervisor loop responsive.

**The Documented P0 Bug:** "All boot stages are stubs — no service is actually started at boot." Honest accounting in the ADR of known implementation gaps, not just design intent.

**Applicability:** Any init system or service manager. Classify services as critical vs non-critical from day one. Sequential ordered boot is simpler to debug. Publish the service dependency graph explicitly (it reveals hidden dependencies that cause mysterious init failures).

## SKILL 34 — Trust Anchor Two-Tier Model (Hardware Root + Software Key)

**Pattern Type:** security, PKI, trust-chain

**The Decision:** GEN.OS uses a two-tier trust model: Tier 0 (TPM 2.0 PCR measurements as hardware root of trust), Tier 1 (GEN.OS Package Authority Ed25519 root key, stored in `/etc/xpkg/trust-anchors/`, sealed by TPM PCR), Tier 2 (per-repository signing keys signed by Tier 1). This prevents a compromised repository from substituting its own key — the substitution would fail against the Tier 1 root. See ADR-S14-08.

**Security Finding SEC-01 Context:** Without out-of-band trust anchor, signature verification proves cryptographic integrity (package signed by whoever holds the key) but not authenticity (holder is actually GEN.OS package authority). The two-tier model adds the authenticity layer.

**Applicability:** Any package manager, update system, or code signing infrastructure. Never use a single-tier trust model. The hardware root of trust (TPM PCR, secure enclave, burned-in ROM key) must be the anchor that cannot be modified by software.

## SKILL 35 — Address Space Constants as CI-Verified Consistency Invariants

**Pattern Type:** systems-programming, consistency-enforcement, CI

**The Decision:** `HHDM_BASE = 0xFFFF888000000000` must appear in exactly 5 header files (xenos.h, xacpi.h, xhpet.h, xlapic.h, boot_handoff.h). Sprint 9 CI workflow includes an HHDM consistency check job that verifies all driver headers use the same constant. A discrepancy indicates a consistency bug. See ADR-S14-02.

**The Tight Coupling Risk:** "Any change requires a full rebuild and coordinated update of all occurrences. A grep for `0xFFFF888000000000` must match exactly 5 locations. Discrepancies indicate a consistency bug."

**Pattern:** When a constant must be consistent across multiple files and a discrepancy causes silent boot failure (not a compile error), add a CI job that greps for the expected value in all expected locations and fails if the count doesn't match.

**Applicability:** Any system with boot-critical constants that appear in multiple files (kernel base address, memory region boundaries, protocol magic numbers). A CI grep for exact occurrence count is a low-cost high-value regression guard.

## SKILL 36 — Immutable Data Structures for Concurrency Safety (XBLOB)

**Pattern Type:** concurrency, data-structures, correctness

**The Decision:** XBLOB blobs are immutable by definition: a key `SHA-256(data)` always refers to the same bytes. This eliminates update-in-place races and simplifies the concurrency model: writers never conflict with readers on the same key. The GC mark phase adds tombstones without locking active reads; the sweep phase is a separate background pass.

**Consequence on Lock Ordering:** The lock ordering bug (PIA-S4-P1-01) was that GC tombstone acquired `store_lock`, set tombstone, then wrote audit log entry INSIDE `store_lock` — creating a potential deadlock with the audit log's own lock. Fix: release `store_lock` before writing audit log entry.

**Rule:** Lock ordering must always acquire locks in the same order across all code paths. Document the lock ordering in a comment. If two locks must be acquired, the order must be consistent — always A then B, never B then A in any path.

**Applicability:** Any concurrent data structure where content never changes after insertion (caches, blob stores, immutable KV stores). Immutability is a concurrency primitive — if data cannot change, there are no write-write conflicts, only read-write conflicts, which are resolved by making the GC non-blocking.

## META-PATTERNS: Cross-Cutting Lessons from 49 Sprints

### Meta-1: Document Known Bugs in ADRs, Not Just Decisions
Every GEN.OS ADR has an "Errata" section. Sprint 14 ADRs openly documented P0 bugs (boot stages are stubs, XKABI_EXEC not implemented, txn_commit missing fdatasync). This is honest engineering: design documents capture both intent and current gaps.

### Meta-2: The "Sky OS Lesson" Pattern — Document Lessons from Previous Systems
`pal.h` explicitly documents "SKY OS LESSON: Sky OS built features on undocumented behavioral dependencies." Documenting failure modes from prior systems is architectural wisdom. Name the lesson, attribute it to a real project (even internal), preserve it in headers.

### Meta-3: Graceful Degradation as a First-Class Requirement
Non-critical service failure (AI, network) must never crash the desktop session. Every settings getter has a three-level fallback (live hardware → last persisted → compile-time default). Every PAL weak extern resolves to a safe fallback. "Crash loudly if critical, degrade gracefully if non-critical" is the GEN.OS resilience contract.

### Meta-4: Static Assertions as Structural Tests
`_Static_assert(sizeof(tss_t) == 104, "tss_t must be exactly 104 bytes")` in `xenos.h`. If the struct layout changes (field added, padding changed), the compile fails immediately. This is the cheapest form of structural testing. Use `_Static_assert` for any struct that maps to hardware-specified memory layouts.

### Meta-5: The Sequential Gate Principle
Boot sequence, package gate, Council gate chain — all are explicitly sequential. The order encodes the dependency graph. Sequential execution is always debuggable. Parallel execution is faster but requires explicit dependency declaration. Always start sequential, prove the correctness, then parallelize if timing data justifies it.

### Meta-6: Append-Only Protocols with Zero-Value Defaults
Every handoff struct, every IPC message format, every XSTORE key namespace is append-only. New fields added at the end with zero meaning "not provided". This enables forward compatibility: an old receiver can process a new struct (ignores unknown trailing fields) and a new receiver can process an old struct (zero-initializes missing fields from fallback defaults).

### Meta-7: CI Gates Enforce Architecture, Not Just Tests
The linux-header-scan gate enforces the "100% original" architectural constraint. The HHDM consistency check enforces address space layout consistency. XJIT compilation gate enforces freestanding compilation. CI is not just a test runner — it is an architecture enforcer. Any architectural invariant that cannot be verified by code review alone should have a CI gate.

## SKILL 31 — TSX-to-C Porting: Deduplication vs New File Decision Matrix

**Pattern Type:** porting-methodology, code-ownership

**The Decision (2026-04-05):** When porting Electron/TSX UI components to native XFRAME C, always audit `devices/desktop/shell/xshell/src/` BEFORE creating any new file.
