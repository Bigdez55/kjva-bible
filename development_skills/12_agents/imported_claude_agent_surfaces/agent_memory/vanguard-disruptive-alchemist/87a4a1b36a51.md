## P0 Regressions (Fix This Week)

### 1. mr_registry.c list_del Ordering Race
- File: kernel/linux-validation/vdram/mr_registry.c:141
- list_del(&mr->node) happens AFTER synchronize_rcu()
- During grace period, pf_find_mr() in pf_handler.c:42 can find MR on mr_list
- pf_find_mr does atomic_inc(&mr->ref) on an MR that is about to be freed
- Fix: move list_del before synchronize_rcu under node->mr_lock, OR use kfree_rcu

### 2. xkabi_capabilities.c cap Pointer UAF After Lock Release
- File: kernel/linux-validation/security/xkabi_capabilities.c:228
- cap pointer obtained under ht->lock at line 218
- ht->lock released at line 221
- cap->generation read at line 228 without ht->lock held
- Concurrent xkabi_cap_revoke can free cap between lines 221 and 228
- Fix: capture generation into local u32 while lock is held

## Radical Alternatives

### SipHash-2-4 for XKABI (replaces HMAC-SHA256)
- 15-25ns vs 120-600ns on 8-byte input
- linux/siphash.h in-tree since 4.13
- Eliminates crypto_alloc_shash lifecycle (40 LOC -> 1 LOC)
- Same 2^64 collision resistance
- Migration: 1 week Strangler Fig

### CLZ/CRC32C for Entropy (replaces 256-entry LUT)
- 0 divisions vs up to 256
- 0 bytes LUT vs 512 bytes
- CRC32C is 3 cycles/byte on Intel/ARM
- Or: sample first 64 bytes only (>0.95 correlation with full payload)
- Migration: 2 weeks with telemetry validation

### kfree_rcu for MR free (replaces synchronize_rcu)
- 0ms vs 10-20ms per free
- Requires adding struct rcu_head to vdram_mr (16 bytes)
- 1000 MR teardown: 0ms vs 10-20 seconds
- Migration: immediate, single commit

### Newton's Law for Thermal (replaces linear extrapolation)
- dT/dt = -k(T - T_ambient) -> exponential, not linear
- Same OLS in log space -> same computational cost
- False positive rate: <5% vs ~35% near thermal equilibrium
- Migration: 1 week A/B test
