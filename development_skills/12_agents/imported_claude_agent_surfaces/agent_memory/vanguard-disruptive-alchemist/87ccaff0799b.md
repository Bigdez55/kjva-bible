## 1. AES-NI DISPATCH: DOES NOT EXIST

### The Damning Evidence

`sec/xsec/crypto/aes_gcm.c` line 24 **comments claim** an AES-NI hardware path exists:

```
 *   1. AES-NI hardware path (x86_64 with AES-NI) -- single-instruction,
 *      inherently constant-time.
```

**This is a lie.** There is no AES-NI code anywhere in the file. Zero inline assembly.
Zero `__builtin_ia32_aesenc`. Zero CPUID detection. The entire 499-line file is pure
software AES with a 256-byte S-box, a constant-time linear scan (`aes_sbox_ct`), and
a software GF(2^8) multiply.

### The Invisible Tax

The i5-8265U (Whiskey Lake) has:
- AES-NI: `AESENC`, `AESDEC`, `AESKEYGENASSIST` (since Westmere, 2010)
- PCLMULQDQ: carryless multiply for GHASH (since Westmere, 2010)

Performance delta:
- Software AES-256 block: ~1,800 cycles (14 rounds x 16 SubBytes x 256-scan + MixColumns)
- AES-NI AES-256 block: ~56 cycles (14 AESENC instructions)
- **Ratio: ~32x slower than hardware**
- Software GHASH (128 iterations of bit-scan with branching): ~3,000 cycles per block
- PCLMULQDQ GHASH: ~10 cycles per block
- **Ratio: ~300x slower than hardware**

The `aes_sbox_ct` function alone does 256 iterations per byte, 16 bytes per SubBytes,
14 rounds = **57,344 iterations per AES block encrypt**. AES-NI does the same work
in a single clock cycle per round.

### P2 Fix Assessment

The P2 fix made `gf_mul` constant-time (lines 84-98) and `aes_sbox_ct` constant-time
(lines 58-68). Both fixes are **technically correct** and **operationally irrelevant**.
On the i5-8265U, `AESENC` is inherently constant-time by construction. We spent
engineering effort making a fallback path secure when the primary path should be
hardware. This is akin to fireproofing the escape raft while the ship has no hull.

### GHASH Still Has Timing Side-Channels (P0 REGRESSION)

Despite the S-box and gf_mul fixes, `ghash_mul` at lines 262-284 **still branches
on secret data**:

```c
if (xi & 0x80u) {                    // line 265: BRANCHES ON HASH INPUT
    for (k = 0u; k < 16u; ++k) { result[k] ^= V[k]; }
}
...
if (lsb) { V[0] ^= 0xe1u; }         // line 280: BRANCHES ON INTERMEDIATE
```

The P2 fix fixed AES but **missed GHASH**. The GHASH multiplication remains
variable-time. An attacker observing GCM decryption timing can recover the hash
subkey H, enabling forgery. This is NIST-grade broken.

## 2. SHA-NI DISPATCH: DOES NOT EXIST

### Ed25519 Bottleneck Analysis

Ed25519 has three computational phases:
1. SHA-512 hashing (key expansion, nonce generation, challenge computation): ~3 calls
2. fe10 field arithmetic (scalar multiplication on Curve25519): ~3000 field ops
3. Scalar reduction mod l

The bottleneck is **field arithmetic, not hashing**. SHA-512 is called 3 times per
sign and 3 times per verify. Each SHA-512 call processes <200 bytes. At ~14 cycles/byte
software, that is ~8,400 cycles total for SHA. The scalar multiply is ~500,000 cycles.

SHA-NI would save ~70% of SHA-512 time = ~5,880 cycles. Against 500,000 cycles of
fe10 arithmetic, this is **~1.2% improvement**. Not worthless, but not the bottleneck.

### The Real Ed25519 Opportunity

The i5-8265U has no dedicated Curve25519 hardware. But it has AVX2 (256-bit SIMD).
The fe10 representation (10 x int32_t limbs) maps perfectly to AVX2 vector operations:
- 8 limbs fit in one __m256i register
- Field add/sub: single `_mm256_add_epi32` / `_mm256_sub_epi32`
- Field mul: `_mm256_mullo_epi32` + carry propagation

This would give **2-4x speedup on the actual bottleneck** vs 1.2% from SHA-NI.
Nobody is looking at this because everyone is staring at the hash function.

## 3. CIPHER SUITE PRIORITY: INVERTED

### The i5-8265U Instruction Set Reality

| Instruction | Available | Used in XSEC |
|-------------|-----------|--------------|
| AES-NI      | YES       | NO           |
| PCLMULQDQ   | YES       | NO           |
| SHA-NI      | YES       | NO           |
| AVX2        | YES       | NO           |

On this CPU, AES-256-GCM with hardware acceleration would be:
- AES-NI + PCLMULQDQ: ~1.2 cycles/byte (Intel measured)
- Software AES-GCM (current): ~40-60 cycles/byte
- ChaCha20-Poly1305 software: ~4-6 cycles/byte

### Current Cipher Priority (tls13.c line 1050)

```c
ctx->cipher = XSEC_TLS_CIPHER_AES_256_GCM_SHA384;  /* prefer AES-GCM */
```

This preference is **correct in intent but catastrophic in practice**. Without AES-NI
dispatch, the "preferred" cipher is 10x slower than ChaCha20-Poly1305. The client
offers AES-GCM first (line 621), the server selects it, and every TLS record pays
a 10x penalty vs the alternative that is sitting right there.

### The SHA-256/SHA-384 Transcript Hash Mismatch (RECONFIRMED)

`xsec_tls_ctx_t.transcript_hash` is `xsec_sha256_ctx_t` (line 348).
Cipher suite 0x1302 (AES-256-GCM-SHA384) requires SHA-384 for the transcript hash.
The code uses SHA-256 everywhere (line 1061, line 438, lines 973-994).

**This means AES-256-GCM-SHA384 produces wrong key material.** The HKDF-Expand-Label
calls use 32-byte SHA-256 outputs where RFC 8446 requires 48-byte SHA-384 outputs.
The handshake will succeed with a conformant peer only by accident (if the peer
also gets it wrong) or fail silently with truncated key derivation.

This was flagged in the Sprint 3 build-vs-adopt audit. It remains unfixed.

## 4. DNS: STUB RESOLVER IS STRUCTURALLY INSECURE

### Random QID Is Necessary But Not Sufficient

The P2 fix replaced sequential QIDs with `pal_random_bytes()` (line 384). This
closes the Kaminsky attack vector for QID prediction. But:

1. **Source port is not randomized.** The `xnet_udp_bind(0, ...)` call (line 427)
   binds an ephemeral port, but there is no evidence the port allocation is random.
   Kaminsky's full attack requires predicting BOTH QID and source port. If the
   ephemeral port allocator is sequential (common in simple stacks), the attack
   surface is still ~2^16, not ~2^32.

2. **No DNSSEC validation.** Without DNSSEC, a race condition between the legitimate
   response and a forged response is always possible. Random QID raises the bar
   from trivial to difficult, not from difficult to impossible.

3. **No DNS-over-TLS/HTTPS.** On a security-focused OS, DNS queries leak the
   user's browsing intent in plaintext to the network. This is a privacy regression
   vs every major OS shipping in 2026.

4. **Single pending query (g_dns_pending).** Only one DNS resolution can be in
   flight at a time. Concurrent hostname lookups serialize on g_dns_pending_lock.
   This is a throughput bottleneck for any application that resolves multiple
   hostnames (e.g., loading a web page with 30+ domains).

### The Architecture Question

The right question is not "is the stub resolver secure?" but "should we have a
stub resolver at all?" DNS-over-TLS (RFC 7858) tunnels DNS through the TLS layer
we already built. This:
- Eliminates QID/port prediction attacks entirely (TLS provides integrity)
- Provides privacy (encrypted queries)
- Reuses existing XSEC TLS code (no new attack surface)
- Is the default in Android 9+, iOS 14+, macOS Big Sur+, Windows 11

A stub resolver speaking plaintext UDP to port 53 is a 1987 design. We are building
a 2026 operating system. The entire plaintext DNS path should not exist.

## 5. TCP CONGESTION CONTROL: RENO AIMD IN 2026

### What We Have (tcp.c lines 458-498)

```c
static void tcp_congestion_init(xnet_tcb_t *t) {
    t->cwnd     = 10u * t->snd_mss;     /* RFC 6928: initial cwnd = 10 MSS */
    t->ssthresh = TCP_SNDBUF_SIZE;
}

/* ... */
/* Congestion avoidance: Reno AIMD */
uint32_t increment = (t->snd_mss * t->snd_mss) / t->cwnd;
t->cwnd += increment;
```

Despite the comment header claiming "simplified CUBIC," the actual implementation
is **Reno AIMD** (additive increase, multiplicative decrease). CUBIC's distinguishing
feature is the cubic function `W(t) = C(t-K)^3 + W_max` -- there is no cubic
polynomial anywhere in this code.

### The State of the Art

| Algorithm | Year | Status | Key Property |
|-----------|------|--------|--------------|
| Reno AIMD | 1988 | Obsolete | Linear increase, halve on loss |
| CUBIC     | 2008 | Linux default since 2006 | Cubic window growth, independent of RTT |
| BBR       | 2016 | Google/YouTube default | Model-based, not loss-based |
| BBRv2     | 2022 | In deployment | Fixes BBRv1 fairness issues |
| BBRv3     | 2024 | Bleeding edge | Improved loss response |

We are shipping **Reno** (1988) and calling it "CUBIC" (2008). This is the TCP
equivalent of shipping Internet Explorer 6 and calling it Chrome.

### initcwnd = 10 MSS Assessment

RFC 6928 (2013) established IW10 as the standard. This is still correct in 2026.
Google experimented with IW32 and IW64 but found diminishing returns on lossy
networks. IW10 is fine. The congestion avoidance algorithm is the problem, not
the initial window.

### Should We Implement BBR?

**No.** BBR requires precise bandwidth estimation (bottleneck bandwidth) and RTT
measurement during specific phases (ProbeRTT). It is deeply stateful and interacts
badly with other BBR flows in ways that took Google years to fix. For a single-user
OS with modest networking needs, CUBIC is the right target. But we should actually
implement CUBIC, not Reno with a misleading comment.

The real fix is 15 lines: replace the linear `mss^2/cwnd` increment with the cubic
function and track `W_max` and `epoch_start` per TCB. This is a weekend task.

## 6. ADDITIONAL FINDINGS

### 6a. ChaCha20-Poly1305 Stack Allocation: 32KB+

`chacha20.c` line 374:
```c
#define POLY_SCRATCH_MAX  (16384u + 16u + 16384u + 16u + 16u)
uint8_t poly_scratch[POLY_SCRATCH_MAX];
```

This allocates **32,848 bytes on the stack**. In a freestanding kernel environment
with typical 8KB-16KB kernel stacks, this will overflow. The comment acknowledges
this ("Stack-local scratch") but does not solve it. This was flagged as a P0 in the
Sprint 3 audit (180KB stack in dep_resolve, 17KB in TLS). The ChaCha20 path adds
another 32KB to the stack pressure.

### 6b. TLS Session Context: 17KB on Stack

`xsec_tls_ctx_t` contains `record_buf[16640]` (line 367). If allocated on stack
(which is the freestanding pattern), this is 17KB+ per TLS session. Combined with
ChaCha20's 32KB scratch, a TLS record decrypt using ChaCha20 requires ~50KB of
stack space. This is not viable on a 16KB kernel stack.

### 6c. Poly1305 Final Reduction Branches on Secret

`chacha20.c` lines 298-304:
```c
if (do_reduce) {
    acc_final = ((u128)(uint64_t)t5h << 64u) | (uint64_t)t5;
} else {
    acc_final = ((u128)acc_hi << 64u) | acc_lo;
}
```

The final reduction in Poly1305 branches on `do_reduce`, which depends on the
accumulator value, which depends on the message and key. This is a timing
side-channel in the MAC. A constant-time select should be used:
`acc_final = ct_select(do_reduce, reduced, unreduced)`.

## LATERAL INSIGHT: The Pharmaceutical Purity Paradox

In pharmaceutical manufacturing, companies spend billions ensuring that every step
of drug synthesis meets purity standards. But the most common cause of drug recalls
is not impure synthesis -- it is wrong formulation: the right chemical at the wrong
dose, or the right tablet with the wrong label.

Our P2 hardening is the crypto equivalent. We are ensuring purity of the software
AES path (constant-time operations, no cache-timing leaks) while the formulation
is wrong: we are dispensing software crypto when the CPU has hardware crypto
instructions that are both faster AND inherently constant-time.

The AES-NI instruction `AESENC` cannot have a timing side-channel because it
executes in fixed cycles by construction. By failing to use it, we created the
timing side-channel vulnerability, then spent effort fixing it, and still missed
two instances (GHASH, Poly1305). The hardware path would have eliminated all
three vulnerabilities simultaneously.

This is the Pharmaceutical Purity Paradox: perfecting the wrong formulation
makes the drug no safer, but creates a false sense of security that prevents
the team from questioning whether the formulation itself is correct.
