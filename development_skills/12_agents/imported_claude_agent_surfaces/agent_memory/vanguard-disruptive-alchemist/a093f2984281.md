## CRITICAL ITEM VERIFICATION

### Item 1: 0x40 Opcode Collision -- STILL PRESENT (P1)

**Status: UNFIXED. Collision confirmed in code.**

Evidence:
- `xisc/include/xisc.h:78` -- `XISC_COMMIT = ((xisc_opcode_t)0x40)`
- `genvirt/jit/xisc_llvm_jit.c:73` -- `XISC_OP_MATMUL = 0x40`

Additionally, secondary collision at 0x41:
- `xisc/include/xisc.h:79` -- `XISC_VERIFY = ((xisc_opcode_t)0x41)`
- `genvirt/jit/xisc_llvm_jit.c:74` -- `XISC_OP_MOELOAD = 0x41`

And 0x20/0x21 are silently redefined:
- `xisc.h:64` -- `XISC_JMP = 0x20`, `XISC_JEQ = 0x21`
- `xisc_llvm_jit.c:76-77` -- `XISC_OP_STORE = 0x20`, `XISC_OP_LOAD = 0x21`

**Impact:** The JIT module defines its OWN opcode namespace that collides with the
canonical ISA header in 6 positions. Any program using XISC_COMMIT(0x40) bytecode
fed to the JIT would be misinterpreted as MATMUL. This is a silent semantic
corruption bug -- the most dangerous kind.

**Root cause:** xisc_llvm_jit.c:72 says `/* from spec/xisc_isa.h */` but was never
actually synchronized with xisc/include/xisc.h. Two separate opcode allocations exist.

**Fix (10 minutes):** Either:
(a) JIT includes xisc.h directly and extends with ML-specific opcodes at 0x60+
(b) Assign ML opcodes to 0x60-0x6F range in xisc.h (Merkle stays at 0x40)

### Item 2: XKABI 2-byte MAC Truncation -- STILL PRESENT (P2)

**Status: DEFERRED (acknowledged risk).**

Evidence: `kernel/linux-validation/security/xkabi_capabilities.c:95`:
```c
memcpy(&handle_out[14], mac, 2);  /* 2-byte truncated MAC */
```

**Quantified risk:** 16-bit MAC = 65,536 possible values. At kernel syscall rate
(~1M/sec on modern hardware), brute-force collision in <66ms. This is NOT a
theoretical concern -- it is a practical attack vector for any local process
that can repeatedly invoke xkabi_cap_validate().

**Mitigating factors (partial):**
- Per-process HMAC key (attacker needs to target specific process)
- Generation counter adds 32-bit epoch check
- idr_find uses bottom 24 bits of object_id (adds noise)

**Net assessment:** The MAC truncation degrades XKABI from "capability-secure"
to "capability-with-integrity-hint." Production deployment MUST expand to
minimum 4 bytes (birthday bound ~1 hour at 1M/sec) or ideally 8 bytes
(birthday bound >500 years). Handle format would expand from 128 to 176 bits.

### Item 3: Entropy Classifier LUT Sparsity -- STILL PRESENT (P3)

**Status: DEFERRED (acknowledged, no change).**

Evidence: `gateway/zc_gw/entropy_classifier.c:18-24`:
```c
static const u16 log2_lut[256] = {
    [0]  = 0,    [1]  = 0,    [2]  = 512,  [3]  = 813,  [4]  = 1024,
    [5]  = 1194, [6]  = 1336, [7]  = 1459, [8]  = 1568, [16] = 2048,
    [32] = 2560, [64] = 3072, [128]= 3584, [255]= 4080,
};
```

13 of 256 entries populated. All uninitialized entries are 0, meaning
frequencies 9-15, 17-31, 33-63, 65-127, 129-254 contribute ZERO to the
entropy calculation. This systematically underestimates entropy for any
non-trivial distribution, biasing classification toward FABRIC_UET (low entropy).

**Impact on routing:** Medium-entropy payloads (gradient sync, mixed workloads)
will be consistently misrouted to UET instead of RoCEv2, losing bandwidth.
