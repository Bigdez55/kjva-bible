## CRITICAL P0 Gaps
1. Q4_0 is lowest-quality GGUF format -- must implement Q4_K_M (~400 LOC)
2. 2048-token context uses only 25% of Llama 3.2 3B's trained capacity -- increase to 8192
3. GGUF parser has no SHA-256 verification before loading (supply chain risk)
4. GGUF array metadata scanning can desync parser (weights_loader.c:310-312)

## Key Technology Decisions Validated
- Linear scan regalloc: correct for XEMU's narrow JIT scope
- SYSCALL->INT3 rewriting: correct for same-architecture (vs full binary translation)
- B+ tree for XSTORE: correct for single-device read-heavy
- SWIM gossip: correct, no superior alternative for this scale

## Key Technology Decisions Challenged
- Q4_0: WRONG. Q4_K_M costs +200MB RAM for +15% perplexity improvement
- 2048 ctx: WRONG. 8192 costs +1GB RAM, uses model's full trained capacity
- No SIMD in tensor.c: acceptable for Sprint 6 but 50-100x perf gap vs AVX2
- Byte-level tokenizer: WRONG. Real BPE needed for coherent output

## GGUF Supply Chain Risk (NEW)
- 6 known attack vectors against GGUF parsers (2024-2025)
- Poisoned GGUF Templates (July 2025): Jinja2 injection via metadata
- XMIND NOT vulnerable to template attacks (no template eval)
- XMIND IS vulnerable to: metadata overflow, array desync, no hash check
- Mitigation: Add xmind_verify_gguf_hash() using XPKG's Ed25519+SHA-256
