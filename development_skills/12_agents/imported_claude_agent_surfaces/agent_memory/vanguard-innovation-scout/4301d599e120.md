## Dimension 4: Secret Management

### What P6 Fixed
- Hardcoded Ed25519 private key REMOVED from platform init
- Hardcoded MinIO root credentials REMOVED
- _require_env() pattern: raises RuntimeError with clear message if env var not set
- All k8s deployments use secretKeyRef pointing to genos-secrets

### Assessment of _require_env()
The pattern is CORRECT for this deployment model. Here is why:

1. GEN.OS is a single-node, airgapped, local-first OS. There is no Vault server, no AWS KMS,
   no cloud HSM. The threat model is local compromise, not network exfiltration.
2. Secrets flow: systemd EnvironmentFile -> k3s Secret -> pod env -> os.environ[]
3. _require_env() is a fail-fast guard -- it prevents boot with missing secrets, which is the
   right behavior for a local OS that must be fully configured before first run.

### What Could Be Better
- The identity service has a dev-key fallback (IDENTITY_ALLOW_DEV_KEYS) that generates a
  random key if no PEM is provided. This is correctly gated behind an env var set to "false"
  in production k8s manifest. Good.
- sync/app.py uses os.environ[] (KeyError on missing) instead of _require_env() -- inconsistent
  but functionally equivalent. Consolidate for readability.
- Consider: python-dotenv for local development ergonomics, but NOT for production path.
- Long-term: If GEN.OS ever connects to a network, implement age (https://github.com/FiloSottile/age)
  for encrypted secrets at rest in the EnvironmentFile.

### Verdict: _require_env() is the RIGHT approach. No library needed.
