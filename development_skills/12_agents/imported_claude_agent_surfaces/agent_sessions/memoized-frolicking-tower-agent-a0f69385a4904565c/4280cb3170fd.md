# Quantitative Strategy & Execution Review: Elson TB2 DRL Trading System

## OVERALL VERDICT: AGREE WITH MODIFICATIONS

**7 of 10 areas require correction before implementation. 2 are sound as-is. 1 should be descoped entirely.**

The plan is architecturally ambitious and demonstrates strong quantitative intuition. However, it contains several infrastructure contradictions that would cause production failures, and proposes compute-intensive workloads on hardware that cannot support them. Below is a point-by-point audit with specific remediation recommendations.

---

## 1. DRL Inference on Cloud Run: CRITICAL BLOCKER -- DISAGREE

### The Problem

The plan proposes PPO/SAC neural network policies for live trade decisions. The production container (`requirements-docker.txt`) explicitly removes `torch`, `transformers`, and `ta-lib` with the comment:

```
# Note: tensorflow, torch, transformers removed - not used in backend, causes build failures
# ta-lib removed - requires system libta-lib not in container
```

**This is not a soft constraint.** Cloud Run builds with `requirements-docker.txt`. PyTorch alone adds ~2GB to the container image. With 2GB RAM allocated in `cloudbuild.yaml` (`--memory=2Gi`), loading a PyTorch model into the same process that runs FastAPI + SQLAlchemy + the auto-trading loop would cause OOM kills.

### Current Reality

The existing `reinforcement_learning.py` (1100+ lines) already uses lazy imports with `TF_AVAILABLE` / `GYM_AVAILABLE` guards and raises `RLUnavailableError` when tensorflow/gymnasium are missing. This means **RL is already disabled in production** -- it only runs locally.

The live auto-trading loop (`auto_trading_service.py`) uses a simpler architecture:
- AI-first: `eft_generate()` calls elson-finance-14b via vLLM HTTP API (running on the L4 GPU VM at 10.138.0.4)
- Fallback: rule-based strategy signals from `StrategyRegistry` (RSI, momentum, etc.)
- No neural network inference in the Cloud Run process

### Recommendation: ONNX Runtime Inference Sidecar

**Option A (Recommended): Export trained models to ONNX, serve via the existing L4 VM**

```
Training (L4 VM) --> PyTorch PPO/SAC --> torch.onnx.export() --> .onnx file
                                                                    |
Inference (L4 VM) --> ONNX Runtime Server (port 8001)              |
                         ^                                          |
Cloud Run auto_trading_service.py --> HTTP POST /predict -----------+
```

- `onnxruntime` is 50MB (vs PyTorch 2GB). Could also run on Cloud Run if needed.
- The L4 VM already runs vLLM on port 8000. Add an ONNX inference endpoint on port 8001.
- Inference latency: ~2-5ms for a small MLP policy network over internal VPC.
- Fits the existing pattern: Cloud Run makes HTTP calls to the GPU VM.

**Option B: Add `onnxruntime` to `requirements-docker.txt`**

- ONNX Runtime CPU is ~50MB, fits within 2GB RAM.
- Load `.onnx` model files from GCS at startup.
- Inference on CPU: ~5-15ms for typical PPO policy networks.
- No GPU needed for inference -- the networks are small (2-3 hidden layers, 256 units).

**Option C (NOT recommended): Add PyTorch to production container**

- Would require increasing Cloud Run memory to 4GB+.
- Would increase cold start time from ~3s to ~15s.
- Would increase container image from ~800MB to ~3GB.
- Contradicts the scale-to-zero cost optimization.

**Verdict: The plan MUST specify the inference runtime. PPO/SAC training uses PyTorch; inference must use ONNX Runtime or a remote serving endpoint. This is a Phase 0 prerequisite, not a detail to defer.**

## 2. Training Compute: DISAGREE -- Unaddressed

### The Problem

Cloud Run: 2GB RAM, 1 CPU, scale-to-zero. PPO training with 2048 steps/update on multi-asset environments requires:
- GPU memory: ~4-8GB for batch processing
- Training time: hours to days depending on environment complexity
- Cannot run on Cloud Run under any configuration

The plan mentions "weekly retraining via Celery" but does not specify WHERE Celery workers run.

### Current Infrastructure

- L4 GPU VM (`elson-dvora-training-l4-2`, us-west1-a): Currently runs vLLM for elson-finance-14b inference. Has 24GB GPU RAM. Could time-share between inference and training if scheduled properly.
- The VM is STOPPED on weekends (cost savings) and started before market open.

### Recommendation: Time-Shared L4 Training Pipeline

```
Weekend (VM running anyway for Monday prep):
  Saturday 00:00 UTC --> Stop vLLM service
  Saturday 00:05 UTC --> Run PPO/SAC training (12-18h)
  Sunday 06:00 UTC  --> Training complete, export ONNX
  Sunday 06:05 UTC  --> Upload .onnx to GCS
  Sunday 06:10 UTC  --> Restart vLLM service
  Monday 06:00 PST  --> Cloud Run pulls latest .onnx from GCS
```

**Key constraints:**
- Training and vLLM inference cannot run simultaneously on a single L4 (24GB). elson-finance-14b (14B params, likely quantized to ~8GB) leaves ~16GB for training, which is tight for PPO with large replay buffers.
- Alternative: Use a preemptible T4 ($0.11/hr) for training. 16GB GPU RAM is sufficient for PPO on a retail-scale environment. Weekly cost: ~$2 for 18h of training.
- NEVER suggest deleting the L4 VM (hard rule from MEMORY.md).

**The plan must add a "Training Infrastructure" section specifying:**
1. Where training runs (L4 VM time-share or preemptible T4)
2. How models are exported (ONNX format + GCS bucket)
3. How Cloud Run loads updated models (GCS pull at startup or periodic refresh)
4. Rollback mechanism (keep last 3 model versions in GCS)

## 3. Feature Engineering Latency: AGREE WITH MODIFICATIONS

### The Problem

The plan targets 200ms total latency for: multi-timeframe OHLCV features + FinBERT sentiment + HMM regime detection + feature normalization. On 1 CPU with 2GB RAM.

### Latency Budget Analysis

| Component | Estimated Latency (1 CPU) | RAM | Notes |
|-----------|--------------------------|-----|-------|
| OHLCV features (RSI, MACD, BB, ATR) | 5-15ms | ~50MB | NumPy/pandas vectorized ops |
| Rolling 252-day z-score normalization | 2-5ms | ~20MB | Simple rolling window |
| FinBERT inference (per headline) | 50-150ms | ~1.3GB | **BLOCKER: model loading alone exceeds budget** |
| HMM regime detection (2-state) | 10-30ms | ~10MB | scipy/hmplearn, small state space |
| Microstructure (OFI, spread, etc.) | 5-10ms | ~20MB | Requires L2 order book data |
| Total | 72-210ms | ~1.4GB | FinBERT dominates |

### Critical Issues

1. **FinBERT requires `transformers` + `torch`** -- both removed from production container. The plan does not acknowledge this.
2. **FinBERT model size**: ~420MB on disk, ~1.3GB in RAM. On a 2GB Cloud Run instance already running FastAPI + SQLAlchemy + the trading loop, this causes OOM.
3. **Microstructure features (OFI, queue depth imbalance)**: Require Level 2 order book data. Alpaca's basic plan provides top-of-book only. The SIP feed for L2 is $9/mo (Alpaca Pro). The plan does not specify the data tier.

### Recommendation

**A. FinBERT: Run on the L4 VM as a microservice**

The L4 VM already runs vLLM. FinBERT (110M params) is trivial to co-host:
- Add a `/sentiment` endpoint on the GPU VM (FastAPI + transformers, port 8002)
- Cloud Run calls it via VPC connector (same pattern as vLLM calls)
- Batch 5-10 headlines per request to amortize network overhead
- Cache results in Redis with 15min TTL (sentiment does not change per-second)
- Latency: ~10ms GPU inference + ~5ms network = 15ms total

**B. HMM: Precompute, do not run per-tick**

- HMM regime detection on 252 days of data should run once per day at market open, not every 60s loop iteration.
- Store regime state in Redis. The trading loop reads the cached regime.
- This removes 10-30ms from the hot path.

**C. ta-lib: Replace with pandas-ta or manual implementations**

- `ta-lib` requires `libta-lib.so` system library (not in Cloud Run container).
- The existing codebase already has a manual RSI implementation in `auto_trading_service.py` (lines 690-700).
- Use `pandas-ta` (pure Python, pip-installable) or continue the manual approach.
- Add `pandas-ta>=0.3.14b` to `requirements-docker.txt`.

**D. Microstructure features: Defer to Phase 2**

- OFI and queue depth require L2 data subscription.
- Confirm Alpaca data tier before implementing.
- These features add ~1-3 bps of alpha on retail-size orders -- diminishing returns vs. complexity.

**Revised latency budget (after modifications):**

| Component | Latency | Location |
|-----------|---------|----------|
| OHLCV + indicators (pandas-ta) | 10-20ms | Cloud Run |
| Z-score normalization | 2-5ms | Cloud Run |
| FinBERT sentiment (cached) | 0ms (cache hit) / 15ms (miss) | Redis cache / L4 VM |
| HMM regime (cached) | 0ms | Redis cache (precomputed daily) |
| Total hot-path | 12-25ms | Fits within 200ms target |

## 4. Reward Shaping: AGREE WITH MODIFICATIONS

### The Proposed Reward

```
R = alpha * delta_PnL - beta * TransactionCosts - gamma * InventoryPenalty - delta * DrawdownPenalty - epsilon * CVaR_Violation
```

### Assessment

**Strengths:**
- Multi-objective reward is correct for trading. Single-objective (PnL only) leads to excessive risk-taking.
- Transaction cost penalty prevents churn (critical for retail accounts with limited capital).
- Drawdown penalty aligns with the circuit breaker architecture already in place.
- CVaR violation penalty is more robust than VaR for tail risk.

**Risks:**

1. **Reward hacking via inventory manipulation**: The agent can learn to hold zero inventory (all cash) to minimize inventory penalty and drawdown penalty simultaneously, achieving reward ~0 (which beats negative reward from bad trades). This is a well-documented PPO failure mode.

   **Fix**: Add a **participation bonus** term: `+ zeta * abs(portfolio_utilization - target_utilization)`. Force the agent to stay invested at a target level (e.g., 60-80% invested). Without this, the agent converges to a no-op policy.

2. **Coefficient sensitivity**: 5 coefficients (alpha through epsilon) create a hyperparameter search space of ~10^5 combinations. PPO is notoriously sensitive to reward scale.

   **Fix**:
   - Normalize each reward component to [-1, 1] range BEFORE applying coefficients.
   - Start with: alpha=1.0, beta=0.3, gamma=0.1, delta=0.5, epsilon=0.3, zeta=0.2.
   - Use population-based training (PBT) to search coefficients during walk-forward.

3. **Reward delay**: PnL is realized only on position close. Between open and close, the agent receives no PnL gradient -- only penalty signals. This creates a sparse reward problem.

   **Fix**: Use **mark-to-market PnL** (unrealized gains/losses) for the delta_PnL component, not just realized PnL. This provides denser gradient signal.

4. **CVaR computation cost**: Computing CVaR at every step requires maintaining a return distribution. On a 60s loop with 20 assets, this is tractable but adds ~5ms per step.

   **Fix**: Compute CVaR on a rolling 252-step window, cached and updated incrementally (O(1) per new observation using sorted insertion).

### Recommended Reward Function

```python
def compute_reward(self, state, action, next_state):
    # All components normalized to [-1, 1]
    pnl_component = np.clip(mark_to_market_pnl / self.pnl_normalizer, -1, 1)
    cost_component = np.clip(transaction_costs / self.cost_normalizer, 0, 1)
    inventory_component = np.clip(abs(inventory_imbalance) / self.max_inventory, 0, 1)
    drawdown_component = np.clip(current_drawdown / self.max_allowed_drawdown, 0, 1)
    cvar_component = np.clip(cvar_violation / self.cvar_limit, 0, 1)
    participation_component = np.clip(
        abs(portfolio_utilization - self.target_utilization) / 0.5, 0, 1
    )

    reward = (
        1.0 * pnl_component
        - 0.3 * cost_component
        - 0.1 * inventory_component
        - 0.5 * drawdown_component
        - 0.3 * cvar_component
        - 0.2 * participation_component
    )
    return reward
```

## 5. CQL Offline Bootstrap: AGREE WITH MODIFICATIONS

### The Proposal

Use Conservative Q-Learning (CQL) to bootstrap policy from historical OHLCV replay buffer before online PPO fine-tuning.

### Assessment

**The approach is directionally correct** but the replay buffer specification is insufficient.

**Issues:**

1. **OHLCV replay buffer is NOT sufficient for CQL.** CQL requires (state, action, reward, next_state) tuples. OHLCV data contains only states (market observations). You need to either:
   - **Synthesize actions** from a heuristic policy (e.g., "what would a momentum strategy have done?") and compute rewards retroactively. This is standard practice but must be documented.
   - **Use the existing TradeDecisionLog table** which stores actual decisions made by the auto-trading service. This is better -- real actions with real outcomes.

2. **Distribution shift**: CQL's conservatism penalty discourages out-of-distribution actions, which is exactly what you want when bootstrapping from suboptimal historical behavior. However, the penalty coefficient (alpha in CQL) must be tuned:
   - Too high: agent never deviates from historical behavior (defeats the purpose of RL).
   - Too low: agent ignores historical data (defeats the purpose of CQL bootstrap).
   - **Recommendation**: Start with alpha=5.0 (CQL paper default), decay to 1.0 over first 100 online PPO updates.

3. **Buffer size**: For statistical significance, need minimum 10,000 transitions. At 60s loop intervals, 6.5 hours of market time per day, ~20 trading days/month: that is ~7,800 transitions/month per asset. With 20 assets: ~156,000 transitions/month. **3 months of paper trading data is the minimum viable buffer.**

### Recommendation

```
Phase 1 (Months 1-3): Run existing rule-based + LLM system, log ALL decisions to TradeDecisionLog
Phase 2 (Month 4): Extract (state, action, reward, next_state) tuples from logs
Phase 3 (Month 4): Train CQL offline policy
Phase 4 (Month 5): Begin online PPO fine-tuning with CQL-initialized weights
```

This is slower than the plan suggests but statistically sound. Rushing to CQL with synthetic actions from OHLCV replay introduces systematic bias from the heuristic used to generate actions.

## 6. Execution Algorithms: AGREE -- Descope Almgren-Chriss

### Assessment

The plan proposes: TWAP, VWAP, Implementation Shortfall (Almgren-Chriss), POV, and RL execution strategies.

**Current codebase already has solid TWAP and VWAP implementations** (verified in `twap_strategy.py` and `vwap_strategy.py`):
- TWAP: 12 slices, randomized timing/size, catch-up mechanism, slippage tracking vs TWAP benchmark
- VWAP: Historical volume profile matching, participation rate limiting, urgency adjustment

**Almgren-Chriss Implementation Shortfall is overkill for a personal account.**

The Almgren-Chriss model solves for the optimal liquidation trajectory that minimizes:
```
E[cost] + lambda * Var[cost]
```
where cost includes temporary market impact (proportional to trade rate) and permanent market impact (proportional to total volume).

**For retail-size orders (typically $500-$50,000), market impact is negligible:**
- A $10,000 order in AAPL (ADV ~$8B) represents 0.000125% of daily volume.
- Temporary market impact at this size: ~0.01 bps. Unmeasurable.
- The Almgren-Chriss optimal trajectory degenerates to "execute immediately" for orders below ~0.1% of ADV.

**Recommendation:**
- Keep TWAP and VWAP (already implemented, well-tested).
- Add POV (Percentage of Volume) -- simple and useful for avoiding market impact on less liquid names.
- **Defer Almgren-Chriss and RL execution** to a future phase when account size exceeds $500K or when trading illiquid small-caps where impact matters.
- The existing fallback chain (RL -> VWAP -> TWAP) should be simplified to: VWAP -> TWAP -> Market.

### Cost-Benefit

| Execution Algo | Implementation Effort | Alpha Added (retail size) | Verdict |
|---------------|----------------------|--------------------------|---------|
| TWAP | DONE | ~1-2 bps | Keep |
| VWAP | DONE | ~2-5 bps | Keep |
| POV | 2 days | ~1-3 bps on illiquid names | Add |
| Almgren-Chriss | 2 weeks | ~0.01 bps | Defer |
| RL Execution | 4 weeks | Unknown, high variance | Defer |

## 7. Walk-Forward + CPCV + Deflated Sharpe: AGREE -- Methodology is Sound

### Assessment

This is the strongest section of the plan. The backtesting methodology is institutionally rigorous.

**Walk-forward (252d train / 63d test / 5d embargo):**
- 252 trading days (~1 year) training window is standard.
- 63 trading days (~3 months) test window provides sufficient out-of-sample data.
- 5-day embargo prevents information leakage from autocorrelation in financial returns. This is correct -- many retail backtesting frameworks skip this.
- **Minor suggestion**: Consider extending embargo to 10 days for strategies using weekly features (SMA20, etc.) to account for feature look-ahead contamination.

**CPCV (Combinatorial Purged Cross-Validation):**
- 10-fold CPCV with purging is the gold standard (de Prado, "Advances in Financial Machine Learning").
- Correctly addresses the non-IID nature of financial time series.
- **Implementation note**: CPCV generates C(10,2) = 45 train/test combinations. Each requires a full training run. On a single L4 GPU, this is ~45 * 2h = 90h = ~4 days. Budget this into the training pipeline.

**Deflated Sharpe Ratio:**
- Correctly accounts for multiple testing bias (the "strategy zoo" problem).
- DSR adjusts for: number of strategies tested, skewness of returns, kurtosis of returns, and estimated Sharpe ratio.
- **Critical addition**: Log EVERY strategy variant tested (including failed ones) to compute the deflation factor accurately. If you test 50 parameter combinations but only report the best, the deflated Sharpe will be optimistic.

### One Gap: Regime-Conditional Evaluation

The plan lists stress tests (2022 bear, COVID, SVB) but does not specify regime-conditional performance reporting.

**Recommendation:** Report Sharpe ratio, max drawdown, and win rate separately for each detected regime (trending up, trending down, ranging, volatile). A strategy with overall Sharpe 1.5 that has Sharpe -0.3 in bear markets is not robust -- it is a bull market momentum strategy masquerading as alpha.

## 8. Options Trading: DISAGREE -- Descope from Phase 1

### Assessment

The plan includes: Greeks tracking, 1-DTE auto-close, multi-leg orders, implied volatility surface modeling.

**This is too ambitious for initial deployment and introduces disproportionate risk:**

1. **Options have non-linear payoffs** that PPO/SAC are not inherently designed for. The continuous action space (target weight vector [-1,1]) maps cleanly to equity positions but poorly to options positions where:
   - Delta changes non-linearly with underlying price (gamma risk)
   - Time decay is non-linear (theta accelerates near expiration)
   - Implied volatility shifts can dominate PnL (vega risk)

2. **1-DTE auto-close** requires millisecond-precision execution at market close. Cloud Run with 60s polling intervals cannot reliably close positions within the last minute of trading. A stuck position going through expiration could result in unexpected assignment on ITM options.

3. **Multi-leg orders** (spreads, condors, butterflies) through Alpaca require specific API handling. Alpaca's options support launched in 2024 and has known limitations with complex orders.

4. **Regulatory risk**: Automated options trading on a personal account triggers additional FINRA scrutiny. Pattern day trading rules apply differently to options.

### Recommendation

- **Phase 1**: Equities only. Get the DRL pipeline production-stable with proven backtesting.
- **Phase 2** (3-6 months later): Add covered calls and cash-secured puts (defined risk, simple Greeks).
- **Phase 3** (6-12 months later): Multi-leg strategies with dedicated Greeks engine.

If options must be included, start with a **rules-based options overlay** (not DRL):
- Covered calls on existing long positions when IV rank > 50%
- Auto-close any option position 2 hours before expiration (not 1-DTE)
- Maximum 10% of portfolio in options positions
- No naked short options ever

## 9. LLM Signal Injection into DRL Observation Space: AGREE WITH MODIFICATIONS

### The Proposal

Project the LLM strategy brief (structured JSON: strategy_bias, risk_appetite, sector_preferences, key_risks, recommended_leverage_cap) into a fixed-dimension vector and inject it into the DRL observation space.

### Assessment

**Architecturally sound but the dimensionality mapping needs specification.**

The LLM outputs are heterogeneous:
- `strategy_bias`: categorical (bullish/bearish/neutral) -- 3-dim one-hot
- `risk_appetite`: continuous [0, 1] -- 1-dim
- `sector_preferences`: variable-length list of sectors -- needs fixed-dim encoding
- `key_risks`: free-text list -- needs embedding
- `recommended_leverage_cap`: continuous [0, 2] -- 1-dim

### Recommended Architecture

```
LLM JSON Output
    |
    v
Fixed Encoder (deterministic, no learning)
    |
    +-- strategy_bias: one-hot [3]
    +-- risk_appetite: scalar [1]
    +-- sector_preferences: binary vector [11] (GICS sectors)
    +-- key_risks: TF-IDF over fixed vocabulary [20] (pre-defined risk categories)
    +-- leverage_cap: scalar [1]
    |
    v
LLM Signal Vector: [36] dimensions
    |
    v
Concatenate with market features + portfolio state
    |
    v
DRL Observation Space: [market_dim + portfolio_dim + 36]
```

**Critical safeguards:**

1. **Stale signal handling**: If vLLM is down (VM stopped on weekends, or timeout), the LLM signal vector must default to a neutral vector (zeros or a learned baseline). The DRL policy must be trained with randomly masked LLM signals (dropout rate 20-30%) so it does not become dependent on LLM availability.

2. **Signal frequency mismatch**: LLM signals are generated max 10/day (per the plan). The DRL loop runs every 60s. The LLM vector is constant for ~40 consecutive observations. This is fine -- it acts as a slowly-varying context signal, similar to how regime detection works.

3. **Gradient isolation**: Do NOT backpropagate DRL gradients through the LLM encoder. The LLM signal is a feature, not a learnable module. Treat it as a fixed input.

4. **Existing integration pattern**: The codebase already injects EFT (elson-finance-14b) signals into the auto-trading loop (lines 512-553 of `auto_trading_service.py`). The proposed architecture extends this pattern into the DRL observation space. This is a natural evolution, not a redesign.

## 10. Overall Viability: AGREE -- With Right-Sizing

### Can a retail personal account realistically benefit from this level of quantitative infrastructure?

**Yes, but only if right-sized.**

### What provides measurable alpha at retail scale:

| Component | Expected Alpha | Confidence | Cost to Implement |
|-----------|---------------|------------|-------------------|
| Regime-aware strategy selection | 50-150 bps/year | High | Medium |
| LLM sentiment + news conditioned risk | 30-100 bps/year | Medium | Low (already built) |
| VWAP/TWAP execution | 5-20 bps/year | High | Done |
| DRL portfolio allocation (PPO) | 0-200 bps/year | Low-Medium | High |
| Multi-timeframe feature engineering | 20-80 bps/year | Medium | Medium |
| CQL offline bootstrap | 10-50 bps/year vs random init | Medium | Medium |

### What does NOT provide measurable alpha at retail scale:

| Component | Expected Alpha | Why |
|-----------|---------------|-----|
| Almgren-Chriss execution | ~0.01 bps | Order sizes too small for market impact |
| Microstructure features (OFI) | ~1-5 bps | Requires L2 data, 60s granularity too slow |
| RL execution agent (SAC) | Negligible | Same reason as Almgren-Chriss |
| Options DRL | Negative (risk) | Non-linear payoffs, gamma risk, assignment risk |

### Realistic Performance Targets

Based on academic literature and industry benchmarks for DRL trading systems:

- **Before fees**: Sharpe 0.8-1.5 is realistic for a well-tuned DRL system on US equities.
- **After fees** (Alpaca: $0 commissions, ~1-3 bps spread cost): Sharpe 0.6-1.3.
- **After slippage** (60s polling = ~5-10 bps slippage on momentum signals): Sharpe 0.4-1.1.
- **Comparison**: Buy-and-hold S&P 500 Sharpe ~0.4 over 2015-2025.

A realistic target: **Sharpe ratio 0.8 after all costs, with max drawdown < 15%, over a 1-year out-of-sample period.**

This would be a genuinely useful system -- beating buy-and-hold on a risk-adjusted basis while maintaining strict drawdown control. Do not promise Sharpe > 2.0 -- that is hedge fund territory requiring co-located infrastructure and proprietary data.
