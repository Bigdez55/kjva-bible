## EXECUTIVE SUMMARY

The proposed plan is architecturally sound in concept but **critically over-scoped for a solo developer maintaining a live production platform**. After deep-diving the codebase, I found that **60-70% of the proposed infrastructure already exists** in various forms across the `trading_engine/` subtree. The plan, as written, would create a parallel system that duplicates existing services, introduces 8 new database tables (migration risk), adds 4 heavy Python packages to the Docker image (tensorflow, torch already stripped from `requirements-docker.txt` for a reason), and requires 30 days of paper validation before any user-facing value is delivered.

**My recommended restructuring reduces 8 phases to 4 milestones, delivers an MVP at milestone 2 (not milestone 8), and reuses existing infrastructure rather than rebuilding it.**

## SECTION 2: DEPENDENCY CHAIN ANALYSIS

### Proposed Chain
```
Bot-0 --> Bot-1 --> Bot-2 --> Bot-2/Bot-3 (parallel) --> Bot-4 --> Bot-5 --> Bot-6 --> Bot-7
```

### Problems with This Chain

1. **Bot-0 is over-indexed on schema.** 8 new tables upfront is the exact anti-pattern that caused the 2026-02-15 outage. Schema changes should be incremental -- add tables only when the service that needs them is being built.

2. **Bot-1 (TradingBotService) creates dual control planes.** `AutoTradingService` is live. Creating `TradingBotService` alongside it means two services can issue trades for the same user. This is a P0 risk -- duplicate orders, conflicting state, split position tracking. The plan does not specify migration/replacement strategy.

3. **Bot-2 and Bot-3 are NOT truly parallelizable.** Bot-3 (DRL) consumes features from Bot-2 (Feature Engineering). You cannot train a PPO agent without the feature pipeline producing the observation space. The proposed parallel execution only works if Bot-2's API contract is frozen before Bot-3 starts -- which means Bot-2 must be COMPLETE, not parallel.

4. **Bot-4 (Risk Engine) is placed too late.** Risk guardrails should be the FIRST thing wired into the bot, not phase 4 of 8. A bot without a kill switch executing trades (even in paper) is a compliance and safety gap.

5. **Bot-6 (LLM Strategy) depends on L4 VM availability.** The L4 scheduler is PAUSED. This phase cannot proceed without the vLLM infrastructure being stable. This is an external dependency not tracked in the plan.

6. **Bot-7 (Dashboard) has no incremental delivery.** The user sees nothing until phase 7. No feedback loop for 3+ months.

### Recommended Dependency Chain (Restructured)

```
Milestone 1 (Foundation):   Schema(incremental) + Extend AutoTradingService + Kill Switch API
Milestone 2 (MVP Bot):      Feature Store + LLM Signal Injection + Dashboard v1
Milestone 3 (Intelligence): DRL Agents (PPO/SAC) + Training Pipeline
Milestone 4 (Validation):   Paper Validation Service + Performance Dashboard
```

## SECTION 3: MVP BREAKPOINT RECOMMENDATION

### The Natural MVP is at Milestone 2

**What Milestone 2 delivers:**
- Enhanced `AutoTradingService` with feature store (persistent indicators)
- LLM signals injected into the strategy loop (elson-finance-14b already live)
- Kill switch API endpoint for emergency halt
- Basic dashboard showing bot status, active strategies, recent signals
- All built on existing infrastructure -- no parallel control plane

**What Milestone 2 does NOT require:**
- Deep RL (PPO/SAC/CQL) -- the existing rule-based + EFT signal loop is already trading
- 8 new database tables -- needs 2-3 tables max (feature_store, bot_signals, kill_switch_log)
- tensorflow/torch in Docker image -- these are EXCLUDED from `requirements-docker.txt` for a reason (image size, build time, cold start)

**User value at MVP:** The bot already trades autonomously. The MVP makes it trade *smarter* (LLM signal injection) and *safer* (kill switch) with *visibility* (dashboard). This is 80% of the user value.

## SECTION 5: SERVICE CONFLICT RESOLUTION

### AutoTradingService vs. TradingBotService

**Current `AutoTradingService` (1,211 lines):**
- Singleton with `start_auto_trading()`, `stop_auto_trading()`
- Uses `StrategyRegistry` to instantiate strategies by name
- Uses `TradeExecutor` for signal execution
- Has market scanner for symbol universe rotation
- Integrated with `eft_generate()` for LLM signals
- Logs decisions to `TradeDecisionLog`
- **LIVE IN PRODUCTION**

**Proposed `TradingBotService`:**
- New orchestrator with 8 endpoints
- Manages bot lifecycle, configuration, strategies
- Presumably replaces AutoTradingService

**VERDICT:** Creating a parallel `TradingBotService` is the Second-System Effect in action. The existing service works. The right approach is:

1. **Extend** `AutoTradingService` with the missing capabilities (kill switch, feature store, enhanced dashboard data)
2. **Add** a `/bot/` API router that provides richer endpoints while delegating to `AutoTradingService`
3. **Deprecate** any endpoints that the new router supersedes
4. **NEVER** have two services both capable of issuing trades simultaneously

## SECTION 7: WHAT'S MISSING FROM THE ORIGINAL PLAN

1. **Alerting / Notification integration** -- The bot should notify the user (email, push) on: kill switch trigger, daily P&L summary, unusual activity. `NotificationService` exists but is not wired into the bot plan.

2. **Observability** -- No mention of structured logging, metrics (Prometheus/custom), or tracing for the bot loop. The existing `structlog` + `record_metric` patterns should be extended, not ignored.

3. **Graceful degradation** -- What happens when vLLM is down? When Alpaca API is degraded? The plan assumes happy path. Each service needs a fallback behavior defined.

4. **Rate limiting on bot endpoints** -- The `/bot/` router needs rate limiting, especially `/bot/kill-switch` (prevent accidental rapid toggles) and `/bot/start` (prevent duplicate starts).

5. **Multi-user isolation** -- `AutoTradingService` uses class-level dicts (`_running_tasks`, `_active_portfolios`). The plan adds more complexity to a singleton. Need to verify memory isolation between users under concurrent load.

6. **CQL (offline RL) feasibility check** -- The plan proposes CQL but does not verify that sufficient historical trade data exists. CQL requires 10K+ transitions minimum. `trade_decision_log` may not have this volume yet. This should be validated before committing to CQL implementation.

7. **Rollback strategy** -- If a milestone introduces a regression, how do we rollback? Each milestone should be behind a feature flag or config toggle, not hardcoded.

## SECTION 8: ANSWERS TO SPECIFIC REVIEW QUESTIONS

### Q1: Is 22 steps / 8 phases realistic for a solo developer?
**NO.** At current velocity (3-4 productive dev days/week), the original plan is 14-19 weeks with no user-facing value until week 14+. The restructured 4-milestone plan delivers MVP value at week 5 and completes at week 11-12.

### Q2: Is the dependency chain optimal?
**NO.** Risk controls (kill switch) should come FIRST, not phase 4. Schema should be incremental, not 8 tables upfront. See Section 2 for restructured chain.

### Q3: Where is the natural MVP break point?
**Milestone 2** -- after feature store + LLM signal injection + dashboard v1. This delivers 80% of user value at 40% of effort.

### Q4: Highest-risk phases?
1. **Bot-3 (DRL)** -- highest technical complexity, longest dev time, dependency on L4 VM, Docker image constraints
2. **Bot-0 (Schema)** -- 8 tables at once is the exact pattern that caused the 2026-02-15 outage
3. **Bot-1 (TradingBotService)** -- dual control plane risk with AutoTradingService

### Q5: AutoTradingService conflict resolution?
**EXTEND, don't replace.** See Section 5. A new `/bot/` router delegates to the existing service. Never have two trade-issuing services.

### Q6: When does integration testing occur?
**At every milestone boundary.** The original plan's approach of testing only at Bot-7 is unacceptable. Each milestone has an integration test step.

### Q7: Production deployment strategy?
**Incremental per milestone.** Milestone 1 deploys immediately (safety feature). Milestone 2 deploys (MVP). Milestones 3-4 deploy incrementally. Each preceded by the standard pre-deploy checklist (schema drift check, requirements sync, Secret Manager audit, health check verification).

### Q8: Should P1 blockers be resolved first?
**YES, unconditionally.** The JSON parse and EFT timeout issues directly affect the LLM signal injection (Milestone 2). Fixing them first (2-3 days) prevents cascading delays. This is the pre-sprint.

### Q9: Burnout risk?
**HIGH in the original plan.** 12 weeks of continuous new development with zero value delivery until week 12+. The restructured plan delivers value at week 5, which provides psychological momentum and stakeholder confidence. One-week cool-down between milestones is mandatory.

### Q10: What's missing?
See Section 7. Alerting, observability, graceful degradation, rate limiting, multi-user isolation, CQL feasibility check, and rollback strategy.

## VERDICT

**AGREE WITH MAJOR MODIFICATIONS**

The destination is correct. The route is wrong. The proposed plan builds a parallel universe of services that duplicates 60-70% of existing infrastructure, deploys 8 tables in a single migration (high outage risk), creates a dual control plane (P0 trade safety risk), and delivers zero user value for 3.5+ months.

- Reuses existing `AutoTradingService`, `FeatureEngineering`, `CircuitBreaker`, `BacktestingEngine`, and `TradingEnvironment`

- Includes integration testing at every milestone (not just the end)
