# Autonomous Trading Bot Plan -- Master Orchestrator Validation

## Executive Summary

The plan at `memoized-frolicking-tower.md` is **ambitious, well-structured, and directionally correct**, but contains **3 FAIL items, 5 WARN items**, and requires significant scheduling adjustments for a single-developer team. The raw timeline of 66-67 dev days (~13 weeks) is **underestimated by 40-60%** when accounting for integration overhead, testing, and the single-developer constraint.

**Revised realistic estimate: 18-22 weeks (with 20% buffer applied: 22-26 weeks)**

### 3. CRITICAL PATH ANALYSIS

The longest dependency chain determines the project's minimum duration:

```
Pre-Sprint (4d) --> M1 (12d) --> M2 (14d) --> M3 (20d) --> M4 (10d) --> Paper (30d)
                                                                          Total: 90 dev days + 30 calendar days
```

**Critical path: 90 dev days = 18 calendar weeks + 6 weeks paper = 24 weeks minimum**

Everything else (M5, M6) can theoretically run in parallel but NOT for a single developer. The critical path is:

```
Pre-Sprint --> M1 --> M2 --> M3 --> M4 --> Paper Validation --> Live
```

M5 and M6 are off the critical path but compete for the same developer's time.

**Bottleneck identification:**
- M3 (DRL) is the single largest block at 16-20 days realistic. If DRL training doesn't converge, the entire M4 validation is meaningless.
- M5 (UI) is the second largest at 20-25 days but is off the critical path.

### 5. TOP 3 RISK ITEMS

**RISK 1: DRL Model Convergence (M3) -- Severity: CRITICAL**

- IQL/d3rlpy on financial data has a high failure rate. The offline RL paradigm requires a replay buffer with sufficient coverage of the state-action space.
- The plan assumes `TradeDecisionLog` has enough data. Current production has 3,559 decisions and 448 AI decisions (from MEMORY.md). For a 46-dim feature space, this is **catastrophically small**. Academic papers on offline RL in trading use 50,000-500,000 trajectories minimum.
- **Mitigation:** Start with a simpler model (linear factor model, or gradient boosting on the 46-dim features) as M3a. DRL becomes M3b, conditional on data accumulation. This prevents the entire M4 validation from being blocked.
- **If DRL fails:** The MVP bot from M2 (rule-based + LLM signals) is still a viable product. The plan should explicitly state this fallback path.

**RISK 2: Schema Drift During Rapid Iteration (M1, M4, M6) -- Severity: HIGH**

- The plan creates 7+ new models across 3 milestones: `bot_performance`, `bot_risk_event`, `bot_audit_log` (M1), `OptionsContract`, `OptionsPosition`, `OptionsOrder` (M6), plus Trade model extensions (M4).
- Our documented production outage on 2026-02-15 was caused by exactly this pattern: new columns deployed before ALTER TABLE.
- Alembic already has 17 migrations with merge heads. Rapid model additions WILL create merge conflicts.
- **Mitigation:** Consolidate all schema changes into a single "schema sprint" at the start of each milestone. Run `alembic upgrade head` on a Cloud SQL clone BEFORE deploying code. Never deploy code and schema in the same Cloud Build step.

**RISK 3: M5 Scope Explosion (Apex UI) -- Severity: HIGH**

- M5 specifies **25+ new frontend components** including a TradingView integration, react-mosaic workspace, DOM visualization, order blotter with WebSocket fills, screener, research panel, and 4 preset layouts.
- This is equivalent to building a second application. For reference, the entire existing frontend is ~150 files after the dead code cleanup.
- The CRA/webpack-dev-server conflict (documented in MEMORY.md) means hot-reload is broken -- every UI change requires a production build to test.
- **Mitigation:** Apply strict MoSCoW to M5:
  - **Must:** TradingView chart, basic order entry, positions panel (5 components)
  - **Should:** Order blotter, watchlist, bot dashboard panel (3 components)
  - **Could:** DOM, screener, research panel, risk dashboard (4 components)
  - **Won't (defer to M7):** Crypto dashboard, volatility surface, preset layouts, workspace save/load

## ADDITIONAL FINDINGS

### Missing from the plan:

1. **No rollback strategy.** If M1's kill switch has a bug and triggers incorrectly in production, what's the rollback? Cloud Run revision rollback? The plan should specify Cloud Run revision pinning for every deploy.

2. **No monitoring/alerting.** The plan doesn't mention Cloud Monitoring dashboards, PagerDuty/alerting for circuit breaker trips, or Sentry error rate thresholds during paper validation. A bot running autonomously NEEDS alerting.

3. **No `models/__init__.py` update checklist.** We have a documented P0 pattern: every new model MUST be imported in `backend/app/models/__init__.py` and added to `__all__`. M1 adds 3 models, M6 adds 3 models. This is 6 opportunities to repeat the 2026-02-21 bug.

4. **`ta-lib` installation.** M2 specifies `ta-lib` for technical indicators. `ta-lib` is a C library requiring system-level installation (`brew install ta-lib` on macOS, `apt-get install libta-lib-dev` on Linux). The Dockerfile needs updating. This is a common 2-4 hour yak-shave.

5. **ONNX model size in Cloud Run.** M3 specifies ONNX for inference on Cloud Run (2Gi memory). An IQL model with 46-dim input is small (~5-10MB), but the plan should specify the model loading strategy (load on startup? lazy load? GCS download?). Cold starts with GCS download could exceed Cloud Run's 300s timeout.

6. **Alpaca options API access.** The plan assumes Alpaca provides options trading API. Alpaca options trading is available but requires a separate approval level. The plan notes `options_approved_level` is already read -- but has the account actually been approved for options? If not, M6 is blocked by a business process, not code.

7. **Data API costs.** The plan lists Polygon, Glassnode, Coingecko at $30/month total. Polygon's starter plan is $29/month alone. Glassnode API starts at $29/month. Coingecko pro is $129/month. Realistic data API cost: $100-200/month, not $30.
