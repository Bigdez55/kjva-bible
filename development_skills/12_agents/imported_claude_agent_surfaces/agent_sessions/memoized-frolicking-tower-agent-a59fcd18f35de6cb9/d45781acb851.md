### ITEM 1 -- CRITICAL: DRLStrategy interface mismatch with StrategyRegistry

**FAIL**

- **What**: The plan proposes a `DRLStrategy` that outputs continuous portfolio weights in the range `[-1, 1]` (short-to-long allocation per asset). The existing `TradingStrategy` base class and `StrategyRegistry.create()` interface expects `generate_signal()` to return a discrete signal dict: `{'action': 'buy'|'sell'|'hold', 'confidence': float, 'price': float}`.
- **Where**: `backend/app/trading_engine/strategies/base.py:56-76` -- the abstract `generate_signal` method contract. Also `backend/app/services/auto_trading_service.py:518-523` -- the consumer that reads `signal.get("action")` and `signal.get("confidence")`.
- **Why**: A DRL model returning a weight vector like `{"AAPL": 0.35, "TSLA": -0.12, "SPY": 0.53}` will be passed into `auto_trading_service._process_strategy()` which checks `signal.get("action") in {"buy", "sell"}` and `float(signal.get("confidence", 0.0)) >= 0.6`. The DRL output has **no `action` key** and **no `confidence` key**. This is a guaranteed runtime `NoneType` crash or silent HOLD-forever behavior.
- **How**: Two options:
  1. **Adapter pattern** (recommended): Create a `DRLSignalAdapter` that wraps the DRL weight output and translates it per-symbol into the existing signal dict format. For example, `weight > threshold` maps to BUY, `weight < -threshold` maps to SELL, `abs(weight)` maps to confidence. This keeps the StrategyRegistry contract intact.
  2. **Dual interface** (more invasive): Extend `TradingStrategy` with an optional `generate_portfolio_weights()` method and add a branching path in `_process_strategy()`. This is riskier because it forces changes to a battle-tested hot loop.

```python
# Option 1: Adapter pattern (recommended)
class DRLSignalAdapter:
    """Translates continuous DRL weights into discrete TradingStrategy signals."""

    def __init__(self, drl_model, buy_threshold: float = 0.15, sell_threshold: float = -0.15):
        self.drl_model = drl_model
        self.buy_threshold = buy_threshold
        self.sell_threshold = sell_threshold

    async def generate_signal(self, market_data: dict) -> dict:
        weights = self.drl_model.predict(market_data)  # e.g. {"AAPL": 0.35}
        symbol = market_data.get("symbol", "")
        weight = weights.get(symbol, 0.0)

        if weight >= self.buy_threshold:
            action = "buy"
        elif weight <= self.sell_threshold:
            action = "sell"
        else:
            action = "hold"

        return {
            "action": action,
            "confidence": min(abs(weight), 1.0),
            "price": float(market_data.get("price", 0)),
            "reason": f"DRL weight={weight:.4f}",
        }
```

### ITEM 2 -- HIGH: Feature engineering service duplicates existing code

**FAIL**

- **What**: The plan proposes a new `feature_engineering_service.py` for computing technical indicators and ML features. There are **already two files** doing this exact work:
  1. `backend/app/trading_engine/data/feature_engineering.py` -- a full `FeatureEngineering` class (336 lines) that computes SMA, EMA, RSI, MACD, Bollinger Bands, Stochastic, ATR, OBV, momentum, volatility, candlestick patterns, and future returns. It also has `SentimentFeatures`, `DataCombiner`, and `prepare_ml_features()` for ML train/test splits.
  2. `backend/app/services/market_data_processor.py` -- a `MarketDataProcessor` class (656 lines) that computes `_calculate_technical_indicators()` (SMA 20/50, EMA 12/26, MACD, RSI, Bollinger Bands, volume ratio), `detect_market_regime()`, `calculate_volatility_metrics()`, and `calculate_correlation_matrix()`.
  3. `backend/app/trading_engine/strategies/moving_average.py:182-219` -- inline `_calculate_indicators()` that also computes SMA, RSI, volume ratio, and MACD.
- **Where**: Proposed file vs. existing `backend/app/trading_engine/data/feature_engineering.py` (lines 16-336) and `backend/app/services/market_data_processor.py` (lines 62-310).
- **Why**: Creating a third implementation of RSI/MACD/Bollinger will produce **three sources of truth** for technical indicator computation. When one is patched (e.g., RSI period change), the others silently drift. This is maintenance debt that compounds exponentially.
- **How**: Do NOT create a new feature engineering service. Instead:
  1. Use the existing `FeatureEngineering` class from `trading_engine/data/feature_engineering.py` as the canonical computation engine.
  2. If the DRL model needs additional features not currently computed (e.g., specific lookback windows, orderbook features), **extend** the existing `FeatureEngineering.generate_features()` method rather than creating a parallel service.
  3. Consider deprecating the inline indicator calculations in `moving_average.py:182-219` and `market_data_processor.py:235-286` to point to the canonical class.

### ITEM 7 -- HIGH: Alembic migration granularity + schema drift risk

**FAIL**

- **What**: The plan proposes 2 migration files. Given the documented **production outage on 2026-02-15** caused by schema drift (new columns added without ALTER TABLE), this needs CRITICAL-level scrutiny.
- **Where**: `backend/alembic/versions/` currently has 16 migration files. The most recent is `add_auto_trading_sessions_2026_02_24.py`.
- **Why**: The MEMORY.md documents that "`create_all()` does NOT add columns to existing tables" and that "Adding columns to models WITHOUT running ALTER TABLE = instant production crash." Two migrations for a sprint that may touch 3+ new tables plus ALTER columns on existing tables is **underspecified**. Each new table should be its own migration. Each ALTER TABLE should be its own migration. This ensures granular rollback capability.
- **How**:
  1. If 3 new tables are proposed: 3 separate migration files (one per `CREATE TABLE`).
  2. If any existing tables get new columns: 1 migration per ALTER TABLE.
  3. **MANDATORY**: Before every deploy, run `git diff HEAD~N -- backend/app/models/ | grep "Column("` and verify every new column has a corresponding ALTER TABLE in a migration.
  4. **MANDATORY**: Test migrations against a Cloud SQL snapshot before production deploy.
