## 1. FEATURE ENGINEERING: 46-dim Equity / 56-dim Crypto Observation Space

### Plan Specification
- 20 OHLCV (ta-lib multi-timeframe)
- 4 sentiment (elson-finance-14b via EFT)
- 6 regime (VolatilityDetector + ADX + VIX)
- 8 portfolio state
- 8 LLM signal
- Crypto extras: 3 on-chain (active addresses, tx volume, whale signal), 2 funding rate, 4 crypto regime one-hot, 1 BTC correlation
- Sentiment residual orthogonalized against recent return

### Existing Code State
The current `backend/app/trading_engine/data/feature_engineering.py` (`FeatureEngineering` class) generates approximately **50+ raw features** from OHLCV data (5 SMA windows, 5 EMA windows, 5 MA strength, 6 Bollinger, 3 RSI, 5 MACD, 6 Stochastic, 2 ATR, 2 OBV, 3 ROC + MFI, 4 volatility, 5 candlestick, 9 returns). However, it does NOT:
- Select/reduce to a fixed 46-dimensional vector
- Include sentiment features (the `SentimentFeatures` class uses `np.random.normal` — placeholder only, not production-ready)
- Include regime features from `VolatilityDetector`
- Include portfolio state or LLM signal dimensions
- Include any crypto-specific features

### Quantitative Assessment

**Equity 46-dim breakdown validation:**

| Block | Dims | Assessment |
|-------|------|------------|
| OHLCV multi-timeframe | 20 | Reasonable. Current code produces ~50 OHLCV-derived features. Reducing to 20 via PCA or manual selection is feasible. However, the plan says "ta-lib multi-timeframe" but existing code uses pandas rolling — no ta-lib dependency exists. **Minor inconsistency.** |
| Sentiment | 4 | Methodologically sound IF orthogonalized. The plan correctly calls for sentiment residual orthogonalization against recent return. This is critical — raw sentiment scores are highly correlated with 1-5d returns (r=0.3-0.5), creating multicollinearity. Orthogonalization via Gram-Schmidt or OLS residual is the correct approach. **Approved.** |
| Regime | 6 | The existing `VolatilityDetector` produces a single regime enum + blended volatility float. Adding ADX (1 dim) and VIX (1 dim) + one-hot encoding of 4 regime states = 6 dims. **Correct.** |
| Portfolio state | 8 | Not specified in detail. Typical: cash_pct, gross_exposure, net_exposure, position_count, unrealized_pnl_pct, drawdown_from_peak, time_in_position, avg_holding_period. **Reasonable but must be documented.** |
| LLM signal | 8 | Risky dimension. LLM signals from elson-finance-14b are non-deterministic. 8 dims is high — what exactly? action_one_hot(3), confidence(1), reasoning_embedding(4)? **Underdefined. Needs specification.** |

**Total: 20+4+6+8+8 = 46. Arithmetic checks out.**

**Crypto 56-dim (46 + 10) breakdown:**

| Block | Dims | Assessment |
|-------|------|------------|
| On-chain: active addresses, tx volume, whale signal | 3 | Methodologically sound for BTC/ETH. However, Alpaca does not provide on-chain data. Plan specifies Glassnode (primary) and Coingecko (fallback). Glassnode free tier has 24h delay — stale for intraday. **Data freshness risk.** |
| Funding rate | 2 | Plan says Coingecko primary, Bybit fallback. Alpaca does not offer perpetual futures so funding rate is external-only. 2 dims (current rate + 8h MA) is standard. **Approved but Alpaca has no perps — funding rate is informational only, not directly tradable on this platform.** |
| Crypto regime one-hot | 4 | 4 states (low/normal/high/extreme) one-hot. Current `VolatilityDetector` thresholds are equity-calibrated (15/25/40%). The plan specifies crypto thresholds (15/40/60%). **These are separate thresholds — the plan correctly identifies this.** |
| BTC correlation | 1 | Rolling correlation of altcoin to BTC. Standard approach. **Approved.** |

**Total: 46+10 = 56. Arithmetic checks out.**

### Critical Gaps

1. **`SentimentFeatures` class is a placeholder** — uses `np.random.normal()`. Must be replaced with actual elson-finance-14b inference. The plan accounts for this (Milestone 2 creates `feature_engineering_service.py` as NEW file, not modifying existing).

2. **No dimensionality reduction pipeline specified.** The existing `FeatureEngineering` produces 50+ raw features. The plan says "20 OHLCV" but does not specify the selection/reduction method. PCA? Manual selection? L1 regularization? This matters because:
   - PPO is sensitive to input dimensionality (curse of dimensionality in policy gradient)
   - Random forests in the existing ensemble are not — different requirements
   - Recommendation: Use mutual information ranking or SHAP feature importance from the existing ensemble to select the 20 most informative OHLCV features.

3. **LLM signal dimensions (8) are underdefined.** If these are raw text embeddings, 8 dims is too few to capture semantic meaning (typical sentence embeddings are 384-768 dim). If these are structured outputs (action, confidence, etc.), 8 is too many and introduces noise. **Must be specified before implementation.**

### Verdict: CONDITIONAL PASS

Pass if: (a) LLM signal 8-dim structure is formally specified, (b) feature selection method for the 20 OHLCV dims is documented, (c) sentiment orthogonalization is implemented before DRL training.

## 2. DRL DESIGN: PPO + SAC + IQL (d3rlpy)

### Plan Specification
- PPO + SAC for online training on H100 VM
- d3rlpy IQL (not CQL) for offline RL from TradeDecisionLog replay buffer
- ONNX export for Cloud Run inference (no PyTorch in prod)
- Beta-hedged reward function
- Continuous action space [-1, 1]

### Existing Code State
The existing `reinforcement_learning.py` in `trading_engine/ml_models/ai_model_engine/` implements:
- A `TradingEnvironment` with **Discrete** action space (strategy selection, not position sizing)
- TensorFlow-based DQN with experience replay
- 15-dim observation space (10 market + 5 strategy metrics)
- No beta-hedging, no ONNX export

The plan proposes a **completely different architecture** (new `trading_env.py` and `reinforcement_learning.py` in `services/`), which is the correct approach — the existing RL module is a strategy-selector, not a position-sizing agent.

### Quantitative Assessment

**PPO + SAC combination:**
- PPO (on-policy, clipped surrogate): Good for continuous action spaces with bounded actions [-1,1]. Stable training, sample-inefficient. Appropriate for H100 training where compute is cheap but data is limited (~3,500 decisions in the log currently).
- SAC (off-policy, max entropy): Better sample efficiency than PPO, automatic temperature tuning. The max-entropy objective provides built-in exploration which is valuable for non-stationary financial environments.
- **Running both and ensembling/selecting** is standard practice (e.g., FinRL framework). Typically you train both, run walk-forward validation, and deploy the better one per regime. **Approved.**

**IQL (Implicit Q-Learning) via d3rlpy:**
- IQL is superior to CQL for this use case. The plan correctly identifies why: CQL requires (s, a, r, s') tuples with properly logged behavior policy. The existing `TradeDecisionLog` does NOT log the behavior policy's action probabilities — only the chosen action. IQL avoids this requirement by using expectile regression.
- d3rlpy is the correct library (not stable-baselines3 which is online-only).
- **Key concern:** IQL performance degrades significantly when the dataset has low action diversity. With 3,559 decisions (448 AI, ~3,111 rule-based), and the rule-based system having limited action variety (mostly HOLD), the effective training signal is thin. **Minimum 5,000 diverse (BUY/SELL/HOLD) decisions recommended before IQL training.**

**Beta-hedged reward:**
The plan states: "Raw PnL: 60-80% market beta. Beta-hedged: 3-5x higher SNR." This is quantitatively correct.

Standard reward: `r_t = PnL_t / NAV_t`

Beta-hedged reward: `r_t = (PnL_t - beta * SPY_return_t * NAV_t) / NAV_t`

The SNR improvement claim of 3-5x is validated by academic literature (Lopez de Prado 2018, Kolm & Ritter 2019). For a portfolio with beta ~0.8 and SPY daily vol ~1%, removing the market component reduces reward noise by approximately `beta * sigma_mkt / sigma_alpha` which for typical alpha strategies is indeed 3-5x.

**For crypto:** The beta-hedge should use BTC return instead of SPY. The plan mentions "Crypto penalties: high BTC correlation, stablecoin >30%, extreme volatility" which are **additional penalty terms** in the reward, not the core beta-hedge. This is correct — the reward should be:

```
r_crypto = (PnL_t - beta_btc * BTC_return_t * NAV_t) / NAV_t
         - lambda_corr * max(0, corr_btc - 0.85)
         - lambda_vol * max(0, vol - 0.60)
         - lambda_stable * max(0, stablecoin_pct - 0.30)
```

**Continuous action space [-1,1]:**
This is a position target (not a delta), where -1 = max short, 0 = flat, +1 = max long. Given that `config.allow_short = False` in the existing backtester, the effective range is [0, 1]. **The plan should specify whether short selling is enabled for DRL.** Current backtester has `allow_short = False`.

### Verdict: PASS

The PPO + SAC + IQL combination is well-justified. The beta-hedged reward is mathematically sound. The ONNX deployment path avoids PyTorch in production. One recommendation: add minimum dataset size gate (5,000 decisions) before enabling IQL training.

## 3. WALK-FORWARD VALIDATION WINDOWS

### Plan Specification
- Equity: 252d train / 63d test / 252d embargo
- Crypto: 60d train / 20d test / 60d embargo

### Quantitative Assessment

**Equity 252d/63d/252d:**

The **252-day embargo** is the concern. The plan states: "252-day embargo (equity) — 12-month momentum lookback. 5-day embargo leaks." The rationale is that standard momentum strategies use 12-month lookback, so a 252-day embargo prevents the test set from being contaminated by features that used the training period's data.

However, this is **overly conservative and wasteful of data:**

- Standard CPCV embargo in financial ML (Lopez de Prado, AFML Chapter 12) uses embargo = max feature lookback period, NOT the full lookback window. If the longest lookback feature is 200-day SMA, the embargo should be **200 days** at most.
- A 252-day embargo on a 252-day training window means you need 252+63+252 = **567 trading days (2.25 years)** of data per fold. With 10 CPCV splits across a typical 10-year dataset (2,520 trading days), you can only fit ~4 non-overlapping folds. This severely reduces statistical power.
- The claim "5-day embargo leaks" is correct for short-horizon features but the solution is too extreme. A **20-day embargo** would prevent leakage from all features up to SMA(20), and a **60-day embargo** would cover RSI(60), MACD(26), etc.

**Recommended equity embargo: 63 days (1 quarter)**
This covers all standard technical indicators (longest lookback is typically SMA(200) at the feature level, but the feature VALUE at time t already incorporates the prior 200 bars — the embargo only needs to cover autocorrelation in the TARGET, not the features).

**Crypto 60d/20d/60d:**

- 60-day training window is extremely short for a DRL agent. PPO typically needs 10,000+ timesteps to converge. 60 trading days at 1 bar/day = 60 timesteps. Even at 1-hour bars (60 * 24 = 1,440), this is marginal for PPO.
- 20-day test window is too short for statistical significance. With ~1 trade per day, you get ~20 trades — well below the 1,000-trade minimum stated in the plan's own verification criteria.
- 60-day embargo is reasonable given crypto regime shifts.

**Recommended crypto windows: 180d train / 60d test / 30d embargo**
This provides 180 daily bars (or 4,320 hourly bars) for training, 60 days for test (~60 trades minimum), and 30 days embargo which is sufficient for crypto's faster regime decay.

### Verdict: FAIL

The equity embargo is too long (252d wastes data, reduces statistical power). The crypto training window is too short (60d insufficient for PPO convergence). Specific recommendations provided above.

## 4. CRYPTO RISK FRAMEWORK

### Plan Specification

| Regime | Crypto Vol | Position Size | Leverage Cap |
|--------|-----------|---------------|-------------|
| Low | < 15% | 100% | 2.0x |
| Normal | 15-40% | 90% | 1.5x |
| High | 40-60% | 50% | 1.0x |
| Extreme | > 60% | 10% | 0.5x |

Plus circuit breakers: vol >60% -> Level 3 DANGER, correlation >0.85 -> Level 2, stablecoin >30% -> Level 2, exchange down -> Level 4.

### Existing Code State

The existing `CircuitBreaker` in `circuit_breaker.py` already has:
- Equity volatility thresholds: extreme 35%, high 20%, normal 15% (line 87-89)
- Cryptocurrency modifier: 1.5x (line 103) — meaning crypto thresholds would be 52.5%, 30%, 22.5%
- Graduated position sizing: extreme 3%, high 10%, normal 90%, low 100% (lines 91-94)
- Correlation circuit breaker at 0.85 (line 146-153)
- Cool-down periods: extreme 90min, high 45min, normal 20min (lines 107-112)
- Daily loss graduated: warning 2%, restricted 2.5%, halt 3% (lines 120-124)

### Quantitative Assessment

**Volatility regime thresholds:**

The plan's crypto thresholds (15/40/60%) vs the existing equity thresholds with 1.5x modifier (22.5/30/52.5%):

| Regime | Plan | Existing (with 1.5x modifier) | Delta |
|--------|------|-------------------------------|-------|
| Low/Normal boundary | 15% | 22.5% | Plan is MORE aggressive (tighter) |
| Normal/High boundary | 40% | 30% | Plan is LESS aggressive (looser) |
| High/Extreme boundary | 60% | 52.5% | Plan is LESS aggressive (looser) |

The plan's thresholds are more aligned with crypto reality. BTC's 30-day realized volatility has historically ranged from 20-120% annualized. A "normal" range of 15-40% captures ~60% of observations (per CryptoCompare data 2020-2025). **Approved.**

**Position sizing:**

| Regime | Plan | Existing CB position multiplier |
|--------|------|---------------------------------|
| Low | 100% | 100% |
| Normal | 90% | 90% |
| High | 50% | 10% |
| Extreme | 10% | 3% |

The plan is significantly more aggressive in High volatility (50% vs 10%). This is a deliberate choice for crypto — higher volatility creates opportunity. However, 50% position sizing at 40-60% annualized volatility is aggressive. Daily 1-sigma move at 50% vol = 3.15%, so a 2-sigma adverse move = 6.3% * 50% position = 3.15% portfolio loss per position. With 5 positions, that is a 15.75% portfolio hit. **This exceeds the 15% max drawdown gate.**

**WARNING 1:** Position sizing at 50% in High regime (40-60% vol) with multiple positions can breach the 15% max DD gate in a single day. Recommend reducing to 30% or capping total crypto exposure at 30% of portfolio.

**Leverage caps:**

| Regime | Cap | Assessment |
|--------|-----|------------|
| Low | 2.0x | Alpaca does NOT offer crypto margin beyond 1x for retail accounts. This cap is theoretical only. **WARNING 2.** |
| Normal | 1.5x | Same issue — Alpaca crypto is spot-only, no leverage. |
| High | 1.0x | Correct — no leverage in volatile markets. |
| Extreme | 0.5x | Effectively mandating 50% cash. Correct for capital preservation. |

**WARNING 2:** Alpaca's crypto trading is spot-only for most account types. The leverage caps of 2.0x and 1.5x are not achievable via Alpaca. The plan should note this and either (a) remove leverage caps > 1.0x for crypto, or (b) specify a different exchange for leveraged crypto (Bybit, Binance) which introduces multi-exchange complexity.

**Circuit breakers:**
All four crypto circuit breakers are methodologically sound and align with the existing `CircuitBreakerType` enum which already includes `VOLATILITY`, `CORRELATION`, `LIQUIDITY`, and `API_FAILURE`. The 30% stablecoin cap is unique — no existing circuit breaker type covers counterparty concentration. This will need a new `CircuitBreakerType.CONCENTRATION` enum value.

### Verdict: PASS with 2 WARNINGS

WARNING 1: High-regime position sizing (50%) can breach max DD gate with correlated multi-position drawdowns.
WARNING 2: Leverage caps > 1.0x are not achievable on Alpaca crypto (spot-only).

## 5. SEPARATE CRYPTO POLICY VS UNIFIED POLICY

### Plan Specification
- Separate training scripts: `train_drl_policy.py` (equity) and `train_drl_policy_crypto.py` (crypto)
- Separate observation spaces (46 vs 56 dims)
- Separate walk-forward windows
- Separate risk parameters

### Quantitative Assessment

This is the **correct design choice** for the following quantitative reasons:

1. **Distributional mismatch:** Equity daily returns are approximately normal with fat tails (kurtosis ~5-7). Crypto daily returns have heavier tails (kurtosis ~15-25) and are positively skewed. A unified policy trained on both would learn a compromise that is suboptimal for each.

2. **Volatility regime mismatch:** Equity "extreme" (35%+ annualized) is crypto "low-normal" (35% is below crypto's median historical vol). A unified agent would be confused by regime labels.

3. **Feature space incompatibility:** The 10 crypto-specific features (on-chain, funding rate, BTC correlation, regime one-hot) are meaningless for equities. Padding with zeros introduces noise; conditional masking adds complexity for marginal benefit.

4. **Market microstructure:** Equity trades during market hours (6.5h/day), crypto trades 24/7. The DRL agent's temporal dynamics are fundamentally different.

5. **Academic support:** Separate policies per asset class is standard in multi-asset RL trading systems (FinRL, Tensortrade, OpenAI Gym trading envs all use per-asset-class environments).

The only downside is **double the training cost** (two H100 runs instead of one). At $2-3/run, this is $4-6 total — negligible.

### Verdict: PASS

Separate policies are unambiguously correct for this use case.

## 6. TradeDecisionLog AS REPLAY BUFFER

### Plan Specification
- d3rlpy IQL reads from `TradeDecisionLog` table
- "Replay buffer from TradeDecisionLog"

### Existing Table Schema (`backend/app/models/trade_decision_log.py`)

```
id                  Integer PK
user_id             Integer FK(users.id)
portfolio_id        Integer FK(portfolios.id)
symbol              String(20)
decided_at          DateTime
strategy_name       String(100)
mode                String(20)          # 'paper' | 'live'

# Inputs
market_context      JSON                # price, rsi, sma20_pct, sma50_pct, 5d_change, volume
model_prompt        Text
model_raw_response  Text

# Decision
action              String(10)          # BUY | SELL | HOLD
confidence          Numeric(5,4)
reasoning           Text
risk_factors        Text                # JSON array as string

# Source
signal_source       String(20)          # 'ai' | 'rule_based'

# Outcome
trade_id            String(36) FK(trades.id) nullable
price_at_decision   Numeric(18,8)
price_at_1h         Numeric(18,8)
price_at_4h         Numeric(18,8)
price_at_1d         Numeric(18,8)
outcome_filled_at   DateTime
```

### IQL Replay Buffer Requirements

d3rlpy IQL requires tuples of `(observation, action, reward, next_observation, terminal)`:

| Required Field | Present in TradeDecisionLog? | Assessment |
|---------------|------------------------------|------------|
| **observation** (s_t) | PARTIAL — `market_context` JSON has price, rsi, sma20_pct, sma50_pct, 5d_change, volume = 6 dims. Missing: 40 of the 46 equity dims. | **FAIL** |
| **action** (a_t) | YES — `action` column (BUY/SELL/HOLD). But IQL needs continuous [-1,1] action, not discrete. | **FAIL** — discrete, not continuous |
| **reward** (r_t) | PARTIAL — Can be computed from `price_at_decision` and `price_at_1d`. But no beta-hedge possible without SPY/BTC return at decision time. | **FAIL** — no benchmark return stored |
| **next_observation** (s_{t+1}) | **NO** — No next-state stored. Would need to reconstruct from the NEXT row for the same symbol. Possible but fragile (gaps, missing rows). | **FAIL** — reconstruction required |
| **terminal** (done) | **NO** — No episode termination signal. | **FAIL** — missing |

### Critical Missing Columns

To use `TradeDecisionLog` as a proper IQL replay buffer, the following columns must be added:

1. **`observation_vector`** — `JSON` or `BYTEA` — The full 46/56-dim feature vector at decision time. The current `market_context` JSON only stores 6 values.
2. **`action_continuous`** — `Numeric(5,4)` — The continuous action value [-1, 1] mapped from the discrete BUY/SELL/HOLD + confidence.
3. **`benchmark_return`** — `Numeric(10,6)` — SPY or BTC return over the same horizon (for beta-hedged reward computation).
4. **`next_observation_vector`** — `JSON` or `BYTEA` — Feature vector at t+1 (or pointer to next row).
5. **`episode_id`** — `Integer` — Groups decisions into trading episodes for terminal signal computation.
6. **`is_terminal`** — `Boolean` — Whether this was the last decision in an episode.

### Dataset Size Assessment

Current: 3,559 decisions (448 AI, ~3,111 rule-based). Minimum for IQL: Lopez de Prado recommends 10x the observation dimension = 460 for 46-dim. However, for meaningful policy learning, 5,000-10,000 diverse transitions are standard. **The current dataset is at the low end of viability, particularly for the 56-dim crypto space.**

### Verdict: FAIL

The `TradeDecisionLog` table is missing 5 critical columns needed for IQL replay buffer. The `market_context` JSON stores only 6 of 46 required observation dimensions. Continuous action values and benchmark returns are not logged. An Alembic migration must add these columns BEFORE Milestone 3 begins. This is a **blocking dependency** for DRL training.

## 7. CPCV WITH DEFLATED SHARPE

### Plan Specification
- Walk-forward with CPCV 10 splits
- Deflated Sharpe at p < 0.05
- Deflated Sharpe > 0.5 gate

### Quantitative Assessment

**CPCV (Combinatorially Purged Cross-Validation):**

CPCV with 10 splits generates C(10,2) = 45 train/test combinations (each using 8 folds for train, 2 for test). This is the correct implementation per Lopez de Prado (AFML 2018, Chapter 12). The purging step removes training observations whose labels overlap with the test period, and the embargo prevents information leakage from feature lookback windows.

With 10 splits over ~2,520 trading days (10 years), each fold is ~252 days. The plan's 252-day embargo would prevent ANY adjacent fold from being used in training alongside its neighbor in the test set — this is equivalent to reducing to ~5 effective combinations. **This further supports the recommendation in Item 3 to reduce the embargo to 63 days.**

**Deflated Sharpe Ratio (DSR):**

The DSR adjusts the observed Sharpe ratio for the number of trials (strategies tested), skewness, and kurtosis:

```
DSR = Phi^{-1}(1 - e * Phi(-SR_hat * sqrt(T-1) / sqrt(1 - gamma_3 * SR_hat + (gamma_4 - 1)/4 * SR_hat^2)))
```

Where:
- `SR_hat` = observed Sharpe ratio
- `T` = number of observations
- `gamma_3` = skewness of returns
- `gamma_4` = kurtosis of returns
- `e` = expected max Sharpe ratio under the null (depends on number of trials)

The plan specifies DSR > 0.5 at p < 0.05. However, there is a **specification ambiguity:**

- "Deflated Sharpe > 0.5" — this means the DSR VALUE must exceed 0.5. But DSR is a probability (0-1), not a Sharpe ratio. If the plan means the underlying Sharpe > 0.5 AND DSR p-value < 0.05, that is two separate tests.
- More likely interpretation: The **observed Sharpe must be > 0.5** AND the **DSR must confirm statistical significance at p < 0.05** (i.e., the probability that this Sharpe could have been achieved by chance, given the number of strategies tested, is < 5%).

This dual gate is appropriate. A Sharpe of 0.5 after transaction costs is achievable for a well-designed swing trading strategy (typical institutional target is 0.5-1.5 net of costs for systematic strategies).

**Implementation concern:**

The existing `PerformanceAnalyzer` in `backtesting/performance.py` calculates standard Sharpe but does NOT implement:
- Deflated Sharpe Ratio
- CPCV splitting logic
- Purging or embargo

These are entirely new implementations needed in `backtesting_service.py`. The backtesting engine (`engine.py`) does simple sequential walk-forward, not CPCV. **Complete rebuild of the validation pipeline is required** — the existing backtester cannot be extended to support CPCV without significant refactoring of the fold generation and data handling.

**Minimum trade count:**

The plan's verification section states "minimum 1000 trades for statistical significance." With CPCV 45 combinations, that means ~22 trades per combination. This is marginal. At 1 trade/day on the 252-day test folds, you get ~252 trades per fold and ~504 per combination (2-fold test), giving ~22,680 total across all 45 combinations. **This is sufficient IF the trading frequency achieves 1 trade/day. At the current rate of ~12 trades/month (observed in production logs), you would only get ~33 trades per fold, which is below the statistical significance threshold.**

### Verdict: CONDITIONAL PASS

The CPCV + DSR methodology is correct and appropriate. Conditional on: (a) clarifying the DSR > 0.5 specification (Sharpe > 0.5 AND DSR p < 0.05), (b) implementing CPCV from scratch (existing backtester cannot support it), (c) ensuring sufficient trade frequency for statistical significance, (d) reducing embargo to 63 days to preserve fold count.
