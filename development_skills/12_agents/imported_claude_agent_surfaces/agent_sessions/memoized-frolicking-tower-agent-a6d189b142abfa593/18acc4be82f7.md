### CRITICAL-1: Massive Duplication -- RiskEngineService vs 4 Existing Risk Modules

**What:** The plan proposes a new `risk_engine_service.py` alongside **four** existing risk modules that already cover the same domain.

**Where (existing files):**
- `backend/app/services/risk_management.py` (1,063 lines) -- `RiskManagementService` with `RiskMetrics`, `TradeRiskAssessment`, VaR, Sharpe, drawdown monitoring, position sizing, 10+ pre-trade risk checks

1. Conflicting risk parameter sources (which config is authoritative?)
2. Trades checked by one risk engine but not another

4. Developer confusion about which risk module to import

### CRITICAL-2: CircuitBreakerService Duplicates Existing 711-Line Circuit Breaker

**What:** The plan creates `circuit_breaker_service.py`. A fully-featured `CircuitBreaker` class already exists.

**Where:** `backend/app/trading_engine/engine/circuit_breaker.py` (711 lines)
- 9 circuit breaker types (`SYSTEM`, `VOLATILITY`, `DAILY_LOSS`, `PER_STRATEGY`, `CORRELATION`, `API_FAILURE`, `LIQUIDITY`, `EXECUTION`, `MANUAL`)
- Graduated responses (`OPEN`, `RESTRICTED`, `CAUTIOUS`, `HALF_OPEN`, `CLOSED`)
- Hysteresis-based volatility regime detection
- Position sizing multipliers per volatility regime
- Auto-reset with cooldown periods
- Thread-safe with `threading.RLock`
- Singleton via `get_circuit_breaker()`

**Why:** The existing circuit breaker is already integrated into `AutoTradingService` (line 23: `from app.trading_engine.engine.circuit_breaker import get_circuit_breaker`) and `TradeExecutor` (line 21: `from .circuit_breaker import CircuitBreakerType, get_circuit_breaker`). Creating a parallel system means the bot could bypass the production circuit breaker.

**How:** Use `get_circuit_breaker()` directly. If the bot needs a "kill switch" concept beyond circuit breakers, create a thin `KillSwitch` class that **trips the existing system-wide circuit breaker**:

```python
from app.trading_engine.engine.circuit_breaker import (
    get_circuit_breaker, CircuitBreakerType
)

class BotKillSwitch:
    """Emergency stop that trips the existing circuit breaker system."""

    def activate(self, reason: str) -> None:
        cb = get_circuit_breaker()
        cb.trip(CircuitBreakerType.SYSTEM, f"BOT KILL SWITCH: {reason}")

    def is_active(self) -> bool:
        cb = get_circuit_breaker()
        allowed, status = cb.check(CircuitBreakerType.SYSTEM)
        return not allowed
```

### CRITICAL-3: OrderManagementService vs Existing Trade Execution Stack

**What:** Plan proposes `order_management_service.py` and `execution_service.py` alongside 3 existing execution modules.

**Where (existing):**
- `backend/app/services/trade_execution.py` (1,015 lines) -- `TradeExecutionService` with TWAP/VWAP/Iceberg chunking, OCO orders, trailing stops, partial fills, execution metrics, slippage tracking. Uses `Decimal` throughout.
- `backend/app/trading_engine/engine/trade_executor.py` (571 lines) -- `TradeExecutor` with broker integration, retry logic, stop-loss/take-profit creation, signal validation, position tracking
- `backend/app/services/paper_trading.py` (847 lines) -- `PaperTradingService` with realistic slippage simulation, commission calculation, fill quality assessment, portfolio holding updates

**Why:** The existing stack already handles:
- Real broker execution (via `BaseBroker` interface)
- Paper trading simulation with realistic fills
- Order chunking (TWAP, VWAP, Iceberg)
- Partial fills, OCO, trailing stops
- Execution metrics (slippage, implementation shortfall, fill rate)

Creating parallel OMS/execution modules means:
1. Two code paths for placing the same trade -- maintenance nightmare
2. Potential for the bot to execute without proper slippage modeling
3. Portfolio holdings updated by one path but not the other

**How:** The bot should use `TradeExecutor.execute_strategy_signal()` which already accepts a signal dict and handles the full lifecycle. Create a thin `BotOrderRouter` that formats RL actions into the signal dict expected by `TradeExecutor`:

```python
class BotOrderRouter:
    """Translates RL agent actions into signals for existing TradeExecutor."""

    def __init__(self, executor: TradeExecutor):
        self._executor = executor

    async def route_action(self, action: dict, portfolio: Portfolio) -> Optional[Trade]:
        signal = {
            "action": action["side"],  # "buy" or "sell"
            "symbol": action["symbol"],
            "price": action["price"],
            "confidence": action["confidence"],
            "stop_loss": action.get("stop_loss"),
            "take_profit": action.get("take_profit"),
        }
        return await self._executor.execute_strategy_signal(signal, portfolio)
```

```python
    def __init__(self, feature_engine: FeatureEngineering):
```

### HIGH-1: Dead Code Creation -- AutoTradingService

**What:** If the new bot system handles autonomous trading, the existing `AutoTradingService` (1,500+ lines, singleton at class level) becomes dead code.

**Where:** `backend/app/services/auto_trading_service.py` -- Currently active in production, integrated with:
- `auto_trading.py` API router (7 endpoints)
- `AutoTradingSession` model (DB persistence across cold starts)
- `autoTradingApi.ts` frontend (RTK Query with 7 endpoints)
- `useAutoTrading` hook (used by Dashboard + AIPage)

**Why:** If both systems coexist, the user has two "start trading" buttons that do different things. If the new system replaces the old, there needs to be a deprecation plan.

**How (mandatory):** The plan MUST explicitly state one of:
1. **Evolution**: The new bot system IS the next version of `AutoTradingService`. Modify `auto_trading_service.py` in-place, adding RL-based signal generation while preserving the existing API contract.
2. **Parallel**: Both coexist (the existing system for rule-based strategies, the new system for RL). In this case, the frontend needs clear separation and the backend needs a unified status endpoint.
3. **Replacement**: The old system is deprecated. Define the migration plan, frontend changes, and API versioning.

My recommendation: **Option 1 (Evolution)**. The existing `AutoTradingService` already has the loop, DB persistence, cold-start recovery, and frontend integration. Add RL signal generation as another strategy type in `StrategyRegistry`.

### HIGH-2: 8-Table Migration Without Rollback Strategy

**What:** The plan proposes 8 new SQLAlchemy tables in a single migration.

**Where:** This violates the established schema drift prevention protocol (MEMORY.md: "create_all() does NOT add columns to existing tables" + "2026-02-15 incident: 7 new User model columns broke login for ALL users")

**Why:** A monolithic 8-table migration:
1. Cannot be partially rolled back -- if table 6 fails, tables 1-5 are orphaned
2. Requires 8 corresponding ALTER TABLE commands on Cloud SQL BEFORE deploy
3. If any column type is wrong (e.g., Float instead of Numeric(18,8)), fixing it requires another migration

**How (mandatory requirements):**
1. Each table gets its own Alembic migration file
2. Each migration must have a working `downgrade()` function
3. All money columns MUST use `Numeric(18,8)`, never `Float` or `Decimal` without precision
4. All datetime columns MUST use `DateTime(timezone=True)`
5. Test migration on a local PostgreSQL (not SQLite) before Cloud SQL
6. Follow the schema drift protocol: run ALTER TABLE on Cloud SQL FIRST, then deploy code

### HIGH-3: Configuration Explosion

**What:** The plan introduces potentially 15+ new environment variables (RL hyperparameters, training schedules, model paths, etc.)

**Where:** `backend/app/core/config.py` is already 408 lines with 100+ settings.

**Why:** Every new env var is a potential production misconfiguration. Missing RL hyperparameters could cause the bot to trade with default (potentially dangerous) settings.

**How:** Group all bot config into a single `BotConfig` Pydantic model with sensible defaults:

```python
class BotConfig(BaseSettings):

    BOT_MAX_POSITION_PCT: float = 0.05  # 5% max per position

    BOT_PAPER_VALIDATION_DAYS: int = 30
    BOT_MIN_SHARPE_FOR_LIVE: float = 1.0
    BOT_KILL_SWITCH_DRAWDOWN_PCT: float = 3.0

    model_config = {"env_prefix": "BOT_"}
```

Add this as a nested config in the main Settings class: `bot: BotConfig = BotConfig()`

### HIGH-5: Circular Import Risk

**What:** 15 new service files with cross-dependencies (e.g., `trading_bot_service -> risk_engine_service -> circuit_breaker_service -> trading_bot_service` for kill switch callbacks).

**Where:** New files would live in `backend/app/services/bot/` (proposed).

**Why:** Python circular imports crash the application at startup. The existing codebase avoids them through:
- Lazy imports (`try/except ImportError`)
- Singleton pattern with module-level `get_*()` functions
- One-directional dependency flow (services import models, never the reverse)

**How (mandatory):** Define a strict import DAG before writing any code:

```
models (DB tables)
  ^
  |
adapters (BotRiskAdapter, BotOrderRouter, BotStateEncoder)
  ^
  |
bot_engine (core RL loop, no DB imports)
  ^
  |
bot_service (FastAPI integration, DB session, ties it all together)
  ^
  |
bot_router (API endpoints)
```

No file may import from a file below it in this DAG.

## 5. Final Verdict

### AGREE WITH MODIFICATIONS

The autonomous trading bot is architecturally sound in concept. The research documents demonstrate deep understanding of RL trading systems. However, the implementation plan as written would create a **parallel codebase** that duplicates 4,000+ lines of existing, battle-tested infrastructure.

### 14 Mandatory Requirements Before Implementation

1. **NO new risk engine** -- Use adapter pattern over existing `RiskManagementService`
2. **NO new circuit breaker** -- Use `get_circuit_breaker()` singleton
3. **NO new OMS/execution** -- Route through existing `TradeExecutor`
4. **NO new feature engineering** -- Extend existing `FeatureEngineering` class
5. **NO new backtesting engine** -- Wrap existing `BacktestEngine`
6. **NO new RL module** -- Extend existing `reinforcement_learning.py`
7. **Declare coexistence strategy** for `AutoTradingService` (evolve, parallel, or replace)
8. **One migration per table** with working `downgrade()`
9. **All money columns** use `Numeric(18,8)` and `Decimal` arithmetic
10. **Group bot config** into a single `BotConfig` Pydantic model
11. **Frontend uses RTK Query** pattern (not axios + thunks)
12. **Define import DAG** before writing any file
13. **No PII to LLM** in `llm_strategy_service`
14. **Create 7+ test files** with financial precision tests

### Net File Count After Modifications

Instead of ~15 new files, the plan should create approximately **7 new files**:

| File | Purpose |
|---|---|
| `backend/app/services/bot/bot_service.py` | Core bot orchestrator (replaces `trading_bot_service.py`) |
| `backend/app/services/bot/bot_risk_adapter.py` | Delegates to existing risk infrastructure |
| `backend/app/services/bot/bot_order_router.py` | Formats RL actions for existing TradeExecutor |

| `backend/app/api/api_v1/endpoints/bot.py` | API router (3-5 endpoints) |
| `frontend/src/services/botApi.ts` | RTK Query API (extends existing patterns) |

Plus 7 test files. Total: ~14 files instead of ~25, with zero duplication.
