### 1. SCHEMA MIGRATIONS: 3 New Tables + Trade Model Columns

**VERDICT: FAIL -- CRITICAL deployment order gap**

**What the plan says:**
- M1 creates 3 new tables: `bot_performance`, `bot_risk_event`, `bot_audit_log`
- M4 adds ~12 columns to the existing `Trade` model (wash sale, crypto tax)
- Plan says "Deploy immediately" for M1 but does NOT specify migration-before-code order

**What I found:**

1. **Dockerfile does NOT copy Alembic directory.** Line 60: `COPY backend/app ./app` -- this copies only the `app/` subdirectory. The `backend/alembic/` directory is never copied into the Docker image. The entrypoint (`entrypoint.sh`) runs gunicorn directly with zero migration steps.

2. **`init_db.py` uses `Base.metadata.create_all()` (line 60)** which will create NEW tables but CANNOT add columns to existing tables. This is the exact pattern that caused the 2026-02-15 production outage documented in MEMORY.md.

3. **The Trade model column additions in M4 are HIGH RISK.** The existing `trades` table has ~20 columns. Adding `is_wash_sale`, `wash_sale_adjustment`, `cost_basis_method`, `lot_id`, `holding_period`, `is_crypto`, `exchange_name`, `transaction_fee`, `cost_basis_usd`, `realized_gain_loss_usd`, `is_staking_reward`, `staking_income_usd` -- that is 12 new columns on a live table. `create_all()` will silently do nothing for these. The code will crash with `UndefinedColumn` errors the moment any endpoint touches these fields.

4. **Alembic migration chain is fragile.** Current head: `add_bot_sessions_20260224`. There are already 2 merge commits in the chain (`f18101facea7`, `merge_heads_2026_02_23`). Adding 3+ new tables plus Trade ALTER TABLEs across multiple milestones will likely create branch conflicts.

**Required fixes:**
- The plan MUST specify: **schema migration runs BEFORE code deployment** for every milestone
- For M1 (new tables only): `create_all()` will work since these are brand-new tables. But Alembic migrations MUST still be created and run to keep `alembic_version` tracking accurate (lesson from 2026-02-21 incident)
- For M4 (Trade column additions): `ALTER TABLE trades ADD COLUMN ...` MUST run on Cloud SQL BEFORE deploying the code revision. This is a hard dependency.
- Either: (a) add Alembic to the Docker image and run `alembic upgrade head` in `entrypoint.sh`, or (b) maintain explicit manual migration steps in a runbook
- Recommendation: Option (a) -- add `COPY backend/alembic ./alembic` and `COPY backend/alembic.ini ./alembic.ini` to Dockerfile, then modify `entrypoint.sh` to run `alembic upgrade head` before gunicorn start

### 2. MODEL DEPLOYMENT: DRL Models H100 -> GCS -> Cloud Run ONNX

**VERDICT: FAIL -- No GCS client in Docker image, no model loading code exists**

**What the plan says:**
- M3: Train DRL policy on H100 VM -> export to ONNX -> upload to `gs://elson-33a95-elson-models/drl-models/`
- Cloud Run loads ONNX models via `onnxruntime` at inference time
- Plan adds `onnxruntime>=1.17.0` to `requirements-docker.txt`

**What I found:**

1. **`google-cloud-storage` is NOT in `requirements-docker.txt`.** Grep returned zero matches. Cloud Run cannot download from GCS without this library.

2. **No model loading infrastructure exists.** There is no code in the backend that downloads files from GCS at runtime. The existing vLLM integration uses HTTP calls to `http://10.138.0.4:8000` (the L4 VM). ONNX model loading from GCS is a completely new capability.

3. **Cloud Run has 2Gi memory limit** (confirmed in `cloudbuild.yaml` line 96). ONNX DRL models for 46-dim/56-dim observation spaces should be small (~5-20MB), so memory is likely fine. But this needs verification after training.

4. **No model versioning or rollback mechanism.** The plan specifies a single GCS path `drl-models/` but does not address how to:
   - Load a specific model version
   - Roll back to a previous model if the new one degrades
   - Handle the case where GCS is temporarily unreachable

**Required fixes:**
- Add `google-cloud-storage>=2.14.0` to `requirements-docker.txt`
- Create a model loading service that downloads from GCS on startup (with caching to local disk)
- Add model versioning: `gs://elson-33a95-elson-models/drl-models/{version}/equity_policy.onnx`
- Add fallback: if ONNX model unavailable, fall back to rule-based strategy (existing behavior)
- Cloud Run service account already has GCS access (used for other operations), but verify `roles/storage.objectViewer` is granted

### 3. NEW DEPENDENCIES IN DOCKER: onnxruntime, polygon-api-client, ccxt

**VERDICT: PASS with ADVISORY**

**What the plan says:**
- `requirements-docker.txt`: Add only `onnxruntime>=1.17.0` (NOT torch, stable-baselines3, etc.)
- `requirements.txt` (dev): Add `stable-baselines3`, `gymnasium`, `hmmlearn`, `d3rlpy`, `polygon-api-client`, `ccxt`, `glassnode-api`

**What I found:**

1. **Good separation.** The plan correctly keeps heavy training dependencies (PyTorch, stable-baselines3, d3rlpy) out of the Docker image. Only ONNX runtime goes to production.

2. **Current Docker image baseline:** Python 3.13-slim + the current `requirements-docker.txt` (87 lines of dependencies including scikit-learn, xgboost, pandas, scipy, matplotlib, plotly, opentelemetry). Image is already substantial.

3. **`onnxruntime` adds ~80MB** (plan correctly notes this). Acceptable.

4. **Missing from the plan:** The plan lists `polygon-api-client` and `ccxt` in dev requirements only. However, if the `crypto_market_analyzer.py` or `feature_engineering_service.py` services run in production (which they must for the bot), these libraries need to be in `requirements-docker.txt` too. The plan contradicts itself -- M2 creates `crypto_market_analyzer.py` as a backend service but M3 lists `polygon-api-client` and `ccxt` as dev-only.

5. **`glassnode-api`** is also a runtime dependency if on-chain metrics are fetched in production.

**Required fixes:**
- Clarify which data-fetching libraries are needed in production. If `crypto_market_analyzer.py` runs in Cloud Run (it must, for live trading), then `ccxt`, `polygon-api-client`, and `glassnode-api` must be added to `requirements-docker.txt`
- Alternative: Use HTTP/REST calls instead of SDK libraries to avoid bloating the Docker image. Polygon and Glassnode both have simple REST APIs.

3. **No additional firewall rules needed** for the new external API connections. Cloud Run already has internet egress by default for non-private-range destinations.

**No fixes required.**

**What I found:**

   - No scheduler job to start the H100 VM
   - No scheduler job to trigger the training script
   - No scheduler job to stop the H100 VM after training completes
   - No mechanism to upload the trained model to GCS and notify Cloud Run

3. **H100 VM start/stop is currently manual** via `gcloud compute instances start/stop elson-h100-spot --zone=us-central1-a` (per GCP_AGENT docs)

- Create Cloud Scheduler jobs for Sunday evening training cycle:

### 7. DEPLOYMENT SEQUENCE: M1 Schema Migration Before Code

**VERDICT: FAIL -- Not explicitly called out; creates_all() will mask the problem for M1 but explode on M4**

**What the plan says:**
- M1: "Deploy immediately. Safety + crypto foundation live first."
- M4: Extends Trade model with 12 new columns + creates `crypto_tax_*` migration

**What I found:**

This is a restatement of finding #1 but from the deployment sequence perspective:

1. **M1 (new tables only):** `create_all()` in `init_db.py` will handle this because these are entirely new tables. However, Alembic will NOT know about them. The next `alembic revision --autogenerate` will try to re-create tables that already exist. This is the exact pattern that caused the 2026-02-21 incident (43 tables existed but `alembic_version` was empty).

2. **M4 (ALTER TABLE on trades):** This WILL fail. `create_all()` does not add columns to existing tables. The 12 new Trade columns will not exist in production. Every query touching `is_wash_sale`, `is_crypto`, etc. will raise `UndefinedColumn`.

3. **The plan has no "run migration" step in any milestone's deployment instructions.** Every milestone ends with "Deploy." but none says "Run Alembic migration first."

**Required deployment sequence for EACH milestone:**
```
1. Create Alembic migration (dev machine)
2. Run migration against Cloud SQL (manual or via entrypoint.sh)
3. Deploy new code to Cloud Run (gcloud builds submit)
4. Verify health check passes
5. Smoke test affected endpoints
```

**The plan MUST be amended to include this sequence explicitly, especially for M4.**

### 8. MULTI-REGION CONCERN: H100 in us-central1-a, Everything Else in us-west1

**VERDICT: PASS -- Acceptable for batch training workload**

**What the plan says:**
- H100 VM: `elson-h100-spot`, us-central1-a
- Everything else: us-west1 (Cloud Run, Cloud SQL, L4 VM, GCS bucket)

**What I found:**

1. **Training artifacts flow:** H100 (us-central1-a) -> GCS (us-west1) -> Cloud Run (us-west1). The cross-region GCS upload adds ~50-100ms latency per request but this is a one-time batch upload after training (not per-inference). ONNX models are small (~5-20MB). Inter-region egress cost: ~$0.01/GB. Negligible.

2. **Training data flow:** Cloud SQL (us-west1) -> H100 (us-central1-a). The training scripts pull `TradeDecisionLog` data for replay buffer. This is a batch read, not real-time. Cross-region latency is acceptable for this use case.

3. **Why H100 is in us-central1-a:** H100 availability. GCP does not guarantee H100 spot availability in us-west1. The training scripts already reference us-central1-a. Changing regions would require re-provisioning.

4. **No latency impact on inference path:** ONNX inference happens in Cloud Run (us-west1), loading models from GCS (us-west1). The H100 is never in the hot path.

**No fixes required.** The multi-region split is architecturally sound for the training-vs-inference separation pattern.
