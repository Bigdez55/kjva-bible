# EXECUTIVE SUMMARY

The proposed autonomous trading bot plan has **3 CRITICAL blockers**, **4 HIGH-severity architectural mismatches**, and **3 MEDIUM items** that must be resolved before any implementation begins. The existing Cloud Run infrastructure is fundamentally incompatible with several requirements of the proposed system. Below is the detailed assessment.

# SECTION 1: DRIFT DETECTION -- IMMEDIATE FINDINGS

## DRIFT-01 (CRITICAL): minScale Contradiction

**Evidence:** User memory states "Cloud Run minScale=0: Set to scale-to-zero" but `cloudbuild.yaml` line 99 reads `--min-instances=1`.

**Impact:** If the live Cloud Run service was manually set to minScale=0 via `gcloud run services update` but `cloudbuild.yaml` still says `--min-instances=1`, then the **next Cloud Build deploy will silently revert to min-instances=1**, costing ~$20/mo extra. Conversely, if the service is truly at 0, the bot loop dies on scale-down.

**Resolution required:** Verify actual Cloud Run state with:
```
gcloud run services describe elson-backend --region=us-west1 --format="value(spec.template.metadata.annotations['autoscaling.knative.dev/minScale'])"
```
Then align `cloudbuild.yaml` to match.

## DRIFT-02 (INFO): Gunicorn Worker Count

`entrypoint.sh` runs gunicorn with `-w 2` (2 workers). Each worker runs its own event loop. The `AutoTradingService` singleton is per-worker, meaning:
- If user starts bot via worker 1, requests hitting worker 2 will show `is_active=False`
- The bot loop lives in ONE worker's asyncio event loop; the other worker has no knowledge of it

This is an existing architectural weakness, not new to this proposal, but it will become a **data integrity issue** if the proposed DRL bot runs across workers.

# SECTION 2: TEN-POINT INFRASTRUCTURE REVIEW

## 1. Cloud Run vs. Long-Running Bot -- CRITICAL MISMATCH

**Current state:** The auto-trading loop (`_auto_trading_loop` at line 304 of `auto_trading_service.py`) is an infinite `while True` loop with `asyncio.sleep(60)` between iterations. It runs as an `asyncio.Task` created during the `/auto-trading/start` POST handler.

**The fundamental problem:** Cloud Run is a request-response platform. The bot loop is kept alive ONLY because it was spawned as a background asyncio task on the same worker process that serves HTTP requests. This works only under these conditions:
1. At least one Cloud Run instance stays alive (min-instances >= 1)
2. The specific gunicorn worker that started the bot does not crash/restart
3. Gunicorn's `--timeout 120` does not kill the worker (it won't -- the bot is async, not blocking)

**Assessment:** The current design ALREADY works on Cloud Run with min-instances=1. The bot loop persists as long as the worker process persists. Adding DRL capabilities does not change this fundamental dynamic.

**Risk with scale-to-zero (minScale=0):** If the instance scales to zero, the bot loop is terminated. On cold start, the lifespan handler (`_restore_trading_sessions`) attempts to restore active sessions from the database. This is the recovery mechanism.

**RECOMMENDATION:**
- For active trading: `min-instances=1` is MANDATORY. The `cloudbuild.yaml` already sets this.
- If user truly wants scale-to-zero for cost savings when bot is OFF, use a deploy-time parameter: `min-instances=0` when bot is not expected to run, `min-instances=1` when it is.
- A Cloud Scheduler job hitting `/auto-trading/health` every 5 minutes during market hours would keep the instance warm even at min-instances=0, but this is fragile. I do NOT recommend it for a system holding open positions.
- **Long-term:** Migrate the bot loop to a Compute Engine VM or Cloud Run Jobs (for batch-mode), NOT Cloud Run services. But this is not a blocker for the current proposal.

## 2. Scale-to-Zero Conflict -- HIGH

**Current state:** `cloudbuild.yaml` says `min-instances=1`. User memory says `minScale=0`. These are contradictory.

**The actual risk:** If the bot is running (has open positions) and Cloud Run scales to zero:
1. Bot loop is killed mid-iteration
2. Open positions are NOT closed (no shutdown hook for position management)
3. On cold-start restore, the bot restarts but has lost all in-memory state (circuit breaker counts, scan cycle counts, last signal times)

**The lifespan restore mechanism** (`_restore_trading_sessions` at main.py line 58-98) queries `AutoTradingSession.is_active == True` and calls `start_auto_trading` for each. This IS a recovery mechanism, but it:
- Loses the `_scan_cycle_counts` (resets to 0, triggering an immediate full rescan)
- Loses `_last_signal_sources` (may generate duplicate signals)
- Does NOT verify that positions opened before the kill are still valid

**RECOMMENDATION:**
- `min-instances=1` during market hours (M-F 6:30am-1:00pm PST)
- Add a shutdown hook in lifespan's yield block that persists `_scan_cycle_counts` and `_last_signal_sources` to the database before the instance dies
- Add position reconciliation to `_restore_trading_sessions` that checks Alpaca for any fills that occurred while the instance was down

## 3. DRL Training Compute -- CRITICAL BLOCKER

**Current state:** Cloud Run backend has 2GB RAM, 1 CPU. The L4 VM (`elson-dvora-training-l4-2`) has 22.5GB VRAM with an L4 GPU. The L4 scheduler is PAUSED.

**DRL training requirements:**
- PPO/SAC with stable-baselines3 requires PyTorch
- Training on 1+ year of market data for even a small policy network needs 4-8GB RAM minimum
- Full training runs take 30-60 minutes per retraining cycle
- Weekly retraining means this runs once per week

**Where it CANNOT run:**
- Cloud Run (2GB RAM, no GPU, 300s request timeout, no persistent disk)
- Cloud Build (E2_HIGHCPU_8 = 16GB RAM -- tight but possible if training is optimized; however, build timeout of 1200s may not be enough)

**Where it CAN run:**
1. **L4 VM (`elson-dvora-training-l4-2`)** -- Already exists, has GPU, has the model ecosystem. Training is a weekly batch job. Run it as a Cloud Scheduler-triggered script.
2. **Dedicated n1-standard-4 VM** (16GB RAM, no GPU) -- CPU-only PPO/SAC training is viable for small policy networks. Cheaper than keeping L4 running.
3. **Vertex AI Custom Job** -- BUT quota is 0 for T4/L4 in us-west1 (documented in memory). Not viable without quota increase.

**RECOMMENDATION:**
- Use the L4 VM for training. It already runs vLLM during market hours. Training can run during off-hours (nights/weekends) when vLLM is idle.
- Create a training script at `gs://elson-33a95-elson-models/scripts/train-drl-bot.sh`
- Trigger via Cloud Scheduler: Sunday night, start L4 VM -> run training -> upload weights to GCS -> stop VM
- Production inference (the DRL model's `predict()` call) runs on Cloud Run -- it is lightweight (forward pass only, <100ms, <500MB RAM)

## 4. Redis in Production -- HIGH (EXISTING BLOCKER)

**Current state:** Redis is running locally via Homebrew. GCP Memorystore is blocked by billing. The `security.py` Redis client has ping-on-init with 2s timeout and graceful fallback.

**The proposed feature store requires Redis with 60s TTL.** This means:
- Without Redis in production, the feature store falls back to... nothing. Features must be recomputed on every cycle.
- The auto-trading loop already runs at 60s intervals, so a 60s TTL cache provides exactly ONE reuse per feature computation. The benefit is marginal.

**RECOMMENDATION:**
- **Do NOT make Redis a hard dependency for the DRL bot.** Design the feature store with an in-memory LRU cache fallback (e.g., `cachetools.TTLCache`).
- The existing `REDIS_URL` secret and `_create_redis_client()` pattern can be used when Redis becomes available.
- When GCP billing is re-enabled, Memorystore provision is a 4-step process (documented in memory).
- For a single-user trading bot, an in-memory TTLCache with 60s expiry is SUFFICIENT. Redis becomes necessary only when you need cross-instance state sharing (which you do NOT need at min-instances=1).

## 5. Migration Safety -- HIGH

**Current state:** 13 Alembic migrations exist. The latest is `add_auto_trading_sessions_2026_02_24.py`. Cloud Build does NOT run migrations -- it only builds and deploys. Migrations are run manually via Cloud SQL Proxy.

**The proposed plan adds 8 new tables.** Based on past incidents:
- 2026-02-15: 7 new User columns deployed without ALTER TABLE = production outage
- 2026-02-21: Cloud SQL was untracked by Alembic (no `alembic_version` table). Fixed with `alembic stamp head`.

**RECOMMENDATION:**
- Create a SINGLE Alembic migration for all 8 tables (not 8 separate migrations)
- Run the migration via Cloud SQL Proxy BEFORE deploying new code:
  ```
  cloud-sql-proxy elson-33a95:us-west1:elson-postgres --port=15432 &
  DATABASE_URL=postgresql://postgres:$DB_PASS@localhost:15432/elson_trading alembic upgrade head
  ```
- Verify with `\dt` that all 8 tables exist before deploying
- Add a Cloud Build step that runs `alembic upgrade head` automatically. This should be step 5.5 (after push-backend, before deploy-backend). The Cloud SQL Proxy sidecar pattern supports this.

**Safe execution order:**
1. Create Alembic migration locally
2. Test migration against local DB
3. Run migration against Cloud SQL (via proxy)
4. Verify tables exist
5. Deploy code via Cloud Build

## 6. WebSocket Keepalive -- MEDIUM

**Current state:** `market_streaming.py` uses FastAPI WebSocket, NOT a client-side Alpaca WebSocket. The streaming loop polls `market_data_service.get_quote()` every 5 seconds and broadcasts to connected browser clients.

**Cloud Run's 300s timeout applies to HTTP requests, NOT WebSocket connections.** Cloud Run supports WebSocket connections with a maximum idle timeout of 3600 seconds (1 hour). The streaming loop sends data every 5 seconds, so it will never be idle.

**The proposed Alpaca WebSocket stream is different.** This would be a SERVER-SIDE WebSocket client (Cloud Run -> Alpaca API). Cloud Run supports outbound WebSocket connections. The connection stays alive as long as:
- The Cloud Run instance is alive (min-instances=1)
- Data flows on the connection (Alpaca sends data continuously during market hours)
- The connection does not exceed the platform's 24-hour maximum connection duration (Cloud Run disconnects WebSocket after 24h)

**RECOMMENDATION:**
- Alpaca WebSocket streaming from Cloud Run IS viable with min-instances=1
- Add automatic reconnection logic with exponential backoff
- For the 24-hour limit: add a daily reconnect at 4:00 PM ET (after market close) to reset the connection timer before the next trading day
- The `websockets` library (already in requirements-docker.txt) supports this pattern

# SECTION 4: ARCHITECTURAL RECOMMENDATION

**The current Cloud Run + L4 VM architecture CAN support the proposed autonomous bot**, provided:

1. `min-instances=1` is maintained during market hours
2. DRL training runs on the L4 VM as a batch job (not on Cloud Run)
3. Redis dependency is softened to in-memory fallback
4. Alembic migrations are run BEFORE code deploy (add to pipeline or run manually)
5. Bot state persistence is added (shutdown hook + cold-start reconciliation)

**The architecture does NOT need to change to Compute Engine or GKE.** The bot loop already works as an asyncio task in the gunicorn worker. The proposed DRL additions (model inference) are lightweight forward passes that fit within the 2GB Cloud Run memory limit. Only the TRAINING pipeline needs dedicated compute, and the L4 VM is the natural target.

**Do NOT add a new VM for the bot.** The user has explicitly forbidden suggesting resource deletion, and adding another always-on VM would double the compute cost for no architectural benefit. Cloud Run with min-instances=1 is equivalent to a small VM for this single-user use case.
