## EXECUTIVE SUMMARY

The plan (`memoized-frolicking-tower.md`) is architecturally sound and demonstrates awareness of critical regulatory domains. However, it contains **4 FAIL items** (P0/P1 severity) that must be remediated before any code ships, **3 CONDITIONAL PASS items** requiring specification tightening, and **3 PASS items** that are correctly addressed. The most critical deficiency is the **complete absence of a PDT enforcement gate in the execution path** -- the plan mentions PDT compliance as a validation gate in paper_validation_service but does NOT wire it into the live trade execution kill-switch. This is a regulatory blocker.

### 1. PDT COMPLIANCE (Pattern Day Trader: $25K minimum, 3 day trades in 5 business days)

**VERDICT: FAIL -- P0 BLOCKER**

**Regulatory Reference:** FINRA Rule 4210(f)(8)(B)(ii) -- Pattern Day Trader designation

**What the plan says:**
- Milestone 4 (`paper_validation_service.py`): "Gates: ... PDT compliant" (line 156)
- Existing code: `get_account_info()` returns `pattern_day_trader` flag from Alpaca (line 293 of `alpaca.py`)
- Existing tests: `TestComplianceEnginePDT` in `test_wealth_llm_service.py` checks advisory-level PDT warnings

**What is MISSING:**
1. **No PDT counter in the execution path.** The `TradingService.validate_trade()` method (trading.py lines 121-360) has zero PDT logic. It checks circuit breakers, position sizing, and order validation, but never counts day trades or checks the $25K threshold before execution.
2. **No day trade counting infrastructure.** The plan does not define a `day_trade_counter` service, table, or column. You need a rolling 5-business-day window tracker that counts round-trip same-day trades (buy+sell or sell+buy of same symbol intraday).
3. **The plan only gates PDT during paper validation (M4), not during live execution.** Paper validation is a one-time 30-day check. PDT must be enforced on EVERY TRADE, in real-time, at the 10ms compliance kill-switch level.
4. **Alpaca's `pattern_day_trader` flag is reactive, not preventive.** By the time Alpaca flags the account, the violation has already occurred. The platform must preemptively block the 4th day trade if equity < $25K.

**FINRA Rule 4210(f)(8)(B)(ii) requires:**
- Accounts with equity < $25,000 are limited to 3 day trades in any 5-business-day rolling period
- The 4th day trade triggers PDT designation, account restriction, and potential 90-day lock
- Cash accounts avoid PDT but require T+1 settlement (T+2 was eliminated in May 2024 per SEC Rule 15c6-1(a) amendment)

**Remediation:**
- Add `day_trade_count` tracking to the Trade model or a new `day_trade_tracker` table
- Wire a PDT pre-trade check into `TradingService.validate_trade()` that runs BEFORE order submission
- Query: count trades where `trade_type=SELL` AND `created_at` is same day as a corresponding BUY for the same symbol, within a 5-business-day rolling window
- HARD_STOP if count >= 3 AND account equity < $25,000
- Log the kill with audit token per Protocol 1

### 2. WASH SALE TRACKING (61-day rolling window)

**VERDICT: CONDITIONAL PASS -- P1 CRITICAL (window calculation is correct, but implementation is underspecified)**

**Regulatory Reference:** IRC Section 1091 (Wash Sale Rule); IRS Publication 550

**What the plan says (line 159):**
> Extend Trade model: `is_wash_sale`, `wash_sale_adjustment`, `cost_basis_method`, `lot_id`, `holding_period` (61-day window)

**Analysis:**
- **The 61-day window is CORRECT.** IRC Section 1091 defines the wash sale period as 30 days before the sale + the sale date + 30 days after = 61 calendar days total. The plan correctly identifies this.
- **"Substantially identical security" is not addressed.** The wash sale rule applies not only to the exact same security but to "substantially identical" securities. For equities, this means: buying a call option on the same stock, buying shares in the same company through a different account, or buying a mutual fund that tracks the same index. The plan must define what constitutes "substantially identical" in the context of this platform.

**What is MISSING:**
1. **No wash sale detection service.** The plan lists model fields but no detection logic. There is no `check_wash_sale()` function described in any milestone.
2. **No pre-trade wash sale warning.** If a user sells AAPL at a loss and then tries to buy AAPL within 30 days, the system should warn (or block for the autonomous bot) that this will trigger a wash sale, disallowing the loss deduction and adding it to the cost basis of the new shares.
3. **Cross-account detection is not mentioned.** IRS Publication 550 states wash sales apply "across all your accounts, including IRAs." If the user has positions outside this platform, the system should at minimum surface a disclaimer.
4. **No mechanism for adjusting the cost basis of the replacement shares.** When a wash sale occurs, the disallowed loss must be added to the cost basis of the replacement security. The plan lists `wash_sale_adjustment` as a field but does not describe the computation.

**Remediation:**
- Create `backend/app/services/wash_sale_service.py` with detection logic
- Pre-trade check: before any BUY, query sells-at-a-loss for the same symbol within the past 30 days; before any SELL-at-a-loss, query future buys within the next 30 days (for autonomous bot, since it controls both sides)
- Adjust cost basis on replacement lots per IRS rules
- Add cross-account disclaimer to user-facing output

### 3. CRYPTO WASH SALES (IRS 2026 treatment)

**VERDICT: FAIL -- P0 BLOCKER**

**Regulatory Reference:** IRC Section 1091 as amended by the Infrastructure Investment and Jobs Act (IIJA) of 2021, effective for tax years beginning after December 31, 2024 (i.e., 2025 tax year onward); IRS Notice 2024-56

**What the plan says:**
- The plan mentions crypto tax tracking (line 160): `is_crypto`, `exchange_name`, `transaction_fee`, `cost_basis_usd`, `realized_gain_loss_usd`, `is_staking_reward`, `staking_income_usd`, "Support FIFO/LIFO/Average Cost methods"
- It does NOT explicitly state that wash sale rules apply to crypto

**Critical Regulatory Context:**
- **Prior to 2025:** Crypto was exempt from wash sale rules because the IRS classified it as "property" (IRS Notice 2014-21), not a "security" or "stock" under IRC Section 1091. Traders could sell crypto at a loss and immediately rebuy to harvest the tax loss.
- **As of January 1, 2025:** The IIJA amendments brought "digital assets" (defined in IRC Section 6045(g)(3)(D)) under the wash sale umbrella. This means for tax year 2025 and beyond (which includes 2026), crypto IS subject to the same 61-day wash sale window as equities.
- **The plan does NOT account for this.** The crypto tax tracking fields (line 160) are listed separately from the wash sale fields (line 159), implying the developer may treat them as independent concerns.

**What is MISSING:**
1. **No explicit statement that `is_wash_sale` applies to crypto trades.** The wash sale fields and crypto tax fields are in separate bullet points.
2. **No "substantially identical" definition for crypto.** Is BTC on Alpaca "substantially identical" to wrapped BTC (WBTC)? The IRS has not issued definitive guidance, but the plan should take a conservative position.
3. **The existing training data (expand_training_data.py line 94) says: "Can't claim loss if you buy substantially identical security within 30 days before or after sale."** This is correct for equities but the training data does NOT mention crypto wash sales. The elson-finance-14b model may give WRONG advice on crypto wash sales if not retrained.

**Remediation:**
- Merge wash sale tracking and crypto tax tracking into a single unified service
- Apply the same 61-day window to ALL asset classes: equities, ETFs, options, AND crypto
- Add a `asset_class` parameter to the wash sale check that does NOT exempt crypto
- Retrain or fine-tune elson-finance-14b with updated crypto wash sale rules for 2025/2026 tax years
- Add disclaimer: "As of 2025, the IRS applies wash sale rules to digital assets. Consult a tax professional."

### 4. OPTIONS ORDER TYPES (BUY_TO_OPEN, SELL_TO_CLOSE, etc.)

**VERDICT: CONDITIONAL PASS -- P1 (Alpaca supports them, but the plan's adapter is underspecified)**

**Regulatory Reference:** OCC (Options Clearing Corporation) order handling rules; FINRA Rule 2360 (Options)

**What the plan says (line 274):**
> `OptionsOrder`: contract_id, side (BUY_TO_OPEN, SELL_TO_CLOSE, etc.), order_type, quantity, price, status

**Alpaca Options API Capabilities (verified against Alpaca docs as of 2025):**
- Alpaca's Trading API v2 supports options trading with these order sides: `buy_to_open`, `sell_to_open`, `buy_to_close`, `sell_to_close`
- Supported order types for options: `market`, `limit`, `stop`, `stop_limit`
- Supported order classes: `simple`, `oco` (One-Cancels-Other), `oto` (One-Triggers-Other)
- Multi-leg strategies are supported via `legs[]` parameter on order submission
- Alpaca requires options approval (Tier 1 or Tier 2) -- the existing `get_account_features()` already reads `options_approved_level`

**What is correct:**
- The plan correctly identifies the four options order sides
- The strategy builder (line 279) covers standard strategies: covered call, protective put, spread, straddle, strangle, iron condor

**What is MISSING:**
1. **No options approval level gate in the execution path.** The plan does not describe checking `options_approved_level` before submitting an options order. Tier 1 allows covered calls and cash-secured puts only. Tier 2 allows spreads. Naked options require Tier 3+ (which Alpaca does not offer). The execution path must enforce these tier limits.
2. **No exercise/assignment handling.** Options can be exercised or assigned. The plan does not address how the system handles ITM options at expiration, early assignment on short positions, or cash settlement vs. physical delivery.
3. **No buying power check specific to options.** Options have different margin requirements than equities. Covered calls need the underlying shares. Cash-secured puts need the cash to cover assignment. Spreads need max-loss as collateral. The current `validate_trade()` does not compute options margin.

**Remediation:**
- Add an options approval tier check to the pre-trade compliance gate
- Define exercise/assignment webhook handling (Alpaca sends these via account activities endpoint)
- Implement options-specific buying power validation

**What the plan says (line 340):**

**Analysis:**
- This is a sound architectural decision. Alpaca supports equities, crypto, and options under a single API key and account. Using a single broker adapter reduces:

The plan's circuit breaker (line 93: "Alpaca exchange down -> Level 4 EMERGENCY -> Flatten all crypto") partially addresses this, but there is no equity-specific outage response. The Guardian Sniper's Protocol 3 (Sniper Scan) mentions multi-venue execution across Binance, Alpaca, and IBKR for optimal fills, but for an initial single-broker implementation, this is acceptable. Multi-venue routing can be added in the Go microservices phase.

### 7. STABLECOIN EXPOSURE CAP AT 30%

**VERDICT: CONDITIONAL PASS -- P2 (appropriate threshold, but rationale needs documentation)**

**What the plan says (lines 67, 93, 341):**
> `_check_stablecoin_exposure()` -- USDC/USDT capped at 30% (counterparty risk)
> "Stablecoin cap 30% | Counterparty risk (Alpaca custody, USDC/USDT solvency)"

**Analysis:**
- **30% is a reasonable cap** from a counterparty risk perspective. The rationale:
  - USDC (Circle): Backed by US Treasuries and cash. Monthly third-party attestations. Lower risk but not zero.
  - USDT (Tether): Historically opaque reserves. Higher counterparty risk.
  - Alpaca custody: Stablecoins held at Alpaca are not SIPC-insured (SIPC covers securities, not crypto). Alpaca uses third-party custodians.
  - A 30% cap means no more than 30% of total portfolio value is exposed to stablecoin de-peg or custodian insolvency risk.
- **There is no regulatory mandate for a specific stablecoin concentration limit.** This is a risk management decision, not a compliance requirement. However, the SEC has signaled interest in stablecoin regulation, and the Lummis-Gillibrand Payment Stablecoin Act (if enacted) may impose reserve requirements.

**What should be improved:**
1. **USDC and USDT should have different risk weights.** Treating them identically at 30% combined is too generous for USDT. Consider: USDC cap 30%, USDT cap 10%, combined cap 30%.
2. **The 30% trigger is a Level 2 CAUTION in the circuit breaker (line 93).** This means it only reduces by liquidating excess -- it does NOT prevent the initial trade that would breach 30%. The pre-trade check should HARD_STOP any trade that would push stablecoin exposure above 30%.
3. **No de-peg monitoring.** If USDC trades at $0.97 (as it did during SVB in March 2023), the system should escalate to Level 3/4 immediately, not wait for the 30% threshold.

### 8. FIFO/LIFO/AVERAGE COST (Tax lot accounting)

**VERDICT: FAIL -- P1 CRITICAL**

**Regulatory Reference:** IRC Section 1012 (Basis of Property); IRS Publication 550; SEC Rule 10b-10 (Trade Confirmation Requirements)

**What the plan says (line 160):**
> "Support FIFO/LIFO/Average Cost methods"

**What currently exists:**
- The `_process_buy_trade()` in `trading.py` (lines 760-803) uses a simple weighted average cost: `holding.average_cost = new_total_cost / float(holding.quantity)`. This is average cost ONLY.
- There is NO tax lot tracking. When you sell, the system does not know WHICH shares you are selling (first-in, last-in, or specific identification).
- The `Holding` model has a single `average_cost` field. There is no `lot_id`, no `acquired_date`, no per-lot cost basis.

**What is WRONG:**
1. **FIFO/LIFO/Specific ID require individual lot tracking.** You cannot implement FIFO or LIFO with a single `average_cost` field. Each purchase creates a separate tax lot with its own acquisition date, quantity, and cost basis.
2. **The plan says "extend Trade model" with `lot_id` and `cost_basis_method`** but does NOT describe a `TaxLot` table or service. A `lot_id` on the Trade is not sufficient -- you need a separate `tax_lots` table that tracks each lot's remaining quantity, acquisition date, and cost basis.
3. **Average cost is only permitted for mutual funds and certain regulated investment companies** per IRS rules. For individual stocks, the default is FIFO unless the taxpayer elects specific identification. The existing implementation (average cost for all holdings) is technically non-compliant for equity tax reporting.
4. **Holding period determination (short-term vs. long-term capital gains) requires per-lot acquisition dates.** The current model has no `acquired_at` field per lot.

**Required data model:**
```
TaxLot:
  - id (PK)
  - trade_id (FK to trades -- the buy trade that created this lot)
  - symbol
  - quantity_remaining (decremented on sell)
  - cost_basis_per_share
  - acquired_at (datetime)
  - holding_period_category (short_term | long_term, computed from acquired_at)
  - is_wash_sale_adjusted (bool)
  - adjusted_cost_basis (if wash sale applies)
```

**Remediation:**
- Create `backend/app/models/tax_lot.py` with the schema above
- Modify `_process_sell_trade()` to consume lots in FIFO order by default, with user-selectable method
- Wire lot_id into the wash sale detection service
- Add holding period computation (>1 year = long-term capital gains rate)

### 9. STAKING INCOME TRACKING (Alpaca staking support)

**VERDICT: FAIL -- P1 CRITICAL (plan includes staking fields but Alpaca does NOT support staking)**

**What the plan says (line 160):**
> `is_staking_reward`, `staking_income_usd`

**Alpaca API Reality:**
- **Alpaca does NOT offer staking services.** Alpaca is a brokerage, not a staking-as-a-service provider. Their crypto offering is limited to spot trading of supported digital assets.
- Alpaca's crypto custody is via third-party partners (currently Fireblocks for institutional, and Alpaca's own custodial solution for retail). Neither offers staking through the Alpaca API.
- There is no `staking` endpoint, no staking rewards reporting, and no staking-related fields in the Alpaca API response schema.

**Why this matters:**
- Adding `is_staking_reward` and `staking_income_usd` fields to the Trade model creates dead columns that will never be populated through Alpaca.
- If the platform ever offers staking (via a different provider), these fields would need to be on a separate `staking_rewards` table, not on the `trades` table, because staking rewards are not trades -- they are income events.
- **Tax treatment:** Staking rewards are taxable as ordinary income at the time of receipt (per IRS Revenue Ruling 2023-14). They are NOT capital gains. Putting them on the Trade model conflates two fundamentally different tax treatment categories.

**Remediation:**
- Remove `is_staking_reward` and `staking_income_usd` from the Trade model extension
- If staking is a future requirement, create a separate `staking_rewards` table with: `asset`, `reward_amount`, `fair_market_value_at_receipt`, `receipt_date`, `source_provider`
- Do NOT implement staking tracking until a staking provider is integrated -- dead code increases audit surface area for no benefit

### 10. AUDIT LOG IMMUTABILITY (BEFORE DELETE/UPDATE trigger approach)

**VERDICT: CONDITIONAL PASS -- P1 (approach is directionally correct but insufficient for SEC Rule 17a-4)**

**Regulatory Reference:** SEC Rule 17a-4 (Records to be preserved by certain exchange members, brokers, and dealers); FINRA Rule 4511 (General Requirements for Books and Records)

**What the plan says (line 75):**
> `backend/app/models/bot_audit_log.py` -- Immutable with BEFORE DELETE + BEFORE UPDATE triggers

**Analysis of the trigger approach:**

**What is CORRECT:**
- Database triggers that block DELETE and UPDATE operations on the audit log table are a valid first layer of defense
- PostgreSQL BEFORE triggers can `RAISE EXCEPTION` to prevent the operation, which is a hard block

**What is INSUFFICIENT for SEC Rule 17a-4:**
1. **Database superusers can disable triggers.** `ALTER TABLE bot_audit_log DISABLE TRIGGER ALL;` bypasses all triggers. A DBA, compromised credential, or SQL injection could circumvent this control. SEC Rule 17a-4 requires WORM (Write Once, Read Many) storage.
2. **No tamper detection.** If someone disables triggers, modifies a row, and re-enables triggers, there is no cryptographic evidence of tampering. The plan does NOT describe hash chaining (where each record's hash includes the previous record's hash, creating a blockchain-like integrity chain).
3. **No separation of duties.** The same database credentials that write audit records can also (in theory) modify the trigger. SEC Rule 17a-4(f) requires that records be stored on non-rewriteable, non-erasable media, OR in a system with automatic integrity verification.
4. **Retention period not enforced.** SEC Rule 17a-4 requires 6-year retention for most records (first 2 years in easily accessible location) and lifetime retention for certain records. The plan does not describe a retention policy or archival strategy.

**What the plan SHOULD add (per Guardian Sniper Protocol 5 -- Bitemporal Audit Log):**
- **Hash chaining:** Each audit record includes `previous_hash` (SHA-256 of the prior record). Any modification breaks the chain. A background job verifies chain integrity hourly.
- **Bitemporal timestamps:** Transaction Time (TT) and Valid Time (VT) as specified in Protocol 5
- **Read-only DB role:** The audit table should be written by a dedicated service account. The application DB user should have INSERT-only, no UPDATE/DELETE permissions at the PostgreSQL GRANT level -- this is stronger than triggers.
- **Offsite replication:** Audit records should be replicated to a separate, append-only store (e.g., GCS bucket with Object Lock / Retention Policy, or SurrealDB as specified in Protocol 5)
- **Periodic integrity attestation:** A cron job that verifies the hash chain and writes the result to a separate monitoring system

**Remediation:**
- Implement INSERT-only GRANT on the audit table for the application DB role
- Add `previous_record_hash` column and SHA-256 chaining
- Add bitemporal fields (`transaction_time`, `valid_time`)
- Replicate to GCS with retention lock for SEC Rule 17a-4 WORM compliance
- Keep the BEFORE DELETE/UPDATE triggers as defense-in-depth, not sole control
