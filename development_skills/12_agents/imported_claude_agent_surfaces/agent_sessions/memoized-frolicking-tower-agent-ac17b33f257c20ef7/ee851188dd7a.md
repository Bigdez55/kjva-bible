# M5 + M6 Frontend Feasibility Assessment — Apex Trading UI + Options Trading

## Architecture Context (Confirmed via Codebase Audit)

- Build system: CRA (react-scripts 5.0.1) — no Vite, no custom webpack
- Current JS bundle: main.b89c75d8.js = 719KB uncompressed; total JS = 6.0MB (multiple chunks)
- Existing chart system: Custom SVG LineChart (WASM-accelerated bounds), NOT recharts
- Navigation: 5-tab MobileBottomNav (dashboard/ai/invest/wealth/account) with hardcoded TabId union
- WebSocket: Single authenticated connection at /ws/market/feed; channels: market_data, orders, portfolio
- RTK Query: marketDataApi + tradingApi; no dedicated DOM or L2 endpoints exist
- No workspace libraries installed (react-mosaic, react-grid-layout, @tanstack/react-table are all absent)
- No lightweight-charts installed (v5.1.0 available on npm, 2886KB unpacked)

## 10-Point Feasibility Assessment

### 1. TradingView Lightweight Charts vs Alternatives — CONDITIONAL PASS

**Verdict: Correct choice for price charts. FAIL for the full feature set described.**

Lightweight Charts v5.1.0 is the right library for the core candlestick/line chart (free, MIT, ~50KB
gzipped, dark theme via layout options). It handles real-time candle updates natively and the
`#04070D` palette maps directly to its `layout.background.color` option.

**Hard capability gaps that kill the described feature set:**

- Drawing tools (trendlines, Fibonacci, annotations): NOT included in lightweight-charts v5. TradingView
  deliberately removed drawing tools from the open-source build. They exist only in the paid
  TradingView Advanced Charts (Widget) or the Charting Library ($). The plan lists these as M5
  deliverables — this is incorrect.

  Options to resolve:
  a) Remove drawing tools from M5 scope entirely (reduce to chart-only)
  b) Build a DIY SVG drawing layer on top of the canvas (3-5 additional dev days, fragile)
  c) Use the free TradingView Widget embed (iframe, very limited theming, breaks design system)

- Volume Profile indicator: Not built-in to lightweight-charts. Requires a custom plugin series
  (TradingView publishes open-source plugin examples but each plugin is ~1-2 day implementation).

- Ichimoku Cloud: Not built-in. Custom plugin required.

- SMA/EMA/RSI/MACD/Bollinger/ATR/Stochastic/OBV: All require manual calculation and rendering
  as custom series. The plan treats "15+ indicators" as a single deliverable. Each non-trivial
  indicator is 0.5–1 day of work. Realistic M5 scope: 5-7 indicators, not 15+.

- VWAP: Requires tick-level intraday volume data from the backend. Current /market-enhanced/historical
  endpoint returns OHLCV daily/weekly bars only. A new intraday bars endpoint is needed.

- Multi-chart view (2x2 or 3x1 grid): Feasible — lightweight-charts supports multiple independent
  chart instances. Synchronizing crosshairs across instances requires manual implementation using
  the subscribeCrosshairMove API. Add 1.5 days.

**Integration note for the existing codebase:** The custom SVG LineChart (WASM-accelerated) will be
replaced for the Trade page only. The InvestPage LineChart stays as-is. TypeScript types for
lightweight-charts are bundled in the package. No @types install needed.

### 2. react-mosaic vs react-grid-layout — FAIL on current choice; REVISED RECOMMENDATION

**Verdict: react-grid-layout is the better choice. react-mosaic has architectural friction with this codebase.**

react-mosaic analysis:
- v6.1.1, last published 2024-12-20 (maintained, MIT license confirmed)
- Unpacked: 254KB (small)
- Dependencies include `react-dnd` + 4 DnD backend packages — this adds ~400KB of DnD stack
- react-dnd has known conflicts with React 18 Concurrent Mode features (StrictMode double-renders
  cause DnD state desync in some versions)
- Bloomberg-style binary splitting layout (each panel splits into halves) — rigid, not free-form
- No built-in panel persistence (must serialize the mosaic tree manually)
- The `immutability-helper` dep is a red flag — it's a dated pattern that fights Redux Toolkit's
  Immer-based state

react-grid-layout analysis:
- v2.2.2, unpacked 1099KB (larger but self-contained)
- No react-dnd dependency — uses its own lightweight drag implementation
- Free-form drag-to-anywhere positioning (more Bloomberg-like than the forced binary split of mosaic)
- Built-in layout serialization as a simple JSON array — trivially persisted to localStorage or
  user profile API
- Responsive grid breakpoints built-in (lg/md/sm/xs/xxs) — crucial for the mobile fallback
- Used in production by Grafana, Retool, many fintech dashboards
- React 18 compatible (no Concurrent Mode conflicts)

**Recommendation:** Use `react-grid-layout`. The forced binary split of react-mosaic produces a
worse UX than a free-grid for a trading terminal — professional terminals (Bloomberg, Thinkorswim)
use free-grid positioning, not mandatory halving.

**Performance with 8+ panels:** react-grid-layout renders all panels to DOM simultaneously. With
8 panels each containing a live chart + table, the frame budget will be stressed. Mitigations:
- Render lightweight-charts instances only when their panel is visible (IntersectionObserver)
- Virtualize the order blotter and DOM table rows (windowing)
- Memo-wrap each panel component so only the changed panel re-renders on price ticks

### 4. Mobile Responsiveness for Multi-Panel Workspace — CONDITIONAL PASS

**Verdict: Feasible but requires a fundamentally different mobile layout — not a simple responsive breakpoint.**

The current app is 100% mobile-first: MobileBottomNav, MobileHeader, MobileLayout, max-width: 480px
content areas. The entire authenticated shell is named "MobileLayout." A Bloomberg-style multi-panel
workspace physically cannot render on a 375px viewport — even at 2 panels the chart becomes unusable.

**Required approach (not in the plan):**
- Viewport detection: `window.innerWidth >= 1024` → render TradingWorkspace; else render "Mobile
  Trade View" — single-panel sequential layout (chart → order entry → blotter as stacked sections)
- Mobile Trade View is a separate component, not a responsive variant of TradingWorkspace
- react-grid-layout's `cols={{ lg: 12, md: 10, sm: 6, xs: 4, xxs: 2 }}` breakpoints handle
  graceful collapse of grid items, but each panel still needs a minimum viable mobile representation
- The plan says "preset layouts (Day Trade, Swing, Options, Crypto)" — on mobile, presets should
  map to single-focus views (e.g., "Day Trade Mobile" = just chart + quick order entry)

**This is design work that must be scoped separately.** Add 3-4 days for mobile trade view design
and implementation.

### 5. Options Chain Viewer — CONDITIONAL PASS with Performance Caveats

**Verdict: Architecturally feasible. Performance requires virtualization from day one.**

An SPX options chain has ~200 strikes × ~12 expirations = 2,400 contracts. Each row shows:
delta, gamma, theta, vega, IV, bid, ask, volume, OI — 9 columns per side × 2 sides = 18 rendered
cells per row. At 200 visible rows without virtualization: 3,600 DOM nodes per chain load.

With real-time Greeks updates (even at 5-second polling), React will diff and repaint all 3,600
nodes on each tick. On a mid-range device this will drop frames.

**Required mitigations:**
- `@tanstack/react-table` with `react-virtual` (or `@tanstack/react-virtual`) for row windowing —
  only render ~30 rows in the DOM at any time
- Debounce Greeks updates to 5s minimum — no per-tick re-render
- ITM/ATM/OTM filter to reduce visible rows to ~60 by default (user can expand)
- The plan does not mention virtualization. This is a P0 requirement, not an optimization.

**Backend dependency:** Greeks at the individual contract level require either Alpaca-provided Greeks
(available on their options API) or local Black-Scholes calculation. Black-Scholes in Python is fast,
but calculating Greeks for 2,400 contracts per refresh on a Cloud Run instance with 2Gi RAM is viable.
The IV surface requires a full 3D solve — more expensive. Stage the VolatilitySurface component for
M6 Phase 2, not M6 Phase 1.

### 6. Strategy Builder P&L Diagram — CONDITIONAL PASS

**Verdict: Use recharts (already installed v3.7.0) — no new dependency needed.**

The P&L payoff diagram for options strategies is a simple 2D chart:
- X-axis: underlying price at expiration (range: current price ± 3 sigma)
- Y-axis: strategy P&L
- Lines: one per expiration scenario (today, 50% to expiry, at expiry)
- Key points: max profit, max loss, breakeven(s) annotated

recharts `ComposedChart` with `Line` series handles this well. The data is small (100-200 price
points, calculated client-side from the strategy legs — pure math, no API call).

`ReferenceLine` components mark breakevens and current price. `ReferenceArea` marks the profit zone
in `C.green` and loss zone in `C.red`. This is a 1-day implementation.

**Do NOT use a 3D library for VolatilitySurface.** The plan describes a 3D IV surface. Three.js or
react-three-fiber adds ~500KB+ to the bundle for a visualization that most users will never interact
with. Recommendation: flatten the IV surface to a 2D heat map (strike × expiration grid, color =
IV level). recharts `Cell`-colored grid achieves this. Move the 3D surface to a future sprint if
there's clear user demand.

### 7. Navigation Changes — FAIL on current approach

**Verdict: Adding "Trade" and "Options" as top-level tabs breaks the mobile nav contract.**

Current state:
- `TabId = 'dashboard' | 'ai' | 'invest' | 'wealth' | 'account'` (5 items, hardcoded)
- `MobileBottomNav` has a hardcoded `tabs` array of 5 items
- At 5 tabs the nav is already dense on 375px (each button ~76px wide)
- Adding Trade + Options = 7 tabs. At 375px viewport: 53px per button. Icon + label becomes unreadable.
  Touch target falls below the 44×44px WCAG minimum.

**Options:**

Option A — Nested navigation (recommended):
- Keep 5 bottom tabs. Rename "Invest" → "Markets" or keep "Invest."
- Inside the Invest/Markets page, add a top-level toggle: "Portfolio | Trade | Options"
- Trade and Options are sub-pages within the Invest space
- Zero navigation contract change. TabId stays as-is. No cascade.

Option B — Replace one tab:
- Remove "Wealth" tab (less frequently used) and add "Trade" tab
- Options lives inside Trade page as a sub-tab
- Requires updating `TabId` union, MobileBottomNav tabs array, MobileLayout routing
- Cascade: every page that receives `nav?: (tab: TabId) => void` prop needs type update
- Estimated impact: ~12 files need `TabId` import update (all page components)

Option C — Hamburger overflow:
- Keep 5 primary tabs, add "More" tab that opens a drawer with Trade + Options + Wealth
- Hides advanced features behind an extra tap — acceptable for power users who will learn it

**Recommendation:** Option A. It requires the least refactoring, preserves mobile UX, and Trade/
Options are naturally "within" the investing context. The type system change of Option B propagates
to too many files given that InvestPage.tsx alone is 2,061 lines.

### 9. WebSocket Requirements for DOM, Blotter, Real-time Charts — CONDITIONAL FAIL

**Verdict: The existing WebSocket supports order blotter and portfolio updates. DOM requires new backend work.**

What the existing /ws/market/feed supports (confirmed from websocketService.ts):
- `market_data` channel: real-time price ticks per symbol (used for chart updates)
- `orders:paper/live` channel: order status updates (fills, cancels, rejects) — blotter use
- `portfolio:paper/live` channel: portfolio value updates — positions panel use
- `level2` and `trade` message types are forwarded as market_data (lines 535-539)

What is MISSING for M5:
- **Depth of Market (DOM):** The plan describes a bid/ask ladder with volume bars at each price level.
  Level 2 quote data (full order book depth) is NOT available from Alpaca's free/Pro market data.
  Alpaca's SIP feed provides best bid/ask only. True Level 2 requires Alpaca's premium data subscription
  ($99/mo or $9/mo for basic) OR Polygon.io subscription.
  - Without L2 data: the DOM panel shows only 1 level deep (just best bid/ask + spread)
  - This fundamentally changes what "DOMVisualization" can display
  - The plan should note this data availability constraint explicitly

- **Real-time candle updates:** Alpaca's streaming data sends individual trades/quotes, not
  aggregated 1-minute OHLCV bars. Building live candles from tick data requires server-side
  aggregation (OHLCV accumulator per symbol per interval). This is new backend work.
  - Workaround: poll /market-enhanced/historical?period=1d every 60s for the current bar.
    This gives ~1-minute "live" candles via polling, not true streaming.
  - True streaming candles = new backend endpoint + WebSocket message type

- **WebSocket subscription scaling:** The current WS subscribes to `market_data:{symbols}` as a
  comma-joined string. With a 20-symbol watchlist + 8 chart panels each on different symbols,
  the subscription management logic in buildSubscriptionMessage() needs to handle symbol deduplication
  and batch re-subscription on workspace layout changes.

**Summary:** Order blotter (PASS), portfolio/positions updates (PASS), live price ticker for charts
(PASS via polling fallback), DOM Level 2 (FAIL — data not available without premium subscription),
true streaming candles (CONDITIONAL — polling workaround viable for initial launch).

### 10. Bundle Size Impact — CONDITIONAL PASS

**Verdict: Manageable with code splitting. Exceeds budget without it.**

Current bundle:
- main.js: 719KB (already large for a mobile-first app; ~250KB gzipped estimated)
- Total JS: 6.0MB across all chunks

New dependencies (gzipped estimates):
- `lightweight-charts` v5: ~50KB gzipped (excellent)
- `react-grid-layout` v2: ~28KB gzipped (good; react-mosaic would be ~35KB + ~120KB react-dnd stack)
- `@tanstack/react-table` v8: ~16KB gzipped (headless, no renderer)
- `react-virtual` (for options chain): ~4KB gzipped

Total new dependency addition: ~98KB gzipped. This is within acceptable budget.

**Risk: CRA code splitting.** CRA (react-scripts) does dynamic chunk splitting via `React.lazy()` +
`Suspense`. If TradingWorkspace, OptionsChainViewer, and their dependencies are lazy-loaded (not
imported in the main bundle), the 98KB new code only loads when the user navigates to Trade or
Options. This is mandatory — the current main.js is already 719KB.

**Implementation requirement:**
```typescript
// In router/MobileLayout — lazy load both new pages
const TradingWorkspace = React.lazy(() => import('../pages/TradePage'));
const OptionsPage = React.lazy(() => import('../pages/OptionsPage'));
```
Without lazy loading, recharts (already 3.7.0, ~180KB unpacked) + lightweight-charts + react-grid-
layout will merge into main.js and push it past 900KB uncompressed (>300KB gzipped), which exceeds
the 200KB gzipped target for a fast initial load.

**Note on recharts:** recharts is currently in package.json but the custom SVG LineChart is used
everywhere in the app. recharts appears unused in production code paths (it was likely carried over
before the LineChart migration). Removing recharts saves ~50KB gzipped and eliminates a dependency.
The P&L payoff diagram (Q6 above) should use the custom chart system or lightweight-charts instead.
