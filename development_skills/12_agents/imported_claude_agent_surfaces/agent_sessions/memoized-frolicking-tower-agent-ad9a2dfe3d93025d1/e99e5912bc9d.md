## SCENARIO 1: FLASH CRASH (May 6, 2010 Style)

**Event:** S&P 500 drops 9% in 5 minutes, then recovers 80% of the loss in 20 minutes.

### Causal Propagation Map
```
Market sell cascade -> ETF arbitrage failure -> Bid liquidity vacuum ->
Circuit breaker HALT -> Stub quotes fill -> Recovery bounce ->
Canceled trades (exchange level) -> Regulatory review
T+0min to T+25min total duration
```

### (a) Does the plan address it?

**Partially.** The circuit breaker system in `circuit_breaker.py` (lines 84-157) has graduated thresholds:
- Daily loss `warning` at 2%, `restricted` at 2.5%, `halt` at 3%
- Volatility `extreme` at 35% annualized (line 87)
- Position sizing drops to 3% of normal at extreme volatility (line 92)

The auto-trading loop in `auto_trading_service.py` checks circuit breakers at the top of every iteration (line 332-338), and the `TradeExecutor._validate_signal()` (lines 213-265) checks system-wide, strategy-specific, and symbol-specific breakers before every trade.

### (b) Likely Outcome

**CRITICAL GAP: The loop runs on a 60-second polling interval** (line 468: `await asyncio.sleep(60)`). In a flash crash that unfolds in 5 minutes, the bot gets at most 5 evaluation opportunities. The circuit breaker check at line 332-338 uses `circuit_breaker.check()` which is a passive check -- it only reads state, it does not actively monitor portfolio P&L in real-time.

The circuit breaker's daily loss thresholds (2%, 2.5%, 3%) are never actively computed against the live portfolio during the loop. The `trip()` method on the circuit breaker requires an external caller to invoke it with the specific loss figure -- but **nothing in the auto-trading loop actually calculates real-time portfolio drawdown and feeds it to the circuit breaker**.

**Most likely outcome:** The bot continues sleeping through the 60-second cycle. If the market drops 9% in 5 minutes and recovers in 20 minutes, the bot:
1. Would NOT flatten at the bottom (no kill switch code exists in the codebase -- grep confirms zero matches for `kill_switch`, `flatten_all`, `cancel_all`, `close_all_positions` as functional code)
2. Would NOT enter new positions during the crash (circuit breaker volatility check would likely trip if volatility data feeds are current)
3. Would survive the crash by doing nothing, purely due to the 60-second sleep acting as an accidental safeguard
4. Would miss the recovery rally for the same reason

### (c) What's Missing

1. **No kill switch implementation exists.** The design document describes mass cancel, flatten positions, halt bot, 2FA restart -- but none of this is implemented in code. The `stop_auto_trading()` method (line 224) merely cancels the asyncio task and marks the session inactive. It does NOT cancel open broker orders or flatten positions.

2. **No real-time P&L monitoring.** The circuit breaker is purely passive. Nothing computes portfolio drawdown intraday and trips the breaker. The `RiskManager.check_portfolio_risk()` method exists but is never called from the auto-trading loop.

3. **No sub-minute monitoring.** The 60-second polling interval means the fastest possible reaction is ~60 seconds. For a flash crash, you need sub-second monitoring via WebSocket price feeds. The WebSocket infrastructure exists (`routes/websocket/market_feed.py`, `services/market_streaming.py`) but is not wired into the auto-trading loop.

4. **No order cancellation on circuit breaker trip.** When the circuit breaker trips, the code waits 300 seconds (line 337). It does NOT cancel open orders that may fill at catastrophic prices during the crash.

**Resilience Rating: 3/10** -- Survives by accident (60s sleep), not by design.

## SCENARIO 2: VOLMAGEDDON (Feb 5, 2018)

**Event:** VIX spikes from 15 to 50 in a single session. XIV/SVXY inverse vol ETPs collapse >90%. Cross-asset correlation spike.

### Causal Propagation Map
```
VIX futures curve inversion -> Inverse vol ETP margin calls ->
Forced liquidation cascade -> Equity selling to meet margin ->
Correlation spike (all assets sell) -> Volatility-targeting funds deleverage ->
Multi-day aftershock via vol-of-vol
T+0 to T+3d for full propagation
```

### (a) Does the plan address it?

**The volatility regime detection exists but has latency issues.** The `VolatilityDetector` in `volatility_detector.py` classifies regimes using a 20-period lookback on realized volatility (line 29), blended with VIX data (30% weight by default, line 30). The thresholds are:
- LOW: 0-15% annualized
- NORMAL: 15-25%
- HIGH: 25-40%
- EXTREME: >40%

The circuit breaker's hysteresis requires 85% of the last 20 samples to be in the new regime before switching (lines 223-244). This is designed to prevent whipsawing but creates a **significant lag** in detecting regime change.

### (b) Likely Outcome

**The hysteresis kills response time.** A VIX spike from 15 to 50 is an instantaneous regime change. The 85%-of-20-samples hysteresis means the bot needs ~17 consecutive EXTREME readings before the circuit breaker acknowledges the new regime. At 60-second intervals, that is a minimum of **17 minutes** before the EXTREME volatility circuit breaker fully engages. During those 17 minutes:

- Position sizing remains at prior regime levels (90% of normal if in NORMAL regime)
- The correlation circuit breaker threshold (0.85, line 146) would need to be actively computed -- but nothing in the auto-trading loop computes cross-asset correlation
- Options positions have no special handling -- there is no Greeks tracking, no 1-DTE auto-close, no multi-leg order management in the current codebase (the design document mentions these but they are not implemented)

The graduated daily loss limits (2% -> 2.5% -> 3%) are reasonable for this scenario IF they were actively monitored, which they are not.

### (c) What's Missing

1. **Hysteresis is too conservative for tail events.** An "override" path should exist: if any single volatility reading exceeds 2x the EXTREME threshold (>70% annualized), bypass hysteresis and trip immediately. The current code has no such override.

2. **No correlation monitoring in the loop.** The circuit breaker has a `CORRELATION` type defined (line 25) with thresholds (lines 146-153), but nothing in the auto-trading loop or any service computes live cross-asset correlation and trips this breaker. It is dead code.

3. **No options infrastructure.** The design mentions Greeks tracking, 1-DTE auto-close, and multi-leg orders. None of this exists in code. The `TradeExecutor` creates stop-loss and take-profit orders (lines 169-174) but has no concept of options contracts, Greeks, or expiration-based logic.

4. **No VIX-conditioned position sizing.** The `VolatilityDetector` can blend VIX data, but this is never called from the auto-trading loop. The feature store concept (60s Redis TTL, multi-timeframe OHLCV) described in the design is not implemented.

**Resilience Rating: 2/10** -- Hysteresis delays response by 17+ minutes; correlation and options gaps are critical.

## SCENARIO 3: SVB COLLAPSE (March 10-13, 2023)

**Event:** Weekend bank run news. Friday close is normal. Monday open gaps banking sector down 30-60%. Bot holds bank stocks. LLM brief from Friday is stale.

### Causal Propagation Map
```
SVB deposit run (Fri after-hours) -> FDIC receivership (Sun) ->
Contagion fear: SIVB/FRC/PACW (Mon pre-market) ->
Sector gap-down at open -> Credit spread widening ->
Fed emergency lending facility (Sun night) ->
Partial recovery (Tue-Wed) but structural repricing of regional banks
T+0 (Fri PM) to T+5d for resolution
```

### (a) Does the plan address it?

**The market-hours gate inadvertently provides some protection, but the gap-down is unaddressed.** The auto-trading loop's `_is_market_open()` check (line 320) prevents trading during the weekend. The LLM daily strategy brief (via `eft_generate` integration in `_generate_ai_trading_signal`) would produce a new signal on Monday, but the signal is generated AFTER the gap has already happened.

The 1% slippage threshold in `TradeExecutor._create_order()` (line 147) would prevent new orders if the current price deviates >1% from the signal price. This is a reasonable safeguard but is calibrated for normal conditions, not a 30% gap.

### (b) Likely Outcome

1. **Bot holds through the gap.** There is no pre-market monitoring, no weekend news scanning, and no mechanism to react between Friday close and Monday open.
2. **Monday's LLM signal would likely say SELL** (if the model is functioning correctly), but by then the position has already taken a 30-60% drawdown.
3. **The rule-based fallback strategies** (RSI, MACD, Bollinger, etc.) would likely generate oversold buy signals on the gap-down, potentially buying MORE of a fundamentally impaired asset.
4. **No sector concentration check.** If the bot's autonomous scan selected multiple banking stocks (which it easily could -- they would score high on the RSI deviation metric in `_scan_market_for_candidates`), the portfolio could be heavily concentrated in the affected sector.

### (c) What's Missing

1. **No pre-market/after-hours monitoring.** The bot sleeps when the market is closed. No pre-market quote check exists. A 30% gap-down would be discovered only after the market opens and the next 60-second cycle fires.

2. **No weekend news integration.** The LLM brief is generated per-iteration, not on a scheduled overnight/weekend basis. There is no mechanism to run a news scan on Sunday evening and adjust positions pre-market.

3. **No sector concentration limits.** The `_scan_market_for_candidates` function (lines 703-778) selects the top 20 stocks by signal score with no sector diversification constraint. In a sector rotation environment, it could load up on correlated names.

4. **No gap-down detection.** When the market opens, the bot's first iteration would see a massive price change, but there is no specific "gap detection" logic that would trigger an emergency review of all open positions.

5. **Rule-based strategies actively hurt.** Mean-reversion strategies (RSI reversion, Bollinger bands) would see the gap as an oversold opportunity and generate BUY signals -- exactly the wrong action for a fundamental credit event.

**Resilience Rating: 2/10** -- Complete blind spot for overnight/weekend risk with no sector concentration guards.

## SCENARIO 4: FED RATE DECISION

**Event:** Scheduled FOMC announcement at 2:00 PM ET. Market whipsaws 200bps in both directions within 30 minutes.

### Causal Propagation Map
```
FOMC statement release (2:00 PM) -> Initial algo-driven spike ->
Headline parsing (rate decision) -> Second move (dot plot interpretation) ->
Press conference (2:30 PM) -> Third move (Chair's tone/guidance) ->
Settling into new regime (3:00-3:30 PM)
T+0 to T+90min, with 3-4 directional reversals
```

### (a) Does the plan address it?

**Minimally.** The auto-trading loop has no concept of an economic calendar. It treats every trading minute equally. The circuit breaker's volatility detection would eventually register elevated volatility, but the 85% hysteresis and 60-second polling interval mean the bot would be generating and executing signals through the whipsaw period.

### (b) Likely Outcome

1. **The bot generates signals during the whipsaw.** At 2:01 PM, momentum strategies fire BUY on the initial spike. At 2:05 PM, mean-reversion strategies fire SELL on the reversal. At 2:10 PM, another reversal triggers another signal. Each signal passes through the `_validate_signal` check (which does not know about scheduled events).

2. **The 1% slippage check provides some protection** -- rapid price movement would cause slippage > 1%, rejecting many orders. But this is not guaranteed.

3. **PDT risk is elevated.** Multiple round-trips in 30 minutes could trigger Pattern Day Trader classification. The design mentions PDT tracking (rolling 5-day window, block 4th trade if account <$25K) but **there is no PDT tracking in the codebase**. The grep for `PDT`, `pattern_day`, `day_trade`, `pdt_count` found matches only in `alpaca.py` (Alpaca's own PDT protection), `trading.py`, `wealth_advisory.py`, `portfolio.py`, `trading.py`, and `risk_management.py` -- but these are references in schemas and advisory text, not enforced trading logic.

4. **The AI signal (elson-finance-14b) might actually help here** if it recognizes an FOMC day and outputs HOLD with rationale. But this depends on the model's training data including FOMC-aware scenarios, which is not guaranteed.

### (c) What's Missing

1. **No economic calendar integration.** The bot should have a list of scheduled macro events (FOMC, CPI, NFP, etc.) and automatically enter a restricted trading mode 30 minutes before and 30 minutes after. This is standard practice and entirely absent.

2. **No whipsaw detection.** If the bot detects 3+ signal reversals within 30 minutes for the same symbol, it should auto-halt that symbol. No such logic exists.

3. **No PDT enforcement in the bot.** Alpaca may enforce PDT at the broker level, but the bot should self-enforce to avoid getting flagged. No PDT counting or blocking logic exists in `auto_trading_service.py` or `trade_executor.py`.

4. **No event-driven risk mode.** The design's "regime detection: HMM (low/high vol), ADX (trending/mean-reverting), VIX (risk-on/off/crisis)" is not wired into the auto-trading loop. The `MarketRegimeDetector` class exists but is never instantiated or called during live trading.

**Resilience Rating: 3/10** -- Slippage check provides minor protection; no scheduled-event awareness exists.

## SCENARIO 5: EARNINGS SURPRISE

**Event:** Company reports after hours, stock gaps 20% on earnings. Bot holds a position. Pre-market liquidity is thin.

### Causal Propagation Map
```
Earnings release (4:05 PM ET) -> After-hours price discovery ->
Analyst revisions (evening) -> Pre-market gap (next day 4:00-9:30 AM) ->
Open auction (9:30 AM) -> Institutional flow (9:30-10:30 AM) ->
Sector sympathy moves -> New price equilibrium
T+0 (4PM) to T+1d (10:30 AM next day)
```

### (a) Does the plan address it?

**No.** The bot does not trade after-hours or pre-market. The `_is_market_open()` check gates all trading to regular hours. There is no earnings calendar integration.

### (b) Likely Outcome

1. **The bot holds through the gap.** For a positive 20% gap, this is fine -- the bot captured upside by doing nothing. For a negative 20% gap, the portfolio takes a full hit.

2. **Day-trade auto-flatten at 3:55 PM ET** (described in the design) does not exist in code. The `_is_market_open()` method only checks if the market is currently open, not if it is close to closing. There is no 3:55 PM flatten logic.

3. **The next morning's signal** would be generated on the gapped price. Momentum strategies would see the gap as a new trend; mean-reversion strategies would see it as an extreme deviation. Conflicting signals with no earnings-awareness.

### (c) What's Missing

1. **No earnings calendar.** The bot should know when held positions have upcoming earnings and either (a) reduce position size pre-earnings, (b) exit entirely, or (c) enter a special "earnings hold" mode with tighter stops.

2. **No 3:55 PM auto-flatten.** The design specifies this for day trades, but it is not implemented. The bot does not distinguish between day trades and swing positions.

3. **No pre-market price check.** Before the first iteration on market open, the bot should check pre-market prices for all held positions and flag any that have gapped significantly. No such check exists.

4. **No after-hours order capability.** Alpaca supports after-hours trading. The bot could use this for protective exits on catastrophic earnings, but the `_is_market_open()` gate prevents all after-hours activity.

**Resilience Rating: 2/10** -- Passive holding through gaps with no earnings awareness or pre-market checks.

## SCENARIO 6: LIQUIDITY VACUUM

**Event:** Low-volume stock's bid-ask spread widens to 5%. Bot tries to execute VWAP strategy.

### Causal Propagation Map
```
Low volume environment -> Market maker widens spread ->
VWAP algo attempts to participate -> Fills at unfavorable prices ->
Market impact exceeds signal alpha -> Strategy produces negative edge
Single-position, T+0 to T+1d
```

### (a) Does the plan address it?

**Partially.** The `VWAPExecutionStrategy` (lines 24-35 of `vwap_strategy.py`) has a `participation_rate` of 10% of volume and a `max_slice_deviation` of 20%. These are designed to prevent excessive market impact.

The `TradeExecutor._create_order()` has a 1% slippage threshold (line 147) that would reject orders with excessive price movement.

The circuit breaker has a `LIQUIDITY` type defined (line 27) but **no code ever trips this breaker type**. It is a dead enum value.

### (b) Likely Outcome

1. **The 1% slippage check would catch a 5% spread** -- the current price at execution time would differ significantly from the signal price, likely exceeding 1%. The order would be rejected.

2. **The VWAP strategy's participation rate** would limit individual slice sizes, but the strategy does not monitor the bid-ask spread directly. It relies on volume-based scheduling, not spread-based abort conditions.

3. **The autonomous market scanner** (`_scan_market_for_candidates`) queries up to 150 assets and selects the top 20 by signal score. It does NOT filter by average daily volume or typical spread width. Low-volume, illiquid stocks can be selected.

### (c) What's Missing

1. **No liquidity pre-filter in market scanner.** The `_scan_market_for_candidates` should filter out assets with average daily volume below a threshold (e.g., <100K shares/day) and typical spread > 0.5%. No such filter exists.

2. **No spread monitoring.** The bot should check the current bid-ask spread before submitting any order and abort if the spread exceeds a configurable threshold (e.g., 0.5% for equities). The `TradeExecutor` does not check spread -- it checks slippage from signal price, which is different.

3. **Liquidity circuit breaker is dead code.** `CircuitBreakerType.LIQUIDITY` is defined but never used. Nothing monitors market depth or order book conditions.

4. **No market impact model.** The design references VWAP and implementation shortfall, but the VWAP strategy does not model its own expected market impact. It simply slices orders mechanically.

**Resilience Rating: 5/10** -- Slippage check provides reasonable protection, but no proactive liquidity screening.

## SCENARIO 7: API OUTAGE

**Event:** Alpaca API is intermittent for 2 hours. WebSocket drops. Bot has open positions.

### Causal Propagation Map
```
API timeout/error -> Market data feed stale ->
Strategy generates signal on stale data ->
Order submission fails -> Retry loop exhausts ->
Open positions unmonitored ->
Stop-loss orders may still be live at Alpaca (broker-side)
T+0 to T+2h
```

### (a) Does the plan address it?

**Yes, partially.** Several mechanisms exist:

- The circuit breaker has `API_FAILURE` type with a 3-consecutive-failure threshold and 5-minute auto-reset (lines 142-145).
- The `TradeExecutor._create_order()` has a retry loop with `max_retries=3` and `retry_delay=1.0` (lines 139-183).
- The `AlpacaBroker` uses the `ApiBaseBroker` base class which has timeout handling (30-second timeout, line 77 of `alpaca.py`).
- The circuit breaker's `EXECUTION` type trips at 30% order failure rate with graduated responses (lines 133-139).

### (b) Likely Outcome

1. **Market data calls fail.** `cls._market_data_service.get_quote(symbol)` returns None or throws. The `_process_strategy` method checks `if not market_data` at line 501 and returns early (GATE 5). **This is correct behavior.**

2. **After 3 consecutive API failures,** the API_FAILURE circuit breaker trips (if something invokes `trip()` -- but **nothing in the auto-trading loop counts API failures and trips this breaker**). The execution failure circuit breaker would need someone to compute the failure rate and trip it.

3. **The main loop would continue cycling every 60 seconds**, calling `get_quote()`, getting errors or None, hitting GATE 5, and sleeping again. This is actually safe -- the bot does nothing.

4. **Open positions remain at the broker** with their stop-loss/take-profit orders intact (if submitted as OCO/bracket orders to Alpaca). This is broker-managed and survives API outages.

5. **WebSocket reconnection** is handled in `market_streaming.py` and `market_data_streaming_enhanced.py`, but the auto-trading loop does NOT use WebSockets -- it polls via `get_quote()` every 60 seconds. So WebSocket drops are irrelevant to the trading bot.

### (c) What's Missing

1. **No API failure counting in the loop.** The circuit breaker's `API_FAILURE` type exists but is never actively tripped by the auto-trading loop. If `get_quote()` fails, the loop just skips the iteration. It should count consecutive failures and trip the breaker after 3.

2. **No alert/notification system.** When the bot detects sustained API failure, it should notify the user. No alerting infrastructure exists in the auto-trading service.

3. **No "last known good" data cache.** If the API fails, the bot has no fallback to cached data. It simply skips. This is safe but means the bot is fully dark during outages.

4. **No position health check on recovery.** When the API comes back, the bot should immediately check all open positions for status changes. The current loop just resumes normal 60-second cycling.

5. **Broker-side stop-losses are NOT guaranteed.** The `TradeExecutor` creates stop-loss orders (line 170: `self._create_stop_loss_order(order, stop_loss)`), but this method's implementation likely submits a separate order to Alpaca. If the main order filled but the stop-loss submission failed due to API issues, the position would be unprotected.

**Resilience Rating: 6/10** -- Fails safe by doing nothing, but no active failure detection or recovery logic.

## SCENARIO 8: CORRELATION SHOCK

**Event:** Multiple positions that appeared uncorrelated suddenly move together (e.g., March 2020 COVID crash, December 2018 rate scare). Portfolio drawdown is faster than single-position circuit breaker can catch.

### Causal Propagation Map
```
Systemic risk event -> All correlations -> 1.0 ->
Portfolio drawdown = sum of individual drawdowns (no diversification benefit) ->
Individual position stops may not trip (each position -1.5%, but portfolio -7.5% with 5 positions) ->
Daily loss limit exceeded BEFORE any single-position trigger fires
T+0 to T+1d for initial correlation shock, T+3d for regime confirmation
```

### (a) Does the plan address it?

**The circuit breaker architecture addresses it in theory, but the implementation is hollow.** The `CircuitBreakerType.CORRELATION` exists (line 25) with graduated thresholds at 75%/80%/85% (lines 146-153) and a 60-minute auto-reset. The daily portfolio loss limits (2%/2.5%/3%) should catch the aggregate drawdown.

BUT: Neither the correlation breaker NOR the daily loss breaker is actively monitored. They are passive data structures waiting to be tripped by external callers that do not exist.

### (b) Likely Outcome

1. **The bot continues operating normally** because no code computes cross-asset correlation or portfolio-level drawdown in real-time.

2. **Individual strategies generate signals independently.** If all 5 correlated positions are dropping, momentum strategies might generate SELL signals (correctly), but mean-reversion strategies might generate BUY signals (incorrectly, treating each position as independent).

3. **The portfolio drawdown compounds** because the bot has no portfolio-level risk view. Each strategy operates in isolation. The `RiskManager.check_portfolio_risk()` method (in `risk_manager.py` line 44) exists and checks concentration and drawdown limits, but it is never called from the auto-trading loop.

4. **Position sizing does not account for correlation.** The `calculate_position_size` method uses individual signal strength and individual volatility, not portfolio-level correlation-adjusted sizing.

### (c) What's Missing

1. **No live correlation matrix computation.** The bot should compute rolling pairwise correlation for all held positions every cycle and trip the CORRELATION circuit breaker when the average pairwise correlation exceeds 0.75. No such computation exists.

2. **No portfolio-level drawdown monitoring.** The `RiskManager.check_portfolio_risk()` exists but is never called during auto-trading. This is the single most dangerous gap in the system.

3. **No portfolio-level circuit breaker integration.** The auto-trading loop checks circuit breakers (line 332) but only the system-level check. It does not pass portfolio-level metrics to the circuit breaker for graduated response.

4. **No correlation-adjusted position sizing.** When correlation is high, total portfolio risk is much larger than the sum of individual position risks. The `calculate_position_size` method should reduce new position sizes when average portfolio correlation exceeds a threshold.

5. **No "all positions" risk aggregation.** The loop processes each strategy independently. There is no per-cycle step that says "before processing any strategies, check: what is my total portfolio exposure, total portfolio drawdown, and portfolio-level correlation? If any exceed limits, stop all trading."

**Resilience Rating: 1/10** -- This is the most dangerous scenario. The infrastructure exists in dead code, but zero integration with the live trading loop.
