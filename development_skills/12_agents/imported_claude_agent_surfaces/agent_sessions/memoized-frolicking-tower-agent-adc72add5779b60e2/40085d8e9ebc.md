## FINDINGS

### Item 1: Kill Switch 2FA Gate
STATUS: CONDITIONAL PASS with P1 findings

FINDING 1a - P1: Kill switch endpoint does NOT YET EXIST
The plan specifies `POST /auto-trading/kill-switch` with pyotp gate. This endpoint
is not in auto_trading.py today. pyotp IS already imported and working in
two_factor.py (verify_totp). The infra is there, but the endpoint must be built.
Risk: Without the kill switch endpoint, the only way to halt the bot is via
`POST /auto-trading/stop` which has no 2FA gate. A compromised JWT would allow
full bot shutdown without 2FA.

FINDING 1b - P1: kill switch useless if Cloud Run is down
The plan has no out-of-band kill mechanism. If Cloud Run is unreachable:
- The bot loop runs in the container — killing the container kills the loop.
- BUT: if the container crashes and restarts, the session is marked is_active=True
  and the loop RESTORES itself (main.py lifespan _restore_trading_sessions()).
- A container restart with a bad circuit breaker state = bot re-enters market.
Mitigation required: Add a `KILL_SWITCH_OVERRIDE` secret in Secret Manager.
On container start, if that secret is "true", skip session restore entirely.
Secondary: Cloud Run min-instances=0 override via gcloud (manually scale to 0
during emergency) should be documented in runbook.

FINDING 1c - P2: pyotp verify_totp() has no replay protection when Redis is unavailable
The existing verify_totp() in two_factor.py does not record used TOTP codes.
A TOTP code is valid for 30 seconds. If an attacker intercepts one (e.g., over
shoulder), they have a 30-second window to replay it. Standard mitigation: store
used JTI/token fingerprint in Redis with 30s TTL. Since Redis is currently
disabled in production, this is unmitigated.

### Item 4: Circuit Breaker DB Migration
STATUS: FAIL — Data loss risk during migration

FINDING 4a - P1: No migration plan for existing circuit_breaker_status.json
The file backend/app/trading_engine/engine/circuit_breaker.py line 160 sets:
  self.status_file = "circuit_breaker_status.json"
  self._load_status()  # reads from JSON file on init

When the new bot_risk_event DB table is deployed:
1. Any open circuit breakers in the JSON file will NOT be loaded from DB.
2. The bot could restart thinking all circuit breakers are CLOSED when they
   were actually OPEN, immediately resuming trading.
Migration sequence required:
  Step 1: Deploy new bot_risk_event table (Alembic migration).
  Step 2: Run one-time migration script: read JSON → insert into bot_risk_event.
  Step 3: Deploy new circuit_breaker.py that reads from DB.
  Step 4: Delete circuit_breaker_status.json from container.

FINDING 4b - P1: circuit_breaker_status.json is a MUTABLE FILE in the container
Cloud Run containers are ephemeral. The JSON file is written to the container
filesystem, which is destroyed on restart. This means ALL circuit breaker state
is lost on every container restart today. The DB migration directly fixes this,
but the existing behavior is already a live safety gap.
Current blast radius: Bot restarts after crash → all circuit breakers reset to
CLOSED → trading resumes without safety checks. This has been live since
the auto-trading feature was deployed.

### Item 5: Shutdown Hook Reliability on Cloud Run SIGTERM
STATUS: CONDITIONAL PASS with P2 finding

FINDING 5a - P2: Current lifespan shutdown does NOT mark sessions inactive
Reviewing main.py lifespan shutdown block (lines 109-118):
The current shutdown only stops the market streaming service. There is NO code
to mark AutoTradingSession.is_active = False on shutdown. The plan adds this,
but it does not currently exist.
On Cloud Run SIGTERM → graceful shutdown window is 10 seconds (configurable
up to 300s via --timeout in the service spec, which is set to 300s).
10 seconds is the DEFAULT SIGTERM grace period for Cloud Run, separate from
the request timeout. Cloud Run sends SIGTERM then SIGKILL after the grace period.

FINDING 5b - P2: Async shutdown with DB write has race condition
When SIGTERM arrives, in-flight trading loop iterations may be mid-execution.
The shutdown hook must:
  1. Signal the loop to stop (asyncio Event or flag).
  2. Wait for the current iteration to complete (with timeout).
  3. Cancel all open orders (broker.cancel_all_orders()).
  4. Mark session is_active=False in DB.
  5. Exit.
If step 3 (cancel orders) takes longer than Cloud Run's grace period,
SIGKILL fires and orders remain open. Mitigation: Set Cloud Run
--timeout=300s (already set) AND configure the graceful shutdown timeout
in uvicorn to 295s to allow cleanup before SIGKILL.

FINDING 5c - P1: Auto-session restore on cold start may re-activate session
during a scheduled maintenance window
If Cloud Run is redeployed (rolling update) while the bot is active:
new instance starts → restores session → starts trading
old instance receives SIGTERM → (does not yet mark inactive) → dies
Two concurrent bot instances may exist briefly, both trading the same portfolio.
The fix is the session restore guard: check if bot is already running before
calling start_auto_trading(). The existing code in main.py line 91 does check
for "already running?" but only within the same process. With multiple instances,
the DB row shows is_active=True and both instances will attempt to start.
Mitigation: Add DB-level advisory lock (PostgreSQL pg_advisory_lock) or a
"claimed_by" instance_id column to AutoTradingSession before session restore.

### Item 6: Immutable Audit Log — BEFORE DELETE/UPDATE Triggers
STATUS: PASS with P2 configuration notes

FINDING 6a - PASS: Cloud SQL PostgreSQL DOES support triggers
Cloud SQL for PostgreSQL (which this project uses — elson-postgres instance)
supports CREATE TRIGGER and CREATE FUNCTION (PL/pgSQL). BEFORE DELETE and
BEFORE UPDATE triggers that RAISE EXCEPTION are valid and enforceable.
Example implementation that the plan should use:
```sql
CREATE OR REPLACE FUNCTION prevent_audit_log_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'Audit log records are immutable: % on %', TG_OP, TG_TABLE_NAME;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER bot_audit_log_immutable
  BEFORE UPDATE OR DELETE ON bot_audit_logs
  FOR EACH ROW EXECUTE FUNCTION prevent_audit_log_mutation();
```

FINDING 6b - P2: Triggers must be in Alembic migration, not just SQLAlchemy model
SQLAlchemy create_all() does NOT create triggers. The trigger DDL must be
in the Alembic migration file for add_bot_safety_tables using op.execute():
```python
op.execute("""
  CREATE OR REPLACE FUNCTION prevent_audit_log_mutation() ...
""")
op.execute("""
  CREATE TRIGGER bot_audit_log_immutable ...
""")
```
And the downgrade() must DROP the trigger and function explicitly.

FINDING 6c - P2: Trigger does not protect against TRUNCATE
PostgreSQL row-level triggers do NOT fire on TRUNCATE. Add a statement-level
TRUNCATE trigger as well, or revoke TRUNCATE privilege from the application
service account:
```sql
REVOKE TRUNCATE ON bot_audit_logs FROM elson_app_user;
```

FINDING 7b - P0: New model files in M1 (bot_performance, bot_risk_event, bot_audit_log)

### Item 9: VPC Connector Firewall (0.0.0.0/0)
STATUS: FAIL — Open firewall is an unacceptable Zero Trust violation

FINDING 9a - P1: vllm-connector VPC firewall allows 0.0.0.0/0 ingress
The vllm-connector (e2-micro, 10.8.0.0/28) currently has an open ingress rule.
Cloud Run uses the VPC connector to reach the L4 VM (10.138.0.4:8000).
With 0.0.0.0/0 allowed:
  - Any resource in the VPC (or project) can reach port 8000 on the L4 VM.
  - The vLLM API has no authentication (standard vLLM default config).
  - Blast radius: Any compromised service in the project can query the model
    or inject prompts directly, bypassing all application-layer guardrails.

Required fix BEFORE any new endpoints that add crypto/options execution:
```bash
# Remove the open rule and replace with source-specific rule
gcloud compute firewall-rules create allow-vllm-from-connector \
  --network=default \
  --direction=INGRESS \
  --action=ALLOW \
  --rules=tcp:8000 \
  --source-ranges=10.8.0.0/28 \
  --target-tags=vllm-server \
  --priority=1000

# Add network tag to the L4 VM
gcloud compute instances add-tags elson-dvora-training-l4-2 \
  --tags=vllm-server \
  --zone=us-west1-a

# Remove or restrict the 0.0.0.0/0 rule
gcloud compute firewall-rules delete default-allow-internal  # or update
```

FINDING 9b - P2: vLLM endpoint has no API key authentication
Standard vLLM serves on port 8000 with no auth. Even with the VPC restriction above,
any Cloud Run service in the project can call vLLM directly.
For production: Enable vLLM's --api-key flag when starting the vLLM server,
and pass the key in VLLM_BASE_URL or as a header via eft_enhance.py.
