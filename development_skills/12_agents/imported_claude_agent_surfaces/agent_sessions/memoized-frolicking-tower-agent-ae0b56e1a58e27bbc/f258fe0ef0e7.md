## 1. DRL for Trading: PPO + SAC

**Plan says:** PPO (stable-baselines3) + SAC for training, ONNX export for inference. d3rlpy IQL for offline RL from replay buffer.

**Verdict: PARTIALLY OUTDATED**

### What has changed since PPO/SAC became the go-to (2020-2023):

**A. Decision Transformers (DT) and variants** -- The paradigm shift happened in 2023-2025 and is now mature:

- **Decision Transformer** (Chen et al., NeurIPS 2021): Reframes RL as sequence modeling. No Bellman backup, no value function. Just a causal transformer that predicts actions conditioned on desired returns.
- **Online Decision Transformer (ODT)** (Zheng et al., ICML 2022): Adds online fine-tuning to DT.
- **Gato-style multi-task** (DeepMind 2022-2024): Single transformer handles multiple "games" -- equity swing, day trading, crypto could all be different conditioning signals to the SAME model.
- **Q-Transformer** (Google DeepMind, 2023): Combines Q-learning with transformer architecture for continuous control. Better sample efficiency than SAC.
- **Diffusion Policy** (Chi et al., RSS 2023, now mature): Uses diffusion models to represent multi-modal action distributions. Particularly strong for portfolio allocation where the action space is inherently multi-modal (hold/buy/sell are not on a smooth continuum).

**Why this matters for Elson TB2 specifically:**

You already have `elson-finance-14b` running on the L4 VM. A Decision Transformer could share the same vLLM serving infrastructure rather than requiring a separate ONNX inference path. The "one model, many agents" EFT pattern already demonstrates this architecture -- a DRL-as-sequence-model would be a natural extension where trading decisions become another "agent config" over the same transformer backbone.

**B. What PPO/SAC still does well:**

- PPO: Excellent for single-asset position sizing with clear reward signals
- SAC: Good entropy regularization prevents premature convergence in noisy financial data
- Both have massive library support and are debuggable

**C. The hybrid approach (RECOMMENDED):**

| Component | Current Plan | SOTA Recommendation |
|-----------|-------------|---------------------|
| **Offline pre-training** | d3rlpy IQL | Decision Transformer (sequence model) from `TradeDecisionLog` history |
| **Online fine-tuning** | PPO + SAC from scratch | Conservative Q-learning (CQL) warm-start OR Online Decision Transformer |
| **Multi-asset allocation** | Separate equity/crypto policies | Single multi-asset DT with asset-class conditioning tokens |
| **Inference** | ONNX on Cloud Run (no GPU) | vLLM on L4 (shared with elson-finance-14b) OR ONNX fallback |

**Concrete alternative library:** `decision-transformer` (HuggingFace-compatible), `minari` (replaces d3rlpy's replay buffer with standardized offline RL datasets), or roll your own on top of the existing transformer stack.

**Risk flag:** Decision Transformers require more training data than PPO/SAC. With only 3,559 trade decisions logged, PPO/SAC may actually be more practical TODAY. The DT approach becomes dominant once you have 50K+ decisions. **Recommendation: Start with PPO/SAC as planned, but architect the feature pipeline and replay buffer to be DT-compatible from day one. Migrate to DT at 50K decisions.**

## 2. Feature Engineering: 46/56 dims with ta-lib

**Plan says:** 20 OHLCV (ta-lib multi-timeframe), 4 sentiment, 6 regime, 8 portfolio state, 8 LLM signal. Manual feature engineering.

**Verdict: PARTIALLY OUTDATED -- but pragmatically correct for now**

### The tension:

**A. Learned representations are now dominant in research:**

- **Temporal Fusion Transformer (TFT)** (Google, 2021): Learns its own feature importance with interpretable attention. No manual TA indicators needed.
- **TimesNet** (ICLR 2023): Learns multi-period temporal patterns automatically. Outperforms manual TA on every published benchmark.
- **PatchTST / iTransformer** (2023-2024): Patching-based transformers that learn features from raw OHLCV directly.
- **TimesFM** (Google, 2024): Foundation model for time series -- pretrained on 100B+ time points. Zero-shot forecasting.
- **Chronos** (Amazon, 2024): Tokenizes time series into discrete tokens and uses language model architectures. Competitive with domain-specific models out of the box.
- **MOMENT** (CMU, 2024): Foundation model for time series that supports classification, forecasting, and anomaly detection with a single pretrained backbone.

**B. Why manual TA still has value in fintech production:**

- Interpretability: Regulators and users can understand "RSI crossed 70" but not "attention head 4 in layer 7 activated"
- Debuggability: When a trade goes wrong, `feature_vector[12] = RSI = 82` is auditable
- Robustness: TA indicators have 50+ years of empirical validation
- Low latency: Computing RSI/MACD is microseconds. Running a transformer forward pass is milliseconds.

**C. RECOMMENDED hybrid approach:**

Keep the 46-dim manual features for the primary DRL agent (interpretability + speed). But add a **Temporal Fusion Transformer sidecar** that:

1. Ingests raw OHLCV + the 46-dim features
2. Outputs a learned 8-dim "market embedding" that captures nonlinear interactions the manual features miss
3. This embedding becomes dims 47-54, making the total 54-dim for equities / 64-dim for crypto
4. TFT attention weights become part of the audit trail (which features mattered for this decision)

**Library:** `pytorch-forecasting` (TFT implementation, actively maintained, 4K+ GitHub stars), or `neuralforecast` (Nixtla, 6K+ stars, includes TFT + N-BEATS + TimesNet).

**BUT -- critical production constraint:** This adds PyTorch to the inference path. The plan explicitly says "requirements-docker.txt removes torch." If you go this route, the TFT sidecar must either (a) also export to ONNX, or (b) run on the L4 VM alongside vLLM and be called via HTTP. Option (b) is architecturally cleaner and avoids bloating Cloud Run.

## 5. ONNX Inference

**Plan says:** ONNX for inference on Cloud Run (no GPU). 80MB vs 1.5GB for PyTorch.

**Verdict: CURRENT -- with one important caveat**

### SOTA scan:

| Format | Status (Feb 2026) | Notes |
|--------|-------------------|-------|
| **ONNX Runtime** | Gold standard for CPU inference | v1.18+ supports quantized int8, which cuts the 80MB further to ~20MB |
| **TensorRT** | NVIDIA GPU only | Not applicable (Cloud Run has no GPU) |
| **OpenVINO** | Intel CPU optimized | Could be 10-20% faster than ONNX on Cloud Run (Intel Xeon) if GCP uses Intel CPUs |
| **GGML/GGUF** | LLM-specific | Not applicable to DRL models |
| **TFLite** | Mobile/edge | Smaller but less capable than ONNX |
| **Apache TVM** | Compiler-based | 2-5x faster than ONNX but much harder to deploy. Not justified for this use case. |
| **ExecuTorch (Meta)** | 2024+ | PyTorch native export. Competitive with ONNX. Less ecosystem. |

**ONNX is the right choice for DRL policy inference on Cloud Run.** The 80MB estimate is conservative -- with int8 quantization via `onnxruntime.quantization`, a PPO/SAC actor network (3-layer MLP, 256 hidden units, 46-dim input) would be under 5MB.

**The caveat:** If you move to a Decision Transformer architecture later, ONNX export of transformer models is well-supported but requires careful handling of the KV cache and sequence length. Plan for this in the model export pipeline from day one.

## 6. d3rlpy for Offline RL

**Plan says:** d3rlpy IQL (Implicit Q-Learning) for offline RL from `TradeDecisionLog` replay buffer.

**Verdict: OUTDATED**

### SOTA scan:

| Library | Status (Feb 2026) | Notes |
|---------|-------------------|-------|
| **d3rlpy** | Last PyPI release: 2024. Sporadic commits. | Functional but stagnating. IQL implementation works. |
| **CORL (Clean Offline RL)** | Active (2025) | Single-file implementations. Better code quality. IQL, CQL, TD3+BC, AWAC. |
| **Minari** | Active (Farama Foundation, 2025) | Standard dataset format for offline RL. Works with any algorithm library. |
| **OfflineRL-Kit** | Active (2024-2025) | Modular, well-tested. Supports IQL, CQL, TD3+BC. |
| **tianshou** | Active (2025) | Full RL library (online + offline). Batch module supports IQL. Better maintained than d3rlpy. |
| **CleanRL** | Active (2025) | Single-file RL implementations. Online only but excellent for PPO/SAC training scripts. |

**RECOMMENDED REPLACEMENT:**

- **For offline RL (IQL):** Use `CORL` -- cleaner code, actively maintained, single-file implementations that are easy to audit (important for fintech compliance). Each algorithm is one self-contained file.
- **For replay buffer format:** Use `Minari` (Farama Foundation standard) instead of custom pickle files. This ensures compatibility with any future algorithm library.
- **For online training (PPO/SAC):** Keep `stable-baselines3` OR switch to `CleanRL`. SB3 is still actively maintained and well-tested.

**Key advantage of CORL over d3rlpy:** d3rlpy uses a complex class hierarchy that makes debugging difficult. CORL's single-file implementations mean you can read the entire IQL algorithm in 200 lines, modify reward shaping directly, and audit the full training loop -- critical for a fintech trading bot where you need to explain every decision.

## 9. Options Greeks: Black-Scholes vs Modern Approaches

**Plan says:** "Greeks calculation (Black-Scholes or Alpaca-provided)." Also mentions IV surface.

**Verdict: OUTDATED for IV surface / advanced strategies. ACCEPTABLE for basic Greeks.**

### The landscape:

| Model | Use Case | Accuracy | Compute Cost |
|-------|----------|----------|-------------|
| **Black-Scholes (BS)** | European options, basic Greeks | Good for ATM, poor for deep OTM/ITM | Microseconds |
| **Binomial/Trinomial Tree** | American options, early exercise | Better than BS for American options | Milliseconds |
| **Heston (1993)** | Stochastic volatility | Captures vol smile, poor short-term skew | 10-50ms |
| **SABR (2002)** | IV smile interpolation | Industry standard for swaptions/FX. Good for equity options skew. | Milliseconds |
| **Rough Volatility (rBergomi)** | State of the art for vol dynamics | Best fit to empirical vol surfaces | 100ms+ (Monte Carlo) |
| **Deep Hedging (Neural)** | Model-free Greeks from data | Captures all market microstructure | Training: hours. Inference: milliseconds. |

**Your training data already knows this.** I found in `backend/training_data/strategic_qa_pairs.json` and `paper_qa_pairs.json` extensive Q&A about Rough Fractional Stochastic Volatility (RFSV), Deep Hedging, and Heston calibration. The `elson-finance-14b` model was literally trained on the superiority of these approaches over Black-Scholes.

**RECOMMENDED tiered approach:**

1. **Basic Greeks (delta, gamma, theta, vega):** Use Black-Scholes. It is fast, well-understood, and Alpaca provides these anyway. No need to recompute.
2. **IV Surface construction:** Use **SABR** calibration, not raw BS IV. SABR gives you a smooth, arbitrage-free IV surface that interpolates correctly between strikes. Library: `py_vollib` (has SABR) or `QuantLib-Python` (comprehensive, actively maintained, used by every quant desk).
3. **For the DRL agent's options signals:** Feed the SABR-calibrated IV surface as input features rather than raw BS IV. This gives the model a more accurate picture of market-implied risk.
4. **Skip Heston/rBergomi** unless you plan to do options market-making. For directional options trading through Alpaca, SABR-calibrated BS Greeks are sufficient.

**Library recommendation:** `QuantLib-Python` (`pip install QuantLib`). Actively maintained, 5K+ GitHub stars, used in production at every major bank. Supports BS, Heston, SABR, American options, and full term structure modeling. Heavy dependency (~200MB) but it runs on the backend only, not in Docker (compute Greeks server-side, cache results).

**Production note:** Add `QuantLib>=1.33` to `requirements.txt` only (not `requirements-docker.txt`). Compute Greeks on the backend during market hours, cache in PostgreSQL. Cloud Run serves cached values. This avoids adding QuantLib to the Docker image.

Actually -- reconsider. If the options endpoints need to compute Greeks on-demand (e.g., user loads an options chain), you DO need QuantLib in Docker. The alternative is pre-computing Greeks for all active contracts on a schedule and serving from cache.

## 10. Missing Innovation Opportunities -- The 10x Plays

### 10A. THE BIGGEST MISS: LLM-as-Decision-Transformer (10x Leverage of Existing Infrastructure)

The plan treats the DRL system and the EFT/LLM system as **two entirely separate stacks:**

- DRL: stable-baselines3 -> ONNX -> Cloud Run
- LLM: elson-finance-14b -> vLLM -> L4 VM

**The 10x opportunity:** Unify them. The EFT pattern already supports 14 agent configs over one base model. A trading agent is just agent config #15.

**How it would work:**

1. Fine-tune `elson-finance-14b` with a new LoRA adapter specifically for trading decisions
2. Input: structured prompt with the 46-dim feature vector + portfolio state + recent decisions
3. Output: JSON with action (BUY/SELL/HOLD), size (0-1), confidence (0-1), reasoning (text)
4. The "reasoning" field is the audit trail -- no separate explainability module needed
5. Inference uses the SAME vLLM instance on the SAME L4 VM -- zero additional infrastructure cost

**Why this is 10x:**

- No ONNX export pipeline to maintain
- No separate training scripts (H100 VM time shared with existing fine-tuning)
- No separate model versioning (same adapter system as existing EFT)
- Built-in explainability (the model generates reasoning alongside actions)
- Reuses the entire EFT enhancement pattern already proven in production
- The model already has financial domain knowledge from 46,137 training examples

**Why the plan's approach is defensible anyway:**

- DRL policies are mathematically grounded; LLM "reasoning" about trades is not
- ONNX inference on Cloud Run is 5-10ms; vLLM call is 200-500ms
- LLM hallucination risk on position sizing is real and dangerous
- PPO/SAC have convergence guarantees; LLM fine-tuning does not

**RECOMMENDED HYBRID:** Use the LLM for trade reasoning and high-level allocation (the "strategist"), and the DRL agent for precise position sizing and timing (the "executor"). The plan already has `TRADING_BOT_STRATEGIST` agent config -- make this the senior decision-maker that sets targets, and the DRL agent the junior that optimizes execution. This is the pattern used by Citadel's hybrid quant/discretionary desks.

### 10B. WASM Greeks Calculator (Leverage Existing Rust Infrastructure)

The plan puts Greeks computation on the Python backend. But you already have a Rust WASM module (`frontend/src/wasm/src/risk.rs`) that computes VaR, HHI, Sharpe, and sector risk aggregation. **Black-Scholes Greeks computation in Rust WASM would be sub-microsecond on the client.**

**Why this is a 10x UX improvement:**

- Options chain with 200 strikes x 6 expirations = 1,200 contracts
- Computing Greeks for all 1,200 contracts server-side: ~200ms round trip
- Computing Greeks for all 1,200 contracts client-side in WASM: <5ms
- User sees instant Greeks updates as they scroll the chain
- Strategy builder P&L diagram updates in real-time as user drags strikes

**Implementation:** Add `src/wasm/src/options.rs` with `calculate_bs_greeks()`, `calculate_strategy_pnl()`, `calibrate_sabr()`. Export via wasm-bindgen. The existing build pipeline (`wasm:build` script) already handles this.

### 10C. Streaming Portfolio Updates via Server-Sent Events (SSE)

The plan uses 15-second polling for bot metrics. The existing WebSocket infrastructure handles market data. But for bot status/metrics, **SSE is simpler, more reliable, and lower overhead than WebSocket or polling.**

**Why SSE over WebSocket for bot metrics:**

- One-directional (server -> client only) -- perfect for status updates
- Auto-reconnect built into the browser API
- Works through HTTP/2 without special proxy config (Cloud Run natively supports SSE)
- No WebSocket connection management complexity

**Implementation:** Add `GET /auto-trading/stream` endpoint that yields `text/event-stream`. Push bot state changes, trade executions, and risk events in real-time. Replace 15s polling with SSE. Time-to-update drops from 15s average to <100ms.

### 10D. Causal Inference for Feature Importance (Beyond Correlation)

The plan uses correlation to detect BTC correlation drift and feature importance. But correlation is not causation, and in financial markets, spurious correlations are everywhere.

**The 10x opportunity:** Use **DoWhy** or **CausalML** for causal feature importance. Instead of asking "which features correlate with profitable trades," ask "which features CAUSE profitable trades."

**Practical application:**
- Before including an on-chain metric (e.g., whale signal), test whether it has a causal effect on next-day returns using a backdoor criterion analysis
- Before the DRL agent acts on a sentiment signal, verify the causal path: sentiment -> order flow -> price (not sentiment <- price -> order flow, which is confounded)
- Periodically re-run causal tests to detect when a previously causal feature has become noise

**Library:** `dowhy>=0.11` (Microsoft, actively maintained, 7K+ stars).

### 10E. Adversarial Robustness Testing

The plan has stress testing (2022 bear, COVID crash, etc.) but these are historical scenarios. The missing piece is **adversarial robustness** -- testing whether the DRL agent breaks under inputs that are plausible but not in the training data.

**Practical implementation:**
1. Train a generative model (simple VAE) on the feature vectors from the training data
2. Generate adversarial feature vectors that are "nearby" but outside the training distribution
3. Verify the DRL agent's behavior is reasonable (e.g., defaults to HOLD, does not take extreme positions)
4. This catches the "flash crash" scenarios that are not in historical data
