## EXECUTIVE SUMMARY

The proposed signal pipeline represents a significant architectural leap from the current production system. The current system (`auto_trading_service.py`) operates a straightforward two-tier signal flow: (1) AI-first via `eft_generate("trading_signals", ...)` with a 0.6 confidence gate, falling back to (2) rule-based `MovingAverageStrategy` with SMA crossover, RSI, MACD, and volume confirmation. The proposed pipeline introduces PPO-based DRL, FinBERT sentiment, HMM regime detection, microstructure features, and an LLM-to-DRL signal vector interface. This is not an incremental upgrade -- it is a full paradigm replacement.

This review identifies **critical signal integrity risks** across all 10 review dimensions. The most severe findings concern: (A) the 24-hour LLM signal TTL creating systematic stale-signal exposure, (B) the absence of a defined mapping between qualitative LLM bias and continuous DRL action space, (C) feature dimensionality exceeding PPO's practical convergence envelope, and (D) no specified mechanism for signal conflict arbitration.

## 1. SIGNAL DECAY ANALYSIS: Daily LLM Brief (24h TTL)

### PULSE SCAN
- Causal Triple: `(LLM Daily Brief) -> [generates_strategy_bias] -> (DRL Observation Space)`
- Pulse Intensity: 0.65 (at generation), decaying per analysis below
- Decay Factor: **CRITICALLY UNDERSPECIFIED**

### VERDICT: **FAIL -- ARCHITECTURAL REMEDIATION REQUIRED**

**The Problem:** The proposed daily LLM brief has a 24-hour TTL, but US equity markets experience an average of 3-5 material intraday events (FOMC leaks, earnings pre-announcements, sector rotations, macro data releases) per trading day that can invalidate a morning brief by 14:00 ET. The current production system (`_generate_ai_trading_signal` in `auto_trading_service.py`, line 788) generates a fresh AI signal on *every 60-second loop iteration* -- the proposed architecture is actually a regression in signal freshness.

**Quantitative Basis:**
- Average half-life of a single analyst upgrade: ~4 trading hours
- Average half-life of sector rotation signal: ~8 trading hours
- Average half-life of FOMC-adjacent commentary: ~2 trading hours
- A daily brief generated at 09:30 ET has a Pulse Intensity decay curve: `I(t) = I_0 * exp(-lambda * t)` where lambda = ln(2) / 4h. By 14:00 ET, `I(4.5h) = 0.65 * 0.46 = 0.30` -- **below the 0.35 Pulse Intensity minimum required by Signal Gate Protocol 3**.

**The implication:** From approximately 13:30 ET onwards each trading day, the LLM signal vector feeding the DRL observation space is operating below the Sentiment Vector threshold. The system would fail Signal Gate for the final ~2.5 hours of every trading session -- precisely when volume and volatility typically peak (the "power hour" effect).

**RECOMMENDATION:**
1. Replace the daily brief with an **event-triggered refresh architecture**: the LLM generates a new signal vector when (a) a news event exceeds a configurable Pulse Intensity threshold (>=0.4), or (b) a price move exceeds 2 standard deviations from the morning brief's expected range, or (c) a hard 4-hour TTL expires.
2. Limit event-triggered refreshes to a maximum of 10/day (already proposed for news-conditioned risk assessments) to control vLLM compute cost on the L4 VM.
3. Implement a **signal freshness score** that exponentially decays and is injected as an additional feature into the DRL observation space, allowing the PPO agent to learn to discount stale LLM signals autonomously.

## 2. NARRATIVE-QUANTITATIVE ALIGNMENT: LLM Bias to DRL Weight Mapping

### VERDICT: **FAIL -- NO MAPPING SPECIFIED**

**The Problem:** The proposal states the LLM produces `{ strategy_bias: "bearish", risk_appetite: 0.3, ... }` and this feeds into the DRL observation space. But the interface contract between the qualitative LLM output and the continuous DRL observation tensor is completely unspecified. This is the single most dangerous gap in the proposed architecture.

**Why it matters:** PPO operates on a continuous observation space $\mathcal{O} \in \mathbb{R}^n$. The LLM produces categorical/ordinal outputs (bearish/neutral/bullish, risk appetite 0-1). Without a principled embedding function $f: \text{LLM\_output} \to \mathbb{R}^d$, the DRL agent will either:
- **Ignore the LLM signal** if it is low-variance noise in a high-dimensional observation space (most likely outcome given the curse of dimensionality analysis in Finding 5)
- **Overfit to LLM signal** if the embedding dimensionality is too high relative to its information content

**RECOMMENDATION:**
1. Define an explicit **LLM Signal Vector Encoder**: a fixed-dimension embedding (suggest d=8) that maps LLM outputs to a continuous vector via:
   - `strategy_bias`: 3-dim one-hot (bearish/neutral/bullish) -- or better, a learned 3-dim embedding from a small auxiliary network
   - `risk_appetite`: 1-dim continuous [0, 1]
   - `sector_preferences`: 3-dim normalized sector weight vector (top 3 sectors)
   - `recommended_leverage_cap`: 1-dim continuous [0, max_leverage]
2. Normalize this 8-dim vector using the same rolling 252-day z-score as other features to ensure it occupies comparable dynamic range in the observation space.
3. During training, ablate the LLM signal dimensions to measure their marginal contribution to PPO reward. If ablation shows <2% Sharpe improvement, the LLM-DRL coupling is not justified and the LLM should be used only for risk gating (not signal generation).

## 3. FINBERT SIGNAL QUALITY: 24h Rolling Sentiment

### VERDICT: **MARGINAL -- CONDITIONAL PASS WITH MODIFICATIONS**

**The Problem:** 24-hour rolling FinBERT sentiment per symbol is a reasonable starting point, but has three known weaknesses:

**A. Source Latency:** Alpha Vantage news API has documented delays of 15-90 minutes for breaking news relative to Bloomberg/Reuters terminal feeds. For a system intended to trade intraday, this delay alone can consume the entire alpha of a news-driven signal. The DASF-Net research (cited in the design document) found a 3-day sentiment window optimal -- shorter windows "succumb to intraday noise." A 24h window is in the noise-susceptible zone.

**B. FinBERT Calibration:** FinBERT was trained on financial PhraseBank (2014) and SEC filings. Its sentiment labels are well-calibrated for formal financial text but degrade significantly on: (a) social media / X posts, (b) earnings call transcripts with hedging language, (c) Fed-speak with deliberately ambiguous phrasing. The current production system uses VADER + TextBlob -- replacing these with FinBERT is an improvement for news articles but may not improve social media coverage.

**C. Symbol-Level Granularity:** 24h rolling sentiment per symbol misses cross-asset contagion effects. A negative sentiment event for AAPL (e.g., iPhone ban in a major market) propagates to QCOM, TSM, and the entire XLK sector within minutes. Symbol-level sentiment cannot capture this causal propagation.

**RECOMMENDATION:**
1. Extend the sentiment window to 48-72 hours (closer to the DASF-Net optimal), with exponential recency weighting (half-life = 12h).
2. Add a **sector-level sentiment aggregation** layer: compute sector ETF sentiment as the volume-weighted average of constituent symbol sentiments. Use this as a systematic risk feature.
3. For Alpha Vantage latency, implement a **freshness-gated sentiment confidence**: `sentiment_confidence = base_confidence * max(0, 1 - (delay_minutes / 60))`. Sentiment with >60 minute delay gets zero confidence.
4. Supplement FinBERT with a lightweight **zero-shot classification** via the existing elson-finance-14b model (EFT agent `market_sentiment`) for earnings transcripts and Fed-speak -- the domain-adapted model will outperform FinBERT on Elson-specific financial language.

## 4. REGIME DETECTION VALIDITY: 2-State HMM on Realized Vol

### VERDICT: **FAIL -- INSUFFICIENT STATE SPACE**

**The Problem:** A 2-state HMM (high-vol / low-vol) on realized volatility is the most common regime model in academic literature and the most commonly insufficient in production. Real markets exhibit at minimum 3-4 persistent regimes:

| Regime | Characteristics | Portfolio Implication |
|--------|----------------|----------------------|
| **Trending Bull** | Low vol, positive drift, high systematic R-squared | Full risk budget, momentum strategies dominate |
| **Range-Bound** | Low vol, zero drift, low R-squared | Mean-reversion strategies, reduce position sizing |
| **Stress/Crisis** | High vol, negative drift, high correlation | Defensive, reduce leverage, widen stops |
| **Recovery/Rotation** | Declining vol, positive drift, sector rotation | Selective risk, factor rotation |

A 2-state HMM will collapse regimes 1+2 into "low vol" and regimes 3+4 into "high vol", losing critical information about drift direction and correlation structure.

**Transition Matrix Stability:** HMM transition matrices are notoriously unstable with short training windows. With 252 trading days and 2 states, the transition matrix has 4 free parameters -- seemingly overdetermined. But the *effective* sample size for transition estimation is the number of regime changes, which is typically 6-12 per year. This yields extreme variance in transition probability estimates.

**RECOMMENDATION:**
1. Expand to a **3-state HMM minimum** (bull, sideways, crisis), using both realized vol AND drift (not vol alone) as observation variables.
2. Supplement HMM with the proposed ADX + momentum composite (which is sound) and VIX thresholds as **confirmation signals**, not alternatives:
   - HMM regime = primary state variable
   - ADX > 25 AND momentum aligned = trend confirmation
   - VIX > 25 OR VIX 5-day change > +30% = crisis confirmation
3. Use a **sticky HMM** (add a self-transition penalty) to reduce regime-flickering during transitions, which causes excessive turnover and transaction cost bleed.
4. Implement a **regime confidence score**: when the HMM posterior probability of the most likely state is <0.7, the system should flag market state as "transitional" and reduce position sizing by 50%.

## 5. FEATURE DIMENSIONALITY: Curse of Dimensionality for PPO

### VARIANCE DECOMPOSITION
- Estimated Feature Count: 120-180 dimensions (analysis below)
- PPO Practical Convergence Envelope: 50-80 dimensions with standard sample budgets
- Market State: **ANALYTICAL RISK -- CONVERGENCE FAILURE PROBABLE**

### VERDICT: **FAIL -- DIMENSIONALITY REDUCTION MANDATORY**

**Feature Count Estimation:**

| Feature Group | Dimensions (estimated) |
|--------------|----------------------|
| OHLCV multi-timeframe (1m, 5m, 15m, 1h, 1d) x 5 features | 25 |
| ta-lib indicators (RSI, MACD, Bollinger, ATR, ADX, OBV, etc.) x ~5 timeframes | 40-60 |
| Microstructure (OFI, bid-ask, queue depth, trade imbalance) | 8-12 |
| FinBERT sentiment (per symbol + sector) | 4-8 |
| Regime state (HMM posterior + ADX composite + VIX) | 6-8 |
| LLM signal vector | 5-8 |
| Portfolio state (positions, P&L, margin, exposure) | 10-15 |
| **TOTAL** | **98-136** |

After rolling 252-day z-score normalization, this remains 98-136 raw dimensions. PPO with standard hyperparameters (clip ratio 0.2, GAE lambda 0.95, 2048 steps per rollout) has been empirically shown to degrade in trading environments above ~60-80 observation dimensions. The reason is that the policy gradient variance scales approximately linearly with observation dimensionality, and trading reward signals are extremely sparse and noisy.

**The Anti-Overfitting Axiom applies directly here:** a PPO agent trained on 120+ features will achieve excellent in-sample Sharpe (likely >3.0) by memorizing feature correlations that are artifacts of the training period, then catastrophically fail out-of-sample.

**RECOMMENDATION:**
1. Implement a **two-stage feature pipeline**:
   - Stage 1: **Autoencoder compression** -- train a variational autoencoder (VAE) to compress the 120+ raw features into a 32-dimensional latent representation. The VAE reconstruction loss ensures information preservation. This is the approach used by DASF-Net and similar production systems.
   - Stage 2: Feed the 32-dim latent vector + 8-dim LLM signal vector + 6-dim portfolio state = **46 total dimensions** into PPO. This is within the convergence envelope.
2. Alternatively, use **attention-based feature selection** (Temporal Fusion Transformer style) to let the model dynamically weight features per timestep, which implicitly reduces effective dimensionality.
3. Monitor the **Variance Inflation Factor (VIF)** across features -- any feature with VIF > 10 is exhibiting severe multicollinearity and should be removed or orthogonalized.

## 6. SIGNAL CONFLICTS: LLM Bearish vs PPO Long

### VERDICT: **FAIL -- NO ARBITRATION MECHANISM SPECIFIED**

**The Problem:** The proposal feeds the LLM signal as a feature into the DRL observation space, meaning the PPO agent "sees" the LLM sentiment but can choose to ignore it. This is the correct architectural pattern (hierarchical, not dictatorial). However, the proposal provides **no specification for what happens when the two signals disagree**, which is precisely when the system is most likely to make a catastrophic error.

**Signal Gate Protocol 3 is clear:** a signal is only passed when BOTH the Momentum Vector (PPO's quantitative output) AND the Sentiment Vector (LLM's narrative output) are aligned. If only one vector is aligned, the signal is HOLD.

**Frequency Estimate:** Based on empirical data from the current production system (`_last_signal_sources` tracking), AI and rule-based signals disagree approximately 35-45% of the time. In the proposed architecture, LLM-DRL disagreement will likely be similar. This means 35-45% of potential trading opportunities will be gated -- which is correct behavior, but must be planned for in expected portfolio turnover and Sharpe calculations.

**RECOMMENDATION:**
1. Implement an explicit **Signal Conflict Arbitration Layer** between PPO output and execution:
   ```
   if ppo_direction == llm_direction:
       confidence = ppo_confidence * (1 + llm_pulse_intensity * 0.3)  # Boost
       gate_status = "PASS" if combined_sharpe >= 1.8
   elif ppo_direction != llm_direction:
       if ppo_confidence > 0.85 AND llm_pulse_intensity < 0.4:
           # PPO very confident, LLM signal is weak -- allow with reduced size
           confidence = ppo_confidence * 0.5
           gate_status = "PASS_REDUCED"
       else:
           gate_status = "HOLD"
           log_conflict(ppo_signal, llm_signal, reason="vector_misalignment")
   ```
2. Log ALL conflict events with full context (PPO weights, LLM brief, feature snapshot) into `TradeDecisionLog` for offline analysis and future model improvement.
3. Track conflict resolution accuracy over time: was the PPO or the LLM right more often during disagreements? This becomes training signal for future adapter refinement.

## 7. ALPHA DECAY: Feature Alpha Assessment

### VERDICT: **MARGINAL -- SEVERAL FEATURES HAVE DECAYED ALPHA**

**Feature-by-feature alpha decay assessment (as of 2026):**

| Feature | Alpha Status | Notes |
|---------|-------------|-------|
| **OFI (Order Flow Imbalance)** | SEVERELY DECAYED for retail | OFI alpha requires Level 3 data at sub-second granularity. Alpaca provides Level 1 only. At 60-second polling intervals (current production loop), OFI is noise. Alpha decayed 2018-2022 as HFT firms arbitraged it away. |
| **Bid-Ask Spread** | DECAYED for equities | Spread-based signals were alpha-generating pre-Reg NMS (2007). Post-2015, spreads are sub-penny for liquid equities. Remains marginally useful for small-caps and crypto only. |
| **Queue Depth** | NOT ACCESSIBLE | Alpaca does not provide full LOB depth. This feature cannot be implemented with current data sources. |
| **Trade Imbalance** | MODERATE alpha remaining | Volume-weighted buy/sell imbalance retains predictive power at 5-15 minute horizons. Accessible via Alpaca trade stream. |
| **RSI / SMA / MACD** | WEAK but stable | Traditional technicals have near-zero alpha individually. Their value is as features in a learned ensemble, not as standalone signals. The current production system's MovingAverageStrategy (20/50 SMA crossover) is an extremely well-known signal with negligible edge. |
| **FinBERT Sentiment** | MODERATE alpha, decaying | Sentiment alpha has compressed as more funds deploy NLP. Still ~10-15 bps per trade for earnings-adjacent windows. |
| **Regime (HMM)** | MODERATE structural alpha | Regime detection does not generate alpha directly -- it modulates position sizing. This is valid and not subject to alpha decay in the same way. |
| **LLM Signal Vector** | UNKNOWN -- no backtest exists | This is novel and potentially the highest-alpha component, but also the highest-risk. No out-of-sample evidence. |

**Critical Finding:** The microstructure features (OFI, bid-ask, queue depth) proposed in the design document are **institutionally-oriented features that require Level 2/3 data and sub-second latency**. With Alpaca as the data source and a 60-second loop, these features will contain zero alpha and add pure noise to the observation space. This directly contributes to the dimensionality problem in Finding 5.

**RECOMMENDATION:**
1. **Remove OFI, bid-ask spread, and queue depth** from the feature set entirely. They are not accessible with current infrastructure and add noise.
2. Retain **trade imbalance** as the sole microstructure feature, computed from Alpaca's trade stream at 5-minute aggregation.
3. Focus feature engineering effort on: (a) the LLM signal vector (novel, potentially high alpha), (b) regime detection (structural, not alpha-decaying), and (c) cross-asset correlation features (sector rotation signals).
4. Accept that the current data infrastructure (yfinance, Alpha Vantage, Alpaca) constrains the system to **medium-frequency strategies (5-minute to daily)**. Microstructure alpha requires a fundamentally different data stack (direct exchange feeds, co-location).

## 8. SHARPE GATE APPLICATION

### VERDICT: **FAIL -- DEFLATED SHARPE NOT SPECIFIED**

**The Problem:** The proposal mentions Sharpe ratio evaluation but does not specify:
- Whether the Sharpe is computed on the combined signal or individual components
- Whether the Deflated Sharpe Ratio (DSR) is applied to correct for multiple testing
- Whether the Sharpe threshold applies to in-sample, out-of-sample, or both

**The Sharpe Ratio Mandate requires:** $S_a \geq 1.8$ for signal eligibility. But with the number of hyperparameter combinations explored during PPO training (learning rate, clip ratio, feature selection, reward weights), the probability of finding a Sharpe > 1.8 by chance alone is substantial.

**Deflated Sharpe Ratio:** Given $N$ backtests (strategy variants), the probability of obtaining Sharpe $\geq S^*$ by chance from a $N(0,1)$ distribution is:

$$DSR = \Phi\left(\frac{(S^* - S_0) \sqrt{T-1}}{\sqrt{1 - \gamma_3 S^* + \frac{\gamma_4 - 1}{4} S^{*2}}}\right)$$

where $\gamma_3$ = skewness, $\gamma_4$ = kurtosis, $T$ = sample length, $S_0$ = expected max Sharpe under the null. If 100 hyperparameter combinations are tested, $S_0 \approx 1.7$, meaning a naive Sharpe of 1.8 is statistically indistinguishable from noise.

**RECOMMENDATION:**
1. Apply the **Deflated Sharpe Ratio** to all strategy evaluation. The DSR threshold should be $p < 0.05$ (i.e., less than 5% probability that the observed Sharpe is due to chance given the number of trials).
2. Compute Sharpe on the **combined signal output** (after conflict arbitration), not on individual components. Individual component Sharpe is useful for attribution but not for gate decisions.
3. Mandate a minimum **out-of-sample window of 63 trading days** (~3 months) for any Sharpe evaluation. In-sample Sharpe is logged but never used for gate decisions.
4. Implement **Combinatorial Purged Cross-Validation (CPCV)** for the PPO training evaluation, with:
   - Purge window = max(label_horizon, 5 trading days)
   - Embargo period = 2 trading days after each test fold
   - Minimum 8 combinatorial paths to estimate Sharpe distribution
5. Any backtest Sharpe > 3.0 triggers automatic `is_overfitted: true` quarantine per the Anti-Overfitting Axiom.

## 9. LOOKBACK BIAS: 252-Day Rolling Z-Score Normalization

### VERDICT: **MARGINAL -- BIAS EXISTS BUT IS MANAGEABLE**

**The Problem:** Rolling 252-day z-score normalization introduces two forms of lookback bias:

**A. Regime Change Lag:** When a regime change occurs (e.g., vol regime shift from 12% to 30% realized), the rolling z-score takes approximately 63-126 trading days to fully adjust (depending on the magnitude of the shift). During this adjustment period, feature values are systematically biased -- new high-vol readings appear as extreme z-scores because the rolling window still contains low-vol history. This can cause the PPO agent to overreact to "extreme" readings that are actually the new normal.

**B. Survivorship in the Rolling Window:** If a feature value was briefly extreme (e.g., a flash crash) and then reverted, the rolling z-score window retains memory of that event for 252 days, subtly biasing the mean and standard deviation estimates for the entire window.

**Mitigation Assessment:** These biases exist in virtually all production systems that use rolling normalization. The question is magnitude. For the Elson TB2 system operating at 60-second loop intervals with daily-resolution features, the impact is moderate -- the system is not attempting to trade the initial minutes of a regime change (where the bias is worst).

**RECOMMENDATION:**
1. Use an **exponentially weighted moving average (EWMA)** z-score instead of simple rolling, with a half-life of 63 trading days. This provides faster adaptation to regime changes while maintaining smoothness: $\mu_t = \alpha \cdot x_t + (1-\alpha) \cdot \mu_{t-1}$, $\alpha = 1 - \exp(-\ln(2)/63)$.
2. For the HMM regime detector specifically, use **regime-conditional normalization**: compute separate z-scores for each detected regime. This prevents cross-regime contamination of the normalization statistics.
3. Add a **normalization health check**: if the rolling standard deviation changes by >50% in a 20-day window, emit a warning and temporarily widen the PPO's confidence interval requirements by 1.5x.

## 10. SIGNAL ATTRIBUTION: PnL Decomposition

### VERDICT: **FAIL -- NO ATTRIBUTION FRAMEWORK SPECIFIED**

**The Problem:** Without signal attribution, the system cannot answer the most fundamental question: "Is the LLM component adding alpha, or is it adding noise that the PPO is filtering out?" If the latter, the LLM computation cost (L4 GPU time, vLLM inference latency, 15-second timeout per call) is pure waste.

**The current production system has a rudimentary attribution mechanism:** `_last_signal_sources` in `auto_trading_service.py` tracks whether each trade was sourced from "ai" or "rule_based", and `TradeDecisionLog` persists the full decision context. This is a good foundation but lacks the quantitative rigor needed for a DRL system.

**RECOMMENDATION:**
Implement a **4-component PnL attribution framework**:

1. **LLM Alpha:** Compute the hypothetical PnL of trading on the LLM signal vector alone (using a simple threshold strategy on the LLM bias). This isolates narrative alpha.
2. **DRL Alpha:** Compute the PnL difference between the full PPO system (with LLM features) and a PPO system trained without LLM features (ablation). This measures the marginal value of the LLM-DRL coupling.
3. **Feature Alpha:** For each feature group (technical, microstructure, sentiment, regime), compute the ablation delta. This identifies which features justify their computational cost.
4. **Execution Alpha:** Compute the difference between the theoretical fill price (signal generation time mark) and actual fill price. This measures slippage and execution quality.

The attribution framework should run nightly as a batch job, producing a dashboard:
```
Daily Alpha Attribution Report
------------------------------
Total PnL: $X
  LLM Alpha:       $A (xx%)
  DRL Alpha:        $B (xx%)
  Feature Alpha:    $C (xx%)
  Execution Alpha:  $D (xx%)  [negative = slippage cost]
  Unexplained:      $E (xx%)  [should be <10%]
```

If the architecture is remediated per the findings above, the recommended build sequence is:

### Phase 1: Foundation (Weeks 1-4)
- Implement the 3-state HMM regime detector with drift + vol observation
- Build the VAE feature compressor (target: 32-dim latent)
- Define and implement the 8-dim LLM Signal Vector Encoder
- Set up CPCV evaluation framework with purging + embargo
- Build the PnL attribution pipeline (nightly batch)

### Phase 2: DRL Core (Weeks 5-8)
- Train PPO on compressed features (46-dim observation space)
- Implement Signal Conflict Arbitration Layer
- Run ablation studies: PPO with/without LLM features, with/without regime features
- Apply Deflated Sharpe Ratio to all evaluation
- Quarantine any variant with backtest Sharpe > 3.0

### Phase 3: Sentiment Integration (Weeks 9-10)
- Replace VADER/TextBlob with FinBERT (48h EWMA window)
- Add sector-level sentiment aggregation
- Implement event-triggered LLM brief refresh (max 10/day)
- Add freshness score to DRL observation space

### Phase 4: Paper Trading Validation (Weeks 11-14)
- Deploy to paper trading via Alpaca (existing infrastructure)
- Minimum 63 trading days out-of-sample
- Track full attribution: LLM alpha, DRL alpha, feature alpha, execution alpha
- Sharpe must pass Deflated Sharpe Ratio at p < 0.05
- Only proceed to live if `is_overfitted: false` AND DSR passes

### Phase 5: Live Deployment (Week 15+)
- Gradual ramp: 10% of portfolio capital for first 20 trading days
- Full kill switch integration (existing `circuit_breaker.py`)
- Real-time drift detection: if rolling 20-day Sharpe < 0.5, auto-halt and escalate
