# Temporal Data Architecture Assessment: Autonomous Trading Bot Schema

**Date:** 2026-02-25
**Reviewer:** Data Infrastructure Lead
**Scope:** Proposed `bot_*` tables for fully autonomous DRL-based trading bot
**Verdict:** CONDITIONAL APPROVAL -- 7 critical deficiencies, 4 structural warnings, 3 design improvements

## Executive Summary

The proposed schema has a workable foundation but contains **seven temporal integrity deficiencies** that, if left unresolved, will cause data corruption, look-ahead bias in backtesting, incorrect PDT (Pattern Day Trader) classification, and silent feature staleness. This assessment is organized by the eight review areas requested.

## 1. TIMESTAMP INTEGRITY: CRITICAL DEFICIENCY

### Finding 1.1: Inconsistent Timezone Handling Across Existing Models (Pre-Existing Debt)

The current Elson codebase already has a dangerous inconsistency. Some models use `DateTime(timezone=True)` with `server_default=func.now()` (correct), while others use bare `DateTime` with `default=lambda: datetime.now(timezone.utc)` (fragile). Evidence from the codebase:

**Pattern A (Correct -- server-side UTC):**
```python
# trade.py, family.py, holding.py, market_data.py, auto_trading_session.py
created_at = Column(DateTime(timezone=True), server_default=func.now())
```

**Pattern B (Fragile -- application-side UTC):**
```python
# account.py, security.py, education.py, notification.py, subscription.py
created_at = Column(DateTime, default=lambda: datetime.now(timezone.utc))
```

Pattern B is dangerous because:
- `DateTime` (without `timezone=True`) stores timezone-naive datetimes in PostgreSQL as `timestamp without time zone`. Even though the Python code generates UTC, PostgreSQL has no metadata to enforce this.
- If Cloud Run instances ever drift (container time skew), or a developer passes a local datetime, the stored value silently diverges from UTC.
- Pattern B timestamps are invisible to PostgreSQL's `AT TIME ZONE` operators.

### Recommendation 1.1: MANDATORY for all new `bot_*` tables

Every timestamp column in the new bot schema MUST use:
```python
Column(DateTime(timezone=True), server_default=func.now())
```
This stores as `timestamptz` in PostgreSQL, which is unambiguous. Never use bare `DateTime` for any new bot table.

### Finding 1.2: Missing Market Time vs. UTC Conversion Layer

NYSE operates 09:30-16:00 ET (Eastern Time, which alternates between EST/UTC-5 and EDT/UTC-4). The proposed schema has no explicit handling for this conversion. This matters because:

- **`bot_sessions.started_at`**: If a bot starts at 09:30 ET, is this stored as 14:30 UTC (EST) or 13:30 UTC (EDT)? The answer depends on the date, and `timestamptz` handles this correctly -- but the application code must be aware when checking "is the market open?"
- **`bot_day_trade_log.open_time` / `close_time`**: These are used to determine if a round-trip constitutes a day trade. FINRA Rule 4210 defines a day trade as a purchase and sale of the same security **in the same trading day**. A "trading day" is defined by market time (ET), not UTC.
- **Extended hours**: Pre-market (04:00-09:30 ET) and after-hours (16:00-20:00 ET) create additional ambiguity. A position opened at 07:00 ET pre-market and closed at 15:00 ET regular session is the SAME calendar day in ET but spans different UTC dates if DST is in effect near midnight.

### Recommendation 1.2: Add `market_date` column and utility function

```python
# On bot_day_trade_log:
market_date = Column(Date, nullable=False, index=True)  # NYSE trading date (ET)

# Utility function (to be placed in a shared temporal utils module):
from zoneinfo import ZoneInfo

NYSE_TZ = ZoneInfo("America/New_York")

def get_nyse_trading_date(utc_dt: datetime) -> date:
    """Convert UTC timestamp to NYSE trading date.

    NYSE trading day boundaries:
    - Pre-market starts at 04:00 ET
    - After-hours ends at 20:00 ET
    - Trades between 20:00 ET and 04:00 ET next day are attributed
      to the NEXT trading day (rare, but possible with overnight orders)
    """
    et_dt = utc_dt.astimezone(NYSE_TZ)
    # If before 04:00 ET, attribute to previous trading date
    if et_dt.hour < 4:
        return (et_dt - timedelta(days=1)).date()
    return et_dt.date()
```

Store `market_date` at write time. Never compute it retroactively from UTC timestamps -- DST transitions make this error-prone when done in batch.

## 2. BITEMPORAL CONCERNS: CRITICAL DEFICIENCY

### Finding 2.1: No Transaction Time / Valid Time Separation in Bot Tables

This is the single most dangerous architectural gap. The proposed schema stores only event timestamps (when things happened in the real world -- `submitted_at`, `filled_at`, etc.) but never records **when the system learned about each fact**.

**Why this matters for DRL training:**

The autonomous bot's training pipeline uses walk-forward backtesting on 2020-2023 data. Consider this scenario:

1. At 10:00 AM, a company announces earnings (the event's `valid_time`).
2. The sentiment score derived from this announcement is computed at 10:05 AM and written to the feature store at 10:07 AM (the `transaction_time` -- when the system actually had this data available).
3. During backtesting, if the model queries "what was the sentiment at 10:02 AM?" and receives the post-announcement sentiment (because valid_time = 10:00 AM <= 10:02 AM), **the model has look-ahead bias**. The system did not actually know that sentiment score until 10:07 AM.

Without `transaction_time`, there is no way to reconstruct what the bot actually knew at any historical point. Every backtest is contaminated.

### Recommendation 2.1: Add bitemporal columns to ALL bot tables that participate in feature computation

```python
# Minimum bitemporal columns for every bot table that feeds the DRL pipeline:
valid_time = Column(DateTime(timezone=True), nullable=False, index=True)
valid_time_end = Column(DateTime(timezone=True), nullable=True)  # NULL = still valid
transaction_time = Column(DateTime(timezone=True), server_default=func.now(), nullable=False, index=True)
transaction_time_end = Column(DateTime(timezone=True), nullable=True)  # NULL = still current

# Composite index for point-in-time queries:
__table_args__ = (
    Index('idx_bot_perf_bitemporal', 'valid_time', 'transaction_time'),
)
```

**Tables that MUST be bitemporal (they feed features):**
- `bot_performance` (PnL, equity, drawdown metrics)
- `bot_risk_events` (regime shifts, circuit breaker activations)
- Feature store snapshots in PostgreSQL

**Tables where bitemporal is optional (operational, not feature inputs):**
- `bot_sessions` (session lifecycle metadata)
- `bot_audit_log` (already an append-only log)
- `bot_orders` / `bot_options_orders` (use Alpaca's authoritative timestamps)

### Finding 2.2: Feature Store Lacks Point-in-Time Reconstruction

The proposed feature store design (Redis 60s TTL + PostgreSQL daily snapshots) creates a gap:
- Redis holds only the latest value. Once it expires (60s TTL), the intermediate states are gone.
- PostgreSQL snapshots are daily. This means 6.5 hours of intraday feature evolution (390 one-minute bars) are collapsed to a single daily snapshot.

During walk-forward training, the DRL agent needs feature vectors at arbitrary historical timestamps. With daily snapshots, the agent can only train on daily-resolution features, even though the live bot operates at sub-minute resolution. The model trains on different data granularity than it infers on.

### Recommendation 2.2: Feature Store Temporal Tiering

```
Tier 1: Redis (hot, 60s TTL) -- Live inference only. No persistence.
Tier 2: PostgreSQL bot_feature_snapshots (warm, 15-second granularity during market hours)
Tier 3: PostgreSQL bot_feature_daily (cold, daily OHLC-style aggregation)
```

The `bot_feature_snapshots` table:
```python
class BotFeatureSnapshot(Base):
    __tablename__ = "bot_feature_snapshots"

    id = Column(Integer, primary_key=True)
    symbol = Column(String(20), nullable=False)
    feature_vector = Column(JSON, nullable=False)  # Full feature dict
    feature_version = Column(String(20), nullable=False)  # Schema version
    valid_time = Column(DateTime(timezone=True), nullable=False)  # When the features describe
    transaction_time = Column(DateTime(timezone=True), server_default=func.now())  # When computed

    __table_args__ = (
        Index('idx_feat_snap_symbol_valid', 'symbol', 'valid_time'),
        Index('idx_feat_snap_symbol_txn', 'symbol', 'transaction_time'),
    )
```

With 15-second snapshots for 50 symbols over 6.5 hours = 78,000 rows/day. At ~1KB per row, this is ~78MB/day, ~1.5GB/month. Acceptable for PostgreSQL with partitioning by `market_date`.

## 3. FEATURE STORE TEMPORAL CONSISTENCY: STRUCTURAL WARNING

### Finding 3.1: Component Staleness in Feature Vectors

The proposed feature vector aggregates components with different update frequencies:

| Component | Update Frequency | Source |
|-----------|-----------------|--------|
| Price/Volume | Every tick (~100ms) | Alpaca WebSocket |
| Technical indicators (RSI, MACD) | Every bar close (1m, 5m) | Computed |
| Sentiment score | Every 5-30 minutes | NLP pipeline |
| Regime classification | Every 1-4 hours | Regime model |
| Fundamental data | Daily / quarterly | Data provider |

When the feature vector is assembled at time T, the price component reflects T, but the sentiment might reflect T-25min and the regime might reflect T-3h. The vector does NOT represent a consistent point in time.

### Recommendation 3.1: Feature Freshness Metadata

Every feature vector should carry a `staleness_map`:
```json
{
  "price": {"value": 185.42, "as_of": "2026-02-25T14:30:00Z", "stale_seconds": 0},
  "rsi_14": {"value": 62.3, "as_of": "2026-02-25T14:30:00Z", "stale_seconds": 0},
  "sentiment": {"value": 0.72, "as_of": "2026-02-25T14:05:00Z", "stale_seconds": 1500},
  "regime": {"value": "bull", "as_of": "2026-02-25T11:00:00Z", "stale_seconds": 12600}
}
```

The DRL training pipeline should include `stale_seconds` as an input feature per component. This way, the model learns to discount stale signals rather than treating all features as equally fresh. This is the same principle as the ANCOVA filter: rather than trying to make the data "perfect," give the model the information it needs to calibrate its own uncertainty.

## 4. ORDER LIFECYCLE TIMESTAMPS: DEFICIENCY

### Finding 4.1: Missing `partially_filled_at` Timestamp

The proposed schema has `submitted_at` and `filled_at` but no `partially_filled_at`. The existing `Trade` model (in `trade.py`) has `filled_quantity` and a `PARTIALLY_FILLED` status, but partial fill events are not individually timestamped.

For the autonomous bot, partial fills matter because:
- A limit order might receive 3 partial fills over 45 seconds before being fully filled.
- The bot's position sizing logic needs to know its *current* net exposure at any instant, which requires knowing the timing of each partial fill -- not just the final aggregated fill.
- Risk calculations (VaR, drawdown) must reflect the actual position trajectory, not a jump from 0 to fully filled.

The existing `TradeExecution` model (`trade.py:208-229`) already models individual execution events with `execution_time`. This is the correct pattern.

### Recommendation 4.1: Ensure `bot_orders` references granular executions

```python
class BotOrderExecution(Base):
    """Individual fill events for a bot order. One order -> many executions."""
    __tablename__ = "bot_order_executions"

    id = Column(Integer, primary_key=True)
    bot_order_id = Column(Integer, ForeignKey("bot_orders.id"), nullable=False, index=True)

    # Alpaca's reported fill time (authoritative)
    exchange_fill_time = Column(DateTime(timezone=True), nullable=False)
    # When our WebSocket handler received the fill event
    received_at = Column(DateTime(timezone=True), server_default=func.now())
    # Latency = received_at - exchange_fill_time (useful for monitoring)

    filled_quantity = Column(Numeric(18, 8), nullable=False)
    filled_price = Column(Numeric(18, 8), nullable=False)

    __table_args__ = (
        Index('idx_bot_exec_order_time', 'bot_order_id', 'exchange_fill_time'),
    )
```

### Finding 4.2: Network Latency Between Alpaca Fill and Bot Receipt

When Alpaca fills an order at T, the fill event arrives at the bot via WebSocket at T + delta. This delta is typically 10-200ms but can spike to 1-5 seconds during high-volume periods (market open, FOMC announcements).

If the bot uses `received_at` (when it processed the WebSocket event) as the fill time for risk calculations, it will underestimate exposure during the delta period. If it uses Alpaca's `filled_at` (exchange time), it is more accurate but introduces a data point that the bot did not *know about* until `received_at`.

This is a textbook bitemporal problem:
- `valid_time` = Alpaca's `filled_at` (when the fill actually occurred in reality)
- `transaction_time` = bot's `received_at` (when the bot learned about it)

### Recommendation 4.2: Store both timestamps, use transaction_time for live decisions, valid_time for analytics

```python
# On bot_orders:
exchange_filled_at = Column(DateTime(timezone=True), nullable=True)  # Alpaca's fill time (valid_time)
bot_received_fill_at = Column(DateTime(timezone=True), nullable=True)  # When bot received WebSocket (transaction_time)
```

Live risk engine uses `bot_received_fill_at`. Backtesting and analytics use `exchange_filled_at`. This ensures the live system never "knows the future" -- it only acts on information it has actually received.

## 5. BOT_PERFORMANCE AT 15s GRANULARITY: DESIGN IMPROVEMENT

### Finding 5.1: Storage Volume and Aggregation Strategy

15-second granularity during a 6.5-hour trading day = 1,560 rows per session per day. Over 252 trading days per year, that is ~393,000 rows per session per year. For a single user with one bot session, this is manageable. But the table will become the most-queried table in the system (dashboards, backtesting, reporting), so it needs a retention and aggregation strategy.

### Recommendation 5.1: Three-Tier Retention with Materialized Aggregation

```
bot_performance (15s granularity)    -- retain 30 days
bot_performance_1m (1-minute OHLC)   -- retain 90 days
bot_performance_daily (daily summary) -- retain indefinitely
```

**Daily aggregation table:**
```python
class BotPerformanceDaily(Base):
    __tablename__ = "bot_performance_daily"

    id = Column(Integer, primary_key=True)
    session_id = Column(Integer, ForeignKey("bot_sessions.id"), nullable=False)
    market_date = Column(Date, nullable=False)  # NYSE trading date

    # OHLC-style metrics for the day
    equity_open = Column(Numeric(18, 8))
    equity_high = Column(Numeric(18, 8))
    equity_low = Column(Numeric(18, 8))
    equity_close = Column(Numeric(18, 8))

    # Aggregate metrics
    total_trades = Column(Integer, default=0)
    total_pnl = Column(Numeric(18, 8), default=0)
    max_drawdown_pct = Column(Numeric(18, 8))
    sharpe_intraday = Column(Numeric(18, 8))  # Intraday Sharpe from 15s returns

    # Volume and activity
    total_volume_traded = Column(Numeric(18, 8), default=0)
    total_commission = Column(Numeric(18, 8), default=0)

    __table_args__ = (
        Index('idx_perf_daily_session_date', 'session_id', 'market_date', unique=True),
    )
```

**Aggregation job:** Run a nightly job (after 20:00 ET when after-hours close) that:
1. Aggregates `bot_performance` rows into `bot_performance_daily`
2. Aggregates into `bot_performance_1m`
3. Purges `bot_performance` rows older than 30 days
4. NEVER purge `bot_performance_daily` -- this is the permanent record

**PostgreSQL partitioning for `bot_performance`:**
```sql
CREATE TABLE bot_performance (
    ...
) PARTITION BY RANGE (valid_time);

-- Create monthly partitions
CREATE TABLE bot_performance_2026_02 PARTITION OF bot_performance
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
```

Partitioning makes both queries and pruning efficient. Dropping old partitions is an O(1) operation vs. DELETE which requires scanning.

## 6. DAY TRADE LOG TEMPORAL EDGE CASES: CRITICAL DEFICIENCY

### Finding 6.1: Day Trade Definition Requires Market Time

FINRA's Pattern Day Trader rule (Rule 4210) counts round-trips within a **single trading day**. The "trading day" is defined by the exchange, not by UTC.

**Edge case 1: DST transition**
- On the second Sunday of March, clocks spring forward. A trade opened at 15:55 ET Friday and closed at 09:31 ET Monday is NOT a day trade (different trading days). But a trade opened at 09:31 ET and closed at 15:59 ET on the same Monday IS a day trade. UTC timestamps alone cannot determine this without the DST conversion.

**Edge case 2: Extended hours**
- A position opened at 05:00 ET pre-market and closed at 19:30 ET after-hours = same calendar day = IS a day trade.
- A position opened at 19:30 ET after-hours Monday and closed at 05:00 ET pre-market Tuesday = different calendar days in ET = NOT a day trade.

**Edge case 3: Overnight holds that close at open**
- Position opened at 15:58 ET Monday, NOT closed by 20:00 ET (after-hours end). Held overnight. Closed at 09:31 ET Tuesday. NOT a day trade (different trading days).

### Recommendation 6.1: Schema additions for bot_day_trade_log

```python
class BotDayTradeLog(Base):
    __tablename__ = "bot_day_trade_log"

    id = Column(Integer, primary_key=True)
    session_id = Column(Integer, ForeignKey("bot_sessions.id"), nullable=False)
    symbol = Column(String(20), nullable=False)

    # UTC timestamps (authoritative for ordering and analytics)
    open_time = Column(DateTime(timezone=True), nullable=False)
    close_time = Column(DateTime(timezone=True), nullable=False)

    # Market time classification (authoritative for PDT determination)
    open_market_date = Column(Date, nullable=False)   # NYSE date of open
    close_market_date = Column(Date, nullable=False)   # NYSE date of close
    is_day_trade = Column(Boolean, nullable=False)      # open_market_date == close_market_date

    # Extended hours flags
    opened_in_extended_hours = Column(Boolean, default=False)
    closed_in_extended_hours = Column(Boolean, default=False)

    # PDT rolling window tracking
    pdt_window_start = Column(Date, nullable=False)  # 5 business days rolling window start
    day_trades_in_window = Column(Integer, nullable=False)  # Count at time of this trade

    __table_args__ = (
        Index('idx_day_trade_session_date', 'session_id', 'close_market_date'),
        Index('idx_day_trade_pdt_window', 'session_id', 'pdt_window_start'),
    )
```

The `is_day_trade` flag MUST be computed at write time using the `get_nyse_trading_date()` function from Recommendation 1.2. Never recompute it retroactively.

## 7. BACKTEST DATA INTEGRITY: STRUCTURAL WARNING

### Finding 7.1: Point-in-Time Reconstruction Without Transaction Time

The proposed DRL training uses walk-forward backtesting on 2020-2023 data. For this to be valid, the training pipeline must answer: **"What data was available to the bot at time T, given only information that existed before T?"**

Without `transaction_time` on historical data records, the pipeline cannot distinguish between:
- A sentiment score computed at 10:00 AM and available at 10:00 AM (normal case)
- A sentiment score that was REVISED at 2:00 PM to correct an earlier error -- the revised version should not be visible to a backtest running at 10:30 AM

If historical data sources (Yahoo Finance, Polygon, etc.) provide revised/adjusted data, the bot's backtest is seeing the "final" version of history, not the version that was available in real-time. This is a form of look-ahead bias.

### Recommendation 7.1: Data Provenance Tracking for Historical Data

```python
class BotHistoricalDataProvenance(Base):
    """Tracks when historical data was fetched and from which source,
    enabling point-in-time reconstruction for backtesting."""
    __tablename__ = "bot_historical_data_provenance"

    id = Column(Integer, primary_key=True)
    symbol = Column(String(20), nullable=False)
    data_type = Column(String(50), nullable=False)  # 'price', 'fundamental', 'sentiment'
    valid_time = Column(DateTime(timezone=True), nullable=False)  # When the fact was true
    fetched_at = Column(DateTime(timezone=True), server_default=func.now())  # When we got it
    source = Column(String(100), nullable=False)  # 'polygon', 'yahoo', 'alpaca'
    is_adjusted = Column(Boolean, default=False)  # Corporate action adjusted?
    adjustment_factor = Column(Numeric(18, 8), nullable=True)  # Split/dividend factor
```

### Finding 7.2: Survivorship Bias in Symbol Universe

If the 2020-2023 replay universe is the current S&P 500 composition, every delisted, acquired, or bankrupt stock from that period is missing. The model never sees the negative outcomes, only survivors. This inflates backtested returns.

### Recommendation 7.2:

- Use a point-in-time index composition (e.g., S&P 500 membership as of January 1, 2020, not February 2026).
- Include delisted symbols with their terminal price events (acquisition price, delisting price, or $0 for bankruptcy).
- Store the symbol universe snapshot with a `as_of_date` column:

```python
class BotSymbolUniverse(Base):
    __tablename__ = "bot_symbol_universe"

    id = Column(Integer, primary_key=True)
    universe_name = Column(String(100), nullable=False)  # 'sp500', 'nasdaq100'
    symbol = Column(String(20), nullable=False)
    as_of_date = Column(Date, nullable=False)  # When this membership was valid
    added_date = Column(Date, nullable=True)
    removed_date = Column(Date, nullable=True)
    removal_reason = Column(String(100), nullable=True)  # 'delisted', 'acquired', 'moved_index'

    __table_args__ = (
        Index('idx_universe_name_date', 'universe_name', 'as_of_date'),
    )
```

## 8. SESSION LIFECYCLE AND MULTI-DAY OPERATIONS: DESIGN IMPROVEMENT

### Finding 8.1: Cumulative vs. Daily Performance Reset

If a bot session starts Monday and runs through Friday, the `bot_performance` table accumulates 7,800 rows (1,560/day x 5 days). The schema is silent on whether metrics like `total_pnl` and `max_drawdown` are:
- **Cumulative from session start** (Monday's starting equity is the baseline through Friday)
- **Reset daily** (each day starts fresh)

Both are needed for different purposes:
- Cumulative: Evaluating overall session performance, DRL reward calculation
- Daily: Risk management (daily loss limits), regulatory reporting (PDT counting)

### Recommendation 8.1: Store both, compute daily from the aggregation table

In `bot_performance` (15s granularity), store cumulative metrics from session start:
```python
# Cumulative from session start
equity = Column(Numeric(18, 8), nullable=False)          # Current total equity
cumulative_pnl = Column(Numeric(18, 8), nullable=False)   # Total PnL since session start
cumulative_max_drawdown = Column(Numeric(18, 8))           # Max drawdown since session start
```

In `bot_performance_daily`, store daily-scoped metrics:
```python
# Daily-scoped (reset each trading day)
daily_pnl = Column(Numeric(18, 8))
daily_max_drawdown = Column(Numeric(18, 8))
daily_trade_count = Column(Integer)
```

### Finding 8.2: Overnight Gap Risk Tracking

Between market close (20:00 ET after-hours) and next open (04:00 ET pre-market), the bot holds positions but cannot trade. If overnight news causes a gap, the bot faces unhedged risk.

The proposed schema has no mechanism to:
1. Record the closing portfolio state at end-of-day
2. Record the opening portfolio state at start-of-day
3. Compute the overnight gap impact

### Recommendation 8.2: Overnight gap event tracking

```python
class BotOvernightGap(Base):
    __tablename__ = "bot_overnight_gaps"

    id = Column(Integer, primary_key=True)
    session_id = Column(Integer, ForeignKey("bot_sessions.id"), nullable=False)
    symbol = Column(String(20), nullable=False)

    # End-of-day state
    eod_market_date = Column(Date, nullable=False)
    eod_price = Column(Numeric(18, 8), nullable=False)
    eod_position_qty = Column(Numeric(18, 8), nullable=False)
    eod_timestamp = Column(DateTime(timezone=True), nullable=False)

    # Start-of-day state
    sod_price = Column(Numeric(18, 8), nullable=False)
    sod_timestamp = Column(DateTime(timezone=True), nullable=False)

    # Gap metrics
    gap_pct = Column(Numeric(18, 8), nullable=False)  # (sod - eod) / eod * 100
    gap_pnl = Column(Numeric(18, 8), nullable=False)  # position_qty * (sod - eod)

    __table_args__ = (
        Index('idx_overnight_session_date', 'session_id', 'eod_market_date'),
    )
```

## INDEX STRATEGY RECOMMENDATIONS

For the proposed bot tables, the following indexes are essential:
