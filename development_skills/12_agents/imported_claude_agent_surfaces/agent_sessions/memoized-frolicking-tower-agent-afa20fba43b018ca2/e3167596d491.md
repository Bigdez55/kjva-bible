### FAIL-1 - CRITICAL: Circuit breaker uses JSON file persistence in Cloud Run
**File:** `backend/app/trading_engine/engine/circuit_breaker.py`, lines 159-190

`_load_status()` and `_save_status()` write to `circuit_breaker_status.json`
on the local filesystem. Cloud Run containers have ephemeral filesystems —
this file is destroyed on every cold start or instance replacement. The plan
identifies DB migration as a Pre-Sprint task but classifies it as a P1 blocker.

**Architectural conflict:** If the plan deploys M1 safety features (crypto circuit
breakers) WITHOUT first migrating persistence to DB, any SYSTEM breaker trip will
be lost on container restart. A LEVEL 4 EMERGENCY (Alpaca exchange down) that
trips the system breaker will be silently cleared on the next container restart.

**Required fix before M1 deploy:** The `bot_risk_event` DB table (proposed in M1)
MUST be created and the `_load_status()`/`_save_status()` methods replaced with
DB reads/writes BEFORE deploying crypto circuit breakers.

**Also missing from plan:** The `threading.RLock()` on line 74 is a threading
primitive. `AutoTradingService` runs in asyncio. Mixed threading/asyncio locking
is a latent deadlock risk. The plan should replace `threading.RLock` with
`asyncio.Lock` when wiring circuit breakers to the async trading loop.

### FAIL-2 - CRITICAL: get_positions() hardcode will silently drop crypto positions
**File:** `backend/app/services/broker/alpaca.py`, line 315

`"asset_type": "equity"` is hardcoded in `get_positions()`. The plan's
`reconcile_holdings_from_alpaca()` (line 839) calls `get_positions()` and will
create all crypto positions with `asset_type="stock"` instead of `"crypto"`.

**Impact:** The `Holding.asset_type` field will be wrong for all crypto positions,
breaking the existing `CRYPTO_SYMBOLS` filtering in `useInvestData.ts` and the
InvestPage crypto tab. The plan notes the hardcode on line 30 of the plan file
but does NOT show the fix being applied in `reconcile_holdings_from_alpaca()`.

**Required fix:** `get_positions()` must read `asset_class` from the Alpaca
response and map it: `"crypto" if position.get("asset_class") == "crypto" else "equity"`.
This must propagate through `reconcile_holdings_from_alpaca()` as well.

### FAIL-3 - CRITICAL: requirements.txt vs requirements-docker.txt sync not addressed
**File:** Plan lines 145-146 (M3 dependency changes)

The plan adds `stable-baselines3`, `gymnasium`, `hmmlearn`, `d3rlpy`,
`polygon-api-client`, `ccxt`, `glassnode-api` to `requirements.txt` and ONLY
`onnxruntime>=1.17.0` to `requirements-docker.txt`.

**Architectural conflict:** `stable-baselines3` depends on PyTorch. If it is
added to `requirements.txt` and a developer runs `pip install -r requirements.txt`
on Cloud Run locally (or if the wrong requirements file is used in cloudbuild.yaml),
the container will attempt to install 1.5GB of PyTorch. More critically: the 2026-02-09
production lesson states these files MUST be kept in sync for non-ML packages.
Any shared utility imported by BOTH training scripts and Cloud Run endpoints
(e.g., `gymnasium` for environment type hints) must be in `requirements-docker.txt`.

**Missing from plan:** No `diff requirements.txt requirements-docker.txt` gate
is mentioned in the deployment sequence for M3.

### FAIL-5 - HIGH: AutoTradingService vLLM semaphore is missing
**File:** `backend/app/services/auto_trading_service.py`, `__init__` / class body

The plan (Pre-Sprint item 2) proposes `asyncio.Semaphore(2)` in
`AutoTradingService.__init__`. But `AutoTradingService` uses `__new__` for
singleton (line 57) and has no `__init__`. The `initialize()` classmethod
(line 63) is the correct injection point.

**Architectural conflict:** The semaphore must be a class-level attribute, not
instance-level, because the singleton never re-initializes. If implemented as
`self._vllm_semaphore` in a non-existent `__init__`, the semaphore will never
be created. Must be: `cls._vllm_semaphore = asyncio.Semaphore(2)` inside
`initialize()`.

**Second issue:** The semaphore is `asyncio`-bound to the event loop at
creation time. If `initialize()` is called from `lifespan()` (which it is —
`main.py` line 53), the loop is already running, so `asyncio.Semaphore(2)` is
safe. But it must be called within the async context, not at module import time.

### FAIL-6 - HIGH: DRLStrategy in StrategyRegistry conflicts with `min_confidence=0.6` hardcode
**File:** `backend/app/services/auto_trading_service.py`, lines 182-191 and 422-428

Every strategy is created with `min_confidence=0.6` hardcoded in both the
initial creation loop AND the rescan loop. `DRLStrategy` returns continuous
actions in [-1, 1] — confidence semantics differ from rule-based strategies
(which return 0.0-1.0 discrete confidence). If `DRLStrategy.generate_signal()`
returns `{"action": "buy", "confidence": 0.73}` where 0.73 is a normalized
softmax probability, the existing 0.6 threshold may be too conservative or
inapplicable.

**Required:** `DRLStrategy` must document its confidence semantics explicitly,
and `_process_strategy()` should check `strategy.name == "drl_strategy"` to
apply a DRL-specific confidence threshold OR `DRLStrategy` must normalize its
output to match the 0.0-1.0 scale used by rule-based strategies.

### FAIL-7 - HIGH: M5 DOM (Depth of Market) polling at 1s conflicts with Cloud Run rate limits
**Plan line 204:** "Real-time updates (polling at 1s or WebSocket if available)"

`DOMVisualization.tsx` polling at 1s per symbol means 1 request/second to
`/market/data/{symbol}/quotes` (or equivalent). With multiple open panels and
multiple symbols, this will hit Alpaca's rate limits (200 requests/min for data
subscriptions) and Cloud Run's concurrent request limit.

**Required fix:** DOM data MUST go through the existing WebSocket infrastructure
(`/ws/market/feed` via `LiveDataProvider.tsx`) not polling. The plan must
explicitly route DOM updates through the existing WebSocket subscription model.
Polling at 1s is a production risk.

### FAIL-8 - HIGH: Options M6 has no Alpaca options API endpoint constant defined
**File:** `backend/app/services/broker/alpaca.py`, lines 24-31

The file defines endpoint constants:
```
PAPER_API_BASE_URL = "https://paper-api.alpaca.markets/v2"
LIVE_API_BASE_URL = "https://api.alpaca.markets/v2"
```

Alpaca's options API uses a DIFFERENT base URL:
`https://paper-api.alpaca.markets/v2/options/` for options chains.
The options quotes require the Market Data API:
`https://data.alpaca.markets/v1beta1/options/`

**Architectural conflict:** The existing `_api_request()` method in
`ApiBaseBroker` uses `self.api_base_url` for ALL requests. Options chains and
Greeks data require a different base URL than trading operations. The plan does
NOT address this dual-URL requirement.

**Required:** `AlpacaBroker` must support a secondary `data_api_base_url`
for market data requests, or options methods must construct full URLs directly.

### FAIL-9 - MEDIUM: No fan-out protection for autonomous scanner + portfolio risk check
**File:** `backend/app/services/auto_trading_service.py`, `_auto_trading_loop()`

M1 proposes adding `_check_portfolio_risk()` at the TOP of the loop. Currently
the loop processes strategies sequentially (line 352). With 20 symbols x N
strategies = potentially 100+ strategy objects, AND the new portfolio risk check
making a broker API call (GET /positions) every 60 seconds per user, this
creates N concurrent API calls per loop iteration for multi-user scenarios.

**The plan does not address:** How many concurrent users can run bots
simultaneously? Each user's loop calls `get_positions()` every 60s for the
portfolio risk check. 10 concurrent users = 10 calls/min just for risk checks,
before any signal generation. This is manageable but not documented.

**Required:** Add explicit note that `_check_portfolio_risk()` should cache the
positions result for the duration of the loop iteration (not re-fetch per
strategy). A single `broker.get_positions()` call at the TOP of the loop,
stored as a local variable, should be passed to all sub-checks.

### FAIL-10 - MEDIUM: `reconcile_holdings_from_alpaca()` creates new holdings with `asset_type="stock"` hardcode
**File:** `backend/app/services/broker/alpaca.py`, line 898

When creating new holdings not in the local DB, the reconcile method hardcodes
`asset_type="stock"`. After M1 adds crypto positions via Alpaca, any crypto
position created through auto-trading will be reconciled back into the DB with
`asset_type="stock"`, breaking the InvestPage crypto tab filtering.

This is a separate bug from FAIL-2 (which is in `get_positions()`). Both must
be fixed together in M1.

### FAIL-11 - MEDIUM: M3 training scripts have no GCS model versioning strategy
**Plan lines 139-140:** Scripts write ONNX models to
`gs://elson-33a95-elson-models/drl-models/`

**Missing from plan:**
1. No model versioning scheme (no timestamp, hash, or semver in path)
2. No atomic swap strategy (Cloud Run could load a partial write)
3. No rollback procedure if new ONNX model degrades performance
4. No model metadata file (training date, validation Sharpe, embargo window)

If the H100 job writes `drl_equity.onnx` and Cloud Run is reading it
simultaneously during an update, the Cloud Run instance will load a corrupt model.
A versioned path (`drl-models/v{timestamp}/drl_equity.onnx`) with an atomic
`latest.json` pointer file is required.

## WARN Items (Non-blocking but require attention)

### WARN-1: minScale=1 in plan but MEMORY.md says minScale=0
Plan line 9: "Cloud Run: minScale=1". MEMORY.md (2026-02-23 entry) says "Cloud Run
minScale=0: Set to scale-to-zero". This is a factual inconsistency. The plan should
be trusted (plan explicitly says "MEMORY.md minScale=0 is stale — needs update").
MEMORY.md must be updated before the next sprint.

### WARN-2: `list_tradable_assets()` hardcodes `asset_class: "us_equity"` (line 943)
This is the same hardcode issue as FAIL-2 but in the asset discovery path.
After M1 adds crypto support, `list_tradable_assets()` will NOT return crypto
assets. A separate `list_tradable_crypto_assets()` method with
`asset_class: "crypto"` is needed for the Discover tab and screener (M5).

### WARN-3: TradeDecisionLog schema for IQL needs outcome_pnl column verification
Plan M3 assumes `TradeDecisionLog` has outcome columns for the IQL replay buffer.
The `_fill_pending_outcomes()` method in `auto_trading_service.py` populates
these retroactively. Before building `reinforcement_learning.py`, verify that
`TradeDecisionLog` has: `outcome_pnl`, `outcome_return_pct`, `outcome_filled_at`.
If these columns are missing, M3 has no training data.

### WARN-4: `d3rlpy` IQL requires `(s, a, r, s')` tuples — plan oversimplifies
Plan states "IQL avoids behavior policy". IQL (Implicit Q-Learning) still requires
transition tuples `(state, action, reward, next_state, done)`. The plan proposes
using `TradeDecisionLog` as the replay buffer, but the log stores individual
decisions, not transitions. A preprocessing step to construct `(s_t, a_t, r_t, s_{t+1})`
pairs from sequential log entries is required and is missing from the plan.

### WARN-5: `VolatilitySurface.tsx` (M6) — 3D IV surface requires THREE.js or similar
A 3D implied volatility surface (strike x expiration x IV) is not feasible with
`lightweight-charts` (TradingView charting, 2D only). This requires either:
- `plotly.js` (large, 3.5MB gzipped)
- `three.js` with custom rendering (complex, significant development time)
- A simplified 2D heatmap (faster, more practical)

The plan does not list a dependency for 3D charting in M6. This is a gap that
will extend the M6 timeline if not resolved early.

### WARN-6: `WorkspaceLayoutManager.tsx` needs backend persistence endpoint
The plan proposes save/load custom layouts but does not include a corresponding
backend endpoint (`POST /users/workspace-layout`, `GET /users/workspace-layout`).
localStorage is insufficient (clears on browser data wipe, not synced across
devices). This endpoint is missing from the API endpoint summary.
