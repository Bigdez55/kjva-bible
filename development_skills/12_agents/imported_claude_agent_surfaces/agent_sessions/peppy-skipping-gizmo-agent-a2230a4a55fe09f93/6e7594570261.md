### INVENTION 1: vDRAM 4-Tier Unified Memory Engine
**Files:** `kernel/vdram/vdram.h`, `pf_handler.c`, `tier_migration.c`, `mr_registry.c`
**LOC:** ~1,195

**Current Paradigm:** Every OS treats memory tiers as separate subsystems. GPU memory (HMM/UVM), swap, and RDMA are three distinct kernel paths with no semantic awareness. You get `mmap()`, `cudaMemcpy()`, and `ibv_reg_mr()` as three unrelated syscall families. Linux 6.x CXL tiering (cgroup2 memory tiers) exists but is flat: no workload classification, no energy gating, no per-page Merkle commitment, no JIT-driven placement hints. NVIDIA's Unified Memory (UVM) only handles GPU-CPU, not NVMe or RDMA.

**Disruptive Alternative:** A single page fault handler that is the OS's central nervous system -- it receives JIT semantic hints (this is a MATMUL, this is a cold checkpoint), gates every access through attestation and energy budgets, places pages across 4 tiers (GPU HBM / DRAM / NVMe / RDMA remote), and commits Merkle hashes before cross-domain migrations. The page fault path becomes a programmable policy engine.

**Technical Assessment:**
- The 8-step orchestration pipeline in `pf_handler.c` (lines 256-362) is architecturally clean: fault -> MR lookup -> attestation -> IOMMU check -> tier selection -> migration -> Merkle commit -> energy accounting -> PTE wiring
- Tier migration in `tier_migration.c` covers all 6 bidirectional paths (T0<->T1, T1<->T2, T1<->T3, T2<->T3) with proper HMM, DMA, and RDMA WRITE semantics
- MR registry has SPDM-bound rkey rotation and cryptographic key zeroization on free
- CRITICAL VULNERABILITY: Linear MR search in `pf_find_mr()` (lines 36-52 of pf_handler.c) -- `list_for_each` on hot path. With thousands of MRs, this is O(n) per page fault. Must be rbtree or radix tree for production
- CRITICAL VULNERABILITY: `rkey_table` in mr_registry.c is a flat 65536-entry array indexed by `rkey % MR_MAX_PER_NODE` -- hash collision risk. Two MRs with rkeys that collide modulo 65536 will corrupt the table
- The heuristic tier selection (lines 152-186 pf_handler.c) is hardcoded: `access_count > 16` for T1, `> 64` for T0. No ML. No learning. This is 1990s page replacement policy with extra steps

**Disruption Score:**
- Novelty: 8/10 -- No OS unifies all 4 tiers through a single programmable fault handler with semantic JIT hints
- Utility: 7/10 -- Eliminates the "which memory API" decision for AI workloads
- Defensibility: 6/10 -- Deep kernel integration is hard to copy, but Linux upstream CXL/TPP efforts are approaching this
- Scalability: 5/10 -- Linear MR search and spinlock contention on hot path will not scale past ~1000 MRs
- **Composite: 6.5/10**

### INVENTION 2: Contra-Rotation Scheduler
**Files:** `kernel/vdram/contra_rotation.c`, `prefetch_engine.c`
**LOC:** ~493

**Current Paradigm:** Every training system uses synchronized phases: all GPUs compute, then all GPUs communicate, then wait. NVIDIA's NCCL + NVLink is "train together, communicate together, idle together." The idle gap is pure waste. Linux's CPU frequency governors (schedutil, ondemand) react to load, but have no concept of inter-device phase coordination.

**Disruptive Alternative:** Stagger devices in 3-phase rotation: COMPUTE -> COOLDOWN -> LOAD. While device A computes, device B prefetches next expert weights into T1 memory. Thermal ceiling IS the scheduling primitive (60-80C in consumer mode). During NCCL idle gaps (enterprise mode), scale RDMA lanes down by 75%.

**Technical Assessment:**
- Consumer scheduling in `cr_schedule_consumer()` (lines 136-176) reads thermal sensors at 500ms intervals and transitions based on temperature thresholds. Simple, robust, correct
- Enterprise scheduling in `cr_schedule_enterprise()` (lines 179-200) uses NCCL idle detection with a 50ms threshold. When NCCL is idle, it transitions to COOLDOWN and scales lanes down
- Prefetch engine (lines 46-94 of prefetch_engine.c) sorts experts by MoE routing weight and pre-migrates them from T2/T3 -> T1 during cooldown. This is the KEY innovation: it converts thermal waste into useful I/O
- CRITICAL VULNERABILITY: The scheduler uses a global static `g_cr_sched` pointer -- single instance only. No multi-tenant support
- VULNERABILITY: 500ms `THERMAL_POLL_MS` is too slow for consumer gaming workloads where thermal spikes can occur in <100ms
- The O(n^2) expert weight update in `vdram_prefetch_update_weights()` (nested loop lines 139-155) will not scale past ~256 experts

**Disruption Score:**
- Novelty: 9/10 -- "Thermal ceiling as a scheduling primitive" is genuinely new thinking. Nobody else treats the idle phase as a free memory bus window
- Utility: 7/10 -- Theoretically eliminates T2/T3 fault latency for MoE workloads by pre-loading experts
- Defensibility: 7/10 -- Requires deep integration with memory subsystem and thermal sensors. Hard to bolt on to existing OSes
- Scalability: 4/10 -- Global singleton, 500ms polling, O(n^2) weight updates
- **Composite: 6.75/10**

### INVENTION 3: RDMA Attestation (SPDM + AES-256-GCM)
**Files:** `kernel/security/rdma_attestation.c`
**LOC:** 289

**Current Paradigm:** RDMA (InfiniBand, RoCEv2) trusts the fabric blindly. There is no mutual attestation before memory registration. rkeys are often sequential or low-entropy. No per-QP encryption -- everything is plaintext on the wire (IB spec has no mandatory encryption below RDMA-KT draft). NVIDIA's ConnectX-7 has inline IPsec, but it is QoS-gated, not identity-gated.

**Disruptive Alternative:** Full SPDM 1.2 mutual attestation before ANY MR registration. rkeys cryptographically bound to peer certificate chain. Per-QP AES-256-GCM with NIC offload. TPM PCR extension on every attestation event. PQ-ready session establishment (ML-KEM-768).

**Technical Assessment:**
- SPDM negotiation in `spdm_negotiate_capabilities()`, `spdm_get_certificate()`, and `spdm_challenge_auth()` follows the SPDM spec correctly (message format, nonce challenge, signature verification)
- Session cache with 1024-entry hashtable and 1-hour max age (lines 52, 32-33). Cached lookup targets <200ns
- rkey-to-session binding (lines 194-212) prevents rkey reuse across sessions -- good security
- TPM PCR extension on every event (lines 168-179) creates a hardware-rooted audit trail
- CRITICAL VULNERABILITY: Session cache uses `*(u32 *)digest` as hash key (line 147) -- only 4 bytes of the SHA-256 digest. With 1024 buckets, collision rate is ~0.02% per entry but degrades rapidly at high load
- CRITICAL VULNERABILITY: `spdm_challenge_auth()` derives session key as `SHA-256(cert_digest || nonce || resp_nonce)` using a non-standard KDF. Should use HKDF-SHA256 per SPDM 1.2 spec section 9.35.2
- ML-KEM-768 is mentioned in comments but not implemented in code (line 9: "PQ-ready: ML-KEM-768 session establishment" -- claim only)

**Disruption Score:**
- Novelty: 7/10 -- Per-QP attestation with SPDM is rare. Most RDMA deployments are trust-the-fabric
- Utility: 8/10 -- Eliminates the entire class of RDMA-based lateral movement attacks
- Defensibility: 6/10 -- SPDM is an industry standard. BlueField-5 will likely have this built-in
- Scalability: 5/10 -- 1024-entry cache with u32 hash key is not enterprise-grade
- **Composite: 6.5/10**

### INVENTION 4: Zero-Copy Gateway (QP Relay)
**Files:** `gateway/zc_gw/qp_relay.c`, `entropy_classifier.c`
**LOC:** 438

**Current Paradigm:** Multi-protocol fabric gateways buffer packets. IB-to-RoCEv2 bridging (e.g., NVIDIA Spectrum switches) requires store-and-forward because header formats differ. This adds 3-10us latency and burns DRAM bandwidth for payloads that just need header rewriting.

**Disruptive Alternative:** Header-only translation at line rate. Payload DMA-mapped once, never copied. Pre-computed header templates cached per QP entry. FPGA path targets <800ns. Entropy classifier drives path selection: high entropy (encrypted gradients) -> IB, medium -> RoCEv2, low (sparse) -> UET.

**Technical Assessment:**
- Header translation functions `translate_ib_to_roce()` and `translate_roce_to_ib()` (lines 110-138) are minimal and correct: just BTH + UDP header rewrites, QPN update
- QP table uses RCU for lockless reads (line 188) -- good for hot path
- Entropy classifier (`entropy_classifier.c`) implements Shannon entropy O(n) with fixed-point log2 LUT
- FPGA vs DPU path selection in `zc_gw_relay_packet()` (lines 278-288) with scatter-gather forwarding
- CRITICAL VULNERABILITY: The `log2_lut[256]` array in entropy_classifier.c is 98% zeros -- only indices 0,1,2,3,4,5,6,7,8,16,32,64,128,255 are populated. Every other index maps to 0, making the entropy calculation severely wrong for most byte distributions. The classifier will underestimate entropy massively
- VULNERABILITY: `translate_ib_to_roce()` hardcodes UDP dest port 4791. This is correct for RoCEv2 but the source port should be hash-based per RoCEv2 spec (annex A.17.1) for ECMP load balancing
- The UET header translation is referenced (`header_translation_ib_to_uet()`) but not implemented in the codebase

**Disruption Score:**
- Novelty: 6/10 -- NVIDIA SHARP already does in-network reduction. Cisco/Arista bridging exists
- Utility: 8/10 -- Eliminates 3-10us of copy latency per packet in mixed-fabric deployments
- Defensibility: 4/10 -- NVIDIA will ship this in Spectrum-X eventually
- Scalability: 7/10 -- RCU + pre-computed headers scale well. 1M QP table is generous
- **Composite: 6.25/10**

### INVENTION 5: XISC JIT (LLVM-based with Policy Annotations)
**Files:** `genvirt/jit/xisc_llvm_jit.c`, `xisc/include/xisc.h`
**LOC:** 530

**Current Paradigm:** JIT compilers translate code. They do not emit placement policy. NVIDIA PTX -> SASS, WebAssembly -> native, BPF JIT -- all produce executable code, period. The runtime system separately decides memory placement, energy budgets, and security gates.

**Disruptive Alternative:** The JIT has SEMANTIC CONTEXT. When it sees a MATMUL opcode, it simultaneously emits: (a) LLVM IR for the computation, (b) a vDRAM placement hint (output goes to T0 GPU HBM), (c) an energy budget annotation (full power), and (d) a Merkle commit requirement for ALL_REDUCE. These annotations flow to vDRAM, EP-RDMA, and the Merkle accumulator simultaneously. The JIT becomes the system's semantic front-end.

**Technical Assessment:**
- `jit_emit_policy_annotation()` (lines 99-154) maps opcodes to multi-dimensional policy hints: tier, workload class, energy budget, Merkle commit, attestation, prefetch weight. This is THE bridge between user intent and system behavior
- LLVM pipeline with MCJIT, -O2 default, auto-vectorization (SLP pass, line 367) is production-grade
- 64K code cache with W^X semantics (line 41) -- proper security for JIT code
- Cache eviction is random (line 305: `ktime_get_ns() % JIT_CODE_CACHE_ENTRIES`) -- no LRU, no LFU. This is lazy
- CRITICAL VULNERABILITY: JIT hot path uses `mutex_lock` for cache lookup (line 280). Mutexes sleep. On the hot path, this CANNOT hit <50ns target. Must use rwlock or RCU
- CRITICAL VULNERABILITY: The XISC instruction encoding in `xisc.h` (lines 82-92) uses 4-bit register fields (16 registers) and 8-bit immediate. This is severely constrained -- most real ISAs need 32+ registers and 16-bit immediates minimum
- CRITICAL: The bytecode parser in `jit_compile_block()` (lines 192-237) skips unknown opcodes by incrementing pc by 1. But XISC instructions are 32-bit (4 bytes per xisc.h line 94). The parser treats bytecode as byte-stream (pc++ per opcode) while the ISA spec says 32-bit words. These two views are inconsistent

**Disruption Score:**
- Novelty: 9/10 -- JIT-as-policy-engine is genuinely new. No existing system does this
- Utility: 8/10 -- Eliminates the disconnect between compiler and runtime resource management
- Defensibility: 8/10 -- Requires co-design of ISA, JIT, and kernel memory subsystem. Very hard to bolt on
- Scalability: 4/10 -- Mutex on hot path, random eviction, inconsistent instruction encoding
- **Composite: 7.25/10**

### INVENTION 6: EP-RDMA (Energy-Proportional Fabric)
**Files:** `tools/ep_rdma/fabric_energy_controller.py`
**LOC:** 433

**Current Paradigm:** RDMA fabrics run at full power continuously. 8x200G (1.6Tbps) lanes burn ~400W per node whether the fabric is in use or not. NCCL collectives are bursty: 200ms all-reduce, then 300ms idle. During idle, the fabric consumes 100% of its peak power for 0% utilization.

**Disruptive Alternative:** Scale SerDes lanes proportionally to demand. During NCCL idle gaps, scale 8x200G -> 2x200G = 75% power reduction on idle. Lossless guarantee via credit drain protocol before lane transition. 10us SerDes PLL re-lock time. 10ms control loop on DPU ARM cores. Demo 9 gate: >20% overall savings.

**Technical Assessment:**
- State machine in `_update_state()` (lines 244-291) is clean: DENSE <-> TRANSITION <-> IDLE. Credit drain before scale-down prevents packet loss
- DPU interface (`DPUInterface` class) has proper simulation mode, credit drain timeout (20ms), and lane verification after scaling
- Energy accounting tracks actual vs baseline Joules with savings percentage
- NCCL simulation pattern (lines 408-422) models realistic training workload: 50ms collective, 250ms dense, then 2s idle gap
- VULNERABILITY: The entire controller is Python running in userspace. 10ms control loop in Python is unreliable -- GC pauses alone can exceed 10ms. This must be C on the DPU ARM cores in production
- The 50ms idle threshold is hardcoded. Different training workloads have different idle profiles (DP vs PP vs EP). Should be adaptive
- No consideration for PCIe Gen5 link state (L0s/L1) which already provides some lane scaling at the link layer

**Disruption Score:**
- Novelty: 6/10 -- IEEE 802.3cg Energy Efficient Ethernet (EEE) exists. SerDes lane scaling for RDMA-specific workloads is incremental
- Utility: 7/10 -- 25-40% fabric power savings is meaningful at datacenter scale ($2-4M/year for a 1000-node cluster)
- Defensibility: 3/10 -- NVIDIA could ship this as a BlueField firmware update. Low barrier
- Scalability: 6/10 -- Control loop per node, no cross-cluster coordination
- **Composite: 5.5/10**

### INVENTION 7: Merkle Verifiable Compute
**Files:** `tools/verifiable/merkle_collective.py`
**LOC:** 370

**Current Paradigm:** Distributed training trusts every node implicitly. If a node's memory is corrupted (bit flip, firmware bug, malicious modification), the gradient contributions are silently wrong. There is no mechanism to detect that rank 47 of a 1024-rank training run contributed corrupted gradients. The entire model is poisoned without detection.

**Disruptive Alternative:** Every collective (all-reduce, all-gather) carries a 64-byte Merkle commitment: 32-byte SHA-256 root over all input tensors, 32-byte context. O(log n) proof generation and verification. Any single-bit tamper in any rank's contribution is detected with 100% certainty. Production target: SHARP switch ASIC as native primitive (zero overhead).

**Technical Assessment:**
- `MerkleTree` class (lines 113-158) is textbook correct: power-of-2 padding, bottom-up build, O(log n) proofs with sibling hashes
- `MerkleCollectiveAccumulator` (lines 165-251) provides clean contribute -> commit -> verify workflow
- `verifiable_all_reduce()` (lines 257-302) integrates with numpy for actual tensor arithmetic
- `demo10_tamper_detection_test()` runs 100 trials with random tamper injection -- 100% detection rate is mathematically guaranteed by hash properties
- CRITICAL WEAKNESS: SHA-256 per 16KB tensor per rank per collective per step. For a 1024-rank all-reduce of 1GB tensors, that is 1024 * 65536 = 67M SHA-256 operations. At ~4GB/s SHA-256 throughput, the Merkle tree build alone takes ~16 seconds. This is UNACCEPTABLE for a 50ms collective
- The SHARP switch ASIC claim is speculative. NVIDIA SHARP does not currently support Merkle commitments. Custom ASIC development is a multi-year, multi-$100M effort
- No incremental update support -- entire tree is rebuilt every collective

**Disruption Score:**
- Novelty: 8/10 -- Verifiable distributed compute in the training loop is genuinely new
- Utility: 6/10 -- Solves a real problem (silent data corruption) but the performance cost may be prohibitive
- Defensibility: 5/10 -- Merkle trees are well-known. The innovation is application to collectives, not the mechanism
- Scalability: 3/10 -- O(n * tensor_size) hashing per collective is a dealbreaker at scale
- **Composite: 5.5/10**

**Disruptive Alternative:** The mesh IS the OS. genos-meshd is a kernel module: boot -> mDNS discovery -> Ed25519 identity pairing -> WireGuard auto-config -> gossip (thermal + GPU + vDRAM state) -> SoftiWARP over WireGuard for vDRAM T3. Target: <10 seconds from cold boot to operational mesh with active gossip.

**Technical Assessment:**
- Discovery thread (lines 74-144) uses mDNS for zero-configuration peer discovery, Ed25519 identity verification, and automatic WireGuard tunnel setup
- Gossip thread (lines 150-222) exchanges thermal, GPU utilization, and vDRAM availability data between peers. Signed with Ed25519
- `genos_meshd_best_t3_peer()` (lines 291-308) selects the best peer for vDRAM T3 placement based on available remote memory -- elegant resource-aware routing
- CRITICAL VULNERABILITY: Private key `meshd.my_privkey[32]` stored in a global static struct in kernel memory. This is fine for kernel modules but the variable is never zeroized on module unload
- VULNERABILITY: `gossip_send()` sends gossip to all peers in a loop while holding spinlock (line 174-184). If a UDP send blocks or takes >1ms, the entire gossip system stalls. Should release lock before sending
- VULNERABILITY: 64-peer limit (`MESHD_MAX_PEERS = 64`) is hardcoded. For consumer mesh this is fine; for enterprise it is too small
- No peer eviction or liveness detection -- stale peers accumulate until reboot

**Disruption Score:**
- Novelty: 7/10 -- Kernel-native mesh with resource gossip is unique. Tailscale is userspace-only
- Utility: 7/10 -- Zero-config mesh for heterogeneous devices is a strong UX proposition
- Defensibility: 5/10 -- Tailscale or Apple could add resource gossip in 6 months
- Scalability: 4/10 -- 64 peers, spinlock in gossip, no peer eviction

### INVENTION 9: XKABI Capabilities (128-bit Handle Table)
**Files:** `kernel/security/xkabi_capabilities.c`, `xisc/conformance/xkabi_conformance.c`
**LOC:** 653

**Current Paradigm:** Unix/Linux uses ambient authority (uid/gid). Root bypasses everything. Android/SELinux adds MAC but is additive complexity. Fuchsia's Zircon and seL4 use capabilities, but they are microkernel-only and not designed for RDMA/GPU workloads. No existing OS has capability handles that embed RDMA rights, vDRAM mapping rights, and GPU execution rights in the same token.

**Disruptive Alternative:** XKABI: 128-bit handles encoding object_id (64b), rights bitmask (16b), generation counter (16b), and HMAC-SHA256 integrity (16b). No ambient authority. No root. Rights only narrow on derivation (never widen). Epoch-based bulk revocation. Audit logging on every operation. 26/26 conformance test suite passing.

**Technical Assessment:**
- Handle encoding/verification (lines 78-122) uses per-table HMAC-SHA256 key with constant-time comparison
- Derivation enforcement (lines 210-233) correctly blocks rights expansion: `if ((derived_rights & ~parent_rights) != 0) return -EACCES`
- Epoch-based bulk revocation (lines 264-272) is elegant: increment global epoch, all pre-epoch handles become ESTALE
- Conformance suite (26 tests) covers allocation, derivation (including 3-level chains), invocation, revocation, epoch revocation, RDMA barriers, and performance
- CRITICAL VULNERABILITY: 2-byte truncated HMAC (lines 95-96: `memcpy(&handle_out[14], mac, 2)`). This means only 16 bits of integrity protection. An attacker who can brute-force 2^16 = 65536 HMAC values can forge any capability handle. This is CATASTROPHICALLY weak for a security primitive. Must be at least 8 bytes (64 bits)
- VULNERABILITY: IDR lookup uses `object_id & 0xFFFFFF` (line 194) -- truncating 64-bit object ID to 24 bits. This creates collision risk: two different object IDs can map to the same IDR entry
- The conformance suite tests in `xkabi_conformance.c` test a simplified userspace model, NOT the actual kernel implementation. The gap between test model and kernel code is a testing fidelity concern

**Disruption Score:**
- Novelty: 8/10 -- Capability handles with RDMA+GPU+vDRAM rights in a unified model is novel
- Utility: 7/10 -- Eliminates the root/ambient authority problem for heterogeneous compute
- Defensibility: 7/10 -- Deep kernel ABI change. Cannot be bolted onto Linux without forking the security model
- Scalability: 6/10 -- IDR-based handle table scales well. Epoch revocation is O(1)
- **Composite: 7.0/10**

This eliminates the 3+ year hypervisor development timeline and lets the JIT ship in 6 months as a PyTorch extension.
