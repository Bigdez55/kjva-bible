## 1. COORDINATION PATTERNS -- How the 10 Inventions Coordinate

### 1.1 The Central Orchestrator Pattern: pf_handler.c

The page fault handler (`/tmp/rfc002_analysis/gen-os/kernel/vdram/pf_handler.c`) is the **single point of orchestration** for the entire system. It implements an 8-step synchronous pipeline that touches 6 of the 10 inventions on every page fault:

```
Page Fault Entry (vm_ops->fault)
  |
  Step 0: Timing instrumentation (ktime_get)
  Step 1: VA -> PA + MR lookup          [Invention #1: vDRAM]
  Step 2: Attestation gate              [Invention #3: RDMA Attestation / Invention #8: Mesh Identity]
  Step 3: IOMMU / MR boundary           [Invention #9: XKABI Capabilities]
  Step 4: Tier selection                 [Invention #5: XISC JIT hints / Invention #6: EP-RDMA energy gate]
  Step 5: Tier migration                 [Invention #1: vDRAM 4-tier engine]
  Step 6: Merkle commitment              [Invention #7: Verifiable Compute]
  Step 7: Energy accounting              [Invention #6: EP-RDMA]
  Step 8: PTE wiring                     [Invention #1: vDRAM]
  |
  Return to Linux mm
```

**Coordination observation**: This is a **Pipeline Choreography** pattern -- not a hub-and-spoke. Each step calls into a distinct subsystem but the fault handler **owns the sequencing contract**. No subsystem calls back into pf_handler; data flows strictly downward through the pipeline. This is correct for a latency-critical hot path (<2us target).

### 1.3 Identified Collision Zones

| Collision Zone | Contending Subsystems | Current Resolution | Risk |
|---|---|---|---|
| `vdram_node->energy_nj` | pf_handler (write), EP-RDMA controller (read), contra-rotation (read) | atomic64 operations | LOW -- reads are non-blocking |
| `vdram_page_meta->hint` | XISC JIT (write via `vdram_jit_hint_update`), pf_handler (read) | per-page spinlock | MEDIUM -- lock contention on hot pages |
| `vdram_page_meta->current_tier` | tier_migration (write), pf_handler (read), prefetch_engine (read) | per-page spinlock | MEDIUM -- same lock as hints |
| `meshd.peers[]` array | discovery_thread (write), gossip_thread (read+write) | global meshd.lock spinlock | HIGH -- single lock for 64 peers, contention at scale |
| `session_cache` hashtable | pf_handler/attestation (read), `rdma_attestation_revoke_peer` (write) | global session_cache_lock | MEDIUM -- revocation path blocks fault path |
| `g_cr_sched->devices[]` | cr_scheduler_thread (write), `contra_rotation_compute_active` (read) | cr->lock spinlock | LOW -- reads are opportunistic |
| `qp_table` hashtable | zc_gw_qp_register (write), zc_gw_relay_packet (read) | RCU (lockless read) + table_lock (write) | LOW -- RCU is correct pattern |
| `rkey_table[]` | mr_registry (write), mr_lookup_rkey (read) | rcu_read_lock for reads, spinlock for writes | LOW -- correct pattern |

**Critical finding**: The meshd peer array lock is the highest-contention collision zone. With 64 peers doing 500ms gossip cycles, the discovery thread and gossip thread will serialize on every cycle. This will not scale past ~20 peers without sharding.

## 3. DATA FLOW ORCHESTRATION

### 3.1 Page Fault Flow (Hot Path)

```
Application accesses VA in vDRAM-managed VMA
  |
  [CPU generates #PF -> Linux mm -> vm_ops->fault]
  |
  vdram_fault_handler()
  |
  +-- pf_find_mr()              -- O(n) MR list walk under spinlock
  +-- pf_get_page_meta()        -- O(1) array index
  +-- pf_attestation_gate()     -- branch: T0/T1 skip, T2 XKABI, T3 SPDM/Ed25519
  |     +-- [T3 cached] ~200ns: spdm_session_find() via hashtable
  |     +-- [T3 new]   ~2ms:   full SPDM handshake + TPM PCR extend
  +-- pf_validate_mr_boundary() -- IOMMU domain check
  +-- pf_select_tier()          -- JIT hint priority -> energy gate -> access heuristic
  |     +-- reads vdram_jit_hint from page_meta (populated by XISC JIT)
  |     +-- calls vdram_energy_gate() (compares power_cap_mw)
  +-- vdram_migrate_page()      -- dispatches to HMM/NVMe DMA/RDMA WRITE
  +-- pf_merkle_commit_if_needed() -- SHA-256 hash under merkle_lock
  +-- pf_energy_account()       -- atomic64_add (nano-joules)
  +-- vdram_tier_resolve_page() -- returns struct page* for PTE
  +-- pf_wire_pte()             -- installs page in VMA
```

**Latency budget breakdown** (target <2us for T0/T1):
- MR lookup: ~100ns (spinlock + list walk)
- Page meta: ~10ns (array index)
- Attestation gate: 0ns (T0/T1 bypass) / ~200ns (T3 cached) / ~2ms (T3 new)
- Tier selection: ~50ns (branch + atomic read)
- Migration: 0ns (same tier) / ~500ns (T0<->T1 HMM) / ~10us (T2 NVMe) / ~100us (T3 RDMA)
- Merkle: 0ns (skip) / ~500ns (SHA-256 of 4K page)
- Energy: ~10ns (atomic add)
- PTE wire: ~50ns

**The latency target of <2us is achievable for T0/T1 only when**: no migration needed, no Merkle commit, T0/T1 bypass on attestation. The comment in pf_handler.c line 15 confirms this is the intended hot-path.

### 3.2 Thermal Event Flow

```
Physical sensor (thermal_zone device)
  |
  cr_read_temp_mc()             -- reads thermal_zone_get_temp() in milli-Celsius
  |
  contra_rotation scheduler thread (500ms poll loop)
  |
  +-- [COMPUTE -> COOLDOWN] when temp >= 80C or (temp >= 60C && elapsed >= 2s)
  |     +-- cr_transition_phase(dev, CR_PHASE_COOLDOWN)
  |           +-- vdram_on_cooldown_phase_start(node)
  |           |     +-- vdram_prefetch_trigger(node)
  |           |           +-- queue_work(prefetch_wq, prefetch_work)
  |           |                 +-- sort experts by routing weight
  |           |                 +-- migrate top-N from T2/T3 -> T1
  |           +-- [enterprise] ep_rdma_scale_lanes_down(lanes/4)
  |
  +-- [COOLDOWN -> LOAD] when temp < 55C && elapsed >= 500ms
  |     +-- cr_transition_phase(dev, CR_PHASE_LOAD)
  |
  +-- [LOAD -> COMPUTE] when contra_rotation_load_complete(device_id)
        +-- cr_transition_phase(dev, CR_PHASE_COMPUTE)
              +-- vdram_on_compute_phase_start(node)
              |     +-- flush_workqueue(prefetch_wq) -- drain prefetch before compute
              +-- [enterprise] ep_rdma_scale_lanes_up(max_lanes)
```

**Cross-subsystem data flow during thermal event**:
1. Thermal sensor -> contra-rotation scheduler (poll)
2. Contra-rotation -> vDRAM prefetch engine (callback)
3. Contra-rotation -> EP-RDMA controller (lane scaling)
4. Prefetch engine -> tier migration engine (workqueue)
5. Prefetch engine checks contra-rotation state (cooperative yield)

This is a **reactive event chain** with cooperative cancellation. The prefetch engine voluntarily yields when compute phase starts (`contra_rotation_compute_active()`). This is well-designed for avoiding resource contention.

### 3.3 Mesh Gossip Flow

```
genos_meshd_init()
  |
  +-- identity_ed25519_generate() -- create node keypair
  +-- kthread_run(meshd_discovery_thread)
  |     +-- [loop 1000ms]
  |           +-- mdns_announce(pubkey, port, hostname)
  |           +-- mdns_discover_peers() -> list of peer_info
  |           +-- for each unknown peer:
  |                 +-- identity_ed25519_verify_peer()
  |                 +-- wireguard_autoconfig_peer() -> genos{N} interface
  |                 +-- register in meshd.peers[] under meshd.lock
  |
  +-- kthread_run(meshd_gossip_thread)
        +-- [loop 500ms]
              +-- build gossip_state_msg (thermal, gpu_util, vdram_t3_avail, cr_phase)
              +-- identity_ed25519_sign(msg)
              +-- fanout to all active peers (under meshd.lock -- COLLISION ZONE)
              +-- gossip_recv_all() -> incoming peer states
              +-- for each received message:
                    +-- identity_ed25519_verify(signature)
                    +-- update peer state in meshd.peers[]
                    +-- if peer overtemp: contra_rotation_peer_overtemp()
```

**Key coordination detail**: Gossip carries the contra-rotation phase state. This means remote nodes know the phase state of their peers. This enables cross-device contra-rotation coordination: while Device A is in COOLDOWN, Device B can be in COMPUTE, and Device A's prefetch engine can pull data from Device B via T3.

## 4. PHASE EXECUTION -- Contra-Rotation as Orchestration Model

### 4.1 The 3-Phase State Machine

File: `/tmp/rfc002_analysis/gen-os/kernel/vdram/contra_rotation.c`

```
State Machine (per device):

    COMPUTE ----[temp >= 80C OR (temp >= 60C && elapsed >= 2s)]----> COOLDOWN
       ^                                                                |
       |                                                                |
       |                                            [temp < 55C && elapsed >= 500ms]
       |                                                                |
       |                                                                v
       +------------------[load complete]---------------------------- LOAD
```

**Staggering strategy**: With N devices, the ideal state is:
- N/3 devices in COMPUTE
- N/3 devices in COOLDOWN (prefetching for their next compute phase)
- N/3 devices in LOAD (receiving data from peer devices)

**Enterprise variant**: The NCCL idle detection replaces thermal polling as the transition trigger. Instead of temperature thresholds, a 50ms NCCL idle gap triggers COOLDOWN. This means the phase machine is **dual-mode**:

```
Consumer: thermal_temp -> phase transition
Enterprise: nccl_idle_ms -> phase transition
```

### 4.2 Cross-Device Phase Coordination

The contra-rotation scheduler supports up to 16 devices (`MAX_DEVICES = 16`). The key insight is in lines 150-154 of contra_rotation.c:

```c
if (temp_c >= THERMAL_CEILING_HIGH_C ||
    (temp_c >= THERMAL_CEILING_LOW_C &&
     elapsed_ms >= PHASE_MIN_COMPUTE_MS)) {
```

The transition logic is **per-device independent**. There is no global "phase coordinator" that assigns phases. Each device transitions based on its own thermal state. The staggering emerges naturally because different devices heat up at different rates.

**Observation**: This is an **emergent coordination** pattern. The contra-rotation name implies deliberate counter-phasing (like gears), but the actual implementation is opportunistic thermal response. The "contra-rotation" effect is a beneficial emergent property, not a hard-scheduled guarantee. In practice, this means:
- Under light load: all devices may sit in COMPUTE indefinitely
- Under heavy load: devices will naturally stagger as the hottest one enters COOLDOWN first
- Under extreme load: all devices may be in COOLDOWN simultaneously (no compute progress)

**Risk**: There is no starvation guard. If all devices are thermally throttled simultaneously, the system has zero compute capacity. A coordinator could enforce a minimum of 1 device in COMPUTE at all times.

### 5.3 Peer Selection for T3 Placement

`genos_meshd_best_t3_peer()` (line 291-308 of genos_meshd.c) selects the peer with the **most available T3 memory** that meets the size requirement. This is a **greedy best-fit** algorithm.

**Coordination gap**: There is no reservation or locking. Two concurrent page faults could both select the same peer and over-commit its T3 capacity. In a 7-device mesh with 500ms gossip cycles, the staleness window for `vdram_t3_avail_bytes` could cause significant over-allocation.

## 6. CONFLICT RESOLUTION -- Competing Resource Demands

### 6.1 Energy Budget Conflicts

The energy gate in pf_handler.c (Step 4, line 162-164) is the arbiter:

```c
if (hint->energy_budget_mw > 0 &&
    !vdram_energy_gate(node, hint->energy_budget_mw))
    t = TIER_T2_NVME_CXL;  /* energy-proportional demotion */
```

When the JIT wants T0 (250W GPU full power) but the energy budget is exhausted, the page is **demoted** to T2 (NVMe). This is a hard priority inversion -- energy conservation overrides JIT semantic hints.

**Resolution hierarchy**:
1. JIT hint (highest semantic priority)
2. Energy gate (overrides JIT if budget exceeded)
3. Access-frequency heuristic (fallback if no JIT hint)

### 6.2 Thermal vs Throughput

Contra-rotation forces COOLDOWN when temperature hits 80C, regardless of workload priority. The prefetch engine yields to compute phase (`contra_rotation_compute_active()` check at line 92 of prefetch_engine.c). This means:
- **Thermal always wins over throughput** (safety first -- correct)
- **Prefetch is preemptable** (correctness maintained, prefetch is advisory)
- **EP-RDMA is cooperative** (scale down on cooldown, scale up on compute)

### 6.3 Attestation vs Latency

The SPDM attestation path has two tiers:
- Cached session: <200ns (non-blocking to hot path)
- New session: <2ms (BLOCKING -- violates T3 latency target of <200us by 10x)

**Conflict**: When a new peer is encountered for the first time via T3, the page fault blocks for ~2ms. This is unavoidable but should be rare (sessions are cached for 1 hour). The TPM PCR extension (`attest_extend_tpm_pcr`) adds additional latency on the slow path.

### 6.4 Zero-Copy Gateway: Entropy vs Relay Table

The entropy classifier can **override** the relay table routing (line 229-241 of qp_relay.c). A packet with high entropy is routed to InfiniBand even if the relay table says RoCEv2. This is a soft override -- the relay table provides the endpoint (QPN, GID), but the entropy score selects the protocol.

**Potential conflict**: If the destination only supports RoCEv2 but entropy says IB, the header translation will produce IB headers for a RoCEv2-only endpoint. There is no capability check before protocol override.

## 9. ADVISORY NOTES

### 9.1 Architectural Risks

1. **Single point of failure**: `pf_handler.c` is the orchestrator for 6/10 inventions. A bug here causes silent data corruption (as the file's own warning on line 16 notes). This file needs the highest test coverage in the project.

2. **Lock hierarchy not documented**: The code uses spinlocks in: `node->mr_lock`, `page->lock`, `node->merkle_lock`, `meshd.lock`, `session_cache_lock`, `rkey_table_lock`, `cr->lock`. There is no documented lock ordering. Deadlock risk exists if any path acquires these in inconsistent order.

3. **meshd peer lock contention**: The single `meshd.lock` spinlock protects a 64-peer array accessed by both the discovery thread (writes) and gossip thread (reads + writes). At full 64-peer capacity with 500ms gossip intervals, this is a bottleneck.

4. **XISC JIT cache eviction is random**: Line 305 of xisc_llvm_jit.c uses `ktime_get_ns() % JIT_CODE_CACHE_ENTRIES` for eviction. This evicts a random entry, potentially a hot one. LRU or LFU would be more appropriate.

5. **No backpressure on T3 migration**: `migrate_to_remote()` blocks for up to 5 seconds on RDMA completion. If multiple page faults trigger T3 migration simultaneously, the migration workqueue (4 threads) can saturate and all faults queue behind it.

### 9.2 Positive Design Patterns

1. **RCU in QP relay table**: The zero-copy gateway uses RCU for lockless read-side access to the 1M QP table. This is the correct pattern for a packet-processing hot path.

2. **Atomic operations for statistics**: All stat counters use `atomic64_t`, avoiding lock contention on the fault path for telemetry.

3. **Cooperative prefetch cancellation**: The prefetch engine checks `contra_rotation_compute_active()` in its work loop, yielding immediately when compute phase starts. This prevents prefetch from stealing memory bandwidth during computation.

4. **Energy-proportional demotion**: The JIT hint -> energy gate -> tier demotion chain ensures the system never exceeds its power budget, even when the JIT requests GPU-tier placement.

5. **Zeroization on cleanup**: Both `mr_registry.c` and `xkabi_capabilities.c` use `memzero_explicit()` to scrub cryptographic material on free. This is correct for security-critical code.
