# RFC-002 v5 POC -- Data Infrastructure Assessment Report

**Analyst:** Data Infrastructure Lead
**Date:** 2026-02-26
**Scope:** READ-ONLY analysis of `/tmp/rfc002_analysis/gen-os/`
**Files Analyzed:** 16 source files (6 kernel C, 3 Python tools, 1 ISA header, 3 test suites, 3 supplementary C modules)

---

## EXECUTIVE SUMMARY

The RFC-002 v5 POC is an ambitious, well-structured kernel-level data system spanning 4-tier unified memory (vDRAM), capability-based security (XKABI), mesh networking (genos-meshd), RDMA attestation, and Python-based tooling for energy/thermal telemetry and Merkle verifiable compute. The C struct design quality is high -- dense, well-documented, and correctly aligned for kernel hot paths. However, from a **provenance data infrastructure** perspective, the system has **six critical gaps** that must be addressed before the Provenance Graph can ingest, index, and replay these events with temporal correctness.

**Overall Rating: STRONG engineering, WEAK provenance instrumentation.**

### 1.2 Normalization Assessment: MODERATE

**Strengths:**
- Clean separation between page-level metadata (`vdram_page_meta`) and region-level metadata (`vdram_mr`). Pages reference MRs via pointer, not embedding.
- XKABI capability objects are properly normalized: `xkabi_cap` stores authoritative state, `xkabi_handle_table` is the per-process container, and the 128-bit handle is an encoded reference with HMAC integrity.
- Tier descriptors (`vdram_tier_desc`) are separate from operational state (`vdram_node`).

**Weaknesses:**
- `meshd_peer` embeds gossip telemetry directly (thermal_temp_mc, gpu_util_pct, vdram_t3_avail_bytes) rather than referencing a time-indexed telemetry ring. This means **every gossip update overwrites** the previous value with no history.
- `spdm_session.bound_rkeys[64][4]` is a fixed-size array inside the session struct. This should be a dynamically allocated list or separate table for proper normalization.
- `vdram_page_meta.peer_id[32]` duplicates the Ed25519 pubkey that is also stored in `vdram_mr.peer_cert[256]` (digest vs full cert) -- the relationship between these two identity representations is implicit, not explicit.

### 1.3 Provenance Fields Assessment: CRITICAL GAP (P0)

**This is the single most important finding in this audit.**

The Provenance Graph requires every event record to carry two independent time dimensions:
- `action_time`: When the event OCCURRED in the system
- `record_time`: When the event was RECORDED in the data store

**Current state across all 16 files: ZERO structs carry both `action_time` and `record_time`.**

| Struct | Has action_time? | Has record_time? | Provenance-compliant? |
|--------|:---:|:---:|:---:|
| `vdram_page_meta` | PARTIAL (`last_access_ns` = monotonic clock, not wall time) | NO | NO |
| `vdram_fault_ctx` | PARTIAL (`fault_start` = ktime, ephemeral, never persisted) | NO | NO |
| `vdram_mr` | NO | NO | NO |
| `spdm_session` | PARTIAL (`established_at`) | NO | NO |
| `xkabi_cap` | NO | NO | NO |
| `meshd_peer` | PARTIAL (`last_gossip_ns`) | NO | NO |
| `cr_device` | PARTIAL (`phase_start_ns`) | NO | NO |
| `qp_relay_entry` | PARTIAL (`last_used_ns`) | NO | NO |
| `FabricMetrics` | PARTIAL (`state_start_ts`) | NO | NO |
| `MerkleCommitment` | YES (`timestamp_ns` in context) | NO | NO |
| `CollectiveEvent` | YES (`timestamp` field) | NO | NO |

**Finding P0-PROV-01: No struct in the entire RFC-002 v5 POC carries `record_time`. This means the Provenance Graph cannot answer the fundamental query: "What did the system believe at time T using only data available at T-epsilon?"**

**Finding P0-PROV-02: No struct carries `action_time_end`. This means bitemporal point-in-time queries cannot be expressed. State transitions (e.g., page moving from T1 to T2) overwrite `current_tier` in-place -- the previous tier is lost forever.**

**Finding P0-PROV-03: Timestamps use mixed clock domains.** Kernel code uses `ktime_get_ns()` (monotonic), `ktime_get()` (also monotonic), Python tools use `time.monotonic()` and `time.time()` (wall clock) and `time.time_ns()` (wall clock nanoseconds). When these events arrive at the Provenance Graph, clock reconciliation will be required.

## 2. STATE MANAGEMENT ASSESSMENT

### 2.1 Kernel State Tracking: CONSISTENT WITHIN SUBSYSTEMS

Each subsystem maintains state using a consistent pattern:

| Subsystem | State Container | Locking | Update Pattern |
|-----------|----------------|---------|----------------|
| vDRAM pages | `vdram_page_meta.lock` (spinlock per page) | Fine-grained | In-place update of `current_tier`, `access_count`, `last_access_ns` |
| MR registry | `vdram_node.mr_lock` + `rkey_table_lock` | Two-phase (MR list + rkey table) | Alloc/free lifecycle, RCU reads for rkey lookup |
| SPDM sessions | `session_cache_lock` + per-session `sess->lock` | Hierarchical | Session cache hash table, `valid` flag for revocation |
| XKABI caps | `xkabi_handle_table.lock` + per-cap `cap->lock` | Hierarchical | IDR allocation, `revoked` flag, epoch-based bulk revocation |
| Mesh peers | `meshd.lock` (global spinlock) | Coarse-grained | Overwrite-in-place on gossip receive |
| Contra-rotation | `contra_rotation_sched.lock` | Single lock | Phase transition with stat accumulation |
| QP relay | `zc_gateway.table_lock` + RCU reads | RCU pattern | Hash table with `kfree_rcu` teardown |

### 2.2 Cross-Subsystem Consistency: WEAK

**Finding P1-STATE-01: No transaction boundary spans multiple subsystems.** When a page fault triggers attestation + tier migration + Merkle commitment + energy accounting (the 8-step pipeline in `pf_handler.c`), each step independently updates state. If step 6 (Merkle) fails after step 5 (migration) succeeds, the system is in an inconsistent state: the page has moved but the Merkle root does not reflect the move. The `pr_warn` on line 327-328 logs the failure but does not roll back the migration.

**Finding P1-STATE-02: Mesh peer state is ephemeral.** `meshd_peer` overwrites telemetry on every gossip cycle (500ms). There is no ring buffer, no history, and no persistence. If the gossip thread crashes, all peer state is lost. The `thermal_guardian.py` tool writes to `/tmp/genos_thermal_state.json` (a single overwritten file) -- same problem.

**Finding P1-STATE-03: Global mutable singletons.** `g_cr_sched`, `g_gateway`, and the `meshd` static struct are all global mutable state. This pattern is standard for kernel modules but creates testability problems and makes it impossible to run multiple isolated instances for Provenance Graph replay.

### 2.3 State Lifecycle Correctness

**Finding P1-STATE-04: `vdram_mr_free()` does hard delete.** When an MR is freed (line 115-134 of `mr_registry.c`), the rkey table entry is set to NULL, crypto material is zeroized, and the struct is `kfree`'d. From a provenance perspective, this is a **data destruction event** -- we have no record that the MR ever existed. The Provenance Graph requires soft-delete: close the `record_time_end`, preserve the record.

**Finding P2-STATE-05: SPDM session revocation sets `valid = false` but does not remove the hash table entry.** This is actually correct for provenance (the session record persists), but there is no cleanup path -- the session cache will grow unboundedly. The 1-hour `SESSION_MAX_AGE_NS` check in `spdm_session_find()` only filters reads, not memory.

### 3.2 XKABI Audit Log Analysis

The `xkabi_audit_log()` function (lines 312-321 of `xkabi_capabilities.c`) is the only structured audit path in the entire codebase:

```c
void xkabi_audit_log(u64 object_id, u16 rights, u32 event_type)
{
    struct audit_buffer *ab = audit_log_start(audit_context(), GFP_ATOMIC,
                                               AUDIT_ANOMALY_ABEND);
    if (!ab) return;
    audit_log_format(ab, "xkabi: obj=%llu rights=0x%x event=%u",
                     object_id, rights, event_type);
    audit_log_end(ab);
}
```

**Finding P0-AUDIT-01: XKABI audit log uses `GFP_ATOMIC` allocation flag.** Under memory pressure, `audit_log_start()` may return NULL, causing the audit event to be **silently dropped**. Security-critical audit events (DENY, REVOKE) must never be dropped.

**Finding P0-AUDIT-02: No timestamps in XKABI audit records.** The kernel audit subsystem adds its own timestamp, but the `xkabi_audit_log()` format string contains no `action_time`. When these records are ingested into the Provenance Graph, we have no way to distinguish between "event happened at T" and "event was logged at T".

**Finding P1-AUDIT-03: Gossip signature verification failures are silently skipped.** In `genos_meshd.c` line 195-198, if `identity_ed25519_verify()` fails, the loop simply `continue`s. This is a security-relevant event (potential impersonation attack) with **zero logging**.

**Finding P1-AUDIT-04: Attestation denials are counter-only.** `atomic64_inc(&node->attestation_denials)` in `pf_handler.c` records THAT a denial happened, but not WHO was denied, WHEN, or WHY. For incident response, this counter is nearly useless.

### 3.3 TPM PCR Extension: CORRECT

The `attest_extend_tpm_pcr()` function (lines 168-179 of `rdma_attestation.c`) correctly extends PCR 16 with the 36-byte event `[peer_cert_digest(32) || rkey(4)]`. This creates a hardware-tamper-proof chain of attestation events that can be independently verified by a remote attestor. This is the **strongest provenance mechanism** in the codebase and should be used as the model for all other audit paths.

## 4. TELEMETRY PIPELINE ASSESSMENT

### 4.1 Data Flow Architecture

```
KERNEL LAYER                           PYTHON TOOLS LAYER
=============                          ==================

ktime_get_ns()                         time.monotonic() / time.time()
     |                                      |
     v                                      v
atomic64_t counters                    Python dataclasses
(fault_count, migration_count,         (FabricMetrics, DeviceThermal,
 energy_nj, attestation_denials)        MerkleCommitment)
     |                                      |
     v                                      v
printk(pr_info/pr_warn)                json.dumps() -> /tmp/*.json
sysfs/debugfs (vdram_print_stats)      logging (stdout/file)
kernel audit subsystem
     |                                      |
     v                                      v
  [NO BRIDGE]  <--- GAP --->          [NO BRIDGE]
     |                                      |
     ?                                      ?
     v                                      v
  Provenance Graph (SQLite/PostgreSQL)
```

**Finding P0-TELEM-01: There is NO data bridge between the kernel event layer and the Python tools layer.** The kernel subsystems use `pr_info`/`pr_warn` (dmesg) and atomic counters. The Python tools read hardware sensors directly (`nvidia-smi`, `/sys/class/thermal`) and write JSON to `/tmp`. Neither layer feeds into a structured event store. The Provenance Graph has no ingestion path.

**Finding P1-TELEM-02: `thermal_guardian.py` writes to `/tmp/genos_thermal_state.json` using `Path.write_text()`.** This is an atomic overwrite with no history. Each write destroys the previous state. For a 7-device fleet polled at 500ms, that is 14 state snapshots/second, all lost except the latest.

**Finding P1-TELEM-03: `fabric_energy_controller.py` saves reports to `/tmp/ep_rdma_report.json` every 5 seconds.** Same overwrite pattern. Energy accounting data (`total_energy_joules`, `baseline_energy_joules`) is cumulative, so the latest snapshot contains the running total -- but the per-interval breakdown is lost.

**Finding P2-TELEM-04: Gossip telemetry (thermal, GPU util, vDRAM availability) flows from kernel `gossip_send()` to peer `gossip_recv_all()` and is consumed in-place.** There is no tap or mirror for the Provenance Graph to observe this data stream. A gossip telemetry mirror endpoint is needed.

### 4.2 Clock Domain Summary

| Source | Clock | Epoch | Granularity | Notes |
|--------|-------|-------|-------------|-------|
| `ktime_get_ns()` | CLOCK_MONOTONIC | Boot | Nanoseconds | Not synchronized across nodes |
| `ktime_get()` | CLOCK_MONOTONIC | Boot | ktime_t (ns) | Same as above |
| `time.monotonic()` | CLOCK_MONOTONIC | Process start | Seconds (float) | Python-specific epoch |
| `time.time()` | CLOCK_REALTIME | Unix epoch | Seconds (float) | Susceptible to NTP jumps |
| `time.time_ns()` | CLOCK_REALTIME | Unix epoch | Nanoseconds | Used in MerkleCommitment context |

**Finding P1-TELEM-05: Mixed clock domains will cause event ordering errors in the Provenance Graph.** A kernel event at `ktime_get_ns() = 123456789` and a Python event at `time.time_ns() = 1740556800000000000` cannot be directly compared. The ingestion pipeline must normalize all timestamps to a unified wall-clock nanosecond epoch.

## 5. SCHEMA EVOLUTION ASSESSMENT

### 5.1 Current Extensibility

**Positive patterns:**
- `vdram_jit_hint` has a `reserved` byte (line 54 of `vdram.h`), indicating forward planning for field additions.
- `xisc_energy_policy_t` is a separate struct, not embedded in the instruction encoding -- clean extension point.
- `gossip_state_msg.version = 1` in `genos_meshd.c` line 157 -- protocol versioning exists.
- `xisc_program_t.version` uses `(MAJOR<<16)|(MINOR<<8)|PATCH` encoding -- standard semver.

**Negative patterns:**
- `vdram_page_meta.peer_id[32]` is a fixed 32-byte Ed25519 pubkey. If the system migrates to post-quantum keys (ML-KEM-768 mentioned in SPDM), this field is too small.
- `spdm_session.bound_rkeys[64][4]` is a fixed 64-entry array. Scaling beyond 64 rkeys per session requires a struct redesign.
- `xkabi_cap.lineage[XKABI_LINEAGE_MAX_DEPTH][16]` is a fixed 16-deep array. Deep capability derivation chains will truncate lineage.
- `MESHD_MAX_PEERS = 64` is a hard compile-time limit. Mesh growth requires recompilation.
- The XKABI handle format is a wire-format struct with fixed bit positions (`[0:63] object_id, [64:79] rights, [80:95] generation, [96:127] HMAC`). Adding fields requires a version bump and dual-format handling.

### 5.2 Migration Hazards

**Finding P1-SCHEMA-01: No schema version field in any kernel struct.** If `vdram_page_meta` gains new fields (e.g., `action_time`, `record_time`), there is no way for existing MRs with old-format page arrays to coexist with new-format arrays. This will require a "drain-and-recreate" migration rather than an in-place upgrade.

**Finding P1-SCHEMA-02: The `xkabi_handle_t` 128-bit wire format uses a 2-byte truncated HMAC (bytes 14-15).** This provides only 16 bits of integrity protection. At 2^15 handle validations, there is a 50% chance of a collision. This is acceptable for a POC but must be extended to at least 8 bytes (64 bits) for production.

## 6. TIME-SERIES DATA MODEL ASSESSMENT

### 6.1 Gossip Telemetry

**Current model:** Each `meshd_peer` has three scalar fields: `thermal_temp_mc`, `gpu_util_pct`, `vdram_t3_avail_bytes`. Updated in-place every 500ms.

**Required model for Provenance Graph:**
```
Table: gossip_telemetry
  node_id         BLOB(32)    -- Ed25519 pubkey
  action_time     INTEGER     -- Wall-clock nanoseconds when reading was taken
  record_time     INTEGER     -- Wall-clock nanoseconds when stored in DB
  thermal_mc      INTEGER     -- milli-Celsius
  gpu_util_pct    INTEGER     -- 0-100
  vdram_avail_b   INTEGER     -- T3 available bytes
  cr_phase        INTEGER     -- contra-rotation phase enum

Indexes:
  (node_id, action_time)     -- Time-series queries per node
  (record_time)              -- Provenance point-in-time queries
  (action_time, thermal_mc)  -- Thermal anomaly range scans
```

**Gap:** The current model stores ZERO history. Even 24 hours of gossip at 2Hz per 64 peers = 11 million rows/day. SQLite can handle this with WAL mode and appropriate page sizes, but the current design has no persistence layer at all.

### 6.2 Thermal Readings

**Current model:** `DeviceThermal` dataclass overwritten to `/tmp/genos_thermal_state.json` every 500ms.

**Required model:**
```
Table: thermal_reading
  device_id       TEXT        -- hostname
  action_time     INTEGER     -- Wall-clock ns
  record_time     INTEGER     -- Wall-clock ns
  gpu_temp_c      REAL
  cpu_temp_c      REAL
  phase           TEXT        -- COMPUTE/COOLDOWN
  alert           TEXT        -- OK/WARN/CRITICAL

Index: (device_id, action_time)
```

### 6.3 Energy Accounting

**Current model:** `FabricMetrics` is a Python dataclass with cumulative `total_energy_joules` and `baseline_energy_joules`. `vdram_node.energy_nj` is a kernel `atomic64_t` counter.

**Required model:** Per-interval energy snapshots (aligned to the 10ms control loop):
```
Table: energy_sample
  node_id         BLOB(32)
  action_time     INTEGER     -- Start of 10ms interval
  record_time     INTEGER
  actual_power_w  REAL
  baseline_w      REAL
  lanes_active    INTEGER
  fabric_state    INTEGER     -- DENSE/IDLE/COOLDOWN/TRANSITION

Index: (node_id, action_time)
```

### 6.4 Summary

**Finding P0-TSERIES-01: The entire POC has no time-series persistence.** Kernel counters are monotonic accumulators with no interval breakdown. Python tools write single-point JSON snapshots. Neither approach supports range queries, anomaly detection, or historical analysis.

## 7. 5-D EVENT FINGERPRINT STORAGE STRATEGY

Given the data sources in this POC, here is how the 5-D Event Fingerprint would be constructed and stored for vDRAM system events.

### 7.1 Fingerprint Dimension Mapping

| Dimension | Source | Encoding | Size |
|-----------|--------|----------|------|
| **D1: Semantic** | Event classification via Llama 3.2 3B on event text (e.g., "T3 page fault on MR 0x1234 with attestation denial") | 256-dim float16 embedding | 512 bytes |
| **D2: Temporal** | `action_time` encoded as sinusoidal position: `[sin(t/10^k), cos(t/10^k)]` for k=0..7 | 16 float32 values | 64 bytes |
| **D3: Causal** | Graph depth from root cause event. For page faults: depth=0 is the fault, depth=1 is the migration, depth=2 is the Merkle commit. For XKABI: depth from root capability in lineage chain. | 1 uint8 | 1 byte |
| **D4: State** | System state classification from contra-rotation phase + thermal state + attestation status. States: NORMAL=0, DEGRADED=1 (thermal warn), RECOVERY=2 (cooldown), CRISIS=3 (attestation denial or Merkle tamper). | 1 uint8 | 1 byte |
| **D5: Component** | Noise-filtered anomaly signal strength per subsystem. 8 subsystems: vDRAM, XKABI, SPDM, mesh, thermal, energy, Merkle, QP-relay. Each is a float16 anomaly residual. | 8 float16 values | 16 bytes |

**Total fingerprint size: 594 bytes** (512 + 64 + 1 + 1 + 16)

### 7.2 Storage Schema

```sql
CREATE TABLE event_fingerprint (
  event_id          TEXT PRIMARY KEY,     -- UUID v7 (time-ordered)
  action_time       INTEGER NOT NULL,     -- Wall-clock nanoseconds
  record_time       INTEGER NOT NULL,     -- Wall-clock nanoseconds (set at INSERT)
  action_time_end   INTEGER,              -- NULL = current state
  record_time_end   INTEGER,              -- NULL = current record

  -- Source identification
  node_id           BLOB(32) NOT NULL,    -- Ed25519 pubkey
  subsystem         INTEGER NOT NULL,     -- enum: vDRAM=0, XKABI=1, ...
  event_type        INTEGER NOT NULL,     -- subsystem-specific event code

  -- Raw event data
  raw_text          TEXT,                 -- Original event text for re-classification

  -- 5-D Fingerprint (stored as contiguous blob for cache efficiency)
  fp_semantic       BLOB(512) NOT NULL,   -- D1: Llama 3.2 3B embedding
  fp_temporal       BLOB(64) NOT NULL,    -- D2: Sinusoidal position encoding
  fp_causal_depth   INTEGER NOT NULL,     -- D3: Graph depth
  fp_state          INTEGER NOT NULL,     -- D4: System state
  fp_component      BLOB(16) NOT NULL,    -- D5: Per-subsystem anomaly

  -- Noise filter metadata
  raw_signal        REAL,                 -- Pre-filter signal value
  baseline_signal   REAL,                 -- Rolling 30-min baseline
  load_factor       REAL DEFAULT 1.0,     -- System load normalization
  anomaly_residual  REAL                  -- raw_signal - baseline * load_factor
);

-- Provenance query indexes
CREATE INDEX idx_fp_record_time ON event_fingerprint(record_time);
CREATE INDEX idx_fp_action_time ON event_fingerprint(action_time);
CREATE INDEX idx_fp_node_action ON event_fingerprint(node_id, action_time);
CREATE INDEX idx_fp_subsystem_action ON event_fingerprint(subsystem, action_time);

-- Point-in-time query: "What did the graph know at T?"
-- SELECT * FROM event_fingerprint

--   AND record_time <= $query_as_of_time
--   AND (action_time_end IS NULL OR action_time_end > $query_action_time)
--   AND (record_time_end IS NULL OR record_time_end > $query_as_of_time);
```

### 7.3 Concatenation Order

The 5-D fingerprint is concatenated in this fixed order for consistency across all subsystems:

```
[D1: Semantic (512B)] [D2: Temporal (64B)] [D3: Causal (1B)] [D4: State (1B)] [D5: Component (16B)]
```

Total: 594 bytes. Padded to 600 bytes (8-byte aligned) for memory layout efficiency.

### 8.1 Cryptographic Mechanisms Inventory

| Mechanism | Location | Algorithm | Applied To | Consistent? |
|-----------|----------|-----------|------------|:-----------:|

### 8.2 Consistency Analysis

**Finding P1-INTEG-01: XKABI handle HMAC is only 2 bytes (16 bits).** At line 95 of `xkabi_capabilities.c`, `memcpy(&handle_out[14], mac, 2)` truncates the 32-byte HMAC to 2 bytes. This provides `2^16 = 65,536` possible MAC values. An attacker with ~256 forged handles has a 50% chance of a valid MAC. The constant-time comparison at line 114 is correct, but the truncation negates much of the HMAC's security value.

**Finding P1-INTEG-02: Merkle root updates are not atomic with migration.** In `pf_handler.c`, step 5 (migration at line 312-322) and step 6 (Merkle commit at line 325-328) are separate operations. If the system crashes between them, the Merkle root does not reflect the current page layout. The `merkle_lock` spinlock in `vdram_node` protects the root, but there is no journaling or WAL to ensure crash consistency.

**Finding P2-INTEG-03: `vdram_page_meta.merkle_hash[32]` is never explicitly set.** The struct field exists (line 66 of `vdram.h`), and `vdram_merkle_commit()` is declared (line 200-202), but the implementation is not in the analyzed files. It is unclear when `merkle_hash` is computed -- at page write? At migration? At explicit commit point? The JIT hint `merkle_commit_point` (line 51) suggests explicit commitment, but the trigger conditions are ambiguous.

**Finding P1-INTEG-04: AES-256-GCM IV is sequence-based, not random.** `vdram_mr.aes_seq` is an `atomic64_t` monotonic counter used to construct the per-packet IV: `aes_iv_base[12] XOR seq`. This is correct for GCM (nonce must never repeat with the same key), BUT if the kernel crashes and `aes_seq` is lost (in-memory only), the next boot could restart at seq=0, causing IV reuse. This must be persisted or protected by a crash counter.

**Finding P1-INTEG-05: Merkle collective `inject_tamper()` test utility is in production code.** At line 244-250 of `merkle_collective.py`, there is a method designed for testing that modifies contributions. This should be guarded by a debug flag or moved to a test module to prevent accidental use.
