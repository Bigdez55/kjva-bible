### 1.3 Temporal Decay Vector (Time2Vec Encoding)

The Time2Vec model encodes the "information half-life" of each event type:

| Event Type | Half-Life | Decay Lambda | Rationale |
|---|---|---|---|
| Thermal ceiling hit | 30s | 0.023 | Direct: contra_rotation THERMAL_POLL_MS=500, but state persists across COOLDOWN |
| Page fault (any tier) | 5s | 0.139 | Fast: access_count increments, but individual faults are high-frequency noise |
| SPDM session establish | 3600s | 0.000193 | Slow: SESSION_MAX_AGE_NS = 1 hour. Cached session valid for entire hour. |
| Migration event | 60s | 0.0116 | Medium: tier state persists until next access pattern shift |
| Prefetch trigger | 2s | 0.347 | Fast: tied to PHASE_MIN_COMPUTE_MS=2000ms cycle time |
| JIT compilation | 86400s | 8.02e-6 | Very slow: AOT cache persists across runs. Cache hit rate stabilizes after 2nd run. |
| EP-RDMA lane transition | 10s | 0.069 | Medium: IDLE_THRESHOLD_MS=50ms but state machine has hysteresis |
| Merkle commitment | 120s | 0.00578 | Medium: commitment valid until next collective. Training step ~120s. |
| Entropy classification | 0.5s | 1.386 | Very fast: per-packet classification. No persistence beyond routing decision. |

**Time2Vec encoding formula:**
```
t2v(t) = [w_0 * t + b_0, sin(w_1 * t + b_1), ..., sin(w_k * t + b_k)]
```
Where t = (T_now - T_event) * decay_lambda, and k=8 harmonics per event type.

### 1.5 State Vector (1D CNN Autoencoder)

Time series compression of system metrics into a latent representation.

**Input channels (8 channels, 128 timestep window at 500ms granularity = 64s window):**
1. CPU temperature (milli-Celsius from thermal zones)
2. GPU temperature (nvidia-smi or hwmon)
3. Fault rate per tier (4 tiers summed weighted: T0*1 + T1*2 + T2*4 + T3*8)
4. Migration rate (events/sec)
5. Energy consumption rate (nJ/sec)
6. EP-RDMA lane count (normalized 0-1 as lanes/LANES_MAX)
7. JIT cache hit rate (rolling 10s window)
8. Prefetch hit rate (rolling 10s window)

**Architecture:**
```
Encoder: Conv1d(8,16,k=5) -> ReLU -> Conv1d(16,32,k=3) -> ReLU -> Conv1d(32,8,k=3) -> AvgPool -> latent(dim=32)
Decoder: ConvTranspose1d mirror
```

Latent dimension = 32. This compressed representation captures the "system state structure"
without requiring raw metric values.

## 4. VARIANCE DECOMPOSITION: PAGE FAULT LATENCY

### 4.1 Source Identification

Page fault latency (pf_handler.c:256-362) is decomposed into 4 variance components:

```
Total_Latency = Tier_Component + Attestation_Component + Migration_Component + Merkle_Component + Error
```

### 4.2 Component Analysis

**Component 1: Tier Resolution (Steps 1 + 4)**
```
Source: pf_handler.c:269-307 (MR lookup + tier selection)
Fixed cost: ~200-500ns (spinlock + list traversal + heuristic evaluation)
Variance contribution:
  - T0/T1 path: 200-500ns (dominated by spinlock contention)
  - T2 path: 200-500ns (same logic, different outcome)
  - T3 path: 200-500ns (same logic)
  Variance within component: VERY LOW (deterministic per-path)
  Estimated eta-squared: 0.05 (5% of total variance)
```

**Component 2: Attestation Gate (Step 2)**
```
Source: pf_handler.c:291 -> rdma_attestation.c:185-262
Variance contribution:
  - T0/T1: 0ns (skip: pf_handler.c:81-85)
  - T2: ~100-500ns (xkabi_cap_validate: pf_handler.c:90)
  - T3 cached: <200ns (hash lookup: rdma_attestation.c:140-162)
  - T3 cold: ~2ms = 2,000,000ns (full SPDM: 3 network round trips)
  Range: 0ns to 2,000,000ns
  Variance within component: EXTREMELY HIGH (bimodal: cached vs cold)
  Estimated eta-squared: 0.35 (35% of total variance)
  NOTE: This is the HIGHEST variance component due to the 10,000x
        ratio between cached (200ns) and cold (2ms) paths.
```

**Component 3: Migration (Step 5)**
```
Source: pf_handler.c:312-322 -> tier_migration.c:302-346
Variance contribution:
  - No migration needed: 0ns (same tier: tier_migration.c:311-314)
  - T0<->T1 (HMM): ~5-50us (GPU cache flush + DMA)
  - T1<->T2 (NVMe): ~15-100us (async DMA + file I/O)
  - T1/T2->T3 (RDMA): ~50-200us (RDMA WRITE + poll completion)
  - T3->T1 (RDMA READ): ~50-200us
  Range: 0ns to 200,000ns
  Variance within component: HIGH (depends on source/destination tier)
  Estimated eta-squared: 0.40 (40% of total variance)
  NOTE: Migration is conditionally triggered. When it fires, it dominates.
```

**Component 4: Merkle Commitment (Step 6)**
```
Source: pf_handler.c:325-328 -> merkle operations
Variance contribution:
  - Not needed: 0ns (most common: neither JIT hint nor T3)
  - SHA-256 commit: ~500-2000ns (single page hash + tree update)
  Range: 0ns to 2000ns
  Variance within component: LOW
  Estimated eta-squared: 0.02 (2% of total variance)
```

**Error Component:**
```
  Sources: spinlock contention, cache misses, interrupt preemption,
           OS scheduler jitter, memory allocator latency
  Estimated: 0.18 (18% of total variance)
```

## 5. MODEL DRIFT DETECTION FOR MoE PREFETCH ENGINE

### 5.1 Drift Surfaces

The MoE prefetch engine (`prefetch_engine.c`) operates on a learned distribution:
MoE routing weights -> expert selection -> prefetch priority.

Three drift types must be monitored:

**Concept Drift (causal structure change):**
```
Definition: The relationship between routing_weight and actual expert usage changes.
Detection signal: prefetch_hits / (prefetch_hits + fault_count[T2] + fault_count[T3])
  Currently: vdram_node.prefetch_hits (prefetch_engine.c:86)
  vs: vdram_node.fault_count[TIER_T2_NVME_CXL] + fault_count[TIER_T3_RDMA_MESH]

Drift indicator: If prefetch_hit_rate drops below 60% over a 30-minute window
  while routing weights remain stable, the MoE gating network has shifted
  its expert selection distribution but the prefetch engine has not adapted.

Detection algorithm:
  1. Maintain rolling 30-min window of prefetch_hit_rate
  2. Compute Wasserstein distance between current expert access distribution
     and prefetch priority distribution
  3. If W_distance > 0.15 AND hit_rate < 0.60: CONCEPT DRIFT flagged

R-squared for detection: 0.55 (marginal SIGNAL -- expert access patterns
  are inherently noisy, but sustained drift is detectable)
```

**Data Drift (input distribution change):**
```
Definition: The distribution of routing_weight values from the MoE gating network changes.
Detection signal: Monitor the sorted weight distribution (prefetch_engine.c:61)
  Currently: sorted by routing_weight descending, threshold at PREFETCH_CONFIDENCE=128

Drift indicator: If the number of experts above PREFETCH_CONFIDENCE threshold
  changes by >50% over a 1-hour window, the MoE model's sparsity has shifted.
  Example: Model starts selecting 8 experts (dense) -> 2 experts (sparse).
  This changes the prefetch workload from O(8 migrations) to O(2 migrations)
  per cooldown phase.

Detection algorithm:
  1. Track n_experts_above_threshold per prefetch cycle
  2. Compute CUSUM (cumulative sum) statistic on the time series
  3. If CUSUM exceeds threshold h (calibrated to 3-sigma): DATA DRIFT flagged

Encoding into 5D Fingerprint:
  The Temporal Decay Vector (half-life=2s for prefetch events) provides
  natural down-weighting of stale expert distributions.
```

**Performance Drift (R-squared decline):**
```
Definition: The prefetch engine's predictive accuracy degrades over time.
Metric: R-squared of the model:
  Y = actual_expert_accessed (binary per expert per compute phase)
  X = routing_weight (prefetch priority assigned)
  R-squared = correlation between predicted and actual expert utilization

Detection:
  1. Compute rolling 30-day R-squared from prefetch_hits vs fault_count
  2. Threshold: R-squared decline > 0.05 over 30 days -> RETRAIN
  3. Report to Guardian Sentinel agent for retraining cycle initiation

Current baseline R-squared: UNKNOWN (no telemetry infrastructure exists)
  Recommendation: Instrument prefetch_work_fn to log expert_id and
  actual_tier_at_fault_time per compute phase.
```

## 9. CROSS-CUTTING FINDINGS AND DEFECTS

### 9.1 Identified Causal Graph Defects

**D-RFC002-01 P1 HIGH: Prefetch confidence threshold is hardcoded**
```
File: /tmp/rfc002_analysis/gen-os/kernel/vdram/prefetch_engine.c:20
Code: #define PREFETCH_CONFIDENCE 128
Issue: The threshold (128/255) is static. It should adapt based on
  historical prefetch hit rate. When the MoE model changes sparsity
  (e.g., 8 experts -> 2 experts), the fixed threshold either:
  - Prefetches too many experts (wasting cooldown bandwidth)
  - Or misses experts if weights are rescaled
Causal impact: Directly affects edges 5 (prefetch->migration) and
  the Model B drift detection accuracy.
Recommendation: Replace with adaptive threshold:
  confidence_threshold = max(64, mean_weight - 1.5 * stddev_weight)
```

**D-RFC002-02 P1 HIGH: JIT cache eviction is random**
```
File: /tmp/rfc002_analysis/gen-os/genvirt/jit/xisc_llvm_jit.c:305
Code: (u32)(ktime_get_ns() % JIT_CODE_CACHE_ENTRIES)  /* evict */
Issue: When the 64K cache is full, eviction uses modular arithmetic
  on ktime_get_ns(). This is effectively random eviction, not LRU or LFU.
  A hot block could be evicted, causing a 500us cold recompile on the
  critical path (vs 50ns cache hit = 10,000x regression).
Causal impact: Edge 8 (XISC bytecode -> JIT) becomes unpredictable.
  The JIT cache hit rate metric (xisc_jit_cache_hit_rate) will show
  artificial variance that is NOT workload-caused.
Recommendation: Replace with LFU eviction using hit_count field
  (already tracked in cache[i].hit_count).
```

**D-RFC002-03 P2 MEDIUM: Thermal hysteresis band too wide**
```
File: /tmp/rfc002_analysis/gen-os/kernel/vdram/contra_rotation.c:25-26
Code: THERMAL_CEILING_LOW_C=60, THERMAL_CEILING_HIGH_C=80
Issue: 20C hysteresis band means the system stays in COMPUTE phase
  from 60C to 80C. On consumer hardware (HP EliteBook, 15W TDP),
  thermal response is fast: 60C->80C in ~5-10s under load.
  But 80C->55C (cooldown floor = LOW-5 = 55C, line 162) takes ~15-30s.
  This creates asymmetric duty cycles: 5-10s compute, 15-30s cooldown.
  Compute utilization: ~25-40%.
Causal impact: Edges 1-7 (entire thermal spine) affected.
  The 60C counterfactual (Section 3.1) demonstrated the collapse risk.
  Optimal band: [65, 75] for consumer, [70, 85] for enterprise.
```

**D-RFC002-04 P2 MEDIUM: Energy cost array assumes uniform page size**
```
File: /tmp/rfc002_analysis/gen-os/kernel/vdram/pf_handler.c:218-223
Code: tier_energy_cost_nj[TIER_MAX] = {50, 100, 2000, 5000}
Issue: Energy costs are per-fault with implicit 4K page assumption.
  Huge pages (2MB, 1GB) would have drastically different energy profiles.
  The vdram.h:73 defines VDRAM_MR_MAX_PAGES = 1M pages (4GB per MR),
  but there is no huge page support in the energy model.
Causal impact: Edge 23 (energy accounting) produces incorrect values
  for non-4K pages, corrupting the energy_nj counter and breaking
  the energy gate feedback loop (edge 16).
```

**D-RFC002-05 P3 LOW: Entropy classifier LUT is sparse**
```
File: /tmp/rfc002_analysis/gen-os/gateway/zc_gw/entropy_classifier.c:18-24
Code: log2_lut[256] with only ~13 populated entries
Issue: Most LUT entries are zero (default-initialized). Any byte frequency
  not matching one of the 13 populated indices contributes 0 to entropy.
```
