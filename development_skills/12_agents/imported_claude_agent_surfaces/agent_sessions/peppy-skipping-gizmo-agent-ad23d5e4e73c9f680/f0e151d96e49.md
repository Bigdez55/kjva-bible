| Tier | Claimed nJ/fault | Physical Estimate | Ratio | Assessment |

| AES-GCM overhead | 800 nJ | ~500-1500 nJ (HW offload on BlueField-4) | 0.53-1.6x | **REASONABLE** -- AES-256-GCM in hardware at ~5 Gbps throughput, ~2W = ~800 nJ per 4K block. This is well-calibrated for NIC offload. |

### 6.3 Energy Model Verdict

The energy model is **directionally correct** (T0 < T1 << T2 << T3) and the tier ordering is physically grounded. However, absolute values are **systematically optimistic by 2-3x for T0, T2, and T3**. The AES-GCM add-on (800 nJ) is the most accurately calibrated value.

**Systematic bias:** The model appears calibrated against datasheet theoretical minimums rather than measured system-level values. This is a **survivorship bias** in the energy model -- only the data-movement energy is counted, ignoring controller overhead, cache hierarchy costs, and idle power amortization.

**Impact on energy gating:** Since `vdram_energy_gate()` compares against `power_cap_mw` (45W consumer, 400W enterprise), and the energy accounting is per-fault in nJ, the absolute calibration error does not affect the gating decision as long as the relative ordering is correct. The gate will trigger on the same workload patterns regardless of whether T2 costs 2000 nJ or 5000 nJ.

**Internal consistency score:** 65/100 -- Relative ordering correct, absolute values need 2-3x correction factor.

```
            |
            +-- vdram_on_compute_phase_start(node)
            |       |
            |       +-- flush_workqueue(prefetch_wq)  [drain prefetch before compute]
            |
            +-- ep_rdma_scale_lanes_up(max_rdma_lanes)
                   |
                   +-- Power: 100W -> 400W
```

### 7.2 Thermal Chain Timing Analysis

| Stage | Latency | Cumulative | Notes |
|-------|---------|------------|-------|
| Sensor read | ~1-5ms | 1-5ms | Kernel thermal_zone_get_temp() |
| Poll detection | 0-500ms | 1-505ms | 500ms poll interval (worst case: just missed) |
| Phase transition logic | ~1us | ~505ms | In-kernel spinlock + state update |
| vDRAM cooldown notification | ~1us | ~505ms | Function call chain |
| Prefetch work queue dispatch | ~10us | ~505ms | queue_work() scheduling |
| Prefetch execution start | ~1-50ms | ~555ms | Workqueue scheduling latency |
| Expert sort (256 entries) | ~10us | ~555ms | sort() with comparator |
| Per-expert migration T2->T1 | ~15us each | ~555ms+ | NVMe read + tier update |
| EP-RDMA scale-down | ~1-20ms | ~575ms | Credit drain timeout |
| **Total: sensor -> action** | **~500-600ms** | | Dominated by poll interval |

### 7.3 Thermal Chain Assessment

**Critical timing gap:** The 500ms polling interval (`THERMAL_POLL_MS = 500`) introduces up to 500ms of detection latency. For consumer GPUs that can heat at ~5C/second under full load, this means up to **2.5C of overshoot** above the 80C ceiling before detection.

**Thermal Guardian (Python layer)** at `thermal_guardian.py` polls at the same 500ms interval with thresholds: WARN=75C, CRITICAL=80C. The 5C headroom (75C trigger for migration, 80C for forced cooldown) correctly compensates for the 500ms poll latency.

**Signal coherence:** The thermal chain is well-designed with proper feedback coupling. The contra-rotation scheduler's `PHASE_MIN_COMPUTE_MS = 2000` ensures at least 2 seconds of compute before allowing thermal-triggered cooldown, preventing oscillation.

**Thermal chain signal score:** 82/100 -- Well-bounded latency, proper hysteresis, 5C headroom adequate for 500ms polling.

## 8. LATENCY BUDGET ANALYSIS -- Full Page Fault Path

### 8.1 T1 Hot Path Budget (Target: <2us)

```
Step 0: ktime_get()                          ~20ns
Step 1: pf_find_mr (spinlock + list scan)    ~50-500ns  (depends on MR count)
Step 1: pf_get_page_meta (offset calc)       ~5ns
Step 1: spin_lock(page->lock)                ~20-100ns
Step 1: access_count++, last_access_ns       ~10ns
Step 2: pf_attestation_gate (T1 bypass)      ~5ns      (immediate return for T0/T1)
Step 3: pf_validate_mr_boundary              ~50-200ns  (iommu_domain_check_va)
Step 4: pf_select_tier                       ~20-50ns   (JIT hint check + heuristic)
Step 4: spin_unlock(page->lock)              ~10ns
Step 5: (no migration if already T1)         ~0ns
Step 6: pf_merkle_commit_if_needed           ~5ns       (should_commit = false for T1)
Step 7: pf_energy_account                    ~15ns      (atomic64_add)
Step 8: vdram_tier_resolve_page              ~20-50ns   (tier backend page resolution)
Step 8: pf_wire_pte                          ~20ns      (vmf->page assignment + get_page)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TOTAL (best case, few MRs):                  ~250ns     WELL WITHIN 2us
TOTAL (worst case, 1000+ MRs):               ~1200ns    WITHIN 2us
TOTAL (pathological, 10K MRs + contention):  ~3000ns    EXCEEDS 2us
```

**Finding:** T1 hot path meets <2us for up to ~5000 MRs. Beyond that, the O(n) MR list scan becomes the bottleneck. Recommendation: replace `list_for_each` in `pf_find_mr()` with an interval tree or rbtree for O(log n) lookup.

### 8.2 T2 NVMe Path Budget (Target: <15us)

```
All T1 steps above:                          ~250ns
Step 2: pf_attestation_gate (T2 XKABI check) ~200-500ns (xkabi_cap_validate HMAC check)
Step 5: vdram_migrate_page T2->T1
  - filp_open(nvme_path)                     ~2-5us    (VFS path lookup)
  - alloc_page(GFP_KERNEL)                   ~100-500ns
  - init_sync_kiocb + vfs_iocb_iter_read     ~5-12us   (NVMe 4K read: ~8us typical)
  - filp_close                               ~500ns
  - spin_lock + tier update                  ~30ns
  - atomic64_add/sub                         ~20ns
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TOTAL (best case, fast NVMe Gen4):           ~8us       WITHIN 15us
TOTAL (typical, Gen3 NVMe + VFS overhead):   ~13us      WITHIN 15us (marginal)
TOTAL (worst case, NVMe queue depth high):   ~25us      EXCEEDS 15us
```

**Finding:** The T2 path is dominated by `filp_open()` + VFS I/O. The `filp_open()` per migration is the most expensive unnecessary operation -- NVMe backing file should be opened once at init and held open. This alone would save ~3us.

### 8.3 T3 Remote Path Budget (Target: <200us)

```
All T1 steps above:                          ~250ns
Step 2: pf_attestation_gate (T3 full)
  - Cached SPDM session:                     ~100-300ns
  - New SPDM session:                        ~2-30ms   (3 round-trips over network)
Step 5: vdram_migrate_page T3->T1
  - alloc_page(GFP_KERNEL):                  ~100-500ns
  - rdma_post_write_wr:                      ~5-50us   (RDMA verb posting)
  - rdma_poll_completion(5s timeout):         ~50-200us (one-sided RDMA READ latency)
    - Enterprise IB: ~2-5us
    - Consumer SoftiWARP+WG: ~100-500us
  - spin_lock + tier update:                 ~30ns
Step 6: Merkle commit (T3 always commits):   ~5-20us   (SHA-256 of 4K page + tree update)
Step 7: Energy account (+800nJ AES):         ~15ns
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TOTAL (enterprise, cached session):          ~55us      WITHIN 200us (easily)
TOTAL (consumer, cached session, LAN):       ~200us     AT TARGET (borderline)
TOTAL (consumer, WiFi, new session):         ~35ms      FAR EXCEEDS 200us
```

**Finding:** T3 target is achievable on enterprise hardware and consumer wired LAN with cached sessions. **On consumer WiFi with new SPDM sessions, the target is unachievable by ~175x.** The code correctly implements a fast-path cache that makes the common case fast, but the cold-start case violates targets significantly.

## CROSS-CUTTING FINDINGS

### FINDING-A: MR List is O(n) -- Single Largest Hot Path Risk

`pf_handler.c:36-52` (`pf_find_mr`) performs a linear scan of all MRs under spinlock. With `VDRAM_MAX_MRS = 65536`, worst case is O(65536) under spinlock in the page fault hot path. This is the **single largest structural risk** to the <2us T0/T1 target.

- Signal Intensity: 0.95
- Causal Direction: -0.90 (negative -- degrades performance)
- Decay Factor: Permanent (architectural)
- Recommended action: Interval tree (`include/linux/interval_tree.h`) already in Linux kernel, designed for exactly this use case.

### FINDING-B: No Temporal Gating in Energy Model

The `energy_nj` atomic counter accumulates forever with no per-epoch reset. The `vdram_energy_gate()` checks against `power_cap_mw` but the comparison semantics are unclear -- nJ (energy) vs mW (power) requires a time window to be meaningful. The code does not define this time window.

- Signal Intensity: 0.70
- Risk: Energy gating may either never trigger (if nJ grows faster than the power cap is checked) or always trigger (if the accumulator never resets). This is a **unit mismatch** bug.

### FINDING-C: Session Cache Unbounded Growth

`rdma_attestation.c:52` defines a 1024-bucket hash table with no eviction policy. Sessions expire after 1 hour (`SESSION_MAX_AGE_NS`), but expired entries are never freed -- they are simply not returned by `spdm_session_find()`. Over time, the hash table accumulates dead entries, causing:
1. Memory leak (each session is ~400 bytes)
2. Hash table degradation (longer chains, slower lookup)

### FINDING-D: Spinlock Held During I/O Dispatch

`contra_rotation.c:214-219`: the scheduler lock is held while calling `cr_schedule_consumer()` and `cr_schedule_enterprise()`, which internally call `cr_read_temp_mc()` (thermal zone read). Thermal zone reads can block or take milliseconds. Holding a spinlock during potential I/O is a lockdep violation and can cause deadlocks or excessive lock contention.

### FINDING-E: Consumer vs Enterprise Signal Divergence

The codebase claims unified API ("MacBook-on-WiFi to DGX-on-IB") but the achievable performance targets diverge by 10-100x between consumer and enterprise:

| Target | Enterprise | Consumer | Ratio |
|--------|-----------|----------|-------|
| T3 latency | ~5us (IB) | ~200us+ (WiFi+WG) | 40x |
| Attestation new | ~300us (IB) | ~30ms (WiFi) | 100x |
| Gateway | ~800ns (FPGA) | ~5us (SW) | 6x |
| EP-RDMA savings | ~35% (8x200G) | N/A (no SerDes) | - |

This is not a bug -- it is an inherent physical constraint. But the unified latency targets in `vdram.h:216-221` do not distinguish between these profiles. The test infrastructure should validate against profile-specific targets.
