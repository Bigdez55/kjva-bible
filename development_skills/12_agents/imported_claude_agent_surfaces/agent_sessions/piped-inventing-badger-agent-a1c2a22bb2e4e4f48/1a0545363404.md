### PHASE 1 — PortfolioSnapshot Cron Job — 2h backend

**File:** `backend/app/services/portfolio_snapshot_service.py` (NEW)

**Logic:**
1. Runs at 4:05 PM EST daily (after market close), via APScheduler or Cloud Scheduler
2. For each active portfolio: fetch current Alpaca equity value
3. Compute `daily_return_pct = (today_value - yesterday_value) / yesterday_value * 100`
4. Fetch SPY close price; normalize to portfolio start value for benchmark_json
5. Insert `PortfolioSnapshot` row
6. Purge snapshots older than 5 years

**APScheduler integration:** Add to `main.py` lifespan:
```python
scheduler.add_job(snapshot_service.run_daily_snapshot, 'cron',
                  hour=21, minute=5, timezone='UTC')  # 4:05 PM EST
```

**Endpoint to backfill (admin only):** `POST /admin/snapshots/backfill?days=90`

**Effort:** 2h backend

### PHASE 2 — Backend Analytics Endpoints — 6h backend

**File:** `backend/app/api/endpoints/analytics.py` (NEW)

Endpoints:

1. `GET /analytics/performance-metrics`
   - Params: `portfolio_id`, `period=1Y`
   - Returns: `{ sharpe_ratio, sortino_ratio, calmar_ratio, information_ratio, win_rate, profit_factor, expectancy, max_drawdown, max_drawdown_pct, recovery_days }`
   - Computation: Pull `PortfolioSnapshot` rows, compute all metrics server-side (Python)
   - Sharpe: `(mean_daily_return - risk_free_daily) / std_daily_return * sqrt(252)`
   - Sortino: `mean_daily_return / downside_deviation * sqrt(252)` (downside_dev = std of negative returns only)
   - Calmar: `annualized_return / abs(max_drawdown_pct)`
   - Information Ratio: `(portfolio_return - benchmark_return) / tracking_error`
   - Win Rate: `filled trades with realized_gain > 0 / total filled trades`
   - Profit Factor: `gross_profit / abs(gross_loss)`
   - Expectancy: `(win_rate * avg_win) + ((1-win_rate) * avg_loss)`

2. `GET /analytics/drawdown-series`
   - Returns: `[{ date, portfolio_value, peak_value, drawdown_pct }]`
   - Computed from PortfolioSnapshot by tracking running peak

3. `GET /analytics/monthly-returns`
   - Returns: `{ grid: { [year]: { [month]: pct } }, annual_returns: { [year]: pct } }`

4. `GET /analytics/return-distribution`
   - Returns: `{ bins: number[], counts: number[], mean: number, std: number, skew: number, kurt: number }`

5. `GET /analytics/benchmark-comparison`
   - Params: `benchmark=SPY&period=1Y`
   - Returns: `{ portfolio_series, benchmark_series, alpha, beta, correlation }`

6. `GET /analytics/pnl-attribution`
   - Returns: `{ by_sector, by_strategy, by_time_of_day, by_symbol_category }`
   - by_strategy groups on `trades.strategy` field + JournalNote.strategy_tag

**Performance metrics cache:** 15-min TTL (recalculate is expensive)

## KEY ARCHITECTURAL DECISION: WASM-Accelerated Statistics

Sharpe, Sortino, and Calmar calculations in the frontend (for instant period-switch response without backend round-trip) should use the existing WASM module at `frontend/src/wasm/wasm.ts`. The backend computes authoritative values; the frontend can compute preview values during period selector changes using the same WASM infrastructure already in place for chart scaling.
