## Key Design Decisions

### Protocol: gRPC (internal) + REST (external)
- Go services expose both gRPC (port 9090) and REST/HTTP (port 8080)
- Python monolith calls Go services via REST during migration phase
- Frontend continues to call Python proxy (unchanged URLs) initially
- Go services call each other via gRPC internally (Order Router -> Risk Engine)

### Migration Strategy: Strangler Fig
1. Phase 1: Go services run alongside Python, Python proxies to Go
2. Phase 2: Frontend directly hits Go services via new /v2/ prefix
3. Phase 3: Python endpoints removed once Go is validated

### Database Access Pattern
- Market Gateway: READ ONLY — market_data, assets tables via pgx pool
- Risk Engine: READ ONLY — portfolios, holdings, trades tables via pgx pool
- Order Router: READ + WRITE — trades table, calls Alpaca API; uses serializable transactions for order writes

### Auth: JWT token forwarded via gRPC metadata / HTTP Authorization header
- Go services validate JWT using shared secret (same as Python ALGORITHM/SECRET_KEY)
- No separate auth service needed during migration

## Source Truth Mapping
- Python `QuoteResponse` → proto `Quote` message
- Python `MarketDataResponse` → proto `Bar` message
- Python `TradeRiskAssessment` → proto `TradeRiskAssessment` message
- Python `RiskMetrics` → proto `PortfolioRiskMetrics` message
- Python `TradeStatus` enum → proto `OrderStatus` enum
- Python `OrderType` enum → proto `OrderType` enum

## Error Format
All Go services return JSON errors compatible with FastAPI's HTTPException format:
```json
{"detail": "error message", "code": "ERROR_CODE", "service": "market-gateway"}
```

## Health Check Format
```json
{
  "status": "healthy",
  "service": "market-gateway",
  "version": "1.0.0",
  "checks": {
    "database": "healthy",
    "upstream_alpaca": "healthy"
  },
  "timestamp": "2026-02-25T10:00:00Z"
}
```
Only "healthy" is acceptable — "degraded" returns HTTP 503.
