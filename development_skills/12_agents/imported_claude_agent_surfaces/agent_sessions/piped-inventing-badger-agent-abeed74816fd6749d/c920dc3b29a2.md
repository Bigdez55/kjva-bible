## 1. Mathematical Correctness Specification

### 1.1 Value at Risk (Parametric, 95% CI, 1-Day)

**Source:** `risk_management.py`, lines 1179-1183

**Formula:**

```
VaR_95 = |mu - z_0.95 * sigma| * P

Where:
  mu    = mean of daily log returns (arithmetic returns used in current Python)
  sigma = standard deviation of daily returns (population std via np.std)
  z     = 1.645 (one-tailed z-score for 95% confidence)
  P     = portfolio value in USD
```

**CRITICAL NOTE:** The Python implementation uses `np.std()` which computes *population* standard deviation (ddof=0). For a sample of 252 daily returns, this introduces a slight downward bias vs. sample std (ddof=1). The Go implementation MUST use the same convention (population std, ddof=0) for shadow-mode parity, then migrate to sample std (ddof=1) as a planned improvement.

**Edge Cases:**

| Condition | Expected Behavior |
|---|---|
| Zero holdings (empty portfolio) | Return VaR = 0.0 |
| Single stock portfolio | sigma = std of that single stock's returns |
| All returns identical (zero variance) | VaR = |mu| * P (approaches 0) |
| NaN in return series | Filter out NaN before calculation. If < 20 valid returns remain, fall back to conservative default (2.5% of portfolio value) |
| Negative portfolio value (short) | Should not occur in Elson (long-only). If detected, log critical error, return default |
| Portfolio value = 0 | Return VaR = 0.0, skip all calculations |

**Numerical Precision:** `float64` is sufficient. VaR is rounded to 2 decimal places for output (`round(daily_var, 2)`). The z-score 1.645 is a hardcoded constant -- no need for a statistics library.

**Test Vectors:**

| # | Daily Returns (last 20) | Portfolio Value | Expected VaR (95%, 1-day) |
|---|---|---|---|
| TV-1 | 20 identical values of 0.001 | $100,000 | mu=0.001, sigma=0.0, VaR = |0.001| * 100000 = $100.00 |
| TV-2 | [-0.01, 0.01, -0.01, 0.01, ...] (alternating, 20 vals) | $100,000 | mu=0.0, sigma=0.01, VaR = 1.645 * 0.01 * 100000 = $1,645.00 |
| TV-3 | All zeros (20 vals) | $50,000 | mu=0.0, sigma=0.0, VaR = $0.00 |
| TV-4 | [-0.05, -0.03, -0.02, ...] (bear market, 20 vals where mean=-0.02, std=0.015) | $200,000 | VaR = |-0.02 - 1.645*0.015| * 200000 = 0.04468 * 200000 = $8,935.00 |
| TV-5 | Empty array / < 20 returns | $100,000 | Fallback: 0.025 * 100000 = $2,500.00 |

### 1.2 Portfolio Beta

**Source:** `risk_management.py`, lines 1174-1177

**Formula:**

```
beta = Cov(R_p, R_m) / Var(R_m)

Where:
  R_p = portfolio daily returns (weighted average of position returns)
  R_m = SPY daily returns (market benchmark)
  Cov = sample covariance (np.cov returns sample covariance, ddof=1)
  Var = population variance (np.var uses ddof=0)
```

**CRITICAL BUG IDENTIFIED:** The Python code mixes `np.cov` (which uses ddof=1 by default, computing sample covariance) with `np.var` (which uses ddof=0 by default, computing population variance). This inconsistency inflates beta by a factor of approximately `n/(n-1)` where `n` is the number of observations. For n=252, the error is ~0.4%, which is within acceptable tolerance but should be documented.

**Go Implementation Decision:** Use consistent ddof. Recommendation: both numerator and denominator use ddof=1 (sample statistics). This is the standard finance convention.

```
beta = sum((R_p[i] - mean_Rp) * (R_m[i] - mean_Rm)) / sum((R_m[i] - mean_Rm)^2)
```

This is mathematically equivalent to Cov(ddof=1) / Var(ddof=1) since the `1/(n-1)` cancels.

**Edge Cases:**

| Condition | Expected Behavior |
|---|---|
| Market variance = 0 (flat market) | Return beta = 1.0 (defensive default) |
| Single holding | Beta = that stock's beta vs SPY |
| SPY data unavailable | Return beta = 1.0 (default) |
| Portfolio 100% cash | Beta = 0.0 (no market exposure) |
| Short series (< 20 observations) | Return beta = 1.0 (default) |

**Test Vectors:**

| # | Portfolio Returns | SPY Returns | Expected Beta |
|---|---|---|---|
| TV-1 | Identical to SPY | SPY | beta = 1.000 |
| TV-2 | 2x SPY (each day) | SPY | beta = 2.000 |
| TV-3 | All zeros | SPY (non-zero variance) | beta = 0.000 |
| TV-4 | Negative of SPY | SPY | beta = -1.000 |
| TV-5 | Random uncorrelated | SPY | beta ~= 0.0 (within tolerance of +/- 0.3 for small samples) |

### 1.3 Sharpe Ratio

**Source:** `risk_management.py`, lines 1186-1190

**Formula:**

```
Sharpe = (mean(R_p - R_f) * 252) / (std(R_p) * sqrt(252))

Simplified:
Sharpe = (mean_excess_daily * 252) / annualized_volatility

Where:
  R_f_daily = 0.04 / 252  (4% annual risk-free rate, daily)
  R_p       = portfolio daily returns
  annualized_volatility = std(R_p) * sqrt(252)  [population std]
```

**Precision Note:** The risk-free rate (4%) should be configurable. As of 2026-02-25, the US 1-year Treasury yield is the appropriate proxy. This value MUST be externalized in the Go config, not hardcoded.

**Edge Cases:**

| Condition | Expected Behavior |
|---|---|
| Annualized volatility = 0 | Return Sharpe = 0.0 (avoid division by zero) |
| All negative excess returns | Return negative Sharpe (valid, indicates underperformance vs. risk-free) |
| Very short history (< 20 days) | Return default Sharpe = 0.5 (conservative) |
| NaN in returns | Filter before calculation |

**Test Vectors:**

| # | Daily Returns (252 days) | Risk-Free | Expected Sharpe |
|---|---|---|---|
| TV-1 | Constant 0.001/day (25.2% annual) | 4% | excess_daily = 0.001 - 0.000159 = 0.000841; annualized_vol = 0; Sharpe = 0.0 (div by zero guard) |
| TV-2 | mean=0.0004, std=0.01 (daily) | 4% | excess = 0.0004 - 0.000159 = 0.000241; Sharpe = (0.000241*252) / (0.01*15.875) = 0.0607/0.1587 = 0.383 |
| TV-3 | mean=-0.001, std=0.02 (daily) | 4% | excess = -0.001159; Sharpe = (-0.001159*252) / (0.02*15.875) = -0.2920/0.3175 = -0.920 |
| TV-4 | mean=0.001, std=0.005 | 4% | excess = 0.000841; Sharpe = (0.000841*252) / (0.005*15.875) = 0.2119/0.0794 = 2.669 |
| TV-5 | Empty / insufficient data | 4% | Default = 0.5 |

### 1.4 Maximum Drawdown

**Source:** `risk_management.py`, lines 1192-1196

**Formula:**

```
cumulative_returns[i] = prod(1 + R_p[0..i])
running_max[i] = max(cumulative_returns[0..i])
drawdown[i] = (cumulative_returns[i] - running_max[i]) / running_max[i]
max_drawdown = abs(min(drawdown))
```

**Edge Cases:**

| Condition | Expected Behavior |
|---|---|
| All positive returns | max_drawdown = 0.0 (no drawdown occurred) |
| Monotonically declining | max_drawdown = total loss percentage |
| Single day of data | max_drawdown = 0.0 |
| Running max = 0 | Should not occur (cumulative return starts at 1.0). If somehow triggered, return 0.0 |

**Test Vectors:**

| # | Daily Returns | Expected Max Drawdown |
|---|---|---|
| TV-1 | [0.10, -0.05, 0.03, -0.08, 0.02] | cum=[1.10, 1.045, 1.076, 0.990, 1.010], max_dd = (0.990-1.10)/1.10 = 0.1000 |
| TV-2 | [0.01, 0.01, 0.01, 0.01] (all up) | max_dd = 0.0000 |

### 1.5 HHI Concentration Index

**Source:** `risk_management.py`, lines 422-425

**Formula:**

```
HHI = sum(w_i^2)

Where:
  w_i = market_value_of_position_i / total_portfolio_value
```

**Interpretation:**
- HHI = 1.0: Single stock portfolio (maximum concentration)
- HHI = 1/N: Equally weighted N-stock portfolio (minimum concentration for N stocks)
- HHI > 0.5: Python flags as "high concentration" warning

**Edge Cases:**

| Condition | Expected Behavior |
|---|---|
| No positions | HHI = 0.0 |
| Single position (100%) | HHI = 1.0 |
| Two positions (50/50) | HHI = 0.50 |
| Total portfolio value = 0 | HHI = 0.0 (guard against division by zero) |
| Negative market value (short) | Not applicable (Elson is long-only); if encountered, use abs() |

## 2. Execution Strategy Safety Review

### 2.1 TWAP (Time-Weighted Average Price)

**Source:** `trade_execution.py`, lines 922-927

**Current Implementation:** TWAP is identical to even chunking. The quantity is divided into equal chunks, each `max_order_chunk_size` (default $10,000 notional).

**Invariants:**

| Invariant | Specification |
|---|---|
| Total filled = original quantity | `sum(chunk_filled_qty) == original_quantity` (within fractional share tolerance of 0.001) |
| Chunk count | `ceil(original_quantity / max_order_chunk_size)` |
| Minimum time between chunks | `min_time_between_chunks` (default: 5 seconds). The Go implementation MUST use `time.Sleep()` or ticker, NOT busy-wait |
| Maximum chunk size | No single chunk exceeds `max_order_chunk_size` |
| Chunk total = original | `sum(chunk_sizes) == original_quantity` (exact, enforced by adjusting last chunk) |

**Partial Fill Handling:**
- If chunk N fails: mark chunk as REJECTED, continue with chunk N+1
- After all chunks: if `executed_quantity < original_quantity`, parent trade status = PARTIALLY_FILLED
- If `executed_quantity == 0`, parent trade status = REJECTED
- The Go service MUST log every chunk result individually for audit

**Max Slippage Tolerance:**
- Current Python: `execution_quality_threshold = 0.001` (0.1% / 10 bps)
- The Go service SHOULD abort the remaining chunks if cumulative slippage exceeds 50 bps (0.5%) from the VWAP of executed chunks vs. arrival price
- This is a NEW safety control not present in Python -- required for the Go upgrade

### 2.2 VWAP (Volume-Weighted Average Price)

**Source:** `trade_execution.py`, lines 929-996

**Current Implementation:** Simplified 3-bucket approximation:
- Opening: 25% of quantity
- Midday: 50% of quantity
- Closing: 25% of quantity

**Invariants:**

| Invariant | Specification |
|---|---|
| Weight sum = 100% | 0.25 + 0.50 + 0.25 = 1.00 (exact) |
| Total filled = original quantity | Same as TWAP |
| Sub-chunk safety | If any bucket exceeds max_order_chunk_size, it is sub-divided into even chunks |
| Last chunk adjustment | `chunks[-1] += quantity - sum(chunks)` to guarantee exact total |

**Go Improvement Opportunity:** The VWAP strategy should eventually use actual historical intraday volume profiles (5-minute bars) to distribute more accurately. The 3-bucket model is a placeholder. The Go implementation should define a `VolumeProfileProvider` interface that can be upgraded later.

### 2.3 Iceberg

**Source:** `trade_execution.py`, lines 998-1014

**Current Implementation:**
- Visible chunk: `min(quantity * 0.20, max_order_chunk_size)`
- Remaining: distributed via even chunking

**Invariants:**

| Invariant | Specification |
|---|---|
| Visible chunk <= 20% of total | `visible <= quantity * 0.20` |
| Visible chunk <= max chunk size | `visible <= max_order_chunk_size` |
| Total = visible + hidden chunks | `visible + sum(remaining_chunks) == quantity` |
| Inter-chunk timing | Same minimum delay as TWAP (5 seconds) |

**Go Safety Addition:** Iceberg orders MUST randomize chunk sizes slightly (+/- 10% of base chunk) to avoid detectable patterns. The Python implementation does NOT do this -- it sends identical chunk sizes, which is a market microstructure fingerprinting risk.

### 2.4 Market Hours Enforcement

**Source:** `alpaca.py`, lines 640-703

**Rules:**

| Asset Class | Trading Hours | Time Zone | `time_in_force` |
|---|---|---|---|
| US Equity | 9:30 AM - 4:00 PM EST (regular) | America/New_York | `day` |
| US Equity (extended) | 4:00 AM - 8:00 PM EST | America/New_York | `day` (Alpaca handles) |
| Crypto | 24/7 | UTC | `gtc` |

**Go Requirement:** The Go Order Router MUST check market hours BEFORE submitting equity orders. If the market is closed and the order is `time_in_force=day`, the order must be queued (not submitted) and the user notified. Crypto orders bypass this check entirely.

**Implementation:** Call Alpaca `GET /v2/clock` and cache the result for 60 seconds. Do NOT rely on local system clock for market hours determination (DST transitions, holidays).

**Source:** `circuit_breaker.py`, lines 121-194

### 3.2 Circuit Breaker State Machine

```
CLOSED -> CAUTIOUS -> RESTRICTED -> OPEN
   ^                                  |
   |        (auto-reset cascade)      |
   +---- CAUTIOUS <-- RESTRICTED <----+
```

**Reset Behavior (graduated, NOT immediate):**
- OPEN -> RESTRICTED (first auto-reset)
- RESTRICTED -> CAUTIOUS (second auto-reset)
- CAUTIOUS -> CLOSED (removal from active breakers)
- Each step requires a full cool-down period

**Persistence:**
- State persisted to PostgreSQL (`circuit_breaker_state` table, singleton row id=1)
- Also persisted to local JSON file as fallback
- Go service MUST persist to PostgreSQL (Cloud Run containers are ephemeral)
- Go service MUST load state from DB on startup

### 3.3 Position Sizing Matrix (Full)

The position sizing multiplier is a function of BOTH circuit breaker status AND volatility regime:

| CB Status \ Vol Level | Low | Normal | High | Extreme |
|---|---|---|---|---|
| CLOSED | 1.00 | 0.90 | 0.10 | 0.03 |
| CAUTIOUS | 0.85 | 0.65 | 0.15 | 0.05 |
| RESTRICTED | 0.60 | 0.40 | 0.10 | 0.03 |
| OPEN | 0.00 | 0.00 | 0.00 | 0.00 |

**Go Invariant:** `position_size = base_size * multiplier`. Multiplier MUST NEVER exceed 1.0. Multiplier of 0.0 MUST result in zero order submission (hard block).

## 4. Kill Switch Protocol

### 4.1 Activation

The kill switch is activated by tripping the SYSTEM circuit breaker:

```go
// Equivalent of Python: circuit_breaker.trip(CircuitBreakerType.SYSTEM, reason)
func (ks *KillSwitch) Activate(reason string) error {
    // 1. Set system circuit breaker to OPEN
    // 2. Cancel ALL open orders via Alpaca API
    // 3. Prevent new order submission
    // 4. Persist state to DB
    // 5. Send alert (PagerDuty / webhook)
}
```

### 4.2 Required Actions on Activation

| Step | Action | Timeout | Failure Mode |
|---|---|---|---|
| 1 | Set CB to OPEN in DB | 5s | Retry 3x, then log critical and proceed |
| 2 | Fetch all open orders from Alpaca: `GET /v2/orders?status=open` | 10s | Log error, continue |
| 3 | Cancel each open order: `DELETE /v2/orders/{id}` | 5s per order | Log failures, continue with next |
| 4 | Bulk cancel fallback: `DELETE /v2/orders` | 10s | Critical alert if this also fails |
| 5 | Block order submission goroutine | Immediate | Atomic bool / channel close |
| 6 | Notify Python auto-trading loop | 5s | Via shared DB flag or Redis pub/sub |

### 4.3 Cross-Service Sync with Python Auto-Trading Loop

**Current Architecture:** The Python `auto_trading_service.py` runs a periodic loop that calls the circuit breaker `check()` method before executing trades.

**Go Integration Options (ranked by recommendation):**

1. **Shared DB State (RECOMMENDED):** Both Go and Python read/write to the `circuit_breaker_state` table. The Python loop already queries this. No new infrastructure needed.

2. **Redis Pub/Sub:** Go publishes kill switch events to a Redis channel. Python subscribes. Requires Redis (currently blocked by GCP billing).

3. **HTTP Callback:** Go sends POST to Python's `/internal/kill-switch` endpoint. Adds coupling.

**Go MUST write the kill switch state to the same `circuit_breaker_state` table (id=1) that Python reads.** The JSON schema must be compatible:

```json
{
  "system": {
    "type": "system",
    "scope": null,
    "status": "open",
    "reason": "Kill switch activated: <reason>",
    "tripped_at": "2026-02-25T14:30:00+00:00",
    "auto_reset_at": null
  }
}
```

### 4.4 Recovery Procedure

1. **Manual reset only.** System breakers have `reset_after_minutes: 0` (no auto-reset)
2. Admin calls `POST /api/v1/circuit-breaker/reset` with type=system
3. Graduated recovery: OPEN -> RESTRICTED -> CAUTIOUS -> CLOSED
4. Each step requires explicit admin action (3 total reset calls)
5. After CLOSED state reached, order submission is re-enabled
6. **IMPORTANT:** Go must verify Alpaca account is not suspended (`trading_blocked: false`) before re-enabling

## 5. Data Integrity for Financial Calculations

### 5.1 Decimal vs. Float

| Domain | Type | Rationale |
|---|---|---|
| Order quantities | `float64` | Alpaca API accepts float strings. Fractional shares are sub-penny. float64 gives 15-17 significant digits. |
| Currency amounts (price, commission, total) | `shopspring/decimal` or `cockroachdb/apd` | Financial amounts MUST use arbitrary-precision decimal. float64 cannot represent $0.10 exactly. |
| Risk metrics (VaR, beta, Sharpe) | `float64` | Statistical calculations. Precision beyond 4 decimal places is noise. |
| Portfolio value | `shopspring/decimal` | Sum of many positions. Floating point accumulation error over 50+ positions can exceed $0.01 |
| Percentage calculations | `float64` | Intermediate calculations. Only use decimal for final currency conversion. |

**Go Library Recommendation:** `github.com/shopspring/decimal` -- widely adopted, thread-safe, supports `MarshalJSON`.

**CRITICAL:** When converting between `decimal.Decimal` and `float64` for risk calculations, always convert decimal -> float64 at the START of the calculation pipeline, and float64 -> decimal at the END for currency amounts. Never mix mid-calculation.

### 5.2 Rounding Rules

| Context | Rounding Mode | Precision |
|---|---|---|
| Currency amounts (USD) | `ROUND_HALF_EVEN` (banker's rounding) | 2 decimal places |
| Share quantities (equities) | `ROUND_DOWN` (truncate) | 0 decimal places (whole shares) |
| Share quantities (fractional) | `ROUND_DOWN` (truncate) | 9 decimal places (Alpaca limit) |
| Share quantities (crypto) | `ROUND_DOWN` (truncate) | 8 decimal places |
| Risk metrics display | `ROUND_HALF_UP` | Per-metric (VaR: 2dp, beta: 3dp, Sharpe: 3dp, drawdown: 4dp) |
| Percentages | `ROUND_HALF_UP` | 2 decimal places |

**Invariant for order quantities:** ALWAYS round DOWN when calculating fill quantities. Rounding UP on a sell order could attempt to sell shares you do not own. Rounding UP on a buy order could exceed buying power.

### 5.3 Timestamp Handling

| Rule | Specification |
|---|---|
| Internal storage | UTC everywhere. All Go `time.Time` values MUST be in UTC. |
| Database columns | `TIMESTAMP WITH TIME ZONE` (PostgreSQL). Go writes UTC. |
| Alpaca API | Returns ISO 8601 with timezone. Parse with `time.Parse(time.RFC3339, ...)` |
| Market hours | Convert to `America/New_York` for comparison ONLY. Never store EST/EDT. |
| Daily reset | Daily loss counters reset at 00:00 UTC (which is roughly 7/8 PM ET). This is DIFFERENT from market close. |
| Display to user | Frontend handles timezone conversion. Go ALWAYS returns UTC. |

### 5.4 Settlement Date

| Asset Class | Settlement | Rule |
|---|---|---|
| US Equities | T+1 | Trade date + 1 business day (SEC Rule 15c6-1, effective May 2024) |
| Crypto | T+0 | Instant settlement |
| Options | T+1 | Same as equities |

**CRITICAL BUG IN PYTHON:** `trade_execution.py` line 887 hardcodes `T+2`:
```python
trade.settlement_date = datetime.now(UTC) + timedelta(days=2)
```
This is WRONG. US equities moved to T+1 on May 28, 2024. The Go implementation MUST use T+1 for equities and T+0 for crypto.

**Go Implementation:**
```go
func CalculateSettlementDate(tradeDate time.Time, assetClass string) time.Time {
    switch assetClass {
    case "crypto":
        return tradeDate  // T+0
    default:  // equity, option
        return nextBusinessDay(tradeDate, 1)  // T+1, skip weekends and NYSE holidays
    }
}
```

The `nextBusinessDay` function must account for NYSE holiday calendar. Use a static calendar or `github.com/rickar/cal/v2`.

## 6. Shadow Mode Validation Protocol

### 6.1 Architecture

```
                  +-------------------+
  Trading Signal  |  Python Order     |----> Alpaca API (LIVE)
  ────────────────>  Router (current) |
       |          +-------------------+
       |
       |          +-------------------+
       +--------->|  Go Order Router  |----> Log-only (NO execution)
                  |  (shadow mode)    |
                  +-------------------+
                           |
                           v
                  +-------------------+
                  |  Comparison DB    |
                  |  (divergence log) |
                  +-------------------+
```

Both services receive the same input (symbol, side, qty, order_type, price, strategy). The Go service computes its output but does NOT submit to Alpaca. A comparison worker reconciles results.

### 6.2 Comparison Fields

| Field | Tolerance | Divergence Type |
|---|---|---|
| Order action (submit/reject) | Exact match | CRITICAL -- blocks promotion |

## 7. Regulatory Compliance

### 7.1 Pattern Day Trader (PDT) Detection

**Rule:** FINRA Rule 4210. An account is flagged as PDT if it executes 4+ day trades in 5 business days AND the account equity is below $25,000.

**Current Python Implementation:** Alpaca returns `pattern_day_trader: bool` in the account response (`alpaca.py` line 479). The Python code reads this but does NOT proactively prevent day trades.

**Go Requirement:**
```go
type PDTTracker struct {
    // Track day trades in a rolling 5-business-day window
    DayTrades []DayTrade  // Timestamped buy-then-sell-same-day pairs
    AccountEquity float64
}

func (t *PDTTracker) CanExecuteDayTrade() (bool, string) {
    recentDayTrades := t.countDayTradesInLast5BusinessDays()
    if recentDayTrades >= 3 && t.AccountEquity < 25000.0 {
        return false, "PDT: 3 day trades in 5 days, equity below $25K. Next day trade would trigger PDT flag."
    }
    return true, ""
}
```

**Counting Method:** A "day trade" is opening AND closing the same position on the same calendar day. Multiple partial fills of the same order count as one trade. The 5-day window is rolling business days.

### 7.2 Wash Sale Tracking

**Rule:** IRS Section 1091. A wash sale occurs when you sell a security at a loss and buy a "substantially identical" security within 30 days before or after the sale.

**Go Requirement:** The Go Order Router does NOT need to enforce wash sale rules (that is a tax reporting concern, not a trading restriction). However, it MUST:
1. Log all sell-at-loss events with the loss amount
2. Log all buy events for the same symbol within 30 days of a loss sale
3. Tag these events for the tax reporting module

### 7.3 Best Execution Obligation

**Rule:** SEC Rule 606 / FINRA Rule 5310. Brokers must seek the best execution reasonably available for customer orders.

**Alpaca Responsibility:** Alpaca, as the broker-dealer, bears the primary best execution obligation. Elson, as the software platform, should:
1. Not artificially delay order submission
2. Log latency from decision to Alpaca API submission
3. Record execution quality metrics (slippage, fill rate)
4. Not route orders to intentionally disadvantageous venues

**Go Metrics to Log:**
- `decision_to_submit_latency_us`: Microseconds from Go risk engine approval to Alpaca API call
- `alpaca_api_latency_ms`: Alpaca round-trip time
- `slippage_bps`: Basis points of slippage from expected price

## 8. Critical Bugs Found in Python Source

During this review, the following issues were identified in the Python source code that MUST be addressed regardless of the Go migration:

### 8.1 CRITICAL: Settlement Date is T+2 (Should be T+1)

**File:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/services/trade_execution.py`
**Line:** 887
```python
trade.settlement_date = datetime.now(UTC) + timedelta(days=2)
```
**Impact:** Settlement date is 1 day late. Affects cash balance availability calculations.
**Fix:** Change to `timedelta(days=1)` for equities, `timedelta(days=0)` for crypto.

### 8.2 MEDIUM: Beta Calculation Uses Mixed ddof

**File:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/services/risk_management.py`
**Lines:** 1175-1177
```python
covariance = np.cov(portfolio_returns, spy_returns_aligned)[0, 1]  # ddof=1
market_variance = np.var(spy_returns_aligned)  # ddof=0
```
**Impact:** Beta is inflated by ~0.4% for 252-day windows. Minimal practical impact but technically incorrect.
**Fix:** Use `np.var(spy_returns_aligned, ddof=1)` to match `np.cov`.

### 8.3 MEDIUM: TWAP Uses Blocking time.sleep() in Async Context

**File:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/services/trade_execution.py`
**Line:** 597
```python
time.sleep(self.min_time_between_chunks)  # Blocks the event loop!
```
**Impact:** In an async context, `time.sleep()` blocks the entire event loop for 5 seconds per chunk. For a 10-chunk order, that is 45 seconds of blocked event loop.
**Fix:** Use `await asyncio.sleep(self.min_time_between_chunks)`.

### 8.4 LOW: VaR Fallback Uses Population Std but Comment Says 95% CI

**File:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/services/risk_management.py`
**Line:** 1225
```python
"daily_var": portfolio_value * 0.025,  # 2.5% VaR (conservative)
```
**Impact:** The 2.5% default is reasonable but not calibrated. For a $100k portfolio, this means VaR defaults to $2,500 which may be too aggressive for a conservative portfolio or too lenient for a speculative one.
**Recommendation:** No code change needed, but the Go implementation should make this configurable.

### 8.5 INFO: Half-Open Circuit Breaker Uses random.random()

**File:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/trading_engine/engine/circuit_breaker.py`
**Line:** 623
```python

    return True, status  # Allow 10% of trades through
```
**Impact:** Non-deterministic behavior in a safety-critical path. For the Go implementation, use a counter-based approach (allow every 10th trade) instead of randomness, so behavior is reproducible in testing.
