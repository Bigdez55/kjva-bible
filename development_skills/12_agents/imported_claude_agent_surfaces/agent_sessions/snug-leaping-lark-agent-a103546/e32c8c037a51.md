## 3. The Audit (Feedback Loop)

### CRITICAL Severity Issues

#### Issue C-1: SQL Injection Vulnerability in System Tasks
- **What:** Dynamic SQL query using f-string without parameterization
- **Where:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/tasks/system.py:453`
- **Why:** If `table` variable is ever derived from user input (even indirectly), this enables SQL injection attacks that could expose or destroy all database data
- **How:** Replace with parameterized query or use SQLAlchemy ORM

```python
# CURRENT (VULNERABLE):
count = db.execute(f"SELECT COUNT(*) FROM {table}").scalar()

# RECOMMENDED FIX:
from sqlalchemy import text, inspect
# Validate table exists in schema first
inspector = inspect(db.get_bind())
if table in inspector.get_table_names():
    count = db.execute(text(f"SELECT COUNT(*) FROM {table}")).scalar()
```

#### Issue C-2: Float Usage in Financial Models (Subscription Model)
- **What:** `Float` type used for monetary amounts in Subscription and SubscriptionPayment models
- **Where:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/models/subscription.py:62,139`
- **Why:** Floating-point arithmetic causes precision errors in financial calculations (e.g., 0.1 + 0.2 != 0.3). This can lead to billing discrepancies.
- **How:** Migrate to `Numeric(18, 2)` for currency fields

```python
# CURRENT (VULNERABLE):
price = Column(Float, nullable=False)  # Line 62
amount = Column(Float, nullable=False)  # Line 139

# RECOMMENDED FIX:
from sqlalchemy import Numeric
price = Column(Numeric(18, 2), nullable=False)
amount = Column(Numeric(18, 2), nullable=False)
```

#### Issue C-3: Float Usage in User Settings (Financial Limits)
- **What:** `Float` type used for financial limits in UserSettings model
- **Where:** `/Users/desmondearly/Library/CloudStorage/OneDrive-Personal/Trading Bot/Elson TB2/Elson/Elson-TB2/backend/app/models/user_settings.py:41,43,59,60`
- **Why:** These limits control investment amounts and thresholds - precision errors could allow users to invest more/less than intended
- **How:** Migrate to `Numeric(18, 4)` for limit fields

```python
# CURRENT:
roundup_multiplier = Column(Float, default=1.0)  # Line 41
roundup_threshold = Column(Float, default=5.0)   # Line 43
max_weekly_roundup = Column(Float, default=50.0) # Line 59
max_monthly_micro_invest = Column(Float, default=200.0)  # Line 60

# RECOMMENDED:
roundup_multiplier = Column(Numeric(8, 4), default=1.0)
roundup_threshold = Column(Numeric(18, 4), default=5.0)
max_weekly_roundup = Column(Numeric(18, 2), default=50.0)
max_monthly_micro_invest = Column(Numeric(18, 2), default=200.0)
```

#### Issue H-3: Excessive `Any` Type Usage (Type Safety)
- **What:** 100+ occurrences of `Dict[str, Any]` and `List[Any]` return types
- **Where:** Multiple service files including:
  - `services/market_simulation.py:67,158`
  - `services/neural_network.py:42,137,203,271,347,383`
  - `services/order_aggregator.py:25,107`
  - `services/advisor.py:64,81,98,145,177,252,281`
  - And many more
- **Why:** Eliminates compile-time type checking, making runtime errors harder to catch
- **How:** Define specific TypedDict or dataclass types for each return structure

#### Issue H-4: Deprecated Pydantic v1 Pattern Usage
- **What:** Using `class Config:` instead of Pydantic v2 `model_config`
- **Where:** 44 schema files still use `class Config:` pattern
  - `schemas/core_schemas.py:223,350,457,564,666`
  - `schemas/portfolio.py:21,38,74`
  - `schemas/family.py:55,73,132,172,226,250,319`
  - `schemas/market_data.py:36,53,97`
  - And many more
- **Why:** While still functional, this is deprecated and may break in future Pydantic versions
- **How:** Migrate to `model_config = ConfigDict(from_attributes=True)`

```python
# CURRENT (DEPRECATED):
class Config:
    from_attributes = True

# RECOMMENDED:
from pydantic import ConfigDict
model_config = ConfigDict(from_attributes=True)
```
