## Key Findings

### P0: LLVM JIT in Kernel Space — Architectural CVE Factory
xisc_llvm_jit.c lines 41-43:
  JIT_CODE_CACHE_SIZE = 256MB W^X code cache
  LLVMCreateMCJITCompilerForModule in kernel context
  cache eviction by timestamp modulo (ktime_get_ns() % JIT_CODE_CACHE_ENTRIES) — NOT LRU

Confirmed vulnerabilities:
1. No bytecode validation before LLVM IR emission (line 203-248)
2. Cache eviction by timestamp modulo — deterministic for attacker with timing oracle
3. Linker symbol resolution via LLVMGetNamedFunction(LLVMGetGlobalContext()) —
   global LLVM context in kernel = shared across all modules
4. genos_matmul_f32 extern symbol — not validated, can be hijacked
5. pf_handler.c line 299-314: attestation called UNDER spinlock —
   T3 slow path calls kzalloc(GFP_KERNEL) = might sleep = BUG_ON in spinlock context

### P1: Confirmed Issues in Non-JIT Code
- rdma_attestation.c W2-SEC-011: *(u32*)digest hash key = 4-byte prefix collision (OPEN)
- rdma_attestation.c W2-SEC-012: session_key on fast-path cache hit not zeroized (OPEN)
- spdm_session_find line 168: XOR-fold improved but still 32-bit key space = 2^16 birthday

## Recommendation: REJECT all new languages for kernel. ONE exception for userspace.

## Language-by-Language Verdict
- Rust: REJECT for kernel (5.15 LTS blocker). CONDITIONAL ACCEPT for XISC userspace shim only.
- Zig: REJECT (pre-1.0, no kernel support, no formal safety guarantees)
- Ada/SPARK: REJECT (kernel space impossible, talent gap, no ROI for this stack)
- Modern C++: REJECT (exceptions in kernel = undefined behavior, vtables + RTTI disabled)

## eBPF + WAMR Verdict
- WAMR replacing xisc_llvm_jit.c: STRONG ACCEPT — eliminates P0 JIT-in-kernel risk
- eBPF for policy annotations: CONDITIONAL ACCEPT — requires BTF type safety enforcement
