The project's `MEMORY.md` specifies a hard language constraint: **"Python, TypeScript, C ONLY. No Go. No Rust. No Java."**

## 2. Domain Boundary Map: What MUST Stay in C

### 2.1 The Iron Rule: Linux Kernel Modules = C (or Rust-for-Linux)

The Linux kernel has exactly two officially supported languages as of kernel 6.12+ (March 2026):
- **C (gnu11)** -- the 30-year incumbent
- **Rust** -- officially permanent, no longer experimental (2025 Kernel Maintainer Summit, Tokyo)

**No other language has upstream kernel support.** This is not a matter of opinion -- it is a matter of build system integration (`Kconfig`, `Kbuild`), ABI compatibility, and maintainer acceptance.

#### Files that MUST use C or Rust-for-Linux:

```
kernel/linux-validation/vdram/pf_handler.c         -- vm_ops->fault callback
kernel/linux-validation/vdram/tier_migration.c      -- HMM, DMA, RDMA verbs
kernel/linux-validation/vdram/prefetch_engine.c     -- kernel workqueue
kernel/linux-validation/vdram/mr_registry.c         -- RCU, spinlock, kernel crypto
kernel/linux-validation/scheduler/contra_rotation.c -- kthread, thermal API
kernel/linux-validation/security/rdma_attestation.c -- kernel crypto, TPM, SPDM
kernel/linux-validation/security/xkabi_capabilities.c -- kernel IDR, HMAC, audit
kernel/linux-validation/mesh/genos_meshd.c          -- kthread, net/sock
gateway/zc_gw/qp_relay.c                           -- RCU, RDMA verbs
gateway/zc_gw/entropy_classifier.c                  -- kernel pure compute
```

### 2.2 The Flexible Zone: Userspace = Any Language

```
xisc/runtime/xkabi_linux_shim/  (14 .c files)      -- pthreads, memfd, POSIX
xisc/runtime/personality_linux/  (2 .c files)       -- syscall dispatch table
xisc/conformance/               (1 .c file)         -- test harness
```

These files link against `-lpthread -lrt`, use `memfd_create`, `pread/pwrite`, `socketpair`, and standard POSIX APIs. They can be written in any language that produces position-independent shared objects with C ABI exports.

## 3. Language-by-Language Deep Analysis

### 3.1 Rust-for-Linux (Kernel Space)

#### Current State (March 2026)

- **Official status:** Permanent core language in the Linux kernel (not experimental). The `experimental` tag was removed at the 2025 Kernel Maintainer Summit.
- **Production deployments:** Android 16 ships ashmem (anonymous shared memory) written in Rust on kernel 6.12. Millions of devices in production.
- **Distribution support:** Fedora and Ubuntu enable Rust in default kernels. Debian 13 (Trixie) provides Rust toolchain. **Debian 12 (Bookworm) does NOT ship Rust kernel support by default** -- it uses kernel 6.1 which has Rust support marked experimental, and the Rust toolchain must be manually installed.
- **Kernel version requirement:** Rust modules require kernel 6.1+ (initial support) but practical production use requires 6.6+ (stable abstractions). The GEN.OS target of Debian 12 ships kernel 6.1, which is the bare minimum.

#### Can Rust Handle GEN.OS Kernel Components?

| Component | Rust Feasibility | Rationale |
|-----------|-----------------|-----------|
| **Page fault handler** (`pf_handler.c`) | MEDIUM-LOW | `vm_operations_struct` Rust bindings do not exist upstream. You would need to write custom unsafe bindings for `vm_fault`, `vm_area_struct`, `struct page`, `spin_lock`, `ktime_get`, and the entire mm subsystem. This is 6-12 months of binding work for a 457-line file. |
| **Tier migration** (`tier_migration.c`) | LOW | Uses HMM (`hmm_range_fault`), `ib_verbs` (RDMA), NVMe `vfs_iocb_iter_write`, `filp_open`. None of these subsystems have Rust abstractions. The migration code touches 6 different kernel subsystems. |
| **Scheduler** (`contra_rotation.c`) | MEDIUM | `kthread_run`, `thermal_zone_get_temp`, `mutex`, `msleep` -- some of these have emerging Rust abstractions. Thermal API bindings would need writing. |
| **RDMA attestation** (`rdma_attestation.c`) | LOW | `crypto_shash`, `tpm_pcr_extend`, `hashtable`, SPDM protocol logic. The kernel crypto API has partial Rust bindings but RDMA verbs and TPM have none. |
| **XKABI capabilities** (`xkabi_capabilities.c`) | MEDIUM | IDR, spinlock, HMAC, audit subsystem. IDR has no Rust bindings. The HMAC-SHA256 validation logic is algorithmically clean and would benefit from Rust's type system, but the kernel plumbing is all C. |
| **Mesh daemon** (`genos_meshd.c`) | MEDIUM | Mostly kthread loops with stubbed-out networking. Could be rewritten in Rust, but it is also a candidate for moving to userspace entirely (it already uses `MODULE_LICENSE` and is structured like a daemon). |
| **QP relay** (`qp_relay.c`) | LOW | RCU (`hash_for_each_possible_rcu`), RDMA verbs, DOCA/FPGA stubs. The <800ns latency target makes this extremely sensitive to any abstraction overhead. |
| **Entropy classifier** (`entropy_classifier.c`) | HIGH | Pure computation: no kernel subsystem dependencies beyond types and `min_t`. 104 LOC. This is the easiest candidate, but rewriting 104 lines of correct pure-math C into Rust gains almost nothing. |

#### Verdict on Rust in Kernel Space

**The binding cost exceeds the safety benefit for GEN.OS's current kernel footprint.** The 2,809 LOC kernel codebase uses 8+ kernel subsystems (mm, HMM, RDMA, crypto, thermal, RCU, IDR, audit, TPM, workqueue, kthread, net/sock). Writing Rust bindings for all of these would cost 6-18 engineer-months and produce ~5,000-10,000 lines of `unsafe` binding code to wrap 2,809 lines of C.

The safety value proposition inverts: you would have MORE unsafe code than safe code.

**Recommendation: Keep kernel modules in C.** Revisit in 18-24 months when upstream Rust abstractions cover more kernel subsystems, particularly mm and RDMA.

#### Exception: Future kernel code for NEW subsystems

If GEN.OS adds kernel-resident components that do NOT depend on HMM/RDMA/TPM (e.g., a new IPC mechanism, a simple procfs/sysfs interface), Rust-for-Linux would be appropriate starting from kernel 6.6+, which means upgrading from Debian 12's kernel 6.1 or using a custom kernel build.

### 3.2 Rust in Userspace (XISC Shim, Mesh Daemon Alternative)

This is where Rust's value proposition becomes compelling.

#### XISC Linux Shim: Strong Candidate

The 14-file XISC shim (`xisc/runtime/xkabi_linux_shim/`) is:
- Pure userspace (pthreads, memfd, socketpair, eventfd, timerfd)
- A capability-enforcement layer (rights checking, handle table, audit)
- Currently C11 with CMake
- ~1,200 LOC with many stubs remaining

Rust would provide:
- **Ownership model maps directly to handle lifecycle management.** The `shim_handle_entry_t` with generation counters and `valid` flags is manually reimplementing what Rust's `Option<Arc<T>>` gives you for free.
- **Enum types for `xk_obj_type_t` and `xk_status_t` errors** -- Rust enums with `#[repr(C)]` provide exhaustive match checking.
- **Thread safety guarantees** -- the shim uses `pthread_mutex_t` everywhere. Rust's `Mutex<T>` ensures the data it protects cannot be accessed without holding the lock (a compile-time guarantee C cannot provide).
- **C ABI export** -- Rust can produce `.so` files with `#[no_mangle] extern "C"` exports that are drop-in replacements for the current CMake-built `libxkabi_shim.so`.

**Cost estimate:** 2-4 weeks for one engineer to rewrite the shim in Rust, given the stub-heavy state of the current code.

**Risk:** The XISC shim must maintain C ABI compatibility for the Linux personality layer and the conformance harness. Rust's `cbindgen` handles this, but it adds a build dependency.

#### Linux Personality Syscall Dispatch: Keep in C

The `linux_syscall_dispatch.c` is a switch statement mapping Linux syscall numbers to XKABI calls. It is trivially correct, needs no memory safety (no allocations, no pointers dereferenced), and benefits from C's direct mapping to the ABI. Rewriting gains nothing.

### 3.3 Zig

#### Kernel Space: NOT VIABLE

Zig cannot produce Linux kernel modules. Period.

- The Linux kernel build system (`Kbuild`) does not support Zig.
- Zig's runtime requires its own allocator model (`std.mem.Allocator`) which is incompatible with `kmalloc`/`kzalloc`/`GFP_KERNEL` flags.
- The zig.ko proof-of-concept ([Mic92/zig.ko](https://github.com/Mic92/zig.ko)) is a toy that hooks into the build system via workarounds and has not been maintained or accepted upstream.
- FreeBSD 14.1 and OpenBSD 7.5 have experimental Zig module support, but GEN.OS targets Linux.

**Verdict: Zig is categorically excluded from GEN.OS kernel space.**

#### Userspace: Potentially Viable but Immature

Zig 1.0 is expected in 2026 but has not shipped yet. Pre-1.0 languages carry API instability risk. Specific concerns:

- **C interop is excellent** -- Zig can `@cImport` C headers directly, better than Rust's `bindgen`.
- **No hidden allocations** -- valuable for the XISC shim's deterministic behavior requirements.
- **Comptime** -- could replace the macro-heavy shim patterns with compile-time type generation.
- **But:** No ecosystem for SPDM, RDMA, or crypto libraries. No formal verification story. No kernel community investment. Pre-1.0 instability means the language semantics may change under your feet.

**Verdict: Monitor Zig post-1.0. Do not adopt pre-1.0 for production OS infrastructure.** The XISC shim rewrite in Zig would be technically possible but provides less safety guarantee than Rust (no borrow checker, no lifetime analysis) while adding ecosystem risk.

### 3.4 Ada/SPARK (Formal Verification)

#### The Dream: Formally Verified Attestation

SPARK can formally verify absence of:
- Buffer overflows
- Integer overflows
- Division by zero
- Uninitialized reads
- Certain classes of logic errors (precondition/postcondition violations)

The GEN.OS attestation layer (`rdma_attestation.c`, `xkabi_capabilities.c`) handles cryptographic session management, capability token validation, and rkey binding. Bugs here mean silent security bypass. SPARK formal verification would provide mathematical proof of correctness for these paths.

#### The Reality: Impractical for GEN.OS

1. **Ada/SPARK cannot produce Linux kernel modules.** The Ada runtime (even the zero-footprint `Ravenscar` profile) is incompatible with the Linux kernel environment. The kernel has its own memory allocator, its own synchronization primitives, and forbids floating-point in most contexts. Ada's runtime assumes a hosted environment.

2. **The Ironclad OS project** (2025) writes its own kernel in SPARK/Ada, but it is a standalone microkernel, not a Linux module. The EwoK microkernel and CuBit OS are similarly standalone.

3. **Formal verification effort:** SPARK proof requires writing contracts (preconditions, postconditions, loop invariants) that are typically 1.5-3x the size of the implementation code. For `xkabi_capabilities.c` (372 LOC), expect 600-1,100 lines of SPARK contracts plus the rewritten implementation. For `rdma_attestation.c` (322 LOC), expect 500-950 lines of contracts. Total effort: 3-6 engineer-months for two files, assuming SPARK expertise (which is rare and expensive).

4. **No kernel integration path:** Even if you wrote a SPARK implementation, you would need to either:
   - (a) Run it as a userspace service (requires IPC to kernel, adding latency to the <200ns cached attestation path), or
   - (b) Transpile SPARK to C and compile as a kernel module (fragile, loses formal verification guarantees at the C boundary), or
   - (c) Move the attestation logic to a standalone microkernel (massive architectural change).

5. **SPARK expertise cost:** There are approximately 500-1,000 engineers worldwide with production SPARK experience. Hiring one for a startup OS project is impractical.

**Verdict: SPARK formal verification is technically desirable for the attestation layer but practically infeasible for GEN.OS.** The kernel execution context, the team expertise requirements, and the integration cost make it a non-starter. If formal guarantees are needed, invest instead in exhaustive property-based testing (C-Check, AFL++, libFuzzer) and CBMC model checking of the existing C code -- these tools work directly on C within the kernel build environment.

### 3.5 Modern C++ in Kernel Space

#### Status: Permanently Blocked by Linux Policy

The Linux kernel explicitly prohibits C++. This is not going to change. The reasons are structural, not stylistic:

1. **Exceptions are incompatible with kernel context.** Stack unwinding in interrupt handlers, under spinlocks, or in NMI context is undefined. There is no kernel-compatible exception ABI.

2. **RTTI (Run-Time Type Information) adds ~5% binary bloat** and is useless in kernel code where all types are known at compile time.

3. **Hidden constructors/destructors** violate the kernel's requirement that every instruction be visible and auditable.

4. **Template instantiation** can cause unexpected binary size explosion in modules.

5. **The C++ ABI is not stable across compilers** -- the kernel must build with both GCC and Clang.

6. **Rust now occupies the "better than C" niche** that C++ would have filled, with stronger safety guarantees and no hidden control flow.

**Verdict: C++ is permanently excluded from Linux kernel space. Do not consider it.**

#### C++ in Userspace

The XISC shim could theoretically use C++ (it's userspace, CMake-built). However:
- It provides no safety advantages over C that Rust doesn't provide better.
- It adds ABI complexity (name mangling, exception tables) to a component that needs C ABI exports.
- The GEN.OS language policy already excludes it.

**Verdict: No path for C++ anywhere in GEN.OS.**

## 4. The XISC Runtime Shim: Language Modernization Evaluation

The 14-file userspace shim is the highest-value modernization target.

### 4.1 Evaluation Matrix

| Criterion | Hardened C (current) | Rust | Zig | C++ |
|-----------|---------------------|------|-----|-----|
| Memory safety | Manual (error-prone) | Compile-time enforced | Partial (no borrow checker) | Partial (smart pointers) |
| Thread safety | Manual (mutex discipline) | Compile-time enforced (`Send`/`Sync`) | Manual | Manual |
| C ABI export | Native | `extern "C"` + cbindgen | Native | Name mangling issues |
| Build complexity | CMake (current) | Cargo + cc crate | Zig build (replaces CMake) | CMake (current) |
| Ecosystem maturity | Infinite | Large | Small, pre-1.0 | Large |
| Team expertise required | Current team has this | Moderate learning curve | Steep (small community) | Common |
| Handle lifecycle modeling | Manual gen counters | `Option<Arc<T>>` natural | Manual | `shared_ptr` overhead |
| Audit trail correctness | Manual ring buffer | Type-safe ring buffer | Manual | Manual |
| Language policy status | ALLOWED | BLOCKED (requires amendment) | BLOCKED (requires amendment) | BLOCKED |
| Risk of rewrite bugs | None (no rewrite) | Moderate (new code) | High (immature tooling) | Moderate |

### 4.2 Recommendation for XISC Shim

**If the language policy is amended:** Rust is the optimal choice for the XISC shim rewrite. The handle table, object store, and rights enforcement logic map directly to Rust's ownership model. The stub-heavy state (many functions return `XK_ERR_NOT_SUPPORTED`) means the rewrite cost is low because most functions are trivial.

**If the language policy stands:** Harden the existing C with:
- `_Static_assert` on struct layouts
- `__attribute__((warn_unused_result))` on all status-returning functions
- Thread sanitizer CI integration (`-fsanitize=thread`)
- AddressSanitizer CI integration (`-fsanitize=address`)
- CBMC model checking on the handle table and object store

## 5. Gateway and Mesh Daemon: Network I/O Hot Paths

### 5.1 Zero-Copy Gateway (`gateway/zc_gw/`)

**Current state:** Kernel module using RCU, RDMA verbs, DOCA/FPGA stubs. Latency target: <800ns (FPGA), <5us (DPU).

**Must stay in C.** This is a kernel module with sub-microsecond latency requirements. Neither Rust-for-Linux bindings for RDMA verbs nor Zig kernel support exist. Even if Rust bindings existed, the <800ns FPGA path cannot tolerate any abstraction overhead.

### 5.2 Mesh Daemon (`genos_meshd.c`)

**Current state:** Built as a kernel module (`MODULE_LICENSE("Proprietary")`), but architecturally it is a daemon with kthreads doing mDNS discovery and gossip on 500ms-1s intervals. It accesses `vdram_node` via exported symbols.

**Architectural question:** Should this move to userspace?

Arguments for userspace:
- Network I/O (UDP sockets for mDNS/gossip) is better handled by userspace event loops.
- WireGuard auto-configuration is done via netlink, which works from userspace.
- Ed25519 signing uses libsodium or openssl, which are userspace libraries.
- A crash in the mesh daemon should NOT crash the kernel.

Arguments for kernel space:
- Direct access to `vdram_node` state (tier availability, thermal data) without IPC overhead.
- The gossip payload includes kernel-internal state (`contra_rotation_get_phase()`).

**Recommendation:** Split the mesh daemon into two components:
1. **Kernel-space thin exporter:** A small procfs/sysfs or netlink interface that exports vDRAM tier stats, thermal data, and CR phase to userspace. ~100 LOC of C.
2. **Userspace daemon:** mDNS, Ed25519, WireGuard autoconfig, gossip protocol. Written in C with `io_uring` for async networking (or Rust with `tokio` if the language policy is amended).

If the daemon stays in kernel space, it must remain C.

If moved to userspace:
- **C + io_uring:** Most natural migration. Reuse existing code with socket API replaced by io_uring SQE/CQE. ~2 weeks.
- **Rust + tokio:** Clean async implementation. Excellent for the gossip protocol (futures, select!, channels). ~3-4 weeks.
- **Zig:** Possible but io_uring support in Zig's standard library is immature. ~4-6 weeks.

## 6. Final Recommendation: Optimal Language Allocation

### 6.1 Decision Matrix

| Component | Domain | Recommended Language | Rationale |
|-----------|--------|---------------------|-----------|
| vDRAM page fault handler | Kernel | **C** | No Rust mm bindings. <2us budget. |
| Tier migration engine | Kernel | **C** | HMM + RDMA + DMA -- no Rust bindings for any of these. |
| Prefetch engine | Kernel | **C** | Workqueue + sort. Simple. No reason to change. |
| MR registry | Kernel | **C** | RCU + crypto. <200ns rkey lookup. |
| Contra-rotation scheduler | Kernel | **C** | kthread + thermal. Straightforward. |
| RDMA attestation | Kernel | **C** | Crypto + TPM + SPDM. No Rust bindings. |
| XKABI capabilities | Kernel | **C** | IDR + HMAC + audit. Core security path. |
| Mesh daemon | Kernel -> **Userspace** | **C** (or **Rust** if policy amended) | Move to userspace. Split: thin kernel exporter + userspace daemon. |
| QP relay gateway | Kernel | **C** | <800ns budget. RCU + RDMA. |
| Entropy classifier | Kernel | **C** | 104 LOC pure math. No reason to change. |
| XISC shim (14 files) | Userspace | **C** (or **Rust** if policy amended) | Rust is technically optimal for handle/rights logic. C is acceptable with hardening. |
| Linux personality | Userspace | **C** | Syscall dispatch table. Trivially correct. |
| Conformance tests | Userspace | **C** | Tests the C shim. Must match. |

### 6.2 Policy Implications

**If the language policy ("Python, TypeScript, C ONLY") remains unchanged:**

All kernel and system components stay in C. Apply the following hardening:
- AddressSanitizer + ThreadSanitizer in CI
- CBMC model checking for the attestation and capability layers
- `cppcheck` + `clang-tidy` (already in use per MEMORY.md)
- Property-based fuzz testing (libFuzzer/AFL++) for capability validation paths
- Formal precondition/postcondition comments (manual SPARK-style contracts in C comments)

**If the policy is amended to allow Rust:**

Phase 1 (immediate): Rewrite XISC shim in Rust (~2-4 weeks)
Phase 2 (6 months): Move mesh daemon to userspace Rust daemon (~3-4 weeks)
Phase 3 (18-24 months): Evaluate Rust-for-Linux for NEW kernel components only (not rewriting existing C)

**Under no scenario should you:**
- Adopt Zig for production code before 1.0 ships
- Attempt Ada/SPARK for kernel-resident code
- Use C++ anywhere in the project
- Rewrite existing, working kernel C into Rust (the binding cost exceeds the safety benefit for the current ~2,800 LOC)

### 6.3 The Counter-Intuitive Conclusion

The GEN.OS kernel C code is actually quite good. The page fault handler, tier migration, and attestation code show:
- Consistent error handling (`unlikely()`, `goto out_*` patterns)
- Proper locking discipline (spinlocks in correct order, documented lock warnings)
- Explicit stub documentation for unimplemented production paths
- Latency tracking and enforcement
- Zeroization of cryptographic material (`memzero_explicit`)
- RCU-correct access patterns in the QP relay and MR registry

Rewriting this code in Rust would not eliminate bugs -- it would introduce new bugs during the rewrite while providing safety guarantees only for memory management patterns that the current code already handles correctly. The ROI is negative for the kernel components.

The real value of Rust is in the XISC shim, where the handle table lifecycle, rights enforcement, and audit trail logic are the exact patterns where Rust's ownership model provides compile-time guarantees that C cannot.

## 7. Compliance and Integrity Notes

- **POSIX conformance:** The XISC shim is POSIX-compliant. A Rust rewrite must maintain C ABI exports for POSIX compatibility.
- **Reproducible builds:** Any language addition must integrate with the existing ISO build pipeline (debootstrap + Docker). Rust toolchains are reproducible via `rust-toolchain.toml`.
- **Supply chain:** Rust crate dependencies must be vendored and audited (`cargo-vet`). Zero network fetches during build.
- **Capability escalation:** No language change affects the XKABI capability model. The rights-narrowing invariant is an application-level property independent of implementation language.
