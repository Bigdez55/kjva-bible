## SPRINT 1: Compiler Hardening + JIT Performance Fix (Week 1)

**Goal:** Harden all C compilation with modern compiler security flags. Fix the O(n) performance bottleneck in `xisc_jit_get_fn()`. Add kernel sanitizers to CI.

### S1-T1: Compiler Hardening Flags -- Makefile.rfc002
- **Owner:** apex-systems-architect (primary), reliability-security-sentinel (review)
- **Input:** Current `Makefile.rfc002` line 9-10:
  ```
  CFLAGS := -O2 -Wall -Wextra -Wpedantic -std=gnu11 ...
  ```
- **Deliverable:** Updated CFLAGS with:
  ```
  -fstack-protector-strong -D_FORTIFY_SOURCE=2 -fPIE -Wformat=2 -Wformat-security
  -Wimplicit-fallthrough -fno-strict-aliasing
  ```
  Updated LDFLAGS with:
  ```
  -pie -Wl,-z,relro -Wl,-z,now
  ```
- **Acceptance Criteria:**
  - All 80 C tests still pass with new flags (zero regressions)
  - `make -f Makefile.rfc002 run-tests` succeeds on ubuntu-22.04, ubuntu-24.04, macos-14
  - No new compiler warnings (existing `-Wextra -Wpedantic` already strict)
- **Risk:** `-D_FORTIFY_SOURCE=2` can cause false positives with certain `memcpy`/`snprintf` patterns. The `snprintf` in `jit_compile_block()` (line 182 of xisc_llvm_jit.c) is safe (bounded by sizeof), but must be verified.
- **Risk:** `-fstack-protector-strong` adds ~2-5% overhead on function calls. Acceptable for validation vehicle; may need tuning for hot paths in production.
- **Rollback:** Revert CFLAGS/LDFLAGS lines in `Makefile.rfc002`. Single-line change.

### S1-T4: Fix O(n) Linear Scan in xisc_jit_get_fn()
- **Owner:** the-architect (primary), system-signal-engine (performance validation)
- **Input:** `genvirt/jit/xisc_llvm_jit.c` lines 283-326 -- the `xisc_jit_get_fn()` function
- **Problem:** Cache lookup is O(n) linear scan over `cache_entries` (up to 64K entries). At 64K entries, worst-case lookup is ~64K iterations under mutex -- far exceeding the 50ns target.
- **Deliverable:** Replace flat array with hash map. Recommended approach:
  ```c
  /* Hash table: block_id → cache slot index
   * Open addressing with power-of-2 table size.
   * Hash function: FNV-1a on block_id (u64). */
  #define JIT_HASH_BUCKETS  (JIT_CODE_CACHE_ENTRIES * 2)  /* 2x load factor */

  struct jit_cache_entry {
      u64   block_id;
      void *native_fn;
      u64   hit_count;
      struct vdram_jit_hint hint;
  };

  /* In struct xisc_jit: */
  u32 hash_table[JIT_HASH_BUCKETS];   /* maps hash → cache index (+1, 0=empty) */
  struct jit_cache_entry cache[JIT_CODE_CACHE_ENTRIES];
  ```
  Lookup becomes O(1) average case.
- **Acceptance Criteria:**
  - `xisc_jit_get_fn()` cache-hit path is O(1) (verified by code inspection)
  - All existing tests that reference JIT functionality still pass
  - No behavior change to callers (same function signature, same semantics)
  - `xisc_jit_cache_hit_rate()` still works correctly
- **Risk:** Hash collision handling adds code complexity. Open addressing with linear probing is sufficient for 50% load factor.
- **Risk:** This file does NOT build in CI (excluded from Makefile.rfc002 -- see line 7: "xisc_llvm_jit.c is explicitly excluded -- it requires LLVM headers"). Testing is manual or deferred to Phase 9 hardware validation. Compile-check with `gcc -fsyntax-only` if LLVM headers are stubbed.
- **Rollback:** Revert changes to xisc_llvm_jit.c and xisc_llvm_jit.h. File is excluded from CI so rollback has zero CI impact.

## SPRINT 2: Formal Verification + Coccinelle CI (Weeks 2-3)

**Goal:** Apply bounded model checking to the two most critical safety invariants. Add Coccinelle semantic patches to enforce C coding patterns in CI.

### S2-T1: CBMC Harness for xkabi_cap_derive() Rights-Narrowing Invariant
- **Owner:** reliability-security-sentinel (primary), the-architect (domain review)
- **Input:** `kernel/linux-validation/security/xkabi_capabilities.c` lines 249-272
- **The Invariant:** `xkabi_cap_derive()` must NEVER produce a child capability with rights that are not a strict subset of the parent's rights. Formally:
  ```
  ASSERT: (child_rights & ~parent_rights) == 0
  ASSERT: if !(parent_rights & XKABI_RIGHT_DERIVE) then return -EACCES
  ```
- **Deliverable:**
  1. New directory: `verification/cbmc/`
  2. Harness file: `verification/cbmc/xkabi_derive_harness.c`
  3. CI job: `cbmc-verification` in `.github/workflows/ci.yml`
  4. Harness structure:
     ```c
     void harness_xkabi_cap_derive(void) {
         u32 parent_rights = nondet_u32();
         u32 derived_rights = nondet_u32();
         // ... setup handle table, encode parent handle ...
         int ret = xkabi_cap_derive(ht, parent_handle, derived_rights, child_handle);
         if (ret == 0) {
             u32 child_rights;
             xkabi_verify_handle(ht, child_handle, &child_obj, &child_rights);
             __CPROVER_assert((child_rights & ~parent_rights) == 0,
                              "rights must only narrow");
         }
     }
     ```
- **Acceptance Criteria:**
  - CBMC proves the invariant holds for ALL possible 24-bit right combinations (2^24 = 16M states)
  - Verification completes in < 5 minutes on CI runner
  - Zero counterexamples
- **Risk:** CBMC requires stubs for kernel APIs (kzalloc, idr_alloc, crypto_shash_*). Building these stubs is the majority of the work. Estimate: 200-300 LOC of stubs.
- **Risk:** CBMC may not handle HMAC crypto operations well. Stub the HMAC as an identity function for verification purposes (we are verifying the rights logic, not the crypto).
- **Rollback:** Remove `verification/cbmc/` directory and CI job. Zero impact on existing code.

### S2-T2: CBMC Harness for pf_attestation_gate() Invariant
- **Owner:** reliability-security-sentinel (primary), guardian-sentinel (attestation domain)
- **Input:** `kernel/linux-validation/vdram/pf_handler.c` lines 75-126
- **The Invariant:** `pf_attestation_gate()` must:
  1. Return 0 (pass) ONLY when `ctx->attestation_passed` is set to true
  2. For T3 tier: always call either `rdma_attestation_verify()` or `mesh_identity_verify()`
  3. Never bypass attestation for T3/RDMA -- no code path reaches `return 0` without verification
- **Deliverable:**
  1. Harness file: `verification/cbmc/pf_attestation_harness.c`
  2. Stubs for `rdma_attestation_verify()` and `mesh_identity_verify()` with nondeterministic return
  3. Properties verified:
     ```
     ASSERT: (ret == 0) => (ctx.attestation_passed == true)
     ASSERT: (tier == T3 && rdma_enabled) => (attestation_fn_called == true)
     ```
- **Acceptance Criteria:**
  - All properties proved for all tier combinations (4 tiers x 2 enterprise modes x 2 rdma modes = 16 states)
  - Verification completes in < 2 minutes
  - Zero counterexamples
- **Risk:** The `WARNING` comment at lines 300-308 of pf_handler.c documents a known spinlock+sleeping violation in T3 slow path. CBMC will not catch this (it verifies logical properties, not locking protocols). Document this as a known limitation in the verification report.
- **Rollback:** Remove harness file. Zero impact.

## SPRINT 3: WAMR + eBPF Evaluation (Weeks 4-6)

**Goal:** Evaluate WAMR (WebAssembly Micro Runtime) as a safer replacement for LLVM JIT in kernel space. Evaluate eBPF for policy annotations and entropy classification. Produce Go/No-Go decision documents -- NOT production code.

### S3-T1: WAMR Feasibility Study + Prototype
- **Owner:** the-architect (primary), vanguard-innovation-scout (SOTA evaluation), system-signal-engine (performance)
- **Input:** `genvirt/jit/xisc_llvm_jit.c` (the component WAMR would replace)
- **Deliverable:**
  1. Document: `docs/evaluation/WAMR_FEASIBILITY.md`
  2. Prototype: `evaluation/wamr/xisc_wamr_poc.c` -- minimal POC that:
     - Compiles XISC ADD + RET opcodes to WASM bytecode
     - Executes via WAMR interpreter (AOT mode evaluation)
     - Measures latency vs. current LLVM JIT targets (50ns cache hit, 500us cold)
  3. Decision matrix covering:
     - Performance: Can WAMR AOT match LLVM -O2 throughput (85-95% native target)?
     - Safety: WAMR sandbox isolation vs. LLVM arbitrary code generation
     - Kernel compatibility: WAMR has a kernel-mode embedding (wasm-micro-runtime)
     - Maintenance: WAMR C API surface vs. LLVM-C API surface
     - Maturity: WAMR production deployments at scale
- **Acceptance Criteria:**
  - Feasibility document with quantitative performance comparison
  - Go/No-Go recommendation with justification
  - If Go: migration plan with estimated LOC and timeline
  - If No-Go: documented reasons and alternative recommendations
- **Risk:** WAMR AOT compilation may not support auto-vectorization (critical for MATMUL performance). This alone could be a No-Go.
- **Risk:** WAMR in-kernel embedding is experimental. May require kernel patches not available in Linux 5.15 LTS.
- **Rollback:** This is an evaluation -- no production code is modified. Remove `evaluation/wamr/` directory.

### S3-T2: eBPF Evaluation for Policy Annotations
- **Owner:** the-architect (primary), vanguard-disruptive-alchemist (alternative approaches)
- **Input:** Current policy annotation system in `jit_emit_policy_annotation()` (xisc_llvm_jit.c lines 109-164)
- **Deliverable:**
  1. Document: `docs/evaluation/EBPF_POLICY_FEASIBILITY.md`
  2. Prototype: `evaluation/ebpf/policy_annotation_poc.c` -- eBPF program that:
     - Attaches to a tracepoint or kprobe
     - Reads JIT block metadata from a BPF map
     - Emits tier placement decisions
  3. Decision matrix:
     - Can eBPF replace the current switch-case policy annotation?
     - Latency impact of eBPF verifier overhead
     - Can eBPF programs be dynamically loaded for new workload classes?
     - Safety: eBPF verifier guarantees vs. current manual C
- **Acceptance Criteria:**
  - Feasibility document with clear Go/No-Go
  - If Go: identify which specific functions migrate to eBPF and which stay in C
  - Performance numbers: eBPF overhead per annotation vs. current inline C
- **Risk:** eBPF programs have stack size limits (512 bytes). Complex policy logic may exceed this.
- **Risk:** eBPF requires Linux 5.15+ with specific config options. Must verify Debian 12 kernel supports required features.
- **Rollback:** Remove `evaluation/ebpf/` directory. No production code modified.
