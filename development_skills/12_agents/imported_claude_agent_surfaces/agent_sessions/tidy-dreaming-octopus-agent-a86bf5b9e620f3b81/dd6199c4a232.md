# GEN.OS Language Architecture Plan — Ground-Up OS
# The Architect — 2026-03-02
# Decision: Definitive Language Allocation for Novel Custom OS

## Summary Decision
**3-language core: C + Python + TypeScript. No new languages added.**

## Rationale Per Component

### Kernel Core (C)
- 4,000+ LOC already exists and is architecturally solid
- linux-validation/ proves algorithms in production-grade environment
- kernel/interfaces/ headers are the porting contracts — they are already C
- XKABI handle encoding (HMAC-SHA256, IDR, spinlock) maps perfectly to C
- vDRAM fault handler is performance-critical path; C gives deterministic latency
- Contra-rotation scheduler uses thermal_zone_get_temp() — replaceable with direct HW register reads in native
- No runtime, no GC pauses on hot paths, direct hardware access

### XISC JIT (C via LLVM-C API)
- xisc_llvm_jit.c already uses LLVM-C API — this is the correct choice
- LLVM-C is stable, well-maintained, and C-linkage compatible
- The JIT feeds policy annotations directly into vDRAM (same address space)
- Switching to C++ (inkwell) would require re-linking the entire kernel in C++
- LLVM-C API covers everything needed: MCJIT, pass managers, IR builder

### Networking Stack (C)
- genos_meshd.c is already C, with clear stub markers for production implementation
- Ed25519 identity: libsodium-style primitives are C
- mDNS + gossip protocol over UDP: raw socket API is C
- WireGuard replacement: custom encrypted tunnel in C, same model

### Driver Model (C)
- HP EliteBook x360 hardware docs are C-oriented
- ACPI, PCIe, I2C, USB — all Linux driver model APIs that native will replace
- The vdram_tier_desc already has nvme_path[], rdma_device[], gpu_hbm_base fields
- C gives direct MMIO access, no abstraction penalty

### Container / Workload Isolation (C + Python)
- XKABI IS the container boundary — capability handles replace cgroups/namespaces
- Python platform services (FastAPI) remain unchanged — they run ABOVE the capability boundary
- No new container runtime needed — XKABI handle tables per process IS the isolation model

### Platform Services (Python — unchanged)
- 9,100 LOC Python services are correct and stay
- FastAPI, Pydantic v2, SQLite — all platform-agnostic, run on any POSIX layer

### Desktop Shell / UI (TypeScript — unchanged)
- 16,000 LOC TypeScript (Electron, Orange Suite) — correct and stays
- Only the IPC channel changes (D-Bus → XKABI IPC ports at the kernel boundary)

## Novel Kernel Architecture Model

GEN.OS kernel is best described as a **Capability-Exokernel**:
- NOT microkernel: no IPC-over-message-passing for every syscall
- NOT monolithic: no ambient authority, no global kernel state accessible to processes
- NOT hybrid: XKABI is the ONLY authority mechanism — no fallback to uid/gid/DAC
- The kernel exposes raw hardware resources (vDRAM tiers, RDMA QPs, GPU rings)
  gated exclusively through XKABI capability handles
- Processes receive handles at creation; they narrow but never widen
- This IS novel: it is an exokernel (raw hardware multiplexing) + capability ABI

## 6-12 Month Feasibility Assessment

### Month 1-2: Boot stub
- Bootloader (GRUB/custom) → kernel entry in C
- Physical memory map, page tables, APIC init
- Serial console output
- This is achievable in C in 6-8 weeks for x86_64

### Month 2-4: XKABI + vDRAM native port
- Port xkabi_capabilities.c — replace Linux IDR with own hash table, spinlock with own primitive
- Port vdram fault handler — replace vm_ops->fault with own page table walker
- These are direct ports of validated C code against the kernel/interfaces/ contracts

### Month 4-6: Scheduler + Mesh
- Port contra_rotation.c — replace kthread with own thread primitive, thermal reads via ACPI EC
- Port genos_meshd.c — implement the stubs (mDNS, Ed25519, gossip)

### Month 6-9: XISC JIT native link
- Decouple xisc_llvm_jit.c from Linux slab/vmalloc → own allocator
- Link LLVM as a library (no kernel module constraints)
- W^X code cache maps directly

### Month 9-12: Platform services + UI reconnection
- Python services boot on top of the GEN.OS POSIX personality layer
- TypeScript Electron connects via XKABI IPC ports instead of D-Bus sockets
- HP EliteBook x360 drivers: ACPI EC, Intel GFX, NVMe, USB

## Final Language Table

| Component | Language | Justification |
|-----------|----------|---------------|
| Kernel core | C | 4K LOC exists, validated, XKABI/vDRAM/scheduler proven |
| XISC JIT | C (LLVM-C API) | Already implemented, correct linkage |
| Networking / mesh | C | genos_meshd.c stubs ready for production implementation |
| Driver model | C | x86_64 hardware access, MMIO, ACPI |
| Workload isolation | C (XKABI handles) | Handles ARE the container boundary |
| Platform services | Python | 9,100 LOC unchanged |
| Desktop shell / UI | TypeScript | 16,000 LOC unchanged |

**Minimum language count: 3 (C, Python, TypeScript)**
**No new languages required.**
