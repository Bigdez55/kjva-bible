# GEN.OS Kernel Architecture Blueprint: The Capability-Woven Execution Fabric (CWEF)

**Document Classification:** System Design Document -- Kernel Architecture
**Status:** PLAN -- Pending Review
**Target Hardware:** HP EliteBook x360 1040 G7/G8 (x86_64, Intel CPU/GPU, NVMe, Thunderbolt)
**Timeline:** 6-12 months to first bootable image
**Date:** 2026-03-02

---

## 0. Executive Summary

After analyzing the existing XISC (`xisc/include/xisc.h`), XKABI (`xisc/include/xkabi_*.h`), vDRAM (`kernel/linux-validation/vdram/vdram.h`), and Encrypted Mesh (`kernel/linux-validation/mesh/genos_meshd.h`) specifications in the repository, a novel kernel architecture emerges that is neither microkernel, nor monolithic, nor hybrid in the traditional sense.

The architecture is a **Capability-Woven Execution Fabric (CWEF)** -- a kernel where:

1. Every kernel object IS a capability (not "has" capabilities -- IS one)
2. The JIT IS the execution model (not an optimization -- the fundamental dispatch path)
3. Memory IS policy-annotated (tier placement is not an afterthought -- it is encoded in every allocation)
4. IPC IS the kernel ABI (there is no separate "kernel API" vs "IPC" -- they are the same thing)

The name "woven" is deliberate: capabilities, JIT annotations, memory tier hints, and energy policies are not layered on top of each other -- they are interlaced at every instruction boundary. A single XISC instruction simultaneously expresses computation, memory tier preference, energy budget, and capability scope. The kernel does not "check" these independently; it resolves them as a single woven decision at each dispatch point.

---

## 1. Why This Is Not Microkernel, Not Monolithic, Not Hybrid

### 1.1 What Microkernels Get Wrong For GEN.OS

Microkernels (L4, seL4, Zircon) move services to userspace and communicate via IPC. The kernel is minimal: scheduling, IPC, memory management. Everything else is a server.

The problem: XISC policy annotations need to flow from JIT compilation through to vDRAM tier placement and energy gating in a single fault-handling path. In a microkernel, this requires a page fault -> IPC to pager -> IPC to energy gater -> IPC to tier manager -> IPC back. That is 4 context switches minimum. At the vDRAM T0 fault target of 2us (from `vdram.h` line 217: `VDRAM_TARGET_T0_FAULT_NS 2000`), there is no latency budget for this.

### 1.2 What Monolithic Kernels Get Wrong For GEN.OS

Monolithic kernels (Linux, Windows NT core) put everything in ring 0. They achieve latency targets but sacrifice isolation. A buggy NVMe driver can corrupt the capability table. A malicious GPU driver can bypass XKABI rights checking.

The problem: XKABI's security model (`xkabi_capabilities.c`) demands that rights enforcement is non-bypassable. "No ambient authority. No root. No ring-0 bypass." A monolithic kernel where drivers share the kernel address space fundamentally cannot guarantee this.

### 1.3 What Hybrid Kernels Get Wrong For GEN.OS

Hybrid kernels (Windows NT, XNU) put some things in-kernel and some in userspace, chosen pragmatically. The boundary is ad-hoc.

The problem: The CWEF boundary is not ad-hoc. It is determined by a formal principle: **code runs inside the fabric if and only if it participates in the woven decision path (capability check + tier placement + energy gate + JIT annotation resolution)**. Everything else runs outside.

### 1.4 The CWEF Principle

The CWEF is defined by what is "woven" into the fabric:

| Woven (inside fabric) | Not woven (outside, in capability-isolated pods) |
|---|---|
| Capability enforcement (handle verification, rights checking, revocation) | Device drivers |
| vDRAM fault handler + tier placement | Filesystem implementations |
| XISC JIT compilation pipeline | Network protocol stacks |
| Energy gating + thermal management | User applications |
| Scheduler (context switch, EDF deadlines) | Mesh daemon |
| IPC dispatch (channels, ports, events) | Display compositor |
| Merkle accumulator | AI inference runtime |

The "fabric" is approximately 15,000-25,000 lines of C (with critical verification-worthy paths potentially in a restricted subset). Everything outside runs in "Driver Pods" -- isolated address spaces with XKABI capability handles as their only interface to the fabric.

This is structurally distinct from:
- **Microkernel**: A microkernel has a thin IPC layer and puts pagers/schedulers in userspace. CWEF keeps the vDRAM fault handler, scheduler, AND JIT inside the fabric because they must resolve together in <2us.
- **Monolithic**: A monolithic kernel puts drivers in ring 0. CWEF isolates all drivers in pods with IOMMU protection. A driver never touches a capability table directly.
- **Hybrid**: A hybrid draws the boundary pragmatically. CWEF draws it formally: woven = participates in the annotation-resolution pipeline.

### 2.1 The Woven Decision Point

The central innovation is the **Woven Decision Point** (WDP). When a page fault occurs, when a syscall is dispatched, or when a JIT-compiled block begins execution, the WDP resolves ALL of these simultaneously:

1. **Capability check**: Does the caller hold a valid, non-revoked handle with sufficient rights? (from `xkabi_capabilities.c`: HMAC-SHA256 verification + epoch check)
2. **Tier placement**: Which vDRAM tier should this data reside in? (from `vdram.h`: `vdram_jit_hint.preferred_tier`)
3. **Energy gate**: Does the current power budget permit this operation? (from `vdram.h`: `vdram_energy_gate()`)
4. **Merkle commitment**: Does this operation require a verifiable commit before proceeding? (from `vdram.h`: `merkle_commit_point`)
5. **Attestation**: Does this operation require SPDM peer attestation? (from `vdram.h`: `attestation_required`)
6. **Scheduling**: Should the current thread yield, or does its EDF deadline take priority?

In a traditional kernel, these are separate subsystems consulted sequentially. In CWEF, they are resolved as a single lookup into a **Policy Resolution Table** (PRT) that the JIT pre-computes and embeds in the code cache alongside the native code.

## 3. Language Allocation Per Subsystem

### 3.1 Constraint Acknowledgment

The existing Language Policy (`docs/LANGUAGE_POLICY.md`) mandates: **Python, TypeScript, C, (Java optional). No Rust. No Go.**

The user's question asks about Rust and Ada/SPARK specifically. I will address these questions honestly from a technical perspective, then provide the architecture within the language policy constraint, and finally present a reasoned recommendation for a policy exception where the technical case is overwhelming.

### 3.2 Language Allocation Table

| Subsystem | Language | Rationale |
|---|---|---|
| **Boot sequence** (UEFI handoff, GDT/IDT, page tables, early memory) | x86_64 Assembly + C | No alternative. UEFI handoff requires real-mode/long-mode transition code. GDT/IDT setup, CR3 manipulation, and initial page table construction are inherently assembly. C takes over once paging is active. ~2,000 lines ASM, ~3,000 lines C. |
| **Memory manager** (physical frame allocator, virtual address spaces, vDRAM tier management) | C | Hot path performance is critical. The vDRAM fault handler targets 2us for T0/T1. Frame allocator is a bitmap/buddy allocator -- straightforward C. VMAR tree operations are pointer-heavy. C is appropriate. ~5,000 lines. |
| **Scheduler** (process/thread management, context switching, contra-rotation) | C | Context switching requires inline assembly for register save/restore. The contra-rotation scheduler (`scheduler_ops.h`) is numerically sensitive (EDF deadline calculations). C with careful `volatile` and barrier usage. ~3,000 lines. |
| **IPC subsystem** (XKABI channels, ports, events, capabilities) | C | The channel fast path (64KB message copy + up to 64 handle transfers) must be extremely tight. The capability verification path (`xkabi_verify_handle` in `xkabi_capabilities.c`) uses HMAC-SHA256 with constant-time comparison -- this is crypto-sensitive C. ~4,000 lines. |
| **Driver model** (Driver Pod framework, IOMMU context management) | C (kernel-side interface) + C or Python (individual drivers) | The IOMMU context management (`xk_iommu_ctx_*` syscalls) is kernel-side C. Individual driver pods are isolated processes that could theoretically be written in any language that can make XKABI syscalls. For MVP, C drivers ported from Linux are most pragmatic. |
| **XISC JIT** (compilation pipeline) | C (already implemented in `genvirt/jit/xisc_llvm_jit.c`) | Uses LLVM C API directly. The existing implementation is production-quality C. The JIT itself generates native x86_64 machine code via LLVM. No language change needed. ~2,000 lines. |
| **Networking** (encrypted mesh, TCP/IP stack) | C (runs in a Driver Pod) | The mesh daemon (`genos_meshd.h`) and TCP/IP stack run as isolated pods. They use XKABI IPC to communicate with the fabric. C for performance. Could also be Python for the gossip/mDNS layer. ~8,000 lines for minimal TCP/IP. |
| **Storage** (filesystem, block layer) | C (filesystem pod) + C (NVMe driver pod) | The NVMe driver needs ring buffer IPC (`XK_DEVICE_TIER_FAST` from `xkabi_objects.h`). The filesystem is a separate pod. For MVP, a simple extent-based filesystem. ~6,000 lines. |
| **Security** (capability enforcement, attestation) | C | The core capability table (`xkabi_capabilities.c`) is already implemented in C. HMAC-SHA256 handle encoding, epoch-based revocation, and lineage tracking are all C. The attestation path uses Ed25519 signatures. ~3,000 lines. |

### 3.3 On the Rust Question

**How does Rust's ownership model map to XKABI's capability model?**

There is a deep structural alignment:

| XKABI Concept | Rust Concept | Mapping |
|---|---|---|
| `xk_handle_t` (owned, process-local) | `T` (owned value) | A handle is an owned resource. When a process exits, all its handles are closed -- like Rust's `Drop`. |
| `xk_handle_dup()` with rights narrowing | `&T` / `&mut T` (references with lifetime) | Duplicating a handle with narrower rights is analogous to creating a reference with fewer permissions. |
| Handle transfer via `xk_channel_write()` | `move` semantics | Sending a handle through a channel transfers ownership. The sender no longer holds it. This is Rust `move`. |
| `XK_RIGHTS_SUBSET(child, parent)` | Lifetime/borrow subtyping | A child capability's rights must be a subset of the parent's. This is subtyping -- a shorter lifetime is a subtype of a longer one. |
| Epoch-based revocation | Lifetime invalidation | When the revocation epoch advances, all handles from prior epochs become invalid. This is similar to lifetime expiration. |
| `XKABI_RIGHT_DERIVE` bit | `Clone` trait | Only handles with the DERIVE right can be duplicated. This is like only `Clone` types can be cloned. |

**The alignment is structural, not superficial.** Rust's type system could enforce at compile time what XKABI enforces at runtime: that rights never widen, that handles are not used after transfer, and that derivation depth does not exceed bounds.

**However**, the alignment is not a sufficient reason to adopt Rust. The XKABI runtime enforcement is mandatory regardless of implementation language (a compromised binary must still be contained). The compile-time guarantees would reduce bugs in kernel development but do not replace the runtime checks.

**Technical recommendation**: If the language policy were to evolve, Rust would be the strongest candidate for the IPC subsystem and capability enforcement layer specifically, because these are the subsystems where use-after-free and rights-widening bugs are most catastrophic. The rest of the kernel (boot, memory manager, scheduler, JIT) has less structural benefit from Rust.

**For this blueprint, per current policy, all kernel code is C + Assembly.**

### 3.4 On the Ada/SPARK Question

**Which subsystems should be formally verified?**

The ideal candidates for formal verification are:

1. **Capability rights checking** (`xkabi_verify_handle`, `xkabi_cap_validate`, `xkabi_cap_derive`) -- These are the security invariants. "Rights only narrow, never widen" is a formally provable property. The code is ~200 lines. SPARK could prove absence of runtime errors AND the narrowing invariant.

2. **Handle encoding/decoding** (`xkabi_encode_handle`, `xkabi_verify_handle`) -- The HMAC integrity path. SPARK could prove that every encoded handle passes verification, and that every tampered handle fails verification.

3. **Epoch-based revocation** -- Proving that `cap_generation < current_epoch => revoked` is monotonic and race-free.

4. **Rights bitmask operations** -- `XK_RIGHTS_SUBSET(child, parent)` and the derivation check `(derived_rights & ~parent_rights) == 0` are bitwise operations that are trivial to verify formally but catastrophic if wrong.

**Practical assessment**: SPARK for the capability enforcement layer is ~200-400 lines of code. It is practical. It would provide mathematical proof that the security invariants hold. The GNATprove toolchain can target x86_64 bare-metal.

**However**: Introducing Ada/SPARK adds a build toolchain dependency (GNAT compiler) and a language that very few developers know. For a team of 17 agents + human engineers working primarily in Python/TS/C, this is a significant friction cost.

**Compromise recommendation**: Write the capability enforcement in C with exhaustive property-based testing (fuzzing + model checking) rather than SPARK formal verification. If/when the security auditors demand formal proof, the capability layer is small enough to rewrite in SPARK or to model-check with CBMC (C Bounded Model Checker) without adding a new language.

## 4. Boot Sequence: UEFI to CWEF Fabric

### 4.1 x86_64 Boot Path

This is the minimum boot code needed for a non-Linux kernel to boot on the HP EliteBook x360 via UEFI.

```
UEFI Firmware
    |
    v
[Stage 0: UEFI Application] (C, compiled with gnu-efi or EDK2)
    - Call UEFI Boot Services to:
      * Get memory map (EFI_MEMORY_DESCRIPTOR array)
      * Locate GOP (Graphics Output Protocol) for framebuffer
      * Locate ACPI RSDP
      * Load kernel image from EFI System Partition (FAT32)
    - ExitBootServices() -- UEFI is now gone, we own the machine
    - Jump to Stage 1 entry point with:
      * Memory map pointer
      * Framebuffer base address + stride + resolution
      * ACPI RSDP pointer
    |
    v
[Stage 1: Long Mode Setup] (x86_64 Assembly, ~500 lines)
    - Already in 64-bit long mode (UEFI guarantees this)
    - Set up GDT (Global Descriptor Table) with kernel code/data segments
    - Set up IDT (Interrupt Descriptor Table) with early exception handlers
    - Set up initial page tables (identity map first 4GB + map kernel to high half)
    - Set up TSS (Task State Segment) for IST (Interrupt Stack Table)
    - Load CR3 with new page table root
    - Enable NX bit (IA32_EFER.NXE)
    - Jump to Stage 2 (C code)
    |
    v
[Stage 2: Early C Initialization] (C, ~2000 lines)
    - Parse UEFI memory map -> build physical frame bitmap
    - Initialize serial console (16550 UART at 0x3F8) for debug output
    - Initialize framebuffer console (write pixels directly to GOP framebuffer)
    - Parse ACPI tables: MADT (find all CPUs), MCFG (find PCIe config space)
    - Initialize local APIC + IO APIC
    - Initialize PIC (disable, route through APIC)
    - Set up HPET or LAPIC timer for preemptive scheduling tick
    - Initialize per-CPU data structures
    - Initialize physical frame allocator (buddy allocator)
    - Initialize virtual address space manager (VMAR tree)
    |
    v
[Stage 3: Fabric Initialization] (C, ~3000 lines)
    - Initialize capability store (HMAC key generation, IDR table)
    - Initialize IPC engine (channel pool, port queue, event objects)
    - Initialize scheduler (idle thread, timer tick handler)
    - Initialize XISC JIT (LLVM context, code cache)
    - Initialize vDRAM tier manager (detect available tiers from ACPI/probing)
    - Initialize Merkle accumulator
    - Initialize IOMMU (Intel VT-d: parse DMAR table, enable DMA remapping)
    |
    v
[Stage 4: First User Process] (C + XISC)
    - Create "init" process with root capability set
    - Create initial VMAR for init process
    - Load init program (driver manager) from initramfs on EFI partition
    - Start driver manager: it enumerates PCI devices and spawns Driver Pods
    |
    v
[Stage 5: Driver Pods Come Online]
    - NVMe driver pod: mounts root filesystem
    - Framebuffer driver pod: initializes display
    - USB/HID driver pod: keyboard + touchpad
    - Init loads shell or desktop environment
```

## 5. The Woven Decision Pipeline -- Detailed Design

### 5.1 Page Fault Path (The Critical Hot Path)

When a process accesses unmapped memory, the CPU generates a #PF exception. The CWEF handles it as follows:

```
#PF Exception (hardware)
    |
    v
[1. Fault Entry] (ASM: save registers, switch to kernel stack)
    |
    v
[2. Fault Address Resolution] (C)
    - Look up VMAR tree for fault address
    - Find owning VMO handle
    |
    v
[3. WOVEN DECISION POINT] (C, single function, ~100 lines)
    - INPUT: fault_ctx (from vdram.h struct vdram_fault_ctx)
    |
    +-- [3a. Capability Check]
    |   Is the faulting process's handle to this VMO valid?
    |   Does it have XK_RIGHT_READ/XK_RIGHT_WRITE/XK_RIGHT_MAP?
    |   Is the handle's generation >= global_revocation_epoch?
    |   COST: ~100ns (HMAC verify from cache) or ~500ns (cold)
    |
    +-- [3b. JIT Hint Lookup]
    |   Does the JIT code cache have an annotation for this address range?
    |   If yes: extract vdram_jit_hint (preferred_tier, energy_budget, etc.)
    |   If no: use default (T1 DRAM, 50W budget)
    |   COST: ~50ns (hash table lookup)
    |
    +-- [3c. Tier Selection]
    |   Given hint.preferred_tier AND current tier utilization:
    |   - If T0 requested and GPU HBM available: select T0
    |   - If T0 requested but full: demote to T1
    |   - If T2/T3: check energy gate first
    |   COST: ~20ns (atomic reads of tier_usage counters)
    |
    +-- [3d. Energy Gate]
    |   Is energy_nj + operation_cost_nj < power_cap_mw * elapsed_time?
    |   If no: shed to lower tier or defer
    |   COST: ~10ns (atomic read + compare)
    |
    +-- [3e. Attestation Gate] (only if hint.attestation_required)
    |   Is there a cached SPDM session for the peer?
    |   If yes: verify session token (~200ns cached target)
    |   If no: establish SPDM session (~2ms -- schedule migration async)
    |   COST: 200ns cached / 2ms cold
    |
    +-- [3f. Merkle Gate] (only if hint.merkle_commit_point)
    |   Compute SHA-256 of page content
    |   Update merkle_root via hash chain
    |   COST: ~1us for 4KB page
    |
    v
[4. Page Population]
    - Allocate physical frame from selected tier
    - Fill page (zero, load from NVMe, or RDMA from remote peer)
    - Map into process page table (update PTE)
    - Update vdram_page_meta
    |
    v
[5. Fault Exit] (ASM: restore registers, IRETQ)
```

**Total hot path latency budget**:
- T0/T1 (local memory, no attestation, no Merkle): ~200ns capability + ~50ns hint + ~20ns tier + ~10ns energy + page allocation = **target <2us** (matches `VDRAM_TARGET_T0_FAULT_NS`)
- T2 (NVMe): + NVMe read latency = **target <15us** (matches `VDRAM_TARGET_T2_FAULT_NS`)
- T3 (remote): + network round trip = **target <200us** (matches `VDRAM_TARGET_T3_FAULT_NS`)

### 5.2 Syscall Path

```
SYSCALL instruction (hardware: RIP -> LSTAR MSR, swap to kernel GS)
    |
    v
[1. Syscall Entry] (ASM: save user registers, load kernel stack from TSS)
    |
    v
[2. Syscall Dispatch] (C: switch on syscall number, 1-60 per xkabi_syscalls.h)
    |
    v
[3. Per-Syscall Handler]
    - EVERY handler begins with capability validation:
      xkabi_cap_validate(handle, required_rights)
    - Then performs the operation
    - Then returns xk_status_t
    |
    v
[4. Syscall Exit] (ASM: restore user registers, SYSRET)
```

### 5.3 Driver Pod Communication

Drivers run in isolated address spaces. They communicate with the fabric via three mechanisms, corresponding to the device tiers in `xkabi_objects.h`:

1. **XK_DEVICE_TIER_STANDARD** (<1ms overhead): Full XKABI channel IPC. Driver sends/receives `xk_channel_msg_t` messages. Used for slow devices (USB HID, Bluetooth).

2. **XK_DEVICE_TIER_FAST** (<50us overhead): Shared-memory ring buffer. The fabric maps a VMO into both the driver pod and the kernel with `XK_MEM_COHERENT` intent. Driver writes to ring, signals via port notification. Used for NVMe, network.

3. **XK_DEVICE_TIER_PASSTHROUGH** (<5us overhead): VFIO-style IOMMU passthrough. Device DMA goes directly to driver pod memory. Fabric only interposes on IOMMU map/unmap via `xk_iommu_ctx_*` syscalls. Used for GPU, Thunderbolt.

## 6. Subsystem Deep Dives

### 6.1 Capability Store

Based on the existing `xkabi_capabilities.c` implementation. For the native kernel, the key changes are:

1. **No Linux IDR**: Replace with a custom slab allocator + handle table using open-addressing hash map. Target: O(1) lookup in <50ns.
2. **No Linux crypto API**: Inline HMAC-SHA256 using a constant-time implementation. No kernel module loading.
3. **Per-CPU handle verification cache**: Cache recent (handle -> cap) lookups in per-CPU memory to avoid hash table contention. Cache invalidation on epoch advance.

**Handle format** (unchanged from `xkabi_capabilities.c`):
```
128-bit token:
  [0:2]   object_id lower 24 bits
  [3:5]   rights bitmask (24 bits: sw 0-11 + hw 12-22 + DERIVE 23)
  [6:7]   generation counter lower 16 bits
  [8:15]  HMAC-SHA256 truncated to 8 bytes
```

### 6.2 vDRAM Tier Manager

Based on the existing `vdram.h` specification. The native kernel implementation removes all Linux dependencies:

1. **Page fault handler**: Registered directly in the IDT exception handler chain (vector 14, #PF). Not a Linux `vm_ops->fault` callback.
2. **Physical frame allocator**: Buddy allocator with per-tier free lists. T0 frames are backed by GPU BAR memory (detected via PCI config space). T1 frames are conventional DRAM. T2 frames are NVMe-backed (demand paging from block device). T3 frames are RDMA-backed (fetched from mesh peers).
3. **Migration work queue**: Kernel work queue with dedicated migration threads. Async DMA for T0<->T1 transfers via GPU DMA engine. Async NVMe commands for T1<->T2. Async RDMA for T1<->T3.

### 6.3 XISC JIT

The existing `xisc_llvm_jit.c` is a Linux kernel module. For native kernel:

1. **LLVM in kernel**: This is the most architecturally significant decision. LLVM is ~30MB of code. It cannot run in the CWEF fabric. **Solution**: The JIT runs in a privileged pod with access to the code cache VMO. It compiles XISC bytecode to native code and writes the result to a W^X (write XOR execute) shared VMO. The fabric maps this VMO as RX (read-execute) for the target process.

2. **JIT-as-a-service**: Instead of the JIT being a function call inside the kernel, it becomes a **JIT Pod** that the fabric communicates with via fast ring-buffer IPC. The sequence:
   - Process loads XISC bytecode
   - Fabric sends bytecode to JIT Pod via ring buffer
   - JIT Pod compiles, emits native code + policy annotations
   - JIT Pod writes native code to shared W^X VMO
   - JIT Pod sends policy annotations to fabric via ring buffer
   - Fabric maps native code into process address space
   - Process executes native code directly

3. **Cache hit path**: Once compiled, the native code is in the code cache VMO. Subsequent executions do not involve the JIT Pod. The code cache is indexed by XISC block ID. Target: <50ns for cache hit (matching `xisc_llvm_jit.c` line 289).

4. **AOT (Ahead-of-Time) path**: On first boot, frequently-used XISC programs are AOT-compiled and stored on NVMe. Subsequent boots load from AOT cache. Target: >90% AOT hit rate after second run (matching `xisc_llvm_jit.c` line 22).

### 6.4 Scheduler

Based on `scheduler_ops.h` and `xkabi_objects.h` scheduling classes:

1. **Three scheduling classes** (from `xkabi_objects.h`):
   - `XK_SCHED_FAIR`: CFS-like proportional share. Default.
   - `XK_SCHED_DEADLINE`: EDF (Earliest Deadline First). For real-time threads.
   - `XK_SCHED_IDLE`: Only runs when no other threads are runnable.

2. **Contra-rotation integration** (from `scheduler_ops.h`):
   - Three phases: COMPUTE, COOLDOWN, LOAD
   - COMPUTE: GPU-heavy threads get priority, vDRAM prefetches T0
   - COOLDOWN: CPU-heavy threads get priority, vDRAM migrates T0->T1 to cool GPU
   - LOAD: I/O-heavy threads get priority, vDRAM prefetches T2->T1

3. **Per-CPU run queues**: Each CPU has a run queue with three sub-queues (deadline, fair, idle). The timer tick handler (LAPIC timer, 1ms period) checks deadline expirations and preempts.

4. **Context switch**: Save/restore x86_64 register state (GPRs, SSE/AVX via XSAVE, FS/GS base). Switch CR3 (address space). Switch kernel stack. ~500ns per context switch target.

### 6.5 IPC Engine

Based on `xkabi_ipc.h` and the syscall definitions:

1. **Channels**: Bidirectional, 64KB max per message, up to 64 handle transfers per message. Implementation: per-channel message queue backed by a ring buffer VMO. Fast path: `memcpy` + handle table manipulation. No data copies for zero-copy mode (share VMO directly).

2. **Ports**: Lightweight async notification. 16-byte notifications (key + payload). Implementation: lock-free MPSC (multi-producer, single-consumer) queue. Used for interrupt delivery from Driver Pods.

3. **Events**: User-controlled signal objects. 64-bit signal mask. Subsumes futexes. Implementation: wait queue with signal mask matching.

4. **`xk_wait_many()`**: Unified wait on multiple handles. Subsumes epoll, kqueue, WaitForMultipleObjects. Implementation: register wait items on per-handle wait queues, sleep until any fires, return all pending signals.

## 9. Risk Analysis

### 9.1 Critical Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| LLVM too large for JIT Pod | Medium | High -- no XISC execution | Use Cranelift (smaller, Rust -- policy exception needed) or interpreter fallback |

## 11. Compliance and Integrity

### 11.1 POSIX Conformance

GEN.OS native does NOT target POSIX directly. The XKABI is the native ABI. POSIX compatibility is provided via a personality layer that translates POSIX syscalls to XKABI syscalls (similar to Windows Subsystem for Linux, or Fuchsia's POSIX runner).

### 11.2 Capability Escalation Risks

- **No setuid**: XKABI has no ambient authority concept. There is no setuid.
- **No CAP_SYS_ADMIN equivalent**: Capabilities are per-handle, not per-process.
- **JIT Pod is privileged**: The JIT Pod has access to write executable code. This is a high-value target. Mitigation: JIT Pod runs with minimal capabilities (only code cache VMO write + ring buffer IPC). It cannot access other processes' memory. W^X is enforced -- the same page is never both writable and executable simultaneously.
- **IOMMU bypass**: If a Driver Pod can program the IOMMU, it can bypass isolation. Mitigation: IOMMU programming is a fabric-only operation. Driver Pods receive pre-configured IOMMU contexts.

### 11.3 Supply Chain Integrity

- All kernel code is built from source with reproducible build toolchain
- SHA-256 checksums of XISC programs verified before JIT compilation (from `xisc.h`: `xisc_program_t.checksum[32]`)
- Merkle root provides verifiable state of all vDRAM page contents
- Ed25519 signatures on mesh gossip messages prevent peer impersonation

## 12. Summary of Architectural Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Kernel architecture | CWEF (Capability-Woven Execution Fabric) | XISC+XKABI+vDRAM demand unified decision path that neither microkernel nor monolithic provides |
| Primary language | C + x86_64 ASM | Language policy constraint; C is adequate for kernel; ~530 lines ASM irreducible |
| XISC JIT location | Privileged Pod (not in-fabric) | LLVM is too large for the fabric; pod isolation prevents JIT bugs from compromising capability store |
| Driver isolation | IOMMU-isolated pods | Mandatory for XKABI "no ambient authority" guarantee |
| GPU driver (MVP) | Linux personality VM | i915 is 500K+ lines; not feasible in 12 months |
| WiFi driver (MVP) | USB Ethernet adapter | iwlwifi is enormous; personality VM provides WiFi |
| Filesystem (MVP) | Simple extent-based | Journaling deferred; FAT32 from EFI partition as fallback |
| First boot target | Month 2: serial console output on real hardware | Prove boot path works before building subsystems |
| SMP | Deferred to month 4-5 | Single-core first reduces early complexity |

## 13. Disclaimer

This is an architectural blueprint and reference implementation plan. Deploying OS-level changes carries risk of rendering the system unbootable or causing data loss. The 6-12 month timeline assumes full-time engineering effort. Regression test results on one device (HP EliteBook x360) do not guarantee behavior across all hardware configurations. Validate on target hardware with a known-good rollback image before shipping any system-critical change.

The CWEF architecture is novel. There is no prior art to reference for operational stability data. The woven decision pipeline is a theoretical design that must be validated empirically through the stress testing protocol (72-hour soak, 1000 boot cycles, cross-device regression).

## Appendix A: Glossary

| Term | Definition |
|---|---|
| CWEF | Capability-Woven Execution Fabric -- the GEN.OS kernel architecture |
| WDP | Woven Decision Point -- the single point where capability, tier, energy, attestation, and Merkle decisions are resolved simultaneously |
| PRT | Policy Resolution Table -- pre-computed decision table from JIT annotations |
| Driver Pod | Isolated process running a device driver with IOMMU protection |
| Personality VM | Virtual machine running a legacy OS (e.g., Linux) for device compatibility |
| Fabric | The trusted kernel core (capability store, vDRAM, scheduler, IPC, JIT interface) |
| Handle | 128-bit HMAC-protected capability token (process-local, unforgeable) |
| Tier | vDRAM memory tier (T0=GPU, T1=DRAM, T2=NVMe, T3=RDMA) |
| Contra-rotation | Thermal-aware scheduling phases (compute/cooldown/load) |

## Appendix B: Language Policy Exception Recommendation

If the language policy evolves in v2, the following exceptions would provide the highest value:

1. **Rust for `kernel/native/cap/`**: The capability enforcement layer (~2000 lines) would benefit most from Rust's ownership model. Compile-time enforcement of rights-narrowing invariant.

2. **Rust for `kernel/native/ipc/`**: Channel handle transfer is a `move` semantic. Rust prevents use-after-transfer at compile time.

3. **Everything else remains C**: Boot, scheduler, memory manager, JIT interface -- C is appropriate. The performance and control requirements align with C's strengths.

This is NOT a recommendation to adopt Rust now. It is a documented option for when the kernel stabilizes and the team evaluates language evolution.
