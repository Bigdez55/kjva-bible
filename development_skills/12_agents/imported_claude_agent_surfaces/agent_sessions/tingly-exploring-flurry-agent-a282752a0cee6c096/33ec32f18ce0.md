# AJV Version Mismatch: Systemic Remediation Plan

## Executive Summary

The frontend `node_modules` contains a **self-inflicted dependency resolution bug** caused by an overly broad `overrides` entry in `package.json`. The override `"schema-utils": { "ajv": "^6.14.0" }` (line 117-119) was intended to remediate Dependabot CVE #405 (ajv@6.12.6 ReDoS), but it has an unintended side effect: it forces **ALL** `schema-utils` instances -- including v4.3.3, which requires ajv@8+ -- to resolve ajv as `^6.14.0`.

This creates a fatal incompatibility: `schema-utils@4.3.3` calls `require("ajv").default`, but `ajv@6.14.0` exports `module.exports = Ajv` (no `.default` property). Only `ajv@8+` has `exports.default = Ajv`.

## Root Cause Analysis

**The override at `package.json` lines 117-119:**
```json
"schema-utils": {
  "ajv": "^6.14.0"
}
```

This was added to remediate Dependabot #405 (CVE-2025-69873, ReDoS in ajv@6.12.6 via `$data` option). The intent was to bump the *root* `schema-utils@2.7.1` dependency on ajv from `^6.12.4` (resolved to 6.12.6) up to `6.14.0`.

**The problem:** npm `overrides` with nested syntax `"schema-utils": { "ajv": "^6.14.0" }` applies to **ALL** `schema-utils` instances in the dependency tree, not just the root one. This includes the six copies of `schema-utils@4.3.3` which legitimately need `ajv@^8.9.0`.

## Agent Dispatch Plan

### Phase 1: The Architect -- Design the Correct Override Strategy

**Task:** Design a `package.json` overrides configuration that:
1. Keeps `ajv@6.12.6` bumped to `6.14.0` for packages that genuinely need ajv@6 (root schema-utils v2.7.1, file-loader, fork-ts-checker, @eslint/eslintrc, eslint)
2. Does NOT interfere with schema-utils v4.3.3 instances that need ajv@8.9.0+
3. Produces a clean lockfile where `npm ci` works correctly

**Proposed fix (needs Architect validation):**

Replace the blanket `"schema-utils"` override with targeted overrides for only the packages that use schema-utils v2.x/v3.x (which need ajv@6):

```json
"overrides": {
  // ... existing overrides ...
  "@eslint/eslintrc": { "ajv": "^6.14.0" },
  "eslint": { "ajv": "^6.14.0" },
  "file-loader": { "ajv": "^6.14.0" },
  "fork-ts-checker-webpack-plugin": { "ajv": "^6.14.0" }
  // REMOVE: "schema-utils": { "ajv": "^6.14.0" }
}
```

The root `schema-utils@2.7.1` already resolves via the `@eslint/eslintrc` and `eslint` overrides. But we need to verify the root schema-utils also gets bumped. Alternative: add `"schema-utils@2": { "ajv": "^6.14.0" }` if npm supports version-scoped overrides (it does not in overrides v1 -- Architect must verify).

## Why The Bug Was Silent

The build appears to work currently because:
1. webpack's schema validation is **lazy** -- it only calls `validate()` when a loader or plugin passes an invalid config. With valid configs, the broken ajv is never invoked.
2. react-scripts wraps webpack and pre-validates most configs internally.
3. The four "latent" locations happen to NOT have ajv@6 on disk (npm deduplication got lucky), so they resolve to the correct root ajv@8.

But this is a **time bomb**: any `npm ci`, any addition of a webpack loader with a schema error, or any change that triggers webpack's schema validation will produce a cryptic `TypeError: Ajv is not a constructor` that is extremely difficult to trace back to this override.
