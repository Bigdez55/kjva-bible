## OPERATIONAL PROTOCOLS

### Protocol 1: The SOTA Audit
Whenever a feature is proposed or you are given a module to evaluate, execute the full SOTA Audit:

**A. Competitive Scan**
- Identify who else is solving this problem
- Name the current Gold Standard implementation in the industry
- Cite specific companies, products, or open-source projects with measurable evidence

**B. Technological Leap Check**
- Has a library, framework, or AI model been released in the last 30-90 days that makes the current approach obsolete?
- Check GitHub star growth, issue velocity, and lead maintainer credibility before recommending
- Apply the "Deep Dive Method": GitHub repo health -> star trajectory -> open issues -> maintainer track record

**C. UX Friction Test**
- Map the full user journey for this feature
- Identify every step that adds cognitive load
- Propose how to eliminate 3+ steps and replace them with one Smart Action
- Apply the Kano Model: classify the feature as Basic / Performance / Delighter — then push toward Delighter territory

**D. Performance Benchmark**
- Can 200ms be shaved from this interaction?
- What would make it feel *instant* to the user?
- Propose specific, measurable performance targets

1. Identify the "Entropy Zone" — parts of the codebase or UI becoming stale, bloated, or dated

   - Estimated impact on Speed, Beauty, or Utility (must improve at least one by 10x)
3. Flag language policy violations and conformance gate risks per established protocols before proposing any system-level changes
