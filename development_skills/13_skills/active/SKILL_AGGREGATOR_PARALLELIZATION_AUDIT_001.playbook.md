# Aggregator-Parallelization Audit

## When to Invoke

You have an aggregator gate (a gate whose primary job is to source/invoke many sub-gates and tally results) that has become the wall-clock bottleneck of your CI or sprint sweep. Common names: `gold`, `self_host_full`, `release_prep`, `full_matrix`, `nightly`.

Triggers:
- "aggregator gate slow", "gold gate slow", "self_host_full slow"
- "parallelize gates", "fan-out gates", "sub-gate serial"
- "wall-clock optimization", "sweep too long", "CI timeout"
- Aggregator gate wall-clock > 60s OR >= 25% of total sweep wall-clock.

## The Rule

**Aggregator gates with >= 5 sub-gate invocations are parallelization candidates.** Audit each sub-gate's write surface; parallelize the write-disjoint set; serialize the write-conflicting remainder. Always per-job log files. Always per-job timeouts.

## Recipe (3-step audit)

### Step 1 - Identify aggregators
```bash
# Find gates with >= 5 `bash <something>.sh` invocations
for f in gates/*.sh; do
  n=$(grep -cE '^[[:space:]]*bash [^|]+\.sh' "$f")
  [ "$n" -ge 5 ] && echo "$n  $f"
done | sort -rn
```
Top hits are your aggregators.

### Step 2 - Classify sub-gates by write-disjointness
For each sub-gate, list what it writes:
- **tmpdir**: `/tmp/foo` vs `/tmp/bar` - disjoint if different paths.
- **build artifacts**: `build/x86/` vs `build/arm64/` - disjoint if different output dirs.
- **shared state**: any file under `superc-v1/build/`, `out/`, or a fixed-name lockfile - CONFLICTING.
- **stdout-tee'd file**: if multiple sub-gates tee to the same log path - CONFLICTING.

Output: two lists - `disjoint[]` and `conflicting[]`.

### Step 3 - Parallelize the disjoint set, serialize the rest
Template below.

## Parallel Template

```bash
#!/bin/bash
# Aggregator gate with bounded parallelism + per-job logs + per-job timeouts.

set -u
PARALLEL_LOG_DIR="${PARALLEL_LOG_DIR:-/tmp/agg-$$}"
mkdir -p "$PARALLEL_LOG_DIR"
PER_GATE_TIMEOUT="${PER_GATE_TIMEOUT:-90}"   # seconds; tune per sub-gate

run_sub() {
  local name="$1"; shift
  local log="$PARALLEL_LOG_DIR/$name.log"
  # kill_after: hard SIGKILL after timeout if SIGTERM ignored
  ( timeout --kill-after=10 "$PER_GATE_TIMEOUT" "$@" > "$log" 2>&1 )
  local rc=$?
  echo "$rc $name" >> "$PARALLEL_LOG_DIR/results.txt"
}

# Disjoint set - fan out
run_sub gate_a  bash gates/a.sh &
run_sub gate_b  bash gates/b.sh &
run_sub gate_c  bash gates/c.sh &
wait

# Conflicting set - must run sequentially (shared state)
run_sub gate_d_shared bash gates/d.sh
run_sub gate_e_shared bash gates/e.sh

# Tally - rc=$? discipline (see outer-harness-rc-discipline)
fails=$(awk '$1 != 0' "$PARALLEL_LOG_DIR/results.txt")
[ -z "$fails" ] && echo PASS || { echo "FAIL"; echo "$fails"; exit 1; }
```

## Critical Invariants

1. **Per-job log file** - never tee to a shared log; you will get interleaved garbage.
2. **Per-job timeout with `--kill-after`** - SIGTERM-ignoring children must die. See `gate-harness-process-isolation`.
3. **Bounded parallelism** - don't `&` all 20 sub-gates on a 4-core box. Use `xargs -P` or batch with `wait` every N.
4. **Tally by RC, not text** - see `outer-harness-rc-discipline`.
5. **Disjointness verified empirically** - actually run two suspected-disjoint gates in parallel 5 times; if any flake, they aren't disjoint.

## Anti-Patterns

- **Parallelize everything**: shared `build/` dir produces silent corruption (gate A reads gate B's half-written .o file). Always classify first.
- **`&` without `wait`**: backgrounded jobs may not have written results when the tally runs.
- **No per-job timeout**: one hung sub-gate hangs the whole aggregator, defeating the parallel win.
- **Optimistic disjointness**: "they SHOULD be disjoint" - verify with empirical parallel runs before shipping.
- **Removing serialization to hit a number**: if a serial step exists for a reason (write conflict, ordering invariant), parallelizing it will give you a faster wrong answer.

## Anchor Case

**SUPER C Wave 7**: top-3 aggregators (`gold`, `self_host_full`, one other) consumed **499s of 1376s total sweep wall-clock = 36%**. Wave 7B converted `self_host_full` to parallel template with `timeout --kill-after` per sub-gate and tiered per-job logs. Wall-clock: **180s timeout window -> 129s honest completion**. ~28% improvement, no flake increase, sub-gate failures still surface cleanly via tally.

## Verification Checklist

- [ ] Aggregator identified empirically (sub-gate count >= 5)
- [ ] Each sub-gate's write surface enumerated
- [ ] Disjoint vs conflicting split is explicit, not assumed
- [ ] Parallel template uses per-job log files
- [ ] Per-job `timeout --kill-after` set
- [ ] Conflicting sub-gates remain serialized
- [ ] Empirical 5x parallel run shows zero new flakes
- [ ] Wall-clock improvement measured and recorded
- [ ] Tally is RC-based (see `outer-harness-rc-discipline`)
