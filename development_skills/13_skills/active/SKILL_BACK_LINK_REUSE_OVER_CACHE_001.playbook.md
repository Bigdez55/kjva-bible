# Back-Link Reuse Over Cache - The Metadata Economy

**Version: v1.1** (2026-05-27)

**DIRECTIVE: If you can recompute it cheaply from an existing back-link, don't cache it.**

---

## Anchor episode

2026-05-27, SUPER C v31.5.I equivalence-fix. The bug: `out.pos = p` for
`out: &mut LexState` param silently no-ops because Pass 4.21 only stamps
Field nodes when base back-links to a StmtLet with `V72_ITY_STRUCT_MASK`.
Param bindings have `ITY_PTR` instead - Pass 4.21 rejects.

To stamp Field-on-param requires resolving the param's pointee struct
type name (e.g., `LexState` from `&mut LexState`). Two design paths:

### Path A - Cache in new struct fields (my first instinct)
- Add `param_pointee_ty_starts[V72_PARAM_CAP]` + `_ends[]` to V72FnEntry_t
- Mirror same fields in V72FnSig_t
- Init in param-init loop (2 new fields zeroed per param)
- Capture during `&`-prefix arm of fn-sig parser
- Mirror FnEntry -> FnSig at sig-construction site
- Read fields in Pass 4.21.c

**Cost:** ~6 separate edit sites across one file. 2324=256 bytes per V72FnEntry_t  128 entries = 32KB static. Mirror logic duplicated. Risk: forget to init or mirror -> silent zero spans -> silent miss.

### Path B - On-demand source scan via existing back-link (the user's approach)
- Pass 4.10 already back-links body Ident refs to their declaring node
- Pass 4.15/4.18 already stamps that declaring node's `int_value = ITY_PTR` for ref-mut params
- At Pass 4.21.c time, check `sl->int_value == V72_ITY_PTR`, then scan source bytes starting at `sl->byte_end` to find the type span - one helper fn `v72_param_pointee_type_span` does it

**Cost:** 1 new helper function. ~30 LOC at one site. Zero struct changes. Zero memory cost. Zero mirror logic.

---

## When B beats A

A design-axis choice. Ask in order:

1. **Is the data derivable from existing back-links?**
   If yes (Pass M already produced the link, Pass N can walk it), consider B.
   If no (would need raw source re-scan from scratch), A wins.

2. **Is the recomputation cheap?**
   Cheap = O(byte_span_length), no allocation, no re-tokenization. The on-demand source scan in Path B is just a forward-byte-iter, which is the same cost as the original parse-time scan for that span.
   If expensive (re-parse a full file, re-allocate big structures), cache.

3. **Will the data shape change?**
   Caching freezes the data shape into a struct definition. Future evolution requires modifying every consumer. On-demand recomputation lets callers ask for what they need without ABI churn.

4. **Risk of stale or missing cache?**
   Every cached field is a potential silent-miss if not populated correctly. B has no such failure mode - if back-link is wrong, recomputation surfaces the failure visibly.

5. **Is the producer pass under your control?**
   If you'd have to modify code OTHERS rely on (FnEntry/FnSig consumers across the codebase), B has lower blast radius.

## Anti-patterns this skill catches

| Anti-pattern | Why it's worse than back-link reuse |
|---|---|
| "Add `param_pointee_ty_*` arrays to V72FnEntry_t" | 2 struct edits + init + capture + mirror + consumer (5 sites) vs 1 helper (1 site) |
| "Mirror FnEntry to FnSig for X" | The mirror itself is a maintenance liability - every new field needs the mirror |
| "Cache type name on every binding so lookups are O(1)" | Type-name lookup is already O(N) byte-compare in `v72_struct_field_lookup`; caching saves nothing |
| "Just add a side table for X" | Side tables decouple from primary data; primary data already has the link. Use it. |
| "Stash N copies of the same span" | Spans are byte-positions in immutable source; never need copying, just re-derive |

## The pattern, generalized

For any multi-pass pipeline (parser -> sema -> lower -> codegen, or ETL Bronze -> Silver -> Gold):

```
Pass M produces: links from B-data back to A-data, plus a type marker on A
Pass N needs:    a property of A that's derivable from raw source

WRONG:  Pass M adds explicit `A.cached_property` field. Init + capture + consume.
RIGHT:  Pass N does `if A.type_marker matches: derive_property(source, A.span)`.
```

The "type marker" is the cheap-test gate that avoids needless re-derivation. In SUPER C, that gate is `sl->int_value == V72_ITY_PTR`. In ETL, it might be a column-type sentinel. In a render pipeline, it might be a node-kind flag.

---

## Apply checklist

Before adding a struct field for cross-pass metadata:

- [ ] Identify the existing back-link from Pass N's data to Pass M's data
- [ ] Identify the existing type-marker/gate that distinguishes the cases where the new property applies
- [ ] Estimate recomputation cost: bytes scanned, allocations made
- [ ] If cost is O(source_span_length) and no allocation -> use back-link reuse
- [ ] If cost is super-linear or requires re-parsing -> cache makes sense
- [ ] If no back-link exists yet -> adding the back-link itself might be cheaper than the cache

## 3 Symmetric Read/Write Pattern

When back-link reuse is applied to field lowering (or any pipeline stage
that has both a READ pass and a WRITE pass for the same data), both sides
of the read/write pair MUST consume the same back-link and stamp pair.
Implementing READ via back-link but leaving WRITE with a separate,
potentially stale cache path - or vice versa - creates an asymmetry that
constitutes its own bug class.

**The v31.5.H failure mode (asymmetric field access):**

- `lower_subscript_field_load` (READ side, Pass 4.21.b) was implemented
  via on-demand source scan using the Pass 4.7 back-link + ITY_PTR stamp.
  It scanned `sl->int_value == V72_ITY_PTR` and derived the pointee type
  span on demand.
- `lower_subscript_field_store` (WRITE side, Pass 4.21.c) initially had
  no parallel implementation - it did not consume the same back-link/stamp
  pair. Field writes through a `&mut [T]` param silently dropped.

Only when Pass 4.21.c was authored with the SAME recipe (on-demand source
scan via Pass 4.7 back-link, gated by the same ITY_PTR marker) did both
READ and WRITE produce consistent, correct IR.

**Symmetry rule:**

For any construct X that can be both read and written (struct field, slice
element, array cell, deref of ref-mut param), the lowering pass for READ
and the lowering pass for WRITE MUST share the same metadata derivation
strategy. If READ uses on-demand back-link reuse, WRITE must also. If READ
uses a cached table, WRITE must also. Asymmetry is a latent bug: whichever
side is "different" will diverge silently when the metadata changes.

**Checklist addition for the Apply Checklist ( above):**

- [ ] If this back-link reuse applies to a READ path, identify the paired
  WRITE path and verify it uses the same derivation strategy.
- [ ] If READ is implemented first, stub the WRITE path immediately (even
  as a `// TODO: implement symmetric with READ`) so the asymmetry is
  visible at code review rather than discovered at runtime.

**Citation.** Pass 4.21.b (ref-mut single-struct READ, `lower_field_read`
helper) + Pass 4.21.c (slice-elem WRITE, `lower_subscript_field_store`
helper) from v31.5.H. Both used on-demand source scan via Pass 4.7 back-link.
Seal anchor `de656486` + preceding `b0b14ac6`.
