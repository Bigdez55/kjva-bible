# Carve-Out -> DEF-ID Discipline v1.0.0

## When to invoke

ANY time you write a seal report, milestone close, sprint retro, or audit
report that contains language like:

- "Deferred to follow-up"
- "Out of scope for this slice"
- "Tracked but not blocking"
- "Honest carve-out"
- "Acceptable XFAIL"
- "Known limitation"

If a carve-out exists without a tracked DEF-* ID, the seal is structurally
dishonest. Future sessions cannot recover the deferred work. This is the
silent-freeze anti-pattern.

## The Discipline

Every carve-out in a closure document MUST carry three artifacts:

### 1. DEF-* Identifier
Format: `DEF-<context>-<topic>-<NNN>` (e.g. `DEF-v22-NATIVE-SC-SCLD-PORT-001`).
- Unique across the project's defect / deferral ledger.
- Recorded in a project-canonical ledger file (e.g. `docs/defects.md`,
  `near_term.md`, `GB-*.md`, project-specific).
- Referenced by the seal report's carve-out section.

### 2. Detection Signal
The empirical observable that will fire when the deferred work is reached:
- A failing gate (currently XFAIL or SKIP, will flip to PASS when closed).
- A test case with explicit `@expected_fail` annotation referencing the DEF-ID.
- A runtime probe with documented failure mode and recovery path.
- A static analysis rule that warns until the carve-out is closed.

"Detection by future-vigilance" is NOT a detection signal. If no automation
fires, the carve-out is unfindable.

### 3. Recovery Class
One of three explicit classifications:

- **NEW-WORK** - the deferred item requires authoring new code/docs/tests.
  Estimated scope in LOC or agent-hours.
- **FIX-EXISTING** - known bug in shipped code; reproduction steps captured.
  Hypothesis for root cause optional but encouraged.
- **WONTFIX-RATIONALE** - explicit decision to leave the gap permanently;
  document the trade-off and the alternative path users should take.

A carve-out without a recovery class is a wish, not a deferral.

## The 3-Step Protocol

Before sealing any document containing a carve-out:

**Step 1 - Inventory.** Grep the draft for carve-out language. List every
hit. Number them.

**Step 2 - Assign or Reject.** For each carve-out:
- Assign a DEF-* ID (allocate new or reuse existing).
- Specify the detection signal (gate name, test ID, probe).
- Tag the recovery class.
- If you cannot do all three, the carve-out is not ready to ship. Either
  resolve it in-scope or remove the language and ship a smaller seal.

**Step 3 - Index.** Append/update the project defect ledger so the DEF-ID
is discoverable by future sessions. The seal report should reference the
ledger, not duplicate the entries.

## Anchor Episode (Reference Pattern)

**v31.5.H super-seal closure (SUPER C / GEN.OS, 2026-05-26):** Six
carve-outs shipped, all with DEF-* IDs, gate signals, and recovery classes.
The Wave 4 main-thread closure was reproducible because each deferred item
had a unique handle.

**Counter-example (anti-pattern):** Pre-v31.4 seal reports contained
"deferred to v31.5" language without IDs. When v31.5 opened, two of those
items were rediscovered only because an agent re-read the prior report -
not because automation flagged them. One was missed for three cycles.

```
| 2 | DEF-<proj>-<topic>-002        | <one-line>            | test_<id> SKIP       | FIX-EXISTING (repro) |

All DEF-IDs registered in <project ledger path>.
```

## Anti-Patterns (forbidden)

- "We'll track this in our heads" - no.
- "It's obvious from context" - no; future sessions don't share context.
- A bare XFAIL annotation without a referencing DEF-ID comment - silent debt.
- DEF-ID assigned but no detection signal - undiscoverable.
- DEF-ID assigned but no recovery class - un-actionable.
