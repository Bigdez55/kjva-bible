# Cascade-Root Triage

## When to Invoke

You are looking at a sweep that reports N>=3 failing gates. The instinct is to open N tickets and grind. **DON'T.** Most gate cascades are M-rooted where M << N, because gates share probes, harnesses, or codegen paths. Fix the M roots and the (N-M) leaves close for free.

Triggers:
- "many gates failing", "all gates red", "cascade", "transitive failure"
- "root failure", "fix-one-close-many", "first what fails"
- Sweep output showing N>=3 FAIL with overlapping probe names, opcodes, or fixture paths.

## The Rule

**Before opening N tickets, build the dependency graph and find the roots.** A root is a failing gate with no upstream dependency in the failing set. A leaf is a failing gate that depends on another failing gate.

Fix roots first. Re-sweep. Watch leaves close without touching them.

## Recipe (4 steps)

### 1. Enumerate the failing set
```
grep -E '^FAIL|^\[FAIL\]|RC=[1-9]' sweep.log | awk '{print $NF}' > failing.txt
```

### 2. Extract each gate's probe surface
For each failing gate, extract: which `.sh` it invokes, which probe binary/fixture it executes, which source files it tests. Store as `gate -> {probes, sources, sub-gates}`.

### 3. Build the directed graph + topo-sort
Edge `A -> B` exists when gate A invokes gate B's probe OR tests source that B also tests. Nodes with no incoming edges from the failing set are **ROOTS**. Topologically sort: roots first, leaves last.

### 4. Fix only roots; re-run full sweep; measure cascade-closure ratio
Cascade-closure ratio = `(N_before - N_after) / fixes_applied`. Target: >= 2.0 (one fix closes >=2 gates). If ratio < 1.5, your root identification is wrong - re-graph.

## Anti-Patterns

- **Open N tickets without graphing.** You will fix the same root M times under M different tags and discover the duplication only when one fix mysteriously closes others.
- **Fix leaves first because "they're simpler".** Leaves often have NO bug - they fail because the root probe miscompiled the shared fixture. Fixing a leaf without the root means re-introducing the leaf-side defect when you finally fix the root.
- **Trust the test name.** A gate named `test_sha256` may be failing because the JIT SIGSEGVs on a 224-byte body, not because SHA-256 logic is wrong. The name lies; the stack trace tells truth.
- **Assume independence.** Gates that look orthogonal often share a single SCIR-lowering helper, a single bash harness, a single tmpdir collision. Always check.

## Anchor Cases

- **SUPER C Wave 7 intrinsic cascade**: 7 gates failing, ALL rooted in ONE probe `d4_c2_store_u32 / store_load_same_fn_via_main`. Closing that probe closed 7 gates with 1 fix (ratio = 7.0).
- **SUPER C Wave 7 validator cascade**: 6 gates failing, ALL rooted in ONE SHA-256 JIT path SIGSEGV at body size 224B. One codegen fix closed 6 gates (ratio = 6.0).
- **General**: a sweep of 26 FAILs reduced to 4 roots after graphing. 22 leaves closed for free.

## Quick Decision Tree

```
N >= 3 gates failing?
  -> Yes: BUILD GRAPH. Do NOT open N tickets.
     -> Roots identified? Fix root #1. Re-sweep.
        -> Ratio >= 2.0? Continue with next root.
        -> Ratio < 1.5? Re-graph; root identification was wrong.
  -> No (N<=2): Fix directly; graphing overhead not justified.
```

## Verification Checklist (before declaring cascade closed)

- [ ] Graph drawn and saved (even as ASCII tree in audit notes)
- [ ] Each root explicitly named with file:line
- [ ] Cascade-closure ratio measured and >= 2.0 OR documented why not
- [ ] Re-sweep run AFTER each root fix (not just at end)
- [ ] No leaf-only fix shipped without its root being closed first
