**Summary.** Empirical diagnostic for silent-drop / miscompile bugs in two-stage compiler bootstraps (seedc -> scc, gcc -> llvm, host -> cross). When a compiler emits much-smaller object files than a reference compiler for the same source, internal helper functions are likely being silently truncated. Use this skill to per-function-localize the emit-size loss BEFORE diving into source-line debugging. Triggers on: "silent drop", "compiler emit-size diff", "scc-built vs seedc-built size mismatch", "scc2 SIGSEGV without sema error", "bootstrap binary smaller than canonical", "scc emit-direct output truncated", or any case where a bootstrap target compiler produces a noticeably smaller .o or binary than the reference toolchain.

# Compile-Output Size-Diff Probe

**Version: v1.1** (2026-05-27)

**DIRECTIVE: When a bootstrap compiler emits less code than the reference for the same source, run a per-function size diff to narrow the failure before touching source.**

---

## Anchor episode

2026-05-27, SUPER C v31.5.I.field-ptr-fix -> next-layer parser miscompile.
After fixing the ref-mut field-write bug (scc1_scc2_chain link strict
GREEN), `scc2 check minimal.sc` still SIGSEGVed with `arena_len=0`. The
question: where is the next miscompile?

**Probe applied** (~30 lines of Python + nm):

```python
def parse(path):
    import subprocess
    lines = subprocess.check_output(['nm', path]).decode().splitlines()
    syms = []
    for l in lines:
        parts = l.split()
        if len(parts) >= 3 and parts[1] == 'T':
            try: syms.append((int(parts[0], 16), parts[2]))
            except: pass
    syms.sort()
    out = {}
    for i, (a, n) in enumerate(syms):
        nxt = syms[i+1][0] if i+1 < len(syms) else a + 5000
        out[n] = nxt - a
    return out

seedc = parse('compiler/scc/build/parser.o')
scc   = parse('/tmp/parser_scc.o')
both  = set(seedc) & set(scc)
diffs = [(seedc[n] - scc[n], n, seedc[n], scc[n])
         for n in both
         if seedc[n] > 200 and scc[n] < seedc[n] * 0.3]
diffs.sort(reverse=True)
for d, n, s, c in diffs[:15]:
    print(f'  -{d:6d}  {n[:60]:60s} ({s} -> {c})')
```

**Output** (parser.o, 21% emit ratio, scc-emit-direct vs seedc):
```
  - 22628  _parse_stmt                 (29192 -> 6564)   77% loss
  - 19084  _parse_expr_bp              (25148 -> 6064)   76% loss
  - 11752  _parse_item_fn              (15780 -> 4028)   74% loss
  -  2020  _ast_write_assign           (2160 -> 140)     94% loss
  -  2020  _ast_write_addr_of          (2160 -> 140)     94% loss
  ... (all ast_write_* writers same ratio)
```

The 94%-loss `ast_write_*` cluster instantly named the next bug class:
they all share `arena[i as usize].field = value` pattern (slice-of-struct
indexed write through a `&mut [T]` parameter). Same bug CLASS as the
ref-mut-single-struct fix just landed, different specific shape.

Without this probe, the team would have spent hours guessing or doing
source-line bisection. The probe took 60 seconds.

## When to invoke

- Bootstrap compiler emits objects << reference compiler size
- Self-compiled binary SIGSEGVs while reference-compiled equivalent works
- Compiler test suite shows "X passed, Y failed" with no clear pattern
- A previous miscompile fix succeeded but a sibling layer still fails
- You're tempted to broadly TU-split or rewrite without a target

## When NOT to invoke

- The compiler errors clearly (E_ARG_TYPE, parse error with line:col) - fix that first
- The size diff is < 20% (within expected codegen-strategy variance)
- The compilers use different ABIs or optimization levels (apples-to-oranges)

---

## Output rubric

A useful probe output names:
- **Total emit ratio** (sum_scc / sum_seedc) - global health signal
- **Top-10 fn-level losses by bytes**, expressed as (loss, name, seedc_bytes, scc_bytes)
- **Cluster pattern** - do all losing fns share a structural shape? (e.g., all
  take `&mut [T]`, all use `out.field = ...`, all do `if ... { return }`)

The CLUSTER is the diagnostic gold. A single big-loss fn might be a
one-off; a CLUSTER points at a code pattern the compiler systematically
miscompiles.

---

## Companion probes (after size diff)

1. **Source-line bisect** of the largest-loss fn - find which line crosses the seedc threshold
2. **nm symbol-table diff** - confirm pub-exports match (otherwise it's silent-drop, not slow-drop)
3. **`otool -tvV` on one losing fn** - see what the truncated emit actually does (often: just prologue + epilogue, no body)
4. **`otool -l` __text section size** - confirm no 64K boundary corruption (separate failure mode)

---

## Anti-patterns this skill catches

| Anti-pattern | Why size-diff probe beats it |
|---|---|
| "Trust the test suite - if it passes, the compiler is fine" | scc_smoke 127/0 ran concurrent with parser.o losing 79% of code |
| "Compare end-to-end behavior of both compilers" | E2E is slow, false-positives easy; per-fn diff is 60s and surgical |
| "TU-split everything proactively" | Without the cluster signal, you might split files that aren't the culprit |
| "Look at the test that fails first" | The failing test may exercise the symptom 10 layers downstream of the cause |
| "Run with verbose / debug output" | Compiler verbose dumps are noisy; size diff is one number per function |

## 6 Reverse Witness Pattern

After a fix lands, the size-diff probe is NOT done. Re-running the probe
serves as the definitive FULLY-RECOVERED witness. The probe's output at
that point must show:

1. **Byte-identity at the `.o` level** - `cmp -s <seedc_built.o> <scc_built.o>`
   returns 0 for every TU in the compiler tree.
2. **100% aggregate emit ratio** - `sum(scc_text) / sum(seedc_text) = 1.000`.
3. **Zero TUs below probe threshold** - no TU reports < 80% ratio.

All three conditions together constitute FULLY-RECOVERED. A fix that moves
the ratio from 21% to 95% is a partial recovery, not FULLY-RECOVERED. Do
not use "no TUs in the top-10 loss list" as a proxy for full recovery -
the top-10 list changes shape as fixes land, and a 5%-residual loss spread
across many small functions would not appear in it.

**The canonical reverse witness** (40/40 byte-identical after v31.5.I
slice-elem-field-write fix at commit `de656486`):

```
Wave 1 / Audit 09 (audits/wave1/09_perf_size_diff_sweep.md)
Verdict: FULLY-RECOVERED

TUs at 100% emit ratio:  40 / 40
Byte-identical .o pairs: 40 / 40
Aggregate seedc __text:  2,153,725 bytes
Aggregate scc   __text:  2,153,725 bytes
Aggregate ratio:         100.0000 %
SHA-256 scc1 vs scc2:   identical
```

The prior probe had identified the `ast_write_*` cluster at 94% loss as
the next bug class. After the slice-indexed-write fix closed that cluster,
the reverse witness confirmed there was NO residual loss in any TU. The
probe output is zero-width (the diff is the empty set) - that is the
correct terminal state.

**Citation.** v31.5.H Wave 1 audit `09_perf_size_diff_sweep.md` (2026-05-27),
FULLY-RECOVERED verdict on 40/40 byte-identical .o objects. Seal lesson L4:
"Compile-output-size-diff probe successfully proves FULLY-RECOVERED at 100%
ratio (all 40 .o byte-identical)."

**Workflow:**

```
1. Run probe before fix  -> identify cluster (N% loss, K functions)
2. Apply fix             -> changes source
3. Rebuild both          -> scc-built .o and seedc-built .o
4. Run probe after fix   -> reverse witness
   a. If 100% + byte-identical: FULLY-RECOVERED - document + seal
   b. If 95-99%: partial, residual cluster exists - iterate
   c. If < 95%: fix incomplete or introduced a different regression
5. Append reverse-witness output to the probe report (F/probe-report
   preservation discipline in sc-empirical-surface-probe)
```
