# Failure-Mode-First Authoring v1.0.0

## When to invoke

BEFORE writing the first line of code or the first design diagram for any
feature that:

- Crosses a service / module / process boundary.
- Handles untrusted input (user, network, file, IPC).
- Mutates shared state (DB, kv-store, registry, capability table).
- Has a non-trivial state machine (>3 states or >5 transitions).
- Will be relied on by automated gates or downstream consumers.

If you find yourself "just sketching" or "just trying it" without a
failure-mode list, stop. Author the list first. Five minutes of FMEA saves
hours of debug.

## The Discipline

A feature design is incomplete until it answers: "How does this fail, and
what defends each failure mode?"

The default reflex is to design the happy path and bolt on error handling
afterward. This reflex is wrong. Error handling bolted on afterward is
nearly always incomplete because it does not cover modes the author never
imagined.

The corrective is to enumerate failure modes FIRST, then design the happy
path as the residual of "everything that isn't a failure mode."

## The 5-Bullet Template

For every non-trivial feature, author this block BEFORE coding:

```
## Feature: <name>

### Failure-Mode Catalog
1. **<mode-name>** - <one-line description of what goes wrong>
   - Trigger: <empirical condition that produces this mode>
   - Defense: <code, test, or architectural choice that prevents/contains it>
   - Detection: <how we observe this mode firing (log, metric, gate, probe)>

2. <repeat for each mode; aim for 3-8>

### Residual Happy Path
<2-4 sentence description of the success flow assuming all defenses hold>

### Defenses-Without-Tests (carve-outs)
<any defense that is asserted but not yet test-covered - assign a DEF-ID
per [[carveout-def-id-discipline]]>
```

## Authoring Heuristics

Aim for **3-8 modes**. Fewer than 3 suggests insufficient imagination
(re-read the feature scope). More than 8 suggests the feature is too large
and should be decomposed.

**Categories to probe** (use as prompts, not as a rigid template):

- **Input-shape failure** - malformed, oversized, empty, adversarial.
- **Resource exhaustion** - memory, disk, file descriptors, time budget.
- **Concurrent-access failure** - race condition, lost update, deadlock.
- **Partial-failure** - half-written state, mid-migration, network split.
- **Trust-boundary violation** - caller bypasses contract, capability leaks.
- **Upstream/downstream change** - dependency upgrade or schema drift.
- **Operator error** - misconfiguration, bad rollout, accidental deletion.
- **Time-related** - clock skew, timeout cascade, retry storm.

You do NOT need a mode from every category. You DO need to confirm each
category was considered (mark "N/A - <reason>" if not applicable).

## Anchor Episode (Reference Pattern)

**v31.5.H Wave 5 Resilience Catalog (SUPER C, 2026-05-26):** The next-
frontier carve-out closure authored an explicit resilience catalog
covering parser, lower, codegen, and runtime failure modes. The catalog
scaled down per-feature and proved load-bearing: three modes that were
catalogued but not yet defended became the v31.5.I slice scope. Without
the catalog, those three modes would have been discovered only after they
fired in production gates.

**Counter-example:** Pre-A4D arithmetic-chain work in v31.4.A4 series
shipped a 2-operand-only design without a failure-mode catalog. The
3+operand chain failure surfaced during runtime, not design. Six cycles
of safe-revert followed before the SCIR-low `op_base=0` root cause was
diagnosed via SCC_O1_DUMP. A per-feature failure-mode list at design time
would have flagged "operand-pool index packing for N-ary expressions" as
mode #1 and saved 4-5 cycles.

## Recipe - Quick Authoring

1. **Write the feature name.** One line.
2. **Brainstorm 3-8 failure modes.** Time-box to 10 minutes. Use the
   category prompts above.
3. **For each mode, write trigger + defense + detection.** If you cannot
   articulate detection, the defense is unverifiable.
4. **Write the residual happy path.** It should feel small, because most
   of the design weight has already moved into defenses.
5. **Carve out defenses-without-tests** with DEF-IDs per
   `[[carveout-def-id-discipline]]`.

## Anti-Patterns (forbidden)

- Starting to code with a vague "I'll handle errors as they come up."
- Skipping the catalog because "this feature is simple" (the simple
  features fail in the most surprising ways).
- Authoring modes but not detection signals - undefended in practice.
- Listing only happy-path edge cases as "failure modes" (input
  validation alone is insufficient; consider concurrent + partial +
  trust-boundary categories).
