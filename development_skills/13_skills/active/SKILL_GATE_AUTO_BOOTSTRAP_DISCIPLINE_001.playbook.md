# Gate Auto-Bootstrap Discipline

## When to invoke

Triggers (any of):
- "gate ordering"
- "prereq artifact"
- "silent-pass missing artifact"
- "gate dependency chain"
- "auto-bootstrap prereq"
- "gate skipped because file missing"
- A gate G consumes an artifact A produced by gate G_prev (transcripts, IR dumps, equivalence proofs, hash manifests, coverage reports).
- A gate harness has multiple shell scripts where order matters.
- A seal is about to be declared based on a gate that reads a file produced elsewhere.

## Core directive

When gate G depends on artifact A produced by gate G_prev, gate G MUST take **exactly one** of these three paths:

1. **Detect-and-bootstrap (preferred).** Top-of-script check: if A is missing or stale, invoke G_prev (or its minimal producer) inline, then proceed. Self-healing chain.
2. **Fail loudly (acceptable).** If A is missing, emit a structured `FAIL prereq=<G_prev> artifact=<A>` line, exit non-zero, and refuse to print PASS counts.
3. **Silent-pass (FORBIDDEN).** Never print FAIL while exiting 0. Never skip the gate body and report 0/0 PASS. Never treat "file not found" as "nothing to check".

The forbidden case is the most common failure mode because it produces a green CI run that certifies nothing. Every seal built on a silent-pass gate is self-certifying - the artifact under test was never read.

## Detection symptoms

1. Gate prints `PASS 5/0` but the script never opened the artifact path it claims to validate.
2. Gate prints `FAIL <reason>` followed by `exit 0` (or no explicit exit, with last command being `echo`).
3. Two consecutive runs of the same gate on the same commit yield different counts depending on whether a prior gate happened to populate `/tmp/<artifact>`.
4. A gate produces no log output when its input file is missing - silent skip masquerading as silent pass.
5. CI green but artifact file absent from the workspace after the run.

## Required pattern: the auto-bootstrap header

Every gate that consumes an upstream artifact starts with this header (bash):

```bash
#!/usr/bin/env bash
set -euo pipefail

# --- Auto-Bootstrap Header ---
GATE_NAME="$(basename "$0" .sh)"
PREREQ_GATE="${PREREQ_GATE:-producer_gate.sh}"
PREREQ_ARTIFACT="${PREREQ_ARTIFACT:-/tmp/upstream_artifact.txt}"

if [[ ! -s "$PREREQ_ARTIFACT" ]]; then
  echo "[$GATE_NAME] prereq artifact missing: $PREREQ_ARTIFACT"
  echo "[$GATE_NAME] attempting auto-bootstrap via $PREREQ_GATE"
  if ! bash "$(dirname "$0")/$PREREQ_GATE"; then
    echo "FAIL gate=$GATE_NAME prereq=$PREREQ_GATE artifact=$PREREQ_ARTIFACT cause=bootstrap_failed"
    exit 2
  fi
fi

if [[ ! -s "$PREREQ_ARTIFACT" ]]; then
  echo "FAIL gate=$GATE_NAME prereq=$PREREQ_GATE artifact=$PREREQ_ARTIFACT cause=bootstrap_did_not_produce"
  exit 2
fi
# --- end Auto-Bootstrap Header ---

# gate body follows; safe to read $PREREQ_ARTIFACT
```

Three invariants:
- `set -euo pipefail` so a typo cannot silently mask the check.
- Exit code 2 is reserved for prereq-failure (distinct from 1 = real fail, 0 = pass).
- The same `[[ ! -s ... ]]` check appears **twice** - once before bootstrap, once after - so a bootstrap that ran but did not produce is also caught.

## Anchor episode

**SUPER C v31.5.H Wave 1 (sealed 2026-05-27, v31.5H-self-host-self-compile-seal, commit 59f44e9c).**

`scc1_scc2_equivalence_gate.sh` was authored assuming `scc_via_scc_h4_proof` artifact was always present in `/tmp/`. When run in isolation (clean state, no prior chain run), the gate printed:

```
FAIL: scc_via_scc_h4_proof artifact missing
PASS 5/0
```

...and exited 0. The `FAIL` line was a `printf` to stdout, not a real failure signal. The `5/0` was the running counter from prior loop iterations (no test ran the missing-artifact path). CI was green. Main-thread auditor previously read `5/0 PASS` because the chain+h4 gates had populated the artifact earlier in the same shell. Parallel-agent auditor (Wave 1.05) ran the gate in a fresh shell and got the same green output - but the artifact was never read.

Without this skill, the v31.5.H seal could have been self-certified on a silent-pass gate. After ordering was reconciled and the auto-bootstrap header was added, the gate runs deterministically 5/0 from any starting state.

## Recipe - three-step audit

When auditing a gate before a seal:

1. **Cold-shell run.** Open a new shell. `cd` to the gate directory. Remove `/tmp/<artifact>` paths the gate might consume. Run the gate. If it prints PASS, the gate is silent-pass-vulnerable.
2. **Exit-code check.** Run with `bash -c '<gate>; echo RC=$?'`. RC must be non-zero whenever the gate body did not execute fully.
3. **Strace/dtruss the read.** `strace -e openat <gate>` (or `dtruss` on macOS) and confirm the artifact path appears in an `openat` call. If the gate prints PASS without ever opening the artifact, it is a silent-pass.

If any of (1)/(2)/(3) fails, the gate must be patched with the auto-bootstrap header before the seal is declared.

## Anti-patterns

- `[ -f "$ART" ] && grep -q PASS "$ART" || echo "no artifact"` - falls through to PASS counter.
- `if [ -f "$ART" ]; then validate; fi` with no `else fail`.
- A `printf "FAIL: ..."` followed by `continue` inside a loop where the outer counter increments unconditionally.
- Reading from a `/tmp/` path that another gate happens to populate, with no explicit producer declaration.
