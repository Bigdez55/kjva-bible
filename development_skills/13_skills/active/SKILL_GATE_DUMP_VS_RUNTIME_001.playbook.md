# Gate Dump vs Runtime

**Version: v1.0.0** (2026-05-27, post-v31.5.I.F1.runtime)

**DIRECTIVE: A structural (IR dump) gate and a runtime RC gate are distinct and non-substitutable. Both are required for any emitter feature.**

---

## Core rule

When verifying that a compiler feature (e.g., field READ, field WRITE,
arithmetic chain) is correctly implemented, TWO classes of gate must pass:

1. **Structural gate** - SHIRL dump shows the expected opcode(s) with the
   expected field values (field_idx, op_base, const_lo, etc.).
2. **Runtime gate** - executing the compiled output yields the expected RC
   (or stdout value); path=bridge (or path=fold) behaves as intended.

A structural gate cannot see whether an emitter function returned `1`
(handled) prematurely on an error path without actually emitting any IR.
A runtime gate cannot pinpoint which IR instruction is malformed.
They are complementary, not redundant.

**Anti-pattern to reject at code review:**

```sh
# WRONG: gate that only checks IR shape
SCC_O1=1 SCC_O1_DUMP=1 scc emit-direct fixture.sc -o /dev/null
grep -q "SHIRL_STORE_32" /tmp/dump.txt && echo "PASS" || { echo "FAIL"; exit 1; }
```

This would pass even if `lower_field_assign` returned `1` (handled) on
pool-exhaust WITHOUT emitting any STORE.

**Correct pattern:**

```sh
# Step 1: structural check
SCC_O1=1 SCC_O1_DUMP=1 scc emit-direct fixture.sc -o /tmp/out.bin
grep -q "SHIRL_STORE_32" /tmp/dump.txt || { echo "FAIL structural"; exit 1; }

# Step 2: runtime check  
scc run-jit fixture.sc
RC=$?
[ "$RC" -eq 42 ] || { echo "FAIL runtime RC=$RC expected=42"; exit 1; }
```

## 1 Silent-Success Smell in Emitter Error Paths

### The failure class

Emitter helper functions that return a "handled" sentinel (e.g., `1` or `true`)
on capacity-exhaust WITHOUT actually emitting the IR instruction they were
supposed to emit. The caller proceeds as if emission succeeded, producing
a structurally-incomplete IR that no dump gate detects.

### Canonical anchor

v31.5.I.F1.runtime episode: `lower_field_assign` at `lower_walker.sc:608-617`
returned `1` (handled) on `arenas.operands_len + 2 > arenas.operands_cap`
WITHOUT emitting STORE. Downstream:
- `field_write_lhs_gate` passed 8/0 (dump showed correct shape because the
  probe fixture didn't saturate the pool).
- Runtime: `s.a = 7; return s.a` returned the initial value, not 7.
- Only a runtime RC probe with a VARIED expected value (not 42 by coincidence)
  surfaced the divergence.

### Mandatory grep on any emitter PR

```bash
grep -nE 'return 1.*pool|return 1.*exhaust|return 1.*cap|return 1.*emit_inst' \
    compiler/scc/src/*.sc
```

Every hit: audit whether the return path emits the intended instruction BEFORE
returning. Convert `return 1` (handled, but missing emit) to `return 0`
(unhandled) so the caller's fallback produces a diagnostic-visible failure.

### Diversify fixture expected values

When probing emitter correctness, use DISTINCT expected RC values per fixture
(e.g., `return 2`, `return 5`, `return 42`) so that a final_ldr leak that
happens to produce 42 doesn't mask a missing STORE. A leaked prior
register value is indistinguishable from a correct write if both happen to
be 42.

## 2 Gate Pair Template

For any new emitter feature (READ, WRITE, compound-assign, arith chain,
param pass, return value):

```sh
#!/usr/bin/env bash
# Gate: <feature>_gate.sh
# Version: pair (structural + runtime)
set -e
PASS=0; FAIL=0

# --- Structural ---
SCC_O1=1 SCC_O1_DUMP=1 scc emit-direct tests/fixtures/<feature>_probe.sc \
    -o /tmp/<feature>_probe.bin > /tmp/<feature>_dump.txt 2>&1
grep -q "<EXPECTED_OPCODE>" /tmp/<feature>_dump.txt \
    && { PASS=$((PASS+1)); echo "PASS structural <feature>"; } \
    || { FAIL=$((FAIL+1)); echo "FAIL structural <feature>"; }

# --- Runtime ---
scc run-jit tests/fixtures/<feature>_probe.sc
RC=$?
EXPECTED=42  # must be distinct per fixture; do not reuse same value across fixtures
[ "$RC" -eq "$EXPECTED" ] \
    && { PASS=$((PASS+1)); echo "PASS runtime RC=$RC"; } \
    || { FAIL=$((FAIL+1)); echo "FAIL runtime RC=$RC expected=$EXPECTED"; }

echo "$PASS passed, $FAIL failed"
[ "$FAIL" -eq 0 ] && exit 0 || exit 1
```

## 3 Artifact-Dependent Gates

**Rule.** When a gate's runtime check depends on a specific build artifact
existing (e.g., a pre-built proof binary, a pre-compiled `.o`, an intermediate
file produced by a prior gate), the gate MUST self-verify the artifact
exists at gate entry, with a clear FAIL path that sets a non-zero exit code.

```sh
# At gate entry - BEFORE any structural or runtime check
if [ ! -f "$ARTIFACT_PATH" ]; then
    echo "FAIL: artifact missing: $ARTIFACT_PATH"
    echo "  This gate requires $(basename $ARTIFACT_PATH) to exist."
    echo "  Run <prerequisite_gate>.sh first."
    exit 1
fi
```

**FORBIDDEN patterns:**

```sh
# WRONG: silently proceeds if artifact missing (treats missing as empty/zero)
[ -f "$ARTIFACT_PATH" ] && run_check || true

# WRONG: sets a SKIP without incrementing FAIL
[ -f "$ARTIFACT_PATH" ] || { echo "SKIP (artifact absent)"; return 0; }
```

**Why.** The `scc1_scc2_equivalence_gate.sh` was found to produce a divergent
reading between a warm session (h4 proof binary present from prior run) and
a cold session (h4 binary absent). Wave 1 audit (`02_gate_suite_forensic.md`)
classified this as MEDIUM - the structural gate still asserted a real
assertion count, but the dependency path was unguarded. In a cold-build
CI context, the gate would silently report a different N/0 than expected,
making the aggregator's "all GREEN" claim fragile.

**Citation.** v31.5.H Wave 1 audit finding, `02_gate_suite_forensic.md`,
MEDIUM finding on `scc1_scc2_equivalence_gate.sh` ordering dependency. Seal
lesson L2: "Silent-pass-on-missing-artifact is a seal-fragility class."
Companion rule in `compiler-discipline` v1.6 12.

## 4 Brittle Regex Hardcoding

**Rule.** Gate scripts MUST NEVER hardcode an exact integer test count in a
regex assertion. Use a range pattern instead.

### The failure class

A gate that asserts `"127 passed"` (exact match) will fail spuriously the moment
any new test case is added - even if all tests pass, even if the new test is
entirely correct. This turns a coverage-growth action (adding tests) into an
apparent gate regression. The fix cycle (wrong diagnosis -> wrong patch -> wasted
investigation) has historically consumed a full sub-session.

### Empirical anchor - Wave 4 brittle-127 fix

In the Wave 4 audit, 11 gates in `scripts/gates/gold_gate.sh` and related gate
scripts were found to contain exact-count assertions of the form:

```sh
grep -q "127 passed, 0 failed" /tmp/smoke_out.txt
```

When Wave 4's test additions pushed scc_smoke from 127 to 129 passing tests,
all 11 gates turned red simultaneously with no functional regression. The "fix"
required updating the hardcoded `127` in each of the 11 greps - a mechanical
exercise that cost investigation time before the root cause was identified.

### Correct pattern: range regex

```sh
# WRONG - brittle exact-count assertion
grep -q "127 passed, 0 failed" /tmp/smoke_out.txt

# CORRECT - range pattern that tolerates forward growth
grep -qE "1[0-9][0-9] passed, 0 failed" /tmp/smoke_out.txt

# For larger expected ranges (150-199):
grep -qE "1[5-9][0-9] passed, 0 failed" /tmp/smoke_out.txt

# For a floor-only assertion (at least N, no upper bound):
PASS_COUNT=$(grep -oE "[0-9]+ passed" /tmp/smoke_out.txt | grep -oE "[0-9]+")
[ "$PASS_COUNT" -ge 127 ] || { echo "FAIL: expected >=127 passed, got $PASS_COUNT"; exit 1; }
```

**Preferred form.** The floor-only assertion is the most robust: it verifies
that pass count has not REGRESSED (< floor) while tolerating any amount of
forward test growth. Use exact-count assertions ONLY when the count is a
semantic invariant (e.g., exactly N opcodes in a fixed opcode table) - not
when the count grows as tests are added.

### Audit rule for gate authors

When authoring or reviewing a gate script, grep for hardcoded counts:

```sh
grep -nE '"[0-9]+ passed"' scripts/gates/*.sh
```

Every match must be converted to a range pattern or floor assertion before
the gate lands. File a follow-up if existing brittle gates are found during
other work - do not defer indefinitely.

### Cross-reference

- `compiler-discipline` 13 - Coverage-Growth Compatibility (same lesson from
  the test-author's perspective: when adding tests, update aggregator gates
  in the same commit or use future-proof regex).

## 5 Outer Harness RC Capture

**Rule.** When a gate script invokes a sub-gate or any command whose exit code
determines PASS/FAIL, capture RC via assignment - NEVER via a pipeline.

### The failure class

A pipeline discards the exit code of every command except the last:

```bash
# WRONG - rc is ALWAYS 0 (exit code of tail/head, not gate)
bash gate.sh | tail -5 | head -3
rc=$?   # rc=0 regardless of gate.sh result
```

This pattern produces a perpetual PASS reading even when the gate fails.
The outer harness reports "all green" while individual gates are red.

### Correct pattern

```bash
# CORRECT - capture to file, read RC from assignment
bash gate.sh > /tmp/gate_out.txt 2>&1
rc=$?
[ "$rc" -eq 0 ] || { echo "FAIL gate.sh RC=$rc"; cat /tmp/gate_out.txt; exit 1; }
```

Or, with an aggregator loop:

```bash
# CORRECT - RC captured per-gate, not piped
FAIL=0
for gate in scripts/gates/*.sh; do
  bash "$gate" > /tmp/gate_out.txt 2>&1
  rc=$?
  if [ "$rc" -ne 0 ]; then
    echo "FAIL: $gate (RC=$rc)"
    FAIL=$((FAIL+1))
  fi
done
[ "$FAIL" -eq 0 ] && echo "ALL PASS" || { echo "$FAIL gate(s) FAILED"; exit 1; }
```

### Empirical anchor - Wave 7 wrong-RC bug

In the SUPER C Wave 7 sweep, an outer harness that reported "all green" was found
to be piping each gate through `tail -20 | head -10` to trim output. Because
`head`'s exit code (0) was what `$?` saw after the pipe, every gate appeared to
pass - including gates that were returning non-zero. The bug was caught only when
the wall-clock sweep showed anomalously fast runtimes for gates known to be slow:
the "fast" gates were actually failing early and the outer harness was not
capturing their failure.

Fix: replace every `bash gate | pipe-chain; rc=$?` with `bash gate > tmp; rc=$?`.

### Interaction with `set -e`

`set -e` does NOT protect against this. A pipeline under `set -e` aborts only if
the LAST command in the pipe fails. `bash gate | tail | head` - only `head` needs
to exit 0 for the shell to continue. Use `set -o pipefail` if you need pipe-aware
abort, but the safest form is to avoid the pipeline entirely.

```bash
# If you must use set -e, add pipefail:
set -euo pipefail
bash gate.sh | tail -5   # now aborts if gate.sh fails - but file-based is still safer
```

### Anti-pattern table

| Pattern | Why it fails |
|---|---|
| `bash gate \| tail -5; rc=$?` | `rc` = exit code of `tail`, always 0 |
| `bash gate \| head -20 \| grep PASS` | `rc` = exit code of `grep`, 1 if no match even on PASS |
| `output=$(bash gate); rc=$?` | Works for RC, but discards stderr - use `2>&1` |
| `bash gate && echo PASS \|\| echo FAIL` | Prints FAIL but script continues, exit code lost |

## Anti-patterns summary

| Anti-pattern | Why it fails |
|---|---|
| Dump-only gate for emitter feature | Cannot detect silent-success on capacity-exhaust path |
| Reuse same expected RC (42) across all probes | Leaking register coincidentally equals expected -> false PASS |
| Missing artifact path -> `|| true` | Turns structural fragility into silent SKIP rather than loud FAIL |
| `return 1` (handled) without emitting | Caller sees "success", IR incomplete, runtime fails silently |
| Gate reads dump file without `-o` on emit-direct | scc prints usage hint, dump file empty, grep vacuously passes |
| Hardcoded exact count in gate regex (`"127 passed"`) | Any test addition turns a green suite red with no functional regression |
