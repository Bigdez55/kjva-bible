# Multi-Wave Audit Cadence v1.0.0

## When to invoke

When the work in front of you is too large for a single agent, too risky
for a single pass, and load-bearing for downstream consumers. Specifically:

- Post-seal reviews (after a milestone tag, before declaring stability).
- Release readiness gates (before tagging an LTS / Gold / external release).
- Deep audits requested by an operator who has lost confidence in a claim.
- Cross-cutting refactors where >5 modules will change in coordinated fashion.
- Recovery from a botched or dishonest prior seal.

Do NOT invoke for routine slices, single-file fixes, or work that one
focused agent can complete in a session. The wave protocol has overhead;
small work should not pay it.

## The 5-Wave Protocol

### Wave 1 - Read-Only Audit (parallel, 6-10 agents)

**Charter:** No source modification. Each agent owns one assertion class
and verifies it against the artifact (commit, tag, repo state, claim).

**Deliverables:** Per-agent audit report in `<project>/audits/wave1/` with
explicit PASS / FAIL / UNCERTAIN verdicts and primary-source citations.

**Discipline:** Each agent must cite line numbers, commit hashes, or
runtime probe output. Hearsay or "looks fine" is a FAIL verdict by
default. Conflicting verdicts between agents trigger
`[[seal-reconciliation-discipline]]`.

### Wave 2 - Source Remediation (write-disjoint parallel, 3-6 agents)

**Charter:** Apply fixes for Wave 1 findings. Each agent owns a
disjoint set of files (write-disjoint = guaranteed merge-clean).

**Deliverables:** A commit per agent with a `audits/wave2/<agent>.md`
note describing the change set, the Wave 1 finding it closes, and any
regression tests added.

**Discipline:** No agent may touch a file owned by another Wave 2 agent.
Pre-flight inventory of file ownership is mandatory. If ownership
overlaps, the conflicting work moves to Wave 4 (main thread).

### Wave 3 - Skill Consolidation (1-3 agents)

**Charter:** Author or refine reusable skills that codify lessons from
Waves 1 and 2. The aim is to make this audit's findings cheaper to detect
next time.

**Deliverables:** New / updated SKILL.md files; a `audits/wave3/<topic>.md`
summary linking the new skills to the Wave 1 findings that motivated them.

**Discipline:** A skill is only worth authoring if its trigger conditions
are clear, its anchor episode is real (from Wave 1/2 evidence), and a
future agent could invoke it without ambiguity. Vague meta-skills are
forbidden.

### Wave 4 - Main-Thread Closure (1 agent, the orchestrator)

**Charter:** Reconcile Waves 1-3 outputs, resolve overlapping file
ownership left from Wave 2, author the super-seal report, and execute the
canonical tag.

**Deliverables:** `audits/wave4/SUPERSEAL.md` (or project-canonical name);
annotated git tag; final memory / agent-memory update with index entry.

**Discipline:** Main-thread only. No parallel agents. The orchestrator
holds the only authority to declare the seal. Per
`[[carveout-def-id-discipline]]`, every carve-out in the super-seal MUST
have a DEF-* ID, detection signal, and recovery class.

### Wave 5 - Next-Frontier Carve-Out Closure (1-3 agents, optional)

**Charter:** Close or scope the carve-outs declared in Wave 4. This is
the "what comes next" wave; it is optional and may be deferred to the
next cycle.

**Deliverables:** Either (a) carve-out closure commits + DEF-ledger
updates, or (b) explicit deferral notes per
`[[carveout-def-id-discipline]]` with refreshed detection signals.

**Discipline:** If Wave 5 produces commits, they ride the same seal tag
as Wave 4. If deferred, the DEF-ledger entry is the artifact.

## Agent Budget

| Wave | Typical | Major audit |
|------|---------|-------------|
| 1    | 6-10    | 10-14       |
| 2    | 3-6     | 5-8         |
| 3    | 1-3     | 2-3         |
| 4    | 1       | 1           |
| 5    | 1-3     | 3-4         |
| **Total** | **<=17** | **<=22**     |

Exceeding 22 agents in a single cadence is a smell; decompose the audit
scope and run two sequential cadences instead.

## Anchor Episode

**v31.5.H super-seal cycle (SUPER C, 2026-05-26):** Originally planned as
4-wave; Wave 5 was added when six carve-outs from Wave 4 needed
next-frontier closure. The final cycle: 8 Wave-1 agents, 4 Wave-2 agents,
2 Wave-3 skill-authoring agents, 1 Wave-4 orchestrator, 3 Wave-5 carve-out
agents = 18 total. Tag: `v31.5.H-superseal`. Zero post-seal regressions
within 7 days.

**Counter-example:** The v31.4 series ran 11 single-agent cycles in a row
with no wave protocol. Each cycle made progress, but findings were not
consolidated and recovery from the A4 / A4_REAL reverts cost 4 extra
sessions. A Wave 1 audit would have surfaced the SCIR-low `op_base=0`
miscompile much earlier.

## Recipe - Directive Template

```
DEEP AUDIT: <subject> @ <commit/tag>
SCOPE: <one-line scope statement>
NON-GOALS: <explicit out-of-scope items>

WAVE 1 (read-only, parallel): <N> agents, one per assertion class.
  Deliverable per agent: audits/wave1/<agent>.md
  Owner: <agent role list>

WAVE 2 (write-disjoint, parallel): <N> agents.
  File ownership map (pre-allocated):
    Agent A: <file paths>
    Agent B: <file paths>
    ...
  Deliverable: commit + audits/wave2/<agent>.md

WAVE 3 (skill consolidation): <N> agents.
  Deliverable: new/updated SKILL.md + audits/wave3/<topic>.md

WAVE 4 (main thread): orchestrator.
  Deliverable: audits/wave4/SUPERSEAL.md + annotated tag.

WAVE 5 (optional, carve-out closure): <N> agents.
  Deliverable: closure commits OR refreshed DEF-ledger entries.
```

## Anti-Patterns (forbidden)

- Skipping Wave 1 because "we know what's wrong" - confirmation bias.
- Wave 2 agents with overlapping file ownership - merge chaos.
- Wave 3 skills with no anchor episode - speculative noise.
- Wave 4 super-seal without DEF-IDs for carve-outs - silent debt.
- Declaring Wave 5 done by waving hands ("we'll get to it") - that is
  exactly the freeze pattern this cadence exists to prevent.
