# Outer Harness RC Capture Discipline

## When to Invoke

You are writing or reviewing an outer bash harness that runs N inner gates and reports pass/fail. **The single most common foot-gun in this space** is capturing `$?` after a pipeline instead of after the gate command. Result: sweep reports 0 FAILs while the actual gate count is mostly red.

Triggers:
- "harness", "outer sweep", "gate sweep", "bash sweep"
- "rc capture", "pipe exit code", "PIPESTATUS"
- "0 FAIL but actually failing", "silent green", "harness lied"
- Any review of a multi-gate sweep script.

## The Rule (1 sentence)

**Capture `rc=$?` IMMEDIATELY after the gate command on its own line, BEFORE any pipe, tail, head, or summary processing.**

## The Foot-Gun (anti-pattern)

```bash
# WRONG - rc reflects head's exit, not the gate's. head always returns 0.
bash gate.sh 2>&1 | tail -5 | head -1
rc=$?    # ALWAYS 0. Gate's true exit is LOST.
```

Variants of the same trap:
```bash
result=$(bash gate.sh 2>&1 | tail -1); rc=$?   # rc = tail's exit (0)
bash gate.sh | tee log.txt; rc=$?              # rc = tee's exit (0 unless disk full)
out=$(bash gate.sh | grep PASS); rc=$?         # rc = grep's exit (1 if no match - wrong direction!)
```

## The Correct Pattern (5-line template)

```bash
# Run gate, redirect both streams to file, capture rc IMMEDIATELY.
bash gate.sh > "$LOG" 2>&1
rc=$?
summary=$(tail -2 "$LOG" | head -1)
if [ "$rc" -ne 0 ]; then echo "FAIL rc=$rc gate=$gate summary=$summary"; fi
```

Three invariants:
1. Gate writes to a **file**, not a pipe.
2. `rc=$?` is the **very next line** after the gate command.
3. Summary extraction happens **after** rc is safe in a variable.

## Acceptable Alternative: PIPESTATUS

If you must pipe (e.g. live tee'ing while running), use `${PIPESTATUS[0]}`:
```bash
bash gate.sh 2>&1 | tee "$LOG"
rc=${PIPESTATUS[0]}    # exit of bash gate.sh, NOT tee
```
Note: `PIPESTATUS` is bash-only (not POSIX sh) and is clobbered by the next command - capture on the next line, not later.

## Anti-Patterns Beyond the Pipe

- **`set -e` instead of explicit rc**: in an outer sweep, `set -e` aborts on first FAIL - you want to **continue and tally**. Disable `set -e` inside the loop or use `bash gate.sh || rc=$?`.
- **Truthy-tests on summary text**: `if grep -q PASS "$LOG"; then ...` is fragile (a gate may print "PASS" in some intermediate step then FAIL). Trust `rc`, not text.
- **Counting only "FAIL" lines**: a SIGSEGV gate may write nothing before dying. Always tally by `rc`, supplement with text.
- **Reusing `$rc` across iterations without reset**: in a loop, `rc=$?` must be inside the iteration; otherwise a previous iteration's value leaks if the current gate is skipped.

## Anchor Case

**SUPER C Wave 6 sweep**: outer harness used `bash gate | tail -1; rc=$?`. Reported **0 FAILs across 60 gates**. Subsequent Wave 7 audit with correct `rc=$?`-before-pipe pattern revealed **26 of 60 were failing**. Root cause: tail always returns 0; the actual gate RCs (139 SIGSEGV, 1, 2, etc.) were silently discarded. Skill exists to prevent the next 26-FAIL silent-green.

## Verification Checklist (before merging any sweep harness)

- [ ] `rc=$?` appears on the line IMMEDIATELY after the gate invocation
- [ ] No pipe between gate command and `$?` capture
- [ ] If pipe is unavoidable, `${PIPESTATUS[0]}` is used and captured on the next line
- [ ] Failure tally is keyed on `rc`, not on text matching
- [ ] `set -e` is OFF inside the iteration loop
- [ ] Sanity test: insert a deliberately-failing gate; harness must report it as FAIL.
