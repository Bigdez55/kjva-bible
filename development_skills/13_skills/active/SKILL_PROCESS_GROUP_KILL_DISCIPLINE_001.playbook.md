# Process-Group Kill Discipline

## When to invoke

Triggers (any of):
- "JIT spin"
- "alarm timeout doesn't kill"
- "orphan compiler children"
- "flake from process accumulation"
- "SIGALRM ignored by child"
- "gate harness flake"
- Any harness that timeboxes a compiler, JIT, interpreter, or runtime that can loop on bad input.
- Gate counts drift across runs on the same commit (e.g., 17/4  18/3) with no source change.
- `ps -ef | grep <child>` shows accumulating instances of a process that the harness should have killed.

This skill is the surgical counterpart to `gate-harness-process-isolation`. That skill addresses **all** shell-timeout leaks on macOS/BSD. This skill is the **specific recipe** for the compiler-spin case: when the child is a compiler/JIT under test that may not respect SIGALRM because it is in a tight CPU loop or has installed its own signal handlers.

## Core directive

**A `perl -e 'alarm N; exec @ARGV'` harness is INSUFFICIENT** for timeboxing a compiler or JIT runtime. Three failure modes:

1. **SIGALRM ignored.** The child may install its own SIGALRM handler (or run in a tight loop where signal delivery is delayed indefinitely on macOS). The alarm fires in the harness, harness exits, child continues.
2. **Grandchild orphan.** If the compiler forks a sub-process (linker, codegen helper, dtrace), killing the harness leaves the grandchild reparented to launchd/init.
3. **CPU saturation.** Accumulating spin-loops from prior gate runs saturate cores, slowing subsequent gates enough to cross their own timeouts -> cascade flake.

**Required pattern:** launch the child in a **new process group** via `setsid` (Linux) or POSIX `setpgid` (portable), then on timeout issue `kill -9 -<pgid>` (negative PID = process group). This wipes the entire descendant tree including spinning grandchildren.

## Detection symptoms

1. Gate counts flake across runs on the same commit, with no source change between runs (e.g., 17/4 then 18/3 then 17/4).
2. `pkill -9 <child_name>` between runs makes the flake disappear.
3. `ps -eo pid,pgid,ppid,stat,command | grep <child>` shows multiple instances where PPID=1 (orphaned to init/launchd).
4. CPU load average climbs across gate runs in the same session.
5. Harness log says `TIMEOUT` but child PID is still alive 30 seconds later.

## Required pattern: the `kill_after.sh` recipe

A 20-line POSIX shell harness - drop in alongside the gate, invoke as `kill_after.sh <seconds> <cmd> [args...]`:

```bash
#!/usr/bin/env bash
# kill_after.sh - run a command in its own process group, kill the whole
# group after N seconds. Survives SIGALRM-ignoring children and grandchild
# orphans. POSIX-portable (uses setsid where available; falls back to
# `exec setpgid` via a perl wrapper on systems without setsid).

set -u
TIMEOUT="${1:?usage: kill_after.sh <seconds> <cmd> [args...]}"
shift

# Launch child as new session leader -> new process group with pgid == pid.
if command -v setsid >/dev/null 2>&1; then
  setsid "$@" &
else
  # macOS lacks setsid by default; perl POSIX::setpgid is reliable.
  perl -e 'use POSIX qw(setsid); setsid(); exec @ARGV' "$@" &
fi
CHILD_PID=$!

# Background timer: after TIMEOUT, SIGKILL the whole process group.
(
  sleep "$TIMEOUT"
  # Negative PID = process group. -9 = SIGKILL (uncatchable, unignorable).
  kill -9 -"$CHILD_PID" 2>/dev/null
) &
KILLER_PID=$!

# Wait for child. If it exits in time, cancel the killer.
wait "$CHILD_PID"
RC=$?
kill "$KILLER_PID" 2>/dev/null
wait "$KILLER_PID" 2>/dev/null

# Distinguish timeout-kill (RC=137) from real exit code.
exit "$RC"
```

Five invariants:
- `setsid` (or `perl POSIX::setsid`) - child becomes session leader, gets its own pgid.
- `kill -9 -<pgid>` - negative PID is the process-group form. Wipes children and grandchildren.
- SIGKILL (`-9`), not SIGTERM. The child cannot install a handler for SIGKILL.
- Killer runs in the background; main thread `wait`s on the child only. No `trap`/`alarm` in the parent shell.
- Exit code `137` (128 + 9) is the canonical signal that timeout-kill fired - distinct from a real non-zero exit.

## Anchor episode

**SUPER C v31.5.H Wave 1.05 (Apple Silicon, macOS Darwin 25.3.0, 2026-05-27).**

`gold_gate.sh` flaked between 17/4 and 18/3 across consecutive runs on the same commit (`59f44e9c`). Reproduction:

```
$ ./gold_gate.sh; ./gold_gate.sh; ./gold_gate.sh
PASS 17/4
PASS 18/3
PASS 17/4
```

Diagnosis: the existing harness used `perl -e 'alarm 30; exec @ARGV' scc emit-direct probe_spin.sc`. On probes that triggered a codegen infinite loop, the alarm fired in the perl wrapper but `scc` (in a tight C loop processing a malformed IR walk) did not surface to a signal-delivery point. The perl process exited; `scc` continued to spin. Each `gold_gate.sh` run leaked 1-4 spinning `scc` processes. By the 3rd run, CPU contention slowed two unrelated gates past their own perl-alarm windows, causing them to count as FAIL.

Reconciliation:
```
$ pkill -9 scc
$ ps -ef | grep scc   # confirmed clean
$ ./gold_gate.sh; ./gold_gate.sh; ./gold_gate.sh
PASS 18/3
PASS 18/3
PASS 18/3
```

Deterministic after the spin-corpses were reaped. Permanent fix: swap perl-alarm for `kill_after.sh` recipe above. No more flake.

## Wave 7 Wins: kill_after.sh Adoption Results

**Rule.** `kill_after.sh` is the MANDATORY timeout wrapper for any gate that
invokes a compiler, JIT, interpreter, or runtime. Local `perl -e 'alarm N; exec @ARGV'`
wrappers are PROHIBITED. Write no new alarm-wrapper code - drop in `kill_after.sh`.

### Measured impact

In SUPER C Wave 7, the `scc1_scc2_equivalence_gate.sh` was converted from its
existing per-invocation perl alarm wrapper to `kill_after.sh`. Measured result:

| Approach | Wall-clock per gate run |
|---|---|
| Local `perl -e 'alarm 30; exec @ARGV'` (per-invocation) | ~1m41s (101 seconds) |
| `kill_after.sh 30 <cmd>` (single process group) | ~0.278 seconds |
| Speedup | **363x** |

The 363x speedup occurred because the old wrapper spawned a fresh perl interpreter
for every sub-invocation inside the gate (gate called scc N times -> N perl
processes spun up and torn down serially). `kill_after.sh` is a 20-line POSIX
shell script with no interpreter startup overhead - it uses `setsid`/`setpgid` in
the shell, not a language runtime.

Additionally, the equivalence gate path previously triggered multiple full builds
via the alarm-wrapper path. Using `kill_after.sh` with `CHECK_GATE_ATTEMPTS=1`
(see `compiler-discipline` 14) cut that gate from 178s to 82s in a separate
measurement (2.2x from attempt-count reduction, compounding with the per-call
speedup).

### Mandate

1. **Do not write new `perl -e 'alarm N; exec ...'` wrappers.** They are
   unreliable against tight C/SC loops AND slow due to interpreter startup cost.
2. **Do not use bare `timeout N`** on macOS without `--kill-after` - the macOS
   `timeout` from BSD coreutils does not send SIGKILL after grace period on all
   versions.
3. **Use `kill_after.sh`** for every new gate that timeboxes an external process.
   Copy the recipe from Required pattern above into `scripts/harness/kill_after.sh`
   if not already present. Reference it with a relative path from the gate.
4. **Migrate existing wrappers** when touching a gate for any other reason.
   Do not leave `perl alarm` wrappers in place during any gate refactor.

### Migration recipe (one-gate adoption, 3 steps)

```bash
# Step 1: Drop kill_after.sh into the harness directory (idempotent)
HARNESS_DIR="$(dirname "$0")"
KILL_AFTER="$HARNESS_DIR/kill_after.sh"
[ -f "$KILL_AFTER" ] || {
  echo "ERROR: kill_after.sh missing at $KILL_AFTER"
  echo "Copy from process-group-kill-discipline SKILL.md Required pattern"
  exit 1
}

# Step 2: Replace every perl-alarm call
# Before:
#   perl -e 'alarm 30; exec @ARGV' scc emit-direct probe.sc -o /dev/null
# After:
"$KILL_AFTER" 30 scc emit-direct probe.sc -o /dev/null

# Step 3: Verify determinism - run the gate 3 consecutive times
bash my_gate.sh && bash my_gate.sh && bash my_gate.sh
# All three must show identical PASS/FAIL counts.
```

## Recipe - three-step adoption

1. **Detect.** Run the suspected gate 3 consecutively. If counts drift, suspect process accumulation. Run `pkill -9 <child>` between runs; if drift disappears, root cause confirmed.
2. **Drop in `kill_after.sh`** alongside the gate.
3. **Migrate the harness.** Replace every `perl -e 'alarm N; exec ...'` and every bare `timeout N ...` (on macOS) with `kill_after.sh N ...`. Re-run 3. Counts must be identical.

## Anti-patterns

- `perl -e 'alarm 30; exec @ARGV'` for a compiler under test. SIGALRM is not reliable against tight C loops.
- `timeout` (GNU coreutils) on macOS without `--kill-after=` and without process-group flag (`-k` semantics vary by version).
- `kill $!` (positive PID, parent only) - leaves the entire descendant tree.
- `trap ... EXIT` in the parent shell to clean up - fires too late if the parent itself was killed.
