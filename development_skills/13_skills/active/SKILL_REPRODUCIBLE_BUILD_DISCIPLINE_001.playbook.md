# Reproducible Build Discipline

**Version: v1.0.0** (2026-05-27)

**DIRECTIVE: Two independent builds of the same source tree MUST produce
byte-identical artifacts. Non-determinism is a defect, not a known limitation.**

---

## Why reproducibility matters for SUPER C

The SUPER C bootstrap closure proof requires byte-identical output from:

```
seedc compiles scc -> scc1
scc1  compiles scc -> scc2
sha256(scc1) == sha256(scc2)    the G6/G5 gate
```

If ANY non-determinism source is active (timestamps, UUID in metadata, random
symbol-table order, non-zero padding bytes), then `sha256(scc1) != sha256(scc2)`
EVEN when the compiler is semantically correct. The closure fails not because
of a real bootstrap defect, but because of build-system noise. These failures
are particularly treacherous because they look like compiler correctness bugs
when they are actually toolchain reproducibility bugs.

**Empirical anchor - v31.5.H Wave 5 `scc1_scc2_identity` closure.**

Wave 5 surface audit surfaced hash drift between scc1 and scc2 that had nothing
to do with SC source behavior. Three sources were active: Mach-O timestamps
(embedded at link time), an unsorted symbol table (linker output order was
randomized by iteration over a hash map), and 12 bytes of non-zero padding in
the `__LINKEDIT` segment. After applying all three fixes, sha256(scc1) ==
sha256(scc2) on every clean rebuild. The closure gate went from FLAKY to
DETERMINISTIC.

## The 4 Main Non-Determinism Sources

### Source 1: Embedded Timestamps

**What it is.** Build tools (compilers, linkers, archivers) often embed the
current wall-clock time into output artifacts. For Mach-O (macOS): the `LC_ID_DYLIB`
timestamp and the `__LINKEDIT` build version timestamp. For ELF: `DT_TIMESTAMP`
in dynamic sections, and some linkers embed a build-ID based on hash-of-timestamp.
For `.a` archives: BSD `ar` embeds per-member modification times.

**Symptom.** Two builds 1 second apart produce different binaries. `xxd scc1 | diff -
<(xxd scc2)` shows differences at predictable offsets (Mach-O: offset 0x18 or
`__LINKEDIT` segment; ELF: note section).

**Recipe - Mach-O / macOS:**

```sh
# Set SOURCE_DATE_EPOCH to a fixed date (e.g., repo's initial commit)
export SOURCE_DATE_EPOCH=1700000000

# For clang/ld64, this suppresses embedded timestamps in most output types:
export ZERO_AR_DATE=1          # for ar archives
export MallocNanoZone=0        # disables some malloc instrumentation metadata

# For lld (if used):
# Pass -reproducible to the linker: LDFLAGS += -Wl,-reproducible
```

**Recipe - ELF / Linux / lld:**

```sh
export SOURCE_DATE_EPOCH=1700000000
# lld respects SOURCE_DATE_EPOCH for build-ID generation
# Add to LDFLAGS:
LDFLAGS += -Wl,--build-id=none   # disable build-ID entirely (strongest)
# OR
LDFLAGS += -Wl,--build-id=sha1   # deterministic build-ID based only on content
```

**Recipe - `.a` archives (ar):**

```sh
# BSD ar (macOS):
ZERO_AR_DATE=1 ar rcs libfoo.a foo.o bar.o

# GNU ar (Linux) - use deterministic mode:
ar D rcs libfoo.a foo.o bar.o    # -D = deterministic; strips timestamps and UIDs
```

**Verification:**

```sh
make clean && make && sha256sum build/scc > /tmp/sha_run1.txt
make clean && make && sha256sum build/scc > /tmp/sha_run2.txt
diff /tmp/sha_run1.txt /tmp/sha_run2.txt  # must be empty
```

### Source 2: UUID / Random Build-ID

**What it is.** Some linkers embed a random UUID in each output binary. macOS `ld64`
embeds a `LC_UUID` load command with a UUID generated fresh at link time. This is
distinct from timestamps - it is a random 16-byte value, not derived from wall-clock
time. The UUID changes on EVERY link invocation even with identical input and
identical `SOURCE_DATE_EPOCH`.

**Symptom.** `sha256(scc1) != sha256(scc2)` even after suppressing timestamps.
`otool -l scc1 | grep uuid -A2` shows a UUID value; compare across builds - differs.

**Recipe - macOS ld64:**

```sh
# Option A: Suppress UUID entirely (strongest - no UUID in output)
LDFLAGS += -Wl,-no_uuid

# Option B: Pin to a content-derived UUID (content-addressable, deterministic)
# Not directly available in ld64; Option A is the standard approach for
# reproducible builds.
```

**Recipe - lld (ELF):**

```sh
# lld uses a content-hash by default when SOURCE_DATE_EPOCH is set.
# Explicit control:
LDFLAGS += -Wl,--build-id=none    # suppress
LDFLAGS += -Wl,--build-id=sha1    # content-derived (deterministic)
```

**Verification:**

```sh
otool -l build/scc | grep -A3 LC_UUID   # must show 00000000-0000-0000-0000-000000000000
                                         # OR the command must be absent entirely
```

### Source 3: Symbol Table Order

**What it is.** Linkers iterate over symbol tables to produce the output. If the
symbol table is stored in a hash map (common in fast linkers), the iteration order
is non-deterministic across runs, JVM instances, or Python dict order (pre-3.7).
The output binary contains symbols in different orders -> different byte layout ->
different hash even with identical source.

**Symptom.** `nm scc1 | sort` matches `nm scc2 | sort`, but `nm scc1` does NOT
match `nm scc2` (ordering differs). Binary diff shows large sections of offset-
shifted content.

**Recipe - macOS ld64:**

```sh
# ld64 in modern Xcode uses a deterministic symbol table order when passed
# -reproducible (Xcode 15+):
LDFLAGS += -Wl,-reproducible

# For older ld64, explicitly sort via -order_file:
# Generate the symbol order from an authoritative build:
nm -n build/scc | awk '{print $3}' | grep -v '^$' > build/scc.order
# Then pass to subsequent builds:
LDFLAGS += -Wl,-order_file,build/scc.order
```

**Recipe - GNU ld / lld (ELF):**

```sh
# Both GNU ld and lld produce deterministic symbol table order by default
# when passed --no-undefined-version --as-needed and a sorted input list.
# Ensure object files are passed in a deterministic order:
OBJS = $(sort $(wildcard src/*.o))   # GNU make 'sort' is deterministic
```

**Recipe - scld.py (SUPER C native linker):**

```python
# In scld.py: sort symbol table before output
symbols = sorted(symbol_table.items(), key=lambda kv: kv[0])
```

**Verification:**

```sh
# Check that symbol lists are identical (not just set-equal):
nm build/scc_run1 > /tmp/sym1.txt
nm build/scc_run2 > /tmp/sym2.txt
diff /tmp/sym1.txt /tmp/sym2.txt   # must be empty (exact order match)
```

### Source 4: Non-Zero Padding Bytes

**What it is.** Section-alignment padding in object files and final binaries is
typically filled with zero bytes. However, some toolchains or compilers leave
uninitialized padding (tail bytes of a section after content, alignment gaps
between sections, `__LINKEDIT` trailing bytes) as-is - containing whatever
was in the memory allocator buffer at the time. These bytes differ between
runs even though they carry no semantic content.

**Symptom.** Binary diff shows short runs of random-looking bytes at section
boundaries, consistent with alignment gaps. `otool -l` shows a section SIZE
larger than the sum of symbols within it. `xxd scc1 | tail` shows non-zero
bytes at offsets after the last symbol.

**Recipe - Mach-O (macOS):**

```sh
# Pass -Wl,-zero_fill_bss to linker (BSS is already zero by ABI,
# but pad bytes in __TEXT need explicit zeroing in some ld64 versions):
LDFLAGS += -Wl,-dead_strip          # removes unreachable sections (reduces padding)

# For __LINKEDIT trailing bytes, use llvm-objcopy to zero-pad explicitly:
llvm-objcopy --set-section-alignment __TEXT,__text=16 build/scc
```

**Recipe - ELF (Linux):**

```sh
# GNU ld: pass --gc-sections to eliminate dead padding + -z,relro
LDFLAGS += -Wl,--gc-sections

# Use objcopy to strip unneeded sections:
objcopy --strip-unneeded build/scc build/scc.stripped
```

**Recipe - general (any toolchain):**

```sh
# After linking, explicitly zero all padding bytes using a post-process script:
# This is the safest approach if linker flags are insufficient.
python3 tools/zero_pad_sections.py build/scc   # see below for reference impl
```

**Reference `zero_pad_sections.py` skeleton:**

```python
#!/usr/bin/env python3
"""Zero-fill all padding bytes between sections in a Mach-O or ELF binary."""
import sys, struct

def zero_fill_macho(path):
    with open(path, 'r+b') as f:
        data = bytearray(f.read())
    # For each LC_SEGMENT_64: find offset + size of each section,
    # then zero bytes from (last_section_end) to (segment_offset + segment_filesize)
    # ... (iterate load commands) ...
    with open(path, 'wb') as f:
        f.write(data)

if __name__ == '__main__':
    zero_fill_macho(sys.argv[1])
```

**Verification:**

```sh
# Confirm no non-zero bytes in alignment gaps:
# After fix, both builds must hash identically:
sha256sum build/scc_run1 build/scc_run2  # must print the same hash twice
```

## Complete Reproducibility Checklist

Run before every bootstrap-closure attempt:

```sh
# 1. Environment setup
export SOURCE_DATE_EPOCH=1700000000   # fixed epoch
export ZERO_AR_DATE=1                 # for ar archives

# 2. Build flags (add to Makefile or build script)
# LDFLAGS += -Wl,-no_uuid            # Mach-O: suppress UUID
# LDFLAGS += -Wl,-reproducible       # Mach-O: deterministic symbol order (Xcode 15+)
# LDFLAGS += -Wl,--build-id=none     # ELF: suppress build-ID

# 3. Symbol table: verify scld.py sorts symbols before emit
grep -n "sort" tools/scld.py         # must show symbol-table sort call

# 4. Two-run identity gate
make clean && make && sha256sum build/scc > /tmp/r1.txt
make clean && make && sha256sum build/scc > /tmp/r2.txt
diff /tmp/r1.txt /tmp/r2.txt && echo "PASS: reproducible" || echo "FAIL: non-deterministic"
```

## Diagnosing a Hash Drift Failure

When `sha256(scc1) != sha256(scc2)` and the source has not changed:

### Step 1: Narrow to a source

```sh
# Are the binaries the same size?
wc -c scc1 scc2

# Is it a timestamp? (compare build timestamps via strings)
strings scc1 | grep -E "[0-9]{4}-[0-9]{2}-[0-9]{2}|[0-9]{10}"
strings scc2 | grep -E "[0-9]{4}-[0-9]{2}-[0-9]{2}|[0-9]{10}"

# Is it a UUID? (Mach-O)
otool -l scc1 | grep -A3 LC_UUID
otool -l scc2 | grep -A3 LC_UUID

# Is it symbol order?
nm scc1 > /tmp/sym1.txt && nm scc2 > /tmp/sym2.txt
diff /tmp/sym1.txt /tmp/sym2.txt | head -20

# Is it padding? (look at diff of hexdumps near section boundaries)
xxd scc1 > /tmp/hex1.txt && xxd scc2 > /tmp/hex2.txt
diff /tmp/hex1.txt /tmp/hex2.txt | head -40
```

### Step 2: Apply the matching recipe

One of the 4 sources above will match the diff pattern. Apply the corresponding
recipe and re-run the two-run identity gate.

### Step 3: Add a gate

Every fix deserves a gate entry so the regression cannot silently re-appear:

```sh
# In scripts/gates/reproducible_build_gate.sh:
export SOURCE_DATE_EPOCH=1700000000 ZERO_AR_DATE=1
make clean && make && sha256sum build/scc > /tmp/rb_r1.txt
make clean && make && sha256sum build/scc > /tmp/rb_r2.txt
diff /tmp/rb_r1.txt /tmp/rb_r2.txt \
    && { echo "PASS: reproducible build identity"; exit 0; } \
    || { echo "FAIL: non-deterministic output"; exit 1; }
```

Register this gate in `gold_gate.sh` so it runs with every CI pass.

## SIGKILL Gate Flake vs Real Non-Determinism

A gold_gate timeout kill (SIGKILL from the perl-alarm wrapper) can masquerade
as a hash-drift failure if the killed build left a partial artifact on disk. The
next build then links against a corrupted intermediate rather than a fresh one.

**Diagnostic:**

```sh
# Check if the previous build was killed mid-flight:
file build/scc         # "truncated or malformed" indicates partial artifact
ls -la build/*.o | sort -k5 -n | tail  # look for a suspiciously small .o
```

**Rule.** Always `make clean` between the two runs of the identity gate.
Never use `make` (incremental) for reproducibility testing - incremental builds
can inherit state from a killed prior build.

**Cross-reference.** `sc-empirical-surface-probe` G (State-Dependent Transient
Behavior) and `gate-harness-process-isolation` for the general SIGKILL harness
pattern. L-W5-5 from Wave 5: "gold_gate flake masquerades as deterministic failure
when SIGKILL harness is not used."
