# Seal Reconciliation Discipline

## When to invoke

Triggers (any of):
- "seal verification"
- "multiple agents conflicting verdicts"
- "evidence conflict"
- "two readings of same commit"
- "audit conflict"
- "before declaring sealed"
- An audit cycle deployed >5 parallel agents and any two report different counts, statuses, or PASS/FAIL verdicts on the same artifact.
- A previously-green gate now reads red (or vice versa) on the same commit hash.
- The seal claim depends on a gate whose result has been read by two or more independent observers.

## Core directive

When parallel-agent audits produce conflicting verdicts on the same artifact, **all** conflicts must be reconciled via empirical re-reproduction BEFORE the seal is declared. The mere existence of a conflict invalidates any single agent's report as the basis for sealing.

Reconciliation is a structured 3-step protocol, not an arbitration vote. Majority-agent consensus is **not** sufficient - the actual cause of the disagreement (ordering, environment, temporality, process leak) must be identified and documented in the seal report.

## Detection symptoms

1. Two audit reports on the same commit cite the same gate but with different counts (`5/0` vs `3/3`).
2. An agent reports an artifact missing; another reports it present.
3. Counts that "should be deterministic" vary across agents' runs.
4. The first agent to report sets the verdict; later agents' contradictory findings are quietly ignored.
5. The seal report cites one agent's run and omits the conflicting agent's run.

## Required protocol: 3-step reconciliation

### Step 1 - Clean-state reproduction

Open a fresh shell. No environment from prior work. Explicitly clear any `/tmp/` artifacts the suspect gate might consume. Run the suspect gate. Record:
- exit code
- count line (`PASS X/Y`)
- whether the artifact path was opened (via `strace -e openat` / `dtruss`)
- whether any prereq files were missing

This isolates the **artifact-bare** behavior.

### Step 2 - Ordered-state reproduction

In a second fresh shell, run the prereq chain explicitly: `producer_gate.sh` then `consumer_gate.sh`. Record the same four data points. This isolates the **artifact-present** behavior.

Compare Step 1 and Step 2:
- If counts differ -> the gate is silent-pass-vulnerable (invoke `gate-auto-bootstrap-discipline`).
- If counts match but exit codes differ -> the gate has loose exit semantics.
- If artifact `openat` is absent in Step 1 but present in Step 2 -> confirmed prereq dependency, must be documented.

### Step 3 - Document the delta in the seal report

The seal report MUST contain a `## Reconciliation` section listing:
- Each agent's reported verdict, with timestamp and commit.
- The clean-state result.
- The ordered-state result.
- The identified cause (ordering / environment / process leak / temporal).
- The canonical reproduction recipe - exact commands that yield the canonical verdict, runnable by any future auditor.

No seal is declared without this section when an agent conflict has been detected.

## Anchor episode

**SUPER C v31.5.H Wave 1 (sealed 2026-05-27, v31.5H-self-host-self-compile-seal, commit 59f44e9c).**

Wave 1 dispatched 7 parallel audit agents. Main-thread auditor read `scc1_scc2_equivalence_gate` as `PASS 5/0`. Flake-agent (Wave 1.05), running in a freshly-spawned shell, read `FAIL: scc_via_scc_h4_proof artifact missing` followed by - confusingly - `PASS 5/0`. Two contradictory readings of the same commit.

Naive reconciliation (majority vote) would have ratified the main-thread `5/0` reading and sealed v31.5.H on a silent-pass gate. Instead:

1. **Step 1 (clean-state).** Empty `/tmp/`, run gate alone -> confirmed silent-pass: gate prints FAIL but counter shows `5/0`, exit 0.
2. **Step 2 (ordered-state).** Run `chain_gate.sh` + `h4_gate.sh` first to populate the artifact, then run equivalence gate -> confirmed real `5/0`, exit 0, `openat` of artifact present.
3. **Step 3 (document).** Seal report carried explicit `## Reconciliation` section with both readings, cause (silent-pass on missing prereq), and the canonical recipe (ordered run).

Outcome: the gate was patched with the auto-bootstrap header (see `gate-auto-bootstrap-discipline`); the seal was declared on the **patched** gate, not the silent-pass one. Without this skill, v31.5.H would have sealed on the unpatched gate and the silent-pass defect would have propagated to later sprints undetected.

## Recipe - the 3-step protocol in operational form

```
# Step 1: Clean-state reproduction
cd $(mktemp -d)
git clone --depth 1 <repo> .
git checkout <commit>
rm -rf /tmp/<suspect_artifact_paths>
strace -e openat ./<gate> 2>strace.log
echo "RC=$?"; grep openat strace.log | grep <artifact>

# Step 2: Ordered-state reproduction
cd $(mktemp -d)
git clone --depth 1 <repo> .
git checkout <commit>
./<producer_gate>
./<consumer_gate>
echo "RC=$?"

# Step 3: Write the Reconciliation section
cat >> seal_report.md <<EOF
## Reconciliation

| Agent | Verdict | Notes |
|-------|---------|-------|
| main-thread | PASS 5/0 | Run after prior chain populated /tmp |
| wave-1.05 flake-agent | FAIL+5/0 silent-pass | Run in clean shell |

**Clean-state result:** silent-pass (FAIL line + exit 0 + counter unchanged).
**Ordered-state result:** real PASS 5/0 with artifact open observed.
**Cause:** gate consumed prereq artifact without bootstrap or fail-loud guard.
**Canonical recipe:** run \`producer_gate\` then \`consumer_gate\`; or apply auto-bootstrap header (see \`gate-auto-bootstrap-discipline\`).
EOF
```

## Anti-patterns

- Sealing on majority-agent vote without re-running the contested gate.
- Quoting only the agent whose verdict matches the desired seal outcome.
- Treating `agent A says PASS, agent B says FAIL` as "flake" without diagnosing the cause (see `gate-contention-isolation`).
- Reconciling by re-running until you get the verdict you want, then sealing on that run.
- Omitting the `## Reconciliation` section when conflicts existed - silent reconciliation is undocumented reconciliation, which means future auditors cannot reproduce the call.
