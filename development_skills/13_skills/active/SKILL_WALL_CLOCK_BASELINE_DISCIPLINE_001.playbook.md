# Wall-Clock Baseline Discipline

**DIRECTIVE: Every optimization wave MUST begin with a per-component wall-clock
baseline. Without the baseline, you cannot measure the win, cannot verify no
regression, and cannot prioritize which component to fix first.**

---

## Core rule

Before touching any gate script, CI pipeline step, or build harness for speed:

1. Run a wall-clock sweep across ALL components. Emit a timestamped CSV.
2. Compute the distribution: min, max, p50, p90, p95, p99.
3. Produce a top-10 slowest list sorted descending.
4. Commit or record the baseline artifact (`audits/waveN/gate_baseline_times.csv`).
5. THEN begin optimization work - targeting the top-3 first.
6. After each fix, re-run the sweep. Confirm the fixed component moved down the list.

Skipping step 1 is an empirical violation. You are claiming an optimization without
the evidence to measure it.

## Recipe: 10-line bash CSV sweep

```bash
#!/usr/bin/env bash
# gate_sweep_timer.sh - measure wall-clock per gate, emit CSV to stdout
# Usage:
#   bash gate_sweep_timer.sh scripts/gates/*.sh > audits/waveN/gate_baseline_times.csv
#   bash gate_sweep_timer.sh scripts/gates/*.sh | tee audits/waveN/gate_baseline_times.csv

echo "gate,wall_seconds,exit_code"
for gate in "$@"; do
  name=$(basename "$gate" .sh)
  start=$(date +%s%N)
  bash "$gate" > /dev/null 2>&1
  rc=$?
  end=$(date +%s%N)
  ms=$(( (end - start) / 1000000 ))
  printf "%s,%.3f,%d\n" "$name" "$(echo "scale=3; $ms / 1000" | bc)" "$rc"
done
```

Notes:
- `date +%s%N` is nanosecond-resolution on Linux and macOS 12+. On older macOS,
  use `perl -MTime::HiRes=time -e 'printf "%.9f\n", time'` for sub-second resolution.
- Redirect gate stderr to `/dev/null` - you are measuring wall-clock, not output.
- Capture `exit_code` alongside time so failed gates are visible in the CSV.

## Recipe: awk distribution analysis

After collecting the CSV:

```bash
#!/usr/bin/env awk -f
# distribution.awk - compute min/max/p50/p90/p95/p99 from a one-column value list
# Usage: awk -F',' 'NR>1 {print $2}' gate_baseline_times.csv | awk -f distribution.awk

BEGIN { n = 0; sum = 0 }
{ vals[n++] = $1 + 0; sum += $1 + 0 }
END {
  # Sort in-place (gawk has asort; otherwise use sort | awk pipeline)
  for (i = 0; i < n-1; i++)
    for (j = i+1; j < n; j++)
      if (vals[i] > vals[j]) { t = vals[i]; vals[i] = vals[j]; vals[j] = t }

  printf "n     = %d\n", n
  printf "total = %.3f s\n", sum
  printf "min   = %.3f s\n", vals[0]
  printf "p50   = %.3f s\n", vals[int(n * 0.50)]
  printf "p90   = %.3f s\n", vals[int(n * 0.90)]
  printf "p95   = %.3f s\n", vals[int(n * 0.95)]
  printf "p99   = %.3f s\n", vals[int(n * 0.99)]
  printf "max   = %.3f s\n", vals[n-1]
  printf "mean  = %.3f s\n", sum / n
}
```

Invocation:

```bash
awk -F',' 'NR>1 {print $2}' audits/waveN/gate_baseline_times.csv \
  | awk -f distribution.awk
```

## Recipe: top-10 slowest list

```bash
# Sort by wall_seconds descending, print top 10 with percent-of-total
awk -F',' 'NR>1 {print $2, $1}' audits/waveN/gate_baseline_times.csv \
  | sort -k1 -rn \
  | head -10 \
  | awk '{
      sum += $1
      rows[NR] = $0
    }
    END {
      # re-read total from file (approximation: sum of top-10 only here)
      for (i = 1; i <= NR; i++) {
        split(rows[i], f, " ")
        printf "%-50s  %7.3f s\n", f[2], f[1]+0
      }
    }'
```

Or simpler - just sort the CSV and eyeball the top-10:

```bash
sort -t',' -k2 -rn audits/waveN/gate_baseline_times.csv | head -11
```

## Baseline artifact format

Record the sweep in a committed artifact before any optimization work begins.
Filename: `audits/wave<N>/gate_baseline_times.csv`

```
# SUPER C Gate Suite Wall-Clock Baseline
# Date: 2026-05-27
# Commit: <SHA>
# Machine: <machine description>
# CHECK_GATE_ATTEMPTS: 3 (default - flake-tolerant)
#
gate,wall_seconds,exit_code
gold_gate,178.341,0
scc1_scc2_equivalence_gate,101.223,0
scc_via_scc_h4_gate,89.431,0
scc_smoke,42.119,0
...
```

This artifact is the falsifiable record. After optimization, produce
`audits/wave<N>/gate_post_optimization_times.csv` using the same sweep.
Diff the two to compute per-gate delta and total delta.

## Empirical anchor: SUPER C Wave 7

In SUPER C Wave 7, a full gate-suite wall-clock sweep was conducted before any
optimization work. Key findings:

| Gate | Baseline | After fix | Delta | Fix type |
|---|---|---|---|---|
| `gold_gate.sh` | 178s | 82s | -96s | `CHECK_GATE_ATTEMPTS=1` |
| `scc1_scc2_equivalence_gate.sh` | 101s | 0.278s | -101s | `kill_after.sh` adoption |
| `scc_smoke.sh` (via aggregator) | ~89s | ~45s | -44s | Sub-gate parallelization |
| Suite total (estimated) | ~1,376s | ~830s | -546s | Top-3 fixes |

The top-3 gates accounted for approximately 36% of total wall-clock. All three were
diagnosable only because the baseline CSV existed. Without it:

- The `kill_after.sh` speedup (363x) would have been invisible - the gate "felt"
  fast interactively because it had been running in a warm session with leftover
  scc processes. The baseline revealed the cold-run cost.
- The `CHECK_GATE_ATTEMPTS=1` win would have been attributed to the wrong cause -
  the gate was "slow because of flaky fixtures," not because the retry loop was
  running 3 serially on a deterministic failure.
- The parallelization opportunity would have been missed - the aggregator gate
  showed up at position 3 in the sorted list only because gates 1 and 2 were
  already optimized.

**The baseline made all three wins legible. Without it, only one of the three
would have been discovered.**

## Integration with other skills

### Before calling `apex-parallel-deploy`

Run the baseline sweep to identify which gates are serial aggregators (candidates
for task-level parallelization). The parallel-deploy skill's independence check
(Step 3) is most effective when you have measured which tasks dominate wall-clock.

### Before calling `process-group-kill-discipline`

If the sweep shows a gate is unexpectedly slow (slower than its logic warrants),
process accumulation is a leading hypothesis. A gate that should take 2s but takes
101s is likely spawning and not cleaning up child processes. The baseline flags
it; `process-group-kill-discipline` fixes it.

### Before calling `compiler-discipline` 14

The `CHECK_GATE_ATTEMPTS=1` envvar is most valuable when you know which gates have
retry loops and how long those loops cost. The baseline CSV, filtered to gates with
high wall-clock times and `exit_code=0` (indicating they passed, so retries
represent waste on intermittent tolerance, not real failures), identifies the targets.

## Pre-optimization checklist

Before starting any harness optimization wave:

```
[ ] Swept all gates with gate_sweep_timer.sh
[ ] Emitted CSV to audits/waveN/gate_baseline_times.csv
[ ] Computed distribution (min/max/p50/p90/p95/p99)
[ ] Produced top-10 slowest list
[ ] Computed percentage contribution of top-3 to total wall-clock
[ ] Identified root cause hypothesis per top-3 (accumulation / retry / serial aggregator)
[ ] Committed baseline artifact (or recorded it as an audit artifact)
[ ] THEN began optimization work - not before
```

Skipping any step before "THEN began optimization work" is an empirical violation.

---

## Anti-patterns

| Anti-pattern | Why it fails |
|---|---|
| "This gate feels slow" -> optimize it | Selection bias. The slow-feeling gate may not dominate total wall-clock |
| Optimize first, measure after | Cannot distinguish win from regression; no falsifiable record |
| Measure once in a warm session | Process accumulation from prior runs inflates or deflates wall-clock; use cold-session measurement |
| Record only total suite time | Cannot identify which gate to fix; cannot verify per-gate wins |
| Estimate from memory ("gold_gate takes ~2 minutes") | Memory underestimates warm-session runtimes; baseline shows cold truth |
| Skip baseline for "obvious" optimizations | The Wave 7 kill_after.sh win (363x) was not obvious before measurement |
