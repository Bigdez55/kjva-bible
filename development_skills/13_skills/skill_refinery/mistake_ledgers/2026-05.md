# Mistake Ledger — 2026-05 (kjva-bible)

## 2026-05-26 — engine.is_ready() blocked retrieval

**Context:** SLICE-0001 implementation review. Pre-slice `backend/routes/complete.py`
called `engine.is_ready()` at the top of the handler and raised 503 immediately when
weights were absent. This made the entire `/api/complete` endpoint unreachable in CI,
in linux/amd64 containers, and on any host without the 72 MB gitignored weights —
even for prompts that retrieval could answer perfectly (e.g. "John 3:16").

**Root cause:** The original implementation pre-dated the retrieval layer. When the
retrieval branches were added, the readiness gate was not moved.

**Remediation:**
- ADR-0003 authored to formalize retrieval-first ordering.
- `engine.is_ready()` check moved below ref-match and prefix-match branches.
- `mlx` import made optional in `inference.py` (try/except ImportError); `is_ready()`
  returns False when MLX absent.
- New tests `test_complete_direct_ref_no_weights`, `test_complete_range_ref_no_weights`,
  `test_complete_prefix_match_no_weights` lock in the behavior.

**Recurrence check:** First occurrence. No skill update needed yet. If a similar
"gate-before-handler" bug recurs, promote a `SKILL_GATE_AFTER_BRANCHES_001` or similar.

## 2026-05-26 — MLX hard-imported in inference.py

**Context:** Even with the retrieval reordering above, the import of `inference.py`
itself failed on environments without MLX, breaking `routes/complete.py` import,
breaking `main.py` import, and breaking pytest collection (`ModuleNotFoundError: No
module named 'mlx'`).

**Root cause:** Top-level `import mlx.core` without graceful degradation.

**Remediation:** Wrapped MLX imports in try/except, set `_MLX_AVAILABLE` flag,
gated `_load()` and `is_ready()` accordingly. Documented the constraint in ADR-0001
and ADR-0003. Container runs retrieval-only on linux/amd64 by design (Dockerfile
strips `mlx` from requirements).

**Recurrence check:** First occurrence. Same family as the readiness bug above —
both are "platform-coupled dependency leaking into general-purpose code paths."
If a third occurrence appears, promote a skill around optional-runtime-imports.
