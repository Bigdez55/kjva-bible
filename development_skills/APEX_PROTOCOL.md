# Apex Development Skills Protocol v2.1

## Purpose

The Apex Development Skills Protocol converts raw ideas into production ready software through structured intent, specs, architecture, diagrams, vertical slices, proof, deployment, and feedback.

## System doctrine

1. Intent compiles into structure.
2. Structure compiles into architecture.
3. Architecture compiles into diagrams.
4. Diagrams guide implementation.
5. Implementation produces evidence.
6. Evidence updates documentation.
7. Mistakes improve skills.
8. Repositories feed the central intelligence system.
9. Every project can deploy a preview early.
10. Every repository maintains a living digital twin.

## Clarification policy

Agents must not stop for clarification unless all conditions are true:

1. The missing answer blocks progress.
2. The answer cannot be inferred from existing sources.
3. The decision materially affects architecture, security, money, compliance, data, deployment, or user experience.

If those conditions are not all true, the agent must make a conservative assumption, record it, continue, and mark it for later review.

## Diagram first rule

No major feature should be built until the following maps exist or are explicitly marked not applicable:

1. Feature context map
2. Component impact map
3. Data flow map
4. Execution sequence map
5. Dependency impact map
6. Test coverage map
7. Deployment impact map

## Skill evolution rule

Every repeated mistake must create or improve a skill.

## Repo twin rule

Every child repository must maintain a living repo twin with:

1. current truth
2. last known state
3. architecture snapshot
4. component graph
5. dependency graph
6. diagram registry
7. skill usage
8. agent activity
9. sync status

## Bookworm rule

Bookworm indexes the living truth of the repository. It should always prefer current truth state, architecture models, source code, diagrams, and proof matrix files over generated documentation.
